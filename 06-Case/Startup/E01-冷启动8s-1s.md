# 3. 优化前：8s 启动的"问题诊断"

## 3.1 bootstat 总耗时

```bash
adb shell dumpsys bootstat | grep -E "boot_total|boot_init|boot_system|boot_zygote"
```

**输出**：
```
boot_init_total_time: 4500ms
boot_zygote_time: 1800ms
boot_system_server_time: 12500ms
boot_total_time: 18500ms
```

**问题诊断**：
- `boot_init_total_time: 4500ms` ← 异常（典型 1500-2500ms）
- `boot_zygote_time: 1800ms` ← 异常（典型 800-1200ms）
- `boot_system_server_time: 12500ms` ← 异常（典型 5000-10000ms）
- `boot_total_time: 18500ms` ← 异常（典型 5000-15000ms）

## 3.2 perfetto 抓取 4 层栈

**配置**：
```python
# perfetto 启动期配置
duration_ms: 30000
enabled_categories: "boot" + "am" + "wm" + "view"
```

**抓取**：
- [见 D01-Perfetto Boot Trace](../../02-Symptom/S11-Startup/D-启动工具/D01-Perfetto-Boot-Trace抓全栈启动时序.md)

**解析**：
- 找到 SystemServer run() 开始 → AMS ready 耗时 12.5s
- 找到 PMS 启动耗时 3s（典型 1.5s）
- 找到某 OEM service 启动耗时 1.5s

## 3.3 bootchart 抓 initcall

**抓取**：
- [见 D03-bootchart 工具链](../../02-Symptom/S11-Startup/D-启动工具/D03-bootchart工具链.md)

**解析**：
- display_init 200ms（典型）
- gpu_init 300ms（典型）
- wlan_init 200ms（典型）

**initcall 总耗时**：1.5s（典型）

## 3.4 8s 启动的 5 大根因

```
   ┌────────────────────────────────────────────────────────────┐
   │  8s 启动的 5 大根因（按耗时排序）                            │
   └────────────────────────────────────────────────────────────┘

   1. App 冷启动 3s（占比 37.5%）
      └─ onCreate 主线程 IO 1.5s
      └─ dex2oat 解释执行慢 1s
      └─ 第一帧 measure + layout + draw 0.5s

   2. SystemServer 启动 2s（占比 25%）
      └─ PMS 扫描 1s
      └─ 5 个 OEM 定制 service 启动 1s

   3. Zygote fork 1.5s（占比 18.75%）
      └─ ART VM 初始化 0.5s
      └─ 预加载类 5000 个 0.5s
      └─ 预加载资源 50 个 0.5s

   4. Init 阶段 1s（占比 12.5%）
      └─ init.rc 解析 300ms
      └─ servicemanager 启动 200ms
      └─ 关键 service 启动 500ms

   5. BootLoader + Kernel 0.5s（占比 6.25%）
      └─ 典型耗时
```

# 4. 优化策略 1：dex2oat AOT 编译

## 4.1 优化前：dex2oat 模式为 quicken

```bash
adb shell dumpsys package com.example.im | grep -A 5 "Dexopt"
# 输出：mode=quicken
```

**问题**：quicken 模式只做部分优化，启动期需解释执行 1s。

## 4.2 优化：dex2oat 模式改为 speed-profile

```bash
# 手动触发 dex2oat speed-profile 模式
adb shell cmd package compile -f -m speed-profile com.example.im

# 验证
adb shell dumpsys package com.example.im | grep -A 5 "Dexopt"
# 输出：mode=speed-profile
```

**原理**：
- speed-profile 模式根据用户实际使用模式（profile）优化
- 提前 AOT 编译热点代码
- 启动期直接执行机器码

## 4.3 量化收益

| 指标 | 优化前（quicken）| 优化后（speed-profile）| 收益 |
|:-----|:---------------|---------------------|:----:|
| **App 冷启动** | 3s | 1.8s | **1.2s（40%）** |
| **onCreate 耗时** | 1.5s | 0.8s | 0.7s |
| **解释执行** | 1s | 0.3s | 0.7s |

> **所以呢**：dex2oat speed-profile 收益 1.2s（占 8s 15%）——**性价比最高**。

## 4.4 风险

- 🟡 **安装时间增加**：speed-profile 让 APK 安装变慢 10s
- 🟡 **profile 数据需要积累**：需要 3-5 次启动数据

## 4.5 解决方案

- 通过 OTA 升级首次安装时使用 speed-profile
- profile 数据自动积累（无需用户操作）

# 5. 优化策略 2：Zygote 预加载优化

## 5.1 优化前：preloaded-classes 5000 个

```bash
adb shell cat /system/etc/preloaded-classes | wc -l
# 输出：5000
```

**问题**：预加载 5000 个类 → Zygote 启动 1.5s（典型 1s）。

## 5.2 优化：精简 preloaded-classes 到 3000 个

```bash
# 1. 抓取实际使用的类
adb shell setprop dalvik.vm.profilebootclasspath true
adb shell am force-stop com.example.im
adb shell am start com.example.im
sleep 5

# 2. 抓取 profile
adb pull /data/local/tmp/boot-profile /tmp/

# 3. 精简 preloaded-classes
# 移除未使用的 2000 个类
```

**原理**：
- Zygote fork 时预加载的类
- 精简后 Zygote 启动 1s

## 5.3 量化收益

| 指标 | 优化前（5000 类）| 优化后（3000 类）| 收益 |
|:-----|:---------------|----------------|:----:|
| **Zygote 启动** | 1.5s | 1.2s | **0.3s（20%）** |
| **App fork 耗时** | 300ms | 200ms | 0.1s |
| **类加载时间** | 500ms | 300ms | 0.2s |

> **所以呢**：Zygote 预加载精简收益 0.3s（占 8s 3.75%）——**小但易实现**。

## 5.4 风险

- 🟡 **Zygote 启动慢 vs App 启动快 trade-off**——需平衡
- 🟡 **类被修改需重启**

# 6. 优化策略 3：SystemServer 服务按需启动

## 6.1 优化前：5 个 OEM 定制 service 启动 1s

```bash
adb shell dumpsys activity services | grep "OEMService" | head -5
# 输出：5 个 OEM service
```

**问题**：5 个 OEM 定制 service 串行启动 1s。

## 6.2 优化：关闭 3 个非关键 service + 延迟 2 个

```java
// SystemServer.java
private void startOtherServices() {
    // 关闭 3 个非关键 OEM service
    // mSystemServiceManager.startService(OEMAnalyticsService.class);  // 关闭
    // mSystemServiceManager.startService(OEMTelemetryService.class);  // 关闭
    // mSystemServiceManager.startService(OEMLocationService.class);  // 关闭
    
    // 延迟 2 个低优先级 OEM service
    CompletableFuture.runAsync(() -> {
        mSystemServiceManager.startService(OEMAdService.class);
        mSystemServiceManager.startService(OEMSocialService.class);
    });
    
    // 启动关键 OEM service
    mSystemServiceManager.startService(OEMCameraService.class);
    mSystemServiceManager.startService(OEMNetworkService.class);
}
```

## 6.3 量化收益

| 指标 | 优化前 | 优化后 | 收益 |
|:-----|:-------|:-------|:----:|
| **SystemServer 启动** | 12.5s | 11s | **1.5s（12%）** |
| **5 个 OEM service** | 1s | 0.5s（启动 2 个）| 0.5s |
| **PMS 扫描** | 3s | 2.5s | 0.5s |
| **关键 service 启动** | 1s | 0.5s（并发）| 0.5s |

> **所以呢**：服务按需启动收益 1.5s（占 8s 18.75%）——**主战场**。

## 6.4 风险

- 🔴 **关闭必要 service = 功能缺失**
- 🟡 **依赖关系**：需明确每个 service 的依赖

# 7. 优化策略 4：渲染优化

## 7.1 优化前：onCreate 主线程 IO 1.5s

```bash
adb shell logcat -d -s ActivityTaskManager:V | grep "Displayed"
# 输出：Displayed 时间 1.5s
```

**问题**：onCreate 主线程 SharedPreferences + 数据库查询。

## 7.2 优化：onCreate 异步化 + SplashScreen

```java
// 优化前
@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_main);
    
    // 🔴 反例：主线程 IO
    SharedPreferences prefs = getSharedPreferences("config", MODE_PRIVATE);
    String token = prefs.getString("token", "");  // 1s
    
    initDatabase();  // 500ms
}

// 优化后
@Override
protected void onCreate(Bundle savedInstanceState) {
    // 🟢 SplashScreen API
    SplashScreen splashScreen = getSplashScreen();
    splashScreen.setOnExitAnimationListener(splashView -> {
        // 启动完成后的动画
    });
    
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_main);
    
    // 🟢 异步化
    CompletableFuture.runAsync(() -> {
        SharedPreferences prefs = getSharedPreferences("config", MODE_PRIVATE);
        String token = prefs.getString("token", "");
    });
    
    CompletableFuture.runAsync(() -> {
        initDatabase();
    });
}
```

## 7.3 量化收益

| 指标 | 优化前 | 优化后 | 收益 |
|:-----|:-------|:-------|:----:|
| **onCreate 耗时** | 1.5s | 0.3s | **1.2s（80%）** |
| **第一帧** | 500ms | 100ms | 0.4s |
| **黑屏时间** | 500ms | 0ms | 0.5s |

> **所以呢**：渲染优化收益 1.2s（占 8s 15%）——**根治黑屏 + 加速**。

## 7.4 风险

- 🟡 **SplashScreen 闪屏**：需正确配置
- 🟡 **异步化顺序**：确保业务正确

# 8. 优化策略 5：内核精简（OEM 主导）

## 8.1 优化前：200+ initcall 1.5s

```bash
grep "initcall.*returned.*[0-9]\{4,\}" /tmp/initcall.log | sort -k 7 -n -r | head -20
# 输出：200+ initcall 总耗时 1.5s
```

**问题**：200+ initcall 串行执行 1.5s。

## 8.2 优化：关闭不必要 initcall + 移到 late_initcall

```c
// drivers/foo/foo.c
// 优化前：device_initcall

// 优化后：late_initcall（晚启动，并行）
```

| **display_init** | 200ms | 150ms | 50ms |
| **gpu_init** | 300ms | 250ms | 50ms |
| **wlan_init** | 200ms | 100ms | 100ms |

> **所以呢**：内核精简收益 0.3s（占 8s 3.75%）——**OEM 主导**。

## 8.4 风险

- 🔴 **关闭必要 initcall = 硬件不工作**

# 11. 风险地图（与 Stability S 系列联动 · 强制）

> **本节是 v4 强制要求**——8s → 1s 优化的风险。

## 11.1 5 大优化策略风险

| 策略 | 风险 | 严重等级 |
|:-----|:-----|:--------:|
| **dex2oat AOT** | 安装时间增加 | 🟡 中 |
| **Zygote 预加载** | Zygote 启动慢 | 🟡 中 |
| **SystemServer 服务按需** | 关闭必要 service | 🔴 高 |
| **渲染优化** | SplashScreen 闪屏 | 🟡 中 |
| **内核精简** | 关闭必要 initcall | 🔴 高 |

## 11.2 优化前/后对比风险

| 风险 | 触发条件 | 应对 |
|:-----|:---------|:-----|
| **功能缺失** | 关闭必要 service | 灰度 + 监控 |
| **黑屏** | SplashScreen 配置错 | 灰度 + 监控 |
| **崩溃** | 关闭 initcall | OEM 主导 |
| **安装变慢** | dex2oat everything | OTA 升级 |

## 11.3 4 大风险控制

| 控制 | 措施 |
|:-----|:-----|
| **灰度发布** | 1% → 5% → 20% → 100% |
| **APM 监控** | bootstat 接入 APM |
| **回退机制** | 异常时回退到优化前 |
| **用户反馈** | 低端机 + 弱网环境专项 |
