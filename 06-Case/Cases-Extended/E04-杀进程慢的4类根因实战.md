# 1. 杀进程慢：4 类根因全景

> **铁律**：杀进程慢的根因**不是 1 个**，是 **4 大类 + N 小类**。本篇覆盖 4 大类。

```
杀进程慢
   ├── 1. 信号丢失（signal loss）        —— 经典但少见的"幽灵"
   ├── 2. 锁竞争（lock contention）      —— 高频 + 难定位
   ├── 3. IO 阻塞（IO block）            —— 看似无关，实则最坑
   └── 4. 内存杀慢（lowmemorykiller 慢）  —— 内存治理失效
```

| 类别 | 占比 | 难度 | 关键信号 |
|:-----|:----:|:----:|:---------|
| 信号丢失 | 10% | 高 | 进程"卡死但没死" |
| 锁竞争 | 35% | 中 | `binder: B hanging` 日志 |
| IO 阻塞 | 30% | 中 | `D` 状态进程 |
| 内存杀慢 | 25% | 低 | `lowmemorykiller` 日志 |

# 3. 案例 1：信号丢失（Signal Loss）

## 3.1 现象

- 用户反馈：某 App 杀进程耗时 8 秒（正常应 < 500ms）
- 监控：APM 告警杀进程 P99 超过 2s
- 频率：每天约 1000+ 次

## 3.2 排查过程

### 抓 bugreport 发现

```
01-01 00:00:01.000  system_server  Process.killProcess()
01-01 00:00:01.500  kernel         send_sigkill to PID 12345
01-01 00:00:01.500  kernel         sigkill delivered
01-01 00:00:08.000  system_server  am force-stop completed
                                              ↑ 7.5s 空白！
```

**关键观察**：`am force-stop` 耗时 7.5s，但 `sigkill` 1.5s 就送达了 —— **进程没死透**。

### 抓 strace 确认

```bash
strace -p 12345 -t -f -e trace=signal,exit,exit_group
```

```
00:00:01.500 < ... sigkill received
00:00:01.501 > ... in futex_wait
00:00:01.502 > ... futex_wait
00:00:08.000 < ... sigkill received (second time)
00:00:08.001 > ... exit_group
```

**根因确认**：进程在 `futex_wait`（内核 futex 等待）阻塞，**第一次 sigkill 没打断**。

## 3.3 5 Whys 根因分析

1. **Why 1**：为什么杀进程耗时 7.5s？—— 进程在 `futex_wait` 阻塞
2. **Why 2**：为什么 `futex_wait` 不响应 sigkill？—— futex 等待是 **MUST_RESTART** 系统调用，信号来时不退出
3. **Why 3**：为什么会有 `futex_wait`？—— App 在主线程用了 `Object.wait()` / `Lock.lock()`
4. **Why 4**：为什么 App 主线程会 wait？—— **异步回调的同步等待**（错误用法）
5. **Why 5**：为什么会有这种错误用法？—— **SDK 文档缺失 + Code Review 没卡**

## 3.4 修复

```java
// 错误写法（主线程 wait 锁）
synchronized (lock) {
    lock.wait();  // ❌ 永远不响应 sigkill
}

// 正确写法（用 Condition + 超时）
Lock lock = new ReentrantLock();
Condition cond = lock.newCondition();
lock.lock();
try {
    cond.await(5, TimeUnit.SECONDS);  // ✅ 超时退出
} finally {
    lock.unlock();
}
```

## 3.5 治理

| 动作 | 责任方 | 时间 |
|:-----|:-------|:-----|
| SDK 文档更新：禁止主线程 `wait()` | SDK 团队 | 1 周 |
| Code Review 增加：检测 `wait()` 出现在主线程 | 工具组 | 2 周 |
| 静态扫描规则：lint 检测主线程 wait | 工具组 | 1 月 |
| APM 增加：`futex_wait` 耗时告警 | APM 团队 | 2 周 |

## 3.6 给"如果是你"的 SOP

> 看到杀进程 5s+，立即：
> 1. 抓 bugreport，看 `am force-stop` 与 `sigkill` 时间差
> 2. 时间差 > 1s = 进程没死透 = **信号丢失**
> 3. strace 看是不是在 `futex_wait` / `epoll_wait` / `read` 等 MUST_RESTART 系统调用
> 4. 让 App 团队改主线程锁等待模式

# 4. 案例 2：锁竞争（Lock Contention）

## 4.1 现象

- 用户反馈：杀进程后偶尔出现"ANR 弹窗"
- 监控：杀进程 P99 抖动到 4s
- 频率：每天约 200 次

## 4.2 排查过程

### logcat 发现

```
01-01 00:00:01.000  ProcessList: killProcessesForRemovedTask
01-01 00:00:01.100  ActivityManager: Waiting for Activity destruction
01-01 00:00:02.500  ActivityManager: Activity destroyed
01-01 00:00:04.000  ProcessList: process killed
```

**关键观察**：`ActivityManager: Waiting for Activity destruction` 耗时 1.4s —— 等待 Activity 销毁。

### 抓 system_server traces

```
"ActivityManager" daemon prio=5 tid=27
  at android.os.BinderProxy.transactNative(Native method)
  at android.app.ActivityManagerProxy.activityDestroyed(ActivityManagerNative.java:...)
  at com.android.server.am.ActivityRecord.activityDestroyed(ActivityRecord.java:...)
  at com.android.server.am.ActivityStack.activityDestroyedLocked(ActivityStack.java:...)
  at android.os.MessageQueue.nativePollOnce(Native method)
  ...
  - waiting on <0x12345678> (a java.lang.Object)  ← 卡在这里
  - locked <0x87654321> (a com.android.server.am.ActivityManagerService)
```

**根因确认**：system_server 在等 Binder 调用，而 **Binder 队列已满**。

## 4.3 5 Whys 根因分析

1. **Why 1**：为什么杀进程卡 4s？—— `activityDestroyed` 等待 1.4s
2. **Why 2**：为什么等 `activityDestroyed`？—— **Activity.onDestroy 没回调**（App 主线程卡住）
3. **Why 3**：为什么 App 主线程卡住？—— **App 持有 AMS 锁 + 在等某个 Binder 回调**
4. **Why 4**：为什么有锁竞争？—— **App 进程与 system_server 互等锁**（死锁边缘）
5. **Why 5**：为什么会有这种死锁？—— **App 通过 AIDL 调系统服务，系统服务又回调 App**（设计错误）

## 4.4 修复

```java
// App 端：错误（同步等系统服务回调）
public void onDestroy() {
    super.onDestroy();
    systemService.doSomethingSync();  // ❌ 等系统服务回调
}

// App 端：正确（异步）
public void onDestroy() {
    super.onDestroy();
    new Thread(() -> systemService.doSomething()).start();  // ✅ 异步
}
```

## 4.5 治理

| 动作 | 责任方 | 时间 |
|:-----|:-------|:-----|
| 主线程检测工具：检测 onDestroy 同步阻塞 | 工具组 | 2 周 |
| AIDL 调用审计：列出所有同步 AIDL 调用 | 全团队 | 1 月 |
| 杀进程超时强制 kill（不等待）| framework | 1 月 |

## 4.6 SOP

> 看到杀进程 2-5s 抖动，立即：
> 1. 看 logcat `Waiting for Activity destruction`
> 2. 抓 system_server traces 看 `waiting on` 哪个锁
> 3. 拉 App 端 onDestroy 实现的 Git blame
> 4. 让 App 团队把同步阻塞改成异步

# 5. 案例 3：IO 阻塞（IO Block）

## 5.1 现象

- 用户反馈：杀进程后偶发"卡死 10s+"
- 监控：杀进程 P99 抖动到 12s
- 频率：每周约 30 次

## 5.2 排查过程

### ftrace 发现

```
kworker/u8:2-1234  [000] d..1  1234.567890: sched_process_exit: comm=app_process pid=5678
kworker/u8:2-1234  [000] d..1  1234.567900: sched_wakeup: comm=logd pid=2345
```

**关键观察**：被杀进程 pid=5678，**触发的是 `sched_process_exit`**（不是正常的 `do_exit`）。

### 抓 strace 发现

```
00:00:00.000  recvfrom(8, ...)  = -1 EINTR (Interrupted system call)  ← 被 sigkill
00:00:00.001  recvfrom(8, ...)  = -1 EINTR
00:00:00.002  recvfrom(8, ...)  = -1 EINTR
... 重复 1000+ 次
00:00:10.000  exit_group(0)
```

**根因确认**：App 主线程在 `recvfrom` 系统调用死循环（EINTR 无限重试），**sigkill 不能立即打断**。

## 5.3 5 Whys 根因分析

1. **Why 1**：为什么杀进程 12s？—— 进程在 `recvfrom` 死循环
2. **Why 2**：为什么 `recvfrom` 不退出？—— **App 用了错误的 errno 处理**（EINTR 应退出或重启）
3. **Why 3**：为什么会有错误处理？—— **老代码从 Linux 移植，Android 没改**
4. **Why 4**：为什么没测过？—— **没有"杀进程兼容性"测试**
5. **Why 5**：为什么没测试？—— **没有 P0 杀进程压测 SOP**

## 5.4 修复

```cpp
// 错误写法（Linux 习惯）
while (true) {
    ssize_t n = recvfrom(fd, buf, sizeof(buf), 0, NULL, NULL);
    if (n < 0) {
        if (errno == EINTR) continue;  // ❌ Android 不能这么写
        break;
    }
    // ...
}

// 正确写法（Android 习惯）
while (true) {
    ssize_t n = recvfrom(fd, buf, sizeof(buf), 0, NULL, NULL);
    if (n < 0) {
        if (errno == EINTR) continue;  // ✅ 但要加超时
        if (errno == EAGAIN) break;    // ✅ 非阻塞返回
        break;
    }
    // ...
}
```

## 5.5 治理

| 动作 | 责任方 | 时间 |
|:-----|:-------|:-----|
| P0 杀进程压测 SOP：每发版前必跑 | 工具组 | 1 月 |
| C/C++ 代码静态扫描：检测 EINTR 死循环 | 工具组 | 1 月 |
| NDK 文档：明确 EINTR 处理 | 官方 | 持续 |

## 5.6 SOP

> 看到杀进程 5-15s，立即：
> 1. ftrace 看 `sched_process_exit` vs `do_exit`
> 2. strace 看是不是 `recvfrom/read/write` EINTR 死循环
> 3. 让 Native 团队改 EINTR 处理

# 6. 案例 4：内存杀慢（lowmemorykiller 慢）

## 6.1 现象

- 用户反馈：低端机杀进程后内存未及时释放
- 监控：内存峰值告警
- 频率：低端机 100% 出现

## 6.2 排查过程

### dumpsys meminfo 发现

```
Total RAM: 2GB
Free RAM: 200MB
Cached: 500MB
lmkd: last kill at 00:00:01 (PID 12345)
```

**关键观察**：`Cached: 500MB` —— **页缓存未及时回收**。

### ftrace 发现

```
lowmemorykiller-1234  [000] ....  1234.567890: mm_vmscan_direct_reclaim_begin
lowmemorykiller-1234  [000] ....  1234.567900: mm_vmscan_direct_reclaim_end
```

**关键观察**：`direct_reclaim` 耗时 10s —— **直接回收页缓存卡住**。

## 6.3 5 Whys 根因分析

1. **Why 1**：为什么内存杀慢？—— `direct_reclaim` 10s
2. **Why 2**：为什么 direct_reclaim 慢？—— **脏页太多，回写 IO 卡住**
3. **Why 3**：为什么脏页多？—— **App 写文件太频繁**（logcat、cache）
4. **Why 4**：为什么 App 写文件多？—— **没有限制写频率**
5. **Why 5**：为什么没限制？—— **没有"内存敏感"接口设计**

## 6.4 修复

```java
// App 端：错误（频繁写）
public void onSensorChanged(SensorEvent event) {
    logFile.write(event.values);  // ❌ 每秒 100 次
}

// App 端：正确（批量 + 节流）
private final Handler handler = new Handler();
public void onSensorChanged(SensorEvent event) {
    handler.removeCallbacks(logTask);
    handler.postDelayed(logTask, 1000);  // ✅ 1s 节流
}
private final Runnable logTask = () -> logFile.writeBatch(buffer);
```

## 6.5 治理

| 动作 | 责任方 | 时间 |
|:-----|:-------|:-----|
| 低端机检测：自动降低写频率 | framework | 1 月 |
| LMKD 参数优化：调低 `vmpressure` 阈值 | framework | 2 周 |
| 内存敏感度标注：App 标注 cache 等级 | 平台 | 持续 |

## 6.6 SOP

> 看到杀进程后内存未释放，立即：
> 1. `dumpsys meminfo` 看 Cached 是否过大
> 2. ftrace 看 `mm_vmscan_direct_reclaim` 耗时
> 3. 找高频写文件的 App，让其节流

# 7. 修复方案库（速查表）

| 根因 | 修复模式 | 通用度 | 复杂度 |
|:-----|:---------|:------:|:------:|
| **信号丢失** | 主线程 `wait` → `await(timeout)` | 高 | 低 |

| **IO 阻塞** | 关闭不必要 fd | 中 | 中 |
| **内存杀慢** | 高频写 → 节流 | 高 | 低 |
| **内存杀慢** | 调整 `vmpressure` 阈值 | 中 | 中 |

# 9. 5 条 Takeaway

1. **杀进程慢的根因分 4 大类**（信号丢失 / 锁竞争 / IO 阻塞 / 内存杀慢）—— 不要假设只有 1 种
2. **MUST_RESTART 系统调用是"幽灵"** —— `futex_wait` / `recvfrom` 等系统调用被 sigkill 后不退出，必须改用超时
3. **锁竞争的主战场是 system_server ↔ App** —— AIDL 同步调用 + Activity 销毁回调 = 高频互锁点
4. **IO 阻塞的 EINTR 是 Linux 习惯** —— Android NDK 必须显式处理 `errno == EINTR`
5. **内存杀慢的 90% 是脏页太多** —— 治本 = 让 App 节流写文件，治标 = 调 `vmpressure` 阈值
