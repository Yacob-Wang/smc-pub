## 一、痛点场景 - 国内 App 的隐私困境

### 1.1 国内 App 生态的特殊性

```
┌─────────────────────────────────────────────────────────────┐
│           国内 App 隐私困境的 4 大表现                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① 不给权限不让用                                            │
│     "拒绝 XX 权限将影响 App 的正常使用,请前往设置开启"        │
│     → 用户被迫授予位置/通讯录/存储等敏感权限                  │
│                                                             │
│  ② 后台偷偷读剪贴板                                          │
│     用户复制密码/银行卡号,App 在后台静默读取                  │
│     → 密码泄露风险                                           │
│                                                             │
│  ③ 设备指纹滥用                                              │
│     IMEI/MAC/Android ID 等设备唯一标识被滥用                  │
│     → 用户画像/追踪/广告定向                                 │
│                                                             │
│  ④ 跨应用数据共享                                            │
│     App A 读取通讯录 → App B 共享                            │
│     → 数据黑产/营销骚扰                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 OEM 的应对策略

```
┌─────────────────────────────────────────────────────────────┐
│           OEM 的两种应对策略对比                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  策略 1:权限拒绝(粗暴)                                       │
│     ┌────────────────────────────────────────┐             │
│     │  App 请求位置权限                         │             │
│     │    ↓                                     │             │
│     │  OEM 直接拒绝(throw SecurityException)    │             │
│     │    ↓                                     │             │
│     │  App 崩溃/功能不可用                       │ ← 用户痛    │
│     └────────────────────────────────────────┘             │
│                                                             │
│  策略 2:假数据返回(温柔)  ← OEM 主流策略                     │
│     ┌────────────────────────────────────────┐             │
│     │  App 请求位置权限                         │             │
│     │    ↓                                     │             │
│     │  OEM 允许权限,但返回"假数据"             │ ← 本篇核心 │
│     │    ↓                                     │             │
│     │  App 拿到假数据,正常运行                  │             │
│     └────────────────────────────────────────┘             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**:OEM 选择"假数据"而非"直接拒绝",是因为**直接拒绝会让 App 不可用**(用户痛),而**假数据让 App 误以为正常工作**,既保护了用户隐私,又不破坏用户体验。

### 1.3 "假数据返回"的 5 大典型场景

| 场景 | 原生行为 | OEM Hook 后行为 |
|---|---|---|
| 获取 IMEI | 抛 SecurityException | 返回全 0 字符串 |
| 获取定位 | 拒绝或返回真实坐标 | 返回模糊坐标(500m 偏移) |
| 读取剪贴板 | 静默返回 | 弹窗提示 + 返回空 |
| 获取已安装应用 | 返回完整列表 | 仅返回主流应用 |
| 读取通讯录 | 弹权限框 | 返回空或用户指定的假数据 |

### 3.3 OEM 假定位实现

```java
// (MIUI 公开技术分享,基于 AOSP 14)
//
// MIUI 隐私替身:LocationManagerService 假定位

public class MiuiPrivacyLocationPolicy {
    
    // OEM 在 getLastLocation 拦截
    public static Location maybeFakeLocation(String packageName, Location realLocation) {
        // [OEM 拦截] 检查 App 是否需要假定位
        if (shouldFakeLocation(packageName)) {
            return createFakeLocation(realLocation);
        }
        return realLocation;
    }
    
    // 创建假定位:在真实坐标基础上偏移 500m
    private static Location createFakeLocation(Location real) {
        if (real == null) return null;
        
        Location fake = new Location(real);
        // [OEM 替换] 随机偏移 300-700m
        double offsetMeters = 300 + new Random().nextInt(400);
        double bearing = new Random().nextInt(360);
        
        // 用 Haversine 公式计算偏移后的经纬度
        double[] offset = calculateOffset(real.getLatitude(), 
                                         real.getLongitude(), 
                                         offsetMeters, 
                                         bearing);
        fake.setLatitude(offset[0]);
        fake.setLongitude(offset[1]);
        // 精度也修改(防止精度异常被检测)
        fake.setAccuracy(50.0f);
        
        return fake;
    }
    
    // 判断是否需要假定位
    private static boolean shouldFakeLocation(String packageName) {
        // 1. 用户没授权位置 → 直接拒绝(不在本场景范围)
        // 2. App 在隐私黑名单 → 假定位
        // 3. App 不在前台 → 假定位
        return MiuiPrivacyBlacklist.contains(packageName) ||
               !isAppInForeground(packageName);
    }
}
```

### 3.4 假定位的"防检测"技巧

```
┌─────────────────────────────────────────────────────────────┐
│           假定位的"防检测"技巧                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① 偏移量不能太规则                                          │
│     每次固定偏移 500m → App 可检测"每次都是同一个假位置"      │
│     → OEM 用随机偏移(300-700m)                               │
│                                                             │
│  ② 时间一致性                                                │
│     假定位的"时间戳"必须和真实 GPS 一致                       │
│     → 否则 App 可以对比"上次真实位置"和"本次假位置"的时间差   │
│                                                             │
│  ③ 精度不能太低                                              │
│     假定位 accuracy=100m → 异常,App 可检测                  │
│     → OEM 设 accuracy=50m(看起来像真实数据)                  │
│                                                             │
│  ④ 速度/航向不能突变                                          │
│     假定位突然从 0 km/h 变成 50 km/h → 异常                  │
│     → OEM 用插值平滑速度/航向                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 五、ClipboardService 拦截 - 静默返回 + 弹窗

### 5.1 剪贴板拦截的特殊性

剪贴板拦截比 IMEI/LBS 更复杂——因为**用户复制的内容对 App 来说往往是"必要"的**。

```
┌─────────────────────────────────────────────────────────────┐
│           剪贴板拦截的 3 种策略                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  策略 1:完全静默                                              │
│     App 读剪贴板 → 返回空                                    │
│     优点:用户感知不到                                        │
│     缺点:用户体验差(复制的内容粘贴不了)                      │
│                                                             │
│  策略 2:静默 + 弹窗提示(主流)                                 │
│     App 读剪贴板 → 弹窗"XX App 读取了你的剪贴板"              │
│              → 同时返回真实内容                              │
│     优点:既提示又保留功能                                    │
│     缺点:实现复杂                                            │
│                                                             │
│  策略 3:智能识别                                              │
│     检测剪贴板内容是否是"敏感"(密码/银行卡)                  │
│     敏感 → 返回空 + 弹窗                                     │
│     普通 → 返回真实内容                                      │
│     优点:平衡隐私和体验                                      │
│     缺点:识别准确率有限                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 剪贴板拦截的源码路径

```java
// frameworks/base/services/core/java/com/android/server/ClipboardService.java
// (AOSP 14.0.0_r1,已校对 cs.android.com)
//
// ClipboardService 的核心方法

public class ClipboardService extends IClipboard.Stub {
    
    // App 获取剪贴板内容
    @Override
    public boolean hasPrimaryClip() throws RemoteException {
        // [OEM 拦截点]
        return mClipboard != null && mClipboard.hasPrimaryClip();
    }
    
    @Override
    public ClipData getPrimaryClip(String callingPackage) throws RemoteException {
        // [OEM 拦截点] 检查 App 是否能读剪贴板
        // 如果不能,返回空 + 弹窗
        if (MiuiPrivacyPolicy.shouldBlockClipboard(callingPackage)) {
            notifyClipboardRead(callingPackage);  // 弹窗
            return ClipData.newPlainText("", "");  // 返回空
        }
        
        // AOSP 原逻辑
        return mClipboard.getPrimaryClip();
    }
}
```

### 5.3 MIUI 剪贴板弹窗实现

```java
// (MIUI 公开技术分享)
//
// MIUI 剪贴板弹窗:App 读取剪贴板时,系统弹窗告知用户

public class MiuiClipboardMonitor {
    
    // OEM 拦截:getPrimaryClip 时弹窗
    public static ClipData interceptGetPrimaryClip(String packageName, 
                                                   ClipData original) {
        // [OEM 拦截] 是否在白名单
        if (MiuiPrivacyWhitelist.isClipboardTrusted(packageName)) {
            return original;
        }
        
        // [OEM 替换] 检测剪贴板内容是否敏感
        if (isSensitiveContent(original)) {
            // 敏感内容 → 返回空 + 弹窗
            notifyUser(packageName, original);  // 弹窗通知
            return ClipData.newPlainText("", "");
        }
        
        // 普通内容 → 返回真实 + 静默记录
        logClipboardAccess(packageName);
        return original;
    }
    
    // 敏感内容检测
    private static boolean isSensitiveContent(ClipData clip) {
        if (clip == null || clip.getItemCount() == 0) return false;
        
        String text = clip.getItemAt(0).getText().toString();
        // 检测规则:银行卡号、身份证号、手机号、密码等
        return PrivacyPatternMatcher.isSensitive(text);
    }
}
```

### 7.1 AppOps 的官方机制

```
┌─────────────────────────────────────────────────────────────┐
│           AppOps 机制的工作原理                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  AppOps(Android 4.3+ 引入):                                  │
│  "Application Operations" 的简称                              │

│  │  AppOpsManager.checkOp() 检查权限          │               │
│  │      ↓                                     │               │
```

### 7.2 AppOps vs OEM 自研的对比

| 维度 | AppOps(官方) | OEM 自研 |
|---|---|---|
| **稳定性** | 高(Google 维护) | 低(依赖 ROM 版本) |
| **细粒度** | 中(标准化 op) | 高(可定制规则) |
| **假数据** | 不支持(只有 allow/deny) | 支持(返回假数据) |
| **静默** | 部分(可 ignore) | 完全静默 |
| **演进趋势** | 越来越强 | 逐渐被替代 |

## 九、实战案例:银行 App 检测到权限欺骗后拒绝运行

### 9.1 案例背景

某银行 App(如招行/工行)在 OEM 设备上运行时,触发"安全检测"逻辑,要求用户"关闭隐私保护功能"才能使用。

### 9.2 现象

- 用户在 MIUI/HyperOS 上安装某银行 App
- 启动时弹出:"检测到您的设备开启了隐私保护,可能影响账户安全"
- 用户被强制要求"关闭空白通行证"才能继续

### 9.3 根因分析

银行 App 的安全检测逻辑:

```java
// 银行 App 自己的反检测逻辑(伪代码)
public class SecurityCheck {
    
    public boolean isDeviceCompromised() {
        // [检测 1] 检查 IMEI 是否异常
        String imei = telephonyManager.getDeviceId();
        if (imei.equals("000000000000000") || imei.length() < 14) {
            return true;  // "可疑设备"
        }
        
        // [检测 2] 检查定位是否异常
        Location loc = locationManager.getLastLocation();
        if (loc != null && loc.getAccuracy() < 1.0f) {
            return true;  // "精度异常,疑似假位置"
        }
        
        // [检测 3] 检查应用列表是否完整
        List<ApplicationInfo> apps = packageManager.getInstalledApplications(0);
        if (apps.size() < 30) {
            return true;  // "应用列表被过滤,疑似黑产设备"
        }
        
        return false;
    }
}
```

### 9.4 OEM 的应对(让 App 难以检测)

```
┌─────────────────────────────────────────────────────────────┐
│           OEM 让 App 难以检测假数据的策略                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① IMEI 不返回全 0,返回符合格式的假值                         │
│     "352099001761481"(15 位数字,符合 IMEI 格式校验)         │
│     通过 App 的 IMEI 格式校验                                │
│                                                             │
│  ② 假定位的 accuracy 设为正常范围(50m 左右)                  │
│     不设 1.0f 的极小精度(太假了)                              │
│     通过 App 的精度合理性检查                                │
│                                                             │
│  ③ 应用列表返回主流 App + 部分冷门 App                        │
│     不是返回 Top 50,而是返回 Top 80 + 随机 20                │
│     通过 App 的"应用数量合理性"检查                          │
│                                                             │
│  ④ 假定位的偏移方向随机化                                    │
│     每次偏移不同的方向,而不是固定北偏东                       │
│     通过 App 的"位置变化模式"检查                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 9.5 兼容性解决方案

**对 App 开发者**:
```java
// 不要用 IMEI 全 0 当作"异常设备"的判断
// 应该用更稳定的设备指纹(如 ANDROID_ID + Build.FINGERPRINT)
String deviceFingerprint = Settings.Secure.getString(
    getContentResolver(), Settings.Secure.ANDROID_ID);

// 或用 SafetyNet/Play Integrity API(Google 官方)
```

**对 OEM**:
- 不要用太"假"的假数据(全 0、固定值等)
- 用更逼真的假数据(格式正确、值合理)
- 提供"白名单"让用户指定哪些 App 可以读到真实数据

**对用户**:
- 银行类 App 建议关闭隐私保护(或在白名单里)
- 一般 App 建议开启隐私保护
- 在 OEM 设置的"隐私保护"里看每个 App 的权限使用情况

### 9.6 环境与复现

- **环境**:MIUI 14(基于 AOSP 13)/ 设备 Xiaomi 13 Pro
- **复现步骤**:
  1. 开启"空白通行证"
  2. 安装某银行 App
  3. 启动 App
  4. 弹出安全检测警告
- **修复**:在 OEM 设置里把该银行 App 加入"隐私白名单"

**稳定性架构师视角**:**隐私保护 vs 业务功能经常冲突**。OEM 必须提供灵活的"白名单/黑名单/灰名单"机制,让用户自己选择。

## 十一、总结 - 架构师视角的 7 条 Takeaway

1. **"假数据返回"是 OEM 的核心隐私策略**——比"权限拒绝"更友好
2. **隐私保护的核心拦截层是 Framework-Binder**——5 大假数据 API 都在这里
3. **AppOps 在收紧,OEM 自研空间在缩小**——Android 12+ AppOps 提供了 READ_CLIPBOARD 等 op
4. **OEM 假数据必须"逼真"**——全 0、固定值容易被 App 检测
5. **隐私保护 vs 业务功能经常冲突**——必须提供灵活的白名单机制
6. **剪贴板拦截需要"识别敏感内容"**——这是技术难点,主流用模式匹配
7. **OEM 隐私保护差异是核心竞争力**——5 大 OEM 各有"杀手锏"功能

**场景 1 速查路径**(遇到问题时):
```
线上问题(银行 App 拒绝运行 / App 拿不到真实数据 / 用户投诉隐私)
   ↓
5 秒定位:是 IMEI?LBS?剪贴板?应用列表?
   ↓
看 logcat:有 "Device compromised" → App 检测到假数据
        有 "permission denied" → 真拒绝,不是假数据
        有 "navigation wrong" → 假定位导致
   ↓
修复:把 App 加入白名单 / 改进假数据逼真度 / 提供关闭选项
```

| **假 IMEI 格式** | 15 位数字 | 必须通过 IMEI 格式校验 | 全 0 容易被检测 |
| **假定位精度** | 50m | 看起来像真实数据 | 太精确或太差都可疑 |
| **剪贴板弹窗** | 静默 + 弹窗 | 平衡隐私和体验 | 全静默用户感知不到 |
| **应用列表白名单** | Top 80 | 太少 App 不可用,太多泄露隐私 | 必须持续更新 |
| **假数据决策耗时** | < 5ms | 超过影响 App 性能 | 不能阻塞 IPC |
| **AppOps checkOp 频率** | 每次 API 调用 | 必须强制 | 否则 AppOps 等于没有 |
| **隐私白名单粒度** | App 级别 | 细粒度更好 | App × API × User 维度 |
| **隐私兼容性测试** | Top 1000 App | 必须覆盖 | 银行类 App 必须 |
| **假数据格式校验** | 与真实一致 | 不能太假 | 否则 App 检测 |
