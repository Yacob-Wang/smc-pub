## 0. 写在前面：分区问题的工程影响范围

01-07 篇把所有"分区相关的架构知识"讲完了——Treble（system ↔ vendor 解耦）、GKI（kernel ↔ SoC 解耦）、APEX（system ↔ 系统组件解耦）、Dynamic Partitions（容器化分配）、Virtual A/B（OTA 期间不打扰用户）。**但对稳定性工程师来说，这些架构本身不是问题**——架构再优雅，被一次 OTA 翻车打回原形。

本篇要回答的是一个完全不同的命题：

> **当线上 P0 告警响起（"某 OEM 机型升级后 5% 设备无法启动"），工程师在 30 分钟内，怎么判断这是 brick / boot loop / OTA 翻车 / APEX 挂载失败 / dlkm 模块加载失败 / VINTF 不匹配 / super 耗尽 / dm-linear 损坏 / OTA 中途断电 / Rollback 失败 中的哪一类？怎么定位到具体哪个分镜像？怎么在 OTA 后续版本中把同类问题堵住？**

换句话说：01-07 篇讲的是"partition 是什么、为什么这样设计"，本篇讲的是"partition **什么时候会咬人**、被咬了怎么查、怎么在 OTA 流程里把同类问题堵死"。

**工程影响范围（按线上 P0 故障分类）**：

```
分区相关稳定性故障的影响范围：

┌────────────────────────────────────────────────────────────┐
│  严重度 Critical（设备直接无法使用）                          │
├────────────────────────────────────────────────────────────┤
│  - brick（vbmeta 验证失败 / bootloader 损坏 / misc 分区损坏） │
│  - OTA 中途断电（slot 损坏 / bootloader 切换失败）            │
│  - Rollback 失败（bootloader rollback protection 触发）       │
│  - super 耗尽（OTA 时 super 重分配空间不足）                  │
│  - APEX 签名失败（com.android.runtime 升级 → bootloop）      │
└────────────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────────────┐
│  严重度 High（设备可用但功能受损）                             │
├────────────────────────────────────────────────────────────┤
│  - boot loop（init_boot 与 boot 版本不匹配）                 │
│  - dlkm 模块加载失败（vendor_boot 损坏 / 内核模块签名失败）   │
│  - VINTF 不匹配（HAL 版本漂移 / CM check 失败）              │
│  - dm-linear 损坏（device-mapper 异常）                      │
│  - APEX staged install 失败                                  │
└────────────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────────────┐
│  严重度 Medium（性能退化 / 用户可感知卡顿）                   │
├────────────────────────────────────────────────────────────┤
│  - super 碎片化（dm-linear 内部 I/O 序列化）                 │
│  - /apex 空间不足（decompressed / active 占用过高）          │
│  - OTA merge 慢（COW 膨胀 / snapuserd 性能瓶颈）             │
│  - VINTF 旧版本（FCM level 不匹配）                          │
│  - 残留 APEX session（/data/apex/sessions/ 异常增长）        │
└────────────────────────────────────────────────────────────┘
```

**心智模型（架构师视角）**：

把整套分区架构看作一个**多层缓存 + 多方协作的分布式事务系统**：

- **bootloader** 是**最终一致性仲裁者**（决定 slot 切换、Rollback Index）
- **init / first_stage_init / fs_mgr** 是**早期协调者**（决定挂载顺序、AVB 校验）
- **update_engine** 是**事务协调者**（写入 OTA 包、协调 snapshot merge）
- **libsnapshot / snapuserd** 是**事务执行者**（COW 读写、user-space-merge）
- **APEX / GKI** 是**版本契约层**（system ↔ kernel ↔ APEX 三方 ABI 锁定）

**任何一个环节出问题，都会被传导到"启动失败"或"功能异常"**。这正是本篇要做的事：把这套分布式事务系统的故障域（failure domain）拆成可识别的模式（patterns），让工程师拿到日志时知道往哪个方向走。

> **跨篇引用**：01 篇已经用 5 张时间线图把 12 年分区演进讲完了；02 篇讲 VINTF（系统 ↔ vendor 接口契约）；03 篇讲 GKI（kernel ↔ SoC 解耦 + 4 镜像独立升级）；04 篇讲 GSI（vendor 升级体检）；05 篇讲 Dynamic Partitions（super 容器）；06 篇讲 APEX（system 内部模块化）；07 篇讲 Virtual A/B（OTA 期间不打扰用户）。**本篇不重复这些机制，只把它们映射到"故障模式 → 排查路径 → 监控指标 → 治理手段"**。

### 1.2 协作链路中的 5 类"故障传导路径"

从架构图看，分区故障的传导路径主要有 5 类：

```
故障传导路径（按发生频率排序）：

路径 ①：bootloader → 内核 dm-linear → fs_mgr → 启动失败
  触发：bootloader AVB 校验失败 / slot metadata 损坏 / misc 写入异常
  表现：brick 或 boot loop
  排查入口：fastboot getvar / fastboot oem unlock-info / bootloader serial output

路径 ②：update_engine → libsnapshot → 内核 dm-snapshot → OTA 翻车
  触发：COW 格式不匹配 / merge 超时 / snapuserd 启动失败
  表现：OTA 后卡 logo / OTA 后系统无响应
  排查入口：logcat -b crash + dmesg + update_engine log

路径 ③：apexd → 内核 dm-verity → APEX 挂载失败 → bootloop
  触发：APEX 包签名验证失败 / fs-verity 校验失败 / classpath jar 缺失
  表现：apexd: Rebooting due to activation failure / 快速重启循环
  排查入口：logcat -s apexd + dmesg + cmd apexd status

路径 ④：init / first_stage_init → dlkm 模块加载失败
  触发：vendor_boot 损坏 / 内核模块签名失败 / 模块依赖不满足
  表现：first_stage_init: Failed to load kernel modules
  排查入口：serial console + dmesg + getprop ro.boot.modules.loaded

路径 ⑤：system_server → VintfObject → CM check failed → 阻断启动
  触发：vendor manifest level 不匹配 / HAL 版本漂移
  表现：boot loop 或部分功能缺失
  排查入口：cmd vintf_object_check / dumpsys vintf
```

> **稳定性架构师视角**：**故障传导路径就是排查路径**。拿到一份 boot loop 或 OTA 翻车的报告，先按这张图找到故障传导路径，就能定位到具体子系统（bootloader / 内核 / fs_mgr / apexd / update_engine），再去对应源码找具体函数。

### 1.3 三层监控点的设计原则

要把这张图变成"可观测系统"，需要在三层都埋点：

```

│  信号：boot reason / slot info / AVB 校验失败                          │

│  价值：唯一能看到 bootloader 决策的窗口                                   │

┌────────────────────────────────────────────────────────────────────────┐
│  第二层：内核层（设备启动的核心信号）                                     │
│  监控点：dmesg（内核 ring buffer）                                      │
│  工具：adb shell dmesg / pstore / ramoops / console-ramoops          │
│  信号：dm-linear/dm-verity/dm-snapshot 错误 / 模块加载失败 /            │
│        AVB 拒绝 / init 阶段异常                                         │
│  监控代价：中（adb 可达即可）                                            │
│  价值：分区相关故障的"主战场"                                             │
└────────────────────────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────────────────────────┐
│  第三层：用户态层（system_server / 服务层）                              │
│  监控点：logcat（system_server + 各 service tag）                       │
│  工具：adb logcat -d -b crash / logcat -b main / logcat -b system     │
│  信号：apexd 状态 / VINTF 错误 / PackageManager 异常 /                  │
│        update_engine 进度 / BootReceiver 事件                          │
│  监控代价：低（logcat 一行命令）                                          │
│  价值：分区相关故障的"业务影响"评估                                       │
└────────────────────────────────────────────────────────────────────────┘
```

**为什么分层重要？**：bootloader 层的故障（vbmeta 校验失败）不会出现在 logcat——它会在 logcat 起来之前就触发 reboot。内核层的故障（dm-linear 错误）会出现在 dmesg，但不会出现在 logcat 的 main buffer（要 logcat -b system）。用户态层的故障（apexd 失败）会出现在 logcat，但不会出现在 dmesg。**三层不能互相替代**。

## 2. 风险地图：10 大分区相关故障类型 + 关键字速查表

本节是本篇的**核心产出**——把整套分区架构相关的故障按严重度排序，整理成 5 列速查表。拿到任何一份 bug 报告（"OTA 后无法启动" / "升级后启动慢" / "升级后某个功能不可用"），对照表格快速匹配问题类型。

### 2.1 严重度排序的 10 类故障

**排序原则**：按线上影响范围（设备数）+ 可恢复性（自动 rollback vs 必须现场刷机）+ 排查难度 综合排序。

#### 故障 ①：brick（设备变砖）

- **触发机制**：bootloader AVB 校验失败 / vbmeta 分区损坏 / misc 分区写入错误导致 slot 切换失败
- **影响**：设备无法启动到 Android，连 recovery 都可能进不去
- **典型场景**：OEM 工厂烧写流程 bug、用户误刷 bootloader、vbmeta 公钥/私钥不匹配
- **Logcat / dmesg 关键字**（在 serial console 可见）：
  ```
  AVB failure: vbmeta: Verification slot is invalid
  AVB failure: vbmeta hash tree error
  Rollback index is invalid
  Device is corrupt
  ```
- **排查入口**：`fastboot --disable-verity`（临时关闭验证）/ `fastboot oem unlock-info` / serial cable
- **修复手段**：重刷 bootloader + vbmeta，工厂解锁后重新烧写

#### 故障 ②：boot loop（init_boot 与 boot 版本不匹配）

- **触发机制**：OTA 包只更新 boot.img，未同步更新 init_boot.img，导致 first_stage_init 与 GKI 内核版本不匹配
- **影响**：升级后第二次重启卡在 bootloader 屏幕，或 fastboot 可进但 boot 仍 bootloop
- **典型场景**：GKI 2.0 设备，OEM OTA 工具的"分镜像升级"逻辑有 bug（详见 §6 实战案例 1 和 [03-GKI 内核分区革命](03-GKI内核分区革命.md) 实战案例）
- **Logcat / dmesg 关键字**：
  ```
  first_stage_init: Failed to load kernel modules
  LoadKernelModules: module_count=0
  init: unable to open '/init' (No such file or directory)
  ```
- **排查入口**：`adb shell cat /proc/version` + `adb shell stat /init`（对比 build_id）/ `adb pull /sys/fs/pstore/console-ramoops`
- **修复手段**：OTA 工具增加 build_id 强制校验 / 推送"boot + init_boot 组合"修复包

#### 故障 ③：OTA 翻车（snapuserd merge 失败 / 压缩 snapshot 解压失败）

- **触发机制**：COW 文件 magic 不匹配（v3 压缩 vs v2 不压缩）/ merge 超时 / snapuserd 启动失败
- **影响**：OTA 完成后首次开机耗时长，最终合并失败需要回滚
- **典型场景**：AOSP 13 → AOSP 14 升级时 BOARD_VIRTUAL_AB_COMPRESSED_SNAPSHOT 未启用（详见 [07-Virtual A/B 与 OTA 链路](07-VirtualA_B与OTA链路.md) 实战案例）
- **Logcat / dmesg 关键字**：
  ```
  E snapuserd: Bad COW header at /data/ota_snapshot/cow/abc123
  E snapuserd: Expected magic 0x434F5733 "COW3", got 0x434F5732 "COW2"
  F snapuserd: Aborting merge
  ```
- **排查入口**：`adb logcat -d -b crash | grep snapuserd` + `adb shell lpdump /dev/block/by-name/super_b`
- **修复手段**：OTA 工具关压缩 / 设备 firmware 启用 BOARD_VIRTUAL_AB_COMPRESSED_SNAPSHOT / 强制回滚 slot

#### 故障 ④：APEX 挂载失败（com.android.runtime 升级导致 ART 异常）

- **触发机制**：APEX 包签名验证失败 / fs-verity 校验失败 / classpath jar 缺失 / 预置 APEX 公钥与系统不匹配
- **影响**：3% 用户进入"快速重启循环"，8% 用户 zygote 立即 crash（详见 [06-APEX 主线模块与运行时升级](06-APEX主线模块与运行时升级.md) 实战案例）
- **典型场景**：OEM 烧写流水线 bug，APEX 增量更新但 system.img 内公钥未同步更新
- **Logcat / dmesg 关键字**：
  ```
  E apexd: signature verification failed for /system/apex/com.android.runtime.apex
  E apexd: Rebooting due to activation failure
  E init: Rebooting: apexd-failed
  ```
- **排查入口**：`adb pull /system/apex/com.android.runtime.apex` + `diff apex_pubkey /system/etc/security/apex/com.android.runtime.avbpubkey`
- **修复手段**：紧急 OTA 推送"旧 APEX + 旧公钥"组合 + 修复 build 流水线增量更新脚本

#### 故障 ⑤：dlkm 模块加载失败（vendor_boot 损坏 / 内核模块签名失败）

- **触发机制**：vendor_boot.img 损坏 / 内核模块签名验证失败 / 模块依赖不满足（modprobe depmod 缺失）
- **影响**：vendor 模块（WiFi / Camera / Sensor）无法加载，相关功能不可用
- **典型场景**：GKI 5.15 升级时 vendor 端未重新签名内核模块，或 vendor_boot.img OTA 包损坏
- **Logcat / dmesg 关键字**：
  ```
  init: Loading module /vendor/lib/modules/foo.ko
  init: Failed to load module foo.ko: Exec format error
  init: Required key not available (内核模块签名失败)
  kernel: module verification failed: signature and/or required key missing
  ```
- **排查入口**：`adb shell dmesg | grep -i "module\|verify"` + `adb shell getprop ro.boot.modules.loaded`（数量异常）
- **修复手段**：重签 vendor 模块 / 修复 vendor_boot.img / 增加 modprobe 依赖检查

#### 故障 ⑥：VINTF 不匹配（HAL 版本漂移 / CM 检查失败）

- **触发机制**：device manifest level 不匹配 / HAL 缺失 / FCM optional→required 转换
- **影响**：boot 阶段被 init 阻断（VintfFm 自动升级失败），或 HAL 服务缺失导致功能异常
- **典型场景**：OEM 在 Android 13→14 升级时未升级 vendor manifest level，未补齐 FCM 11 新增 HAL（详见 [02-VINTF 与 Treble 接口契约](02-VINTF与Treble接口契约.md) 和 [04-GSI 通用系统镜像](04-GSI通用系统镜像.md) 实战案例）
- **Logcat / dmesg 关键字**：
  ```
  init: VINTF for device: 11
  VintfObject: ERROR: device manifest level (8) < required FCM level (9)
  init: VINTF check failed: INCOMPATIBLE
  init: android.hardware.camera.provider@2.6::ICameraProvider is required but not declared
  ```
- **排查入口**：`adb shell cmd vintf_object_check` + `adb pull /vendor/etc/vintf/manifest.xml`
- **修复手段**：升级 vendor manifest level + 补齐 HAL 声明 + 重新编译 vendor HAL 实现

#### 故障 ⑦：super 耗尽（OTA 重分配空间不足）

- **触发机制**：OTA 新版本 partition 累计 > group maximum_size，dm-linear 写入失败
- **影响**：OTA 中途失败，设备卡在 OTA 进度条
- **典型场景**：release 2 system + product 都扩容（system 3→3.5GB, product 1.5→2GB），但 group 配额仍是 7GB
- **Logcat / dmesg 关键字**：
  ```
  liblp: UpdateSuper failed: not enough space in group
  fs_mgr: Failed to mount logical partitions
  first_stage_init: Cannot mount /system
  ```
- **排查入口**：`adb shell lpdump /dev/block/by-name/super_b --slot=1` + 检查 group maximum_size
- **修复手段**：扩容 super 分区 / 减小 partition size / 重新分配 group 配额

#### 故障 ⑧：dm-linear 损坏（device-mapper 异常）

- **触发机制**：super partition metadata 写入越界 / lp_metadata 损坏 / dm-linear 表物理扇区错位
- **影响**：fs_mgr 读取 lpmetadata 失败，逻辑分区无法挂载
- **典型场景**：OEM OTA 流程中 UpdateSuper 未做 active/inactive slot 边界检查（详见 [05-动态分区与 super 容器](05-动态分区与super容器.md) 实战案例）
- **Logcat / dmesg 关键字**：
  ```
  lp_metadata: Failed to read metadata: bad magic
  lp_metadata: Falling back to backup slot
  lp_metadata: Backup slot also bad, aborting
  dm-linear: Invalid mapping
  ```
- **排查入口**：`adb shell lpdump /dev/block/by-name/super_a` + `adb shell dmesg | grep -i "dm-linear\|lp_metadata"`
- **修复手段**：recovery 模式下重新格式化 super / 重新刷 super 分区 / 强制回滚到 inactive slot

#### 故障 ⑨：OTA 中途断电（slot 损坏 / bootloader 切换失败）

- **触发机制**：OTA 写入过程中设备掉电，bootloader 状态不一致 / active slot 与 inactive slot 都有半成品
- **影响**：重启后无法进入 active slot，反复尝试后进入 recovery
- **典型场景**：用户 OTA 下载完成后未充电，电量低自动重启
- **Logcat / dmesg 关键字**：
  ```
  bootloader: Slot _a is not bootable
  bootloader: Slot _b is not bootable
  init: Failed to read slot info from /misc
  ```
- **排查入口**：`fastboot getvar current-slot` + `fastboot getvar slot-count` + 检查 /misc 分区
- **修复手段**：recovery 模式刷完整 OTA 包 / fastboot 强制刷写 active slot / 重置 misc 分区

#### 故障 ⑩：Rollback 失败（bootloader rollback protection 触发）

- **触发机制**：当前启动 slot 的 Rollback Index 高于 bootloader 允许的最大值（常见于跨版本降级尝试）
- **影响**：bootloader 直接拒绝启动，显示 "Device is corrupt" 或类似警告
- **典型场景**：用户尝试降级到旧版本（OEM 强制 rollback protection）/ OTA 中 bootloader 元数据损坏
- **Logcat / dmesg 关键字**：
  ```
  AVB failure: rollback index is invalid
  Rollback protection triggered
  ```
- **排查入口**：`fastboot getvar rollback-index` + serial cable 读 bootloader 版本
- **修复手段**：OEM 工厂工具重写 Rollback Index（无法 OTA 修复）/ 重烧 bootloader

## 3. 诊断工具链：adb / fastboot / recovery / dumpsys / dm-verity 工具速查

把 §1 的协作链路 + §2 的故障类型 → 落到具体的"诊断命令"。本节按"诊断层级"组织，每条命令都有：触发场景 + 预期输出 + 解读要点。

### 3.1 早期诊断工具（bootloader / 内核层）

#### 工具 ①：`fastboot getvar all`（bootloader 层状态）

```
触发场景：设备卡在 fastboot mode 或 recovery
命令：fastboot getvar all

关键输出（分区相关）：
  current-slot: a
  slot-count: 2
  slot-retry-count:b: 7
  partition-type:userdata: ext4
  is-logical:system_a: yes
  is-logical:vendor_a: yes
  is-logical:product_a: yes
  super-partition-name: super
  snapshot-update-status: merge-completed
  vbmeta-digest: ...
  rollback-index: 14

解读要点：
- current-slot 是 boot_a 还是 boot_b → 决定从哪个 slot 启动
- slot-retry-count 接近 7（最大重试次数）→ 表明 OTA merge 多次失败
- is-logical:xxx 是 yes → 走 dm-linear，是 no → 走 raw partition
- snapshot-update-status: merge-pending → OTA merge 未完成
- rollback-index: 14 → 当前 Rollback Index（Android 14 = 14）
```

#### 工具 ②：`adb shell dmesg`（内核层完整状态）

```
触发场景：设备启动到 Android，但功能异常
命令：adb shell dmesg | grep -iE "dm-linear|dm-verity|dm-snapshot|lp_metadata|first_stage_init|module|verif"

关键输出（分区相关）：
  [    1.234567] lp_metadata: Successfully read metadata from slot 0
  [    1.345678] dm-linear: device-mapper: linear: dm-linear table: 2048 7340032 linear 8:1 2048
  [    1.456789] device-mapper: verity: hashtree matches
  [    2.123456] init: Loading module /vendor/lib/modules/wlan.ko
  [    2.234567] init: Loaded 47 kernel modules

解读要点：
- 出现 lp_metadata 成功读取 → super metadata 正常
- dm-linear table 输出 → logical partition 映射关系
- hashtree matches → dm-verity 校验通过
- Loaded N kernel modules → N 应 > 30（GKI 5.15 标配）
- N < 5 → vendor_boot 损坏或模块签名失败
```

#### 工具 ③：serial console + pstore（启动早期日志）

```
触发场景：设备卡在 bootloader，无法启动到 Android，adb 无法连接
工具：
  - UART serial cable（115200 baud）
  - adb shell cat /sys/fs/pstore/console-ramoops
  - adb shell cat /proc/last_kmsg（旧内核 log）

典型日志（init_boot 不匹配）：
  [    2.123] Loading module /vendor/lib/modules/foo.ko
  [    2.345] Failed to load module foo.ko: Exec format error
  [    2.456] Loaded 0 kernel modules took 0 ms
  [    3.123] init: Failed to mount /system

解读要点：
- "Loaded 0 kernel modules" → first_stage_init 加载 0 个模块，明确的"init_boot 不匹配"特征
- "Exec format error" → 内核版本与模块版本不匹配
- 完整启动序列需要看到 mount /system → mount /vendor → start zygote 三步齐全
```

#### 工具 ⑤：`dumpsys diskstats`（磁盘 / dm-linear / super 使用率）

```
触发场景：分析 OTA 后性能退化（system_server 启动慢）
命令：adb shell dumpsys diskstats

关键输出（分区相关）：
  Bootai
    Pools:
      Boot (size=83886080, reads=12450, writes=458)
      System (size=209715200, reads=8234, writes=12567)

- Boot pool reads 远大于 System pool → first stage boot 加载密集
```

#### 工具 ⑥：`cmd apexd status`（APEX 状态机）

```
触发场景：APEX 升级失败 / staged install 异常
命令：adb shell cmd apexd status

关键输出（分区相关）：
  Active APEX packages:
    com.android.runtime (version 341411030, last update = 2024-01-15 12:00:00)
    com.android.tzdata (version 341411000, last update = 2024-01-15 12:00:00)
    com.android.adbd (version 341411000, last update = 2024-01-15 12:00:00)
    ...
  Staged sessions:
    (none)
  Factory packages:
    com.android.runtime (version 341411030)

解读要点：
- Active APEX 的 version 异常低 → APEX 升级未生效
- Staged sessions 不为空 → 上次 OTA 未完成 APEX 升级，需要 reboot 激活
- Factory packages 与 Active APEX version 不一致 → 烧写流水线不一致
```

#### 工具 ⑦：`lpdump`（super partition metadata 读取）

```
触发场景：super 耗尽 / dm-linear 损坏 / OTA 后无法读取 logical partition
命令：adb shell lpdump /dev/block/by-name/super_a --slot=0

关键输出（partition table + group table）：
  Partition table:
  -------------------------
    Name: system
    Group: default
    Attributes: readonly
    Extents:
      0 .. 7340031 linear super_a 2048    ← system 起始扇区 2048 (1MB 偏移)
  -------------------------
    Name: vendor
    Group: default
    Attributes: readonly
    Extents:
      0 .. 1048575 linear super_a 7342080  ← vendor 起始扇区 7342080
  -------------------------

  Group table:
  -------------------------
    Name: default
    Maximum size: 7516192768 bytes (7 GB)  ← group 配额 = 7 GB
    Flags: none
  -------------------------

解读要点：
- Extents 起始扇区跨 partition 一致 → metadata 完整
- Group maximum_size < partition 总和 → super 耗尽风险
- 起始扇区在 release 1 vs release 2 之间变化 → OTA 已切换 slot
```

### 3.3 工具组合实战

**最常用的 3 步排查组合**：

```
Step 1: adb shell dmesg | grep -iE "dm-linear|lp_metadata|module verif"
        → 判断内核层是否有故障信号

Step 2: adb logcat -d -b crash | grep -iE "apexd|snapuserd|update_engine|VintfObject"
        → 判断用户态层是否有 crash

Step 3: adb shell cmd <vintf_object_check|apexd status|lpdump> <args>
        → 针对性查具体子系统
```

**快速判断矩阵**：

```
故障在哪个层（基于日志位置）：
┌─────────────────────┬──────────────────────┬──────────────────────┐
│ 故障层              │ 关键日志位置          │ 首选诊断命令         │
├─────────────────────┼──────────────────────┼──────────────────────┤
│ bootloader 层       │ serial console       │ fastboot getvar all │
│ 内核层（dm-linear）  │ dmesg                │ dmesg | grep dm-... │
│ 内核层（模块）      │ dmesg + dmesg -t L   │ dmesg | grep module │
│ init / first_stage  │ dmesg + console-ramoops │ dmesg | grep init   │
│ 用户态（APEX）      │ logcat -s apexd     │ cmd apexd status    │
│ 用户态（OTA）       │ logcat -s update_engine | logcat -b crash     │
│ 用户态（VINTF）     │ logcat -s VintfObject | cmd vintf_object_check│
│ 用户态（snapshot）  │ logcat -s snapuserd | lpdump                │
└─────────────────────┴──────────────────────┴──────────────────────┘
```

## 4. 监控体系设计：super 使用率 / apex 空间 / OTA merge 成功率等指标

排查是被动响应，**监控是主动发现**。本节设计一套"分区相关的稳定性监控指标体系"，按严重度分级输出告警。

### 4.1 三层监控指标

#### Layer 1：bootloader 层指标（serial cable / fastboot）

| 指标 | 采集方式 | 阈值 | 告警级别 |
| :--- | :--- | :--- | :--- |
| `current-slot` 切换失败率 | fastboot getvar | > 1% | P1 |
| `rollback-index` 异常 | fastboot getvar rollback-index | ≠ 当前 Android 版本号 | P0 |
| `slot-retry-count` 接近 7 | fastboot getvar slot-retry-count:b | ≥ 5 | P1 |
| `vbmeta-digest` 失败率 | serial console + boot reason | > 0.1% | P0 |

#### Layer 2：内核层指标（dmesg / 监控 daemon）

| 指标 | 采集方式 | 阈值 | 告警级别 |
| :--- | :--- | :--- | :--- |
| 内核模块加载数量 | `dmesg \| grep "Loaded N kernel modules"` | < 30（GKI 5.15 标配）| P1 |
| dm-linear 错误率 | `dmesg \| grep "dm-linear.*Invalid"` | > 0.1% | P1 |
| dm-verity 失败率 | `dmesg \| grep "hashtree.*mismatch"` | > 0.01% | P0 |
| first_stage_init 耗时 P99 | dmesg timestamp | > 30s | P1 |
| `lp_metadata: bad magic` 错误 | `dmesg \| grep lp_metadata` | > 0 次 | P0 |

#### Layer 3：用户态层指标（logcat / dumpsys / cmd）

| 指标 | 采集方式 | 阈值 | 告警级别 |
| :--- | :--- | :--- | :--- |
| **APEX 挂载成功率** | `cmd apexd status` + 解析 | < 99.9% | P0 |
| **APEX active 数量** | `cmd apexd status` | ≠ 25（基线 25 个核心 APEX）| P1 |
| `/data/apex/active/` 占用 | `du -sh /data/apex/active/` | > 80% 分区容量 | P1 |
| `/data/apex/decompressed/` 占用 | `du -sh /data/apex/decompressed/` | > 200 MB | P1 |
| `/data/apex/sessions/` 数量 | `ls /data/apex/sessions/ \| wc -l` | > 5（残留 session）| P1 |
| **OTA merge 成功率** | update_engine callback | < 99.9% | P0 |
| OTA merge 完成时间 P99 | update_engine callback | > 30 分钟 | P1 |
| **snapuserd 启动失败率** | logcat -b crash \| grep snapuserd | > 0.1% | P0 |
| COW 校验失败率 | logcat -b crash \| grep "Bad COW header" | > 0.01% | P0 |
| **VINTF 不匹配率** | `cmd vintf_object_check` + `dumpsys vintf` | > 0 | P0 |
| **super partition 使用率** | `dumpsys diskstats` + lpdump | > 90% | P1 |
| /apex 空间使用率 | `df -h /data` + `du -sh /data/apex/` | > 80% | P1 |
| system_server 启动耗时 P99 | `dumpsys activity boot complete time` | > 60s | P1 |
| `reboot_on_failure` 触发次数 | logcat \| grep "Rebooting: " | > 0.01% | P0 |

```

│    - update_engine 通过 callback 汇报状态到 GMS                       │

┌────────────────────────────────────────────────────────────────────────┐
│  指标：VINTF 不匹配率                                                    │

│    adb shell cmd vintf_object_check > /tmp/vintf.txt                   │
│    → 解析输出（COMPATIBLE / INCOMPATIBLE）                              │
│    → dumpsys vintf 详细差异                                            │
│  告警触发：输出 INCOMPATIBLE                                            │
│  采集频率：每次启动后采集（boot_completed callback）                     │
│  采集代价：低（cmd 命令 + 文本解析）                                     │
└────────────────────────────────────────────────────────────────────────┘
```

### 4.3 监控告警分级原则

```
告警分级（按线上影响 + 恢复成本）：

P0（Critical - 立即响应）：
  - vbmeta 验证失败率 > 0.1%
  - APEX 挂载成功率 < 99.9%（com.android.runtime / art 类）
  - OTA merge 失败率 > 1%
  - Rollback 失败触发
  - 系统层 reboot_on_failure 触发
  → 自动停发 OTA + 立即工程调查 + 推送紧急修复

P1（High - 30 分钟响应）：
  - dm-verity / dm-linear 错误率 > 0.1%
  - 内核模块加载数量 < 30
  - /apex 空间使用率 > 80%
  - super partition 使用率 > 90%
  - VINTF 不匹配
  → 监控告警 + 30 分钟响应 + 计划下次 OTA 修复

P2（Medium - 当日响应）：
  - 残留 APEX session 数量 > 5
  - /data/apex/decompressed/ > 200 MB
  - OTA merge 完成时间 P99 > 30 分钟
  → 监控告警 + 当日调查 + 后续优化

P3（Low - 周报跟踪）：
  - system_server 启动耗时 P99 > 60s（无明显异常）
  - super partition 预留 < 1 GB
  → 周报跟踪 + 季度优化
```

## 5. 治理最佳实践：OTA 前/后 checklist + 不同分区类型的 OTA 策略

### 5.1 OTA 前：发布门禁（pre-OTA gate）

```
OTA 发布前必须通过的 11 项 checklist（CI + QA）：

[ ] 1. VINTF check：check_vintf host 端对新 vendor image + system image 校验通过
    责任方：QA 测试团队
    命令：./artifacts/test/vintf/test/vintf_test.py --system system.img --vendor vendor.img
    通过标准：输出 COMPATIBLE，0 条 INCOMPATIBLE

[ ] 2. APEX 签名一致性：system.img 中所有 APEX 的 apex_pubkey 与 system/etc/security/apex/*.avbpubkey 一致
    责任方：build system 维护方
    工具：build/verify_apex_signatures.py（OEM 自研，参考 06 篇 §8.4）
    通过标准：0 处 mismatch

[ ] 3. 4 镜像版本一致性：boot.img、init_boot.img、vendor_boot.img、system_dlkm.img 来自同一 build（同一 commit）
    责任方：build system + OTA 工具
    工具：update_engine 预检 check_build_id（参考 03 篇 §7.4）
    通过标准：4 个镜像的 build_id 完全一致

[ ] 4. super partition 配额：OTA 后所有 partition 总和 < group maximum_size
    责任方：partition 架构师
    工具：lpdump + 解析 group table
    通过标准：预留至少 1 GB free space（避免 OTA 中途耗尽）

[ ] 5. dm-verity 校验：所有分区的 dm-verity hashtree 已正确生成
    责任方：build system
    工具：avbtool verify_image --image system.img
    通过标准：Verification successful

[ ] 6. AVB 校验：vbmeta header 包含所有需要的 partition descriptor
    责任方：build system
    工具：avbtool dump_image / avbtool verify_image
    通过标准：vbmeta digest 与各 partition 一致

[ ] 7. Rollback Index 检查：当前 OTA 包的 Rollback Index ≥ device 当前 Rollback Index
    责任方：OTA 平台
    工具：fastboot getvar rollback-index + OTA 包的 version
    通过标准：OTA version ≥ device Rollback Index

[ ] 8. OTA 兼容性矩阵：OTA 工具版本 × 设备 firmware × COW 格式 兼容
    责任方：OTA 工具维护方
    工具：CI 中维护 compat matrix，任意组合不匹配都报错
    通过标准：CI 绿灯

[ ] 9. dry-run 测试：1% 灰度升级监控 24 小时
    责任方：OTA 平台 + 监控
    监控：APEX 挂载成功率、OTA merge 成功率、boot loop 率
    通过标准：三类指标全部 ≥ 99.9%

[ ] 10. 回滚演练：1% 灰度用户强制回滚到 inactive slot，验证回滚链路
    责任方：OTA 平台
    工具：fastboot set_active b + 重启

[ ] 11. 灰度放量阶段：5% → 25% → 100% 三阶段，每阶段 24h 监控
    责任方：OTA 平台
    监控：所有 §4 监控指标
    通过标准：每阶段全部 P0 指标未触发告警
```

## 6. 实战案例 1：OEM 升级 init_boot 时未同步 boot 导致无法启动

> **典型模式（generic pattern）**——本案例基于 03 篇 §7 实战案例的"init_boot 与 boot 版本不匹配"通用模式重写，**不复述具体 OEM 内部事故**。该案例在 03 篇中已完整覆盖，本节聚焦"作为稳定性工程师如何从 §2 速查表定位 + 排查 + 修复"。

### 6.1 现象

某 OEM 厂商推送 AOSP 14 GKI 5.15 升级包到其 5 款设备。**升级流程中只更新了 `boot.img`，但 OTA 工具的"分镜像升级"逻辑有 bug，导致 `init_boot.img` 没有被同步更新**。

**线上现象**（升级后 24 小时内）：

1. **10% 设备**：升级后第一次启动正常，但**第二次重启**（如 OTA 完成提示"reboot to apply update"）后卡在 bootloader 屏幕
2. **20% 设备**：升级后开机直接卡在 "Your device is corrupt" 警告页
3. **70% 设备**：升级后启动到 second stage，但 **WiFi/Display 全部不可用**（vendor modules 加载失败）

### 6.2 分析

- 关键字 `Failed to load kernel modules` → boot loop（init_boot 不匹配）
- 关键字 `module_count=0` → 同上
- 严重度：Critical（70% 设备功能异常 + 30% 设备变砖）

```bash
# 对比 boot.img 与 init_boot.img 的版本
```

**根因诊断（OEM build system 流程还原）**：

```
build_script.sh
├─ 编译 boot.img（kernel commit A → boot-5.15.124）
├─ 编译 init_boot.img（init commit B → init_boot-5.15.125）   ← 不同 commit!
└─ 打包 OTA 包
    ├─ boot.img 来自 5.15.124
    └─ init_boot.img 来自 5.15.125
```

1. **build system 原子性缺失**——boot.img 与 init_boot.img 由不同 build target 生成（make bootimage vs make init_boot），没有"两个 target 必须来自同一 commit"的强制约束
2. **OTA 工具预检缺失**——update_engine 没有"build_id 匹配"校验
3. **启动时校验缺失**——first_stage_init.cpp 的 LoadKernelModules() 在加载失败时只记 ERROR / FATAL，没有"模块加载数量异常时主动回滚"逻辑

## 7. 实战案例 2：OEM OTA 后 system_server 启动慢 → dm-linear 检查 → super 碎片化

> **典型模式（generic pattern）**——本案例基于"OTA 后 dm-linear 性能退化"的通用模式构造，聚焦 super partition 在多次 OTA 后的碎片化问题。

### 7.1 现象

某 OEM 厂商在 12 个月内连续推送 8 次 OTA 升级（每月一次），每次 OTA 都会调整 partition 大小（system / vendor / product 扩容）。

**用户现象**：

1. OTA 完成后首次开机耗时从正常的 25s 增长到 75s
2. system_server 启动耗时从 8s 增长到 35s
3. App 启动延迟 P99 增长 200%（从 800ms 到 2400ms）
4. 用户反馈"升级到 Android 14 后手机越用越慢，重启也不管用"

### 7.2 分析

**Step 1：定位故障类型**

按 §2.2 速查表匹配：

- 关键字：无显式日志（沉默型退化）
- 类型：**super 碎片化**（严重度 Medium，performance degradation）

**Step 2：抓取 dm-linear 性能数据**

```bash
adb shell dumpsys diskstats | grep -A 50 "dm-linear"

# 输出：
dm-linear stats
  /dev/block/dm-0 (system)
    Reads: 8.5GB Writes: 120MB
    Avg latency: 28ms
    P99 latency: 185ms        ← 正常应 < 50ms，当前 185ms
  /dev/block/dm-1 (vendor)
    Reads: 1.2GB Writes: 50MB
    Avg latency: 12ms
    P99 latency: 68ms
  /dev/block/dm-2 (product)
    Reads: 3.4GB Writes: 200MB
    Avg latency: 18ms
    P99 latency: 125ms        ← product P99 异常高
```

**Step 3：检查 super partition 碎片化程度**

```bash
adb shell lpdump /dev/block/by-name/super_a --slot=0

# 输出（关键）：
Partition table:
-------------------------
  Name: system
  Group: default
  Attributes: readonly
  Extents:
    0 .. 7340031 linear super_a 2048
    7340032 .. 10485759 linear super_a 8388608   ← system 跨两个 extent！
    10485760 .. 13631487 linear super_a 14680064 ← 第三个 extent
-------------------------
  Name: vendor
  Group: default
  Attributes: readonly
  Extents:
    0 .. 1048575 linear super_a 20971520
-------------------------
```

**关键发现**：

- system partition 跨**三个 extent**（不是连续的 1 个 extent）
- 这意味着 dm-linear 在读取 system 时需要**多次 I/O**（extent 0 → 1 → 2）
- 每次 I/O 切换都有延迟，叠加导致 P99 latency 升高

**Step 4：分析碎片化原因**

```
super partition 物理布局（release 1 vs release 8）：

release 1（OTA 前）：
  super_a: [lpmetadata 4KB][gap 1MB][system 3GB][vendor 512MB][product 1.5GB][system_ext 1GB][free 2GB][lpmetadata 4KB]
                                  ↑ system extent: super_a[1MB, 3GB+1MB]

release 8（OTA 后多次扩容）：
  super_a: [lpmetadata 4KB][gap 1MB][system 3.5GB][vendor 512MB][product 2GB][system_ext 1GB][free 1GB][lpmetadata 4KB]
                                  ↑ system extent: super_a[1MB, 3.5GB+1MB]

问题：每次 OTA 扩容都会在 super 内重新分配 extent，由于 metadata 中保留的 extent 不能"原地扩展"，
      系统会创建新的 extent 块（不在原位置），导致 single logical partition 跨多个 physical extent。

假设 12 个月内 8 次 OTA 扩容：
  system partition 可能跨 3 个 extent（最大连续块 + 2 次碎片）
  vendor partition 可能跨 1-2 个 extent
  product partition 可能跨 2 个 extent（product 扩容 2 次）

碎片化累积效应：
  每次 I/O 跨 extent → dm-linear 内部 I/O 序列化 → 系统 I/O latency 累积
```

### 7.3 根因

```
根因链（5 个 Why）：

Why 1：为什么 system_server 启动慢？
  → 读取 system.img 内文件时 dm-linear P99 latency 高（185ms）

Why 2：为什么 dm-linear P99 latency 高？
  → system partition 跨多个 extent，每次读取需要多次 I/O

Why 3：为什么 system 跨多个 extent？
  → super partition 在多次 OTA 后碎片化

Why 4：为什么 super 碎片化？
  → 每次 OTA 扩容都在 super 内重新分配 extent，没有"原地扩展"或"碎片合并"机制

Why 5：为什么没有"碎片合并"机制？
  → AOSP 14 liblp 的 UpdateSuper 函数只做"修改 metadata"，不做"物理重写"
     （物理重写意味着把 data 从一个 extent 拷贝到另一个 extent，然后修改 metadata）
```

### 7.4 修复

**短期修复（recovery 模式重新格式化 super）**：

```bash
# 进入 recovery 模式
adb reboot recovery

# 备份当前 metadata
adb shell lpdump /dev/block/by-name/super_a --slot=0 > /tmp/lpdump_pre.txt

# 重新格式化 super（按当前 OTA 的 partition 表）
adb shell update_engine_client --reset-super

# 重启设备
adb reboot

# 验证 super 已重新格式化
adb shell lpdump /dev/block/by-name/super_a --slot=0
# 输出应显示每个 partition 都是 single extent
```

**中期修复（OEM OTA 流程优化）**：

```
新增 OTA 后碎片化监控：
  1. OTA 完成后立即 lpdump 检查每个 logical partition 的 extent 数量
  2. 任何 logical partition extent 数量 > 2 → 标记为"碎片化"
  3. 累计碎片化程度（extent 数 - 1）超过 5 → 触发下次 OTA 前重新格式化 super

新增 build system 优化：
  - 每次 OTA 分配新 partition 时，优先使用"连续 extent"而非"分散 extent"
  - 在 liblp::UpdateSuper 中增加"extent 合并"逻辑：
    - 如果新 partition 可以合并到现有 extent → 合并
    - 否则创建新 extent
```

**长期修复（libsnapshot + liblp 协作改造）**：

```cpp
// system/core/fs_mgr/libsnapshot/liblp/builder.cpp
// 新增"碎片整理"逻辑
bool PartitionBuilder::Defragment() {
    // 对每个 logical partition，按当前 size 检查是否有连续 extent 可用
    // 如果当前 extent 不连续 → 尝试物理拷贝到连续 extent → 修改 metadata
    // 注意：这个操作需要 active slot 是 inactive slot（不能修改当前 active 的数据）
    // 因此只能在 OTA 写入阶段执行，不能在运行时执行
}
```

### 7.5 治理与监控

```
线上监控指标（应纳入 §4 监控体系）：

| 指标 | 阈值 | 告警级别 |
|------|------|---------|
| super partition logical partition extent 数量 | > 2 | P2 |
| dm-linear P99 latency（system partition） | > 100ms | P1 |
| dm-linear P99 latency（product partition） | > 100ms | P1 |
| system_server 启动耗时 P99 | > 30s | P1 |
| App 启动延迟 P99 | > 2s | P1 |

应急方案：
- recovery 模式 update_engine_client --reset-super（立即生效）
- 下次 OTA 前规划"重新格式化 super"（需要用户备份数据）

预防方案：
- 每 6 个月定期执行一次"super 重新格式化"OTA
- 监控 dm-linear P99 latency 趋势，提前告警
```

### 7.6 经验教训

> **稳定性架构师视角**：super partition 碎片化是"渐进式退化"的典型代表——**无显式日志、无用户报错、无监控告警**，但系统会越来越慢。**它是 OTA 累积效应的"灰犀牛"**：单次 OTA 不出问题，但 8 次 OTA 之后性能可能退化 200%。**稳定性工程师必须建立"长期性能趋势监控"，不要只看 OTA 当前的 P0/P1 指标**。建议把 `dm-linear P99 latency` 加入月度趋势图，> 100ms 触发 P1 告警。

## 总结：架构师视角的 5 条 Takeaway

> **资深架构师视角**：从 8 篇分区架构演进看稳定性全景，本质是 **"把分布式事务系统翻译成可观测的故障模式"**。以下 5 条 Takeaway 是稳定性工程师排查分区类问题的心智锚点：

**Takeaway 1：分区故障 = 分布式事务系统的失败**

- **三方协作**：bootloader（仲裁）+ 内核 dm-linear/dm-verity/dm-snapshot（执行）+ 用户态 update_engine/apexd/init（协调）
- **四类故障域**：bootloader 层（brick / Rollback）、内核层（dm-linear / dm-verity / dlkm）、用户态 init/first_stage（VINTF / 模块加载）、用户态服务（apexd / update_engine / VintfObject）
- **故障传导**：任何一个环节失败都会传导到"启动失败"或"功能异常"，**关键是定位到具体子系统**

**Takeaway 2：5 列速查表 + 3 层日志是排查核心**

- **5 列速查表**：风险类型 + 严重度 + logcat/dmesg 关键字 + 排查入口 + 修复手段
- **3 层日志**：bootloader（serial console / fastboot getvar）+ 内核（dmesg）+ 用户态（logcat -b crash / dumpsys）
- **速查表使用**：拿到 bug 报告 → 复制关键字 → 5 列匹配 → 30 秒定位方向

**Takeaway 3：监控体系是"主动发现"而非"被动响应"**

- **P0 监控**：APEX 挂载成功率、OTA merge 成功率、vbmeta 验证失败率、Rollback Index 异常
- **P1 监控**：dm-verity / dm-linear 错误率、内核模块加载数量、/apex 空间使用率、super partition 使用率
- **P2 监控**：残留 APEX session、/data/apex/decompressed/、OTA merge 完成时间 P99
- **关键设计**：**监控指标必须按"线上影响 + 恢复成本"分级**，不能一刀切

**Takeaway 4：OTA 前/中/后 3 阶段 checklist 是治理核心**

- **OTA 前**：VINTF check、APEX 签名一致性、4 镜像版本一致性、super 配额检查、AVB 校验、Rollback Index 检查
- **OTA 中**：下载成功率、写入成功率、merge 启动率、merge 完成率、用户回退率、异常重启率
- **OTA 后**：APEX 挂载成功率、boot loop 率、VINTF 不匹配率、/apex 空间、super 碎片化、system_server 启动耗时
- **关键设计**：**checklist 不是文档摆设，是 CI / OTA 平台强制的门禁**

**Takeaway 5：4 类分区有 4 套 OTA 策略，不能混用**

- **raw partition**（boot / init_boot / vendor_boot / vbmeta）：直接写入 + AVB + Rollback Index
- **logical partition in super**（system / vendor / product）：Virtual A/B + COW + dm-snapshot merge
- **APEX 包**（com.android.runtime 等）：staged install + apexd 激活 + fs-verity
- **内核模块**（system_dlkm / vendor_boot）：KMI 强制一致 + modprobe depmod + 模块签名
- **关键设计**：**每类分区有独立的 OTA 工具链 + 校验机制**，不能用"通用 OTA 工具"覆盖所有场景
