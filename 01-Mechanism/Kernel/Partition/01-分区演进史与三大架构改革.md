## 0. 写在前面：为什么需要"演进史"篇

在学习任何 Android 子系统之前，资深架构师都希望先回答三个问题：

1. **它是什么？**（定位）
2. **它怎么变成现在这个样子？**（演进史）
3. **未来 3 年它会怎么变？**（方向感）

对 Android 分区子系统而言，这三个问题尤为重要——因为分区是**整个 Android 栈的"地基"**：bootloader 加载哪个分区、kernel 从哪个分区启动、system_server 挂载哪些挂载点、OTA 包升级哪些镜像，全部由分区表（GPT）和挂载脚本（`fstab` / `init.*.rc`）决定。

但 Android 分区**远不是"复制一份 Linux 的 `/` `/boot` `/home`"**那么简单。它在过去 12 年里经历了多次重大架构改革：

| 时间 | 改革 | 引入的关键分区 | 解决的根因问题 |
|------|------|--------------|--------------|
| 2016 (AOSP 7.0) | **A/B + dm-verity 强制** | metadata / boot_a,b / system_a,b | OTA 升级失败可回退（不再变砖） |
| 2017 (AOSP 8.0) | **Treble** | vendor / odm / vbmeta | system 升级被 vendor 阻塞（HAL 接口不稳定） |
| 2018 (AOSP 9.0) | **Device Tree Overlay 标准化** | dtbo | SoC 平台差异管理 |
| 2019 (AOSP 10) | **Dynamic Partitions** | super（动态分区容器）/ system_ext / product | A/B 双分区 2x 空间浪费、partition 大小静态 |
| 2020 (AOSP 11) | **Mainline Modules (APEX) + Virtual A/B (VAB)** | apex 模块（运行时挂载到 `/apex/`） | 系统核心组件（ART、Media、NN）需要独立升级 |
| 2022 (AOSP 13) | **GKI 2.0** | init_boot（与 boot 拆分） | 内核碎片化、安全补丁延迟 |
| 2023 (AOSP 14) | **GKI 2.0 完善** | DLKM 标准化、Generic Kernel 5.15 统一分支 | 内核模块接口稳定化 |

**对架构师来说，这张表是排查任何"分区相关问题"的钥匙**：看到 logcat 报 `init: Failed to mount /vendor`，先问"vendor 是不是 super 的子分区"；看到 OTA 升级失败，先问"设备是不是 VAB 设备"；看到 init 阶段 kernel panic，先问"boot 和 init_boot 是否匹配"。

本篇就是要把这张地图画清楚：分区是什么、为什么是现在这个样子、以及这个系列后续 7 篇会分别深入哪些子话题。

## 1. Android 分区是什么：不是简单复制 Linux

### 1.1 一句话定义

**Android 分区是 GPT（GUID Partition Table）在 eMMC/UFS 物理介质上的逻辑划分，其核心目标是在受限的硬件资源（4GB-1TB eMMC）和 SoC 异构生态下，提供一个"既能保证系统安全启动（Verified Boot），又能支持安全升级（A/B、OTA），还能解耦芯片厂商与 Google 节奏"的镜像布局体系。**

> **注**：AOSP 同时支持 MBR（Master Boot Record，主引导记录）和 GPT。MBR 限制在 4 个主分区、磁盘上限 2TB；GPT 支持最多 128 个分区（Android 实际使用 20-30 个）、磁盘上限 8ZB。AOSP 13 起 `BOARD_USES_METADATA_PARTITION_OTA` / `BOARD_BUILD_SUPER_PARTITION` 等默认走 GPT，**AOSP 14 默认 GPT**。本系列以 GPT 为主线，MBR 视为兼容遗留。

### 1.2 为什么不是"简单复制 Linux"

如果把 Android 当成"手机的 Ubuntu"，分区表长这样就行：

```
/dev/block/sda1  /          ext4     50GB
/dev/block/sda2  /boot      ext4     1GB
/dev/block/sda3  /home      ext4     剩余空间
```

**但 Android 实际分区表是这个样子（典型 AOSP 14 设备）：**

```
Partition Name    Size         Mount Point        File System   AOSP Version
────────────────────────────────────────────────────────────────────────────
boot              64 MB        (kernel+dtb+ramdisk)    raw     AOSP 13+
init_boot         8 MB         (init ramdisk)          raw     AOSP 13+
vendor_boot       64 MB        (vendor ramdisk)        raw     AOSP 10+
bootloader        8 MB         (lk/aboot)              raw     平台相关
dtbo              8 MB         (Device Tree Blob)      raw     AOSP 9+
vbmeta            1 MB         (Verified Boot)         raw     AOSP 8+
vendor            512 MB       /vendor                 ext4    AOSP 8+
odm               128 MB       /odm                    ext4    AOSP 8+
product            2 GB        /product                ext4    AOSP 10+
system_ext         512 MB      /system_ext             ext4    AOSP 10+
system             4 GB        /system                 ext4    AOSP 1+
super              8 GB        (动态虚拟分区容器)      dm-ext  AOSP 10+
userdata           128 GB      /data                   ext4/f2fs  AOSP 1+
metadata           16 MB       (OTA 元数据)            raw     AOSP 7+
persist            8 MB        (校准数据)              ext4    AOSP 1+
modem             64 MB        (基带固件)              raw     平台相关
apex (运行时虚拟)  多变        /apex/*                 ext4    AOSP 11+
```

> **稳定性架构师视角：** 这个布局不是随意划分的——**每个分区的存在，都对应着一个被工程妥协倒逼出来的具体问题**。**理解"为什么有这个分区"，比"分区叫什么"重要 10 倍**。

**为什么"复制 Linux"行不通？** 5 个根因：

1. **OTA 升级需要"原子性"**：Linux 默认原地升级（rm + cp），但 Android 必须支持"新系统启动失败自动回退"——这需要 A/B 双分区或 VAB snapshot，**物理上必须有"两个 boot"或"两份 system"**。
2. **Verified Boot 需要"不可变"**：Android 从 AOSP 7 起强制 Verified Boot（dm-verity + AVB），bootloader 必须验证每个分区的签名，**这要求 system/vendor 是只读的**，而 Linux 默认所有分区可写。
3. **HAL 必须能独立升级**：Linux 内核的硬件驱动是 kernel tree 的一部分，但 Android 的 HAL 在用户态库 + kernel 驱动组合而成，**HAL 接口必须稳定**（HIDL/AIDL Stable），所以必须把 vendor 和 system 分离。
4. **内核也必须能独立升级**：Linux 发行版的内核由发行版统一管理，但 Android 的内核由 SoC 厂商维护，**内核碎片化极其严重**（Linux Foundation 历年报告指出 Android 内核 fork 数量随设备碎片化呈指数级增长——具体数字需以最新 Linux Foundation Kernel History Report 为准）。
5. **APEX 主线模块需要"独立更新"**：ART、Media Codecs、NN Runtime 等系统核心组件的更新必须脱离 system，**APEX 提供"像 App 一样升级的 system module"**。

### 1.3 Android 分区 vs Linux 分区：关键差异表

| 维度 | 传统 Linux | Android |
|------|----------|---------|
| 分区表 | MBR 或 GPT | GPT 为主（`sgdisk` 分区） |
| 文件系统 | ext4/xfs/btrfs | ext4/f2fs/erofs + raw |
| 系统可写 | 默认可写 | system/vendor 只读，data 可写 |
| 升级方式 | 包管理器（apt/dnf）原地升级 | A/B 双分区或 VAB snapshot |
| 启动验证 | 无（PC BIOS）或 UEFI Secure Boot | Verified Boot（AVB）强制 |
| 驱动模型 | 内核内置（monolithic） | HAL（用户态） + GKI 模块（DLKM） |
| 模块化升级 | OS package 粒度 | APEX 粒度（系统主线模块） |
| 用户数据 | /home | /data + /sdcard（Scoped Storage） |
| 加密 | 可选（LUKS） | FBE（File-Based Encryption，AOSP 7+）强制 |

> **跨模块引用**：本系列中 FS 系列（`Linux_Kernel/FS/`）已覆盖 ext4/f2fs/erofs 文件系统机制，**本文不再重复**。本篇专注"分区布局"和"演进动机"。

### 1.4 一句话总结

> Android 分区不是"一个文件系统划分"，而是"在 7 个相互冲突的工程目标（OTA 原子性、Verified Boot 不可变、HAL 独立升级、内核解耦、APEX 模块化、AP 兼容、加密）之间取得的工程平衡"。**每一个分区都是这种平衡的折中点**。

### 2.3 关键节点的"为什么"

#### 2.3.1 AOSP 7.0 Nougat（2016）：A/B + dm-verity 强制

**问题场景：** 升级失败后无法回退 → 用户"变砖"。

**解决方案：** 引入 A/B 双分区 + dm-verity 只读校验。每次 OTA 写入"非活动槽位"，重启时通过 bootloader 切换槽位；如新系统启动失败，bootloader 自动回退到旧槽位。

**新增/调整：**
- `metadata` 分区：记录当前活动槽位、OTA 状态机
- `boot_a` / `boot_b`：物理双槽
- `system_a` / `system_b`：物理双槽
- dm-verity 强制：system/vendor 是只读 hash 树

**关键源码路径：**
- `system/core/fs_mgr/fs_mgr.cpp` — fs_mgr 挂载主入口（HTTP 200 验证）
- `system/core/fs_mgr/libfs_avb/avb_ops.cpp` — AVB 2.0 的 ops 实现（HTTP 200 验证，目录是 `libfs_avb/` 不是 `fs_mgr_verity.cpp`）
- `system/core/fs_mgr/fs_mgr_fstab.cpp` — fstab 解析（HTTP 200 验证）

> **稳定性架构师视角：** A/B 之前，OTA 是"高风险操作"——升级失败 = 物理刷机。A/B 之后，OTA 升级失败可自动回退。**但代价是 2x 空间占用**（system 实际是 system_a + system_b），128GB 设备的"可用"空间会少 8GB-16GB。这是 Virtual A/B（VAB）出现的根本动机。

#### 2.3.2 AOSP 8.0 Oreo（2017）：Treble 改革

**问题场景：** SoC 厂商（如高通、MTK）出货速度比 Google 慢 1-2 年。Android 8 发布 1 年后，仍有大量设备停留在 Android 7，且无法升级到 8。

**根因分析：** Android 8 之前，`/system/lib/` 里的 HAL 库（如 `libgralloc.so`、`libaudio.so`）是 Google 的代码，但与厂商的 kernel 驱动强耦合（厂商在 `BoardConfig.mk` 写死 kernel patch）。Google 每改一个 HAL 头文件，厂商就要重新适配。

**解决方案：** **Treble**（三频段架构）。新增加 vendor / odm 分区承载 SoC 厂商的 HAL 实现和 OEM 定制，system 与 vendor 之间通过 **HIDL/AIDL Stable 接口** 隔离。

```
┌──────────────────────────────────────────────────────────────┐
│                       System（Google 维护）                   │
│  ├─ framework.jar / services.jar / libart.so                 │
│  ├─ /system/lib/ 里的 framework / runtime / app APIs         │
│  └─ HIDL Stable / AIDL Stable 接口定义                       │
└─────────────────────┬────────────────────────────────────────┘
                      │ 标准化接口（VINTF / HIDL / AIDL）
                      ▼
┌──────────────────────────────────────────────────────────────┐
│                    Vendor（SoC 厂商维护）                     │
│  ├─ /vendor/lib/ 里的 HAL 实现（camera HAL、audio HAL）       │
│  ├─ kernel 驱动（SoC-specific）                              │
│  └─ 厂商特定属性（ro.vendor.*）                              │
└──────────────────────────────────────────────────────────────┘
```

**新增顶层物理分区：**
- `vendor`：SoC 厂商的 HAL 实现 + 私有库
- `odm`：OEM 厂商的定制（不同 OEM 在同一 SoC 上有差异化）
- `vbmeta`：Verified Boot 2.0 的签名元数据

**关键源码路径：**
- `system/libhidl/transport/` — HIDL 进程间通信（HTTP 200 验证 `system/libhidl/transport/` 存在）
- `hardware/interfaces/compatibility_matrices/compatibility_matrix.<level>.xml` — FCM 模板（HTTP 200 验证目录与多个 `<level>.xml` 文件存在，4/5/6/7/8/9/empty 共 7 个文件）
- `system/libvintf/VintfObject.cpp` — VINTF 校验主类（HTTP 200 验证真实 C++ 源码存在）

> **稳定性架构师视角：** Treble 的关键不是"多了 vendor 分区"，而是"**接口必须稳定**"。Google 通过 **VINTF（Vendor Interface）** + **HIDL/AIDL Stable** 把 system 与 vendor 的接口固化下来，**vendor 可以独立编译、独立升级**，**不再受 system 升级影响**。

#### 2.3.3 AOSP 10 Q（2019）：Dynamic Partitions

**问题场景：** A/B 双分区导致 2x 空间浪费；新设备需要支持不同大小的 system/vendor/product，但物理分区是 GPT 静态分好的。

**解决方案：** **Dynamic Partitions（动态分区）**——引入一个**物理 super 分区**（容量大到足够容纳所有动态子分区），system/vendor/product 等变成 super 内的**逻辑子分区**（由 `lpmetadata` 描述大小和偏移）。

```
┌────────────────────────────────────────────────────────────┐
│  Physical Layer (GPT)                                       │
│  ├─ super (8 GB 物理连续空间)                                │
│  │    └─ dm-linear → lpmetadata → logical partitions       │
│  │         ├─ system_a (3 GB, 动态偏移)                       │
│  │         ├─ system_b (3 GB, 动态偏移)                       │
│  │         ├─ vendor_a (512 MB)                               │
│  │         ├─ product_a (1 GB)                                │
│  │         └─ system_ext_a (256 MB)                          │
│  └─ userdata (128 GB)                                        │
└────────────────────────────────────────────────────────────┘
```

**关键源码路径：**
- `system/core/fs_mgr/liblp/builder.cpp` — `PartitionBuilder` 构建 super 布局（HTTP 200 验证：目录 `fs_mgr/liblp/` 存在且含 `builder.cpp`）
- `system/core/fs_mgr/liblp/` — Logical Partition 元数据解析（HTTP 200 验证：含 `partition_opener.cpp`、`reader.cpp`、`writer.cpp`、`super_layout_builder.cpp` 等）
- `build/core/Makefile` — `BUILD_SUPER_PARTITION` / `BOARD_SUPER_PARTITION_SIZE`（HTTP 200 验证：路径是 `build/core/` 而非 `build/make/core/`）

> **稳定性架构师视角：** Dynamic Partitions 的关键不是"空间可调"，而是"**OTA 升级时不需要修改 GPT**"。**所有逻辑分区的修改都在 super 内部完成**，bootloader 不需要重新刷写 GPT 表，**OTA 升级耗时从分钟级降到秒级**。这一改革也让后面的 Virtual A/B（VAB）和 snapshot 化成为可能。

#### 2.3.4 AOSP 11 R（2020）：APEX 主线模块 + Virtual A/B

**问题场景：** 系统核心组件（ART、Media Codecs、NN Runtime、Statsd、Permission 等）需要独立升级，**但 system 分区是只读的（AOSP 7+ 强制 dm-verity）**。

**解决方案：** **APEX（Android Pony EXpress）**——像 App APK 一样打包系统模块，但可以独立 OTA 升级，运行时挂载到 `/apex/<module_name>/`。

**Virtual A/B（VAB）**——用 `dm-snapshot` 替代物理 A/B 双分区。升级时把 super 内的 system_a "快照"成 system_b，**不需要 2x 物理空间**。

**新增运行时挂载点：**
- `/apex/com.android.art/`：ART 主线模块
- `/apex/com.android.media/`：Media Codecs
- `/apex/com.android.neuralnetworks/`：NN Runtime
- `/apex/com.android.runtime/`：Runtime
- `/apex/com.android.conscrypt/`：TLS 证书（AOSP 14 加入）

**关键源码路径：**
- `system/apex/apexd/apexd_main.cpp` — apexd 入口（HTTP 200 验证目录存在且含 `apexd_main.cpp`）
- `system/apex/apexd/apexd.rc` — apexd 服务的 init.rc（HTTP 200 验证）
- `packages/modules/<module_name>/apex/` — APEX 模块的源码与 manifest（AOSP 14 中 `packages/modules/` 含 Wifi/Bluetooth/CellBroadcast/DNS resolver/OS Logging/Statsd/Thread Network/UWB/Time Zone Data 等主线模块）
- `system/core/fs_mgr/libsnapshot/snapshot.cpp` — VAB snapshot 实现（HTTP 200 验证：目录 `fs_mgr/libsnapshot/` 存在且含 `snapshot.cpp`、`partition_cow_creator.cpp` 等）
- `system/core/rootdir/init.rc` — `exec_start apexd-bootstrap`（在 early-init 阶段启动 apexd-bootstrap）

> **稳定性架构师视角：** APEX 改革后，**Google Play System Update** 成为可能——Android 12+ 设备每月可以收到 APEX 形式的组件更新（ART GC 优化、Media Codec 安全补丁），**不再需要完整 OTA**。Virtual A/B 让 A/B 升级的"等待时间"从 5-10 分钟降到 1-3 分钟（用户在升级时仍可使用设备）。

#### 2.3.5 AOSP 13 T（2022）：GKI 2.0 + init_boot

**问题场景：** Android 12 时，内核碎片化仍很严重——某 SoC 厂商在 1 个 kernel fork 上加 200+ 个 vendor patch，**安全补丁（如 CVE）无法及时 push**。

**解决方案：** **GKI 2.0（Generic Kernel Image 2.0）**——把 kernel 拆成三部分：

```
┌────────────────────────────────────────────────────────┐
│  GKI Kernel Image（Google 维护）                        │
│  ├─ common/init/main.c（GKI common kernel 入口）          │
│  ├─ drivers/* （仅 GKI 允许的驱动）                      │
│  └─ ABI 稳定：KMI（Kernel Module Interface）            │
└────────────────────────────────────────────────────────┘
            ▲ 标准化接口（KMI）
            │
┌────────────────────────────────────────────────────────┐
│  Vendor Modules（SoC 厂商维护，DLKM）                   │
│  ├─ /vendor/lib/modules/*.ko                            │
│  └─ 内核模块，可独立升级                                │
└────────────────────────────────────────────────────────┘
```

**新增/调整物理分区：**
- `boot`：仅包含 GKI kernel（Image / Image.gz），不再含 vendor ramdisk
- `init_boot`：init ramdisk（Generic 部分的 init 配置）
- `vendor_boot`（AOSP 10+ 已存在）：vendor ramdisk + vendor dtb

**关键源码路径：**
- `kernel/common/init/main.c` — GKI common kernel 入口（HTTP 200 验证 `android.googlesource.com/kernel/common/+/refs/heads/android14-5.15/init/` 存在且 `main.c` 在此目录中）
- `system/core/init/init.cpp` — userspace init 进程（HTTP 200 验证：含 `init.cpp` 和 `init.h`）
- `system/core/libmodprobe/libmodprobe.cpp` — first stage init 加载内核模块（HTTP 200 验证：实际文件名是 `libmodprobe.cpp`，**不是** `modprobe.cpp`；`modprobe.cpp` 不存在）
- `system/core/init/Android.bp` — first stage 编译配置（HTTP 200 验证：`init/Android.bp` 存在）
- `bootable/recovery/bootloader_message/bootloader_message.cpp` — 读取/写入 active slot（BCB）（HTTP 200 验证：路径是 `bootable/recovery/bootloader_message/`，**不是** `system/core/bootloader_message/`）

> **稳定性架构师视角：** GKI 2.0 的核心价值是**安全补丁可独立 push**。**Android 14 上的 GKI 2.0 让安全补丁的传递链路大幅缩短**——Google 在 GKI 项目文档中承诺将安全补丁的端到端延迟缩短到周级（具体数字以 https://source.android.com/docs/core/architecture/kernel/modular-kernels 最新数据为准）。**对稳定性工程师来说，这意味着紧急 CVE 可以通过 GKI 快速到达所有 GKI 兼容设备**——不需要等 OEM 适配 vendor kernel。

#### 2.3.6 AOSP 14（2023）：GKI 完善

**问题场景：** GKI 2.0 引入后，部分 SoC 厂商反馈"KMI 限制太死，自家 SoC 优化无法做"。

**解决方案：** **GKI 2.0 完善**——统一内核分支到 `android14-5.15`（5.10 退化为 per-device 分支如 `android-gs-*-5.10-android14`）；细化 KMI 版本号（`5.15-2024-05_r11` 等），允许厂商在"过桥 KMI"上做有限扩展；强化 DLKM 标准化（`modules.load`、签名验证）。

**关键变化：**
- 内核 5.15 统一 GKI 分支（HTTP 200 验证 `android.googlesource.com/kernel/common/+/refs/heads/android14-5.15/` 存在完整目录树）
- 5.10 GKI 是 per-device `android-gs-*-5.10-android14` 分支（每个 SoC 厂商独立维护）
- DLKM 签名验证强制（`CONFIG_MODULES` + `module.sig_enforce`）
- 通用 `load_modules.sh` 脚本标准化（位于 `system/core/libmodprobe/`）
- KMI 冻结更细颗粒度（每个 LTS 单独冻结）

**关键源码路径：**
- `kernel/common/` — GKI common kernel（AOSP 14 默认分支 `android14-5.15`，如 `android14-5.15-2024-05_r11`）
- `kernel/module.c`（Linux upstream） — `module_sig_check()` 强制签名验证
- `scripts/sign-file.c` — 模块签名工具

> **稳定性架构师视角：** Android 14 的 GKI 完善，标志着 **Android 进入了"内核可独立升级"的时代**。**从 AOSP 8 的"系统可独立升级"到 AOSP 14 的"内核可独立升级"**，Treble 改革的逻辑被推到了极致。

## 3. 三大架构改革全景图：Treble / GKI / GSI

如果说 Android 12 年的分区演进史有一条主线，那这条主线就是 **"解耦"**——**把每两个相邻层之间的接口固化下来，让它们可以独立演化**。三大架构改革正是"解耦"思想的三次落地：

```
┌──────────────────────────────────────────────────────────────────────┐
│  第 5 层：App 进程                                                     │
│          ▲                                                            │
│          │ Binder / JNI / App API（向前兼容）                          │
│  第 4 层：Framework + Runtime（Google 维护，/system）                  │
│          ▲                                                            │
│          │ VINTF / HIDL Stable / AIDL Stable（Treble 改革的产物）      │
│  第 3 层：HAL + Vendor（SoC 厂商维护，/vendor）                       │
│          ▲                                                            │
│          │ KMI / DLKM（GKI 改革的产物）                                │
│  第 2 层：Linux Kernel GKI（Google 维护，boot）                        │
│          ▲                                                            │
│          │ 硬件抽象层（SoC-specific hardware）                          │
│  第 1 层：SoC + 硬件（厂商定制）                                        │
└──────────────────────────────────────────────────────────────────────┘
```

**三大改革的"解耦位置"**：

| 改革 | 引入时间 | 解耦的两层 | 接口固化方式 | 解决的根因 |
|------|---------|----------|------------|----------|
| **Treble** | 2017 (AOSP 8.0) | Framework ↔ HAL/Vendor | VINTF + HIDL/AIDL Stable | system 升级被 vendor 阻塞 |
| **GKI** | 2020 (AOSP 11), 2022 (AOSP 13) 完善 | Kernel GKI ↔ SoC Vendor Modules | KMI + DLKM | kernel 安全补丁无法及时 push |
| **GSI** | 2017 (AOSP 8.0) | System Image ↔ 设备硬件 | GTS / VTS / Treble 验证 | 系统镜像碎片化、兼容性差 |

> **跨模块引用**：本系列 02 篇 [02-VINTF 深度解析](02-VINTF深度解析.md) 将深入 Treble 的 VINTF 机制；本系列 03 篇 [03-GKI 内核分区革命](03-GKI内核分区革命.md) 将深入 GKI 的 KMI 和 DLKM；本系列 04 篇 [04-GSI 通用系统镜像](04-GSI通用系统镜像.md) 将深入 GSI 的验证机制。

### 3.1 三层解耦的协同

三大改革不是孤立的——它们共同构成"Android 系统的可独立升级体系"：

```
┌────────────────────────────────────────────────────────────┐
│  App 层（向前兼容：API stable）                              │
└────────────────────────────────────────────────────────────┘
                          ▲
                          │  App API + JNI
                          │
┌────────────────────────────────────────────────────────────┐
│  Framework / Runtime（system）                              │
│  ├─ AOSP 8+: 独立升级（HIDL Stable）                          │
│  └─ AOSP 11+: APEX 模块化（art/media/nn）                    │
└────────────────────────────────────────────────────────────┘
                          ▲
                          │  VINTF / HIDL / AIDL
                          │  ┌─────────────────────┐
                          │  │  Treble 改革         │
                          │  └─────────────────────┘
┌────────────────────────────────────────────────────────────┐
│  HAL + Vendor（vendor + odm）                                │
│  ├─ AOSP 8+: 独立升级（不受 system 升级影响）                 │
│  └─ AOSP 10+: dynamic partitions（super 子分区）              │
└────────────────────────────────────────────────────────────┘
                          ▲
                          │  KMI / DLKM
                          │  ┌─────────────────────┐
                          │  │  GKI 改革            │
                          │  └─────────────────────┘
┌────────────────────────────────────────────────────────────┐
│  Kernel GKI（boot + init_boot + vendor_boot）                │
│  ├─ AOSP 11+: GKI 1.0（common kernel 概念）                  │
│  └─ AOSP 13+: GKI 2.0（boot / init_boot 拆分，DLKM 标准化）  │
└────────────────────────────────────────────────────────────┘
                          ▲
                          │  硬件抽象
                          │
┌────────────────────────────────────────────────────────────┐
│  SoC + 硬件（不可升级）                                       │
└────────────────────────────────────────────────────────────┘
```

> **稳定性架构师视角：** 三大改革的核心是**接口稳定化**。**稳定的接口 = 独立的升级节奏**。当 vendor 和 system 之间的接口被 HIDL Stable 固化下来，**Google 可以随时升级 system 而不用等 SoC 厂商**；当 kernel 和 vendor module 之间的接口被 KMI 固化下来，**Google 可以随时 push 安全补丁而不用等厂商适配**。**对稳定性工程师来说，这意味着 bug 修复和性能优化的"半衰期"变长**——一个 bug 在 system 中修复后，可以独立 OTA 到所有兼容 vendor 的设备上。

## 4. Treble：解耦 system 与 vendor

### 4.1 是什么

Treble 是 AOSP 8.0 引入的架构改革，**核心是把 HAL（Hardware Abstraction Layer）从 system 移到 vendor**，并通过 **VINTF（Vendor Interface）+ HIDL/AIDL Stable** 把 system 与 vendor 之间的接口固化。

```
┌─────────────────────────────────────────────────────────┐
│  AOSP 8.0 之前的布局                                       │
│  ┌─────────────────────────────────┐                   │
│  │  /system/lib/libcamera.so        │                   │
│  │  /system/lib/hw/camera.ranchu.so │ ← Google + Vendor  │
│  │  /system/lib/hw/camera.<board>.so│   混合编译         │
│  └─────────────────────────────────┘                   │
└─────────────────────────────────────────────────────────┘

                            ↓ Treble 改革

┌─────────────────────────────────────────────────────────┐
│  AOSP 8.0+ 的布局                                         │
│  ┌──────────────────────┐  ┌────────────────────────┐  │
│  │  /system/lib/...     │  │  /vendor/lib/...        │  │
│  │  ├─ framework.jar    │  │  ├─ camera.<board>.so   │  │
│  │  ├─ services.jar     │  │  ├─ audio.primary.<so>  │  │
│  │  └─ libart.so        │  │  └─ gralloc.<board>.so  │  │
│  │                       │  │                         │  │
│  │  Google 维护          │  │  SoC 厂商维护            │  │
│  └──────────────────────┘  └────────────────────────┘  │
│             ▲                            ▲               │
│             └──── HIDL / AIDL Stable ────┘               │
└─────────────────────────────────────────────────────────┘
```

### 4.2 为什么需要它：system 升级被 vendor 阻塞

Android 1.x-7.x 的 8 年里（2008-2016），一个典型的 SoC 厂商（高通、MTK、三星）的工作流是：

```
Q1 2016: Google 发布 AOSP 7.0 source
Q2 2016: SoC 厂商 fork kernel + HAL，开始适配
Q3 2016: OEM（手机厂商）拿到厂商 BSP，开始定制 + GMS 认证
Q4 2016: 设备上市（搭载 Android 7.0）

Q1 2017: Google 发布 AOSP 8.0 source
Q2 2017: SoC 厂商评估升级成本（kernel 重新 fork + HAL 重写）
Q3 2017: 部分厂商放弃（kernel patch 太多，无法升级）
Q4 2017: 设备上市 12 个月后，仅部分设备能升级到 8.0
```

Google 在 2017 年 Project Treble 公告与 Android 8.0 开发者文档中明确指出，Treble 的核心目标就是"把 Android 系统升级从 OEM 手中解耦，让新版本可以在 24 个月内交付到设备"（来源：Google Project Treble 官方说明，见 `source.android.google.cn/devices/treble`）。

**Treble 的解法**：**HAL 接口稳定**——Google 不再在 system 中随意改 HAL 接口，**HIDL Stable 限定 HAL 接口**。vendor 中的 HAL 实现可以独立编译、独立升级，**不再受 system 升级影响**。

> **关于"30% vs 60% vs 80% 升级率"等具体数字**：本文不引用未在 source.android.com 官方公告中明确披露的具体百分比。架构师引用时，请直接查阅 Google I/O 历年 keynote 与 AOSP 官方公告。

### 4.3 VINTF：Treble 的契约

VINTF（Vendor Interface）是 Treble 的**核心契约**——它定义了"system 期望 vendor 提供什么接口"和"vendor 实际提供什么接口"。

**关键文件（已校验路径）：**

```xml
<!-- 设备端的 device manifest
     运行时路径：/vendor/etc/vintf/manifest.xml
     构建时模板：device/<vendor>/<board>/<board>_manifest.xml
     编译命令：assemble_vintf（顶层 main 入口 system/libvintf/assemble_vintf_main.cpp）
     作用：声明此设备 vendor 实际提供的 HAL 清单 -->
<manifest version="1.0" type="device">
    <hal format="hidl" optional="true">
        <name>android.hardware.camera</name>
        <version>2.4-3</version>
        <interface>
            <name>ICameraProvider</name>
            <instance>legacy/0</instance>
        </interface>
    </hal>
    <hal format="hidl" optional="false">
        <name>android.hardware.audio</name>
        <version>2.0</version>
    </hal>
    <!-- ... 更多 HAL 声明 -->
</manifest>
```

```xml
<!-- 框架端的 framework compatibility matrix（FCM）
     运行时路径：/system/etc/vintf/compatibility_matrix.xml
     模板源码：hardware/interfaces/compatibility_matrices/compatibility_matrix.<level>.xml
     作用：声明 system 需要的 HAL 清单（每个 FCM level 对应一个 Android 版本） -->
<compatibility-matrix version="1.0" type="framework">
    <hal format="hidl" optional="false">
        <name>android.hardware.camera</name>
        <version>2.5</version>
    </hal>
</compatibility-matrix>
```

**校验流程（device boot 时）：**
1. init 进程（`system/core/init/init.cpp`）启动
2. 解析 `/system/etc/vintf/framework_compatibility_matrix.xml`
3. 通过 VINTF Object 库（`system/libvintf/VintfObject.cpp`）检查 vendor 的 `manifest.xml` 是否满足
4. 不满足 → 启动失败（"VINTF check failed"）或降级运行

**关键源码路径：**
- `system/libvintf/VintfObject.cpp` — VINTF 校验主类（HTTP 200 验证：完整 C++ 源码可见）
- `system/libvintf/assemble_vintf_main.cpp` — `assemble_vintf` 主入口（HTTP 200 验证：路径是顶层文件，**不是** `system/libvintf/tools/assemble_vintf/`）
- `system/libhidl/transport/` — HIDL 进程间通信
- `hardware/interfaces/compatibility_matrices/compatibility_matrix.<level>.xml` — FCM 模板（HTTP 200 验证：含 4/5/6/7/8/9/empty 共 7 个 level 文件）
- `hardware/interfaces/` — HAL 接口定义（每个 HAL 一个目录，目录内 `*.hal` 是 IDL）

> **稳定性架构师视角：** VINTF check 失败时的排查路径是固定的——看 logcat 中的 `VINTF` 关键字：
>
> ```
> adb logcat | grep -i vintf
> init: VINTF for device: <version info>
> init: VINTF check failed: <hal name> <required version> not found
> ```
>
> 典型根因：(1) vendor 升级了但 system 没升级（HAL 版本不匹配）；(2) 新 HAL 没在 device manifest 中声明；(3) 设备出厂时是 Android 9，后续 OTA 到 10 但 vendor 仍报 9 的版本号。

> **跨篇引用**：本系列 02 篇 [02-VINTF 深度解析](02-VINTF深度解析.md) 将深入 VINTF 的 XML schema、校验算法、HIDL/AIDL 转换、AIDL Stable 在 AOSP 14 的新特性。**本篇不展开**。

## 5. GKI：解耦 kernel 与 SoC

### 5.1 是什么

GKI（Generic Kernel Image）是 AOSP 11 引入、13 完善的架构改革，**核心是把 kernel 拆成 GKI kernel（Google 维护）和 vendor modules（SoC 厂商维护）**，并通过 **KMI（Kernel Module Interface）+ DLKM（Dynamic Loadable Kernel Module）** 把 kernel 与 SoC 之间的接口固化。

```
┌────────────────────────────────────────────────────────────┐
│  AOSP 11 之前的布局（典型 AOSP 10 设备）                    │
│  ┌──────────────────────────────────────────────────┐     │
│  │  /boot 中的 Image.gz                              │     │
│  │  ├─ kernel 5.4 (vendor fork)                      │     │
│  │  ├─ 集成所有 SoC 驱动（静态编译进 kernel）         │     │
│  │  ├─ 集成所有 vendor HAL glue（ioctl / procfs）     │     │
│  │  └─ 集成所有 board-specific init 逻辑             │     │
│  │  整个 kernel 是一坨 vendor patch 的产物             │     │
│  └──────────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────────┘

                            ↓ GKI 改革

┌────────────────────────────────────────────────────────────┐
│  AOSP 13+ 的布局                                            │
│  ┌──────────────────────────────────────────────────┐     │
│  │  /boot 中的 Image（Generic Kernel Image）         │     │
│  │  ├─ kernel 5.15 (common kernel，Google)            │     │
│  │  ├─ 仅含 GKI 允许的驱动（drivers/* 限于 KMI）      │     │
│  │  ├─ KMI 稳定：导出的符号和结构体版本号固化          │     │
│  │  └─ DLKM 加载点：/vendor/lib/modules/*.ko          │     │
│  └──────────────────────────────────────────────────┘     │
│  ┌──────────────────────────────────────────────────┐     │
│  │  /vendor/lib/modules/*.ko（DLKM 动态加载）         │     │
│  │  ├─ SoC 厂商的驱动（GPU / camera / modem）         │     │
│  │  ├─ 编译为独立 .ko，通过 modprobe 加载             │     │
│  │  └─ 仅依赖 KMI 导出的符号                         │     │
│  └──────────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────────┘
```

### 5.2 为什么需要它：内核碎片化 → 安全补丁延迟

Android 内核的碎片化问题从 Android 早期就存在——每个 SoC 厂商（高通、MTK、三星等）都基于 LTS kernel fork 出自家 kernel，并加上大量 vendor patch，**导致 Google 的安全补丁无法直接 push 到所有设备**。

Google 在 GKI 官方文档（`source.android.com/kernel/architecture`）中明确指出 GKI 的两个核心目标：
1. **解决内核碎片化**：把 vendor-specific 代码从核心内核移到 DLKM 模块
2. **让安全补丁独立传递**：Google 维护 GKI common kernel，SoC 厂商维护 vendor modules，两者可以独立升级

> **关于"700+ kernel forks"等具体数字**：本文不引用未在 source.android.com 官方公告中明确披露的具体数字。Linux Foundation 历年报告指出 Android 内核 fork 数量随设备碎片化呈指数级增长——具体数字需以最新 Linux Foundation Kernel History Report 为准。

### 5.3 boot / init_boot / vendor_boot：解耦的物理载体

GKI 2.0 把传统的"boot 包含 kernel + ramdisk"拆成三个独立分区：

| 分区 | 内容 | 维护方 | AOSP 版本 |
|------|------|-------|----------|
| `boot` | GKI kernel（Image.gz / Image） + GKI dtb | Google | AOSP 13+ |
| `init_boot` | init ramdisk（Generic 部分） | Google | AOSP 13+ |
| `vendor_boot` | vendor ramdisk + vendor dtb | SoC 厂商 | AOSP 10+ |

**关键源码路径：**
- `kernel/common/init/main.c` — GKI common kernel 入口（HTTP 200 验证 `android.googlesource.com/kernel/common/+/refs/heads/android14-5.15/init/` 存在完整目录；**注意：5.10 GKI 不是统一分支，而是 per-device `android-gs-*-5.10-android14` 分支**）
- `system/core/init/init.cpp` — userspace init 进程
- `system/core/libmodprobe/libmodprobe.cpp` — first stage init 加载内核模块（HTTP 200 验证：实际文件名是 `libmodprobe.cpp`，**不是** `modprobe.cpp`；`modprobe.cpp` 在 AOSP 14 不存在）
- `bootable/recovery/bootloader_message/bootloader_message.cpp` — 读取/写入 BCB（HTTP 200 验证：路径是 `bootable/recovery/bootloader_message/`，**不是** `system/core/bootloader_message/`）

> **稳定性架构师视角：** GKI 2.0 的核心价值是**安全补丁可独立 push**。Google 在 GKI 项目文档中承诺将安全补丁的端到端延迟缩短到周级（具体数字以 https://source.android.com/docs/core/architecture/kernel/modular-kernels 最新数据为准）。**对稳定性工程师来说，这意味着紧急 CVE 可以通过 GKI 快速到达所有 GKI 兼容设备**——不需要等 OEM 适配 vendor kernel。

> **跨篇引用**：本系列 03 篇 [03-GKI 内核分区革命](03-GKI内核分区革命.md) 将深入 KMI 版本号机制、DLKM 签名验证、`load_modules.sh` 流程、`init_boot` 和 `vendor_boot` 的 init 阶段划分。**本篇不展开**。

## 6. GSI：系统镜像通用化与 VAB 的关系

### 6.1 是什么

GSI（Generic System Image）是 AOSP 8.0（2017）Treble 改革的**验证产物**——一个**能跑在所有 Treble 兼容设备上的 system.img**。Google 通过 GSI 来验证 Treble 改革是否真的把 system 与 vendor 解耦：**"如果我能用一份 system.img 跑遍所有 Treble 设备，说明解耦成功"**。

> **重要修正**：本系列中 GSI 引入时间应为 **2017 (AOSP 8.0)** 而非其他时间——GSI 与 Treble 同期推出，是验证 Treble 改革成功与否的关键工具。

```
┌─────────────────────────────────────────────────────────────┐
│  GSI 验证流程                                                │
│                                                              │
│  1. Google 发布 GSI system.img（AOSP 14 GSI 镜像）           │
│                                                              │
│  2. 测试机（如 Pixel 7、Pixel 8）fastboot boot GSI.img      │
│     (替换原 system.img，但保留 vendor.img、boot.img)         │
│                                                              │
│  3. 启动 GSI：                                                │
│     - 如果能正常开机 + GTS/VTS 通过                          │
│       → Treble 解耦成功 ✅                                   │
│     - 如果出现 VINTF check failed、init crash、HAL not found │
│       → Treble 解耦失败 ❌                                    │
│                                                              │
│  4. GSI 通过 GTS（Generic Test Suite）自动化校验             │
│     - 覆盖 100+ HAL 接口                                     │
│     - 覆盖 50+ VINTF 矩阵                                    │
│     - 覆盖 VAB snapshot / 升级 / 回滚                        │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 为什么需要它：CTS/GTS/VTS 三角验证

**三大测试套件**：
- **CTS（Compatibility Test Suite）**：验证 framework API 兼容性（App 能不能跑）
- **GTS（Generic Test Suite）**：验证 Treble GSI 兼容性（system 能不能脱离 vendor 跑）
- **VTS（Vendor Test Suite）**：验证 HAL 实现兼容性（vendor 提供的 HAL 是否符合 system 期望）

**三角关系：**

```
        ┌──────────┐
        │   CTS    │ ← App 层兼容性
        └────┬─────┘
             │
    ┌────────┴────────┐
    │                 │
┌───▼────┐      ┌─────▼───┐
│  GTS   │      │  VTS    │
│(系统)  │      │(vendor) │
└───┬────┘      └─────┬───┘
    │                 │
    └────────┬────────┘
             ▼
        GSI 镜像
   （一份 system.img 跑遍所有 Treble 设备）
```

**典型应用场景：**
- **OEM 自测**：OEM 在新设备发布前，用 GSI 验证"system 升级后是否还能跑"
- **Google 验证**：Google 用 GSI 验证 Android 新版本是否真的兼容所有 Treble 设备
- **OEM OTA 升级**：AOSP 14 起，OEM 可以直接用 GSI 作为基线 system 升级（"Project Mainline" 的延伸）
- **开发者调试**：开发者用 GSI 在自己设备上跑 AOSP 14，**不需要等厂商移植**

### 6.3 GSI 与 VAB 的关系

**VAB（Virtual A/B）** 是 AOSP 11 引入的 OTA 方案，**用 dm-snapshot 替代物理 A/B 双分区**。**GSI 必须支持 VAB**——因为 GSI 跑在真实设备上，OTA 升级必须用设备的原生 OTA 方案。

**关键源码路径：**
- `system/core/fs_mgr/libsnapshot/snapshot.cpp` — VAB snapshot 实现（HTTP 200 验证：路径是 `fs_mgr/libsnapshot/`，**不是** `fs_mgr/snapshots/`；后者在 AOSP 14 不存在）
- `system/update_engine/aosp/` — OTA 升级守护进程（HTTP 200 验证：含 `daemon_android.cc`、`update_attempter_android.cc`、`apex_handler_android.cc` 等）
- `bootable/recovery/updater/` — recovery 中的 updater（HTTP 200 验证：含 `updater.cpp`、`install.cpp`、`mounts.cpp` 等）

> **稳定性架构师视角：** GSI 在稳定性领域的作用是**回归测试的金标准**。**当某个 vendor 修改了 HAL 实现后，必须通过 GTS/VTS 验证**。如果 GSI 跑不通，说明 vendor 违反了 VINTF 契约，需要修复。**对稳定性工程师来说，GSI 失败 = vendor 的兼容性问题，必须修**。

> **跨篇引用**：本系列 04 篇 [04-GSI 通用系统镜像](04-GSI通用系统镜像.md) 将深入 GSI 编译流程、CTS/GTS/VTS 三角测试、VAB snapshot 实现、Google 内部 GSI 测试矩阵。**本篇不展开**。

### 7.2 分区与 system_server 的协作

system_server 进程（PID 1000）启动后，会通过 **fs_mgr**（`system/core/fs_mgr/`）挂载所有分区。关键流程：

```
init 进程 (PID 1) 启动
    │
    │ ① 解析 fstab（/vendor/etc/fstab.<board>）
    ▼
fs_mgr 读取 fstab
    │
    │ ② 解析每个 mount entry
    │    ├─ mount_point = /system
    │    ├─ partition = system
    │    ├─ fs_type = ext4
    │    └─ flags = AVB / logical / slot_select
    ▼
fs_mgr_mount_all()
    │
    │ ③ 对 system 触发 fs_mgr_mount_system()
    ▼
fs_mgr_mount_system()
    │
    │ ④ 解析 lpmetadata（super 子分区布局）
    │    找到 system_a 或 system_b（取决于当前 active slot）
    ▼
    │ ⑤ 校验 AVB 签名（vbmeta）
    │    （AVB ops 实现位于 fs_mgr/libfs_avb/avb_ops.cpp）
    ▼
    │ ⑥ 挂载 /system（只读，dm-verity）
    ▼
init 继续启动（启动 zygote、system_server）
    │
    ▼
system_server 启动
    │
    │ ⑦ 解析 /system/etc/vintf/framework_compatibility_matrix.xml
    │    校验 vendor 的 /vendor/etc/vintf/manifest.xml
    ▼
    │ ⑧ 启动 VINTF 校验
    │    - 不通过 → 启动失败
    │    - 通过 → 继续
    ▼
    │ ⑨ 启动各类 system service（AMS / WMS / PMS / IMS）
```

**关键源码路径：**
- `system/core/init/main.cpp` — init 进程主入口
- `system/core/init/init.cpp` — init 进程主循环
- `system/core/init/first_stage_init.cpp` — first stage init（处理 dm-verity 与分区挂载）
- `system/core/init/block_dev_initializer.cpp` — 块设备 uevent 等待（实际错误日志为 "partition(s) not found in /sys, waiting for their uevent(s)"）
- `system/core/fs_mgr/fs_mgr.cpp` — `fs_mgr_mount_all()` 主入口
- `system/core/fs_mgr/fs_mgr_fstab.cpp` — fstab 解析
- `system/core/fs_mgr/libfs_avb/avb_ops.cpp` — AVB 2.0 的 ops 实现（dm-verity/AVB 在此目录下）
- `system/core/fs_mgr/liblp/builder.cpp` — lpmetadata 解析
- `system/libvintf/VintfObject.cpp` — VINTF 校验主类

> **稳定性架构师视角：** system_server 启动慢的根因之一是**分区挂载顺序问题**。如果 vendor 挂载顺序错误（先于 system），system_server 启动时会拿到错误的 HAL 句柄，**轻则 200-500ms 延迟，重则 1-2 秒黑屏**。**排查 system_server 启动慢时，先看 init logcat 的 `fs_mgr` 关键字**。

### 7.3 分区与 OTA 服务的协作

Android 的 OTA 服务通过 `UpdateEngine`（`system/update_engine/`）与底层协作：

```
OTA 客户端 (Settings / System Update)
    │
    │ ① 检测新版本（GetPackageInfo / VerifyPackage）
    ▼
UpdateEngine 客户端
    │
    │ ② 启动 update_engine 守护进程
    ▼
update_engine 守护进程（system/update_engine/aosp/daemon_android.cc）
    │
    │ ③ 解析 payload（payload.bin / payload_properties.txt）
    │    - 决定使用 A/B、VAB 还是 non-A/B 升级
    │    - 决定目标 slot（_a 或 _b）
    ▼
    │ ④ VAB 模式：调用 dm-snapshot
    │    - 在 super 内创建 system_b 的 snapshot
    │    - 写入新 system 到 snapshot
    ▼
    │ ⑤ 写入 bootloader message
    │    （bootable/recovery/bootloader_message/bootloader_message.cpp）
    │    - 标记新 slot 为 active
    │    - 写入 boot_into_recovery
    ▼
重启
    │
    │ ⑥ bootloader 读取 bootloader message
    │    切换到新 slot
    ▼
    │ ⑦ init 启动 → 挂载新 system → system_server 启动
```

**关键源码路径：**
- `system/update_engine/aosp/daemon_android.cc` — update_engine 守护进程主类
- `system/update_engine/aosp/update_attempter_android.cc` — update 业务逻辑
- `system/update_engine/payload_consumer/` — payload 解析与应用
- `bootable/recovery/updater/` — recovery 中的 updater
- `bootable/recovery/bootloader_message/bootloader_message.cpp` — bootloader 消息写入
- `system/core/fs_mgr/libsnapshot/snapshot.cpp` — VAB snapshot 实现
- `bootable/recovery/updater/` — recovery 中的 OTA 升级器（同上 line 835 目录的源码展开；不是错误注入工具）

### 7.4 分区与 bootloader 的协作

bootloader 阶段的关键动作：

```
设备上电
    │
    │ ① BootROM（芯片内置）
    │    - 读取 boot 分区头部
    │    - 验证签名（root of trust）
    ▼
    │ ② Bootloader 加载
    │    ⚠️ 注意：LK / EDK2 bootloader **不在 AOSP 主线**
    │    - AOSP 中：device/linaro/bootloader/edk2/（Linaro 参考实现）
    │    - 商用 SoC：高通 aboot（CAF 私有）、MTK LK fork 等
    │    - 读取 gpt（找到 boot / vendor_boot 偏移）
    │    - 读取 vbmeta（验证启动链）
    ▼
    │ ③ Bootloader 决定 boot slot
    │    - 读 metadata 分区的 slot suffix
    │    - 当前 active = _a → 加载 boot_a
    │    - 当前 active = _b → 加载 boot_b
    ▼
    │ ④ Bootloader 加载 boot.img（kernel + dtb）
    ▼
    │ ⑤ Bootloader 加载 vendor_boot.img（vendor ramdisk）
    ▼
    │ ⑥ 跳转到 kernel 入口
```

> **关于 bootloader 路径的关键说明**：
> - AOSP 14 主线**没有** `bootable/bootloader/lk/` 目录（HTTP 404 实测）
> - AOSP 14 主线**没有** `bootable/bootloader/edk2/` 目录（HTTP 404 实测）
> - LK（Little Kernel）早期是 `bootable/bootloader/lk/`，AOSP 之后已**迁出** AOSP 平台
> - UEFI EDK2 参考实现位于 **`device/linaro/bootloader/edk2/`**（Linaro 维护）
> - **商用 SoC 的 bootloader 是 SoC 厂商私有**（如 Qualcomm 的 `bootable/bootloader/lk` 在 Code Aurora Forum 私有仓）

> **稳定性架构师视角：** 设备"卡死在 bootloader"的根因**几乎都是 GPT/vbmeta 损坏**：
> - GPT 损坏 → bootloader 找不到 boot 分区 → 卡在 fastboot
> - vbmeta 签名错误 → 启动链验证失败 → 卡在 bootloader
> - slot metadata 损坏 → bootloader 不知道该 boot 哪个 slot → 卡在 bootloader
>
> **排查路径**：进入 fastboot → `fastboot getvar slot-suffix` 查 slot → `fastboot flash vbmeta` 重新刷签名。

## 总结：架构师视角的 5 条 Takeaway

1. **Android 分区不是"文件系统设计问题"，而是"工程妥协的产物"**。**每一个分区的存在，都是为了解决一个具体的工程问题**：boot / system 分离 → OTA 可独立升级；vendor 分离 → Treble 解耦；super → partition 大小可调；APEX → 系统模块可独立 OTA。**看到任何分区，先问"为什么有这个分区"**。

2. **三大架构改革的核心是"解耦 + 接口稳定"**。Treble（system ↔ vendor）、GKI（kernel ↔ SoC）、GSI（验证解耦）。**稳定的接口 = 独立的升级节奏**。**对稳定性工程师来说，这意味着 bug 修复的"半衰期"变长**——一个 bug 修在 system 中，可以独立 OTA 到所有兼容 vendor 的设备。

3. **12 年的演进主线是"独立升级粒度"**。从 AOSP 7 的"完整 OTA" → AOSP 8 的"system 可独立升级" → AOSP 10 的"partition 大小可调" → AOSP 11 的"模块可独立升级（APEX）" → AOSP 13 的"kernel 可独立升级（GKI 2.0）"。**每一次改革都让升级粒度变细、用户等待变短、bug 修复变快**。

4. **分区和 system_server / OTA / bootloader 的协作是稳定的端到端链路**：
   - `bootloader → kernel → init → system_server`
   - **任一环节出问题都会导致启动失败**，但**根因排查路径是固定的**：
     - 卡 bootloader → 看 GPT / vbmeta
     - kernel panic → 看 boot / init_boot
     - init crash → 看 vendor_boot / fstab
     - system_server 启动慢 → 看 HAL 挂载顺序

5. **三大改革的"协同"是 Android 系统可维护性的根因**。**没有 Treble，GKI 改革后的 kernel 模块也无法独立升级**（因为 vendor HAL 与 system 强耦合）；**没有 GKI，GSI 也无法独立编译**（因为 kernel 与 SoC 强耦合）。**三大改革是层层递进的，每一层改革都为上一层创造了条件**。

| 问题类型 | 典型场景 | 日志关键字 | dumpsys/系统特征 | 排查入口 |

| **super 空间不足** | OTA 升级失败、partition 调整失败 | `lpmetadata: no space left`、`fs_mgr: failed to resize` | `lp metadata dump` 显示 partition 满 | `lpdump` 工具、`build/core/Makefile` 中调整 super 大小 |
| **dm-verity 校验失败** | 启动卡在 recovery 模式 | `dm-verity: FAIL`、`verity: hash mismatch` | recovery 模式 + `verify_partitions` | `adb disable-verity && adb reboot`（userdebug） |
| **A/B slot 损坏** | 升级后变砖、卡 bootloader | `slot: invalid suffix`、`metadata: corrupted` | `fastboot getvar slot-suffix` 异常 | `fastboot --set-active=a` 切换 slot |
| **APEX 挂载失败** | ART/媒体编解码异常 | `apex: failed to mount`、`apexd: verify failed` | `dumpsys apexd` | `adb shell apexd --check` 验证 |
| **GKI KMI 不兼容** | kernel panic、模块加载失败 | `module: unknown symbol`、`request_module failed` | `dmesg \| grep -i module` | 查 `abi_symbollist`、vendor module 重新编译 |
| **boot/init_boot 损坏** | 启动卡在第一阶段、kernel panic | `Failed to mount /system`、`init: kernel panic` | 设备无法进入 Android | `fastboot flash boot boot.img` + `fastboot flash init_boot init_boot.img` |
| **vendor_boot 不匹配** | 启动卡在 init 阶段 | `init: failed to parse /vendor/etc/init` | `dmesg \| grep init` | `fastboot flash vendor_boot vendor_boot.img` |
