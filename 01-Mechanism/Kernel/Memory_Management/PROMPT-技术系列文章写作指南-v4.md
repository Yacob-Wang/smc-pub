# 技术系列文章写作 Prompting 指南 v4

> 本文件是一套经过多系列实战验证的 AI 提示词模板，用于指导 AI 生成**有深度、有广度、有源码依据、有工程基线、可验证**的技术系列文章。
> 适用于任何技术模块（内核子系统、Framework 服务、Native 库、协议层等）的系列化学习输出。
> **目标读者：稳定性架构师**。所有约束以"可治理、可机制化、可追溯"为标准。
>
> **v4 增量**（相对 v3）：
> - 新增 **§7 校准与迭代**：把"AI 写完就完事"升级为"机制化迭代流程"——架构师交付的是机制，不是一篇文章
> - 新增 **§8 跨系列一致性治理**：术语库 / 版本基线 / 引用矩阵 / 案例索引的可治理性，是 10+ 系列规模下的核心痛点
> - 新增 **§9 破例准则**：破例不是"破罐子破摔"，必须有"决策记录 + 影响范围分析"
> - 反例库 10→**12**：新增「数据堆砌」「AI 自嗨」——架构师读者最反感的两个反例
> - 适用场景列表扩展：新增 Binder / Linux Kernel Layer / Process / Interrupt 等已落地系列
> - v3→v4 变更记录在文末

## 一、核心原则

````
你写的不是科普文，不是面试八股，而是给【资深架构师】看的技术深度文章。
必须满足以下硬性要求：

1. 言必有据，码必有源：每个技术论点必须标注对应的源码文件路径，关键流程必须贴核心源码片段。
   每篇文章末尾必须给出"源码路径对账表"，逐条标注【已校对 / 待确认】。
2. 自顶向下：每个章节先讲"是什么、为什么需要它"，建立上下文后再深入源码细节。禁止上来就贴代码。
3. 关联实战：每个知识点必须关联到真实的工程问题（Crash/ANR/OOM/性能退化等），说清楚"这个机制会在什么场景下出问题"。
4. 实战案例收尾：每篇文章末尾必须有 1-2 个完整的线上问题排查案例（现象→分析→根因→修复），案例必须可验证（含 dump/日志片段、版本号、复现步骤）。
5. 内容量级：每篇文章不少于 300 行（约 8000-15000 字），确保广度和深度兼具。
6. 版本基线（多版本矩阵）：
   - 应用层 / Framework：以 Android 14（AOSP android-14.0.0_r1）为主线
   - Linux 内核：以 android14-5.10/android14-5.15/android15-6.1/android15-6.6 为基线
   - 涉及 GKI 厂商分支差异时，标注"厂商 GKI"或具体 vendor tag
   - 涉及历史演进或多版本差异时，必须标注版本范围
7. 工程基线：涉及可调参数（timeout、buffer size、maxevents、优先级等）时，必须给出"工程默认值"与"选用准则"，说清楚"线上我应该用啥、为啥"。
````

## 二、系列文章的设计流程

### 第一步：定义角色与目标

````
我的角色是 Android 稳定性高级工程师。

我要系统学习【填入模块名称，如：Binder / SurfaceFlinger / Linux 调度器 / epoll】。

学习目标不是"了解原理"，而是：
- 能在遇到线上问题时，快速定位到该模块的对应子系统和源码文件
- 能理解问题的根因机制
- 能给出治理方案或监控手段
````

### 第二步：让 AI 先输出大纲（不要直接写文章）

````
请按照以下架构师思维链，为【模块名称】设计一套系列文章的大纲：

架构师看一个系统的逻辑链：
  它是什么？解决什么问题？（定位）
      ↓
  它在系统中处于什么位置？和谁协作？（边界与交互）
      ↓
  它内部是怎么运转的？（核心机制）
      ↓
  它会在什么地方出问题？（风险地图）
      ↓
  出了问题我怎么查？怎么防？（诊断与治理）

大纲要求：
1. 分为 3-5 个篇章（全局观 → 核心机制 → 跨模块交互 → 实战/治理）
2. 每篇文章列出具体章节，每个章节说清楚：
   - 讲什么内容
   - 对应哪个源码文件/目录（含内核版本基线）
   - 关联什么工程问题
3. 用表格形式呈现每篇文章的章节规划
4. 明确标注与其他系列（如已有的 Input / ART / MM_v2 / IO / Socket / Syscalls）的交叉引用点
   - 引用矩阵格式：| 本篇章节 | 引用系列 | 引用文章 | 引用原因 |
5. 如果该模块与其他模块有深度交互（如 WMS 与 Input 的焦点管理），
   应为交互部分单独设置专题篇章，而非分散在多篇中一笔带过
6. 明确标注"本篇定位"：在本系列中属于什么角色（全局观/核心机制/横切专题/风险/治理），
   强依赖哪几篇已发布文章
````

### 第三步：生成系列 README

````
大纲审核通过后，请生成一份系列导读文档（README-【模块名】系列.md），包含：

1. 为什么要写这个系列
   - 该模块在稳定性领域的重要性（用数据说话，如"占线上 ANR 的 X%"）
   - 对稳定性工程师的核心价值

2. 系列设计思路
   - 用架构师思维链的具体展开形式，展示从定位到治理的逻辑链
   - 给出"依赖关系图"：本系列各篇之间的依赖方向
   - 给出"跨系列引用矩阵"：本系列与已有系列哪几篇有双向引用

3. 每篇文章的章节规划表格
   - 格式：| 章节 | 内容 | 核心源码路径 | 内核版本基线 | 稳定性关联 |
   - 必须有"本篇定位"列

4. 与其他系列的交叉引用
   - 列出与已有系列的关联文章和互相引用关系
   - 列出与已有系列内容的"边界声明"（什么不重复讲、什么一笔带过）

5. 阅读建议
   - 时间有限优先阅读哪几篇
   - 系统学习的推荐顺序
   - 每篇文章的设计逻辑（背景 → 架构 → 源码 → 风险 → 案例）

6. 质量基线（可选，仅横切型系列需要）
   - 若本系列涉及可调参数，给出"工程默认值表"：参数 / 典型默认 / 选用准则 / 踩坑提醒
````

### 第四步：逐篇撰写（使用一站式模板）

大纲和 README 审核通过后，使用下方**一站式写作模板**逐篇撰写。每次只需替换 `【】` 中的内容。

## 三、一站式写作模板

以下是一个可直接复制使用的完整提示词模板，整合了角色设定、写作约束、章节结构和禁止事项。

> **公开站说明（必读）**：`本篇定位` / `校准决策日志` / `角色设定` / `上下文` / `写作标准` 是**写稿与校准用**的作者元信息，读者不需要看。  
> GitHub Pages 构建（`00-Meta/scripts/prepare_web_docs.py` → `preamble_transform.py`）会在生成 `docs/` 时**自动剥离**这些文首段落，只保留标题 + 版本/系列 blockquote + 正文（对齐 Activity 文首形态）。  
> **仓库源文件请继续按本模板写齐前言**；不要为了“好看”删掉源文件里的定位/校准日志。公开站若仍露出前言，说明变换器未命中，应修 `preamble_transform.py` 而非删源。

````markdown
# 本篇定位（强制开头段，先写它再写正文）
- 本篇系列角色：【全局观 / 核心机制 / 横切专题 / 风险地图 / 诊断治理】
- 强依赖：必须先读【文章 A】的【章节 X】、【文章 B】的【章节 Y】
- 承接自：上一篇【文章 C】已覆盖【简述】，本篇不重复展开
- 衔接去：下一篇【文章 D】将覆盖【简述】，本篇末尾会预告
- 不重复内容：与本系列已发布/已规划文章中【哪些内容】的边界声明（一句话）

# 角色设定
我是一名【Android 稳定性架构师】，正在系统学习【模块名称】。
本篇是系列的第【N】篇，主题是【具体主题】。

# 上下文
- 上一篇是【上一篇标题】，已覆盖了【简述上一篇核心内容】
- 下一篇是【下一篇标题】，将覆盖【简述下一篇核心内容】
- 本系列的 README 见【README 文件名】

# 写作标准

## 硬性要求
1. 目标读者：资深架构师，不是初学者。不需要解释基础概念（如什么是线程、什么是指针），
   但需要解释该模块特有的术语和机制。
2. 每个章节必须先讲"这个东西是什么、为什么需要它、解决什么问题"，然后再深入源码。
3. 涉及源码时：
   - 必须标注源码文件路径（如 `frameworks/native/services/inputflinger/dispatcher/InputDispatcher.cpp`
     或 `fs/eventpoll.c`），**并标注内核版本基线**
   - 只贴核心逻辑，不要全贴
   - 贴代码前先用自然语言解释这段代码要干什么
   - 贴代码后紧跟"稳定性架构师视角"分析
   - 如果不确定某个文件路径是否准确，用 `// 路径待确认` 标注，并在"源码路径对账表"中标记【待确认】
4. 每个技术点必须关联到实际的工程问题（Crash、ANR、OOM、JE、NE、Watchdog、性能退化、兼容性问题），
   说清楚"它会在什么场景下咬你一口"。
5. 涉及量化描述时，必须给出数量级（"ms 级""秒级""占比 X%"），禁止使用"大约""通常"等模糊用词。
   量化依据须填入"量化数据自检表"。
6. 源码版本基线：
   - 应用层 / Framework：以 Android 14（AOSP android-14.0.0_r1）为主线
   - Linux 内核：以 android14-5.10 / android14-5.15 / android15-6.1 / android15-6.6 为基线
   - 涉及厂商 GKI 分支差异时，标注"厂商 GKI"或具体 vendor tag
   - 涉及历史演进或多版本差异时，必须标注版本范围
7. 工程基线要求：涉及可调参数（timeout、buffer size、maxevents、nice/ionice 等级等）时，
   必须给出"工程默认值"与"选用准则"，说清楚"线上我应该用啥、为啥"。
8. 文章长度不少于 300 行（约 8000-15000 字）。

## 章节结构
请严格按照以下结构组织内容：

```
┌─ 背景与定义（是什么、为什么需要它、解决什么问题）
│     不要假设读者已经知道，先用 1-2 段建立上下文
│
├─ 架构与交互（在系统中的位置、上下游依赖、数据流向）
│     用 ASCII Art 画架构图或层级图，说清楚边界
│     标注本篇重点在全局架构中的位置
│
├─ 核心机制与源码（关键数据结构、核心流程、源码走读）
│     每段源码必须标注 AOSP/内核 源码路径 + 内核版本基线
│     贴代码前先用自然语言解释这段代码要干什么
│     代码后紧跟"稳定性架构师视角"分析
│
├─ 风险地图（这个机制会在哪些场景下出问题）
│     列出具体的异常类型、日志特征、触发条件
│     用表格形式呈现"问题类型 / 日志关键字 / 排查入口"
│
├─ 实战案例（1-2 个线上问题的完整排查过程）
│     格式：现象 → 分析思路 → 根因 → 修复方案
│     案例必须可验证：
│       · 现象：贴关键 logcat / dmesg / systrace 片段
│       · 环境：Android 版本 / 内核版本 / 设备 / 复现步骤
│       · 修复：贴修复 commit 或配置 diff
│     案例基于通用模式构造时标注"典型模式"，
│     基于公开 bug report 时标注来源
│
├─ 总结（3-5 条架构师视角的关键 Takeaway）
│     浓缩全篇为"排查时需要记住的几件事"
│     给出该主题的排查路径速查
│
├─ 附录 A：核心源码路径索引（表格）
│     格式：| 文件名 | 完整路径 | 内核版本基线 | 说明 |
│     集中列出本篇涉及的所有源码文件，方便检索
│
├─ 附录 B：源码路径对账表（强制）
│     格式：| 序号 | 文章中出现的路径 | 已校对/待确认 | 校对来源 |
│     用于在 AOSP Code Search / LXR / Elixir 上逐条核对，杜绝路径幻觉
│
├─ 附录 C：量化数据自检表（强制）
│     格式：| 序号 | 量化描述 | 数量级 | 依据来源（源码常量 / 线上统计 / 文档）|
│
├─ 附录 D：工程基线表（若涉及可调参数）
│     格式：| 参数 | 典型默认 | 选用准则 | 踩坑提醒 |
│
└─ 篇尾衔接
      用 1-2 句说明下一篇将深入什么内容
      格式：下一篇 [文章标题](文件名.md) 将深入...
```

## 图表格式
- 架构图/层级图/流程图：统一使用 ASCII Art（纯文本方框图），
  确保在任何 Markdown 渲染器中正常显示，不依赖外部渲染工具
- 对比信息：使用 Markdown 表格
- 时序说明：使用缩进文本 + 箭头符号

## 图表密度规则
- **平均每 1500 字 1 张图**
- **单篇 4-6 张图（不含源码中的小型 ASCII 示意）**
- 上限 6 张：超过说明信息密度没跟上
- 下限 4 张：少于说明架构/流程讲得不到位
- 表格不计入图数
- **例外**（破例适用，按 §9 破例准则处理）：
  - 横切专题型（机制 + 案例一篇覆盖）可放宽到 3 张图
  - 演进型（重点是版本对比表）可放宽到 3 张图 + 1-2 张对比表

## 跨模块引用规范
- 当涉及本系列其他篇章已覆盖的内容时，用 `[文章标题](文件名.md)` 引用，不重复展开
- 当涉及其他系列已覆盖的内容时，用 `[系列名-文章标题](相对路径)` 引用，只用 1-2 句概述核心结论
- 当涉及其他系列未覆盖的内容时，简要说明核心逻辑（3-5 句），不要深入展开
- 涉及引用时，必须在文章开头"本篇定位"中显式声明

## 章节规划
请按照以下章节规划撰写：

1. 【章节 1 标题】
   - 要讲的内容：...
   - 核心源码路径（含内核版本基线）：...
   - 关联的工程问题：...

2. 【章节 2 标题】
   - 要讲的内容：...
   - 核心源码路径（含内核版本基线）：...
   - 关联的工程问题：...

...（按需添加）

## 禁止事项
- 禁止写成面试八股（罗列概念但不深入）
- 禁止上来就贴源码而不给上下文
- 禁止只讲原理不关联实际问题
- 禁止使用"据说""通常认为""一般来说"等模糊表述，要么给源码依据，要么给数据依据
- 禁止编造不存在的源码文件路径（不确定时用 `// 路径待确认` 标注，并在附录 B 中记录）
- 禁止混用不同 Android 版本 / 内核版本的 API 而不标注版本
- 禁止在篇末缺少"总结"和"附录 A/B/C/D"（按需）
- 禁止在篇末缺少下一篇衔接（系列最后一篇除外）
- 禁止图表密度不达标（少于 4 张或多于 6 张核心图）
- 禁止出现"先挖坑后填"（如"我们将在后续文章中详细讲"）—— 既然挖了就要当场讲清楚
- 禁止"数据堆砌"（见反例 #11）和"AI 自嗨"（见反例 #12）—— 宁可少写，不要凑字数
````

## 四、质量检查清单（26 项 · 4 个维度）

文章写完后用以下清单自查。

### 4.1 内容质量（10 项）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | **有没有回答"是什么"** | 文章开头 2 段内能让读者明白这个东西的定位和职责 |
| 2 | **有没有回答"为什么"** | 解释了这个设计选择的背景和动机，而不仅仅是描述现状 |
| 3 | **有没有架构图/层级图** | 至少有一张 ASCII Art 图说明该模块在系统中的位置和上下游关系 |
| 4 | **源码有没有标路径+版本基线** | 每段源码都标注了对应的文件路径，且标注了 AOSP/内核 版本基线 |
| 5 | **源码前有没有上下文** | 每段源码前都有自然语言解释"这段代码要干什么" |
| 6 | **有没有关联实际问题** | 每个知识点都能回答"这和我排查线上问题有什么关系" |
| 7 | **有没有实战案例** | 文章末尾有 1-2 个完整的排查案例（现象→分析→根因→修复） |
| 8 | **案例是否可验证** | 案例含 logcat/dmesg/systrace 片段、Android+内核版本、复现步骤、修复 diff |
| 9 | **深度够不够** | 不是停留在 API 使用层面，而是深入到数据结构和算法层面 |
| 10 | **广度够不够** | 覆盖了该主题的主要方面，没有重大遗漏 |

### 4.2 结构完整性（6 项）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 11 | **有没有本篇定位声明** | 文章开头有"本篇定位"段，明确系列角色、强依赖、衔接、不重复内容 |
| 12 | **有没有总结** | 篇末有 3-5 条架构师视角的关键 Takeaway |

## §7 校准与迭代机制

> **核心立场**：架构师交付的是**机制**（可复用、可治理），不是"一篇文章"。
> 一篇文章质量从 70 分到 90 分靠的不是 AI 更聪明，是**校准机制设计得更合理**。

### 7.1 AI 输出的三个天然倾向

必须在校准前识别，否则会被带偏：

1. **平滑化**：倾向于写"看起来都对"的内容，但缺乏锐度
2. **冗余化**：同一个意思换 3 种说法重复 3 次
3. **自信化**：即使不确定也写得很笃定（"一定是……"）

### 7.2 校准的三轮节奏

| 轮次 | 目标 | 触发 | 动作 | 不动 |
|------|------|------|------|------|
| **第 1 轮 · 结构** | 让骨架立住 | 26 项清单扫描后 | 缺本篇定位 / 总结 / 附录 → 补；章节顺序乱 → 调整；跨篇重复 → 改引用 | 细节、风格 |
| **第 2 轮 · 硬伤** | 经得起查 | 路径对账 / 量化自检 / API 版本核对后 | 路径幻觉 → 重查或删；量化没依据 → 补来源；API 签名不对 → 重读源码 | 风格、措辞 |
| **第 3 轮 · 锐度** | 读完有收获 | 通读全文，凭"读起来是否废话"判断 | 数据堆砌 → 加"所以呢"；AI 自嗨 → 删"非常精妙"；模糊量化 → 改具体数字；冗余 → 合并 | 骨架、硬伤 |

**铁律**：每轮校准**只动一类问题**，不交叉。否则丢失已校准的进度。

### 7.3 校准决策日志（v4 强制要求）

每轮校准后，必须在文章开头（本篇定位段下方）记录**决策日志**：

```markdown
# 校准决策日志
| 轮次 | 类别 | 决策 | 理由 | 影响范围 |
|------|------|------|------|----------|
| 1    | 结构 | 附录 D 改为按需 | §9 破例：本篇无参数表 | 仅本篇 |
| 2    | 硬伤 | 删除路径 fs/foo/bar.c | 校对发现该文件不存在 | 全文 3 处 |
| 3    | 锐度 | 删"非常精妙"段落 | 反例 #12 | 全文 2 处 |
```

**为什么需要决策日志**：
- 6 个月后回头看，能复盘"当时为什么这么改"
- 跨系列共享经验（"Input 系列第 2 轮校准时这样处理，Binder 系列可以参考"）
- 当 AI 之后又"建议"把已校准的段落改回去时，能用决策日志挡住

### 7.4 校准的边际效用与停止准则

| 轮次 | 典型分 | 边际收益 | 何时停 |
|------|--------|---------|--------|
| 第 1 稿 | 65-75 | —— | —— |
| 校准 1 轮 | 75-82 | +7-10 | 26 项清单基本通过 |
| 校准 2 轮 | 82-88 | +5-7 | 附录 B/C 全量核对完毕 |
| 校准 3 轮 | 88-92 | +3-4 | 通读没有废话 |
| 校准 4 轮 | 边际递减 | <+2 | 改的比改坏的多 |

**停止信号**：当一轮校准里，AI 的修改有 ≥30% 引发新问题，**立即停止校准**——这是 AI 已经"理解饱和"的信号，再压会写崩。改为人工润色。

### 7.5 校准时的提示词模板（定向修正，避免整篇重写）

````markdown
# 校准第 N 轮（结构 / 硬伤 / 锐度）

只修改以下指出的问题，**其他段落完全保留**：

## 必须修改的（按优先级）
1. 【第 X 章 第 Y 节】"那段话" → 改为"……"（理由：反例 #5 模糊量化）
2. 【附录 B 第 Z 行】路径 fs/eventpoll.c → 实际不存在，请重新确认或标注"路径待确认"
3. ……

## 可以改进的（不强制）
- 【第 3 章】"非常精妙"这种话可以删
- ……

## 绝对不要动
- 第 1-2 章（已经校准过）
- 附录 A 索引（已经核对）
- ……

修改后请在末尾输出"本轮修改清单"，列清楚改了哪几处，并更新文首"校准决策日志"。
````

## §8 跨系列一致性治理

> **为什么需要这一节**：当系列数 ≥ 3（如 ART + Input + MM_v2），"术语漂移""基线漂移""引用断裂"会指数级恶化。
> 10+ 系列规模下，**一致性治理是架构师的核心责任**，不是作者个人习惯。

### 8.1 治理范围与责任主体

| 治理对象 | 漂移风险 | 治理手段 | 责任人 |
|---------|---------|---------|--------|
| **术语** | 同一概念在不同系列用不同名字 | 全局术语表 | 架构师 |
| **版本基线** | 新 AOSP 版本出来，旧文章基线过时 | 基线升级流程 | 各系列 owner |
| **跨篇引用** | 文章删除/移动，引用断裂 | 引用矩阵治理 | 各系列 owner |
| **架构图风格** | 不同系列画的架构图风格不一致 | 模板规范 | 架构师 |
| **案例编号** | 案例无法跨系列检索 | 案例索引 | 架构师 |

### 8.2 全局术语表（强制维护）

`Reference/术语表.md` 维护一份全局术语，**所有系列必须遵守**：

```markdown
# 全局术语表
| 中文 | 英文 | 别名（禁止使用） | 首次定义于 | 备注 |
|------|------|----------------|-----------|------|
| 主线程 | main thread | UI 线程、UI thread | ART 01 | 明确不用"UI 线程"避免和 UI 渲染混淆 |
| 焦点窗口 | focused window | 当前焦点、front window | Input 05 | |
| 分代回收 | generational GC | 分代收集 | ART 05 | 统一用"回收" |
| …… | | | | |
```

**使用规则**：
- 任何系列在新引入术语前，先查术语表
- 已有术语，**必须使用**，禁止"别名"（除非术语表里明确登记）
- 跨系列文章里如果发现术语漂移，立刻开 issue 修复

### 8.3 版本基线升级流程

当 AOSP 新版本（android-15.0.0_r1）或新内核（android15-6.6）发布时：

```
步骤 1：基线评估
  - 新版本相对旧版本的关键 API 变化（cs.android.com diff）
  - 哪些系列受影响（按引用矩阵过滤）
  - 评估工作量（篇数 × 平均改 30% 章节）

步骤 2：批次升级
  - 优先升级引用次数最多的文章（影响面最大）
  - 不在主线的系列（如 Window 11 篇）可以延后

步骤 3：跨系列一致性回归
  - 升级后跑一遍 26 项清单
  - 重点检查：版本基线声明（§4.3-20/21）、跨篇引用是否还指向存在的文章

步骤 4：基线切换公告
  - 在 README 顶部加 "本系列基线：android-15.0.0_r1（自 YYYY-MM-DD 起）"
  - 在术语表/版本基线登记表登记
```

**反模式**：一次性把所有系列基线全切——会导致某段时期"50% 文章是 A14，50% 是 A15"，读者困惑。

### 8.4 引用矩阵治理

每篇 README 维护一份**本系列 → 跨系列**的引用矩阵：

```markdown
# 跨系列引用矩阵（本系列 → 其他系列）
| 本系列文章 | 引用其他系列 | 引用章节 | 引用原因 | 链接有效性最后核查 |
|-----------|-------------|---------|---------|-------------------|
| ART 05-内存与GC | MM_v2 03-ART堆内存与GC全景 | §3 ART 堆结构 | 从内核视角看 ART 堆的 VMA 映射 | 2026-07-10 |
| ART 05-内存与GC | Process 05-ART进程内世界 | §2 zygote 进程 | zygote 进程内的 ART 初始化 | 2026-07-10 |
| …… | | | | |
```

**治理动作**：
- 每月跑一次"链接有效性核查"（点击所有引用链接，确认未 404）
- 目标链接的文章删除/移动前，必须先在引用矩阵里找到所有引用方，逐个更新
- 用脚本辅助（`grep -r "ART 05" --include="*.md"` 找到所有引用方）

### 8.5 架构图风格规范

为避免视觉漂移，**全仓库统一遵循以下规范**：

- 架构图/层级图：ASCII Art（纯文本方框图），不依赖外部渲染
- 方向：左→右 或 上→下，**全仓库只用这两种**
- 边框：用 `┌─┐│└─┘` 而不是 `+-|+`（更清晰）
- 颜色：Markdown 不支持，**统一用文字标注层级**（如 [FW]、[Native]、[Kernel]）

### 8.6 案例索引

跨系列共享的实战案例建索引：

```markdown
# 实战案例索引（按主题分类）
| 案例 ID | 主题 | 系列 | 文章 | 环境 | 案例类型 |
|---------|------|------|------|------|---------|
| CASE-INPUT-01 | Input ANR - 主线程 Looper 阻塞 | Input | 06-Input ANR | A14 / K5.10 / Pixel 6 | 典型模式 |
| CASE-MEM-01 | ART 老年代泄漏 → OOM | MM_v2 | 12-内存稳定性风险全景 | A14 / K5.10 | 真实案例 |
| …… | | | | | |
```

**价值**：
- 同一个症状（如 OOM）能在多个系列里找到不同视角的案例
- 排查时按症状反向检索案例
- 避免同一案例被多个系列各自构造

## §9 破例准则

> **核心立场**：模板不是"必须逐字遵循的教条"，但**破例必须有决策记录**，不能"我想怎么写就怎么写"。

### 9.1 破例的三类合法场景

| 场景 | 允许的破例 | 破例决策记录内容 |
|------|-----------|----------------|
| **横切专题型**（如 epoll 横切、信号横切） | 图表数 → 3 张；案例 1 个即可；章节合并 | 写在 README 的"系列设计思路"段 |
| **演进型**（如 ART vs JVM 对比、Kernel 版本演进） | 图表数 3 张 + 1-2 张对比表；源码片段可精简 | 写在 README 的"系列设计思路"段 |
| **总览篇**（系列第 1 篇） | 风险地图可只列 3 类常见问题；实战案例可省略 | 写在文章开头"本篇定位"段 |
| **诊断工具型**（如 trace 工具、内存诊断） | 章节结构重排为"工具使用 → 工具源码 → 实战" | 写在 README 的"系列设计思路"段 |
| **小众模块**（如 kworker、PSI） | 实战案例可标注"行业案例"代替"线上案例" | 写在文章开头"本篇定位"段 |

### 9.2 破例决策记录格式

每条破例必须按以下格式记录，**位置在"本篇定位"段**或 README 的"系列设计思路"段：

```markdown
## 破例决策记录
| 破例项 | 破例内容 | 破例理由 | 影响范围 | 是否传染 |
|--------|---------|---------|---------|---------|
| 图表密度 | 3 张图（规则要求 4-6 张） | 横切专题型，机制+案例一篇覆盖 | 仅本篇 | 否 |
| 实战案例 | 1 个行业案例（规则要求 1-2 个真实案例） | 该模块线上案例数据不足 | 仅本篇 | 否 |
```

**字段说明**：
- **影响范围**：破例只影响本篇，还是整个系列都会这么写？
- **是否传染**：是否会成为后续文章的"先例"，导致模板整体松弛？
  - 若 `是` → 必须升级到 README 的"系列设计思路"段登记
  - 若 `否` → 写在本篇即可

### 9.3 不允许的破例

无论什么理由，以下**永远不允许破例**：

| 项 | 不允许原因 |
|----|----------|
| **本篇定位段** | 没有定位就不知道这文章在整个系列里的角色，跨篇引用一定乱 |
| **附录 B 路径对账** | AI 幻觉的最大风险点，省了就是赌运气 |
| **数据后必须有洞察**（"所以呢"原则） | 反例 #11 的核心防御 |
| **多版本基线矩阵** | 没有基线声明，读者无法判断"我这版本是不是这样" |
| **跨篇引用用 Markdown 链接** | 不引就是抄，破坏可治理性 |

### 9.4 破例传染性治理

当某篇破例后，发现"这破例其实适用于一类文章"时，**必须升级为模板规则**：

```
本篇破例：横切 epoll 文章用 3 张图
  ↓
3 篇横切文章都用了 3 张图
  ↓
发现 §4 质量清单 #26 写"4-6 张"是个一刀切
  ↓
升级为：横切专题型 → 3 张图；普通型 → 4-6 张图
  ↓
更新 §4 质量清单，更新 §9.1 破例表
```

**反模式**：破例→传染→破例→传染——最后模板完全失效。

## 附录 B：本指南的自迭代规则

这份指南**自己也用自己**。每次有新的系列落笔、踩到新的反例、发现新的技巧，就更新本文件：

1. **新反例** → 加入 §5 反例库，附触发场景和防范方法
2. **新硬要求** → 加入 §1 核心原则 + §3 硬性要求 + §4 质量清单对应项
3. **新破例** → 加入 §9.1 破例表，写明触发条件和传染性评估
4. **新治理手段** → 加入 §8 跨系列一致性治理对应子节
5. **校准经验** → 更新 §7.2 节奏表和 §7.4 边际效用表
6. **版本号** → 每次有结构性变更就 +1（v3 → v4 → v5），并在附录 A 追加变更记录

**铁律**：本指南**不删除旧版反例**——曾经踩过的坑要保留，作为新写手的警示。
