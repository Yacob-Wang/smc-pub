## 0. 写在前面：为什么需要"总览"篇

在开始深入 Linux 内核 `mm/` 子系统之前，必须先建立一个**端到端的心智模型**——把 Android 内存系统的每一层都定位清楚：它在哪里、它管什么、它和邻居的边界在哪里。原因是：内存问题**几乎没有单点根因**。

一个典型的"App 报 OOM"故障，根因可能藏在以下任意一层：

| 故障表象 | 可能的根因层 | 排查方向 |
|---------|------------|---------|
| `OutOfMemoryError: Java heap space` | ART 堆分代 / GC 策略 | dumpsys meminfo + ART GC trace |
| `OutOfMemoryError: Direct buffer memory` | Native 堆（ByteBuffer.allocateDirect） | procrank + malloc_debug |
| 进程被杀（logcat 中 `Process xxx has died`） | LMKD 误杀 / OOM Killer 杀 | LMKD 日志 + dmesg |
| 应用冷启动慢 30%+ | Zygote fork 慢 → 物理内存碎裂 | /proc/buddyinfo + /proc/pagetypeinfo |
| 切换应用卡 2-3 秒 | zRAM swap 风暴 / Direct Reclaim 阻塞 | /proc/vmstat + PSI 指标 |
| 视频播放内存持续增长 | ION/DMA-BUF 泄漏、Gralloc 泄漏 | /d/ion/heaps + graphics buffer dump |

**对架构师来说，"内存去哪了"比"内存还剩多少"重要一万倍**。本篇就是要把这张地图画清楚。

## 1. Android 内存系统是什么：五层架构定义

### 1.1 一句话定义

**Android 内存系统是一个由 5 层（App 进程、ART 运行时、Framework 服务、Linux 内核 mm/、硬件）协同构成的、跨进程-跨内核-跨硬件的分布式资源管理系统，其核心目标是在 2GB-16GB 的物理 RAM 资源约束下，让数十个进程、上百个组件同时高效运行，并在内存压力出现时做出符合用户体验的取舍。**

### 1.2 为什么"五层"

任何一个合格的资源管理系统都需要解决三个基本问题：**抽象（Abstraction）、隔离（Isolation）、调度（Scheduling）**。Android 内存系统对这三个问题的回答分布在五层：

```
┌────────────────────────────────────────────────────────────────────┐
│  Layer 1: App 进程层（Java 堆 + Native 堆 + 资源句柄）              │
│  职责：业务逻辑；分配诉求的发源地                                    │
│  抽象：Java 对象 / Native 句柄 / Bitmap / GraphicBuffer            │
├────────────────────────────────────────────────────────────────────┤
│  Layer 2: ART 运行时层（GC、堆管理、Reference、JNI 引用表）         │
│  职责：Java 堆生命周期、内存可见性、GC 调度                          │
│  抽象：Java 堆、LOS、Card Table、Concurrent GC                     │
├────────────────────────────────────────────────────────────────────┤
│  Layer 3: Framework 服务层（AMS 进程治理、LMKD 杀手、dumpsys/procrank）│
│  职责：进程级内存治理（adj 计算、kill 决策、监控导出）                │
│  抽象：oom_score_adj、ProcessRecord、MemoryInfo                    │
├────────────────────────────────────────────────────────────────────┤
│  Layer 4: Linux 内核 mm/（VMA、页分配器、SLAB、回收、OOM、memcg、PSI）│
│  职责：物理内存的分配、回收、隔离、统计                                │
│  抽象：VMA、struct page、page frame、cgroup、psi metric            │
├────────────────────────────────────────────────────────────────────┤
│  Layer 5: 硬件（DRAM、MMU、IOMMU、zRAM、ION/DMA-BUF）              │
│  职责：物理存储；页错误的中转                                         │
│  抽象：物理页帧、DMA-BUF fd、zRAM block                             │
└────────────────────────────────────────────────────────────────────┘
```

### 1.3 与"单片机内存管理"的本质区别

如果你是从嵌入式 / RTOS 出身，第一次接触 Android 内存会被"这么多层"劝退。可以用一组对比来快速建立直觉：

| 维度 | 裸机/RTOS（无 OS 或 μC/OS） | Android（Linux + ART + Framework） |
|------|--------------------------|----------------------------------|
| 物理 RAM 视角 | 单一全局数组 `uint8_t ram[8KB]` | 几十个进程共享 2-16 GB 物理 RAM |
| 谁负责分配 | 程序员手写 | malloc → scudo/jemalloc → brk/mmap → 页分配器 → 物理页帧 |
| 隔离机制 | 无（单地址空间） | MMU 硬件 + 进程页表 + ASLR |
| 容量约束 | 编译期确定 | 运行时动态：swap、zRAM、cache 回收 |
| 故障域 | 单进程崩溃 = 系统死 | 单进程崩溃可独立重启 |
| 治理策略 | 无 | LMKD 杀进程、memcg 限流、PSI 压力传播 |
| 监控指标 | 静态内存池水位 | PSS/RSS/USS/PSI/swap/compaction 等 50+ 指标 |

**最关键的差异**是：在 Android 上，"内存"不是一个静态资源池，而是一个**多维度的、跨进程动态博弈的对象**。同一个物理页帧，可能在 t1 时刻属于 App A 的 Java 堆，在 t2 时刻被回收为 page cache，在 t3 时刻被 swap 到 zRAM，在 t4 时刻又被 reclaim 回某个新进程的匿名页。这种"内存是流动的"特性，决定了 Android 内存治理**必须分层**。

### 1.4 一句话总结

> Android 内存系统不是"一个分配器"，而是"一套由进程、运行时、Framework、内核、硬件共同构成的分层治理体系"。**每一层都是其他层的不可信依赖，每一层都有自己的故障模式**。

> **稳定性架构师视角：** 排查内存问题的"第一步错误"是把内存当成一个全局资源池来看。例如，看到 `dumpsys meminfo` 报 PSS 8GB 就判断"内存不够"——但 Android 12+ 设备的物理 RAM 也就 8GB，PSS 大于物理 RAM 并不意味着 OOM，因为 PSS 包含共享部分（去重后才是实际占用）。**理解 PSS/RSS/VSS/USS 的区别，是入门内存架构的第一道关卡**（见第 5 章）。

## 2. 为什么需要多层架构：单一层管不住的 4 个核心问题

"为什么不能像单片机一样直接管物理 RAM？"——这个问题的答案，藏在以下 4 个核心矛盾里。**每一个矛盾，都只能由某一层（或层与层之间的协作）来解决**。

### 2.1 矛盾一：地址空间隔离（Address Space Isolation）

**问题场景：** 假设 App A 写入地址 `0x7f1234560000`，App B 也在同一地址写入。会发生什么？

- 没有隔离：App B 的数据被 App A 覆盖 → 数据错乱；App A 的指针被 App B 误用 → 崩溃
- 后果：任何一个 App 野指针就能让整个系统崩溃

**解决层级：** Layer 4 (Linux 内核 mm/) + Layer 5 (硬件 MMU)

**机制要点：**

1. **进程虚拟地址空间**：每个进程都有独立的 0x0 ~ 0x7fffffffffff（用户态 128TB，arm64）的虚拟地址空间，由 `mm_struct` 描述。

2. **页表隔离**：每个进程的页表（`pgd` → `p4d` → `pud` → `pmd` → `pte`）由 MMU 硬件查询，硬件保证 A 进程的虚拟地址映射到 B 进程看不到的物理页帧。

3. **硬件 MMU 兜底**：即使内核有 bug 导致错误的页表写入，MMU 会在转换时抛"页错误"或"权限错误"，触发内核的 page fault handler，最坏情况下 `SIGSEGV` 终止违规进程。

**关键源码入口：**

```c
// mm/mmap.c （AOSP android14-5.10 GKI）
// 进程地址空间的核心抽象
struct mm_struct {
    struct vm_area_struct *mmap;        // VMA 链表
    struct rb_root mm_rb;               // VMA 红黑树
    pgd_t *pgd;                         // 页全局目录
    atomic_t mm_users;                  // 引用计数（线程共享）
    atomic_t mm_count;                  // 主引用计数
    int map_count;                      // VMA 数量
    spinlock_t page_table_lock;         // 页表锁
    // ... 60+ 字段
};
```

> **稳定性架构师视角：** 隔离机制的"代价"是**两次地址转换的开销**（虚拟→物理）。在低内存设备上，TLB miss 率高时，page walk 可能消耗 100ns-1μs。**对内存敏感型代码（如游戏、相机），大页（HugePage）能减少 TLB miss**。Android 11+ 在部分场景（vulkan、大尺寸 bitmap）已支持 2MB 大页，但全局启用仍有稳定性风险，故默认关闭。

### 2.2 矛盾二：资源配额（Resource Quota）

**问题场景：** App A 在循环里 `new byte[1MB]`，1000 次后占用 1GB 物理内存。如果放任不管，会发生什么？

- App A 单进程占用过多 → 系统进入低内存 → LMKD 杀其他 App → 整个设备卡顿
- 即使没杀进程，App A 的"内存扩张"会挤占系统其他组件的可用内存

**解决层级：** Layer 2 (ART 堆配额) + Layer 3 (Framework oom_score_adj) + Layer 4 (memcg 限流)

**机制要点：**

1. **ART 堆配额**（Layer 2）：每个进程的 Java 堆有上限（`dalvik.vm.heapmaxfree` 等），默认 256MB-512MB，达到上限后 `new` 抛 `OutOfMemoryError`。

2. **oom_score_adj 调度**（Layer 3）：Framework 维护每个进程的 `oom_score_adj`（-1000 ~ +1000），值越大越容易被杀。`dumpsys meminfo` 中的 `oom_score_adj` 字段直接对应此值。

3. **memcg 限流**（Layer 4）：Linux 内核的 cgroup v2 机制允许为进程组设置 `memory.max`（硬上限）和 `memory.high`（软上限），达到上限后进程分配会触发回收或 OOM。

**关键源码入口：**

```c
// kernel/sched/psi.c （AOSP android14-5.10，真实定义）
// PSI（Pressure Stall Information）是 Layer 3 感知 Layer 4 压力的关键
// NR_PSI_STATES = 6 (PSI_SOME / PSI_FULL × CPU/MEM/IO 共 6 个 state)
// NR_PSI_AGGREGATORS = 2 (PSI_AVGS = 0, PSI_POLL = 1)
struct psi_group {
    /* 保护 avg 数组的锁（普通 mutex，不是 seqlock） */
    struct mutex avgs_lock;

    /* Per-cpu state */
    struct psi_group_cpu __percpu *pcpu;

    /* 累计 stall 时间和总累计（先 state 后 aggregator） */
    u64 avg_total[NR_PSI_STATES - 1];
    u64 avg_last_update;
    u64 avg_next_update;

    /* 工作队列：定期把 per-cpu 状态聚合到 group */
    struct delayed_work avgs_work;

    /* 累计 stall 时间（维度：先 state 后 aggregator） */
    u64 total[NR_PSI_STATES - 1][NR_PSI_AGGREGATORS];

    /* avg 数组：先 state 后 3 个时间窗口（avg10/avg60/avg300） */
    unsigned long avg[NR_PSI_STATES - 1][3];

    /* Polling 模式（无触发器时主动 poll） */
    struct task_struct __rcu *poll_task;
    struct timer_list poll_timer;          /* 普通 timer_list，不是 hrtimer */
    wait_queue_head_t poll_wait;
    atomic_t poll_wakeup;

    /* 触发器（epoll 消费者） */
    struct mutex trigger_lock;
    struct list_head triggers;
    u32 nr_triggers[NR_PSI_STATES - 1];
    u32 poll_states;                       /* 不是 trigger_states */
    u64 poll_min_period;
    u64 polling_total[NR_PSI_STATES - 1];
    u64 polling_next_update;
    u64 polling_until;
};
```

**为什么不能只用单层解决？**

| 方案 | 不足 |
|------|------|
| 只用 ART 配额 | Native 堆不归 ART 管；Bitmap 分配走 native 路径；JIT 代码占内存 |
| 只用 memcg | memcg 不能区分"前台重要"和"后台无关"——后台服务 memcg 限额到后不能被优先杀 |
| 只用 oom_score_adj | oom_score_adj 是杀进程的优先级，不是分配的限制——不能预防，只能事后 |

> **稳定性架构师视角：** 多层配额**互为冗余**。ART 堆溢出 → 抛 OOM 异常 → App 崩溃（最干净）。memcg 触发 → 进程回收 → 仍在跑但分配变慢。oom_score_adj 被杀 → 整个进程消失（最暴力）。**对一个 App 来说，最希望"死在 Layer 2"，最不希望"死在 Layer 3"**。这也是为什么内存优化的核心是"在 ART 堆内收敛内存"。

### 2.3 矛盾三：调度公平（Scheduling Fairness）

**问题场景：** 系统同时运行 50 个 App + 100 个系统服务，谁的分配请求优先满足？

- 如果按"先来先服务"：后启动的小 App 可能饿死；前台 App 可能因后台 GC 抢占而卡顿
- 如果按"按重要性"：谁定义"重要性"？Foreground App 一定比音乐 App 重要吗？

**解决层级：** Layer 1 (App 自管理) + Layer 2 (ART GC 调度) + Layer 3 (AMS 进程优先级)

**机制要点：**

1. **App 层自律**：开发者通过 `Application.onTrimMemory()` 响应系统压力，主动释放缓存（图片、内存缓存）。

2. **ART GC 调度**（Layer 2）：Concurrent GC 优先；前台 App 触发 GC 时，stop-the-world 时间被严格限制（5-15ms 级别，Android 14 的 CC collector 目标 2-5ms）。

3. **AMS 优先级调整**（Layer 3）：App 切后台 → `oom_score_adj` 升高 → 优先被回收。

**关键源码入口：**

```java
// art/runtime/gc/heap.cc （AOSP android-14.0.0_r1）
// ART 堆的 GC 触发条件
void Heap::Trim() {
    // 1. 软引用、弱引用、PhantomReference 清理
    // 2. 根据 thread priority 决定是否阻塞
    // 3. Concurrent GC 优先；前台触发走 Background GC
}

// art/runtime/gc/collector/concurrent_copying.cc
// Concurrent Copying (CC) collector - Android 10+ 默认
// 关键约束：pause time < 5ms（前台）
// 关键约束：pause time < 100ms（后台）
```

> **稳定性架构师视角：** "调度公平"在 Android 上有两个常见误判：
> 1. "前台 App 一定不会被 LMKD 杀"——**错**。Android 11+ 的 LMKD 2.0 在极端压力下也会杀前台 App（典型 case：相机预览占 1.5GB 内存 + 后台有大块分配）。
> 2. "后台 App 一定会被先杀"——**也不全对**。后台但高优先级（音乐播放、SyncAdapter）的 App 仍可能比低优先级的前台 App 存活更久（典型的 adj 计算）。
>
> 排查"为什么这个进程被杀了"的第一步，是看 `dumpsys meminfo --oom --local` 中的 `oom_score_adj` 历史变化。

### 2.4 矛盾四：回收策略（Reclaim Policy）

**问题场景：** 系统物理内存只有 8GB，但实际使用的"虚拟内存"（含 file cache、swap）可能达 12GB+。当分配新内存时，内核从哪里回收？

- 回收 file cache？→ 影响下次文件读性能
- 回收匿名页（要 swap）？→ 写 zRAM 慢，影响性能
- 杀进程？→ 影响用户体验
- 三者如何权衡？

**解决层级：** Layer 4 (Linux 内核 vmscan/kswapd/Direct Reclaim) + Layer 3 (LMKD 决策)

**机制要点：**

1. **kswapd 异步回收**（Layer 4）：当 zone watermark 低于 `low` 时唤醒，异步回收 inactive LRU 链表中的页。**不阻塞分配者**。

2. **Direct Reclaim 同步回收**（Layer 4）：当 kswapd 来不及，分配者进入 `__alloc_pages_slowpath()`，亲自回收。**会阻塞分配者**，是性能抖动的主因。

3. **回收优先级**：file cache > 匿名 swap > 杀进程。`/proc/sys/vm/swappiness` 控制两者比例（0=只回收 file，100=倾向 swap）。

4. **LMKD 决策**（Layer 3）：当 PSI 指标超过阈值，LMKD 按 `min_score_adj` 选择目标进程 kill。

**关键源码入口：**

```c
// mm/vmscan.c （AOSP android14-5.10）
// kswapd 主循环
static int balance_pgdat(pg_data_t *pgdat, int order, int highest_zoneidx) {
    // 1. 尝试从 inactive LRU 回收
    // 2. 如不足，回收 active LRU（refault 检查）
    // 3. 如不足，回收 slab（KASAN 关闭时）
    // 4. 如不足，触发 shrink_slab（LMK shrinker）
    // 5. 如仍不足，触发 OOM Killer
}

// mm/page_alloc.c
// 慢路径
static inline struct page *__alloc_pages_slowpath(gfp_t gfp_mask, ...)
{
    // 1. 尝试唤醒 kswapd
    // 2. 尝试 Direct Reclaim
    // 3. 尝试 compact（碎片整理）
    // 4. 尝试 CMA
    // 5. 触发 OOM Killer（numa-aware）
}
```

> **稳定性架构师视角：** 回收策略的"性能代价"分四档：
> 1. **kswapd 异步回收**：< 5% CPU，影响可忽略
> 2. **Direct Reclaim 同步回收**：被阻塞进程 5-50ms 卡顿（worst case 100ms+）
> 3. **swap 写 zRAM**：阻塞时间翻倍（zRAM 是 CPU-bound 压缩）
> 4. **LMKD 杀进程**：进程消失，被杀 App 重新启动耗时 1-3 秒
>
> **Android 14 的关键优化**是 PSI（Pressure Stall Information）——让 LMKD 提前知道"系统即将进入 Direct Reclaim"，从而在卡顿发生前就杀进程。但 PSI 的阈值配置错误反而会让 LMKD 误杀。

### 2.5 四个矛盾的层次对应关系

| 矛盾 | 主解决层 | 协同层 | 关键机制 |
|------|---------|--------|---------|
| 地址空间隔离 | 内核 mm/ | 硬件 MMU | 进程页表 |
| 资源配额 | ART + memcg | Framework oom_adj | 堆上限 / cgroup / adj |
| 调度公平 | ART GC + AMS | App 自律 | Concurrent GC / trim |
| 回收策略 | 内核 vmscan | LMKD | kswapd / Direct Reclaim / PSI |

**结论：没有单层能解决所有问题。Android 内存架构是分层的，是被"四个矛盾"逼出来的。**

### 3.2 关键数据流（"一个分配请求的生命周期"）

```
[App] new byte[1024]
       │
       │ ① Java 层的 new 由 ART 拦截
       ▼
[ART] Heap::TryToAllocate / ConcurrentCopying::Alloc
       │
       │ ② 命中 TLAB（Thread Local Allocation Buffer）→ 快速路径
       │ ③ 未命中 → CAS 申请 RegionSpace → 触发 GC
       ▼
[ART] mmap(anonymous)  ←── 第一次区域耗尽
       │
       │ ④ libc 包装的 syscall
       ▼
[内核] do_mmap() → mm/mmap.c
       │
       │ ⑤ 查找空闲 VMA 区间 → 扩展 VMA 链表/红黑树
       │ ⑥ 仅建立虚拟映射，不分配物理页（lazy allocation）
       ▼
[App 写 byte[0]] ←── 首次写入
       │
       │ ⑦ 触发 page fault（缺页异常）
       ▼
[内核] handle_mm_fault() → alloc_pages() → fast path
       │
       │ ⑧ pcp 缓存有 → 直接拿
       │ ⑨ pcp 没有 → __alloc_pages_slowpath()
       ▼
[内核] __alloc_pages_slowpath()
       │
       │ ⑩ 唤醒 kswapd（如水位低）
       │ ⑪ 尝试 Direct Reclaim
       │ ⑫ 尝试 compaction
       │ ⑬ 仍失败 → 触发 OOM Killer
       ▼
[硬件] MMU 更新 PTE → DRAM 分配页帧 → CPU 继续执行
```

**这张图揭示了三个关键事实：**

1. **分配不是"一次原子操作"**，而是 13 步以上的协作；任何一步失败都会引发上一层的应对（GC、回退、OOM）
2. **虚拟→物理是 lazy 的**，只有首次写入才真正分配物理页
3. **回收和分配是耦合的**（步骤 ⑩⑪），这就是为什么"内存抖动"会引发分配延迟

### 3.3 各层职责与边界（一表速记）

| 层 | 关键对象 | 管什么 | 不管什么 |
|---|---------|-------|---------|
| **App 进程** | Java 堆对象、Native 句柄、Bitmap | 业务逻辑、内存使用习惯 | 物理地址、回收 |
| **ART 运行时** | Heap、RegionSpace、Reference | Java 对象生命周期、可见性 | Native 堆分配、mmap 策略 |
| **Framework** | ProcessRecord、oom_score_adj、ProcessList | 进程分类、kill 决策、监控 | 物理分配、回收策略 |
| **LMKD** | proc adj slot table | 杀谁不杀谁、阈值管理 | 分配、回收、调度 |
| **内核 mm/** | VMA、struct page、memcg | 物理分配、回收、隔离 | Java 语义、进程业务 |
| **硬件** | DRAM、MMU、IOMMU | 物理存储、地址转换 | 一切软件逻辑 |

> **稳定性架构师视角：** **"边界"决定了排查路径**。当你看到"OOM"时：
> - 是不是 Java 堆？→ 看 `dumpsys meminfo` 的 `Java Heap` 行；如果是，看 ART GC trace
> - 是不是 Native 堆？→ 看 `dumpsys meminfo` 的 `Native Heap` 和 `Graphics`；如果是，看 procrank 和 malloc_debug
> - 是不是系统级？→ 看 `/proc/meminfo` 的 `MemAvailable` 和 PSI 指标
> - 是不是被杀？→ 看 logcat 的 `Process xxx has died` 和 LMKD 日志
>
> **任何一个"OOM"问题，跳过边界判断就开始调参，都是在赌博。**

## 4. 一个 byte 的旅程：从 `new byte[1024]` 到 256 个 4KB 页帧落定

> 这里的"1024"是逻辑表述（实际 `new byte[1024]` 在 ART 中会做对齐到 16 字节），但**我们追踪的是 1KB 逻辑 → 4KB 物理页的端到端过程**。

### 4.1 旅程时序图

```

### 4.2 阶段一：Java 层的 `new byte[1024]`

**Java 字节数组的分配由 ART 拦截**。在 Android 5.0 之前，Dalvik 用 `dvmAllocObject()`；在 Android 14，路径是 `art::Heap::AllocObjectWithAllocator<>`。

```cpp
// art/runtime/gc/heap-inl.h （AOSP android-14.0.0_r1）
// Heap::AllocObjectWithAllocator 是 Java 分配的核心入口（内联实现）
template <bool kInstrumented, bool kCheckLargeObject, typename PreFenceVisitor>
inline mirror::Object* Heap::AllocObjectWithAllocator(
    Thread* self,
    ObjPtr<mirror::Class> klass,
    size_t byte_count,
    AllocatorType allocator,
    const PreFenceVisitor& pre_fence_visitor) {
  // ... 前置检查 ...

  // 1. TLAB 快速路径：当前线程的 TLAB 还能装下
  if (IsTLABAllocator(allocator) && byte_count <= self->TlabSize()) {
    obj = self->AllocTlab(byte_count);   // 用户态 bump pointer，无锁
    // TLAB 已计入 num_bytes_allocated_，这次不再增加
  } else {
    // 2. 走 TryToAllocate → RegionSpace / RosAlloc / LOS
    obj = TryToAllocate<kInstrumented, false>(self, allocator, byte_count, ...);

    if (UNLIKELY(obj == nullptr)) {
      // 3. 分配失败 → 触发 GC 后重试
      obj = AllocateInternalWithGc(self, allocator, kInstrumented,
                                   byte_count, ..., &klass);
    }
  }
  // ... GC 触发检查 ...
  return obj;
}
```

`new byte[1024]` 是小对象（1024 字节 + 16 字节头 ≈ 1KB），走 TLAB 快速路径。TLAB 是 per-thread 的连续 Region 子集，**分配在用户态完成，无锁**。

**但 ART 堆不是物理内存**。它分配的是 RegionSpace 中的"逻辑页"。当 RegionSpace 容量耗尽或主动 GC 时，会触发 `ConcurrentCopying`（CC）回收，**把死亡对象的物理页释放回操作系统**。

```cpp
// art/runtime/gc/space/region_space-inl.h （AOSP android-14.0.0_r1）
// Region 分配的两个关键路径
inline mirror::Object* RegionSpace::Alloc(Thread* self, size_t num_bytes, ...) {
  num_bytes = RoundUp(num_bytes, kAlignment);
  return AllocNonvirtual<false>(num_bytes, ...);
}

// art/runtime/gc/space/region_space.cc
// AllocNewTlab：把整块 Region 分配给当前线程作 TLAB
bool RegionSpace::AllocNewTlab(Thread* self) {
  // 1. 释放当前线程原 TLAB
  RevokeThreadLocalBuffersLocked(self);
  // 2. 找一个 free region（默认 1MB 大小）→ 设置为 TLAB
  for (size_t i = 0; i < num_regions_; ++i) {
    Region* r = &regions_[i];
    if (r->IsFree()) {
      r->Unfree(time_);
      ++num_non_free_regions_;
      r->SetTop(r->End());
      r->is_a_tlab_ = true;
      r->thread_ = self;
      self->SetTlab(r->Begin(), r->End());
      return true;
    }
  }
  return false;
}

// AllocateRegion：分配非 TLAB 用的 region（用于大对象、LOS）
RegionSpace::Region* RegionSpace::AllocateRegion(RegionUse use) {
  // 类似 AllocNewTlab，但不绑定到线程
}
```

> **稳定性架构师视角：** ART 堆分配是"廉价"的——但**廉价不等于无代价**。**1KB 逻辑分配可能触发以下连锁**：
> 1. Region 不足 → 触发 Concurrent GC
> 2. GC 慢 → 触发 Background GC（阻塞 GC）
> 3. GC 失败 → 抛 OOM
>
> 排查 ART 堆分配导致的卡顿，关键看 `dumpsys gfxinfo` + ART GC trace。

### 4.3 阶段二：bionic libc 的 scudo 分配器

ART 堆在底层仍然依赖 malloc/free。Android 14 的 bionic 默认使用 **scudo 分配器**（10.0+ 替代 jemalloc，详见第 7 章）。

```cpp
// bionic/libc/bionic/malloc.cpp （AOSP android-14.0.0_r1）
// malloc 的入口分发
void* malloc(size_t byte_count) {
  if (byte_count == 0) byte_count = 1;
  // Android 平台的 allocator 决策：
  // - 调试模式：malloc_debug
  // - 静态链接：使用静态 allocator
  // - 默认：scudo
  return scudo::Allocator::allocate(...);
}
```

scudo 是基于 **Size Class + 线程本地缓存 + Quarantine** 的现代分配器：

```
scudo 分配路径
──────────────
1. 线程本地缓存 (TSD - Thread Specific Data)
   └─ size class 命中 → 直接返回（无锁，最快）

2. Size Class 不命中
   └─ 从 region 取一块 → 切分成小对象
   └─ region 不足 → mmap 申请新 region（向内核）

3. mmap 也失败
   └─ 返回 NULL（malloc 返回 NULL，触发 OOM 异常）
```

**关键代码：**

```cpp
// external/scudo/standalone/allocator.cpp （AOSP android-14.0.0_r1，bionic 通过头文件 include）
// 注意：Android 12+ 起，scudo 实现在 external/scudo/，bionic/libc/bionic/scudo.h 是适配头
void* ScudoAllocator::allocate(SizeClassType size_class, size_t size) {
  // 1. TSD 快速路径
  if (TSD::getCache().isValid()) {
    void* ptr = TSD::getCache().retrieve(size_class);
    if (ptr != nullptr) return ptr;
  }

  // 2. 从 SizeClass 池子拿
  void* ptr = SizeClassAllocator::allocate(size_class);

  // 3. 池子空了，pop 出一个 region
  if (ptr == nullptr) {
    ptr = popRegion(size_class);
  }
  return ptr;
}
```

scudo 的 region 申请最终会调到 `mmap(0, 64KB, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0)`，向内核要内存。

> **稳定性架构师视角：** **scudo 的 mmap 是用户态→内核态的唯一桥梁**。**每一次 mmap 都是一次 syscall 开销**（arm64 上 100-200ns）。**频繁的小 mmap 会让分配性能塌方**——scudo 用 64KB 大小 region 缓解，但仍有极限。**native 内存优化的核心就是"减少 mmap 次数"**：对象池、内存复用、避免大量小对象。

### 4.4 阶段三：内核的 `do_mmap` —— 创建 VMA

**mmap 系统调用是 Linux 内核进程内存的核心入口**。当 scudo 调用 `mmap(0, 64KB, ...)`，内核走 `do_mmap()` 流程：

```c
// mm/mmap.c （AOSP android14-5.10 GKI）
// do_mmap 是 mmap 入口（8 参数：注意 5.10 真实签名没有 vm_flags_t 参数；
// vm_flags 在 do_mmap 内部由 calc_vm_prot_bits/calc_vm_flag_bits 计算得到）
unsigned long do_mmap(struct file *file, unsigned long addr,
                      unsigned long len, unsigned long prot,
                      unsigned long flags, unsigned long pgoff,
                      unsigned long *populate, struct list_head *uf) {
    struct mm_struct *mm = current->mm;
    int pkey = 0;

    *populate = 0;  // 出参：是否预填充页表
    // ...

    // 1. 参数检查（长度对齐、map_count 不超限）
    if (mm->map_count > sysctl_max_map_count)
        return -ENOMEM;

    // 2. 查找空闲虚拟地址区间
    addr = get_unmapped_area(file, addr, len, pgoff, flags);

    // 3. 由 prot/flags 计算 vm_flags（这里才是 vm_flags_t 真正出现的地方，
    //    之前的版本有 vm_flags 参数，5.10 把它移到内部计算了）
    //    然后调用 mmap_region 创建 VMA；populate 标记 MAP_POPULATE 时由 mmap_region 设置为 len
    addr = mmap_region(file, addr, len, vm_flags, pgoff, uf);

    if (!IS_ERR_VALUE(addr) &&
        ((vm_flags & VM_LOCKED) ||
         (flags & (MAP_POPULATE | MAP_NONBLOCK)) == MAP_POPULATE))
        *populate = len;

    return addr;
}
```

**关键洞察**：**mmap 不分配物理内存**。它只创建 VMA（Virtual Memory Area），把虚拟地址范围标记为"我可使用"。**真正的物理分配要等首次写入**——这是 Linux 的"demand paging"机制。

> **稳定性架构师视角：** mmap 的 VMA 数量是有限度的。**每个进程默认 VMA 数量上限由 `vm.max_map_count` 控制，默认 65536**。如果 App 频繁 mmap/munmap（如加载大量 .so、频繁读文件），VMA 数量可能爆炸：
> - VMA 红黑树查找变慢（O(log n)）
> - mmap 锁竞争加剧（`mmap_write_lock`）
> - fork 时 VMA 复制变慢（Zygote 派生慢）
>
> 排查方向：`/proc/pid/maps` 数量；超过 5000 VMA 即为异常。

### 4.5 阶段四：缺页异常（Page Fault）—— 真正的物理分配

**首次写入 `byte[0]` 触发 page fault**。CPU MMU 查页表发现 PTE（Page Table Entry）无效，触发缺页异常，内核执行 `handle_mm_fault`：

```c
// mm/memory.c （AOSP android14-5.10）
// handle_page_fault → handle_mm_fault → __handle_mm_fault
vm_fault_t __handle_mm_fault(struct vm_area_struct *vma,
                              unsigned long address,
                              unsigned int flags) {
    // 1. 查 PTE，确认是匿名页
    // 2. 调 alloc_zeroed_user_highpage_movable() 分配
    // 3. 写入 PTE，建立虚拟→物理映射
    // 4. 进程从 page fault 返回，继续执行
}
```

**分配路径**：

```c
// mm/page_alloc.c
struct page *alloc_pages(gfp_t gfp, unsigned int order) {
    // fast path: pcp → buddy
    return __alloc_pages(gfp, order, ...);
}

struct page *__alloc_pages(gfp_t gfp, unsigned int order, ...) {
    // 1. fast path: get_page_from_freelist
    //    └─ 从 pcp (per-CPU page frame cache) 拿 → 命中返回
    //    └─ 从 buddy system 拿 → 命中返回
    // 2. slow path: __alloc_pages_slowpath
    //    └─ wakeup_kswapd
    //    └─ try_to_free_pages (Direct Reclaim)
    //    └─ try_to_compact_pages
    //    └─ 失败 → OOM Killer
}
```

**关键路径：pcp 缓存**

```c
// mm/page_alloc.c
// pcp (per-CPU page cache) 是 hot path 优化
struct per_cpu_pages {
    spinlock_t lock;
    int count;        // 当前缓存的页数
    int high;         // high watermark
    int batch;        // 批量操作的页数
    struct list_head lists[MIGRATE_PCPTYPES];
};
```

**pcp 缓存的意义**：避免每次分配都加 zone 锁。每个 CPU 维护自己的缓存，分配时只需要本地 CPU 原子操作。

> **稳定性架构师视角：** **pcp 是分配性能的"红利"**——但红利是有限度的：
> - 高阶分配（order ≥ 1）不走 pcp，直接进 buddy
> - 跨 NUMA 节点不走 pcp
> - pcp 缓存耗尽 → 加 zone 锁 → 慢路径
>
> 性能优化方向：**大块连续分配（如 camera buffer、Bitmap）应用大页（HugePage）减少 order 阶**。

### 4.6 阶段五：buddy system 慢路径（Direct Reclaim / OOM）

**如果 pcp 和 fast path 失败**，进入 `__alloc_pages_slowpath`：

```c
// mm/page_alloc.c
static struct page *__alloc_pages_slowpath(gfp_t gfp_mask, unsigned int order,
                                            struct alloc_context *ac) {
    // 1. 尝试 wakeup kswapd（异步回收，让 kswapd 帮忙）
    if (wait_iff_congested(ac->preferred_zone, gfp_mask, order) && ...) {
        goto retry;  // 有进展，重试 fast path
    }

    // 2. Direct Reclaim：当前进程亲自回收
    if (!(gfp_mask & __GFP_DIRECT_RECLAIM)) {
        goto nopage;  // 不允许 Direct Reclaim（如 NMI 上下文）→ 失败
    }
    __perform_reclaim(gfp_mask, order, ac);  // 阻塞回收

    // 3. 尝试 compaction（整理碎片，腾出连续大页）
    if (order > 0 && !(gfp_mask & __GFP_NOFAIL)) {
        if (compaction_ready(ac)) {
            try_to_compact_pages(gfp_mask, order, ac, ...);
        }
    }

    // 4. 重试 fast path
    page = get_page_from_freelist(gfp_mask, order, ...);
    if (page) goto got_pg;

    // 5. 都失败了 → OOM Killer
    __check_panic_on_oom(...);
    page = __alloc_pages_may_oom(gfp_mask, order, ac, &did_some_progress);
    if (page) goto got_pg;

nopage:
    return NULL;
}
```

**关键路径：Direct Reclaim**

```c
// mm/vmscan.c
static unsigned long do_try_to_free_pages(struct zonelist *zonelist,
                                           struct scan_control *sc) {
    // 1. 遍历所有 zone，回收 inactive LRU
    // 2. inactive 不足 → 回收 active LRU（refault 检查）
    // 3. 仍不足 → 回收 slab（KASAN 关闭时）
    // 4. 仍不足 → 触发 shrinker（LMK、binder 等）
    // 5. 仍不足 → 触发 memcg OOM 或全局 OOM
}
```

**OOM Killer**：

```c
// mm/oom_kill.c
// oom_badness 计算每个进程的"该杀分数"
// 分数 = RSS * 1000 / (uptime * oom_score_adj_norm)
static long oom_badness(struct task_struct *p, struct mem_cgroup *memcg,
                        const nodemask_t *nodemask, unsigned long totalpages) {
    // 1. oom_score_adj 越低越不该杀
    // 2. RSS 越大越该杀（占用多，释放多）
    // 3. 选分数最高的 → 杀
}
```

> **稳定性架构师视角：** **Direct Reclaim 是 Android 卡顿的"头号嫌疑"**。一旦某次分配进入慢路径：
> - 用户线程阻塞 5-50ms（worst case 100ms+）
> - UI 渲染线程阻塞 → 帧率下降
> - Input 事件处理阻塞 → ANR 风险
>
> 排查命令：`echo 1 > /proc/sys/vm/drop_caches`（先确认是 page cache 问题），再 `cat /proc/vmstat | grep -E 'pgscan|pgsteal'` 看 Direct Reclaim 频率。
>
> **真正的高阶优化**是：在 `__alloc_pages_slowpath` 之前用 PSI 提前感知——这就是 Android 14 LMKD 的工作。

### 4.7 一个 byte 旅程的"四张地图"

把旅程总结为四张地图，方便排查时快速定位：

| 阶段 | 关键源码 | 失败模式 | 排查命令 |
|------|---------|---------|---------|
| ① ART 堆分配 | `art/runtime/gc/heap.cc` | OOM 异常抛出 | `dumpsys meminfo` Java Heap |

## 5. 关键名词速查：PSS/RSS/VSS/USS/memcg/oom_adj/oom_score_adj/PSI some-full

> 这一章是"术语字典"。每一条都关联到具体的工程场景。**对资深架构师，本章是速查 + 跨团队沟通的统一语言**。

### 5.1 进程内存视角：VSS / RSS / PSS / USS

| 缩写 | 全称 | 定义 | 包含什么 | 用途 | 工程示例 |
|------|------|------|---------|------|---------|
| **VSS** | Virtual Set Size | 进程虚拟地址空间总大小 | 全部 mmap + brk 范围（含未分配的） | 几乎不用 | App 启动时 VSS 3-6GB（含 libc、art 等大库） |
| **RSS** | Resident Set Size | 进程实际占用的物理页数 | 实际驻留在 RAM 的页（去重前） | 性能监控、LMK 历史参考 | 一个 App RSS 4GB 表示它实际占了 4GB 物理页 |
| **PSS** | Proportional Set Size | 按共享比例分摊的物理页数 | 共享页 × (1/共享数) + 私有页 | **dumpsys meminfo 主指标** | App PSS 1.2GB 才是真正的"占用"（去重后） |
| **USS** | Unique Set Size | 进程独占的物理页数 | 完全不共享的页 | 排查泄漏的"金标准" | 内存泄漏时 USS 单调递增，PSS 不一定 |

**关键区别示例：**

```
场景：App A 和 App B 都加载了 libc.so（占 2MB）
      App A 还独占分配了 100MB 堆

VSS: A=3GB, B=3GB（虚拟地址空间，意义不大）
RSS: A=102MB, B=2MB（实际驻留，重复计算了 libc）
PSS: A=101MB (= 100MB + 1MB), B=1MB（libc 分摊）
USS: A=100MB, B=0（只算独占）

→ "App A 占多少内存？" 答案是 PSS 101MB，不是 RSS 102MB 也不是 VSS 3GB
```

> **稳定性架构师视角：** 90% 的"内存问题"沟通不畅是因为混用了这些术语。**报告内存时统一用 PSS**（dumpsys meminfo 默认输出），看独占问题时用 USS（`procrank -u`）。

### 5.2 内核视角：page / page frame / pcp / buddy / zone

| 术语 | 含义 | 出现位置 |
|------|------|---------|
| **page** | 4KB 的逻辑单位（arm64 也支持 16KB/64KB page） | `struct page` |
| **page frame** | 物理页，内核用 `struct page` 描述 | `mm/page_alloc.c` |
| **pcp** | per-CPU page frame cache，hot path 优化 | `struct per_cpu_pages` |
| **buddy** | 伙伴系统，按 order 0-10 组织空闲页 | `free_area[MIGRATE_TYPES]` |
| **zone** | ZONE_DMA / ZONE_NORMAL / ZONE_MOVABLE | `struct zone` |
| **migrate type** | UNMOVABLE / MOVABLE / RECLAIMABLE / CMA | 影响碎片 |
| **watermark** | min / low / high，触发回收的阈值 | `zone->_watermark[WMARK_*]` |
| **LRU** | 链表：active_anon / inactive_anon / active_file / inactive_file | `struct lruvec` |

### 5.3 治理视角：oom_adj / oom_score_adj / oom_score

| 术语 | 范围 | 含义 | 谁设置 |
|------|------|------|-------|
| **oom_adj** | -16 ~ +15（旧 API） | 进程被 OOM Killer 选中的"折扣" | 旧内核，已废弃 |
| **oom_score_adj** | -1000 ~ +1000 | 进程被 OOM Killer 选中的权重（新版） | Framework AMS 写 `/proc/pid/oom_score_adj` |
| **oom_score** | 0 ~ 1000 | 实时计算（基于 RSS 和 oom_score_adj） | 内核动态算 |
| **oom_badness** | 0 ~ 1000 | oom_badness() 选目标进程时的评分 | `mm/oom_kill.c` |

**关键洞察：**

```c
// oom_score_adj 是"调整因子"；oom_score 是"实时评分"
// 内核最终选 oom_score 最高的进程杀
// oom_score ≈ RSS比例 × (1000 + oom_score_adj)
```

**Android 进程 adj 等级**（`frameworks/base/services/core/java/com/android/server/am/ProcessList.java`）：

| 等级 | 含义 | 例子 |
|------|------|------|
| -1000 | NATIVE（不受系统管理） | init 子进程、native 守护进程 |
| -900 | SYSTEM | system_server |
| -800 | PERSISTENT | system、surfaceflinger |
| -700 | PERSISTENT_SERVICE | systemui 等持久化服务 |
| 0 | FOREGROUND_APP | 前台 App |
| 100 | VISIBLE_APP | 可见但非前台 |
| 200 | PERCEPTIBLE | 后台音乐（用户可感知） |
| 300 | BACKUP | 备份任务 |
| 400 | HEAVY_WEIGHT | 大型 App |
| 500 | SERVICE | 后台服务 |
| 600 | HOME | launcher |
| 700 | PREVIOUS | 上一个前台 |
| 800 | SERVICE_B | 老服务 |
| 900-999 | CACHED_APP | 后台缓存（CACHED_APP_MIN_ADJ ~ CACHED_APP_MAX_ADJ） |

> **稳定性架构师视角：** **看到进程被杀，第一件事是看它的 oom_score_adj**。如果是 999（最老缓存），被 LMKD 杀是正常；如果是 -800（系统服务），被杀是 P0 事故。**Adj 是定位"为什么被这个进程杀"的入口**。

### 5.4 压力视角：PSI some / PSI full / memcg

| 术语 | 含义 | 出现位置 |
|------|------|---------|
| **PSI some** | 至少一个任务因等内存而阻塞的时间占比 | `/proc/pressure/memory:some` |
| **PSI full** | 所有任务都因等内存而阻塞的时间占比（=系统 hang 风险） | `/proc/pressure/memory:full` |
| **avg10/60/300** | 10/60/300 秒窗口的移动平均 | 同一文件 |
| **memcg** | memory cgroup，进程组的内存配额 | `/sys/fs/cgroup/memory/<path>/` |
| **memory.high** | memcg 软上限（触发慢回收） | memcg 控制文件 |
| **memory.max** | memcg 硬上限（触发 OOM kill） | memcg 控制文件 |
| **memory.swap.high/max** | memcg swap 限制 | memcg 控制文件 |
| **memory.events** | memcg 事件：low/high/max/oom | memcg 控制文件 |
| **memory.pressure** | memcg 级别的 PSI 指标 | memcg 控制文件 |

**PSI 字段示例：**

```
$ cat /proc/pressure/memory
some avg10=12.34 avg60=8.91 avg300=5.67 total=12345678
full avg10=2.45  avg60=1.23 avg300=0.45 total=1234567
```

- `some avg10=12.34%` —— 过去 10 秒，**至少一个任务**有 12.34% 的时间在等内存
- `full avg10=2.45%` —— 过去 10 秒，**所有任务**有 2.45% 的时间都在等内存

**工程含义**：

- `some > 50%` 持续：系统进入"内存压力"模式，**LMKD 应该开始杀进程**
- `full > 5%` 持续：**用户能感受到卡顿**（所有任务都被阻塞）

> **稳定性架构师视角：** **PSI 是 Android 10+ LMKD 的"眼睛"**。在 LMK 时代（vmscan 触发），"内存压力"是个模糊概念；在 PSI 时代，"10 秒内 30% 任务在等内存"是精确可量化的。**PSI 阈值配置错误 = LMKD 行为异常**。

### 5.5 设备/分区分组：memcg / cgroup v2

| 术语 | 含义 | Android 中的应用 |
|------|------|-----------------|
| **cgroup v1** | 第一代 cgroup，按子系统分（如 memory、cpu 各自一个 hierarchy） | Android 12 之前 |
| **cgroup v2** | 统一 hierarchy，所有子系统在一个树中 | Android 13+ 默认 |
| **memcg** | cgroup 的 memory 子系统 | 进程级内存隔离 |
| **/sys/fs/cgroup/** | cgroup v2 挂载点 | Android 14 默认 |

**Android 14 的 cgroup 布局（简化）：**

```
/sys/fs/cgroup/
├── init/                    # init 进程
├── system/                  # system_server 等
├── foreground/              # 前台 App
├── background/              # 后台 App
├── system-background/       # 后台系统服务
├── apps/                    # 通用 App 容器
│   ├── uid_10001/           # App UID 10001
│   └── uid_10002/
├── top-app/                 # 当前 top app
└── ...                      # 其他
```

**关键：每个 cgroup 有独立的 memory.pressure**——这意味着 LMKD 可以**只针对 foreground/top-app** 触发 kill，而不影响后台。

## 6. 内存指标全景：dumpsys meminfo / procrank / /proc/meminfo / /proc/vmstat

> 这一章是"工具字典"。每条命令都附字段含义 + 排查场景。

### 6.2 `procrank` —— 系统级 TopN

**命令**：`adb shell procrank`（或 `/system/bin/procrank`）

**输出**：

```
  PID       Vss      Rss      Pss      Uss  Swap    SwapPSS   RTGroup   ThreadCount   Name
  1234  3123456K  234567K  123456K  100234K  12345K     1234K   foreground        85  com.example.app
   567  2345678K  198765K  100234K   90234K   2345K      234K   foreground        62  com.android.settings
   ...
```

**字段解读**：

- `VSS` / `RSS` / `PSS` / `USS`：见 5.1
- `Swap`：进程占用的 swap 空间
- `SwapPSS`：swap 按比例分摊
- `RTGroup`：cgroup 归属（foreground / background / system）

**典型用法**：

```bash
# 1. 找 PSS Top 5
adb shell procrank | head -6

# 2. 找 USS 异常增长
adb shell procrank | sort -k5 -n -r | head -10

# 3. 看特定 cgroup
adb shell procrank -c background | head
```

> **稳定性架构师视角：** **procrank 是"看全局"**——`dumpsys meminfo` 只能看一个进程，`procrank` 看全部。当系统级内存压力大时，先 `procrank` 找 Top 5 进程，再逐个 `dumpsys meminfo`。

|------|---------|
| `MemAvailable` | "系统还剩多少内存" 的金标准（比 MemFree 准确） |

### 6.4 `/proc/vmstat` —— 虚拟内存事件计数

**命令**：`adb shell cat /proc/vmstat`

**关键字段**：

```
# 页面回收
pgscan_kswapd 1234567        ← kswapd 扫描的页数
pgscan_direct 234567         ← Direct Reclaim 扫描的页数（卡顿指标）
pgsteal_kswapd 123456        ← kswapd 实际回收的页数
pgsteal_direct 12345         ← Direct Reclaim 实际回收

# 页面错误
pgfault 12345678             ← 总 page fault
pgmajfault 12345             ← major fault（需 I/O，slow）

# Swap
swap_in 1234                ← swap 读次数
swap_out 5678               ← swap 写次数

# OOM
oom_kill 12                 ← OOM Killer 触发次数
```

**关键指标**：

| 指标 | 含义 | 阈值（参考） |
|------|------|------------|
| `pgscan_direct` 增长率 | Direct Reclaim 频率 | > 100/秒 = 卡顿风险 |
| `pgmajfault` 增长率 | Major fault 频率 | > 50/秒 = 冷启动慢 |
| `swap_out` 增长 | swap 写频率 | > 10/秒 = zRAM 风暴 |
| `oom_kill` 累计 | 进程被杀次数 | > 0 = 已经杀过 |

> **稳定性架构师视角：** **`pgscan_direct` 持续增长是"内存不够"的最强信号**。当 `pgscan_direct > 0` 时，分配者已经进入慢路径。**性能优化的最高优先级是降低 `pgscan_direct` 频率**——增加 MemAvailable、降低 RSS、增加 zRAM 容量。

## 7. 历史演进：内核 LMK → 用户态 LMKD、内核 4.x → 5.x → 6.x PSI、jemalloc → scudo

> 这一章讲"为什么是这样设计的"——通过版本演进，看到 Android 内存系统的"选型压力"。

### 7.1 演进总表

| 维度 | 旧（Android 4.x - 7） | 中（Android 8 - 12） | 新（Android 13+） |
|------|---------------------|-------------------|-----------------|
| **杀手** | 内核 LMK (`lowmemorykiller.c`) | 用户态 lmkd（`lmkd.cpp`） | lmkd 2.0 + PSI |
| **事件源** | shrinker（kswapd 回调） | vmpressure | PSI + memcg |
| **分配器** | jemalloc | jemalloc → scudo 过渡 | scudo 默认 |
| **回收算法** | 简化 LRU | 4-链表 LRU | 多代 LRU (MLN) |
| **memcg** | v1，hierarchy 复杂 | v1/v2 过渡 | v2 |
| **PSI** | 不存在 | 引入（4.10+） | Android 默认 |
| **压力传播** | shrinker 全局视角 | memcg 隔离 | memcg + PSI 细粒度 |

### 7.2 内核 LMK → 用户态 LMKD

**旧：内核 LMK（drivers/staging/android/lowmemorykiller.c）**

```c
// drivers/staging/android/lowmemorykiller.c （旧，4.9 内核）
// 核心逻辑：kswapd 触发 → 遍历所有进程 → 选 adj 最大的 → kill
static unsigned long lowmem_scan(struct shrinker *s, struct shrink_control *sc) {
    int other_free = global_page_state(NR_FREE_PAGES) - totalreserve_pages;
    int other_file = global_page_state(NR_FILE_PAGES) - ...;

    for (i = 0; i < array_size; i++) {
        minfree = lowmem_minfree[i];
        if (other_free < minfree && other_file < minfree) {
            min_score_adj = lowmem_adj[i];
            break;
        }
    }

    for_each_process(tsk) {
        // 选 adj 最大 + RSS 最大的进程
        // send_sig(SIGKILL, selected, 0);
    }
}
```

**问题：**

1. **杀进程在内核态**：触发 watchdog、crash 难以调试
2. **无法跨 cgroup 隔离**：foreground 和 background 进程同台竞争
3. **策略僵化**：水线/adj 数组写死在代码，运行时调参受限
4. **缺乏标准化接口**：vendor 修改不统一，碎片化严重

**新：用户态 LMKD（system/memory/lmkd/，Android 10+ 默认）**

```cpp
// system/memory/lmkd/lmkd.cpp （AOSP android-14.0.0_r1）
// 用户态主循环
int main(int argc, char **argv) {
    // 1. mlockall(MCL_FUTURE) - 锁内存，避免自己被换出
    // 2. SCHED_FIFO 实时调度
    // 3. init() - 创建 epoll + 监听 PSI
    // 4. mainloop() - 事件循环
    while (1) {
        int nevents = epoll_wait(epollfd, events, maxevents, -1);
        for (i = 0; i < nevents; ++i) {
            // 处理 PSI 事件 / ctrl socket 事件
        }
    }
}
```

**改进：**

1. **杀进程在用户态**：root 权限，可访问 vendor 策略，可写日志
2. **基于 cgroup**：foreground/background 隔离
3. **可热配置**：`/sys/fs/cgroup/.../memory.*` 动态调
4. **PSI 事件驱动**：提前感知，避免 Direct Reclaim 抖

**演进图**：

```
Android 1.x - 7.x:    内核 LMK
Android 8.x (O):      用户态 lmkd 引入（默认仍走内核）
Android 10.x (Q):     lmkd 默认（内核 LMK deprecated）
Android 12.x (S):     lmkd 2.0（PSI 为主，vmpressure 兜底）
Android 13.x (T):     lmkd 2.0 + 内存分级（low/critical）
Android 14.x (U):     lmkd 2.1 + memcg-aware
```

> **稳定性架构师视角：** **看 bug report 第一眼要分清是哪个版本**：
> - Android 7- 报告：可能是内核 LMK 误杀（看 `/sys/module/lowmemorykiller/parameters/adj`）
> - Android 10+ 报告：大概率是 lmkd 决策（看 `logcat -s lmkd`）
> - 跨版本对比时尤其注意 adj 等级含义变了

### 7.3 内核 4.x → 5.x → 6.x PSI

**PSI 是什么**：Pressure Stall Information，统计"任务因某种资源而阻塞的时间占比"。

**演进**：

| 内核 | PSI 特性 | Android 集成 |
|------|---------|-------------|
| 4.20+ | PSI 引入 | 不集成 |
| 5.4+ | memcg-aware PSI | 准备集成 |
| 5.10+ | cgroup v2 PSI | Android 12+ |
| 6.1+ | PSI 优化 | Android 14+ |

**Android 14 内核 5.10 的 PSI 读取**：

```bash
# 系统级 PSI
$ cat /proc/pressure/memory
some avg10=12.34 avg60=8.91 avg300=5.67 total=12345678
full avg10=2.45  avg60=1.23 avg300=0.45 total=1234567

# cgroup 级 PSI
$ cat /sys/fs/cgroup/foreground/memory.pressure
some avg10=5.67 avg60=3.45 avg300=1.23 total=1234567
full avg10=0.12 avg60=0.05 avg300=0.01 total=123456
```

**Android 14 LMKD 的 PSI 阈值（典型配置）**：

```
ro.lmk.psi_partial_stall_ms = 70      ← some > 70ms 触发
ro.lmk.psi_complete_stall_ms = 100    ← full > 100ms 触发
ro.lmk.psi_some_threshold_pct = 50    ← 任务在 50% 时间等
ro.lmk.psi_full_threshold_pct = 70
```

> **稳定性架构师视角：** **PSI 让"内存压力"从黑盒变白盒**。但 PSI 阈值配置错误会让 LMKD 行为异常：
> - 阈值太低 → LMKD 太激进 → 误杀
> - 阈值太高 → LMKD 反应慢 → 卡顿后才杀
>
> **生产经验**：先开 `ro.lmk.debug = 1` 观察 1-2 天，根据 PSI 趋势调阈值。

### 7.4 jemalloc → scudo 分配器

**演进**：

| Android 版本 | 默认分配器 | 理由 |
|------------|----------|------|
| 1.0 - 4.x | dlmalloc | 简单 |
| 5.0 - 9 | jemalloc | 性能更好 |
| 10.0 - 11 | jemalloc + scudo（可切换） | 引入 scudo |
| 12.0+ | scudo 默认 | 安全 + 性能 |
| 14.0+ | scudo | 加 Hardening |

**scudo vs jemalloc**：

| 维度 | jemalloc | scudo |
|------|---------|-------|
| **线程缓存** | 8 个 size class 的 TSD | 16 个 size class 的 TSD |
| **Quarantine** | 无 | 有（防 UAF） |
| **Hardening** | 弱 | 强（chunk 头校验、随机化） |
| **调试模式** | mallopt | malloc_debug（独立库） |
| **性能** | 高 | 接近 jemalloc |
| **安全** | 一般 | 业界领先（强化版本） |

**关键源码（scudo 核心）**：

```cpp
// external/scudo/standalone/allocator.cpp （AOSP android-14.0.0_r1）
class ScudoAllocator {
public:
    void* allocate(SizeClassType size_class, size_t size) {
        // 1. TSD 快速路径（线程本地，无锁）
        if (TSD::getCache().isValid()) {
            if (void* ptr = TSD::getCache().retrieve(size_class)) {
                return ptr;
            }
        }
        // 2. Region 池
        void* ptr = SizeClassAllocator::allocate(size_class);
        if (UNLIKELY(!ptr)) {
            ptr = popRegion(size_class);
        }
        // 3. 校验 Hardening
        return chunk->getMem();
    }
};
```

> **稳定性架构师视角：** **scudo 的 Quarantine 是"安全红利"**——释放的内存不会立刻复用，会进入隔离队列一段时间。这对调试 use-after-free 极有帮助。**但 Quarantine 也有副作用**：极端分配/释放场景会多占 5-15% 内存。

### 7.5 演进对工程师的影响

| 版本 | 影响 |
|------|------|
| Android 4.x 报告 | 看不到 cgroup，内存调试工具少 |
| Android 8.x 报告 | lmkd 引入但默认内核 LMK，需要看 build 配置 |
| Android 10+ 报告 | PSI 可用，可调阈值 |
| Android 12+ 报告 | memcg v2 默认启用，cgroup 路径变了 |
| Android 14+ 报告 | scudo 完全替代 jemalloc，注意 chunk 头变化 |

## 8. 五大类稳定性问题映射：OOM / 泄漏 / 抖动 / 杀进程 / 卡顿

> 这一章是"风险地图"——把线上 5 类问题映射到层，定位排查入口。

### 8.1 五大类问题全景图

```
稳定性问题
├── OOM（Out of Memory）
│   ├── Java OOM            → ART 堆满（heap max）
│   ├── Native OOM          → scudo 失败 / mmap 失败
│   ├── System OOM          → 内核 OOM Killer
│   └── Address Space OOM   → VMA 用尽 (max_map_count)
│
├── 内存泄漏（Leak）
│   ├── Java 堆泄漏          → ART GC 不回收（强引用）
│   ├── Native 泄漏          → malloc 不释放 / fd 泄漏
│   ├── 资源泄漏            → fd / 线程 / Window
│   └── Bitmap 泄漏         → Native 堆泄漏
│
├── 抖动（Thrashing）
│   ├── GC 抖动              → ART 频繁 GC
│   ├── Direct Reclaim 抖动  → 慢路径
│   ├── swap 风暴            → zRAM 反复压缩
│   └── 锁竞争抖动          → mmap 锁 / page_table_lock
│
├── 杀进程（Kill）
│   ├── LMKD 杀              → 内存压力
│   ├── OOM Killer 杀        → 内核级压力
│   ├── Watchdog 杀          → ANR 超时
│   └── Crash 杀             → SIGSEGV
│
└── 卡顿（Jank）
    ├── PSI some 高          → 内存压力
    ├── GC pause 长          → STW
    ├── Direct Reclaim 阻塞  → 慢分配
    └── Swap 阻塞            → zRAM 压缩慢
```

### 8.2 问题 → 层 → 排查入口 速查

| 问题类型 | 主要层 | 日志关键字 | 排查命令 | 治理方向 |
|---------|-------|----------|---------|---------|
| **Java OOM** | ART | `OutOfMemoryError`、`GC freed` | `dumpsys meminfo` Java Heap | 减小对象大小、释放引用 |
| **Native OOM** | bionic scudo | `scudo error`、`malloc failed` | `procrank`、`malloc_debug` | 检查 Native 分配、Bitmap |
| **System OOM** | 内核 | `oom-kill`、`Out of memory` | dmesg | 减少内存占用 |
| **泄漏 Java** | ART | `LeakCanary`、Finalizer 队列 | heap dump、MAT | Reference 链分析 |
| **泄漏 Native** | bionic | `unreachable memory` | libmemunreachable、mallopt | 资源释放、生命周期 |
| **GC 抖动** | ART | `GC_Wait`、`Concurrent Mark` | Perfetto、ART trace | 减少分配、对象池 |
| **Direct Reclaim 抖动** | 内核 mm/ | `shrink_inactive_list`、ftrace mm events | `/proc/vmstat` pgscan_direct | 增加 zRAM、降 RSS |
| **swap 风暴** | 内核 mm/ | `swap_in`、`swap_out` | `/proc/vmstat` swap_* | swappiness 调优 |
| **LMKD 误杀** | LMKD | `lmkd kill`、`lowmem` | logcat -s lmkd | adj 调整、白名单 |
| **OOM Killer 杀** | 内核 mm/ | `oom-kill` | dmesg | 减小 RSS |
| **杀进程→冷启动** | LMKD/AMS | `Process killed` | logcat、procstats | 后台保活策略 |

## 9. 实战案例（典型模式）

> 本节给出两个端到端排查案例，覆盖 App 视角（Layer 1-2）和系统视角（Layer 3-4）。**所有案例基于通用模式构造，标注"典型模式"**。详细数据、Perfetto trace 走读放在 02-13 篇展开。

### 9.1 案例 A：某 App 启动后 PSS 持续增长（典型模式）

**现象描述**

某社交类 App 在 Android 14 设备上反复启动 - 浏览图片流 5 分钟 - 退出 - 重新启动的循环中，**单进程 PSS 从 380MB 单调增长到 1.2GB，10 次循环后触发系统 OOM Killer**。Bug report 中 `dumpsys meminfo` 显示如下异常：

```
                   Pss  Private  Private
                 Total    Dirty    Clean
                ------   ------   ------
  Native Heap    156MB    156MB        0     ← 持续增长
  Java Heap       48MB     48MB        0     ← 正常
  Graphics       620MB    620MB        0     ← 异常增长
  Code            82MB      2MB     80MB
  Stack            3MB      3MB        0
  TOTAL         1.18GB    980MB    200MB
```

**第一步：定位层（PSS/USS 拆分）**

`dumpsys meminfo` 中 `Native Heap 156MB` + `Graphics 620MB` 占了 PSS 增长的 90% 以上。**Graphics 是 Native 堆中的图形缓冲区（包括 SurfaceView、HWUI TextureCache、Glide Bitmap Pool）**。运行 `procrank` 跨多次启动对比 USS：

```bash
# 第 1 次启动后
$ adb shell procrank | grep com.example.app
  5678  3.1GB  456MB  380MB  195MB  ...  com.example.app

# 第 5 次启动后
  5678  3.2GB  892MB  780MB  510MB  ...  com.example.app   ← USS 涨了 315MB

# 第 10 次启动后
  5678  3.3GB  1.3GB  1.2GB  980MB  ...  com.example.app   ← USS 单调递增
```

**USS 单调递增**是 Native 内存泄漏的"金标准"——说明泄漏不在共享库里。

**第二步：dumpsys gfxinfo 拆分 Graphics**

```bash
$ adb shell dumpsys gfxinfo com.example.app reset
# 浏览 5 分钟后
$ adb shell dumpsys gfxinfo com.example.app
 Total memory allocated:
   HWUI:    18MB     ← 正常
   Skia:   240MB     ← 异常
   GL:     362MB     ← 异常（Texture 缓存）
```

`GL: 362MB` 锁定根因：GL Texture 缓存（即 Glide 的 BitmapPool / ImageView 的 TextureView 缓存）未释放。

**第三步：根因分析**

抓取 Perfetto trace，看到如下关键路径：

```
T=0      App.onCreate() → ImageView 加载图片
T=10ms   Glide.with(ctx).load(url).into(imageView)
T=50ms   BitmapFactory.decodeStream() → 申请 4MB Bitmap
T=51ms   HWUI TextureCache.put(bitmap) → Native 堆 +4MB
T=15s    列表滑动到第 N 项
T=20s    旧 ImageView 被 RecyclerView 回收
T=21s    Bitmap 引用断开 → GC 标记
T=22s    [GL TextureCache.put() 旧 bitmap 应在此处 evict]
         ─── 实际：texture cache 未配置上限，永不 evict ───
```

**根因**：GL TextureCache 是个无上限 LRU 缓存，**从未配置 `BitmapPool` 大小上限**。每个新图片进入缓存，旧图片被挤掉一部分，但 Native 堆（Layer 2 的 scudo 视角）的总占用持续增长。

**修复方案**

1. **限制 Glide BitmapPool 上限**：
   ```java
   // Application.onCreate()
   Glide.get(this).setMemoryCategory(Glide.MemoryCategory.NORMAL);
   // 或者自定义 MemorySizeCalculator
   GlideBuilder builder = new GlideBuilder(this);
   builder.setMemoryCache(new LruResourceCache(20 * 1024 * 1024));  // 20MB 上限
   builder.setBitmapPool(new LruBitmapPool(20 * 1024 * 1024));      // 20MB 上限
   Glide.init(builder, /*registry*/ null);
   ```
2. **App 端响应 trim 回调**：
   ```java
   @Override
   public void onTrimMemory(int level) {
       super.onTrimMemory(level);
       if (level >= TRIM_MEMORY_UI_HIDDEN) {
           Glide.get(this).clearMemory();   // 关键：UI 不可见时清空
       }
   }
   ```
3. **Bitmap.recycle()** 在 Activity.onDestroy 中强制释放大图：
   ```java
   @Override
   protected void onDestroy() {
       super.onDestroy();
       imageView.setImageBitmap(null);
       if (bitmap != null && !bitmap.isRecycled()) {
           bitmap.recycle();
       }
   }
   ```

**修复效果验证**：

| 指标 | 修复前 | 修复后 |
|------|-------|-------|
| 单次浏览 5min 后 USS | 510MB | 95MB |
| 10 次循环后 USS | 980MB | 110MB |
| 触发 OOM 次数 | 1/10 次 | 0/100 次 |
| dumpsys meminfo Graphics | 620MB | 35MB |

> **稳定性架构师视角：** **USS 是诊断 Native 泄漏的金标准**。**一旦看到 USS 单调递增且重启不归零**——立刻怀疑 Bitmap / GraphicBuffer / Native IO 缓存。**dumpsys gfxinfo 的 GL/HWUI/Skia 三段拆分**是定位图形层泄漏的入口。**修复的核心是"配置上限 + 响应 trim"**——任何无上限的 Native 缓存，最终都会成为 OOM 源头。深入 Native 内存优化详见 [04-Native 堆内存与分配器（AOSP 14）](04-Native 堆内存与分配器（AOSP 14）.md)。

### 9.2 案例 B：某后台 Service 被 LMKD 误杀（典型模式）

**现象描述**

某 IM 类的后台长连接 Service（通过 `startForegroundService()` 持有通知），在低内存设备（4GB RAM，启用 `ro.config.low_ram=true`）上**每 5-10 分钟被系统杀一次**。`logcat` 关键日志：

```
03-12 14:23:45 lmkd: Process com.example.im killed (adj 400), oom_score_adj 400
03-12 14:23:45 ProcessRecord: Process com.example.im has died
03-12 14:23:45 Notification: cancelAllPck com.example.im
```

**关键观察**：`adj 400`（即 `HEAVY_WEIGHT_APP_ADJ`）被杀——但这是前台 Service 的预期 adj？**问题就出在这里**。

**第一步：检查进程的 adj 状态**

```bash
$ adb shell dumpsys meminfo com.example.im
  ...
  App Summary
                       Pss(KB)
           Java Heap:     5430
         Native Heap:    18932
                Code:    12344
               Stack:      892
            Graphics:     2145
       Private Other:     1203
            System:     3456
               TOTAL:    44402
  ...
  Adj:           400     ← 异常
  Sched Group:   Background
  ProcState:     16 (PROCESS_STATE_CACHED_EMPTY)
```

**`ProcState: PROCESS_STATE_CACHED_EMPTY` 是关键**。这意味着 AMS 判定当前进程"空进程"（没有可见 Activity、没有活跃 Service）。**但 App 开发者认为自己是前台 Service**——冲突在哪里？

**第二步：检查 Service 的 onStartCommand 返回值**

查看 App 端代码：
```java
// 错误示例
public class IMService extends Service {
    @Override
    public int onStartCommand(Intent intent, int flags, int startId) {
        // 缺了 startForeground() 也没 return START_STICKY
        doConnect();   // 异步建立长连接
        return START_NOT_STICKY;   // 关键错误
    }
}
```

**两个问题叠加**：
1. `START_NOT_STICKY` —— Service 被杀后不会重启
2. `startForegroundService` 后 5 秒内没调 `startForeground()`（即违规 §8 Service 启动限制）—— **AMS 直接把 adj 设为 400，procState 设为 CACHED_EMPTY**

**第三步：dmesg + /proc/pressure/memory 验证**

```bash
# 看系统是否真的进入低内存
$ adb shell cat /proc/pressure/memory
some avg10=32.45 avg60=28.91 avg300=18.34 total=1234567890
full avg10=5.67  avg60=4.12  avg300=2.45  total=123456789

# lmkd 决策日志
$ adb logcat -d | grep -i "lmkd\|low memory"
03-12 14:23:45 lmkd PSI some=32% > 30% threshold, start kill scan
03-12 14:23:45 lmkd Selected pid=12345 adj=400 rss=44000KB to kill
```

PSI 指标显示系统确实处于压力中，LMKD 按规则在找最低 adj 候选——**adj 400 的 Service 是被规则选中的"最该杀"对象**。

**第四步：根因分析**

**根因链**：

```
App: startForegroundService(intent)
  → AMS: processRecord.setForegroundService(true)  // 临时设 adj=200
  → App: 5 秒内未调 Service.startForeground()       // 启动违规
  → AMS: 取消前台 Service 标记，adj 回到 400
  → AMS: 1 分钟后未发现活跃 Service，procState 降为 16 (CACHED_EMPTY)
  → 内存压力: PSI some > 30%
  → LMKD: 扫描 adj slot[400] → 选中 com.example.im
  → SIGKILL
```

**修复方案**：

1. **修复 Service onStartCommand**：
   ```java
   public class IMService extends Service {
       private static final int NOTIF_ID = 1001;
       
       @Override
       public void onCreate() {
           super.onCreate();
           // 关键：5 秒内必须调 startForeground()，且 notification 必须常驻
           startForeground(NOTIF_ID, buildNotification());
       }
       
       @Override
       public int onStartCommand(Intent intent, int flags, int startId) {
           doConnect();
           return START_STICKY;   // 被杀后系统会重启
       }
   }
   ```

2. **响应 onTrimMemory 主动释放**：
   ```java
   @Override
   public void onTrimMemory(int level) {
       super.onTrimMemory(level);
       if (level >= TRIM_MEMORY_RUNNING_LOW) {
           // 释放非关键内存
           clearOldMessages();
       }
   }
   ```

3. **检测 adj 异常并告警**（生产端）：
   ```java
   // 通过 dumpsys meminfo 抓取 adj，做线上监控
   public void logAdjAnomaly() {
       int adj = Process.getOomAdj();
       if (adj > PERCEPTIBLE_APP_ADJ && isForegroundService) {
           reportAnomaly("IMService adj=" + adj);
       }
   }
   ```

**修复效果**：

| 指标 | 修复前 | 修复后 |
|------|-------|-------|
| 1 小时内被杀次数 | 8-12 次 | 0 次 |
| dumpsys meminfo Adj | 400 (CACHED_EMPTY) | 200 (FOREGROUND_SERVICE) |
| 长连接断线率 | 60% | < 5% |

> **稳定性架构师视角：** **LMKD 杀的是 adj 高的进程，不是 RSS 大的进程**。**一个 adj=200 的 Service，RSS 1GB 也不会被杀**；**一个 adj=400 的 Service，RSS 50MB 也会被杀**。**排查"为什么我的进程被杀"的顺序是：① 确认 adj ② 确认 procState ③ 确认 PSI 指标 ④ 确认 LMKD 决策日志**。**adj 异常 90%+ 场景是 AMS 的问题，不是 LMKD 的问题**——是你 App 的 Component 状态没让 AMS 正确判定。深入 adj 治理详见 [05-AMS 内存治理与进程优先级](05-AMS 内存治理与进程优先级.md)。

## 总结：架构师视角的 5 条 Takeaway

> 作为稳定性架构师，排查 Android 内存问题时需要记住以下关键认知：

1. **Android 内存是"五层分布式系统"，不是"一个分配器"**。**单点调参无用，必须按层定位**。从 App Java 堆到 DRAM，经历 ART → bionic → 内核 mm/ → MMU 四道边界。**任何一个边界的瓶颈都会成为问题表象**。**排查的第一步永远是"问题在哪一层"**。

2. **PSS 是和团队沟通的统一语言**。**报告内存时统一说 PSS**，看独占问题时用 USS。**VSS 和 RSS 是历史遗留，在新代码评审中应避免出现**。一份好的 bug report：`App X 在 Y 场景 PSS 增长 Z MB，其中 Native Heap 增长 W MB`——而不是"App X 内存涨了 500MB"。

3. **"内存去哪了"比"内存还剩多少"重要一万倍**。`MemAvailable < 1GB` 是结论，不是原因。**`procrank` + `dumpsys meminfo` + `/proc/vmstat` 联合使用**才能定位"被谁吃掉了"——是 App 自身 RSS 增长？是 SLAB 泄漏？是 page cache 占用？是 zRAM 写满？

4. **"卡顿"和"OOM"是内存问题的两个出口**。**卡顿 = Direct Reclaim 阻塞或 PSI some 高**，**OOM = 慢路径走到 oom_kill**。**架构师的目标是让系统"在卡顿发生前就杀进程"**——这正是 PSI 和 LMKD 2.0 的设计哲学。**老 LMK 时代是"卡顿到极限了才杀"**，**新 LMKD 时代是"还没卡顿就先杀"**。

5. **每个 Android 版本都是"新坑"**。**LMK → LMKD、jemalloc → scudo、cgroup v1 → v2、shrinker → PSI**——**每一代演进都改变了"问题表象"**。**看一份 bug report 之前，先看 Android 版本**——同一个"杀进程"问题，在 Android 7、10、12 上的根因可能完全不同。

**排查路径速查**：

```
问题现象 → 排查入口
─────────────────────────────
Java OOM              → dumpsys meminfo Java Heap + ART GC trace
Native OOM            → procrank + malloc_debug
进程被杀（LMKD）       → logcat -s lmkd
进程被杀（OOM）        → dmesg | grep -i oom
系统卡顿              → /proc/pressure/memory + /proc/vmstat
冷启动慢              → procrank + Zygote fork time
Bitmap 泄漏           → dumpsys meminfo Graphics + libmemunreachable
Window 泄漏           → dumpsys window | grep leaked
```
