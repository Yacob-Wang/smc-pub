## 第 0 章 阅读路线图

承接 [08] 已经把物理 RAM 翻译成 `struct page × N` 并按 node / zone 索引。本篇回答的是"分配请求来了之后，内核怎么快速、可预测、低碎片地把一段连续物理页交给调用方"。五个层层递进的问题：

```
Q1: 为什么不能简单地"找个连续 2^order 段就返回"？                ←  §1  伙伴系统的动机：避免外碎片
Q2: alloc_pages(gfp, order) 调进来之后，代码路径上发生了什么？   ←  §2  fast path：get_page_from_freelist
Q3: zone 什么时候才"有资格"被这次分配选上？                     ←  §3  zone_watermark_ok / zone_watermark_fast
Q4: 满足不了 watermark 时，内核的"兜底"动作是什么？              ←  §4  slow path：kswapd / direct reclaim / compact
Q5: 释放路径上怎么"主动合并"以避免碎片膨胀？                    ←  §5  free_one_page + buddy merge
Q6: 多核竞争下，per-CPU 缓存为什么能减少 zone lock？            ←  §6  PCP / rmqueue_bulk
Q7: 真实项目里我从哪儿开始调优？                                ←  §7  架构师 Takeaway
```

这七个问题，对应 6 个核心机制：binary buddy merge、fast path 分配、watermark 守卫、slow path 回收、释放时的反向合并、PCP 批量缓存；以及 4 个核心数据结构：`struct zone->free_area[]`（11 阶双向链表）、`struct per_cpu_pages`（per-cpu 热缓存）、`struct page`（[08] 已讲）、`enum migratetype`（5+1 种迁移类型）。它们的关系：

```
alloc_pages(gfp, order)
    │
    ▼
__alloc_pages(gfp, order)            ← fast path 入口
    │
    ▼
get_page_from_freelist() ─→ 遍历 zonelist，按 order 取 free_area[order]
    │                              │
    │                              ├─ zone_watermark_ok() 守卫（§3）
    │                              ├─ rmqueue() → __rmqueue_smallest()（§2）
    │                              └─ rmqueue_bulk() 从 PCP 拉批量（§6）
    │
    └─→ __alloc_pages_slowpath()    ← 失败 / watermark 不达标
            │
            ├─ wakeup_kswapd()              异步回收（[11] 详讲）
            ├─ __perform_reclaim() / 直接 reclaim   同步回收
            └─ __compact_memory() / compaction    内存规整（页迁移）
                                          │
                                          ▼
                                  最终仍然失败 → warn_alloc / dump_stack
```

**本篇核心价值**：把 `alloc_pages` 这条"全内核调用频率 TOP 5 的函数"在 GKI 5.10 上的完整路径讲透——让你在看到 `__alloc_pages: 1-page allocation failure`、看到 `order: 3`、看到 `Normal` zone 时，**5 秒内定位到 fast path 还是 slow path、是哪一档 watermark 不达标、是 fragmentation 还是 true OOM**。

## 第 1 章 引子：伙伴系统为什么是页分配器

### 1.1 一句话定义

**伙伴系统（binary buddy system）是 Linux 内核 `mm/page_alloc.c` 在 zone 上实现的"按 2^k 大小组织空闲页"的算法和数据结构，作用是把零散的 `struct page` 回收成"连续、可拆分、可合并、可按 order 拉取"的物理页块，从而以 O(log N) 时间和接近 0 的外碎片代价满足任意 `alloc_pages(gfp, order)` 请求（order ∈ [0, MAX_ORDER-1]，GKI 5.10 默认 MAX_ORDER = 11）。**

### 1.2 为什么需要伙伴系统——单一平坦视角会撞上的 4 类问题

如果内核用"一块大数组，按线性扫描找一段连续 N 页"做分配器，会立即撞上 4 类不可调和的问题：

| # | 问题 | 线性扫描的后果 | 伙伴系统的解法 |
|---|------|--------------|--------------|
| 1 | **外碎片（external fragmentation）** | 多次分配/释放后出现"总空闲页够但无连续 N 页"的死局；典型线上 `order-5 (128KB)` 失败 | 严格按 2^k 切分与合并，每一次合并只发生在一对"大小相等 + 物理地址相邻 + 同一 migrate type"的 buddy 之间 |
| 2 | **合并时机不可控** | 释放时只能简单归还，无法主动合并同伴 → 碎片指数增长 | 释放时递归 `free_one_page` 检查 partner 是否也是 free、order 相同 → 合并 → 检查更高 order 的 partner |
| 3 | **分配/释放时间复杂度** | 线性扫描是 O(N)，8GB RAM 下扫一遍需要数十毫秒 | 每一阶 order 是一个双向链表，`__rmqueue_smallest` 是 O(1) 取链表头 + O(order) 拆分 |
| 4 | **NUMA / 迁移类型亲和** | 线性视角丢失 NUMA 局部性、无法按 migrate type 隔离 → CMA / 不可移动页混在一起 → 后续迁移失败 | `zone->free_area[order].free_list[MIGRATE_TYPES]` 数组把每个 order × 每个 migrate type 独立挂链，PCP 同样按 `MIGRATE_TYPES` 隔离 |

这 4 个问题层层递进——外碎片是表象，合并是机制，O(log N) 是性能保证，migrate type 隔离是 NUMA/anti-frag 的协作手段。本章按"动机 → 算法 → 数据结构 → 数量级"的顺序展开。

### 1.3 伙伴系统的核心思想：二进制伙伴（binary buddy）

"buddy" 这个词来自于它的合并规则：**两个大小相等的相邻块（"伙伴"）在两者都空闲时合并成一块大一阶的块**。Linux 内核用的是"二进制 buddy"——阶只取 2 的幂。

```
设阶 order = k，对应块大小 B = 2^k 个连续页（4KB × 2^k = 2^(k+12) 字节）。
块起始物理页号 pfn 满足：pfn & (B - 1) == 0   （即按 B 对齐）

伙伴块的 pfn：
  partner_pfn = pfn ^ B   ←  XOR 操作；异或 B 把对齐位翻转，定位到物理上相邻的伙伴
```

关键性质：**伙伴块是唯一确定的、与块大小一一对应的**。XOR 操作之所以成立，是因为伙伴两块在二进制地址上的差别只在"低 k 位之外的那一位"——也就是 2^k 那一 bit。

举 4 阶（order=4，64 页 = 256KB）块为例：

```
假设起始 pfn = 0x1000_0000（256MB 处），块大小 B = 2^4 = 16 页 = 64KB
其伙伴块的 pfn = 0x1000_0000 ^ 16 = 0x1000_0010
                          块 1 起始           块 2 起始
```

### 1.4 外碎片 vs 内碎片

伙伴系统解决**外碎片**（free pages 足够但物理上不连续），代价是引入**内碎片**（分配 order=0 时给你一整页没商量，分配 order=5 时给你 32 页，但申请方只要 25 页——浪费 7 页）。这是经典的"空间换时间 / 灵活性换可控性"权衡：

```
            内碎片代价                            外碎片代价
        （每次浪费 ≤ 50%）                  （找一段连续 N 页）
                      │                                 │
线性扫描分配器          │         完美匹配分配器           │      伙伴系统
（无内碎片）            │         （外碎片累积到死局）       │      （内碎片 + 完美外碎片控制）
                      │                                 │
       0 ~ 49%          │           不可预测             │      0 ~ 49%（理论最大）
```

GKI 5.10 / Android 14 实战中，"浪费"那部分页大多数会被 SLAB/SLUB（[10]）立即拆给小对象，所以实际上内碎片代价远低于理论 50%。

### 1.5 关键数据结构（v5.10 真实字段）

**`struct free_area`**——每一个 zone 内有 `free_area[MAX_ORDER]`，每个 free_area 描述一个阶的空闲页块：

```c
/* include/linux/mmzone.h — GKI 5.10 真实字段 */
struct free_area {
    struct list_head    free_list[MIGRATE_TYPES];   /* 5+1 种迁移类型的双向链表头 */
    unsigned long       nr_free;                    /* 当前阶的总空闲页数（含所有 migrate type） */
};
```

**关键字段解释**：

| 字段 | 含义 | 关键不变量 |
|------|------|----------|
| `free_list[i]` (i ∈ MIGRATE_TYPES) | migrate type 为 i 的空闲块链表头 | 一个 free block 通过 `page->lru` 链入 |
| `nr_free` | 该阶所有 migrate type 的总空闲页数（按 order 计，不是按块计） | `__rmqueue_smallest` 取块后减 `1 << order`；`free_one_page` 还块后加 `1 << order` |

**`struct per_cpu_pages`**——per-CPU 热缓存（详细见 [§6](#6-pcpper-cpu-pages批量分配--migrate-type-隔离)）：

```c
/* include/linux/mmzone.h — GKI 5.10 真实字段 */
struct per_cpu_pages {
    spinlock_t      lock;            /* pcp 自旋锁（统计上报用，正常路径 lockless） */
    int             count;           /* 当前 pcp 缓存的总页数 */
    int             high;            /* 高水位（达到则停止 drain） */
    int             high_min;        /* 5.10 新增：high 的下限 */
    int             high_max;        /* 5.10 新增：high 的上限 */
    int             batch;           /* 单次 refill/drain 的批量大小 */
    u8              expire;          /* 5.10 新增：上次 refill 距今的代数（0=刚 refill） */
    u16             flags;           /* PCP 行为 flag（如 PCPF_ALLOW_MIGRATION） */
    short           free_count;      /* 5.10 新增：lists[] 总页数（与 count 同义，5.10 起双计） */
    struct list_head lists[MIGRATE_TYPES]; /* 5+1 条 migrate type 维度的 freelist */
};
```

**`enum migratetype`**（`include/linux/mmzone.h`）——迁移类型，决定"这块页能不能搬"：

```c
/* GKI 5.10 真实定义；MIGRATE_TYPES 是数组维度（=6），不是有效类型 */
enum migratetype {
    MIGRATE_UNMOVABLE     = 0,   /* 不可移动：内核页表、某些 DMA buffer */
    MIGRATE_MOVABLE       = 1,   /* 可移动：用户页、page cache、匿名页 */
    MIGRATE_RECLAIMABLE   = 2,   /* 可回收：内核缓存、不需要保留 */
    MIGRATE_HIGHATOMIC    = 3,   /* 5.10 保留：原子上下文的不可移动预留（早期为 CMA 预留） */
    MIGRATE_CMA           = 4,   /* CMA：连续内存预留，分配时与其他类型物理隔离 */
    MIGRATE_ISOLATE       = 5,   /* 隔离：内存规整 / 故障下线时使用 */
    MIGRATE_TYPES         = 6,   /* 数组维度上限 */
};
```

**关键设计意图**：

- **MIGRATE_UNMOVABLE**：被分配的页之后基本不会被迁移（页表 / 设备绑定 DMA），属于"必须落到固定位置"。占比通常 < 5%，但碎片化敏感。
- **MIGRATE_MOVABLE**：用户态 mmap / 匿名页 / page cache，可以被 `__compact_memory` 搬移。占比通常 60-80%。
- **MIGRATE_RECLAIMABLE**：内核缓存（dentry / inode），回收路径上优先释放。占比 < 5%。
- **MIGRATE_CMA**：连续物理内存预留，给 DMA / camera ISP / 大块页分配。典型 32-256MB。
- **MIGRATE_ISOLATE**：虚拟状态，所有分配都跳过它；只在 `mm/memory_hotplug.c` 或 `mm/page_isolation.c` 主动隔离一段内存时使用。
- **MIGRATE_HIGHATOMIC**：保留类型，5.10 默认不上线；早期为 atomic 上下文预留。

### 1.6 数量级（GKI 5.10 / Android 14 典型机型 8GB RAM）

| 量 | 数值 | 备注 |
|----|------|------|
| `MAX_ORDER` | 11（默认）/ 10/9（部分内核配置） | `include/linux/mmzone.h`，对应 2^10 = 1024 页 = 4MB（MAX_ORDER=11 → 8MB 块） |
| `MIGRATE_TYPES` | 6 | 数组维度上限 |
| `free_area[]` 总条数 | 11 × 6 = 66 | 每个 zone 一组 |
| `per_cpu_pages->lists[]` | 6 条链表 | 每个 CPU × 每个 zone 一组 |
| `pcp->high` 默认 | zone->managed_pages / 32（典型 50-200） | `/proc/zoneinfo` 可查 `pcp` 字段 |
| `pcp->batch` 默认 | `pcp->high / 4`（典型 16-64） | refill/drain 单次批量 |
| `free_area[0].nr_free`（空闲 order-0 页数） | 典型 10000-50000 | 8GB 机型空闲时 |
| `free_area[10].nr_free`（空闲 order-10 块数） | 典型 0-50（高度碎片敏感） | 8GB 机型空闲时；启动后大部分为 0 |

**稳定性意义**：理解这些数量级之后，看到 `/proc/buddyinfo` 第 10 列只有个位数时，立即知道"系统在 order-10 (4MB) 级别已经出现碎片，需要 compact 或者 ZONE_MOVABLE 重映射"——这是 [11] 内存回收那篇要展开的内容。

## 第 2 章 核心分配路径：`alloc_pages` → `__alloc_pages` → `get_page_from_freelist`

### 2.1 入口层次（5 层包装 → 1 个 fast path）

GKI 5.10 上分配页帧的入口一共 5 层，最终都收敛到 `__alloc_pages()`：

```
L1: alloc_pages(gfp, order)        include/linux/gfp.h
     ↑ 最常用宏：直接按当前 task 的 NUMA 亲和

L2: alloc_pages_node(nid, gfp, order)
     ↑ 显式指定 NUMA 节点

L3: alloc_pages_nodemask(gfp, order, nodemask)  mm/page_alloc.c
     ↑ 接受 nodemask（NULL 时退化到 L1）

L4: __alloc_pages_nodemask(gfp, order, nodemask, preferred_nid)
     ↑ 真实实现，处理 cpuset/memcg/高水位 + 调用 fast/slow path

L5: __alloc_pages(gfp, order)   ← 由 __alloc_pages_nodemask 间接调用
     ↑ 老接口（保留兼容）；新代码应直接调 L4
                              ↓
                  ┌───────────┴────────────┐
                  │                        │
            fast path (常规)         slow path (兜底)
            zone_watermark_fast       __alloc_pages_slowpath
            get_page_from_freelist    wakeup_kswapd / reclaim / compact
```

**关键源码**（`mm/page_alloc.c`，GKI 5.10 真实函数名）：

```c
/* L1 — include/linux/gfp.h */
#define alloc_pages(gfp_mask, order) \
        alloc_pages_node(numa_node_id(), gfp_mask, order)

/* L3 — mm/page_alloc.c */
struct page *alloc_pages_nodemask(gfp_t gfp_mask, unsigned int order,
                                  nodemask_t *nodemask)
{
    return __alloc_pages_nodemask(gfp_mask, order,
                                  nodemask, numa_node_id());
}
EXPORT_SYMBOL(alloc_pages_nodemask);

/* L4 — mm/page_alloc.c（fast path 主体） */
struct page *__alloc_pages_nodemask(gfp_t gfp_mask, unsigned int order,
                                    nodemask_t *nodemask,
                                    int preferred_nid)
{
    struct page *page;
    /* ... cpuset / memcg / gfp_zone / alloc_flags 准备 ... */

    page = get_page_from_freelist(alloc_mask, order, alloc_flags,
                                    &ac);                    /* ← fast path */
    if (likely(page))
        goto out;

    page = __alloc_pages_slowpath(alloc_mask, order, &ac);  /* ← slow path */

out:
    return page;
}
EXPORT_SYMBOL(__alloc_pages_nodemask);
```

**关键不变量**：

- `alloc_mask` = `gfp_mask` 经 `gfp_zone` 重算后的"实际允许的 zone 标志"。
- `alloc_flags` = `ALLOC_FAIR`（默认）/ `ALLOC_HIGHATOMIC` / `ALLOC_HARDER` / `ALLOC_OOM` 等（取自 `gfp_mask` 的 `__GFP_ATOMIC` 等高位）。
- `preferred_nid` 控制 zonelist 的 fallback 顺序（`node_zonelists[]`）。
- fast path 命中立即返回；不命中才进 slow path。

### 2.2 `get_page_from_freelist` —— fast path 主体

```c
/* mm/page_alloc.c —— fast path 完整实现（GKI 5.10） */
struct page *get_page_from_freelist(gfp_t gfp_mask, unsigned int order,
                                    int alloc_flags,
                                    const struct alloc_context *ac)
{
    struct zoneref *z = ac->preferred_zoneref;
    struct zone  *zone;
    struct page  *page;

    /* 遍历 zonelist：从 preferred_nid 的 preferred zone 开始 → 跨 node fallback */
    for_next_zone_zoneref(zone, z, ac->highest_zoneidx,
                          ac->nodemask) {
        struct per_cpu_pages *pcp;
        int cold = -1;   /* -1 表示 pcp 路径未尝试 */

        /* §3 的 zone_watermark_fast 守卫；不达标直接跳过 */
        if (!zone_watermark_fast(zone, order, mark_high, highest, alloc_flags))
            continue;

        /* §6 PCP 优先：尝试从 per-CPU 热缓存拿 */
        pcp = &this_cpu_ptr(zone->pageset)->pcp;
        page = rmqueue(pcp, zone, order, gfp_flags, alloc_flags, &cold);
        if (likely(page))
            return page;

        /* PCP miss 后 fallback 到 zone->free_area 取 */
        page = __rmqueue(zone, order, gfp_mask, alloc_flags, &cold);
        if (likely(page))
            return page;
    }
    return NULL;
}
```

**fast path 三步走**：

```
Step 1: zone_watermark_fast() 守卫
        ↓ 通过
Step 2: rmqueue() 优先从 per_cpu_pages (PCP) 拉
        ↓ 命中 → return page (最快路径)
        ↓ miss
Step 3: __rmqueue() → __rmqueue_smallest() 从 free_area[] 拉
        ↓ 命中 → return page
        ↓ miss（高阶 fragmentation）
        → 继续循环尝试下一阶的 __rmqueue_smallest()
        → 最终 NULL → 返回上层走 slow path
```

### 2.3 `__rmqueue_smallest` —— 真正的"取一块"操作

```c
/* mm/page_alloc.c —— 从 free_area 取最小可用块 */
static __always_inline
struct page *__rmqueue_smallest(struct zone *zone, unsigned int order,
                                int migratetype, int cold)
{
    unsigned int current_order;
    struct free_area *area;
    struct page *page;

    /* 从请求的 order 开始向上扫，直到找到一个有 free_list 的阶 */
    for (current_order = order; current_order < MAX_ORDER; ++current_order) {
        area = &zone->free_area[current_order];
        list_for_each_entry(page, &area->free_list[migratetype], lru) {
            list_del(&page->lru);                 /* 从 freelist 摘下 */
            rmv_page_order(page);                 /* 清 page->private 的 order 标记 */
            area->nr_free--;                      /* 计数减 1<<order */
            /* 关键：可能把本块从更高阶"拆下来"了 */
            expand(zone, page, order, current_order, migratetype);
            return page;
        }
    }
    return NULL;
}
```

**关键不变量**：

- 扫描顺序：低 order → 高 order。低阶 miss 自动升级到高一阶，**以"拆大块给小需求"为代价换取分配成功率**。
- 每个 `page->lru` 链入 `free_list[migratetype]`；摘下时 `list_del`，并清掉 `page->private`（private 在 free 状态下保存 order）。
- `nr_free` 减少的是"块所包含的页数"，即 `1 << order`，不是 1。

### 2.4 `expand` —— 把高阶块拆给低阶请求

当 `__rmqueue_smallest` 在 `current_order > order` 命中时，必须把高阶块逐级拆开，剩余部分按 buddy 算法回挂到对应阶的 freelist：

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 expand() */
static void expand(struct zone *zone, struct page *page,
                   int low, int high, int migratetype)
{
    unsigned long size = 1 << high;

    /* 从 high-1 开始向下拆，每步产生两个 (high-k) 块 */
    while (high > low) {
        unsigned long pfn = page_to_pfn(page);
        struct page *buddy = page + (size >> 1);    /* 伙伴块地址 = 中点 */

        high--;
        size >>= 1;

        /* 关键 1：标记 buddy 的 order（低一位阶） */
        set_page_order(buddy, high);
        /* 关键 2：把 buddy 挂回对应 migrate type 的 freelist */
        list_add(&buddy->lru,
                 &zone->free_area[high].free_list[migratetype]);

        /* 标记 page 仍属于剩余的左半部分 */
        set_page_order(page, high);
    }
}
```

**ASCII 演示**（请求 order=2，但当前 free 的是 order=4 块）：

```
             ┌──────────────────────┐
             │  order-4 块 (16 页)   │  ← free_area[4].free_list 命中
             │  page 起始 = 0x1000   │
             └──────────┬───────────┘
                        │ expand(order=2, low=2, high=4)
                        │
            ┌───────────┴───────────┐
            ▼                       ▼
     ┌──────────────┐        ┌──────────────┐
     │  page (8 页) │        │  buddy (8 页)│  ← 挂回 free_area[3].free_list
     │  0x1000..0x8 │        │  0x1008..0x10│
     └──────┬───────┘        └──────────────┘
            │ expand 继续（高 3 → 低 2）
            │
     ┌──────┴───────┐        ┌──────────────┐
     │ page (4 页)  │        │ buddy (4 页) │  ← 挂回 free_area[2].free_list
     │ 0x1000..0x4  │        │ 0x1004..0x8  │
     └──────┬───────┘        └──────────────┘
            │ expand 结束（high=2=low）
            │
            ▼
     ┌──────────────┐
     │ 最终 page    │  ← 返回给 alloc_pages 调用者
     │ order=2 (4页)│
     └──────────────┘
```

### 2.5 `prep_new_page` —— 分配后的初始化

`get_page_from_freelist` 拿到 page 之后，调用 `prep_new_page` 做"上线前体检"：

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 */
static __always_inline bool prep_new_page(struct page *page,
                                          unsigned int order, gfp_t gfp_flags,
                                          unsigned int alloc_flags,
                                          int memcg_kmem_flags)
{
    int i;

    /* 关键检查 1：PG_reserved 残留？内核自用页有时被强制 reserved */
    for (i = 0; i < (1 << order); i++) {
        struct page *p = page + i;
        if (unlikely(PageReserved(p)))
            return false;
    }

    /* 关键检查 2：高阶页的 compound page 设置 */
    prep_compound_page(page, order);

    /* 关键检查 3：memory init-on-alloc */
    if (unlikely(init_on_alloc(gfp_flags)))
        kernel_init_free_pages(page, 1 << order, gfp_flags);

    return true;
}
```

**关键不变量**：

- 高阶页（order >= 1）会走 `prep_compound_page`，把首页的 `compound_head` / `compound_mapcount` 字段设置好。
- `init_on_alloc`（`/sys/kernel/mm/init_on_alloc`，AOSP 默认 OFF，GKI 默认 OFF）为 ON 时新页写 0x00，能缓解 use-after-free 但影响分配性能 5-15%。
- `prep_new_page` 返回 false 时 `__alloc_pages` 会重试或失败——典型是 PG_reserved 残留，发生在内存热插拔失败场景。

### 2.6 fast path 的真实耗时（数量级）

| 步骤 | 命中 PCP | 命中 free_area | 两者都 miss |
|------|---------|---------------|------------|
| `zone_watermark_fast` | ~50ns | ~50ns | ~50ns |
| `rmqueue` (PCP) | **~100ns** | — | — |
| `__rmqueue_smallest` (1 阶命中) | — | **~300ns** | — |
| `__rmqueue_smallest` (3-4 阶展开) | — | ~800ns | — |
| 全部 miss | — | — | ~1μs → 跳 slow path |

数量级参考：8GB 机型空闲时，**fast path 命中率 > 99%**，绝大部分请求在 100-300ns 内返回；只有大块连续页（order >= 5）或内存压力期才走 slow path。

## 第 3 章 水位线判断：`zone_watermark_ok` / `zone_watermark_fast`

### 3.1 为什么需要水位线

伙伴系统解决了"怎么挂链 / 怎么取块 / 怎么合并"，但没有解决"什么时候取块会被允许"。如果 zone 里 free pages 已经非常少，还允许分配 → 后续可能撞上 OOM Killer 或进程卡死。**水位线（watermark）就是 zone 上"剩余多少 free pages 才允许分配"的门槛**。

GKI 5.10 每个 zone 有 3 档水位（`include/linux/mmzone.h`）：

```c
/* include/linux/mmzone.h —— GKI 5.10 真实枚举 */
enum zone_watermarks {
    WMARK_MIN,        /* 0: 极限水位。低于此值时直接 reclaim + 可能 OOM */
    WMARK_LOW,        /* 1: 低水位。低于此值时唤醒 kswapd */
    WMARK_HIGH,       /* 2: 高水位。fast path 分配必须 ≥ 此值 */
    NR_WMARK          /* 3: 数组维度 */
};

#define min_wmark_pages(z) (z->_watermark[WMARK_MIN] + z->watermark_boost)
#define low_wmark_pages(z) (z->_watermark[WMARK_LOW] + z->watermark_boost)
#define high_wmark_pages(z) (z->_watermark[WMARK_HIGH] + z->watermark_boost)
```

**关键不变量**：

- `_watermark[]` 是**启动时**按 `zone->managed_pages` × 比例计算的（典型 MIN ≈ 0.25% × pages；LOW ≈ MIN × 1.25；HIGH ≈ MIN × 1.5）。
- `watermark_boost`（5.10 新增）由 `/proc/sys/vm/watermark_boost_factor` 决定，可在 PSI 触发时临时抬高水位，等价于"提前准备更多空闲页"。
- **三档水位的语义不同**：

| 档位 | 含义 | 触发动作 |
|------|------|--------|
| WMARK_MIN | zone 已近乎耗尽 | 同步 direct reclaim + 必要时 OOM |
| WMARK_LOW | zone 开始紧张 | **唤醒 kswapd 异步回收**（不阻塞当前分配） |
| WMARK_HIGH | zone 健康 | fast path 允许分配 |

### 3.2 `zone_watermark_ok` —— 完整守卫（slow path 用）

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 zone_watermark_ok() */
bool zone_watermark_ok(struct zone *z, unsigned int order,
                       unsigned long mark, int highest_zoneidx,
                       int alloc_flags)
{
    long min = mark;
    int oom;
    /* 关键：每个请求 order 都需要 (1 << order) 个连续 free */
    long free_pages = zone_page_state(z, NR_FREE_PAGES) - (1 << order);

    /* 考虑 ALLOC_HARDER / ALLOC_OOM / __GFP_HIGH 等"放宽水位"标记 */
    if (alloc_flags & ALLOC_HARDER)
        min -= min / 4;     /* 放宽 25%（高优先级 / cpuset 路径） */
    if (alloc_flags & ALLOC_OOM)
        min -= min / 2;     /* 放宽 50%（OOM 路径；最后一搏） */

    /* 关键：free_pages 加上 lowmem_reserve 后才与 min 比 */
    if (free_pages <= min + z->lowmem_reserve[highest_zoneidx])
        return false;

    /* 检查 order 连续：water_mark_ok 是单页数保证；order 连续保证由 free_area 配合 */
    for (o = 0; o < order; o++) {
        struct free_area *area = &z->free_area[o];
        int mt;
        for_each_migratetype_order(mt, o) {
            if (area->free_list[mt].next != &area->free_list[mt]) {
                /* 找到任意一个低阶 free_list 非空就 OK */
                return true;
            }
        }
    }
    /* 关键兜底：order 0 全空也要尽力返回 true（让上层 slow path 处理） */
    return true;
}
```

**关键行为**：

- 这是 slow path 的"完整"判断——既看 free_pages 总数，也看 low-order 是否有可拆分的块。
- `ALLOC_HARDER`（cpuset、kmemcache 高优路径）和 `ALLOC_OOM`（OOM 兜底）会**降低水位门槛**，让紧缺情况下还能分到。
- `lowmem_reserve[highest_zoneidx]` 是 [08] 讲的低端 zone 保护——保证本次分配不抢光 DMA/DMA32。

### 3.3 `zone_watermark_fast` —— fast path 用精简版

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 */
static __always_inline bool zone_watermark_fast(struct zone *z,
                                               unsigned int order,
                                               unsigned long mark,
                                               int highest_zoneidx,
                                               int alloc_flags)
{
    long free_pages;
    long min = mark;

    /* 关键 1：预读 NR_FREE_PAGES（per-CPU 计数器，无锁） */
    free_pages = zone_page_state(z, NR_FREE_PAGES);

    /* 关键 2：hot-path 优化：先按"无序 1 页"快速判断 */
    if (!order && (free_pages > min + z->lowmem_reserve[highest_zoneidx]))
        return true;       /* ← 99% 的 alloc_pages(0) 在此直接返回 */

    /* 否则走完整 zone_watermark_ok */
    return zone_watermark_ok(z, order, mark, highest_zoneidx, alloc_flags);
}
```

**性能含义**：fast path 的 watermark 判断是关键热路径。`zone_page_state(NR_FREE_PAGES)` 走 per-CPU counter，无锁，~20ns。绝大多数 order-0 分配在 `zone_watermark_fast` 第 1 行就直接返回 true。

### 3.4 watermark 触发的实际行为

```ascii
            free_pages
                 │
                 ▼
   ┌─────────────────────────────────────┐
   │  > high + lowmem_reserve           │  ← fast path 全部允许
   ├─────────────────────────────────────┤
   │  [low, high]                        │  ← fast path 允许，但 kswapd 已被唤醒
   │                                      │     （异步回收中）
   ├─────────────────────────────────────┤
   │  [min, low)                         │  ← fast path 仍允许（fast path 看 high）
   │                                      │     kswapd 持续回收中
   ├─────────────────────────────────────┤
   │  < min                               │  ← slow path 才允许（必须先 reclaim）
   └─────────────────────────────────────┘
```

**线上现象对照表**（dmesg / `/proc/zoneinfo` / `/proc/vmstat`）：

| 现象 | watermark 状态 | 内核动作 |
|------|--------------|---------|
| `pgscan_kswapd` 持续增长 | free 在 low ~ high 之间 | kswapd 异步回收，**不阻塞 alloc** |
| `pgscan_direct` 突增 | free 在 min 附近 | 同步 reclaim，alloc 线程阻塞数十毫秒 |
| `alloc_pages: order-N failure` | free < min 且 reclaim 也没救回 | warn_alloc + dump_stack（详见 §4.7） |
| `__GFP_ATOMIC` 分配仍失败 | free < min 且 ALLOC_HARDER 也救不回 | warn_alloc + OOM Killer |

### 3.5 数量级（典型 8GB 机型）

| 量 | 典型值 | 备注 |
|----|-------|------|
| ZONE_NORMAL `_watermark[WMARK_MIN]` | 16384 页 = 64MB | 来自 `init_per_zone_wmark_min()` |
| ZONE_NORMAL `_watermark[WMARK_LOW]` | 20480 页 = 80MB | MIN × 1.25 |
| ZONE_NORMAL `_watermark[WMARK_HIGH]` | 24576 页 = 96MB | MIN × 1.5 |
| ZONE_DMA `_watermark[WMARK_MIN]` | 256 页 = 1MB | DMA zone 很小 |
| `watermark_boost_factor` | 0（默认）/ 15000（PSI 触发） | `/proc/sys/vm/watermark_boost_factor` |
| `min_free_kbytes`（全局） | 16384 KB = 16MB | `/proc/sys/vm/min_free_kbytes`，由 watermark 计算反推 |

## 第 4 章 慢路径：`__alloc_pages_slowpath`

### 4.1 慢路径总览

当 fast path 的 `get_page_from_freelist` 返回 NULL（即所有候选 zone 都不满足 watermark / 取不到块）时，进入 `__alloc_pages_slowpath`：

```
                fast path 失败
                       ↓
        ┌──────────────────────────────┐
        │  __alloc_pages_slowpath        │
        │  ─────────────────────────── │
        │  1. wakeup_kswapd             │   异步回收（不阻塞当前 alloc）
        │  2. 重试 get_page_from_freelist │
        │  3. 若仍失败：                │
        │     a. try_to_free_pages       │   同步回收（直接 reclaim）
        │     b. __compact_memory        │   内存规整（页迁移）
        │     c. 重试 get_page_from_freelist │
        │  4. 若仍失败：                │
        │     a. check OOM              │   是否触发 OOM Killer
        │     b. 重试最终一次            │
        │  5. 仍失败 → warn_alloc       │   dump_stack + return NULL
        └──────────────────────────────┘
```

### 4.2 `wakeup_kswapd` —— 异步回收

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 */
void wakeup_kswapd(struct zone *zone, gfp_t gfp_flags, int order,
                   enum zone_type classzone_idx)
{
    /* 关键判断：只有当 free < LOW 时才唤醒，避免 kswapd 永远在线 */
    if (!managed_zone(zone))
        return;
    if (!zone_watermark_ok(zone, order, low_wmark_pages(zone),
                            classzone_idx, 0))
        wake_up_interruptible(&zone->zone_pgdat->kswapd_wait);
}
```

**关键不变量**：

- `wakeup_kswapd` 只把 kswapd 唤醒，**不等回收完成**——当前 alloc 线程继续走 slow path 的后续步骤。
- kswapd 在哪个 zone 上被唤醒，就只对那个 zone 做后台回收；但通过 `balance_pgdat` 可以跨 zone 协作（详见 [11]）。
- kswapd 唤醒频率受 `pgdat->kswapd_order` 控制：5.10 上默认一次回收 1 个 order，越压越紧时按 2/4/8 升级。

### 4.3 `__perform_reclaim` —— 同步回收

如果 wakeup_kswapd 之后 fast path 重试仍失败，slow path 改为**当前线程直接 reclaim**：

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 */
static unsigned long __perform_reclaim(gfp_t gfp_mask, unsigned int order,
                                        const struct alloc_context *ac)
{
    unsigned long nr_reclaimed = 0;
    unsigned int noreclaim_flag;
    /* 关键：开启 PF_MEMALLOC 让当前线程"豁免"某些限制 */
    noreclaim_flag = memalloc_noreclaim_save();

    /* 调用 vmscan.c 的 try_to_free_pages（详见 [11]） */
    nr_reclaimed = try_to_free_pages(ac->zonelist, order, gfp_mask,
                                      ac->nodemask);
    memalloc_noreclaim_restore(noreclaim_flag);
    return nr_reclaimed;
}
```

**关键不变量**：

- `try_to_free_pages` 走的是 **kswapd 的同一条路径**（`shrink_zones` → `shrink_lruvec` → `shrink_page_list`），但跑在 alloc 线程上，**阻塞当前 alloc**。
- 一次 `try_to_free_pages` 默认回收 32 个页（`SWAP_CLUSTER_MAX` × `sc.order`），典型耗时 1-10ms；高内存压力下可能跑多次。
- `PF_MEMALLOC` 标记让当前 alloc 绕过 cgroup 限制（memcg 不再 reject），但也会"霸占"回收资源——这是 direct reclaim 抖动的根因。

### 4.4 `__compact_memory` —— 内存规整

如果 reclaim 之后 fast path 仍失败，且请求是**高阶（order >= 1）** 分配，slow path 走 `__compact_memory`：

```c
/* mm/compaction.c —— GKI 5.10 真实函数名 */
static int __compact_memory(struct zone *zone, gfp_t gfp_mask,
                            int order, int alloc_flags,
                            const struct alloc_context *ac,
                            enum migrate_mode mode)
{
    /* compaction 阶段 1：isolate_migratepages 把 MOVABLE 页隔离 */
    /* compaction 阶段 2：isolate_freepages 在 zone 末尾找 free 页 */
    /* compaction 阶段 3：migrate_pages 把 MOVABLE 页搬到 free 页 */
    /* compaction 阶段 4：原来 MOVABLE 页的位置变成新的连续 free 块 */
    return compaction_suitable(zone, order, alloc_flags, mode);
}
```

**关键不变量**：

- compaction 只对 **MIGRATE_MOVABLE** 类型生效（不可移动页 kernel 自己占着，搬不动）。
- 高频 order-3/4/5 失败时通常建议查 `/proc/sys/vm/compact_memory` 主动触发 compaction，或在 ftrace 里观察 `mm_compaction_*` 事件。
- 5.10 上 `compact_memory` 默认 `proactive_compaction` 关闭，由 `vm.compaction_proactiveness` 控制（典型 0-100）。

### 4.5 慢路径的真实耗时（数量级）

| 阶段 | 典型耗时 | 阻塞当前 alloc？ |
|------|---------|-----------------|
| `wakeup_kswapd` | ~1μs（仅唤醒） | 否 |
| 重试 fast path | ~1μs | 否（仍可能命中） |
| `__perform_reclaim` | 1-30ms | **是** |
| `__compact_memory` | 10-500ms | **是** |
| `warn_alloc` 后 `dump_stack` | ~1ms | 是 |

**关键不变量**：slow path 的最大威胁是**直接 reclaim + compaction 的同步阻塞**——它会把 alloc 线程卡在直接 reclaim 的 LRU 扫描上，**这是 Android 上 Input 事件延迟 / SurfaceFlinger 卡顿的常见根因**（详见 [11] 的 §4 direct reclaim）。

### 4.6 `try_to_free_pages` 失败 / OOM 触发

当 reclaim 也救不回来时，slow path 检查是否要触发 OOM Killer：

```c
/* mm/page_alloc.c —— GKI 5.10 真实代码片段 */
if (likely(!did_some_progress)) {
    /* 关键：检查 cpuset / memcg OOM 是否允许 */
    if (oom_reaping)
        schedule_timeout_killable(1);
    /* 关键：进入 __alloc_pages_may_oom */
    page = __alloc_pages_may_oom(gfp_mask, order, ac, &did_some_progress);
    if (page)
        goto out;
}
```

**关键不变量**：

- `__alloc_pages_may_oom` 选 oom_score_adj 最大的进程杀，**不是**杀当前 alloc 线程。
- OOM 决策要 `cap_sys_admin` 才允许；受限进程会被 OOM skip。
- 杀掉后释放的页是否够救回当前 alloc？看运气——所以 OOM Killer 不保证当前 alloc 成功。

### 4.7 `warn_alloc` —— 分配失败的 dump_stack

### 4.8 慢路径上的 7 步尝试（5.10 真实顺序）

把 4.1 ~ 4.7 串起来，`__alloc_pages_slowpath` 在 GKI 5.10 上一共做 7 步：

```
Step 1: __alloc_pages_direct_compact          直接 compaction（高阶分配路径）
Step 2: wakeup_kswapd + 重试 fast path
Step 3: __alloc_pages_direct_reclaim          同步 reclaim + 重试
Step 4: __alloc_pages_may_oom                 OOM Killer 检查
Step 5: __alloc_pages_direct_compact 再试      （如果 order >= PAGE_ALLOC_COSTLY_ORDER）
Step 6: 最后一次 reclaim
Step 7: warn_alloc + return NULL              ← 失败兜底
```

每一段都被独立的 alloc_flags / `gfp_to_alloc_flags()` 控制，例如 `__GFP_RECLAIM` / `__GFP_RECLAIMABLE` / `__GFP_IO` / `__GFP_FS` 决定是否能做 IO reclaim。

## 第 5 章 释放路径：`__free_pages` → `free_one_page` → 合并伙伴

### 5.1 释放路径总览

释放与分配是对偶的，但更复杂——释放时要主动尝试合并伙伴块以减少外碎片：

```
__free_pages(page, order)
    ↓
__free_pages_ok(page, order, fpi_flags)
    ↓   ← 包含基础校验 + migratetype 决策
free_one_page(page, pfn, zone, order, fpi_flags, migratetype)
    ↓   ← 核心：尝试合并伙伴
    ├─ 检查 partner_pfn 是否也在 free_area[order] 中
    ├─ 若 partner 是 free 的：合并到高一阶 → 递归 free_one_page 高阶
    └─ 若 partner 不在 freelist：把本块挂回 free_area[order].free_list[migratetype]
```

### 5.2 `__free_pages` / `__free_pages_ok` 入口

```c
/* include/linux/gfp.h —— GKI 5.10 真实宏 */
#define __free_pages(page, order) \
        __free_pages_ok(page, order, __GFP_BITS_MASK & ~__GFP_MOVABLE)

/* mm/page_alloc.c —— GKI 5.10 真实函数名 __free_pages_ok() */
void __free_pages_ok(struct page *page, unsigned int order,
                     fpi_t fpi_flags)
{
    unsigned long combined_flags;
    int migratetype;
    unsigned long pfn = page_to_pfn(page);

    /* 关键校验 1：order 在合法范围 */
    VM_BUG_ON_PAGE(order > MAX_ORDER, page);
    /* 关键校验 2：page 状态正确（不应当是 PageReserved） */
    VM_BUG_ON_PAGE(PageReserved(page), page);

    /* 关键决策：migratetype 选择 —— 这里非常微妙 */
    migratetype = get_pfnblock_migratetype(page, pfn);    /* 默认走 pageblock 推断 */
    if (unlikely(order >= MAX_ORDER)) {
        /* 兜底：超大 order（不应该发生） */
        migratetype = MIGRATE_UNMOVABLE;
    } else {
        /* 关键：根据 pageblock 的类型判断 + __GFP_MOVABLE 标志 */
        if (is_migrate_isolate(migratetype))
            migratetype = MIGRATE_UNMOVABLE;     /* 隔离页不能挂 freelist */
        /* ... */
    }

    /* 关键：把 free 标志打上，并清 _refcount（这是 v5.10 关键不变量） */
    page->flags &= ~PAGE_FLAGS_CHECK_AT_FREE;
    set_page_private(page, order);    /* 用 private 存 order（free 状态下） */
    /* 进入真正的合并 */
    free_one_page(page, pfn, page_zone(page), order, fpi_flags, migratetype);
}
```

**关键不变量**：

- `page->private` 在 page 是 free 状态时保存 order；分配时被 `rmv_page_order` 清空。这是个典型的"复用 union 字段"技巧。
- `get_pfnblock_migratetype` 通过 `pageblock_order`（5.10 = 9，512 页 = 2MB）的位图查"这一段连续内存默认属于哪个 migrate type"——所以**即使申请方传 `__GFP_MOVABLE`，内核也会按 pageblock 范围判断**（这是 anti-frag 的核心机制）。

### 5.3 `free_one_page` —— 合并伙伴的核心

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 free_one_page() */
static void free_one_page(struct page *page, unsigned long pfn,
                          struct zone *zone, unsigned int order,
                          fpi_t fpi_flags, int migratetype)
{
    struct page *buddy;
    unsigned long buddy_pfn;
    unsigned long combined_pfn;
    int current_order;

    /* 关键：开启 PCP 路径或直接走 zone->free_area */
    if (pcp_allowed_order(order)) {
        /* order 较小（order <= PAGE_ALLOC_COSTLY_ORDER）走 pcp 优化 */
        __free_one_page(zone, page, pfn, order, migratetype, fpi_flags);
        return;
    }

    /* 高阶（order > PAGE_ALLOC_COSTLY_ORDER）走 zone->free_area */
    __free_pages_ok(page, order, fpi_flags);   /* 递归到底 */
}

static void __free_one_page(struct page *page, unsigned long pfn,
                            struct zone *zone, unsigned int order,
                            int migratetype, fpi_t fpi_flags)
{
    struct page *buddy;
    unsigned long buddy_pfn;
    int current_order;

    /* 关键循环：尝试合并到尽可能高的 order */
    while (order < MAX_ORDER - 1) {
        buddy_pfn = __find_buddy_pfn(pfn, order);    /* pfn ^ (1 << order) */
        buddy = page + (buddy_pfn - pfn);            /* 伙伴的 page 地址 */

        /* 关键检查 1：partner 必须是 free 的（PG_buddy 标记） */
        if (!PageBuddy(buddy))
            goto done_merging;

        /* 关键检查 2：partner 的 order 必须等于当前 order */
        if (buddy_order(buddy) != order)
            goto done_merging;

        /* 关键检查 3：partner 的 migrate type 必须一致 */
        if (migratetype_is_mergeable(get_pageblock_migratetype(buddy),
                                      migratetype))
            /* 可合并 */
            ;
        else
            goto done_merging;

        /* 合并：从 freelist 摘下 buddy */
        list_del(&buddy->lru);
        zone->free_area[order].nr_free--;

        /* 关键：合并后，pfn 取两者中较小者（保证地址对齐） */
        combined_pfn = buddy_pfn & pfn;
        pfn = combined_pfn;
        page = page + (combined_pfn - pfn);
        order++;       /* 升阶 */
    }

done_merging:
    /* 把合并后的块挂回 freelist */
    set_page_order(page, order);
    list_add(&page->lru, &zone->free_area[order].free_list[migratetype]);
    zone->free_area[order].nr_free++;
}
```

**关键不变量**：

- **合并只能与"特定一个伙伴"合并**——XOR 决定了它是唯一确定的；不是找"任何相邻 free 块"。
- 合并条件三连：① `PageBuddy(partner)` ② partner order == 当前 order ③ migrate type 可合并（相同 / CMA 可以跨）。
- 合并后 pfn 取两者中较小者，order +1；继续向上合并（最多到 MAX_ORDER-1）。

### 5.4 合并的 ASCII 演示

**释放场景**：先分配 order-2 块，再分配相邻的另一个 order-2 块，再依次释放。

```

状态 1：分配 order-2 块 A（0x1000_0000）
        free_area[2].nr_free = 0（已经分配）

        同上

### 5.5 `__find_buddy_pfn` —— 找伙伴的 XOR 公式

```c
/* mm/page_alloc.c —— GKI 5.10 真实内联函数 */
static inline unsigned long __find_buddy_pfn(unsigned long page_pfn,
                                             unsigned int order)
{
    /* 关键：XOR (1 << order) 把对齐位翻转，定位到物理上相邻的伙伴 */
    return page_pfn ^ (1 << order);
}
```

**关键不变量**：

- 这条公式只对 **2^k 对齐** 的块有效——伙伴系统的前提是所有块都 2^k 对齐。
- XOR 之后**严格落在另一侧**（前半段 / 后半段），保证地址上恰好相邻。

### 5.6 释放路径的常见问题

| 现象 | 根因 | 排查入口 |
|------|------|---------|
| 合并失败率高、碎片指数高 | `migrate type` 隔离过强（不同 type 不合并） | `/proc/pagetypeinfo` 看每种 type 的分布 |
| `PG_buddy` 标记残留 | `PageBuddy()` 状态没正确清 → 后续合并误判 | 看 ftrace `mm_page_free` / `mm_page_alloc` |
| 释放路径 OOPS | `__free_pages` 收到 PG_reserved 页 | `VM_BUG_ON_PAGE(PageReserved(page))` 会先 dump |
| 高阶块释放后立刻被再次拆分 | 合并成功但立即被另一线程分配走 | rare，正常 |

## 第 6 章 PCP（per-cpu pages）：批量分配 + migrate type 隔离

### 6.1 为什么需要 PCP

多核场景下，**所有 CPU 都通过 `zone->lock` 抢同一把 zone 自旋锁**，即使单次分配仅 300ns，8 核并发时锁竞争也会成为瓶颈。Linux 2.6.24 起引入 **per-CPU pageset（PCP）**：每个 CPU × 每个 zone 维护一个**私有热缓存**，单核独占使用，**不抢 zone->lock**。

```
              内存分配请求（per-CPU 视角）
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   CPU 0 的 pcp    CPU 1 的 pcp    CPU 2 的 pcp
   (zone->pageset) (zone->pageset) (zone->pageset)
        │              │              │
        └──────────────┴──────────────┘
                       │  pcp miss / pcp 满了
                       ▼
            zone->free_area[] + zone->lock  ← 全局共享
                       │
                       ▼
                   __rmqueue / rmqueue_bulk
```

### 6.2 `struct per_cpu_pages` 字段详解

承接 [§1.5](#15-关键数据结构v510-真实字段) 的结构定义，关键字段含义：

| 字段 | 含义 | 关键不变量 |
|------|------|----------|
| `count` / `free_count` | 当前 pcp 缓存总页数 | `0 <= count <= high` |
| `high` | 高水位（缓存上限） | `high_min <= high <= high_max`（5.10 新约束） |
| `batch` | refill/drain 单次批量 | `batch = high / 4`（典型 16-64） |
| `lists[MIGRATE_TYPES]` | 按 migrate type 隔离的 freelist | 6 条链表，与 zone->free_area[] 同构 |
| `expire` | 5.10 新增：上次 refill 距今代数（0=刚 refill） | 用作"是否需要 batch drain"的标志 |
| `flags` | PCP 行为 flag（如 `PCPF_ALLOW_MIGRATION`） | 5.10 新增 |
| `lock` | pcp 自旋锁（仅在统计 / 跨 CPU 操作时用） | 正常路径 lockless |

### 6.3 `rmqueue_bulk` / `__rmqueue_bulk` —— PCP refill

当 pcp `count < batch` 时，`__rmqueue_bulk` 从 `zone->free_area[]` 拉一批页进 pcp：

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 __rmqueue_bulk() */
static void __rmqueue_bulk(struct zone *zone, unsigned int order,
                           unsigned long count,
                           struct list_head *list,
                           int migratetype, unsigned long flags)
{
    unsigned long allocated = 0;
    struct page *page;

    /* 关键循环：count 次分配，全部挂入 pcp 的 list */
    while (allocated < count) {
        page = __rmqueue_smallest(zone, order, migratetype);
        if (unlikely(!page))
            break;
        /* 关键：挂入 pcp->lists[migratetype]，不直接返回 alloc 线程 */
        list_add_tail(&page->lru, list);
        allocated++;
    }
}
```

**关键不变量**：

- `__rmqueue_bulk` **不返回 page**——它把多个 page 一次性塞进 pcp 的 list，后续由 `rmqueue` 单个取。
- 单次 refill 量是 `pcp->batch`（典型 16-64）；如果 batch 个全成功，count 增 batch；中途 `__rmqueue_smallest` 返回 NULL 就提前结束。

### 6.4 `rmqueue` —— PCP + free_area 统一取页入口

```c
/* mm/page_alloc.c —— GKI 5.10 真实函数名 rmqueue() */
static inline struct page *rmqueue(struct per_cpu_pages *pcp,
                                    struct zone *zone, unsigned int order,
                                    gfp_t gfp_flags, int alloc_flags,
                                    int *cold)
{
    struct page *page;
    int migratetype = gfp_migratetype(gfp_flags);

    /* 关键 1：单页（order==0）且 pcp 有货，直接从 pcp 拉 */
    if (likely(order == 0) && !pcp_guard(pcp)) {
        page = rmqueue_pcp_list(pcp, zone, migratetype, cold);
        if (likely(page))
            return page;
    }

    /* 关键 2：order >= 1 或 pcp miss → 从 zone->free_area 拉 */
    do {
        page = __rmqueue_smallest(zone, order, migratetype);
        if (unlikely(!page)) {
            /* miss，尝试 fallback migrate type */
            page = __rmqueue(zone, order, gfp_flags, alloc_flags, cold);
            break;
        }
    } while (0);

    return page;
}
```

**关键不变量**：

- `rmqueue` 是 `rmqueue` 宏 + `__rmqueue` 内联的统一入口；`cold` 标记是否冷取（影响 LRU/统计）。
- `pcp_guard(pcp)` 5.10 新增：检查 `pcp->expire` 是否"已过期"，避免在 lockless 路径上反复 miss 后频繁 refill。

### 6.5 PCP refill / drain 时序图（核心 ASCII 图）

```
                  时间 →
   CPU 0 (PCP)        ┌──────────────────────────────────────┐
                      │                                      │
   alloc_pages(0)     │ ──1──→ rmqueue()                     │
                      │       ├─ pcp->lists[MOV] 非空         │
                      │       ├─ 取首页                       │
                      │       └─ 直接返回 alloc 线程           │
                      │                                      │
                      │ ──2──→ alloc_pages(0)                │
                              ├─ pcp->lists[MOV] 空            │
                              ├─ count < batch                │
                              ├─ 触发 pcp_refill:              │
                              │   __rmqueue_bulk(zone, 0,      │
                              │     batch, &pcp->lists[MOV])  │
                              │   ↳ 进入 zone->lock            │
                              │   ↳ __rmqueue_smallest 拉 batch│
                              │   ↳ 挂入 pcp->lists[MOV]       │
                              │   ↳ count += batch            │
                              │   ↳ exit zone->lock            │
                              └─ 然后从 pcp 取一页返回         │
                                                              │
   free_pages(page,0)  │ ──3──→ free_unref_page()            │
                      │       ├─ migratetype 推断             │
                      │       ├─ 把 page 加到 pcp->lists[MOV]│
                      │       ├─ count++                     │
                      │       └─ 若 count > high：             │
                              ↳ pcp_drain:                   │
                                ↳ 进入 zone->lock            │
                                ↳ 把 batch 个页挂回           │
                                  zone->free_area[MOV][0]    │
                                ↳ count -= batch            │
                                ↳ exit zone->lock            │
                                                              │
   batch 时间窗口后     │ ──4──→ zone_reclaim / pcp_expire   │
                      │       pcp_drain 触发（按 expire/老化） │
                      │       把多余页回收到 zone->free_area  │
                      └──────────────────────────────────────┘

   关键点：
   - 1/2/3 是 per-CPU lockless 路径（"快"）
   - 4 是 zone->lock 路径（"慢"，但只在 count 越界时）
```

### 6.6 migrate type 在 PCP 上的隔离

PCP 的 `lists[MIGRATE_TYPES]` 与 `zone->free_area[].free_list[MIGRATE_TYPES]` 同构：

- 单核分配 UNMOVABLE：先看 pcp->lists[UNMOVABLE]，再 fallback 到 zone->free_area[0].free_list[UNMOVABLE]。
- PCP 在 refill 时按调用方请求的 migrate type 拉，**不跨 type 混拉**——这就是 anti-frag 的 PCP 层面保证。
- `__rmqueue` 在 fast path 失败时按 `fallback migratetype list` 顺序回退（5.10 默认顺序：`MOVABLE → RECLAIMABLE → UNMOVABLE → HIGHATOMIC`）。

### 6.7 PCP 性能数量级

| 指标 | 典型值 |
|------|-------|
| pcp hit 时分配耗时 | **50-150ns**（无锁） |
| pcp miss → refill 一次耗时 | 1-10μs（含 zone->lock 抢占） |
| 8 核并发下 pcp 命中率 | 95-99%（典型 App 场景） |
| pcp `high` 默认值 | zone->managed_pages / 32（典型 50-200 页） |
| pcp `batch` 默认值 | `high / 4`（典型 16-64 页） |
| `/proc/zoneinfo` 中 `pcp` 字段 | 每 zone 显示当前 count + 高低水位 |

**稳定性意义**：

- 看到 `/proc/zoneinfo` 的 `pcp count` 接近 `high`，说明 pcp 缓存正常；看到 `pcp count` 长期 < `batch/2`，说明该 zone 实际很紧张。
- PCP 配置可调：`/sys/module/page_alloc/parameters/per_cpu_pct`（5.10 已 deprecated，但仍是查询入口）。

### 6.8 `page_frag_cache` / `__page_frag_cache_refill` —— 4.10+ 的"page 内碎片"优化

**问题背景**：传统的 `alloc_pages(0)` 拿一整页 4KB，但网卡 / socket / 内核线程栈很多场景只用到几十字节——剩余 99% 立即被 free 回去，又被其他 alloc 拿走，造成"page 内碎片"（同一页内多个对象共存但争用 cache line）。4.10 起 Linux 引入 `page_frag_cache` 让每个 CPU 上缓存"半页"，按字节粒度切片使用：

```c
/* include/linux/mm_types_task.h —— GKI 5.10 真实字段 */
struct page_frag_cache {
    struct page  *page;          /* 当前正在切的页（NULL 表示未 cache） */
    unsigned int  offset;        /* 下一个可切起点（按 L1_CACHE_BYTES 对齐） */
    unsigned int  size;          /* 当前页剩余可切字节数 */
    unsigned int  flags;         /* 行为标志（__GFP_xxx 等的镜像） */
};
```

**关键 API**（`mm/page_alloc.c`，GKI 5.10）：

```c
/* 拿一个 page 内碎片 —— 失败时自动调 __page_frag_cache_refill */
void *page_frag_alloc(struct page_frag_cache *nc, unsigned int fragsz,
                      gfp_t gfp_mask);

/* refill：从 zone 拉一个新页 + 重置 offset/size */
static void __page_frag_cache_refill(struct page_frag_cache *nc,
                                     gfp_t gfp_mask)
{
    struct page *page;
    int order = (PAGE_SIZE / fragsz) >= 1 ? 0 : 0;  /* 始终拿 order-0 */

    page = __alloc_pages(gfp_mask, 0);
    nc->page  = page;
    nc->offset = 0;
    nc->size  = PAGE_SIZE;     /* 整页可用 */
}

/* 释放（实际只是把 nc->page 清掉，原 page 留给后续 refill 时直接复用） */
void page_frag_free(void *addr);
```

**关键不变量**：

- `page_frag_cache` 是**单 CPU 独占**，类似 PCP 的扩展（但 PCP 按 page 计数，`page_frag_cache` 按字节计数）。
- `fragsz` 必须是 2 的幂（典型 64/128/256/512/1024/2048 字节）。
- 同一 `nc` 在 refill 后，**多个调用方共享同一页**——这意味着"page 内字节级切片"，但也意味着"分配方与释放方不在同一页"时无法用 `page_frag_free` 释放。
- `__page_frag_cache_refill` 在 page 用尽或 fragsz 超过剩余 size 时触发。
- **线上常见误用**：在高并发 socket 路径上，`page_frag_cache` 被 socket 层 `sk_page_frag()` 调用，平均每次拿 2KB；8 核并发 × 4KB page = 每秒 refill 约 8000 次，对应 8k × 4KB = 32MB/s 的 alloc_pages 流量。

**稳定性意义**：

- 在 `ftrace mm_page_alloc` 事件里频繁看到 `order:0` + 短 stack（`sk_page_frag` / `tcp_sendmsg`）→ 这就是 `page_frag_cache` 在工作。
- 看到 `__page_frag_cache_refill` 频次 > 1万/秒 → 检查上游 socket 收发是否存在"长生命周期 buffer"导致 fragsz 被压缩。
- 5.10 与 5.4 / 4.19 行为不同：5.10 起 `page_frag_cache` 默认 per-CPU 缓存命中率 > 95%，4.19 上是 per-task，命中率较低。

## 第 7 章 架构师 Takeaway：5 条伙伴系统调优经验

### 7.1 Takeaway 1 — 高阶分配（order >= 5）必查 fragmentation，不要直接调 watermark

高阶分配失败（`order >= 5` 即 32 页 ≈ 128KB 以上）几乎**全部是 fragmentation 导致**，不是 watermark 不够。调大 watermark / min_free_kbytes **不能解决 fragmentation**，反而会因为"提前保留更多空闲页"导致可用于 compaction 的页更少。

**实战做法**：

1. 先看 `/proc/buddyinfo` 第 N 列（order=N）：如果整列是 0 或个位数 → fragmentation。
2. 看 `/proc/pagetypeinfo` 中 `Unmovable` / `Reclaimable` 在每个 migrate type 下的分布——fragmentation 90% 是 **MIGRATE_UNMOVABLE 太多**导致 free_area[5..10] 被"占满"。
3. 解决手段：
   - 短期：`/proc/sys/vm/compact_memory`（写 1 触发 compaction）→ 注意这是手动操作，不要放线上 cron。
   - 中期：减少上游不可移动分配（典型是设备驱动的 `kmalloc(order=5, GFP_KERNEL)`）。
   - 长期：把高阶分配改为 `vmalloc` / 多次小分配拼接 / 用 CMA（[01] §7 演进）。

### 7.2 Takeaway 2 — `min_free_kbytes` 是全局变量，不是 zone 级，调大前必须先算比例

`/proc/sys/vm/min_free_kbytes` 是**全局**的，由它反推每个 zone 的 `_watermark[WMARK_MIN]`：

```
WMARK_MIN_zone = (zone->managed_pages × min_free_kbytes / total_managed) - safety_margin
```

调大 `min_free_kbytes` 的副作用：

- kswapd 唤醒频率变高（low 水位上升）；
- Direct reclaim 频率降低（min 水位上升）；
- 但**可用内存减少**（被 watermark 占着）；
- **更容易 fragmentation**（free area 高阶块被切成低阶）。

**经验比例**：

- 8GB RAM：`min_free_kbytes = 16384 KB`（= 16MB）默认值合理。
- 4GB RAM：`min_free_kbytes = 8192 KB`（= 8MB）。
- 16GB RAM：`min_free_kbytes = 32768 KB`（= 32MB）。
- **禁止调到 < 4MB 或 > 64MB**——前者会导致 kswapd 几乎不工作（频繁 OOM），后者会导致"空闲内存虚假高"。

### 7.3 Takeaway 3 — `MIGRATE_UNMOVABLE` 碎片化是 anti-frag 的最大敌人

[§1.5](#15-关键数据结构v510-真实字段) 提到 `MIGRATE_TYPES = 6`。实战中：

- `MIGRATE_MOVABLE` 占比 60-80% 时，anti-frag 工作良好（compaction 容易）。
- `MIGRATE_UNMOVABLE` 占比 > 30% 时，**高阶分配几乎必然 fragmentation 失败**。

**典型 UNMOVABLE 增长源**：

- 设备驱动的 `kmalloc(size, GFP_KERNEL)` 隐式变成 UNMOVABLE；
- binder 驱动的 `alloc_page(GFP_KERNEL | __GFP_ZERO)`（binder buffer 不可迁移）；
- `vm_struct` 区域（vmalloc 后的内核页）；
- THP（transparent hugepage）失败回退路径上的 unmovable 残留。

**实战做法**：

- 用 `/proc/pagetypeinfo` 看每个 zone × migrate type 的分布；
- UNMOVABLE 比例 > 30% 触发上游 review：哪些 `kmalloc` 可以改为 `kmalloc(..., GFP_KERNEL_ACCOUNT | __GFP_MOVABLE)` 强制走 MOVABLE？
- 注意：unmovable 是**默认的**，必须显式传 `__GFP_MOVABLE` 才能落到 MOVABLE。

### 7.4 Takeaway 4 — PCP 命中率是"多核分配健康度"的第一指标

`/proc/zoneinfo` 的 `pcp` 字段提供 `count` / `high` / `batch` / `free_count`：

```
$ cat /proc/zoneinfo | grep -A 5 "per CPU" | head -20
   per CPU stats:
     count      :  87
     high       :  186
     batch      :  46
     free_count :  87
     ...
```

**判定规则**：

- `count` 在 `[batch/2, high]` 区间 → 健康；
- `count` 长期 < `batch/2` → 该 zone 实际很紧，分配大量走 zone->lock 慢路径；
- `count` 长期 = `high` → pcp 缓存"装满"，可能是上游大量 free 导致，可能伴随 fragmentation。

**8 核 / 6GB 机型典型值**：

- `count` ≈ 50-150；
- `high` ≈ 100-200；
- `batch` ≈ 25-50；
- `count/high` 比例 ≈ 30%-80%。

### 7.5 Takeaway 5 — `__alloc_pages_slowpath` 走几步可以判断问题类型

每次 `alloc_pages` 失败时，slow path 走的"步数"反映了问题严重度。实战中按以下规则对照：

| 走到第几步 | 问题类型 | 排查方向 |
|-----------|---------|---------|
| 第 1 步（compact）就失败 | fragmentation 严重（高阶分配） | `/proc/pagetypeinfo` 看 UNMOVABLE 比例 |
| 第 2-3 步（reclaim）成功 | 短暂内存压力 | `pgscan_direct` 增量，看是否周期性 |
| 第 4 步（OOM 触发） | 真 OOM 或 cgroup 限额 | `/proc/meminfo` 总量 + `cgroup.event` |
| 第 7 步（warn_alloc） | 系统已无能为力 | dump_stack 必看 |

**关键不变量**：

- `pgscan_kswapd` 增长 → 异步回收（不阻塞 alloc）；
- `pgscan_direct` 增长 → 同步回收（阻塞 alloc，**这是 STW 卡顿的真凶**）；
- `allocstall` 计数器增长 → 至少有 1 次走到 reclaim；
- `compact_fail` 增长 → 走到 compaction 仍失败。

## 总结

**本篇核心**：页分配器是 Linux 内核 mm/ 子系统的"调度器"——它把"任何 order 的物理页请求"翻译成 zone free_area 的 binary buddy 算法操作，同时通过 watermark / PCP / migrate type 三套机制做并发控制、压力管理、抗碎片化。

**5 条 Takeaway 浓缩**：

1. **高阶失败必查 fragmentation**，不要直接调 watermark。
2. **`min_free_kbytes` 是全局且影响 compaction**，4MB-64MB 区间外调通常有害。
3. **MIGRATE_UNMOVABLE 比例 < 30%** 是 anti-frag 的硬性健康线。
4. **PCP 命中率**（count / high 比例）是多核分配健康度的第一指标。
5. **slow path 步数**对应不同问题类型：1 = fragmentation、3 = 短暂压力、4 = OOM、7 = 系统无能为力。

**承接下一篇**：[10-SLAB,SLUB 分配器与小对象分配(GKI 5.10)](10-SLAB,SLUB 分配器与小对象分配(GKI 5.10).md) 将解决"kmalloc(8 字节) 不能直接调 alloc_pages(0)"的问题——SLAB/SLUB 在 buddy 之上建一层 per-cpu / per-node 的对象缓存，把"小对象分配"从 buddy 的 100-300ns 降到 30-80ns，同时提供 KASAN / KFENCE 的越界 / UAF 检测能力。

## 篇尾衔接

**本篇核心**：伙伴系统是 Linux 内核 mm/ 子系统的"调度中枢"——它把"任何 order 的物理页请求"翻译成 zone free_area 的 binary buddy 操作，并通过 watermark / PCP / migrate type 三套机制做并发控制、压力管理、抗碎片化。掌握 fast path 与 slow path 的分界、watermark 三档水位的语义、PCP 的 refill/drain 时序、migrate type 的合并规则，是稳定性架构师排查"分配失败"问题的硬门槛。

**下一篇**：[10-SLAB,SLUB 分配器与小对象分配(GKI 5.10)](10-SLAB,SLUB 分配器与小对象分配(GKI 5.10).md) 将解决"kmalloc(8 字节) 不能直接调 alloc_pages"的问题——SLAB/SLUB 在 buddy 之上建一层 per-cpu / per-node 的对象缓存，把"小对象分配"从 buddy 的 100-300ns 降到 30-80ns，同时提供 KASAN / KFENCE 的越界 / UAF 检测能力。

**系列尾预告**：[12-内存稳定性风险全景](12-内存稳定性风险全景.md) 将整合 01-11 给出五大类稳定性问题（OOM / 泄漏 / 抖动 / 杀进程 / 系统卡顿）的风险地图与速查表，让架构师在线上救火时 5 分钟内定位到层与子系统。
