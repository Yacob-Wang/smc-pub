## 第 1 章 五大类问题分类

### 1.1 顶层分类与占比

从线上稳定性故障归因样本看（典型 OEM 内部统计），内存类问题在所有稳定性故障中占比约 **35-45%**，且具有"症状相似、根因多样、跨层传递"的特征。

```
                ┌───────────────────────────────────────────────┐
                │         内存稳定性故障（占稳定性 35-45%）       │
                └───────────────────┬───────────────────────────┘
                                    │
        ┌───────────┬───────────────┼───────────────┬───────────┐
        │           │               │               │           │
     [OOM]      [泄漏]          [抖动]         [杀进程]      [卡顿]
    20-25%      15-20%         20-25%         20-25%       10-15%
        │           │               │               │           │
   Java OOM    Java 堆         GC 抖动       LMKD 误杀    主线程 STW
   Native OOM  Native 堆      Direct Reclaim  OOM Killer   Input 延迟
   系统 OOM    资源泄漏       swap 抖动       Watchdog    Surface 分配
   杀触发 OOM  fd/线程/Win     锁竞争抖动     Kernel OOM   锁竞争
```

> **稳定性架构师视角**：五大类之间有**级联关系**——Java 堆泄漏会触发 GC 抖动，GC 抖动会触发 Direct Reclaim，Direct Reclaim 触发 LMKD，LMKD 杀错关键进程又触发 ANR/卡顿。**看到现象 A，要往下追 2-3 层根因**。

### 1.2 五大类问题的一句话定义

| 大类 | 一句话定义 | 核心特征 | 量化阈值（经验） |
|------|----------|---------|---------------|
| **OOM** | 进程或系统在分配内存时获得"无内存"错误 | 抛 OutOfMemoryError、收到 SIGKILL、ANR `lowmemory`、dumpsys 出现 `Killed by LMK/OOM` | Java 堆使用率 > 90% 持续 5s、Native PSS > 上限、可用内存 < 4% |
| **泄漏** | 内存分配后无法被 GC/回收/释放，长期累积导致可用内存单调下降 | dumpsys 多次采样 PSS 单调增长、fd 数 / 线程数 / Window 数 持续增长 | Java 堆增长 > 20MB/h、fd 数 > 1024、线程数 > 500、Window 数 > 100 |
| **抖动** | 内存分配/回收的"次秒级波动"导致延迟尖刺 | 周期性的 STW、Direct Reclaim 耗时 > 100ms、swap I/O 突增、kswapd 占用 CPU > 30% | STW > 50ms、Direct Reclaim > 200ms、PSI `some` > 20% |
| **杀进程** | 因内存压力被 LMKD/OOM Killer/Watchdog 强制结束 | logcat `Killed by LMK`、`Out of memory: Killed process`、`ANR in` 关联 `lowmemory` | 30s 内杀进程 > 3 个、关键进程 adj 异常 |
| **卡顿** | 内存压力传导到主线程调度、Input 分发、Surface 分配，表现为 UI 卡顿 | 帧率 < 50fps、TTID/TTFD 退化、Input 事件延迟 > 100ms | 主线程 STW > 16ms、PSI `full` > 5% |

### 1.3 五大类问题的层归属

| 大类 | App 层 | ART 层 | Framework 层 | 内核层 | 硬件层 |
|------|-------|-------|------------|-------|-------|
| OOM | ✓ | ✓ | ✓ | ✓ | — |
| 泄漏 | ✓ | ✓ | ✓ | ✓ | — |
| 抖动 | — | ✓ | ✓ | ✓ | ✓ |
| 杀进程 | — | — | ✓ | ✓ | — |
| 卡顿 | — | ✓ | ✓ | ✓ | ✓ |

> **稳定性架构师视角**：**OOM 和泄漏跨所有层**，**杀进程集中在 Framework + 内核**，**卡顿集中在 ART + Framework + 内核**。先看现象落在哪一层，再下钻到具体子类型。

## 第 3 章 OOM 分类

OOM 是"申请不到内存"的统称，但根因差异巨大。**同样报 OOM，Java 堆满、Native 堆满、Address Space 满、cgroup 满、内核满的治理方案完全不同**。

### 3.1 Java OOM

#### 3.1.1 是什么 / 为什么

Java 堆（ART 堆）被填满，无法分配新对象时抛 `OutOfMemoryError`。ART 堆分 Young/Old/Zygote/Region，详见 [03-ART 堆内存与 GC 全景](03-ART 堆内存与 GC 全景.md)。

**关键判定**：`dumpsys meminfo <pkg>` 中 `Java Heap` 一行：
- `Heap Size`：当前堆大小（弹性伸缩）
- `Heap Alloc`：已分配字节
- `Heap Free`：剩余可用

`Heap Alloc / Heap Size > 0.95` 持续 5s 以上 + logcat `OutOfMemoryError` + 堆栈含 `Java heap space` → 确认 Java OOM。

#### 3.1.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | App Crash、ANR（Foreground）、后续请求失败 |
| **影响** | 用户级崩溃；后台 OOM 可能触发 ANR |
| **日志关键字** | `OutOfMemoryError`、`Java heap space`、`Failed allocating <N> bytes`、`GC overhead limit exceeded` |
| **dumpsys 特征** | `Java Heap Heap Alloc / Heap Size > 0.95`；`Views`/`Activities` 暴涨；`Object[]` 占用异常 |
| **proc 特征** | `/proc/<pid>/status` VmRSS Java 段增长 |
| **源码入口** | `art/runtime/gc/heap.cc:Heap::ThrowOutOfMemoryError` |
| **排查入口** | hprof + MAT / LeakCanary / `am dumpheap` |
| **跨篇链接** | [03-ART 堆内存与 GC 全景 §7](03-ART 堆内存与 GC 全景.md) |

#### 3.1.3 子类型细分

```
Java OOM
├── 一次性分配过大
│   ├── 现象：Bitmap/byte[]/byte[] 数 MB 一次性分配
│   ├── 关键代码：BitmapFactory.decodeStream 加载大图
│   └── 治理：BitmapFactory.Options.inSampleSize、BitmapRegionDecoder
├── 累积分配无释放（堆泄漏）
│   ├── 现象：长时间使用后 PSS 单调增长
│   ├── 关键代码：单例持有 Context、Static 集合未清理
│   └── 治理：LeakCanary + WeakReference + onTrimMemory
├── 分代溢出
│   ├── 现象：Young GC 频繁但 Old 区不收敛
│   ├── 关键代码：长生命周期对象误用分代
│   └── 治理：profile + Reference 调整
├── HUMONGOUS 对象
│   ├── 现象：大于 Region（AOSP 14 默认 32KB，可通过 `-XX:RegionSize` 调整）的对象
│   ├── 关键代码：large list/array
│   └── 治理：拆分 / 分块
├── 线程栈内分配大对象
│   └── 关键代码：Java 线程栈内分配数组（少见）
├── Finalizer/Reference 队列死锁
│   └── 关键代码：finalize() 慢、Reference 队列积压
└── 触发后 ART 报错的"伪 OOM"
    ├── GC overhead limit exceeded：98% 时间在 GC、回收 < 2% 堆
    └── 治理：避免大内存 + 弱引用
```

#### 3.1.4 稳定性架构师视角

Java OOM 的根因 80% 是"分配合理但释放路径错"或"单次分配过大"。**真正的"堆不够大"反而是少数**——业务方加 -Xmx 是个临时方案。架构师要追问：
1. 堆里什么东西涨了？→ hprof 找支配树
2. 这些对象是谁在持有？→ GC Roots
3. 持有路径合理吗？→ 是否经过长生命周期引用
4. 有没有更轻的实现？→ SparseArray 替 HashMap、ProtoBuf 替 JSON

详见 [03-ART 堆内存与 GC 全景](03-ART 堆内存与 GC 全景.md) 第 7 章风险地图。

### 3.2 Native OOM

#### 3.2.1 是什么 / 为什么

Native 堆（malloc 家族管理）被填满或分配失败。Android Native 内存使用 bionic `scudo` 分配器（Android 11+），详见 [04-Native 堆内存与分配器（AOSP 14）](04-Native 堆内存与分配器（AOSP 14）.md)。

**关键判定**：`dumpsys meminfo <pkg>` 中 `Native Heap` 一行：
- `Native Heap`：Native 堆总分配
- `Native Heap / Native Pss` 比例高（> 0.8）→ 大部分是真占用
- 比例低 → 大量被 swap/page cache 回收

**典型场景**：Bitmap 分配在 Native（API 26+ 的 `NativeAllocationRegistry`），glide/coil 等图片库默认在 Native 分配。Native 堆未配上限，可能吃满整个 Address Space。

#### 3.2.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | App Crash（`art_sigsegv`、`SIGSEGV`）、tombstone 含 `malloc/free`、Bitmap decode 失败 |
| **影响** | 用户级崩溃；Native 分配是 OOM 的"看不见的角落" |
| **日志关键字** | `art_sigsegv`、`Native allocation`、`tcmalloc`、`scudo`、`abort: heap corruption`、`double free or corruption` |
| **dumpsys 特征** | `Native Heap` 持续增长；`Native Heap / Pss Total` 比例高；`Graphics` 一栏异常大 |
| **proc 特征** | `/proc/<pid>/status` VmRSS 整体增长但 Java 堆未满 |
| **源码入口** | `bionic/libc/bionic/malloc.cpp:__libc_malloc` → `bionic/libc/bionic/scudo/` |
| **排查入口** | `dumpsys meminfo -d` + `malloc_debug` 模式（userdebug）+ `address sanitizer` |
| **跨篇链接** | [04-Native 堆内存与分配器（AOSP 14）](04-Native 堆内存与分配器（AOSP 14）.md) |

#### 3.2.3 子类型细分

```
Native OOM
├── malloc 返回 NULL
│   ├── 一次性大分配（> 256MB）触发 scudo/内核拒绝
│   └── 治理：拆分、mmap 替代 malloc
├── Bitmap 泄漏
│   ├── 现象：Native Heap 涨，Java 堆没动
│   ├── 关键代码：Bitmap.recycle() 漏调；Glide/Coil 缓存配置不当
│   └── 治理：recycle + 缓存上限 + image library 配置
├── JNI 引用泄漏
│   ├── 现象：JNI global reference 涨
│   ├── 关键代码：NewGlobalRef 后未 DeleteGlobalRef
│   └── 治理：Reference 封装 + LeakCanary JNI 模块
├── ByteBuffer.allocateDirect
│   ├── 现象：Direct buffer 不被 Java GC 回收
│   ├── 关键代码：allocateDirect 大 DirectByteBuffer
│   └── 治理：Cleaner / 显式释放
├── 图形缓冲区（ION/DMA-BUF）
│   ├── 现象：ION heap 用尽
│   ├── 关键代码：Surface/ImageReader 持有未 release
│   └── 治理：close() + SurfaceView destroy
├── 第三方 Native 库泄漏
│   ├── WebView (Chromium)、音视频编解码、3D 引擎
│   └── 治理：库版本升级 + LeakCanary NDK
└── Address Space 满
    ├── 现象：32 位进程虚拟地址空间用尽（4GB / 3GB 用户态）
    ├── 关键代码：大量 mmap、共享库过多
    └── 治理：64 位迁移 + VMA 合并
```

#### 3.2.4 稳定性架构师视角

Native OOM 是"最难查"的 OOM：
1. **没有像 Java 堆那么好的工具**——hprof 不覆盖 Native
2. **`dumpsys meminfo` 的 Native 数字是估值**——PSS 是按比例摊销 page cache
3. **malloc_debug 有性能损耗**——线上不能开
4. **很多泄漏在第三方库**——WebView、Chromium Native

**架构师的核心建议**：
- App 启动时 dump 一次 Native baseline，运行时监控差值
- 关键 Native 资源（Bitmap、Surface、AudioRecord）必须有 recycle/close 路径
- 优先排查"持续增长"的项，阶跃性增长通常是业务正常请求

详见 [04-Native 堆内存与分配器（AOSP 14）](04-Native 堆内存与分配器（AOSP 14）.md) 第 6 章风险地图。

### 3.3 系统 OOM

#### 3.3.1 是什么 / 为什么

整机可用内存耗尽（`/proc/meminfo` MemAvailable < 5%），内核回收路径已尽力（Direct Reclaim 失败、kswapd 无果），最终触发 OOM Killer。**Android 场景下**：通常是 LMKD 先杀，但 LMKD 没杀干净（min_score_adj 阈值过宽）就会落到内核 OOM Killer。

**关键判定**：
- `dumpsys meminfo` Total PSS 接近 Total RAM
- `/proc/meminfo` MemAvailable 接近 0
- `/proc/pressure/memory` `full` avg10 > 80%
- logcat 同时出现 `Killed by LMK` 和 `Out of memory: Killed process`

#### 3.3.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | 多个进程短时间内连环被杀；前台 App 频繁闪退；系统"无响应" |
| **影响** | 整机型故障；最严重的稳定性问题 |
| **日志关键字** | `lowmemorykiller: Killing`、`low memory on global`、`Out of memory: Killed process`、`page allocation failure: order:N` |
| **dumpsys 特征** | `dumpsys meminfo` 整机 PSS 接近 RAM；`/proc/meminfo` MemAvailable < 5% |
| **proc 特征** | `/proc/pressure/memory` `full` 持续 100%；`/proc/vmstat` `pgscan_*` 暴增 |
| **源码入口** | `mm/oom_kill.c:out_of_memory` → `select_bad_process` → `oom_kill_process` |
| **排查入口** | `dumpsys lmkd` + `dumpsys meminfo` + PSI 历史 |
| **跨篇链接** | [06-LMKD 用户态内存杀手](06-LMKD 用户态内存杀手.md) [11-内存回收](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md) |

#### 3.3.3 子类型细分

```
系统 OOM
├── 前台 App 占用过大
│   ├── 现象：单个 App 占用 1-2GB+
│   ├── 关键代码：低内存设备（4GB RAM）跑大 App
│   └── 治理：低内存模式 + App 内存瘦身
├── 后台 App 大量缓存
│   ├── 现象：cached 进程 PSS 累计 > 系统 RAM
│   ├── 关键代码：AMS cached 列表过长
│   └── 治理：trimMemory + 进程数上限
├── LMKD 配置过宽
│   ├── 现象：LMKD min_score_adj 阈值过高
│   ├── 关键代码：/vendor/etc/lmkd.conf
│   └── 治理：调低阈值（详见 06 篇）
├── 启动阶段内存不足
│   ├── 现象：开机后 30s 内大量杀进程
│   ├── 关键代码：Zygote 启动慢 + 多 App 同时启动
│   └── 治理：启动优化（详见 Window 08 篇）
├── 内存泄漏累积到系统级
│   └── 治理：top-app 内存基线 + 长期监控
└── 内核 OOM Killer 误杀
    ├── 现象：adj 不该被杀但被内核 OOM Killer 干掉
    ├── 关键代码：oom_score_adj_min_threshold 配置错误
    └── 治理：oom_score_adj 配置 + 进程白名单
```

#### 3.3.4 稳定性架构师视角

系统 OOM 通常是"最后一根稻草"——前面四类问题累积到整机扛不住。**架构师要建"水位线告警"**：在 MemAvailable 下降到 20% 时就告警，下降到 10% 时启动自动治理（清理缓存、降低 cached 进程数）。

详见 [06-LMKD 用户态内存杀手](06-LMKD 用户态内存杀手.md) 第 5 章风险地图。

### 3.4 杀进程触发 OOM

#### 3.4.1 是什么 / 为什么

**反向因果**：进程 A 内存压力触发 LMKD，LMKD 杀了进程 B（B 是无辜的，但 adj 略高），B 死了之后，A 仍持续 OOM（因为 A 自己有泄漏），最后 A 也被杀。**表象是"杀进程"，根因是"杀错了"或"没解决问题"**。

**关键判定**：
- 短时间内（30s）logcat 出现 3+ 个 `Killed by LMK`
- 被杀进程不一定是 PSS 最大的
- 杀进程后系统内存并没有恢复

#### 3.4.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | 短时间内多个进程被标记 kill；用户体验"切回 App 变冷启动" |
| **影响** | 业务级故障（消息推送、后台服务）；用户感知强 |
| **日志关键字** | `Killed by LMK`（连续多条）、`Sent SIGKILL to <pid> (adj <score>)` |
| **dumpsys 特征** | `dumpsys lmkd` 显示 kill 事件密集；`dumpsys meminfo` 显示被杀进程消失但总 PSS 仍高 |
| **proc 特征** | `/proc/<pid>` 在杀进程后消失（logcat 中可见） |
| **源码入口** | `system/memory/lmkd/lmkd.c:cmd_kill` |
| **排查入口** | `dumpsys lmkd` + logcat 关联 + `procstats` |
| **跨篇链接** | [05-AMS 内存治理与进程优先级](05-AMS 内存治理与进程优先级.md) [06-LMKD 用户态内存杀手](06-LMKD 用户态内存杀手.md) |

详见 [第 6 章杀进程分类](#第-6-章-杀进程分类)。

## 第 4 章 泄漏分类

### 4.1 Java 堆泄漏

#### 4.1.1 是什么 / 为什么

分配的对象应该被 GC 回收但实际没被回收，长期累积导致 PSS 单调增长。**根因**：从 GC Roots 到对象的引用路径未断开。

**关键判定**：连续 3 次以上 `dumpsys meminfo` 间隔 5 分钟，Java Heap Heap Alloc 单调增长且不收敛。

#### 4.1.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | App 使用越久越卡；最终触发 Java OOM |
| **影响** | 业务级 PSS 异常；用户感知"App 越来越慢" |
| **日志关键字** | `finalize` 频繁、`ReferenceQueue` 满、LeakCanary 报告 |
| **dumpsys 特征** | `Java Heap` 增长；`Views` / `Activities` / `AppContexts` 持续增长；`Object[]` 占用异常 |
| **proc 特征** | `/proc/<pid>/status` VmRSS Java 段增长 |
| **源码入口** | `art/runtime/gc/heap.cc` + `art/runtime/gc/collector/` |
| **排查入口** | hprof + MAT/LeakCanary + `am dumpheap <pkg>` |
| **跨篇链接** | [03-ART 堆内存与 GC 全景 §7](03-ART 堆内存与 GC 全景.md) |

#### 4.1.3 子类型细分

```
Java 堆泄漏
├── Activity/Fragment 泄漏
│   ├── 现象：旋转屏幕、关闭 Activity 后 GC Roots 仍可达
│   ├── 关键代码：内部类持有外部 Activity 引用；Handler 持有 Activity
│   └── 治理：WeakReference + lifecycle 监听
├── 单例/SystemService 持有 Context
│   ├── 关键代码：单例 init 时传入 Activity Context
│   └── 治理：ApplicationContext
├── 静态集合未清理
│   ├── 关键代码：static List/Map 缓存对象
│   └── 治理：LruCache + 容量上限
├── Listener/Callback 注销失败
│   ├── 关键代码：registerReceiver/onClick 未 unregister
│   └── 治理：Lifecycle 感知 + onDestroy 清理
├── 线程/Handler 持有 Activity
│   ├── 关键代码：内部 Runnable 持有外部 Activity
│   └── 治理：static Runnable + WeakReference
├── WebView 泄漏
│   ├── 关键代码：WebView 持有 Activity Context
│   └── 治理：独立进程 + destroy() 调用
├── 资源未关闭
│   ├── Cursor/Stream/Bitmap
│   └── 治理：try-with-resources / use {}
└── Reference 链异常
    ├── SoftReference/WeakReference 用错
    └── 治理：熟悉四种引用语义
```

详见 [03-ART 堆内存与 GC 全景](03-ART 堆内存与 GC 全景.md) 第 7 章。

### 4.2 Native 堆泄漏

#### 4.2.1 是什么 / 为什么

malloc 分配的内存在 free 后未真正归还 scudo/内核，或者 free 路径缺失。**根因**：C/C++ 代码未配对 malloc/free，或第三方 Native 库内部泄漏。

**关键判定**：`dumpsys meminfo <pkg>` 中 `Native Heap` 持续增长，但 `Native Heap` 没有被 page cache 回收迹象（`Native Heap / Pss` 比例高且稳定）。

#### 4.2.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | Native Heap 涨、最终触发 Native OOM |
| **影响** | 排查困难（hprof 不覆盖）；用户感知"App 越来越重" |
| **日志关键字** | `malloc debug`、`scudo` quarantine 满、`ASAN report` |
| **dumpsys 特征** | `Native Heap` 单调增长；`Native Heap / Pss` 比例 > 0.8 持续 |
| **proc 特征** | `/proc/<pid>/status` VmRSS 整体增长（Native 段） |
| **源码入口** | `bionic/libc/bionic/scudo/` |
| **排查入口** | `malloc_debug` + LeakCanary NDK + `am send-trim-memory` |
| **跨篇链接** | [04-Native 堆内存与分配器（AOSP 14）](04-Native 堆内存与分配器（AOSP 14）.md) |

#### 4.2.3 子类型细分

```
Native 堆泄漏
├── Bitmap Native 分配未 recycle
│   ├── 现象：Native Heap 涨但 Java 堆没动
│   ├── 关键代码：Bitmap.recycle() 漏调
│   └── 治理：bitmap 库配置 + 显式 recycle
├── JNI 引用未释放
│   ├── NewGlobalRef 未 DeleteGlobalRef
│   └── 治理：JNI 引用封装
├── Native 库内部泄漏
│   ├── WebView、Chromium、音视频、3D 引擎
│   └── 治理：库版本 + 升级
├── malloc 大块内存未 free
│   ├── 关键代码：循环中 malloc 累积
│   └── 治理：tcmalloc 报告 + 代码 review
├── 内存映射（mmap）未 munmap
│   ├── 现象：/proc/<pid>/maps 中 VMA 持续增长
│   └── 治理：配合 munmap
└── ION/DMA-BUF 泄漏
    ├── 现象：ION heap 用尽；SurfaceView 持有
    └── 治理：close() 路径
```

### 4.3 资源泄漏（fd / 线程 / Window）

#### 4.3.1 是什么 / 为什么

除了内存字节，"系统资源"也会泄漏。Linux 进程有 fd 上限（默认 1024，可调到 1M+）、线程数限制（`/proc/<pid>/limits`）。Android 还有 Window 数（Activity 管理）。

**关键判定**：
- fd：`ls /proc/<pid>/fd | wc -l` 持续增长超过 1024
- 线程：`cat /proc/<pid>/status` Threads 持续增长超过 500
- Window：logcat `WindowLeaked` 或 `dumpsys window` 中窗口数 vs Activity 数

#### 4.3.2 现象、影响、关键字、特征、入口

| **proc 特征** | `cat /proc/<pid>/limits` No file | ulimit 触顶 | — |
| **源码入口** | `fs/file.c` | `kernel/fork.c` | `frameworks/base/services/core/java/com/android/server/wm/WindowManagerService.java` |
| **跨篇链接** | [01 §1](01-内存系统总览：从进程视角到硬件的完整链路.md) | [01 §1](01-内存系统总览：从进程视角到硬件的完整链路.md) | [02 §7](02-进程内存地图与 VMA 体系.md) |

### 4.4 泄漏治理的"四个统一"原则

> **稳定性架构师视角**：资源泄漏治理的**通用方法论**——"四个统一"：

1. **统一回收路径**：每个 `open`/`alloc`/`register` 都必须有一个对应的 `close`/`free`/`unregister`，且在 `onDestroy`/`finalize` 中兜底
2. **统一上限**：每个容器/缓存/池都要有上限（LruCache、bounded queue、连接池）
3. **统一监控**：上线前 dumpsys baseline，运行时监控 PSS、fd、Threads 差值
4. **统一降级**：达到上限时的降级策略——LRU 淘汰、拒绝新请求、清空缓存

## 第 5 章 抖动分类

抖动 = "次秒级波动"。OOM 和泄漏是长期累积，抖动是短时间尖刺，**最容易让用户感知"卡顿"**。

### 5.1 GC 抖动

#### 5.1.1 是什么 / 为什么

ART GC 的 STW（Stop-The-World）阶段让所有 Java 线程暂停。**CC（Concurrent Copying）算法**在 5.10+ Android 上是默认，STW 目标 < 5ms，但在内存压力下会退化。

**关键判定**：
- `dumpsys gfxinfo <pkg>` 中 `Janky frames > 5%`
- logcat `Concurrent copying paused` 或 `WaitForGcToComplete`
- `dumpsys meminfo <pkg>` 中 `GC` 时间占比 > 10%

#### 5.1.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | UI 卡顿、滚动掉帧、动画不流畅 |
| **影响** | 帧率退化、用户感知明显 |
| **日志关键字** | `Concurrent copying paused`、`WaitForGcToComplete`、`Background concurrent`、`Foreground` GC |
| **dumpsys 特征** | `dumpsys gfxinfo` Janky frames > 5%；art `ConcurrentMarkSweep` pause time |
| **proc 特征** | `/proc/<pid>/status` VmRSS 短期波动 |
| **源码入口** | `art/runtime/gc/collector/concurrent_copying.cc` |
| **排查入口** | `dumpsys gfxinfo` + `am send-trim-memory` + LeakCanary |
| **跨篇链接** | [03-ART 堆内存与 GC 全景](03-ART 堆内存与 GC 全景.md) |

#### 5.1.3 子类型细分

```
GC 抖动
├── Full GC 频繁
│   ├── 现象：1 分钟内多次 `Background concurrent`
│   ├── 关键代码：堆配置不当 + 内存压力
│   └── 治理：调大堆 / 减少长生命周期对象
├── Concurrent 失败退到 STW
│   ├── 现象：logcat `Concurrent copying paused: <N> ms`
│   ├── 关键代码：分配速率超过回收速率
│   └── 治理：减少短生命周期分配
├── 软引用/弱引用回收风暴
│   ├── 现象：GC 频繁但堆未释放
│   ├── 关键代码：弱引用持有大对象
│   └── 治理：少用软引用
├── Reference 队列清理阻塞
│   └── 治理：避免在 finalize() 阻塞
├── Finalizer 死锁
│   ├── 现象：GC 长时间不返回
│   ├── 关键代码：finalize() 持有锁
│   └── 治理：禁用 finalize
├── 大对象（HUMONGOUS）触发 Ref/RefToSpace
│   └── 治理：拆分 / 复用
└── image space 损坏
    └── 治理：清除 + 重启
```

详见 [03-ART 堆内存与 GC 全景](03-ART 堆内存与 GC 全景.md) 第 7 章。

### 5.2 Direct Reclaim 抖动

#### 5.2.1 是什么 / 为什么

内核在慢路径上**同步回收**（详见 [11-内存回收](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md)）。分配线程被阻塞直到回收完成。**这是最常见的"卡顿"来源**——任何分配都可能被 Direct Reclaim 阻塞 100ms-1s。

**关键判定**：
- `/proc/vmstat` 中 `pgscan_direct` / `pgsteal_direct` 突增
- ftrace `mm_vmscan_direct_reclaim_begin/end` 事件持续时间长
- 应用主线程被 `__alloc_pages_slowpath` 阻塞

#### 5.2.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | 任意分配阻塞；Binder 通信延迟；主线程卡顿 |
| **影响** | 应用卡顿（不限 Java）；跨进程通信延迟 |
| **日志关键字** | `__alloc_pages_slowpath`、`direct reclaim`、`shrink_inactive_list`、`compact_zone` |
| **dumpsys 特征** | `/proc/vmstat` `pgscan_direct` 突增 |
| **proc 特征** | `/proc/<pid>/stack` 显示 `shrink_inactive_list` |
| **源码入口** | `mm/page_alloc.c:__alloc_pages_slowpath` → `mm/vmscan.c:__perform_reclaim` |
| **排查入口** | ftrace + `/proc/vmstat` + perfetto |
| **跨篇链接** | [11-内存回收 §3](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md) [09-页分配器与伙伴系统(GKI 5.10)(GKI 5.10)(GKI 5.10)](09-页分配器与伙伴系统(GKI 5.10)(GKI 5.10)(GKI 5.10)(GKI 5.10).md) |

#### 5.2.3 子类型细分

```
Direct Reclaim 抖动
├── 高阶分配（order > 3）
│   ├── 现象：16MB / 32MB 连续页申请
│   ├── 关键代码：camera、视频编解码、gralloc
│   └── 治理：CMA / 预留内存池
├── 文件页回收慢
│   ├── 现象：大量 dirty page 回写耗时
│   └── 治理：调整 vm.dirty_*
├── 匿名页 swap 慢
│   ├── 现象：zRAM 写满后落到磁盘
│   └── 治理：调大 zRAM / 减少匿名页
├── 锁竞争
│   ├── 现象：lru_lock 持有时间长
│   └── 治理：memcg 分组隔离
├── compaction 耗时
│   ├── 现象：logcat `compact_zone`
│   └── 治理：避免高阶分配
└── memcg 限制触发
    ├── 现象：foreground cgroup 达到 limit
    └── 治理：limit 调整 / PSI 监控
```

详见 [11-内存回收](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md) 第 9 章风险地图。

### 5.3 swap 抖动

#### 5.3.1 是什么 / 为什么

匿名页被换出（swap out）后再次访问（swap in），I/O 抖动。Android 默认使用 zRAM（压缩内存到 RAM 中），当 zRAM 满时回退到磁盘 swap（如果有配置）。

**关键判定**：
- `/proc/vmstat` `pswpin` / `pswpout` 持续增长
- `/proc/meminfo` `SwapTotal - SwapFree` 持续减少（说明有 swap out）
- IO 监控 `zram0` I/O 突增

#### 5.3.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | App 切换回前台时卡顿（cold start 退化）；Zygote 派生新进程慢 |
| **影响** | 业务级卡顿；内存压力持续 |
| **日志关键字** | `swap_writepage`、`zram0: I/O error`（少见）、`out_of_memory` |
| **dumpsys 特征** | `/proc/vmstat` swap 事件突增 |
| **proc 特征** | `/proc/meminfo` SwapUsed 增长 |
| **源码入口** | `mm/swap.c`、`mm/swapfile.c` |
| **排查入口** | `/proc/vmstat` + `iostat` + perfetto |
| **跨篇链接** | [11-内存回收 §5](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md) |

#### 5.3.3 子类型细分

```
swap 抖动
├── zRAM 满
│   ├── 现象：zRAM 压缩率下降
│   └── 治理：调大 zRAM 大小
├── swap 设备 I/O 慢
│   ├── 现象：磁盘 swap（极少配置）
│   └── 治理：禁用磁盘 swap，仅用 zRAM
├── 冷热页反复 swap
│   ├── 现象：refault 风暴
│   ├── 关键代码：active_anon / inactive_anon 调优不当
│   └── 治理：调整 swappiness（Android 默认 100）
├── swap 抖动引发整体 I/O 抖动
│   └── 治理：I/O 调度器 + 内存压力
└── 匿名页太多（Java 堆）
    └── 治理：堆配置 + 业务瘦身
```

详见 [11-内存回收](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md) 第 9 章。

### 5.4 锁竞争抖动

#### 5.4.1 是什么 / 为什么

内核或 ART 内部锁的持有者长时间不释放，其他等待者阻塞。**锁竞争在内存压力下会被放大**——Direct Reclaim 时持有 `lru_lock`，所有分配线程等待。

**关键判定**：
- ftrace `sched_blocked_reason` 显示等待锁
- `/proc/<pid>/stack` 显示在锁等待栈
- Perfetto track 看到调度空隙

#### 5.4.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | 任意线程阻塞；锁等待者长尾 |
| **影响** | 系统级卡顿；不仅是内存问题 |
| **日志关键字** | `sched_blocked_reason`、kernel 锁警告 |
| **dumpsys 特征** | `dumpsys cpuinfo` 显示锁等待 |
| **proc 特征** | `/proc/<pid>/wchan` 显示锁名 |
| **源码入口** | `kernel/locking/`、`include/linux/lockdep.h` |
| **排查入口** | perfetto + ftrace + `cat /proc/<pid>/wchan` |
| **跨篇链接** | [10-SLAB/SLUB 分配器](10-SLAB,SLUB 分配器与小对象分配(GKI 5.10).md) [Window 10-WMS 锁竞争](../01-Mechanism/Framework/Window/10-WMS锁竞争与Watchdog.md) |

#### 5.4.3 子类型细分

```
锁竞争抖动
├── lru_lock 持有
│   ├── 现象：Direct Reclaim 持有 lru_lock
│   ├── 关键代码：shrink_inactive_list 持锁
│   └── 治理：memcg 隔离
├── zone->lock 持有
│   ├── 现象：pcp 路径竞争
│   └── 治理：per-CPU 缓存
├── alloc_mutex 持有
│   ├── 现象：OOM 路径
│   └── 治理：避免 OOM
├── ART 内部锁
│   ├── mutator_lock
│   └── 治理：减少 GC 触发
├── WMS 锁（与 Window 体系交叉）
│   ├── 现象：WindowManagerService.mWindowMap 锁
│   └── 治理：见 Window 10
└── AMS 锁
    ├── 现象：Activity 启动时持锁
    └── 治理：见 Window 系列
```

详见 [10-SLAB/SLUB 分配器与小对象](10-SLAB,SLUB 分配器与小对象分配(GKI 5.10).md) 第 6 章 + [11-内存回收](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md) 第 9 章。

## 第 6 章 杀进程分类

杀进程是"被动发生"的——某个组件判定需要杀，进程就被 SIGKILL。**判定错位是最大的根因**。

### 6.1 LMKD 误杀

#### 6.1.1 是什么 / 为什么

LMKD（LowMemoryKiller Daemon）根据 PSI / memcg 压力选择进程杀掉。**误杀**指：
- 杀错 adj（应杀 cached 杀了 service）
- 杀太频繁（min_score_adj 阈值过宽）
- 杀得太慢（PSI 持续 30s+ 才响应）

详见 [06-LMKD 用户态内存杀手](06-LMKD 用户态内存杀手.md)。

#### 6.1.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | 关键进程消失；后台服务中断；用户感知"消息推送丢失" |
| **影响** | 业务级故障；用户感知强 |
| **日志关键字** | `lmkd: Killing`、`Sent SIGKILL to <pid> (adj <score>)`、`lowmemorykiller` |
| **dumpsys 特征** | `dumpsys lmkd` 显示 kill 事件；`dumpsys meminfo` 显示被杀进程消失 |
| **proc 特征** | `/proc/<pid>/oom_score_adj` 查看 adj |
| **源码入口** | `system/memory/lmkd/lmkd.c`、`system/memory/lmkd/event.cpp` |
| **排查入口** | `dumpsys lmkd` + logcat 关联 |
| **跨篇链接** | [06-LMKD 用户态内存杀手](06-LMKD 用户态内存杀手.md) [05-AMS 内存治理与进程优先级](05-AMS 内存治理与进程优先级.md) |

#### 6.1.3 子类型细分

```
LMKD 误杀
├── 杀错 adj
│   ├── 现象：service 被杀，cached 仍在
│   ├── 关键代码：adj 计算错误（详见 05 篇）
│   └── 治理：adj 重新计算 + 监控
├── 阈值过宽
│   ├── 现象：min_score_adj 阈值过高
│   ├── 关键代码：lmkd.conf / Property 配置
│   └── 治理：调低阈值
├── PSI 配置错误
│   ├── 现象：PSI 持续 5s+ 仍未触发
│   ├── 关键代码：/proc/pressure/memory 阈值
│   └── 治理：调整 PSI 阈值
├── 关键进程保护缺失
│   ├── 现象：Persist 进程被杀
│   ├── 关键代码：oom_score_adj 错误
│   └── 治理：调整 adj 范围
└── 频繁杀进程
    ├── 现象：30s 内 5+ 个进程被杀
    └── 治理：调高阈值 + 治理泄漏源
```

### 6.2 OOM Killer 杀

#### 6.2.1 是什么 / 为什么

内核 OOM Killer（`mm/oom_kill.c`）在 LMKD 未杀干净、kernel 内存仍无法满足时触发。**与 LMKD 区别**：内核 OOM Killer 是"最后手段"，是低效的杀手。

**关键判定**：
- logcat `Out of memory: Killed process <pid> (<name>) total-vm:<N>, anon-rss:<M>, file-rss:<K>`
- 该日志由 `oom_kill.c:oom_kill_process` 输出
- 进程被 SIGKILL 但无 `Killed by LMK` 前缀

#### 6.2.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | 关键进程被强制结束；系统级故障 |
| **影响** | 业务级故障 + 系统不稳定 |
| **日志关键字** | `Out of memory: Killed process`、`oom-kill:constraint=CONSTRAINT_MEMCG` |
| **dumpsys 特征** | `dumpsys meminfo` 整机 PSS 接近 RAM |
| **proc 特征** | `/proc/sys/vm/oom_kill_allocating_task` 决定是否杀当前 |
| **源码入口** | `mm/oom_kill.c:out_of_memory` → `select_bad_process` |
| **排查入口** | `dmesg` + logcat + `/proc/vmstat` |
| **跨篇链接** | [08-物理内存组织](08-物理内存组织-Node,Zone,Page,memblock(GKI 5.10).md) [11-内存回收](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md) |

#### 6.2.3 子类型细分

```
OOM Killer 杀
├── oom_score_adj 配置过低
│   ├── 现象：关键进程被内核选中
│   ├── 关键代码：adj 范围 [-1000, 1000]
│   └── 治理：调整 adj
├── memcg limit 过严
│   ├── 现象：memcg OOM kill
│   └── 治理：调整 cgroup limit
├── LMKD 没杀干净
│   ├── 现象：内核 OOM Killer 兜底
│   └── 治理：优化 LMKD 阈值
├── total-vm 估算错误
│   ├── 现象：oversized PSS 的进程被选
│   └── 治理：oom_score_adj 调整
└── cgroup 嵌套导致误判
    └── 治理：cgroup 层级简化
```

### 6.3 Watchdog 杀

#### 6.3.1 是什么 / 为什么

Android Watchdog 监控 system_server 等关键线程，5s+ 阻塞判定为 ANR，最终可能杀进程或重启 system_server。**虽然不是纯内存问题，但内存压力（Direct Reclaim / GC 抖动）会触发 Watchdog**。

**关键判定**：
- logcat `ANR in <process>` + `subject: <reason>` + 包含 `lowmemory` 或 `WaitForGcToComplete`
- `data/anr/traces.txt` 中显示主线程被 GC 阻塞

#### 6.3.2 现象、影响、关键字、特征、入口

| 维度 | 内容 |
|------|------|
| **现象** | ANR 弹窗；system_server 重启；系统级卡顿 |
| **影响** | 业务级 ANR；系统级故障 |
| **日志关键字** | `ANR in`、`Input event injection timed out`、`Not responding`、`Watchdog` |
| **dumpsys 特征** | `dumpsys activity activities` 中 `mLastANR` 时间戳 |
| **proc 特征** | `data/anr/traces.txt` |
| **源码入口** | `frameworks/base/services/core/java/com/android/server/Watchdog.java` |
| **排查入口** | traces.txt + `dumpsys activity` + perfetto |
| **跨篇链接** | [Window 10-WMS 锁竞争与 Watchdog](../01-Mechanism/Framework/Window/10-WMS锁竞争与Watchdog.md) [07-PSI 压力传递](07-PSI、vmpressure、memcg 压力传递.md) |

#### 6.3.3 子类型细分

```
Watchdog 杀
├── 主线程 GC 阻塞
│   ├── 现象：traces.txt 显示 `Waiting for GC`
│   └── 治理：减少 Java 堆压力
├── 主线程 Direct Reclaim
│   ├── 现象：traces.txt 显示 `__alloc_pages_slowpath`
│   └── 治理：减少高阶分配
├── Binder 通信阻塞
│   ├── 现象：traces.txt 显示 `Binder Proxy`
│   └── 治理：减少同步 Binder
├── Input 事件分发阻塞
│   ├── 现象：`Input event injection timed out`
│   └── 治理：主线程耗时操作治理
├── Service Binder 阻塞
│   └── 治理：服务拆异步
└── system_server 健康监控
    └── 治理：见 Window 10
```

### 6.4 杀进程的反向思维：什么时候**不**应该被杀

> **稳定性架构师视角**：杀进程是手段不是目的。**判断杀进程是否合理的三问**：
> 1. 杀完后内存恢复了吗？→ 没恢复说明没杀对（杀小了）
> 2. 杀的是不是 cached 进程？→ 杀 service 是反模式
> 3. 杀的频率合理吗？→ 30s 杀 3+ 个说明阈值过宽

详见 [05-AMS 内存治理与进程优先级](05-AMS 内存治理与进程优先级.md) + [06-LMKD 用户态内存杀手](06-LMKD 用户态内存杀手.md)。

## 第 7 章 排查流程图（5 分钟内定位）

### 7.1 通用内存问题排查流程

```
现象（用户投诉 / 监控告警 / ANR）
    ↓
【30 秒】第 2 章速查表 - 确认大类（OOM/泄漏/抖动/杀进程/卡顿）
    ↓
【1 分钟】日志关键字（5 条 shell 命令）
    ├── 看到 OutOfMemoryError → Java OOM
    ├── 看到 Killed by LMK → 杀进程
    ├── 看到 ANR in + lowmemory → 卡顿/杀进程
    ├── 看到 GC overhead limit → Java OOM
    └── 看到 direct reclaim → 抖动
    ↓
【2 分钟】dumpsys meminfo 采样
    ├── Native Heap 涨 → Native 泄漏/OOM
    ├── Java Heap 涨 → Java 泄漏/OOM
    ├── fd/Threads 涨 → 资源泄漏
    ├── Total PSS 接近 RAM → 系统 OOM
    └── 单个进程消失 → 杀进程
    ↓
【5 分钟】/proc 关键指标
    ├── /proc/pressure/memory → PSI 压力
    ├── /proc/vmstat → 回收事件
    ├── /proc/meminfo → 整机内存
    └── /proc/<pid>/oom_score_adj → adj 配置
    ↓
【10-30 分钟】深入定位
    ├── hprof + MAT（Java）
    ├── malloc_debug（Native）
    ├── ftrace + perfetto（内核）
    └── dmesg（OOM killer）
    ↓
跨篇链接 → 找到根因 + 治理方案
```

### 7.2 OOM 排查流程

```
OOM 报错
    ↓
logcat 堆栈定位
    ├── Java heap space → Java OOM
    │   ↓
    │   dumpsys meminfo Java Heap 涨？
    │   ├── 是 → hprof + MAT 找支配树
    │   │   ↓
    │   │   找到泄漏 → 修复 + 监控
    │   └── 否 → 业务一次性大分配 → 拆分/分块
    │
    ├── Native allocation → Native OOM
    │   ↓
    │   dumpsys meminfo Native Heap 涨？
    │   ├── 是 → malloc_debug + LeakCanary NDK
    │   └── 否 → 一次性大分配（Bitmap/Buffer）→ 拆分
    │
    └── Killed by ... → 杀进程触发
        ↓
        dumpsys lmkd 看 kill 事件
        ├── Killed by LMK → LMKD 配置或 adj 异常
        └── Out of memory: Killed → 内核 OOM Killer
            ↓
            /proc/meminfo MemAvailable
            ├── 接近 0 → 系统 OOM → 整机治理
            └── 还有 → 进程内 OOM → 进程治理

```
怀疑泄漏（PSS 单调增长）

连续 3 次 dumpsys meminfo（间隔 5min）
    ├── Java Heap 涨 → Java 堆泄漏
    │   ↓
    │   hprof 采样 + MAT 支配树
    │   ├── Activity/Fragment 泄漏 → WeakReference
    │   ├── 单例持 Context → ApplicationContext
    │   └── 静态集合未清 → LruCache 上限
    │
    ├── Native Heap 涨 → Native 堆泄漏
    │   ↓
    │   malloc_debug（userdebug）
    │   ├── Bitmap 未 recycle → 显式 recycle
    │   ├── JNI 引用 → DeleteGlobalRef
    │   └── 第三方库 → 升级
    │
    ├── fd 涨 → fd 泄漏
    │   ↓
    │   ls /proc/<pid>/fd | wc -l
    │   ├── socket 池泄漏 → 调连接池
    │   ├── 文件未 close → try-with-resources

    │
    ├── Threads 涨 → 线程泄漏

    └── Window 涨 → Window 泄漏

        logcat WindowLeaked

        └── Toast 用 Activity Context → ApplicationContext
```

```
卡顿（帧率 < 50fps / TTID 退化 / ANR）

    │   │   ↓

    │   │

## 第 9 章 5 分钟内定位内存问题的方法论

> **稳定性架构师视角**：5 分钟看似不可能，但 80% 的内存问题遵循以下 3-5 条规律：

### Takeaway 1：**先分类，再下钻**——"四分半"决策

```
0:00 - 0:30  监控告警 / 用户投诉 → 大类（OOM/泄漏/抖动/杀进程/卡顿）
0:30 - 1:30  关键字匹配（logcat + dumpsys）
1:30 - 3:00  采样对比（dumpsys meminfo 连续 3 次）
3:00 - 4:30  /proc 关键指标（PSI / vmstat / meminfo）
4:30 - 5:00  跨篇链接 → 找到根因机制
```

**核心动作**：**永远不要在没分类前看 hprof**。Java 堆 / Native 堆 / 系统层的 hprof 工具链完全不同。

### Takeaway 2：**连续采样比单次 dump 更有效**

单次 dumpsys meminfo 只能告诉"现在是什么"，**连续 3-5 次间隔 5 分钟**才能告诉"在怎么变"。架构师要建**采样基线**：
- 启动时 1 次（baseline）
- 运行中每 5 分钟 1 次（runtime）
- 关键操作（打开页面/退出）前后 1 次

### Takeaway 3：**先看进程级，再看系统级**

> "**App 自己有问题，看 App；整机都卡，看系统。**"

**反向思维**：很多架构师看到系统卡顿就去看 PSI、vmstat，但其实 80% 的"系统卡顿"是某个 App 占用过高。先 `dumpsys meminfo` 排序看 Top 5 进程，能省一半时间。

### Takeaway 4：**OOM 的 80/20 法则**

- **80% 的 Java OOM** 是泄漏 → hprof + MAT 找支配树
- **80% 的 Native OOM** 是 Bitmap → recycle + 缓存配置
- **80% 的系统 OOM** 是 cached 进程过多 → trimMemory + LMKD 阈值
- **80% 的抖动** 是 Direct Reclaim → 减少高阶分配

剩下 20% 才是真正复杂的情况（image space 损坏、SLAB 泄漏、cgroup 嵌套误判）。

### Takeaway 5：**杀进程的合理性三问**

1. **杀完后内存恢复了吗？** 没恢复 = 没杀对
2. **杀的是不是 cached 进程？** 杀 service = 反模式
3. **杀的频率合理吗？** 30s 杀 3+ = 阈值过宽

### Takeaway 6（锦上添花）：**建监控而不是救火**

**架构师的天花板**不是"5 分钟内定位"，而是"5 分钟内定位后**建监控**让下次不再发生**"。每解决一个线上问题，问自己：
- 这个 PSS / 句柄 / 线程的"正常范围"是什么？
- 监控上有没有对应告警？
- 治理后是否回滚到基线？

## 第 10 章 总结

### 10.1 核心观点回顾

1. **五大类问题（OOM/泄漏/抖动/杀进程/卡顿）**覆盖线上 90% 内存稳定性问题
2. **速查表（第 2 章）是 5 分钟定位的核心**——关键字 + dumpsys 特征 + 排查入口
3. **OOM 内部差异巨大**——Java OOM / Native OOM / 系统 OOM 治理完全不同
4. **泄漏要分内存 vs 资源**——Java 堆 / Native 堆 / fd / 线程 / Window
5. **抖动的根因在内核**——Direct Reclaim / swap / 锁竞争 三大类
6. **杀进程要看合理性**——LMKD / OOM Killer / Watchdog 各自的责任边界
7. **跨篇引用**——本篇是 01-11 各篇的"风险地图汇总"

### 10.2 架构师视角的金句

> **"看到现象 A，要往下追 2-3 层根因。"** ——五大类问题之间有级联关系，Java 堆泄漏 → GC 抖动 → Direct Reclaim → LMKD 杀进程。

> **"Android 内存不是单一系统，是五层分布式系统。"** ——不懂全栈就无法定位根因。

> **"不要在没分类前看 hprof。"** ——Java 堆 / Native 堆 / 系统层的工具链完全不同。

> **"80/20 法则：80% 的问题遵循 4 个根因，剩下 20% 才是真正复杂的。"**

> **"杀进程是手段不是目的。"** ——杀完内存没恢复就是没杀对。

### 10.3 本篇的"未完成项"

本篇只覆盖**风险分类和识别**，**具体工具用法**留给 [13-内存稳定性诊断工具链](13-内存稳定性诊断工具链.md)：
- dumpsys meminfo 字段全解
- procrank / procmem 实战
- PSI / memcg 监控体系
- Perfetto + ftrace 跟踪模板
- 监控体系设计（阈值、告警、根因定位）
