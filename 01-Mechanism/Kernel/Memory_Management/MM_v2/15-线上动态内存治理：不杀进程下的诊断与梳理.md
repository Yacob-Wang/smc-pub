## 0. 写在最前:为什么需要"不杀进程"治理

### 0.1 生产环境的 4 大约束

**"杀进程" 在生产环境不总是合适的**。前 14 篇讲的"通过 adj / LMKD / OOM killer 杀进程" 是**终极方案**,但**有 4 类场景**你必须用"不杀进程" 治理:

| 约束 | 场景 | 杀进程代价 |
|------|------|----------|
| **① 不能杀** | 用户正在用的 app(adj < 0);系统关键服务(system_server / phone / systemui) | 杀 = 用户感知崩溃/系统卡顿 |
| **② 杀了也复现** | 内存增长是**间歇性**的(bug 触发条件复杂),杀了下次还会复现 | 杀 = 治标不治本,下次还要查 |
| **③ 多个进程** | 5+ 个 app 同时增长,不知道哪个是"主犯" | 杀 = 把所有候选都杀了 |
| **④ 治理时间窗** | 治理团队需要**长时间持续观察**(7×24h),不能"杀一下看效果" | 杀 = 中断观察链 |

**结论**:**生产环境的内存治理,80% 时间是"不杀进程下识别 + 隔离 + 动态梳理"**,只有 20% 是"杀进程治理"(LMKD/AMK/force-stop)。

### 0.2 跟"杀进程治理"的本质差异

| 维度 | 杀进程治理(05/06/12) | **不杀进程治理(本篇)** |
|------|---------------------|---------------------|
| **核心目标** | 快速腾内存 | 持续观察 + 精准识别 |
| **决策速度** | ms 级(直接发 SIGKILL) | 分钟-小时级(等数据)|
| **数据来源** | adj 数值 + PSI 触发 | PSS 趋势 + baseline diff + heapprofd |
| **副作用** | 用户感知崩溃(可能性) | 几乎无 |
| **诊断精度** | 中(知道"哪类进程"被杀) | 高(知道"哪行代码泄漏")|
| **持续时间** | 单次动作 | 7×24h 持续 |
| **工具性能损耗** | 0(杀是瞬时) | 0-5%(取决于工具) |

## 1. FWK 层如何标识记录内存:ProcessRecord 内存视图

> **本节核心问题**:**FWK 层用什么字段记录一个进程的"内存状态"?线上怎么读这些字段?**
>
> **本节是 v2 系列的补强**——05 篇讲了 adj / 进程优先级,但**没讲 ProcessRecord 怎么记录内存状态**。本节补齐这块。

### 1.1 ProcessRecord 5 个内存维度

`frameworks/base/services/core/java/com/android/server/am/ProcessRecord.java` 里有 5 个字段**直接记录"内存状态"**(不是内存数字本身,而是**与内存治理相关的元数据**):

| 字段 | 类型 | 含义 | 线上读法 | 治理价值 |
|------|------|------|---------|---------|
| **`lastTrimMemoryLevel`** | `int` | 上次 trimMemory 调用的 level | `dumpsys activity processes \| grep -A2 trimMemory` | 看 app 收到了哪些 trim 请求 |
| **`lastPssTime`** | `long` | 上次 dumpsys 记录 PSS 的时间 | `dumpsys activity processes \| grep -A1 lastPss` | PSS 采样间隔(默认 60s)|
| **`lastPss`** | `long` | 上次采样的 PSS(单位 KB) | 同上 | PSS 趋势基线 |
| **`lastRequestedGc`** | `long` | 上次主动触发 GC 的时间 | `dumpsys activity processes \| grep lastRequestedGc` | 主动 GC 频率(可能反映 OOM 压力) |
| **`lastLowMemory`** | `long` | 上次标记为低内存的时间 | `dumpsys activity processes \| grep lastLowMemory` | 内存压力告警的时序 |

**完整 dumpsys activity processes 输出片段**(简化):

```text
ProcessRecord{3f4a8c 1500:com.example.im/u0a100}
  userId=10010
  curAdj=900
  setAdj=900
  curProcState=PROCESS_STATE_CACHED_ACTIVITY   ← ★ 内存治理触发态
  lastTrimMemoryLevel=MODERATE                ← ★ 关键字段
  lastPss=245760                                ← ★ 240 MB
  lastPssTime=2026-07-15 14:23:45             ← ★ 1 min 前采样
  lastRequestedGc=2026-07-15 14:21:30         ← ★ 2 min 前主动 GC
  lastLowMemory=2026-07-15 14:20:00          ← ★ 4 min 前低内存告警
  ...
```

**线上排查典型用法**:

```bash
# 1. 找最近 1 小时被 trimMemory 调用过 >= 3 次的进程(内存压力证据)
adb shell dumpsys activity processes | \
  awk '/ProcessRecord/{p=$0} /lastTrimMemoryLevel/{print p, $0}'

# 2. 找 lastPss > 500MB 且 lastPssTime 在最近 1 min 的进程(疑似泄漏)
adb shell dumpsys activity processes | \
  awk '/lastPss=/{split($2,a,"="); pss=a[2]/1024; if(pss>500) print $0}'

# 3. 找 lastRequestedGc 频率 > 5 次/小时(被频繁触发 GC,可能 OOM 边缘)
adb shell dumpsys activity processes | \
  grep "lastRequestedGc" | sort | uniq -c
```

### 1.2 ActivityRecord / ServiceRecord 内存记账

ProcessRecord 是**进程级**,但**每个进程内的 Activity / Service 也有自己的内存记账字段**:

**`ActivityRecord.java` 中**:

| 字段 | 含义 | 治理价值 |
|------|------|---------|
| `mLastReportedConfiguration` | Activity 配置(屏幕/方向) | 配置变化触发内存重分配 |
| `mIcicle` | onSaveInstanceState 的 Bundle 内存 | Activity 重建时的内存快照 |
| `thumbnail` | Activity 缩略图(用 mThumbnailFile) | 缩略图内存占用 |

**`ServiceRecord.java` 中**:

| 字段 | 含义 | 治理价值 |
|------|------|---------|
| `lastActivity` | Service 最后活跃时间 | 识别"僵尸 Service" |
| `startRequested` | 是否 startService() | startService = 持锁,可能不释放 |
| `connections` | 当前 binder 连接数 | 识别泄漏的连接 |
| `pendingConnectionGroup` | 待处理连接组 | 积压 = 内存风险 |

**线上排查用法**(放在本节是为了**说明 FWK 层内存记账的全貌**):

```bash
# 4. 找所有 connections > 5 的 Service(可能是泄漏)
adb shell dumpsys activity services | \
  awk '/ServiceRecord/{p=$0} /connections=/{n=$2; gsub("=","",n); if(n+0>5) print p, $0}'

# 5. 找 lastActivity > 24h 的 Service(僵尸)
adb shell dumpsys activity services | grep "lastActivity" | \
  awk '{...时间判断...}'
```

### 1.3 dumpsys meminfo 怎么把"ProcessRecord 5 维 + 内存详情"缝起来

`dumpsys meminfo` 是**进程级**的内存快照,但**和 ProcessRecord 5 维是不同维度**:

```
ProcessRecord 5 维     ←  元数据(状态/时间/触发)
       ↓
       与
       ↓
dumpsys meminfo 内存详情 ← 数字(PSS/RSS/Heap)
       ↓
       关联
       ↓
procstats 长期趋势   ← 历史(小时/天/周)
```

**完整 dumpsys meminfo 关键字段**(本节只列**线上常用**的,完整字段看 13 §5.1):

```
TOTAL PSS: 240 MB           ← ★ 进程 PSS 总和
  Native Heap:   80 MB      ← ★ C/C++ 堆(04 篇)
  Java Heap:    100 MB      ← ★ ART Java 堆(03 篇)
  Code:          20 MB      ← ★ dex + so
  Stack:         5 MB       ← ★ 线程栈
  Graphics:      15 MB      ← ★ 图形缓冲(04 §5)
  Private Other:  3 MB      ← ★ 私有内存
  System:        17 MB      ← ★ 系统级(TLS 等)

Heap:        100 MB / 256 MB  ← ★ 已用 / 堆大小
Heap Alloc:   80 MB           ← ★ 已分配
Heap Free:   176 MB           ← ★ 空闲

Views:        1500            ← ★ View 数量(界面复杂度)
Activities:      3            ← ★ Activity 数量
AppContexts:     5            ← ★ Context 数量(可能泄漏)
Assets:          8            ← ★ AssetManager 数量
```

**关键关联**:当 `dumpsys meminfo` 的 `Java Heap: 100MB` 突然上涨,**同时** `ProcessRecord.lastPss` 也在涨,**且** `lastTrimMemoryLevel` 从 `RUNNING_MODERATE` 升到 `COMPLETE`——**这就是典型 Java 堆泄漏信号**。

### 1.4 dumpsys procstats:PSS 历史数据(可用于"不杀进程"做趋势分析)

`dumpsys procstats` 是**system_server 持有的 `ProcessStatsService`** 维护的**长期 PSS 历史**:

**关键命令**:

```bash
# 1. 进程级 PSS 历史(最近 24h)
adb shell dumpsys procstats --hours 24 | head -100

# 2. 单进程详细 PSS 历史
adb shell dumpsys procstats com.example.im

# 3. 内存状态时间线(配合 grep 找增长)
adb shell dumpsys procstats com.example.im | \
  awk '/^\s*Sample/{print}'
```

**输出格式**(简化):

```
Process com.example.im / u0a100:
  ProcessSummary:
    State: PERSISTENT  ★
    TotalRSS:    180 MB
    TotalPss:    120 MB
    TotalSwapPss:  5 MB

  Last 24 hours PSS samples:
    Sample at 2026-07-15 14:00:00: PSS=120 MB
    Sample at 2026-07-15 15:00:00: PSS=125 MB    ← +5 MB/h
    Sample at 2026-07-15 16:00:00: PSS=140 MB    ← +15 MB/h  ← ★ 加速!
    Sample at 2026-07-15 17:00:00: PSS=160 MB    ← +20 MB/h  ← ★ 确认泄漏
```

**线上治理价值**:
- **不杀进程,只看趋势**——`dumpsys procstats` 跑一遍,**1 分钟看到 24h PSS 曲线**
- 配合 5 个 ProcessRecord 字段,**可以做"不杀进程下定位泄漏模块"**

### 1.5 ProcessStatsService 的内存账本机制

`frameworks/base/services/core/java/com/android/server/am/ProcessStatsService.java` 维护**所有进程的 PSS 长期账本**:

| 内存机制 | 源码位置 | 数据保留时长 |
|---------|---------|------------|
| `mProcessStats` (SparseArray) | ProcessStatsService.java | 24h(默认) |
| `mLastReportMemUsage` (long) | ProcessStatsService.java | 24h |
| `mProcessStatsCounter` (int) | ProcessStatsService.java | 进程整个生命周期 |
| `mLastRss` (long) | ProcessStatsService.java | 24h |
| `mMemFactor` (enum) | ProcessStatsService.java | 进程整个生命周期 |
| `mStartTime` (long) | ProcessRecord.java | 进程整个生命周期 |

**关键概念——`mMemFactor`**:

```java
// ProcessStatsService.java 中的 memFactor
public enum MemFactor {
    UNKNOWN,        // 未知
    NORMAL,          // 正常
    MODERATE,        // 内存中度压力
    LOW,             // 内存低压
    CRITICAL,        // 内存临界
    NOT_RESPONDING   // 进程不响应(ANR 边缘)
}
```

**线上读法**:

```bash
# 看所有进程的 memFactor
adb shell dumpsys procstats | grep -A1 "MemFactor"
```

**关键观察**:`mMemFactor` 从 `NORMAL` 升到 `MODERATE` 通常意味着**多个进程同时在涨**——这是 cgroup 级别压力的信号,**不是单进程问题**。

## 2. 线上诊断:不杀进程下识别内存问题

> **本节核心问题**:**生产环境能开什么工具?哪些不能开?开了之后如何解读?**

### 2.1 受限诊断矩阵(对比 13 的完整工具)

13 §5-§8 讲了完整工具集(perfetto、ftrace、simpleperf、heapprofd 等),但**没区分"线上可用 vs 不能用"**。本节给出**线上受限诊断矩阵**:

| 工具 | 13 篇位置 | 性能损耗 | 生产环境**推荐** | 备注 |
|------|---------|---------|-------------------|------|
| `dumpsys meminfo` | §5.1 | < 0.1% | ✅ **必用** | 零损耗,关键工具 |
| `dumpsys procstats` | §5.2 | < 0.1% | ✅ **必用** | 长期趋势,看增长 |
| `dumpsys activity processes` | §5.7 | < 0.1% | ✅ **必用** | ProcessRecord 5 维 |
| `cat /proc/<pid>/smaps_rollup` | §3.6 | < 0.1% | ✅ **必用** | 进程级 PSS 汇总 |
| `cat /proc/<pid>/status` | §3.7 | < 0.1% | ✅ **必用** | 进程基础状态 |
| `cat /proc/meminfo` | §3.1 | < 0.1% | ✅ **必用** | 系统级 |
| `cat /proc/vmstat` | §3.2 | < 0.1% | ✅ **必用** | 内核事件 |
| `cat /proc/pressure/memory` | §3.9 | < 0.1% | ✅ **必用** | PSI 压力 |
| `cat /proc/cgroups` | — | < 0.1% | ✅ **必用** | cgroup 视图 |
| `cat /proc/<pid>/cgroup` | — | < 0.1% | ✅ **必用** | 进程 cgroup 归属 |
| `cat /proc/lmkd/debug` | §4.2 | < 0.1% | ✅ **必用** | lmkd 事件 |
| **perfetto heap_profile** | §8.5 | **5-10%** | ⚠️ **可短开** | Native 堆采样(开 5-10 min 后停)|
| **ART heapprofd** | §2.2 | **3-5%** | ⚠️ **可短开** | Java 堆采样(开 5 min)|
| **ftrace(指定 tracepoint)** | §7 | **1-3%** | ⚠️ **可短开** | ftrace mm/tracepoint |
| **simpleperf** | §6 | **5-15%** | ⚠️ **可灰度** | 高损耗,需用户授权 |
| **KMALLOC_TRACE** | §10 | **10-20%** | ❌ **不能开** | SLAB 分配追踪,损耗巨大 |
| **KASAN** | §10 | **100%** | ❌ **不能开** | 调试模式,release 不开 |
| **malloc_debug** | 04 篇 | **50-100%** | ❌ **不能开** | 12 篇明确说"线上不能开" |
| **gator / stream** | — | **20-50%** | ❌ **不能开** | 性能剖析工具,大损耗 |

**架构师观察**:
- **可必用工具(性能 < 0.1%)** 占 11 个——**满足 80% 的线上诊断需求**
- **可短开工具(性能 1-10%)** 占 4 个——perfetto heapprofd / ftrace / ART heapprofd / simpleperf
- **不能开工具(性能 > 10%)** 占 5 个——**只能本地复现用**

### 2.2 ART heap profile 录制(Guppy 风格)

**AOSP 14+ 新增能力**——可以用 `cmd activity` 远程触发 ART heap profile,采样某个 app 的 Java 堆分配热路径。

**关键命令**:

```bash
# 1. 启动 heap profile(录制 30 秒)
adb shell cmd activity start-heap-profile com.example.im
# 等待 30 秒
# 停止录制(输出到 /data/local/tmp/heap.prof)
adb shell cmd activity stop-heap-profile com.example.im

# 2. 拉取 profile
adb pull /data/local/tmp/heap.prof .

# 3. 解析(用 simpleperf 或直接 heap-help 工具)
# Android Studio 中:View → Tool Windows → Profiler → Memory → Java/Kotlin → Heap profile
# 或命令行: simpleprof report heap.prof --json
```

**性能损耗**:实测 3-5%(OpenJDK 11 时代的 HPROF 是 50-100%,**Android 14 的 Guppy 风格 heapprofd 是优化版**)

**线上推荐用法**:
- **5-10 min 短开**——录 5 min 拿 50-100 MB profile
- **配合 5.1 系统级 PSS 监控**——只在 PSS 上涨期间开 heapprofd

### 2.3 采样类工具(perfetto heap_profile / heapprofd)

**AOSP 14+ perfetto 原生支持** `process_memory` 和 `heap_profile` 数据源:

**关键命令**:

```bash
# 配置 perfetto 录制 30 秒的 process_memory + heap_profile
cat > /tmp/heap.pbtx << 'EOF'
duration_ms: 30000
buffers:
  - size_kb: 4096
data_sources:
  - config {
      name: "android.process_memory"
      process_memory_config {
        sampling_interval_ms: 100
        procfs_sampling_interval_ms: 1000
      }
    }
  - config {
      name: "android.heapprofd"
      heapprofd_config {
        sampling_interval_bytes: 4096
        process_cmdline: "com.example.im"
      }
    }
EOF

# 录制
adb shell perfetto --text --config /tmp/heap.pbtx --out /data/local/tmp/heap.pftrace

# 拉取
adb pull /data/local/tmp/heap.pftrace .

# 上传 Perfetto UI 查看
# https://ui.perfetto.dev/
```

**线上推荐**:
- **`sampling_interval_bytes: 4096`** 是合理采样粒度(每 4KB 采样一次)
- **30 秒录制**足够捕获泄漏热路径
- **录完立即停**——不要持续

### 2.4 ftrace 短期启用(5s snapshot,不重启)

ftrace 是**最便宜的动态追踪工具**——只开 5 秒,**损耗 1-3%**。

**关键命令**(线上 30 秒完成全流程):

```bash
# 1. 5 秒 ftrace 录制 mm 相关事件
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_process_exit/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/kmem/kmalloc/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/vmscan/mm_vmscan_direct_reclaim_begin/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/tracing_on"
sleep 5
adb shell "echo 0 > /sys/kernel/debug/tracing/tracing_on"
adb shell "cat /sys/kernel/debug/tracing/trace" > /tmp/trace.log

# 2. 拉取(本地分析)
adb pull /tmp/trace.log .

# 3. 看关键事件
grep "sched_process_exit" /tmp/trace.log | head -50  # 杀进程时点
grep "kmalloc.*size=4096" /tmp/trace.log | head -50  # 大块分配
```

**线上推荐**:
- **5 秒是上限**——5 秒以上 CPU 占用累积
- **单 tracepoint 而非全开**——只开与现象相关的 2-3 个 tracepoint
- **适合快速验证**:某进程是否在某时点被杀、某函数是否被大量调用

### 2.5 /proc/pid/smaps_rollup 多次采样(识别"持续增长")

**最便宜也最关键的"不杀进程"诊断手段**——**多次采样 smaps_rollup,做趋势对比**:

**关键命令**:

```bash
# 1. 采样 1(基线)
T1=$(date +%s)
adb shell cat /proc/<pid>/smaps_rollup > /tmp/smaps_T1.txt

# 2. 等待 5 min
sleep 300

# 3. 采样 2(5 min 后)
T2=$(date +%s)
adb shell cat /proc/<pid>/smaps_rollup > /tmp/smaps_T2.txt

# 4. 对比两次 PSS 增量
diff /tmp/smaps_T1.txt /tmp/smaps_T2.txt

# 5. 输出关键字段(脚本化)
echo "PSS_5min_Δ: $(($(grep Pss /tmp/smaps_T2.txt | awk '{print $2}') - $(grep Pss /tmp/smaps_T1.txt | awk '{print $2}'))) KB"
```

**关键观察**:
- **PSS_5min_Δ > 10 MB** —— 强泄漏信号
- **PSS_5min_Δ > 0 且 SwapPss_5min_Δ > 0** —— 内存压力已经传导到 swap
- **PSS_5min_Δ 稳定不变** —— **不是泄漏**,是稳态占用

**线上推荐**:作为**默认工具**——零损耗,5 min 拿一个数据点。

### 2.6 baseline + 增量 diff(本案 02 §2.5 思路)

**核心方法学**:**没有 baseline,就没有 diff;没有 diff,就没有泄漏判断**。

**完整流程**:

```
Step 1: 采集 baseline(设备出厂版本 / 上一稳定版本)
   ↓
Step 2: 跑同场景 24h(标准化操作:启动 3 个 app、切到后台 1h、回到前台)
   ↓
Step 3: 多次 smaps_rollup + dumpsys meminfo 采样
   ↓
Step 4: 对比 baseline,看哪些模块增长 > 10%
   ↓
Step 5: 定位到具体模块
   ↓
Step 6: 不杀进程,在生产环境做 1-2 周的"持续观察"
```

**线上推荐**:把 baseline 固化到 `dumpsys procstats --start-time <基准时间>`,**任何时间点都能回溯到 baseline**。
