## 0. 写在最前:为什么需要"不杀进程"治理

### 0.1 生产环境的 4 大约束

**"杀进程" 在生产环境不总是合适的**。前 14 篇讲的"通过 adj / LMKD / OOM killer 杀进程" 是**终极方案**,但**有 4 类场景**你必须用"不杀进程" 治理:

| 约束 | 场景 | 杀进程代价 |
|------|------|----------|
| **① 不能杀** | 用户正在用的 app(adj < 0);系统关键服务(system_server / phone / systemui) | 杀 = 用户感知崩溃/系统卡顿 |
| **② 杀了也复现** | 内存增长是**间歇性**的(bug 触发条件复杂),杀了下次还会复现 | 杀 = 治标不治本,下次还要查 |
| **③ 多个进程** | 5+ 个 app 同时增长,不知道哪个是"主犯" | 杀 = 把所有候选都杀了 |
| **④ 治理时间窗** | 治理团队需要**长时间持续观察**(7×24h),不能"杀一下看效果" | 杀 = 中断观察链 |

**结论**:**生产环境的内存治理,80% 时间是"不杀进程下识别 + 隔离 + 动态梳理"**,只有 20% 是"杀进程治理"(LMKD/AMK/force-stop)。

### 0.2 跟"杀进程治理"的本质差异

| 维度 | 杀进程治理(05/06/12) | **不杀进程治理(本篇)** |
|------|---------------------|---------------------|
| **核心目标** | 快速腾内存 | 持续观察 + 精准识别 |
| **决策速度** | ms 级(直接发 SIGKILL) | 分钟-小时级(等数据)|
| **数据来源** | adj 数值 + PSI 触发 | PSS 趋势 + baseline diff + heapprofd |
| **副作用** | 用户感知崩溃(可能性) | 几乎无 |
| **诊断精度** | 中(知道"哪类进程"被杀) | 高(知道"哪行代码泄漏")|
| **持续时间** | 单次动作 | 7×24h 持续 |
| **工具性能损耗** | 0(杀是瞬时) | 0-5%(取决于工具) |

## 1. FWK 层如何标识记录内存:ProcessRecord 内存视图

> **本节核心问题**:**FWK 层用什么字段记录一个进程的"内存状态"?线上怎么读这些字段?**
>
> **本节是 v2 系列的补强**——05 篇讲了 adj / 进程优先级,但**没讲 ProcessRecord 怎么记录内存状态**。本节补齐这块。

### 1.1 ProcessRecord 5 个内存维度

`frameworks/base/services/core/java/com/android/server/am/ProcessRecord.java` 里有 5 个字段**直接记录"内存状态"**(不是内存数字本身,而是**与内存治理相关的元数据**):

| 字段 | 类型 | 含义 | 线上读法 | 治理价值 |
|------|------|------|---------|---------|
| **`lastTrimMemoryLevel`** | `int` | 上次 trimMemory 调用的 level | `dumpsys activity processes \| grep -A2 trimMemory` | 看 app 收到了哪些 trim 请求 |
| **`lastPssTime`** | `long` | 上次 dumpsys 记录 PSS 的时间 | `dumpsys activity processes \| grep -A1 lastPss` | PSS 采样间隔(默认 60s)|
| **`lastPss`** | `long` | 上次采样的 PSS(单位 KB) | 同上 | PSS 趋势基线 |
| **`lastRequestedGc`** | `long` | 上次主动触发 GC 的时间 | `dumpsys activity processes \| grep lastRequestedGc` | 主动 GC 频率(可能反映 OOM 压力) |
| **`lastLowMemory`** | `long` | 上次标记为低内存的时间 | `dumpsys activity processes \| grep lastLowMemory` | 内存压力告警的时序 |

**完整 dumpsys activity processes 输出片段**(简化):

```text
ProcessRecord{3f4a8c 1500:com.example.im/u0a100}
  userId=10010
  curAdj=900
  setAdj=900
  curProcState=PROCESS_STATE_CACHED_ACTIVITY   ← ★ 内存治理触发态
  lastTrimMemoryLevel=MODERATE                ← ★ 关键字段
  lastPss=245760                                ← ★ 240 MB
  lastPssTime=2026-07-15 14:23:45             ← ★ 1 min 前采样
  lastRequestedGc=2026-07-15 14:21:30         ← ★ 2 min 前主动 GC
  lastLowMemory=2026-07-15 14:20:00          ← ★ 4 min 前低内存告警
  ...
```

**线上排查典型用法**:

```bash
# 1. 找最近 1 小时被 trimMemory 调用过 >= 3 次的进程(内存压力证据)
adb shell dumpsys activity processes | \
  awk '/ProcessRecord/{p=$0} /lastTrimMemoryLevel/{print p, $0}'

# 2. 找 lastPss > 500MB 且 lastPssTime 在最近 1 min 的进程(疑似泄漏)
adb shell dumpsys activity processes | \
  awk '/lastPss=/{split($2,a,"="); pss=a[2]/1024; if(pss>500) print $0}'

# 3. 找 lastRequestedGc 频率 > 5 次/小时(被频繁触发 GC,可能 OOM 边缘)
adb shell dumpsys activity processes | \
  grep "lastRequestedGc" | sort | uniq -c
```

### 1.2 ActivityRecord / ServiceRecord 内存记账

ProcessRecord 是**进程级**,但**每个进程内的 Activity / Service 也有自己的内存记账字段**:

**`ActivityRecord.java` 中**:

| 字段 | 含义 | 治理价值 |
|------|------|---------|
| `mLastReportedConfiguration` | Activity 配置(屏幕/方向) | 配置变化触发内存重分配 |
| `mIcicle` | onSaveInstanceState 的 Bundle 内存 | Activity 重建时的内存快照 |
| `thumbnail` | Activity 缩略图(用 mThumbnailFile) | 缩略图内存占用 |

**`ServiceRecord.java` 中**:

| 字段 | 含义 | 治理价值 |
|------|------|---------|
| `lastActivity` | Service 最后活跃时间 | 识别"僵尸 Service" |
| `startRequested` | 是否 startService() | startService = 持锁,可能不释放 |
| `connections` | 当前 binder 连接数 | 识别泄漏的连接 |
| `pendingConnectionGroup` | 待处理连接组 | 积压 = 内存风险 |

**线上排查用法**(放在本节是为了**说明 FWK 层内存记账的全貌**):

```bash
# 4. 找所有 connections > 5 的 Service(可能是泄漏)
adb shell dumpsys activity services | \
  awk '/ServiceRecord/{p=$0} /connections=/{n=$2; gsub("=","",n); if(n+0>5) print p, $0}'

# 5. 找 lastActivity > 24h 的 Service(僵尸)
adb shell dumpsys activity services | grep "lastActivity" | \
  awk '{...时间判断...}'
```

### 1.3 dumpsys meminfo 怎么把"ProcessRecord 5 维 + 内存详情"缝起来

`dumpsys meminfo` 是**进程级**的内存快照,但**和 ProcessRecord 5 维是不同维度**:

```
ProcessRecord 5 维     ←  元数据(状态/时间/触发)
       ↓
       与
       ↓
dumpsys meminfo 内存详情 ← 数字(PSS/RSS/Heap)
       ↓
       关联
       ↓
procstats 长期趋势   ← 历史(小时/天/周)
```

**完整 dumpsys meminfo 关键字段**(本节只列**线上常用**的,完整字段看 13 §5.1):

```
TOTAL PSS: 240 MB           ← ★ 进程 PSS 总和
  Native Heap:   80 MB      ← ★ C/C++ 堆(04 篇)
  Java Heap:    100 MB      ← ★ ART Java 堆(03 篇)
  Code:          20 MB      ← ★ dex + so
  Stack:         5 MB       ← ★ 线程栈
  Graphics:      15 MB      ← ★ 图形缓冲(04 §5)
  Private Other:  3 MB      ← ★ 私有内存
  System:        17 MB      ← ★ 系统级(TLS 等)

Heap:        100 MB / 256 MB  ← ★ 已用 / 堆大小
Heap Alloc:   80 MB           ← ★ 已分配
Heap Free:   176 MB           ← ★ 空闲

Views:        1500            ← ★ View 数量(界面复杂度)
Activities:      3            ← ★ Activity 数量
AppContexts:     5            ← ★ Context 数量(可能泄漏)
Assets:          8            ← ★ AssetManager 数量
```

**关键关联**:当 `dumpsys meminfo` 的 `Java Heap: 100MB` 突然上涨,**同时** `ProcessRecord.lastPss` 也在涨,**且** `lastTrimMemoryLevel` 从 `RUNNING_MODERATE` 升到 `COMPLETE`——**这就是典型 Java 堆泄漏信号**。

### 1.4 dumpsys procstats:PSS 历史数据(可用于"不杀进程"做趋势分析)

`dumpsys procstats` 是**system_server 持有的 `ProcessStatsService`** 维护的**长期 PSS 历史**:

**关键命令**:

```bash
# 1. 进程级 PSS 历史(最近 24h)
adb shell dumpsys procstats --hours 24 | head -100

# 2. 单进程详细 PSS 历史
adb shell dumpsys procstats com.example.im

# 3. 内存状态时间线(配合 grep 找增长)
adb shell dumpsys procstats com.example.im | \
  awk '/^\s*Sample/{print}'
```

**输出格式**(简化):

```
Process com.example.im / u0a100:
  ProcessSummary:
    State: PERSISTENT  ★
    TotalRSS:    180 MB
    TotalPss:    120 MB
    TotalSwapPss:  5 MB

  Last 24 hours PSS samples:
    Sample at 2026-07-15 14:00:00: PSS=120 MB
    Sample at 2026-07-15 15:00:00: PSS=125 MB    ← +5 MB/h
    Sample at 2026-07-15 16:00:00: PSS=140 MB    ← +15 MB/h  ← ★ 加速!
    Sample at 2026-07-15 17:00:00: PSS=160 MB    ← +20 MB/h  ← ★ 确认泄漏
```

**线上治理价值**:
- **不杀进程,只看趋势**——`dumpsys procstats` 跑一遍,**1 分钟看到 24h PSS 曲线**
- 配合 5 个 ProcessRecord 字段,**可以做"不杀进程下定位泄漏模块"**

### 1.5 ProcessStatsService 的内存账本机制

`frameworks/base/services/core/java/com/android/server/am/ProcessStatsService.java` 维护**所有进程的 PSS 长期账本**:

| 内存机制 | 源码位置 | 数据保留时长 |
|---------|---------|------------|
| `mProcessStats` (SparseArray) | ProcessStatsService.java | 24h(默认) |
| `mLastReportMemUsage` (long) | ProcessStatsService.java | 24h |
| `mProcessStatsCounter` (int) | ProcessStatsService.java | 进程整个生命周期 |
| `mLastRss` (long) | ProcessStatsService.java | 24h |
| `mMemFactor` (enum) | ProcessStatsService.java | 进程整个生命周期 |
| `mStartTime` (long) | ProcessRecord.java | 进程整个生命周期 |

**关键概念——`mMemFactor`**:

```java
// ProcessStatsService.java 中的 memFactor
public enum MemFactor {
    UNKNOWN,        // 未知
    NORMAL,          // 正常
    MODERATE,        // 内存中度压力
    LOW,             // 内存低压
    CRITICAL,        // 内存临界
    NOT_RESPONDING   // 进程不响应(ANR 边缘)
}
```

**线上读法**:

```bash
# 看所有进程的 memFactor
adb shell dumpsys procstats | grep -A1 "MemFactor"
```

**关键观察**:`mMemFactor` 从 `NORMAL` 升到 `MODERATE` 通常意味着**多个进程同时在涨**——这是 cgroup 级别压力的信号,**不是单进程问题**。

## 2. 线上诊断:不杀进程下识别内存问题

> **本节核心问题**:**生产环境能开什么工具?哪些不能开?开了之后如何解读?**

### 2.1 受限诊断矩阵(对比 13 的完整工具)

13 §5-§8 讲了完整工具集(perfetto、ftrace、simpleperf、heapprofd 等),但**没区分"线上可用 vs 不能用"**。本节给出**线上受限诊断矩阵**:

| 工具 | 13 篇位置 | 性能损耗 | 生产环境**推荐** | 备注 |
|------|---------|---------|-------------------|------|
| `dumpsys meminfo` | §5.1 | < 0.1% | ✅ **必用** | 零损耗,关键工具 |
| `dumpsys procstats` | §5.2 | < 0.1% | ✅ **必用** | 长期趋势,看增长 |
| `dumpsys activity processes` | §5.7 | < 0.1% | ✅ **必用** | ProcessRecord 5 维 |
| `cat /proc/<pid>/smaps_rollup` | §3.6 | < 0.1% | ✅ **必用** | 进程级 PSS 汇总 |
| `cat /proc/<pid>/status` | §3.7 | < 0.1% | ✅ **必用** | 进程基础状态 |
| `cat /proc/meminfo` | §3.1 | < 0.1% | ✅ **必用** | 系统级 |
| `cat /proc/vmstat` | §3.2 | < 0.1% | ✅ **必用** | 内核事件 |
| `cat /proc/pressure/memory` | §3.9 | < 0.1% | ✅ **必用** | PSI 压力 |
| `cat /proc/cgroups` | — | < 0.1% | ✅ **必用** | cgroup 视图 |
| `cat /proc/<pid>/cgroup` | — | < 0.1% | ✅ **必用** | 进程 cgroup 归属 |
| `cat /proc/lmkd/debug` | §4.2 | < 0.1% | ✅ **必用** | lmkd 事件 |

### 2.4 ftrace 短期启用(5s snapshot,不重启)

ftrace 是**最便宜的动态追踪工具**——只开 5 秒,**损耗 1-3%**。

**关键命令**(线上 30 秒完成全流程):

```bash
# 1. 5 秒 ftrace 录制 mm 相关事件
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_process_exit/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/kmem/kmalloc/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/vmscan/mm_vmscan_direct_reclaim_begin/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/tracing_on"
sleep 5
adb shell "echo 0 > /sys/kernel/debug/tracing/tracing_on"
adb shell "cat /sys/kernel/debug/tracing/trace" > /tmp/trace.log

# 2. 拉取(本地分析)
adb pull /tmp/trace.log .

# 3. 看关键事件

grep "kmalloc.*size=4096" /tmp/trace.log | head -50  # 大块分配
```

**线上推荐**:
- **5 秒是上限**——5 秒以上 CPU 占用累积
- **单 tracepoint 而非全开**——只开与现象相关的 2-3 个 tracepoint
- **适合快速验证**:某进程是否在某时点被杀、某函数是否被大量调用

### 2.5 /proc/pid/smaps_rollup 多次采样(识别"持续增长")

**最便宜也最关键的"不杀进程"诊断手段**——**多次采样 smaps_rollup,做趋势对比**:

**关键命令**:

```bash
# 1. 采样 1(基线)
T1=$(date +%s)
adb shell cat /proc/<pid>/smaps_rollup > /tmp/smaps_T1.txt

# 2. 等待 5 min
sleep 300

# 3. 采样 2(5 min 后)
T2=$(date +%s)
adb shell cat /proc/<pid>/smaps_rollup > /tmp/smaps_T2.txt

# 4. 对比两次 PSS 增量
diff /tmp/smaps_T1.txt /tmp/smaps_T2.txt

# 5. 输出关键字段(脚本化)
echo "PSS_5min_Δ: $(($(grep Pss /tmp/smaps_T2.txt | awk '{print $2}') - $(grep Pss /tmp/smaps_T1.txt | awk '{print $2}'))) KB"
```

**关键观察**:
- **PSS_5min_Δ > 10 MB** —— 强泄漏信号
- **PSS_5min_Δ > 0 且 SwapPss_5min_Δ > 0** —— 内存压力已经传导到 swap
- **PSS_5min_Δ 稳定不变** —— **不是泄漏**,是稳态占用

**线上推荐**:作为**默认工具**——零损耗,5 min 拿一个数据点。

### 2.6 baseline + 增量 diff(本案 02 §2.5 思路)

**核心方法学**:**没有 baseline,就没有 diff;没有 diff,就没有泄漏判断**。

**完整流程**:

```
Step 1: 采集 baseline(设备出厂版本 / 上一稳定版本)
   ↓
Step 2: 跑同场景 24h(标准化操作:启动 3 个 app、切到后台 1h、回到前台)
   ↓
Step 3: 多次 smaps_rollup + dumpsys meminfo 采样
   ↓
Step 4: 对比 baseline,看哪些模块增长 > 10%
   ↓
Step 5: 定位到具体模块
   ↓
Step 6: 不杀进程,在生产环境做 1-2 周的"持续观察"
```

**线上推荐**:把 baseline 固化到 `dumpsys procstats --start-time <基准时间>`,**任何时间点都能回溯到 baseline**。

## 3. 线上隔离:不杀进程下"框定"问题范围

> **本节核心问题**:**确认了"某进程在泄漏"之后,怎么不杀它但把它"框定"在影响范围?**
>
> **核心思路**:**用 cgroup v2 的"节点级隔离",把问题进程"圈"在它自己的 cgroup 节点里**,即使内存涨,也不会影响其他 app。

### 3.1 cgroup v2 节点级隔离(Android 14+)

**Android 14+ 全面切到 cgroup v2**(不再用 cgroup v1)。cgroup v2 节点级隔离是**线上治理的核心手段**。

**cgroup v2 树形结构(AOSP 14+ 设备实际)**:

```
/sys/fs/cgroup/
├── init.scope/                          ← init 进程
├── system.slice/                         ← system_server 等系统服务
│   ├── system-server/                    ← system_server(大节点)
│   ├── webview/                          ← webview
│   └── ...
├── apps.slice/                           ← 所有 app
│   ├── apps/uid_10010/                   ← app 1(com.example)
│   ├── apps/uid_10020/                   ← app 2(...)
│   └── ...
├── top-app.slice/                        ← 当前前台 app(API 31+)
│   └── uid_10030/                        ← 当前前台 app
└── ...
```

**关键 cgroup v2 文件**(每个 cgroup 节点下都有):

| 文件 | 含义 | 治理用法 |
|------|------|---------|
| `cgroup.procs` | 节点内所有 pid 列表 | 看哪些进程在该 cgroup |
| `memory.current` | 节点当前内存使用(字节) | 实时监控 |
| `memory.peak` | 节点历史最大内存 | 看历史峰值 |
| `memory.max` | 节点内存硬限制 | **关键** — 隔离入口 |
| `memory.high` | 节点内存软限制 | 软限制,触发 throttle |
| `memory.min` | 节点内存保底 | **关键** — 临时冻结 |
| `memory.events` | 节点内存事件(low/high/max/oom) | 看触发过什么 |
| `memory.pressure` | PSI 接口 | 节点 PSI |
| `cgroup.subtree_control` | 子树资源控制 | 父节点配置 |

### 3.2 /proc/pid/cgroup + /proc/pid/smaps_rollup 联动

**关联查看命令**:

```bash
# 1. 找进程属于哪个 cgroup
adb shell cat /proc/<pid>/cgroup
# 输出:
# 0::/system.slice/apps/uid_10010/...
# → 进程在 /sys/fs/cgroup/system.slice/apps/uid_10010/

# 2. 看该 cgroup 节点的实时内存
adb shell cat /sys/fs/cgroup/system.slice/apps/uid_10010/memory.current
# 输出: 245678592 (= 234 MB)

# 3. 看该 cgroup 节点是否触发过 max
adb shell cat /sys/fs/cgroup/system.slice/apps/uid_10010/memory.events
# 输出:
#   low 0
#   high 5
#   max 12
#   oom 0
# → 该 cgroup 12 次触发过 max(内存超限)

# 4. 看进程级 PSS 和 cgroup 内存是否对得上
adb shell cat /proc/<pid>/smaps_rollup
# 输出: 240000 KB PSS
# → cgroup memory.current 234 MB ≈ 进程 PSS 240 MB(差 4% 是 cgroup 内部其他进程)
```

**线上排查用法**:

```bash
# 5. 找"cgroup 已触发 max 但进程没被 OOM"的进程(隔离生效)
adb shell "for pid in \$(ls /proc/ | grep -E '^[0-9]+\$'); do
  cgroup=\$(cat /proc/\$pid/cgroup 2>/dev/null | grep '0::' | head -1)
  if [ -n \"\$cgroup\" ]; then
    path=\${cgroup#0::}
    events=\$(cat /sys/fs/cgroup\$path/memory.events 2>/dev/null | grep 'max')
    if [ \"\$events\" != \"max 0\" ]; then
      echo \"PID=\$pid cgroup=\$path \$events\"
    fi
  fi
done" 2>/dev/null
```

### 3.3 lmkd 主动配合(给"问题进程"打 oom_score_adj 让 LMKD 知道"它可以杀")

**核心思路**:**线上"不杀进程" 不是"永远不杀",而是"先治理 + 让 LMKD 知道它可以杀"**。

**关键命令**:

```bash
# 1. 把"问题进程"的 oom_score_adj 调高(让 LMKD 优先杀)
# 注意:这**不杀**进程,只调整 LMKD 的"候选列表"
# 通常 oom_score_adj 范围: -1000(永不杀) ~ 1000(优先杀)
# 默认 app: 0~1000(根据 adj 状态)
# cached 进程: 900+
adb shell "echo 800 > /proc/<pid>/oom_score_adj"

# 2. 让问题进程有"逃生通道"——但不是现在杀
# 在 dumpsys 验证:
adb shell dumpsys activity processes | grep oom_score
```

**关键观察**:
- **不要**直接把 oom_score_adj 调到 1000(可能立即被杀)
- **推荐**:调到 800~900,让 LMKD 在**内存真正吃紧**时把它当候选
- **配套**:同时设置 `memory.high` 软限制(§3.5),让进程被 throttle 而不是杀

### 3.4 临时冻结 cgroup 内存回收(memory.min 调高)

**反向思路**:**如果"问题进程" 是关键进程(system_server),不能被 throttle 也不能被回收**,那**把它的 cgroup 节点** `memory.min` **调高,让它在内存压力下不被回收**:

**关键命令**:

```bash
# 1. 看 system_server 当前 memory.min
adb shell cat /sys/fs/cgroup/system.slice/system-server/memory.min
# 输出: 0(没设)

# 2. 临时把 system_server memory.min 调到 1 GB(让它不被回收)
adb shell "echo 1073741824 > /sys/fs/cgroup/system.slice/system-server/memory.min"

# 3. 验证
adb shell cat /sys/fs/cgroup/system.slice/system-server/memory.min
# 输出: 1073741824 (= 1 GB)
```

**关键观察**:
- **`memory.min` 是保底**——内核会保证该 cgroup 至少有 1 GB 可用
- **如果该 cgroup 实际用 < 1 GB**,多余会被**借给其他 cgroup**
- **如果该 cgroup 实际用 > 1 GB**,保底不生效(自己负责)
- **风险**:`memory.min` 总和 > 系统总内存,会让其他 cgroup 饿死

### 3.5 实战:cgroup 节点级 throttle 模板

**完整模板**(可复制执行):

```bash
# === 场景:某 app(com.example)内存持续增长,要"不杀但限制" ===

# 1. 找到 app 的 cgroup
APP_CGROUP=$(adb shell "cat /proc/\$(pidof com.example)/cgroup 2>/dev/null | grep '0::' | head -1 | cut -d: -f3")
echo "App cgroup: $APP_CGROUP"

# 2. 设置 memory.high(软限制 — 触发 throttle)
# 软限制超过后,内核会放慢该 cgroup 的内存分配(不是杀)
# 推荐: 设为进程当前 PSS 的 1.5 倍
PSS_NOW=$(adb shell "cat /proc/\$(pidof com.example)/smaps_rollup | grep Pss | head -1 | awk '{print \$2}'")
HIGH=$((PSS_NOW * 3 / 2))  # 1.5x in KB
adb shell "echo ${HIGH}*1024 | bc > /sys/fs/cgroup$APP_CGROUP/memory.high"

# 3. 设置 memory.max(硬限制 — 触发 reclaim)
# 硬限制超过后,内核会强制回收该 cgroup(不是杀进程)
MAX=$((PSS_NOW * 2))  # 2x in KB
adb shell "echo ${MAX}*1024 | bc > /sys/fs/cgroup$APP_CGROUP/memory.max"

# 4. 观察 30 min
for i in {1..6}; do
  sleep 300
  CURRENT=$(adb shell "cat /sys/fs/cgroup$APP_CGROUP/memory.current")
  EVENTS=$(adb shell "cat /sys/fs/cgroup$APP_CGROUP/memory.events")
  echo "[$i/6] current=$CURRENT events=$EVENTS"
done

# 5. 清理(恢复 — 移除限制)
adb shell "echo max > /sys/fs/cgroup$APP_CGROUP/memory.max"
adb shell "echo 0 > /sys/fs/cgroup$APP_CGROUP/memory.high"
```

**关键观察**:
- **`memory.high` = 软限制**——推荐用
- **`memory.max` = 硬限制**——小心用,可能触发 OOM
- **监控 events.low/high/max 计数器**——如果 max 持续增长,**意味着 cgroup 在被强 reclaim**

## 4. 线上动态梳理:不杀进程下"治理"内存

> **本节核心问题**:**"隔离" + "诊断"之后,怎么让进程"自己"释放内存?**
>
> **核心机制**:**FWK 层提供 4 个"远程触发内存释放"接口,线上不杀进程下做"主动梳理"**。

### 4.1 AppProcessImportance 动态降级

**通过设置 `app.processImportance`** 让系统认为该 app 不重要,从而**降低它的 adj**(adj 升高 = 重要性降低 = 更容易被 LMKD 杀):

**关键命令**:

```bash
# 1. 把 app adj 临时降到 CACHED(900) — 模拟"用户切到后台很久"
# 注意:这**不杀**进程,只是让系统觉得它不重要
adb shell am send-trim-memory com.example RUNNING_MODERATE
adb shell am send-trim-memory com.example COMPLETE
```

**关键 API**:`am send-trim-memory <pkg> <level>`:
- `RUNNING_MODERATE` — 内存压力中
- `RUNNING_LOW` — 内存压力高
- `RUNNING_CRITICAL` — 内存压力临界
- `UI_HIDDEN` — UI 隐藏(等同 onTrimMemory UI_HIDDEN)
- `BACKGROUND` — 进入后台
- `MODERATE` / `COMPLETE` — 详细档位

### 4.2 trimMemory 远程触发(ActivityManager.appTrimMemory)

**Finer-grained 接口**——`ActivityManager.appTrimMemory()`,从 FWK 远程给所有 app 发 trim 通知:

**关键命令**:

```bash
# 1. 远程触发所有 app 的 trimMemory(COMPLETE 档)
adb shell am send-trim-memory --user 0 COMPLETE

# 2. 远程触发某个进程的 trimMemory
adb shell am send-trim-memory com.example MODERATE

# 3. 远程触发指定 uid 的 trimMemory
adb shell am send-trim-memory --user 0 --uid 10010 COMPLETE
```

**Finer API 源码路径**:
- `frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java#appTrimMemory`
- `frameworks/base/core/java/android/app/ActivityManager.java#appTrimMemory`

### 4.3 Application.onTrimMemory() 的 FWK 端契约

**关键**:`onTrimMemory` 是个**契约**——App 端必须实现,FWK 才能有效 trim:

**App 端正确实现**(Java):

```java
public class MyApplication extends Application {
    @Override
    public void onTrimMemory(int level) {
        super.onTrimMemory(level);
        // TRIM_MEMORY_RUNNING_MODERATE:释放可重建的缓存
        if (level >= TRIM_MEMORY_RUNNING_MODERATE) {
            // 1. 释放图片缓存
            imageCache.evictAll();
            // 2. 释放可丢弃的 List
            recentItemsCache.clear();
        }
        // TRIM_MEMORY_COMPLETE:释放一切可释放的
        if (level >= TRIM_MEMORY_COMPLETE) {
            // 1. 关闭可选的 Service
            // 2. 释放单例
            // 3. 释放 viewModelStore
        }
    }
}
```

**关键观察**:
- **`onTrimMemory` 是 App 侧的**——App 必须实现才有意义
- **如果 App 没实现**——FWK 发 trim 通知,App 收不到 → 内存不会减少
- **线上验证 App 是否实现**:用 `am send-trim-memory` 触发,看 App 内存是否真的减少

### 4.4 binder 端点级"清堆"指令

**AOSP 14+ 新增能力**——可以通过 binder 直接对**指定进程**的 ART 触发"清理堆"。

**关键命令**:

```bash
# 1. 触发指定进程的 Runtime GC
adb shell "kill -SIGRTMIN+4 <pid>"  # ART 内部信号

# 2. 更精确的:通过 am 命令触发
adb shell am gc <pid>  # 直接对指定进程发 GC
```

**注意**:**`am gc`** 是**用户态命令**,但**实际效果**取决于 ART 是否处理。
- ART 看到 GC 信号 → 触发 **Concurrent Mark Sweep (CMS)** 或 **Concurrent Copying (CC)**
- **不是 full GC**——只能释放可回收的内存,不能解决真正的泄漏

### 4.5 Per-Process ART compact / GC 触发

**AOSP 14+ ART 内部能力**:
- `art::Runtime::RequestHeapTrim()` — 主动请求 trim
- `art::gc::Heap::Trim()` — 主动 trim heap(释放空闲 pages)
- `art::gc::Heap::CollectGarbage()` — 触发 GC

**触发方式**:

```bash
# 1. 通过 setprop + kill -3 触发 ART profile save + GC
adb shell setprop ctl.start siggcoree

# 2. 通过 debuggerd 触发(只 root)
# 不推荐线上用
```

**线上推荐**:**`am gc` 是最简单的"远程 GC 触发"**——效果有限(只能释放空闲页),但 0 风险。

### 4.6 heap growth inhibitor 模式(Android 15+)

**Android 15+ 新增**——`ActivityManager.setProcessMemoryTrimLevel()` API,允许系统**抑制 Java 堆增长**。

**关键 API**:

```java
// Android 15+ Framework API
ActivityManager.setProcessMemoryTrimLevel(int level, int fromUid);
```

**level 取值**:

| Level | 含义 | 行为 |
|-------|------|------|
| `MEMORY_TRIM_LEVEL_NORMAL` | 正常 | 无抑制 |
| `MEMORY_TRIM_LEVEL_MODERATE` | 中度抑制 | 限制新建对象大小 |
| `MEMORY_TRIM_LEVEL_COMPLETE` | 完全抑制 | 强制 GC + 限制新分配 |

**线上推荐**:Android 15+ 可以**系统级**抑制 Java 堆,**是治理 system_server 内存爆炸的新手段**。

## 5. 实战 3 例(基于真实脱敏)

> **本节给 3 个真实脱敏案例,展示"不杀进程下"治理的全流程**。

### 5.1 system_server Java 堆 2GB 不杀进程定位

**场景**:某设备,system_server 进程 Java 堆持续增长,12h 内从 800MB 涨到 2GB,**触发系统卡顿**。但**不能杀 system_server**——杀了整机重启,用户感知崩溃。

**完整诊断流程**(3 天):

```
Day 1:
  Step 1: 抓 baseline
    adb shell dumpsys procstats --start-time 7d_ago
  Step 2: 看 PSS 曲线
    adb shell dumpsys procstats | grep system_server -A 30
  Step 3: 看到 system_server 的 PSS 在 12h 内从 800MB → 2GB
  Step 4: 抓 smaps_rollup
    adb shell cat /proc/<system_server_pid>/smaps_rollup
  Step 5: 看到 Java Heap: 1800 MB, Native Heap: 200 MB
  Step 6: 关键观察:Heap Alloc / Heap Free 比例异常(Alloc 1750 MB / Total 2048 MB)

Day 2:
  Step 7: 远程触发 trimMemory COMPLETE
    adb shell am send-trim-memory android COMPLETE
  Step 8: 10 min 后再看 PSS
    → 1900 MB(只降了 100MB) → 不是 trimMemory 能解决的
  Step 9: 启用 perfetto heap_profile 5 min
    perfetto --text --config /tmp/heap.pbtx --out /data/local/tmp/heap.pftrace
  Step 10: 拉取分析
    → 看 Top by size:ActivityManager$Service 占据 800 MB
    → 进一步定位:ActivityManager$Lifecycle 占据 500 MB

Day 3:
  Step 11: 验证假设
    adb shell dumpsys activity activities | grep -i "ActivityRecord" | wc -l
    → 4500 个 ActivityRecord 残留(异常!正常应该 < 200)
  Step 12: 根因定位
    → 某个第三方 app 启动 Activity 没 finish,ActivityRecord 全部堆在 system_server
  Step 13: 不杀进程
    → 远程触发 system_server GC + trimMemory
    → 临时设置 system_server cgroup memory.min = 2 GB(冻结回收)
    → 联系 app 端修复
```

**根因**:**第三方 app 的 Activity 泄漏 → ActivityRecord 堆在 system_server**。

**治理手段**:
- **临时**:cgroup `memory.min = 2 GB` 冻结 system_server 回收
- **短期**:am force-stop 第三方 app(`adb shell am force-stop com.example.leaky`)
- **长期**:通知 app 端修复 Activity 泄漏

### 5.2 某 app Native 持续增长,cgroup 隔离后定位泄漏模块

**场景**:某 app 24h 内 PSS 从 200MB 涨到 600MB,**且 Native Heap 占比 70%**——典型 Native 内存泄漏。**不能杀**——杀 = 用户感知崩溃。

**完整诊断流程**(2 天):

```
Day 1:
  Step 1: 多次采样 smaps_rollup
    adb shell cat /proc/<pid>/smaps_rollup (每 1h 一次)
  Step 2: 看到 PSS_5min_Δ ≈ 5 MB(每小时涨 60 MB) — 稳定泄漏
  Step 3: 看 smaps 详细(memory map)
    adb shell cat /proc/<pid>/smaps | grep -A1 "Native\|heap"
  Step 4: 看到 [heap] 段持续增长 — Native heap 增长
  Step 5: cgroup 节点级隔离(防止影响其他 app)
    APP_CGROUP=$(adb shell cat /proc/<pid>/cgroup | grep '0::' | head -1 | cut -d: -f3)
    adb shell "echo 600*1024*1024 > /sys/fs/cgroup$APP_CGROUP/memory.max"  # 600MB 硬限制
    adb shell "echo 400*1024*1024 > /sys/fs/cgroup$APP_CGROUP/memory.high"  # 400MB 软限制

Day 2:
  Step 6: perfetto heap_profile 录制 5 min
    perfetto --config /tmp/heap.pbtx (含 native heap_profile)
  Step 7: 拉取分析
    → 看 Top by size:bitmap 占据 200 MB(Native Bitmap!)
  Step 8: 进一步定位
    → 某些 Bitmap 没 recycle() — 某个 ViewModel 持有
  Step 9: 远程 trimMemory
    adb shell am send-trim-memory com.example COMPLETE
    → PSS 降回 250 MB(释放了 350 MB 泄漏的 Bitmap)
  Step 10: 清理 cgroup 限制
    adb shell "echo max > /sys/fs/cgroup$APP_CGROUP/memory.max"
```

**根因**:**Bitmap 未 recycle + ViewModel 持有**。

**治理手段**:
- **cgroup 隔离**(已做)
- **远程 trimMemory** 释放
- **联系 app 端**修复 Bitmap 泄漏

### 5.3 zygote preload 内存膨胀,动态梳理方案

**场景**:某 OEM 设备,zygote 进程从 100MB 涨到 500MB,**导致所有 app 冷启动慢**。**不能杀 zygote**——杀 = 整机重启。

**完整诊断流程**(5 天):

```
Day 1:
  Step 1: 看 zygote 进程 smaps
    adb shell cat /proc/<zygote_pid>/smaps_rollup
  Step 2: 看到 Native Heap 占据 300 MB(异常)
  Step 3: 看 smaps 详细
    adb shell cat /proc/<zygote_pid>/smaps | grep -B1 -A1 "r-xp" | head -20
  Step 4: 看到很多 preload 的 so 文件占用

Day 2-3:
  Step 5: 跑 perfetto trace 看 zygote 进程启动
    perfetto --config --text -d /tmp/zygote.cfg
  Step 6: 看哪些 so 被 preload
  Step 7: 找到 2 个 so 占 200 MB(某个 vendor HAL)

Day 4-5:
  Step 8: 远程不能直接 trim zygote(zygote 没有 trimMemory 入口)
  Step 9: 方案 A:动态 unload preload classes
    adb shell "cmd activity update-preload-info com.vendor.hal"  (仅 Android 13+ 部分设备)
  Step 10: 方案 B:减少 zygote preload 项
    修改 frameworks/base/config/preloaded-classes,减少 vendor preload
  Step 11: 方案 C:从 zygote 进程中强制 GC(只 root)
    adb shell am gc <zygote_pid>
  Step 12: 治理效果
    zygote 内存: 500MB → 350MB
    所有 app 冷启动: 3s → 1.5s
```

**根因**:**vendor HAL preload 过度,zygote 印钞机模板过大**。

**治理手段**:
- **短期**:zygote 强制 GC(只 root)
- **中期**:减少 preload 项
- **长期**:vendor HAL 优化加载时机

## 6. 监控与告警(配套)

> **本节核心问题**:**"不杀进程"治理需要 7×24h 持续监控,怎么设计告警体系?**

### 6.1 内存增长告警的 4 个关键指标

**基于 dumpsys procstats 长期采样**(零损耗):

| 指标 | 公式 | 阈值(典型) | 告警 |
|------|------|----------|------|
| **PSS_1h 增长率** | (PSS_now - PSS_1h_ago) / PSS_1h_ago | > 30% | WARNING |
| **PSS_24h 增长率** | (PSS_now - PSS_24h_ago) / PSS_24h_ago | > 100% | CRITICAL |
| **Heap Alloc / Heap Total 比例** | Heap_Alloc / Heap_Total | > 80% | WARNING |
| **cgroup max 触发次数** | memory.events.max | > 0/小时 | WARNING |

**完整监控脚本**(零损耗,7×24h 可跑):

```bash
#!/bin/bash
# /usr/local/bin/mem_monitor.sh
# 用法:每 5 min 跑一次

LOG_DIR=/data/local/tmp/mem_monitor
mkdir -p $LOG_DIR

# 1. 抓 system_server PSS
SYSTEM_SERVER_PID=$(pidof system_server)
SYSTEM_SERVER_PSS=$(cat /proc/$SYSTEM_SERVER_PID/smaps_rollup | grep Pss | head -1 | awk '{print $2}')

# 2. 抓 1h 前 PSS(从 procstats)
PSS_1H_AGO=$(dumpsys procstats system_server | head -200 | grep "Sample at.*1 hour" | tail -1 | awk '{print $NF}')

# 3. 算增长率
if [ -n "$PSS_1H_AGO" ]; then
  GROWTH=$((($SYSTEM_SERVER_PSS - $PSS_1H_AGO) * 100 / $PSS_1H_AGO))
  if [ $GROWTH -gt 30 ]; then
    echo "[WARNING] system_server PSS grew ${GROWTH}% in 1h" >> $LOG_DIR/alerts.log
  fi
fi

# 4. cgroup max 触发
APP_CGROUPS=$(ls /sys/fs/cgroup/apps.slice/apps/uid_*/ 2>/dev/null)
for c in $APP_CGROUPS; do
  MAX_HIT=$(cat $c/memory.events 2>/dev/null | awk '/max/ {print $2}')
  if [ "$MAX_HIT" != "0" ] && [ -n "$MAX_HIT" ]; then
    echo "[WARNING] $c hit memory.max $MAX_HIT times" >> $LOG_DIR/alerts.log
  fi
done
```

### 6.2 哪些是"必须杀"的信号,哪些是"可以拖"

**判断决策表**:

| 信号 | 必须杀? | 治理手段 |
|------|--------|---------|
| **PSS 5h 涨 200%** | ⚠️ **必须杀**(泄漏已经失控)| 杀 + dumpsys 抓最后状态 |
| **PSS 24h 涨 50%** | 可拖(缓慢泄漏)| cgroup 隔离 + 远程 trimMemory |
| **Heap Alloc > 95%** | ⚠️ **必须杀**(OOM 即将触发)| 杀 + force-stop app |
| **Heap Alloc > 80%** | 可拖(还能撑)| cgroup `memory.high` 软限制 |
| **cgroup max 触发 > 10 次/小时** | 可拖(已被强 reclaim)| 观察 1h,如果持续 → 杀 |
| **cgroup OOM 触发** | ⚠️ **必须杀**(已经到上限)| 立即杀 |
| **system_server Java Heap > 1.5 GB** | ⚠️ **不杀但要治理**| cgroup `memory.min` 冻结 + 远程 trimMemory |
| **app 进程 RSS > 1 GB** | 可拖(单 app 占用过大)| cgroup 隔离 + 通知 app 端 |
| **多个 app 同时涨** | 可拖(系统级压力)| 等 LMKD 触发,不要手动杀 |

**核心原则**:
- **单 app 失控** → 可以不杀
- **系统级(system_server)失控** → 不杀但要治理
- **LMKD 自动杀 + cgroup 隔离** 是首选
- **手动 force-stop** 是**最后手段**

### 6.3 治理灰度平台设计

**推荐架构**(线上治理平台):

```
┌─────────────────────────────────────────────────────────────┐
│  治理灰度平台(线上)                                          │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  数据采集层(零损耗)                                  │  │
│  │  - dumpsys meminfo(每 5 min)                          │  │
│  │  - dumpsys procstats(每 1 hour)                       │  │
│  │  - /proc/pid/smaps_rollup(每 1 min)                  │  │
│  │  - /sys/fs/cgroup/memory.events(每 5 min)           │  │
│  └───────────────────────────────────────────────────────┘  │
│  ↓  ←  数据流(MQ + 持久化)                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  告警引擎                                              │  │
│  │  - 增长率告警(§6.1 4 指标)                          │  │
│  │  - 趋势预测(ML 模型,提前 24h 告警)                  │  │
│  │  - 多 app 关联分析(系统级 vs 单 app)                 │  │
│  └───────────────────────────────────────────────────────┘  │
│  ↓  ←  告警                                                  │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  治理动作库(可灰度)                                   │  │
│  │  - am send-trim-memory <pkg> <level>                 │  │

│  │  - am force-stop <pkg> (灰度)                        │  │
│  │  - am kill <pkg> (灰度)                              │  │
│  │  - lmkd 主动配合(oom_score_adj)                       │  │
│  └───────────────────────────────────────────────────────┘  │
│  ↓  ←  灰度下发                                              │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  效果验证                                              │  │
│  │  - 治理后 PSS 变化(必须下降)                        │  │
│  │  - 用户体验(冷启动时间 / 卡顿率)                    │  │

│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**关键设计原则**:
- **数据采集零损耗**(只用 < 0.1% 工具)
- **治理动作可灰度**(先 1% 设备 → 10% → 50% → 100%)
- **效果验证多维度**(不只是 PSS 下降,要看用户体验)

## 7. 总结:架构师视角 5 条 Takeaway

1. **生产环境 80% 的内存治理是"不杀进程下"**——杀进程是终极方案,但**有 4 大约束**(不能杀、杀了复现、多进程、时间窗)
2. **ProcessRecord 5 维 + dumpsys procstats** 是"不杀进程"治理的**核心数据源**——零损耗,可以 7×24h 跑
3. **cgroup v2 节点级隔离**是"不杀进程"治理的**关键手段**——`memory.max/high/min` 三个值组合使用
4. **远程 trimMemory + onTrimMemory() 契约**是"动态梳理"的核心——但**前提是 App 端必须实现**
5. **必须杀 vs 可以拖**的判断要**看 PSS 增长率、Heap 比例、cgroup 触发次数**——单看 PSS 不够,要看趋势

```bash
# 1. 找进程的 cgroup 节点
adb shell cat /proc/<pid>/cgroup | grep '0::' | head -1 | cut -d: -f3

# 2. 查 cgroup 节点实时内存
adb shell cat /sys/fs/cgroup<path>/memory.current

# 3. 查 cgroup 节点历史峰值
adb shell cat /sys/fs/cgroup<path>/memory.peak

# 4. 查 cgroup 节点 PSI
adb shell cat /sys/fs/cgroup<path>/memory.pressure

# 5. 查 cgroup 节点事件
adb shell cat /sys/fs/cgroup<path>/memory.events

# 6. 设置 cgroup 节点软限制
adb shell "echo 400*1024*1024 > /sys/fs/cgroup<path>/memory.high"

# 7. 设置 cgroup 节点硬限制
adb shell "echo 600*1024*1024 > /sys/fs/cgroup<path>/memory.max"

# 8. 设置 cgroup 节点保底
adb shell "echo 1*1024*1024*1024 > /sys/fs/cgroup<path>/memory.min"

# 9. 清理(恢复)
adb shell "echo max > /sys/fs/cgroup<path>/memory.max"
adb shell "echo 0 > /sys/fs/cgroup<path>/memory.high"
adb shell "echo 0 > /sys/fs/cgroup<path>/memory.min"

# 10. 找所有触发过 max 的 cgroup 节点
adb shell "for d in /sys/fs/cgroup/apps.slice/apps/uid_*; do
  if [ \"\$(cat \$d/memory.events | awk '/max/ {print \$2}')\" != \"0\" ]; then
    echo \"\$d: \$(cat \$d/memory.events | awk '/max/ {print \$2}')\"
  fi
done"
```
