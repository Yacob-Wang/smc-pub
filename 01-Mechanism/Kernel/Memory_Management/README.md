# Android 内存架构深度解析（基于 AOSP 17 + android17-6.18）

> **视角**：架构师视角——讲"Android 怎么管理内存"，不讲"工程师怎么排查 bug"
>
> **源码基线**：AOSP `android-17.0.0_r1`（API 37，CinnamonBun）+ Kernel `android17-6.18` GKI
>
> **目标读者**：Android 稳定性高级工程师 / 平台架构师 / Framework 工程师
>
> **历史版本**：原 `MM_v2/` 系列（15 篇）已归档至 `_archive/MM_v2_v1/`

---

## 核心立场

> **本课程不是"工具手册"——讲 dumpsys 怎么读、hprof 怎么分析、杀进程怎么定位。**
> **本课程是"架构指南"——讲 Android 作为一个系统，为什么要这样设计内存管理，5 层架构怎么整体协作，20 年演进的逻辑是什么。**

读完本课程，你应该能：

1. **画出一张 Android 内存管理的全景图**——5 大管理职责 × 5 层物理架构 × N 个子系统的二维矩阵
2. **解释每个子系统"为什么这么设计"**——不只知道"是什么"，更知道"为什么"
3. **看懂跨层协作**——一次 page fault 跨 5 层怎么协作，一次杀进程跨 4 层怎么协作
4. **预判演进方向**——从 20 年演进史中看出 Google 的设计哲学，从而预判 AOSP 18/19 的方向

## 课程地图（5 大管理职责 × 5 层物理架构）

```
                  App        ART       FWK      Kernel mm/    Hardware
                 ──────────────────────────────────────────────────────
  分配            ○         ★         ○         ★             ○
  跟踪            ○         ★         ★         ★             -
  限额            -         ★         ○         ★             -
  保护            -         -         ★         ★             -
  释放            ○         ★         ○         ★             -
```

**矩阵解读**：
- **分配**：ART 负责 Java 堆；Kernel mm/ 负责物理页；Framework 负责调度记账
- **跟踪**：3 层账本（ART / FWK / Kernel）独立维护、协作同步
- **限额**：ART 限额（Java 堆）+ Kernel 限额（cgroup memcg）
- **保护**：Framework 决定杀谁（adj）+ Kernel 决定怎么杀（LMKD / OOM）
- **释放**：4 大释放源协同（trimMemory / GC / kswapd / LMKD）
