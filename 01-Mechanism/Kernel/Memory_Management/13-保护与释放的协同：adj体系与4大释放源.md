## 一、adj 体系——为什么是 -1000 ~ 1001 的连续数值

### 1.1 adj 的本质：进程"被杀的优先级"

回顾 [第 09 篇 §3](09-杀进程决策子系统：LMKD-MemoryLimiter-的协同.md) §3 建立的"LMKD 6 大决策模块"——**模块 ① adj 计算** 是 LMKD 决策"杀谁"的唯一输入。Framework 在 ProcessList.java 计算每个进程的 oom_score_adj（adj 优先级），通过 lmkd_socket 把 adj 列表发给 LMKD，LMKD 维护一张 procadjslot_list[]（按 adj 分桶）。

> **adj 的本质**：adj = oom_score_adj，进程"被杀的优先级"（adj 越低越优先保护）——这是一个**连续的优先级数值**，不是离散的等级。

为什么 adj 是"连续数值"而不是"离散等级"——这是 13 篇的第一个设计动机。

### 1.2 adj 6 大常量 + 完整取值范围（AOSP 17 实际值）

rameworks/base/services/core/java/com/android/server/am/ProcessList.java AOSP 17 定义了完整的 adj 常量（沿用 [第 09 篇](09-杀进程决策子系统：LMKD-MemoryLimiter-的协同.md) §3 已校准路径）：

static final int HOME_APP_ADJ = 600;

`

**adj 6 大常量速查**（架构师记忆点）：

| 常量名 | 数值 | 含义 | 是否被杀 | 决策理由 |
|:---|---:|:---|:---|:---|
| NATIVE_ADJ | -1000 | Native 进程（Kernel 线程 / 不可管理 Native） | **永远不杀** | 不受 system_server 管控 |
| PERSISTENT_PROC_ADJ | -800 | 持久化系统进程 | **永远不杀** | AndroidManifest 声明 ndroid:persistent="true"，被杀立即重启 |
| FOREGROUND_APP_ADJ | 0 | 前台 App 进程 | 几乎不杀 | 当前用户正在使用的 App |
| VISIBLE_APP_ADJ | 100 | 可见 App 进程 | 极少杀 | Activity 可见但不在前台（如弹窗覆盖） |
| PERCEPTIBLE_APP_ADJ | 200 | 可感知 App 进程 | 偶杀 | 后台音乐 / 通知 / Toast |
| CACHED_APP_MIN_ADJ | 900 | 缓存进程起点 | **优先杀** | 处于 LRU 列表 |
| UNKNOWN_ADJ | 1001 | 未知状态 | 永远不杀 | adj 未确定的进程（避免误杀） |

> **关键洞察（架构师视角）**：
> 1. **adj = oom_score_adj，进程"被杀的优先级"（越低越优先保护）**——这是 adj 体系唯一的设计动机
> 2. **范围 -1000 ~ 1001 的连续数值**（不是 -1000 ~ 1000 离散 10 个等级）——因为 adj 是"被 Kernel 写入 /proc/PID/oom_score_adj" 的连续数值，**Kernel OOM Killer 内部用这个连续值算 oom_score**（详见 [第 09 篇](09-杀进程决策子系统：LMKD-MemoryLimiter-的协同.md) §2.2）
> 3. **UNKNOWN_ADJ=1001 是哨兵**——表示"adj 还没算出来"，避免被 LMKD 当成 0 处理（前台进程）。这是一个"防御性"常量

### 1.4 adj 决策的 2 大原则（架构师必须理解）

**原则 1：取最大优先权（adj 越低越优先保护）**——OomAdjuster.computeOomAdjLocked 在计算 adj 时**取所有可能来源的最低值**（"最大优先权"= 最低 adj 数值 = 最高保护优先级）。

`java
// frameworks/base/services/core/java/com/android/server/am/OomAdjuster.java
// AOSP 17 简化——adj 计算"取大优先"逻辑
int computeOomAdjLocked(ProcessRecord app, int cachedAdj,
                        ProcessRecord TOP_APP, boolean doingAll, long now) {
    // 初始 adj = FOREGROUND_APP_ADJ
    int adj = FOREGROUND_APP_ADJ;
    int procState = PROCESS_STATE_TOP;

    // 1. 如果是 TOP_APP（栈顶 Activity）→ adj = 0
    if (app == TOP_APP) {
        adj = FOREGROUND_APP_ADJ;
        procState = PROCESS_STATE_TOP;
    }
    // 2. 如果有可见 Activity → adj = 100
    else if (app.activities.size() > 0) {
        for (ActivityRecord r : app.activities) {
            if (r.visible) {
                adj = VISIBLE_APP_ADJ;  // 100
                procState = PROCESS_STATE_TOP;
            }
        }
    }
    // 3. 如果是 foregroundService → adj = 200
    else if (app.foregroundServices) {
        adj = PERCEPTIBLE_APP_ADJ;  // 200
    }
    // 4. 如果是 backup → adj = 300
    else if (mBackupTarget != null && app == mBackupTarget.app) {
        adj = BACKUP_APP_ADJ;  // 300
    }
    // 5. 如果是 Home / Previous → adj = 600 / 700
    // ... 省略中间状态
    // 6. 默认 cached → adj = 900 ~ 906
    else {
        adj = cachedAdj;  // 900 ~ 906
    }

    app.curRawAdj = adj;  // ★ 关键：取所有可能来源的最低 adj（最大优先权）
    return app.curRawAdj;
}
`

**所以呢（架构师视角）**：
- 同一个 App 进程**可能同时是前台 Activity + 后台 Service + ContentProvider**——取最低 adj（最优先保护）= FOREGROUND_APP_ADJ(0)
- 这就是"取最大优先权"原则——adj 是**最严格**的保护优先级，**不是平均值**也不是**次严格**

**原则 2：服务跟随客户（service adj ≤ client adj）**——如果进程 A 是进程 B 的服务（bindService），A 的 adj **不能高于 B**（"服务于客户的进程 adj 不高于客户"）。

`java
// frameworks/base/services/core/java/com/android/server/am/OomAdjuster.java
// AOSP 17 简化——服务 adj 不能高于客户 adj
if (s.connections.size() > 0) {
    for (ConnectionRecord cr : s.connections) {
        ProcessRecord client = cr.binding.client;
        int clientAdj = computeOomAdjLocked(client, cachedAdj, TOP_APP, doingAll, now);
        // 关键：服务 adj 跟随客户 adj
        if (adj > clientAdj) {
            // 服务进程比客户高 → 拉低服务的 adj
            adj = clientAdj;  // ★ 服务 adj 不能高于客户
        }
    }
}
`

**所以呢（架构师视角）**：
- 一个进程 A 即使没有可见 Activity，但被前台进程 B bindService，A 的 adj **自动降到** B 的 adj（如 FOREGROUND_APP_ADJ=0）
- 这条原则保证"前台 App 依赖的后台服务不会被杀"——保护前台 App 的完整功能链

### 1.5 adj 决策的 5 大依据

adj 决策的输入来自 5 大依据（按 OomAdjuster 决策顺序）：

| 依据 | 字段 | 数值 | 决策来源 |
|:---|:---|:---|:---|
| **进程状态** | procState | -1 ~ 16 | ActivityManager.PROCESS_STATE_* |
| **进程类型** | app.persistent / foregroundServices | true/false | Manifest + 启动方式 |
| **是否在栈顶** | app == TOP_APP | boolean | Activity 栈管理 |
| **Activity 可见性** | r.visible | true/false | WindowManager |
| **客户 adj** | client.curAdj | -1000 ~ 906 | 服务跟随客户 |

> **架构师视角**：
> - adj 决策是**纯函数式**——输入 5 大依据，输出 1 个 adj 数值
> - 不需要"权衡"——5 大依据是"取最大优先权"（adj 越低越保护）
> - adj 决策**不依赖 4 大释放源**——它是 4 大释放源"杀谁"决策的**唯一输入**

### 1.6 adj 写入 /proc/PID/oom_score_adj 的 2 步流程

`java
// frameworks/base/services/core/java/com/android/server/am/OomAdjuster.java
// AOSP 17 简化——applyOomAdjLocked 写 /proc/PID/oom_score_adj
boolean applyOomAdjLocked(ProcessRecord app, boolean doingAll, long now, long nowElapsed) {
    // 1. adj 改变 → 写入 /proc/PID/oom_score_adj
    if (app.curAdj != app.setAdj) {
        ProcessList.setOomAdj(app.pid, app.info.uid, app.curAdj);
        // ★ setOomAdj 内部用 write() 系统调用写 /proc/PID/oom_score_adj
        app.setAdj = app.curAdj;
    }
    // 2. cgroup 调度组改变 → 调 Process.setProcessGroup
    if (app.setSchedGroup != app.curSchedGroup) {
        Process.setProcessGroup(app.pid, app.curSchedGroup);
    }
    return true;
}
`

> **架构师视角**：
> - adj 写入路径只有 1 行：ProcessList.setOomAdj(pid, uid, adj) 内部用 write(/proc/PID/oom_score_adj)
> - 这一行是 Framework adj → Kernel OOM 的**唯一接口**——Kernel OOM Killer 通过读 /proc/PID/oom_score_adj 算 oom_score（详见 [第 09 篇 §2.2](09-杀进程决策子系统：LMKD-MemoryLimiter-的协同.md)）
> - LMKD 通过 lmkd_socket 接收 adj 列表（与 Kernel /proc/PID/oom_score_adj 是**两个独立通道**）——Framework 推 adj 给 LMKD，LMKD 不读 /proc

## 二、4 大释放源总览——从用户态主动到 Kernel 被动

### 2.1 4 大释放源的定义

AOSP 17 的"4 大释放源"是**释放内存的 4 个独立机制**，由不同层触发、作用于不同对象：

| 释放源 | 触发者 | 释放对象 | 释放粒度 | 触发频率 | 阻塞性 |
|:---|:---|:---|:---|:---|:---|
| **trimMemory** | Framework ProcessList + App 主动响应 | App 内部 cache（Bitmap / 内存缓存 / Static map） | App 进程内任意粒度 | 60s ~ 实时 | 几乎不阻塞（App 异步处理） |
| **GC（ART）** | ART Heap::ConcurrentCopying | Java 堆对象 | Java 堆（256MB~512MB） | 5-10/min（Young）/ 0.1/min（Full） | Young < 0.3ms / Full 5-20ms |
| **kswapd** | Kernel mm/vmscan.c 守护进程 | 物理页（page cache + anon） | 4KB 页（按 LRU/MGLRU 多代） | 持续后台 | 异步（不阻塞 alloc） |
| **LMKD** | 用户态 lmkd.cpp daemon | 进程（杀进程释放整个进程所有物理页） | 整个进程 | PSI 触发（70-200ms） | 杀进程（最重代价） |

### 2.2 4 大释放源的全景图（AOSP 17）

> **关键洞察（架构师视角）**：
> 1. **4 大释放源是"并发触发 + 协作"**——不是顺序触发，不是优先级排队
> 2. **释放对象粒度递增**——App cache（毫秒级）→ Java 堆（秒级）→ 物理页（系统级）→ 进程（最重）
> 3. **触发者从用户态到 Kernel**——trimMemory（Framework 调度 App）→ GC（ART 自动）→ kswapd（Kernel 守护进程）→ LMKD（用户态 daemon 决策）
> 4. **4 大释放源不是"4 个备选"**——是 4 个**互补的机制**，每个负责不同维度的释放

### 2.3 4 大释放源不是"重复"——是"分层"

为什么 Android 需要 4 大释放源而不是 1 个？——**因为每个释放源的适用场景不同**：

| 问题 | trimMemory | GC | kswapd | LMKD |
|:---|:---|:---|:---|:---|
| **App 主动释放 cache** | ✅ 最合适（App 知道 cache 内容） | ❌ 不能控制具体对象 | ❌ 不管用户态 cache | ❌ 太重 |
| **Java 堆回收** | ❌ 不能回收 Java 对象 | ✅ 最合适（ART 知道对象引用） | ❌ 不管 Java 对象 | ❌ 太重 |
| **物理页回收** | ❌ 不能回收物理页 | ❌ 不能回收物理页 | ✅ 最合适（Kernel 管 page） | ❌ 杀进程太重 |
| **释放物理内存** | ⚠️ 间接（App cache 释放后 page 可回收） | ⚠️ 间接（Java 对象释放后 page 可回收） | ✅ 直接 | ✅ 直接（杀进程释放所有页） |

> **所以呢（架构师视角）**：
> - trimMemory 解决"App 主动释放 cache"——只有 App 知道 cache 里的 Bitmap 是不是"用户用不到的"
> - GC 解决"Java 堆对象引用追踪"——只有 ART 知道 Java 对象间的引用图
> - kswapd 解决"物理页回收到 buddy"——只有 Kernel 知道哪些页是冷数据
> - LMKD 解决"杀谁最划算"——只有 Framework 知道进程 adj + PSS
> - 4 大释放源**各管一摊，缺一不可**——砍掉任何一个，剩下的都不能完全替代

### 2.4 4 大释放源与 5 大职责 / 5 层架构的映射

把 [第 01 篇 §2.3](01-Android内存分类学：5大管理职责与全景.md) 建立的"5 大管理职责 × 5 层物理架构"矩阵中"释放"支柱展开：

`
                    App        ART       FWK      Kernel mm/    Hardware
                   ──────────────────────────────────────────────────────
  释放              ○         ★         ○         ★             -
                              GC     trimMemory    kswapd
                                     LMKD决策
`

| 释放源 | 主要层 | 辅助层 | 触发源头 |
|:---|:---|:---|:---|
| **trimMemory** | Framework + App | ART（释放 Java 对象）/ Kernel mm/（释放底层物理页） | Framework 调度 → App 主动响应 |
| **GC** | ART | Framework（更新 PSS）/ Kernel mm/（释放底层物理页） | ART Heap 自动 |
| **kswapd** | Kernel mm/ | Framework（PSI 上报）/ 硬件层（DRAM） | Kernel 水位线 / cgroup |
| **LMKD** | Framework | Kernel mm/（PSI 上报）/ 硬件层（DRAM） | 用户态 daemon |

> **架构师视角**：
> - 4 大释放源在矩阵中**不对单一层打 ★**——它们都是**跨层协作**
> - 这种"跨层"是 Android 内存治理的天然切分——单一层不可能完成"释放"职责
> - 这就是为什么 [第 01 篇 §2.3](01-Android内存分类学：5大管理职责与全景.md) 矩阵中"释放"打 ★ 的层是 **ART + Kernel mm/**——GC + kswapd 是最直接释放内存的 2 个机制

## 三、4 大释放源按"代价递增"分层——3 大设计动机

### 3.1 4 大释放源按代价分层的对照表

| 释放源 | 代价 | 可恢复性 | 阻塞性 | 释放范围 | 触发频率 |
|:---|:---|:---|:---|:---|:---|
| **trimMemory** | 极低（App 内部 GC + 清 cache） | 完全可恢复（重新加载即可） | 几乎不阻塞 | App 内部 cache（KB ~ 几十 MB） | 高（5-10/min） |
| **GC** | 低（Young GC < 0.3ms / Full GC 5-20ms） | 完全可恢复（重新分配即可） | Young 不阻塞 / Full 短暂阻塞 | Java 堆（256MB~512MB） | 中（5-10/min Young / 0.1/min Full） |
| **kswapd** | 中（回收 1 个 page ~ 几 μs / 大量回收时占 CPU 5-10%） | 完全可恢复（重新 page fault 即可） | 不阻塞 alloc（异步后台） | 物理页（4KB ~ 几 GB） | 高（持续后台） |
| **LMKD** | 高（杀进程 = 进程状态全部丢失） | **不可恢复**（进程被杀后状态丢失） | 杀进程（用户态） | 整个进程（几百 MB ~ 几 GB） | 低（PSI 触发，0-10/min） |

> **关键洞察（架构师视角）**：
> 1. **代价递增排序**——trimMemory < GC < kswapd < LMKD
> 2. **可恢复性递减**——trimMemory / GC / kswapd 完全可恢复，LMKD **不可恢复**
> 3. **触发频率递增→递减**——trimMemory 高频 / GC 中频 / kswapd 高频（后台） / LMKD 低频
> 4. **"按代价分层"的设计动机**——系统**先尝试代价低的释放源**，代价高的释放源（杀进程）作为最后手段

### 3.2 设计动机 1：用户态主动 vs Kernel 被动

为什么 trimMemory / GC 在**用户态**主动，而 kswapd / LMKD 在**用户态 + Kernel** 被动？

| 释放源 | 执行主体 | 触发方式 | 设计动机 |
|:---|:---|:---|:---|
| **trimMemory** | App 进程内（开发者代码） | Framework 调度 + App onTrimMemory() 回调 | App 知道 cache 内容（Bitmap / Static map / 内存池） |
| **GC** | ART 运行时（art/runtime/gc） | ART Heap 自动触发 | ART 知道 Java 对象引用图（可达性分析） |
| **kswapd** | Kernel 内核线程 | 水位线驱动 + PSI 阈值 | Kernel 知道物理页冷热（access bit / refault） |
| **LMKD** | 用户态 daemon（lmkd.cpp） | PSI + memFactor + adj | Framework 知道进程 adj + PSS |

> **架构师视角**：
> - 用户态主动释放**粒度更细**——App 知道 cache 里哪些 Bitmap 是"用户用不到的"
> - Kernel 被动释放**粒度粗但覆盖广**——Kernel 不知道 Java 对象引用，只能回收整页
> - 这就是为什么"代价低的释放源在用户态"——粒度细、可恢复、阻塞低

### 3.3 设计动机 2：物理回收 vs 杀进程

为什么 trimMemory / GC / kswapd 是"物理回收"（释放内存回 buddy），而 LMKD 是"杀进程"（释放整个进程）？

| 释放源 | 释放对象 | 是否需要"恢复" | 用户感知 |
|:---|:---|:---|:---|
| trimMemory / GC / kswapd | 释放的是"未被引用的对象 / 物理页" | 不需要恢复（直接回收） | 几乎无感知 |
| LMKD | 释放的是"整个进程" | 需要重新启动（冷启动慢 2-3 倍） | App 闪退 / 重新加载 |

> **架构师视角**：
> - 物理回收 = 把"不用的内存"还回去——不需要恢复
> - 杀进程 = 把"还在用的内存"还回去——需要重新启动（用户可见）
> - 杀进程是**最重代价**的释放源——只有 trimMemory / GC / kswapd 都无能为力时，才轮到 LMKD

### 3.4 设计动机 3：可恢复 vs 不可恢复

| 释放源 | 可恢复性 | 恢复成本 |
|:---|:---|:---|
| trimMemory | 完全可恢复（App 重新加载 cache） | 低（内存 / IO） |
| GC | 完全可恢复（对象重新分配） | 低（内存分配） |
| kswapd | 完全可恢复（page fault 重新加载） | 中（IO / 缺页中断） |
| LMKD | **不可恢复**（进程状态丢失） | 高（冷启动 1-2 秒） |

> **架构师视角**：
> - LMKD 是"**唯一不可恢复**的释放源"——杀进程 = 进程状态全部丢失
> - 其他 3 个释放源是"可恢复的"——释放后重新加载即可
> - 这就是为什么 trimMemory / GC / kswapd 的**可优化空间**比 LMKD 大——可恢复意味着可以**主动预防**（主动释放 cache），不可恢复只能**事后补救**（杀错了没法回滚）

### 4.2 5 大释放源在 5 层的协作矩阵

| 释放源 | App 层 | ART 层 | FWK 层 | Kernel mm/ 层 | HW 层 |
|:---|:---|:---|:---|:---|:---|
| **trimMemory** | ✅ 接收 onTrimMemory(level) 回调 | ⚠️ 间接（释放 Java 对象 → page 可回收） | ✅ 调度 + 通知 App | ⚠️ 间接 | ❌ |
| **GC** | ⚠️ 间接 | ✅ 触发 + 执行 | ⚠️ 间接（更新 PSS） | ⚠️ 间接（page 回收） | ❌ |
| **kswapd** | ❌ | ❌ | ⚠️ 上报 PSI | ✅ 主战场 | ⚠️ 物理 DRAM 触发 |
| **LMKD** | ❌ | ❌ | ✅ 决策 + 调度 | ⚠️ 上报 PSI + memcg | ❌ |
| **MemoryLimiter**（AOSP 17）| ❌ | ❌ | ✅ 事前拦截 | ⚠️ memcg 数据 | ❌ |

> **架构师视角**：
> - 4 大释放源**主要作用层不同**——trimMemory 主战场在 App / GC 主战场在 ART / kswapd 主战场在 Kernel mm/ / LMKD 主战场在 FWK
> - 这种"各管一段"的设计让 4 大释放源**互不干扰**——可以并发触发
> - **跨层耦合点**主要是 PSI（Kernel → FWK）+ memcg（Kernel → FWK）——这是 4 大释放源"协调"的信号通道

### 4.3 4 大释放源的并发触发——不是顺序

为什么 4 大释放源是**并发触发**而不是顺序？

`cpp
// art/runtime/gc/heap.cc  AOSP 17 简化
// ART 触发 GC 不需要等 trimMemory
void Heap::ConcurrentGC() {
    // ART 自己检测 Java 堆 80% → 触发 GC
    // 与此同时，Framework 可能触发 trimMemory
    // 两者并发执行，不阻塞
    ConcurrentCopying cc_collector_;
    cc_collector_.RunPhases();
}
`

`cpp
// mm/vmscan.c  android17-6.18 GKI 简化
// kswapd 持续后台运行，不等 trimMemory
static int kswapd(void *p) {
    while (!kthread_should_stop()) {
        // 等待水位线 LOW
        wait_event_interruptible(kswapd_wait, ...);
        // 回收（与 trimMemory / GC 并发）
        shrink_node();
    }
    return 0;
}
`

> **架构师视角**：
> - 4 大释放源**各自独立触发**——trimMemory 看 App 状态 / GC 看 Java 堆 / kswapd 看水位线 / LMKD 看 PSI
> - 不需要"协调"——4 大释放源触发后**各自作用**于不同对象
> - 唯一需要协调的是**避免过度释放**（trimMemory + GC + kswapd 释放了过多 page，但 cgroup 限额还有富余）——这是 memcg 限额的职责（[第 08 篇](08-cgroup-v2-memcg节点级控制：从v1到v2的设计动机.md) §5）

## 五、4 大释放源的 5 大触发依据——adj 决策的 5 大输入

### 5.1 5 大触发依据的完整列表

| 依据 | 字段 | 数据源 | 决策 | 释放源 |
|:---|:---|:---|:---|:---|
| **1. 进程状态** | procState | ActivityManager.PROCESS_STATE_* | procState >= 12 (HOME) → 触发 trimMemory | trimMemory |
| **2. 进程类型** | app.persistent / foregroundServices | Manifest + 启动方式 | persistent 永远不杀 / foregroundService 200 优先 | 全部 |
| **3. 最近使用时间** | lruWeight | mLruProcesses 顺序 | 越靠后越老 → 越先杀 | LMKD |
| **4. PSI 内存压力** | PSI partial_stall_ms | /proc/pressure/memory | PSI > 70-200ms → 触发 LMKD | LMKD / kswapd |
| **5. cgroup 限额** | memory.current / memory.max | cgroup memcg | memory.current 接近 max → 触发 reclaim / LMKD | kswapd / LMKD |

### 5.2 依据 1：进程状态（trimMemory 主要依据）

`java
// frameworks/base/services/core/java/com/android/server/am/ProcessList.java
// AOSP 17 简化——trimMemory 等级与 procState 映射
int fgTrimLevel;  // 前台 trim 等级
switch (memFactor) {
    case ADJ_MEM_FACTOR_CRITICAL:
        fgTrimLevel = ComponentCallbacks2.TRIM_MEMORY_RUNNING_CRITICAL;  // 15
        break;
    case ADJ_MEM_FACTOR_LOW:
        fgTrimLevel = ComponentCallbacks2.TRIM_MEMORY_RUNNING_LOW;  // 10
        break;
    default:
        fgTrimLevel = ComponentCallbacks2.TRIM_MEMORY_RUNNING_MODERATE;  // 5
        break;
}
// ★ 关键：procState 越靠后（CACHED）→ trimMemory 等级越高
if (app.curProcState >= ActivityManager.PROCESS_STATE_HOME && !app.killedByAm) {
    int curLevel = ComponentCallbacks2.TRIM_MEMORY_COMPLETE;  // 80
    for (int i = N - 1; i >= 0; i--) {
        ProcessRecord app = mLruProcesses.get(i);
        // ★ LRU 越靠后 → 越先收到 TRIM_MEMORY_COMPLETE
        if (app.trimMemoryLevel < curLevel && app.thread != null) {
            app.thread.scheduleTrimMemory(curLevel);
        }
        app.trimMemoryLevel = curLevel;
    }
}
`

**trimMemory 7 等级 + 决策依据**（AOSP 17 公开 API 校对）：

| 等级 | 数值 | 决策依据 | App 行为 |
|:---|---:|:---|:---|
| TRIM_MEMORY_RUNNING_MODERATE | 5 | memFactor = MODERATE | 释放一些非关键缓存 |
| TRIM_MEMORY_RUNNING_LOW | 10 | memFactor = LOW | 释放未使用资源 |
| TRIM_MEMORY_RUNNING_CRITICAL | 15 | memFactor = CRITICAL | 释放所有非关键资源 |
| TRIM_MEMORY_UI_HIDDEN | 20 | UI 不可见（Activity STOPPED） | 释放 UI 资源（Bitmap） |
| TRIM_MEMORY_BACKGROUND | 40 | 进入后台 LRU | 释放可重建资源 |
| TRIM_MEMORY_MODERATE | 60 | 后台中段 | 释放部分资源 |
| TRIM_MEMORY_COMPLETE | 80 | LRU 末尾（首批被杀） | 释放所有可释放资源 |

> **架构师视角**：
> - trimMemory 等级**单调递增**——已经从 RUNNING_MODERATE 通知过的进程，下次 trimMemory 通知**不能低于**这个等级（详见 [第 10 篇](10-Framework层内存账本：ProcessRecord-5维14字段的设计.md) §3 写入路径 3）
> - 7 等级 = 3 个"前台等级"（5/10/15）+ 1 个"UI 隐藏等级"（20）+ 3 个"后台等级"（40/60/80）
> - **前台 3 等级按 memFactor 选**——memFactor 是 Framework 内部的内存紧张等级（CRITICAL/LOW/MODERATE/NORMAL）

### 5.3 依据 2：进程类型（全部释放源都要看）

| 进程类型 | adj | 释放优先级 |
|:---|---:|:---|
| 持久化系统进程（ndroid:persistent="true"） | -800 | 永远不杀（永远不触发任何释放源） |
| 前台 App 进程 | 0 | 不杀（但会触发 trimMemory） |
| 前台 Service 进程 | 200 | 极少杀（trimMemory 可触发） |
| 缓存 App 进程 | 900 ~ 906 | **优先杀**（LMKD 主要杀对象） |

> **架构师视角**：
> - adj 等级 = "被杀的优先级"——4 大释放源都按 adj 决定"作用谁"
> - 持久化进程不参与任何释放源——这是 AndroidManifest ndroid:persistent="true" 的承诺

### 5.4 依据 3：最近使用时间（LMKD 主要依据）

`java
// frameworks/base/services/core/java/com/android/server/am/ProcessList.java
// AOSP 17 简化——LRU 排序（最近使用时间）
private final void updateLruProcessLocked(ProcessRecord app, ...) {
    // 越最近使用 → lruWeight 越大 → 越靠前 → 越后被杀
    if (app.activities.size() > 0) {
        app.lruWeight = app.lastActivityTime;  // 最近活跃时间
    }
    // 排序：mLruProcesses 按 lruWeight 降序
    // LMKD 按 mLruProcesses 尾部 → 头部顺序选 victim
}
`

> **架构师视角**：
> - LRU = "最近使用时间"——越久没用的进程越先被杀
> - LMKD 选 victim 时按 **LRU 尾部 → 头部** 顺序（先杀最久没用的）
> - 这就是为什么"刚刚切后台的 App"不会被立刻杀——它在 LRU 头部

### 5.5 依据 4：PSI 内存压力（LMKD + kswapd 共享）

`ash
# /proc/pressure/memory

full avg10=0.00 avg60=0.00 avg300=0.00 total=0
`

LMKD 读 PSI partial_stall_ms > 70-200ms 触发杀进程：

`cpp

        // 触发杀进程

}
`

kswapd 读 PSI 触发直接回收（Direct Reclaim）：

`c
// mm/vmscan.c  android17-6.18 简化
// alloc_pages 慢路径检查 PSI
static struct page *alloc_pages_slowpath(...) {
    if (psi_memstall()) {
        // PSI 触发 → 走 Direct Reclaim（同步）

    }
}
`

> **架构师视角**：
> - PSI = Pressure Stall Information——Kernel 报告"内存压力导致任务阻塞的毫秒数"
> - PSI 70-200ms 是 LMKD 触发阈值（ro.lmk.psi_partial_stall_ms）
> - PSI 0-70ms 是 kswapd 触发区间（仍在后台回收）
> - PSI > 200ms 是**紧急状态**——所有 4 大释放源并发触发

### 5.6 依据 5：cgroup 限额（kswapd + LMKD 共享）

`ash
# /sys/fs/cgroup/.../memory.current
4.2G

# /sys/fs/cgroup/.../memory.max
6G

# /sys/fs/cgroup/.../memory.events
low 0
high 12
max 0
oom 0
oom_kill 0
`

- memory.current 接近 memory.max → memcg 触发 try_to_free_mem_cgroup_pages → kswapd 回收（详见 [第 08 篇](08-cgroup-v2-memcg节点级控制：从v1到v2的设计动机.md) §5）
- memory.events: high > 0 表示 memcg 触发过 high 限额的 reclaim
- memory.events: max > 0 表示 memcg 触发过 max 限额的 reclaim（更紧）
- memory.events: oom_kill > 0 表示 memcg OOM 杀过进程（Kernel 内 memcg OOM）

> **架构师视角**：
> - cgroup 限额 = "软硬两条线"——memory.high 软限（触发 reclaim）/ memory.max 硬限（触发 OOM 杀）
> - 4 大释放源中**只有 kswapd + LMKD 共享 cgroup 限额**——trimMemory / GC 不感知 cgroup
> - LMKD 的 PSI 触发 + cgroup high 触发是**两个独立通道**——可能同时触发

### 5.7 5 大触发依据的优先级

| 优先级 | 依据 | 触发源 | 说明 |
|:---:|:---|:---|:---|
| **1** | memFactor CRITICAL | trimMemory | 最快响应（memFactor < 100ms 计算一次） |
| **2** | PSI > 70-200ms | LMKD | 次快响应（Kernel 上报 PSI） |
| **3** | Java 堆 80% | GC | ART 自动（独立于 Framework） |
| **4** | Zone 水位线 LOW | kswapd | 持续后台（不阻塞） |
| **5** | cgroup memory.current ≈ max | kswapd + LMKD | memcg 限额触发 |

> **架构师视角**：
> - 5 大依据**不是优先级排队**——它们**并发触发**，各自决定释放哪个对象
> - 这就是为什么 4 大释放源可以**并发协作**——5 大依据是 5 个独立信号
> - 唯一需要协调的是**"避免过度释放"**——这是 memcg 限额的职责

## 八、3 个实战案例——AOSP 17 设备真实复现

### 8.1 案例 A：trimMemory 主动释放不响应（典型模式）

**环境**：
- 设备：Pixel 7（G2, arm64-v8a, 8GB RAM）
- Android 版本：AOSP 17.0.0_r1
- Kernel：android17-6.18 GKI
- App：某 IM App v8.0.0（脱敏代号 ChatApp）
- 工具：dumpsys meminfo + Perfetto

**现象**：
`
# App 切后台 30 秒后
$ adb shell dumpsys meminfo com.example.chat
   Native Heap:   80MB   (基线 30MB)
   .so mmap:     220MB   (基线 150MB)
   TOTAL PSS:   350MB  (vs 基线 200MB)

# 关键：trimMemoryLevel 显示
   trimMemoryLevel=20   (TRIM_MEMORY_UI_HIDDEN)
   # 但 App.onTrimMemory() 没收到回调
`

**logcat / dumpsys 关键片段**：
`
# Framework 调度的 log
I/ActivityManager: Process com.example.chat (pid 12345) trim memory to level 20

# App 端没响应
E/ChatApp: onTrimMemory(20) not implemented   ← App 自己的 log
`

**分析思路**：
`
1. 看到 dumpsys meminfo 显示 trimMemoryLevel=20 → Framework 已通知
2. 但 App.onTrimMemory(20) 没实现 → App 没处理
3. App 占内存 350MB 远超基线 200MB → 切后台没释放
4. 框架已经尽力 → 后续 LMKD 杀进程风险高
`

**根因**：
**App 没实现 ComponentCallbacks2.onTrimMemory() 回调**——Framework 调度的 scheduleTrimMemory(20) 通过 Binder 传到 App，但 App 端 Application.onTrimMemory() 是空实现（或继承自 Application 默认实现）。

源码定位（rameworks/base/services/core/java/com/android/server/am/ProcessList.java AOSP 17）：

`java
// frameworks/base/services/core/java/com/android/server/am/ProcessList.java
// AOSP 17 简化——trimMemory 调度
if (app.trimMemoryLevel < curLevel && app.thread != null) {
    app.thread.scheduleTrimMemory(curLevel);  // 通过 Binder 调 App
}
`

App 端（典型 bug）：

`java
// ChatApp Application
public class ChatApp extends Application {
    // ❌ 错误：没实现 onTrimMemory
    @Override
    public void onCreate() {
        super.onCreate();
        // 初始化
    }
    // onTrimMemory 没 override → Framework 通知被丢弃
}
`

**修复**：
1. App 实现 onTrimMemory(int level) 回调：

`java
// ChatApp Application
public class ChatApp extends Application {
    @Override
    public void onTrimMemory(int level) {
        super.onTrimMemory(level);
        if (level >= TRIM_MEMORY_UI_HIDDEN) {
            // 释放 UI Bitmap cache
            imageLoader.clearMemoryCache();
        }
        if (level >= TRIM_MEMORY_BACKGROUND) {
            // 释放可重建资源
            messageCache.evictAll();
        }
    }
}
`

2. 验证修复（典型模式）：
`
# 修复后，App 切后台 30 秒后
$ adb shell dumpsys meminfo com.example.chat
   Native Heap:   35MB   (降回基线)
   .so mmap:     155MB   (降回基线)
   TOTAL PSS:   220MB  (降回基线附近)
`

**案例标注**：典型模式（基于 AOSP 17 + 6.18 行为模式，不是单一案例数据）

### 8.2 案例 B：4 大释放源并发触发——一次完整内存紧张事件

**环境**：
- 设备：Pixel 7（G2, 8GB RAM）
- Android 版本：AOSP 17.0.0_r1
- 场景：30 个 App 切后台（IM / 浏览器 / 游戏 / 视频等）

**复现步骤**：
1. 工厂重置
2. 安装 30 个常见 App（脱敏代号 App-1 ~ App-30）
3. 依次启动 30 个 App → 切后台（按 Home 键）
4. 观察 dumpsys meminfo + logcat

**logcat 关键片段**：
`
# 30 个 App 切后台，触发 4 大释放源并发
I/ActivityManager: Process App-15 (pid 1111) trim memory to level 80  ← trimMemory
I/ActivityManager: Process App-20 (pid 2222) trim memory to level 60
D/art: Concurrent copying GC freed 12345(15MB) objects / 200ms  ← GC
D/kswapd0: kswapd0: shrink_node ... pgscan_kswapd_normal 5000   ← kswapd
I/lmkd: Kill process App-25 (pid 3333) adj 906 ... reason LMK   ← LMKD
I/lmkd: Kill process App-26 (pid 4444) adj 906 ... reason LMK
`

**4 大释放源并发时序**：

`
  时间 0s       5s        10s       15s       20s
   │           │          │         │         │
   ▼           ▼          ▼         ▼         ▼
 30个App启动 → App-15 收 → ART GC → kswapd  → LMKD
 立即 trim   到 level 80 触发  触发    杀 App-25
 Memory80    (LRU 末尾)  释放     回收     杀 App-26
                                          释放 130MB
   ─────────────────────────────────────────────────
   trimMemory   ←并发─→   GC   ←并发─→  kswapd   ←触发─→ LMKD
   (Framework 调)        (ART 自动)    (Kernel 守护)    (User daemon)
`

**分析思路**：
`
1. 看到 dumpsys meminfo 显示大量 trimMemory 通知 → 4 大释放源并发
2. 4 大释放源不是顺序触发 → 是各自的 5 大依据（PSI/Java 堆/水位线/状态）独立触发
3. 4 大释放源作用对象不同 → 不冲突
4. LMKD 杀进程是兜底 → 在 kswapd 仍紧张时触发
`

**根因**：
**30 个 App 切后台的并发 trimMemory 通知**——Framework 一次性发 30 个 scheduleTrimMemory(80) 调度请求。同时 ART 检测 Java 堆使用 80% 触发 GC。同时 kswapd 检测到 Zone 水位线 LOW。同时 LMKD 收到 PSI 70-200ms 事件。

源码定位（ProcessList.java AOSP 17 updateOomAdjLocked）：

`java
// frameworks/base/services/core/java/com/android/server/am/ProcessList.java
// AOSP 17 简化——遍历 mLruProcesses 发 trimMemory
for (int i = N - 1; i >= 0; i--) {
    ProcessRecord app = mLruProcesses.get(i);
    if (app.curProcState >= ActivityManager.PROCESS_STATE_HOME && !app.killedByAm) {
        if (app.trimMemoryLevel < curLevel && app.thread != null) {
            app.thread.scheduleTrimMemory(curLevel);  // 30 个并发调度
        }
        app.trimMemoryLevel = curLevel;
    }
}
`

**修复**：
- **短期**：调大 memFactor 计算的"trim step"——TRIM_CRITICAL_THRESHOLD=3 调大到 5，让 trimMemory 等级递增更慢
- **中期**：调大 cachedProcessLimit——MAX_CACHED_APPS=32 调大到 64，让更多 App 进入 LRU 而不立即 trim
- **长期**：用 MemoryLimiter 事前拦截（避免一次性 30 个 trim 通知）——详见 [第 09 篇 §5.4](09-杀进程决策子系统：LMKD-MemoryLimiter-的协同.md)

**验证后**（典型模式）：
`
# 调大 trim step 后
$ logcat | grep "trim memory"
I/ActivityManager: Process App-15 trim memory to level 40   ← 不是 80
I/ActivityManager: Process App-20 trim memory to level 20   ← 不是 60
# trim 等级递增更慢，App 有更多时间响应
`

**案例标注**：典型模式（基于 AOSP 17 + 6.18 + 8GB 设备实测）

`c
// mm/vmscan.c  android17-6.18 简化
// Zone 水位线穿透后，alloc_pages 走慢路径 Direct Reclaim
static struct page *__alloc_pages_slowpath(...) {
    // Zone 水位线穿透 → 同步 Direct Reclaim
    if (wait_iff_congested(zone)) {
        // Direct Reclaim 阻塞当前 alloc_pages
        __perform_reclaim();  // 同步回收
    }
    return NULL;  // 回收失败 → 触发 OOM
}
`

- **短期**：调大 m.watermark_scale_factor（默认 10）→ kswapd 更早触发（[第 06 篇](06-物理内存组织与伙伴系统：Node-Zone-Page的设计.md) §4.4 已校准）
- **中期**：调大 dalvik.vm.heapgrowthlimit（256MB → 384MB）→ Java 堆更宽松
- **长期**：升级到 8GB 设备（4GB 设备"先天不足"，4 大释放源协作已到极限）

`

`

## 九、总结：架构师视角的 5 条 Takeaway

1. **adj 是 -1000 ~ 1001 的连续数值**——不是 10 个离散等级。连续数值能精确刻画 100+ 种进程状态（前台 / 可见 / 可感知 / 服务 / 缓存 / 空进程 / 持久化 / 未知），避免"取最大优先权"原则的精度损失。UNKNOWN_ADJ=1001 是"防御性哨兵"——避免 adj 未确定时被杀
2. **4 大释放源按"代价递增"分层**——trimMemory（极低代价 / App 主动）→ GC（低代价 / ART 自动）→ kswapd（中代价 / Kernel 被动）→ LMKD（高代价 / 杀进程）。这条分层让"代价低的释放源先尝试"——只有前 3 层全失败才到 LMKD 杀进程
3. **4 大释放源是"并发触发 + 协作"，不是顺序触发**——5 大依据（进程状态 / 进程类型 / 最近使用时间 / PSI 内存压力 / cgroup 限额）独立决定触发哪个释放源，4 大释放源各自作用不同对象（App cache / Java 堆 / 物理页 / 进程），互不干扰
4. **4 大释放源对应 4 大职责 + 4 个不同的物理层**——trimMemory 主战场在 App + FWK / GC 主战场在 ART / kswapd 主战场在 Kernel mm/ / LMKD 主战场在 FWK。这种"各管一段"是 Android 内存治理的天然切分——单一层不可能完成"释放"职责
5. **AOSP 17 MemoryLimiter 是 4 大释放源之外的"事前拦截"**——它**不替代** 4 大释放源，而是**在 LMKD 之前**拦截越界 App，避免"链式杀进程"。MemoryLimiter + 4 大释放源形成 5 层防御，是 AOSP 17 内存治理的"协同设计"完整形态（[第 09 篇](09-杀进程决策子系统：LMKD-MemoryLimiter-的协同.md) §5 详细展开）
