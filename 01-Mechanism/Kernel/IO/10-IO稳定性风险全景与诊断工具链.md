#### §0 锚点案例的可验证 4 件套:线上 ANR 5min 定位演练:从 dumpsys 到 IO 子系统的完整路径

> **环境**:
> - 设备:某线上用户上报 Pixel 7(G2, arm64-v8a, 8GB RAM, UFS 3.1)
> - Android 版本:AOSP `android-14.0.0_r1`
> - Kernel:`android14-5.15` GKI
> - App:某 IM App(脱敏代号 `ChatApp`,ANR 时在主线程)
> - 工具:`adb bugreport` + `dumpsys input` + `simpleperf` + `perfetto` + `iostat`(事后回放)

> **复现步骤**(线上用户 ANR,事后排查):
> 1. 拿到 bugreport → 解压,`grep -r "ANR" bugreport/` 找 ANR 文件
> 2. `cat bugreport/anr/anr_2026-XX-XX-XX.txt` 找到 ANR 主线程 trace
> 3. 主线程 state=D + waiting on `<...>` 等待 → 怀疑 IO 阻塞
> 4. `cat bugreport/proc/<pid>/stack` 验证 D 状态
> 5. `dumpsys input | grep -A 20 "Recent Events"` 看输入分发延迟
> 6. 5 分钟定位:ANR 类型 → 主线程阻塞点 → IO 链路 → 根因 + 修复

## 一、背景与定义：为什么需要 IO 风险全景 + 工具链

### 1.1 稳定性架构师排查 IO 问题的痛点

**没有系统化工具链时的痛点**：

```
现象：App 启动慢
  ↓
看代码？看不出问题
看 systrace？信息太杂
看 iostat？不知道哪个指标关键
看 logcat？没看到明显错误
看 ANR trace？知道是 IO，但不知道是哪类 IO
  ↓
浪费 2 小时定位 → 终于找到"是 FUSE daemon 卡死"
```

**痛点的根因**：
1. **风险分类不清晰**——不知道"卡顿"属于 4 类 IO 问题中的哪一类
2. **工具不熟悉**——知道有 iostat / blktrace / Perfetto，但不知道什么时候用哪个
3. **指标不明确**——不知道 iostat 的 `await` / `r_await` / `w_await` 含义
4. **没有排查路径速查**——每次都从头想

| **① 风险地图**：4 类 IO 稳定性问题速查表 | §3-§6 |

## 三、风险 ① IO hang 速查

### 3.1 IO hang 的 4 大子类

| 子类 | 触发 | 占比 | 排查入口 |
|------|------|------|---------|
| **Page Cache IO** | wait_on_page_bit | 60-70% | ANR trace 主线程栈帧 |
| **direct reclaim** | shrink_inactive_list | 10-15% | kswapd 栈帧 |
| **inode lock** | __wait_on_inode | 10-15% | inode 操作栈帧 |
| **高阶页分配** | __alloc_pages_slowpath | 5-10% | alloc_pages 栈帧 |

### 3.2 IO hang 的 5 分钟排查路径

```
ANR 触发（D 状态 / IO hang）
  ↓
① 抓 ANR trace → 看主线程栈帧
  ↓
② 栈帧在哪一层？
  ├── io_schedule + wait_on_page_bit → Page Cache IO → 检查设备 IO
  ├── balance_dirty_pages + io_schedule → 写卡顿 → 检查 dirty_ratio
  ├── throtl_schedule → cgroup 限速 → 检查 cgroup weight
  ├── shrink_inactive_list → reclaim → 检查内存压力
  └── mutex_lock → 上游等待链 → 找原始阻塞点
  ↓
③ dmesg | grep "blocked for more than"
  ↓
④ hung_task 检测是否触发？
  ↓
⑤ 治理：调参 / 优化应用 / 升级 kernel
```

详见 [06-IO 与进程](06-IO与进程的深度耦合：D状态、iowait、IO-hang、进程阻塞.md) §11 案例 1。

## 四、风险 ② IO 延迟抖动速查

### 4.1 延迟抖动的 5 大子类

| 子类 | 触发 | 排查工具 |
|------|------|---------|
| **设备层延迟** | UFS thermal / GC / Sleep 唤醒 | iostat await + Perfetto |
| **调度器延迟** | writes_starved / fifo_batch 配置错 | blktrace + sysfs scheduler |
| **Block 层延迟** | tag 耗尽 / merge 失败 | debugfs block + blktrace |
| **cgroup 限流** | cgroup bps/iops 限制 | blk-throttle debug |
| **Page Cache miss** | 冷启动 / 低内存 | ftrace + proc/vmstat |

### 4.2 延迟抖动的 5 分钟排查路径

```
应用响应慢（怀疑 IO）
  ↓
① iostat -xz 1 → 看 await / r_await / w_await / %util
  ↓
② await > 10ms？ → 看是 r_await 还是 w_await
  ├── r_await 高 → 读慢 → 检查设备 / 调度器 / cgroup 读限流
  └── w_await 高 → 写慢 → 检查 balance_dirty_pages / 写 IO 设备
  ↓
③ %util > 95% → 设备打满 → 看 blktrace
  ↓
④ Perfetto IO events → 看具体哪个 IO 慢
  ↓
⑤ 治理：调调度器 / 调 cgroup / 优化应用 / 优化设备性能
```

### 4.3 关键的延迟指标解读

```
iostat 输出解读：
await：所有 IO 平均延迟（ms）包括排队
r_await：读 IO 平均延迟（ms）
w_await：写 IO 平均延迟（ms）
svctm：服务时间（已废弃，不准）
%util：设备利用率

健康值（UFS 3.1）：
├── await < 5ms（正常）
├── await 5-10ms（监控）
├── await > 10ms（异常）
├── await > 100ms（严重卡顿）
└── await > 1s（IO hang）

%util 健康值：
├── < 70%：正常
├── 70-90%：监控
├── > 95%：异常（设备打满）
└── 注意：%util 不等于"磁盘忙"，可能是 IO 队列拥塞
```

## 五、风险 ③ IO 资源耗尽速查

### 5.1 资源耗尽的 4 类

| 资源 | 限制 | 现象 |
|------|------|------|
| **磁盘空间** | 分区大小 | `ENOSPC: No space left on device` |
| **fd 数** | `RLIMIT_NOFILE` | `EMFILE: Too many open files` |
| **inode 数** | 文件系统 | `ENOSPC: No space left on device`（其实是 inode）|
| **bio / tag** | blk-mq tag | `BLK_STS_RESOURCE` |

### 5.2 资源耗尽的排查路径

```
写入失败 / 应用 IO 异常
  ↓
① ENOSPC？ → df -h 检查所有挂载点
  ├── /data 满？→ 清理 app 缓存
  ├── /cache 满？→ 清理 system cache
  └── /system 满？→ 严重（vendor 异常）
  ↓
② EMFILE？ → ulimit -n 检查 / /proc/<pid>/limits
  ├── fd 泄漏？ → lsof | grep <pid> | wc -l
  └── 单进程 fd 过多？ → 优化 app
  ↓
③ BLK_STS_RESOURCE？ → 检查 blk-mq tag
  └── 监控 /sys/kernel/debug/block/*/tags
```

## 六、风险 ④ IO 优先级反转速查

### 6.1 优先级反转的 4 类

| 反转类型 | 触发 | 排查 |
|---------|------|------|
| **CPU 优先 → IO 阻塞** | 高 nice 进程执行大量 IO | 检查 ioprio |
| **IO 优先 → CPU 阻塞** | 高 ioprio 进程等 CPU | 检查 nice vs ioprio |
| **cgroup 反转** | cgroup IO 权重 < CPU 权重 | 检查 cgroup io.weight |
| **FUSE 反转** | FUSE daemon 被拖累 | 检查 sdcard daemon |

### 6.2 优先级反转的排查路径

```
应用响应慢但磁盘不忙
  ↓
① 看 cgroup IO 流量 → cat /sys/fs/cgroup/<cgroup>/io.stat
  ├── background IO > top-app IO？ → cgroup 权重错
  └── top-app IO < background IO？ → 同上
  ↓
② 看 blk-throttle debug → 限速情况
  ↓
③ 检查 oom_adj vs ioprio 一致性
  ↓
④ 治理：调 cgroup weight / 升级 Android 14（自动协同）
```

详见 [04-IO 优先级](04-IO优先级与cgroup-IO控制器.md) §11-12 案例。

## 八、诊断工具链：内核态指标（持续监控）

### 8.1 /proc/diskstats（块设备流量统计）

```bash
# 查看所有块设备的 IO 统计
cat /proc/diskstats
# 字段：major minor name reads_completed reads_merged sectors_read time_reading 
#       writes_completed writes_merged sectors_written time_writing io_in_progress 
#       time_doing_io weighted_time_doing_io

# 示例输出：
#   8       0 sda 12345 100 567890 5678 2345 50 123456 2345 0 8023 8023
#   major: 8（SCSI 设备）
#   minor: 0
#   reads_completed: 12345 次
#   time_reading: 5678 ms
#   writes_completed: 2345 次
#   time_writing: 2345 ms
#   io_in_progress: 0（当前正在进行的 IO）
#   weighted_time_doing_io: 8023 ms（重要：队列加权时间）

# 衍生指标：
#   await = weighted_time / (reads + writes)
#   r_await = time_reading / reads
#   w_await = time_writing / writes
```

### 8.2 /sys/block/<dev>/queue/*（调度器参数与状态）

```bash
# 查看 / 修改调度器
cat /sys/block/sda/queue/scheduler
# [mq-deadline] kyber bfq none

# 查看调度器参数（mq-deadline）
cat /sys/block/sda/queue/iosched/*
# writes_starved: 2
# fifo_batch: 16
# read_expire: 500
# write_expire: 5000

# 物理参数
cat /sys/block/sda/queue/rotational
# 0（非旋转，SSD/UFS）

cat /sys/block/sda/queue/nr_requests
# 64（队列深度）

cat /sys/block/sda/queue/read_ahead_kb
# 128（预读窗口）
```

### 8.3 /proc/pressure/io（IO 压力监控，Linux 4.20+）

```bash
cat /proc/pressure/io
# some avg10=0.20 avg60=0.15 avg300=0.10 total=...
# full avg10=0.05 avg60=0.04 avg300=0.03 total=...

# some: 至少一个任务在 IO 等待的时间占比
# full: 所有任务同时在 IO 等待的时间占比

# 告警阈值：
#   some > 30% → IO 压力大
#   full > 5% → IO 严重等待
```

### 8.4 cgroup IO 统计

```bash
# cgroup v1
cat /sys/fs/cgroup/blkio/<cgroup>/blkio.throttle.io_service_bytes
cat /sys/fs/cgroup/blkio/<cgroup>/blkio.throttle.io_serviced

# cgroup v2
cat /sys/fs/cgroup/<cgroup>/io.stat
cat /sys/fs/cgroup/<cgroup>/io.pressure

# cgroup v2 的 io.pressure 是关键 PSI 指标
```

### 8.5 /proc/<pid>/io（进程级 IO 统计）

```bash
cat /proc/<pid>/io
# rchar: 累计 read() 调用字节（包括 Page Cache 命中）
# wchar: 累计 write() 调用字节
# syscr: read() 调用次数
# syscw: write() 调用次数
# read_bytes: 实际磁盘读字节（Page Cache miss 部分）
# write_bytes: 实际磁盘写字节（脏页回写部分）
# cancelled_write_bytes: 被覆盖的脏字节

# 关键派生指标：
#   Page Cache 命中率 ≈ 1 - read_bytes / rchar
#   平均单次 read 字节 = rchar / syscr
#   平均单次 write 字节 = wchar / syscw
```

## 九、诊断工具链：性能分析工具（深度排查）

### 9.1 iostat（系统级 IO 概览）

```bash
# 安装：sysstat 包（Android 通过 toybox / busybox）
# 基础用法
iostat -xz 1
# -x：扩展统计（await / r_await / w_await / %util 等）
# -z：隐藏零活跃的设备
# 1：每秒刷新

# 关键指标解读（详见 §4.3）
# await > 10ms = 异常
# %util > 95% = 设备打满（但可能是队列拥塞，不是物理打满）
```

### 9.2 iotop（进程级 IO 监控）

```bash
# 安装：iotop 包
# 用法
iotop -o -t -n 3
# -o：只显示有 IO 的进程
# -t：time-stamp
# -n 3：3 次刷新后退出

# 输出：每个进程实时 IO 速率
#   TID  PRIO  USER  DISK READ  DISK WRITE  SWAPIN  IO>  COMMAND
#   1234 be/4  app   0.00 B/s   5.12 M/s    0.00 %  65.10 %  com.example
#   5678 be/4  system  12.34 M/s  1.23 M/s   0.00 %  30.20 %  system_server

# 稳定性视角：哪个进程 IO 最多一目了然
```

### 9.3 fio（IO 性能基准测试）

```bash
# 安装：fio 包
# 用法：测 IO 性能 + 监控

# 4K 随机读（测设备 IOPS 和延迟）
fio --name=randread --ioengine=libaio --direct=1 \
    --filename=/dev/block/sda --bs=4k --size=1G \
    --rw=randread --iodepth=1 --numjobs=1 \
    --runtime=30 --time_based \
    --percentile_list=50:90:99:99.9:99.99

# 顺序写（测设备吞吐）
fio --name=seqwrite --ioengine=libaio --direct=1 \
    --filename=/data/test --bs=128k --size=1G \
    --rw=write --iodepth=32 --numjobs=1 --runtime=10

# 关键输出：
#   read: io=...bw=...iops=...
#   lat (usec): 50=...90=...99=...99.90=...
#   lat (msec): 4=...10=...20=...
```

### 9.4 blktrace + btt（块层 IO 追踪）

```bash
# blktrace：抓所有块层 IO 事件
blktrace -d /dev/block/sda -w 30 -o /tmp/trace
# -d：目标设备
# -w 30：抓 30 秒
# -o：输出目录

# blkparse：合并多 CPU 的 trace
blkparse -i /tmp/trace -d /tmp/parse.bin

# btt：分析 trace，输出 IO 生命周期
btt -i /tmp/parse.bin -o /tmp/btt_output

# 输出：
#   Q2Q（队列到队列）：两次 IO 派发的间隔
#   D2C（设备到完成）：设备 IO 总耗时 ← 关键！
#   Q2C（队列到完成）：IO 总延迟 = 排队 + D2C
```

**稳定性视角**：
- D2C 长 → 设备慢
- Q2C - D2C 长 → 调度器慢
- Q2Q 短但 Q2C 长 → 队列拥塞

### 9.5 perf + bpf（高级 IO 分析）

```bash
# perf 跟踪 IO 事件
perf list 'block:*'
# block:block_rq_insert
# block:block_rq_issue
# block:block_rq_complete
# block:block_bio_remap

perf trace -e 'block:block_rq_insert' -e 'block:block_rq_complete' \
    -p <pid> -- sleep 10

# eBPF：高级 IO 分析（需要内核支持）
# bpftrace -e 'kprobe:blk_mq_make_request { printf("rq=%p\n", args->bio); }'
```

## 十一、监控体系设计（生产环境必备）

### 11.1 监控指标的 3 个层级

```
第 1 层：设备层（必监控）
├── /proc/diskstats（read / write 完成数）
├── /sys/block/*/queue/scheduler（当前调度器）
├── /sys/class/thermal/（设备温度）
└── /sys/kernel/debug/ufs/（UFS 状态）

第 2 层：内核层（推荐）
├── /proc/pressure/io（PSI IO 压力）
├── /proc/vmstat 中的 nr_dirty / nr_writeback（脏页状态）
├── /sys/fs/cgroup/*/io.stat / io.pressure（cgroup IO 流量）
└── /proc/<pid>/io（关键进程 IO 统计）

第 3 层：用户态层（按需）
├── Perfetto IO events（深度追踪）
├── blktrace（块层追踪）
└── fio（性能基准）
```

### 11.2 告警阈值（推荐）

| 指标 | 阈值 | 严重度 |
|------|------|-------|
| `await > 10ms` (iostat) | 持续 5 分钟 | 警告 |
| `await > 100ms` | 持续 1 分钟 | 严重 |
| `await > 1s` | 持续 10 秒 | 紧急 |
| `iostat %util > 95%` | 持续 1 分钟 | 警告 |
| `/proc/pressure/io full > 5%` | 持续 1 分钟 | 严重 |
| `/proc/pressure/io some > 30%` | 持续 5 分钟 | 警告 |
| `nr_dirty > 80% of dirty_ratio` | 持续 30 秒 | 警告 |
| `cgroup top-app io.stat < background io.stat` | 持续 1 分钟 | 严重 |
| `UFS throttle_state = 1` | 持续 1 分钟 | 警告 |
| `dmesg "hung_task blocked > 30s"` | 单次 | 紧急 |
| `dmesg "task blocked for more than N seconds"` | 单次 | 警告 |

### 11.3 监控采集方案

```bash
# 推荐采集脚本（简化版）
cat > /system/bin/io_monitor.sh <<'EOF'
#!/system/bin/sh

# 设备层
iostat -xz 1 1 | tail -n +4 | head -n -2

# IO 压力
cat /proc/pressure/io

# 脏页状态
grep -E 'nr_dirty|nr_writeback' /proc/vmstat

# cgroup 流量（top-app）

cat /sys/fs/cgroup/top-app/io.pressure

# 设备温度
cat /sys/class/thermal/thermal_zone*/temp

# hung_task 检测
dmesg | grep -E "blocked for more than|hung_task"

# 调度器状态
cat /sys/block/sda/queue/scheduler

# UFS 状态
cat /sys/class/scsi_host/host*/state
EOF

chmod +x /system/bin/io_monitor.sh

# 定时采集（每 5 分钟）
*/5 * * * * /system/bin/io_monitor.sh >> /data/local/tmp/io_monitor.log 2>&1
```

## 十二、治理最佳实践

### 12.1 调度器调优决策树

```
设备类型？
├── UFS 3.1/4.0 → mq-deadline（默认）
├── UFS 2.1 / eMMC → mq-deadline
├── NVMe → kyber
└── 桌面 / 多租户 → bfq

工作负载特征？
├── 顺序 / 混合 IO → mq-deadline（默认参数）
├── 数据库随机 IO → 考虑 none 或 bfq
└── 高并发低延迟 → kyber

调参方向？
├── 读敏感（在线服务）→ read_expire 调小
├── 写敏感（日志服务）→ fifo_batch 调大
└── 功耗敏感（移动设备）→ auto_hibern8 调长
```

详见 [02-IO 调度器](02-IO调度器与多队列架构.md) §8。

### 12.2 cgroup 调优决策树

```
进程类型？
├── system_server → system cgroup / 不限速
├── 前台 App → foreground / top-app cgroup
│   ├── io.weight = 100-200
│   ├── io.max（推荐 200MB/s 读 / 100MB/s 写）
│   └── io.latency = 100ms
├── 后台 App → background cgroup
│   ├── io.weight = 50
│   ├── io.max（推荐 riops=1000 / wiops=500 / bps 30MB/s）
│   └── 不设 latency（保护前台）
└── 系统批处理 → dexopt / camera-daemon cgroup
    └── 按业务需要

Android 14 优先？
├── 是 → OomAdjuster 自动协同 ioprio + oom_adj
└── 否 → 手动 ionice + oom_adj 同步
```

详见 [04-IO 优先级](04-IO优先级与cgroup-IO控制器.md) §8。

### 12.3 应用层 IO 优化清单

```
□ 减少 .so 数量（合并 / 移除无用）
□ 启用 AOT（dex2oat 在安装时完成）
□ 资源压缩（PNG → WebP / 矢量图）
□ 资源按需加载（启动期只加载必要资源）
□ 异步化（IO 放工作线程，不阻塞主线程）
□ 批量写入（避免频繁小写入）

□ 避免在 onCreate / onResume 做同步 IO
□ scoped storage 适配（减少 query 次数）
```

## 十三、总结：架构师视角的 5 条 Takeaway

读完本篇 + 整个 IO 系列，请记住这 5 件事——它们是稳定性架构师排查 IO 问题的"金钥匙"：

1. **"IO 是横跨 5 层的横切系统"**——应用层 ↔ Framework ↔ 内存 ↔ Block ↔ 设备。排查时**必须穿透到具体层**，不能停留在表象（"IO 慢"）。
2. **"80%+ 的 ANR / 卡顿 / 冷启动慢 涉及 IO"**——稳定性架构师必备技能是**从现象反推到 IO 子系统的具体层**。
3. **"5 分钟定位 = 风险分类 + 工具选择 + 指标解读"**——4 类风险 × 5 个工具 × 关键指标 = 排查闭环。
4. **"Page Cache 是 MM 和 IO 的共享数据结构"**——dirty page 限流、reclaim IO 风暴、COW 风暴都是 IO 与 MM 耦合的产物。**不要把它们分开看**。
5. **"治理 = 应用层 + 内核层 + 设备层 三管齐下"**——单独调任何一层都不够。**应用层是收益最大的优化空间**，内核调优是最后手段。

### IO 系列完整阅读路径

```
5 分钟建立认知：
└── 01 总览 → 全局观

10 分钟横切专题（高价值）：
├── 05 IO ↔ MM → 三大传导链
├── 06 IO ↔ Process → D 状态 / ANR
└── 07 程序加载 IO → 冷启动优化

20 分钟核心机制：
├── 02 IO 调度器 → 算法与选型
├── 03 Block 层 → bio/request/plug/merge
└── 04 IO 优先级 → cgroup 隔离

10 分钟 Android 特化：
├── 08 Android 存储栈 → FUSE / sdcardfs
└── 09 存储设备 → UFS / eMMC / NVMe

5 分钟收官：
└── 10 风险 + 工具链 → 排查速查

完整阅读顺序：01 → 02 → 03 → 04 → 05 → 06 → 07 → 08 → 09 → 10
```

### 排查路径速查（终极版）

```
IO 类故障
  ↓
① 现象分类
  ├── 卡死 / ANR → ① IO hang → 抓 ANR trace / sysrq-w / hung_task
  ├── 响应慢 → ② IO 延迟抖动 → iostat / Perfetto IO events
  ├── 写入失败 → ③ IO 资源耗尽 → df / lsof
  └── 前台被后台拖累 → ④ IO 优先级反转 → cgroup stat
  ↓
② 工具选择
  ├── 系统概览 → iostat / /proc/diskstats / /proc/pressure/io
  ├── 进程级 → iotop / /proc/<pid>/io
  ├── 块层 → blktrace / debugfs block
  ├── 端到端 → Perfetto IO events
  └── 性能基准 → fio
  ↓
③ 根因定位
  ├── 应用层 → 减少 IO / 异步化 / 合并
  ├── 内核层 → 调度器 / cgroup / 内核参数
  ├── 内存层 → dirty_ratio / zRAM / reclaim 调优
  └── 设备层 → 散热 / 更换设备
  ↓
④ 治理落地
  ├── 短期：调参 + 配置
  ├── 中期：应用优化
  └── 长期：升级 kernel / Android
```
