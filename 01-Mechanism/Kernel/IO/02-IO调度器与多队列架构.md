## 一、背景与定义：IO 调度器是什么、为什么需要它

### 1.1 朴素的问题：为什么不能让进程直接写磁盘

如果没有 IO 调度器，每个进程都直接调用 `submit_bio` 提交到磁盘，会出现 4 类问题：

```
问题 1：磁盘寻道抖动（seek storm）
├── 进程 A 提交：扇区 100
├── 进程 B 提交：扇区 100000
├── 进程 A 又提交：扇区 200
├── 进程 B 又提交：扇区 100100
└── 磁盘磁头在 100 / 100000 之间反复抖动
    → 平均 IO 延迟从 1ms 飙升到 50ms+（机械硬盘时代）

问题 2：读写饿死
├── 进程 A 疯狂提交读请求（每个 1ms IO）
├── 进程 B 偶尔提交写请求（每个 10ms IO）
└── 如果 A 一直不结束，B 的写永远轮不到
    → 写延迟无上限

问题 3：公平性
├── 进程 A 一次提交 1000 个 bio
├── 进程 B 一次提交 1 个 bio
└── A 的请求挤占整个 Block 队列
    → B 严重卡顿

问题 4：性能（合并 / 排序机会丢失）
├── 进程 A 提交扇区 100-104（5 次 4K IO）
├── 进程 B 提交扇区 200-204（5 次 4K IO）
└── 没有合并 = 10 次 IO
    → 优化后可以合并为 2 次 IO（合并后变成扇区 100-104 和 200-104）
```

**IO 调度器就是解决这 4 类问题**——它是"IO 子系统的调度器"，与 CFS（CPU 调度器）形成"CPU + IO"双调度架构。

### 1.2 IO 调度器 vs CPU 调度器的对比

| 维度 | CPU 调度器（CFS）| IO 调度器 |
|------|---------------|----------|
| **资源** | CPU 时间 | 磁盘队列位置 |
| **调度单位** | task（进程/线程）| bio / request |
| **核心策略** | 公平（vruntime 红黑树）| 公平 + 排序 + 合并 + 优先级 |
| **影响维度** | 谁先跑 | IO 什么时候提交到设备 |
| **优先级反转** | 高优先级任务等低优先级锁 | 高优先级进程等低优先级 IO |
| **协同关系** | 见 [Process 08](../Process/08-进程调度基础架构.md) | 见 §10 |

**关键认知**：**IO 调度器与 CPU 调度器是平级关系**——它们各管各的资源，但通过 task_struct 与 blk_plug 协同工作。

### 1.3 稳定性意义

| 现象 | 真实根因（IO 调度器） | 排查方向 |
|------|------------------|---------|
| **应用响应慢但磁盘不忙** | 调度器配置不当（write 饿死 / 权重错）| `cat /sys/block/sda/queue/scheduler` |
| **写卡顿但读正常** | mq-deadline writes_starved 太小 | 调 `writes_starved` |
| **UFS 高温降频 + 卡顿** | 调度器合并窗口太大 → 单次 IO 大 | 调 `fifo_batch` |
| **多 cgroup 互相干扰** | bfq 权重配置错 | 调 cgroup weight |
| **NVMe 性能上不去** | 用了 cfq（早被废弃）| 切换到 kyber / none |

## 二、架构与交互：IO 调度器在 IO 链路中的位置

### 2.1 调度器在 5 层 IO 链路中的位置

```
用户进程（read/write）
    ↓ syscall
VFS 层（fs/read_write.c）
    ↓
Page Cache 层（mm/filemap.c）
    ↓ submit_bio()
┌───────────────────────────────────────────────────────────┐
│  Block 层：generic_make_request → blk_mq_make_request     │
│       ↓                                                    │
│  【IO 调度器】（本篇核心）                                  │
│  - bio → request（合并 + 排序）                             │
│  - 选 request 调度策略（mq-deadline / bfq / kyber）        │
│       ↓                                                    │
│  request 派发到 hardware queue                              │
└───────────────────────────────────────────────────────────┘
    ↓ queue_rq
驱动层 + 设备层（UFS / eMMC / NVMe）
```

### 2.2 blk-mq 多队列的整体架构

```
                  request_queue (per block device)
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
   ┌────▼────┐          ┌─────▼─────┐         ┌─────▼────┐
   │ hw-queue│          │  hw-queue │         │ hw-queue │
   │   #0    │          │    #1     │         │   #2     │
   └────┬────┘          └─────┬─────┘         └────┬────┘
        │                     │                     │
   ┌────▼────┐          ┌─────▼─────┐         ┌─────▼────┐
   │ sw-queue│          │  sw-queue │         │ sw-queue │
   │ (CPU 0) │          │  (CPU 1)  │         │ (CPU 2)  │
   └────┬────┘          └─────┬─────┘         └────┬────┘
        │                     │                     │
   ┌────▼────┐          ┌─────▼─────┐         ┌─────▼────┐
   │ 调度器  │          │  调度器   │         │  调度器   │
   │ (elevator)        │           │         │           │
   └─────────┘          └───────────┘         └───────────┘

关键点：
- 每个 CPU 有自己的 software queue（减少锁竞争）
- 每个硬件队列对应设备的一个 DMA / 中断向量
- 调度器在 software queue 和 hardware queue 之间决策
```

## 三、单队列时代：cfq / deadline / noop（历史背景）

### 3.1 单队列的问题

Linux 2.6 - 4.x 时代，所有 IO 都通过**单一队列**（`struct request_queue`）：

```
单队列时代（2.6 - 4.x）：
                       request_queue (单队列)
                              │
                              ▼
                  ┌──────────────────────┐
                  │     IO 调度器          │
                  │  cfq / deadline / noop│
                  └──────────────────────┘
                              │
                              ▼
                       block driver

问题：
1. 队列锁竞争严重（多核同时提交 IO）
2. 调度器是单线程的（无法利用多核）
3. CFQ 在多核下扩展性差
```

### 3.2 cfq（Completely Fair Queueing）

```c
// block/cfq-iosched.c（Linux 4.x 之前，5.x 已废弃）
// 核心思路：按进程分组，每个进程一个 service tree
struct cfq_data {
    struct rb_root service_tree;          // 红黑树
    struct cfq_queue *active_queue;       // 当前活跃队列
    // ...
};

struct cfq_queue {
    struct rb_root sort_list;             // 该进程的请求排序列表
    // ...
};
```

**cfq 的特点**：
- ✅ 按进程公平分配 IO 带宽
- ✅ 支持 cgroup（blkio controller）
- ❌ 单队列锁竞争严重
- ❌ 算法复杂，配置项多
- ❌ **Android GKI 5.10+ 已废弃**

### 3.3 deadline

```c
// block/deadline-iosched.c（Linux 4.x 之前，5.x 已废弃）
// 核心思路：每个 IO 都有"截止时间"，避免 IO 饿死
struct deadline_data {
    struct rb_root sort_list[2];          // 读 / 写 排序树
    struct list_head fifo_list[2];        // 读 / 写 FIFO 队列
    
    unsigned int writes_starved;          // 写饿死计数（默认 2）
    unsigned int fifo_batch;              // 单次批量处理数（默认 16）
    
    // ...
};
```

**deadline 的特点**：
- ✅ 简单可靠，读写公平
- ✅ 适合数据库等需要读延迟可预测的场景
- ❌ 仍是单队列，多核扩展差
- ❌ 已废弃（mq-deadline 是其 blk-mq 版本）

### 3.4 noop

```c
// block/noop-iosched.c
// 核心思路：FIFO + 简单合并，不做调度
structnoop_data {
    struct list_head queue;
};
```

**noop 的特点**：
- ✅ 最低开销
- ✅ 适合**真随机 IO**（SSD / NVMe，调度器帮不上忙）
- ❌ 没有公平性 / 优先级 / 合并优化

### 3.5 单队列时代总结

| 调度器 | 公平性 | 读延迟 | 复杂度 | Android GKI 默认 |
|-------|-------|--------|--------|----------------|
| cfq | ✅ 优 | 中 | 高 | ❌ 已废弃 |
| deadline | 中 | ✅ 优 | 低 | ❌ 已废弃 |
| noop | ❌ | 中 | 极低 | ❌ 已废弃 |

**为什么单队列时代一去不复返？**
- NVMe 设备有多队列（最多 64K 队列），单队列调度器跟不上
- 多核扩展性差
- cfq 维护成本高，bug 多

## 四、blk-mq 多队列架构（5.10+ 主流）

### 4.1 blk-mq 核心数据结构

```c
// include/linux/blk-mq.h
struct blk_mq_tag_set {
    struct blk_mq_ops *ops;              // 驱动 ops
    unsigned int nr_hw_queues;           // 硬件队列数
    unsigned int queue_depth;            // 每个队列深度
    
    struct blk_mq_tags **tags;           // tag set 数组
    
    // ...
};

struct request_queue {
    // ...
    struct blk_mq_tag_set *tag_set;
    struct blk_mq_hw_ctx **queue_hw_ctx; // 硬件队列数组
    unsigned int nr_hw_queues;
    
    struct elevator_queue *elevator;     // IO 调度器
    
    // ...
};

struct blk_mq_hw_ctx {
    struct blk_mq_tags *sched_tags;      // 调度器 tag
    struct blk_mq_tags *tags;            // 驱动 tag
    
    unsigned long dispatched[2];          // 派发计数（sync/async）
    
    struct blk_mq_ctx *dispatch_from;    // 当前派发来源
    
    // 调度器 ops
    struct request *(*sched_dispatch_request_fn)(struct blk_mq_hw_ctx *);
    // ...
};
```

### 4.2 blk-mq 的三级架构

```
┌─────────────────────────────────────────────────────────────┐
│  Software Staging Queue（per-CPU）                             │
│  - 进程 submit_bio 后先到 sw queue                            │
│  - 减少跨 CPU 锁竞争                                          │
│  - 调度器在这里做合并 / 排序                                   │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼ (调度器 dispatch_request)
┌─────────────────────────────────────────────────────────────┐
│  Hardware Dispatch Queue（per-hardware-queue）                │
│  - 调度器选中的 request 进入 hw queue                         │
│  - 准备派发到设备                                             │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼ (driver queue_rq)
┌─────────────────────────────────────────────────────────────┐
│  Device Hardware Queue                                        │
│  - 设备自身的命令队列（UFS command queue / NVMe SQ）          │
│  - 设备内部调度                                               │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 submit_bio 在 blk-mq 下的路径

```c
// block/blk-mq.c
blk_status_t blk_mq_make_request(struct request_queue *q, struct bio *bio) {
    // ① plug：如果当前 task 有 plug，先尝试合并
    if (current->plug)
        blk_mq_attempt_bio_merge(q, bio, &nr_segs);
    
    // ② 调度器 insert_request
    //    mq-deadline → 读 fifo / 写 fifo
    //    bfq → service tree
    //    kyber → token bucket
    blk_mq_sched_insert_request(bio, nr_segs, ...);
    
    // ③ 唤醒硬件队列派发
    blk_mq_run_hw_queue(hctx, async);
    
    return BLK_STS_OK;
}
```

### 4.4 blk-mq 的核心优势

| 优势 | 说明 | 性能提升 |
|------|------|---------|
| **per-CPU sw queue** | 减少跨 CPU 锁竞争 | 多核 IO 吞吐 +200%+ |
| **per-device hw queue** | 与设备多队列匹配 | NVMe 性能充分发挥 |
| **tag 机制** | O(1) request 分配 / 回收 | 锁开销降到最低 |
| **无锁派发** | dispatch 阶段几乎无锁 | 延迟波动减小 |

---## 五、mq-deadline 调度器（Android GKI 默认）

### 5.1 mq-deadline 核心数据结构

```c
// block/mq-deadline.c
struct deadline_data {
    // 排序树（按 LBA 排序，便于合并）
    struct rb_root sort_list[2];          // READ / WRITE
    // FIFO 队列（按时间顺序，到达时间记录在 bio/req 里）
    struct list_head fifo_list[2];        // READ / WRITE
    
    // 下一个要派发的读 / 写 request
    struct request *next_rq[2];
    
    // 配置参数
    unsigned int writes_starved;          // 写饿死计数（默认 2）
    unsigned int fifo_batch;              // 单次批量处理数（默认 16）
    
    // 读 / 写方向的限制（LBA 寻道距离等）
    int read_expire;                       // 读过期时间（ms，默认 500）
    int write_expire;                      // 写过期时间（ms，默认 5000）
    
    // ...
};
```

### 5.2 mq-deadline 调度算法

```
mq-deadline 调度核心（dispatch_request 时）：

if (下一个要派的 READ) {
    // ① 优先派发读
    return next_rq[READ];
}

if (写 FIFO 长度 >= fifo_batch) {
    // ② 单次批量派发写
    return 取 batch 个写；
}

if (写饥饿计数 writes_starved 到达阈值) {
    // ③ 处理写（避免写饿死读的处理反过来饿死写）
    return next_rq[WRITE];
}

// ④ 默认：等待更多读
return NULL;
```

**核心策略**：
- **读优先**：进入 `fifo_list[READ]` 的读请求立即派发（除非超过 read_expire）
- **写批量**：写请求先在 `sort_list[WRITE]` 合并，达到 `fifo_batch` 一次性派发
- **公平约束**：`writes_starved` 防止写因读永远优先而被饿死

### 5.3 关键参数（Android 默认值）

```bash
# 查看 / 设置 mq-deadline 参数
cat /sys/block/sda/queue/iosched/writes_starved
# 2

cat /sys/block/sda/queue/iosched/fifo_batch
# 16

cat /sys/block/sda/queue/iosched/read_expire
# 500 (ms)

cat /sys/block/sda/queue/iosched/write_expire
# 5000 (ms)
```

| 参数 | 默认 | 调大效果 | 调小效果 |
|------|------|---------|---------|
| **writes_starved** | 2 | 写更容易派发（牺牲读）| 读更容易饿死写（更好读延迟）|
| **fifo_batch** | 16 | 写合并更多（吞吐↑、写延迟↑）| 写更快派发（写延迟↓、吞吐↓）|
| **read_expire** | 500ms | 读容忍更长等待 | 读更快派发（但可能被饿死）|
| **write_expire** | 5000ms | 写容忍更长等待 | 写更快派发 |

### 5.4 mq-deadline 的工程取舍

**优势**：
- ✅ 算法简单稳定（5 行核心逻辑）
- ✅ 读延迟可预测（read_expire 限制）
- ✅ 适合移动设备（UFS 顺序 / 混合 IO）
- ✅ 内存开销小

**劣势**：
- ❌ 没有 cgroup 权重（要靠 blk-throttle 配合，见 [04-IO 优先级](04-IO优先级与cgroup-IO控制器.md)）
- ❌ 对 SSD 内部调度"帮不上太多忙"
- ❌ 极端混合负载下可能不够灵活

### 5.5 mq-deadline 适用场景

| 场景 | 推荐度 | 原因 |
|------|------|------|
| **Android 主流移动设备** | ⭐⭐⭐⭐⭐ | 顺序 + 混合 IO，mq-deadline 最匹配 |
| **UFS 3.1/4.0** | ⭐⭐⭐⭐⭐ | UFS 有 command queue，mq-deadline 配合良好 |
| **eMMC 入门机** | ⭐⭐⭐⭐ | 同上 |
| **NVMe 高端** | ⭐⭐⭐ | 性能有提升空间，kyber 可能更好 |
| **数据库（高随机 IO）** | ⭐⭐ | 随机 IO 调度帮不上忙 |
| **服务器多租户** | ⭐ | bfq 更适合（有 cgroup 权重）|

## 六、bfq 调度器（cgroup 感知的公平调度）

### 6.1 bfq 核心数据结构

```c
// block/bfq-iosched.c
struct bfq_data {
    struct rb_root_cached service_tree;   // 红黑树（按 budget 排序）
    
    struct bfq_queue *active_queue;       // 当前活跃 bfq_queue
    
    // 全局配置
    u64 bfq_timeout;                       // 同步 IO 超时（默认 8s）
    unsigned int bfq_weight;               // 默认权重
    unsigned int bfq_max_budget;           // 最大 budget
    // ...
};

struct bfq_queue {
    struct rb_node entity;                 // service tree 节点
    struct bfq_data *bfqd;
    
    pid_t pid;                             // 关联的 task
    
    // budget 机制
    unsigned int max_budget;               // 单次预算
    unsigned int budget;                   // 剩余预算
    
    // IO 统计
    unsigned long num_IO;
    u64 tot_IO_size;
    // ...
};
```

### 6.2 bfq 调度算法（核心：budget + weight）

```
bfq 的核心思想：
- 每个 bfq_queue（per-process 或 per-cgroup）有"budget"（能跑多少 IO 时间）
- 按 weight（权重）分配全局 IO 带宽
- 同步 IO 有"加速"机制（latency target）

调度循环：
for each bfq_queue in service_tree:
    if (有同步 IO) {
        // 同步 IO 优先派发
        return sync_request;
    }
    if (budget 用完) {
        // 轮到下一个 queue
        continue;
    }
    return async_request;
```

### 6.3 bfq 的工程取舍

**优势**：
- ✅ cgroup 感知（不同 cgroup 权重不同）
- ✅ 同步 IO 延迟保证（latency target）
- ✅ 桌面场景体验好（前台 app 不被后台拖累）
- ✅ 适合多租户场景

**劣势**：
- ❌ 算法复杂（最难维护的调度器）
- ❌ 内存开销大（每个进程一个 bfq_queue）
- ❌ 调参不直观
- ❌ Android GKI 默认不开启（部分厂商如三星部分机型启用）

### 6.4 bfq 适用场景

| 场景 | 推荐度 | 原因 |
|------|------|------|
| **桌面 / 笔记本** | ⭐⭐⭐⭐⭐ | 前台 / 后台隔离，体验优化 |
| **服务器多租户** | ⭐⭐⭐⭐⭐ | cgroup 公平 |
| **Android 桌面模式** | ⭐⭐⭐⭐ | 同桌面 |
| **Android 手机前台/后台** | ⭐⭐⭐ | 部分场景有提升 |
| **UFS 高端手机** | ⭐⭐ | 默认 mq-deadline 够用 |

## 七、kyber 调度器（NVMe / 低延迟专用）

### 7.1 kyber 核心数据结构

```c
// block/kyber-iosched.c
struct kyber_queue_data {
    struct kyber_hctx_data *hctx;          // per-hw-ctx 数据
    
    // 4 个 domain：读同步 / 读异步 / 写同步 / 写异步
    struct kyber_domain domain[KYBER_NUM_DOMAINS];
    
    unsigned int latency_target[KYBER_NUM_DOMAINS];
    //   [READ_SYNC] = 2000us
    //   [READ]      = 8000us
    //   [WRITE]     = 8000us
    //   [SYNC]      = 8000us
};

struct kyber_domain {
    struct rb_root read_buckets;           // 红黑树
    struct list_head dispatch;             // 已派发列表
    
    atomic_t tokens;                       // 当前 token 数
    int limit_depth;                       // 最大并发数
    // ...
};
```

### 7.2 kyber 调度算法（核心：token bucket）

```
kyber 的核心思想：
- 每个 domain 有 token bucket
- 派发请求消耗 token
- 完成请求释放 token
- token 数 = 当前可并发 IO 数

调度循环：
for each domain:
    if (domain.tokens > 0) {
        // 减少 token
        atomic_dec(&domain.tokens);
        return domain_request;
    }

// 全部 domain token 耗尽 → 不派发
return NULL;

// 后台：异步完成 → 释放 token → 调度恢复
```

**关键洞察**：kyber 用 token 控制**并发度**——不是按"时间"分配，而是按"同时能跑多少 IO"分配。

### 7.3 kyber 适用场景

| 场景 | 推荐度 | 原因 |
|------|------|------|
| **NVMe 高端** | ⭐⭐⭐⭐⭐ | 低延迟特性匹配 |
| **eMMC 入门** | ⭐⭐⭐⭐ | 性能提升明显 |
| **UFS 高端** | ⭐⭐⭐ | 可用但 mq-deadline 也够 |
| **UFS 主流** | ⭐⭐ | mq-deadline 更合适 |
| **服务器** | ⭐⭐ | 服务器场景复杂 IO，kyber 单一策略不够 |

### 7.4 三大调度器对比

| 维度 | mq-deadline | bfq | kyber |
|------|------------|-----|-------|
| **核心策略** | 读优先 + 写批量 | budget + weight | token bucket |
| **cgroup 权重** | ❌（靠 blk-throttle）| ✅ 内置 | ❌ |
| **同步 IO 优先** | ✅ | ✅ | ✅ |
| **算法复杂度** | 低 | 高 | 中 |
| **内存开销** | 低 | 高 | 中 |
| **Android GKI 默认** | ✅（UFS）| 部分厂商 | NVMe 专用 |
| **典型延迟** | UFS ~1ms | UFS ~2ms | NVMe ~50μs |
| **典型吞吐** | 高 | 中 | 极高 |

---## 九、IO 调度器 ↔ CFS 调度的联动（双调度视角）

### 9.1 双调度的工作模型

```
进程视角（CFS 调度器）：
├── CPU 1：跑进程 A（系统服务）
├── CPU 2：跑进程 B（前台 App）  
└── CPU 3：跑进程 C（后台 App）

IO 视角（IO 调度器）：
├── 进程 A 提交 bio → sw queue 0
├── 进程 B 提交 bio → sw queue 1
└── 进程 C 提交 bio → sw queue 2
    → 调度器统一处理 sw queue 0/1/2 → 派发到 UFS hw queue
    → UFS 设备执行 IO
    → 完成 → 唤醒对应进程

关键问题：
- 进程 B（前台）提交 IO 后，B 立刻进入 D 状态
- 调度器切走 B，跑其他进程
- IO 完成时，调度器切回 B（如果 B 的 vruntime 最小）
```

### 9.2 优先级反转问题（双调度视角）

```
典型场景：
- 高优先级进程 A（nice=-10）执行
- A 等一个锁，锁被低优先级进程 C 持有
- C 持有锁后被高优先级进程 B 抢走（nice=-20 比 C 更优先）
- B 现在持有锁，正在执行 IO
- B 的 IO 排队等待 IO 调度器派发
- 但 B 是低 IO 优先级（ioprio=BE/7）
- IO 调度器先派发其他请求，B 的 IO 迟迟不完成
- B 不释放锁 → C 不释放锁 → A 永远拿不到锁
```

**IO 调度器与 CFS 调度的优先级反转**：

| 反转类型 | 触发 | 影响 |
|---------|------|------|
| **CPU 优先 → IO 阻塞** | 高 nice 进程执行大量 IO | 调度器先派发其他请求，高 nice 进程 IO 卡 |
| **IO 优先 → CPU 阻塞** | 高 ioprio 进程等 CPU | 调度器等待 CPU，高 ioprio 没意义 |
| **cgroup 反转** | cgroup IO 权重 < CPU 权重 | 后台 cgroup 拖累前台 IO |

### 9.3 ioprio 与 nice 的协同（Android 实践）

```java
// frameworks/base/services/core/java/com/android/server/am/OomAdjuster.java
// Android 在调整进程优先级时，同步调整 IO 优先级
private void applyOomAdjLocked(...) {
    // ...
    if (app.setProcState != oldProcState) {
        // CPU 调度优先级调整
        Process.setOomAdj(pid, oom_adj);
        
        // IO 优先级调整（Android 14+ 新增）
        Process.setIoPriority(pid, ioprio_class, ioprio_data);
    }
}
```

详见 [04-IO 优先级](04-IO优先级与cgroup-IO控制器.md) §9。

### 9.4 双调度协同的稳定性意义

| 现象 | 真实根因（双调度协同问题） | 排查方向 |
|------|------------------------|---------|
| **前台 App 慢但磁盘不忙** | 前台进程被 cgroup 限流，后台 cgroup 抢 IO | 检查 cgroup weight 配置 |
| **系统服务响应慢** | 高 IO 负载时 CFS 误判"系统空闲" | 检查 %wa 与 %idle 综合判断 |
| **关键进程饿死** | IO 调度器 write_starved 配置错 | 调 writes_starved |

## 十一、实战案例：某机型 bfq 调度器在 IO 密集场景导致游戏 loading 延迟飙高（典型模式）

### 现象

某品牌旗舰手机在**游戏 loading 阶段卡顿 3-5s**，FPS 从 60 跌到 10。多发生在用户安装大量 App 后。

### 环境

- Android 13 / Kernel 5.10 / 某厂商 GKI（启用了 bfq）/ 设备 8GB RAM

### 分析思路

**第一步：抓 game loading 的 IO 行为**：

```bash
# 1. 抓 IO trace
cat /sys/kernel/debug/tracing/trace_pipe | grep "block:" &
# 或者用 perfetto 抓 IO events
perfetto -o game_load.pftrace -c config.pbtx

# 2. 看调度器状态
cat /sys/block/sda/queue/scheduler
# [bfq] mq-deadline kyber none  ← 启用了 bfq

# 3. 看 bfq 内部状态
cat /sys/kernel/debug/block/sda/bfq/queue/budget
# budget 平均值 8ms（理论最大值 16ms）
```

**第二步：分析根因**：

```
游戏启动 → 大量 read IO
    ↓
bfq 调度器按 weight 分配
    ↓
但此时 system_server + 其他 App 也在提交 IO
    ↓
前台 game（cgroup weight = 800）vs 后台 cgroup（weight = 200）
    ↓
按比例 game 应该得 80% IO 带宽，但实际只得到 50%
    ↓
原因：bfq 的"budget 机制"在 burst IO 下受限
    ↓
game 单次需要 100ms IO 时间，budget 只够 8ms → 频繁切换 → 吞吐下降
```

**第三步：定位 bfq 配置问题**：

```bash
cat /sys/block/sda/queue/iosched/bfq_max_budget
# 16 （太小！默认 12，标准应该是 64）

cat /sys/block/sda/queue/iosched/bfq_timeout
# 50ms （太短！默认 8s，但游戏需要长 burst）
```

### 根因诊断

1. 该厂商启用了 bfq（桌面体验导向）
2. 但 bfq 参数配置不当：`bfq_max_budget` 仅 16（默认值太低），导致单次 IO burst 受限
3. 游戏启动期大量 burst IO 被频繁切走 → 加载延迟飙高
4. **这不是 bfq 算法问题，是配置问题**

### 修复方案

1. **调参**：`bfq_max_budget = 64`（提升 4x）
2. **调参**：`bfq_timeout = 8000`（默认 8s）
3. **重启验证**：游戏 loading 延迟从 3s 降到 1s
4. **长期**：评估是否切换到 mq-deadline（UFS 主流场景更稳定）

### 排查路径速查

```

## 十二、实战案例：UFS 设备用 cfq 导致 Kernel Panic（典型模式）

### 现象

某 OEM 在 GKI 5.10 上**误保留 cfq 调度器配置**，导致 UFS 设备在高负载时随机触发 Kernel Panic（UE），重启后偶发。

### 环境

- Kernel 5.10 GKI（误配置）/ UFS 3.1 / 设备 Pixel 6 改造机

### 分析思路

**第一步：抓 panic 日志**：

```
[12345.678] BUG: unable to handle kernel NULL pointer dereference at 0000000000000018
[12345.679] PGD 0 P4D 0
[12345.680] Oops: 0000 [#1] PREEMPT SMP NOPTI
[12345.681] CPU: 3 PID: 1234 Comm: kworker/u16:2 Not tainted 5.10.0-xxx
[12345.682] Hardware name: ...
[12345.683] Task: ... (kworker)
[12345.684] Call trace:
[12345.685]  cfq_dispatch_requests+0x14c/0x4b0
[12345.686]  __blk_mq_run_hw_queue+0x38/0x90
[12345.687]  blk_mq_sched_dispatch_requests+0x40/0x80
[12345.688]  __blk_mq_sched_dispatch_requests+0x40/0x80
```

**第二步：根因分析**：

- `cfq_dispatch_requests` 在 5.10 中**已被删除**——但驱动代码仍然调用
- 这是**旧版本配置残留**导致的**API mismatch**
- UFS 设备压力下高频触发 → Kernel Panic

### 修复方案

1. **移除 cfq 配置**：`elevator=  `（留空，让内核选默认 mq-deadline）
2. **代码修复**：移除 GKI 配置文件中 cfq 相关项
3. **验证**：压力测试 1 小时不再 panic

### 排查路径速查

```
Kernel Panic + cfq_dispatch_requests 栈帧
  ↓
检查内核是否启用 cfq → 5.10+ 应该用 mq-deadline
  ↓
检查 GKI 配置 → 移除 cfq
  ↓
压力测试验证
```

## 十三、总结：架构师视角的 5 条 Takeaway

读完本篇，请记住这 5 件事——它们是排查 IO 调度器故障的"金钥匙"：

1. **"IO 调度器与 CFS 调度器是平级双调度"**——CPU 调度决定"什么时候跑"，IO 调度决定"IO 什么时候提交"。一个高优先级进程被 IO 卡住，调度器也救不了它。
2. **"Android GKI 5.10+ 默认是 mq-deadline"**——但部分厂商改用 bfq。**稳定性视角必须先确认设备当前调度器**，再决定调参方向。
3. **"blk-mq 多队列是 5.10+ 唯一支持"**——单队列时代（cfq/deadline/noop）已彻底过时。不要在新代码中看到这些名词就以为它们还在用。
4. **"调度器参数要按工作负载调"**——`writes_starved`、`fifo_batch`、`bfq_max_budget` 等不是默认值就最好。**关键场景（游戏 / 相机 / 大数据写入）需要专项调优**。
5. **"调度器选错会让设备随机 UE"**——最典型的就是 5.10+ 残留 cfq 配置。**OEM 升级 kernel 必须同步清理旧调度器配置**。

### 排查路径速查（IO 调度器问题）

```
IO 性能 / 卡顿 / UE
  ↓
看现象：游戏慢 / 系统卡 / 写卡 / panic
  ↓
① 调度器配置 → cat /sys/block/sda/queue/scheduler
  ↓
② 调度器参数 → cat /sys/block/sda/queue/iosched/*
  ↓
③ IO 延迟分布 → iostat -xz 1
  ↓
④ 调度器内部状态 → debugfs 或 sysfs
  ↓
⑤ 治理 → 选调度器 / 调参数 / 切 mq-deadline（兜底）
```

## 附录 D：工程基线表

| 参数 | 典型默认 | 选用准则 | 踩坑提醒 |
|------|---------|---------|---------|
| **IO 调度器（UFS 设备）** | mq-deadline | 移动设备首选 mq-deadline | 不要用 cfq（5.10+ 已废弃）|

| **mq-deadline writes_starved** | 2 | 读密集场景调到 1（写更快）；写密集调到 4 | 太小 → 写饿死读 |
| **mq-deadline fifo_batch** | 16 | 顺序写多调到 32；随机写多调到 8 | 太大 → 单次 IO 大 → 卡顿 |
| **mq-deadline read_expire** | 500ms | 移动设备调到 250-500ms | 太短 → 读饿死 |
| **mq-deadline write_expire** | 5000ms | 移动设备调到 5000-10000ms | 太长 → 写延迟高 |

| **blk-mq nr_hw_queues** | 设备硬件队列数 | 与设备能力匹配 | 太多 → 调度开销 |
