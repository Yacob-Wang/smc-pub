## 0. 写在前面:为什么"风险地图"作为收官

### 0.1 一个真实的紧急故障

**场景**:凌晨 3 点,生产环境告警,某 App P99 冷启动从 1200ms 退化到 5000ms。

**症状**:
- 告警群炸锅
- 多个用户反馈"打开 App 卡死"
- 工程师打开 logcat,看到大量 `ClassNotFoundException`、`UnsatisfiedLinkError`、`<clinit>` 死锁

**根因(事后复盘)**:
- App 升级时,升级了某个第三方 SDK
- 新 SDK 引入了新的 .so,但 R8 配置没有保留 SDK 类的 keep 规则
- R8 把 SDK 类 strip 了 → `ClassNotFoundException`
- R8 把 SDK 的 JNI 引用 strip 了 → `UnsatisfiedLinkError`
- R8 把 SDK 的 static 块重排了 → `<clinit>` 死锁

**修复时间**:2 小时(如果工程师熟悉启动期故障速查)

**这个案例需要的就是本篇要提供的能力**:
- 看到异常关键字,直接定位到 PLE 阶段
- 看到阶段,直接定位到 PLE 文章
- 看到文章,直接拿到诊断流程

#### 0.1.1 §0.1 案例的可验证 4 件套

> **环境**:
> - 设备:某线上 App 的全量用户群(主要是 arm64-v8a,中端机 Pixel 5/6 + 旗舰 S22/P50)
> - Android 版本:8.0 / 9.0 / 10 / 11 / 12 / 13 / 14 全分布
> - App:某社交 App v5.0.0(集成 8 个三方 SDK,multidex,DEX 60MB,native 库 15 个)
> - 工具:`Bugsnag` + `Sentry` + 自研 APM + `dumpsys`

> **复现步骤**(线上场景):
> 1. 凌晨 3 点告警群:P99 冷启动从 1200ms 退化到 5000ms+
> 2. APM 显示三类异常同时爆发:
>    - `ClassNotFoundException`(新增 1000 例/小时)
>    - `UnsatisfiedLinkError`(新增 500 例/小时)
>    - `<clinit>` 死锁(新增 200 例/小时)
> 3. 工程师 5 分钟内打开本篇速查表,锁定"SDK 升级 + R8 误 strip"根因
> 4. 2 小时内修复并发布 hotfix

> **logcat 关键片段**(线上抓取,去敏感):
> ```
> # 模式 1:ClassNotFoundException
> E AndroidRuntime: java.lang.ClassNotFoundException: Didn't find class "com.thirdparty.NewSDKManager"
>
> # 模式 2:UnsatisfiedLinkError
> E AndroidRuntime: java.lang.UnsatisfiedLinkError: dlopen failed: library "libnewsdk.so" not found
>
> # 模式 3:ExceptionInInitializerError(<clinit> 死锁)
> E AndroidRuntime: java.lang.ExceptionInInitializerError
> E AndroidRuntime: Caused by: java.lang.RuntimeException: timeout waiting for static init
> ```

> **根因诊断命令序列**(5 秒定位 → 30 分钟修复):
> ```bash
> # Step 1:看异常类型 → 锁定 PLE 阶段
> # ClassNotFoundException → PLE-07 ClassLoader
> # UnsatisfiedLinkError → PLE-03 linker64 / PLE-04 RELRO
> # <clinit> 失败 → PLE-08 Verify / Init
>
> # Step 2:看 SDK 升级日志,确认是否近期升级
> $ git log --oneline -- com.example.app/dependencies.gradle | head -10
>
> # Step 3:看 R8 mapping.txt,确认 strip
> $ grep "NewSDKManager\|libnewsdk" mapping.txt
>
> # Step 4:回滚 SDK 或补 R8 keep 规则
> ```

> **修复 commit-style diff**(紧急 hotfix):
> ```diff
> - # 旧:升级 SDK 没加 R8 keep 规则
> - implementation 'com.thirdparty:new-sdk:2.0.0'
> + # 新:补 keep 规则

> + # proguard-rules.pro:
> + -keep class com.thirdparty.newsdk.** { *; }
> + -keep class com.thirdparty.newsdk.**$* { *; }
> + -keepclasseswithmembernames class * {
> +     native <methods>;
> + }
> ```

> **架构师视角**:凌晨 3 点的 P0 故障,**"5 秒定位 → 30 分钟修复"** 是稳定性的核心 KPI。本篇速查表是工程师贴在工位上的"作战地图",**必须每半年随 PLE 系列主版本升级同步刷新**。

## 2. SO 加载失败:4 个根因

### 2.1 4 类 SO 故障

| 故障 | 触发条件 | 错误关键字 |
|---|---|---|
| **符号找不到** | .dynsym 没暴露 / 没加载 | `cannot locate symbol` |
| **架构错配** | 32 位 .so 加载到 64 位进程 | `wrong ELF class` |
| **版本不匹配** | libstdc++ vs libc++ | `undefined symbol: _ZNSt...` |
| **namespace 冲突** | 私有路径 .so | `cannot open from namespace` |

### 2.2 故障 1:符号找不到

**典型症状**:
```
W linker: cannot locate symbol "_ZN7android14MediaCodecC2Ev" referenced by /data/app/~~xyz/lib/arm64/libnative.so
F libc : Fatal signal 11 (SIGSEGV), code 1 (SEGV_MAPERR)
```

**根因诊断**:

```bash
# 1. 看 libnative.so 的 .rela.plt
$ readelf -r libnative.so | grep "_ZN7android14MediaCodecC2Ev"
# 0x0001a87d0  R_AARCH64_JUMP_SLOT  _ZN7android14MediaCodecC2Ev

# 2. 看 libart.so 的 .dynsym 是否有这个符号
$ readelf -Ds libart.so | grep "_ZN7android14MediaCodecC2Ev"
# 空 → libart.so 没导出这个符号

# 3. 看 libart.so 的 .symtab
$ readelf -s libart.so | grep "_ZN7android14MediaCodecC2Ev"
# 找到 → 符号存在,但只在 .symtab,不在 .dynsym
```

**根因**:**符号被 visibility=hidden 限制,不出现在 .dynsym**。

**修复**:
- 修改源码:`__attribute__((visibility("default")))`
- 链接选项:加 `-fvisibility=default`
- NDK 升级时检查 STL 链接方式

**详细流程**:见 [PLE 04 §8.3](04-符号解析与重定位-plt-got-relro全景.md)

### 2.3 故障 2:架构错配

**典型症状**:
```
W linker: dlopen failed: "libfoo.so" has bad ELF magic
W linker: 32-bit instead of 64-bit
```

**根因**:**32 位 .so 加载到 64 位进程**。

**诊断**:

```bash
# 1. 看 .so 是什么架构
$ readelf -h libfoo.so | grep Machine
Machine:                           ARM          ← 32 位

# 2. 看进程是 32 位还是 64 位
$ adb shell getprop ro.product.cpu.abi
arm64-v8a                          ← 64 位进程

# 3. 看 abi 列表
$ adb shell getprop ro.product.cpu.abilist
arm64-v8a,armeabi-v7a,armeabi       ← 包含 32 位
```

**修复**:
- 用 64 位 .so
- 或在 build.gradle.kts 设置 `ndk.abiFilters = ['arm64-v8a']`

### 2.4 故障 3:版本不匹配(NDK STL)

**典型症状**:
```
W linker: cannot locate symbol "_ZNSt6vectorIiSaIiEE9push_backEi" referenced by /data/app/~~xyz/lib/arm64/libnative.so
```

**根因**:**NDK 升级时 STL 库变化**:
- NDK r18 之前:默认 `c++_static`(libstdc++)
- NDK r18 之后:默认 `c++_shared`(libc++)

**诊断**:

```bash
# 1. 看 .so 的 NEEDED
$ readelf -d libnative.so | grep NEEDED
# 0x0000000000000001 (NEEDED)             Shared library: [libc++.so]  ← libc++

# 2. 看代码里用的是哪个 STL
$ nm -D libnative.so | grep "_ZNSt"
# 找到 _ZNSt... → libstdc++ 符号(不是 libc++)
```

**修复**:
- 修改 CMakeLists.txt:`target_link_libraries(native PRIVATE c++_shared)` 或 `c++_static`
- 确保和 NDK 默认一致

### 2.5 故障 4:namespace 冲突

**典型症状**:
```
W linker: cannot open "libfoo.so" from namespace "default"
W linker: namespace "default" does not have path "/data/local/tmp"
```

**根因**:**Android 7+ namespace 隔离,App 不能加载 `/data/local/tmp` 的 .so**。

**诊断**:

```bash
# 1. 确认 .so 存在
$ adb shell ls -la /data/local/tmp/libfoo.so

# 2. 看 namespace 配置
$ adb shell cat /system/etc/ld.config.txt

# 3. (debug)用 strace 看 dlopen 系统调用
$ adb shell strace -e openat -p PID
```

**修复**:
- debug:关 SELinux(`setenforce 0`)
- 正常:把 .so 移到 App 的 lib/ 目录

| 异常关键字 | 故障 | PLE 阶段 | PLE 文章 |
|---|---|---|---|
| `cannot locate symbol` | 符号未导出 | 阶段 1 | P04 |
| `wrong ELF class` | 32/64 位错配 | 阶段 1 | P02 |

## 3. DEX 加载失败:5 个根因

### 3.1 5 类 DEX 故障

| 故障 | 触发条件 | 错误关键字 |
|---|---|---|
| **类找不到** | R8 strip / multidex 错 | `ClassNotFoundException` |
| **Verify 错误** | 字节码非法 | `VerifyError` |
| **NoClassDefFoundError** | 类存在但链接失败 | `NoClassDefFoundError` |
| **NoSuchMethodError** | 方法签名变更 | `NoSuchMethodError` |
| **dex2oat 失败** | DEX 太大 / 内存不够 | `dex2oat failed: out of memory` |

### 3.2 故障 1:类找不到

**典型症状**(本系列 P07 §0.1):
```
E AndroidRuntime: FATAL EXCEPTION: main
E AndroidRuntime: java.lang.ClassNotFoundException: Didn't find class "com.thirdparty.SDKManager"
```

**根因诊断**:

```bash
# 1. 看 APK 是否包含这个类
$ baksmali disassemble app.apk -o smali/
$ find smali/ -name "SDKManager.smali"
# 空 → R8 strip 了

# 2. 看 R8 配置
$ cat proguard-rules.pro
# 确认 -keep class com.thirdparty.** { *; }
```

**修复**:
```proguard
-keep class com.thirdparty.** { *; }
-keep class com.thirdparty.**$* { *; }
```

**详细流程**:见 [PLE 07 §8.2](07-ART-ClassLoader体系-从BootClassLoader到PathClassLoader.md)

### 3.3 故障 2:VerifyError

**典型症状**(本系列 P08 §0.1):
```
E AndroidRuntime: FATAL EXCEPTION: main
E AndroidRuntime: java.lang.VerifyError: Rejecting class com.example.MyClass
```

**根因诊断**:

```bash
# 1. 看 ART verify 日志
$ adb logcat -d | grep -i "verify" | head -30

# 2. 看 smali
$ baksmali disassemble app.apk -o smali/
$ cat smali/com/example/MyClass.smali | head -50

# 3. 看是不是第三方库
$ cat mapping.txt | grep "MyClass"
```

**修复**:
- 升级第三方库(修复字节码)
- 加 R8 -keep
- 用 `android:vmSafeMode="true"` (debug only)

**详细流程**:见 [PLE 08 §3.4](08-类加载生命周期-Loading-Linking-Initializing.md)

### 3.4 故障 3:NoClassDefFoundError

**典型症状**:
```
E AndroidRuntime: java.lang.NoClassDefFoundError: com.example.MyClass
    at MyClass.<clinit>(MyClass.java:10)
```

**根因**:**`<clinit>` 失败,导致类永远不可用**。

**诊断**:

```bash
# 1. 看 ART 日志
$ adb logcat -d | grep -i "class.*def\|<clinit>\|ExceptionInInitializer"
```

**修复**:
- 修复 `<clinit>` 中的失败逻辑
- 重启进程

### 3.5 故障 4:NoSuchMethodError

**典型症状**:
```
E AndroidRuntime: java.lang.NoSuchMethodError: 
    com.example.MyClass.someMethod
```

**根因**:**方法签名变更**(SDK 升级 / 反射调用失败)。

**诊断**:

```bash
# 1. 看 dex2oat 输出
$ readelf -s app.oat | grep "someMethod"

# 2. 看 mapping.txt
$ cat mapping.txt | grep "MyClass"
```

**修复**:
- 同步更新 SDK 版本
- 保留反射调用的方法

### 3.6 故障 5:dex2oat 失败

**典型症状**(本系列 P09 §0.1):
```
I dex2oat: dex2oat took 175.3s for /data/app/~~xyz/base.apk
I PackageManager: Install finished in 180.5s
```

**根因**:**DEX 太大,dex2oat 内存不够**。

**修复**:

```bash
# 1. 增大 dex2oat 内存
$ adb shell setprop dalvik.vm.dex2oat-Xmx 2g

# 2. 用 speed-profile
# 默认是 speed-profile,但部分 OEM 改了

# 3. R8 优化
android {
    buildTypes {
        release {
            isMinifyEnabled = true
            isShrinkResources = true
        }
    }
}

# 4. 不预编译(只 verify)
$ adb install --no-dex2oat app.apk
```

## 4. Resources 加载失败:4 个根因

### 4.1 4 类 Resources 故障

| 故障 | 触发条件 | 错误关键字 |
|---|---|---|
| **资源 ID 找不到** | R8 strip / R.id 错误 | `Resources$NotFoundException` |
| **arsc 解析失败** | arsc 损坏 / 截断 | `Failed to parse resources` |
| **Configuration 静默 fallback** | 资源变体不全 | (无错误,UI 错乱) |
| **Drawable OOM** | Bitmap 太大 | `OutOfMemoryError` |

### 4.2 故障 1:Resources$NotFoundException

**典型症状**(本系列 P10 §4.6):
```
E AndroidRuntime: android.content.res.Resources$NotFoundException: 
    String resource ID #0x7f0a0001
```

**根因诊断**:

```bash
# 1. 看 R.id 对应的资源名
$ aapt2 dump resources app.apk | grep 0x7f0a0001

# 2. 看 ProGuard 是否 strip 了
$ cat mapping.txt | grep "R.string.app_name"
```

**修复**:
- 加 `-keep class **.R$* { *; }`

### 4.3 故障 2:arsc 解析失败

**典型症状**:
```
E AndroidRuntime: Failed to load resource: Resources$NotFoundException
```

**根因**:**arsc 文件被截断 / CRC 校验失败**。

**诊断**:

```bash
# 1. 看 arsc 是否在 APK 中
$ unzip -l app.apk | grep arsc

# 2. 解压 arsc 看大小
$ unzip -p app.apk resources.arsc > arsc.bin
$ ls -la arsc.bin

# 3. 看 arsc 头部
$ hexdump -C arsc.bin | head -1
```

### 4.4 故障 3:Configuration 静默 fallback

**典型症状**:**中文设备显示英文资源**(本系列 P10 §6.4)。

**诊断**:

```bash
$ aapt2 dump resources app.apk | grep "string/app_name"
# (default) "MyApp"
# (zh) "我的应用"
# 没有 (zh-rCN) → zh-CN 设备 fallback 到 zh
```

**修复**:
- 补齐语言变体
- 或只保留 default

### 4.5 故障 4:Drawable OOM

**典型症状**:
```
java.lang.OutOfMemoryError: Failed to allocate a 12 MB byte allocation
    at android.graphics.Bitmap.nativeCreate(Native Method)
```

**根因**:**Bitmap 太大 / Drawable 缓存堆积**。

**修复**:

```java
// 用 BitmapFactory.Options 压缩
BitmapFactory.Options options = new BitmapFactory.Options();
options.inSampleSize = 2;
Bitmap bitmap = BitmapFactory.decodeFile(path, options);

// 用 BitmapPool 复用
BitmapPool pool = BitmapPool.getInstance();
Bitmap bitmap = pool.get(width, height, Config.ARGB_8888);

// 加载完成后及时释放
drawable.setCallback(null);
bitmap.recycle();
```

## 5. 进程侧故障:4 个根因

### 5.1 4 类进程故障

| 故障 | 触发条件 | 错误关键字 |
|---|---|---|
| **Zygote fork 失败** | 系统资源不足 | `Zygote: fork failed` |
| **进程残留 / 僵尸** | 异常退出 | `Process gone` |
| **ActivityThread 启动崩溃** | attach 阶段崩溃 | `FATAL EXCEPTION at ActivityThread.main` |
| **JNI_OnLoad 失败** | libart 启动失败 | `JNI_ERR returned from JNI_OnLoad` |

### 5.2 故障 1:Zygote fork 失败

**典型症状**:
```
E Zygote: Fork failed
E ActivityManager: Failed to start process
```

**根因**:**系统资源不足**(PID 耗尽 / 内存不足 / 文件描述符耗尽)。

**诊断**:

```bash
# 1. 看系统资源
$ adb shell cat /proc/sys/kernel/pid_max
# 32768 → 可能接近上限

# 2. 看进程数
$ adb shell ps -A | wc -l
# 3000+ → 接近上限

# 3. 看内存
$ adb shell cat /proc/meminfo | head -5
```

**修复**:
- 杀进程释放资源
- 调整 pid_max

### 5.3 故障 2:进程残留 / 僵尸

**典型症状**:
```
W ActivityManager: Process com.example.app has crashed too many times
```

**根因**:**App 频繁崩溃,AMS 标记为"bad process"**。

**诊断**:

```bash
# 看 AMS 日志
$ adb logcat -d | grep "ActivityManager.*com.example.app"
# 看 kill 决策
```

**修复**:
- 修复崩溃根因
- 重启设备
- `adb shell am force-stop com.example.app`

### 5.4 故障 3:ActivityThread 启动崩溃

**典型症状**:
```
F libc : Fatal signal 11 (SIGSEGV), code 1 (SEGV_MAPERR), fault addr 0x0
E AndroidRuntime: FATAL EXCEPTION: main
E AndroidRuntime: at android.app.ActivityThread.handleBindApplication
```

**根因**:**Application.attachBaseContext 或 onCreate 阶段崩溃**。

**诊断**:

```bash
# 1. 看完整 crash dump
$ adb logcat -d | grep -A 30 "FATAL EXCEPTION" | head -50

# 2. 看 ART trace
$ adb shell setprop debug.alloc-count-max 10000
$ adb shell am start -W -n com.example.app/.MainActivity
$ adb logcat -d | grep -E "art.*class|art.*method"
```

**详细流程**:见 [PLE 12 §7](12-进程启动全景-Zygote-fork-第一帧.md)

### 5.5 故障 4:JNI_OnLoad 失败

**典型症状**:
```
F libc : Fatal signal 11 (SIGSEGV), code 1 (SEGV_MAPERR)
E linker: "libart.so" calling DT_INIT 0x17ec28
```

**根因**:**libart 启动失败**(本系列 P05 §4.4)。

**诊断**:

```bash
# 1. 看 ART 启动日志
$ adb logcat -d | grep -i "art\|runtime"

# 2. 看 .so 加载日志
$ adb logcat -d | grep -i "linker"
```

**修复**:
- 通常是 libart 本身 bug,需要厂商发包

### 5.6 进程故障速查表

| 异常关键字 | 故障 | PLE 阶段 | PLE 文章 |
|---|---|---|---|
| `Zygote: fork failed` | fork 失败 | 阶段 3 | P12 |
| `Process gone` / `crashed too many times` | 进程残留 | 阶段 3-4 | P12-13 |
| `FATAL EXCEPTION at ActivityThread.main` | attach 崩溃 | 阶段 4 | P12 |
| `JNI_ERR returned from JNI_OnLoad` | libart 启动失败 | 阶段 2 | P05 |
| `am_proc_start` 没有 | 进程没起来 | 阶段 3 | P12-13 |

## 6. 启动期 OOM:5 个根因

### 6.1 5 类启动期 OOM

| 故障 | 触发条件 | 错误关键字 |
|---|---|---|
| **linear alloc 失败** | DEX 加载 | `Failed to allocate a X byte linear-allocation` |
| **scudo reserve 失败** | .so 加载 | `scudo: out of memory` |
| **native 堆 OOM** | 启动期 mmap | `Out of memory: Kill process` |
| **Java 堆 OOM** | Application onCreate | `OutOfMemoryError: Java heap space` |
| **dex2oat OOM** | 安装期 | `dex2oat failed: out of memory` |

### 6.2 故障 1:linear alloc 失败

**典型症状**(本系列 P06 §0.1):
```
W art : Throwing OutOfMemoryError with VFY status: "Failed to allocate a 4194316 byte linear-allocation"
```

**根因**:**DEX 加载需要大块连续内存,启动期内存碎片化**。

**修复**:
- R8 减少 DEX 大小
- multidex 拆分
- 增大 ART 堆

### 6.3 故障 2:scudo reserve 失败

**典型症状**:
```
F libc : Fatal signal 6 (SIGABRT), code -1 (SI_TKILL)
E libc : malloc_failed
```

**根因**:**Native 堆分配失败**。

**修复**:
- 减少 .so 数量
- 减少全局对象

### 6.4 故障 3:native 堆 OOM

**典型症状**:
```
W libc : Out of memory: Kill process 1234 (com.example.app) score 999 or sacrifice child
```

**根因**:**系统内存压力 + App 启动期 mmap 大量内存**。

**修复**:
- 减少 App 自身 mmap
- 增大 App 堆
- 优化启动期 mmap 顺序

### 6.5 故障 4:Java 堆 OOM

**典型症状**:
```
java.lang.OutOfMemoryError: Java heap space
    at java.util.Arrays.copyOf(Arrays.java:...)
```

**根因**:**Application.onCreate 里分配大量对象**。

**修复**:
- 减少 onCreate 里的对象分配
- 用对象池
- 增大 Java 堆

### 6.6 启动期 OOM 速查表

| 异常关键字 | 故障 | PLE 阶段 | PLE 文章 |
|---|---|---|---|
| `linear-allocation` | linear alloc OOM | 阶段 5 | P06 |
| `scudo: out of memory` | scudo OOM | 阶段 1 | P03 |
| `Out of memory: Kill process` | native 堆 OOM | 阶段 1-6 | P03 |
| `Java heap space` | Java 堆 OOM | 阶段 7 | (本系列外) |
| `dex2oat failed: out of memory` | dex2oat OOM | 阶段 5 | P09 |

## 8. 启动期"5 秒定位法"

### 8.1 5 步快速定位

**线上故障的 5 秒定位法**(在工程师查 logcat 时使用):

```
Step 1:看异常关键字
├─ 关键字 → 速查矩阵(§7.1) → PLE 阶段
├─ 1 秒

Step 2:看进程类型
├─ pid 100+ 通常是 system_server
├─ pid 1000+ 通常是 app
├─ 5 秒

Step 3:看 PLE 阶段
├─ execve / linker / JNI_OnLoad / fork / attach / ClassLoader / Resources / 渲染
├─ 1 秒

Step 4:看 PLE 文章
├─ 找到对应文章(本系列)
├─ 看诊断流程
├─ 1-3 分钟

Step 5:看修复方案
├─ 按文章"修复"段操作
├─ 5-30 分钟
```

**总耗时**:**5 秒定位 + 5-30 分钟修复**。

### 8.2 真实案例:5 秒定位

**场景**:凌晨 3 点告警,某 App `ClassNotFoundException`。

**Step 1(1 秒)**:看到 `ClassNotFoundException` → 速查矩阵 → 阶段 5 → P07

**Step 2(1 秒)**:看进程类型 → App 进程(pid 15000+)

**Step 3(1 秒)**:确认是 PLE 阶段 5(ClassLoader 加载)

**Step 4(1 分钟)**:打开 P07 §8.2

**Step 5(10 分钟)**:按文章操作

```bash
# 1. 看 APK 是否包含
$ baksmali disassemble app.apk -o smali/
$ find smali/ -name "SDKManager.smali"
# 空 → R8 strip 了

# 2. 修复 R8 配置
$ cat proguard-rules.pro
# 加 -keep class com.thirdparty.** { *; }
```

**总耗时**:**~12 分钟**。

**架构师必记**:**本系列的价值不是"读完所有文章"**——**而是"5 秒定位 + 5-30 分钟修复"**。

## 10. 架构师视角:启动期故障的 5 个核心洞察

### 10.1 洞察 1:SO + DEX + Resources 占启动期故障 60%

**这 3 类是排查重点**。**对应 PLE 04 + P07 + P10**。

### 10.2 洞察 2:R8 是 60% 故障的根因或解药

**R8 既可能 strip 类导致 ClassNotFoundException,也可能优化 DEX 减少故障**。**关键是配置 R8**。

### 10.3 洞察 3:Android 14 强化了 verify 和签名校验

**Android 14+ 失败率比 Android 10+ 高**。**升级系统时务必做兼容性测试**。

### 10.4 洞察 4:启动期 OOM 多是"碎片化"而非"绝对不够"

**linear alloc 失败 / scudo reserve 失败 通常是碎片化**。**R8 减少 mmap 数量 = 减少碎片**。

### 10.5 洞察 5:本系列的价值 = "5 秒定位 + 5-30 分钟修复"

**不是"读完所有文章"**——**是"异常关键字 → 速查矩阵 → 文章 → 修复"**。
