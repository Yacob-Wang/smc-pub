## 一、为什么写这个系列

### 1.1 现状：Android 工程师读 Kernel 进程源码的痛点

做稳定性架构这几年，我观察到一个普遍现象：

> 团队里大多数人能读懂 `frameworks/base/services/core/java/com/android/server/am/ProcessList.java`，但当问题下沉到 Kernel 层，需要去看 `kernel/sched/fair.c` 的 `pick_next_task_fair` 时，就开始卡壳。

卡壳的原因是 **Kernel 进程子系统本身的教材太散**：

- 《Linux Kernel Development》（Robert Love）是经典，但偏概念，对调度器算法的讲解停在 v3.0 之前；
- 《Understanding the Linux Kernel》覆盖广但 1300 页，进程管理这一块就有 6 章，看不完；
- LWN.net 上的文章（特别是 scheduler 系列的"rotting"文章）单篇质量极高，但缺一条主线串起来；
- Android 特有的部分（UClamp、EAS、LMKD、cgroup freezer）分散在 Google 官方文档和各 OEM 厂商的 wiki 里。

本系列的目标是：**用 13 篇，把"Kernel 进程子系统"这条主线讲透，每一篇都能直接对应到你排查线上问题的某个工具或某个现象**。

### 1.2 主线（一句话贯穿）

> **一个进程，在 Linux Kernel 内部，是如何被"管起来"的——从结构（task_struct）→ 行为（诞生 / 执行 / 退出）→ 调度（被谁挑中跑）→ 控制（被谁约束资源）→ 调试（出问题怎么定位）。**

按这条主线分 5 个阶段、13 篇：

| 阶段 | 篇数 | 主题 | 核心问题 |
|---|---|---|---|
| A 建立总览 | 01-02 | 全景 + task_struct | 拿到地图 |
| B 生命周期 | 03-05 | 诞生 / 执行 / 退出 | 进程怎么活 |
| C 调度 | 06-09 | 基础架构 / CFS / RT-DL / 多核 | 进程怎么跑 |
| D 控制 | 10-12 | cgroup / 信号 / IPC | 进程怎么被管 |
| E 调试收口 | 13 | ftrace / PSI / 实战案例 | 排查怎么用 |

### 1.3 与 Framework 06 的边界契约（一句话）

**Framework 06（Framework 视角的 Kernel 进程接口：procfs / cgroup fs / pidfd）讲的是"从 Android Framework 这层看，能用 Kernel 提供的哪些用户态接口来排查问题"**；**本系列讲的是"这些接口在 Kernel 内部到底是怎么实现的，看完能去读 `kernel/sched/fair.c`"**。

判断标准只有一条：

> **读完后想去看 `kernel/sched/` 或 `kernel/fork.c` → 本系列**
> **读完后想去看 `frameworks/base/services/core/java/com/android/server/am/` → Framework 系列**

详细分工表见 §五。

## 三、五大子系统的边界与协作

### 3.1 子系统职责的本质——它们是"管进程的某个维度"

子系统的本质是 **把进程管理的某个职责抽象成独立模块**：

| 子系统 | 抽象的是什么 | 不管什么 |
|---|---|---|
| 进程生命周期 | 进程的"生老病死" | 不管进程跑得快不快 |
| 调度 | 进程"什么时候跑、跑在哪颗 CPU 上" | 不管进程消耗多少内存 |
| 信号 | 进程之间的"异步事件通知" | 不管进程之间怎么共享数据 |
| cgroup | 进程"能消耗多少资源" | 不管进程之间怎么通信 |
| 命名空间 | 进程"看到的世界" | 不管进程怎么调度 |

每个子系统有自己的"账本"：

- 调度子系统账本 = `struct rq` + `struct cfs_rq` + `struct rt_rq`
- 内存子系统账本（不展开）= `struct mm_struct`
- cgroup 子系统账本 = `struct cgroup_subsys_state`
- 信号子系统账本 = `struct signal_struct` + `struct sighand_struct` + `struct sigpending`
- 命名空间账本 = `struct nsproxy`

### 3.2 协作模式：task_struct 作为枢纽

五大子系统之间 **不直接互相调用**。它们之间的协作通过一个数据结构枢纽——`struct task_struct`：

```c
// include/linux/sched.h  简版（真实字段 600+ 行）
struct task_struct {
    /* ---------- 调度子系统账本 ---------- */
    struct sched_entity     se;       // CFS 调度实体（fair.c）
    struct sched_rt_entity  rt;       // RT 调度实体（rt.c）
    struct sched_dl_entity  dl;       // Deadline 调度实体（deadline.c）
    int                     on_rq;    // 是否在某个 runqueue 上
    int                     prio;     // 动态优先级
    int                     static_prio;
    unsigned int            policy;   // SCHED_NORMAL / SCHED_RT / SCHED_DEADLINE ...

    /* ---------- 内存子系统账本（不展开）---------- */
    struct mm_struct       *mm;
    struct mm_struct       *active_mm;

    /* ---------- 文件子系统账本（不展开）---------- */
    struct files_struct    *files;
    struct fs_struct       *fs;

    /* ---------- 信号子系统账本 ---------- */
    struct signal_struct   *signal;
    struct sighand_struct  *sighand;
    sigset_t                blocked;
    sigset_t                real_blocked;
    sigset_t                saved_sigmask;

    /* ---------- cgroup 子系统账本 ---------- */
    struct css_set         *cgroups;

    /* ---------- 命名空间账本 ---------- */
    struct nsproxy         *nsproxy;

    /* ... 还有凭证 / io / ipc / audit 等 50+ 子结构 ... */
};
```

**关键认知**：

1. **调度子系统**改的是 `se` `rt` `dl` `prio` `policy` 这组字段——这些字段怎么改、什么时候改、改完有什么效果，是 06-09 篇的核心。
2. **信号子系统**改的是 `signal` `sighand` `blocked` 这组字段——信号怎么排队、怎么处理、什么时候强制投递，是 11 篇的核心。
3. **cgroup 子系统**改的是 `cgroups` 指针指向的 `css_set`——账本怎么算、什么时候拒绝继续消耗资源，是 10 篇的核心。
4. **生命周期子系统**改的是 `state` `exit_state` `pid` `comm` 这组字段——状态机怎么转、退出时怎么通知父进程，是 03-05 篇的核心。

02 篇会把这个枢纽展开成"task_struct 全景拆解"，每一组字段配一组"实战对应"。本篇先建立这个认知就够了。

### 3.3 子系统间的耦合点——两处必须知道

虽然五大子系统不直接互相调用，但有两处**显式耦合**必须知道，否则读源码时会绕：

#### 耦合点 A：调度子系统需要"通知"信号子系统

场景：一个进程在等信号（`ps` 看是 `S` 状态），信号到了，调度子系统要把这个进程唤醒。

代码路径：

```
信号投递（kernel/signal.c）
  └─ signal_wake_up()
      └─ kick_process() / wake_up_state()
          └─ try_to_wake_up() ← kernel/sched/core.c
              └─ enqueue_task() / activate_task()
                  └─ 进程状态从 S 变 R，进入 runqueue
```

**关键点**：信号子系统不直接调用调度子系统，它通过 `wake_up_state()` 这种通用唤醒接口，而该接口最终落到调度子系统的 `try_to_wake_up()`。

#### 耦合点 B：调度子系统需要"尊重" cgroup 的资源上限

场景：一个 cgroup 把 memory 限额设成 1GB，进程触发了 OOM，调度子系统要把这个进程挑出来杀掉。

代码路径：

```
进程分配内存（mm/ 页分配器）
  └─ try_charge() / mem_cgroup_charge()
      └─ mem_cgroup_oom() ← kernel/cgroup/memory.c
          └─ mem_cgroup_oom_select_bad_process()
              └─ 选中一个进程 → oom_kill_process() ← kernel/oom.c
                  └─ 发送 SIGKILL（信号子系统）
                      └─ schedule() 重新调度（调度子系统）
```

**关键点**：cgroup 的"账本"由内存子系统维护，但"超出上限杀进程"这个动作会触发信号子系统 + 调度子系统。这也是为什么本系列 10 篇（cgroup）会和 11 篇（信号）有交叉引用。

## 四、task_struct 作为枢纽（本章埋的钩子）

### 4.1 task_struct 为什么是枢纽

把 3.2 节的表做一个 **反向解读**：

| 子系统 | 看 task_struct 的哪个字段 | 改这个字段的时机 |
|---|---|---|
| 调度 | `se` `rt` `dl` `policy` `prio` | 入队 / 出队 / 优先级改变 / 时间片用尽 |
| 内存 | `mm` `active_mm` | fork / exec / exit |
| 文件 | `files` `fs` | open / close / chdir / fork |
| 信号 | `signal` `sighand` `blocked` | signal 投递 / sigaction 设置 / 解阻塞 |
| cgroup | `cgroups`（指针指向 css_set） | 进程创建 / 移动到新 cgroup |
| 命名空间 | `nsproxy` | clone（CLONE_NEWNS / CLONE_NEWPID 等） |
| 凭证 | `cred` `real_cred` | setuid / setgid / 切换用户 |
| 审计 | `audit_context` | 系统调用进入 / 退出 |

每个子系统对 task_struct 的"看法"不一样——**这就是为什么 task_struct 是枢纽，而不是说所有子系统共享同一份数据**。它是"会面点"，不是"会议室"。

### 4.2 给 02 篇留的钩子

02 篇《task_struct 全景拆解》会做三件事：

1. 把 task_struct 的所有字段按子系统分组，给每个字段标"作用 + 排查什么问题时会用到"。
2. 在结尾给一份"任务卡"——给一个真实的稳定性问题（例如 `kswapd0` 持续高 CPU），让你顺着 task_struct 的字段一步步定位。
3. 留下 02 → 03 的钩子：task_struct 怎么从"空"变"满"？03 篇讲 fork。

### 4.3 一个认知陷阱：不要把 task_struct 当成"完整档案"

很多人初学 Kernel 时把 task_struct 当成进程的"完整描述"。**这是一个误导**。

真实的进程状态是 **分散在多个数据结构里的**，task_struct 只是"指针汇聚点"：

- 进程的虚拟地址空间在 `mm_struct`，不在 task_struct
- 进程打开的文件在 `files_struct`，不在 task_struct
- 进程收到的信号 pending 队列在 `signal_struct->shared_pending`，不在 task_struct
- 进程的 cgroup 归属在 `css_set`，不在 task_struct
- 进程的凭证在 `struct cred`，不在 task_struct

task_struct 真正的内容只有：

- **调度相关**：状态、优先级、调度实体
- **生命周期相关**：PID、状态机、退出码
- **关联关系**：指针（指向 mm / files / signal 等）
- **统计信息**：utime / stime / nvcsw / nivcsw 等

**为什么这么设计？** 因为很多资源（地址空间、文件表、信号表）在进程退出时仍可能被子进程继承，需要独立管理。task_struct 只在进程还"活着"时存在（do_exit 时大部分字段会被清理）。

### 5.3 一个具体例子

**线上问题**：某 App 启动后 1 秒内被系统杀掉，logcat 显示 `LMK: kill`.

```
Q1: 需要看内核吗？
    A: 是（要确认是 LMKD 触发还是 OOM killer 触发，还是其他）

Q2: 看哪？
    A: 先看 /proc/<pid>/oom_score_adj → Framework 06 §X
       再看 /sys/fs/cgroup/.../memory.events → Framework 06 §4
       确认是 cgroup memory 上限触发 → 本系列 10 篇（cgroup 内核账本）

    如果 cgroup 没触发，看 dmesg 中的 OOM 记录：
       cat /proc/vmstat | grep oom → 本系列 13 篇（OOM 排查）
```

**结论**：**绝大多数稳定性问题，从 Framework 系列入手（特别是 06）；只有 Framework 看完仍定位不到根因时，才下沉到 Kernel 系列**。Kernel 系列不是入门读物，是排查的"最后一公里"。

**重点解读**：

- `State: S (sleeping)` ← 进程状态机（02 篇 + 03-05 篇会展开）

- `Threads: 1` ← task_struct 通过 `thread_group` 链表组织

### 7.2 调度子系统（runqueue 投影到 `/proc`）

```bash
# CPU 调度统计
adb shell "cat /proc/stat | grep -E '^cpu[0-9]+'"
```

输出（典型 4 核）：

```
cpu0 12345 12 3456 67890 100 0 0 0 0 0
cpu1 11234 11 3345 68901 200 0 0 0 0 0
cpu2 12001 13 3400 68500 300 0 0 0 0 0
cpu3 11500 10 3500 68000 400 0 0 0 0 0
```

**字段含义**（`man proc` 查 `/proc/stat`）：

| 字段 | 含义 | 对应调度子系统 |
|---|---|---|
| user | 用户态时间 | 调度器记录  |
| nice | 低优先级用户态时间 | 调度器按 nice 加权 |
| system | 内核态时间 | CFS 计入 vruntime |
| idle | 空闲时间 | 无进程在跑的时间 |
| iowait | 等待 IO 时间 | D 状态累计（13 篇实战） |
| irq / softirq | 中断 / 软中断时间 | 中断子系统不算调度账本 |
| steal | 被其他虚拟机偷走的时间 | 容器场景 |

07-09 篇会深入解释这些时间怎么被 CFS 累计进 vruntime。

### 7.3 cgroup 子系统（账本在 `/sys/fs/cgroup`）

```bash
# 查看当前进程在哪个 cgroup
adb shell "cat /proc/self/cgroup"
```

输出（典型 Android 14）：

```
0::/uid_1000/pid_1234
```

**解读**：

- `0` 表示 cgroup v2（v1 是 `N:N::/path` 多行格式）
- `/uid_1000/pid_1234` 是 cgroup 层级路径
- v2 下所有控制器（memory / cpu / io）统一挂在这棵树

```bash
# 查看这个 cgroup 的 memory 账本
adb shell "cat /sys/fs/cgroup/uid_1000/pid_1234/memory.pressure"
adb shell "cat /sys/fs/cgroup/uid_1000/pid_1234/memory.current"
adb shell "cat /sys/fs/cgroup/uid_1000/pid_1234/memory.events"
```

输出：

```
some avg10=0.00 avg60=0.00 avg300=0.00 total=0
full avg10=0.00 avg60=0.00 avg300=0.00 total=0
```

```
2097152
```

```
low 0
high 0
max 0
oom 0
oom_kill 0
```

**重点解读**：

- `memory.pressure` ← PSI（Pressure Stall Information），13 篇会讲
- `memory.current` ← 当前已用内存字节数
- `memory.events` ← `oom_kill` 计数是这个 cgroup 内被杀进程的累计次数

10 篇会把这套账本的内核实现讲透。

### 7.4 信号子系统（pending 队列在 `/proc`）

```bash
# 查看当前进程的待处理信号
adb shell "cat /proc/self/status | grep -E '^(Sig|Shd)'"
```

输出（已截取）：

```
SigQ:   0/30472    ← 已排队信号数 / 上限
SigPnd: 0000000000000000   ← 线程私有 pending
ShdPnd: 0000000000000000   ← 线程组共享 pending
SigBlk: 0000000000000000   ← blocked 掩码
SigIgn: 0000000000000000   ← 忽略掩码
SigCgt: 0000000000000000   ← 捕获掩码
```

**重点解读**：

- `SigQ: 0/30472` ← 这个上限 `30472` 是 `RLIMIT_SIGPENDING`，超过后 `kill()` 会失败
- `SigPnd` vs `ShdPnd` ← 多线程进程下，私有信号 vs 共享信号的区分（11 篇会展开）

### 7.5 命名空间子系统（投影到 `/proc`）

```bash
# 查看进程的命名空间
adb shell "ls -la /proc/self/ns/"
```

输出：

```
lrwxrwxrwx ... cgroup -> cgroup:[4026531835]
lrwxrwxrwx ... ipc -> ipc:[4026531839]
lrwxrwxrwx ... mnt -> mnt:[4026531841]
lrwxrwxrwx ... net -> net:[4026531992]
lrwxrwxrwx ... pid -> pid:[4026531836]
lrwxrwxrwx ... pid_for_children -> pid:[4026531836]
lrwxrwxrwx ... time -> time:[4026531834]
lrwxrwxrwx ... user -> user:[4026531837]
lrwxrwxrwx ... uts -> uts:[4026531838]
```

**重点解读**：

- 数字 `[402653xxxx]` 是 namespace inode 号，每个进程看到的"世界"由这组 inodes 定义
- 同一 inode 号 → 同 namespace → 互相能看到
- 不同 inode 号 → 不同 namespace → 互相隔离

本系列不单独成篇讲 namespace，但 03 篇会讲 `clone(CLONE_NEWPID)` 怎么创建 PID namespace。
