## 一、信号是什么

### 1.1 信号定位

**信号** = **signal**，Linux 进程间通信的"异步通知"机制。

**本质**：内核发给进程的"软件中断"——通知某个事件发生，让进程决定怎么处理。

**关键认知**：
- 信号是"通知"——不是"数据交换"
- 携带的信息有限（最多 32 位 data）
- 接收方决定怎么响应——terminate / core dump / stop / ignore / custom handler

### 1.2 信号的常见来源

```
来源 1：内核发信号
  - 子进程退出 → 父进程 SIGCHLD
  - 非法内存访问 → SIGSEGV
  - 算术错误 → SIGFPE
  - 非法指令 → SIGILL
  - 终端关闭 → SIGHUP
  - 进程写已关闭的管道 → SIGPIPE

来源 2：用户态发信号
  - kill(pid, sig) → sys_kill
  - pthread_kill(tid, sig) → sys_tkill
  - raise(sig) → kill(getpid(), sig)
  - alarm(n) → SIGALRM
  - Ctrl+C → SIGINT（来自终端）

来源 3：硬件发信号（内核转发）
  - 除 0 → SIGFPE
  - 非法地址 → SIGSEGV
```

### 1.3 信号 vs IPC

| 维度 | 信号 | IPC（pipe / shm / Binder） |
|---|---|---|
| 数据量 | 0 字节 + 32 位 data | 任意字节 |
| 通信模式 | 异步通知 | 同步 / 异步数据交换 |
| 接收方选择 | 不能选择 | 通常点对点 |
| 排队 | 标准信号不排队 | pipe / shm 排队 |
| 适用场景 | 通知 + 简单同步 | 数据交换 |

**关键**：
- 信号不适合大量数据交换——用 IPC
- IPC 不适合"通知"——用信号
- Android 14 上：binder 用 IPC、状态变化用 signal

### 1.4 Android 14 上信号的典型使用

```bash
# 1. 看进程收到的信号
adb shell "cat /proc/<pid>/status | grep -E 'SigQ|SigPnd|ShdPnd|SigBlk|SigCgt|SigIgn'"
# SigQ:   0/15475         ← 排队 / 总配额
# SigPnd: 0000000000000000 ← 进程级 pending
# ShdPnd: 0000000000000000 ← 线程组 pending
# SigBlk: 0000000000000000 ← 阻塞的信号
# SigCgt: 0000000180000000 ← 捕获的信号
# SigIgn: 0000000000000000 ← 忽略的信号

# 2. 看进程注册的 signal handler
adb shell "cat /proc/<pid>/status | grep SigCgt"
# 0000000180000000  ← 二进制表示哪些信号被捕获
# bit 28-31 是 SIGRTMIN-SIGRTMAX
# bit 16 = SIGSTKFLT
# ...
```

**关键**：
- Android 14 上 ART / framework 注册了很多信号 handler
- SigCgt 显示具体哪些信号被捕获
- 这是排查"信号是否被处理"的入口

## 二、信号生命周期

### 2.1 完整生命周期

```
产生（Generate）
  ↓
  内核：sys_kill / 异常 / 内核事件
  ↓
  标记：task->pending.signal 或 shared_pending.signal
  ↓
  pending（排队）
  ↓
  唤醒目标 task（如果它在 sleep）
  ↓
  调度目标 task 回到用户态
  ↓
dequeue（取信号）
  ↓
  从 pending 取一个未阻塞的信号
  ↓
handle_signal（处理）
  ↓
  设置用户态栈帧
  ↓
  返回用户态时跳到 handler
  ↓
handler 执行
  ↓
sys_sigreturn
  ↓
返回原执行点
```

**关键**：
- 信号从产生到处理可能要等很久（target 在 sleep）
- 但产生是即时的——内核立刻标记 pending
- 处理时机：target 调度回用户态时

### 2.2 信号的"瞬时状态"

```
产生瞬间：
  signal = SIGKILL
  target_pid = 1234

内核态：
  1. 找到 target (pid 1234)
  2. 检查 signal 是否被屏蔽
  3. 设置 pending.signal.sig[0] |= SIGKILL_MASK
  4. 如果 target 在 sleep，唤醒（wake_up_state）
  5. 设置 TIF_SIGPENDING 标志
  6. 返回

target 调度到时：
  1. 检查 TIF_SIGPENDING
  2. dequeue_signal 取信号
  3. handle_signal 设置栈帧
  4. 返回用户态时跳到 handler
```

**关键**：
- 信号"产生"是异步的——内核只是标记
- "处理"是同步的——target 调度到时
- 处理和产生可能差几秒（target 在 D 状态）

### 2.3 哪些信号会被排队？

```
标准信号（1-31）：
  - 不排队——同一个信号多次发只记一次
  - 例：3 次 SIGTERM 在 pending 里只有 1 个

实时信号（SIGRTMIN-SIGRTMAX）：
  - 排队——每次都加到 pending 尾部
  - 例：3 次 SIGRTMIN+5 在 pending 里 3 个

阻塞期间：
  - 即使多次发，也只在 pending 里记一次（标准）
  - 解阻塞后只 deliver 一次
```

**关键认知**：
- 标准信号：可能丢失——多次发只一次
- 实时信号：不丢失——排队
- Android 14 上大多数信号是标准的

### 2.4 Android 14 上的实际场景

```bash
# 1. 进程被杀
adb shell "kill -9 $(pidof com.example.app)"
# SIGKILL 立即生效——但内核先标记 pending

# 2. 进程收到 SIGSEGV
adb shell "kill -11 $(pidof com.example.app)"
# SIGSEGV 被标记——task 调度到时处理

# 3. ART 主动 SIGABRT
adb logcat -b crash | grep SIGABRT
# ART 调用 raise(SIGABRT) 触发 abort

# 4. system_server 杀应用
adb logcat | grep "killProcess"
# system_server → killProcess → 内核 kill()
```

## 三、信号编号与默认动作

### 3.1 信号编号

```c
// include/uapi/asm-generic/signal.h
#define SIGHUP    1    /* Hangup (terminal closed) */
#define SIGINT    2    /* Interrupt (Ctrl+C) */
#define SIGQUIT   3    /* Quit */
#define SIGILL    4    /* Illegal instruction */
#define SIGTRAP   5    /* Trace/breakpoint trap */
#define SIGABRT   6    /* Abort */
#define SIGBUS    7    /* Bus error */
#define SIGFPE    8    /* Floating point exception */
#define SIGKILL   9    /* Kill (uncatchable) */
#define SIGUSR1   10   /* User-defined 1 */
#define SIGSEGV   11   /* Segmentation fault */
#define SIGUSR2   12   /* User-defined 2 */
#define SIGPIPE   13   /* Broken pipe */
#define SIGALRM   14   /* Alarm */
#define SIGTERM   15   /* Termination */
#define SIGSTKFLT 16   /* Stack fault */
#define SIGCHLD   17   /* Child stopped/terminated */
#define SIGCONT   18   /* Continue */
#define SIGSTOP   19   /* Stop (uncatchable) */
#define SIGTSTP   20   /* Terminal stop (Ctrl+Z) */
#define SIGTTIN   21   /* Background read from tty */
#define SIGTTOU   22   /* Background write to tty */
#define SIGURG    23   /* Urgent data on socket */
#define SIGXCPU   24   /* CPU time limit exceeded */
#define SIGXFSZ   25   /* File size limit exceeded */
#define SIGVTALRM 26   /* Virtual timer */
#define SIGPROF   27   /* Profiling timer */
#define SIGWINCH  28   /* Window resize */
#define SIGIO     29   /* I/O now possible */
#define SIGPWR    30   /* Power failure */
#define SIGSYS    31   /* Bad system call */

// 实时信号
#define SIGRTMIN  32
#define SIGRTMAX  64
```

**关键**：
- 标准信号：1-31
- 实时信号：32-64
- SIGKILL (9) / SIGSTOP (19) 不可捕获——本篇核心

### 3.2 默认动作

```c
// 每个信号的默认动作（kernel/signal.c get_signal）
static int get_signal(struct kernel_siginfo *info)
{
    // ...
    if (sig_kernel_only(sig)) {
        // 内核处理——不返回用户态
        if (sig == SIGKILL)
            do_group_exit(SIGKILL);
        if (sig == SIGSTOP)
            do_group_exit(SIGSTOP);
    }

    // 默认动作
    switch (sig) {
    case SIGHUP:
    case SIGINT:
    case SIGQUIT:
    case SIGILL:
    case SIGTRAP:
    case SIGABRT:
    case SIGBUS:
    case SIGFPE:
    case SIGKILL:
    case SIGSEGV:
    case SIGPIPE:
    case SIGALRM:
    case SIGTERM:
    case SIGUSR1:
    case SIGUSR2:
        // Term + Core (如果是 core dumpable)
        do_group_exit(info->si_signo);
        break;
    case SIGSTOP:
        // Stop
        do_group_exit(SIGSTOP);
        break;
    case SIGCHLD:
        // Ignore
        break;
    // ...
    }
}
```

**关键**：
- 大部分标准信号的默认动作是 Term
- SIGKILL/SIGSTOP 默认 + 不可捕获
- SIGCHLD 默认 Ignore——必须主动 catch

### 3.3 Android 14 上常见信号的处理

| 信号 | 默认 | Android 14 实际 |
|---|---|---|
| SIGSEGV | Term + Core | ART 捕获 → 写 dropbox → Term |
| SIGABRT | Term + Core | ART 捕获 → 写 dropbox → Term |
| SIGBUS | Term + Core | ART 捕获 → 写 dropbox → Term |
| SIGCHLD | Ignore | Zygote 捕获 → 收尸 |
| SIGPIPE | Term | 捕获忽略（TCP 场景） |
| SIGTERM | Term | framework 可能捕获或默认 |
| SIGKILL | Term | 不可捕获——直接 kill |

**关键**：
- ART 注册了 SIGSEGV/SIGABRT/SIGBUS 的 handler
- ART 的 handler 写 dropbox + 调 exit
- Zygote 注册 SIGCHLD——收尸

## 四、sys_kill / sys_tkill / sys_rt_sigqueueinfo

### 4.1 三个系统调用

```c
// kernel/signal.c

// 给整个进程组发信号
SYSCALL_DEFINE2(kill, pid_t, pid, int, sig)
{
    return kill_something_info(sig, &info, pid);
}

// 给指定线程发信号
SYSCALL_DEFINE2(tkill, pid_t, tgid, int, sig)
{
    return do_tkill(tgid, sig, NULL);
}

// 给指定线程发带数据的实时信号
SYSCALL_DEFINE4(rt_sigqueueinfo, pid_t, tgid, int, sig,
                siginfo_t __user *, uinfo)
{
    return do_rt_sigqueueinfo(tgid, sig, uinfo, SEND_SIG_NOINFO);
}
```

**关键差异**：

| 系统调用 | 接收方 | 信号类型 | 带数据？ |
|---|---|---|---|
| `kill` | 进程或进程组 | 标准/实时 | 否 |
| `tkill` | 指定线程 | 标准/实时 | 否 |
| `rt_sigqueueinfo` | 指定线程 | 仅实时 | 是（siginfo_t） |
| `rt_tgsigqueueinfo` | 指定线程 | 仅实时 | 是 |

**关键**：
- `kill` 是最常用——Android 上大多数信号走这个
- `tkill` 给特定线程发——pthread_kill 内部走这个
- `rt_sigqueueinfo` 带数据——但实际很少用

### 4.2 kill 的完整路径

```c
// kernel/signal.c kill_something_info
static int kill_something_info(int sig, struct kernel_siginfo *info, pid_t pid)
{
    // 1. pid > 0：发到指定进程
    if (pid > 0)
        return kill_pid_info(sig, info, find_vpid(pid));

    // 2. pid == 0：发到当前进程组
    if (pid == 0)
        return kill_pgrp_info(sig, info, task_pgrp(current));

    // 3. pid == -1：发到所有进程（init 除外）
    if (pid == -1)
        return kill_all_info(sig, info);

    // 4. pid < -1：发到指定进程组
    return kill_pgrp_info(sig, info, find_vpid(-pid));
}
```

**关键认知**：
- `kill(pid, sig)` 实际有 4 种语义（pid > 0 / == 0 / == -1 / < -1）
- Android 14 上 framework 用 pid > 0——指定进程
- `kill -0` 是"测试进程是否存在"——不真发信号

### 4.3 kill_pid_info 的实现

```c
// kernel/signal.c kill_pid_info
int kill_pid_info(int sig, struct kernel_siginfo *info, struct pid *pid)
{
    // 1. 找 pid 对应的 task
    p = pid_task(pid, PIDTYPE_PID);
    if (!p)
        return -ESRCH;

    // 2. 检查权限
    if (info && task_pid_vnr(p) != info->si_pid)
        // 需要权限——发信号给自己可以，别人不行
        // check_kill_permission 等

    // 3. 投递信号
    return group_send_sig_info(sig, info, p, PIDTYPE_PID);
}
```

**关键**：
- 权限检查：发信号给其他用户进程需要权限
- 内核自己发信号不受限制
- Android 14 上 SELinux 也限制——SELinux context 不允许不能发

### 4.4 group_send_sig_info

```c
// kernel/signal.c group_send_sig_info
int group_send_sig_info(int sig, struct kernel_siginfo *info,
                        struct task_struct *p, enum pid_type type)
{
    // 1. 检查权限
    if (!check_kill_permission(sig, info, p))
        return -EPERM;

    // 2. SIGKILL / SIGSTOP 特判
    if (sig_kernel_only(sig)) {
        // 走 force_sig_info 路径——绕过所有阻塞
        return force_sig_info_to_task(sig, info, p, type);
    }

    // 3. 普通信号——可能阻塞
    return send_signal(sig, info, p, type);
}
```

**关键认知**：
- SIGKILL / SIGSTOP 走 `force_sig_info_to_task`——绕过阻塞
- 其他信号走 `send_signal`——可能被屏蔽
- 这就是 SIGKILL 不可阻塞的根因——12 篇详讲

## 五、signal_struct 与 sighand_struct

### 5.1 两个数据结构的角色

```c
// include/linux/sched/signal.h
// 线程组共享：signal_struct
struct signal_struct {
    refcount_t sigcnt;        // 引用计数
    int nr_threads;           // 线程数
    atomic_t live;            // 活线程数
    // ...

    // 1. shared_pending：线程组共享的 pending 信号
    struct sigpending shared_pending;

    // 2. 进程级信号动作
    // 注意：sighand_struct 单独存在

    // 3. group_exit
    int group_exit_code;
    int group_exit_task;
    int group_stop_count;

    // 4. oom_score_adj / oom_adj
    short oom_score_adj_min;
    short oom_score_adj;
    struct mm_struct *oom_mm;

    // ...
};

// 每个线程独立：sighand_struct
struct sighand_struct {
    spinlock_t siglock;       // 保护信号表
    refcount_t count;         // 引用计数
    struct k_sigaction action[_NSIG];  // 信号动作数组（每个信号一个）
    // ...
};
```

**关键认知**：
- `signal_struct` 是**线程组共享**——所有线程共享
- `sighand_struct` 是**handler 表**——每个线程独立的 handler 表（fork 时共享，exec 时复制）
- `shared_pending` 是线程组共享的 pending——任何线程都能收到

### 5.2 task_struct 中的信号字段

```c
// include/linux/sched/signal.h
struct task_struct {
    // ...
    struct signal_struct *signal;       // 线程组共享
    struct sighand_struct *sighand;     // handler 表
    sigset_t blocked;                   // 阻塞的信号（per-thread）
    sigset_t real_blocked;              // 真实阻塞的信号
    sigset_t saved_sigmask;             // sigsuspend 时的 mask
    struct sigpending pending;          // 进程级 pending（per-thread）
    // ...
};

// fork 时
//   - signal: 线程组共享 → fork 增加 refcount
//   - sighand: 线程组共享 → fork 增加 refcount
//   - blocked: 复制
//   - pending: 新建（per-thread）
//   - action[]: 复制（sighand）
```

**关键**：
- `task->pending` 是 per-thread——只有当前线程能看到
- `task->signal->shared_pending` 是 thread-group-wide
- 阻塞信号是 per-thread——不同线程可以阻塞不同信号

### 5.3 信号的两种 pending

```
thread-group pending（shared_pending）：
  - 发给进程组的信号（kill(pid, sig)）
  - 任何线程都能接收

per-thread pending（task->pending）：
  - 发给指定线程的信号（tkill / pthread_kill）
  - 只有指定线程接收
```

**关键认知**：
- `kill(pid, sig)` 走 shared_pending——任何线程都能 dequeue
- `tkill(tid, sig)` 走 per-thread pending——只有该线程 dequeue
- `dequeue_signal` 先看 per-thread，再看 shared_pending

### 5.4 sigpending 结构

```c
// include/linux/signal.h
struct sigpending {
    struct list_head list;    // 链表头（实时信号排队）
    sigset_t signal;          // 位图（标准信号 + 实时信号）
};

// 实时信号：每个信号一个 sigqueue
struct sigqueue {
    struct list_head list;
    int flags;
    kernel_siginfo_t info;
    struct user_struct *user;
};
```

**关键**：
- `signal`：位图——标记哪些信号有 pending
- `list`：实时信号的 sigqueue 链表——携带具体 data
- 标准信号不排队——位图标记即可

### 5.5 pending signal 怎么被 dequeue

```c
// kernel/signal.c dequeue_signal
bool dequeue_signal(struct task_struct *tsk, sigset_t *mask,
                    kernel_siginfo_t *info, bool *resched_timer)
{
    // 1. 先看 per-thread pending
    // 2. 再看 shared_pending
    // 3. 取一个未阻塞的信号（mask 之外）
    // 4. 如果是实时信号，从 list 取一个 sigqueue
}
```

**关键**：
- dequeue 按信号编号从小到大——优先 SIGTERM / SIGINT / SIGKILL
- 实时信号：按 sigqueue 顺序
- 这是"标准信号不排队"的实现

## 六、信号 mask 与阻塞

### 6.1 sigprocmask 的语义

```c
// kernel/signal.c
SYSCALL_DEFINE3(sigprocmask, int, how, sigset_t __user *, nset,
                sigset_t __user *, oset)
{
    // 1. 读 old mask
    // 2. 根据 how 修改 task->blocked
    // 3. 写回 new mask
    return 0;
}

// how 的取值
#define SIG_BLOCK    0    /* 阻塞 nset 中的信号 */
#define SIG_UNBLOCK  1    /* 解除阻塞 nset 中的信号 */
#define SIG_SETMASK  2    /* 直接设置 mask 为 nset */
```

**关键认知**：
- `sigprocmask(SIG_BLOCK, ...)`：把指定信号加入阻塞
- `sigprocmask(SIG_UNBLOCK, ...)`：解除阻塞——可能立刻 deliver
- `sigprocmask(SIG_SETMASK, ...)`：直接覆盖

### 6.2 阻塞期间的信号去哪了

```
signal 产生
  ↓
  task->blocked 检查
  ↓
  被阻塞：
    标记 pending（不 deliver）
    阻塞期间多次发同一信号：
      - 标准信号：只记 1 个 pending
      - 实时信号：每个 sigqueue 都加到链表
  ↓
  解除阻塞：
    检查 pending
    deliver pending 中的信号
```

**关键**：
- 阻塞的信号都进 pending

- 实时信号不丢失——排队

## 七、dequeue_signal 与 handle_signal

### 7.1 dequeue_signal 取出信号

```c
// kernel/signal.c dequeue_signal
bool dequeue_signal(struct task_struct *tsk, sigset_t *mask,
                    kernel_siginfo_t *info, bool *resched_timer)
{
    struct sighand_struct *sighand = tsk->sighand;
    struct sigpending *pending;
    struct sigqueue *q, *first;
    sigset_t *sigpending = &tsk->pending.signal;
    sigset_t *shrpending = &tsk->signal->shared_pending.signal;
    sigset_t unblocked, *signals;

    // 1. 找到未被阻塞的信号
    unblocked = *sigpending & ~(*mask);
    if (!unblocked) {
        unblocked = *shrpending & ~(*mask);
        if (!unblocked)
            return false;
        pending = &tsk->signal->shared_pending;
        signals = shrpending;
    } else {
        pending = &tsk->pending;
        signals = sigpending;
    }

    // 2. 找最小信号编号
    sig = ffz_unmasked(unblocked, ...);
    // 优先 per-thread pending
    // 然后 shared_pending

    // 3. 标记已处理
    sigdelset(signals, sig);
    // ...

    // 4. 实时信号：从链表取 sigqueue
    if (sig >= SIGRTMIN) {
        first = list_first_entry(&pending->list, struct sigqueue, list);
        list_del(&first->list);
        copy_siginfo(info, &first->info);
        __sigqueue_free(first);
    } else {
        // 标准信号：构造 siginfo
        prepare_signal(SIG_DFL, tsk, sig, info);
    }
    // ...
}
```

**关键**：
- 先 per-thread，后 shared_pending
- 取最小信号编号
- 标准信号不取 sigqueue——只清位
- 实时信号取 sigqueue——携带 data

### 7.2 handle_signal 准备栈帧

```c
// arch/arm64/kernel/signal.c
static void handle_signal(struct ksignal *ksig, struct pt_regs *regs)
{
    struct task_struct *tsk = current;
    struct sighand_struct *sighand = tsk->sighand;
    struct sigframe __user *frame;
    sigset_t *set;
    int ret;

    // 1. 设置 sigframe 在用户栈
    frame = get_sigframe(ksig, regs, sizeof(*frame));

    // 2. 写 siginfo 到栈帧
    if (copy_siginfo_to_user(&frame->info, &ksig->info))
        goto bad;

    // 3. 写 ucontext 到栈帧（保存寄存器）
    if (setup_ucontext(regs, &frame->uc, ...))
        goto bad;

    // 4. 写信号 trampoline
    // (Linux 上是 sa_restorer 或 vDSO)

    // 5. 修改 pt_regs——让返回用户态时跳到 handler
    regs->pc = (unsigned long)ksig->ka.sa.sa_handler;
    regs->regs[0] = ksig->sig;       // x0 = signal number
    regs->sp = (unsigned long)frame;  // sp = sigframe
    regs->regs[29] = ...;             // fp

    // 6. 设置 signal mask
    if (ksig->ka.sa.sa_flags & SA_NODEFER)
        // 不阻塞
        ;
    else
        // 阻塞该信号
        sigaddset(&tsk->blocked, ksig->sig);
    set_current_blocked(&tsk->blocked);
}
```

**关键**：
- 修改 pt_regs：pc = handler、x0 = signal number
- 保存原寄存器到 user stack
- 设置 signal mask（除非 SA_NODEFER）
- 返回用户态时跳到 handler

### 7.3 用户态执行 handler

```c
// 用户态的 signal handler
void my_sig_handler(int sig) {
    // 1. 收到信号——sig 变量是信号编号
    // 2. 处理信号
    // 3. 返回——触发 sys_sigreturn

    // 在 handler 内可调用的函数受限制——async-signal-safe
    // 详见 §10
}
```

**关键**：
- 用户态 handler 收到 signal number 作为参数
- handler 返回时自动触发 sys_sigreturn
- 这个 sys_sigreturn 不需要显式调用

### 7.4 sys_sigreturn

```c
// kernel/signal.c sys_rt_sigreturn
SYSCALL_DEFINE0(rt_sigreturn)
{
    struct pt_regs *regs = current_pt_regs();
    struct rt_sigframe __user *frame;
    sigset_t set;
    unsigned long uc_flags;

    // 1. 读 sigframe
    frame = (struct rt_sigframe __user *)(regs->sp - sizeof(long));

    // 2. 恢复 signal mask
    if (__get_user(set.sig[0], &frame->uc.uc_sigmask.sig[0]))
        goto badframe;
    set_current_blocked(&set);

    // 3. 恢复 ucontext（所有寄存器）
    if (restore_sigcontext(regs, &frame->uc.uc_mcontext, uc_flags))
        goto badframe;

    // 4. 恢复 aarch64 寄存器 + 浮点寄存器
    // ...

    // 5. 返回用户态——回到原执行点
    return regs->regs[0];

badframe:
    signal_fault(regs, frame, "rt_sigreturn");
    return 0;
}
```

**关键**：
- sys_sigreturn 恢复原寄存器 + signal mask
- 返回用户态时跳到原执行点——handler"返回"不是真的返回
- handler 内修改的栈变量不影响原流程

### 7.5 用户态栈帧：sigframe 结构

```c
// 用户态栈帧（处理信号时的栈结构）
struct sigframe {
    char __user *pretcode;          // 返回地址（指向 sigreturn trampoline）
    struct siginfo info;            // 信号信息
    struct ucontext uc;             // 上下文
};

struct ucontext {
    unsigned long uc_flags;
    struct ucontext *uc_link;
    stack_t uc_stack;                // sigaltstack
    struct sigcontext uc_mcontext;  // 寄存器上下文
    sigset_t uc_sigmask;            // signal mask
    // ...
};

struct sigcontext {
    unsigned long fault_address;    // 出错地址（SIGSEGV）
    // 32 个通用寄存器
    __u64 regs[31];
    // FP 状态
    __u8 fp_regs[512]; // 视架构
    // ...
};
```

**关键认知**：
- sigframe 是"信号上下文快照"——所有寄存器
- sys_sigreturn 恢复 sigframe——回到原执行点
- sigframe 在用户栈上——handler 完成后栈恢复

### 7.6 SA_RESTORER 标志

```c
// kernel/signal.c setup_rt_frame
static int setup_rt_frame(int usig, struct ksignal *ksig,
                           sigset_t *set, struct pt_regs *regs)
{
    // 如果 SA_RESTORER 标志设置
    if (ksig->ka.sa.sa_flags & SA_RESTORER) {
        // 用用户态提供的 trampoline
        restorer = ksig->ka.sa.sa_restorer;
    } else {
        // 用 vDSO 提供的 trampoline
        restorer = (void __user *)current->mm->context.vdso_base + vdso_offset_sigreturn;
    }

    // 写 restorer 到栈帧
    if (put_user(restorer, &frame->pretcode))
        return -EFAULT;
}
```

**关键**：
- 用户态 glibc 提供 SA_RESTORER——避免 vdso 依赖
- Android 14 上 Bionic 用 vDSO trampoline
- 这是为什么 signal 处理不依赖 vdso 也能工作

## 八、SIGKILL 不可捕获的底层原因

### 8.1 sig_kernel_only 的特判

```c
// kernel/signal.c sig_kernel_only
static inline bool sig_kernel_only(int sig)
{
    return sig == SIGKILL || sig == SIGSTOP;
}
```

**关键**：
- `SIGKILL` 和 `SIGSTOP` 是 kernel-only——handler 不能是用户态
- 即使 `signal(SIGKILL, handler)` 也会失败
- 这是 kernel 强制——sigaction 拒绝注册 SIGKILL handler

### 8.2 sigaction 拒绝 SIGKILL

```c
// kernel/signal.c do_sigaction
static int do_sigaction(int sig, struct k_sigaction *act,
                         struct k_sigaction *oact)
{
    struct k_sigaction *s;
    struct task_struct *tsk = current;

    // 1. 检查合法性
    if (sig_kernel_only(sig) || sig < 0 || sig >= _NSIG) {
        // SIGKILL / SIGSTOP 拒绝
        return -EINVAL;
    }

    // 2. 设置 action
    s = &tsk->sighand->action[sig - 1];
    // ...
}
```

**关键**：
- `signal(SIGKILL, my_handler)` 返回错误
- `sigaction(SIGKILL, ...)` 返回错误
- 这是从用户态的第一个保护

### 8.3 force_sig_info_to_task 绕过 handler

```c
// kernel/signal.c force_sig_info_to_task
int force_sig_info_to_task(struct kernel_siginfo *info,
                            struct task_struct *t, enum pid_type type)
{
    struct sighand_struct *sighand;
    // ...

    // 1. 不经过 normal signal path
    spin_lock(&sighand->siglock);

    // 2. 直接强制 deliver——绕过阻塞和 ignore
    if (t->signal->group_exit_task)
        goto out;

    // 3. 关键：直接 do_send_sig_info
    ret = send_signal_locked(true, info, t, type);

    // 4. 强制唤醒
    wake = (info->si_signo == SIGKILL) || ...;
    if (wake)
        signal_wake_up(t, info->si_signo == SIGKILL ? TASK_DEAD : 0);

out:
    spin_unlock(&sighand->siglock);
    // ...
}
```

**关键**：
- `force_sig_info_to_task` 是 kernel 强制路径
- 绕过 blocked mask
- 绕过 SIG_IGN
- 直接 dequeue + do_group_exit

### 8.4 dequeue 时 SIGKILL 不可被屏蔽

```c
// kernel/signal.c get_signal
static int get_signal(struct kernel_siginfo *info)
{
    // ...
    // 即使 blocked mask 包含 SIGKILL，dequeue 也优先取
    unblocked = *sigpending & ~tsk->blocked.sig[0];
    // 但 SIGKILL 不会被 blocked
    // 因为 task->blocked 不能包含 SIGKILL
    // ...
}
```

**关键**：
- `task->blocked` 不可能包含 SIGKILL——内核不允许
- 即使强行设置也会被忽略
- 这是 SIGKILL 不可阻塞的根本

### 8.5 SIGKILL 立刻执行——不可推迟

```c
// kernel/signal.c do_group_exit
void do_group_exit(int exit_code)
{
    // ...
    // 直接退出——不可推迟
    do_exit(exit_code);
}
```

**关键认知**：
- SIGKILL 在 get_signal 路径上立刻 do_group_exit
- 即使 task 在 atomic 上下文也能立刻 kill（D 状态除外）
- 这就是 SIGKILL "强制杀进程"的本质

### 8.6 SIGSTOP 与 SIGCONT

```c
// SIGSTOP 默认动作：停止进程
// kernel/signal.c get_signal
case SIGSTOP:
    do_group_exit(SIGSTOP);
    break;

// 实际上是 do_signal_stop
static int do_signal_stop(int signr)
{
    // 1. 设置 TASK_STOPPED 状态
    current->jobctl |= JOBCTL_STOP_DEQUEUED;
    // 2. 通知父进程（SIGCHLD）
    // 3. 调度器跳过该 task
    // 4. 等 SIGCONT 唤醒
}
```

**关键**：
- SIGSTOP 让 task 进入 TASK_STOPPED——不调度
- SIGCONT 唤醒 task
- SIGSTOP 不可捕获——用户态不能注册 handler

## 九、实时信号 vs 标准信号

### 9.1 实时信号（SIGRTMIN-SIGRTMAX）

```c
// include/linux/signal.h
#define SIGRTMIN  32
#define SIGRTMAX  _NSIG // 通常是 64
// Android 14 上 _NSIG = 64

// 实时信号数量
#define RT_SIG_MAX  (SIGRTMAX - SIGRTMIN + 1)
```

**关键**：
- 实时信号 33 个（SIGRTMIN 到 SIGRTMAX，Linux 5.10 上）
- Android 14 用 rt_sigtimedwait / sigtimedwait 等待实时信号
- libc / Bionic 几乎不直接用——但 ART 可能用

### 9.2 实时信号的核心特点

| 维度 | 标准信号 | 实时信号 |
|---|---|---|
| 编号 | 1-31 | 32-64 |
| 排队 | 不排队（合并） | 排队（sigqueue 链表） |
| 数据 | 无 | 携带 siginfo_t |
| 投递 | 一次 | 每次 sigqueue 都投递 |

**关键**：
- 标准信号：丢失信息（多次发只 1 次）
- 实时信号：不丢失（每个 sigqueue 都投递）
- Android 14 上 framework 主要用标准信号——实时信号是"高级"用法

### 9.3 实时信号的使用场景

```
使用场景：
1. 数据通信
   - 用 SIGRTMIN 携带 32 位数据
   - 不需要 pipe / socket

2. 多线程同步
   - 一个线程等信号——A 发 SIGRTMIN+B 携带 id
   - B 醒来后知道是哪个 id

3. 实时通知
   - 多次发不同信号——全部投递
```

### 9.4 Android 14 上的实时信号

```bash
# 看进程捕获的实时信号
adb shell "cat /proc/<pid>/status | grep SigCgt"
# 假设看到 0xfffff000
# 高位对应实时信号

# ART 可能用实时信号做内部通信
# 不过 framework 层主要用标准信号
```

**关键**：
- Android 14 上实时信号用得少
- 大多数场景用标准信号
- ART 内部有部分用实时信号——但很少

### 10.1 为什么 printf 在 signal handler 里危险

// 危险代码

## 十一、信号与线程

### 11.1 pthread_kill 与线程信号

```c
// 给指定线程发信号
int pthread_kill(pthread_t thread, int sig);

// 内部走 sys_tkill
// kernel/signal.c do_tkill
static int do_tkill(pid_t tgid, int sig, struct kernel_siginfo *info)
{
    // 1. 找到指定 tid
    // 2. 检查 task 是否活着
    // 3. 发信号——走 per-thread pending
    return send_signal_info(sig, info, p, PIDTYPE_PID);
}
```

**关键**：
- pthread_kill 走 per-thread pending
- 只有指定线程能收到
- 这是 Java Thread.interrupt() 的内核路径

### 11.2 kill 给进程 vs 线程

```c
// kill(pid, sig) 给进程组：
//   → 走 shared_pending
//   → 任意线程都能 dequeue

// pthread_kill(tid, sig) 给线程：
//   → 走 per-thread pending
//   → 只有该线程能 dequeue
```

**关键认知**：
- 同一个信号号，kill 和 pthread_kill 是不同路径
- Android 14 上 ART 通常用 kill——便于 broadcast
- pthread_kill 主要用于线程特定场景

### 11.3 Java Thread.interrupt 的底层

```java
// Java
Thread.currentThread().interrupt();  // 给当前线程发 SIGINT? 不对，是设置中断标志

// 实际上是：Thread.interrupt() 不发信号——只是设置 Java 标志
// 但底层 pthread_kill pthread_self(), SIGUSR1 在某些 ART 实现中
```

**关键**：
- Java interrupt 主要是 Java 层语义
- 底层 ART 可能用 pthread_kill + SIGUSR1
- 大多数情况下 Java interrupt 不发信号

### 11.4 信号与 cgroup

```c
// 给 cgroup 内 task 发信号
// kernel/signal.c __send_signal
// 1. 找到 cgroup
// 2. 遍历 cgroup 内 task
// 3. 给每个 task 发信号
```

**关键**：
- 没有"给整个 cgroup 发信号"的直接 API
- framework 自己遍历 cgroup.procs + kill
- 这是 Android 14 上杀一组进程的方式

## 十三、稳定性排查

### 13.1 信号丢失

```bash
# 症状：handler 没收到信号

# 排查 1：看 signal mask
adb shell "cat /proc/<pid>/status | grep SigBlk"
# SigBlk: 0000000000000004  ← bit 2 = SIGINT 被阻塞

# 排查 2：看 pending
adb shell "cat /proc/<pid>/status | grep -E 'SigPnd|ShdPnd'"
# SigPnd: 0000000000000004  ← SIGINT 在 pending 里
```

**关键**：
- 信号被阻塞 → pending 等解阻塞
- 但解阻塞后只 deliver 一次（标准信号）
- 如果期间发了 10 次——只收到 1 次

### 13.2 handler 死循环

```bash
# 症状：进程卡在 signal handler
# 排查：
# 1. 看进程的栈
adb shell "cat /proc/<pid>/stack"
# 看到 handler 函数的栈帧

# 2. 看进程的状态
adb shell "cat /proc/<pid>/status | grep State"
# State: S (sleeping) 或 R (running)
```

**关键**：
- handler 卡 = 死循环 / 阻塞系统调用
- 无法发 SIGKILL——handler 不响应（SIGKILL 仍能杀）
- SIGKILL 不依赖 handler——直接 kill 进程

### 13.3 SIGSEGV 排查

```bash
# 1. 看 dropbox
adb shell "dumpsys dropbox | grep -A 20 'com.example.app'"

# 2. 看 tombstones
adb shell "ls /data/tombstones/"
adb shell "cat /data/tombstones/tombstone_01"  # 看具体 tombstone

# 3. 看 signal
adb shell "dmesg | grep -i segv"
```

**关键**：
- SIGSEGV 写到 dropbox + tombstones
- tombstone 含寄存器 + 栈 + 内存映射
- Android 14 上 ART 的 crash handler 负责

### 13.4 信号与线程死锁

```c
// 经典死锁：
// 主线程：sigsuspend()
// handler：等主线程释放锁
// 结果：handler 等主线程，主线程等 SIGCONT
// 永远卡死

// 解决：handler 不用锁 / 用 async-signal-safe 函数
```

**关键**：
- handler 中不能用锁——主线程可能持锁
- 这是 signal handler 编程的核心约束

### 13.5 signal storm

```bash
# 症状：大量信号发送，系统慢

# 排查：
adb shell "cat /proc/<pid>/status | grep -E 'SigQ|SigPnd'"
# SigQ: 100/15475  ← 排队很多

# 解决：
# 1. 修复发信号的应用
# 2. 限制发信号频率
# 3. 用 cgroup 隔离
```

**关键**：
- signal storm 罕见但破坏大
- 通常是 bug 引起——大量发同一信号

## 十四、信号与 cgroup OOM 的联动

### 14.1 cgroup OOM 的信号触发

```c
// mm/memcontrol-v1.c mem_cgroup_out_of_memory
// 选 victim 后通过 signal 通知

static bool mem_cgroup_out_of_memory(struct mem_cgroup *memcg, ...)
{
    // 1. 选 victim
    victim = select_victim(memcg);

    // 2. 通过 SIGKILL 杀 victim
    // memcg_oom_reaper 走 OOM killer 路径

}
```

**关键**：
- cgroup OOM 用 SIGKILL 杀进程——强制
- victim 收到 SIGKILL 后走 do_group_exit
- 不可阻挡——OOM killer 强制 kill

### 14.2 LMKD 的信号使用

**关键**：
- LMKD 用 SIGKILL——确保能杀
