## 一、Android 收紧的三大压力

### 1.1 Hidden API 限制的演进

```
┌─────────────────────────────────────────────────────────────┐
│           Hidden API 限制的演进(Android 9 → 14)              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Android 8 及之前:无限制                                    │
│     反射 + 隐藏 API 调用 = 黄金期                            │
│     → 所有第三方 Hook 工具都能用                             │
│                                                             │
│  Android 9:黑名单机制(2018)                                │
│     reflection 访问隐藏 API 抛 NoSuchFieldError             │
│     → 必须用 reflection 黑魔法(异常捕获+重试)              │
│                                                             │
│  Android 10:扩展黑名单(2019)                              │
│     灰名单 + 白名单机制                                    │
│     → OEM Hook 越来越难做                                  │
│                                                             │
│  Android 12:白名单强制(2021)                              │
│     targetSdk=31+ 的 App 强制走白名单                      │
│     → 多数反射都被拦截                                     │
│                                                             │
│  Android 14:进一步收紧(2023)                              │
│     增加更多 op 到黑名单                                   │
│     → 反射访问 ART 内部类几乎不可行                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 SELinux + GKI 收紧的演进

```
┌─────────────────────────────────────────────────────────────┐
│           SELinux + GKI 收紧的演进                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Android 8:引入 Treble + SELinux 强制                       │
│     → 修改系统分区越来越难                                  │
│                                                             │
│  Android 10:GKI(Generic Kernel Image)强制                   │
│     → 内核不再允许随意修改                                  │
│     → 改 Syscall Table 彻底封死                              │
│                                                             │
│  Android 11:vendor Hooks 引入                              │
│     → 提供"官方"的内核扩展点                                │
│     → OEM 必须在 Vendor Hooks 上做 Hook                     │
│                                                             │
│  Android 12:GKI 进一步收紧                                 │
│     → Kernel 升级必须经过 Google 审核                       │
│     → 私自改内核会导致 CTS 失败                             │
│                                                             │
│  Android 14:GKI 2.0                                        │
│     → Kernel 模块签名强制                                   │
│     → 几乎所有"野路子"内核 Hook 都被封死                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 兼容性反噬的演进

```
┌─────────────────────────────────────────────────────────────┐
│           兼容性反噬的演进                                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Android 8-9 时代:Hook 简单,兼容性好                        │
│     → OEM Hook 数量少,误杀率低                             │
│                                                             │
│  Android 10-11 时代:Hook 激增,开始出问题                    │
│     → 后台冻结导致微信抢不到红包                            │
│     → 假数据导致银行 App 拒绝运行                           │
│                                                             │
│  Android 12-13 时代:兼容性问题爆发                          │
│     → 微信双开 Bootloop 案例(详见 15)                     │
│     → OEM 维护成本急剧上升                                  │
│                                                             │
│  Android 14 时代:OEM 不得不砍 Hook                          │
│     → 升级周期延长 12-18 个月                               │
│     → 部分 OEM 选择"少做少错"                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、演进方向 1:运行时 → 编译期

### 2.1 运行时 Hook 的局限性

```
┌─────────────────────────────────────────────────────────────┐
│           运行时 Hook 的 4 大局限性                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① 性能损耗                                                  │
│     每次方法调用都跳到 trampoline → 5-15% 性能损耗          │
│                                                             │
│  ② 兼容性问题                                                │
│     依赖反射访问 ART 内部类,Android 大版本经常破坏           │
│                                                             │
│  ③ 维护成本                                                  │
│     每个 Android 大版本都要重写 ART 兼容层                   │
│                                                             │
│  ④ 调试困难                                                  │
│     运行时 Hook 的崩溃栈很难调试                             │
│                                                             │
│  → OEM 必须迁移到编译期源码修改                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 编译期源码修改的优势

```
┌─────────────────────────────────────────────────────────────┐
│           编译期源码修改的 4 大优势                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① 性能更好                                                  │
│     编译期直接修改字节码,运行时无 trampoline 开销           │
│                                                             │
│  ② 兼容性更稳                                                │
│     编译时决定,不受运行时环境变化影响                       │
│                                                             │
│  ③ 维护成本更低                                              │
│     AOSP 大版本升级时,只需要适配新的 API                    │
│                                                             │
│  ④ 调试更简单                                                │
│     崩溃栈是正常的 Java 栈,不需要特殊工具                   │
│                                                             │
│  → OEM 的主流方案:从运行时迁移到编译期                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 OEM 实际迁移案例

```
┌─────────────────────────────────────────────────────────────┐
│           OEM 实际迁移案例                                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  小米 HyperOS:                                              │
│     旧:MIUI 12 大量运行时 Hook(基于 Xposed 框架)            │
│     新:HyperOS 主要编译期源码修改                            │
│     → 性能提升 30%+,兼容性问题下降 50%                       │
│                                                             │
│  华为 HarmonyOS:                                            │
│     旧:EMUI 11 部分运行时 Hook + 部分编译期                 │
│     新:HarmonyOS NEXT 完全编译期 + 自研框架                  │
│     → 不再依赖运行时 Hook                                   │
│                                                             │
│  OPPO ColorOS:                                              │
│     旧:ColorOS 11 部分运行时 Skia Hook                      │
│     新:ColorOS 14 全部编译期 Skia 修改                      │
│     → 量子动画性能提升 20%                                  │
│                                                             │
│  vivo OriginOS:                                             │
│     旧:OriginOS 1.0 有运行时 ART Hook                       │
│     新:OriginOS 4 全部编译期 + Native Hook                   │
│     → 兼容性大幅提升                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、演进方向 2:Framework → HAL/Kernel

### 3.1 Framework 层 Hook 的局限性

```
┌─────────────────────────────────────────────────────────────┐
│           Framework 层 Hook 的局限性                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① 容易被检测                                                │
│     App 通过反射 / Smali 能检测 Framework 是否被拦截         │
│     → App 加固后的检测越来越严                              │
│                                                             │
│  ② 兼容性问题                                                │
│     Framework 层 API 在 AOSP 大版本经常变化                  │
│     → OEM Hook 经常失效                                     │
│                                                             │
│  ③ 性能开销                                                  │
│     Framework 层 Hook 会增加 IPC 延迟                       │
│     → 用户感知"App 启动慢"                                  │
│                                                             │
│  → OEM 必须下移到 HAL/Kernel 层                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 HAL/Kernel 层 Hook 的优势

```
┌─────────────────────────────────────────────────────────────┐
│           HAL/Kernel 层 Hook 的优势                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① 更难检测                                                  │
│     HAL/Kernel 层接口对 App 完全不可见                      │
│     → App 无法通过反射检测                                  │
│                                                             │
│  ② 更稳定                                                    │
│     HAL/Kernel 接口相对稳定                                 │
│     → AOSP 大版本升级影响小                                 │
│                                                             │
│  ③ 更高效                                                    │
│     HAL/Kernel 层直接操作硬件                               │
│     → 没有 IPC 开销                                         │
│                                                             │
│  → OEM 的演进方向:Framework → HAL → Kernel                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 GKI Vendor Hook:Kernel 层的官方扩展点

```
┌─────────────────────────────────────────────────────────────┐
│           GKI Vendor Hook:Kernel 层的官方扩展点               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Android 11+ 引入 GKI Vendor Hooks:                          │
│  → OEM 不再需要改 Syscall Table                              │
│  → 通过官方预留的 Hook 干预内核                              │
│                                                             │
│  Android 14 Vendor Hooks(android14-5.10):                   │
│  ├── 80+ 个钩子点                                           │
│  ├── 调度器、内存、进程管理全覆盖                           │
│  └── OEM 在 driver/vendor_hooks/ 下实现                     │
│                                                             │
│  主流 OEM 的 Vendor Hook 实现:                              │
│  ├── 华为:30+ 个 Vendor Hook(游戏调度、EROFS 等)            │
│  ├── 小米:20+ 个(HyperOS 调度优化)                          │
│  ├── iQOO/vivo:15+ 个(游戏调度、内存优化)                   │
│  └── OPPO:15+ 个(ColorOS 后台墓碑)                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 四、演进方向 3:OEM 自研 → 官方扩展

### 4.1 Android 12-14 引入的官方扩展机制

```
┌─────────────────────────────────────────────────────────────┐
│           Android 12-14 引入的官方扩展机制                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① AppOps 扩展                                              │
│     Android 12+ 引入 OP_READ_CLIPBOARD 等 op                │
│     → OEM 自研"剪贴板拦截"逐渐被 AppOps 替代               │
│                                                             │
│  ② RRO(Runtime Resource Overlay)扩展                        │
│     Android 10+ RRO API 增强                                │
│     → OEM 自研"资源拦截"被 RRO 替代                         │
│                                                             │
│  ③ Mainline 模块化                                          │
│     Android 10+ Mainline(系统模块化)                       │
│     → 部分系统服务通过 Mainline 更新,OEM 改动失效           │
│                                                             │
│  ④ WindowInsets 扩展                                        │
│     Android 11+ WindowInsets API 增强                      │
│     → OEM 自研"异形屏填充"被 WindowInsets 替代             │
│                                                             │
│  ⑤ TaskFragment(Android 14)                                │
│     Android 14 引入 TaskFragment 官方 API                   │
│     → OEM 自研"折叠屏适配"被官方 API 替代                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 官方扩展机制 vs OEM 自研的对比

| 机制 | 官方 vs 自研 | OEM 迁移路径 |
|---|---|---|
| **剪贴板拦截** | AppOps OP_READ_CLIPBOARD vs OEM 自研 | OEM 自研 → AppOps |
| **资源拦截** | RRO vs OEM 自研 | OEM 自研 → RRO |
| **后台冻结** | AppStandby Bucket + Doze vs cgroup freezer | 保留 cgroup,精简策略 |
| **异形屏** | WindowInsets vs OEM 自研 | OEM 自研 → WindowInsets |
| **折叠屏** | TaskFragment vs OEM 自研 | OEM 自研 → TaskFragment |
| **应用双开** | 多用户模式 vs OEM 自研 | 保留 OEM 自研(官方不支持双开) |

### 4.3 官方扩展机制不能完全替代 OEM 自研

```
┌─────────────────────────────────────────────────────────────┐
│           官方机制不能完全替代 OEM 自研的场景                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① 性能极致优化                                              │
│     官方 API 只提供"通用"能力,无法做"激进"优化              │
│     例:游戏调度、iQOO Monster 模式仍需 OEM 自研             │
│                                                             │
│  ② 隐私保护                                                  │
│     官方 AppOps 太标准化,无法做"假数据"返回                │
│     例:空白通行证仍需 OEM 自研                              │
│                                                             │
│  ③ 后台治理                                                  │
│     官方策略太保守,无法应对国内 App 生态                    │
│     例:cgroup freezer 仍需 OEM 自研,但用官方 API 调用       │
│                                                             │
│  ④ 折叠屏适配                                                │
│     官方 TaskFragment 不够灵活,OEM 仍需自研扩展              │
│     例:平行视界仍是华为自研                                 │
│                                                             │
│  ⑤ 应用双开                                                  │
│     官方不支持双开,完全 OEM 自研                            │
│                                                             │
│  总结:官方机制能替代 60-70% 的 OEM 自研                      │
│       剩余 30-40% 必须 OEM 自研                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 五、Android 12 → 13 → 14 → 15 关键收紧节点

### 5.1 Android 12 (S, 2021)

```
┌─────────────────────────────────────────────────────────────┐
│           Android 12 关键收紧节点                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① Foreground Service 严格化                                │
│     后台启动前台服务必须明确类型                             │
│     → 拦截 startForegroundService 的策略更严格              │
│                                                             │
│  ② Approximate Location 引入                                │
│     模糊定位 vs 精确定位 必须分别申请                       │
│     → 假定位需要返回"模糊"而不是完全空                      │
│                                                             │
│  ③ Hidden API 白名单强制                                    │
│     targetSdk=31+ 必须走白名单                              │
│     → 反射访问 ART 内部类越来越难                           │
│                                                             │
│  ④ Mainline 模块扩展                                        │
│     新增 Advertising ID、Scheduled Job 等模块              │
│     → OEM 部分 Hook 被 Mainline 绕过                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 Android 13 (T, 2022)

```
┌─────────────────────────────────────────────────────────────┐
│           Android 13 关键收紧节点                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ① READ_CLIPBOARD AppOps 引入                               │
│     App 读剪贴板必须明确申请                                 │
│     → OEM 自研"剪贴板拦截"可简化                            │

│     更多 op 进入黑名单                                       │
│     → ART 反射 Hook 风险增大                                │
│                                                             │
│  ④ Photo Picker 引入                                        │
│     系统级照片选择器                                         │
│     → 部分 App 不再需要存储权限                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 Android 14 (U, 2023)

```
┌─────────────────────────────────────────────────────────────┐
│           Android 14 关键收紧节点                             │

│     → OEM 不用自研区域适配                                  │
│                                                             │
│  ④ Hidden API 黑名单扩展                                    │
│     更多 reflection 黑名单                                   │
│     → ART Hook 必须用兼容层                                 │
│                                                             │
│  ⑤ Mainline 模块化进一步推进                                 │

### 5.5 Android 12-15 收紧节点时间线

```

## 九、总结 - 架构师视角的 7 条 Takeaway

1. **Android 收紧是必然趋势**——Hidden API / SELinux / GKI / 官方扩展都在挤压 OEM Hook 空间
2. **运行时 → 编译期是核心演进**——编译期源码修改更稳、更快、更易维护
3. **Framework → HAL/Kernel 是空间转移**——官方扩展机制覆盖 Framework,OEM 必须下移
4. **OEM 自研 → 官方扩展是大势所趋**——但仍有 20-30% 必须 OEM 自研
5. **HarmonyOS NEXT 是"激进派"**——彻底去 Android 化,但失去生态
6. **Android 15+ 的 Private Space / App Pair / SAT** 进一步替代 OEM Hook
7. **OEM Hook 维护成本随 Android 版本指数增长**——必须有专门团队

**演进趋势速查**(遇到新问题时):
```
Android 大版本升级后,OEM Hook 失效
   ↓
5 秒定位:是 Hidden API 限制?ART 字段变化?官方机制替代?
   ↓
应对:
   ├── Hidden API 限制 → 用 reflection 黑魔法
   ├── ART 字段变化 → 动态检测 + 兼容层
   └── 官方机制替代 → 迁移到官方 API
```
