# 1. 背景：为什么四大组件启动是"性能优化金矿"

## 1.1 一句话定位

**四大组件（Activity / Service / BroadcastReceiver / ContentProvider）的启动顺序与启动效率，直接决定用户感知的"启动快慢"**——冷启动 1s 行业基准，每多 100ms 用户可感知，每多 1s 转化率 -7%。

## 1.2 启动期四大组件的 4 个独特性

| 独特性 | 表现 | 后果 |
|:-------|:-----|:-----|
| **高频路径** | 每个 App 启动都走 Activity | 性能瓶颈放大 |
| **强联动** | Activity ↔ Service ↔ Broadcast | 顺序错 = 启动慢 |
| **资源敏感** | onCreate 分配大对象 → GC 触发 | 卡顿高发 |
| **跨进程** | Activity 启动触发 SystemServer | Binder 通信开销 |

## 1.3 行业数据

| 指标 | 数据 | 来源 |
|:-----|:-----|:-----|
| **冷启动平均时间** | 1.5-2.5s | Android Vitals |
| **冷启动 1s 行业基准** | < 1s | 头部 App 目标 |
| **冷启动 3s 用户感知阈值** | > 3s 用户可感知 | 行业研究 |
| **首屏慢 1s 转化率** | -7% | Akamai 研究 |
| **启动期 ANR 占比** | 15-20% 总 ANR | 字节 / 阿里内部数据 |
| **冷热温三态占比** | 60% 冷 / 30% 温 / 10% 热 | 头部 App 实测 |

> **所以呢**：四大组件启动 = 性能优化"金矿"——优化 100ms 即可被用户感知。

# 4. 冷启动 / 热启动 / 温启动 三态判定

## 4.1 三态定义

| 类型 | 进程状态 | 资源状态 | 典型耗时 | 用户感知阈值 |
|:-----|:---------|:---------|:---------|:------------|
| **冷启动** | 进程不存在 | 资源未加载 | 1-2s | > 3s 异常 |
| **热启动** | 进程存在 | 资源在内存 | 200-500ms | > 1s 异常 |
| **温启动** | 进程被杀 | 资源在文件系统 | 500ms-1s | > 2s 异常 |

## 4.2 AMS 冷启动判定

```java
// frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java
public final int startActivityAsUser(...) {
    // 1. 冷启动判定：进程不存在
    ProcessRecord proc = mProcessList.getProcessRecordLocked(
        processName, info.uid, true);
    if (proc == null) {
        // 冷启动：需要创建新进程
        return startActivityCold(...);
    }
    
    // 2. 热启动：进程存在
    if (proc.thread != null) {
        // 热启动：直接复用进程
        return startActivityWarm(...);
    }
    
    // 3. 温启动：进程被杀但资源在文件
    return startActivityTemp(...);
}
```

**冷启动 vs 热启动 vs 温启动的关键差异**：

| 步骤 | 冷启动 | 热启动 | 温启动 |
|:-----|:-------|:-------|:-------|
| 进程创建 | ✅ Zygote fork | ❌ 跳过 | ✅ Zygote fork |
| dex2oat | ✅ 全量加载 | ❌ 已在内存 | ✅ 文件系统加载 |
| 类加载 | ✅ 全量加载 | ❌ 已加载 | ✅ 重新加载 |
| onCreate | ✅ 完整 | ❌ 跳过 | ✅ 重新执行 |
| 资源加载 | ✅ 全量加载 | ❌ 已在内存 | ✅ 文件系统加载 |
| **典型耗时** | **1-2s** | **200-500ms** | **500ms-1s** |

## 4.3 冷启动 4 步判定（AMS 内部）

```java
// 1. 进程是否存在
ProcessRecord proc = mProcessList.getProcessRecordLocked(
    processName, info.uid, true);

// 2. 进程的 Thread 是否存在
if (proc != null && proc.thread != null) {
    // 热启动：直接调度
    return startActivityHot(...);
}

// 3. 进程的 Provider 是否已发布
if (proc != null && proc.pubProviders != null) {
    // 温启动：部分资源已发布
    return startActivityWarm(...);
}

// 4. 完全冷启动
return startActivityCold(...);
```

> **所以呢**：冷启动 = 进程 + Thread + Provider 都不存在；热启动 = 三者都存在；温启动 = 介于两者之间。

## 5.2 Activity 启动 8 步详解

### Step 1 · Launcher.startActivity(Intent)

**关键事件**：用户点击图标 → Launcher 进程发起 startActivity 请求。

**关键源码**：
- `packages/apps/Launcher3/src/com/android/launcher3/Launcher.java`

**耗时**：< 10ms

### Step 2 · ActivityTaskManager.startActivity（跨进程）

**关键事件**：通过 Binder IPC 调用 `IActivityTaskManager.startActivity`。

**关键源码**：
- `frameworks/base/services/core/java/com/android/server/wm/ActivityTaskManagerService.java`
- `frameworks/base/core/java/android/app/IActivityTaskManager.aidl`

**耗时**：1-5ms（Binder IPC）

### Step 3 · AMS.startActivityLocked

**关键事件**：AMS 创建 ActivityRecord + 处理启动模式 + 任务栈管理。

**关键源码**：
- `frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java`
- `frameworks/base/services/core/java/com/android/server/wm/ActivityStack.java`

**耗时**：5-20ms

### Step 4 · ActivityStarter.execute

**关键事件**：ActivityStarter 进一步处理启动逻辑——权限校验 + 启动模式处理。

**关键源码**：
- `frameworks/base/services/core/java/com/android/server/wm/ActivityStarter.java`

**耗时**：5-10ms

### Step 5 · 进程创建（仅冷启动）

**关键事件**：如果进程不存在，通过 Zygote fork 创建新进程。

**关键步骤**：
- `ActivityThread.main()`：主线程入口
- `attachApplication()`：绑定到 AMS
- `bindApplication()`：创建 Application
- `Application.onCreate()`：Application 初始化

**关键源码**：
- `frameworks/base/core/java/android/app/ActivityThread.java`
- `frameworks/base/core/java/android/app/Application.java`
- [A04 §3 Zygote](SystemServer/A04-Zygote+SystemServer.md)（Zygote 详解）

**耗时**：100-300ms（冷启动）/ 0ms（热启动）

### Step 6 · Activity.onCreate

**关键事件**：Activity 生命周期第一站——onCreate。

**关键源码**：
- `frameworks/base/core/java/android/app/Activity.java`

**耗时**：100-500ms（典型）

**onCreate 优化要点**（B02 详述）：
- **不要做 IO**：磁盘读取放异步线程
- **不要做网络请求**：放异步线程
- **不要做复杂计算**：放异步线程
- **不要 inflate 复杂 layout**：用 ViewStub / ConstraintLayout

### Step 7 · Activity.onResume

**关键事件**：Activity 进入前台——onResume。

**关键步骤**：
- `Activity.onResume()` 回调
- `WindowManager.addView()` 绑定到 WMS
- `ViewRootImpl` 创建
- 通知 WMS 显示窗口

**耗时**：50-200ms

### Step 8 · 第一帧绘制（measure + layout + draw）

**关键事件**：View 树 measure → layout → draw → SurfaceFlinger 合成 → 显示。

**关键源码**：
- `frameworks/base/core/java/android/view/ViewRootImpl.java`
- `frameworks/base/core/java/android/view/Choreographer.java`
- `frameworks/native/services/surfaceflinger/SurfaceFlinger.cpp`

**耗时**：100-500ms（A06 详解）

## 5.3 Activity 启动耗时分布（AOSP 17 实测）

| 阶段 | 耗时 | 占比 |
|:-----|:----:|:----:|
| **Step 1-2 Launcher 发起** | 10ms | 1% |
| **Step 3-4 AMS 处理** | 30ms | 2% |
| **Step 5 进程创建** | 200ms | 15% |

# 6. Service / Broadcast / ContentProvider 启动详解

## 6.1 Service 启动

**启动方式**：
- `startService()`：startService 模式（onCreate + onStartCommand）
- `bindService()`：bind 模式（onCreate + onBind）
- `startForegroundService()`：前台 Service（5s 内必须 startForeground）

**关键源码**：
- `frameworks/base/services/core/java/com/android/server/am/ActiveServices.java`
- `frameworks/base/core/java/android/app/Service.java`

**启动期 Service 风险**（C01 详述）：
- **20s ANR**：前台 Service 20s 内未 onCreate
- **200s ANR**：后台 Service 200s 内未 onCreate
- **5s FGS ANR**：前台 Service 5s 内未 startForeground（**AOSP 17 新增**）

**AOSP 17 硬变化**：
- 🆕 **Foreground Service 强化**：必须显式声明 `foregroundServiceType`
- 🆕 **Background Service 限制**：后台启动 Service 抛 `BackgroundServiceStartNotAllowedException`
- 🆕 **FGS ANR 阈值 5s**：前台 Service 必须 5s 内 startForeground

## 6.2 BroadcastReceiver 启动

**启动方式**：
- `sendBroadcast()`：标准广播
- `sendOrderedBroadcast()`：有序广播
- `sendStickyBroadcast()`：粘性广播（AOSP 17 已废弃）

**关键源码**：
- `frameworks/base/services/core/java/com/android/server/am/BroadcastQueue.java`
- `frameworks/base/services/core/java/com/android/server/am/BroadcastRecord.java`
- `frameworks/base/core/java/android/content/BroadcastReceiver.java`

**启动期 Broadcast 风险**（C01 详述）：
- **10s ANR**：前台 Broadcast 10s 内未 onReceive
- **60s ANR**：后台 Broadcast 60s 内未 onReceive
- **BOOT_COMPLETED 10s**：启动期 BOOT_COMPLETED 10s 内必须处理完

**AOSP 17 硬变化**：
- 🆕 **BOOT_COMPLETED 限流**：所有 BOOT_COMPLETED 接收器 10s 内必须处理完
- 🆕 **Implicit Broadcast 限制**：禁止隐式广播（除系统白名单）
- 🆕 **Broadcast 队列优化**：`BroadcastQueueModern` 替代 `BroadcastQueue`（部分场景）

## 6.3 ContentProvider 启动

**启动方式**：
- `query()` / `insert()` / `update()` / `delete()`
- ContentProvider 启动时自动 publish（在 `Application.onCreate` 之后）

**关键源码**：
- `frameworks/base/services/core/java/com/android/server/am/ContentProviderHelper.java`
- `frameworks/base/core/java/android/content/ContentProvider.java`

**启动期 Provider 风险**（C01 详述）：
- **10s ANR**：启动期 Provider 10s 内未 publish
- **Provider 死锁**：主线程等 Provider 加载

**AOSP 17 硬变化**：
- 🆕 **Provider 启动并行化**：ContentProvider 启动不再阻塞 Application
- 🆕 **Provider 优先级**：可为 Provider 设置启动优先级
- 🆕 **Lazy Provider**：标记为 lazy 后，仅在被访问时启动

# 7. 启动期关键联动关系

## 7.1 AMS ↔ PMS 联动

**AMS 启动时依赖 PMS**：
- AMS 需要查询包信息（`ApplicationInfo`、`ActivityInfo`）
- PMS 在 `startBootstrapServices()` 中先于 AMS ready

**关键调用**：
```java
// AMS.<init>()
mPackageManager = context.getPackageManager();

// AMS.systemReady() → PMS.queryIntentActivities()
Intent intent = new Intent(Intent.ACTION_MAIN);
intent.addCategory(Intent.CATEGORY_LAUNCHER);
List<ResolveInfo> apps = mPackageManager.queryIntentActivities(intent, 0);
```

**PMS 启动慢的影响**：
- AMS 必须等 PMS ready（增加启动时间 1-3s）
- 所有 App 启动都会等 PMS（10-50ms/启动）

## 7.2 AMS ↔ WMS 联动

**WMS 启动时依赖 AMS**：
- WMS 需要 AMS 管理 Task / Stack
- WMS 在 `startOtherServices()` 中启动（晚于 AMS）

**关键调用**：
```java
// WMS.<init>()
mActivityManager = ActivityManager.getService();

// WMS.addWindow() → AMS 检查权限
int res = mActivityManager.checkPermission(...);
```

**WMS 启动慢的影响**：
- 启动后触摸不响应（IMS 也依赖 WMS）
- 启动后黑屏

## 7.3 WMS ↔ IMS 联动

**IMS 启动时依赖 WMS**：
- IMS 通过 WMS 转发输入事件到 focused window
- IMS 在 WMS 之后启动

**关键调用**：
```java
// IMS.<init>()
mWindowManager = WindowManagerGlobal.getInstance();

// IMS 事件分发 → WMS
mWindowManager.getFocusedWindow();
```

**IMS 启动慢的影响**：
- 启动后触摸不响应
- Input ANR 5s 触发

## 7.4 四大组件 ↔ ART 联动

**所有四大组件都依赖 ART**：
- onCreate / onResume / onReceive 都运行在 ART 之上
- ART GC 触发会卡顿

**ART 启动期风险**：
- **类加载失败**：Zygote 死
- **GC 暂停**：启动卡 50-200ms
- **JIT 失败**：解释执行慢

# 8. 风险地图（与 Stability S01 联动 · 强制）

> **本节是 v4 强制要求**——启动期 ANR 与稳态不同，阈值有差异。

## 8.1 启动期 ANR 阈值（S01 联动 · 与稳态不同）

| ANR 类型 | 稳态阈值 | 启动期阈值 | 差异 |
|:-------|:---------|:----------|:-----|
| **Input ANR** | 5s | 5s | 相同 |
| **Broadcast 前台** | 10s | 10s | 相同 |
| **Broadcast 后台** | 60s | 60s | 相同 |
| **Service 前台** | 20s | 20s | 相同 |
| **Service 后台** | 200s | 200s | 相同 |
| **Provider** | 10s | 10s | 相同 |
| **FGS** | 5s | 5s | 相同（AOSP 17 强化）|
| **BOOT_COMPLETED 接收器** | 10s | 10s | 启动期严格 |
| **onCreate 主线程** | 5s | 5s | 相同（onCreate 卡 = ANR）|

> **关键洞察**：启动期 ANR 阈值**与稳态相同**——但启动期高发，所以占比高。

## 8.2 启动期 ANR 的 5 大根因

| 根因 | 占比 | 表现 |
|:-----|:----:|:-----|
| **onCreate 主线程 IO** | 30% | 磁盘读取 / SharedPreferences |
| **onCreate 主线程网络** | 20% | 网络请求未异步 |
| **ContentProvider 慢** | 15% | Provider publish 慢 |
| **BOOT_COMPLETED 慢** | 15% | 接收器未及时消费 |
| **类加载 / GC 卡顿** | 10% | onCreate 触发 GC |
| **其他** | 10% | 杂项 |

## 8.3 启动期 Service / Broadcast / Provider ANR

| 组件 | 阈值 | 启动期风险 |
|:-----|:-----|:----------|
| **BOOT_COMPLETED 接收器** | 10s | 🔴 启动期 100+ 接收器 |
| **前台 Service** | 20s | 🟡 启动期少 |
| **前台 Service startForeground** | 5s | 🟢 启动期少 |
| **ContentProvider publish** | 10s | 🟡 启动期多 |
| **其他 Broadcast** | 10s | 🟡 启动期少 |

## 8.4 启动期黑屏（C03 联动）

启动期黑屏 = 启动后 1-3s 内屏幕仍黑（无任何画面）：

| 黑屏原因 | 表现 | 怎么查 |
|:-------|:-----|:------|
| **WMS 未 ready** | 卡在 Boot Logo | `dumpsys window` |
| **SurfaceFlinger 卡** | 卡在 Boot Animation | `dumpsys SurfaceFlinger` |
| **AMS 未 startHome** | 启动后一直黑屏 | `dumpsys activity` |
| **Launcher onCreate 卡** | 启动后黑屏但能输入 | `dumpsys activity <pkg>` |
| **onCreate 主线程 IO** | 启动 1-3s 后才显示 | traces.txt |

# 13. 性能优化方向

> **本节为 B01-B04 做铺垫**——四大组件启动优化是性能优化"金矿"。

## 13.1 冷启动优化（B02 详述）

- **SplashScreen**：使用 SplashScreen API（AOSP 12+）
- **onCreate 异步化**：把 IO / 网络 / 复杂计算放异步线程
- **类预加载**：使用 `MultiDex` + 启动期 dex2oat
- **类去重**：依赖 ART 17 类去重
- **延迟初始化**：非关键功能放 `Application.onCreate` 之后

## 13.2 热启动优化（B04 详述）

- **TaskStack 复用**：使用 `singleTask` / `singleTop` 启动模式
- **状态恢复**：使用 `onSaveInstanceState` 恢复
- **避免重启**：尽量复用 Activity

## 13.3 启动期 ANR 优化（C01 详述）

- **BOOT_COMPLETED 接收器精简**：减少 10s 内必须消费的接收器
- **ContentProvider 拆分**：避免大 Provider 阻塞 Application
- **前台 Service 限时**：5s 内 startForeground
- **onCreate 不做 IO**：磁盘读取放异步线程

## 13.4 启动期黑屏优化（C03 详述）

- **SplashScreen 主题**：避免冷启动黑屏
- **启动图片**：使用 launchImage 配置
- **第一帧加速**：onCreate 精简 + ViewStub 优化
