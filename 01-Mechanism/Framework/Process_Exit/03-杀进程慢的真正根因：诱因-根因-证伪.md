## 1. 是什么：4 条判定标准（核心）

### 1.1 问题的提出

线上 P0 故障 30%+ 跟"杀进程"相关，但每次复盘都会发现：**"杀进程慢"的原因众说纷纭**——有人说是"swap 55% 高"，有人说是"rss 太大"，有人说是"进程数多"。

**但这些都是表面的"诱因"，不是真正的"根因"**——把诱因当根因治理是线上常见误区，治理投入打水漂。

**对读者有什么用**：本节给出**4 条判定标准**——任何"杀进程慢"的因素，用这 4 条标尺一卡，立刻知道是诱因还是根因。

### 1.2 4 条判定标准

| 规则 | 含义 | 检验方法 | 通过门槛 |
|---|---|---|---|
| **重现性** | 根因必须能在受控条件下独立重现 | ftrace 抓 trace，删除/保留该因素对比 | 删除该因素 → 慢消失；保留 → 慢重现 |
| **充分性** | 仅有该因素就能触发杀进程慢 | 单变量实验：只造这个因素，看是否触发 | **只造这一个因素就能 10s+** |
| **必要性** | 没有该因素就不会慢 | 阻断该因素，看是否还慢 | 阻断后依然不慢 → 该因素非必要 |
| **可证伪** | 必须有反例能推翻这个说法 | 找"高 swap 50% 但杀进程不慢"的 case | 找到反例 → 充分性不成立 → 不是根因 |

**所以**：4 条规则中**充分性最关键**——"只造这一个因素就能 10s+" 是根因的"硬指标"。

### 1.3 诱因 vs 根因 对照

| 类型 | 满足判定标准 | 角色 | 治理 |
|---|---|---|---|
| **诱因**（必要非充分） | 只满足 1-3 条（通常必要性 + 重现性，但**不满足充分性**） | **放大器** | 减少诱因（但**不解决根因**） |
| **根因**（必要 + 充分） | 满足 4 条 | **真正原因** | 直接治理（**主战场**） |

**对读者有什么用**：当听到"杀进程慢是因为 X"，立刻问 4 条：
- X 满足充分性吗？（"只造 X 就能 10s+"）
- X 满足可证伪吗？（"高 X 场景下不慢的反例有吗"）

只要 1 条不满足，**X 就不是根因**，不要在 X 上投入治理。

### 1.4 反例库 #11 #12 在本篇的应用

**v5 反例 #11 数据堆砌**：很多"杀进程慢"分析会列 10 个参数（swap 55% / rss 170MB / 进程 100+ / zRAM 8GB / ...），但**没有洞察**——看完不知道"所以呢"。

**本篇防御**：每个因素都标"诱因/根因" + "治理投入 ROI"——如果只是诱因，治理投入低；如果根因，治理投入高。

**v5 反例 #12 AI 自嗨**：很多"杀进程慢"分析会写"swap 机制非常精妙"这种空话，但**对读者没用**。

**本篇防御**：每个判断都加"对读者有什么用"段——读者看完能立刻用 4 条标尺卡自己的 case。

## 2. 6 大伪根因的证伪（每个配反例）

> **本节是本篇核心**——6 大"伪根因"每个配 1 个真实反例，证伪"它不是根因"。

### 2.1 伪根因 ①：swap 55% 高使用率

**说法**：杀进程慢是因为系统 swap 使用率 55% 太高，释放 swap 慢导致 do_exit 卡。

**证伪（4 条判定标准）**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ✅ | 高 swap 场景下杀进程确实**可能**慢 |
| 充分性 | ❌ | **只造 swap 55% 不能 10s+**（见 §7 反例实验） |
| 必要性 | ❌ | 没有高 swap 也能 10s+（vma 异常 case） |
| 可证伪 | ✅ | **高 swap 50%+ 场景线上很多，但杀进程慢 case 极少** |

**真实角色**：**诱因**（放大器），不是根因。

**反例（真实场景）**：
- 场景：某设备跑长跑测试 24h，系统 swap 使用率 60%（比 55% 还高）
- 表现：杀进程总耗时 ~200ms（健康范围）
- 为什么：do_exit ②-c swap_free 期间虽然要走 swap_map 锁，但**锁等待时间 ms 级**，不会拖到 10s+

**对读者有什么用**：当听到"杀进程慢是因为 swap"时，**用 §7 反例实验模板**5 分钟内证伪——高 swap 场景下杀进程不慢，反例成立。

**怎么识别这个伪根因**（反向应用）：
- 如果案发 kernel log 有 `binder_alloc: no vma` 关键字 → **真根因是 vma 异常，不是 swap**
- 如果案发 kernel log 有 GPU 资源 flush 慢 → **真根因是 fd 关闭慢，不是 swap**
- 如果两个都没有，但系统 swap 55%+ → **诱因**，治理投入 ROI 低

### 2.2 伪根因 ②：rss 170MB 大

**说法**：杀进程慢是因为进程 rss 170MB 太大，unmap 这么多物理页慢。

**证伪**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ✅ | rss 大场景下 unmap 数量大，可能慢 |
| 充分性 | ❌ | **只造 rss 170MB 不能 10s+** |
| 必要性 | ❌ | rss 30MB 也能 10s+（vma 异常 + swap 78MB） |
| 可证伪 | ✅ | **rss 几百 MB 场景很多，单纯 unmap rss 大不会 10s+** |

**真实角色**：**诱因**（unmap 数量大），不是根因。

**反例（实测）**：
- 实测：rss 200MB 进程，单纯 unmap 在 ARM64 上 ~50ms
- 关键：unmap 主要是 atomic_dec × 44000 PTE，us 级/次
- 200MB / 50MB/s 理想速率 = 4s（**但实际是 ms 级**，因为 atomic_dec 不走 IO）

**对读者有什么用**：当听到"杀进程慢是因为 rss 大"，立刻反问"rss 多大能 10s+"——答案是**单 rss 大根本到不了 10s+**，必须有 swap / vma 异常叠加。

**怎么识别这个伪根因**（反向应用）：
- 案发 DFX `VmRSS` 大，但 `VmSwap` 小 + kernel log 无 `no vma` → **rss 是诱因不是根因**
- 案发 `VmRSS` 不大，但 `VmSwap` 大 + `no vma` → **swap / vma 异常是根因不是 rss**

### 2.3 伪根因 ③：进程数量多

**说法**：杀进程慢是因为桌面上有 20 个 app，Clear All 时杀 20 个进程，叠加成 10s+。

**证伪**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ✅ | 多 task 场景下确实可能慢 |
| 充分性 | ❌ | **20 个 task 独立 do_exit，不互相阻塞** |
| 必要性 | ❌ | 单 task 也能 10s+（Process 09 案就是单 task） |
| 可证伪 | ✅ | **单 task 杀进程慢的 case 很多** |

**真实角色**：**诱因**（叠加），不是根因。

**反例（实测）**：
- 实测：Clear All 杀 20 个 task，总耗时 ~6s（不是 20 × 12s）
- 关键：**每个 task 的 do_exit 独立**，kernel 内核是抢占式调度
- 20 个 task 加起来看似叠加，但**单 task 平均 300ms**（健康范围）

**对读者有什么用**：当听到"杀进程慢是因为 app 多"，立刻反问"单 app 也慢吗"——单 app 慢 → 真根因是 do_exit 内部，不是 app 数量。

**怎么识别这个伪根因**：
- 案发 am_kill 日志只杀单 task，进程也慢 → 进程数量不是根因
- 多 task 加起来 < 单 task × 2 → 没有叠加效应，进程数量不是根因

### 2.4 伪根因 ④：zRAM 压缩比高

**说法**：杀进程慢是因为 zRAM 压缩比高，swap IO 慢。

**证伪**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ❌ | zRAM 压缩比高**不影响 swap_free 路径**（swap_free 是位图操作） |
| 充分性 | ❌ | zRAM 压缩比高不能 10s+ |
| 必要性 | ❌ | 没有 zRAM（裸 swap 设备）也能 10s+（vma 异常） |
| 可证伪 | ✅ | zRAM 压缩比 0% 也能 10s+ |

**真实角色**：**完全无关**。

**反例**：
- AOSP 16 部分设备用裸 swap（无 zRAM）—— 同样能 10s+
- zRAM 压缩比只影响 swap IO 性能，**不影响 swap_free 路径**（swap_free 是位图操作，无 IO）

**对读者有什么用**：当听到"杀进程慢是因为 zRAM"，立刻反问"zRAM 关闭后还慢吗"——**不会解决任何问题**。

### 2.5 伪根因 ⑤：系统 MemFree 80MB

**说法**：杀进程慢是因为系统 MemFree 80MB 太低，direct_reclaim 触发慢。

**证伪**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ✅ | MemFree 低场景下 unmap 路径可能走 direct_reclaim |
| 充分性 | ❌ | **只造 MemFree 80MB 不能 10s+**（释放不需要新分配） |
| 必要性 | ❌ | MemFree 800MB 也能 10s+（vma 异常） |
| 可证伪 | ✅ | **MemFree 低场景很多，单纯 unmap 不会 10s+** |

**真实角色**：**诱因**（叠加），不是根因。

**反例**：
- 关键：**释放不需要新分配**——unmap 是取消页表映射 + 释放 refcount，**不分配新页**
- MemFree 低只让"新分配"走 direct_reclaim，但**unmap 本身不走新分配**
- 除非 unmap 期间**同时要分配**（罕见，②-c swap_free 内的 `free_swap_slot` 可能），但通常 us 级

**对读者有什么用**：当听到"杀进程慢是因为 MemFree 低"，立刻反问"MemFree 高时还慢吗"——**核心慢因不在这**。

### 2.6 伪根因 ⑥：单 app 复杂业务逻辑

**说法**：杀进程慢是因为 app 业务复杂，cleanup 慢。

**证伪**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ❌ | **SIGKILL 不走用户态**，业务逻辑不会跑 |
| 充分性 | ❌ | 业务复杂不导致 SIGKILL 路径慢 |
| 必要性 | ❌ | 业务简单的 app 也能 10s+（system_server 自己杀自己？不可能） |
| 可证伪 | ✅ | SIGKILL 路径上**没有用户态代码** |

**真实角色**：**完全无关**。

**反例**：
- 关键：**SIGKILL 不可被 catch/ignore/block**（POSIX 定义）
- kernel 收到 SIGKILL 后直接 `do_group_exit(SIGKILL)`，**不进入用户态 signal handler**
- 所以 app 业务逻辑再复杂，**杀进程路径上 0 行 app 代码执行**

**对读者有什么用**：当听到"杀进程慢是因为 app 业务复杂"，**直接反驳**——SIGKILL 不走用户态，业务复杂 0 影响。

### 2.7 6 大伪根因总结表

| 伪根因 | 真实角色 | 反例数 | 反例可证伪度 |
|---|---|---|---|
| ① swap 55% 高 | 诱因（放大器） | 2 | 100% |
| ② rss 170MB 大 | 诱因（数量大） | 2 | 100% |
| ③ 进程数量多 | 诱因（叠加） | 1 | 100% |
| ④ zRAM 压缩比高 | 完全无关 | 1 | 100% |
| ⑤ MemFree 80MB | 诱因（叠加） | 1 | 100% |
| ⑥ 业务逻辑复杂 | 完全无关 | 1 | 100% |

**对读者有什么用**：6 大伪根因每个都**有反例**——"swap 55% 是根因"这种说法在 §7 实验下 100% 可证伪。

## 3. 3 大真正根因的判定逻辑

### 3.1 真正根因 ①：vma 状态异常

**是什么**：进程在被杀前 mmap 已被预回收（vma 状态与页表不一致）——vendor `process_reclaim` 预回收过 mmap 但没清 `vm_flags` / `page state`。

**4 条判定标准验证**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ✅ | 案发前 24min kernel log 4 次 `binder_alloc: <pid>: no vma` 直接证据 |
| 充分性 | ✅ | **只造 vma 异常 + 中等 swap 能 5-10s+** |
| 必要性 | ✅ | Process 09 案 12.24s 中 10s 在 ②-b unmap_page_range 反复校验 vm_flags |
| 可证伪 | ✅ | 健康 vma 场景下 unmap_page_range 不会反复校验 → 不会 1s+ |

**真实角色**：**真正根因**。

**关键观测**：
- kernel log `binder_alloc: <pid>: no vma` 是 vma 异常的**直接证据**
- 这种日志在 binder 通信失败时出现，意味着进程的 mmap 已被回收
- unmap 时 vma 状态与页表不一致，unmap 路径反复校验 `vm_flags` 累计 5-8s

**判定流程**（ASCII）：
```
[案发] 杀进程 10s+
    ↓
[kernel log 检索] grep "no vma" /tmp/kernel.log
    ↓
[命中] → vma 异常
    ↓
[验证] unmap_page_range 期间 vm_flags 反复校验
    ↓
[确认] 真正根因 ①
```

**治理**：
- vendor 修复 `process_reclaim` 同步（unmap 后清 `vm_flags`）
- 减少 vma 预回收频率
- kernel log 监控 `binder_alloc: no vma` 关键字

### 3.2 真正根因 ②：fd 关闭慢

**是什么**：慢 FS（fuse / network）或 GPU 资源释放让 `filp_close` 卡 1-2s+。

**4 条判定标准验证**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ✅ | 相机案 lsof 看到 `/dev/ashmem/CameraBuffer` 直接证据 |
| 充分性 | ✅ | **只造 GPU flush 慢能 1-2s+** |
| 必要性 | ✅ | 案发 1.5s 中 ~1s 在 ③ exit_files filp_close |
| 可证伪 | ✅ | 无 GPU 资源 / 慢 FS 的进程不会 1s+ |

**真实角色**：**真正根因**。

**关键观测**：
- `lsof -p <pid>` 看到 `/dev/ashmem` / `/dev/dri` 等 fd → GPU 资源
- `mount | grep fuse` 看到 fuse FS → 慢 FS 候选

**判定流程**：
```
[案发] 杀进程 1-5s
    ↓
[lsof -p <pid>] 看 fd 类型
    ↓
[命中 /dev/ashmem 或 fuse] → fd 关闭慢
    ↓
[确认] 真正根因 ②
```

**治理**：
- App 侧：在 `onPause` / `onTrimMemory(TRIM_MEMORY_UI_HIDDEN)` 主动 release GPU 资源
- 系统侧：SurfaceFlinger 在 App 不可见时主动通知 GPU 释放
- 监控侧：检测 `filp_close` 期间 `dma_buf_release` 耗时

### 3.3 真正根因 ③：资源回收拥堵

**是什么**：cgroup 销毁 / reaper 排队 / 锁竞争让 task_struct 释放慢。

**4 条判定标准验证**：

| 规则 | 通过？ | 证据 |
|---|---|---|
| 重现性 | ✅ | Process 09 案 56.016 "Still waiting ... after 2209 ms" 直接证据 |
| 充分性 | ✅ | **只造 cgroup 锁持有能 24s+** |
| 必要性 | ✅ | am_proc_died 后 24s 还没清理完 |
| 可证伪 | ✅ | 健康 cgroup 状态不会 1s+ |

**真实角色**：**真正根因**（但**不影响 am_proc_died 时点**——只影响 am_proc_died 后的 cgroup 清理）。

**关键观测**：
- am_proc_died 后 24s 仍出现 "Device or resource busy" → cgroup.workqueue 锁竞争
- 但**不影响杀进程主耗时**——12.24s 中 cgroup 销毁只是 async 部分

**对读者有什么用**：这是 Process 09 案的**第三个根因**——前两个是 ② exit_mm 慢（10s），这个是 cgroup 残留（24s+）。两者**不互相阻塞**，但要分开治理。

**治理**：
- vendor 修复 cgroup 配置（`memory.max` / `memory.high` / `cgroup.workqueue` 调度）
- kernel 修复 cgroup_mutex 持有时间过长
- 监控侧：am_proc_died 后 30s 内 cgroup 节点还没清理 → 告警

### 3.4 3 大真正根因总结

| 真正根因 | 影响哪个阶段 | 典型耗时 | 关键观测 | 治理 |
|---|---|---|---|---|
| ① vma 异常 | 阶段 5 ② exit_mm | 5-10s | `binder_alloc: no vma` | 修复 vendor process_reclaim |
| ② fd 关闭慢 | 阶段 5 ③ exit_files | 1-5s | `/dev/ashmem` / fuse | 主动 release GPU 资源 |
| ③ 资源回收拥堵 | 阶段 6 async | 1-24s | am_proc_died 后 24s+ | vendor cgroup 配置 |

**对读者有什么用**：3 大根因分别对应 3 个不同阶段——**不要混在一起治理**。

## 7. 实战案例 3：证伪实验（swap 55% 不是根因）

> **本节是本篇关键——用反例证伪"swap 55% 是根因"这种说法**

### 7.1 实验目标

用 §4 模板证伪"swap 55% 高使用率是杀进程 10s+ 的根因"——4 条判定标准只满足 3 条（不满足充分性）。

### 7.2 实验设计

**变量**：swap 使用率（唯一变量）
**对照**：swap 30%（健康范围）
**实验组**：swap 60%（超过 55% 阈值）
**其他变量控制**：同一个简单 app、fd < 100、健康 vma、MemFree 充足

### 7.3 实验步骤

```bash
# 对照组: swap 30%
echo 1 > /proc/sys/vm/swappiness  # 调高使用 swap
# 制造约 30% swap
stress-ng --vm 4 --vm-bytes 4G &

# 抓 ftrace
echo 1 > events/sched/sched_process_exit/enable
echo 1 > events/sched/sched_process_wait/enable
echo 1 > tracing_on

# 杀简单 app
adb shell am force-stop com.example.test
sleep 5

echo 0 > tracing_on
# 提取 T_exit_mm (对照组)
grep "exit_mmap end" /tmp/trace.log | head -1

# 实验组: swap 60%
# 制造约 60% swap
stress-ng --vm 8 --vm-bytes 8G &
sleep 5

# 同样的杀进程操作
echo 1 > tracing_on
adb shell am force-stop com.example.test
sleep 5
echo 0 > tracing_on
# 提取 T_exit_mm (实验组)
grep "exit_mmap end" /tmp/trace.log | head -1
```

### 7.4 预期结果

| 组 | swap 使用率 | T_exit_mm |
|---|---|---|
| 对照组 | 30% | < 200ms |
| 实验组 | 60% | < 300ms（**没有 10x 放大**） |

**关键观察**：swap 从 30% 提升到 60%，T_exit_mm 增加 < 50%——**不满足"5x 放大"门槛**，**充分性不成立**。

### 7.5 证伪结论

**"swap 55% 高使用率是杀进程 10s+ 的根因" 不成立**：
- 不满足 4 条判定标准的"充分性"（只造 swap 55% 不能 10s+）
- 满足"必要性"（高 swap 场景下杀进程可能慢——swap_free 锁竞争）
- 满足"重现性"（同一设备可重现）
- **不满足"可证伪"的反面**（高 swap 场景下杀进程不慢的反例很多）

**真实角色**：**诱因（放大器）**，不是根因。

### 7.6 这个实验对实际工作的指导

**对读者有什么用**：
- 下次听到"杀进程慢是因为 swap"，**直接用这个实验反证**
- 实验模板可以改成"vma 异常 vs 健康 vma"，"GPU 资源 vs 无 GPU 资源"等
- 5 分钟内能反证一个说法，**避免在诱因上浪费治理投入**

## 8. 怎么把"诱因 / 根因"写到故障报告

### 8.1 故障报告模板

```
[杀进程慢故障报告 v1.0]
- 现象: am_kill → am_proc_died 间隔 X 秒
- 案发设备: 某 Android 16 设备
- 复现步骤: ...

## 诱因(必要非充分,只放慢不造成)
- swap 55% 高使用率
- rss 170MB
- 进程 20 个 task

## 真正根因(必要+充分,可独立重现)
- 真正根因 ① vma 异常: 案发前 24min kernel log 4 次 binder_alloc: no vma
- 真正根因 ② fd 关闭慢: lsof 看到 /dev/ashmem/GPUBuffer
- 真正根因 ③ 资源回收拥堵: am_proc_died 后 24s+ cgroup 残留

## 治理建议
- P0: 修复 vendor process_reclaim 同步(根因 ①)
- P1: 主动 release GPU 资源(根因 ②)
- P2: vendor cgroup 配置优化(根因 ③)
- P3: 减少诱因(ROI 低)
```

### 8.2 关键要点

| 段 | 必填 | 目的 |
|---|---|---|
| 现象 | ✅ | 量化损失 |
| 诱因 | ✅ | 排除 6 大伪根因（不混淆视听） |
| 真正根因 | ✅ | 治理主战场 |
| 治理建议 | ✅ | 标优先级 + ROI |

**对读者有什么用**：故障报告不区分诱因和根因，**治理团队就会把 80% 投入放在诱因上**——这是稳定性治理的常见误区。

### 8.3 治理 ROI 排序

| 根因 | 治理投入 | 治理收益 | ROI |
|---|---|---|---|
| ① vma 异常 | 中（vendor 修复） | 高（消除 10s 慢） | **★★★★★** |
| ② fd 关闭慢 | 低（App 改 onPause） | 中（消除 1-2s 慢） | **★★★★** |
| ③ 资源回收拥堵 | 高（kernel + cgroup） | 中（消除 24s 残留） | **★★★** |
| 诱因（swap/rss/进程数） | 中（系统调优） | 低（不解决慢） | **★** |

**所以**：治理投入**优先根因 ①**——一笔修复解决 10s 慢；治理诱因 ROI 最低。

## 9. 总结 + 跨篇索引

### 9.1 架构师视角 5 条 Takeaway

1. **4 条判定标准是金标尺**——任何"杀进程慢是因为 X"的说法，用"重现性 / 充分性 / 必要性 / 可证伪"卡一下，**不满足 4 条的不是根因**。
2. **6 大伪根因都有可证伪的反例**——swap 55% / rss 170MB / 进程数多 / zRAM / MemFree 80MB / 业务复杂，**每个都有真实反例**。
3. **3 大真正根因对应 3 个不同阶段**——vma 异常（② exit_mm，5-10s）/ fd 关闭慢（③ exit_files，1-5s）/ 资源回收拥堵（⑥ async，1-24s）。**不混在一起治理**。
4. **证伪实验模板（§4）5 分钟可复现**——下次听到"X 是根因"，**用对照实验 + 5x 放大门槛**直接证伪。
5. **治理 ROI 优先根因**——投入一笔修根因 ① 解决 10s 慢；治理诱因 ROI 最低。故障报告必须区分诱因和根因（§8）。
