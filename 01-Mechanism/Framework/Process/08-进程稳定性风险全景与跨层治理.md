> **架构师视角的"终极第一性问题"**:**进程类问题永远不可能从单层定位**——
> - App 层看到"ANR",根因可能在 FWK(AMS 决策慢)、ART(GC 卡)、Kernel(调度延迟)

## 3. 10 大故障 × 4 层根因矩阵

> **核心方法论**:**任何进程类问题,从 4 层逐层排查**——
> - **App 层**:业务代码?对象分配?线程管理?fd 关闭?
> - **FWK 层**:AMS 决策?ProcessList 调度?lmkd 选进程?
> - **ART 层**:GC 卡?JIT 抢占?OAT 缺失?SignalCatcher 卡?
> - **Kernel 层**:cgroup 失配?OOM kill?调度延迟?blk-throttle 节流?

| 故障 # | App 层根因 | FWK 层根因 | ART 层根因 | Kernel 层根因 |
|------|----------|-----------|-----------|------------|
| 1 冷启动慢 | 业务在 `Application.onCreate` 跑 IO | AMS 决策慢 (mLruProcesses 漂移) | OAT 缺失 (dex2oat 同步) | cgroup `cpu.weight` 低 |
| 2 冷启动 ANR | 业务死锁/重活 | startProcessLocked 超时 (10s) | ART 启动失败 Runtime::Init | cgroup fork 超时 |
| 3 运行时 ANR | 主线程 IO / 锁 | Binder 调用阻塞 5s | FinalizerWatchdog 10s 超时 | cpuset 限制主线程在大核外 |
| 4 GC 频繁 | 业务对象分配过快 | (无直接关系) | ConcurrentCopying mark 不收敛 | memcg `memory.high` 软限触发 direct reclaim |
| 5 OOM 误杀 | (无直接关系) | lmkd 选错进程 (adj 漂移) | Native 分配抖动 (Bitmap) | cgroup `memory.max` 触发 / cgroup 节点残留 |
| 6 进程泄漏 | 业务没注销 / Service 持有 | (无直接关系) | (无直接关系) | 父进程没 wait4() → zombie |
| 7 进程卡但不死 | (无直接关系) | (无直接关系) | ART SignalCatcher 卡死 | CFS 调度延迟 / cgroup 失配 |
| 8 fd / 线程泄漏 | 业务没 close / 线程池泄漏 | (无直接关系) | (无直接关系) | `task_struct` / `files_struct` 不释放 |
| 9 fork 失败 | (无直接关系) | ProcessList startProcessLocked 失败 | (无直接关系) | cgroup 满 / `pid_max` 满 |
| 10 进程间死锁 | 业务循环等 Binder | Binder 死锁 (同线程池) | (无直接关系) | (无直接关系,通常 FWK 层) |

**关键观察**:
- **故障 1 (冷启动慢) 跨 4 层** —— 这是为什么"冷启动" 是稳定性领域最难解决的问题
- **故障 5 (OOM 误杀)** 主要在 Kernel + FWK + ART 三层
- **故障 6/8 (资源泄漏)** 主要在 App + Kernel

## 5. 监控指标 + 数据采集

> **本节是"治理的前提"** —— 任何故障排查都需要 1) 实时数据 2) 历史数据 3) 趋势分析。

### 5.1 App 层 监控

| 指标 | 数据源 | Android 命令 | 关键阈值 |
|------|--------|-------------|---------|
| 冷启动时间 | `ActivityTaskManager` | `am start -W` | > 1s 警告,> 2s 严重 |
| 首帧时间 | `dumpsys gfxinfo` | `dumpsys gfxinfo <pkg>` | > 500ms 警告 |
| ANR 次数 | `logcat` + `dropbox` | `dumpsys dropbox --print` | 任何 ANR 严重 |
| OOM 次数 | `logcat` `OutOfMemoryError` | `dumpsys meminfo` | 任何 OOM 警告 |
| Java 堆 PSS | `dumpsys meminfo` | `dumpsys meminfo <pkg>` | > 200MB 警告 |
| 线程数 | `/proc/<pid>/status` | `cat /proc/<pid>/status` | > 100 警告,> 300 严重 |
| fd 数 | `/proc/<pid>/fd` | `ls /proc/<pid>/fd \| wc -l` | > 500 警告 |

### 5.2 FWK 层 监控

| 指标 | 数据源 | Android 命令 | 关键阈值 |
|------|--------|-------------|---------|
| AMS 决策时间 | `ActivityTaskManager` log | `dumpsys activity` | > 200ms 警告 |
| lmkd 杀进程次数 | lmkd logcat | `dumpsys activity lmk` | 任何杀进程都该审计 |
| mLruProcesses 大小 | `dumpsys activity processes` | 同左 | > 200 警告 |
| oom_score_adj 设置 | `/proc/<pid>/oom_score_adj` | `dumpsys activity oom` | 异常值审计 |
| system_server 启动时间 | `dumpsys activity` System Ready | `am start` logcat | > 10s 严重 |

### 5.3 ART 层 监控

| 指标 | 数据源 | Android 命令 | 关键阈值 |
|------|--------|-------------|---------|
| GC 频率 | `dumpsys meminfo` `Last GC` | `dumpsys gfxinfo` Janky | > 1次/s 警告 |

## 6. 治理清单(7 类治理动作)

> **本节按"治理目标"分类** —— 实际做稳定性工程时,先选"我要做哪类治理",再对应到本节的具体动作。

### 6.1 治理 1:预防 ANR

- [ ] **业务层**:主线程禁 IO / 锁 / sleep
- [ ] **业务层**:所有 Binder 调用设 timeout
- [ ] **业务层**:Application.onCreate 禁 IO,推到子线程
- [ ] **FWK 层**:AMS startProcessLocked 超时检测
- [ ] **Kernel 层**:主线程 cpuset.cpus 锁大核
- [ ] **Kernel 层**:主线程 uclamp.min ≥ 50%

### 6.2 治理 2:预防 OOM 误杀

- [ ] **App 层**:Bitmap 用 `inSampleSize` + `BitmapPool`
- [ ] **App 层**:Native 分配后及时 release
- [ ] **ART 层**:`dalvik.vm.heapgrowthlimit` 按需调
- [ ] **Kernel 层**:`memory.high` 软限设合理值
- [ ] **Kernel 层**:`memory.max` 硬限不要设,或远大于 peak
- [ ] **Kernel 层**:监控 `memory.peak` 趋势,peak > 80% max 警告

### 6.3 治理 3:预防进程泄漏

- [ ] **App 层**:Service 用 `JobService` / `WorkManager`,不用常驻 Service
- [ ] **App 层**:Activity 用 `lifecycleScope` 替代手动管理
- [ ] **FWK 层**:ams `killPackageProcessesLSP` 时机
- [ ] **Kernel 层**:父进程(zygote)wait4() 正常

### 6.4 治理 4:预防调度延迟

- [ ] **Kernel 层**:`top-app` cgroup `cpu.weight = 100`(默认)
- [ ] **Kernel 层**:`top-app` cgroup `uclamp.min ≥ 50%`
- [ ] **Kernel 层**:`top-app` cgroup `cpuset.cpus = 0-7`(含大核)
- [ ] **Kernel 层**:Game Mode 启用,`cpuset.cpus = 6-7` + `uclamp.min=80%`
- [ ] **业务层**:长任务拆小,不要让主线程占满 100% CPU

### 6.5 治理 5:预防冷启动慢

- [ ] **业务层**:不打 dexopt 主动重 AOT
- [ ] **业务层**:不打 `cmd package compile -m speed -f` 频繁重 AOT
- [ ] **业务层**:发版前打 baseline profile
- [ ] **FWK 层**:`pm install` 时机正确
- [ ] **ART 层**:dex2oat 任务放在 zygote 预编译阶段
- [ ] **Kernel 层**:冷启动进程的 `cpu.weight` + `cpuset.cpus` 不被压缩

### 6.6 治理 6:预防死锁 / 调度卡

- [ ] **业务层**:避免循环 wait(经典:两个线程各持一把锁等对方)
- [ ] **FWK 层**:Binder 线程池大小按需设,不要过小
- [ ] **FWK 层**:监控 `dumpsys binder` 状态
- [ ] **ART 层**:Finalizer 业务禁 IO,FinalizerWatchdog 才能监控
- [ ] **Kernel 层**:SignalCatcher 线程优先级 ≥ 主线程

### 6.7 治理 7:治理流程性动作

- [ ] **监控**:配置 5.1-5.4 全部指标,趋势图存 30 天
- [ ] **告警**:memory.events.high > 0 / cgroup memory.peak > 80% / wait_sum > 50ms 立即告警
- [ ] **文档**:10 大故障 × 4 层根因矩阵 打印贴工位
- [ ] **演练**:每季度做 1 次 ANR / OOM 模拟,验证工具链
- [ ] **升级**:AOSP 14+ 升级,享受 pidfd + UClamp + cgroup v2 默认

## 7. 实战案例:跨层故障完整排查链

> **本节给 2 个"真实风格的" 跨层完整排查链**——把 §3 矩阵 + §4 速查表串成"工程师具体怎么干活"。

### 7.1 案例 1:冷启动 5s+ ANR 的 4 层溯源

> **典型模式**:用户报"app 冷启动要 5 秒以上才出首屏,经常 ANR" —— 这是"冷启动相关" 故障,跨 4 层。

**Step 1:App 层症状**

```bash
$ adb shell am start -W com.example/.MainActivity
WaitTime: 5800       # <-- 5.8s
ThisTime: 4500       # <-- 4.5s 启动耗时
TotalTime: 5300      # <-- 5.3s 总耗时
```

**Step 2:FWK 层排查(AMS 决策)**

```bash
$ adb logcat -d -s ActivityTaskManager:I
START u0 {act=android.intent.action.MAIN ...} from uid 10086
Displayed com.example/.MainActivity for user 0: +4s500ms
```

→ `Displayed` 显示 4.5s,**AMS 决策本身** 只占约 200ms (TotalTime - ThisTime 早期值),**不是 AMS 慢**。

**Step 3:ART 层排查(冷启动期 GC / dexopt)**

```bash
$ adb logcat -d -s art:I PackageManager:I
art: Starting ART
art: After preload
art: Background concurrent copying GC freed 12MB
PackageManager: Running dex2oat on /data/app/com.example-XXX.apk   # <-- 关键!
```

→ **`dex2oat` 在冷启动期同步跑** —— 这是 [05 篇 §3.2 + §6.1](05-ART进程内世界:JIT-AOT与GC.md) 描述的"OAT 缺失" 现象。**根因**:`/data/dalvik-cache/<arch>/system@app@com.example` 在某次清理后没了,**首次冷启动要重新 AOT**。

**Step 4:Kernel 层排查(冷启动期调度)**

```bash
$ adb shell cat /sys/fs/cgroup/top-app/com.example-XXX/cpu.weight
100     # <-- 正常
$ adb shell cat /sys/fs/cgroup/top-app/com.example-XXX/cpuset.cpus
0-7     # <-- 正常
$ adb shell cat /proc/<pid>/sched | grep wait_sum
wait_sum (01.000s) 250000.0    # <-- 250ms 等待
```

→ Kernel 侧 cgroup 配置正常,wait_sum 在 ART 启动期也正常 —— **问题不在 Kernel**。

**跨层根因**:
- **App 层**:无业务问题
- **FWK 层**:无 AMS 决策问题
- **ART 层**:**OAT 缺失,dex2oat 同步执行占 4.2s**(占总时间 70%+)
- **Kernel 层**:无问题

**修复方案**:
- [ ] **业务层**:发版前打 baseline profile
- [ ] **业务层**:`adb shell cmd package compile -m speed -f <pkg>` 强制重 AOT
- [ ] **OEM 层**:确保 `/data/dalvik-cache` 不被无故清理
- [ ] **监控**:`dumpsys package <pkg>` 看 `codePath` 是否有 `odex` 文件

### 7.2 案例 2:app 偶发 OOM 误杀的 4 层溯源

> **典型模式**:用户报"app 用着用着突然消失" —— 这是"OOM 误杀" 故障,跨 3 层。

**Step 1:App 层症状**

```bash
$ adb logcat -d | grep -E "killed|lowmemory"
lowmemorykiller: Killing 'com.example' (20461) adj 900 ...
```

**Step 2:FWK 层排查(lmkd 选进程)**

```bash
$ adb shell dumpsys activity oom | grep -A 2 com.example
ProcessRecord{... com.example pid=20461}
  oom_score_adj=900  # <-- 后台
  mLastPss=200MB
```

→ lmkd 选 900 正确 —— **不是 lmkd 选错**。

**Step 3:ART 层排查(内存波动)**

```bash
$ adb shell dumpsys meminfo com.example
App Summary
  Pss Total:    220000 KB
  Java Heap:    150000 KB
  Native Heap:  50000 KB    # <-- 50MB
```

→ Native heap 50MB 不大,**实时 RSS 也不大** —— 但 lmkd 还是杀了。

**Step 4:Kernel 层排查(cgroup 历史峰值)**

```bash
$ adb shell cat /sys/fs/cgroup/apps.slice/com.example-XXX/memory.peak
838860800    # <-- 800MB!  # <-- 关键!
$ adb shell cat /sys/fs/cgroup/apps.slice/com.example-XXX/memory.events
high 5
max 1
oom 1
```

→ **`memory.peak=800MB`,但实时 `memory.current=220MB`** —— cgroup 在历史某刻用了 800MB! `memory.events max=1` 表示触发过 1 次 OOM kill。**根因**:`memory.peak` 显示 cgroup 历史峰值,实时内存已释放但"被 lmkd 记了一笔"。

**跨层根因**:
- **App 层**:**Native 内存分配有抖动**(可能 Bitmap 大图解码)
- **FWK 层**:lmkd 选正确
- **ART 层**:Native 分配触顶 cgroup memory.max
- **Kernel 层**:`memory.peak=800MB` 触发 OOM kill

**修复方案**:
- [ ] **业务层**:Bitmap 用 `inSampleSize` + `BitmapPool`
- [ ] **业务层**:Native 分配后及时 release
- [ ] **Kernel 层**:监控 `memory.peak` 趋势,peak > 80% max 告警
- [ ] **架构层**:拆"图片处理" 到独立子进程,避免污染主进程 cgroup

## 8. 总结:架构师视角的 5 条 Takeaway

> **本篇(也是本系列) 的最终 5 条 Takeaway** —— 把 01-08 全部浓缩成 5 句话。

### Takeaway 1:**进程类问题 = 跨层问题 —— 80% 必须从 4 层联调**

- **App 层** 看业务代码、对象分配、线程管理
- **FWK 层** 看 AMS / lmkd / 杀进程策略
- **ART 层** 看 GC / JIT / OAT / SignalCatcher
- **Kernel 层** 看 cgroup / 调度 / 内存

**任何只查 1 层的"解决方案" 都是错的。**

### Takeaway 2:**"10 大故障" 是工程师的"启动工具集"**

```bash
# 看到 1 个症状 → 反查 §2 表格 → 锁定 T 编号 → 走 §4 速查路径 → 走 §6 治理清单
```

**这是本系列 8 篇 50+ 万字符的"压成 1 张表"** —— 直接打印出来贴工位。

### Takeaway 3:**监控比修复更重要 —— 5.1-5.4 全部指标是"稳定性的护城河"**

- **App 层**:冷启动时间、首帧、ANR 次数、堆 PSS、线程数、fd 数
- **FWK 层**:AMS 决策时间、lmkd 杀进程次数、mLruProcesses 大小
- **ART 层**:GC 频率、暂停时长、JIT 池 CPU、OAT 加载耗时
- **Kernel 层**:`memory.current / peak / events`、`cpu.weight`、`cpuset.cpus`、`/proc/<pid>/sched`

**没有这 24+ 指标的实时监控,90% 的故障会"事后才被发现"**。

### Takeaway 4:**Android 14 演进:pidfd / UClamp / cgroup v2 + baseline profile**

- **`pidfd_send_signal` 取代 `kill()`** —— 避免 PID 复用误杀
- **UClamp 取代 `schedtune`** —— utilization 夹紧更精确
- **cgroup v2 默认** —— `memory.peak` / `cpu.weight` 可直接读
- **baseline profile 强制 AOT 编译热点** —— 冷启动时无需 JIT
- **AOSP 14 升级一次性解决 5+ 稳定性问题** —— 性价比极高

### Takeaway 5:**本系列 8 篇 = 8 层 50+ 万字符 = 1 张工程师工具书**

```
01 全局观 + 12 个时间点
   ↓
02/03/04 上三层:AMS 决策 / Zygote / 进程首生
   ↓
05/06/07 进程内:ART / Kernel / 调度资源
   ↓
08 收口:10 大故障 + 4 层矩阵 + 监控 + 治理
```

**这是"从点击图标到首帧到 ANR 到 OOM 到治理" 的完整链路** —— 8 篇合一,资深架构师能看懂整个 Android 进程管理栈。
