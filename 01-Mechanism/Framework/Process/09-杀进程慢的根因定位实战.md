## 0. 写在最前:一个真实黑屏 12.2s 案例

> **本节先把"为什么会有这篇文章"钉死。** 后面的 7 阶段拆解,所有"慢在哪"的论断都可以回看本节的时间轴。

### 0.1 现象与一句话根因

**现象(用户侧)**:在某 transsion Android 16 设备上,`com.sh.smart.caller` 在 Clear All 后立刻点图标,**黑屏约 10s**,直到用户第三次点击才看到首帧(`launch_time` ≈ 3808ms)。

**一句话根因**:

> **旧进程 `pid=15770` 在收到 SIGKILL 后,内核侧资源回收在"低 MemFree + 78MB swap + 全机批量杀进程"三重压力下拖到 12.2s 才完成,导致 AMS 启动新进程时被 `refused to die` 判定主动 cancel,出现黑屏。**

**3 个关键观察**(可作为所有同类故障的"判别 checklist"):

1. **不是"没发 kill"**——`am_kill [0,15770,com.sh.smart.caller,754,remove task,151024]` 在 `19.658` 已发
2. **不是"没响应点击"**——Splash 在 `20.150` 已上屏(用户看到的是"假启动" → 黑屏)
3. **是"AMS 主动取消启动"**——`29.982 am_wtf: 15770 refused to die ... cancelling the process start` → 启动被取消

```
         100ms  ──── 理想情况(空进程,MemFree 充足)───→ ~12s ──── 本案(78MB swap + MemFree 80MB)───→ 12.2s
```

### 1.2 各阶段"快"与"慢"基线

| # | 阶段 | 理想耗时 | 本案耗时 | 慢在哪 | 占比(本案)|
|---|------|---------|---------|--------|----------|
| §2 | 触发源 | < 1ms | < 1ms | - | 0% |
| §3 | AMS 决策 | ~5ms | ~5ms | - | 0% |
| §4 | 信号发送 | ~5ms | ~5ms | - | 0% |
| §5 | ART 退出 | 0ms(SIGKILL 不走 OnExit) | 0ms | - | 0% |
| §6 | **do_exit** | **~50ms** | **~10s** | **exit_mm 回收 swap** | **82%** |
| §7 | **reaper/cgroup** | **~30ms** | **~2s** | cgroup "Device or resource busy" × 2 | **16%** |
| §8 | FWK 收尾 | ~10ms | 0.2s | ALL_TIMEOUT 兜底 | 2% |
| | **合计** | **~100ms** | **~12.2s** | | **100%** |

**核心结论**:**杀进程慢的 98% 在 Kernel**(§6 + §7)。FWK 阶段本身是 ms 级。

### 1.3 与"启动链路"的对称

| 启动链路(Process/01 T0-T8) | 杀进程链路(本篇 §2-§8)|
|----------------------|-----------------|
| T0 桌面点击 | §2 触发源 |
| T1 Launcher → ATMS | §2 用户主动 |
| T2 AMS 决策(冷启动) | §3 AMS 决策(杀进程)|
| T3-T5 Zygote fork | §4 信号发送(pidfd_send_signal)|
| T6 子进程首生 | §5 ART 退出(对偶)|
| T7-T8 启动期 lifecycle | §6 do_exit(资源释放) |
| T9 驻留期 | §7 reaper/cgroup(收尸)|
| T10-T12 驻留→死亡 | §8 FWK 收尾(handleAppDied)|

**对称性**:**启动 = 资源申请**,**杀进程 = 资源释放**。启动链路越长说明越慢,杀进程链路越长同样说明越慢。

## 2. 阶段 1:触发源(谁在杀这个进程)

> **本节核心问题**:**谁**决定这个进程要死?
>
> **常见误区**:把"杀进程"等同于"AMS 调用 killProcess"。但 AMS 只是"执行者",真正"决定者"有 4 类:
> 1. 用户主动(force-stop / Clear All)
> 2. AMS 决策(lowmem / remove-task / 升级安装)
> 3. Watchdog(SWT)
> 4. 进程自退(`System.exit` / `Process.killProcess` / `Looper.quit`)

### 2.2 触发源 ①:用户主动 force-stop

**核心调用栈**:

```

  → ActivityManager.forceStopPackage(pkg, userId)
  → frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java
```

**AOSP 16 关键源码路径**:

- `frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java` — `forceStopPackage`
- `frameworks/base/services/core/java/com/android/server/am/ProcessList.java` — `killPackageProcessesLocked`

**AMS 入口代码(精简)**:

```java
// ActivityManagerService.java
public void forceStopPackage(String packageName, int userId) {
    // ... 权限校验 ...
    final int appId = UserHandle.getAppId(uid);

    synchronized(this) {
        // 1. 杀所有该 package 名的进程
        int[] users = new int[] { userId };
        mProcessList.killPackageProcessesLocked(
            packageName, appId, users, /* allowRestart= */ false,
            /* isolatedPid= */ -1, /* reason= */ "force stop",
            /* mustRunInProcessRec= */ null);
        // 2. 杀该 package 所有的 service
        mServices.bringDownDisabledPackageServicesLocked(...);
        // 3. 杀所有的 broadcast receiver
        mReceiverList.killPackageReceiversLocked(...);
        // 4. 清掉所有的 PendingIntent
        mPendingIntentController.removePendingIntentsForPackage(...);
    }
}
```

**与 remove-task 的差异**:

| 维度 | force-stop | remove-task |
|------|------------|-------------|
| 是否发 broadcast | 发 `PACKAGE_RESTARTED` | 不发 |
| 是否清 PendingIntent | 是 | 否 |
| 是否清 alarm/job | 是 | 否 |
| 是否影响其他 user 的进程 | 是(uid 维度) | 仅本 task |
| 用户预期 | "卸载级别清理" | "单纯退出当前任务" |

**架构师观察**:force-stop 是**最重的杀进程路径**,它会做 4 件事,影响远大于 remove-task。

### 2.3 触发源 ②:AMS 决策(lowmem / remove-task / 升级安装)

#### 2.3.1 lowmemorykiller(lmkd)

**入口**:`/system/bin/lmkd` 守护进程,**不是** AMS。

```
lowmemorykiller(lmkd)
  ↓ poll /proc/pressure/memory 或 /proc/meminfo
  ↓ 计算 oom_score_adj 选择目标进程
  ↓ PidfdProcess.killProcess(pid)  ← 直接绕过 AMS
  ↓
  SIGKILL → do_exit → ...
```

**lmkd 与 AMS 的关系**:

- lmkd **不**经过 AMS
- lmkd 用 `PidfdProcess.killProcess` 直接发 SIGKILL(详见 §4)
- lmkd 选进程通过 `oom_score_adj`(adj 越高越优先被杀)
- AMS **不知道** lmkd 杀了谁(只能从 `am_proc_died` 被动感知)

**源码路径**:
- `system/memory/lmkd/lmkd.cpp` — 主循环
- `frameworks/native/libs/binder/PidfdProcess.cpp` — 杀进程接口
- 详见 [Process/06 §5.3](../Process/06-Framework视角的Kernel进程接口_procfs_cgroup_pidfd.md)

**为什么 Recents 动画要"错开"**?——为了视觉流畅。但代价是**杀进程被串行化**:
- 20 个 task × 每 100ms 错开 = **2s 才开始杀第 20 个 task**
- 这段时间内,第 1 个 task 可能已经 do_exit 走完 → reaper/cgroup 释放中 → cgroup_mutex 等待
- 多个 task 并发做 do_exit,加剧系统内存压力

**架构师观察**:

1. **"Clear All 杀进程慢"= 多 task 串行 + 每个 task 的 do_exit 慢 × N**
   - 假设桌面 20 个 app,Clear All 后 → 20 次 do_exit(部分并发) → 系统级锁竞争 → 总耗时可能 20-30s
2. **如果单 app 的 do_exit 已经 12s(本案)**,Clear All 触发 → **12s + 动画错开** = 用户感知 15-20s 黑屏
3. **本案 com.sh.smart.caller 只是一个 task**,所以单 task 也吃了 12s;如果同时清 5 个电话 app,总耗时可能 60s+

**实战建议(架构师)**:

- **不要在低内存设备上 Clear All**——在 MemFree < 200MB 时,Clear All 一次可能导致整机 30s+ 假死
- **LMKD + Clear All 是高风险组合**——LMKD 已经在批量杀,Clear All 又叠加 → 雪崩
- **监控建议**:
  ```bash
  # 抓 Clear All 的总耗时
  adb logcat -b events | grep -E "am_kill|am_proc_died" | \
    awk -F'[][]' '/am_kill/ {start=$5} /am_proc_died/ {end=$5; if(start) print end-start, "kill耗时"; start=0}'
  # 多个 task 全部 am_proc_died 完毕的总时长
  ```
- **治理**:OEM 可以把 Recents 动画错开间隔从 100ms 调到 50ms(代价:视觉不流畅);或并行发 kill(代价:SNR 风险)

**关键源码路径**:
- `frameworks/base/services/core/java/com/android/server/wm/ActivityTaskManagerService.java` — `removeTask`
- `frameworks/base/services/core/java/com/android/server/wm/Task.java` — `removeTaskIfNeeded`

### 2.4 触发源 ③:Watchdog SWT

**入口**:`Watchdog` 类(`frameworks/base/services/core/java/com/android/server/Watchdog.java`)。

**触发条件**:`system_server` 主线程 30s+ 无响应。

**被杀进程**:**只杀 system_server 自己**(不杀 app)。

**架构师观察**:
- Watchdog 杀 system_server → system_server 死 → 触发 zygote 重启 system_server
- 这个路径下"杀进程慢"不是 system_server 本身,而是**系统重启耗时**
- 与本篇杀 app 进程的链路**完全不同**,本文不展开

### 2.5 触发源 ④:进程自退

**4 种自退路径**:

```
路径 1:Java 主动退出
  System.exit(0)  → Runtime.exit
  → ZygoteInit.zygoteExit
  → libcore.os.exit(code)
  → 进程结束

路径 2:Native 主动退出
  exit(2) / _exit(2)
  → sys_exit_group
  → do_group_exit
  → do_exit

路径 3:进程自杀
  Process.killProcess(myOwnPid)
  → 走 FWK killProcess 路径(同阶段 4)
  → 注意:自己的进程杀自己,SIGKILL 后无法做收尾

路径 4:Crash 自杀
  uncaughtExceptionHandler 触发
  → Process.killProcess(myOwnPid)
  → 注意:是 SIGKILL 自己,不做清理
```

**关键观察**:
- `System.exit(0)` 走 `ZygoteInit.zygoteExit` → 主动调用 `Runtime::OnExit`(走 ART §5)
- `Process.killProcess(myOwnPid)` 走 SIGKILL 路径(不走 ART 主动清理)
- 业务方常见 bug:用 `Process.killProcess` 杀自己,期望走清理逻辑——**不会**。SIGKILL 不走任何清理。

### 2.7 阶段 1 慢在哪

> **架构师结论**:**阶段 1 几乎不慢**。触发源确定后,决策是同步的、毫秒级的。
>
> **真正可能慢的子路径**:

| 场景 | 慢在哪 | 占比 |
|------|-------|------|
| lmkd 杀进程 | 选进程算法(遍历进程列表算 oom_score) | < 5% |
| removeTask 多 task | 串行遍历 task,每个 task 发 kill | < 5% |
| 升级安装杀进程 | 等 broadcast 处理完成 | < 10% |
| force-stop | 同步发 PACKAGE_RESTARTED + 清 PendingIntent | < 10% |
| **本案(remove-task)** | - | **0%** |

**本案阶段 1 耗时**:**~140ms**(从 19.517 触发到 19.658 发出 am_kill)——但这 140ms 是 **WindowManager 销毁 Activity + Usf 调整 + AMS 决策**的合计,**实际触发源确定 < 1ms**。

## 3. 阶段 2:AMS 决策(杀谁、怎么杀、等多久)

> **本节核心问题**:**AMS 拿到"要杀 pid=15770"之后,做了什么决策?**
>
> **决策三件事**:
> 1. 选 pid(谁被杀——多进程 app 要全杀)
> 2. 选 killer(用什么 API 杀——PidfdProcess vs Process)
> 3. 等多久(`wait_for_death` + `ALL_TIMEOUT`)

### 3.1 入口:`ProcessList.killPackageProcessesLocked`

**AOSP 16 签名**:

```java
// frameworks/base/services/core/java/com/android/server/am/ProcessList.java

        int[] userIds, boolean allowRestart, int isolatedPid,

            }
        }
    }
    // 2. 杀进程(关键调用)

```

**关键源码路径**:
- `frameworks/base/services/core/java/com/android/server/am/ProcessList.java#killPackageProcessesLocked`
- `frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java#killProcessesForRemovedTask`

### 3.2 killer 选择:PidfdProcess vs Process

**AMS 收到"要杀 pid=15770"后**:

```java
// ActivityManagerService.java
private void killProcessesForRemovedTaskLocked(ProcessRecord app, String reason) {
    // 1. 校验:进程存在且未被杀
    if (app.thread == null) return;  // 已死

    // 2. ★ 选 killer
    if (app.pid > 0 && app.pid != MY_PID) {
        // 路径 A:PidfdProcess(Android 13+ 默认)
        Process.killProcess(app.pid);
    }

    // 3. 标记 ProcessRecord
    app.removed = true;
}
```

**killer 的两条路径**(详见 §4 阶段 4):

| 路径 | API | 适用版本 | 风险 |
|------|-----|---------|------|
| A | `PidfdProcess.killProcess(pid)` | Android 13+ 默认 | fd 泄露(必须 close) |
| B | `Process.killProcessQuiet(pid)` | Android 12- | PID race(信号发错目标) |

**架构师观察**:本案走的是**路径 A**(Android 16 默认 PidfdProcess),但 vendor hook 可能改回**路径 B**(这是已知 OEM 行为,见 §10.4 案例 4)。

### 3.3 等多久:`wait_for_death` + `ALL_TIMEOUT`

**AMS 不立即把 ProcessRecord 摘掉,等内核报死亡**。等待机制有 2 个:

#### 3.3.1 `wait_for_death`:binder death 通知

```java
// ProcessRecord.java
// 在 attachApplication 时设置:
app.thread.linkToDeath(deathRecipient, 0);

// deathRecipient.onBinderDied():
//   1. 进入 handleAppDiedLocked(...)
//   2. 清理 ProcessRecord
//   3. 通知 observers
```

**原理**:进程死时,binder 驱动会通过 `linkToDeath` 通知到 system_server。

**问题**:**binder death 通知依赖进程死前能发出通知**。如果进程**已经卡在不可中断状态**(D state),binder 通知可能延迟。

#### 3.3.2 `ALL_TIMEOUT cleanup`:超时兜底

**核心常量**:

```java
// ProcessList.java
static final int PROC_START_TIMEOUT = 10 * 1000;        // 10s(冷启动)
static final int WAIT_FOR_DEATH_TIMEOUT = 5 * 1000;     // 5s(等死亡)
static final long ALL_TIMEOUT = 5000;                   // 5s(本节重点)
```

**超时逻辑**(节选):

```java
// ProcessList.java
// 启动时注册的 watchdog:
private final Runnable mAllProcessesDied = new Runnable() {
    @Override
    public void run() {
        // ★ 每 5s 跑一次,清理所有"marked died but still alive"的 ProcessRecord
        synchronized (ActivityManagerService.this) {
            for (int i = 0; i < lruProcesses.size(); i++) {
                ProcessRecord app = lruProcesses.get(i);
                if (app.removed && app.thread != null) {
                    // ★ ALL_TIMEOUT 兜底
                    Slog.wtf(TAG, "remove ALL_TIMEOUT cleanup " + app);
                    cleanUpApplicationRecordLocked(app, ...);
                }
            }
        }
    }
};
```

**关键点**:**`ALL_TIMEOUT` 不是"杀后等 5s 强制清",是"每 5s 跑一次检查"**——所以本案从 19.658 到 31.892 实际是**等了多个 5s 周期**,而不是"5s 超时"。

### 3.4 ProcessRecord 状态机

```
                  ┌─────────────────┐
                  │  ATTACHED       │ ← process 已 attach,binder thread 在
                  │  (thread != null│
                  │   removed=false)│

                           │ AMS.killProcess(pid)
                           │ app.removed = true
                           ▼
                  ┌─────────────────┐
                  │  KILLED         │ ← 等待 binder death 通知
                  │  (thread != null│
                  │   removed=true) │ ★ 本案长期停留在此
                  └────────┬────────┘
                           │ Process.killProcess(pid) 发出 SIGKILL
                           │ Kernel do_exit 完成
                           │ binder 驱动发 BINDER_THREAD_EXIT
                           │ linkToDeath 回调
                           ▼
                  ┌─────────────────┐
                  │  DEAD           │ ← onProcessDied
                  │  (thread == null│

                  ┌─────────────────┐

                  │  (不在 lruProc) │
                  └─────────────────┘

```

**本案轨迹**:

```
T+0     19.658  AMS:app.removed=true  ──→ 进入 KILLED 态

T+10s   20.xx   再次启动,app.removed=true → attached 检查失败
T+12s   31.892  ALL_TIMEOUT cleanup → REMOVED
```

**关键观察**:**`thread == null` 是"真死"的标志**。本案中 thread 字段**12s 一直 != null**,说明 binder death 通知**从未收到**——内核侧 `do_exit` 还没走完。

### 3.6 阶段 2 慢在哪

> **架构师结论**:**阶段 2(AMS 决策)本案例也不慢,5ms 内完成**。
>
> 但阶段 2 **为后续阶段埋雷**:
>
> 1. **不立即摘 ProcessRecord**——等 binder death 通知,本案等了 12s
> 2. **不预检 `app.thread == null`**——启动逻辑被旧的 attached 状态污染
> 3. **ALL_TIMEOUT 是 5s,但本案 12s 才触发**——5s 不是硬超时,是"每 5s 检查一次"——首次 5s 跑时 app.thread 仍 != null

**本案阶段 2 耗时**:**295ms**(从 19.658 到 19.953)+ 12s(到 ALL_TIMEOUT) = **~12.2s 总**,但**其中 99% 是阶段 6+7 决定的**。

## 4. 阶段 3:信号发送(FWK → Kernel)

> **本节核心问题**:**AMS 已经决定杀 pid=15770**,那 FWK 怎么把"杀"的信号送到 Kernel?
>
> **三种路径**:
> 1. **新路径**:`PidfdProcess.killProcess(pid)` → `pidfd_send_signal` 系统调用
> 2. **旧路径**:`Process.killProcessQuiet(pid)` → `kill(pid, SIGKILL)` 系统调用
> 3. **最旧路径**:`Runtime.exec("kill -9 <pid>")` → shell 工具

### 4.1 三条路径:pidfd(新)/ PID signal(旧)/ kill -9(最旧)

| 维度 | 新路径(pidfd)| 旧路径(PID signal)| 最旧路径(kill -9)|
|------|------------|-------------------|------------------|
| **API** | `pidfd_open` + `pidfd_send_signal` | `kill(pid, SIGKILL)` | `Runtime.exec("kill")` |
| **系统调用** | `SYS_pidfd_send_signal` | `SYS_kill` | shell fork+exec |
| **PID race** | ✅ 无(pidfd 绑定 task) | ❌ 有(PID 复用) | ❌ 有 |
| **fd 泄露风险** | ⚠️ 必须 close(pidfd) | ✅ 无 fd | ✅ 无 fd |
| **性能** | 微秒级 | 微秒级 | 毫秒级(fork) |
| **Android 16 默认** | ✅ 是 | fallback | 已废弃 |
| **OEM 改回风险** | 中(hook 改回 PID) | 高(vendor 改回旧)| - |

**架构师观察**:**Android 16 默认走新路径(pidfd)**,但 OEM 经常改回旧路径(详见 §10.4)。

### 4.2 `PidfdProcess.killProcess` 源码

**核心实现**:

```cpp
// frameworks/native/libs/binder/PidfdProcess.cpp
int PidfdProcess::killProcess(pid_t pid) {
    // 1. 打开 pidfd
    int pidfd = pidfd_open(pid, 0);
    if (pidfd < 0) {
        // fallback 到旧路径
        return kill(pid, SIGKILL);
    }
    // 2. 通过 pidfd 发 SIGKILL
    int ret = pidfd_send_signal(pidfd, SIGKILL, nullptr, 0);
    // 3. ★ 必须 close,否则 fd 泄露
    close(pidfd);
    return ret;
}
```

**关键源码路径**:
- `frameworks/native/libs/binder/PidfdProcess.cpp#killProcess`
- `frameworks/native/libs/binder/PidfdCache.cpp` — 缓存层(避免反复 open)

**对应 Java 入口**:
- `frameworks/base/core/java/android/os/Process.java#killProcess` — 静态封装

**详细单接口说明**详见 [Process/06 §5](../Process/06-Framework视角的Kernel进程接口_procfs_cgroup_pidfd.md)(本篇不重复)。

### 4.3 `pidfd_send_signal` syscall 链路

**kernel 内部流转**(参考 [`../../Kernel/Process/11-信号机制_从产生到投递.md`](../../Kernel/Process/11-信号机制_从产生到投递.md)):

```
userspace:PidfdProcess::killProcess(pid)
  ↓
syscall SYS_pidfd_send_signal(pidfd, SIGKILL, info=NULL, flags=0)
  ↓
kernel:__sys_pidfd_send_signal()
  ↓
  pidfd_to_pid(pidfd)            ← ★ 从 fd 反查 task_struct
  ↓
  kill_pid_info(&pid, SIGKILL)   ← 复用 kill() 的同一路径
  ↓
  __send_signal_locked()          ← 投递信号到 task->pending
  ↓
  wake_up_state(t, TASK_WAKEKILL)  ← 唤醒目标进程(如果它在睡眠)
  ↓
  target task 调度回来,执行 SIGKILL handler
  ↓ (handler 默认是 SIG_DFL → 直接终止)
  do_exit(SIGKILL)
```

**关键源码路径**:
- `kernel/pid.c#SYSCALL_DEFINE3(pidfd_send_signal, ...)`
- `kernel/pid.c#pidfd_to_pid`
- `kernel/signal.c#__send_signal_locked`

**架构师观察**:**`pidfd_send_signal` 走的是与 `kill()` 相同的 `kill_pid_info` 路径**——所以本节慢不慢完全取决于 Kernel 信号投递速度,与 pidfd 本身**无关**。

### 4.4 路径选择的"默认"与"fallback"

**Android 16 默认行为**:

```java
// frameworks/base/core/java/android/os/Process.java
public static final void killProcess(int pid) {
    // ★ 优先级:
    //   1. PidfdProcess.killProcess(pid)  (Android 13+ 默认)
    //   2. Libcore.os.killProcess(pid)    (Android 12-)
    //   3. Runtime.exec("kill -9")         (最旧,vendor hook fallback)
    sendSignalQuiet(pid, Signal.KILL);
}
```

**OEM 改回的常见场景**:

- **MTK 平台**:`vendor/mediatek/proprietary/frameworks/base/.../Process.java` 可能改回 `Runtime.exec`
- **展锐(UNISOC)**:`vendor/sprd/platform/.../Process.java` 类似改法
- **三方 ROM**(LineageOS):通常沿用 AOSP

**如何检测**:
```bash
# 看 PidfdProcess.killProcess 是否被改写
adb shell objdump -d /system/lib64/libandroid_runtime.so | grep -A 30 "pidfd_open" | head -50
```

**本案**:从 logs 看 vendor 路径未改,走的是 Android 16 默认 PidfdProcess。

### 4.5 阶段 3 慢在哪

> **架构师结论**:**阶段 3(信号发送)不慢**——所有路径都是微秒级系统调用。
>
> **但阶段 3 可能被**"**"伪成功"**"**骗到**:
>
> - `pidfd_send_signal` 成功返回 0 ≠ 目标进程已死
> - 目标进程收到信号后才进入 do_exit,do_exit 还要 ~50ms(理想)或 10s(本案)
> - **如果只看 `pidfd_send_signal` 返回值就认为"已杀完"——会误判**
>
> **本案**:`pidfd_send_signal` 19.658 已成功 → 实际 am_proc_died 在 31.898 才到。**12.2s 的差距在阶段 5+6,不在阶段 3**。

## 5. 阶段 4:ART 进程内退出

> **本节核心问题**:**进程收到 SIGKILL 后,ART 运行时内部能做什么?**
>
> **一句话答案**:**几乎不能**。SIGKILL 不可捕获、不可阻塞,ART 不会做任何"主动清理"。

### 5.1 SIGKILL 在 ART 内部的处理路径

**kernel 视角**(见 §4.3 + [`../../Kernel/Process/11-信号机制_从产生到投递.md`](../../Kernel/Process/11-信号机制_从产生到投递.md)):

```
SIGKILL 投递到 task->pending
  ↓
  task 调度回来,从内核返回用户态前
  ↓
  do_signal_exit(task, SIGKILL)  ← 立刻 do_exit,不走用户态
  ↓
  do_exit(SIGKILL)
  ↓
  ... 资源回收(§6)...
```

**ART 视角**:

```
进程收到 SIGKILL
  ↓
  ★ 没有 ART 处理 ★
  ↓
  ART 不会"主动"清理(JIT code cache / OAT 缓存 / GC 状态等)
  ↓
  Kernel 直接 do_exit
  ↓
  ART 的所有数据结构由 Kernel 释放(munmap、close fd 等)
```

**关键源码路径**:
- `art/runtime/signal_catcher.cc#SignalCatcher::Run` — 接收 SIGQUIT/SIGUSR1 等,**不处理 SIGKILL**
- `art/runtime/runtime.cc#Runtime::OnExit` — 只在 `System.exit(0)` 路径调用

### 5.2 为什么 SIGKILL 不走 `Runtime::OnExit`

**`Runtime::OnExit` 路径**(主动退出才走):

```cpp
// art/runtime/runtime.cc
void Runtime::OnExit() {
    // 1. 通知所有线程退出
    {
        Thread* self = Thread::Current();
        MutexLock mu(self, *shutdown_lock_);
        shutdown_ = true;
    }
    // 2. 等待所有 non-daemon 线程退出
    // 3. 关闭 Heap(CC / GC)
    // 4. 关闭 JIT code cache
    // 5. 卸载 OAT
    // 6. 关闭 class linker
    // 7. 关闭 ThreadPool
}
```

**只有以下路径会触发 `OnExit`**:

1. **`System.exit(0)`(Java 主动退出)**
   - `Runtime.exit` → `ZygoteInit.zygoteExit` → `libcore.os.exit` → 进程结束
2. **Native `exit(2) / _exit(2)`**
3. **未捕获异常导致 `KillApplication`**(AOSP 16 仍保留该机制,主要在 native crash 后的 java 侧清理路径使用)
4. **RuntimeInit.commonInit 中的 explicit exit**

**`SIGKILL` 不会触发 `OnExit`**——因为:

- SIGKILL 不可捕获、不可阻塞
- ART 不会进入任何"主动清理"流程
- 进程从内核态直接走 `do_exit`

### 5.3 SignalCatcher 的兜底行为

**SignalCatcher 是 ART 唯一的信号接收线程**:

```cpp
// art/runtime/signal_catcher.cc
void* SignalCatcher::Run(void* arg) {
    // 等待信号
    sigwait(sigset, &sig_info);
    switch (sig_info.si_signo) {
        case SIGQUIT:
            HandleSigQuit(sig_info);
            break;
        case SIGUSR1:
            HandleSigUsr1(sig_info);
            break;
        // ★ 没有 case SIGKILL
        default:
            LOG(WARNING) << "Unexpected signal " << sig_info.si_signo;
    }
}
```

**SignalCatcher 处理的信号**:

| 信号 | 处理 |
|------|------|
| `SIGQUIT`(3) | dump 堆栈 / 触发 ANR dump |
| `SIGUSR1`(10) | 触发 profile saver / GC |
| `SIGUSR2`(12) | 部分 OEM 自定义 |
| `SIGPIPE`(13) | 忽略(写到已关闭 socket) |
| **`SIGKILL`(9)** | **★ 不处理,直接被 do_exit 接管** |
| **`SIGTERM`(15)** | **★ 不处理,直接被 do_exit 接管** |

**架构师观察**:
- SignalCatcher 用 `sigwait` 等待信号——这意味着它**屏蔽了这些信号**
- 即使 SIGKILL 触发了 SignalCatcher 的 sigwait,内核也会在 do_signal_exit 中绕过它直接 do_exit
- 所以 ART 看到 SIGKILL 也**不做事**

### 5.4 ART 退出可能 hook 的位置

**理论上,ART 进程"退出"是没有任何 hook 的**。但有两个**观察点**:

#### 5.4.1 `Java 进程退出回调`(主动路径)

```java
// RuntimeInit.java
public static final void commonInit() {
    // 注册 uncaughtExceptionHandler
    Thread.setDefaultUncaughtExceptionHandler(new UncaughtHandler());
}

private static class UncaughtHandler implements Thread.UncaughtExceptionHandler {
    public void uncaughtException(Thread t, Throwable e) {
        // 1. 同步阻塞,等 dump 完成
        // 2. 调用 System.exit(10)
        // 3. 不走 SIGKILL
    }
}
```

**只在 uncaught exception 时触发**,SIGKILL 不会到这里。

#### 5.4.2 `Thread.setDefaultUncaughtExceptionHandler`(App 侧)

**App 自己的 `Thread.UncaughtExceptionHandler`** 可能在进程退出前做最后清理(如 flush 日志到磁盘)。但 SIGKILL 同样不会触发。

**架构师建议**:
- 业务方在 SIGKILL 路径上**不要期望任何回调**
- 关键数据**必须定期 flush**到磁盘,不能依赖"退出前"
- 任何"退出前发信号" / "退出前 flush" 设计在 SIGKILL 路径上**全部失效**

### 5.5 阶段 4 慢在哪

> **架构师结论**:**阶段 4(ART 退出)** **几乎不耗时**——ART 收到 SIGKILL 直接被 do_exit 接管。
>
> **阶段 4 慢的唯二可能**:
> 1. **主动退出路径**:`Runtime::OnExit` 等所有 non-daemon 线程退出,可能 1-2s(业务线程死循环)
> 2. **Art 自己**:`profile_saver` 还在写 profile → 阻塞 → 但 SIGKILL 不走这个路径
>
> **本案**:SIGKILL 路径,ART 0ms。**不慢**。

### 6.2 `exit_mm`:释放地址空间(本案关键 — swap 78MB + vma 异常)

**`exit_mm` 干 3 件事**:

1. **释放所有 VMA**(virtual memory area)——每个 VMA 对应一个 mmap 区域
2. **释放匿名页 + 文件页**(包括 swap 出去的页)
3. **释放页表**(PGD/PUD/PMD/PTE)

#### 6.2.1 `exit_mm` 内部 4 个 sub-sub-step

```
exit_mm(tsk)
  ↓
  mmput(tsk->mm)
    ↓
    exit_mmap(mm)                            ← ★ 释放所有 VMA + 物理页
      ↓
      unmap_vmas(mm->mmap, ...)              ← sub-step ③-a:遍历 VMA 链表
        ↓
        for each VMA:
          unmap_page_range(VMA, ...)         ← sub-step ③-b:释放 VMA 内的所有页
            ↓
            for each PTE in this VMA:
              zap_pte_range(...)
                ↓
                if pte is swap entry:
                  swap_free(entry)           ← sub-step ③-c:释放 swap slot
                    ↓
                    swap_entry_free(...)
                      ↓
                      add to swap_map[slot]  ← 写 swap 位图
                else if pte is file:
                  release_page(...)
                else if pte is anonymous:
                  put_page(...)
          tlb_finish_mmu(...)                ← sub-step ③-d:刷新 TLB
      ↓
    mmdrop(mm)                               ← 释放 mm_struct
```

**4 个 sub-sub-step × 5 栏对账表**(本案 ~10s 的理论分布):

| sub-sub-step | 源码 | 理想耗时 | 本案耗时(估算) | 日志证据 | 慢在哪(本案) |
|---|---|---|---|---|---|
| **③-a unmap_vmas 遍历** | `mm/mmap.c#unmap_vmas` | ~10ms | ~100ms(估) | 推论: 进程 VMA 数 < 200(健康范围),§9.8 显示 vma 状态异常(部分已被 process_reclaim 预回收) | vma 状态异常,`vm_area_struct.vm_flags` 反复校验 |
| **③-b unmap_page_range** | `mm/memory.c#unmap_page_range` | ~5ms | ~5-8s(估) | 推论: rss=170MB + swap 78MB ≈ 248MB 物理页要 unmap,理想 ~50MB/s,本案 ~30MB/s 慢于预期 | (a) 物理页多(248MB);(b) vma 状态异常导致 page 状态反复校验 |
| **③-c swap_free** | `mm/swapfile.c#swap_entry_free` | ~30ms | ~2-3s(估) | 推论: swap 78MB 释放 8000+ PTE,每个 PTE 都要 `swap_map` 位图操作(全局锁);`/proc/meminfo` 设备 swap 使用率 55% | (a) `swap_map` 全局锁竞争;(b) 高 swap 使用率锁更紧;(c) system-wide 正在并发回收 |
| **③-d tlb_finish_mmu** | `arch/arm64/mm/tlb.c#tlb_finish_mmu` | ~5ms | < 50ms(估) | 无直接证据(只 flush TLB) | 不慢(只是硬件 TLB 刷新) |
| **③ 合计** | — | **~50ms** | **~10s** | — | **swap 78MB + vma 状态异常 + 系统级内存压力** 三重叠加 |

> **⚠️ 重要诚实声明**:
> - **③-a / ③-b / ③-c / ③-d 的"本案耗时"都是估算,不是 kernel log 直接测量**。
> - **③-b 5-8s 的估算逻辑**:248MB 物理页要 unmap,理想 `unmap_page_range` 速率 ~50MB/s(arm64 实测),本案 ~30MB/s 慢于预期,主因是 vma 状态异常导致 `vm_area_struct.vm_flags` 反复校验(每次校验 ~5μs × 数千个 PTE 累计)。
> - **③-c 2-3s 的估算逻辑**:78MB swap = ~8000 PTE,每次 `swap_entry_free` 要 (1)取 `swap_map` 锁 + (2)位图置位 + (3)引用计数减 1,本案设备 swap 55% 使用率,锁等待时间 ~300μs/PTE,8000 × 300μs = 2.4s。
> - **要精确测 ③-b / ③-c 耗时**,需提前 `echo 1 > /sys/kernel/debug/tracing/events/mm/mm_page_free/enable`(详见 §9.6)。

**为什么本案 `exit_mm` 慢?—— 3 重压力叠加**

**压力 1:swap 78MB 释放(主要)**

本案 DFX 显示该进程:
```
rss=170MB  mVmSwap=80240(~78MB)  mIsHiber=false
```

**swap 78MB** 意味着:
- 该进程有 78MB 的匿名页**被换出到 swap**
- 释放这些页时,需要**先把 swap 里的页**"换入或丢弃"
- 如果**当前 MemFree 极低**(< swap 总量),回收只能**直接丢弃**这些页(swap_free),本身需要写 swap 位图 → 慢

**具体路径**(同 6.2.1 ③-c):
```
unmap_page_range → zap_pte_range
  ↓
  if pte is swap entry:
    swap_free(entry)  ← swap slot 释放
      ↓
      swap_entry_free
        ↓
        add to swap_map[slot]  ← 写 swap 位图
```

**压力 2:vma 状态异常(关键伏笔)**

§9.8 揭示:案发前 24min(14:21:16)transsion `process_reclaim` 已把 15770 进程 mmap 资源预回收过一次,kernel log 有 4 次 `binder_alloc: 15770: no vma` 记录。

**含义**:
- 进程到达 do_exit 时,**部分 VMA 已经是"空壳"**(vm_flags 标记但页表项已经被回收过)
- `unmap_vmas` 遍历这些 VMA 时,需要**反复校验** `vm_area_struct.vm_flags` 和 page state
- 每次校验 ~5μs × 数千个 PTE 累计 ~数十 ms 到几秒

**压力 3:系统级内存压力(雪上加霜)**

本案数据印证:

```
07-17 20:45:19.854  rms_NormalReclaimer: package: com.sh.smart.caller, uid: 10131,
                    (request + memFreeThreshold) - memFree: (222754 + 204800) - 86736 = 340818
```

**含义**:
- am_kill 之前 100ms 系统主动回收 340MB
- MemFree 只有 **86.7MB**(请求 222.7MB + 阈值 204.8MB)
- 这意味着**设备整体处于内存压力**下,`swap_map` 锁被并发回收占用,锁等待时间 ×3

**3 重压力叠加的最终耗时推算**:
- ③-b unmap_page_range(物理页多 + vma 异常):5-8s
- ③-c swap_free(全局锁竞争):2-3s
- ③-a / ③-d(其他):< 200ms
- **合计 ~10s**

### 6.3 `exit_files`:关闭 fd

**`exit_files` 关闭进程所有打开的 fd**:

```
exit_files(tsk)
  ↓
  close_files(tsk->files)
    ↓
    for each fd in files_struct:
      filp_close(fd, ...)
        ↓
        fput(file)
          ↓
          if refcount == 0:
            file->f_op->release(inode, file)  ← 文件系统 close
```

**本案风险**:fd 泄露
- 假设进程开了 500 个 fd,关闭每个 fd 需要微秒级,合计 ~5ms(理想)
- 但如果有**慢文件系统**(如 fuse),单个 fd 关闭可能 100ms+
- 如果有**网络 socket** 关闭要做 FIN,可能更长

**本案日志**:
- 没看到"Too many open files"相关日志 → fd 数在合理范围
- 但 GPU memory data 显示 `pid_15770:6787072`(6.7MB GPU)——可能是 SurfaceFlinger 的资源,关闭要 flush GPU

### 6.4 `exit_fs` / `exit_thread` / `exit_namespaces` / `exit_task_stack`

**这 4 个 sub-step 通常是 ms 级**——只释放本进程元数据,不做实际 IO。

```
exit_fs:        释放 fs_struct(当前 fs root / pwd),< 1ms
exit_thread:    释放 thread_info(每线程 1 个),< 10ms
exit_namespaces:释放 nsproxy(pid/mnt/net/uts/ipc/user/cgroup),< 1ms
exit_task_stack:释放 task_struct.stack(16KB 通常),< 1ms
```

**本案**:这 4 步**不慢**。

### 6.5 `schedule()` 切走 + TASK_DEAD 状态

**`do_exit` 走到最后**:

```c
// kernel/exit.c
void do_exit(long code)
{
    // ... 上面的所有 sub-step ...
    sched_set_group_id(tsk);    // 从调度组移除
    sched_dead(tsk);             // 标记为不可调度

    // ★ ★ ★ 切走!
    preempt_disable();
    schedule();                  // 进入 TASK_DEAD 状态
}
```

**TASK_DEAD 状态**:
- task_struct **仍然存在**(`/proc/<pid>` 仍可见)
- 但**不再被调度**——CPU 永远不会选它
- 等待父进程 `wait()` / `waitid()` 收尸
- 如果是孤儿进程,init 进程会领养并 wait

**关键观察**:
- 进程到达 TASK_DEAD **≠** task_struct 被释放
- 释放要等 §7 reaper 收尸
- 12.2s 的延迟可能是**到 TASK_DEAD 已经完成**(~10s),但 **reaper 收尸被 cgroup 拥堵拖了 2s**

### 6.6 阶段 5 慢在哪:5 大根因

> **5 大根因**(按本案概率从高到低):

| 根因 | 表现 | 本案 | 排查 |
|------|------|------|------|
| **① swap 回收慢** | exit_mm 在 swap_free 路径上等待 | ★ 主因 | `cat /proc/<pid>/status` 看 `VmSwap` |
| **② MemFree 极低** | 所有页回收都要走 direct reclaim | ★ 加剧 | `cat /proc/meminfo` 看 `MemFree` |
| **③ 锁竞争** | swap_map / mmap_lock 等锁被并发回收占用 | ★ 隐含 | `cat /proc/lockdep` / ftrace lockdep |
| **④ fd 关闭慢** | 慢文件系统 / 网络 FIN / GPU 资源 | 可能 | `lsof -p <pid>` 看 fd 类型 |
| **⑤ page cache 刷盘** | 文件页 dirty 太多,mmput 等待 | 可能 | `cat /proc/<pid>/smaps` 看 `Locked` / `Dirty` |

**本案根因 ①+②+③ 复合**:
- 进程 swap 78MB
- 设备 MemFree 80MB
- 系统并发大量回收(rms_NormalReclaimer)

**为什么是 12.2s**:**10s 都在 exit_mm**(释放 swap),**2.24s 是 do_exit 其他 sub-step 累计**(exit_files + exit_thread + sched_dead + schedule)。reaper/cgroup 主干是**同步包含在 do_exit 末尾(< 10ms),不占独立阶段**。**真正的 cgroup 销毁残留**("Device or resource busy")是 am_proc_died 之后 24s+ 的事(详见 §7.4)。

## 7. 阶段 6:reaper 收尸 + cgroup 清理

> **本节核心问题**:**进程到达 TASK_DEAD 之后,谁负责释放 task_struct?**
>
> **答案**:父进程调用 `wait` / `waitid` → kernel 走 `release_task` → 释放 task_struct。

### 7.1 父进程 `wait` / `waitid(P_PIDFD)`

**PidfdProcess 收尸的标准做法**:

```cpp
// frameworks/native/libs/binder/PidfdCache.cpp
int PidfdCache::reap(pid_t pid) {
    int pidfd = acquire(pid);  // 拿到 pidfd
    siginfo_t info;
    // ★ 用 P_PIDFD 等待,不依赖 PID race
    int ret = waitid(P_PIDFD, pidfd, &info, WEXITED);
    close(pidfd);
    return ret;
}
```

**kernel 视角**:

```
waitid(P_PIDFD, pidfd, &info, WEXITED)
  ↓
  __waitid(P_PIDFD, ...)
    ↓
    do_wait(...)
      ↓
      wait_task_zombie(tsk, ...)   ← 处理僵尸
        ↓
        release_task(tsk)         ← 释放 task_struct
```

### 7.2 `release_task` 释放 `task_struct`

**`release_task` 干这些事**:

```c
// kernel/exit.c
void release_task(struct task_struct *tsk)
{
    // 1. 从 pid hash 移除
    detach_pid(tsk, PIDTYPE_PID);
    detach_pid(tsk, PIDTYPE_TGID);
    detach_pid(tsk, PIDTYPE_PGID);
    detach_pid(tsk, PIDTYPE_SID);

    // 2. cgroup 清理
    cgroup_release(tsk);                ← ★ 释放 cgroup 节点

    // 3. 释放 task_struct 元数据
    put_cred(tsk->cred);
    put_cred(tsk->real_cred);
    free_task_struct(tsk);               ← ★ kfree(task_struct)
}
```

**本案关键**:**`cgroup_release` 释放 cgroup 节点**——这是 cgroup "Device or resource busy" 错误的来源。

### 7.3 `cgroup_release` + `put_pid`

**`cgroup_release` 走 cgroup 内部清理**:

```c
// kernel/cgroup/cgroup.c
void cgroup_release(struct task_struct *tsk) {
    struct css_set *cset = tsk->cgroups;
    // ...
    cset->task_count--;
    // 如果 cset 没任务了,从 cgroup 节点删除
    if (atomic_read(&cset->refcount) == 1) {
        cgroup_destroy_css_killed(cset);  // 删除 cgroup 节点
    }
}
```

**为什么 "Device or resource busy"**:

- cgroup_destroy_css_killed 内部需要 `cgroup_mutex` 锁
- 如果**其他进程正在写这个 cgroup**(如 `oom_score_adj` 更新),锁被持有
- cgroup 销毁会等待(不能强制打断)
- 本案日志中的 `Still waiting ... uid_10131/pid_15770 after 2209 ms` 就是这个锁等待

**`put_pid` 释放 PID**:

```c
// kernel/pid.c
static void put_pid(struct pid *pid) {
    // 减少 refcount
    if (atomic_dec_and_test(&pid->count)) {
        // refcount == 0,真正释放
        free_pid(pid);  ← ★ kfree(pid_struct)
    }
}
```

**为什么本案 put_pid 不慢**:`put_pid` 主要是 kfree,us 级。

### 7.4 阶段 6 慢在哪

> **架构师结论**:**阶段 6 慢的几乎只有 cgroup 释放**。
>
> **⚠️ 重要诚实声明**:**12.24s 总耗时中,§7 reaper 主干是同步包含在 §6 do_exit 末尾,不占独立阶段**——am_proc_died 31.898 ≈ PidfdProcess.waitid 返回时点,所以 waitid + release_task 主体都在 31.898 之前完成。
>
> 但**真正的 cgroup 销毁等待**(Device or resource busy)是 am_proc_died **之后 24s+** 才出现的,属于"杀进程后 cgroup 残留"问题,**不在 12.24s 区间内**。

#### 7.4.1 正常时序:am_proc_died vs cgroup 释放的精确顺序

**回答用户问题"am_proc_died 打印后才发生 cgroup 的移除吗?":** **是,正常情况下 am_proc_died 早于 cgroup 节点真销毁**——am_proc_died 在 do_exit 末尾的 `do_notify_parent_dead` 阶段触发;cgroup 释放在父进程 `waitid` 返回后的 `release_task` 阶段触发;cgroup 节点真从 `/sys/fs/cgroup` 树消失还要再走 cgroup.workqueue 异步处理。

**Kernel 代码顺序**(单 do_exit vs 父进程 wait 两条线):

```c
// kernel/exit.c do_exit() — 进程 15770 单线程
void do_exit(long code)
{
    // ... 上面的所有 sub-step ...
    exit_mm(tsk);                  // 释放 mmap
    exit_files(tsk);
    // ...
    do_notify_parent_dead(tsk);    // ★ ① 触发 am_proc_died (FWK 感知进程死亡)
    schedule();                    // ② 进入 TASK_DEAD
}

// 父进程 wait() 内核侧 — 另一线程(PidfdCache.reap())
// kernel/exit.c wait_task_zombie → release_task
void release_task(struct task_struct *tsk)
{
    detach_pid(tsk, ...);
    cgroup_release(tsk);           // ★ ③ 释放 cgroup 节点
    put_cred(...);
    free_task_struct(tsk);         // ④ kfree(task_struct)
}

// kernel/cgroup/cgroup.c cgroup_release()
void cgroup_release(struct task_struct *tsk)
{
    cset->task_count--;
    if (atomic_read(&cset->refcount) == 1) {
        cgroup_destroy_css_killed(cset);   // ★ 标记 + 排队
    }
}

// kernel/cgroup/cgroup.c (异步 workqueue 处理)
cgroup_destroy_work() → 实际从 /sys/fs/cgroup 树删除节点   // ★ ⑤ 节点真消失
```

**5 阶段时序图**:

```
进程 15770 do_exit (单线程)              父进程 waitid (另一线程)
─────────────────────                ─────────────────────
exit_mm (10s)                         [阻塞等 child TASK_DEAD]
exit_files / ...                      
do_notify_parent_dead ──────→ am_proc_died (31.898)  ← FWK 感知死亡
schedule() → TASK_DEAD ──────────→ wake_up(wait_queue)  ← 父进程被唤醒
                                      ↓
                                      waitid 返回 (31.898)
                                      ↓
                                      release_task
                                      ↓
                                      cgroup_release (微秒级)
                                      ↓
                                      cgroup_destroy_css_killed (标记)
                                      ↓
                                      [异步 workqueue 排队]
                                      ↓
                                      cgroup_destroy_work
                                      ↓
                                      /sys/fs/cgroup/.../pid_15770 真消失
                                      (几 ms - 几秒,看 cgroup.workqueue 压力)
```

**正常时序差**:
- am_proc_died → waitid 返回:**差 < 1ms** (同步)
- waitid 返回 → cgroup_release:**差 < 1ms** (同步)
- cgroup_release → cgroup 节点真消失:**差几 ms 到几秒** (异步 workqueue)

**本案异常**:
- am_proc_died 在 31.898
- 56.016 才出现 "Still waiting ... after 2209 ms" (cgroup workqueue 等待锁)
- 58.224 才出现 "Device or resource busy" (cgroup 销毁失败)

**24s 缺口** = cgroup.workqueue 被其他进程的 cgroup 清理任务排到后面(系统级锁竞争)+ cgroup_mutex 被持有(vendor 配置问题)。

#### 7.4.2 4 个 sub-step × 5 栏对账表

| sub-step | 源码 | 理想耗时 | 本案耗时(估算) | 日志证据 | 慢在哪(本案) |
|---|---|---|---|---|---|
| **⑦-a 父进程 waitid 返回** | `kernel/exit.c#do_wait` + `wait_task_zombie` | < 1ms | < 1ms | events_log: 31.898 am_proc_died ≈ waitid 返回时点 | 不慢(pidfd 同步) |
| **⑦-b release_task** | `kernel/exit.c#release_task` | < 5ms | < 5ms | 无直接证据(同步包含在 do_exit 末尾) | 不慢(只 detach_pid + cgroup_release + put_cred + kfree) |
| **⑦-c cgroup_release** | `kernel/cgroup/cgroup.c#cgroup_release` | < 1ms | < 1ms | 无直接证据(同步) | 不慢(只减少 refcount) |
| **⑦-d put_pid** | `kernel/pid.c#put_pid` | < 1ms | < 1ms | 无直接证据 | 不慢(us 级 kfree) |
| **⑦ 合计** | — | **< 10ms** | **< 10ms(同步)** | — | 同步不占独立阶段 |

**cgroup 销毁残留(am_proc_died 之后 24s+)**:

```
31.898  am_proc_died (kernel 报死亡)
        ↓ (24s+ 后)
56.016  Still waiting ... uid_10131/pid_15770 after 2209 ms (events_log line 1855)
        ↓ (2.2s 后)
58.224  Unable to remove cgroup ... Device or resource busy (line 1856)
```

**含义**:am_proc_died **之后 24s+**,cgroup 销毁还在等待 `cgroup_mutex` 锁——这是 transsion vendor 的 cgroup 配置 + 系统级锁竞争导致的,不是 do_exit 主耗时。**这部分不影响 12.24s 总耗时**,但**影响 cgroup 节点清理**,对后续 11.3 / 12.3 治理有意义。

**5 大根因(原版)**:

| 根因 | 表现 | 本案 |
|------|------|------|
| **① cgroup 锁竞争** | `cgroup_destroy_css_killed` 等待 `cgroup_mutex` | ★ 主因(但发生在 am_proc_died 之后) |

## 8. 阶段 7:FWK 收尾 `handleAppDied` + 进程完全退出

> **本节核心问题**:**进程在 Kernel 已 TASK_DEAD,FWK 还要做什么?**
>
> **答案**:`onProcessDied` → `cleanUpApplicationRecordLocked` → `removeProcessLocked` → 通知 observers。

### 8.1 `onProcessDied` → `cleanUpApplicationRecordLocked`

**binder death 通知触发**:

```java
private final IBinder.DeathRecipient deathRecipient = new DeathRecipient() {
```

```java
final void handleAppDiedLocked(ProcessRecord app, ...) {

    mProcessList.cleanUpApplicationRecordLocked(app, ...);
        // - 清理所有 ActivityRecord
        // - 清理所有 ServiceRecord
```

## 9. 实战根因分析:`com.sh.smart.caller` 黑屏 12.2s

> **本节把所有"杀进程流程"拉回本案,逐阶段做根因对账**。

### 9.1 现象锁定(events_log + main_log 双向校验)

**events_log 关键节点(时间戳单位 ms)**:

### 9.3 根因 ①:do_exit 在 78MB swap + MemFree 80MB 下拖长

**直接证据(events_log + main_log)**:

```
events_log 19.546:  Usf_appStateManager: handleUpdateTaskOverLap uid: 10131 remove
main_log 19.854:   rms_NormalReclaimer: package: com.sh.smart.caller,uid:10131,
                    (request + memFreeThreshold) - memFree: (222754 + 204800) - 86736 = 340818
                    curMemFree: 84.7MB, swapUsage: 55.35%
```

**含义**:
- **进程在退出前已经触发回收**——`memFreeThreshold=204800KB=200MB`,**当前 MemFree 只有 86.7MB,差 113MB**
- 系统发起 340MB 回收 → 启动这个进程需要先腾空间
- swap 使用率 55.35%——**swap 空间在大量使用**,回收时锁竞争激烈
- 进程自身 DFX(本案设备私有):`mVmSwap=80240KB ≈ 78MB`

**根因链条**:

```
进程 swap 78MB(电话 app 大量缓存)
  +
系统 MemFree 80MB(全局内存压力)
  +
swap 使用率 55%(全局 swap 拥堵)
  +
并发:同窗口大量进程被 lmkd 杀(vending/webview/facebook/redotpay 等)
  ↓
do_exit → exit_mm → unmap_vmas → swap_free
  ↓
swap_map 锁竞争激烈
  ↓
释放 78MB swap 耗时 ~10s
  ↓
进程长时间停留在 do_exit,未到 TASK_DEAD
  ↓
lruProcesses 中 app.thread != null
  ↓
ProcessRecord 长期"removed" 状态
```

**架构师观察**:**这是物理硬件约束,不是 bug**。在低内存设备(8GB RAM)上,如果用户长时间挂着电话 app(2.58h)+ 大量后台 app,这个延迟是必然的。

### 9.4 根因 ②:启动逻辑不预检 `app.thread == null`,沿用旧 pid

**直接证据(events_log)**:

**根因**:ActivityTaskSupervisor 试图把 `ActivityStart` 事务投递到旧进程 15770 的 binder thread,但 15770 已不可用(binder 已死)→ DeadObjectException。

**代码层**(`frameworks/base/services/core/java/com/android/server/wm/ActivityTaskSupervisor.java#startSpecificActivityLocked`):


**架构师观察**:
- 这是 **AOSP 已知 bug**,部分 OEM fix 了,部分没
- 即使 fix 了 `app.removed` 检查,**真正能解决**还要等 do_exit 完成

### 9.5 根因 ③:cgroup 残留 2 次,系统侧清理队列拥堵

**直接证据(events_log,本案设备私有日志)**:

```
20:45:56.016  ... Still waiting ... uid_10131/pid_15770 after 2209 ms
20:45:58.224  ... Unable to remove cgroup ... Device or resource busy
```

**含义**:
- 31.898 am_proc_died 之后,**24s 时(56.0)** 系统仍在尝试清理 cgroup
- 2209ms 后(58.2)报"Device or resource busy"——cgroup 销毁失败
- **kernel cgroup 子系统**和 **system_server** 都在抢这个 cgroup 节点

**根因**:
- 全机批量杀进程(本案窗口 vending/webview/facebook/redotpay 等)
- cgroup 销毁要拿 `cgroup_mutex` 全局锁
- 锁被其他正在写 cgroup 的进程持有
- 释放只能等待

**架构师观察**:
- cgroup "Device or resource busy" 失败后,通常会**重试**或**下次清理**时带走
- 不会无限卡住
- 但**对用户而言**,这个 cgroup 节点残留意味着**未来启动同一包名时,可能会踩到 cgroup 配额问题**

### 9.6 根因 ④:AMS 启动逻辑被 "removed" 状态污染,`refused to die` 主动 cancel

**直接证据(events_log)**:

```
20:45:29.981625  wm_finish_activity: [0,168725480,937,...,starting-proc-removed]
20:45:29.982166  am_wtf  : [0,1624,system_server,-1,ActivityManager,
                  ProcessRecord{...15770...} 15770 refused to die
                  while trying to launch ProcessRecord{...0:com.sh.smart.caller...},
                  cancelling the process start]
```

**AMS 代码层**(`frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java#cleanUpApplicationRecordLocked`):

```java
// AOSP 16 当前实现
final void cleanUpApplicationRecordLocked(ProcessRecord app, ...) {
    // 启动新进程时,被 attachProcessLocked 拒绝
    if (app.removed) {
        // 旧进程"标记要杀但还没死",且 thread 还存在
        Slog.wtf(TAG, app + " refused to die while trying to launch ...");
        // ★ 直接 cancel 新进程启动!
        mProcessList.removeProcessLocked(newApp, ...);
    }
}
```

**含义**:**AMS 不愿意"脏启动"——如果旧进程没死透,就拒绝启动新进程**。这是 AMS 的**保守策略**,但用户体验上表现为"黑屏"。

**架构师观察**:
- 这是 AMS 的**设计选择**,不是 bug
- 设计动机:避免"新进程 attach 到旧 pid"导致不可预测行为
- 但在"旧进程死得慢"的场景下,这个策略会**主动黑屏**

**修复方向**(架构师视角):
- 短期:超时后**强制**启动新进程(`forceRestart=true`)
- 长期:启动前 fork 新进程前先 `kill -9` + `waitpid` 同步等死
- 风险:可能造成两个相同 package 的进程短暂共存

### 9.7 根因 ⑤:`ALL_TIMEOUT` 5s 不是硬超时,是"每 5s 一次检查"

**直接证据(events_log)**:

```
20:45:31.892  am_wtf: remove ALL_TIMEOUT cleanup
              ProcessRecord{...15770:com.sh.smart.caller}
```

**计算**:
- 19.658 发 kill
- 31.892 ALL_TIMEOUT cleanup 触发
- **间隔 12.234s**——远超 5s

**这意味着什么**:`ALL_TIMEOUT` 在 AOSP 16 是**每 5s 跑一次检查**的 watchdog,**不是"杀后等 5s 强制清"**。所以从 19.658 到 31.892 之间跑了 **2-3 个 5s 周期**,每个周期都检查 `app.removed && app.thread != null`,但前几个周期 thread 仍未 null。

**代码层**(`frameworks/base/services/core/java/com/android/server/am/ProcessList.java`):

```java
// mAllProcessesDied 是 handler message,每 5s 触发一次
private static final long ALL_TIMEOUT = 5 * 1000;  // 5s

void scheduleAppGcsLocked() {
    // ... 定期 mAllProcessesDied.run() ...
}
```

**架构师观察**:
- 命名误导:`ALL_TIMEOUT` 让人以为"5s 兜底",实际是"5s 检查周期"
- **真正的清理时机是 kernel 报死亡**,ALL_TIMEOUT 只是兜底
- 改进方向:把 `ALL_TIMEOUT` 改为**渐进式**——1s / 2s / 5s 检查 → 1s / 5s / 30s

**`no vma` 的真正含义**——`binder_alloc_buf failed to map pages in userspace, no vma`:

```c
// drivers/android/binder.c
static int binder_alloc_buf(struct binder_alloc *alloc, size_t size)
{
    // ...
    // 尝试 mmap 一个 page 给 binder transaction buffer
    ret = vmalloc_user(buf, size);  // 或 alloc->pages[i] 路径
    if (ret < 0) {
        // ★ 失败原因:no vma = 该进程没有任何 vma 可以 map
        // 意味着该进程的 mmap 资源已经被内核回收
        return -ENOMEM;
    }
}
```

**为什么 mmap 会被回收?**——本案中至少有 4 个机制可能触发(其中 1 个是本案 transsion vendor 加的):

| 机制 | 触发条件 | 本案 | kernel log 证据 |
|------|---------|------|-----------------|
| **① vmscan.c try_to_unmap** (标准) | kswapd 走 inactive_list 找到该进程 vma,unmap 但保留 vma 结构 | ✅ 可能 | kswapd 持续 30%+ 负载 |
| **② oom_reaper** (标准) | OOM 时,**unmap + reap 整个 mmap**,但**不杀进程** | ❌ | kernel log 无 oom_reaper 记录 |
| **③ memcg force_uncharge** (标准) | cgroup 内存超限,unmap 该 cgroup 进程 vma | ❌ | kernel log 无 memcg 相关 |
| **④ transsion process_reclaim** (vendor hook) | OEM 加的更激进的回收 hook | ✅ **可能** | TranMemoryInfo 是 transsion 自家工具,推断有配套 vendor hook |

**架构师观察**:**kernel log 0 命中 oom_reaper / direct_reclaim / PSI / D state**——这强烈暗示**"no vma" 不是标准内核机制触发的**,而是 transsion 在 vmscan 路径上**vendor hook 了一个"提前回收"机制**(`tran_hang` 等关键字大量出现说明 transsion 在 kernel 加了监控/回收 hook)。

**进程变成"半死"状态**:

```
14:21 UTC 之前(案发前 24 min):  task_struct + mm_struct + vma + page table + 物理页  ← 完整进程
            ↓ kswapd 30%+ + transsion process_reclaim
14:21:16 UTC  mmap 资源被回收 → binder_alloc 失败
            ↓ 主线程尝试发 binder transaction 失败 → 走错误处理
14:21:19 UTC  主线程开始 hang 在 binder 错误处理
            ↓
14:21:46 UTC  主线程 hang 30.6s ← ANR 阈值(5s)早已过
            ↓ 但进程没死
            ↓
(案发前 23 min - 案发 = 24 分钟静默期,进程持续"半死"状态,mmap 资源持续异常)
            ↓
14:45:19 UTC  am_kill 触发
            ↓
            do_exit
            ↓
            exit_mm 试图 unmap 已经被回收过一次的 vma
            ↓
            内核要做大量状态修正 + 锁竞争
            ↓
            12.2s
```

**这才是 12.2s 的真正根因**——`do_exit` 在 mmap 状态异常的 task 上被迫"修复"内核资源,而不是简单地"释放"。

**为什么 `do_exit` 12.2s 这么慢**(机制层,**所有数字为估算,非精确测量**):

| sub-step | 理想耗时 | 本案耗时(估算) | 日志证据 | 慢在哪 |
|---------|---------|---------|---------|-------|
| `exit_mm` → unmap_vmas | ~10ms | **~10s** | 推论: rss=170MB + swap 78MB = 248MB 物理页要 unmap;`binder_alloc: 15770: no vma` 4 次 + `process_reclaim` 伏笔 | **vma 状态异常,unmap 时反复查 vma flags / page state** |
| `exit_mm` → swap_free | ~30ms | **~2-3s** | 推论: 78MB swap = ~8000 PTE,`swap_map` 全局锁,swap 55% 使用率锁竞争 | swap_map 锁等待 + vma 状态异常 |
| `exit_files` → close fd | ~5ms | ~10ms | 推论: 进程 fd < 500(健康),部分 fd 在 §9.8 "no vma" 阶段已释放 | 不慢(健康范围) |
| `exit_thread` / `exit_stack` | ~5ms | ~5ms | 无直接证据 | 正常 |
| `schedule()` → TASK_DEAD + am_proc_died | ~1ms | ~1ms | events_log: 31.898 am_proc_died ≈ 同步触发 | 正常 |
| reaper / release_task(同步) | < 10ms | < 10ms | events_log: 31.898 am_proc_died ≈ waitid 返回 | 同步不占独立阶段 |
| FWK 收尾 (ALL_TIMEOUT) | ~10ms | ~200ms | events_log: 31.892 ALL_TIMEOUT cleanup 触发(每 5s 检查) | 兜底 |
| **合计** | **~80ms** | **~12.24s** | **直接观测**: 19.658 am_kill → 31.898 am_proc_died | **99% 在 exit_mm 内部(82% unmap_vmas + 16% swap_free)** |

> **⚠️ 关键诚实声明**:
> - **总时长 12.24s 直接观测**(events_log 19.658 am_kill 与 31.898 am_proc_died 间隔)。
> - **每个 sub-step 的"本案耗时"都是估算**,基于"理想 ms 级 + 已知慢点(swap 78MB / vma 异常 / cgroup_mutex 拥堵)"反推。
> - **cgroup "Device or resource busy" 实际出现在 am_proc_died 之后 24s+**(56.0/58.2s,events_log line 1855-1856),**不在 12.24s 区间内**——见 §7.4 说明。
> - **要精确测每个 sub-step 耗时**,需提前开 ftrace(详见 §6.7.3)。

**`exit_mm` 在 mmap 状态异常 task 上变慢的原因**:

```
exit_mm(tsk)
  ↓
mmput(tsk->mm)
  ↓
exit_mmap(mm)
  ↓
unmap_vmas(mm->mmap, ...)         ← vma 列表(很多状态异常)
  ↓
  for each vma in mm->mmap:
    unmap_page_range(vma, ...)     ← ★ 这步会非常慢!
      ↓
      unmap_single_vma(vma, ...)
        ↓
        zap_pte_range(...)
          ↓
          if pte is swap entry: swap_free()   ← swap slot 已被人动过
          if pte_present(...): release_pte()   ← 部分 page 已被 process_reclaim 释放
          if vma is VM_DENYWRITE: ...        ← vma 状态不一致
        ↓
        tlb_finish_mmu(...)                  ← TLB flush(可能因为 SMP cross-call 慢)
  ↓
mmdrop(mm)                        ← 释放 mm_struct
  ↓
kfree(mm)                         ← 释放 mm_struct 内存
```

**慢的关键路径**:
1. **`unmap_page_range` 反复查 vma 的状态**——`no vma` 阶段回收后,vma 标志位可能不一致(VM_IO / VM_PFNMAP / VM_HUGEPAGE 等)
2. **`swap_free` 重复释放**——`process_reclaim` 已经释放过 swap slot,do_exit 再释放一次 → 触发 swap_map lock contention
3. **TLB flush 跨 CPU**——多核情况下,`tlb_finish_mmu` 需要 IPI 其他 CPU,本案 kswapd 持续跑意味着 CPU 忙
4. **vm_area_struct 自身状态异常**——vendor hook 改过某些 vma 但没清标记,do_exit 走标准路径会反复检查

**架构师观察**:

1. **不要把"杀进程慢"当成纯 FWK/Kernel 主干问题**——`do_exit` 慢可能源于**进程在被杀前已经被各种机制"半死"**
2. **kernel log + events_log 必须双源对比**——只看一个会漏掉"伏笔"
3. **transsion 这类 OEM 设备的 vendor hook**是杀进程慢的隐性根因——AOSP 上不存在的"process_reclaim"在 MTK/transsion 平台上常见
4. **`no vma` 错误是 golden signal**——出现意味着进程 mmap 资源被某种回收机制(标准/非标)动了

**修正项**:

| 之前的判断 | 修正后 |
|----------|------|
| "do_exit 慢 = exit_mm 回收 swap" | 真实原因:exit_mm unmap 已经被 process_reclaim 预回收的 vma,状态异常 |
| "本案是单事件 |" | 实际:本案前 24min 进程已经"半死",am_kill 触发收尸 |
| "swap 78MB 是主因" | swap 78MB 是诱因(系统级内存压力),但 exit_mm 慢的真正原因是 vma 状态异常 |
| "5 大根因矩阵" | 应该是 **6 大根因**:加上 **"mmap 已被预回收(0 状态 vma)"** |

**新增到 §10 根因矩阵的 1 大根因**(更新):

| 6 大根因 | 主要阶段 | 占比(经验)| 本案 | 排查命令 |

| ... | ... | ... | ... | ... |

**预防建议**:

### 11.1 30 秒定位(关键词扫日志)

```bash
# - "am_wtf: 15770 refused to die"  → 阶段 §3 + §6/§7 复合
# - "remove ALL_TIMEOUT cleanup"  → 阶段 §8 兜底
```

**30 秒判读规则**:

| 看到 | 判断 |
|------|------|
| am_kill 后 200ms 内 am_proc_died | 杀进程正常,问题在别处 |
| am_kill 后 1-5s am_proc_died | 轻微杀进程慢,关注 §6 |
| am_kill 后 5s+ am_proc_died | 杀进程严重慢,优先看 §6+§7 |
| "refused to die" 出现 | 阶段 §3 + §6/§7 复合,本案 |
| "ALL_TIMEOUT cleanup" 出现 | 阶段 §8 兜底生效 |

**30 秒输出**:**结论 + 阶段归因**。

### 11.2 5 分钟定位(阶段拆解)

**适用场景**:**30 秒定位后,需要进一步定位具体阶段**。


**ftrace 抓 do_exit**:
```bash

adb shell "echo 0 > /sys/kernel/debug/tracing/tracing_on"
adb shell "echo > /sys/kernel/debug/tracing/trace"

# 2. 打开 exit 相关 tracepoints
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_process_exit/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/syscalls/sys_enter_kill/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/signal/signal_generate/enable"

# 3. 复现杀进程
adb shell am force-stop <package>

# 4. 抓 5s

adb shell "cat /sys/kernel/debug/tracing/trace" > /tmp/trace.log
adb pull /tmp/trace.log .
```

**ftrace 关键事件**:
```
kworker/R  ... signal_generate: sig=9 ...
<...>-1234 ... sched_process_exit: comm=<package> pid=1234 prio=...
<...>-1234 ... do_exit: code=9
<...>-1234 ... mmput
<...>-1234 ... exit_mmap start
```

**30 分钟输出**:**精确到函数级的耗时分布**。

## 12. 总结:架构师视角 5 条 Takeaway

### 12.1 5 条 Takeaway

1. **杀进程慢 80%+ 在 Kernel(§6 do_exit)**,FWK 阶段是 ms 级。看到 "启动黑屏" 不要先查 AMS,先去 `cat /proc/<pid>/status` 看 VmSwap。

2. **`ALL_TIMEOUT` 不是"5s 超时",是"5s 检查周期"**——所以本案 12.2s 才兜底,不是"应该 5s 就清"。看代码不要被命名误导。

3. **"refused to die" 是 AMS 的保守策略**——宁愿黑屏也不愿"脏启动"短期可接受,长期要把"同步等死"做到 fork 之前。

4. **5 大根因对应 5 个监控指标**——VmSwap / MemFree / cgroup 销毁耗时 / am_kill→am_proc_died 间隔 / ALL_TIMEOUT 出现频率。每个都该有线上 dashboard。

5. **SIGKILL 路径 ART 几乎不做事**——任何"退出前 flush"设计在 SIGKILL 路径上**全部失效**。业务方要定期 flush,不能依赖"退出前"。

### 12.2 排查路径速查表

| 看到症状 | 30 秒判断 | 5 分钟定位 | 30 分钟取证 |
|---------|----------|----------|------------|
| 启动黑屏 ≥ 1s | events_log 查 am_kill→am_proc_died 间隔 | 7 阶段时间轴 | ftrace sched_process_exit |
| `am_wtf refused to die` | 阶段 §3 + §6/§7 复合 | 看 VmSwap | exit_mmap ftrace |
| `remove ALL_TIMEOUT cleanup` | 阶段 §8 兜底生效 | 看 am_kill 时间 | 等下次复现抓 ftrace |
| `Unable to remove cgroup Device or resource busy` | 阶段 §7 拥堵 | 看 cgroup.procs | perfetto cgroup trace |
| 进程 cgroup 残留 | 阶段 §7 残留未清 | `ls /sys/fs/cgroup/uid_*/pid_*` | cgroup_mutex 锁 trace |

### 12.3 治理清单(7 类治理动作)

| # | 治理动作 | 解决根因 | 难度 | 优先级 |
|---|---------|---------|------|-------|
| **1** | 启动前预检 `app.thread == null && app.removed` | 根因 ② | 中 | **P0** |
| **2** | `am_wtf refused to die` 时**强制**启动新进程 | 根因 ④ | 中 | **P0** |
| **3** | fork 新进程前同步 `kill -9` + `waitpid` 等死 | 根因 ①+④ | 高 | **P1** |
| **4** | 监控 VmSwap > 100MB 触发主动 trim | 根因 ① | 低 | **P1** |
| **5** | cgroup 销毁失败**重试** + 异步清理 | 根因 ② | 中 | **P1** |
| **6** | ALL_TIMEOUT 改为渐进式(1s/2s/5s/30s) | 根因 ⑤ | 中 | **P2** |
| **7** | lmkd 避免在低 MemFree 时全机批量杀 | 根因 ① 加剧 | 高 | **P2** |

**架构师建议**:**先做 #1 + #2**(成本低,效果显著),再做 #4 + #5**(成本中,治本),最后做 #3**(成本高,需要 framework 大改)。
