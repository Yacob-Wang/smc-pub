# 06-窗口动画与转场

## 1. 窗口动画在 Window 架构中的位置

### 1.1 动画组件在 WMS 架构中的定位

窗口动画是 WMS 中"让用户感知到变化"的模块——窗口的出现、消失、切换、大小变化，如果没有动画，用户看到的将是瞬间跳变的画面。窗口动画操作的对象不是 View 属性（alpha、translationX 等），而是 **SurfaceControl**——直接在 SurfaceFlinger 层面对整个窗口的 Surface 进行变换（位移、缩放、透明度、裁剪）。

### 1.2 窗口动画 vs View 动画：本质区别

理解窗口动画必须先厘清它与 View 动画的根本差异：

| 维度 | 窗口动画 (Window Animation) | View 动画 (View/Property Animation) |
|:---|:---|:---|
| 操作对象 | `SurfaceControl`（整个窗口的 Surface） | `View` 属性（alpha、translationX、scaleX 等） |
| 执行位置 | `system_server` 的动画线程 | App 进程的主线程 |
| 渲染方式 | 直接修改 SurfaceFlinger Layer 的变换矩阵 | 通过 `invalidate()` 触发 View 树重绘 |
| 影响范围 | 整个窗口（包括 DecorView 及其所有子 View） | 单个 View 及其子 View |
| 性能特征 | 不触发 App 端 measure/layout/draw，极轻量 | 可能触发重绘、重排，CPU/GPU 开销大 |
| 典型场景 | Activity 进入/退出、窗口最小化/最大化、分屏切换 | 按钮点击反馈、列表项入场、Loading 转圈 |

**关键认知：** 当你看到 Activity 切换时整个画面的滑入/滑出/缩放效果，那是窗口动画——`system_server` 在操作 SurfaceControl；当你看到一个按钮在点击后缩放回弹，那是 View 动画——App 进程在操作 View 属性。窗口动画不会触发 App 端的任何绘制操作，因此即使 App 主线程卡住，窗口动画仍然可以流畅播放。

### 2.1 三大核心组件

### 2.2 WindowAnimator — 动画驱动器

> 源码：`frameworks/base/services/core/java/com/android/server/wm/WindowAnimator.java`

`WindowAnimator` 是整个窗口动画系统的"心脏"，它通过 `Choreographer` 注册 VSYNC 回调，每帧调用 `animate()` 驱动所有动画前进。

```java
// frameworks/base/services/core/java/com/android/server/wm/WindowAnimator.java（简化）
class WindowAnimator {
    final WindowManagerService mService;
    
    // 在 AnimationThread 上的 Choreographer
    private final Choreographer mChoreographer;
    
    // 标记是否有动画正在运行
    private boolean mAnimating;
    private boolean mLastRootAnimating;

    // 动画帧回调——注册到 Choreographer 的 CALLBACK_TRAVERSAL
    private final Choreographer.FrameCallback mAnimationFrameCallback = frameTimeNs -> {
        synchronized (mService.mGlobalLock) {
            mAnimationFrameCallbackScheduled = false;
            animate(frameTimeNs);
        }
    };

    // 每帧的动画核心方法
    private void animate(long frameTimeNs) {
        // 遍历所有 DisplayContent
        for (int i = mService.mRoot.getChildCount() - 1; i >= 0; i--) {
            final DisplayContent dc = mService.mRoot.getChildAt(i);

            // 执行该 Display 上所有窗口的动画
            dc.prepareSurfaces();
        }

        // 检查是否还有动画在运行
        boolean rootAnimating = mService.mRoot.isAnimating(TRANSITION | CHILDREN);

        if (rootAnimating && !mLastRootAnimating) {
            // 动画开始：提升 system_server 优先级
            Trace.traceBegin(Trace.TRACE_TAG_WINDOW_MANAGER, "wmAnimate");
        }

        if (rootAnimating) {
            // 还有动画 → 请求下一帧
            scheduleAnimation();
        }

        // 提交 SurfaceControl.Transaction
        mService.mRoot.getSyncTransaction().apply();
    }

    void scheduleAnimation() {
        if (!mAnimationFrameCallbackScheduled) {
            mAnimationFrameCallbackScheduled = true;
            mChoreographer.postFrameCallback(mAnimationFrameCallback);
        }
    }
}
```

> **稳定性架构师视角：** `animate()` 在持有 `mGlobalLock` 的情况下执行。如果单帧动画计算耗时过长（如窗口数量极多、Surface 变换复杂），会延长 `mGlobalLock` 的持有时间，阻塞其他线程的 `addWindow`/`relayout` 等操作。更严重的是，如果 `animate()` 中出现异常导致 `scheduleAnimation()` 不再被调用，所有正在进行的窗口动画将冻结——表现为窗口卡在转场的中间状态，用户看到半透明或错位的画面。

### 2.3 SurfaceAnimator — Leash 动画机制

> 源码：`frameworks/base/services/core/java/com/android/server/wm/SurfaceAnimator.java`

`SurfaceAnimator` 是每个 `WindowContainer` 都持有的动画控制器。它的核心设计是 **Leash（缰绳）机制**——在动画期间，在目标 SurfaceControl 之上插入一个临时的 Leash SurfaceControl，动画的所有变换都作用在 Leash 上，而非直接操作目标 Surface。这样做的好处是：动画结束后移除 Leash 即可恢复原始状态，无需手动还原所有变换。

```
动画前:                          动画中:                          动画后:
Parent SurfaceControl           Parent SurfaceControl           Parent SurfaceControl
  └── Window SurfaceControl       └── Leash SurfaceControl        └── Window SurfaceControl
       (原始位置)                      │ (应用变换矩阵:              (恢复原始位置)
                                       │  scale, alpha, position)
                                       └── Window SurfaceControl
                                            (位置不变，由 Leash 带动)
```

```java
// frameworks/base/services/core/java/com/android/server/wm/SurfaceAnimator.java（简化）
class SurfaceAnimator {
    private final WindowContainer mAnimatable;
    
    // 当前动画正在使用的 Leash
    private SurfaceControl mLeash;
    
    // 当前正在运行的动画适配器
    private AnimationAdapter mAnimation;

    void startAnimation(SurfaceControl.Transaction t,
            AnimationAdapter anim, boolean hidden,
            @AnimationType int type) {
        cancelAnimation(t, true /* restarting */, true /* forwardCancel */);
        
        // 1. 创建 Leash
        mLeash = createAnimationLeash(mAnimatable, t);
        
        // 2. 将目标 SurfaceControl 重新挂到 Leash 下
        t.reparent(mAnimatable.getSurfaceControl(), mLeash);
        
        // 3. 启动动画
        mAnimation = anim;
        mAnimation.startAnimation(mLeash, t,
                type, (animType, adapter) -> {
                    // 动画完成回调
                    synchronized (mAnimatable.mWmService.mGlobalLock) {
                        onAnimationFinished(animType, adapter);
                    }
                });
    }

    private void onAnimationFinished(@AnimationType int type,
            AnimationAdapter anim) {
        // 1. 将目标 SurfaceControl 从 Leash 下移回原父节点
        SurfaceControl.Transaction t = mAnimatable.getSyncTransaction();
        t.reparent(mAnimatable.getSurfaceControl(),
                mAnimatable.getParentSurfaceControl());
        
        // 2. 移除 Leash
        t.remove(mLeash);
        mLeash = null;
        mAnimation = null;
        
        // 3. 通知 WindowContainer 动画结束
        mAnimatable.onAnimationFinished(type, anim);
    }

    private SurfaceControl createAnimationLeash(
            WindowContainer animatable, SurfaceControl.Transaction t) {
        SurfaceControl leash = animatable.makeAnimationLeash()
                .setName(animatable.getSurfaceControl() + " - animation-leash")
                .setCallsite("SurfaceAnimator.createAnimationLeash")
                .build();
        // Leash 继承目标 Surface 的位置和大小
        t.setPosition(leash, 0, 0);
        t.show(leash);
        return leash;
    }
}
```

> **稳定性架构师视角：** Leash 泄漏是窗口动画最常见的稳定性问题之一。如果 `onAnimationFinished()` 没有被正确调用（如动画被意外中断、AnimationAdapter 抛出异常），Leash SurfaceControl 将永远存在于 SurfaceFlinger 的 Layer 树中。每个泄漏的 Leash 至少占用一个 Layer 和对应的 fd/内存。累积泄漏会导致 Layer 数量超限或显存增长。排查命令：`dumpsys SurfaceFlinger --list` 中搜索 `animation-leash`，正常情况下动画结束后不应有残留的 leash Layer。

### 2.4 AnimationThread 与 SurfaceAnimationThread

WMS 动画使用两个专用线程，避免占用 WMS 主线程（`android.display`）的时间：

```
system_server 进程
├── "android.display" (WMS 主线程)
│     └── 窗口管理逻辑: addWindow / removeWindow / 焦点计算 / 布局
│         ⚠️ 不执行动画计算，避免阻塞窗口管理
│
├── "android.anim" (AnimationThread)
│     └── WindowAnimator.animate() 的执行线程
│         Choreographer 注册在此线程的 Looper 上
│         持有 mGlobalLock 执行动画帧计算
│
├── "android.anim.lf" (SurfaceAnimationThread)
│     └── SurfaceAnimationRunner 的执行线程
│         ValueAnimator 驱动的 SurfaceControl 变换
│         不持有 mGlobalLock，直接操作 SurfaceControl.Transaction
│
└── Binder 线程池
      └── App 端的窗口请求入口
```

```java
// frameworks/base/services/core/java/com/android/server/wm/AnimationThread.java
public final class AnimationThread extends ServiceThread {
    private static AnimationThread sInstance;
    private static Handler sHandler;

    private AnimationThread() {
        super("android.anim", THREAD_PRIORITY_DISPLAY, false);
    }
    
    public static AnimationThread get() {
        if (sInstance == null) {
            sInstance = new AnimationThread();
            sInstance.start();
            sHandler = new Handler(sInstance.getLooper());
        }
        return sInstance;
    }
}
```

```java
// frameworks/base/services/core/java/com/android/server/wm/SurfaceAnimationThread.java
public final class SurfaceAnimationThread extends ServiceThread {
    private SurfaceAnimationThread() {
        super("android.anim.lf", THREAD_PRIORITY_DISPLAY, false);
    }
}
```

> **稳定性架构师视角：** 两个动画线程的分工至关重要。`AnimationThread` 持有 `mGlobalLock`，执行的是动画状态的管理和协调；`SurfaceAnimationThread` 不持有 `mGlobalLock`，只负责通过 `ValueAnimator` 驱动 SurfaceControl 变换。如果这两个线程发生死锁或被长时间阻塞（如 SurfaceFlinger Transaction 提交卡住），所有窗口动画将冻结。在 traces.txt 中，如果看到 `android.anim` 或 `android.anim.lf` 线程 blocked on lock 或 stuck in native call，这就是动画冻结的直接原因。

### 2.5 SurfaceAnimationRunner — 动画执行器

> 源码：`frameworks/base/services/core/java/com/android/server/wm/SurfaceAnimationRunner.java`

`SurfaceAnimationRunner` 在 `SurfaceAnimationThread` 上执行实际的动画插值和 SurfaceControl 变换：

```java
// frameworks/base/services/core/java/com/android/server/wm/SurfaceAnimationRunner.java（简化）
class SurfaceAnimationRunner {
    // 待启动的动画
    private final ArrayMap<SurfaceControl, RunningAnimation> mPendingAnimations =
            new ArrayMap<>();
    // 正在运行的动画
    private final ArrayMap<SurfaceControl, RunningAnimation> mRunningAnimations =
            new ArrayMap<>();

    void startAnimations(long frameTimeNs) {
        // 在 SurfaceAnimationThread 上执行
        for (int i = mPendingAnimations.size() - 1; i >= 0; i--) {
            RunningAnimation anim = mPendingAnimations.valueAt(i);
            // 创建 ValueAnimator
            ValueAnimator animator = ObjectAnimator.ofFloat(0f, 1f);
            animator.setDuration(anim.mAnimSpec.getDuration());
            animator.setInterpolator(anim.mAnimSpec.getInterpolator());
            animator.addUpdateListener(animation -> {
                // 每帧更新 SurfaceControl 的变换
                applyTransformation(anim, animator);
            });
            animator.start();
            mRunningAnimations.put(anim.mLeash, anim);
        }
        mPendingAnimations.clear();
    }

    private void applyTransformation(RunningAnimation anim,
            ValueAnimator animator) {
        SurfaceControl.Transaction t = new SurfaceControl.Transaction();
        // 获取当前动画进度对应的变换矩阵
        anim.mAnimSpec.apply(t, anim.mLeash, animator.getCurrentPlayTime());
        // 提交变换
        t.apply();
    }
}
```

## 3. AppTransition — Activity 转场动画

### 3.1 转场类型

> 源码：`frameworks/base/services/core/java/com/android/server/wm/AppTransition.java`

`AppTransition` 管理 Activity 切换时的转场动画。每次 Activity 变化（打开、关闭、移到前台/后台）都对应一种转场类型：

| 转场类型 | 常量值 | 触发场景 | 动画效果（默认） |
|:---|:---|:---|:---|
| `TRANSIT_OPEN` | 打开 | 启动新 Activity（栈中无同类实例） | 新窗口从右侧滑入，旧窗口向左淡出 |
| `TRANSIT_CLOSE` | 关闭 | finish Activity | 当前窗口向右滑出，底层窗口从左侧滑入 |
| `TRANSIT_TO_FRONT` | 移到前台 | 从 Recents 恢复已存在的 Activity | 类似 OPEN，但不创建新实例 |
| `TRANSIT_TO_BACK` | 移到后台 | 按 Home 键 / 切到其他 Task | 当前窗口缩小/淡出 |
| `TRANSIT_CHANGE` | 窗口变化 | 分屏切换、PiP 进入/退出、旋转 | 窗口 bounds 变化动画 |
| `TRANSIT_KEYGUARD_GOING_AWAY` | 锁屏消失 | 解锁屏幕 | 锁屏界面淡出/滑出 |
| `TRANSIT_KEYGUARD_OCCLUDE` | 锁屏遮挡 | 在锁屏上显示某些 Activity | 特殊遮挡动画 |

```java

    static final int TRANSIT_TO_FRONT = 8;
```

### 3.2 转场状态机

`AppTransition` 内部维护了一个状态机，管理转场动画的完整生命周期：

```
                    ┌────────────────────────────────────────────────────┐
                    │              AppTransition 状态机                   │
                    │                                                    │
                    │                                                    │
   Activity 切换 ──▶│  IDLE ──setAppTransition()──▶ SET                 │
   请求到来         │                                │                   │
                    │                                │ 等待参与窗口就绪    │
                    │                                ▼                   │
                    │                              READY                 │
                    │                                │                   │
                    │                    goodToGo() │ 所有窗口就绪        │
                    │                                ▼                   │
                    │                             RUNNING                │
                    │                                │                   │
                    │                 动画播放完成    │                   │
                    │                                ▼                   │
                    │                            FINISHED ──▶ IDLE      │
                    │                                                    │
                    │  超时路径:                                          │
                    │  SET/READY 状态停留超过阈值 → 强制 goodToGo()      │
                    │  → 可能导致动画异常（窗口未绘制完就开始转场）       │
                    └────────────────────────────────────────────────────┘
```

```java
// frameworks/base/services/core/java/com/android/server/wm/AppTransition.java（简化）
public class AppTransition implements Dump {

    // 状态常量
    static final int APP_STATE_IDLE = 0;
    static final int APP_STATE_READY = 1;
    static final int APP_STATE_RUNNING = 2;
    static final int APP_STATE_TIMEOUT = 3;

    private int mAppTransitionState = APP_STATE_IDLE;
    private @TransitionType int mNextAppTransitionType = TRANSIT_UNSET;

    // 设置转场类型
    void setAppTransition(@TransitionType int transit, int flags) {
        mNextAppTransitionType = transit;
        mAppTransitionState = APP_STATE_IDLE;
        // 状态从 IDLE → 等待 goodToGo
    }

    // 标记转场已准备就绪
    void setReady() {
        mAppTransitionState = APP_STATE_READY;
        // 接下来由 goodToGo() 判断是否可以开始
    }

    // 判断是否可以开始转场动画
    int goodToGo(@TransitionType int transit, ActivityRecord topOpeningApp,
            ActivityRecord topClosingApp) {
        // 检查所有参与转场的窗口是否已经绘制完毕
        // 如果 topOpeningApp 的主窗口 mHasSurface == false → 还不能开始
        // 如果 Starting Window 已显示 → 可以先用 Starting Window 开始转场
        mAppTransitionState = APP_STATE_RUNNING;
        return transit;
    }

    void postFinishTransition() {
        mAppTransitionState = APP_STATE_IDLE;
        mNextAppTransitionType = TRANSIT_UNSET;
    }
}
```

### 3.3 AppTransitionController — 转场编排

> 源码：`frameworks/base/services/core/java/com/android/server/wm/AppTransitionController.java`

`AppTransitionController` 负责编排转场动画的具体执行——决定哪些窗口参与、选择什么动画、何时启动：

```java
// frameworks/base/services/core/java/com/android/server/wm/AppTransitionController.java（简化）
class AppTransitionController {

    void handleAppTransitionReady() {
        // 1. 收集参与转场的 ActivityRecord
        ArraySet<ActivityRecord> openingApps = new ArraySet<>();
        ArraySet<ActivityRecord> closingApps = new ArraySet<>();
        collectActivityRecords(openingApps, closingApps);

        // 2. 确定转场类型
        @TransitionType int transit = getTransitType(openingApps, closingApps);

        // 3. 为每个参与者选择动画
        for (ActivityRecord app : openingApps) {
            Animation anim = mService.mAppTransition.loadAnimation(
                    app, transit, true /* enter */);
            app.applyAnimation(anim, transit, true /* enter */);
        }
        for (ActivityRecord app : closingApps) {
            Animation anim = mService.mAppTransition.loadAnimation(
                    app, transit, false /* enter */);
            app.applyAnimation(anim, transit, false /* enter */);
        }

        // 4. 更新焦点
        mDisplayContent.updateFocusedWindowLocked(
                UPDATE_FOCUS_PLACING_SURFACES, true /* updateInputWindows */);
    }
}
```

### 3.4 焦点切换与转场的危险时间窗

Activity 转场期间，焦点窗口的切换存在一个关键时间窗——旧窗口已经失去焦点，但新窗口尚未获得焦点。这段时间内如果有 Key 事件到达 InputDispatcher，会触发 ANR：

```
T=0ms     AMS: startActivity(B)
          WMS: setFocusedApp(B.appToken)
          → InputDispatcher: FocusedApplication = B
          → InputDispatcher: FocusedWindow = A (旧窗口仍有焦点)
 
T=10ms    AppTransition: 开始 TRANSIT_OPEN 动画
          WMS: A 的窗口开始退出动画 (closing)
          WMS: updateFocusedWindowLocked()
          → A 正在 closing → canReceiveKeys() = false
          → B 的窗口还没 addWindow
          → FocusedWindow = null  ←── 危险！
          → InputDispatcher: FocusedWindow = null

T=10ms ~ T=?ms   ★★★ 焦点空窗期 ★★★
                  InputDispatcher: FocusedApplication = B, FocusedWindow = null
                  此时 Key 事件到达 → 等待 FocusedWindow 出现
                  等待超过 5000ms → ANR!

T=100ms   B 的 Activity 完成 onCreate/onResume
          ViewRootImpl.setView() → WMS.addWindow(B 的窗口)
          WMS: updateFocusedWindowLocked()
          → FocusedWindow = B 的窗口
          → InputDispatcher: FocusedWindow = B ←── 焦点恢复
```

> **稳定性架构师视角：** 转场期间的焦点空窗期是 Input ANR 的高发区。空窗期的长度取决于：① 新 Activity 的启动速度（Application.onCreate 耗时 + Activity.onCreate 耗时 + 首帧绘制时间）；② WMS 锁竞争导致的 `addWindow` 延迟。当空窗期超过 5000ms，InputDispatcher 会对 `FocusedApplication`（即新 Activity B）发出 ANR。排查 ANR 时，`dumpsys input` 中 `FocusedApplications` 有值但 `FocusedWindows` 为 null/空，是典型的转场焦点空窗 ANR 特征。详细分析参见 [07-WMS 与 Input 焦点管理](07-WMS与Input焦点管理.md)。

### 3.5 goodToGo() — 转场启动的判定

`goodToGo()` 是转场动画能否启动的"门禁"——它检查所有参与转场的窗口是否已准备就绪：

```java
// frameworks/base/services/core/java/com/android/server/wm/AppTransitionController.java（简化）
// 判断转场是否可以开始的核心逻辑
private boolean transitionGoodToGo(ArraySet<ActivityRecord> apps,
        ArrayMap<ActivityRecord, Integer> outReasons) {
    for (int i = apps.size() - 1; i >= 0; i--) {
        ActivityRecord app = apps.valueAt(i);
        
        // 检查窗口是否已经绘制
        if (!app.allDrawn && !app.startingDisplayed) {
            // 窗口还没绘制完，也没有 Starting Window 可以替代
            // → 不能开始转场
            return false;
        }
    }
    return true;
}
```

**`allDrawn` 和 `startingDisplayed` 的含义：**

| 标志 | 含义 | 何时为 true |
|:---|:---|:---|
| `allDrawn` | Activity 的所有窗口都完成了首帧绘制 | `finishDrawingWindow()` 被所有窗口调用后 |
| `startingDisplayed` | Starting Window（启动窗口）已显示 | Starting Window 完成绘制并可见 |

当新 Activity 的主窗口还没绘制完成时，如果有 Starting Window，`goodToGo()` 会放行——用户先看到 Starting Window 的转场动画，等主窗口绘制完成后再替换。这就是 Android 12+ SplashScreen 的工作机制。

如果既没有 `allDrawn` 也没有 `startingDisplayed`，转场会被阻塞。但 WMS 有超时保护——超过一定时间后强制 `goodToGo()`：

```java
// AppTransition.java
// 转场就绪的超时时间
static final int APP_TRANSITION_TIMEOUT_MS = 5000;
```

> **稳定性架构师视角：** `goodToGo()` 无法通过意味着转场动画被阻塞。最常见的原因是新 Activity 的首帧绘制过慢——`Application.onCreate()` 耗时、`Activity.onCreate()` 中执行了重量级 IO、`setContentView()` 的布局过于复杂。在阻塞期间，旧窗口可能已经不可见（被系统隐藏），但新窗口还没显示，用户看到的是黑屏或白屏。超时强制启动后，用户可能看到未绘制完成的窗口一闪而过再被正式内容替换，表现为"闪烁"。

## 4. 远程动画（RemoteAnimation）

### 4.1 RemoteAnimation 机制

> 源码：`frameworks/base/services/core/java/com/android/server/wm/RemoteAnimationController.java`

Android 10 引入了 `RemoteAnimationAdapter`，允许 App（通常是 Launcher 或 SystemUI）自定义 Activity 转场动画。与传统的窗口动画不同，远程动画的控制权从 `system_server` 交给了 App 进程——App 直接操作 SurfaceControl 实现任意自定义效果。

```
传统窗口动画:
  WMS 选择预定义动画 → WindowAnimator/SurfaceAnimator 执行
  （动画逻辑全在 system_server 中）

远程动画 (RemoteAnimation):
  App 注册 RemoteAnimationAdapter
  → WMS 通知 App: onAnimationStart(targets[], wallpaperTargets[])
  → App 操作 SurfaceControl.Transaction 自定义动画
  → App 动画完成后调用 finishCallback
  → WMS 清理动画状态

   system_server                              App 进程 (Launcher/SystemUI)
   ─────────────                             ──────────────────────────────
   Activity 切换触发                          
        │                                    
   AppTransition 判断有 RemoteAnimation       
        │                                    
   RemoteAnimationController                  
        │ Binder 回调                         
        │──onAnimationStart()──────────────▶ IRemoteAnimationRunner.onAnimationStart()
        │    参数: RemoteAnimationTarget[]       │
        │          (包含 SurfaceControl Leash)   │ 获得 Leash SurfaceControl
        │                                        │ 开始自定义动画
        │                                        │ 每帧:
        │                                        │   Transaction.setScale(leash, ...)
        │                                        │   Transaction.setAlpha(leash, ...)
        │                                        │   Transaction.apply()
        │                                        │
        │   2 秒超时计时开始                      │ ... 动画播放中 ...
        │                                        │
        │◀──finishCallback()──────────────────── │ 动画完成
        │                                        
   清理 Leash, 恢复窗口状态                     
```

### 4.2 RemoteAnimationController 源码

```java
// frameworks/base/services/core/java/com/android/server/wm/RemoteAnimationController.java（简化）
class RemoteAnimationController {
    private final RemoteAnimationAdapter mRemoteAnimationAdapter;
    private final ArrayList<RemoteAnimationRecord> mPendingAnimations = new ArrayList<>();
    
    // 超时时间：2 秒
    private static final long TIMEOUT_MS = 2000;
    
    private final Runnable mTimeoutRunnable = () -> {
        synchronized (mService.mGlobalLock) {
            onAnimationFinished();
            // 超时后强制结束动画
            mService.mAnimator.addAfterPrepareSurfacesRunnable(() -> {
                cancelAnimation("timeout");
            });
        }
    };

    void goodToGo(@TransitionType int transit) {
        // 1. 收集所有动画目标的信息
        RemoteAnimationTarget[] targets = createAnimationTargets();
        RemoteAnimationTarget[] wallpaperTargets = createWallpaperTargets();
        
        // 2. 启动超时计时
        mHandler.postDelayed(mTimeoutRunnable, TIMEOUT_MS);
        
        // 3. 通过 Binder 回调通知 App 开始动画
        mRemoteAnimationAdapter.getRunner().onAnimationStart(
                transit,
                targets,
                wallpaperTargets,
                new IRemoteAnimationFinishedCallback.Stub() {
                    @Override
                    public void onAnimationFinished() {
                        // App 通知动画完成
                        RemoteAnimationController.this.onAnimationFinished();
                    }
                });
    }

    private void onAnimationFinished() {
        // 取消超时
        mHandler.removeCallbacks(mTimeoutRunnable);
        
        // 清理所有 Leash
        for (RemoteAnimationRecord record : mPendingAnimations) {
            record.mAdapter.mCapturedFinishCallback
                    .onAnimationFinished(record.mAdapter.mAnimationType,
                            record.mAdapter);
        }
        mPendingAnimations.clear();
    }
}
```

### 4.4 RemoteAnimation 的稳定性风险

| 风险场景 | 触发条件 | 后果 | 防护机制 |
|:---|:---|:---|:---|
| App 未调用 `finishCallback` | App Crash 或逻辑 Bug | 动画永远不结束 → 窗口卡在中间状态 | 2 秒超时强制结束 |
| App Crash 期间持有 Leash | App 进程被杀 | Leash SurfaceControl 泄漏 | Binder DeathRecipient 清理 |
| `onAnimationStart` Binder 调用失败 | App 进程冻结或 Binder 繁忙 | 转场动画无法启动 | 降级为默认窗口动画 |
| App 动画执行过慢 | 动画逻辑复杂或主线程阻塞 | 转场卡顿、帧率低 | 无自动防护，依赖 App 质量 |
| App 操作非法 SurfaceControl | App 操作了非 Leash 的 Surface | 窗口状态被破坏 | SurfaceControl 权限校验 |

```java
// Binder DeathRecipient 防护：App 进程死亡时清理
// RemoteAnimationController.java
mRemoteAnimationAdapter.getRunner().asBinder().linkToDeath(() -> {
    synchronized (mService.mGlobalLock) {
        cancelAnimation("runner died");
    }
}, 0);
```

> **稳定性架构师视角：** RemoteAnimation 是一把双刃剑——它给了 App 极大的动画自由度，但也将稳定性风险引入了 App 侧。Launcher 应用是 RemoteAnimation 的最大用户（App 切换动画、Recents 界面），如果 Launcher 的动画逻辑有 Bug，会影响所有 Activity 的转场体验。2 秒超时虽然提供了兜底保护，但超时后的强制结束会导致画面突变——用户会看到动画突然跳到终态，体验非常差。监控建议：对 RemoteAnimation 超时事件进行埋点统计，超时率超过 1% 应告警排查 Launcher 代码。

## 5. 窗口内动画与 Choreographer

### 5.1 View 动画与属性动画

除了 WMS 侧的窗口动画，App 内部的动画（View 动画、属性动画）也会影响窗口的显示性能。这类动画在 App 主线程执行，通过 Choreographer 的 VSYNC 回调驱动。

```
┌──────────────────────────────────────────────────────────────────────┐
│ App 主线程 Choreographer.doFrame() 的回调顺序                         │
│                                                                      │
│ VSYNC 信号到达                                                        │
│    ↓                                                                  │
│ ① CALLBACK_INPUT         → 处理输入事件（InputEventReceiver）         │
│    ↓                                                                  │
│ ② CALLBACK_ANIMATION     → 执行属性动画 (ValueAnimator/ObjectAnimator)│
│    ↓                                                                  │
│ ③ CALLBACK_INSETS_ANIMATION → 执行 Insets 动画（IME 弹出/收起动画）   │
│    ↓                                                                  │
│ ④ CALLBACK_TRAVERSAL     → ViewRootImpl.doTraversal()                 │
│    │                       → performTraversals()                      │
│    │                         → measure → layout → draw                │
│    ↓                                                                  │
│ ⑤ CALLBACK_COMMIT        → 提交帧（reportDrawFinished 等）           │
│                                                                      │
│ 总耗时必须 < 1 VSYNC 周期 (16.6ms @60Hz / 8.3ms @120Hz)             │
└──────────────────────────────────────────────────────────────────────┘
```

### 5.2 Choreographer 与动画的关系

> 源码：`frameworks/base/core/java/android/view/Choreographer.java`

```java
// frameworks/base/core/java/android/view/Choreographer.java（简化）
public final class Choreographer {
    // 五种回调类型，按优先级排序执行
    public static final int CALLBACK_INPUT = 0;
    public static final int CALLBACK_ANIMATION = 1;
    public static final int CALLBACK_INSETS_ANIMATION = 2;
    public static final int CALLBACK_TRAVERSAL = 3;
    public static final int CALLBACK_COMMIT = 4;

    void doFrame(long frameTimeNanos, int frame,
            DisplayEventReceiver.VsyncEventData vsyncEventData) {
        try {
            Trace.traceBegin(Trace.TRACE_TAG_VIEW, "Choreographer#doFrame");
            
            mFrameInfo.markInputHandlingStart();
            doCallbacks(CALLBACK_INPUT, frameTimeNanos, frame);
            
            mFrameInfo.markAnimationsStart();
            doCallbacks(CALLBACK_ANIMATION, frameTimeNanos, frame);
            
            mFrameInfo.markInsetsAnimationsStart();
            doCallbacks(CALLBACK_INSETS_ANIMATION, frameTimeNanos, frame);
            
            mFrameInfo.markPerformTraversalsStart();
            doCallbacks(CALLBACK_TRAVERSAL, frameTimeNanos, frame);
            
            doCallbacks(CALLBACK_COMMIT, frameTimeNanos, frame);
        } finally {
            Trace.traceEnd(Trace.TRACE_TAG_VIEW);
        }
    }
}
```

### 5.3 AnimationHandler — 属性动画的调度中心

> 源码：`frameworks/base/core/java/android/animation/AnimationHandler.java`

```java
// frameworks/base/core/java/android/animation/AnimationHandler.java（简化）
public class AnimationHandler {
    // 每个线程一个 AnimationHandler（ThreadLocal）
    private static final ThreadLocal<AnimationHandler> sAnimatorHandler =
            new ThreadLocal<>();
    
    // 活跃的动画列表
    private final ArrayList<AnimationFrameCallback> mAnimationCallbacks =
            new ArrayList<>();

    // Choreographer 回调——在 CALLBACK_ANIMATION 阶段执行
    private final Choreographer.FrameCallback mFrameCallback = frameTimeNanos -> {
        doAnimationFrame(getProvider().getFrameTime());
        if (mAnimationCallbacks.size() > 0) {
            // 还有动画 → 注册下一帧回调
            getProvider().postFrameCallback(this);
        }
    };

    private void doAnimationFrame(long frameTime) {
        for (int i = 0; i < mAnimationCallbacks.size(); i++) {
            final AnimationFrameCallback callback = mAnimationCallbacks.get(i);
            // 调用每个 ValueAnimator 的 doAnimationFrame
            callback.doAnimationFrame(frameTime);
        }
    }
}
```

### 5.4 动画过多导致的性能问题

当 App 同时运行大量属性动画时，`CALLBACK_ANIMATION` 阶段的执行时间会显著增加，挤压 `CALLBACK_TRAVERSAL`（布局和绘制）的时间预算，导致掉帧：

```
正常帧 (16.6ms 预算):
  ┌─INPUT─┬──ANIMATION──┬───TRAVERSAL────┬COMMIT┐
  │ 1ms   │    2ms      │     8ms        │ 1ms  │  总计 12ms ✓
  └───────┴─────────────┴────────────────┴──────┘

动画过多的帧 (超出预算):
  ┌─INPUT─┬──────────ANIMATION──────────┬───TRAVERSAL────┬COMMIT┐
  │ 1ms   │          12ms              │     8ms        │ 1ms  │  总计 22ms ✗ 掉帧!
  └───────┴────────────────────────────┴────────────────┴──────┘
                  ↑ 太多动画同时执行
```

```java
// frameworks/base/core/java/android/animation/ValueAnimator.java（简化）
public class ValueAnimator extends Animator implements AnimationHandler.AnimationFrameCallback {
    
    // 动画帧回调
    public final boolean doAnimationFrame(long frameTime) {
        // 计算当前进度
        final long currentTime = Math.max(frameTime, mStartTime);
        boolean finished = animateBasedOnTime(currentTime);
        
        if (finished) {
            endAnimation();
        }
        return finished;
    }
    
    // 通知值变化 → 触发 View.invalidate() → 下一帧重绘
    void animateValue(float fraction) {
        fraction = mInterpolator.getInterpolation(fraction);
        for (int i = 0; i < mUpdateListeners.size(); i++) {
            mUpdateListeners.get(i).onAnimationUpdate(this);
        }
    }
}
```

> **稳定性架构师视角：** 属性动画泄漏是 App 内存泄漏的常见原因之一。`ValueAnimator` 通过 `AnimationHandler` 注册到 Choreographer，如果动画未被 `cancel()` 或 `end()`，AnimationHandler 会一直持有对 ValueAnimator 的引用，而 ValueAnimator 的 `UpdateListener` 通常持有 Activity 的引用，形成泄漏链：`AnimationHandler(ThreadLocal) → ValueAnimator → UpdateListener → Activity`。更隐蔽的是，无限循环动画（`setRepeatCount(INFINITE)`）如果在 Activity `onDestroy()` 时未 cancel，会永远占用 Choreographer 的回调槽位，持续消耗 CPU。

## 6. Shell Transition（Android 13+）

### 6.1 从 AppTransition 到 Shell Transition 的演进

Android 13 引入了 Shell Transition 框架，逐步替代经典的 `AppTransition` 机制。核心变化是将转场动画的管理从 `system_server`（WMS 内部）迁移到 SystemUI 的 Shell 模块中：

```
经典 AppTransition (Android 12 及之前):
┌───────────────────────────────────────────┐
│ system_server (WMS)                       │
│   AppTransition + AppTransitionController │
│   → 决定转场类型                          │
│   → 加载动画资源                          │
│   → SurfaceAnimator 执行动画              │
│   → 动画完成后清理状态                    │
│   (动画逻辑与窗口管理耦合在一起)          │
└───────────────────────────────────────────┘

Shell Transition (Android 13+):
┌───────────────────────────────────┐     ┌──────────────────────────────┐
│ system_server (WMS Core)          │     │ SystemUI (Shell 模块)         │
│   TransitionController            │     │   Transitions.java            │
│   → 收集 transition 参与者        │     │   → 接收 transition 信息       │
│   → 创建 TransitionInfo           │────▶│   → 选择/创建动画              │
│   → 等待 Shell 完成动画           │     │   → 操作 SurfaceControl 动画   │
│   → Shell 通知完成后清理          │◀────│   → 动画完成后通知 WMS Core    │
│   (只做协调，不做动画)            │     │   (专注于动画效果)             │
└───────────────────────────────────┘     └──────────────────────────────┘
```

### 6.2 Shell Transition 的设计优势

| 维度 | 经典 AppTransition | Shell Transition |
|:---|:---|:---|
| 动画代码位置 | `system_server` 内部 | SystemUI Shell 模块 |
| 解耦程度 | 动画逻辑与 WMS 窗口管理混合 | 动画逻辑独立于 WMS |
| 可定制性 | 修改动画需要修改 framework 代码 | OEM 可在 SystemUI 层定制 |
| 调试便利性 | 需要 root / userdebug 版本调试 system_server | 可通过 SystemUI 的日志/工具调试 |
| 动画卡死影响 | 影响 WMS 的 mGlobalLock，可能级联阻塞 | Shell 侧卡死不影响 WMS 核心逻辑 |
| 超时处理 | AppTransition 超时 → 强制 goodToGo | Shell Transition 超时 → 回退到默认动画 |

### 6.4 Shell 侧的 Transitions

> 源码位于 SystemUI 模块：`frameworks/base/libs/WindowManager/Shell/src/com/android/wm/shell/transition/Transitions.java`

Shell 侧的 `Transitions` 类负责接收 WMS 发来的转场请求，选择合适的 `TransitionHandler` 来处理动画：

```java
// frameworks/base/libs/WindowManager/Shell/src/.../transition/Transitions.java（简化）
public class Transitions implements TransitionPlayerProto {
    private final ArrayList<TransitionHandler> mHandlers = new ArrayList<>();
    
    // WMS 调用此方法请求播放转场
    @Override
    public void requestStartTransition(IBinder transitionToken,
            TransitionInfo info) {
        // 找到合适的 handler
        for (TransitionHandler handler : mHandlers) {
            if (handler.handleRequest(transitionToken, info)) {
                handler.startAnimation(transitionToken, info,
                        t -> onTransitionFinished(transitionToken, t));
                return;
            }
        }
        // 无 handler 处理 → 使用默认动画
        playDefaultAnimation(transitionToken, info);
    }
}
```

> **稳定性架构师视角：** Shell Transition 的最大稳定性收益是**动画卡死不再直接影响 WMS 核心逻辑**。在经典模式下，`AppTransitionController.handleAppTransitionReady()` 在 `mGlobalLock` 内执行，动画选择和启动逻辑的任何延迟都会延长锁持有时间。Shell Transition 将这部分逻辑移到了 SystemUI 进程，即使 Shell 动画处理卡住，WMS 的 `mGlobalLock` 不会被阻塞。但需要注意，Shell Transition 引入了跨进程通信的开销——WMS 与 Shell 之间的 Binder 调用延迟如果过高，也会导致转场启动变慢。Android 14+ 中 Shell Transition 已成为默认路径，经典 AppTransition 作为 fallback 保留。

## 7. 稳定性风险总结

### 7.1 动画相关稳定性风险地图

| 风险类型 | 触发条件 | 表现 | 影响等级 | 排查入口 |
|:---|:---|:---|:---|:---|
| **Leash 泄漏** | `onAnimationFinished` 未被调用（异常中断、回调丢失） | SurfaceFlinger Layer 数量持续增长，显存上升 | 系统级 | `dumpsys SurfaceFlinger --list` 搜索 `animation-leash` |
| **转场 goodToGo 阻塞** | 新 Activity 首帧绘制慢，allDrawn 迟迟不为 true | 黑屏/白屏等待，超时后闪烁 | 用户感知 | `dumpsys window` 检查 `allDrawn`/`startingDisplayed` |
| **转场焦点空窗 ANR** | 转场期间旧窗口失焦、新窗口未添加，Key 事件到达 | Input ANR: "no focused window" | ANR | `dumpsys input` 检查 FocusedWindow |
| **RemoteAnimation 超时** | App 未在 2s 内调用 finishCallback | 动画被强制结束，画面突变 | 用户感知 | `dumpsys window` 搜索 "remote animation timeout" |
| **RemoteAnimation App Crash** | 持有 Leash 的 App 进程崩溃 | Leash 可能泄漏（依赖 DeathRecipient 清理） | 系统级 | Logcat 检查 App Crash + `dumpsys SurfaceFlinger` |
| **WindowAnimator 动画卡住** | `animate()` 中异常导致 `scheduleAnimation` 不再调用 | 所有窗口动画冻结，画面停在中间态 | 用户感知 | Systrace 检查 `wmAnimate` 是否停止 |
| **SurfaceAnimationThread 阻塞** | Transaction.apply() 卡住（SurfaceFlinger 无响应） | 动画变换不生效，画面冻结 | 系统级 | traces.txt 检查 `android.anim.lf` 线程状态 |
| **过多属性动画导致掉帧** | App 同时运行大量 ValueAnimator/ObjectAnimator | UI 卡顿，帧率下降 | 用户感知 | Systrace 检查 `CALLBACK_ANIMATION` 耗时 |
| **属性动画泄漏** | 无限循环动画未在 Activity onDestroy 时 cancel | Activity 内存泄漏，CPU 持续消耗 | 内存泄漏 | LeakCanary / MAT 检查 AnimationHandler 引用链 |
| **mGlobalLock 竞争加剧** | 动画帧计算在 AnimationThread 上持有 mGlobalLock | 其他线程的窗口操作被阻塞 | 性能 | traces.txt 检查 mGlobalLock 竞争 |
| **Shell Transition 超时** | Shell 侧动画处理时间过长 | 转场延迟，WMS 回退到默认动画 | 用户感知 | SystemUI 日志 + WMS 日志 |
| **动画期间 Surface 提前释放** | 窗口在动画播放中被 removeWindow 移除 | 动画操作已释放的 SurfaceControl → Crash 或画面残留 | Crash/异常 | Logcat 搜索 "Surface has been released" |

### 7.2 dumpsys 输出模式解读

**模式 1：正常的转场状态**

```
AppTransition:
  mAppTransitionState=APP_STATE_IDLE
  mNextAppTransitionType=TRANSIT_UNSET
```

无转场进行中，状态正常。

**模式 2：转场被阻塞（高风险）**

```
AppTransition:
  mAppTransitionState=APP_STATE_READY
  mNextAppTransitionType=TRANSIT_OPEN
  # allDrawn=false for com.example.app/.SlowActivity
```

转场类型已确定，但 `goodToGo()` 未通过——新 Activity 的窗口还没绘制完成。如果此状态持续超过 5 秒，WMS 将强制启动转场。

**模式 3：Leash 泄漏（SurfaceFlinger 层）**

```bash
$ adb shell dumpsys SurfaceFlinger --list | grep "animation-leash"
  Task=5#0 - animation-leash
  ActivityRecord{abc1234}#0 - animation-leash
```

正常情况下不应有残留的 `animation-leash`。如果 dumpsys 中持续出现，说明有 Leash 泄漏。

## 8. 实战案例

### Case 1：Activity 转场黑屏 — goodToGo() 永远不满足

**现象**

用户从 App 的列表页点击进入详情页（`DetailActivity`），屏幕黑了约 3 秒后详情页才出现。在黑屏期间，手机不是无响应——状态栏和导航栏正常，但 App 区域是纯黑色。

**排查过程**

**Step 1：确认转场状态**

```bash
$ adb shell dumpsys window | grep -A 5 "AppTransition"
AppTransition:
  mAppTransitionState=APP_STATE_READY
  mNextAppTransitionType=TRANSIT_OPEN
  mNextAppTransitionQualifier=0
```

转场处于 `READY` 状态但未进入 `RUNNING`——`goodToGo()` 没有通过。

**Step 2：检查新 Activity 的窗口状态**

```bash
$ adb shell dumpsys window windows | grep -A 10 "DetailActivity"
  Window #3 Window{abc1234 u0 com.example.app/.DetailActivity}:
    mHasSurface=true
    isReadyForDisplay()=false
    allDrawn=false
    startingDisplayed=false
    mViewVisibility=0
```

关键发现：`allDrawn=false`，`startingDisplayed=false`。详情页的窗口虽然已经 `addWindow`（`mHasSurface=true`），但还没完成首帧绘制（`allDrawn=false`），也没有 Starting Window（`startingDisplayed=false`）。

**Step 3：分析首帧绘制为何慢**

用 Systrace 捕获启动过程：

```
T=0ms      startActivity(DetailActivity)
T=5ms      AMS: ActivityRecord 创建, setFocusedApp(DetailActivity)
T=15ms     AppTransition: setAppTransition(TRANSIT_OPEN)
T=20ms     DetailActivity.onCreate() 开始
T=25ms     → setContentView(R.layout.detail_activity)
T=30ms     → DetailViewModel.loadData() [同步 IO!]
             → Room Database 查询 (耗时 2500ms)
T=2530ms   → loadData 返回，UI 数据绑定
T=2535ms   DetailActivity.onResume()
T=2540ms   ViewRootImpl.setView() → WMS.addWindow()
T=2600ms   relayoutWindow() → Surface 创建
T=2700ms   performTraversals() → measure → layout → draw
T=2800ms   finishDrawingWindow() → allDrawn = true
T=2810ms   AppTransition: goodToGo() 通过!
T=2820ms   转场动画开始播放
T=3000ms   动画结束，详情页完全显示
```

**根因**

`DetailActivity.onCreate()` 中同步执行了 Room 数据库查询（`loadData()`），耗时 2500ms。在此期间：

1. AppTransition 已进入 `READY` 状态，等待 `goodToGo()`
2. 但 `allDrawn` 为 false（Activity 还没完成 `onCreate` → 没有 `addView` → 没有绘制）
3. 同时没有 Starting Window（App 未配置 `windowSplashScreenAnimatedIcon` 等 SplashScreen 属性，且该 Activity 的 Theme 也没有设置 `windowBackground`，导致系统判定不需要 Starting Window）
4. 旧窗口（列表页）在转场开始后被设为不可见（`closing` 状态）
5. 新窗口还没显示 → 用户看到的就是黑屏

```
旧窗口（列表页）                  新窗口（详情页）
  visible → closing               不存在 → addWindow → 未绘制
  (被 WMS 标记为正在关闭)         (allDrawn=false, 还在 onCreate 中)
       ↓                                ↓
  不再显示 (hide)                  未开始显示 (show 被推迟)
       ↓                                ↓
  ════════ 黑屏区间 (2.5 秒) ════════════
```

**修复方案**

```java
// 方案 1：将耗时操作移到异步（根本修复）
@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.detail_activity);
    
    // 先显示骨架屏/Loading 状态，让 performTraversals 尽快完成
    showLoadingState();
    
    // 异步加载数据
    viewModel.loadDataAsync().observe(this, data -> {
        hideLoadingState();
        bindData(data);
        // 可选：通知系统内容已完全就绪
        reportFullyDrawn();
    });
}

// 方案 2：配置 SplashScreen 提供过渡画面（缓解方案）
// res/values/themes.xml
// <item name="android:windowSplashScreenBackground">@color/white</item>
// <item name="android:windowSplashScreenAnimatedIcon">@drawable/app_icon</item>
```

> **稳定性架构师视角：** 这个案例是 `goodToGo()` 阻塞的典型模式——新 Activity 的 `onCreate` 耗时过长导致首帧绘制延迟。`allDrawn` 和 `startingDisplayed` 两个标志都为 false 时，WMS 不会启动转场动画，黑屏不可避免。防御策略有三层：① App 端将耗时操作异步化，确保 `onCreate → onResume → 首帧绘制` 的路径尽可能短（< 500ms）；② 配置 Theme 的 `windowBackground` 或 SplashScreen，确保系统创建 Starting Window；③ 系统层面的 5 秒超时保护，虽然是兜底但不应依赖。

### Case 2：ObjectAnimator 泄漏导致 Activity 内存泄漏

**现象**

LeakCanary 报告 `AnimationActivity` 存在内存泄漏，retained heap 约 12MB。GC Root 路径如下：

```
┌─ GC Root: Thread (main)
│   ├─ Looper.mQueue
│   │   ├─ MessageQueue.mMessages
│   │   │   ├─ Message.callback (Choreographer$FrameDisplayEventReceiver)
│   │   │   │   ├─ Choreographer.mCallbackQueues[1]  (CALLBACK_ANIMATION)
│   │   │   │   │   ├─ AnimationHandler.mAnimationCallbacks
│   │   │   │   │   │   ├─ ObjectAnimator
│   │   │   │   │   │   │   ├─ ObjectAnimator.mTarget → View (ImageView)
│   │   │   │   │   │   │   │   └─ View.mContext → AnimationActivity (LEAKED)
│   │   │   │   │   │   │   │       ├─ AnimationActivity.mBitmap (8MB)
│   │   │   │   │   │   │   │       ├─ AnimationActivity.mDataList (3.5MB)
│   │   │   │   │   │   │   │       └─ ... (total retained: 12MB)
```

**分析思路**

1. GC Root 路径显示 `AnimationHandler.mAnimationCallbacks` 持有 `ObjectAnimator` 的引用
2. `ObjectAnimator.mTarget` 指向一个 `ImageView`
3. `ImageView.mContext` 指向已泄漏的 `AnimationActivity`
4. `AnimationHandler` 是 ThreadLocal 的，绑定在主线程上，不会被 GC

**问题代码**

```java
// AnimationActivity.java（问题代码）
public class AnimationActivity extends Activity {
    private ImageView mLogoView;
    private Bitmap mBitmap;  // 8MB

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_animation);
        mLogoView = findViewById(R.id.logo);
        mBitmap = BitmapFactory.decodeResource(getResources(), R.drawable.large_image);
        mLogoView.setImageBitmap(mBitmap);

        // 创建无限旋转动画
        ObjectAnimator rotateAnimator = ObjectAnimator.ofFloat(
                mLogoView, "rotation", 0f, 360f);
        rotateAnimator.setDuration(2000);
        rotateAnimator.setRepeatCount(ValueAnimator.INFINITE);
        rotateAnimator.setInterpolator(new LinearInterpolator());
        rotateAnimator.start();
        // ⚠️ 未在 onDestroy 中 cancel！
    }

    // onDestroy 中没有取消动画
}
```

**泄漏链路**

```
Choreographer (主线程, 永不释放)
  → CALLBACK_ANIMATION 回调队列
    → AnimationHandler (ThreadLocal, 永不释放)
      → mAnimationCallbacks
        → ObjectAnimator (repeatCount=INFINITE, 永不自动结束)
          → mTarget = ImageView
            → mContext = AnimationActivity  ← LEAKED
              → mBitmap (8MB)
              → mDataList (3.5MB)
```

关键点：`ObjectAnimator.setRepeatCount(INFINITE)` 意味着动画永远不会自动结束。`AnimationHandler` 会一直持有它的引用，每帧都调用 `doAnimationFrame`。即使 Activity 已经 `onDestroy()`，动画仍在运行，通过 `mTarget → View → Context` 的引用链阻止了 Activity 的 GC。

**修复方案**

```java
// 修复后
public class AnimationActivity extends Activity {
    private ImageView mLogoView;
    private ObjectAnimator mRotateAnimator;

    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_animation);
        mLogoView = findViewById(R.id.logo);

        mRotateAnimator = ObjectAnimator.ofFloat(
                mLogoView, "rotation", 0f, 360f);
        mRotateAnimator.setDuration(2000);
        mRotateAnimator.setRepeatCount(ValueAnimator.INFINITE);
        mRotateAnimator.setInterpolator(new LinearInterpolator());
        mRotateAnimator.start();
    }

    @Override
    protected void onDestroy() {
        super.onDestroy();
        if (mRotateAnimator != null) {
            mRotateAnimator.cancel();
            mRotateAnimator = null;
        }
    }
}
```

更规范的做法——使用 Lifecycle 自动管理：

```java
public class AnimationActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_animation);
        ImageView logoView = findViewById(R.id.logo);

        ObjectAnimator rotateAnimator = ObjectAnimator.ofFloat(
                logoView, "rotation", 0f, 360f);
        rotateAnimator.setDuration(2000);
        rotateAnimator.setRepeatCount(ValueAnimator.INFINITE);
        rotateAnimator.setInterpolator(new LinearInterpolator());

        getLifecycle().addObserver(new DefaultLifecycleObserver() {
            @Override
            public void onResume(@NonNull LifecycleOwner owner) {
                rotateAnimator.start();
            }
            @Override
            public void onPause(@NonNull LifecycleOwner owner) {
                rotateAnimator.pause();
            }
            @Override
            public void onDestroy(@NonNull LifecycleOwner owner) {
                rotateAnimator.cancel();
            }
        });
    }
}
```

> **稳定性架构师视角：** 属性动画泄漏是 Android App 中 Top 10 的内存泄漏类型。根因是 `INFINITE` 重复的动画不会自动从 `AnimationHandler` 中注销。防御策略：① 所有在 `onCreate`/`onResume` 中启动的动画必须在 `onDestroy`/`onPause` 中 cancel；② 使用 `Lifecycle` 感知组件自动管理动画生命周期；③ 在代码审查中对 `setRepeatCount(INFINITE)` 强制要求配套的 cancel 逻辑；④ 对 LeakCanary 报告中 GC Root 路径包含 `AnimationHandler` 的泄漏重点治理。框架层面可以考虑在 `Activity.onDestroy()` 时自动 cancel 所有以该 Activity View 为 target 的 Animator，但目前 AOSP 未实现此兜底逻辑。

## 总结

窗口动画是 WMS 中直接影响用户体验的模块。对于稳定性架构师，以下是必须掌握的关键要点：

1. **窗口动画操作 SurfaceControl，不触发 App 端重绘。** 窗口动画（WindowAnimator / SurfaceAnimator）在 `system_server` 的动画线程上执行，通过 SurfaceControl.Transaction 直接变换 SurfaceFlinger 的 Layer。即使 App 主线程阻塞，窗口动画仍然流畅。但 Leash 泄漏、动画线程阻塞会在系统层面造成问题。

2. **AppTransition 的 goodToGo() 是转场黑屏的关键判定点。** 如果新 Activity 的首帧绘制慢且没有 Starting Window，`goodToGo()` 无法通过，转场动画被阻塞，用户看到黑屏。优化首帧速度（异步化耗时操作）和配置 SplashScreen 是两个防御方向。

3. **转场期间的焦点空窗期是 Input ANR 的高发区。** 旧窗口失焦到新窗口获焦之间的时间窗内，`FocusedWindow` 为 null。任何 Key 事件到达都会触发 ANR 倒计时。缩短 Activity 启动时间是根本解决方案。

4. **RemoteAnimation 将稳定性风险引入 App 侧。** Launcher 和 SystemUI 通过 RemoteAnimation 自定义转场效果，但 2 秒超时保护是兜底。App Crash 或未调用 finishCallback 都会导致动画异常。

5. **属性动画泄漏是 App 内存泄漏的常见原因。** `setRepeatCount(INFINITE)` 的动画必须在 Activity 销毁时 cancel，否则 `AnimationHandler → ObjectAnimator → View → Activity` 的引用链会阻止 GC。

6. **Shell Transition（Android 13+）改善了动画与 WMS 核心逻辑的解耦。** 动画处理移到 SystemUI Shell 模块后，动画卡死不再直接阻塞 `mGlobalLock`，降低了 system_server Watchdog 超时的风险。

**排查路径速查：**

```
转场黑屏/白屏?
  → dumpsys window → 检查 AppTransition 状态
  → 确认 allDrawn / startingDisplayed
  → Systrace 检查 Activity 启动到首帧绘制的耗时

动画冻结?
  → traces.txt → 检查 "android.anim" / "android.anim.lf" 线程状态
  → Systrace → 搜索 wmAnimate 是否停止
  → dumpsys SurfaceFlinger → 检查 animation-leash 残留

转场期间 ANR?
  → dumpsys input → FocusedApplication vs FocusedWindow
  → 确认是否存在焦点空窗期
  → 分析新 Activity 的启动耗时

RemoteAnimation 异常?
  → Logcat → 搜索 "remote animation timeout" / "runner died"
  → dumpsys window → 检查 RemoteAnimationController 状态

掉帧/卡顿?
  → Systrace → 检查 Choreographer doFrame 各阶段耗时
  → 重点关注 CALLBACK_ANIMATION 阶段
  → 检查同时运行的属性动画数量

内存泄漏?
  → LeakCanary → GC Root 路径包含 AnimationHandler?
  → 检查 ObjectAnimator.repeatCount == INFINITE
  → 确认 onDestroy 中是否 cancel 了所有动画
```
