## 2. 7 大监控维度

### 2.1 ANR 监控

**核心指标**：
- ANR 率（次 / 千次启动）
- ANR 类型分布（Input / Broadcast / Service / Provider）
- ANR trace 内容（含 ART 17 内部状态）

**核心工具**：
- `adb shell am trace-ipc-start` + `adb shell am dumpheap`
- `adb pull /data/anr/`
- `traces.txt` 解析工具

**ART 17 增强**：ANR trace 增加 ART 内部状态（GC / JIT / ClassLoader / JNI）。

详见 [02-ANR_Trace 完整链路 v2](../06-信号与ANR-Trace/02-ANR_Trace完整链路-v2.md)。

### 2.2 Crash 监控

**核心指标**：
- Crash 率（次 / 千次启动）
- Crash 类型分布（Java / Native / ANR / Tombstone）
- Crash 现场（堆栈 + ART 17 内部状态）

**核心工具**：
- `adb logcat -b crash`
- `adb shell ls /data/tombstones/`
- Firebase Crashlytics / 友盟 / Bugly

**ART 17 增强**：Crash 快速路径让 crash dump 速度 +200%。

### 2.3 性能监控

**核心指标**：
- 冷启动 / 热启动 / Warm 启动时间
- 帧率 / 卡顿率
- 主线程阻塞时间

**核心工具**：
- Jetpack Macrobenchmark
- `adb shell dumpsys gfxinfo`
- `adb shell atrace`
- `simpleperf`

**ART 17 增强**：AOSP 17 启动优化让冷启动 -30-40%。

### 2.4 GC 监控

**核心指标**：
- GC 频率（次 / 分钟）
- GC 暂停时间
- GC 原因分布（9 种 GcCause）
- 堆内存使用

**核心工具**：
- `adb logcat -s art:V`
- `adb shell dumpsys meminfo`
- `adb shell setprop dalvik.vm.dex2oat-threads 4`（调节 GC 线程）

**ART 17 增强**：GenCC + kSoftThresholdPercent=30% 软阈值。

详见 [10-ART17分代GC强化专章 v2](../03-GC系统/10-ART17分代GC强化专章-v2.md)。

### 2.5 内存监控

**核心指标**：
- Java 堆使用 / 峰值
- Native 堆使用
- 显存（GPU）
- Leak（LeakCanary）

**核心工具**：
- `adb shell dumpsys meminfo`
- `adb shell procrank`
- LeakCanary
- Android Studio Profiler

**ART 17 增强**：Linux 6.18 sheaves 让 Native 堆内存占用降低 15-20%。

### 2.6 启动监控

**核心指标**：
- 冷启动时间（TotalTime / ThisTime / WaitTime）
- 第一行 Java 代码时间
- Activity.onCreate 时间

**核心工具**：
- `adb shell am start -W`
- Jetpack Macrobenchmark
- `adb shell atrace`

**ART 17 增强**：启动期 Lazy Load + Class 去重 + Quickened Bytecode 综合优化。

详见 [01-从 app_process 到第一行 Java 代码 v2](../07-启动流程/01-从app_process到第一行Java代码-v2.md)。

### 2.7 监控决策树

```
线上问题出现
  ↓
是什么问题？
  ↓
├─ App 卡死 / 无响应
│   └─ ANR 监控（traces.txt）
│
├─ App 闪退
│   └─ Crash 监控（tombstone / logcat crash）
│
├─ App 慢 / 卡顿
│   └─ 性能监控（gfxinfo / simpleperf / Macrobenchmark）
│
├─ App OOM
│   └─ 内存监控（dumpsys meminfo / LeakCanary）
│
└─ App 冷启动慢
    └─ 启动监控（am start -W / Macrobenchmark）
```

## 3. ART 17 内部状态监控

### 3.1 监控 ART 内部状态

AOSP 17 引入**ART 内部状态监控**——通过 `adb shell dumpsys art` 或新 API：

```bash
# AOSP 17 新增
adb shell dumpsys art
# 输出 ART 内部状态：GC / JIT / AOT / ClassLoader / JNI

# ART 17 输出示例
=== ART 17 内部状态 ===
GC state: Concurrent
GC strategy: GenCC
GC threshold: soft=30%, hard=50%
JIT queue: 3 methods pending
JIT code cache: 5MB / 256MB
AOT profile: hot
ClassLoader: 1 active, 3 cached
JNI refs: Local 25 / Global 3
```

### 3.2 实战应用

**场景 1：GC 频繁**
```
JIT queue: 50 methods pending
GC state: Concurrent
GC frequency: 30/min
→ 推测：JIT 编译压力大，触发频繁 GC
→ 优化：调高 JIT 阈值
```

**场景 2：JNI 内存泄漏**
```
JNI refs: Global 50000 / 50000
→ 推测：Global Reference 泄漏
→ 排查：JNI 代码
```

**场景 3：ClassLoader 异常**
```
ClassLoader: 50 active, 0 cached
→ 推测：插件化框架 ClassLoader 泄漏
→ 排查：插件化代码
```

## 8. 实战案例：ART 17 内部状态监控实战

**现象**：某 App 偶发卡顿，传统监控（gfxinfo）无异常。

**环境**：AOSP 17.0.0_r1（API 37）/ Pixel 8。

### 步骤 1：传统监控

```bash
adb shell dumpsys gfxinfo com.example.app
# 帧率正常（55-60fps），但用户投诉卡顿
```

### 步骤 2：ART 17 内部状态监控

```bash
adb shell dumpsys art
# 输出：
=== ART 17 内部状态 ===
GC state: Concurrent
GC frequency: 30/min  ← 异常高
JIT queue: 0 methods
ClassLoader: 1 active
JNI refs: Local 25 / Global 3
```

**根因**：GC 频率 30/min 异常高（行业标准 < 5/min），**触发卡顿**。

### 步骤 3：定位 GC 频繁原因

```bash
adb logcat -d -s art:V | grep "GC"
# 输出大量 "Concurrent GC: 30MB freed in 5ms"
```

**根因**：App 持续分配短生命周期对象（30MB / 5ms），**触发 GenCC Young GC 频繁**。

### 步骤 4：优化

```java
// 优化前：循环内 new 对象
for (int i = 0; i < 1000; i++) {
    String s = new String("hello " + i);  // 1000 个短生命周期对象
}

// 优化后：复用对象
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.setLength(0);
    sb.append("hello ").append(i);
    // 复用 sb，避免分配
}
```

### 步骤 5：验证

```
┌──────────────────────────────────────┬───────────┬───────────┐
│ 指标                                  │ 修复前     │ 修复后     │
├──────────────────────────────────────┼───────────┼───────────┤
│ GC 频率                              │ 30/min    │ 2/min     │
│ 帧率（用户感知）                      │ 55-60fps  │ 60fps     │
│ 卡顿投诉（每周）                      │ 5 次      │ 0 次      │
│ ART 17 内部状态可观测性               │ ✓         │ ✓         │
└──────────────────────────────────────┴───────────┴───────────┘
```

**典型模式说明**：上述数据基于"循环内分配 + ART 17 GC 监控 + 复用对象"的典型场景。**具体数值因 App 复杂度、机型而异**。

## 9. 总结（架构师视角的 5 条 Takeaway）

1. **监控诊断是稳定性架构师的"工具箱"**——7 大维度（ANR / Crash / 性能 / GC / 内存 / 启动 / ART 内部）+ ART 17 增强。**ART 17 内部状态监控让"ART 内部阻塞"排查成为可能**。
2. **ANR / Crash 监控是基础**——traces.txt + tombstone 是核心数据。**ART 17 trace 增强 + Crash 快速路径让排查更高效**。详见 [02-ANR_Trace 完整链路 v2](../06-信号与ANR-Trace/02-ANR_Trace完整链路-v2.md)。
3. **性能 / GC / 内存监控是性能基线**——没有基线就没有"异常"的概念。**ART 17 GenCC + kSoftThresholdPercent 强化**详见 [10-ART17分代GC强化专章 v2](../03-GC系统/10-ART17分代GC强化专章-v2.md)。
4. **AI Agent 监控是 ART 17 新增维度**——AppFunctions 行为监控让 AI Agent 问题排查成为可能。**com.android.profiling APEX 让持续性能监控成为系统能力**。
5. **Linux 6.18 增强监控能力**——eBPF 加密签名 + pidfds 扩展 + sheaves 内存，**让 ART 17 监控工具在 Linux 层有更强支撑**。
