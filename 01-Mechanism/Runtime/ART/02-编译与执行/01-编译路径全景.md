## 1. 背景与定义：三种执行模式

### 1.1 一句话定义

**ART 采用"解释器 + JIT + AOT"三态切换模式**：方法首次调用用解释器（快启动），热点方法 JIT 编译（运行时优化），下次启动用 AOT 编译产物（已优化）+ Cloud Profile 下发热点（云端优化）。**ART 17（API 37+）在此基础上引入 static final 不可变硬约束 + 编译器 IR 增强 + Cloud Profile 增量下发**。

### 1.2 三种模式对比

| 模式 | 触发时机 | 性能 | 内存 | 存储 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **解释器** | 方法首次调用 | ★★ | ★ | 0 | 未编译 |
| **JIT** | 热度阈值（默认 10,000 次） | ★★★★ | ★★ | 0（运行时编译） | 热方法运行时 |
| **AOT** | 安装时 / 后台 / 下次启动 | ★★★★★ | ★★★★ | 10-100MB / App | 热方法编译产物 |

### 1.3 三态切换状态机

```
方法首次调用
  ↓
解释器执行（同时记录调用次数）
  ↓
达到 JIT 阈值（10,000 次，ART 17 可调至 5,000）
  ↓
JIT 编译（后台线程，OSR 替换入口）
  ↓
JIT 代码执行（Profile 记录）
  ↓
后台 dex2oat 编译 AOT（speed-profile 模式）
  ↓
下次启动 → 直接 AOT 代码（Cloud Profile 让首次安装就有 AOT）
```

**关键设计**：
- **混合模式** = 启动快 + 跑起来快 + 长期最优
- **Profile 跨版本累积**（Cloud Profile 让首次安装就有 AOT）
- **OSR** 让长循环方法能从解释器平滑切到 JIT
- **ART 17 新增**：static final 字段在 AOT 产物中直接编码为字面量，跳过运行时常量池查找（见 §7.1）

## 2. dex2oat 完整流程

### 2.1 dex2oat 是什么

**dex2oat（DEX to OAT）** 是 AOT 编译工具，把 .dex 文件编译成 .oat（OAT 文件包含编译后的本地代码 + 元数据）。在 AOSP 17 中，dex2oat 进一步拆分为 `dex2oat`（全量 AOT）和 `dexopt`（增量优化）两个二进制，`dexopt` 专用于后台 speed-profile 模式。

## 3. PGO 与 Baseline Profile

### 3.1 什么是 PGO

**PGO（Profile-Guided Optimization）** 是基于运行时 Profile 数据的优化技术：

1. **收集阶段**：在用户设备上运行 → 收集热方法 / 类 / 启动路径
2. **Profile 文件**：把热数据保存为 `.prof` 文件
3. **优化编译**：dex2oat 根据 Profile 优先编译热方法
4. **云端下发**：Cloud Profile 跨用户共享（Google Play）

### 3.2 Baseline Profile 机制

```
┌────────────────────────────────────────────────────────────────┐
│ Baseline Profile 工作机制（AOSP 17 增强）                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. 开发阶段                                                    │
│     ├─ 在测试设备上跑应用                                        │
│     └─ 用 macrobenchmark / 手动测试 生成 Baseline Profile       │
│                                                                │
│  2. 上传 Play Store                                              │
│     └─ Baseline Profile 文件随 APK 上传                        │
│                                                                │
│  3. 用户安装                                                     │
│     └─ Google Play 把 Baseline Profile 下发给设备                │
│     └─ AOSP 17 增强：Cloud Profile 增量下发（见 §3.4）            │
│                                                                │
│  4. 首次启动                                                     │
│     └─ ART 使用 Baseline Profile 中的热点方法做 AOT 编译         │
```

### 3.3 Baseline Profile 效果

**Google 官方数据（AOSP 17，Pixel 8）**：

| 指标 | 无 Baseline | 有 Baseline | 提升 |
| :--- | :--- | :--- | :--- |

- 冷启动 1800ms → 750ms（-58%）
- 首帧绘制 1500ms → 600ms（-60%）

## 4. 蹦床机制（Trampolines）

### 4.1 什么是蹦床

**蹦床（Trampoline）** 是 ART 的 Native 入口桩代码，用于在解释器模式、JIT 模式、AOT 模式之间无缝切换。

### 4.2 art_quick_* 系列

**蹦床函数族（`art/runtime/arch/`）**：

```cpp
// arm64 蹦床实现（AOSP 17 强化 PAC/BTI 集成）
extern "C" void art_quick_invoke_stub(...);
extern "C" void art_quick_invoke_static_stub(...);
extern "C" void art_quick_resolution_trampoline(...);
extern "C" void art_quick_imt_conflict_trampoline(...);
extern "C" void art_quick_generic_jni_trampoline(...);
```

**AOSP 17 蹦床安全增强**：
- **PAC（Pointer Authentication Codes）**：在 arm64 蹦床跳转时插入指针签名验证，**防止 ROP/JOP 攻击**
- **BTI（Branch Target Identification）**：限制蹦床目标地址必须在合法分支目标位，**防止任意跳转**
- 这两项是 ARMv8.3-A+ 硬件特性，AOSP 17 在 arm64 蹦床中默认启用

**蹦床作用**：

```
Java 调用 → 蹦床 → 实际执行

示例：invoke-virtual
  ↓
art_quick_invoke_stub（含 PAC 签名验证）
  ↓
检查方法状态（解释器 / JIT / AOT）
  ↓
跳转到对应代码入口
```

### 4.3 蹦床分类

| 类型 | 作用 |
| :--- | :--- |
| **art_quick_invoke_stub** | invoke-virtual 入口 |
| **art_quick_invoke_static_stub** | invoke-static 入口 |
| **art_quick_resolution_trampoline** | 符号解析 |
| **art_quick_imt_conflict_trampoline** | interface 多方法冲突 |
| **art_quick_generic_jni_trampoline** | JNI 调用入口 |

## 5. OSR（On-Stack Replacement）

### 5.1 什么是 OSR

**OSR（On-Stack Replacement）** 是在方法执行过程中**热替换**为编译版本的技术。

### 5.2 为什么需要 OSR

**场景**：
- 长循环方法（如游戏渲染循环）
- 解释器执行 → 方法被调用 10,000+ 次
- 如果等待方法返回再 JIT → 用户感知卡顿（数秒延迟）
- **OSR 在循环中途替换** → 用户无感知

### 5.3 OSR 工作机制

```
方法首次调用（解释器）
  ↓
调用次数达到阈值（10,000）
  ↓
JIT 编译器在后台编译方法
  ↓
编译完成后 → OSR 在循环回边替换入口
  ↓
方法继续执行（已经是 JIT 代码）
```

**栈帧转换**：
- 解释器栈帧（ShadowFrame + Dex PC）
- JIT 栈帧（Native 栈帧 + 机器 PC）
- OSR 转换时需要拷贝 / 转换栈状态

### 5.4 ART 17 OSR 去优化机制（AOSP 17 新增）

AOSP 17 引入了 **OSR 去优化（Deoptimization）**机制：当 JIT 编译的方法因为 Profile 变化或代码失效需要回退到解释器时，**OSR 去优化在循环回边反向替换入口**，保证长循环方法不会因为代码失效而卡死。

```
JIT 代码执行 ──代码失效──> 标记为去优化点
  ↓
下次循环回边 → 切换回解释器执行
  ↓
方法重新被 JIT 编译
  ↓
再次 OSR 切换回 JIT
```

**架构师视角**：去优化机制是 ART 17 长循环方法稳定性的关键补丁——之前在 ART 14 上偶发的"长循环方法卡死"问题在 ART 17 上消失。

## 7. ART 17 硬变化专章

### 7.1 static final 不可变（API 37+）

**AOSP 17 起，Java 层的 `public static final` 字段被标记为真正不可变**——不仅编译期常量，运行期也不允许通过反射修改。

**编译器侧变化**：
- AOT 编译时直接编码字面量到调用点（**字面量内联**）
- 跳过运行时常量池查找（性能 +5-10%）
- JIT 编译时同样优化

**反射侧变化**：
```java
// AOSP 14：反射修改成功（仅警告）
Field field = MyClass.class.getDeclaredField("CONSTANT");
field.setAccessible(true);
field.set(null, "new value");  // 成功

// AOSP 17：直接抛 IllegalAccessException
Field field = MyClass.class.getDeclaredField("CONSTANT");
field.setAccessible(true);
field.set(null, "new value");  // 抛 IllegalAccessException
```

**实战影响**：
- **Hook 框架失效**：很多 Hook 框架通过反射修改 static final 字段（如 hook View.mAccessibilityDelegate），**在 API 37+ 上完全失效**
- **配置覆盖失效**：很多 App 用反射覆盖 BuildConfig 常量，**在 API 37+ 上失效**
- **测试代码失效**：单元测试用反射准备 mock 数据，**在 API 37+ 上需要重写**

**架构师建议**：
- **新代码**：避免通过反射修改 static final，改用 `MutableLiveData` / `AtomicReference` 等可变容器
- **老代码**：在 API 37+ 上检测到 `IllegalAccessException` 时降级为非 final 字段
- **Hook 框架**：升级到 ART 17 兼容的 Hook 方案（如 Frida + ART 17 内置 bypass）

### 7.2 AOSP 17 编译器 IR 增强

```
┌────────────────────────────────────────────────────────────────┐
│ AOSP 17 编译器 IR 增强                                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. 内联深度从 48 层 → 96 层（性能 +10-15%）                      │
│  2. 去虚拟化增强（devirtualization）                              │

```

### 7.3 AOSP 17 与 Linux 6.18 关联

- **dex2oat 内存**：Linux 6.18 引入 `sheaves` 内存分配器，**让 dex2oat 在大 App 编译时内存峰值降低 20-30%**
- **PGO Profile 写盘**：Linux 6.18 `io_uring` 增强，**Profile 写盘延迟降低 40%**
- **跨系列引用**：详见 [Linux_Kernel/DM/09-DM-调优-性能与pcache](../01-Mechanism/Kernel/DM/09-DM-调优-性能与pcache.md) 了解 Linux 6.18 IO 优化如何间接提升 ART 启动速度

## 9. 总结（架构师视角的 5 条 Takeaway）

1. **三态切换是 ART 性能的核心**——解释器 + JIT + AOT 混合模式让"启动快 + 跑起来快 + 长期最优"三者兼得。**Profile 是切换的钥匙**。AOSP 17 通过 Cloud Profile 增量下发让这把钥匙"更准"。
2. **dex2oat 4 产物文件**（dex / vdex / odex / oat）——理解这些文件的作用是排查 APK 体积 / 安装慢 / 启动慢问题的前提。AOSP 17 把 dex2oat 拆分为 dex2oat + dexopt，让后台编译更轻量。
3. **Baseline Profile 是 PGO 的标准化**——AOSP 9+ 引入，让开发者主动声明热点方法。**冷启动优化必备**。AOSP 17 Cloud Profile 增量下发让命中率提升 15-25%。
4. **蹦床机制是 ART Native 入口**——所有 Java → Native 调用都经过 `art_quick_*` 系列。**理解蹦床是理解 ART Native 调用的前提**。AOSP 17 在蹦床中默认集成 PAC/BTI，**安全增强**。
5. **static final 不可变是 ART 17 的硬约束**（API 37+）——AOT 字面量内联 + 反射失效。**新代码避免反射改 final，老代码检测 IllegalAccessException 降级**。详见 [02-ART17无锁MessageQueue与static-final不可变 v2](02-ART17无锁MessageQueue与static-final不可变-v2.md)。
