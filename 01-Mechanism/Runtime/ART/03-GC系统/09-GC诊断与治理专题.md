## 一、dumpsys meminfo 全字段解读(ART 17 增强版)

`dumpsys meminfo` 是 GC 诊断的"第一站"——所有内存问题的起点都是 dumpsys meminfo。**所以呢**:线上 OOM 投诉时,先 dumpsys 确认分类(Java / Native / Graphics / Code),再决定走 hprof 还是 smaps 路径。

### 1.1 命令格式与 AOSP 17 增强

```bash
# 基本调用
adb shell dumpsys meminfo <package_name>     # 基础输出(14 类汇总)
adb shell dumpsys meminfo -d <package_name>  # AOSP 17 增强:含 ART Internal State
adb shell dumpsys meminfo -h                 # 帮助
```

AOSP 17 在原 14 类汇总之上**新增 2 段**:

```
┌────────────────────────────────────────────────────────────────┐
│ dumpsys meminfo 输出结构(AOSP 17)                              │
├────────────────────────────────────────────────────────────────┤
│ 1. 顶部:App 基础信息(Uptime / PSS Total / Heap Summary)        │
│ 2. 中部:内存分类(Native / Dalvik / Stack / Graphics / Code)    │
│ 3. 底部:对象分布(Views / AppContexts / Activities / Assets)    │
│ 4.【AOSP 17 新增】ART 内部状态(GC / JIT / ClassLoader / JNI)   │
```

### 1.4 完整字段含义(架构师视角)

| 字段 | 含义 | 关键判断 |
|:---|:---|:---|
| Pss Total | 按比例分摊的物理内存(最准) | > 500 MB 紧急,可能 LMK |
| Private Dirty | 进程独占的脏页(已修改) | > Native 堆 80% → 独占内存高 |
| Heap Size | 堆总大小 | 64 MB 默认;largeHeap 模式可到 512 MB |
| Heap Alloc | 堆已分配 | Alloc ≈ Size → 真实 OOM |
| Heap Free | 堆空闲 | Free 大但仍 OOM → 碎片化 |
| .so mmap | .so 库占用 | > 30 MB 异常(太多 .so) |
| .dex mmap | DEX 文件 | AOSP 17 典型 5-15 MB(AOT 优化) |
| Activities | Activity 数 | > 5 异常,高度怀疑泄漏 |
| ViewRootImpl | 根 View 数 | 与 Activity 不一致 → 泄漏 |

**AOSP 17 软阈值 `kSoftThresholdPercent=30%` 是 AOSP 17 最有价值的工程参数**——它在 `art/runtime/options.h` 中是**编译时常量**(`static constexpr size_t kSoftThresholdPercent = 30;`),**所以呢**:Heap Alloc 达到 30% 触发 Young GC(轻量、频繁、暂停 < 1ms),达到 80% 触发 Full GC(重量、罕见、暂停 5-20ms)。监控"距离软阈值的距离"比监控 GC 频率更精准。

### 1.5 真实 OOM vs 碎片化 OOM(关键判断)

```
情况 1:真实 OOM(堆真的满了)
  Heap Size:   65536
  Heap Alloc:  65000  ← 接近 Heap Size
  Heap Free:   536
  → 真实 OOM,需要修复泄漏

情况 2:碎片化 OOM(堆还有空闲但碎片化)
  Heap Size:   65536
  Heap Alloc:  30000  ← 远小于 Heap Size
  Heap Free:   35536
  → 碎片化 OOM,需要优化 Bitmap / byte[] 等大对象管理
```

**所以呢**:看到 OOM 投诉,先看 `Heap Alloc / Heap Size` 比例——Alloc ≈ Size 是泄漏(走 hprof+LeakCanary 路径),Alloc << Size 是碎片化(走 Bitmap 优化路径)。**AOSP 17 的 Distance to soft/hard threshold 字段让判断更精准**。

### 1.6 快速排查决策树

```
dumpsys meminfo 异常
│
├─ 1. 看 TOTAL PSS
│    ├─ < 100 MB → 正常
│    ├─ 100-300 MB → 警告

│    ├─ GC count 增长 > 10/min → GC 频繁
```

### 1.7 ART 17 软阈值示例(JNI 泄漏排查)

**场景**:某图像处理 App 在反复加载/卸载图片 100 次后,PSS 增长 200MB,最终 OOM。

```bash
$ adb shell dumpsys meminfo -d com.example.app

ART Internal State:
  ...
  JNI refs:   Global refs: 8500  Local refs: 8
              Weak global refs: 12
# Global refs = 8500(异常!正常应该 < 500)
```

**根因**:JNI Global refs = 8500 → **JNI 引用泄漏**(每次加载图片创建 85 个 Global ref,未释放)。每次加载图片:85 个 Global ref × 100 次 = 8500 个 ref → 8500 个 native 对象无法 GC → 200MB 泄漏。

**修复**:
```c
// 错误写法:每次 NewGlobalRef 但不 DeleteGlobalRef
jobject globalRef = (*env)->NewGlobalRef(env, localRef);
// 忘记调用 (*env)->DeleteGlobalRef(env, globalRef);

// 正确写法:配对使用
jobject globalRef = (*env)->NewGlobalRef(env, localRef);
// ...使用...
(*env)->DeleteGlobalRef(env, globalRef);  // 必须释放!
```

**所以呢**:**AOSP 17 的 JNI refs 输出是泄漏排查利器**——AOSP 14 看不到这个数据。**反复加载/卸载场景(图片、文件、Bitmap)最容易出 JNI 泄漏**。

### 1.8 dumpsys meminfo 的限制

```
dumpsys meminfo 不显示:
1. LOS(Large Object Space)大对象详情 — 只能看到 LOS 总占用,看不到具体哪个 Bitmap
2. 跨进程内存引用 — Bitmap / Surface 等跨进程共享看不到
3. Java 对象的具体类型 — 不知道哪个 Bitmap 占内存最大,需要 hprof + MAT
```

**所以呢**:dumpsys meminfo 是"内存概览",详细分析必须走 hprof 路径(详见 §四 MAT)。

## 二、procrank / smaps VMA 详细分析(Linux 6.18 增强)

`procrank` 给进程排名(`Pss > Uss > Rss > Vss` 优先级),`smaps` 给 VMA 粒度(Linux 内核 `/proc/$pid/smaps`)。**所以呢**:dumpsys meminfo 给"分类",procrank/smaps 给"VMA 粒度"——深度内存排查的下一站。

### 2.1 procrank 详解

```bash
# 基本命令
adb shell procrank
# 输出:PID Vss Rss Pss Uss Swap SwapPSs FD Process
#  12345  512MB 234MB 123MB 98MB 0 0 256 com.example.app

#【AOSP 17】按 Uss 排序(看独占内存)
adb shell procrank -u
```

字段优先级:`Pss > Uss > Rss > Vss`——**Pss 最准**(按比例分摊共享库),**Uss 看独占**(Private Dirty + Private Clean),**Rss 含共享**,**Vss 含未分配**。

AOSP 17 字段变化:
- 新增 `Pss_Cache`(缓存的 PSS,AOSP 17 优化)
- 新增 `SwapPss_Anon / SwapPss_Shmem`(区分匿名/共享 Swap)

**procrank 异常的诊断**:

| PSS | 状态 | 诊断 |
|:---|:---|:---|
| < 100 MB | 正常 | — |
| 100-300 MB | 警告 | 可能内存泄漏 |
| 300-500 MB | 严重 | 紧急优化 |
| > 500 MB | 紧急 | 即将被 LMK 杀 |

**所以呢**:**Pss > 500 MB 是紧急信号**,但**持续增长 > 1 MB/分钟才是泄漏预警**——单次 PSS 高不一定是泄漏,需要看趋势。

### 2.2 smaps VMA 粒度

```bash
# 1. 获取 smaps
adb shell run-as <package> cat /proc/self/smaps > smaps.txt

# 2.【AOSP 17 + Linux 6.18】smaps_rollup(汇总版,文件小 100 倍)
adb shell cat /proc/self/smaps_rollup > smaps_rollup.txt
```

**smaps vs smaps_rollup 关键差异**:
- 传统 smaps:每个 VMA 一段(~30 行),大进程文件 ~30 MB

### 2.3 AOSP 17 + Linux 6.18 smaps 硬变化(sheaves 段)

```
旧版(AOSP 14 / Linux 5.10):
[heap]                                # 只有 [heap] 段
Size:               80 kB
Pss:                80 kB

新版(AOSP 17 / Linux 6.18):
[heap]                                # 大对象(> 8KB)用 [heap]
Size:               70 kB
Pss:                70 kB

[sheaves]                             # 小对象(< 8KB)用 [sheaves](新!)
Size:               16 kB
Pss:                16 kB
```

**架构师解读**:
- **`[heap]` 段**:大对象(> 8KB)分配,仍是传统 malloc
- **`[sheaves]` 段**:小对象(< 8KB)分配,用 Linux 6.18 sheaves slab
- **优势**:sheaves 减少 slab 内部碎片,**Native 堆总占用降 15-20%**
- **smaps 价值**:能直接看到 sheaves 段大小,验证优化效果

**所以呢**:**别把 sheaves 优化的内存下降误判为"内存泄漏已修复"**——AOSP 17 升级后 Native 堆自然下降 15-20% 是 Linux 6.18 sheaves 的功劳,不是业务改动。

```bash
#!/bin/bash
# monitor_memory.sh - 自动化内存监控脚本
PACKAGE=$1
LOG_FILE="memory_monitor.log"

while true; do
    TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')
    
    # 1. procrank
    PSS=$(adb shell procrank | grep "$PACKAGE" | awk '{print $4}')
    
    # 2. dumpsys meminfo(Heap Alloc)
    HEAP_ALLOC=$(adb shell dumpsys meminfo "$PACKAGE" | grep "Dalvik Heap" | awk '{print $9}')
    
    # 3.【AOSP 17】ART 内部状态(GC count + JNI Global refs)
    GC_COUNT=$(adb shell dumpsys meminfo -d "$PACKAGE" | grep "Cumulative GC count" | awk '{print $NF}')
    JNI_GLOBAL=$(adb shell dumpsys meminfo -d "$PACKAGE" | grep "Global refs" | awk '{print $NF}')
    
    # 4.【AOSP 17】smaps_rollup 快速汇总(每 10 分钟)
    if [ $(($(date +%M) % 10)) -eq 0 ]; then
        adb shell run-as "$PACKAGE" cat /proc/self/smaps_rollup > "smaps_rollup_$(date +%Y%m%d%H%M).txt"
    fi
    
    # 5. 记录 + 告警
    echo "$TIMESTAMP PSS=$PSS HeapAlloc=$HEAP_ALLOC GCCount=$GC_COUNT JNIGlobal=$JNI_GLOBAL" >> "$LOG_FILE"
    if [ "${PSS%MB}" -gt 500 ]; then
        echo "[ALERT] $TIMESTAMP PSS=$PSS > 500MB"
    fi
    if [ "${JNI_GLOBAL}" -gt 1000 ]; then
        echo "[ALERT] $TIMESTAMP JNI Global refs=$JNI_GLOBAL > 1000"
    fi
    sleep 60
done
```

### 2.5 监控告警阈值

| 指标 | 阈值 | 监控方式 |
|:---|:---|:---|
| PSS | > 500 MB | procrank |
| Native Heap | > 100 MB | dumpsys meminfo / smaps |
| Heap Alloc / Heap Size | > 80% | dumpsys meminfo |
| **Distance to soft threshold** | **< 5%** | **dumpsys meminfo -d (AOSP 17)** |
| **JNI Global refs** | **> 1000** | **dumpsys meminfo -d (AOSP 17)** |
| .so 数量 | > 30 | smaps |
| VMA 数量 | > 1000 | smaps |
| **sheaves PSS** | **> 50 MB** | **smaps (AOSP 17 + Linux 6.18)** |

**工具选择原则**(AOSP 17 优化版):
- 日常监控 → dumpsys meminfo(轻量,14-16 类汇总)
- 找内存大户 → procrank(进程排名,按 PSS / Uss 排序)
- 深度排查 → smaps_rollup(快速汇总,AOSP 17 推荐)→ smaps(详细 VMA)
- 对象级分析 → hprof + MAT(看对象类型和引用链,详见 §四)
- GC 事件分析 → Perfetto / Systrace(看 GC 触发时机,详见 §五)

## 三、LeakCanary 原理(ReferenceQueue + WeakReference + Heap Dump)

LeakCanary 是"自动内存泄漏检测"的事实标准。**所以呢**:**生产环境必须集成 LeakCanary 3.x**(适配 AOSP 17),不要等用户投诉 OOM 才查。

### 3.1 核心原理:KeyedWeakReference + 5 秒延迟 + Shark 引擎

```
LeakCanary 检测内存泄漏的核心原理(AOSP 17 优化版):

1. 监控对象销毁
   - Activity.onDestroy / Fragment.onDestroy / View.onDetachedFromWindow
2. 用 KeyedWeakReference 包装已销毁对象
   - 让 GC 能回收已销毁对象(如果正确清理)
   - 如果对象未被回收 → 泄漏
3.【AOSP 17】触发 1-3 次 Young GC
   - GenCC 软阈值 30% 触发,5 秒延迟期间自动触发多次
4. 延迟检测(5 秒后)
   - 检查 WeakReference.get()
   - 还非 null → 泄漏
5. 触发 Heap Dump
   - LeakCanary HeapDumper 或 Android 11+ Heap Dump API
6.【AOSP 17】解析时处理类去重
   - hprof 中类去重后,类加载器引用要重新映射
7. Shark 引擎分析
   - 找出泄漏链(GC Root → 泄漏对象)
8.【AOSP 17】FinalReference 改进
   - Finalizer 池化(4 线程),泄漏检测不受 Finalizer 阻塞影响
9. 报告 + 修复
```

### 3.3 KeyedWeakReference 实现

```java
    private final String key;
    private final String name;
    private final long watchUptimeMillis;
```

## 四、MAT 使用指南(支配树 + Leak Suspects + 浅堆/深堆)

MAT(Memory Analyzer,Eclipse 基金会开源工具)用于 hprof 深度分析。**所以呢**:**LeakCanary 找泄漏点, MAT 深度验证**——两者结合覆盖完整 GC 治理流程。

### 4.1 浅堆 vs 深堆(核心概念)

```
Shallow Size(浅大小):对象自身占用的内存(不含引用的对象)
  包括:对象头 + 实例字段 + 对齐填充

Retained Size(保留堆):如果该对象被 GC 回收,能释放的总内存
  包括:自身 Shallow Size + 该对象引用的所有对象的 Shallow Size(递归)
```

```
┌────────────────────────────────────────────────────────────────┐
│ Shallow Size vs Retained Size                                  │
├────────────────────────────────────────────────────────────────┤
│  ┌─────┐ 引用 ┌─────┐ 引用 ┌─────┐                              │
│  │  A  │──────→│  B  │──────→│  C  │                            │
│  └──┬──┘      └──┬──┘      └──┬──┘                              │
│  Shallow: 100  Shallow: 200  Shallow: 300                       │
│                                                                │
│  A.Retained = A + B + C = 100 + 200 + 300 = 600                 │
│  B.Retained = B + C = 200 + 300 = 500                           │
│  C.Retained = C = 300                                           │
│                                                                │
│  → A 被回收 → B 和 C 也被回收 → 释放 600 bytes                  │
└────────────────────────────────────────────────────────────────┘
```

**关键区别**:
- Shallow Heap:对象自身大小(直接看得到)
- Retained Heap:对象能"保留"的总内存(释放这个对象能释放多少)
- **MAT 中看 Retained Heap 找最大内存占用**
- **MAT 中看 Shallow Heap 找最多的对象实例**

### 4.2 支配树(Dominator Tree)

```
A 支配 B = 从 GC Root 到 B 的所有路径都必须经过 A
特点:如果 A 被回收,B 也必然被回收
用途:找内存中最大的子树、找可能的内存泄漏点
```

```
┌────────────────────────────────────────────────────────────────┐
│ Dominator Tree 示例                                            │
├────────────────────────────────────────────────────────────────┤
│           GC Root                                              │
│              │                                                 │
│              ▼                                                 │
│         ┌───────┐                                              │
│         │   A   │ ← 支配所有                                   │
│         │ 600 B │                                              │
│         └───┬───┘                                              │
│       ┌─────┼─────┐                                            │
│       ▼     ▼     ▼                                            │
│   ┌─────┐ ┌─────┐ ┌─────┐                                      │
│   │  B  │ │  C  │ │  D  │                                      │
│   │ 200 │ │ 150 │ │ 250 │                                      │
│   └──┬──┘ └─────┘ └─────┘                                      │
│      ▼                                                         │
│   ┌─────┐                                                      │
│   │  E  │                                                      │
│   │ 100 │                                                      │
│   └─────┘                                                      │
│   A.Retained = A + B + C + D + E = 1300                         │
└────────────────────────────────────────────────────────────────┘
```

### 4.7 MAT 实战:GenCC Old 代泄漏定位

**场景**:AOSP 17 GenCC 下,Old 代有泄漏对象导致 PSS 持续增长。

```sql
-- OQL:找 GenCC Old 代的泄漏对象(retained > 1 MB)
SELECT obj, obj.@displayName, obj.@retainedHeapSize
FROM java.lang.Object obj
WHERE obj.@youngGen = false
  AND obj.@retainedHeapSize > 1048576
ORDER BY obj.@retainedHeapSize DESC
```

结果示例:
```
#1 Bitmap   retainedHeap=8.4 MB   youngGen=false
#2 byte[]   retainedHeap=5.2 MB   youngGen=false
#3 HashMap  retainedHeap=3.1 MB   youngGen=false
```

**根因**:这些对象是 GenCC Old 代对象(youngGen=false)→ 长期持有导致 Old 代增长 → Young GC(软阈值 30% 触发)回收不掉。

**修复**:
- 用 WeakReference / SoftReference 替代强引用
- 检查 static 字段 / 单例是否持有大对象
- 监控 Old 代大小,> 100 MB 告警

**所以呢**:**AOSP 17 GenCC 下 Young 代对象会自然回收**——但 Old 代对象不会。**Old 代泄漏是 AOSP 17 主要的 OOM 根因**,用 OQL `obj.@youngGen = false` 精准定位。

### 6.5 JVMTI 实战:ObjectFree 找泄漏候选

**场景**:用 ObjectFree 事件找泄漏候选——未触发 ObjectFree 的对象就是泄漏候选。

```cpp
// 对象 tag 跟踪
struct ObjectTag {
    jlong created_time;
    std::string class_name;
};

static std::map<jlong, ObjectTag> g_object_tags;
static std::set<jlong> g_freed_objects;

// 1. SetTag 时记录对象信息
void JNICALL OnObjectAlloc(jvmtiEnv* env, jlong tag, ...) {
    g_object_tags[tag] = {currentTimeMillis(), "..."};
}

// 2. ObjectFree 时记录释放
void JNICALL OnObjectFree(jvmtiEnv* env, jlong tag) {
    g_freed_objects.insert(tag);
}

// 3. 周期检测泄漏候选
void checkLeakCandidates() {
    for (auto& [tag, info] : g_object_tags) {
        if (g_freed_objects.count(tag) == 0) {
            long age = currentTimeMillis() - info.created_time;
            if (age > 60_000) {  // 1 分钟未释放
                apmClient.alert("leak.candidate", 
                    "Object not freed after 60s: " + info.class_name);
            }
        }
    }
}
```

**所以呢**:
- **JVMTI ObjectFree 事件是 AOSP 17 新利器**——找泄漏候选
- **不要单独依赖 ObjectFree**——它只告诉你"未释放",不告诉你"为什么未释放"
- **JVMTI + LeakCanary 配合**:JVMTI 找候选,LeakCanary 找根因

### 6.6 JVMTI 最佳实践

```
JVMTI 最佳实践(AOSP 17):
1. 异步上报 — JVMTI 回调中不要做耗时操作,异步队列 + 后台线程上报
2. 采样上报 — 不是每次 GC 都上报,采样(如 1/10),ObjectFree 必须采样(高频)
3. 过滤重要 GC — 只上报长 STW(> 50ms),过滤 Young GC(除非异常)
4.【AOSP 17】利用扩展元数据 — 不要只算 pause time,用 GC 类型/原因/软阈值距离做精细告警
5.【AOSP 17】多工具联动 — JVMTI(实时)+ Perfetto(事后)+ LeakCanary(泄漏)
```

## 七、监控指标体系(GC 频率/暂停/吞吐/分配速率)

监控指标体系是 GC 治理的"度量衡"。**所以呢**:**没有指标就没有治理**——所有优化都要有量化依据。

### 7.1 指标族分类(AOSP 17 增强版)

```
GC 监控指标族分类(AOSP 17 增强):

2. 时长指标族
   - STW 时间(gc.pause)
   - 总 GC 时间(含 Concurrent)
   - 95 / 99 分位
   -【AOSP 17】GenCC 各阶段耗时

3. 原因指标族
   - 各 GcCause 的次数
   -【AOSP 17 增强】通过 JVMTI 直接获取 gc_cause

4. 堆水位指标族
   - Java 堆 当前使用 / 总量 / 最大
   - Native 堆 当前使用 / 最大
   - 堆使用率
   -【AOSP 17 新增】软阈值触发距离(kSoftThresholdPercent=30%)
   -【AOSP 17 新增】硬阈值触发距离(80%)

5. ART 内部状态指标族【AOSP 17 新增】
   - JIT Code Cache 大小

6.【AOSP 17 新增】Native 堆分类指标族
   - sheaves slab 占用
   - malloc 占用

7. 衍生指标族
   - GC 分配速率(MB/s)
   - GC 暂停占总时长比例
   - 晋升速率(Young → Old)
```

### 7.2 四大核心指标详解

**指标 1:GC 频率**

| 范围 | 含义 |
|:---|:---|
| 静止应用 | < 1 次/分钟 |
| 轻度使用 | 1-5 次/分钟 |
| 重度使用 | 5-20 次/分钟 |
| 异常 | > 50 次/分钟 |

AOSP 17 增强:通过 JVMTI 直接获取 GC 类型,分 Young GC / Old GC / Sticky GC 分别统计。

**指标 2:GC 时长**

| 暂停时长 | 分类 |
|:---|:---|
| < 5ms | 正常 |
| 5-10ms | 轻微 |
| 10-50ms | 中等 |
| 50-100ms | 严重 |
| > 100ms | 紧急 |

卡顿阈值:**16ms(1 帧)→ 用户能感知;100ms → 明显卡顿;300ms → ANR 风险**。

AOSP 17 增强:GenCC 各阶段独立计时(Initialize / Marking / Reclaim / Promotion)。

**指标 3:GC 原因**

```
各 GcCause 的次数:
- kGcCauseForAlloc — 分配时触发(最常见)
- kGcCauseForNativeAlloc — Native 分配触发
- kGcCauseBackground — 后台并发触发
- kGcCauseExplicit — 手动 GC(System.gc)
- kGcCauseForTrim — Trim 内存触发
- kGcCauseJitArenaFull — JIT arena 满
```

AOSP 17 增强:通过 JVMTI `GetGarbageCollectionExtendedInfo` 直接获取 gc_cause,不需要额外采集。

**指标 4:堆水位**

```
Java 堆:
  - Java Heap Total / Used / Free / Max
  -【AOSP 17】分代指标:Young Gen Used / Size、Old Gen Used / Size、LOS Used / Size

【AOSP 17 增强】Native 堆:
  - Native Heap Used / Max
  -【AOSP 17】Native 堆分类:sheaves slab、malloc、mmap

堆使用率:
  - Java Heap Used / Java Heap Max
  - > 80% → 高风险
  - > 90% → 紧急

【AOSP 17 新增】软阈值触发距离:
  - Distance to soft threshold(30%)
  - > -5% → 软阈值即将触发

增长趋势:
  - 堆随时间增长
  - 持续增长 → 可能的泄漏
```

### 7.3 AOSP 17 新增指标族(3 大族)

**新增指标族 1:ART 内部状态指标族**

```
ART 内部状态指标(AOSP 17 新增):

1. JIT 状态
   - jit.code_cache.size(JIT Code Cache 大小)
   - jit.code_cache.max(Code Cache 上限)
   - jit.compiled_methods(JIT 已编译方法数)
   → Code Cache 满会导致 JIT 失效

2. ClassLoader 状态
   - classloader.loaded_classes(已加载类数)
   - classloader.count(ClassLoader 实例数)

3. JNI 引用
   - jni.global_refs(Global refs 数)
   - jni.local_refs(Local refs 数)
   - jni.weak_global_refs(Weak global refs 数)
   → JNI 引用泄漏 = 引用泄漏

4. GC 累计
   - gc.cumulative_count(累计 GC 次数)
   - gc.cumulative_time(累计 GC 耗时)
   - gc.last_type(上次 GC 类型)
   - gc.last_time(上次 GC 时间)
```

**所以呢**:**AOSP 14 看不到这些指标,只能从 ART 内部估算**。AOSP 17 通过 dumpsys meminfo -d / JVMTI 直接暴露——可观测性大幅提升。

**新增指标族 2:软阈值指标族**

```
软阈值指标(AOSP 17 新增):
1. soft_threshold.percent(kSoftThresholdPercent=30%)
2. hard_threshold.percent(80%)
3. distance_to_soft(软阈值距离)
   - > -5% → 软阈值即将触发
4. distance_to_hard(硬阈值距离)
```

**所以呢**:**软阈值是"轻量预警"**(Young GC 触发,暂停 < 1ms),**硬阈值是"紧急预警"**(Full GC 触发,暂停 5-20ms)。**Distance 字段明确告诉运维"距离下次 GC 触发还有多远"**。

**所以呢**:**Native 堆下降不一定是泄漏修复**——可能是 sheaves 优化。监控 Native 堆时必须看 sheaves / malloc / mmap 分类。

### 7.7 JVMTI 采集实现(AOSP 17 增强版)

```cpp
//【AOSP 17 增强】JVMTI 采集
static int gc_count = 0;
static int young_gc_count = 0;  //【AOSP 17】
static int old_gc_count = 0;    //【AOSP 17】
static long total_pause_time = 0;
static int soft_threshold_reached = 0;  //【AOSP 17】

void JNICALL OnGCFinish(jvmtiEnv* env) {
    long pause_time = currentTimeMillis() - g_gc_start_time;
    gc_count++;
    total_pause_time += pause_time;
    
    //【AOSP 17】获取 GC 扩展元数据
    jvmtiGarbageCollectionInfo gc_info = {0};
    env->GetGarbageCollectionExtendedInfo(&gc_info);
    
    if (gc_info.gc_type == GC_TYPE_YOUNG) young_gc_count++;
    if (gc_info.gc_type == GC_TYPE_OLD) old_gc_count++;
    
    //【AOSP 17】软阈值统计
    if (gc_info.distance_to_soft_threshold > -5) {
        soft_threshold_reached++;
    }
}

// 定时上报
void report() {
    apmClient.report("gc.count", gc_count);
    apmClient.report("gc.young.count", young_gc_count);  //【AOSP 17】
    apmClient.report("gc.old.count", old_gc_count);      //【AOSP 17】
    apmClient.report("gc.pause.avg", total_pause_time / gc_count);
    apmClient.report("soft.threshold.reached", soft_threshold_reached);  //【AOSP 17】
}
```

### 8.5 代码优化(4 大原则)

**原则 1:减少对象分配**

```java
// 1. 避免在循环中分配对象
// Bad
for (int i = 0; i < n; i++) {
    String s = String.valueOf(i);  // 每次创建 String
}
// Good
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) sb.append(i);

// 2. 使用对象池
// Good
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>()
);
executor.execute(runnable);

// 3. 避免装箱
// Bad: map.put(i, "value") 装箱
// Good: int[] keys = new int[n]; String[] values = new String[n];

// 4. 避免 String 拼接
// Bad: result += s(每次创建新 String)
// Good: StringBuilder.append(s)
```

**原则 2:对象池化(Bitmap / View)**

```java
// Bitmap 池化
public class BitmapPool {
    private final Stack<Bitmap> pool = new Stack<>();
    
    public Bitmap obtain(int width, int height) {
        if (!pool.isEmpty()) {
            Bitmap bmp = pool.pop();
            if (bmp.getWidth() == width && bmp.getHeight() == height) return bmp;
            bmp.recycle();
        }
        return Bitmap.createBitmap(width, height, Bitmap.Config.ARGB_8888);
    }
    public void recycle(Bitmap bmp) {
        if (pool.size() < MAX_POOL_SIZE) pool.push(bmp);
        else bmp.recycle();
    }
}
```

**原则 3:避免大对象**

```java
// Bitmap 优化
BitmapFactory.Options options = new BitmapFactory.Options();
options.inSampleSize = 2;                       // 缩小一半
options.inPreferredConfig = Bitmap.Config.RGB_565;  // 565 而非 8888
Bitmap bmp = BitmapFactory.decodeFile(path, options);

// 集合大小限制
List<String> list = new ArrayList<>(MAX_SIZE);  // 限制容量
```

**原则 4:避免 Finalize(AOSP 17 强化)**

```java
// 1. 用 Cleaner 替代 finalize
public class Resource {
    private static final Cleaner CLEANER = Cleaner.create();
    private final Cleaner.Cleanable cleanable;
    public Resource() {
        cleanable = CLEANER.register(this, new CleanupAction());
    }
    private static class CleanupAction implements Runnable {
        public void run() { close(); }
    }
}

// 2. 用 AutoCloseable + try-with-resources(首选)
try (Resource r = new Resource()) { /* 使用 res */ }
```

**所以呢**:**AOSP 17 池化主要解决"漏的 finalize",不是"慢的 finalize"**——新代码严禁使用 finalize(),用 AutoCloseable + try-with-resources 替代。

### 8.6 治理优先级与 ROI

```
治理优先级:
P0(立即):
  - OOM
  - 长 STW(> 100ms)
  - 严重内存泄漏

P1(紧急):
  - GC 频繁(> 50 次/分钟)
  - 中等卡顿(50-100ms)

P2(重要):
  - GC 频率中等(10-50 次/分钟)
  - 轻微卡顿(16-50ms)

P3(优化):
  - GC 频率低(< 10 次/分钟)
  - 偶发卡顿
```

```
ROI(投入产出比):
高 ROI(先做):
  - 减少对象分配(成本低,收益大)
  - 避免 Finalize(成本低,收益大)
  - 监控告警(成本中,收益高)

中 ROI(接着做):
  - 调大堆(成本低,收益中)
  - 对象池化(成本中,收益中)

低 ROI(后做):
  - CC GC 调优(成本高,收益中)
  - 升级到 GenCC(成本高,收益高,但需要平台升级)
```

### 9.1 实战案例 1:从 dumpsys 到 Heap Dump 完整诊断

**案例背景**:某社交 App 进入聊天页面后,再回到首页,内存持续增长。多次进出后内存不释放,最终 OOM 闪退。

**环境**:AOSP 17 + Linux 6.18;聊天页面 5 次以上进出触发 OOM;软阈值频繁触发,Young GC 频率高;Native 堆占用异常(疑似泄漏)。

**第一步:dumpsys meminfo -d(AOSP 17 增强)**

```bash
$ adb shell dumpsys meminfo -d com.example.app

  GC:  Last GC: 2s ago (ConcurrentCopying Young)
       Cumulative GC count: 234
       Soft threshold (30%): not reached (current 18%, threshold 30%)
  JNI refs:   Global refs: 142  Local refs: 8

Heap Summary:
  Dalvik Heap:    Alloc 45678 KB / Size 65536 KB (69.7%)
                  Distance to soft threshold (30%): -39.7% (way below)
```

**关键观察**:
1. **Activities: 5**(异常,应为 1)→ 高度怀疑 Activity 泄漏
2. **ViewRootImpl: 12**(异常)→ 确认泄漏
3. **Native Heap Distance to hard threshold: -5.6% (CRITICAL)** → 紧急
4. **JNI Global refs: 142** → 正常(阈值 < 1000)

**第二步:Heap Dump + hprof-conv 转换**

```bash

adb shell am dumpheap <pid> /data/local/tmp/heap.hprof
adb pull /data/local/tmp/heap.hprof

# 2.【AOSP 17 + Linux 6.18】hprof-conv 转换(io_uring 优化,1 GB 转换从 30 秒 → 10 秒)
hprof-conv heap.hprof heap-conv.hprof
```

**第三步:MAT 1.14.0+ 分析**

MAT 关键发现(AOSP 17 增强):
- 5 个 `com.example.app.ChatActivity` 实例(应为 0,已 finish)
- Dominator Tree 顶层是 `ChatManager` 单例,**Retained Heap 50MB**
- 包含:5 个 ChatActivity + Bitmap + ...
- OQL 查询确认:5 个 ChatActivity 全部被 `ChatManager` 持有

**引用链(GC Root → ChatActivity,AOSP 17 增强版)**:

```
ChatManager(静态单例)
  → List<ChatSession> mSessions
    → ChatSession
      → Context(ContextImpl)
        → ChatActivity(泄漏)
```

**AOSP 17 增强**:GC Root 路径查找时间从 30 秒(AOSP 14)→ 3 秒(AOSP 17),**快 10 倍**。

**第四步:LeakCanary 3.x 验证**

```
====================================
HEAP ANALYSIS RESULT
====================================
2  ChatActivity instances found.
0  ChatActivity instances are kept alive.

┬───
│ GC Root: Static field
│
├─ com.example.app.ChatManager class
│   Leaking: NO (a class is never leaking)
│
├─ ChatManager INSTANCE
│   Leaking: UNKNOWN
│
├─ ChatManager.mSessions
│   Leaking: YES (ArrayList retained)
│
└─ java.util.ArrayList instance
    Leaking: YES (Object was never GCed)
    Retained Heap: 50 MB
```

**根因**:`ChatManager` 是单例,静态引用 `ChatSession`,`ChatSession` 持有 Context。Activity finish 后仍被持有 → 泄漏。

**修复方案**:

```java
// 错误:ChatManager 单例持有 Activity Context
public class ChatManager {
    private static ChatManager sInstance;
    private final Context mContext;  // 持有 Activity!
    private ChatManager(Context context) {
        mContext = context;  // 传入 Activity → 泄漏
    }
}

// 正确:使用 Application Context + WeakReference
public class ChatManager {
    private static ChatManager sInstance;
    private final Context mAppContext;  // Application Context
    private final WeakReference<ChatSession> mSessionRef;  // 弱引用
    
    private ChatManager(Context context) {
        mAppContext = context.getApplicationContext();  // 转 Application
        mSessionRef = new WeakReference<>(new ChatSession());
    }
}
```

**修复后验证**:
- Activities: 1(正常)
- Heap Alloc: 稳定,不持续增长
- LeakCanary: 0 leaks found

**架构师 Takeaway**:
- **诊断流程:dumpsys → Heap Dump → MAT → LeakCanary**(五步递进,从概览到细节)
- **AOSP 17 GC Root 快速定位**:找泄漏链快 5-10 倍——大型 hprof(GB 级别)也能快速分析
- **Activity 泄漏 80% 来自 static 字段**——用 Application Context + WeakReference 替代

### 9.2 实战案例 2:LeakCanary 定位 AOSP 17 类去重误报

**案例背景**:升级到 AOSP 17 后,LeakCanary 2.x 大规模误报"Class 泄漏",CI 无法运行。

**环境**:AOSP 17 + LeakCanary 2.x + Android 11+ 设备

**现象**:LeakCanary 报告多个 `com.example.*` Class 泄漏,Retained Heap 5-10 MB。

**修复方案**:

```groovy
dependencies {
    debugImplementation 'com.squareup.leakcanary:leakcanary-android:3.0'
}
```

LeakCanary 3.x 适配:
- 正确识别"共享 Class"是正常情况
- 类加载器引用链重新映射
- **整体误报率降低 30-50%**

**验证**:
```

====================================
HEAP ANALYSIS RESULT
====================================
0 leaks found  ← 正常
====================================
```

**架构师 Takeaway**:
- **AOSP 17 升级必须同步升级 LeakCanary 3.x**——2.x 会因类去重大量误报
- 类去重是 AOSP 17 重要优化(节省 30-50% metaspace),**不要禁用**
- 用 LeakCanary 3.x 的"共享 Class"识别能力消除误报

**GcCollector 核心(AOSP 17 增强)**:

```java
public class GcCollector {
    private long gcCount = 0;
    private long totalPauseTime = 0;
    
    //【AOSP 17】分代 GC 统计
    private long youngGcCount = 0;
    private long oldGcCount = 0;
    private long stickyGcCount = 0;
    private long totalPromotedObjects = 0;
    
    //【AOSP 17】软阈值统计
    private long softThresholdReached = 0;
    private long hardThresholdReached = 0;
    
    //【AOSP 17】ObjectFree 统计
    private long objectFreeCount = 0;
    
    public void onGcFinish(jvmtiGarbageCollectionInfo gcInfo) {
        long pauseTime = System.currentTimeMillis() - gcStartTime;
        gcCount++;
        totalPauseTime += pauseTime;
        
        //【AOSP 17】分代 GC 统计
        switch (gcInfo.gc_type) {
            case GC_TYPE_YOUNG: youngGcCount++; break;
            case GC_TYPE_OLD: oldGcCount++; break;
            case GC_TYPE_STICKY: stickyGcCount++; break;
        }
        totalPromotedObjects += gcInfo.promoted_objects;
        
        //【AOSP 17】软阈值统计
        if (gcInfo.distance_to_soft_threshold > -5) softThresholdReached++;
        if (gcInfo.distance_to_hard_threshold > 0) hardThresholdReached++;
        
        // 告警
        if (pauseTime > 100) {
            alertManager.alert("gc.pause.long", pauseTime);
        }
    }
    
    public void onObjectFree(String className) {  //【AOSP 17】
        objectFreeCount++;
    }
    
    public void report() {
        Map<String, Object> metrics = new HashMap<>();
        metrics.put("gc.count", gcCount);
        metrics.put("gc.young.count", youngGcCount);  //【AOSP 17】
        metrics.put("gc.old.count", oldGcCount);      //【AOSP 17】
        metrics.put("gc.promoted", totalPromotedObjects);  //【AOSP 17】
        metrics.put("gc.soft.threshold.reached", softThresholdReached);  //【AOSP 17】
        metrics.put("object.free.count", objectFreeCount);  //【AOSP 17】
        reporter.report("gc.metrics", metrics);
        reset();
    }
}
```

## 十、ART 17 硬变化专章(综合 4 个工具强化)

AOSP 17 在诊断治理工具链上做了系统性强化,横跨 dumpsys / perfetto / JVMTI / Linux 内核 4 个层面。

### 10.1 dumpsys meminfo 增强(API 37+ 硬变化)

AOSP 17 在 dumpsys meminfo 中**新增 ART 内部状态输出**,把 ART Runtime 内部状态暴露给开发者:

```bash
ART Internal State:
  GC:  Last GC: 2s ago (ConcurrentCopying Young)
       Cumulative GC count: 234
       Cumulative GC time: 1.2s
       Soft threshold (30%): not reached (current 18%, threshold 30%)
  JIT: Code cache size: 8 MB / 16 MB
       JIT compiled methods: 1247
  ClassLoader:  Loaded classes: 8765
                Total class loader count: 23
  JNI refs:   Global refs: 142  Local refs: 8
              Weak global refs: 12
```

**架构师解读**:
- **GC 状态**:Last GC 类型/时间、累计 GC 次数/耗时——直接用于性能监控
- **JIT 状态**:Code cache / 已编译方法数——Code Cache 满会导致 JIT 失效
- **ClassLoader 状态**:已加载类数 / ClassLoader 实例数——ClassLoader 泄漏是常见 OOM 根因
- **JNI 引用**:Global/Local/Weak global——**JNI 泄漏 = 引用泄漏的直接证据**

**软阈值 kSoftThresholdPercent=30% 是 AOSP 17 最有价值的工程参数**:
- **软阈值 30%**:Heap Alloc 达到 30% 触发 Young GC(轻量、频繁、暂停 < 1ms)
- **硬阈值 80%**:Heap Alloc 达到 80% 触发 Full GC(重量、罕见、暂停 5-20ms)
- **Distance 字段**:明确告诉运维"距离下次 GC 触发还有多远"

5. **三件套协同(JVMTI + Perfetto + LeakCanary)是 AOSP 17 监控最佳组合**——同一对象/同一 GC 事件可通过时间戳关联

## 十二、总结(架构师视角 5 条 Takeaway)

1. **6 大工具组合覆盖 GC 治理的完整闭环**——**dumpsys meminfo 看分类**(PSS / Native / Dalvik / Heap Alloc)+ **procrank/smaps 看 VMA 粒度**(sheaves + smaps_rollup)+ **LeakCanary 看泄漏**(3.x 类去重适配)+ **MAT 看引用链**(1.14.0+ Class Extent)+ **Perfetto 看时序**(7 类 GenCC 事件 + ART 内部状态时间轴)+ **JVMTI 看实时**(GCStart/Finish + ObjectFree + GC 扩展元数据)。**6 大工具协同**是 AOSP 17 时代 GC 治理的标配——单一工具能力有限,组合才有完整价值。详见 §一-六。

2. **AOSP 17 dumpsys 增强 + 软阈值 kSoftThresholdPercent=30% 是最大亮点**——把 ART 内部状态(GC/JIT/ClassLoader/JNI refs)**全暴露**给开发者;**软阈值距离字段明确告诉运维"距离下次 GC 触发还有多远"**。**JNI Global refs > 1000 是 JNI 泄漏的直接证据**(AOSP 14 看不到)。**软阈值频繁触发**不一定是泄漏,可能是堆太小或分配过快。详见 §一 §十。

3. **Linux 6.18 io_uring + sheaves 是性能优化基础**——**hprof-conv 转换快 3 倍**(1 GB 从 30 秒 → 10 秒),**heap dump 写盘延迟降 30%**,**Native 堆 -15-20%**。**AOSP 17 + Linux 6.18 让生产环境频繁 dump hprof 成为可能**。**别把 sheaves 优化的内存下降误判为"内存泄漏已修复"**。详见 §二 §四 §八 §十。

4. **AOSP 17 升级必须配套升级工具链**——**MAT 1.14.0+**(Class Extent 元数据 + GC Root 索引 + hprof 1.2.0)+ **LeakCanary 3.x**(类去重适配 + Finalizer 池化)+ **Java 17+**(解析 AOSP 17 hprof)。**AOSP 14 工具 + AOSP 17 hprof = 误判/崩溃**。**新代码严禁使用 `finalize()`**——AOSP 17 池化也救不了慢 finalize,用 `AutoCloseable` + try-with-resources 替代。详见 §三 §四 §八。

5. **三件套(JVMTI + Perfetto + LeakCanary) + 多维度指标 = AOSP 17 监控最佳组合**——**JVMTI**(实时告警毫秒级触发,生产 APM 首选)+ **Perfetto**(事后 trace,SQL 批量分析)+ **LeakCanary 3.x**(自动泄漏检测)。**监控指标必须多维度**:GC 频率 + STW + 堆使用率 + **AOSP 17 软阈值距离** + **JNI Global Refs** + **晋升速率**——单维度信息不完整。**软阈值 30% 触发 = "轻量预警",硬阈值 80% 触发 = "紧急预警"**——别把软阈值当 OOM 信号。详见 §五 §六 §七 §十一。
