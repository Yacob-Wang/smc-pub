## 一、读屏障的本质

### 1.1 一句话定义

**读屏障（Read Barrier）** 是 **业务线程在读对象引用指令前后插入的钩子代码**——它让业务线程在 GC 并发移动对象时，仍能读到正确的对象地址。

### 1.2 读屏障存在的根本原因

```
┌────────────────────────────────────────────────────────────────┐
│ 读屏障存在的根本原因                                              │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  CC GC（Concurrent Copying）的根本挑战：                          │
│    1. GC 在并发标记 + 复制中持续移动对象                          │
│    2. 业务线程在 GC 移动对象的瞬间仍在读对象引用                   │
│    3. 业务线程读到旧地址 → 对象已被移动 → 野指针 / 数据错乱        │
│                                                                │
│  读屏障 = 解决方案：                                            │
│    1. 读引用时检查"对象是否已被移动"                               │
│    2. 如果被移动，更新指针到新地址（自愈）                          │
│    3. 业务线程总是读到最新地址                                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 1.3 读屏障在读引用指令前后的执行

```cpp
// 业务代码
Object value = obj.field;

// 实际执行（读屏障插入后）
Object tmp = obj.field;
tmp = ReadBarrier(tmp);   // 读屏障：检查是否被移动，自愈指针
Object value = tmp;
```

### 1.4 读屏障的两种语义

| 语义 | 触发时机 | 用途 | ART 中的实现 |
| :--- | :--- | :--- | :--- |
| **逻辑读屏障（Logical Read Barrier）** | 在源代码层面（编译时）插入 | 编译器在每个 obj.field 后插入 | C++ 抽象层 |
| **物理读屏障（Physical Read Barrier）** | 在机器码层面（运行时）插入 | 通过 trampoline / code patching | **ART 用物理读屏障** |

**架构师视角**：ART 用 **物理读屏障**——在 AOT/JIT 编译码中插入对 read barrier stub 的调用，**性能比逻辑读屏障高**。

### 1.5 读屏障 vs 写屏障（关键 insight）

| 维度 | 读屏障 | 写屏障 |
| :--- | :--- | :--- |
| 触发时机 | **读**对象引用时 | **指针赋值**时 |
| 触发频率 | **读多（~80-90%）** | 写少（~10-20%） |
| 开销 | **较高** | 较低 |
| 维护的不变式 | 弱三色不变式 | 强三色不变式 |
| ART 中的代表 | CC / GenCC | CMS（不变量）/ GenCC（Card Table） |
| 核心收益 | **STW < 1ms** | 漏标防护 |
| 核心代价 | 运行时开销 | 屏障调用开销 |

**关键 insight**：读屏障的开销 **远高于** 写屏障（因为读操作远多于写操作），**但 CC GC 选择读屏障，因为能把 STW 时间降到 < 1ms**——这是用户体验的根本提升。

详见 [03-写屏障机制](03-写屏障机制.md)。

## 二、为什么 CC GC 需要读屏障

### 2.1 CMS 的 STW 瓶颈（回顾）

回顾 CMS 的 STW 时间分布：
- Initial Mark（STW）：~5ms
- Concurrent Mark（并发）：0ms
- **Remark（STW）：~50ms**
- Concurrent Sweep（并发）：0ms

→ **Remark 阶段是 CMS 的硬伤**。

**Remark 为什么慢**？
- Incremental Update 把"被删除引用的对象"重新染灰
- 但这些对象可能已经有大量子对象
- STW 期间要重新扫描这些对象 → STW 时间不可控

详见 [03-写屏障机制](03-写屏障机制.md) §2.5。

### 2.2 CC GC 的设计目标

CC GC（AOSP 8+ 默认）想：
1. **STW < 1ms**（用户几乎无感知）
2. **不碎片化**（复制算法天然无碎片）
3. **避免 Remark 阶段**（直接并发移动对象）

但有个根本问题：**对象在并发移动中，业务线程还在跑，怎么保证读到正确的对象？**

→ **必须用读屏障**。

### 2.3 CC GC 的并发复制策略

```
时刻 T0: GC 开始，从 from-space 复制活对象到 to-space
时刻 T1: 业务线程访问 obj.field，obj 还在 from-space
时刻 T2: GC 移动 obj 到 to-space
时刻 T3: 业务线程再访问 obj.field → 读 old address → 错！
```

**问题**：obj 已经被移动，旧地址无效。

**解法**：读屏障检查"对象是否已被移动"，如果被移动，更新指针到新地址。

```cpp
// 读屏障伪代码
Object ReadBarrier(Object* field) {
    Object obj = *field;
    if (IsInFromSpace(obj)) {
        // 已被移动到 to-space
        Object new_obj = GetForwardingAddress(obj);
        *field = new_obj;  // 自愈指针
        return new_obj;
    }
    return obj;  // 快速路径
}
```

## 三、读屏障的实现机制

### 3.1 ART 读屏障的源码入口

```cpp
// art/runtime/read_barrier.h
class ReadBarrier {
 public:
  // 模板函数：对任意类型 T（T 通常是 mirror::Object* 或子类指针）
  template <typename T>
  static ALWAYS_INLINE T Barrier(T* field, ...) {
    // 1. 读取字段
    mirror::Object* obj = reinterpret_cast<mirror::Object*>(*field);

    // 2. 检查对象是否在 from-space
    if (UNLIKELY(obj == nullptr)) {
      return nullptr;
    }
    if (kUseBakerReadBarrier && obj->GetMarkWord<...>().IsMarked()) {
      // Baker 读屏障：检查对象头的 mark bit
      return obj;
    }
    if (kUseTableBasedReadBarrier) {
      // 表查读屏障：查 from-space 表
      return obj;
    }

    return obj;
  }

  // 简化的 CC GC 读屏障
  static mirror::Object* BarrierForRoot(mirror::Object* obj) {
    // 1. 检查是否在 from-space
    if (heap_->ConcurrentCopying()->IsInFromSpace(obj)) {
      // 2. 获取 to-space 的 forwarding address
      mirror::Object* new_obj = heap_->ConcurrentCopying()
                                      ->GetForwardingAddress(obj);
      // 3. 自愈
      return new_obj;
    }
    return obj;
  }
};
```

### 3.2 自愈指针（Self-Healing Pointer）

**关键设计**：读屏障是 **自愈的**。

```cpp
mirror::Object* ReadBarrier(mirror::Object** field) {
    mirror::Object* obj = *field;

    if (IsInFromSpace(obj)) {
        // 已经被移动到 to-space
        mirror::Object* new_obj = GetForwardingAddress(obj);

        // 关键：把新地址写回原字段
        *field = new_obj;  // 自愈！

        return new_obj;
    }
    return obj;
}
```

**自愈的意义**：
- 第一次访问 obj.field 会更新指针
- 后续访问走快速路径（不再触发 from-space 检查）
- **热路径开销接近零**

### 3.3 自愈指针的执行流程

```
┌────────────────────────────────────────────────────────────────┐
│ 自愈指针执行流程                                                  │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  业务线程第一次访问 obj.field：                                   │
│    1. 读 obj.field → obj (old_addr)                              │
│    2. 检查 obj 是否在 from-space                                  │
│    3. obj 在 from-space → 获取 forwarding_address                │
│    4. 写回 *field = new_addr (自愈)                              │
│    5. 返回 new_addr                                              │
│                                                                │
│  业务线程后续访问 obj.field：                                     │
│    1. 读 obj.field → obj (new_addr) ← 已被自愈                   │
│    2. 检查 obj 是否在 from-space                                  │
│    3. obj 不在 from-space → 快速路径                              │
│    4. 返回 obj (new_addr)                                        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 3.4 ART 中读屏障的三种模式

| 模式 | 含义 | 用途 | AOSP 版本 |
| :--- | :--- | :--- | :--- |
| `kWithoutReadBarrier` | 不开启读屏障 | 仅用于 Zygote / System Server 等特殊场景 | 通用 |
| `kWithReadBarrier` | 开启读屏障 | CC GC 默认模式 | AOSP 8+ |
| `kGrayImmuneReadBarrier` | 灰色对象免疫读屏障 | 编译器 / 反射等"信任场景" | AOSP 8+ |

```cpp
// art/runtime/read_barrier.h
enum ReadBarrierMode {
  kWithoutReadBarrier,
  kWithReadBarrier,
  kGrayImmuneReadBarrier,
};

// 不同模式的实现差异
mirror::Object* ReadBarrier::Barrier(T* field, ReadBarrierMode mode) {
    switch (mode) {
        case kWithoutReadBarrier:
            return *field;  // 直接返回，不检查

        case kWithReadBarrier:
            return BarrierForRoot(*field);  // 完整读屏障

        case kGrayImmuneReadBarrier:
            // 灰色对象免疫读屏障（信任 GC 已标记的对象）
            return *field;
    }
}
```

## 四、读屏障在编译码（AOT/JIT）中的实现

### 4.1 AArch64 上的读屏障机器码

```asm
; AArch64 读屏障的实现（art/runtime/arch/arm64/quick_entrypoints_arm64.S）
;
; 调用 ReadBarrier 的代码路径
; 入口：x0 = field_addr

read_barrier_entry:
    ; 1. 加载字段值
    ldr x1, [x0]                ; x1 = *field_addr（对象引用）

    ; 2. 检查是否非空
    cbz x1, .Lskip              ; 如果是 null，跳过读屏障

    ; 3. 检查对象是否在 from-space
    ;    ART 的实现：在对象头 Mark Word 中有特殊标记
    ldr x2, [x1, #mark_word_offset]   ; x2 = obj.mark_word
    tbnz x2, #kMarkBit, .Lskip        ; 如果 mark bit 已设置，跳过

    ; 4. 调用读屏障 stub
    bl artReadBarrier

.Lskip:
    ; 5. 返回对象引用（x1）
    ret
```

### 4.2 JIT 模式下的读屏障优化

ART 的 JIT 编译器做以下优化：

1. **内联（Inlining）**：简单的读屏障直接内联到调用点
2. **冗余消除**：如果连续两次读同一个字段，第二次可以消除
3. **循环优化**：循环外的读屏障提到循环外
4. **Baker 风格**：对象头 1 bit 标记是否已被读屏障处理

```cpp
// 优化前（每次循环都触发读屏障）
for (int i = 0; i < N; i++) {
    Object value = array[i].field;  // 读屏障
    process(value);
}

// 优化后（循环外触发一次读屏障）
Object first = ReadBarrier(&array[0].field);  // 循环外读屏障
for (int i = 0; i < N; i++) {
    Object value = array[i].field;  // 已经是新地址，不触发读屏障
    process(value);
}
```

### 4.3 解释器模式下的读屏障

解释器模式下读屏障是普通的 C++ 函数调用：

```cpp
// art/runtime/interpreter/interpreter.cc
void Interpreter::DoFieldGet(Thread* self, ...) {
  // ...

  // 读屏障（解释器模式下相对低开销）
  mirror::Object* result = field_value;
  if (kUseReadBarrier && !obj->IsReadBarrierProtected()) {
    result = ReadBarrier::BarrierForRoot(field_value);
  }

  // ...
}
```

### 4.4 ART 17 AArch64 读屏障的硬件加速

AOSP 17 利用 ARMv8.2 的 **CAS（Compare-And-Swap）指令** + **TSO 内存序** 优化读屏障：

```asm
; ART 17 AArch64 读屏障的硬件加速
; 旧版本：查 from-space 表
; 新版本：CAS + TSO 内存序

read_barrier_cas:
    ldr x1, [x0]                  ; x1 = *field_addr
    cbz x1, .Lskip                ; null 检查

    ; 检查对象头 mark bit
    ldrb w2, [x1, #mark_offset]   ; w2 = obj.mark_byte
    tbnz w2, #kReadBarrierBit, .Lskip  ; mark bit 已设置 → 快速路径

    ; 调用读屏障 stub（CAS 优化）
    bl artReadBarrierCAS

.Lskip:
    ret
```

**收益**：AOSP 17 上读屏障的多线程竞争开销降低 50%。

## 五、Baker 风格的读屏障优化

### 5.1 Baker 风格的核心思想

**Baker 风格**（David Bacon 等人提出）通过 **对象头 Mark Word 的额外 1 bit** 来记录对象是否已被读屏障处理：

```cpp
// Baker 风格读屏障伪代码
template <typename T>
inline T BakerReadBarrier(T* field) {
    T obj = *field;

    if (obj == nullptr) {
        return obj;
    }

    // 1. 检查对象头的 mark bit
    if (!obj->IsReadBarrierMarked()) {
        // 2. 未标记 → 需要读屏障
        T new_obj = ReadBarrierSlowPath(obj);

        // 3. 更新对象头 mark bit
        obj->SetReadBarrierMarked();

        // 4. 自愈指针
        *field = new_obj;

        return new_obj;
    }

    // 5. 已标记 → 快速路径
    return obj;
}
```

### 5.2 Baker 风格的优劣

**优点**：
- 快速路径开销极低（只读 1 bit）
- 自愈指针只需要更新一次
- 不需要查 from-space 表

**缺点**：
- 对象头需要 1 bit 空间（与 hash、lock、age 竞争）
- 写屏障需要维护 mark bit（与写屏障冲突）
- 不适合多线程并发标记

### 5.3 ART 14+ 的 rbcc 优化

**rbcc（Read Barrier Copy Collector）** 把读屏障 + 写屏障结合：

```cpp
// 优化后的读屏障伪代码（AOSP 14+）
template <typename T>
inline T ReadBarrierOptimized(T* field) {
    T obj = *field;

    // 1. 检查对象头
    if (!obj->IsMoved()) {
        // 2. 没被移动 → 直接返回
        return obj;
    }

    // 3. 被移动了 → 获取 forwarding address
    T new_obj = obj->GetForwardingAddress();

    // 4. 自愈
    *field = new_obj;

    return new_obj;
}
```

**关键改进**：
- **被移动的对象** 在对象头有特殊标记
- 读屏障只检查这个标记，不需要查全局表
- 写屏障在移动时设置这个标记

### 5.4 ART 17 rbcc 进一步优化

AOSP 17 在 rbcc 基础上进一步优化：

```
┌────────────────────────────────────────────────────────────────┐
│ ART 17 rbcc 进一步优化                                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. 对象头 mark bit 用 CAS 设置（无锁）                           │
│    └─ 多线程同时读屏障时冲突减少 50%                              │
│                                                                │
│  2. 读屏障内联（AOT 编译时）                                      │
│    └─ 简单读屏障直接内联到调用点                                   │
│    └─ 函数调用开销消除                                            │
│                                                                │
│  3. 循环提升（JIT）                                                │
```

## 六、读屏障的工程坑点

### 6.1 坑点 1：Hook 框架绕过读屏障

```cpp
// Xposed 旧版本 Hook 代码
void* HookMethod(void* method, void* new_entrypoint) {
    ArtMethod* art_method = reinterpret_cast<ArtMethod*>(method);

    // 直接修改 ArtMethod.entrypoint，绕过读屏障
    void* old_entry = art_method->entry_point_from_quick_compiled_code_;
    art_method->entry_point_from_quick_compiled_code_ = new_entrypoint;

    return old_entry;
}

// CC GC 移动 ArtMethod 到 to-space
// 业务线程调用 method.entrypoint → 读屏障检查
// 但 Hook 框架缓存的 method 指针是旧地址
// → 调用旧地址 → 崩溃
```

**修复**：

```cpp
// 新版本 Hook 框架
void* HookMethod(void* method, void* new_entrypoint) {
    ArtMethod* art_method = reinterpret_cast<ArtMethod*>(method);

    // 1. 调用读屏障（获取正确地址）
    art_method = ReadBarrier::BarrierForRoot(art_method);

    // 2. 修改 entrypoint
    void* old_entry = art_method->entry_point_from_quick_compiled_code_;
    art_method->entry_point_from_quick_compiled_code_ = new_entrypoint;

    return old_entry;
}
```

### 6.2 坑点 2：JNI 直接访问对象字段绕过读屏障

```cpp
// 错误的 JNI 代码
jstring GetFieldDirect(JNIEnv* env, jobject obj) {
    jclass cls = env->GetObjectClass(obj);
    jfieldID fid = env->GetFieldID(cls, "field", "Ljava/lang/String;");

    // ❌ 直接内存操作绕过读屏障
    return *(jstring*)((char*)obj + offset);
}
```

**修复**：

```cpp
// 正确做法
jstring GetFieldCorrect(JNIEnv* env, jobject obj) {
    jclass cls = env->GetObjectClass(obj);
    jfieldID fid = env->GetFieldID(cls, "field", "Ljava/lang/String;");

    // ✅ 内部自动调用读屏障
    return env->GetObjectField(obj, fid);
}
```

### 6.3 坑点 3：反射 / Unsafe 操作绕过读屏障

```java
// ❌ Unsafe.getObject() 不调用读屏障
Object value = unsafe.getObject(obj, offset);
// 风险：CC GC 移动对象后，offset 失效

// ✅ Field.get() 内部会调用读屏障
Object value = field.get(obj);
```

**修复**：
- 业务代码尽量用 Field.get()
- 必要时用 `ReadBarrier::BarrierForRoot()` 包装 Unsafe.getObject

## 八、读屏障与 CC/GenCC 的关系

### 8.1 CC GC 的核心机制（与读屏障协同）

```
┌────────────────────────────────────────────────────────────────┐
│ CC GC 核心机制（与读屏障协同）                                      │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. 初始标记（STW）                                                │
│    └─ 扫描 GC Root，标记为灰色                                    │
│                                                                │
│  2. 并发标记（与业务线程并行）                                      │
│    ├─ 从灰色对象出发，遍历子对象                                    │
│    ├─ 灰色 → 黑色                                                  │
│    ├─ 业务线程读引用 → 读屏障检查                                  │
│    └─ 业务线程写引用 → 写屏障记录                                  │
│                                                                │
│  3. 并发复制（与业务线程并行）                                      │
│    ├─ 活对象从 from-space 复制到 to-space                          │
│    ├─ 设置 forwarding address                                     │
│    └─ 业务线程读引用 → 读屏障 + 自愈指针                           │
│                                                                │
│  4. 并发清理（与业务线程并行）                                      │
│    └─ 释放 from-space 内存                                         │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**关键 insight**：**读屏障让 CC GC 在 STW < 1ms 的同时保证业务线程安全**——这是 ART 8+ 整个 GC 体系的基石。

### 8.2 GenCC 的核心机制（与读屏障协同）

AOSP 17 GenCC 在 CC GC 基础上分代处理：

```
┌────────────────────────────────────────────────────────────────┐
│ GenCC 核心机制（与读屏障协同）                                      │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Young GC（Minor GC）：                                          │
│    ├─ 根集合 = GC Root + Old → Young 引用（Remembered Set）      │
│    ├─ 标记范围：Young + Remembered Set                            │
│    ├─ 读屏障：Young→Young 不触发，Old→Young 触发                 │
│    └─ STW < 1ms                                                  │
│                                                                │
│  Full GC（Major GC）：                                            │
│    ├─ 根集合 = 全部 GC Root                                       │
│    ├─ 标记范围：整个堆                                              │
│    ├─ 读屏障：所有跨区引用触发                                      │
│    └─ STW 5-20ms                                                 │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 十二、总结（架构师视角的 5 条 Takeaway）

1. **读屏障 = 读对象引用时插入的钩子代码**——它的根本职责是**让 CC GC 在并发移动对象时，业务线程仍能读到正确地址**。**自愈指针让热路径开销接近零**。详见 [04-CC-GC](../04-CC-GC/)。

2. **CC GC 选读屏障的根本原因**——CMS 的 Remark 阶段（~50ms STW）是硬伤，**CC GC 用读屏障 + 自愈指针把 STW 降到 < 1ms**，代价是更高的运行时开销（读比写频繁）。详见 [03-写屏障机制](03-写屏障机制.md) §2.5。

3. **Baker 风格 + rbcc 是 ART 14+ 的关键优化**——对象头 1-2 bit 状态机让读屏障快速路径只读 1 bit；**自愈指针让第一次读屏障后后续访问零开销**。ART 17 进一步扩展 rbcc 状态机到 3 bit。

4. **ART 17 读屏障性能强化是高频读场景的关键改进**——单次读屏障从 30ns 降至 10ns（-67%），多线程读屏障冲突减少 50%。**GenCC 让 Young→Young 的读屏障不触发，整体开销再降 30%**。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.2。

5. **读屏障被绕过是高频读场景的稳定性坑**——Hook 框架、JNI、Unsafe、反射都可能导致读屏障被绕过，读到旧地址崩溃。**AOSP 17 已修复反射 Field.get / Method.invoke 屏障覆盖**，但 Hook 和 Unsafe 仍需业务代码用 ReadBarrier 包裹。
