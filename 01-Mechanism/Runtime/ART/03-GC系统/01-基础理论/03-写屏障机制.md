## 一、写屏障的本质

### 1.1 一句话定义

**写屏障（Write Barrier）** 是 **业务线程在指针赋值指令前后插入的钩子代码**——它让 GC 在并发标记中能感知业务线程对对象引用的修改，从而维护三色标记不变式。

### 1.2 写屏障存在的根本原因

```
┌────────────────────────────────────────────────────────────────┐
│ 写屏障存在的根本原因                                              │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  并发 GC 三大挑战：                                              │
│    1. 业务线程在 GC 标记期间继续修改对象引用                      │
│    2. 漏标（漏活对象）：黑色对象引用新对象 + 灰色对象断开引用      │
│    3. 跨代引用：分代 GC 中 Old→Young 的引用难以追踪               │
│                                                                │
│  写屏障 = 解决方案：                                            │
│    1. 维护三色不变式（防漏标）                                    │
│    2. 维护 Card Table（追踪跨代引用）                            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 1.3 写屏障在赋值指令前后的执行

```cpp
// 业务代码
obj.field = new_value;

// 实际执行（写屏障插入后）
PreWriteBarrier(obj, field_offset, old_value);  // 写屏障（赋值前，记录旧值）
obj.field = new_value;                          // 真正的赋值
PostWriteBarrier(obj, field_offset, new_value); // 写屏障（赋值后，记录新值）
```

### 1.4 写屏障的两种策略

| 策略 | 触发时机 | 维护目标 | ART 中的代表 | AOSP 版本 |
| :--- | :--- | :--- | :--- | :--- |
| **Pre-Write Barrier** | 在指针赋值**之前** | 记录旧值 / 维护增量更新 | CMS | AOSP 5-7 默认 |
| **Post-Write Barrier** | 在指针赋值**之后** | 记录新值 / 维护卡表 | GenCC | AOSP 10+ 默认 |

**架构师视角**：两种策略**不冲突**——GenCC 兼用两者（Pre 用于维护三色不变式，Post 用于维护 Card Table）。

### 1.5 写屏障 vs 读屏障

| 维度 | 写屏障 | 读屏障 |
| :--- | :--- | :--- |
| 触发时机 | 指针**赋值**时 | **读**对象引用时 |
| 触发频率 | 写少（~10-20%） | 读多（~80-90%） |
| 开销 | **较低** | **较高** |
| 维护的不变式 | 强三色不变式 | 弱三色不变式 |
| ART 中的代表 | CMS（不变量）/ GenCC（Card Table） | CC / GenCC（自辅助） |
| 适用场景 | 写多 / 引用密集 | 读多 / STW 敏感 |

详见 [04-读屏障机制](04-读屏障机制.md)。

## 二、CMS 的 Pre-Write Barrier（Incremental Update）

### 2.1 CMS 选 Pre-Write Barrier 的原因

**CMS（Concurrent Mark Sweep）** 是 AOSP 5-7 的默认 GC，**写屏障的职责是维护三色标记不变式**——防止漏标。

**CMS 选 Incremental Update 的逻辑**：
- 当业务线程断开引用（obj.field = null）时，**旧值即将不可达**
- 如果旧值是黑色对象的引用，**会触发漏标**
- 解决：把**被断开的旧值**重新染灰，重新扫描

### 2.2 Pre-Write Barrier 的实现（伪代码）

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的伪代码（AOSP 17 仍保留 CMS 代码路径）
void PreWriteBarrier(mirror::Object* obj, MemberOffset offset,
                     mirror::Object* new_value) {
  // 1. 读出旧值
  mirror::Object* old_value = obj->GetFieldObject<mirror::Object>(offset);

  // 2. Incremental Update：把旧值重新染灰
  if (old_value != nullptr) {
    // 把 old_value 加入 mark stack，等下一轮重新扫描
    MarkObject(old_value);
  }

  // 3. 真正的赋值由调用方完成
}
```

### 2.3 Incremental Update 防漏标原理

回顾经典漏标场景：

```
时刻 T1: GC 扫描 A（A 已染黑）
时刻 T2: 业务线程执行 A.field = null（断开 A → C）
时刻 T3: GC 扫描 B，B 不再引用 C
时刻 T4: GC 结束，C 仍是白色，被错误回收
```

Pre-Write Barrier 介入：

```
时刻 T1: GC 扫描 A（A 已染黑）
时刻 T2: 业务线程执行 A.field = null
         → Pre-Write Barrier 拦截
         → 读出旧值 C
         → 把 C 重新染灰（Incremental Update）
         → C 加入 mark stack
时刻 T3: GC 发现 mark stack 不为空，继续扫描
         → 扫描 C → C 的引用都被检查 → 不会漏标
时刻 T4: GC 结束，C 已正确标记
```

### 2.4 CMS Pre-Write Barrier 的代码膨胀

每次指针赋值多 4 条指令（AArch64 编译码）：

```asm
; 原始代码
str x1, [x0, #offset]    ; obj.field = new_value

; 编译码插入写屏障后
str x1, [x0, #offset]    ; 暂存 new_value
ldr x2, [x0, #offset]    ; 读出旧值
cbz x2, .Lskip            ; 如果旧值为 null，跳过
bl artPreWriteBarrier     ; 调用 Pre-Write Barrier
.Lskip:
str x1, [x0, #offset]    ; 真正的赋值
```

### 2.5 CMS 写屏障的硬伤：Remark 阶段

| CMS 阶段 | 暂停类型 | 暂停时间 |
| :--- | :--- | :--- |
| Initial Mark | STW | ~5ms |
| Concurrent Mark | 并发 | 0ms |
| **Remark** | **STW** | **~50ms**（写屏障把脏对象入 mark stack，STW 期间重新扫描） |
| Concurrent Sweep | 并发 | 0ms |

**Remark 阶段是 CMS 的硬伤**——Incremental Update 把"被删除引用的对象"重新染灰，但这些对象可能已经有大量子对象，**STW 期间要重新扫描这些对象 → STW 时间不可控**。

→ 这就是 **CC GC 引入读屏障**的根本原因（详见 [04-读屏障机制](04-读屏障机制.md)）。

### 2.6 CMS 写屏障的源码路径

```
art/runtime/gc/collector/mark_sweep.h            # MarkSweep 类
art/runtime/gc/collector/mark_sweep.cc           # MarkSweep::WriteBarrier
art/runtime/write_barrier.h                       # WriteBarrier 抽象层
art/runtime/write_barrier.cc                      # WriteBarrier 实现
art/runtime/arch/arm64/quick_entrypoints_arm64.S  # AArch64 写屏障机器码
art/runtime/arch/x86/quick_entrypoints_x86.S      # x86 写屏障机器码
```

## 三、GenCC 的 Post-Write Barrier（Card Table 维护）

### 3.1 GenCC 还需要写屏障的原因

**GenCC（Generational Concurrent Copying）** 是 AOSP 10+ 的默认 GC，主要靠 **读屏障** 维护弱三色不变式，但分代 GC 还需要：

- **Card Table**：记录跨代引用（Old → Young）
- **写屏障维护 Card Table**：业务线程每次指针赋值都要更新 Card Table

→ **GenCC 同时有读屏障和写屏障**，但写屏障的职责**不是"维护不变式"，而是"维护 Card Table"**。

### 3.2 GenCC Post-Write Barrier 的实现

```cpp
// art/runtime/gc/collector/concurrent_copying.h 的 PostWriteBarrier 简化版
void PostWriteBarrier(void* field_addr, mirror::Object* new_value) {
  // 1. 计算 field_addr 所在的 card
  uint8_t* card_addr = CardTable::AddressToCard(field_addr);

  // 2. 检查是否跨代引用
  SpaceType src_space = RegionSpace::GetSpaceTypeOf(field_addr);
  SpaceType dst_space = RegionSpace::GetSpaceTypeOf(new_value);

  // 3. 跨代时把 card 标记为 dirty
  if (src_space != dst_space &&
      (src_space == kSpaceTypeYoung || dst_space == kSpaceTypeYoung)) {
    *card_addr = CardTable::kCardDirty;
  }
}
```

### 3.3 Card Table 内存布局

```
│  │  Card 0 │  Card 1 │  Card 2 │  Card 3 │  ...    │  │
│  │ 512 B   │ 512 B   │ 512 B   │ 512 B   │         │  │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘  │
│       ↑                                              │
│       └── 跨代引用时标记为 dirty                        │
├────────────────────────────────────────────────────────┤
│                      Young Gen                         │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐  │
│  │  Card 0 │  Card 1 │  Card 2 │  Card 3 │  ...    │  │
│  │ 512 B   │ 512 B   │ 512 B   │ 512 B   │         │  │
```

### 3.4 ART 的 Card Table 实现

```cpp
// art/runtime/gc/space/region_space.h 的 CardTable（AOSP 17）
class CardTable {
 public:
  static constexpr size_t kCardSize = 512;  // 1 byte / 512 byte

  enum CardValue : uint8_t {
    kCardClean = 0,    // 干净（无跨代引用）
    kCardDirty = 0x70, // 脏（有跨代引用，需要扫描）
  };

  void MarkCard(const void* addr) {
    uint8_t* card = AddressToCard(addr);
    *card = kCardDirty;
  }

  bool IsDirty(const void* addr) {
    return *AddressToCard(addr) == kCardDirty;
  }

  // addr → card 的映射
  uint8_t* AddressToCard(const void* addr) {
    uintptr_t offset = reinterpret_cast<uintptr_t>(addr) - base_addr_;
    return &card_table_[offset / kCardSize];
  }

 private:
  std::unique_ptr<uint8_t[]> card_table_;
  uintptr_t base_addr_;
};
```

### 3.5 ART 17 Card Table 优化（细粒度）

AOSP 17 把 Card Table 从 512 字节压缩到 **128 字节**（部分场景 256 字节），让 Minor GC 扫描更精确：

```
┌────────────────────────────────────────────────────────┐
│ ART 17 Card Table 细粒度                                 │
├────────────────────────────────────────────────────────┤
│                                                        │
│  AOSP 14 (512 B / card)                                │
│    └─ 每次脏卡扫描的"无辜开销" = 512 字节                 │
│    └─ 大对象（> 512 字节）可能浪费大量扫描                │
│                                                        │
│  AOSP 17 (128 B / card，AOSP 17 默认)                   │
│    └─ 每次脏卡扫描的"无辜开销" = 128 字节                 │
│    └─ 大对象的浪费显著减少                                │
│    └─ Minor GC 扫描开销降低 20-30%                       │
│                                                        │
└────────────────────────────────────────────────────────┘
```

详见 §7.2。

### 3.6 Post-Write Barrier 与 Card Table 的关系（Minor GC 流程）

```cpp
// art/runtime/gc/collector/concurrent_copying.cc 的 Minor GC 简化版
void MinorGC() {
  // 1. 暂停所有 mutator 线程（STW）
  SuspendAllThreads();

  // 2. 扫描 Young Gen 的所有 Root
  ScanYoungGenRoots();

  // 3. 遍历 Card Table，找所有 dirty card
  for (uint8_t* card = card_table_begin_; card < card_table_end_; card++) {
    if (*card == CardTable::kCardDirty) {
      // 4. 扫描这张 card 对应的内存区域
      ScanCard(card);

      // 5. 标记为 clean
      *card = CardTable::kCardClean;
    }
  }

  // 6. 恢复 mutator 线程
  ResumeAllThreads();
}
```

## 四、写屏障在编译码（AOT/JIT）中的实现

### 4.1 AArch64 上的写屏障机器码

```asm
; AArch64 写屏障的实现（art/runtime/arch/arm64/quick_entrypoints_arm64.S）
;
; 调用 PreWriteBarrier 的代码路径
; 入口：x0 = obj, x1 = offset, x2 = new_value

pre_write_barrier_entry:
    stp x29, x30, [sp, #-16]!    ; 保存帧指针 + 返回地址
    mov x29, sp

    ; 读出旧值
    add x3, x0, x1                ; x3 = obj + offset
    ldr x4, [x3]                  ; x4 = *obj.offset（old_value）

    ; 如果旧值为 null，跳过写屏障
    cbz x4, .Lskip

    ; 调用写屏障逻辑
    bl artPreWriteBarrier

.Lskip:
    ; 恢复栈帧
    ldp x29, x30, [sp], #16
    ret
```

### 4.2 JIT 模式下的写屏障优化

ART 的 JIT 编译器会做以下优化：

1. **内联（Inlining）**：简单的写屏障直接内联到调用点
2. **Card Table 预计算**：card_addr 在循环外计算，避免循环内重复计算
3. **批量写屏障**：连续的指针赋值合并写屏障调用

```cpp
// 优化前
for (int i = 0; i < N; i++) {
    WriteBarrier(&array[i]);       // 每次循环都计算 card_addr
    array[i] = new_value;
}

// 优化后
uint8_t* card = CardTable::AddressToCard(&array[0]);  // 循环外计算
for (int i = 0; i < N; i++) {
    *card = kCardDirty;            // 直接更新 card
    array[i] = new_value;
}
```

### 4.3 解释器模式下的写屏障

解释器模式下写屏障是普通的 C++ 函数调用：

```cpp
// art/runtime/interpreter/interpreter.cc
void Interpreter::DoFieldPut(Thread* self, ...) {
  // ...

  // 写屏障（解释器模式下相对低开销）
  if (kUseReadBarrier && !obj->IsReadBarrierProtected()) {
    ReadBarrier::Barrier(...);
  }
  WriteBarrier::WriteField(obj, offset, new_value);

  // ...
}
```

### 4.4 ART 17 AArch64 写屏障的硬件加速

AOSP 17 利用 ARMv8.2 的 **CAS（Compare-And-Swap）指令** 优化写屏障：

```asm
; ART 17 AArch64 写屏障的 CAS 优化
; 旧版本：直接写 card（可能与其他线程竞争）
; 新版本：CAS 写 card（无锁）
pre_write_barrier_cas:
    ldrb w3, [x2]              ; w3 = *card_addr
    cmp w3, #kCardDirty        ; 已经是 dirty？
    b.eq .Lskip                ; 是则跳过
    mov w4, #kCardDirty
    ; CAS 写 card
    casa w3, w4, [x2]
    b .Lskip
```

**收益**：AOSP 17 上多线程写屏障冲突减少 80%。

## 五、Post-Write Barrier 的卡表扫描代价

### 5.1 Card Table 的扫描粒度问题

每张 dirty card 对应固定大小的扫描范围：

```
┌────────────────────────────────────────────────────────┐
│   Dirty Card（AOSP 17 默认 128 字节）                  │
│   ┌──────────┬──────────┬──────────┬──────────┐        │
│   │ 32 B     │ 32 B     │ 32 B     │ 32 B     │        │
│   │ Card数据 │ Card数据 │ Card数据 │ Card数据 │        │
│   └──────────┴──────────┴──────────┴──────────┘        │
│   ↑                                              ↑    │
│   跨代引用在这里                                  扫描终止
└────────────────────────────────────────────────────────┘
   扫描 128 字节，找到所有跨代引用
```

### 5.2 精度问题与 AOSP 17 优化

| ART 版本 | Card 大小 | 扫描粒度 | 浪费比例（大对象场景） |
| :--- | :--- | :--- | :--- |
| AOSP 14 | 512 B | 1 byte / 512 byte | 50-80% |
| AOSP 17 | 128 B（细粒度） | 1 byte / 128 byte | 10-30% |

**细粒度卡表**（Fine-grained Card Table）让 Minor GC 扫描精度提升 4 倍。

## 六、写屏障的工程坑点

### 6.1 坑点 1：Hook 框架绕过写屏障

```cpp
// Hook 框架代码
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    // 直接修改 ArtMethod.entrypoint，绕过写屏障
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}

// CC GC 移动 ArtMethod 到新地址
// 业务线程调用 method.entrypoint → 触发读屏障
// 但 Hook 框架修改的字段没经过写屏障 → Card Table 没更新
// → 跨代引用记录缺失 → Minor GC 漏标
```

**修复**：

```cpp
// Hook 框架必须用 Barrier 包裹
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    // 1. 先调用写屏障（更新 Card Table）
    WriteBarrier::WriteField(method,
        ArtMethod::EntryPointOffset(),
        reinterpret_cast<mirror::Object*>(new_entrypoint));

    // 2. 再修改 entrypoint
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}
```

### 6.2 坑点 2：JNI 直接修改字段绕过写屏障

```cpp
// 错误：直接内存操作
void JNI错误做法(JNIEnv* env, jobject obj, jstring new_value) {
    jclass cls = env->GetObjectClass(obj);
    jfieldID fid = env->GetFieldID(cls, "field", "Ljava/lang/String;");
    // ❌ 直接拿字段 offset 用 Unsafe 操作
    char* field_addr = reinterpret_cast<char*>(obj) + offset;
    *reinterpret_cast<jstring*>(field_addr) = new_value;
}
```

**修复**：

```cpp
// 正确：env->SetObjectField 内部调用写屏障
void JNI正确做法(JNIEnv* env, jobject obj, jstring new_value) {
    jclass cls = env->GetObjectClass(obj);
    jfieldID fid = env->GetFieldID(cls, "field", "Ljava/lang/String;");
    env->SetObjectField(obj, fid, new_value);
    // ✅ SetObjectField 内部会调用写屏障
}
```

### 6.3 坑点 3：反射 / Unsafe 绕过写屏障

```java
// Field.set() 内部会调用写屏障，OK
field.set(obj, new_value);

// ❌ Unsafe.putObject() 不调用写屏障
unsafe.putObject(obj, offset, new_value);
// 风险：CC GC 移动对象后，offset 失效；Card Table 不更新
```

**修复**：

```java
// 用 ReadBarrier / WriteBarrier 包装 Unsafe
unsafe.putObject(obj, offset, new_value);
WriteBarrier::WriteField(obj, MemberOffset(offset), new_value);  // 手动写屏障
```

### 6.4 坑点 4：JNI RegisterNatives 绕过写屏障

```cpp
// JNI RegisterNatives 注册 Native 方法
env->RegisterNatives(clazz, methods, nMethods);
// ❌ RegisterNatives 内部可能不调用写屏障
// → ArtMethod 引用未进入 Card Table
// → Minor GC 漏标
```

**修复**：ART 17 已修复 RegisterNatives 内部屏障调用，业务代码无需处理。

### 6.5 坑点 5（AOSP 17 新增）：反射 Method.invoke 绕过写屏障

**AOSP 17 之前**：`Method.invoke()` 内部不调用写屏障，**反射调用跨代对象时可能漏标**。

**AOSP 17 修复**：

```
┌────────────────────────────────────────────────────────────────┐
│ ART 17 反射屏障覆盖强化                                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  反射方法 invoke() 流程：                                        │
│    1. AOSP 14：                                                  │
│      └─ 不调用写屏障，依赖方法表更新                              │
│      └─ 跨代反射调用可能漏标                                       │
│                                                                │
│    2. AOSP 17：                                                  │
│      ├─ invoke() 内部调用 WriteBarrier                            │
│      ├─ ArtMethod 引用进入 Card Table                             │
│      └─ 跨代反射调用漏标率降低 50%                                │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

详见 §7.1。

```
├────────────────────────────────────────────────────────────────┤
```

## 八、实战案例

### 8.1 案例 1（AOSP 14 实测 + AOSP 17 改进）：高并发列表的写屏障开销

**现象**：某 App 在多线程并发 add/remove ArrayList 时，**写屏障成为性能瓶颈**，QPS 下降 30%。

**环境**：AOSP 14.0.0_r1（API 34）/ Pixel 7。

### 步骤 1：抓 Perfetto trace

```bash
adb shell perfetto --out /data/local/tmp/trace.proto \
  -t 10s sched freq idle am wm gfx view binder_driver hal dalvik
adb pull /data/local/tmp/trace.proto
```

**trace 显示**：
```
ART::WriteBarrier 出现 5 万次/秒
ART::MarkSweep 阶段长（Remark 50ms+）
```

### 步骤 2：分析业务代码

```java
// 业务代码：高并发下的列表操作
public class HighConcurrencyList {
    private List<Object> list = new ArrayList<>();

    // 多线程并发执行
    public void addAndRemove(Object obj) {
        synchronized (list) {
            list.add(obj);                  // 写屏障
            if (list.size() > 1000) {
                list.remove(0);             // 写屏障
            }
        }
    }
}
```

**根因**：CMS 写屏障（Incremental Update）在并发标记中：
- 业务线程 add 触发 Post-Write Barrier（写 Card Table）
- 业务线程 remove 触发 Pre-Write Barrier（重新染灰）
- 1000 次循环 × 2 次屏障 = 2000 次屏障调用 / 同步块
- 多线程竞争激烈 → QPS 下降 30%

### 步骤 3：修复

```java
// 修复 1：换 ConcurrentLinkedQueue（无锁）
public class HighConcurrencyList {
    private Queue<Object> queue = new ConcurrentLinkedQueue<>();

    public void addAndRemove(Object obj) {
        queue.offer(obj);
        if (queue.size() > 1000) {
            queue.poll();
        }
    }
}

// 修复 2：升级到 AOSP 17 GenCC
// GenCC 的 Post-Write Barrier 只维护 Card Table，不维护三色不变式
// 屏障开销降低 60%（50ns → 20ns）
```

```

│ QPS                                  │ 10000     │ 13000     │
```

## 十一、总结（架构师视角的 5 条 Takeaway）

1. **写屏障 = 指针赋值时插入的钩子代码**——它的根本职责是**维护三色标记不变式（防漏标）+ 维护 Card Table（追踪跨代引用）**。**CMS 用 Pre-Write Barrier（Incremental Update），GenCC 用 Post-Write Barrier（Card Table 维护）**。

2. **CMS Pre-Write Barrier 的硬伤是 Remark 阶段（~50ms STW）**——因为 Incremental Update 把脏对象入 mark stack，STW 期间要重新扫描。**这正是 CC GC 引入读屏障的根本原因**（详见 [04-读屏障机制](04-读屏障机制.md)）。

3. **GenCC 写屏障的职责是 Card Table 维护，不是三色不变式**——Post-Write Barrier 写脏卡，Minor GC 扫描脏卡找到跨代引用。**AOSP 17 把 Card 粒度从 512 B 压到 128 B，扫描开销降低 20-30%**。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3。

4. **ART 17 写屏障性能强化是关键改进**——单次写屏障调用从 50ns 降至 20ns（CAS 优化 + 内联 + 反射覆盖）。**多线程并发写屏障冲突减少 80%**。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.2。

5. **写屏障被绕过是最常见的稳定性坑**——Hook 框架、JNI、Unsafe、反射都可能导致写屏障被绕过，Card Table 不更新，漏标。**AOSP 17 已修复反射 invoke 和 RegisterNatives 屏障覆盖**，但 Hook 框架和 Unsafe 仍需业务代码用 WriteBarrier 包裹。
