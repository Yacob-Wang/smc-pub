## 一、案例 1：图片编辑器 LOS 碎片化导致大 Bitmap OOM

### 1.1 案例信息

| 维度 | 信息 |
| :--- | :--- |
| **App** | 某图片编辑器 |
| **AOSP 版本** | AOSP 14.0.0_r1（API 34）→ 升级 AOSP 17（API 37） |
| **设备** | Pixel 5 → Pixel 8 |
| **问题** | 编辑 4K 图片时 OOM 崩溃 |
| **OOM 类型** | Java heap OOM（看似） |
| **真实根因** | LOS 碎片化 |

### 1.2 问题描述

用户反馈：
> 在编辑 4K 图片（3840×2160）时，App 偶发性崩溃，显示 `OutOfMemoryError: Failed to allocate a 4194304 byte allocation`。

但 dumpsys meminfo 显示 Java 堆还有很多空闲内存（看起来不可能 OOM）。

### 1.3 排查过程

#### Step 1：抓取 dumpsys meminfo

```bash
$ adb shell dumpsys meminfo com.example.imageeditor

# 关键输出
                       Pss    Private   Private   SwapPss      Rss     Heap     Heap     Heap
                     Total    Dirty    Clean    Dirty    Total     Size    Alloc     Free
  Native Heap      56789    40000    16789      200    67890   102400    56789    45611
  Dalvik Heap      34567    30000     4567      100    40000    98304    45000    53304
   TOTAL           118706    89390    29316      300  140450  102400    56789    45611
```

#### Step 2：分析数据

```
Dalvik Heap:
  Heap Size: 98304 KB (96 MB)
  Heap Alloc: 45000 KB (44 MB)  ← 使用了 44 MB
  Heap Free: 53304 KB (52 MB)   ← 空闲 52 MB
```

**看起来**：堆空闲 52 MB，分配 4 MB 应该没问题。

**实际**：OOM 发生了。

**矛盾！**

#### Step 3：分析 LOS

```bash
$ adb shell am gc  # 触发 GC

$ adb shell cmd activity dumpheap <pid> /data/local/tmp/large_objects.hprof
$ adb pull /data/local/tmp/large_objects.hprof

# 用 hprof-conv 转换
$ hprof-conv large_objects.hprof large_objects-conv.hprof

# 用 MAT 打开
# 找到 LOS 中的大对象（> 12 KB）
```

#### Step 4：分析 LOS 大对象

用 MAT 分析 hprof，发现 LOS 中的 Bitmap 分布：

```
LOS 中的 Bitmap（按大小排序）：
┌────────────────────────────────────────────────┐
│  Bitmap A (8 MB) - 缩略图缓存 1                │
│  Bitmap B (12 MB) - 原始图片                   │
│  [FREE 1 MB]                                   │
│  Bitmap C (4 MB) - 编辑缩略图                  │
│  [FREE 2 MB]                                   │
│  Bitmap D (16 MB) - 4K 编辑图层                │
│  [FREE 1 MB]                                   │
│  Bitmap E (8 MB) - 缩略图缓存 2                │
│  [FREE 0.5 MB]                                 │
│  Bitmap F (4 MB) - 滤镜预览                    │
└────────────────────────────────────────────────┘
总占用：52 MB
最大连续空洞：2 MB
```

#### Step 5：找到根因

```
需要分配的 Bitmap G: 4 MB（编辑图层）
LOS 状态：
  - Bitmap A (8 MB) 已用
  - [FREE 1 MB] 空洞
  - Bitmap C (4 MB) 已用
  - [FREE 2 MB] 空洞  ← 最大空洞
  - Bitmap D (16 MB) 已用
  - [FREE 1 MB] 空洞
  - Bitmap E (8 MB) 已用
  - [FREE 0.5 MB] 空洞
  - Bitmap F (4 MB) 已用

→ 最大空洞 2 MB < 需要 4 MB
→ 即使总空闲 4.5 MB，依然无法分配 4 MB 连续空间
→ OOM！
```

**根因**：**LOS 碎片化** —— 大量小空洞，最大空洞 < 单个 Bitmap 大小。

#### Step 6：确认业务代码

```java
// 用户的图片编辑器代码
public class ImageEditor {
    private List<Bitmap> layerCache = new ArrayList<>();

    public void addLayer(Bitmap layer) {
        layerCache.add(layer);  // Bitmap 进入 LOS
    }

    public void removeLayer(int index) {
        Bitmap removed = layerCache.remove(index);
        // ❌ 忘记调用 removed.recycle() 释放 native 像素
        // ❌ Bitmap 对象本身被 GC 回收，但 LOS 留下空洞
    }
}
```

**问题**：
- 删除图层时 **忘记 recycle()** → Bitmap 对象被 GC 回收，但 LOS 留下空洞
- 长期运行后，LOS 充满小空洞
- 即使总空闲足够，也无法分配大 Bitmap

### 1.4 修复方案

#### 方案 1：及时回收 Bitmap

```java
// 修复后
public void removeLayer(int index) {
    Bitmap removed = layerCache.remove(index);
    if (removed != null && !removed.isRecycled()) {
        removed.recycle();  // 立即释放 native 像素
        // Bitmap 对象本身在下次 GC 时被回收 → LOS 空间也释放
    }
}
```

#### 方案 2：使用 inBitmap 复用

```java
// 修复后：复用已有的 Bitmap
public void addLayer(BitmapFactory.Options options) {
    // 检查是否能复用
    if (canUseInBitmap(layerCache, options)) {
        options.inBitmap = findReusableBitmap(layerCache, options);
    }
    Bitmap layer = BitmapFactory.decodeFile(path, options);
    layerCache.add(layer);
}

private boolean canUseInBitmap(List<Bitmap> cache, BitmapFactory.Options options) {
    // 检查是否有兼容的 Bitmap 可复用
    for (Bitmap bitmap : cache) {
        if (BitmapUtilities.canUseForInBitmap(bitmap, options)) {
            return true;
        }
    }
    return false;
}
```

#### 方案 3：分块 Bitmap

```java
// 修复后：大 Bitmap 分块
public Bitmap createTiledBitmap(int width, int height) {
    int tileSize = 1024;
    int cols = (width + tileSize - 1) / tileSize;
    int rows = (height + tileSize - 1) / tileSize;

    Bitmap[] tiles = new Bitmap[cols * rows];
    for (int i = 0; i < tiles.length; i++) {
        tiles[i] = Bitmap.createBitmap(tileSize, tileSize, Bitmap.Config.ARGB_8888);
        // 每个 tile 只有 4 MB，更易分配和回收
    }
    return new TiledBitmap(tiles, width, height);
}
```

#### 方案 4：限制 LOS 大小

```java
// 修复后：主动管理 LOS
public class LRUBitmapCache {
    private LinkedHashMap<String, Bitmap> cache;
    private long maxLOSUsage;  // 限制 LOS 总占用
    private long currentLOSUsage;

    public void put(String key, Bitmap bitmap) {
        // 检查 LOS 占用
        long bitmapSize = bitmap.getByteCount();
        if (currentLOSUsage + bitmapSize > maxLOSUsage) {
            evictUntil(bitmapSize);  // 淘汰直到能放下
        }
        cache.put(key, bitmap);
        currentLOSUsage += bitmapSize;
    }
}
```

#### 方案 5（AOSP 17 升级）：依赖 LOS 压缩 + 增量压缩

```bash
# AOSP 17 LOS 压缩默认启用
# 无需业务层调整
# -XX:LOSCompactThreshold=20  # 碎片化 > 20% 就触发（更激进）
# -XX:LOSCompactBatchSize=5   # 增量压缩分 5 次 Minor GC（更激进）
```

详见 [07-慢速路径与碎片化](07-慢速路径与碎片化.md) §7.1。

### 1.5 修复后的验证

```bash
$ adb shell dumpsys meminfo com.example.imageeditor

  Dalvik Heap      23456    20000     3456       50    28000    65536    23456    42080
                                                              ↑        ↑       ↑
                                                              Size    Alloc   Free
                                                              64 MB   22 MB   41 MB (空闲)
```

**修复效果**：

```
┌──────────────────────────────────────┬───────────┬───────────┬───────────┐
│ 指标                                  │ AOSP 14   │ AOSP 17   │ AOSP 17 升级│
├──────────────────────────────────────┼───────────┼───────────┼───────────┤
```

## 二、案例 2：频繁 Minor GC 导致 CPU 占用飙升

### 2.1 案例信息

| 维度 | 信息 |
| :--- | :--- |
| **App** | 某新闻 App |
| **AOSP 版本** | AOSP 14.0.0_r1（API 34）→ 升级 AOSP 17（API 37） |
| **设备** | Pixel 6 → Pixel 8 |
| **问题** | App 启动后 1 分钟，CPU 占用 30%，全部在 GC |
| **真实根因** | 软阈值未启用（AOSP 14 默认 0% = Young Gen 满才 GC） |

### 2.2 问题描述

用户反馈：
> 打开新闻 App 后，滑动列表时偶发卡顿，用 Perfetto trace 看到 GC 频率高（每秒 2-3 次），单次 STW 时间 1-2ms。

### 2.3 排查过程

#### Step 1：抓 Perfetto trace

```bash
adb shell perfetto --out /data/local/tmp/trace.proto \
  -t 30s sched freq idle am wm gfx view binder_driver hal dalvik
adb pull /data/local/tmp/trace.proto
```

**trace 显示**：
```
Minor GC 频率：2-3 次 / 秒
Minor GC STW：1-2ms / 次
30 秒内 Minor GC 次数：80 次
总 STW 时间：120ms（30 秒内）
```

#### Step 2：分析 GC log

```bash
adb logcat -d -s art:V | grep "Background concurrent"
# 输出：
# Background concurrent mark sweep GC: 80 times, 120ms total
```

**问题**：
- AOSP 14 默认软阈值 = 0%（Young Gen 满才触发 GC）
- 业务线程疯狂分配 → Young Gen 快速填满 → Minor GC 频繁
- 单次 STW 不长（1-2ms），但**频繁触发 → CPU 占用 30%**

#### Step 3：分析业务代码

```java
// 业务代码：新闻列表分配大量短生命周期对象
public class NewsListAdapter extends RecyclerView.Adapter {
    @Override
    public void onBindViewHolder(ViewHolder holder, int position) {
        NewsItem item = items.get(position);
        // 每次 onBind 创建 StringBuilder + DateFormat + 临时对象
        holder.title.setText(item.title);
        holder.time.setText(formatTime(item.publishTime));  // 创建 StringBuilder
    }
}
```

**根因**：
- AOSP 14 默认软阈值 0% → Young Gen 满才 GC
- Young Gen 满时 GC → 单次 STW 长（> 1ms）
- 业务分配频率高 → Minor GC 频繁触发

### 2.4 修复方案

#### 方案 1：业务层减少分配

```java
// 优化前：每次 onBind 创建 StringBuilder
public void onBindViewHolder(ViewHolder holder, int position) {
    holder.time.setText(formatTime(item.publishTime));
}

private String formatTime(long time) {
    StringBuilder sb = new StringBuilder();  // ❌ 每次创建
    sb.append(formatDate(time));
    sb.append(" ");
    sb.append(formatHour(time));
    return sb.toString();
}

// 优化后：复用 StringBuilder
private final StringBuilder sb = new StringBuilder();
private final SimpleDateFormat dateFormat = new SimpleDateFormat("yyyy-MM-dd HH:mm");

public void onBindViewHolder(ViewHolder holder, int position) {
    sb.setLength(0);
    sb.append(dateFormat.format(new Date(item.publishTime)));
    holder.time.setText(sb.toString());
}
```

#### 方案 2：升级到 AOSP 17 软阈值

```bash
# AOSP 17 默认启用软阈值 kSoftThresholdPercent=30%
# 即 Young Gen 到 30% 容量就触发 GC
# -XX:SoftThresholdPercent=30  # AOSP 17 默认
# -XX:SoftThresholdPercent=50  # 更激进
```

**软阈值的影响**：

```
┌──────────────────────────────────────┬───────────┬───────────┬───────────┐
│ 指标                                  │ AOSP 14（0%）│ AOSP 17（30%）│ AOSP 17（50%）│
├──────────────────────────────────────┼───────────┼───────────┼───────────┤
│ Minor GC 频率                          │ 2-3 次/秒  │ 5-6 次/秒  │ 8-10 次/秒 │
│ Minor GC 单次 STW                      │ 1-2ms     │ 0.2-0.5ms  │ 0.1-0.3ms  │
│ 30 秒 GC 次数                          │ 80 次     │ 150 次     │ 250 次     │
│ 30 秒总 STW 时间                       │ 120ms     │ 60ms       │ 50ms       │
│ CPU 占用（GC 部分）                    │ 30%       │ 10%        │ 8%         │
│ 滑动卡顿                              │ 明显       │ 几乎无      │ 几乎无      │
└──────────────────────────────────────┴───────────┴───────────┴───────────┘
```

**AOSP 17 软阈值（30%）让 Minor GC 更频繁但单次 STW < 0.3ms**——**总 STW 时间反而降低 50%**。

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.6。

### 2.5 验证

```bash
# 升级到 AOSP 17 后
adb logcat -d -s art:V | grep "Background concurrent"
# 输出：
# Background concurrent mark sweep GC: 150 times, 60ms total
```

**典型模式说明**：上述数据基于"新闻列表频繁分配 + AOSP 14 软阈值 0% + 升级 AOSP 17 软阈值 30%"的典型场景。**具体数值因机型、业务分配模式而异**。

## 三、案例 3：高并发 Region Pool 锁竞争

### 3.1 案例信息

| 维度 | 信息 |
| :--- | :--- |
| **App** | 某 IM App（消息推送 + 实时聊天） |
| **AOSP 版本** | AOSP 14.0.0_r1（API 34）→ 升级 AOSP 17（API 37） |
| **设备** | Pixel 7 → Pixel 8 |
| **问题** | 100 线程并发分配消息对象，分配延迟 500ns，CPU 占用 100% |
| **真实根因** | Region Pool 全局锁竞争 |

### 3.2 问题描述

用户反馈：
> IM App 收到群消息轰炸时（100 线程并发处理消息），分配延迟从 1ns 飙升到 500ns，CPU 占用 100%，消息处理 QPS 下降 80%。

### 3.3 排查过程

#### Step 1：抓 Perfetto trace

```bash
adb shell perfetto --out /data/local/tmp/trace.proto \
  -t 10s sched freq idle am wm gfx view binder_driver hal dalvik
adb pull /data/local/tmp/trace.proto
```

**trace 显示**：
```
RegionPool::AllocateRegion 锁等待时间 500ns/次
TLAB 用完导致申请新 Region 频率 5000次/秒
CAS 失败次数 5000次/秒
sched_yield 调用 1000次/秒
```

#### Step 2：分析业务代码

```java
// 业务代码：100 线程并发处理消息
ExecutorService pool = Executors.newFixedThreadPool(100);
for (int i = 0; i < 100; i++) {
    pool.submit(() -> {
        while (true) {
            // 每线程每秒处理 1000 条消息（每条 1KB 对象）
            List<Message> messages = receiveMessages();
            for (Message msg : messages) {
                processMessage(msg);  // 创建临时对象
            }
        }
    });
}
```

**根因**：
- 100 线程同时申请新 Region → Region Pool 全局锁竞争
- 单次分配延迟 1ns → 500ns（500 倍）
- 99% CAS 失败 → 无限重试 → CPU 100%

### 3.4 修复方案

#### 方案 1：业务层减少分配

```java
// 优化前：每条消息创建新对象
public void processMessage(Message msg) {
    String text = new String(msg.body);  // ❌ 每次创建
    // ...
}

// 优化后：复用对象池
private final ObjectPool<StringBuilder> sbPool = new ObjectPool<>(100);

    StringBuilder sb = sbPool.acquire();
    sb.setLength(0);
    sb.append(msg.body);
    // ...
    sbPool.release(sb);
}
```

## 九、总结（架构师视角的 5 条 Takeaway）

1. **OOM ≠ 堆用完**——最常见原因是 LOS 碎片化（"堆里还有 100MB 为什么 OOM"）。**dumpsys meminfo 不显示 LOS 详细分布，必须用 hprof + MAT 分析**。AOSP 17 LOS 压缩默认启用 + 增量压缩让碎片化 OOM 减少 70%，详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.5。

2. **Bitmap 必须及时 recycle()**——否则 LOS 留下空洞，**即使业务层用 inBitmap 复用仍可能有碎片化残留**。AOSP 17 LOS 压缩 + 增量压缩自动消除空洞，但业务层仍应保持好习惯。

3. **频繁 Minor GC = 软阈值未启用 + 业务高分配**——AOSP 14 默认软阈值 0%（Young Gen 满才 GC）导致单次 STW 长（> 1ms）。AOSP 17 软阈值 30% 让 Minor GC 更频繁但单次 STW < 0.3ms，**总 STW 时间反而降低 50%**，详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.6。

4. **高并发分配慢 = Region Pool 全局锁竞争**——100 线程并发 + TLAB 64 KB → 每秒 5000 次 Region 申请 → 锁等待 500ns / 次。AOSP 17 **CAS 优化 + 后台预分配（RegionPrefetcher）**把锁等待降到 10ns / 次（-98%），CPU 占用从 100% 降到 60%，详见 [06-Concurrent分配器](06-Concurrent分配器.md) §7.2、§7.3。

5. **Heap 调优是系统性问题**——单点优化（业务层 / ART 17）效果有限，**必须综合应用**：业务层（对象池 + Bitmap 复用）+ AOSP 17（软阈值 + LOS 压缩 + Humongous Region + CAS + 后台预分配）+ 工具链（Perfetto + hprof + MAT）。**本篇 3 个案例展示了完整的综合调优方法**。
