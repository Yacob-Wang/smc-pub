### 9.8.8 【AOSP 17 新增】软阈值调优

```bash
#【AOSP 17 新增】软阈值 kSoftThresholdPercent=30%（默认）
# 调低软阈值 → Young GC 更频繁（适合分配速率稳定的应用）
# 调高软阈值 → Young GC 更少（适合大对象分配多的应用）

# 必须重新编译 ART 才能修改

# 编译时修改：
# art/runtime/options.h: static constexpr size_t kSoftThresholdPercent = 30;
# → 改为 20（更频繁）或 50（更少）
```

## 三、代码优化

### 9.8.10 减少对象分配

```java
// 1. 避免在循环中分配对象
// Bad
for (int i = 0; i < n; i++) {
    String s = String.valueOf(i);  // 每次创建 String
}

// Good
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) {
    sb.append(i);
}

// 2. 使用对象池
// Bad
new Thread(runnable).start();  // 每次创建 Thread

// Good
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>()
);
executor.execute(runnable);

// 3. 避免装箱
// Bad
Map<Integer, String> map = new HashMap<>();
for (int i = 0; i < n; i++) {
    map.put(i, "value");  // 装箱
}

// Good
Map<Integer, String> map = new HashMap<>();
int[] keys = new int[n];
String[] values = new String[n];
for (int i = 0; i < n; i++) {
    map.put(keys[i], values[i]);  // 不装箱
}

// 4. 避免 String 拼接
// Bad
String result = "";
for (String s : list) {
    result += s;  // 每次创建新 String
}

// Good
StringBuilder sb = new StringBuilder();
for (String s : list) {
    sb.append(s);
}
String result = sb.toString();
```

### 9.8.11 对象池化

```java
// Bitmap 池化（Android）
public class BitmapPool {
    private final Stack<Bitmap> pool = new Stack<>();
    
    public Bitmap obtain(int width, int height) {
        if (!pool.isEmpty()) {
            Bitmap bmp = pool.pop();
            if (bmp.getWidth() == width && bmp.getHeight() == height) {
                return bmp;
            }
            bmp.recycle();
        }
        return Bitmap.createBitmap(width, height, Bitmap.Config.ARGB_8888);
    }
    
    public void recycle(Bitmap bmp) {
        if (pool.size() < MAX_POOL_SIZE) {
            pool.push(bmp);
        } else {
            bmp.recycle();
        }
    }
}

// View 池化（RecyclerView）
public class MyViewHolder extends RecyclerView.ViewHolder {
    public MyViewHolder(View view) {
        super(view);
    }
}
```

### 9.8.13 避免 Finalize（AOSP 17 增强）

```java
// 1. 用 Cleaner 替代 finalize
// Bad
public class Resource {
    @Override
    protected void finalize() throws Throwable {
        close();
    }
}

// Good
public class Resource {
    private static final Cleaner CLEANER = Cleaner.create();
    
    private final Cleaner.Cleanable cleanable;
    
    public Resource() {
        cleanable = CLEANER.register(this, new CleanupAction());
    }
    
    private static class CleanupAction implements Runnable {
        @Override
        public void run() {
            close();
        }
    }
}

// 2. 用 AutoCloseable + try-with-resources
public class Resource implements AutoCloseable {
    @Override
    public void close() {
        // 释放资源
    }
}

try (Resource r = new Resource()) {
    // 使用资源
}
```

**AOSP 17 Finalizer 池化说明**：
```
【AOSP 17 强化】
- Finalizer 线程池化（4 线程）后，"漏的 finalize" 不再是问题
- 但"慢的 finalize" 仍是问题（每个 finalizer 都要执行）
- 最佳实践仍是"用 Cleaner 替代 finalize"

→ AOSP 17 池化主要解决"漏的 finalize"，不是"慢的 finalize"
→ 新代码严禁使用 finalize()，用 AutoCloseable + try-with-resources 替代
```

详见 [03-LeakCanary原理](03-LeakCanary原理.md) §6.2。

## 六、ART 17 治理工具（API 37+ 强化）

### 9.8.20 【ART 17 硬变化】hprof 格式变更（Class Extent 元数据）

AOSP 17 在 hprof 中**新增 Class Extent 元数据**——类去重后引用追踪的硬变化：

```
AOSP 14 hprof 格式：
  - Class 对象：每个 ClassLoader 独立的 Class 对象
  - Class Extent：每个 Class 的实例集合（按 ClassLoader 分）
  - 元数据：无

AOSP 17 hprof 格式：
  - Class 对象：类去重后，多个 ClassLoader 共享 Class
  - Class Extent：每个 Class 的实例集合（合并去重后的实例）
  -【新增】元数据：Class Deduplication Map
  -【新增】元数据：GenCC Young/Old 代标记
```

**对治理工具的影响**：
- **AOSP 14 的 MAT 1.13.0 解析 AOSP 17 hprof 会出错**——不识别 Class Extent
- **AOSP 17 的 MAT 1.14.0+ 正确解析**——支持 Class Extent 元数据
- **升级 MAT 是必需的**——AOSP 17 升级必须配套升级 MAT
- **LeakCanary 2.x 会误报"Class 泄漏"**——必须升级 LeakCanary 3.x

**架构师建议**：
- AOSP 17 升级后**必须升级 MAT 1.14.0+**——否则会误判类去重为泄漏
- 用 OQL `class.@deduplicated` 字段查询去重后的 Class
- 用 OQL `obj.@youngGen` 字段查询 GenCC Young 代对象

**源码定位**：
- `art/runtime/hprof/hprof.cc#WriteHeapDump`（AOSP 17 新增 Class Extent 元数据）
- `art/runtime/hprof/hprof.cc#WriteClassExtent`（AOSP 17 新增）
- `art/runtime/gc/class_linker.cc#ClassDeduplication`（AOSP 17 类去重实现）

详见 [04-MAT使用指南](04-MAT使用指南.md) §6.1。

### 9.8.21 【ART 17 硬变化】GC Root 快速定位

AOSP 17 在 hprof 中**新增 GC Root 索引**——找泄漏链快 5-10 倍：

```
AOSP 14 hprof：
  - GC Root 信息：每个 Root 单独记录
  - 找 Root 路径：O(n) 遍历
  - 大型 hprof（GB 级别）：几分钟

AOSP 17 hprof：
  - GC Root 信息：带索引的 Root 表
  - 找 Root 路径：O(1) 索引查询
  - 大型 hprof（GB 级别）：几秒
  -【新增】GenCC Young/Old 标记
```

**对治理工具的影响**：
- "Path to GC Roots" 功能在 AOSP 17 hprof 下快 5-10 倍
- 大型 hprof（GB 级别）也能快速定位
- MAT 1.14.0+ 利用新索引优化分析
- LeakCanary 3.x Shark 引擎也利用新索引

**实战价值**：
- 找泄漏链：从几分钟降到几秒
- 生产环境频繁 dump hprof 成为可能
- 大型 App（GB hprof）也能快速分析

**源码定位**：
- `art/runtime/hprof/hprof.cc#WriteGCRootIndex`（AOSP 17 新增）
- `art/runtime/hprof/hprof.cc#WriteHeapDump`（AOSP 17 集成）

### 9.8.22 【ART 17 + Linux 6.18 增强】hprof-conv 优化

AOSP 17 + Linux 6.18 优化 hprof-conv 转换——让 heap dump 写盘快 3 倍：

```bash
# AOSP 14：Android hprof → Java SE hprof
# 转换时间：~30 秒（1 GB hprof）
hprof-conv android.hprof java.hprof

# AOSP 17 + Linux 6.18：转换时间 ~10 秒
# 优化点：
# 1. io_uring 异步 I/O：写盘快 3x
# 2. mmap 零拷贝：大文件快 5x
# 3. sheaves slab：小对象分配快 2x
hprof-conv android.hprof java.hprof
```

**架构师解读**：
- AOSP 17 + Linux 6.18 让 hprof 转换整体快 3 倍
- 1 GB hprof 转换从 30 秒降到 10 秒
- **生产环境可频繁 dump hprof**（不影响用户体验）
- **Linux 6.18 io_uring 增强**让 heap dump 写盘延迟降低 30%

**源码定位**：
- `external/robolectric-shadows/hprof-conv/`（AOSP 17 优化）
- `kernel/io_uring.c`（Linux 6.18 io_uring 增强）
- `mm/slab_common.c`（Linux 6.18 sheaves）

### 9.8.23 【Linux 6.18 增强】heap dump 写盘优化

Linux 6.18 优化 heap dump 写盘：

```
Linux 6.18 heap dump 写盘优化：

1. io_uring 异步 I/O
   - 传统：同步 write（每次写盘都阻塞）
   - Linux 6.18：io_uring 异步 write（批量写盘）
   - 效果：heap dump 写盘延迟降低 30%

2. mmap 零拷贝
   - 传统：read + write（两次拷贝）
   - Linux 6.18：mmap + sendfile（零拷贝）
   - 效果：大文件写盘快 5 倍

3. sheaves slab
   - 传统：malloc 分配（碎片化）
   - Linux 6.18：sheaves slab（碎片化少）
   - 效果：hprof 转换中小对象分配快 2 倍

→ 让 AOSP 17 hprof 治理工具在生产环境可用
```

**架构师解读**：
- **生产环境友好**：heap dump 写盘延迟降低 30%，不再影响用户体验
- **频繁 dump 成为可能**：可以每分钟 dump 一次
- **大 hprof 友好**：GB 级别 hprof 也能快速处理

**源码定位**：
- `kernel/io_uring.c`（Linux 6.18 io_uring 增强）
- `mm/filemap.c`（Linux 6.18 mmap 零拷贝）
- `mm/slab_common.c`（Linux 6.18 sheaves）

## 八、治理的优先级

### 9.8.29 优先级评估

```
治理优先级：

P0（立即）：
- OOM
- 长 STW（> 100ms）
- 严重内存泄漏

P1（紧急）：
- GC 频繁（> 50 次/分钟）
- 中等卡顿（50-100ms）

P2（重要）：
- GC 频率中等（10-50 次/分钟）
- 轻微卡顿（16-50ms）

P3（优化）：
- GC 频率低（< 10 次/分钟）
- 偶发卡顿

→ 按优先级分配治理资源
```

### 9.8.30 治理的 ROI

```
ROI（投入产出比）：

高 ROI（先做）：
- 减少对象分配（成本低，收益大）
- 避免 Finalize（成本低，收益大）
- 监控告警（成本中，收益高）

中 ROI（接着做）：
- 调大堆（成本低，收益中）
- 对象池化（成本中，收益中）

低 ROI（后做）：
- CC GC 调优（成本高，收益中）
- 升级到 GenCC（成本高，收益高，但需要平台升级）

→ 按 ROI 分配资源
```

## 十、总结（架构师视角的 5 条 Takeaway）

1. **治理工具箱是 GC 治理的"武器库"**——四大手段（参数调优/代码优化/监控告警/工具链）+ 治理决策树。**没有工具箱就没有治理**——所有优化都要有配套工具。详见 §1 + [07-监控指标体系](07-监控指标体系.md)（重写为 v2 升级版）。

2. **AOSP 17 hprof 格式变更是硬性升级要求**——Class Extent 元数据 + GenCC Young/Old 标记。**旧版 MAT 1.13.0 解析 AOSP 17 hprof 会出错**——AOSP 17 升级必须配套升级 MAT 1.14.0+。详见 §6.1 + [04-MAT使用指南](04-MAT使用指南.md) §6.1。

3. **AOSP 17 GC Root 快速定位是泄漏分析利器**——hprof 中新增 GC Root 索引，找泄漏链快 5-10 倍。**大 hprof（GB 级别）也能快速分析**——生产环境频繁 dump 成为可能。详见 §6.2 + [03-LeakCanary原理](03-LeakCanary原理.md)（重写为 v2 升级版）。

4. **AOSP 17 + Linux 6.18 让 hprof-conv 转换快 3 倍**——io_uring 异步 I/O + mmap 零拷贝 + sheaves slab。**生产环境可频繁 dump hprof**（不影响用户体验）。详见 §6.3。

5. **AOSP 17 软阈值 kSoftThresholdPercent=30% 是 Young GC 的"轻量预警"**——通过 [07-监控指标体系](07-监控指标体系.md) 的软阈值告警，**软阈值频繁触发不一定是泄漏**——可能是堆太小或分配过快。详见 §2.8 + [10-ART17分代GC强化专章 v2](../../03-GC系统/10-ART17分代GC强化专章-v2.md) §3。
