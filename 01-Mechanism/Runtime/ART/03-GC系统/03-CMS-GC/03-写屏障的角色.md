## 一、漏标的本质

### 1.1 什么是漏标

**漏标**（漏活对象）= **本应存活的对象被错误地判定为死亡**。

```cpp
// 时序问题
// T1: GC 扫描 obj_A，将 obj_A 染黑（已完成扫描）
// T2: 业务线程：obj_A.field = null（断开 obj_A → obj_C 的引用）
// T3: 业务线程：obj_B.field = obj_C（建立 obj_B → obj_C 的引用）
//     但 obj_B 也已经被 GC 扫描（染黑）
// T4: GC 完成，obj_C 仍是白色 → 被回收
// T5: 业务线程访问 obj_C → 野指针 / 崩溃
```

### 1.2 漏标的充要条件（Wilson 1992）

**Wilson 1992 年证明**：漏标发生的充要条件是 **同时满足**：

1. 业务线程**插入**了一个从**黑色对象**到**白色对象**的新引用
2. 业务线程**删除**了从**灰色对象**到这个白色对象的所有可达路径

```cpp
// 漏标的两个条件（缺一不可）
条件 1: black_obj.field = white_obj  // 黑色引用白色（漏标源头）
条件 2: gray_obj.field = null  // 灰色断开引用（让 white_obj 不可达）
```

### 1.3 漏标的具体例子

```
初始状态（GC 已扫描部分）：
  obj_A = 黑色（已扫描）
  obj_B = 灰色（待扫描）
  obj_C = 白色（未扫描）

  obj_A.field = obj_C
  obj_B.field = obj_C

GC 继续扫描：
  扫描 obj_B（灰色）→ obj_B 引用 obj_C → obj_C 染灰

漏标场景：
  T1: 业务线程：obj_A.field = null（断开 A → C）
      写屏障：PreWriteBarrier 拦截
      → 把 obj_C（被断开的对象）染灰

  T2: 业务线程：obj_B.field = obj_C（建立 B → C）
      → obj_C 已经被染灰，OK

  T3: GC 扫描 obj_B → obj_C 已经在 mark stack → 继续扫描 obj_C
      → obj_C 存活，正确标记

如果不写屏障：
  T1': obj_A.field = null（无拦截）→ obj_C 仍是白色
  T2': obj_B.field = obj_C → obj_C 仍是白色
  T3': GC 扫描 obj_B（已染黑）→ 不再扫描 obj_B 的引用
       obj_C 永远白色 → 漏标 → 回收 → 崩溃
```

### 1.4 漏标的后果

**业务线程访问已回收对象**：
```cpp
Object* obj_C = some_list.get(0);
// 此时 obj_C 已被 GC 回收（漏标）
// obj_C 指向已回收的内存 → 野指针
// 访问 obj_C.field → 崩溃 / 内存错误 / 安全漏洞
```

→ **漏标是 GC 最严重的正确性问题之一**。

### 1.5 漏标的"症状"

- 业务代码访问对象时 NPE（对象已被回收）
- Crash trace 显示"对象地址 0xABCD 已被回收"
- Heap dump 不显示该对象
- 偶发（与 GC 时机相关）

## 二、CMS 的 Pre-Write Barrier（Incremental Update）

### 2.1 Pre-Write Barrier 的定义

CMS 用 **Pre-Write Barrier**（增量更新）防止漏标：

```cpp
// 业务代码
obj.field = new_value;

// CMS 实际执行（写屏障插入后）
PreWriteBarrier(obj, field_offset, new_value);  // 写屏障：标记旧值
obj.field = new_value;                          // 真正的赋值
```

### 2.2 Pre-Write Barrier 的核心逻辑

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::WriteBarrier
void MarkSweep::WriteBarrier(mirror::Object* obj, MemberOffset offset,
                              mirror::Object* new_value) {
    // 1. 读出旧值
    mirror::Object* old_value = obj->GetFieldObject<mirror::Object>(offset);

    // 2. 把旧值重新染灰（Incremental Update）
    if (old_value != nullptr) {
        // 把"被断开引用的对象"重新加入 mark stack
        MarkObject(old_value);
    }

    // 3. 真正的赋值由调用方完成
}
```

### 2.3 Incremental Update 的逻辑

**Incremental Update** 的精髓：**当黑色对象断开引用时，把被断开的对象"拯救"为灰色**。

```
场景：black_obj.field = null（断开引用）

Pre-Write Barrier 拦截：
1. 读出 old_value（即将被覆盖的引用）
2. 把 old_value 重新染灰（加入 mark stack）

下次 GC 扫描 mark stack：
→ 重新扫描 old_value
→ old_value 的引用都被检查
→ 即使 old_value 已经不被 black_obj 引用，也能被其他对象引用
→ old_value 不漏标
```

### 2.4 Pre-Write Barrier 的源码

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::MarkObject
void MarkSweep::MarkObject(mirror::Object* obj) {
    // 1. 检查是否已标记
    if (mark_bitmap_->Test(obj)) {
        return;  // 已标记，不重复
    }

    // 2. 设置 mark bit
    mark_bitmap_->Set(obj);

    // 3. 加入 mark stack（等下一轮扫描）
    mark_stack_->Push(obj);
}
```

### 2.5 Pre-Write Barrier 的执行时机

**关键**：Pre-Write Barrier 在赋值 **之前** 执行。

```
时间线：
T1: 业务线程调用 obj.field = new_value
T2: Pre-Write Barrier 执行（读 old_value + 染灰）
T3: 真正的赋值 obj.field = new_value
T4: Post-Write Barrier（CMS 不使用，仅 GenCC 用于 Card Table）
```

### 2.6 Pre-Write Barrier 的开销

**每次指针赋值都有开销**：

```cpp
// 简单赋值：obj.field = new_value
// 优化前：
str x1, [x0, #offset]    ; 真正的赋值（1 条指令）

// 优化后（带 Pre-Write Barrier）：
ldr x2, [x0, #offset]    ; 读旧值（1 条指令）
cbz x2, .Lskip            ; 检查旧值是否为 null（1 条指令）
bl artWriteBarrier         ; 调用写屏障（1 条指令）
.Lskip:
str x1, [x0, #offset]    ; 真正的赋值（1 条指令）
```

**开销**：每次指针赋值多 3 条指令（约 5-10ns）。

## 三、CMS 写屏障如何防漏标

### 3.1 防漏标的关键

**关键洞察**：漏标的两个条件必须 **同时满足**。写屏障只要 **破坏其中一个条件**，漏标就不会发生。

CMS Pre-Write Barrier 破坏的是 **条件 1**：
```
条件 1: black_obj.field = white_obj（黑色引用白色）

CMS 写屏障：在赋值前，把 old_value（被断开的对象）染灰
          → 即使后续赋值 obj.field = new_value
          → 被断开的对象（旧值）已染灰
          → GC 下一轮会扫描它
```

### 3.2 完整时序示例

```
初始状态：
  obj_A = 黑色（已扫描完成）
  obj_B = 黑色（已扫描完成）
  obj_C = 白色（未扫描）

业务线程：执行 obj_B.field = obj_C（建立新引用）
         但 obj_A.field 也要断开

CMS 写屏障的介入：
T1: 业务线程：obj_A.field = null（断开 A → C）
    Pre-Write Barrier 拦截：
    → 读 old_value = obj_C
    → 把 obj_C 染灰（加入 mark stack）

T2: 真正的赋值 obj_A.field = null

T3: 业务线程：obj_B.field = obj_C（建立 B → C）
    Pre-Write Barrier 拦截：
    → 读 old_value = null（无需染灰）
    → 真正的赋值

T4: GC 完成 Remark
    → 发现 obj_C 在 mark stack
    → 重新扫描 obj_C
    → obj_C 的引用都被检查
    → obj_C 存活，正确标记

结论：漏标被 CMS 写屏障阻止！
```

### 3.3 漏标的边界情况

**情况 1：业务线程修改不触发写屏障**

```cpp
// 如果业务线程绕过写屏障
// 例如直接修改内存（Unsafe）
unsafe.putObject(obj, offset, new_value);
// → 没有 Pre-Write Barrier
// → 漏标风险
```

**修复**：ART 11+ 引入了 **JIT 代码校验**，检测绕过写屏障的代码。

**情况 2：写屏障失效**

```cpp
// 极端并发场景
// 写屏障未生效（多线程竞争）
// → 极小概率漏标
// → ART 通过自愈机制减少影响
```

**修复**：ART 通过 **多次 GC** + **STW 校验** 减少漏标概率。

### 3.4 CMS 写屏障的局限性

**局限 1：开销大**

每次指针赋值都有 3 条额外指令。循环内多次赋值开销显著。

**局限 2：不能完全防漏标**

极端并发场景下仍可能漏标（虽然概率极小）。

**局限 3：与 JIT 代码集成复杂**

JIT 编译的代码需要手动插入写屏障调用 → 编译复杂度高。

## 四、CMS 写屏障 vs 其他 GC 的写屏障

### 4.1 三种写屏障对比

| GC | 写屏障类型 | 维护的不变式 | 性能开销 |
|:---|:---|:---|:---|
| **CMS** | Pre-Write Barrier（Incremental Update） | 强三色不变式 | 中（每次指针赋值） |
| **G1** | Pre-Write Barrier（SATB） | 弱三色不变式 | 中 |
| **GenCC** | Post-Write Barrier（Card Table） | 弱三色不变式 | 中 |

### 4.2 Incremental Update vs SATB

**Incremental Update（CMS 用）**：
- 关注"被删除的引用"
- 旧值染灰

**SATB（Snapshot At The Beginning，G1 用）**：
- 关注"新建立的引用"
- 记录新值，等 Remark 重新扫描

### 4.3 CMS 写屏障的工程选择

CMS 选 Incremental Update 的原因：

| 优势 | 说明 |
|:---|:---|
| **实现简单** | Pre-Write Barrier 只读旧值 |
| **正确性保证** | 强三色不变式的形式化证明 |
| **工程成熟** | ART 5-7 长期使用 |
| **跨平台支持** | x86 / ARM / ARM64 一致 |

## 五、CMS 写屏障在编译码中的实现

### 5.1 解释器模式的写屏障

```cpp
// art/runtime/interpreter/interpreter.cc
void Interpreter::DoFieldPut(Thread* self, const Instruction* inst, ...) {
    // 1. 解析指令
    // ...

    // 2. 调用写屏障
    if (kUseWriteBarrier) {
        WriteBarrier::WriteField(obj, offset, new_value);
    }

    // 3. 真正的赋值
    obj->SetFieldObject(offset, new_value);
}
```

### 5.2 AOT 编译码的写屏障

```asm
; AArch64 写屏障的实现
; 入口：x0 = obj, x1 = offset, x2 = new_value
pre_write_barrier_entry:
    ; 1. 保存现场
    stp x29, x30, [sp, #-16]!
    mov x29, sp

    ; 2. 计算字段地址
    add x3, x0, x1                ; x3 = obj + offset

    ; 3. 读旧值
    ldr x4, [x3]                  ; x4 = *obj.offset

    ; 4. 检查旧值
    cbz x4, .Lskip                 ; 如果 null，跳过

    ; 5. 调用写屏障
    bl artWriteBarrier

.Lskip:
    ; 6. 恢复现场
    ldp x29, x30, [sp], #16
    ret
```

### 5.3 JIT 编译码的写屏障

ART JIT 编译器会做以下优化：

1. **冗余消除**：连续的写屏障合并
2. **死代码消除**：明显不需要写屏障的代码
3. **内联**：简单写屏障内联到调用点

```cpp
// 优化前（每次循环都触发）
for (int i = 0; i < N; i++) {
    WriteBarrier(&arr[i]);        // 写屏障
    arr[i] = value;
}

// 优化后（循环外判断）
for (int i = 0; i < N; i++) {
    if (i % 100 == 0) {           // 抽样写屏障
        WriteBarrier(&arr[i]);
    }
    arr[i] = value;
}
```

## 六、Mod Union Table 优化

### 6.1 Mod Union Table 的定义

**Mod Union Table**（修改并集表）是 ART 17 引入的关键优化，记录"哪些 Card 被修改过"，让 CMS 在 Concurrent Mark 期间增量处理。

```cpp
// art/runtime/gc/space/mod_union_table.h
class ModUnionTable {
 public:
  // 记录 Card 被修改
  void MarkCardDirty(const uint8_t* card);

  // 获取所有 dirty card
  void GetDirtyCards(std::vector<uint8_t*>* cards);

  // 并发标记期间增量处理
  void ProcessCards();

 private:
  std::vector<uint8_t*> dirty_cards_;
  CardTable* card_table_;
};
```

### 6.2 Mod Union Table 的工作原理

```
┌────────────────────────────────────────────────────────────────┐
│ Mod Union Table 工作原理（ART 17）                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  CMS 时代（AOSP 14）：                                          │
│    └─ 写屏障记录"哪些对象被修改"                                 │
│    └─ Remark 阶段遍历所有 dirty 对象                              │
│    └─ 时间复杂度：O(dirty 对象数)                                 │
│                                                                │
│  ART 17 优化：                                                  │
│    ├─ 写屏障记录"哪些 Card 被修改"（Card = 256 字节内存块）       │
│    ├─ Remark 阶段只处理 dirty card（而不是所有对象）              │
│    ├─ 时间复杂度：O(dirty card 数) << O(dirty 对象数)             │
│    ├─ Card Table 压缩：256B → 64B                                │
│    └─ Remark STW 从 50ms+ 降至 20-30ms                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 6.3 Mod Union Table 的优势

| 维度 | dirty 对象记录（CMS 时代） | dirty card 记录（ART 17） |
|:---|:---|:---|
| **记录粒度** | 单个对象 | 256 字节内存块（压缩后 64B） |
| **记录数量** | 100K 对象 = 100K 条 | 100K 对象 ≈ 10K card |
| **Remark 耗时** | 50ms+ | 20-30ms |
| **空间开销** | 8 字节/对象 | 1 字节/card |
| **时间复杂度** | O(n) | O(√n) |

**架构师视角**：Mod Union Table 让 Remark STW 从 O(对象数) 降到 O(Card 数)，**性能提升 50-70%**。

### 6.4 Mod Union Table 的源码

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::RemarkPhase
void MarkSweep::RemarkPhase() {
    // 1. 暂停所有 mutator 线程（STW）
    SuspendAllThreads();

    // 2. 处理 dirty card（ART 17 优化）
    std::vector<uint8_t*> dirty_cards;
    mod_union_table_->GetDirtyCards(&dirty_cards);

    for (uint8_t* card : dirty_cards) {
        // 3. 扫描 card 内的所有对象
        for (mirror::Object* obj : CardTable::GetObjectsInCard(card)) {
            // 重新扫描这个对象
            obj->VisitReferences([this](mirror::Object* ref) {
                if (ref != nullptr) {
                    MarkObjectParallel(ref);
                }
            });
        }
    }

    // 4. 清理 dirty card 列表
    mod_union_table_->ClearDirtyCards();
}
```

## 七、ART 17 硬变化专章

### 7.1 ART 17 写屏障 Incremental Update 策略强化

AOSP 17 对 Incremental Update 策略做了强化：

- **CMS 时代**：每次指针赋值 3 条额外指令（读旧值 + 检查 + 调用）
- **ART 17 强化**：
  - 写屏障 + Card Table 协同，dirty card 数量减少 50-70%
  - 写屏障指令从 3 条降至 1-2 条（编译期内联）
  - 寄存器分配优化，写屏障总开销降低 30%
  - **漏标概率降低 30-50%**

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §5。

### 7.2 ART 17 Mod Union Table 优化

AOSP 17 引入 Mod Union Table 优化：

- **CMS 时代**：Remark 阶段遍历所有 dirty 对象，时间复杂度 O(n)
- **ART 17 优化**：
  - 写屏障记录"哪些 Card 被修改"（Card = 256 字节内存块）
  - Remark 阶段只处理 dirty card，时间复杂度 O(√n)
  - Card Table 压缩：256B → 64B
  - **Remark STW 从 50ms+ 降至 20-30ms**

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §5。

## 八、CMS 写屏障的工程坑点

### 8.1 坑点 1：Hook 框架绕过写屏障

**场景**：
```cpp
// Hook 框架代码
void* HookMethod(void* method, void* new_entrypoint) {
    ArtMethod* art_method = reinterpret_cast<ArtMethod*>(method);
    // 直接修改 entrypoint，不调用写屏障
    art_method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}

// CMS 在并发标记期间扫描 ArtMethod
// 业务线程修改了 ArtMethod.entrypoint
// 但没有触发写屏障
// → CMS 不记录这个修改
// → ArtMethod 可能被错误回收（如果没被 GC Root 引用）
// → 崩溃
```

**修复**：ART 13+ 引入了 **JIT 代码校验**，部分解决此问题。

### 8.2 坑点 2：JNI 直接修改对象字段

**场景**：
```cpp
// JNI 代码直接修改字段
void JNI修改字段(JNIEnv* env, jobject obj, jstring new_value) {
    // 直接写字段内存
    char* field_addr = reinterpret_cast<char*>(obj) + offset;
    *reinterpret_cast<jstring*>(field_addr) = new_value;
    // ❌ 没调用写屏障！
}
```

**修复**：
```cpp
// 正确做法
void JNI修改字段(JNIEnv* env, jobject obj, jstring new_value) {
    // 调用 JNI 接口，内部自动调用写屏障
    env->SetObjectField(obj, fieldID, new_value);
}
```

### 8.3 坑点 3：反射 Unsafe 操作

**场景**：
```java
// Unsafe 操作不调用写屏障
Object value = unsafe.getObject(obj, offset);
unsafe.putObject(obj, offset, new_value);
// ❌ 风险：CMS 并发标记期间可能漏标
```

**修复**：用 `Field.set()` 替代 `Unsafe.putObject()`。

### 8.4 坑点 4：ART 17 Mod Union Table 边界

**ART 17 Mod Union Table 的边界情况**：

```
┌────────────────────────────────────────────────────────────────┐
│ Mod Union Table 边界情况                                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. Card 跨多个对象                                              │
│    └─ 1 个 dirty card 可能含多个对象                              │
│    └─ 需要遍历 card 内所有对象                                    │
│                                                                │
│  2. Card 与对象大小不匹配                                         │
│    └─ 大对象（LOS）占用多个 card                                  │
│    └─ 单独处理（不计入 Mod Union Table）                          │
│                                                                │
│  3. Card 压缩的精度损失                                          │
│    └─ 256B → 64B 压缩后，边界对象可能漏标                        │
│    └─ ART 17 用保守策略避免                                      │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 8.5 漏标问题快速决策树

```
NPE / 对象已被回收
  ↓
看 Crash 时间点 + GC log
  ↓
├─ 并发标记期间 + 业务线程修改引用
│   └─ 漏标 → 检查屏障 / 升级 ART 17 Mod Union Table
│
├─ finalize() 期间
│   └─ 复活对象 → 禁止在 finalize 中复活
│
├─ JNI / Native 代码
│   └─ 检查 DeleteGlobalRef / SetObjectField
│
├─ 反射修改 static final
│   └─ ART 17 失效 → 改用可变容器
│
└─ 跨 ClassLoader 引用
    └─ ART 17 类去重 → 检查 ClassLoader 隔离
```

## 十一、总结（架构师视角的 5 条 Takeaway）

1. **漏标 = 本应存活的对象被错误回收**。**Wilson 1992 证明**：漏标需要两个条件同时发生——黑色对象建立新引用 + 灰色对象断开引用。**写屏障 + 读屏障是防护机制**。
2. **CMS 用 Pre-Write Barrier（Incremental Update）防漏标**——把被断开的引用对象重新染灰。**强三色不变式的形式化保证**。**每次指针赋值多 3 条指令**（约 5-10ns）。
3. **ART 17 Mod Union Table 优化**——写屏障记录"哪些 Card 被修改"（而不是所有对象），**Remark STW 从 50ms+ 降至 20-30ms**。**Card Table 压缩：256B → 64B**。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §5。
4. **ART 17 写屏障指令开销降低**——编译期内联，从 +3 条降至 +1-2 条。**大型 App 性能提升 5-10%**。**漏标概率降低 30-50%**。
5. **写屏障局限性的根治靠代码**——Hook 框架用 ART 提供的 WriteBarrier API，JNI 用 SetObjectField，**禁止 Unsafe.putObject 绕过写屏障**。**ART 17 强化了反射 / Unsafe 的屏障覆盖**。
