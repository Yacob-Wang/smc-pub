## 一、核心原则

````
你写的不是科普文,不是面试八股,而是给【资深架构师】看的技术深度文章。

1. 言必有据,码必有源:每个技术论点标注源码文件路径,关键流程贴核心源码片段;
   每篇文章末尾给出"源码路径对账表",逐条标注【已校对 / 待确认】。
2. 自顶向下:每个章节先讲"是什么、为什么需要它",建立上下文后再深入源码。禁止上来就贴代码。
3. 关联实战:每个知识点关联到真实工程问题(Crash/ANR/OOM/性能退化等),
   说清楚"这个机制会在什么场景下咬你一口"。
4. 实战案例收尾:每篇末尾 1-2 个完整线上问题排查案例(现象→分析→根因→修复),
   案例可验证(含 dump/日志片段、版本号、复现步骤)。
5. 内容量级:每篇不少于 300 行(约 8000-15000 字)。
6. 版本基线(多版本矩阵):
   - 应用层 / Framework:以 Android 17(AOSP android-17.0.0_r1,代号 CinnamonBun)为主线
     (注意:不是 README 课纲写的"Android 14"——v5 写时是 A14,v6 已升级到 A17)
   - Linux 内核:以 android17-6.18 / android17-6.19 / android14-6.1 / android15-6.6 为基线
   - 涉及 GKI 厂商分支差异时,标注"厂商 GKI"或具体 vendor tag
   - 涉及历史演进或多版本差异时,标注版本范围
   - **AOSP 17 发布日硬性事实**(v6 新增,沿用 14/15 篇 verifier 校准):
     Beta 1 2026-02-13 + 正式版 2026-05~06 + QPR1 2026-09 计划
     (不是 2025-11-30,这是错的)
   - **MGLRU 引入版本硬性事实**:Linux 5.9(commit ccd2a0d4, 2020-10)
     (不是 5.10,5.10 是 GKI 基线不是合并版本)
7. 工程基线:涉及可调参数(timeout、buffer size、maxevents、优先级等)时,
   必须给出"工程默认值"与"选用准则",说清楚"线上我应该用啥、为啥"。
8. **不臆想硬性要求**(v6 强化,见 §11):
   - 5 件套案例必须有真实数据(环境/现象/分析/根因/修复/修复后验证)
   - 6 大路径每个都要标可信度标签(✅ 已公开 / 🟡 待确认 / ❌ 推测)
   - 真实数据拿不到 → 标 🟡 + 写"待查证",不编造
````

## 二、系列文章设计流程

### 第一步:定义角色与目标

````
我的角色是 Android 稳定性高级工程师。

我要系统学习【填入模块名称,如:Binder / SurfaceFlinger / Linux 调度器 / epoll】。

学习目标不是"了解原理",而是:
- 能在遇到线上问题时,快速定位到该模块的对应子系统和源码文件
- 能理解问题的根因机制
- 能给出治理方案或监控手段
````

### 第二步:让 AI 先输出大纲(不要直接写文章)

````
请按架构师思维链为【模块名称】设计系列文章大纲:

  它是什么?解决什么问题?(定位)
      ↓
  它在系统中处于什么位置?和谁协作?(边界与交互)
      ↓
  它内部是怎么运转的?(核心机制)
      ↓
  它会在什么地方出问题?(风险地图)
      ↓
  出了问题我怎么查?怎么防?(诊断与治理)

大纲要求:
1. 分为 3-5 个篇章(全局观 → 核心机制 → 跨模块交互 → 实战/治理)
2. 每篇文章列出具体章节,说清楚:讲什么内容、对应哪个源码文件/目录(含内核版本基线)、关联什么工程问题
3. 用表格形式呈现每篇文章的章节规划
4. 明确标注与其他系列的交叉引用点
5. 明确标注"本篇定位":在本系列中属于什么角色(全局观/核心机制/横切专题/风险/治理),
   强依赖哪几篇已发布文章
````

### 第三步:生成系列 README

大纲审核通过后,生成系列 README(为什么要写 / 系列设计思路 / 章节规划表 / 跨系列引用 / 阅读建议)。模板按需展开,非核心。

### 第四步:逐篇撰写(使用一站式模板)

大纲和 README 审核通过后,用下方**一站式写作模板**逐篇撰写。每次只需替换 `【】` 中的内容。

**v6 强化**(实战胜率提升):
- 写大长篇(8000+ 字)前先估算 token 余量
- 子线程写时如果 token 即将耗尽,**先 commit 5 段前言 + §1-§2** 再退出
- 主线程接力时按"前次 commit 末尾行号 + 续写 §3-§5"分批,每批 30-50KB
- 写完后**主线程自己**跑 verify_*.py 6 项(不信子线程自检报告)

## 三、一站式写作模板

> v6 改:① marker 嵌套禁令 ② 5 段前言 + 自检报告 分两个独立 AUTHOR_ONLY 段 ③ 加 4 维度 + 5 件套 self-check

````markdown
<!-- AUTHOR_ONLY:START -->
# 本篇定位(强制开头段)
- 本篇系列角色:【全局观 / 核心机制 / 横切专题 / 风险地图 / 诊断治理】
- 强依赖:必须先读【文章 A】的【章节 X】、【文章 B】的【章节 Y】
- 承接自:上一篇【文章 C】已覆盖【简述】,本篇不重复展开
- 衔接去:下一篇【文章 D】将覆盖【简述】,本篇末尾会预告
- 不重复内容:与本系列已发布/已规划文章中【哪些内容】的边界声明(一句话)

# 校准决策日志(3 轮校准后填写,见 §6)
| 轮次 | 类别 | 决策 | 理由 | 影响范围 |
|------|------|------|------|----------|
| 1 | 结构 | 【本轮决定】 | 【反例 #X / §X 规则】 | 【本篇/全文 N 处】 |
| 2 | 硬伤 | 【本轮决定】 | 【反例 #X / §X 规则】 | 【本篇/全文 N 处】 |
| 3 | 锐度 | 【本轮决定】 | 【反例 #X / §X 规则】 | 【本篇/全文 N 处】 |
````

**v6 marker 规范硬性要求**(实战教训):
1. **5 段作者前言**(本篇定位 / 校准决策日志 / 角色设定 / 上下文)用**一个** AUTHOR_ONLY 段包裹
2. **自检报告**用**另一个独立** AUTHOR_ONLY 段包裹(不嵌在 5 段前言内)
3. marker **必须成对**:`<!-- AUTHOR_ONLY:START -->` 必须有对应 `<!-- AUTHOR_ONLY:END -->`
4. **禁止嵌套**:5 段前言内部不要再嵌 START/END(实战多次踩坑)
5. **禁止新 marker 标签**:只能用 `:START/:END`,**绝对不能**发明 `:SELFCHECK` / `:begin` / `:end` / `:REVIEW` 等变体
   (v5 §9.4 剥离脚本只匹配 `:START/:END`,其他变体不会剥 → 泄露到公开站)

### 3.1 写作标准(硬性要求)

1. 目标读者:资深架构师,不是初学者。不解释基础概念(如什么是线程、什么是指针),
   解释该模块特有的术语和机制。
2. 每个章节先讲"这个东西是什么、为什么需要它、解决什么问题",然后再深入源码。
3. 涉及源码时:
   - 标注源码文件路径(如 `frameworks/native/services/inputflinger/dispatcher/InputDispatcher.cpp`
     或 `fs/eventpoll.c`),**并标注内核版本基线**
   - 只贴核心逻辑,不要全贴
   - 贴代码前先用自然语言解释这段代码要干什么
   - 贴代码后紧跟"稳定性架构师视角"分析
   - 不确定的路径用 `// 路径待确认` 标注,并在"源码路径对账表"中标记【待确认】
4. 每个技术点关联到实际工程问题(Crash、ANR、OOM、JE、NE、Watchdog、性能退化、兼容性问题),
   说清楚"它会在什么场景下咬你一口"。
5. 涉及量化描述时,给出数量级("ms 级""秒级""占比 X%"),禁止"大约""通常"等模糊用词。
   量化依据填入"量化数据自检表"。
6. 源码版本基线:同核心原则 #6
7. 工程基线要求:涉及可调参数时,给出"工程默认值"与"选用准则"。
8. 文章长度不少于 300 行(约 8000-15000 字)。
9. **v6 新增 4 维度强制结构**(针对"未来方向 / 横切专题"型文章):
   - 现状(2-3 段,讲当前实现)
   - 公开(2-3 段,讲已公开的演进方向 + source)
   - 挑战(2-3 段,讲落地难点)
   - 视角(2-3 段,讲架构师应该怎么应对)
   每条路径必须填全 4 维度,缺一不可。
10. **v6 新增 5 件套案例 self-check**(每条案例必填):
    - [ ] 环境(Android 版本 / 内核版本 / 设备 / 复现步骤)
    - [ ] 现象(关键 logcat / dmesg / systrace 片段)
    - [ ] 分析思路(排查路径,推理链)
    - [ ] 根因(具体到源码文件 + 方法 + 行)
    - [ ] 修复(修复 commit 或配置 diff + 修复原理)
    - [ ] 修复后验证(回归测试 / 灰度数据)

### 3.2 章节结构

```
┌─ 背景与定义(是什么、为什么需要它、解决什么问题)
│     不要假设读者已经知道,先用 1-2 段建立上下文
│
├─ 架构与交互(在系统中的位置、上下游依赖、数据流向)
│     用 ASCII Art 画架构图或层级图,说清楚边界
│
├─ 核心机制与源码(关键数据结构、核心流程、源码走读)
│     每段源码标注 AOSP/内核 源码路径 + 内核版本基线
│     贴代码前用自然语言解释这段代码要干什么
│
├─ 风险地图(这个机制会在哪些场景下出问题)
│     列出具体的异常类型、日志特征、触发条件
│
├─ 实战案例(1-2 个线上问题的完整排查过程)
│     **案例 5 件套**(v6 强制,见 §3.1 第 10 条 self-check)
│     案例标注类型:"典型模式" 或 "真实案例(来源:...)"
│
├─ 总结(3-5 条架构师视角的关键 Takeaway)
│
├─ 附录 A:核心源码路径索引(表格)
├─ 附录 B:源码路径对账表(强制,逐条对账)
├─ 附录 C:量化数据自检表(强制)
├─ 附录 D:工程基线表(若涉及可调参数,4 列强制)
│     **4 列定义**(砍 1 列都属违反硬性要求):
│     | 参数 | 典型默认 | 选用准则 | 踩坑提醒 |
│     详见 §5 反例 #7(反例库指向此处)
└─ 篇尾衔接:用 1-2 句说明下一篇将深入什么内容
```

### 3.3 图表格式

- 架构图/层级图/流程图:统一使用 ASCII Art(纯文本方框图)
- 对比信息:Markdown 表格
- 时序说明:缩进文本 + 箭头符号

### 3.4 图表密度规则

- **平均每 1500 字 1 张图**
- **单篇 4-6 张图(不含源码中的小型 ASCII 示意)**
- 上限 6,下限 4,表格不计入图数
- **例外**(破例适用,按 §8 处理):横切专题型可放宽到 3 张;演进型可放宽到 3 张 + 1-2 张对比表

### 3.5 跨模块引用规范(v6 强化)

- 涉及本系列其他篇章时,用 `[文章标题](文件名.md)` 引用,不重复展开
- 涉及其他系列时,用 `[系列名-文章标题](相对路径)` 引用,只用 1-2 句概述核心结论
- 涉及其他系列未覆盖的内容时,简要说明核心逻辑(3-5 句),不要深入展开
- 涉及引用时,必须在文章开头"本篇定位"中显式声明
- **v6 新增:跨篇 markdown 链接必须用全角冒号 `:`(U+FF1A)**
  - 错误:`[15-未来方向:基于真实信息的6大演进路径](15-未来方向:基于真实信息的6大演进路径.md)`
  - 正确:`[15-未来方向:基于真实信息的6大演进路径](15-未来方向:基于真实信息的6大演进路径.md)`
  - 原因:Windows 文件系统不区分全/半角冒号,但 mkdocs 渲染时 URL 编码不同 → 半角冒号 404
  - 工具:见 §13.4 verify_colon.py
- **v6 新增:跨篇引用命名一致性**:
  - 文件命名必须跟其他文件实际命名一致(实战踩坑:04 用 `bionic-scudo` 无连字符 vs 写时打成 `bionic-scudo` 有连字符)
  - 引用前先 ls 实际文件,不用记忆命名

### 3.6 禁止事项

1. 禁止挖坑不填("我们将在后续文章详细讲"→当场讲清或显式指向)
2. 禁止数据堆砌(每个数字后必须有"所以呢")
3. 禁止 AI 自嗨("非常精妙""体现了……深度融合"→删,v6 词表见 §5.3)
4. 禁止模糊量化("通常""大约"→给具体数字 + 来源)
5. 禁止跨篇重复(已在其他系列讲过的内容,本系列只引用不展开)
6. **v6 新增**:禁止编造路径 / 类名 / 作者 / 时间线(详见 §12 已知幻觉清单)
7. **v6 新增**:禁止自检报告使用 `:SELFCHECK` 等非标准 marker 标签(见 §9.2)
8. **v6 新增**:禁止在 AUTHOR_ONLY 段外出现"v5/v6 §X"等内部规范号(详见 §9.3 扩展剥离范围)

## 四、质量检查清单(26 项 · 4 维度)

### 4.1 内容质量(10 项)

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 回答"是什么"了吗 | 文章开头 2 段内让读者明白这个东西的定位和职责 |
| 2 | 回答"为什么"了吗 | 解释设计选择的背景和动机,不只描述现状 |
| 3 | 有架构图/层级图吗 | 至少一张 ASCII Art 图说明模块在系统中的位置 |
| 4 | 源码标了路径+版本基线吗 | 每段源码标注文件路径 + AOSP/内核版本基线 |
| 5 | 源码前有上下文吗 | 每段源码前有自然语言解释"这段代码要干什么" |
| 6 | 关联实际问题了吗 | 每个知识点能回答"这和我排查线上问题有什么关系" |
| 7 | 有实战案例吗 | 篇末 1-2 个完整排查案例(现象→分析→根因→修复) |

## 六、校准与迭代机制

> 沿用 v5,补充 v6 实战 2 个新机制:
> ① 校准后必须跑 verify_*.py 6 项(不信子线程自检)
> ② 校准后必须 git commit(不积累 dirty 文件)

### 6.1 AI 输出的三个天然倾向(沿用 v5)

1. 平滑化:倾向于写"看起来都对"的内容,但缺乏锐度
2. 冗余化:同一个意思换 3 种说法重复 3 次
3. 自信化:即使不确定也写得很笃定("一定是……")

### 6.2 校准的三轮节奏(沿用 v5)

| 轮次 | 目标 | 触发 | 动作 |
|------|------|------|------|
| **第 1 轮 · 结构** | 让骨架立住 | 26 项清单扫描后 | 缺本篇定位 / 总结 / 附录 → 补;章节顺序乱 → 调整;跨篇重复 → 改引用 |
| **第 2 轮 · 硬伤** | 经得起查 | 路径对账 / 量化自检 / API 版本核对后 | 路径幻觉 → 重查或删;量化没依据 → 补来源;API 签名不对 → 重读源码 |
| **第 3 轮 · 锐度** | 读完有收获 | 通读全文,凭"读起来是否废话"判断 | 数据堆砌 → 加"所以呢";AI 自嗨 → 删"非常精妙";模糊量化 → 改具体数字;冗余 → 合并 |

**铁律**:每轮校准**只动一类问题**,不交叉。否则丢失已校准的进度。

### 6.6 校准后的 verifier 复审机制(v6 新增)

```markdown
2. **重新跑完整【26 项 / 40 项 / 自定义数】质量清单**
```

### 9.4 构建 hook(mkdocs,v6 强化)

```python
# scripts/mkdocs_strip_author_meta.py (v6 强化版)
import re
from mkdocs.plugins import get_plugin_logger

log = get_plugin_logger(__name__)

# v6:只匹配标准 AUTHOR_ONLY:START/END,变体(:SELFCHECK/:begin/:end 等)不剥
PATTERN = re.compile(
    r'<!--\s*AUTHOR_ONLY:START\s*-->.*?<!--\s*AUTHOR_ONLY:END\s*-->\n?',
    re.DOTALL
)

def on_page_markdown(markdown, page, config, files):
    cleaned = PATTERN.sub('', markdown)
    if cleaned != markdown:
        log.info(f"stripped AUTHOR_ONLY from {page.file.src_path}")
    return cleaned
```

### 10.1 system prompt 字符串渲染陷阱(v6 实战沉淀 P0 坑)

**症状**:
- 在 Edit 工具 old_string 和 new_string 里输入两个不同但视觉相似的字符串(如 `mm/memcontrol.c` 和 `mm/memcontrol.c`)
- 系统报 "No changes made. The replacement produced identical content."(实际不同)
- 或者 Python `assert 'mm/memcontrol.c' == 'mm/memcontrol.c'` 返回 True(实际不同)

**根因**:
- system prompt 在渲染时把所有"看起来像 memcontrol.c"的字符串规范化为同一个字符序列
- 无论 `m-e-m-c-o-n-t-r-o-l` 还是 `m-e-m-c-g-r-o-u-p-c-o-n-t-r-o-l`,显示都一样
- 严重影响 Edit 工具、assert 验证、grep 匹配、文件 IO

**影响范围**:
- 任何涉及"子串关系 / 视觉相似 / 拼写错位"的字符串比较都会失效
- 实战例子:
  - `mm/memcontrol.c` 和 `mm/memcontrol.c` 被规范化成同一个
  - `aart/` 和 `art/` 可能被规范化成同一个
  - `vvmscan` 和 `vmscan` 可能被规范化成同一个

**绕过方法**:
1. **Python 必须用 `chr()` 拼接**,如:
   ```python
   target = chr(109) + chr(101) + chr(109) + chr(99) + chr(103) + chr(114) + chr(111) + chr(117) + chr(112) + chr(46) + chr(99)  # memcgroup.c
   ```
2. **`bytes([0x6d, 0x65, ...])` hex 验证**,绕过字符串渲染
3. **用 `re.escape` 拼字符串时也要用 `chr()` 拼**,不能直接 `re.escape('mm/memcontrol.c')` (也会被规范化)
4. **字符串字面量用 ASCII ord 校验**:
   ```python
   expected_hex = '6d656d6367726f75702e63'  # memcgroup.c
   assert target.encode('utf-8').hex() == expected_hex
   ```

**教训**:
- 写 verify 脚本时,涉及容易冲突的字符串(子串关系、视觉相似),必须用 `chr()` 拼接 + `bytes()` hex 验证
- **不能依赖字符串字面量**

### 12.2 子线程写入 bug 6 类(v6 实战沉淀)

实战中子线程被 token 截断时,容易出现的 6 类 bug:

| # | ❌ 错例 | ✅ 正确 | 出现频次 |
|---|---------|---------|---------|
| 1 | `aart/` (多 a) | `art/` | 1+ 次 |
| 2 | `vvmscan` (多 v) | `vmscan` | 1+ 次 |
| 3 | `rameworks` (缺 f) | `frameworks` | 1+ 次 |
| 4 | `ndroid:` (缺 a) | `android:` | 1+ 次 |
| 5 | `m_kill` (缺 a) | `am_kill` | 1+ 次 |
| 6 | `o.lmk` (缺 r) | `ro.lmk` | 1+ 次 |
| 7 | 控制字符 0x07/0x08/0x0b/0x0c | 无 | 25+ 次(Memory 13 实战) |

**根因**:
- 子线程在 token 截断时,部分字符被吞/多
- 子线程在写入时,前面多 1 字符(变成 `aart/`/`vvmscan` 等)
- 控制字符是 token 截断时输出乱码

**防范**:
- 子线程 prompt 必加"绝对不能写错文件名/类名/作者名,每行写到磁盘前必须脑内复述"
- 主线程 verify 必查这 6 类(见 §10.4 verify_bug6.py + verify_control.py)

## 十三、子线程协作规范(v6 新增)

> v5 没有子线程协作规范,实战 15 篇遇到 3 类问题:
> ① 子线程 token 截断(04/06 + 15 实战 2 次)
> ② 子线程违抗 prompt(Memory 09 用 `:SELFCHECK`)
> ③ 子线程写入 bug 6 类(Memory 13 实战 25+ 处)
> v6 给出规范化的子线程 prompt 模板 + 主线程接力协议 + verifier 复审机制。

### 13.1 子线程 prompt 模板(v6 强制)

> 派活给子线程时,**必须**用以下模板,顶部 5 行硬性要求不可省。

````markdown
# 任务:写 [系列名] [文章名] v6

## 顶部 5 行硬性要求(违反任一 = 文章作废)

1. **marker 规范**:用 `<!-- AUTHOR_ONLY:START -->` 和 `<!-- AUTHOR_ONLY:END -->` 包裹。
   **绝对不能**发明新 marker 标签(`:SELFCHECK` / `:begin` / `:end` / `:REVIEW` 等都不行,只能 `:START/:END`)。
   v6 §9.4 剥离脚本只匹配 `:START/:END`,其他变体不会剥 → 泄露到公开站。
2. **marker 嵌套禁令**:`<!-- AUTHOR_ONLY:START -->` 必须有对应 `<!-- AUTHOR_ONLY:END -->`,禁止孤立 START。
   5 段前言用一个 AUTHOR_ONLY 段,自检报告用另一个独立段,**不嵌套**。
3. **绝对不能写错文件名 / 类名 / 作者名**:
   - 不能写 `aart/`(多 a) → 正确是 `art/`
   - 不能写 `vvmscan`(多 v) → 正确是 `vmscan`
   - 不能写 `rameworks`(缺 f) → 正确是 `frameworks`
   - 不能写 `ndroid:`(缺 a) → 正确是 `android:`
   - 不能写 `m_kill`(缺 a) → 正确是 `am_kill`
   - 不能写 `o.lmk`(缺 r) → 正确是 `ro.lmk`
   - 不能嵌入控制字符 0x07/0x08/0x0b/0x0c(来源 token 截断乱码)
   - 每行写到磁盘前必须脑内复述。
4. **跨篇 markdown 链接必须用全角冒号 `:`(U+FF1A)**,不能用半角 `:`(U+003A)。
   错误:`[15-未来方向:...](15-未来方向:...md)`
   正确:`[15-未来方向:...](15-未来方向:...md)`
   (Windows 文件系统不区分全/半角冒号,但 mkdocs 渲染时 URL 编码不同 → 半角冒号 404)
5. **已知幻觉清单**:见 v6 §12,所有 `kernel/cgroup/memcontrol.c`(应 `mm/memcontrol.c`)/
   `class GenerationalCC` / `Heap::IsAIAgentApp()` / `MGLRU 5.10 合并` /
   `2025-11-30 AOSP 17 发布` 都是错的,绝对不能写。

## 写任务

(此处写具体的写作要求,沿用 v5 模板 + v6 强化项)

## 写完后必须跑

(此处列具体的 verify_*.py 工具,子线程必须自己跑)

## token 截断时的退出协议

如果你 token 即将耗尽:
1. 立即 commit 当前已写内容(只含 5 段前言 + §1 + §2,可能不完整)
2. 退出前在 output 报告接力点:"主线程请从 LXXX 行 §X.Y 末续写 §X+1-§Y"
3. 不要试图"塞下更多内容"——会导致部分内容被吞,出现 6 类 bug

## 不信子线程自检报告(主线程 hygiene)

主线程在收到子线程 output 后,**必须自己跑** §10.4 的 verify_*.py 6 项,
不信子线程的"我已自检全部通过"声明。
````

### 13.2 主线程接力协议(v6 实战沉淀)

> 实战 04/06 + 15 踩了 2 次子线程 token 截断,v6 给出标准化接力协议。

**子线程截断信号**:
- 子线程 output 含 "已达到 Token Plan 用量上限" 错误
- 或子线程 commit 末尾不完整(只到 §2 中间)

**主线程接力步骤**:
1. **Read 接力点**:子线程最后一次 commit 的实际行号 + 章节号(不要凭子线程 prompt 估计)
2. **跑 verify 已写部分**:确保子线程已写部分没 6 类 bug + marker 嵌套
3. **分批续写**:每批 30-50KB,先写 §3-§5,再写 §6-§11,再写附录 A-D
4. **每批 commit**:避免再次截断时全部丢失
5. **跑综合 verify**:§10.4 全部 6 项
6. **报告用户**:接力完成 + commit 列表

**接力示例**(Memory 15 v5 实战):
```
1. 子线程 bg_xxx 在 00:25:24 报"已达到 Token Plan 用量上限"
2. 主线程 git log 看子线程最后一次 commit:00d8091 (只含 §1-§2, 共 350 行)
3. 主线程 Read L350,确认接力点
4. 主线程分 3 批续写:
   - 批 1:§3-§5 (commit 9396cef, "Memory 15 v5: 未来方向")
   - 批 2:§6-§11 + 附录 (commit 90d0e99, "Memory 15 复审修复")
   - 批 3:最终打磨(commit 29d8091, "Memory 15 v2 P0 修复")
5. 主线程跑 verify_15_v3.py 6 项
6. 报告用户 PASS
```

### 13.3 marker 嵌套禁令(v6 实战沉淀)

**禁令**(实战多次踩坑):
1. ❌ 5 段前言内嵌孤立 START(子线程常用)
2. ❌ 自检报告嵌在 5 段前言内(应该独立段)
3. ❌ 自检报告用 `:SELFCHECK` 等非标准标签
4. ❌ 嵌套 START 无对应 END(被外层 END 一锅端剥)
5. ❌ 自检报告里嵌入 `<!-- xxx -->` 字符串(破坏 `re.DOTALL` 非贪婪匹配)

**正确结构**:
```markdown
<!-- AUTHOR_ONLY:START -->
# 5 段前言(本篇定位 / 校准决策日志 / 角色设定 / 上下文)
...
<!-- AUTHOR_ONLY:END -->

# 正文...

<!-- AUTHOR_ONLY:START -->
# 26 项质量清单自检
...
<!-- AUTHOR_ONLY:END -->
```

**v6 §9.4 剥离脚本**:只匹配 `<!--\s*AUTHOR_ONLY:START\s*-->.*?<!--\s*AUTHOR_ONLY:END\s*-->\n?`,**不匹配**所有变体。

### 13.4 verify_*.py 工具集(v6 实战沉淀)

> v5 没有统一的 verify 工具集,实战中每个 agent 各自写,容易遗漏。
> v6 规定标准工具集,主线程写完必跑(不信子线程自检)。

| 工具 | 检查项 | 实现提示 |
|------|--------|---------|
| `verify_marker.py` | 0 rogue + 0 嵌套 + 2 段配对 | 用 §9.4 regex + `re.findall` |
| `verify_strip.py` | 0 公开站元信息残留 | 剥后扫 `AUTHOR_ONLY` / `本篇定位` / `校准决策日志` |
| `verify_colon.py` | 0 半角冒号链接 | `re.findall(r'\]\([^)]*\.md:[^)]*\)', out)` |
| `verify_paths.py` | §12.1 blacklist | 错路径 hex 验证 |
| `verify_internal_ref.py` | 0 公开站内部规范号 | 扫 `v[0-9] §X` / `校准` / `硬性要求` / `人工搜索` |
| `verify_bug6.py` | 0 6 类子线程 bug | 扫 `aart/` `vvmscan` `rameworks` `ndroid:` `m_kill` `o.lmk` |
| `verify_control.py` | 0 控制字符 | 扫 `[\x00-\x08\x0b-\x0c\x0e-\x1f]` |
| `verify_words.py` | 0 反 AI 自嗨词(公开站) | 扫 §5.3 词表 20 个 |
| `verify_ai_words.py` | 0 反 AI 自嗨词(全文件) | 同上 |
| `verify_struct.py` | 26 项质量清单 | 各种 |
| `verify_facts.py` | 关键事实校准 | AOSP 17 发布日 / MGLRU 版本 等 |
| `verify_chinese.py` | 中文字数 ≥ 8000 | `re.findall(r'[\u4e00-\u9fff]', src)` |
| `verify_chapter.py` | 章节完整性 | H1 数量 + 标题完整性 |
| `verify_quotes.py` | 量化数据附录 C | 各种 |
| `verify_dual_view.py` | 公开站 vs 作者视图一致性 | 公开站 = 全文 - AUTHOR_ONLY 段 |

**主线程 hygiene**:
- 写完每篇后必跑这 6 项:marker / strip / colon / paths / bug6 / control
- 不信子线程自检报告
- 子线程违抗 prompt 时的修复:主线程 Edit 修,git commit,再开 verifier 复审

### 13.5 verifier 复审机制(v6 实战沉淀)

> 实战 14/15 篇都开了 verifier 复审(独立子代理),v6 规范化为强制机制。

**verifier 复审触发条件**:
- 每篇主写 commit 后,自动触发 verifier 复审
- 复审 FAIL → 主线程修 + 再开复审
- 复审 PASS → 标记收官

**verifier 复审 prompt 模板**:见 §6.6。

**verifier 复审 hygiene**:
- 强制用 Python `bytes([...])` hex 验证(因为 system prompt 渲染陷阱)
- 不信子线程剥离报告
- 必须自己跑 verify 6 项
- 报告明确 PASS/FAIL + 剩余问题分级
