## 一、用户态视角：调度相关的 API

### 1.1 调度策略 / 优先级 / CPU 亲和性

用户态有几类 API 可以影响调度：

| API | 作用 | 典型场景 |
|---|---|---|
| `nice(value)` | 设置 CFS 类的 nice 值 | 调低优先级 |
| `sched_setscheduler(pid, policy, param)` | 设置调度策略 | 改 SCHED_FIFO / SCHED_RR / SCHED_DEADLINE |
| `sched_setattr(pid, attr, flags)` | 扩展的调度参数 | 设置 deadline 的 runtime/deadline/period |
| `sched_setaffinity(pid, cpuset)` | 设置 CPU 亲和性 | 绑核 |
| `taskset -c 0-3 command` | 命令行绑核 | 启动时绑核 |
| `chrt -p <pid>` | 查看 / 设置 RT 调度参数 | 调 RT 优先级 |
| `cgroup cpu.uclamp.min/max` | UClamp 调度提示 | Android 14 top-app |

**关键认知**：
- 普通进程默认走 `SCHED_NORMAL`（CFS 调度类）
- `nice` 只能改 CFS 类的权重
- 想用 RT 调度需要 root + `CAP_SYS_NICE`
- Android 14 上 `cgroup cpu.uclamp` 是普通应用能用的"软"调度控制

### 1.2 Android 14 上能看到的调度状态

```bash
# 1. 看进程的调度策略
adb shell "chrt -p <pid>"
# 输出: pid 1234's current scheduling policy: SCHED_NORMAL
#        pid 1234's current scheduling priority: 0

# 2. 看进程的 CPU 亲和性
adb shell "taskset -p <pid>"
# 输出: pid 1234's current affinity mask: ff

# 3. 看 cgroup 的 UClamp 设置
adb shell "cat /dev/cgroup_info/cpu/<pid>/cpu.uclamp.min 2>/dev/null"
# 或者
adb shell "cat /sys/fs/cgroup/cpu/<slice>/cpu.uclamp.min"

# 4. 看 /proc/<pid>/stat 里的调度字段
adb shell "cat /proc/<pid>/stat"
# 第 16-19 字段：priority / nice / num_threads / itrealvalue
# 第 36 字段：start_code
# 第 39 字段：policy（CFS=0, RT=1, ...）
```

**关键字段**：
- `policy=0`：CFS（普通进程）
- `policy=1`：FIFO（RT）
- `policy=2`：RR（RT）
- `policy=3`：BATCH（CFS 子类）
- `policy=4`：ISO（CFS 子类，几乎不用）
- `policy=5`：IDLE
- `policy=6`：DEADLINE

### 1.3 Android 14 上的 cgroup cpu 视图

```bash
# Android 14 上的 cgroup v2 路径
adb shell "ls /sys/fs/cgroup/"

# 典型输出
# cgroup.controllers  cgroup.max.descendants  cgroup.pressure
# cgroup.events  cgroup.max.depth          cgroup.procs
# cgroup.freeze  cgroup.stat               cgroup.subtree_control
# cgroup.id      cgroup.threads            cpu.pressure
# cpu.stat       cpu.uclamp.max            cpu.uclamp.min
# init.scope     system.slice              top-app.slice  ← top-app 是独立 slice
# foreground.slice
```

**关键**：
- Android 14 用 cgroup v2——CPU 控制 + UClamp
- `top-app.slice` 是前台应用的独立 slice
- `background.slice` 是后台应用的 slice
- 应用启动时被分配到对应 slice——这是 Android 14 调度优化的关键

## 二、调度子系统全景

### 2.1 调度子系统的位置

```
Linux Kernel
  │
  ├── 进程管理 (kernel/fork.c, kernel/exit.c)
  │
  ├── 调度子系统 (kernel/sched/)
  │   ├── core.c            ← 调度器主入口
  │   ├── fair.c            ← CFS 调度类
  │   ├── rt.c              ← RT 调度类
  │   ├── deadline.c        ← Deadline 调度类
  │   ├── idle.c            ← Idle 调度类
  │   ├── stop_task.c       ← Stop 调度类
  │   ├── wait.c            ← 等待队列
  │   ├── completion.c      ← 完成量
  │   ├── cpufreq.c         ← CPU 频率（依赖调度）
  │   ├── cpuidle.c         ← CPU 空闲（依赖调度）
  │   ├── membarrier.c      ← 内存屏障
  │   ├── topology.c        ← CPU 拓扑
  │   ├── stats.c           ← 调度统计
  │   └── ...
  │
  ├── 内存管理 (mm/)
  │   └── ...
  │
  ├── 文件系统 (fs/)
  │   └── ...
  │
  └── ...
```

**关键**：
- 调度子系统是 Linux 内核**最复杂**的子系统之一
- 涉及 core.c（基础设施） + 5 个调度类 + 各种 helper
- Android 14 还扩展了 EAS（Energy Aware Scheduling）——在 fair.c 中实现

### 2.2 调度子系统跟其他子系统的协作

```
调度子系统
  ↑
  │ 被以下子系统使用
  ├── 进程管理 (fork/exit → 调 wake_up_new_task / schedule)
  ├── 中断子系统 (中断返回 → schedule)
  ├── 定时器子系统 (tick 中断 → scheduler_tick)
  ├── 信号子系统 (deliver_signal → wake_up_state)
  ├── 内存管理 (page fault → wake_up_kthread)
  ├── 文件系统 (IO 完成 → wake_up)
  └── 进程间通信 (futex wake → wake_up_q)
```

**关键**：
- 调度器**几乎**被每个子系统调用
- "wake_up" 是调度器的入口——任何阻塞的进程被唤醒都要走调度器
- "schedule()" 是调度器的出口——主动让出 CPU

### 2.3 调度子系统的核心数据结构

```c
// kernel/sched/sched.h（简化）
struct rq {
    raw_spinlock_t lock;          // runqueue 自旋锁
    unsigned int cpu;             // 哪个 CPU 的 rq
    struct task_struct *curr;     // 当前跑在 rq 上的 task
    struct task_struct *idle;     // 空闲 task
    struct task_struct *stop;     // stop task（migration / CPU hotplug）

    struct cfs_rq cfs;           // CFS 调度类的 runqueue
    struct rt_rq rt;             // RT 调度类的 runqueue
    struct dl_rq dl;             // Deadline 调度类的 runqueue

    struct root_domain *rd;      // 多核调度的根域
    struct sched_domain *sd;     // 调度域

    unsigned long nr_running;    // runqueue 上有多少 runnable task
    u64 nr_switches;             // 切换计数
    u64 cpu_load[];              // CPU 负载

    struct sched_avg avg;        // 平均负载（PELT 算法）

    // ... 大量其他字段
};

// per-CPU 的 runqueue 数组
DECLARE_PER_CPU(struct rq, runqueues);
```

**关键认知**：
- 每个 CPU 一个 rq（runqueue）
- rq 包含 5 个调度类的子 rq（CFS / RT / Deadline / Idle / Stop 各一个）
- 锁 `rq->lock` 是调度器最热的锁——所有调度操作都要拿这个锁

### 2.4 调度子系统的入口函数

```c
// kernel/sched/core.c
// 主动调度
void schedule(void);
// 带睡眠的 schedule
long schedule_timeout(long timeout);
// 抢占检查
void preempt_enable(void);
void might_resched(void);
// 唤醒
int wake_up_process(struct task_struct *tsk);
int wake_up_state(struct task_struct *tsk, unsigned int state);
// 任务迁移
void set_cpus_allowed(struct task_struct *p, const struct cpumask *new_mask);
```

**关键**：
- `schedule()` 是**调度的主入口**——任何主动让出 CPU 都走这个
- `wake_up_process()` 是**唤醒的主入口**——把 task 从 sleep 状态变 runnable
- `preempt_enable()` 是抢占检查点——可能触发 schedule

## 三、sched_class 体系：5 个调度类

### 3.1 sched_class 是什么

`sched_class` 是调度器的**抽象基类**——每个调度类实现一组 ops（操作），调度器主入口通过 ops 调用具体实现：

```c
// kernel/sched/sched.h
struct sched_class {
    const struct sched_class *next;   // 链表 next 指针

    // 1. 入队 / 出队
    void (*enqueue_task)(struct rq *rq, struct task_struct *p, int flags);
    void (*dequeue_task)(struct rq *rq, struct task_struct *p, int flags);

    // 2. 主动让出 CPU
    void (*yield_task)(struct rq *rq);

    // 3. wake_up 唤醒路径
    void (*wakeup_preempt)(struct rq *rq, struct task_struct *p, int flags);
    void (*task_woken)(struct rq *rq, struct task_struct *p);

    // 4. 修改调度参数
    void (*set_cpus_allowed)(struct task_struct *p, const struct cpumask *newmask);

    // 5. 选下一个 task
    struct task_struct *(*pick_next_task)(struct rq *rq);

    // 6. put_prev_task / set_next_task
    void (*put_prev_task)(struct rq *rq, struct task_struct *p);
    void (*set_next_task)(struct rq *rq, struct task_struct *p, bool first);

    // 7. tick 处理
    void (*task_tick)(struct rq *rq, struct task_struct *p, int queued);

    // 8. 新建进程 / fork / exit
    void (*task_fork)(struct task_struct *p);
    void (*task_dead)(struct task_struct *p);

    // ... 30+ ops
};
```

**关键认知**：
- sched_class 是 C 语言的"多态"——用结构体指针模拟继承
- 每个调度类实现这一组 ops
- 调度器主入口 `schedule()` 不关心具体调度类——只调 ops

### 3.2 5 个调度类实例

```c
// kernel/sched/fair.c
DEFINE_SCHED_CLASS(fair) = {
    .next = &idle_sched_class,
    .enqueue_task = enqueue_task_fair,
    .dequeue_task = dequeue_task_fair,
    .pick_next_task = pick_next_task_fair,
    // ...
};

// kernel/sched/rt.c
DEFINE_SCHED_CLASS(rt) = {
    .next = &fair_sched_class,
    .enqueue_task = enqueue_task_rt,
    .pick_next_task = pick_next_task_rt,
    // ...
};

// kernel/sched/deadline.c
DEFINE_SCHED_CLASS(dl) = {
    .next = &rt_sched_class,
    .enqueue_task = enqueue_task_dl,
    .pick_next_task = pick_next_task_dl,
    // ...
};

// kernel/sched/idle.c
DEFINE_SCHED_CLASS(idle) = {
    .next = &fair_sched_class,
    .enqueue_task = enqueue_task_idle,
    .pick_next_task = pick_next_task_idle,
    // ...
};

// kernel/sched/stop_task.c
DEFINE_SCHED_CLASS(stop) = {
    .next = &dl_sched_class,
    .enqueue_task = enqueue_task_stop,
    .pick_next_task = pick_next_task_stop,
    // ...
};
```

**5 个调度类**：

| 调度类 | 实现 | 用途 | 优先级 |
|---|---|---|---|
| `stop` | stop_task.c | CPU 热插拔 / 迁移 | 最高（内核专用） |
| `deadline` | deadline.c | SCHED_DEADLINE（截止时间） | 高 |
| `rt` | rt.c | SCHED_FIFO / SCHED_RR | 中 |
| `fair` | fair.c | SCHED_NORMAL / SCHED_BATCH | 默认 |
| `idle` | idle.c | SCHED_IDLE | 最低 |

### 3.3 sched_class 链表与优先级

调度类通过 `next` 指针形成链表：

```
stop_task → dl → rt → fair → idle
   │
   └──> schedule() 按链表顺序检查每个调度类的 runqueue
        找到非空的就选它
```

**关键认知**：
- 链表顺序 = 优先级——`schedule()` 从链表头开始找
- 如果 dl_rq 非空，先跑 dl 的 task；否则找 rt，再否则 fair
- stop_task 是内核专用的（migration / hotplug），用户态不能选

### 3.4 调度策略 vs 调度类

```c
// include/uapi/linux/sched.h
#define SCHED_NORMAL    0    // → fair 类
#define SCHED_FIFO      1    // → rt 类
#define SCHED_RR        2    // → rt 类
#define SCHED_BATCH     3    // → fair 类
#define SCHED_IDLE      5    // → idle 类
#define SCHED_DEADLINE  6    // → dl 类
```

**关键**：
- 用户态用 `policy`（调度策略）
- 内核用 `sched_class`（调度类）
- 多个 policy 可能映射到同一个 sched_class——例如 SCHED_NORMAL 和 SCHED_BATCH 都走 fair

### 3.5 Android 14 上调度类的使用

```bash
# 1. 看系统中各调度类的进程数
adb shell "for pid in \$(ls /proc/ | grep -E '^[0-9]+$'); do
    policy=\$(cat /proc/\$pid/stat | awk '{print \$36}')
    if [ \"\$policy\" = \"0\" ]; then echo \"\$pid CFS\";
    elif [ \"\$policy\" = \"1\" ]; then echo \"\$pid FIFO\";
    elif [ \"\$policy\" = \"2\" ]; then echo \"\$pid RR\";
    elif [ \"\$policy\" = \"6\" ]; then echo \"\$pid DL\";
    fi
done | head -20"
```

**典型 Android 14 进程分布**：
- 系统服务（system_server / zygote64 / surfaceflinger）：SCHED_NORMAL（CFS）
- audio HAL：高优先级 SCHED_FIFO / SCHED_RR
- display HAL：SCHED_FIFO
- 其他：SCHED_NORMAL

### 3.6 调度类的 cgroup 影响

```bash
# Android 14 上 cgroup v2 的 cpu.uclamp
# UClamp 在内核走 fair_sched_class 的特殊路径
adb shell "cat /sys/fs/cgroup/top-app.slice/cpu.uclamp.min"
# 输出: 0   ← 默认 min 0%（无约束）
adb shell "cat /sys/fs/cgroup/top-app.slice/cpu.uclamp.max"
# 输出: max   ← 默认 max 100%
```

**关键**：
- UClamp 作用于 fair 类——通过 `p->uclamp_min` / `p->uclamp_max` 影响选 task
- top-app slice 的 UClamp 是 Android 14 调度优化的核心
- 这部分会在 09 篇（多核调度）展开

## 四、runqueue 数据结构

### 4.1 per-CPU runqueue

Linux 每个 CPU 有独立的 runqueue——这样能**减少锁竞争**：

```c
// kernel/sched/sched.h
extern struct rq runqueues[NR_CPUS];

// 或者 per-CPU 变量（Linux 5.x）
DECLARE_PER_CPU_SHARED_ALIGNED(struct rq, runqueues);
```

**为什么 per-CPU？**
- 减少锁竞争——不同 CPU 的 rq 操作不需要互斥
- cache locality——CPU 经常访问自己的 rq
- 多核调度时只迁移 task，不迁移整个 rq

### 4.2 rq 结构的关键字段

```c
// kernel/sched/sched.h
struct rq {
    raw_spinlock_t lock;          // 自旋锁——保护 rq
    unsigned int cpu;             // 哪个 CPU 的 rq
    struct task_struct *curr;     // 当前跑在这个 CPU 的 task
    struct task_struct *idle;     // idle task（CPU 空闲时跑）
    struct task_struct *stop;     // stop task（CPU hotplug 用）

    struct cfs_rq cfs;           // CFS 调度类的 rq
    struct rt_rq rt;             // RT 调度类的 rq
    struct dl_rq dl;             // Deadline 调度类的 rq

    u64 clock;                    // rq 时钟（单调递增）
    u64 clock_task;               // task 视角的时钟（被 throttled 时停）

    unsigned long nr_running;    // runnable task 总数
    unsigned long nr_uninterruptible;  // uninterruptible task 数

    // CPU 负载（PELT 算法）
    struct sched_avg avg;
    struct util_est util_est;

    // ... 其他
};
```

**关键字段**：
- `lock`：调度器最热的锁——任何 schedule 操作都要拿
- `curr`：当前跑在这个 CPU 的 task
- `cfs` / `rt` / `dl`：三个调度类的子 rq——pick_next_task 找 task 从这里找
- `clock` / `clock_task`：调度器的时间基

### 4.3 子 rq：cfs_rq / rt_rq / dl_rq

每个调度类有自己的 runqueue 结构：

```c
// CFS 的 runqueue
struct cfs_rq {
    struct rb_root_cached tasks_timeline;  // 红黑树根——07 篇详讲
    struct rb_node *rb_leftmost;            // 红黑树最左节点——最小 vruntime

    struct sched_entity *curr;              // 当前 CFS task 的 sched_entity
    struct sched_entity *next;              // 下一个要跑的 sched_entity
    struct sched_entity *last;              // 上一个跑的 sched_entity
    struct sched_entity *skip;              // skip entity（用于 group scheduling）

    unsigned int nr_running;                // CFS rq 上的 task 数
    unsigned int h_nr_running;              // 包含组的 task 数

    u64 min_vruntime;                       // 最小 vruntime（用于新 task）
    u64 exec_clock;                         // 累计执行时间

    int on_list;                            // 是否在 parent rq 的 leaf_cfs_rq_list
    struct cfs_rq *tg_runnable;             // task_group runnable 列表
    struct rq *rq;                          // 所属的 CPU rq
    // ... task_group 相关
};

// RT 的 runqueue
struct rt_rq {
    struct rt_prio_array active;  // 位图 + 链表数组——08 篇详讲
    unsigned int rt_nr_running;
    unsigned int rr_nr_running;

    int highest_prio;             // 最高优先级
    int overloaded;               // 是否过载（rq > 1 个 task）

    // ...
};

// Deadline 的 runqueue
struct dl_rq {
    struct rb_root_cached root;   // 红黑树——按 deadline 排序
    unsigned int dl_nr_running;
    struct dl_bw dl_bw;           // bandwidth 控制
    // ...
};
```

**关键认知**：
- CFS rq 用红黑树——按 vruntime 排序
- RT rq 用位图 + 链表——按优先级索引
- DL rq 用红黑树——按 deadline 排序
- pick_next_task 调用 `class->pick_next_task(rq)` 即可——每个调度类自己实现

### 4.4 rq.lock：调度器的最热锁

```c
// kernel/sched/core.c
// 拿 rq 锁
raw_spin_lock(&rq->lock);

// 释放
raw_spin_unlock(&rq->lock);
```

**为什么是 raw_spinlock？**
- 调度器上下文不能用 sleeping lock（mutex）——会死锁
- `raw_spin_lock` 是最轻量的锁——不关中断
- 但 `rq->lock` 在中断上下文也可能被拿——所以有些路径用 `raw_spin_lock_irq`

**关键**：
- `schedule()` 拿 `rq->lock` 时不关中断
- 中断处理中的调度操作（`sched_tick`）需要 `raw_spin_lock_irq`
- 双锁顺序：先 local rq lock，再 remote rq lock（防止死锁）

### 4.5 rq 的 lock 顺序（双锁）

```c
// kernel/sched/core.c
// 任务迁移需要两个 rq 的 lock
static int migrate_task(struct task_struct *p, ...)
{
    // 1. 拿 src rq lock
    raw_spin_lock(&src_rq->lock);
    // 2. 拿 dst rq lock
    double_rq_lock(src_rq, dst_rq);
    // 3. 迁移 task
    // 4. 释放
    double_rq_unlock(src_rq, dst_rq);
}
```

**关键**：
- 双锁顺序：先 src 再 dst——必须按 CPU 编号顺序
- 否则会死锁——CPU 1 拿 1→2，CPU 2 拿 2→1，互相等待
- 这个死锁是调度器历史上著名的 bug 来源

### 4.6 rq 的可观测性

```bash
# 1. 看每个 CPU 的负载
adb shell "cat /proc/loadavg"
# 输出: 1.50 0.80 0.30 1/512 12345
# 解释: 1分钟负载 1.50 / 5分钟 0.80 / 15分钟 0.30
#      1 个 runnable / 总共 512 个 / 最近 PID 12345

# 2. 看每个 CPU 的 nr_running
adb shell "cat /sys/devices/system/cpu/cpu*/cpufreq/scaling_cur_freq"
# 这看的是 CPU 频率——间接反映 CPU 负载

# 3. 用 perfetto 看 runqueue 上的 task
adb shell "perfetto --record -o /data/local/tmp/trace.proto \
    -e 'sched:sched_wakeup sched:sched_switch sched:sched_migrate_task' --time 30"
```

**关键**：
- `/proc/loadavg` 是系统全局的负载
- perfetto 能看每个 CPU 的 runqueue 事件

## 五、schedule() 主入口：调度器的核心

### 5.1 schedule() 源码

```c
// kernel/sched/core.c
void schedule(void)
{
    struct task_struct *prev, *next;
    unsigned long flags;
    struct rq *rq;
    int cpu;

    // 1. 关抢占（防止当前 task 在 schedule 中被切换）
    preempt_disable();

    cpu = smp_processor_id();
    rq = cpu_rq(cpu);
    prev = rq->curr;

    // 2. 检查新 task 是否跑完——如果跑完会触发 resched
    // (sched_submit_to_load_balance 等)

    // 3. 核心：释放当前 CPU 的 rq lock
    raw_spin_lock_irqsave(&rq->lock, flags);

    // 4. 更新 task 的各种统计（vruntime、cpu 时间）
    update_rq_clock(rq);

    // 5. 选下一个 task
    next = pick_next_task(rq, prev, &rf);

    // 6. 清除当前 task 的 need_resched 标志
    clear_tsk_need_resched(prev);

    // 7. 真正切换：context_switch
    if (likely(prev != next)) {
        rq->nr_switches++;
        rq->curr = next;
        ++*switch_count;

        // 上下文切换
        context_switch(rq, prev, next, &rf);
    } else {
        // 没有切换——恢复
        __balance_callbacks(rq);
        raw_spin_unlock_irqrestore(&rq->lock, flags);
    }

    // 8. 开抢占
    preempt_enable();
}
```

**关键路径**：
1. `pick_next_task` 选下一个 task（§6 详讲）
2. `context_switch` 切换上下文（§7 详讲）
3. schedule() 不返回——直到 task 被切走

### 5.2 schedule 的 4 个调用入口

```c
// 1. 主动让出 CPU
// kernel/sched/core.c
asmlinkage __visible void __sched schedule(void);

// 2. 阻塞系统调用（schedule_timeout）
// 例：nanosleep / sleep / read on pipe
long __sched schedule_timeout(signed long timeout);

// 3. 抢占点（preempt_enable）
// kernel/sched/core.c
#define preempt_enable() \
do { \
    barrier(); \
    if (unlikely(preempt_count_dec_and_test())) \
        __preempt_schedule(); \
} while (0)

// 4. 唤醒路径（wake_up_process）
// kernel/sched/core.c
int wake_up_process(struct task_struct *p);
```

**关键认知**：
- 主动 schedule：task 自己想睡（`schedule()` 直接调）
- schedule_timeout：阻塞 IO / 睡眠（nanosleep / sleep）
- preempt_enable：抢占点（内核返回用户态前）
- wake_up_process：唤醒路径——把 task 放上 runqueue，可能触发 schedule

### 5.3 need_resched：调度触发器

```c
// kernel/sched/sched.h
static inline void set_tsk_need_resched(struct task_struct *tsk)
{
    set_tsk_thread_flag(tsk, TIF_NEED_RESCHED);
}

static inline void clear_tsk_need_resched(struct task_struct *tsk)
{
    clear_tsk_thread_flag(tsk, TIF_NEED_RESCHED);
}

static inline int test_tsk_need_resched(struct task_struct *tsk)
{
    return unlikely(test_tsk_thread_flag(tsk, TIF_NEED_RESCHED));
}
```

**关键**：
- `TIF_NEED_RESCHED` 是 task 的 thread flag
- 设置位置：scheduler_tick / wake_up_process / set_cpus_allowed
- 检查位置：返回用户态前 / preempt_enable / 中断返回

### 5.4 返回用户态前的调度点

```c
// arch/arm64/kernel/entry.S
// 从中断返回用户态
ret_to_user:
    // ...
    ldr    x19, [sp, #S_PSTATE]  // 检查 pstate
    // 检查 TIF_NEED_RESCHED
    tbz    x19, #TIF_NEED_RESCHED, no_work_pending
    // 有 need_resched——调 schedule
    bl    el0_svc_disable
    bl    schedule
    // 重新检查 work
    b    ret_to_user
no_work_pending:
    // 继续返回用户态
```

**关键认知**：
- ARM64 上 `TIF_NEED_RESCHED` 检查在中断返回路径上
- 一旦检查到，就调 schedule
- 这是"抢占式调度"的硬件配合机制——内核不需要主动检查

### 5.5 schedule_timeout：阻塞型调度

```c
// kernel/sched/core.c
long __sched schedule_timeout(signed long timeout)
{
    struct process_timer timer;
    unsigned long expire;

    switch (timeout) {
    case MAX_SCHEDULE_TIMEOUT:
        // 无限等待——直到被唤醒
        schedule();
        return 0;
    default:
        // 计算到期时间
        expire = timeout + jiffies;
        // 设置定时器
        init_timer(&timer);
        timer.expires = expire;
        timer.data = (unsigned long)current;
        timer.function = process_timeout;
        add_timer(&timer);

        // 调度出去
        schedule();

        // 醒来后删除定时器
        del_singleshot_timer_sync(&timer);

        timeout = expire - jiffies;
        return timeout < 0 ? 0 : timeout;
    }
}
```

**关键**：
- `schedule_timeout` 设置 timer，然后 schedule
- 醒来时返回剩余时间
- 几乎所有阻塞系统调用底层都走这个

### 5.6 为什么 schedule 不能失败

schedule 是**不可逆**的——一旦调用，task 一定被切走（除非没找到 next task，调度 idle）。

```c
// kernel/sched/core.c
void schedule(void)
{
    // ...
    next = pick_next_task(rq, prev, &rf);
    if (likely(prev != next)) {
        context_switch(rq, prev, next, &rf);
        // 不会返回这里——除非这个 task 被再次调度
    }
    // 如果 prev == next：不切换，直接 unlock
}
```

**关键认知**：
- schedule() 不返回调用方（除非 prev == next）
- 调用方代码在 task 被切走之前不会继续
- 任务代码可以认为是"暂停"了——直到被切回来

## 六、pick_next_task：挑下一个 task

### 6.1 优先级检查链

```c
// kernel/sched/core.c
/*
 * Pick up the highest-prio task:
 */
static inline struct task_struct *
pick_next_task(struct rq *rq, struct task_struct *prev, struct rq_flags *rf)
{
    // 优化：如果 prev 的调度类有 pick_next_task，并且 dl_rq 不空，调 class->pick_next_task
    // 否则走完整链表

    const struct sched_class *class;
    struct task_struct *p;

    if (likely(rq->nr_running == rq->cfs.h_nr_running)) {
        // 优化路径：CFS 是唯一非空调度类
        p = fair_sched_class.pick_next_task(rq);
        if (unlikely(p == RETRY_TASK))
            goto again;
        return p;
    }

again:
    // 完整路径：按调度类优先级链表
    for_each_class(class) {
        p = class->pick_next_task(rq);
        if (p)
            return p;
    }

    BUG();   // 不可能——至少 idle
}
```

**关键认知**：
- 调度器按优先级从 stop → dl → rt → fair → idle 顺序检查
- 每个调度类的 `pick_next_task` 检查自己的 rq 非空
- 命中第一个非空的就返回

### 6.2 优化：单一调度类路径

```c
// 调度器有 90%+ 时间只跑 CFS——所以优化
if (likely(rq->nr_running == rq->cfs.h_nr_running)) {
    // CFS 是唯一的非空调度类
    // 直接调 CFS 的 pick_next_task，跳过其他检查
    p = fair_sched_class.pick_next_task(rq);
}
```

**关键**：
- Android 14 上大多数时候只有 CFS 在跑——优化路径覆盖 90%+
- 节省的指令数 ~50 条——在调度器这么热的路径上影响很大

### 6.3 fair_sched_class.pick_next_task 概览

```c
// kernel/sched/fair.c
struct task_struct *
pick_next_task_fair(struct rq *rq)
{
    struct task_struct *p;
    struct sched_entity *se;
    struct cfs_rq *cfs_rq = &rq->cfs;
    struct sched_entity *curr = cfs_rq->curr;

    // 1. 把当前 task 的 vruntime 更新到 cfs_rq
    if (curr) {
        update_curr(cfs_rq);
        // ...
    }

    // 2. 红黑树最左节点——vruntime 最小的
    se = pick_next_entity(cfs_rq, curr);
    if (!se)
        return NULL;

    // 3. 从 sched_entity 反向找到 task_struct
    p = task_of(se);

    return p;
}
```

**关键**：
- CFS 选 vruntime 最小的 task——07 篇详讲
- 这是 O(log n) 操作——红黑树
- "最左节点"就是红黑树的最小 vruntime

### 6.4 RT / Deadline 的 pick_next_task

```c
// kernel/sched/rt.c
static struct task_struct *pick_next_task_rt(struct rq *rq)
{
    struct task_struct *p;
    struct rt_rq *rt_rq = &rq->rt;

    // 1. 找到最高优先级的 RT task
    p = _pick_next_task_rt(rt_rq);

    return p;
}

// kernel/sched/deadline.c
static struct task_struct *pick_next_task_dl(struct rq *rq)
{
    struct task_struct *p;
    struct dl_rq *dl_rq = &rq->dl;

    // 1. 找到 deadline 最早的 DL task
    p = _pick_next_task_dl(dl_rq);

    return p;
}
```

**关键**：
- RT 选最高优先级（数值最小）——08 篇详讲
- DL 选 deadline 最早——08 篇详讲
- 都是 O(1) 操作（位图 / 红黑树最小节点）

### 6.5 put_prev_task：放下当前 task

```c
// kernel/sched/core.c
// 在 context_switch 之前，先把当前 task 放回它所属的调度类
static void put_prev_task(struct rq *rq, struct task_struct *prev)
{
    if (prev->on_rq) {
        prev->sched_class->put_prev_task(rq, prev);
    }
}
```

**关键**：
- `put_prev_task` 把当前 task 重新入队到它所属的调度类
- 例如 fair 类：把 sched_entity 重新加入红黑树
- `set_next_task` 把下一个 task 从 rq 移除——标记"正在跑"

## 七、context_switch 详解

### 7.1 context_switch 的两个子操作

```c
// kernel/sched/core.c
context_switch(rq, prev, next, &rf)
  │
  ├── 1. prepare_task_switch(rq, prev, next)
  │       // 通知 trace / perf
  │
  ├── 2. arch_start_context_switch(prev)
  │       // 架构相关准备（如 ARM64 的 cpu_switch_to 准备）
  │
  ├── 3. mm switch（如果 prev != next）
  │       // 切换 mm_struct
  │       if (!next->mm) {
  │           // next 是内核线程——借 prev 的 mm
  │           enter_lazy_tlb(prev->active_mm, next);
  │           next->active_mm = prev->active_mm;
  │       } else {
  │           switch_mm(prev->active_mm, next->mm, next);
  │       }
  │
  ├── 4. switch_to(prev, next, prev)
  │       // 架构相关：保存 prev 上下文 + 加载 next 上下文
  │
  └── 5. finish_task_switch(prev)
          // 收尾：prev 可能就是当前 task 了（被切回）
```

### 7.2 switch_mm：切换内存描述符

```c
// include/linux/mmu_context.h
// 架构相关实现
static inline void switch_mm(struct mm_struct *prev, struct mm_struct *next,
                             struct task_struct *tsk)
{
    // ARM64 实现
    // 1. 设置 TTBR0_EL1 = next->pgd
    //    （用户态页表基址）
    // 2. flush local TLB
    // 3. 更新 ASID
    // 4. 设置 cpu_tlbstate
}
```

**关键**：
- `TTBR0_EL1` 是 ARM64 的用户态页表基址寄存器
- 切换时硬件 MMU 用新的页表
- TLB 失效——所有地址翻译重新走页表

### 7.3 switch_to：架构相关的上下文切换

ARM64 上的 switch_to 用汇编实现：

```assembly
// arch/arm64/kernel/entry.S
SYM_FUNC_START(cpu_switch_to)
    mov     x10, #THREAD_CPU_CONTEXT
    add     x8, x0, x10           // x8 = prev->thread.cpu_context
    mov     x9, sp
    stp     x9, x30, [x8], #16    // 保存 prev 的 sp 和 x30 (LR)
    stp     x19, x20, [x8], #16
    stp     x21, x22, [x8], #16
    stp     x23, x24, [x8], #16
    stp     x25, x26, [x8], #16
    stp     x27, x28, [x8], #16
    stp     x29, x8,  [x8], #16   // 保存 fp 和 cpu_context 地址

    add     x8, x1, x10           // x8 = next->thread.cpu_context
    ldp     x29, x8,  [x8], #16   // 恢复 fp 和 cpu_context 地址
    ldp     x27, x28, [x8], #16
    ldp     x25, x26, [x8], #16
    ldp     x23, x24, [x8], #16
    ldp     x21, x22, [x8], #16
    ldp     x19, x20, [x8], #16
    ldr     x9, [x8]              // 恢复 sp
    mov     sp, x9
    // x30 是 LR——切到 next 时会从这里"返回"——但其实跳到 ret 标签
    and     x29, x29, #0xf
    mov     sp, x9
    ret
SYM_FUNC_END(cpu_switch_to)
```

**关键认知**：
- 保存 19 个寄存器（x19-x29 + sp + x30）
- ARM64 的 PC 由 LR (x30) 控制——`ret` 指令从 LR 跳转
- 切到 next 时，next 的 LR 指向它的 `ret_from_fork` 或调度入口——这就是 context switch 后 CPU 跑 next 的原因

### 7.4 finish_task_switch 收尾

```c
// kernel/sched/core.c
static struct rq *finish_task_switch(struct task_struct *prev)
{
    struct rq *rq = this_rq();
    struct mm_struct *mm = rq->prev_mm;

    // 1. 释放 prev 的 mm
    if (mm)
        mmdrop(mm);

    // 2. 收尾：rseq、cgroup、RCU、schedstats、perf
    // ...

    // 3. 平衡 callback（load_balance 等）
    __balance_callbacks(rq);

    return rq;
}
```

**关键**：
- `prev` 在 finish_task_switch 时已经是"被切走的 task"——它的 mm 等需要清理
- 但**当前跑的是 next**——所以 prev 的清理是"清理别人"

### 7.5 context_switch 的不可见成本

**context_switch 不是免费的**：

| 阶段 | 耗时（典型） | 说明 |
|---|---|---|
| 拿锁 / 放锁 | ~100-500 ns | rq lock 竞争 |
| switch_mm (TTBR0 + TLB flush) | ~500 ns - 5 μs | TLB flush 慢 |
| switch_to (寄存器保存/恢复) | ~100 ns | 纯寄存器操作 |
| 调度器其他开销 | ~1-2 μs | 统计、callback |

**关键**：
- 单次 context_switch ~2-10 μs（具体看 CPU）
- 这是"线程上下文切换开销"的来源——Android 14 上切线程慢就会卡
- perfetto / ftrace 能看 sched_switch 的耗时

### 7.6 上下文切换在 perfetto 上

```bash
# 抓 sched_switch 事件
adb shell "perfetto --record -o /data/local/tmp/trace.proto \
    -e 'sched:sched_switch' --time 30"
```

UI 上看到的：

```
[CPU 0] pid=1234 (comm=main) → pid=1235 (comm=worker)
        prev_state='S'   next_pid=1235
        duration_ns=12345
```

**字段含义**：
- `prev_pid` / `prev_comm`：被切走的 task
- `next_pid` / `next_comm`：新跑起来的 task
- `prev_state`：被切走时的状态（S / R / D / ...）
- 调度条上的色块宽度 = task 跑的时间

### 7.7 为什么 context_switch 是性能瓶颈

**几个常见原因**：

1. **TLB flush**：每次 mm 切换要 flush TLB——后面访问虚拟地址都要重新翻译
2. **cache miss**：新 task 的代码 / 数据不在 cache 里——前几百 ns 是冷启
3. **rq lock 竞争**：多核调度时锁 rq 是热点
4. **抢占抖动**：抢占频率高 → context switch 频率高 → 业务跑不完

**Android 14 上的缓解**：
- **binder 线程池**：减少不必要的进程切换
- **EAS**：智能选择 idle CPU，减少冷启
- **UClamp**：减少 top-app 抢占
- **big.LITTLE 调度**：大核跑前台，小核跑后台——减少 cache miss

## 八、调度触发点：tick / preempt / voluntary

### 8.1 三大调度触发点

调度器什么时候被触发？三个地方：

```
1. tick 中断（scheduler_tick）
   ↓
   周期性检查当前 task 的时间片是否用完
   ↓
   用完则 set_tsk_need_resched()

2. preempt_enable 抢占点
   ↓
   preempt_count == 0 时检查 need_resched
   ↓
   设置则触发 schedule

3. 主动 schedule（schedule() 直接调）
   ↓
   task 主动让出 CPU
```

### 8.2 scheduler_tick：tick 中断

```c
// kernel/sched/core.c
void scheduler_tick(void)
{
    int cpu = smp_processor_id();
    struct rq *rq = cpu_rq(cpu);
    struct task_struct *curr = rq->curr;
    struct sched_class *class;

    // 1. 更新 rq 时钟
    update_rq_clock(rq);

    // 2. 当前 task 累计运行时间（task 时间记账）
    curr->sched_class->task_tick(rq, curr, 0);
    // 内部：
    // - fair: entity_tick() → check_preempt_tick() → 可能 set need_resched
    // - rt: task_tick_rt() → task 移到末尾（RR 调度）
    // - dl: task_tick_dl() → 检查 deadline

    // 3. 多核调度相关
    trigger_load_balance(rq);
    // 触发软中断 SCHED_SOFTIRQ → run_rebalance_domains → load_balance
}
```

**关键**：
- tick 中断每秒触发 `HZ` 次（Android 14 通常 `CONFIG_HZ=250` 或 `300`）
- 每次 tick 都检查当前 task——"该不该让出 CPU"
- tick 也是触发多核负载均衡的入口

### 8.3 preempt_enable 抢占点

```c
// kernel/sched/core.c
#define preempt_enable() \
do { \
    barrier(); \
    if (unlikely(preempt_count_dec_and_test())) \
        __preempt_schedule(); \
} while (0)
```

**关键**：
- 内核代码用 `preempt_disable()` / `preempt_enable()` 包裹临界区
- 退出临界区时 `preempt_count == 0` 才会检查 need_resched
- 这就是"内核态抢占"——CONFIG_PREEMPT 开启后内核可被抢占

### 8.4 主动 schedule：阻塞型系统调用

```c
// 例：read() 在 pipe 上阻塞
ssize_t read(int fd, void *buf, size_t count) {
    // ...
    if (pipe_empty) {
        // 阻塞——调 schedule_timeout
        schedule_timeout(pipe_timeout);
    }
    // ...
}

// 例：nanosleep
int nanosleep(const struct timespec *req, struct timespec *rem) {
    // ...
    schedule_timeout(nanos_to_jiffies(req));
}
```

**关键认知**：
- 阻塞型系统调用底层都调 schedule_timeout
- 醒来后看是否超时、是否被信号打断
- 这是用户态 IO 阻塞 → 内核态 schedule 的"自然"路径

### 8.5 中断上下文 vs 进程上下文

```c
// 中断上下文不能直接调 schedule
irq_handler_t my_irq_handler(int irq, void *dev_id) {
    // ...
    schedule();  // ❌ 错误！中断不能 schedule
    // ...
}

// 正确方式：deferred work
irqreturn_t my_irq_handler(int irq, void *dev_id) {
    tasklet_schedule(&my_tasklet);
    return IRQ_HANDLED;
}

void my_tasklet_func(unsigned long data) {
    // 在软中断上下文——也不能 schedule
    // 但可以 wake_up_process()
    wake_up_process(&my_thread);
}
```

**关键**：
- 中断 / 软中断上下文不能 schedule（没有 rq 上下文）
- 只能 wake_up_process 让进程去 schedule
- 这是调度器和中断协作的关键约束

## 九、Android 14 实战：调度延迟与 perfetto

### 9.1 调度延迟的概念

**调度延迟** = task 从 runnable 到真正开始执行的时间：

```
task wake_up（变成 runnable）
  ↓
  delay 1: 等待 CPU 空闲
  delay 2: 等待更高优先级 task
  delay 3: 等待调度器选中
  delay 4: context_switch 本身
  ↓
task 开始执行
```

**关键指标**：
- `task->se.sum_exec_runtime`：累计执行时间
- `task->se.vruntime`：虚拟时间（07 篇详讲）
- perfetto 的 sched_wakeup → sched_switch 间隔

### 9.2 perfetto 抓调度事件

```bash
# 抓 30 秒的调度事件
adb shell "perfetto --record -o /data/local/tmp/trace.proto \
    -e 'sched:sched_switch sched:sched_wakeup sched:sched_wakeup_new sched:sched_migrate_task' --time 30"
```

UI 上看到的事件类型：

| 事件 | 含义 |
|---|---|
| `sched_wakeup` | task 被唤醒（sleep → runnable） |
| `sched_wakeup_new` | 新 task 第一次被唤醒 |
| `sched_switch` | 一次 context_switch |
| `sched_migrate_task` | task 迁移到另一颗 CPU |
| `sched_pi_setprio` | priority inheritance 设置 |
| `sched_process_exit` | 进程退出 |

### 9.3 调度延迟的计算

```bash
# perfetto query
# 用 SQL 查 "从 wakeup 到执行"的延迟
SELECT
    ts,
    dur,
    sched_wakeup.pid,
    sched_wakeup.comm,
    sched_switch.next_comm
FROM sched_wakeup
JOIN sched_switch
  ON sched_wakeup.pid = sched_switch.next_pid
WHERE sched_wakeup.ts < sched_switch.ts
ORDER BY sched_switch.ts - sched_wakeup.ts DESC
LIMIT 10;
```

**关键**：
- 调度延迟 > 10ms 通常意味着调度问题
- Android 14 上 input 调度延迟 > 50ms 会被用户感知

### 9.4 Android 14 上的调度延迟基准

| 场景 | 典型延迟 |
|---|---|
| 应用响应 touch 事件 | < 16ms（60Hz 一帧） |
| 应用冷启动 | 50-200ms |
| 普通 context_switch | < 10 μs |
| EAS 迁移 task | < 1 ms |
| wake_up → 实际执行（idle 系统） | < 1 ms |
| wake_up → 实际执行（高负载） | 1-100 ms |

### 9.5 调度延迟的可观测性

```bash
# 1. ftrace 直接看
adb shell "echo 1 > /sys/kernel/tracing/events/sched/sched_stat_wait/enable"
adb shell "echo 1 > /sys/kernel/tracing/tracing_on"
adb shell "cat /sys/kernel/tracing/trace_pipe"
```

**输出**：

```
sched_stat_wait: task=system_server delay=1.234ms
sched_stat_sleep: task=worker delay=0.567ms
sched_stat_iowait: task=loader delay=2.345ms
```

**关键**：
- `sched_stat_wait`：在 runqueue 上等待的时间
- `sched_stat_sleep`：在睡眠状态的时间
- `sched_stat_iowait`：等 IO 的时间

### 9.6 binder 调用作为调度延迟的代理

Android 14 上 binder 调用耗时**间接反映调度延迟**：

```bash
# 看 binder 调用耗时
adb shell "dumpsys binder_calls_stats | head -30"
```

输出（典型）：

```
Binder call stats for system_server:
  PIDs: 1234 5678 9012
  Top slow calls (last 5s):
    call to ActivityManager.getTasks: avg 1234μs max 5678μs
    call to InputManager.injectInput: avg 234μs max 1234μs
```

**关键**：
- binder 调用包含"调度延迟 + 业务耗时"
- binder 调用慢往往是调度问题
- "binder 延迟" 是 Android 14 性能问题的常见指标

## 十、cgroup v1/v2 + cpu.uclamp 怎么影响调度

### 10.1 cgroup v1 的 CPU 控制

```bash
# cgroup v1 路径（旧版 Android）
adb shell "ls /dev/cpuctl/"
# cgroup.procs  cpu.cfs_period_us  cpu.cfs_quota_us  cpu.shares  tasks
```

**关键**：
- v1 用 `cpu.cfs_quota_us` / `cpu.cfs_period_us` 控制 CPU 时间
- v1 用 `cpu.shares` 控制 CPU 权重（跟 CFS weight 相关）

### 10.2 cgroup v2 的 CPU 控制

```bash
# cgroup v2 路径（Android 14）
adb shell "ls /sys/fs/cgroup/top-app.slice/"
# cgroup.procs  cpu.max  cpu.weight  cpu.uclamp.min  cpu.uclamp.max
```

**关键**：
- v2 用 `cpu.max`（`quota period`）控制带宽
- v2 用 `cpu.weight` 控制权重
- v2 新增 `cpu.uclamp.min/max`——UClamp 调度提示

### 10.3 UClamp 怎么影响调度

```c
// include/linux/sched.h
struct task_struct {
    // ...
    struct uclamp_se uclamp_req[UCLAMP_CNT];   // 任务请求的 UClamp
    struct uclamp_se uclamp[UCLAMP_CNT];        // 实际生效的 UClamp
};

struct uclamp_se {
    unsigned int value;          // 0-1024（百分比 × 10）
    unsigned int bucket_id;      // bucket 索引
};
```

**关键**：
- `uclamp_min`：task 的最低利用率保证——调度器会"挤"出这个 CPU 时间
- `uclamp_max`：task 的最高利用率上限——调度器不让它跑得比这高
- 作用于 fair 类（07 篇详讲）

### 10.4 Android 14 的 UClamp 配置

```bash
# 1. top-app slice 的 UClamp
adb shell "cat /sys/fs/cgroup/top-app.slice/cpu.uclamp.min"
# 0   ← 默认 min 0%（无保证）
adb shell "cat /sys/fs/cgroup/top-app.slice/cpu.uclamp.max"
# max ← 默认 max 100%

# 2. background slice 的 UClamp
adb shell "cat /sys/fs/cgroup/background.slice/cpu.uclamp.min"
# 0

# 3. system slice 的 UClamp
adb shell "cat /sys/fs/cgroup/system.slice/cpu.uclamp.min"
# 0
```

**关键**：
- 默认都是 0 / max——不约束
- Android 14 上 Framework 可调这些参数——比如 `libprocessgroup` 或 `TaskOrganizer`
- 这是 Android 14 调度优化（"前台优先"）的关键开关

### 10.5 UClamp 与 CFS 的耦合

```c
// kernel/sched/fair.c uclamp_eff_value
// 影响 pick_next_task 的选 task 决策
// task 实际利用率受 uclamp 约束
// uclamp_min > 0 时，调度器倾向于选这个 task
// uclamp_max < max 时，调度器限制这个 task 跑太久
```

**关键**：
- UClamp 是"软"控制——不像 cgroup 配额那么硬
- 但 UClamp 影响 CFS 的 task 选择——可以"挤"出 top-app 的 CPU 时间
- 09 篇（多核调度）会展开
