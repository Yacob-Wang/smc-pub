## 一、调试工具全景

### 1.1 工具层次

```
用户态工具：
  - strace：跟踪系统调用
  - ltrace：跟踪库调用
  - gdb：源码级调试
  - top / htop：进程状态
  - ps：进程列表
  - vmstat / mpstat：系统状态

Android 14 工具：
  - dumpsys：系统服务状态
  - logcat：日志
  - perfetto：性能 trace
  - atrace：Android trace
  - bugreport：完整报告

/proc / sys：
  - /proc/<pid>/{status, stat, maps, sched, oom_score_adj}
  - /proc/sched_debug
  - /proc/pressure/{cpu, memory, io}
  - /sys/kernel/debug/{sched_features, tracing}

内核态工具：
  - ftrace：函数跟踪
  - bpftrace：BPF 跟踪
  - perf：性能分析
  - kprobes：动态插桩
  - tracepoints：静态跟踪点
```

### 1.2 Android 14 上的关键工具

```
最常用的 5 个：
1. logcat -b crash        ← 看 crash / SIGSEGV
2. dumpsys meminfo        ← 看内存
3. dumpsys binder_calls_stats  ← 看 binder
4. perfetto --record      ← 性能 trace
5. cat /proc/<pid>/status ← 看进程状态

性能问题的 5 个：
1. perfetto --record -e sched:sched_switch
2. perfetto --record -e sched:sched_wakeup
3. ftrace /sys/kernel/debug/tracing
4. bpftrace
5. /proc/pressure/memory
```

**关键认知**：
- 80% 的问题用前 5 个工具能定位
- 性能问题需要 perfetto + ftrace + bpftrace

### 1.3 工具的优先级

```
第一层：定位现象
  - logcat -b crash
  - logcat -b main
  - ps / top

第二层：定位资源
  - dumpsys meminfo
  - /proc/<pid>/status
  - /proc/sched_debug

第三层：定位 trace
  - perfetto --record
  - ftrace

第四层：深度排查
  - bpftrace
  - perf
  - kprobes
```

**关键**：
- 第一层就能解决 50% 问题
- 性能问题需要第三、四层

## 二、ftrace 实战

### 2.1 ftrace 是什么

**ftrace** = **F**unction **Trace**，内核内置的 trace 工具。

**关键认知**：
- ftrace 通过 tracepoint（静态） / kprobe（动态）跟踪
- 输出写到 ring buffer
- Android 14 默认开启 tracefs

### 2.2 ftrace 的入口

```bash
# 1. 看 ftrace mount
adb shell "ls /sys/kernel/debug/tracing/"

# 2. 看可用的 tracepoint
adb shell "ls /sys/kernel/debug/tracing/events/sched/ | head -20"

# 3. 看某个 tracepoint 的格式
adb shell "cat /sys/kernel/debug/tracing/events/sched/sched_switch/format"

# 4. 启用某个 tracepoint
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_switch/enable"

# 5. 启用 tracing
adb shell "echo 1 > /sys/kernel/debug/tracing/tracing_on"

# 6. 看输出
adb shell "cat /sys/kernel/debug/tracing/trace_pipe"

# 7. 关闭
adb shell "echo 0 > /sys/kernel/debug/tracing/tracing_on"
```

**关键**：
- `events/` 下面是所有 tracepoint
- `tracing_on` 是全局开关
- `trace_pipe` 是流式输出
- `trace` 是 buffer 输出

### 2.3 sched_switch trace

```bash
# 启用 sched_switch
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_switch/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/tracing_on"

# 跑一段时间后停止
adb shell "sleep 10 && echo 0 > /sys/kernel/debug/tracing/tracing_on"

# 看输出
adb shell "cat /sys/kernel/debug/tracing/trace | head -50"
```

**输出格式**：

```
# tracer: nop
#
# entries-in-buffer/entries-written: 1000/1000   #P:8
#
#                                      _-----=> irqs-off
#                                     / _----=> need-resched
#                                    | / _---=> hardirq/softirq
#                                   || / _--=> preempt-depth
#                                   ||| /     delay
#           TASK-PID    TGID   CPU#  ||||    TIMESTAMP  FUNCTION
#              | |        |      |   ||||       |         |
      system_server-1234 (1234) [001] d..3    12.345678: sched_switch: prev_comm=system_server prev_pid=1234 prev_prio=120 ... ==> next_comm=app_thread next_pid=5678 next_prio=120
```

**字段解读**：
- `d..3`：delay 标记
- `12.345678`：时间戳（秒）
- `sched_switch`：事件类型
- `prev_*` / `next_*`：切换前后的 task

### 2.4 sched_wakeup trace

```bash
# 看 wakeup 事件
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_wakeup/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_wakeup_new/enable"

# 输出例子：
# kworker/0:1-89  [000] d..3   12.345678: sched_wakeup: comm=app_thread pid=5678 prio=120 target_cpu=001
```

**关键**：
- `sched_wakeup`：task 被唤醒（变成 runnable）
- `target_cpu`：被唤醒的目标 CPU
- `prio`：被唤醒 task 的优先级

### 2.5 signal 事件 trace

```bash
# 信号事件
adb shell "echo 1 > /sys/kernel/debug/tracing/events/signal/signal_generate/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/signal/signal_deliver/enable"

# 输出例子：
# system_server-1234 [001] d..3  12.345678: signal_generate: sig=9 (SIGKILL) errno=0 code=0 comm=system_server pid=1234
# app_thread-5678    [001] d..3  12.345679: signal_deliver: sig=9 (SIGKILL) irq=0 res=0
```

**关键**：
- `signal_generate`：信号产生
- `signal_deliver`：信号投递
- 这是 11 篇知识的具体应用

### 2.6 binder 事件 trace

```bash
# binder 事件
adb shell "echo 1 > /sys/kernel/debug/tracing/events/binder/binder_transaction/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/binder/binder_transaction_received/enable"

# 输出例子：
# app-1234          [001] d..3  12.345678: binder_transaction: transaction=12345 dest_node=1 dest_proc=1000 dest_thread=0 reply=0
# system_server-1000 [001] d..3  12.345679: binder_transaction_received: transaction=12345
```

**关键**：
- `binder_transaction`：发起事务
- `binder_transaction_received`：接收事务
- 这是 12 篇知识的具体应用

### 2.7 ftrace 的过滤

```bash
# 过滤特定 PID
adb shell "echo 1234 > /sys/kernel/debug/tracing/set_ftrace_pid"

# 过滤特定 CPU
adb shell "echo 1 > /sys/kernel/debug/tracing/tracing_cpumask"

# 过滤特定函数
adb shell "echo schedule > /sys/kernel/debug/tracing/set_ftrace_filter"

# 清除过滤
adb shell "echo > /sys/kernel/debug/tracing/set_ftrace_pid"
```

**关键**：
- ftrace 支持过滤——减少无关输出
- 排查特定 PID 时很有用

## 三、perfetto 实战

### 3.1 perfetto 是什么

**perfetto** = Google 主导的 trace 工具（替代 atrace）。

**关键认知**：
- 统一 trace 格式——支持 ftrace + atrace + 用户态 annotation
- Web UI 可视化——调度条 / 火焰图 / 计数器
- Android 14 默认 trace 工具

### 3.2 perfetto 的使用

```bash
# 1. 简单 trace
adb shell "perfetto --record -o /data/local/tmp/trace.proto --time 30"
# 跑 30 秒

# 2. 指定 events
adb shell "perfetto --record -o /data/local/tmp/trace.proto \
    -e 'sched:sched_switch sched:sched_wakeup' --time 30"

# 3. 多个 events
adb shell "perfetto --record -o /data/local/tmp/trace.proto \
    -e 'sched:sched_switch sched:sched_wakeup binder:transaction' --time 30"

# 4. 看 config
adb shell "perfetto --record -c /data/local/tmp/trace_config.txt --time 30"
```

**关键**：
- `-e` 选 events
- `-o` 选输出
- `--time` 选时长

### 3.3 perfetto config 详解

```text
# /data/local/tmp/trace_config.txt

# 1. duration
duration_ms: 30000

# 2. buffers
buffers {
    size_kb: 65536
    fill_policy: DISCARD
}

# 3. data sources
data_sources {
    config {
        name: "linux.ftrace"
        ftrace_config {
            ftrace_events: "sched:sched_switch"
            ftrace_events: "sched:sched_wakeup"
            ftrace_events: "sched:sched_wakeup_new"
            ftrace_events: "sched:sched_migrate_task"
            ftrace_events: "sched:sched_process_exit"
            ftrace_events: "signal:signal_generate"
            ftrace_events: "signal:signal_deliver"
            ftrace_events: "binder:transaction"
            ftrace_events: "binder:transaction_received"
            ftrace_events: "futex:futex_wait_start"
            ftrace_events: "futex:futex_wake"
            atrace_categories: "am"
            atrace_categories: "pm"
            atrace_categories: "view"
            atrace_categories: "gfx"
            atrace_categories: "input"
            atrace_categories: "wm"
        }
    }
}

# 4. 输出
write_into_file: true
output_path: "/data/local/tmp/trace.proto"
```

**关键**：
- perfetto config 是 protobuf 格式
- 可以指定多个 ftrace_events
- atrace_categories 是 Android 特有的
- 输出 .proto 文件

### 3.4 perfetto UI 看调度条

```bash
# 把 trace 拉出来
adb pull /data/local/tmp/trace.proto

# 在 UI 打开：https://ui.perfetto.dev/
```

**关键事件**：
- **调度条**（每个 task 一条）
  - 颜色：执行状态
  - 长度：执行时间
  - 紧邻的：调度切换
- **wakeup 事件**（箭头）
  - 从 wakeup 到执行：调度延迟
- **binder transaction**（彩色块）
  - 从发起 → 接收 → reply

### 3.5 perfetto SQL 查询

```sql
-- 找调度延迟 > 10ms 的事件
SELECT
    sched_wakeup.ts AS wakeup_ts,
    sched_switch.ts AS run_ts,
    sched_switch.ts - sched_wakeup.ts AS delay_ns,
    sched_wakeup.pid,
    sched_wakeup.comm
FROM sched_wakeup
JOIN sched_switch
  ON sched_wakeup.pid = sched_switch.next_pid
WHERE sched_wakeup.ts < sched_switch.ts
  AND (sched_switch.ts - sched_wakeup.ts) > 10000000
ORDER BY delay_ns DESC
LIMIT 20;
```

**关键**：
- perfetto UI 内置 SQL 查询
- 能聚合分析——找规律

### 3.6 perfetto 看 binder 链

```sql
-- 找 binder 调用链
SELECT
    s.ts AS start_ts,
    s.tid AS caller_tid,
    s.idx AS transaction_id,
    r.ts AS received_ts,
    r.ts - s.ts AS latency_ns,
    r.tid AS callee_tid
FROM binder_transaction s
LEFT JOIN binder_transaction_received r
  ON s.idx = r.idx
ORDER BY latency_ns DESC
LIMIT 20;
```

**关键**：
- 能看到 binder 端到端延迟
- 这是排查 binder 慢的关键

```bash

```

**输出格式**：

```

7f9a8c0000-7f9b000000 r-xp 00000000 fd:05 1234  /system/bin/linker64
```

### 4.7 /proc/sched_debug

```bash
adb shell "cat /proc/sched_debug" | head -100
```

**关键字段**：
- `cfs_rq[N]`：每个 CPU 的 CFS runqueue
  - `exec_clock`
  - `min_vruntime`
  - `tasks_timeline`
  - `load_avg` / `util_avg`
- `rt_rq[N]`：每个 CPU 的 RT runqueue
- `dl_rq[N]`：每个 CPU 的 DL runqueue
- 任务详情：每个 task 的 vruntime / exec_start 等

**关键**：
- /proc/sched_debug 是调度器内部状态的全量 dump
- 调试调度问题时必看

### 4.8 /proc/pressure/\{cpu,memory,io\}

```bash
# memory pressure
adb shell "cat /proc/pressure/memory"
# some avg10=0.00 avg60=0.00 avg300=0.00 total=0
# full avg10=0.00 avg60=0.00 avg300=0.00 total=0

# cpu pressure
adb shell "cat /proc/pressure/cpu"
# some avg10=12.34 avg60=8.90 avg300=5.67 total=1234567

# io pressure
adb shell "cat /proc/pressure/io"
# some avg10=0.00 avg60=0.00 avg300=0.00 total=0
# full avg10=0.00 avg60=0.00 avg300=0.00 total=0
```

**关键认知**：
- `some`：至少 1 个任务受影响
- `full`：所有任务受影响
- `avg10/60/300`：10 秒/60 秒/300 秒平均
- `total`：累计 stall 时间

**PSI 的意义**：
- `cpu.pressure > 10%`：CPU 受压力
- `memory.pressure > 20%`：内存受压力
- `io.pressure > 5%`：IO 受压力

## 六、实战案例 1：卡顿定位

### 6.1 现象

```
用户反馈：
  - 应用启动慢（5 秒）
  - 滑动卡顿（> 16ms）
  - ANR（5 秒无响应）
```

### 6.2 排查步骤

**Step 1：看现象**

```bash
# 1.1 看进程状态
adb shell "ps -A | grep com.example.app"
# 看进程是否活着

# 1.2 看应用是否在前台
adb shell "dumpsys activity processes | grep -A 5 com.example.app"
# 看 procstate

# 1.3 看 CPU
adb shell "top -p $(pidof com.example.app) -n 1"
# 看 CPU 占用
```

**Step 2：抓 trace**

```bash
# 2.1 perfetto 抓 30 秒
adb shell "perfetto --record -o /data/local/tmp/lag_trace.proto \
    -e 'sched:sched_switch sched:sched_wakeup binder:transaction' \
    --time 30"

# 2.2 触发问题场景
# - 滑动
# - 启动应用
# - 跳转页面

# 2.3 拉出 trace
adb pull /data/local/tmp/lag_trace.proto
```

**Step 3：分析 trace**

```sql
-- 找调度延迟 > 16ms（一帧）
SELECT
    sched_wakeup.ts AS wakeup_ts,
    sched_switch.ts AS run_ts,
    sched_switch.ts - sched_wakeup.ts AS delay_ns,
    sched_wakeup.comm,
    sched_wakeup.pid
FROM sched_wakeup
JOIN sched_switch
  ON sched_wakeup.pid = sched_switch.next_pid
WHERE sched_wakeup.ts < sched_switch.ts
  AND (sched_switch.ts - sched_wakeup.ts) > 16000000
  AND sched_wakeup.comm = 'com.example.app'
ORDER BY delay_ns DESC
LIMIT 20;
```

**Step 4：找根因**

根据 trace 分析，常见原因：

| 现象 | 根因 |
|---|---|
| main thread 调度延迟 | main thread 没在大核跑 |
| binder 调用慢 | system_server 拥塞 |
| wakeup 后 long delay | CPU 满载 |
| voluntary_ctxt_switches 高 | pthread_mutex 竞争 |
| 非自愿切换（preempt）高 | RT task 抢占 |

**Step 5：解决**

```bash
# 5.1 调 framework 配置
# - main thread 跑在大核：cpuset
# - 减少 binder 调用：合并
# - 减少锁竞争：减少临界区

# 5.2 应用层优化
# - 用协程（Kotlin）
# - 减少主线程 IO
# - 预加载

# 5.3 系统层优化
# - UClamp 调整
# - 调整 cgroup 配额
```

### 6.3 案例分析：滑动卡顿

```
Trace 显示：
  - main thread 调度延迟 18ms
  - main thread 跑了 12ms
  - 然后被 RT 任务抢占
  - RT 任务跑了 5ms（audio 处理）
  - main thread 回来跑 1ms
  - 这一帧用了 36ms（> 16ms = 卡顿）

根因：
  - audio RT 任务抢占 main thread
  - main thread 不能完全跑 16ms
  - 滑动掉帧

解决：
  - 把 audio RT 任务优先级降低
  - 或把 main thread 优先级提高（CFS 类内部）
  - 或减少 audio 处理时间
```

**关键认知**：
- 卡顿通常是多因素叠加
- perfetto 能定位"哪一段"出问题
- 根因可能是 main thread / binder / RT 抢占

## 七、实战案例 2：ANR 系统侧归因

### 7.1 现象

```
logcat:
  ANR in com.example.app
  Reason: Input dispatching timed out (Application does not have a focused window)
```

### 7.2 排查步骤

**Step 1：看 ANR 类型**

```bash
# 1.1 看 logcat
adb logcat -b crash | grep -A 30 "ANR in"

# 1.2 看 dropbox
adb shell "dumpsys dropbox | grep -A 50 'com.example.app'"
```

**ANR 类型**：

| 类型 | 触发条件 | 根因方向 |
|---|---|---|
| Input dispatching | 输入事件 5 秒无响应 | main thread 卡 |
| BroadcastQueue | 广播 60 秒无处理 | receiver 慢 |
| Service | Service 20 秒无响应 | service 慢 |
| ContentProvider | provider query 慢 | provider 慢 |
| InputEventInjection | input event 注入慢 | input 路径卡 |

**Step 2：看主线程做了什么**

```bash
# 2.1 看主线程栈
adb shell "cat /proc/$(pidof com.example.app)/task/<main_tid>/stack"
# 看到主线程在等什么 / 卡在哪

# 2.2 看 systrace
adb shell "perfetto --record -o /data/local/tmp/anr_trace.proto \
    -e 'sched:sched_switch signal:signal_generate binder:transaction' \
    --time 30"
# 看到 ANR 之前主线程的状态
```

**Step 3：系统侧归因**

```sql
-- 找 ANR 期间所有主线程调度事件
SELECT
    ts,
    prev_comm,
    prev_state,
    next_comm,
    next_pid
FROM sched_switch
WHERE next_pid = <main_tid>
  AND ts > <anr_start_ts>
ORDER BY ts
LIMIT 100;
```

**常见系统侧 ANR 原因**：

| 现象 | 根因 | 解决 |
|---|---|---|
| main thread 在等 binder reply | system_server 慢 | 排查 system_server |
| main thread 被 RT 抢占 | audio / display HAL bug | 排查 HAL |
| main thread 在等 GC | Java heap 满 | 排查内存 |
| main thread 在等 IO | 文件系统慢 | 排查 IO |
| main thread 在等 cgroup throttle | background slice | 排查 cgroup |

### 7.3 案例分析：binder reply 延迟导致 ANR

```
Trace 显示：
  - main thread 调 ActivityManager.foo()
  - 调 binder_transaction
  - 等 reply——但 system_server 拥塞
  - 5 秒后：Input event timeout
  - ANR

根因：
  - 不是应用的问题——是 system_server 拥塞
  - 应用能做的就是减少 binder 调用

解决：
  - 应用层：减少主线程 binder 调用
  - framework 层：优化 system_server 处理
  - 系统层：增加 system_server 资源
```

**关键认知**：
- ANR 不一定是应用 bug——系统侧问题也会
- 必须看到 main thread 5 秒的 trace 才能定位
- perfetto 是核心工具

## 八、实战案例 3：LMKD 误杀排查

### 8.1 现象

```
用户反馈：
  - 应用被频繁杀掉
  - 重启后又被杀
  - logcat: LowMemoryKiller killing com.example.app
```

### 8.2 排查步骤

**Step 1：看 LMKD 日志**

```bash
# 1.1 看 LMKD 日志
adb shell "dmesg | grep -i 'lmk\|lowmem'"
# 输出: lowmemorykiller: Killing 'com.example.app' (pid 1234) (adj 800) \
#        to free 102400kB on behalf of system (12345)

# 1.2 看 LMKD 状态
adb shell "dumpsys lmkd"
```

**Step 2：看 PSI 和 cgroup**

```bash
# 2.1 看 memory pressure
adb shell "cat /proc/pressure/memory"
# some avg10=80.00 avg60=60.00 avg300=40.00 total=1234567890
# 高——内存压力大

# 2.2 看 cgroup memory
adb shell "for slice in \$(ls /sys/fs/cgroup/); do
    if [ -f /sys/fs/cgroup/\$slice/memory.current ]; then
        current=\$(cat /sys/fs/cgroup/\$slice/memory.current)
        max=\$(cat /sys/fs/cgroup/\$slice/memory.max 2>/dev/null)
        echo \"\$slice: \$current / \$max\"
    fi
done"
```

**Step 3：找被杀的进程**

```bash
# 3.1 看进程 adj
adb shell "cat /proc/<pid>/oom_score_adj"
# 800 —— 易杀

# 3.2 看应用最近被杀时间
adb shell "dumpsys activity processes | grep -A 5 com.example.app"
```

### 8.3 LMKD 决策路径

```
LMKD 决策：
  1. 监听 PSI / cgroup memory 压力
  2. PSI 超过阈值时扫描进程
  3. 按 oom_score_adj 从高到低选 victim
  4. 检查 cgroup memory 占用
  5. 杀 victim（kill -9）

"误杀" 通常原因：
  1. 应用 adj 设错——应该不易杀但被杀了
  2. 应用内存占用大——但应该有大内存配额
  3. cgroup 配额错——应用被 quota 限制
  4. LMKD 阈值错——杀得太激进
```

### 8.4 案例分析：background 应用频繁被杀

```
现象：
  - 后台应用频繁被杀
  - logcat 显示 adj 800
  - LMKD 日志显示 PSI 80%

根因：
  - background slice cpu.max 30% + memory.max 500MB
  - 应用占用内存超 500MB
  - LMKD 杀掉

解决：
  1. 应用层：减少内存占用
  2. framework 层：增加 background slice 的 memory.max
  3. 系统层：调 LMKD 阈值
```

### 8.5 案例分析：cached 应用被误杀

```
现象：
  - cached 应用被 LMKD 杀
  - 用户切回时需要冷启动

根因：
  - cached 应用 adj 900（默认 cached）
  - PSI 高时被 LMKD 杀
  - cached 本来就是可杀——不算误杀

"误杀" 的判断：
  - cached 应用被杀不算 bug
  - 但如果用户切回时被杀的体验差
  - 解决：增加 memory 或减少 cached 应用数量
```

**关键认知**：
- "误杀"判断：cached 应用被杀是设计
- 真正的误杀：foreground / top-app / service 被杀
- 排查方向：adj 配置 + cgroup 配置 + LMKD 阈值

## 九、bpftrace 进阶

### 9.1 bpftrace 是什么

**bpftrace** = 基于 **eBPF** 的高级 trace 工具。

**关键认知**：
- bpftrace 比 ftrace 更灵活——可以自定义脚本
- 基于 BPF——内核态安全执行
- Android 14 上需要 root

### 9.2 bpftrace 的使用

```bash
# 1. 简单的 syscall 计数
adb shell "bpftrace -e 'kprobe:sys_clone { @[comm] = count(); }'"

# 2. 看 wakeup 延迟
adb shell "bpftrace -e '
kprobe:try_to_wake_up {
    $ts = nsecs;
    @start[arg1] = $ts;
}

kretprobe:try_to_wake_up /@start[arg1]/ {
    $delay = nsecs - @start[arg1];
    @delay = hist($delay);
    delete(@start[arg1]);
}
'"

# 3. 看 task 在 cpu 上的 util
adb shell "bpftrace -e '
kprobe:update_curr {
    @util[arg0] = arg1;
}
' 5"
```

**关键**：
- bpftrace 语法：kprobe / kretprobe + 内置函数
- 输出：hist / count / avg
- 灵活但需要学习

### 9.3 bpftrace vs ftrace

| 维度 | ftrace | bpftrace |
|---|---|---|
| 灵活度 | 中（filter / tracepoint） | 高（自定义脚本） |
| 学习成本 | 低 | 高 |
| 性能开销 | 低 | 中 |
| 输出格式 | 文本 | 自定义 |
| Android 14 | 默认 | 需要 root |

**关键**：
- ftrace 用于标准场景——足够了
- bpftrace 用于深度分析——灵活

### 9.4 bpftrace 实战案例

```bash
# 1. 看所有 cgroup OOM 触发
adb shell "bpftrace -e '
kprobe:mem_cgroup_out_of_memory {
    printf(\"OOM: cgroup=%s pid=%d\n\", arg0->name, arg1);
}
'"

# 2. 看所有 binder transaction 大小
adb shell "bpftrace -e '
kprobe:binder_transaction {
    $t = (struct binder_transaction_data *)arg1;
    @size = hist($t->data_size);
}
'"

# 3. 看所有 task 调度延迟
adb shell "bpftrace -e '
kprobe:sched_wakeup {
    @start[arg1] = nsecs;
}

kretprobe:finish_task_switch /@start[arg1]/ {
    @delay = hist(nsecs - @start[arg1]);
    delete(@start[arg1]);
}
'"
```

**关键**：
- bpftrace 能写很深的探针
- 适用于 ftrace 不够用的场景
