## 第 1 章 引子：内存问题的 5 类症状

### 1.1 5 类症状（用户视角）

| 症状 | 用户语言 | 内存相关特征 | 五大类问题对应 |
|------|---------|------------|--------------|
| **卡顿** | "App 怎么这么慢"、"滑动掉帧" | 主线程 STW > 16ms、PSI `full` > 5%、Input 分发延迟 | 抖动 + 卡顿 |
| **闪退** | "App 一打开就崩"、"突然没了" | `OutOfMemoryError`、堆栈含 `alloc`/`grow`、LMKD 杀进程 | OOM + 杀进程 |
| **杀进程** | "我后台的 App 被关了"、"回到 App 重启了" | `Killed by LMK`、`Sent SIGKILL to`、`low memory on global` | 杀进程 |
| **电量高** | "手机发烫"、"待机耗电多" | GC 占用 CPU > 30%、Direct Reclaim 长时间占用、App 后台 | 抖动 + 泄漏 |
| **磁盘满** | "存储空间不足"、"微信装不了" | `/data` 使用率 > 95%、`/data/local/tmp` 满、logcat 写入失败 | 泄漏（log 累积） |

> **稳定性架构师视角**：用户描述的"症状"与底层"根因"经常隔了 2-3 层。例如用户说"App 卡顿"，根因可能是 GC 抖动 → Direct Reclaim → LMKD 杀错关键进程 → 进程重启 → 重新加载慢。**5 类症状是入口，工具链才是路径**。

### 1.2 症状与工具的对应（速查表）

| 症状 | 第一工具 | 第二工具 | 进阶工具 |
|------|---------|---------|---------|
| 卡顿 | `dumpsys gfxinfo <pkg>` | `simpleperf record -g` | perfetto trace |
| 闪退（OOM） | `dumpsys meminfo <pid>` | `/proc/<pid>/smaps_rollup` | ftrace `lowmemorykiller_kill` |
| 杀进程 | `/proc/lmkd/debug` | `dumpsys meminfo` | perfetto `lowmemorykiller_kill` tracepoint |
| 电量高 | `dumpsys batterystats` | `simpleperf record -e cpu-cycles` | perfetto `power` track |
| 磁盘满 | `dumpsys diskstats` | `/data` df | logcat ring buffer 是否满 |

## 第 2 章 工具链全景：5 层观察视角

### 2.1 5 层观察视角（从内核到 Framework）

内存问题诊断必须**多视角交叉验证**。任何一个数字单独看都会失真，必须从 5 层同时取样：

```
┌────────────────────────────────────────────────────────────┐
│ Layer 5: perfetto / simpleperf / Systrace                   │  ← 时间线视角
│         （ftrace_events_mm_vmscan + gpu_mem + process_memory）│
├────────────────────────────────────────────────────────────┤
│ Layer 4: 用户态 daemon                                     │  ← Framework 视角
│         （lmkd / ActivityManager / SurfaceFlinger）         │
├────────────────────────────────────────────────────────────┤
│ Layer 3: 工具命令层（dumpsys / procrank / procmem）        │  ← 进程级视角
│         （dumpsys meminfo / procstats / gfxinfo / memtrack）│
├────────────────────────────────────────────────────────────┤
│ Layer 2: 内核调试节点（debugfs / sysfs）                    │  ← 内核子系统视角
│         （/sys/kernel/debug/lmkd、/sys/kernel/mm/vmscan/*） │
├────────────────────────────────────────────────────────────┤
│ Layer 1: 内核 procfs 接口                                  │  ← 内核全局视角
│         （/proc/meminfo、/proc/vmstat、/proc/zoneinfo）     │
└────────────────────────────────────────────────────────────┘
```

### 2.4 工具选择决策树

```
是系统全局问题（多个进程被杀）？
├─ 是 → L1 procfs（/proc/meminfo MemAvailable、/proc/vmstat pgscan_kswapd）
└─ 否（单进程问题）
     ├─ 是杀进程？
     │  ├─ 是 → L4 lmkd debug（/proc/lmkd/debug）→ L5 perfetto lowmemorykiller tracepoint
     │  └─ 否
     │       ├─ 是 OOM？
     │       │  ├─ Java OOM → L3 dumpsys meminfo + L5 ftrace mm_page_alloc
     │       │  └─ Native OOM → L3 dumpsys meminfo (Native Heap) + L5 simpleperf -e kmem_cache_alloc
     │       └─ 是泄漏？
     │           ├─ Java 堆 → dumpsys meminfo -d (Views/Activities/AppContexts)
     │           ├─ Native 堆 → dumpsys meminfo (Native Heap 多次采样)
     │           └─ 资源 → /proc/<pid>/fd 数量、/proc/<pid>/status Threads
     └─ 是卡顿？
         ├─ 是 GC → dumpsys gfxinfo + perfetto ftrace_events_mm (heap growth)
         └─ 是分配慢 → ftrace mm_vmscan_direct_reclaim_begin/end
```

> **稳定性架构师视角**：5-30 分钟的故障现场，决策树是"树形快查"——每个问题都能在 3-5 个分支内落到具体工具命令。不要用"先看什么再试什么"的线性思维，决策树更快。

## 第 3 章 内核 procfs 接口

procfs 是最底层的诊断入口——所有数字都是**内核态直接采集**，没有 Framework 层的转换开销。本章覆盖 9 个最常用的 procfs 路径。

### 3.1 /proc/meminfo

```bash
# 8GB 设备典型输出（AOSP 14）
$ adb shell cat /proc/meminfo
MemTotal:        7829700 kB        # 物理 RAM 总大小（减去内核预留）
MemFree:          312840 kB         # 完全空闲（未被任何用途占用）
MemAvailable:    2812640 kB         # 可分配内存估算（= Free + Reclaimable）
Buffers:          148200 kB         # 块设备缓冲区（块缓存的元数据）
Cached:          2103680 kB         # 页缓存（可回收）
SwapCached:        12800 kB         # swap 缓存（已 swap out 但仍在内存）
Active:          1856720 kB         # 活跃内存（最近使用过）
Inactive:        1943120 kB         # 不活跃内存（候选回收）
Active(anon):    1432100 kB         # 活跃匿名页（进程堆）
Inactive(anon):   420800 kB         # 不活跃匿名页（回收候选）
Active(file):     424620 kB         # 活跃文件页
Inactive(file):  1522320 kB         # 不活跃文件页
Unevictable:       32400 kB         # 不可回收（mlocked 等）
Mlocked:           32400 kB         # mlock() 锁定
SwapTotal:       2097148 kB         # swap/zRAM 总大小
SwapFree:        1854000 kB         # swap 剩余
Dirty:              8400 kB         # 待写盘脏页

AnonPages:       1432100 kB         # 匿名页（同 Active(anon)）

KReclaimable:     186400 kB         # 内核可回收（SLAB 可回收部分）
Slab:             412800 kB         # SLAB 分配器占用
SReclaimable:     186400 kB         # SLAB 可回收部分
SUnreclaim:       226400 kB         # SLAB 不可回收
KernelStack:       58400 kB         # 内核栈占用
PageTables:        80200 kB         # 页表占用
NFS_Unstable:          0 kB

WritebackTmp:          0 kB

VmallocUsed:       72800 kB

Percpu:             4352 kB

CmaFree:          312840 kB         # CMA 剩余
```

**关键字段速查**：

| 字段 | 含义 | 诊断阈值（经验） |
|------|------|----------------|
| `MemAvailable` | 系统当前可分配内存估算 | < 4% MemTotal → 接近 OOM 触发线 |
| `Cached` | 页缓存（可回收） | 持续 > 30% MemTotal → 文件缓存过多 |
| `SwapUsed` = `SwapTotal - SwapFree` | swap 占用 | 持续 > 50% SwapTotal → swap 压力大 |
| `AnonPages` | 匿名页（进程堆） | > 50% MemTotal → 进程堆过大 |
| `Slab` | 内核 SLAB 占用 | > 10% MemTotal → SLAB 泄漏嫌疑 |
| `Committed_AS` vs `CommitLimit` | 已 commit vs overcommit 限制 | Committed_AS > 90% CommitLimit → 触发 OOM Killer 风险 |
| `Dirty` | 脏页 | > 5% MemTotal → I/O 压力大 |

> **稳定性架构师视角**：`MemAvailable` 是最重要的单一数字——它把 Free、Reclaimable、Cached 综合后给出一个"还能分配多少"的估算。AOSP 14 内核在 4.6+ 已支持该字段，**优先看 MemAvailable 而不是 MemFree**。

**关键计数器分类**：

| 分类 | 计数器 | 含义 | 告警阈值 |
|------|-------|------|---------|
| **回收压力** | `pgscan_direct` / `pgsteal_direct` | Direct Reclaim 触发次数 | 任意分配阻塞 200ms+ 即 1 次 |
| **回收压力** | `allocstall` | 进入 slowpath 次数 | > 100/分钟 → 内存压力 |
| **回收压力** | `pgscan_kswapd` / `pgsteal_kswapd` | kswapd 异步回收 | 持续 > 1000/s → kswapd 压力大 |
| **回收压力** | `pgrefill` | refill inactive list | > 0 → 主动 refill，调试 vmscan 行为 |
| **swap** | `pswpin` / `pswpout` | swap in/out | pswpout > 1000/s → swap 风暴 |
| **page fault** | `pgfault` / `pgmajfault` | minor/major fault | pgmajfault 持续增长 → 内存不足 |
| **回收策略** | `pgactivate` / `pgdeactivate` | LRU 状态切换 | deactivate > activate → 内存压力大 |

> **稳定性架构师视角**：`allocstall` 是"分配阻塞"的权威计数器——每一次 alloc 进入 slowpath 都会 +1。线上监控必加 `allocstall > 100/分钟` 告警。**详见 [11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10)](11-内存回收-kswapd,Direct Reclaim,LRU,MGLRU(GKI 5.10).md)**。

### 3.4 /proc/buddyinfo

```bash

Node 0, zone Normal 842 240 110 56 24 8 4 2 1 0 0
```

**字段含义**：每行 11 个数字（order 0-10），分别表示当前 zone 中各 order 的空闲页块数。例如 order=3 有 14 个空闲块 = `14 × 8 pages × 4KB = 448KB` 可用连续内存。

**关键观察**：

| 观察项 | 含义 | 告警 |
|--------|------|------|
| order 0-2 高，order 3+ 为 0 | 大量碎片化，无法分配大块连续内存 | 高分辨率相机/ION 分配可能失败 |
| 所有 order 都为 0 | 内存完全耗尽 | OOM 即将触发 |
| order 8+ 有连续空闲 | 内核可分配 4MB+ 连续内存 | 健康状态 |

| `ViewRootImpl` | 窗口数 | > 10 → Window 泄漏嫌疑 |
| `Activities` | Activity 数 | > 5 → Activity 泄漏嫌疑 |
| `AppContexts` | Context 数 | > 50 → Context 泄漏 |
| `Local Binders` | 本地 Binder 引用数 | > 200 → Binder 泄漏 |
| `WebView count` | WebView 数 | > 3 → WebView 泄漏 |

> **稳定性架构师视角**：**`-d` 深度模式是 Java 堆泄漏诊断的核心入口**。连续两次 dumpsys meminfo -d 对比，`Activities/ViewRootImpl/AppContexts` 单调增长 → Java 堆泄漏。详见 [12-内存稳定性风险全景 §4.1](12-内存稳定性风险全景.md)。

## 第 6 章 simpleperf

simpleperf 是 Android 自带的 CPU 采样工具，**对内存问题诊断的价值是"找热点函数"**——GC 抖动时哪个函数在跑、Direct Reclaim 时哪个分配路径在调用。

### 6.1 simpleperf 基础命令

```bash
# 录制 30 秒，记录所有线程调用栈
$ adb shell simpleperf record -g -e cpu-cycles -o /data/local/tmp/perf.data --duration 30
$ adb pull /data/local/tmp/perf.data
$ simpleperf report -i perf.data --sort comm
```

**关键 flag**：

| flag | 含义 | 用途 |
|------|------|------|
| `-g` | 记录调用栈 | 看热点路径 |
| `-e cpu-cycles` | 采样 CPU cycles | 默认值，适合 CPU 密集型问题 |
| `-e sched:sched_switch` | 调度事件 | 看线程切换 |
| `--duration <N>` | 录制时长（秒） | 默认 10s |
| `-o <file>` | 输出文件 | 后续 report |

### 6.2 内存问题专项事件

```bash

## 第 7 章 ftrace

ftrace 是内核自带的追踪框架，**对内存问题诊断的核心价值是"看事件时序"**——GC、Direct Reclaim、LMKD 杀进程的时间关联。

### 7.1 ftrace 核心 tracepoint（5.10 真实存在）

| tracepoint | 触发时机 | 用途 |
|------------|---------|------|
| `mm_vmscan_lru_isolate` | LRU 隔离一个页 | 看回收路径 |
| `mm_vmscan_lru_shrink_inactive` | shrink inactive list | Direct Reclaim 主路径 |
| `mm_vmscan_direct_reclaim_begin` / `_end` | Direct Reclaim 起止 | **核心诊断点** |
| `mm_vmscan_kswapd_wake` / `mm_vmscan_kswapd_sleep` | kswapd 唤醒/睡眠 | kswapd 行为 |
| `mm_page_alloc` | 每次页分配 | 看分配热点 |
| `mm_page_free` | 每次页释放 | 看释放节奏 |
| `lowmemorykiller_kill` | LMKD 杀进程（旧内核 LMK） | 看杀进程上下文 |
| `kmem_cache_alloc` | SLAB 分配 | Native 内存热点 |
| `ion_alloc` / `ion_free` | ION 分配/释放 | 图形缓冲区 |

## 第 8 章 perfetto

perfetto 是 AOSP 10+ 取代 Systrace 的全栈追踪系统，**对内存问题诊断的核心价值是"全栈时间线"**——把内核事件、Framework 事件、App 事件统一在时间轴上展示。

### 8.1 perfetto trace 数据源（内存相关）

| 数据源 | 类型 | 用途 |
|--------|------|------|
| `ftrace_events_mm_vmscan` | 内核 | 回收事件（Direct Reclaim / kswapd / LRU） |
| `ftrace_events_mm_page_alloc` | 内核 | 页分配/释放 |
| `ftrace_events_lowmemorykiller` | 内核 | LMK 杀进程（旧内核） |
| `ftrace_events_ion` | 内核 | ION 分配/释放 |

## 第 9 章 实战 Case Study：OOM 全链路追踪

### 9.1 触发场景

某 OEM 反馈：某第三方 App `com.example.app` 在低端 3GB 设备上频繁闪退。现象：

1. 启动 App → 打开商品列表页 → 滑动 5 分钟 → 闪退
2. logcat 显示 `OutOfMemoryError: Failed to allocate a 16MB byte array`
3. 用户投诉：1 小时内闪退 3 次

### 9.2 收集步骤

#### 步骤 1：采集全局内存状态（L1 procfs）

```bash
# 确认系统级内存压力
$ adb shell cat /proc/meminfo | grep -E "MemTotal|MemAvailable|Cached|Slab|Swap"
MemTotal:        3072100 kB
MemAvailable:    124080 kB        # 4% → 系统级压力
Cached:          1240000 kB
Slab:             184000 kB
SwapTotal:       1048572 kB
SwapFree:         320000 kB        # swap 已用 70%
```

→ **结论**：MemAvailable 仅 4%，系统级内存压力。Swap 已用 70%，存在 swap 风暴。

#### 步骤 2：看 vmstat 回收压力（L1 procfs）

```bash
$ adb shell cat /proc/vmstat | grep -E "allocstall|pgscan|pgsteal|pswp"
allocstall 2840                    # slowpath 2840 次
pgscan_direct 5210340              # Direct Reclaim 5M+ 页
pgsteal_direct 3128900             # 实际回收 3M+
pswpout 168000                     # swap out 168K 页
```

→ **结论**：分配进入 slowpath 2840 次（高频），Direct Reclaim 严重，swap 活跃。

#### 步骤 3：看进程级内存（L3 dumpsys）

```bash
# 多次采样（间隔 30s）看趋势
$ for i in 1 2 3 4 5; do
    adb shell dumpsys meminfo com.example.app | grep "TOTAL PSS\|Native Heap\|Dalvik Heap"
    sleep 30
done
TOTAL PSS: 184200 kB       Native Heap: 84200 kB        Dalvik Heap: 32400 kB
TOTAL PSS: 224800 kB       Native Heap: 124200 kB       Dalvik Heap: 38100 kB
TOTAL PSS: 280200 kB       Native Heap: 184800 kB       Dalvik Heap: 42400 kB
TOTAL PSS: 320400 kB       Native Heap: 224200 kB       Dalvik Heap: 48100 kB
TOTAL PSS: 380200 kB       Native Heap: 284000 kB       Dalvik Heap: 52800 kB
```

→ **结论**：**Native Heap 5 分钟内从 84MB 增长到 284MB，单调递增 → Native 堆泄漏**。

#### 步骤 4：看 smaps_rollup 详细分类（L1 procfs）

```bash
$ adb shell cat /proc/18240/smaps_rollup
...
Rss:                412800 kB
Pss:                380200 kB
Private_Dirty:      340200 kB
Anonymous:          320400 kB
...
```

→ **结论**：Anonymous 320MB / PSS 380MB = 84%，是进程堆（Native 匿名分配）占比极高。

#### 步骤 5：看 lmkd 调试日志（L2 lmkd debug）

```bash
$ adb shell cat /proc/lmkd/debug
...
Last kill info:
  time: 2026-06-15 11:12:34
  pid: 18240
  uid: 10001
  process: com.example.app
  oom_score_adj: 0                  # App adj = 0（普通后台）
  min_score_adj: 906
  rss: 380 MB
  kill reason: pressure (PSI full 0.55)
  last event log:
    [11:11:50] Pressure event some=0.40 full=0.20
    [11:12:00] Pressure event some=0.85 full=0.55
    [11:12:34] Kill pid=18240 (com.example.app) adj=0 rss=380MB reason=pressure
```

→ **结论**：lmkd 因 PSI full > 0.55 杀进程，候选是 `com.example.app`（adj=0）。

#### 步骤 6：perfetto 录制 Native 分配热点（L5 perfetto）

```bash
$ adb shell perfetto \
    -o /data/local/tmp/heap.perfetto-trace \
    -t 60s \
    --buffer 8192 \
    --heap-profile-interval 4096 \
    --heap-stack-filter com.example.app

$ adb pull /data/local/tmp/heap.perfetto-trace /tmp/
```

在 perfetto UI 中打开 → "Top by size" → 看到热点：
```
Total: 280 MB
  com.example.app.util.ImageCache
    .put(Bitmap)                     120 MB (240 instances)
    .get(String)                       0 MB (no leak)
  com.example.app.network.HttpClient
    .read(ByteBuffer)                 80 MB (40 instances)
  com.example.app.view.ViewRenderer
    .createTexture(int, int)          60 MB (20 instances)
  misc                                 20 MB
```

→ **结论**：`ImageCache.put(Bitmap)` 持有 240 个 Bitmap 未释放 → **Bitmap 泄漏嫌疑**。

#### 步骤 7：ftrace 验证分配路径（L5 ftrace）

```bash
$ adb shell "echo 1 > /sys/kernel/debug/tracing/events/kmem/kmalloc/enable"
$ adb shell "echo 1 > /sys/kernel/debug/tracing/tracing_on"
sleep 30
$ adb shell "echo 0 > /sys/kernel/debug/tracing/tracing_on"
$ adb shell cat /sys/kernel/debug/tracing/trace | grep "18240" | head -20
```

### 9.3 关键发现

| 维度 | 现象 | 结论 |
|------|------|------|
| 全局 | MemAvailable 4%，slab/swap 高 | 系统级内存压力 |
| 进程 | Native Heap 84→284 MB（5 分钟） | Native 堆泄漏 |
| 杀进程 | lmkd 杀 com.example.app | adj=0 候选被杀 |
| 热点 | ImageCache.put 持有 240 个 Bitmap | Bitmap 未回收 |
| 根因 | `Bitmap.recycle()` 漏调 + 缓存无 LRU 上限 | 设计缺陷 |

### 9.4 修复建议

1. **短期**：在 ImageCache 加 LRU 上限（建议 ≤ 100 个）+ LRU 淘汰时调 `Bitmap.recycle()`
2. **中期**：把 ImageCache 改为弱引用缓存（WeakReference）
3. **长期**：增加 Native Heap 监控告警（> 100MB 持续 10 分钟告警）

### 9.5 时序图（Case Study 流程）

```
时间   现象                     工具               数据                结论
─────────────────────────────────────────────────────────────────────────────
T+0s   App 启动                 dumpsys meminfo    Native=84MB         正常
T+30s  列表页打开                /proc/meminfo      MemAvail=4%         系统级压力
T+60s  滚动 1 分钟               /proc/vmstat       allocstall=200/min  分配慢
T+90s  Native=124MB             dumpsys meminfo    增长 47MB           持续增长
T+120s GC 抖动                  perfetto           GC pause 200ms      GC STW
T+180s Native=184MB             dumpsys meminfo    累计 100MB          泄漏
T+240s Native=224MB             smaps_rollup       Private=200MB       匿名页
T+300s 闪退                     logcat             OutOfMemoryError    OOM 触发
T+302s 进程被杀                 /proc/lmkd/debug   PSI full 0.55       LMKD 杀
```

> 图 2：OOM 全链路追踪时序图。5 分钟内 Native Heap 从 84MB 增长到 284MB，最终触发 OOM。**30 秒内的多个工具采样是关键**——单工具可能只看到局部，无法串联根因。

> **稳定性架构师视角**：**5 层工具链同时取样**是本案例的核心。MemAvailable 看到全局压力，vmstat 看到分配阻塞，dumpsys meminfo 看到进程增长，lmkd debug 看到决策，perfetto 看到热点函数。**任何一个单工具都不会给出这个完整链路**。详见 [12-内存稳定性风险全景 §4.2 实战案例](12-内存稳定性风险全景.md)。

## 第 10 章 架构师 Takeaway

5 条诊断经验沉淀：

**Takeaway 1：5 层工具链同时取样，避免单点失真**
- 单看 `dumpsys meminfo` 只能看到"这个进程内存大"，看不到全局背景
- 单看 `/proc/meminfo` 只能看到"系统压力大"，看不到具体进程
- **必须 5 层同时取样**（procfs + 调试节点 + dumpsys + daemon + perfetto）

**Takeaway 2：先看全局，后看进程，最后看时间线**
- 故障现场的标准顺序：L1 procfs（30 秒）→ L3 dumpsys（2 分钟）→ L5 perfetto（10 分钟）
- 反向顺序（先 perfetto）会浪费 30 分钟看不相干的数据

**Takeaway 3：procfs 是"免费"诊断入口，优先 cat**
- `/proc/meminfo` / `/proc/vmstat` / `/proc/pressure/memory` 都不需要 root
- 每次故障现场第一件事：cat 这三个文件 → 30 秒判断全局状态

**Takeaway 4：AOSP 14 新增 `/proc/lmkd/debug` 是在线调试的核心入口**
- 不用复现问题，cat 就能看到最近一次杀进程的完整上下文
- 包含 kill 决策时序、候选进程 adj、PSI 曲线

**Takeaway 5：perfetto 的 `memory_breakdown` + `heap_profile` 是 Native 泄漏的杀手锏**
- `memory_breakdown` 看 Pss/Rss 时间趋势
- `heap_profile` 看 Native 分配热点函数 + 调用栈
- 两者结合能定位到具体代码行
