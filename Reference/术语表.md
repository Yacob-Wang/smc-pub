# 全局术语表

> **目的**：跨系列统一术语，避免"主线程 hang" / "主线程卡死" / "UI 线程卡顿"等别名漂移。
>
> **使用规则**：
> - 任何系列新引入术语前，先查本表
> - 已有术语**必须使用**，禁止"别名"（除非本表明确登记）
> - 跨系列文章里发现术语漂移，立刻开 issue 修复
>
> **维护责任**：架构师本人（每个系列 owner 可提 PR，但术语统一由架构师拍板）
>
> **基线**：AOSP 17.0.0_r1（API 37）+ Linux 6.18 LTS

## 1. 稳定性症状术语（Stability 系列专用）

> **重要说明**：本节术语由 Stability 系列首次定义，部分术语在历史文章中可能使用了别名，
> 遇到时**以本表为准**统一替换。

| 中文 | 英文 | 别名（禁止使用） | 严格定义 | 首次定义于 |
|------|------|----------------|---------|-----------|
| **ANR** | Application Not Responding | 应用无响应 | 主线程 looper 在指定超时窗口（Input 5s / Broadcast 10s / Service 20s / Provider 10s）内未响应消息 | Stability-S00 |
| **JE** | Java Exception（未捕获） | Java 崩溃 | Java/Kotlin 层 `Throwable` 沿调用栈冒泡且**未被 catch**走完主流程，导致进程退出 | Stability-S00 |
| **NE** | Native Exception | Native 崩溃 | Native 层接收到 kernel 投递的致命信号（SIGSEGV/SIGABRT/SIGBUS/SIGFPE/SIGILL/SIGSYS 等）且未被处理 | Stability-S00 |
| **SWT** | SystemServer Watchdog Timeout | SystemServer 卡死 | SystemServer 主线程 / 关键线程卡死超过 Watchdog 阈值（默认 30s）触发 AMS 杀进程或重启 | Stability-S00 |
| **HANG** | 挂起 / 卡死（未被捕获） | 软卡死、卡顿 | 进程/线程/子系统功能失效但**未被任何超时机制捕获**（区别于 ANR 和 SWT） | Stability-S00 |
| **REBOOT** | 重启（非预期） | 异常重启 | 进程或系统的非预期重启，区别于用户主动重启或 OTA 重启 | Stability-S00 |
| **KE** | Kernel Exception | 内核异常 | Kernel 异常：panic（不可恢复）/ oops（可恢复）/ BUG / WARN / hung_task / softlockup / hardlockup / RCU stall | Stability-S00 |

### 1.1 易混淆术语对的严格区分

| 对比 | 区分准则 | 谁主动检测 | 谁触发杀进程 | 谁演变为 REBOOT |
|------|---------|----------|------------|----------------|
| **ANR vs HANG** | 阈值是否被触发 | ANR 主动检测 | ANR 杀进程 | 都不直接 |
| **SWT vs ANR** | 检测对象不同 | SWT 检测 SystemServer / ANR 检测 app | 都杀进程 | 都可能 |
| **SWT vs HANG** | 是否被 Watchdog 捕获 | SWT 主动检测 | SWT 杀 SystemServer | 都可能 |
| **NE vs KE** | 崩溃发生在哪一层 | Kernel 投递信号 | KE 杀进程或重启整机 | KE 几乎必然 |
| **REBOOT vs 其他 6 类** | REBOOT 是结果态，不是触发态 | 取决于上层 | 取决于上层 | REBOOT 自身 |

> **架构师防混淆口诀**（速记）：
> - **ANR 主动 / HANG 被动**
> - **SWT 杀的是 SystemServer / ANR 杀的是 App**
> - **NE 在用户态 / KE 在内核态**
> - **REBOOT 是结果，不是原因**

## 2. 主线程相关术语（跨系列共用）

| 中文 | 英文 | 别名（禁止使用） | 严格定义 | 首次定义于 |
|------|------|----------------|---------|-----------|
| **主线程** | main thread | UI 线程、UI thread | Android App 进程创建的第一个线程，承载 Activity/View/Looper | ART-01 |
| **主线程 Looper** | main looper | 主线程消息循环 | 主线程上挂载的 Looper，循环处理 MessageQueue 中的 Message | App/Handler |
| **主线程 hang** | main thread hang | 主线程卡死 | 主线程 Looper 在指定时间窗口（≠ ANR 阈值）内未处理新消息 | Stability-S00 |
| **主线程 ANR** | main thread ANR | — | 主线程 Looper 超过 ANR 阈值（Input 5s 等）未处理 | Stability-S00 |

|------|------|----------------|---------|-----------|

## 5. 版本基线（v4 §8 强制）

> 当前系列基线**升级决定**（2026-07-17 决策日志 · 2026-07-18 基线纠正）：
> - **Framework/应用层**：AOSP `android-17.0.0_r1`（API 37）
> - **Linux 内核**：`android17-6.18`（Linux 6.18 LTS，**2025-11-30 发布，EOL Dec 2031**（kernel.org longterm，6 年 LTS）；AOSP 17 官方 GKI 内核，2026 Q2/Q3 推 6.18 分支）
> - **Linux 6.18 LTS 配套特性**：Rust 版 Binder 上主线（K 6.4-6.6 期间合入，K 6.12/6.18 生产化）、sheaves 内存分配（K 6.10 mainline 引入，6.18 保留）、io_uring 持续增强、pidfds 扩展支持内核命名空间、eBPF 加密签名、bcachefs 移除、XFS 在线 check/repair、exFAT 16x 加速、smaps_rollup（6.5+ 引入，6.18 包含）
> - **AOSP manifest 分支建议**：`android-latest-release`
>
> 存量旧文章基线**保持不动**（AOSP 14 + 5.10/5.15），等用户单点触发再升级。
