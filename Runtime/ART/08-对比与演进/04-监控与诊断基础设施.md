## 1. 背景与定义：为什么需要 ART 监控基础设施

### 1.1 一句话定义

**ART 监控与诊断基础设施是稳定性工程师排查 Java/Kotlin 层性能、ANR、Crash、内存等问题的工具集**，包括 JVMTI 实现、ART instrumentation、Systrace/Perfetto 时序追踪、Simpleperf 性能采样等。Android 14 上这些工具的能力边界和应用场景各不相同。

### 1.2 ART 监控工具全景

```

│ ART 监控与诊断工具全景                                            │
├────────────────────────────────────────────────────────────────┤

│    └─ Method Entry/Exit / Field Access / Thread Start            │

│                                                                │

│    └─ 时序追踪（线程 + 系统调用 + GC）                           │
```

## 2. JVMTI 与 ART instrumentation

### 2.1 ART JVMTI 能力边界

| 能力 | 支持 | 备注 |
| :--- | :--- | :--- |
| **Method Entry/Exit** | ✅ | 高频事件，需谨慎使用 |
| **Field Access/Modify** | ✅ | 高频事件 |
| **Class Load/Unload** | ✅ | 启动期高频 |
| **Thread Start/End** | ✅ | 启动期高频 |
| **Garbage Collection Start/Finish** | ✅ | 用于 GC 监控 |
| **Exception** | ✅ | Crash 监控 |
| **Method Modification（Retransform）** | ❌ | ART 不支持 |
| **Object Allocation** | ✅ | 可用于内存分析 |
| **Monitor Contention** | ✅ | 死锁检测 |
| **Native Method Bind** | ✅ | JNI 监控 |

### 2.2 ART instrumentation 实现

```cpp
// art/runtime/instrumentation.cc
void Instrumentation::MethodEnterEvent(Thread* thread, ArtMethod* method) {
    if (!HasMethodEntryListeners()) return;
    
    for (auto listener : method_entry_listeners_) {
        listener->MethodEnter(thread, method);
    }
}
```

### 2.3 性能影响

| 能力 | 性能损耗 | 备注 |
| :--- | :--- | :--- |
| **Method Entry/Exit** | 10-50% | 高频事件，慎用 |
| **Field Access** | 5-20% | — |
| **Class Load** | < 1% | 启动期一次 |
| **Thread Start/End** | < 1% | 启动期一次 |
| **GC Event** | < 1% | 低频 |
| **Exception** | 5-10% | Crash 时触发 |
| **Allocation** | 10-30% | 高频事件 |
| **Monitor Contention** | < 1% | 低频 |

**架构师视角**：JVMTI 是**功能强大但性能开销大**的接口。**生产环境必须谨慎使用，避免全量埋点**。

## 3. Systrace / Perfetto

### 3.1 Systrace 简介

**Systrace** 是 Android 4.3+ 的系统级追踪工具，基于 Linux `ftrace`。

**核心能力**：
- 捕获线程状态（Running / Runnable / Sleeping / IO / Uninterruptible）
- 捕获系统调用
- 捕获 ART GC 事件
- 捕获 Binder 调用

### 3.2 Perfetto 简介

**Perfetto**（https://perfetto.dev）是 Google 在 Android 10+ 推出的新一代追踪工具，是 Systrace 的继任者。

**核心能力**：
- 兼容 Systrace 接口
- 支持更细粒度的追踪
- 支持 native 进程追踪
- 支持长时追踪（小时级）
- 支持 SQL 查询

### 3.3 ART GC 事件追踪

**Perfetto 中 ART GC 事件**：

```
GC Begin: type=Concurrent, reason=Background, duration=10ms
  ├─ Mark Begin: duration=5ms
  ├─ Mark End
  ├─ Sweep Begin: duration=3ms
  └─ Sweep End
GC End: freed=10MB
```

**关键指标**：
- GC 类型（Concurrent / STW）
- GC 原因（Background / Alloc / Native / etc.）
- GC 持续时间
- 释放内存量

### 3.4 使用场景

| 场景 | 工具 | 用途 |
| :--- | :--- | :--- |
| **ANR 排查** | Perfetto + traces.txt | 锁等待 / Binder 阻塞定位 |
| **冷启动优化** | Perfetto + am start -W | 启动期耗时分析 |
| **GC 卡顿** | Perfetto | GC 暂停定位 |
| **JIT 行为** | Perfetto | JIT 编译时机 |
| **Binder 慢调用** | Perfetto | Binder transaction 时长 |

## 4. Simpleperf

### 4.1 Simpleperf 简介

**Simpleperf**（https://android.googlesource.com/platform/system/extras/+/master/simpleperf/）是 Android NDK 提供的性能分析工具。

**核心能力**：
- CPU 采样（sampling profile）
- 硬件事件（PMU counters）
- 符号化（symbols）
- 火焰图生成

### 4.2 使用场景

| 场景 | 工具 | 用途 |
| :--- | :--- | :--- |
| **CPU 热点函数** | Simpleperf record + report | 找最耗 CPU 的函数 |
| **JIT 编译开销** | Simpleperf + ART JIT log | JIT 编译时机 |
| **Native 库性能** | Simpleperf record | libart.so 等 Native 性能 |
| **Java 方法耗时** | Simpleperf + Java 符号 | Java 方法 CPU 占用 |

### 4.3 Simpleperf 与 ART 的协同

**Simpleperf 可以分析 ART 内部**：
- libart.so 中 JIT 编译开销
- libart.so 中 GC 暂停
- app_process 启动开销

## 5. 字节码插桩（APM SDK 主流方案）

### 5.1 字节码插桩 vs JVMTI

| 维度 | 字节码插桩 | JVMTI |
| :--- | :--- | :--- |
| **时机** | 编译期 | 运行期 |
| **性能损耗** | < 5% | 10-50% |
| **灵活性** | 低（编译期固定） | 高（运行时调整） |
| **稳定性** | 高（静态代码） | 中（运行时可能崩溃） |
| **适用场景** | APM 埋点 | Profiler / Hook |

### 5.2 字节码插桩工具

| 工具 | 出品方 | 特点 |
| :--- | :--- | :--- |
| **R8** | Google | AOSP 标准 |
| **Redex** | Meta | Facebook 出品 |
| **ASM** | OW2 | 通用字节码操作库 |
| **Javassist** | JBoss | 运行时字节码 |

### 5.3 APM SDK 字节码插桩示例

```java
// 字节码插桩前
public void onCreate() {
    super.onCreate();
    init();
}

// 字节码插桩后（自动注入）
public void onCreate() {
    APM.startMethod("onCreate");  // 注入
    super.onCreate();
    init();
    APM.endMethod("onCreate");    // 注入
}
```

**实现方式**：
- 使用 Gradle Transform API（在 AGP 7.0+ 已被 ASM API 替代）
- 使用 AGP 8.0+ 的 ASM Transform
- 使用 Redex 静态插桩（编译后）

## 7. bugreport 与 dropbox

### 7.1 bugreport

- logcat
- ANR traces
- Tombstone
- dumpsys meminfo
- dumpsys cpuinfo
- 等等

**生成方式**：
```bash
adb bugreport > bugreport.zip
```

### 7.2 dropbox

**dropbox** 是系统级日志缓冲区，分类存储各类事件：

- `system_app_crash`

## 8. 实战案例：某 App 冷启动慢 1500ms 排查（综合监控）

**现象**：某 App 冷启动 1500ms，远超行业标准。

**环境**：Android 14 / Pixel 6。

### 步骤 1：am start -W

```bash
adb shell am start -W com.example.app/.MainActivity
```

输出：
```
Status: ok
Activity: com.example.app/.MainActivity
TotalTime: 1500
WaitTime: 1600
```

### 步骤 2：Perfetto 抓 trace

```bash
adb shell perfetto --txt -o /data/misc/perfetto-traces/boot.txt \
  -t 30s am wm gfx view binder_driver hal
```

### 步骤 3：分析 trace

```
0.000s: Process start
0.300s: ActivityThread.main
0.500s: Application.onCreate（耗时 600ms）
1.200s: MainActivity.onCreate（耗时 200ms）
1.500s: 首帧绘制
```

### 步骤 4：Simpleperf 找热函数

```bash
simpleperf record -p <pid> -g -e cpu-cycles --duration 10
simpleperf report
```

发现：
- `App.onCreate` 调用 `SharedPreferences.load()` 占 300ms
- `App.onCreate` 调用 `ThirdPartySDK.init()` 占 200ms

### 步骤 5：修复

- SharedPreferences 延迟加载
- ThirdPartySDK 延迟初始化

### 步骤 6：验证

| 指标 | 修复前 | 修复后 |
| :--- | :--- | :--- |
| 冷启动 | 1500ms | 800ms |
| Application.onCreate | 600ms | 100ms |

## 9. 总结（架构师视角的 5 条 Takeaway）

1. **ART JVMTI 是强大的监控接口，但性能开销大**——Method Entry/Exit 开销 10-50%，**生产环境必须谨慎使用**。
2. **Perfetto 是 Android 10+ 的标准追踪工具**——替代 Systrace，支持更细粒度和长时追踪。**所有性能分析都应首选 Perfetto**。
3. **字节码插桩是 APM SDK 的主流方案**——性能损耗 < 5%，比 JVMTI 稳定得多。**生产 APM 首选字节码插桩**。
4. **dumpsys 是快速诊断的第一站**——meminfo / cpuinfo / gfxinfo 1 秒出结果。**遇到问题先 dumpsys**。
5. **bugreport 是综合问题快照**——logcat + ANR + Tombstone + dumpsys 一次性收集。**复杂问题必拉 bugreport**。

## 10. 进阶实战：综合监控方案设计

### 10.1 监控分层架构

```
┌────────────────────────────────────────────────────────────────┐
│ ART 监控分层架构                                                 │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  第 1 层：字节码插桩（APM SDK）                                   │
│    └─ APM 埋点 / 启动期监控 / 关键路径耗时                        │
│    └─ 性能损耗 < 5%                                             │
│                                                                │
│  第 2 层：dumpsys（系统级诊断）                                   │
│    └─ meminfo / cpuinfo / gfxinfo                               │
│    └─ 性能损耗 0%                                               │
│                                                                │
│  第 3 层：Perfetto / Simpleperf（性能分析）                       │
│    └─ 时序追踪 / CPU 采样 / GC 分析                              │
│    └─ 性能损耗 < 1%                                             │
│                                                                │
│  第 4 层：JVMTI / instrumentation（深度诊断）                     │
│    └─ Method Entry / Field Access / Allocation                 │
│    └─ 性能损耗 10-50%（仅用于深度诊断）                          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 10.2 排查方法论

**Step 1：dumpsys 快速诊断（30 秒）**

```bash
# 内存
adb shell dumpsys meminfo <package> | head -30

# CPU
adb shell dumpsys cpuinfo

# 渲染
adb shell dumpsys gfxinfo <package>
```

**Step 2：logcat + bugreport（5 分钟）**

```bash
# 抓 ANR / Crash
adb logcat -d -v threadtime > logcat.txt

# 拉 bugreport
adb bugreport > bugreport.zip
```

**Step 3：Perfetto 时序追踪（10 分钟）**

```bash
# 启动追踪
adb shell perfetto --txt -o /data/misc/perfetto-traces/boot.txt \
  -t 30s am wm gfx view binder_driver hal

# 拉 trace
adb pull /data/misc/perfetto-traces/boot.txt
```

**Step 4：Simpleperf CPU 分析（10 分钟）**

```bash
# 采样
adb shell simpleperf record -p <pid> -g -e cpu-cycles --duration 10

# 拉数据
adb pull /data/local/tmp/simpleperf.data

# 报告
simpleperf report
```

**Step 5：深度诊断（JVMTI / instrumentation）**

只在 Step 1-4 找不到根因时使用。
