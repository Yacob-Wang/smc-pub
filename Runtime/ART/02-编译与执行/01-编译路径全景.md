## 1. 背景与定义：三种执行模式

### 1.1 一句话定义

**ART 采用"解释器 + JIT + AOT"三态切换模式**：方法首次调用用解释器（快启动），热点方法 JIT 编译（运行时优化），下次启动用 AOT 编译产物（已优化）+ Cloud Profile 下发热点（云端优化）。

### 1.2 三种模式对比

| 模式 | 触发时机 | 性能 | 内存 | 存储 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **解释器** | 方法首次调用 | ★★ | ★ | 0 | 未编译 |
| **JIT** | 热度阈值（默认 10,000 次） | ★★★★ | ★★ | 0（运行时编译） | 热方法运行时 |
| **AOT** | 安装时 / 后台 / 下次启动 | ★★★★★ | ★★★★ | 10-100MB / App | 热方法编译产物 |

### 1.3 三态切换状态机

```
方法首次调用
  ↓
解释器执行（同时记录调用次数）
  ↓
达到 JIT 阈值（10,000 次）
  ↓
JIT 编译（后台线程，OSR 替换入口）
  ↓
JIT 代码执行（Profile 记录）
  ↓
后台 dex2oat 编译 AOT
  ↓
下次启动 → 直接 AOT 代码
```

**关键设计**：
- **混合模式** = 启动快 + 跑起来快 + 长期最优
- **Profile 跨版本累积**（Cloud Profile 让首次安装就有 AOT）
- **OSR** 让长循环方法能从解释器平滑切到 JIT

## 2. dex2oat 完整流程

### 2.1 dex2oat 是什么

**dex2oat（DEX to OAT）** 是 AOT 编译工具，把 .dex 文件编译成 .oat（OAT 文件包含编译后的本地代码 + 元数据）。

### 2.2 四种产物文件

```
┌────────────────────────────────────────────────────────────────┐
│ dex2oat 4 种产物文件                                             │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  .dex（原始字节码）                                              │
│    └─ 应用安装的原始字节码（也称 dex 文件）                       │
│                                                                │
│  .vdex（校验后的 dex）                                           │
│    └─ 包含 DEX 校验信息 + 原始 dex                                │
│    └─ Android 8.0 引入（AOSP 8+）                                │
│                                                                │
│  .odex（已优化的 dex）                                           │
│    └─ 包含已优化的 dex + 部分 AOT 代码                           │
│    └─ 主要用于系统应用（framework）                                │
│                                                                │
│  .oat（AOT 编译产物）                                             │
│    └─ 包含编译后的本地代码 + 元数据                                │
│    └─ App 启动时优先使用                                         │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 2.3 dex2oat 4 阶段

```
┌────────────────────────────────────────────────────────────────┐
│ dex2oat 编译 4 阶段                                              │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  阶段 1：OatFileAssistant 准备                                   │
│    └─ 校验输入 dex / vdex / odex                                │
│    └─ 决定 AOT 模式（speed / speed-profile / everything）         │
│                                                                │
│  阶段 2：CompilerDriver 编译                                     │
│    ├─ 方法级编译（method compiler）                              │
│    │   └─ HGraph（SSA IR）→ MIR（机器 IR）→ 机器码               │
│    ├─ 类级优化（class linker）                                    │
│    │   └─ 类链接（vtable / itable）                              │
│    └─ dex2oat 优化（inlining / 死代码消除）                       │
│                                                                │
│  阶段 3：OutputWriter 写入                                       │
│    ├─ .vdex 写入（DEX 校验信息）                                  │
│    ├─ .oat 写入（编译代码 + 元数据）                              │
│    └─ .art 写入（Image 文件）                                    │
│                                                                │
│  阶段 4：Image 写入                                              │
│    └─ boot.art / boot-core.art / boot-oat.art                    │
│    └─ 包含预加载的类 + 类加载器 + 镜像对象                        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 2.4 三种 AOT 编译模式

| 模式 | 编译范围 | 编译时机 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **everything** | 全部方法 | 安装时 | 旧版本（Android 5-6） |
| **speed** | 全部方法 | 后台 / 设备空闲 | 性能优先 |
| **speed-profile** | Profile 标记的方法 | 后台 / 设备空闲 | 平衡（Android 7+ 默认） |

**架构师视角**：Android 7+ 默认 speed-profile，只编译 Profile 标记的热点方法。**Profile 是 ART 性能优化的核心**。

### 2.5 dex2oat 性能优化技术

| 优化技术 | 说明 | 性能影响 |
| :--- | :--- | :--- |
| **方法内联（Inlining）** | 把小方法调用展开为内联代码 | +10-30% |
| **死代码消除** | 移除不可达代码 | +5-10% |
| **循环优化** | 循环展开 / 循环不变代码外提 | +20-50% |
| **SSA 优化** | 静态单赋值形式 + 常量传播 | +10-20% |
| **寄存器分配** | 线性扫描 / 图着色 | +5-15% |

## 3. PGO 与 Baseline Profile

### 3.1 什么是 PGO

**PGO（Profile-Guided Optimization）** 是基于运行时 Profile 数据的优化技术：

1. **收集阶段**：在用户设备上运行 → 收集热方法 / 类 / 启动路径
2. **Profile 文件**：把热数据保存为 `.prof` 文件
3. **优化编译**：dex2oat 根据 Profile 优先编译热方法
4. **云端下发**：Cloud Profile 跨用户共享（Google Play）

### 3.2 Baseline Profile 机制

**Baseline Profile** 是 Android 9+ 引入的 PGO 标准化方案：

```
┌────────────────────────────────────────────────────────────────┐
│ Baseline Profile 工作机制                                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. 开发阶段                                                    │
│     ├─ 在测试设备上跑应用                                        │
│     └─ 用 macrobenchmark / 手动测试 生成 Baseline Profile       │
│                                                                │
│  2. 上传 Play Store                                              │
│     └─ Baseline Profile 文件随 APK 上传                        │
│                                                                │
│  3. 用户安装                                                     │
│     └─ Google Play 把 Baseline Profile 下发给设备                │
│                                                                │
│  4. 首次启动                                                     │
│     └─ ART 使用 Baseline Profile 中的热点方法做 AOT 编译         │
│     └─ 跳过 Verify 阶段（Profile 中的方法已预验证）               │
│                                                                │
│  5. 运行时持续优化                                               │
│     └─ Cloud Profile 自动收集新热点                              │
│     └─ 下次更新 Profile 时下发新热点                              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 3.3 Baseline Profile 效果

**Google 官方数据（Android 14）**：

| 指标 | 无 Baseline | 有 Baseline | 提升 |
| :--- | :--- | :--- | :--- |
| **冷启动** | 1500ms | 800-1000ms | -30-50% |
| **JIT 编译次数** | 1000+ / 启动 | 100-200 / 启动 | -80% |
| **热方法首次调用** | 解释器（慢） | AOT（快） | -50-70% |

**实战数据（某头部 App）**：
- 冷启动 2000ms → 800ms（-60%）
- 首帧绘制 1500ms → 600ms（-60%）

## 4. 蹦床机制（Trampolines）

### 4.1 什么是蹦床

**蹦床（Trampoline）** 是 ART 的 Native 入口桩代码，用于在解释器模式、JIT 模式、AOT 模式之间无缝切换。

### 4.2 art_quick_* 系列

**蹦床函数族（`art/runtime/arch/`）**：

```cpp
// arm64 蹦床实现
extern "C" void art_quick_invoke_stub(...);
extern "C" void art_quick_invoke_static_stub(...);
extern "C" void art_quick_resolution_trampoline(...);
extern "C" void art_quick_imt_conflict_trampoline(...);
extern "C" void art_quick_generic_jni_trampoline(...);
```

**蹦床作用**：

```
Java 调用 → 蹦床 → 实际执行

示例：invoke-virtual
  ↓
art_quick_invoke_stub
  ↓
检查方法状态（解释器 / JIT / AOT）
  ↓
跳转到对应代码入口
```

### 4.3 蹦床分类

| 类型 | 作用 |
| :--- | :--- |
| **art_quick_invoke_stub** | invoke-virtual 入口 |
| **art_quick_invoke_static_stub** | invoke-static 入口 |
| **art_quick_resolution_trampoline** | 符号解析 |
| **art_quick_imt_conflict_trampoline** | interface 多方法冲突 |
| **art_quick_generic_jni_trampoline** | JNI 调用入口 |

## 5. OSR（On-Stack Replacement）

### 5.1 什么是 OSR

**OSR（On-Stack Replacement）** 是在方法执行过程中**热替换**为编译版本的技术。

### 5.2 为什么需要 OSR

**场景**：
- 长循环方法（如游戏渲染循环）
- 解释器执行 → 方法被调用 10,000+ 次
- 如果等待方法返回再 JIT → 用户感知卡顿（数秒延迟）
- **OSR 在循环中途替换** → 用户无感知

### 5.3 OSR 工作机制

```
方法首次调用（解释器）
  ↓
调用次数达到阈值（10,000）
  ↓
JIT 编译器在后台编译方法
  ↓
编译完成后 → OSR 在循环回边替换入口
  ↓
方法继续执行（已经是 JIT 代码）
```

**栈帧转换**：
- 解释器栈帧（ShadowFrame + Dex PC）
- JIT 栈帧（Native 栈帧 + 机器 PC）
- OSR 转换时需要拷贝 / 转换栈状态

## 6. 风险地图

| 风险类型 | 触发条件 | 现象 | 排查入口 |
| :--- | :--- | :--- | :--- |
| **JIT 卡顿** | 热方法首次 JIT 编译 | 主线程 50-100ms 卡顿 | simpleperf + JIT log |
| **dex2oat 慢** | 后台 AOT 编译 | 设备卡顿 / 续航下降 | `dex2oat` log |
| **Baseline Profile 失效** | Profile 文件未上传 / 错误 | 首次启动无优化 | Play Console |
| **OSR 失败** | 长循环方法栈帧转换错误 | 解释器继续运行（性能损失） | `simpleperf` |
| **蹦床错误** | 跨架构 trampoline 错误 | SIGSEGV in art:: | debuggerd |
| **AOT 编译 OOM** | 后台 AOT 内存不足 | 后台 kill | `dumpsys meminfo` |

## 8. 总结（架构师视角的 5 条 Takeaway）

1. **三态切换是 ART 性能的核心**——解释器 + JIT + AOT 混合模式让"启动快 + 跑起来快 + 长期最优"三者兼得。**Profile 是切换的钥匙**。
2. **dex2oat 4 产物文件**（dex / vdex / odex / oat）——理解这些文件的作用是排查 APK 体积 / 安装慢 / 启动慢问题的前提。
3. **Baseline Profile 是 PGO 的标准化**——Android 9+ 引入，让开发者主动声明热点方法。**冷启动优化必备**。
4. **蹦床机制是 ART Native 入口**——所有 Java → Native 调用都经过 `art_quick_*` 系列。**理解蹦床是理解 ART Native 调用的前提**。
5. **OSR 让长循环方法 JIT 无感知**——栈帧转换 + 循环回边替换，让用户在游戏渲染循环等场景下无感知 JIT 卡顿。

## 10. 总结（架构师视角的 7 条 Takeaway）

1. **ART 编译路径是分层的**——Interpreter（兜底）/ Mterp（解释器快路径）/ JIT（热方法）/ AOT（已编译）。**理解分层是排查编译问题的前提**。
2. **dex2oat 是 AOT 的核心**——后台运行，根据 Profile 选择 speed-profile / speed / everything。**dex2oat 失败会导致 App 启动失败**。
3. **Baseline Profile 是冷启动加速器**——把热方法预先 AOT，省掉 JIT 编译开销。**冷启动敏感 App 必启用**。
4. **AOT 占用空间 vs 性能 trade-off**——speed-profile 平衡空间和性能，speed 强调性能，everything 是极限性能但 APK 大。
5. **JIT 触发条件是方法调用计数**——默认 10,000 次。**调整阈值需要重新编译 ART**。
6. **Verify-at-runtime vs Verify-at-install**——AOSP 14 默认 verify-at-install（AOT 阶段完成）。**老设备 / 低内存设备用 verify-at-runtime**。
7. **Cloud Profile 让 AOT 跨设备迁移**——Google Play Cloud Profile 让 AOT 命中更准。**接入 Play Store 的 App 自动获益**。
