# 1.2 三色标记不变式

> **本节回答一个根本问题**：并发 GC（与业务线程并行执行）怎么保证正确性？
>
> **答案**：用 **三色标记不变式**（Tri-Color Invariant）作为数学表达 + **屏障**作为执行机制。
>
> **理解本节，就理解了"漏标"（漏活对象）的本质**——这是所有 GC Bug 的根因。

---

## 一、三色标记的定义

### 1.2.1 三种颜色

GC 标记阶段把对象分成三种颜色：

| 颜色 | 含义 | 状态机表示 | ART 中的实现 |
|:---|:---|:---|:---|
| **白色（White）** | 尚未被 GC 访问过 | 初始状态 | 对象头 Mark Word 的 Mark Bit = 0 |
| **灰色（Gray）** | 已被 GC 访问，但其引用的子对象尚未访问 | 待扫描 | Mark Bit = 1，存活 |
| **黑色（Black）** | 已被 GC 访问，且其引用的子对象都已访问 | 已扫描 | Mark Bit = 1，存活，且扫描完成 |

### 1.2.2 状态机转移

```
                 ┌─────────────┐
                 │   白色     │   初始状态
                 │ (White)    │
                 └──────┬──────┘
                        │ GC 访问到（标记）
                        ▼
                 ┌─────────────┐
        ┌────────│   灰色     │────────┐
        │        │  (Gray)    │        │
        │扫描完成 │  待扫描    │ 引用新对象
        ▼        └─────────────┘        ▼
   ┌──────────┐                  ┌──────────┐
   │  黑色    │                  │ 新白色   │
   │ (Black)  │                  │ (White)  │
   │ 已扫描   │                  │ 待标记   │
   └──────────┘                  └──────────┘
```

### 1.2.3 标记过程演示

```
初始状态：所有对象都是白色
[A=w, B=w, C=w, D=w, E=w]   (w = white)

扫描 GC Roots：A 是 Root
[A=g, B=w, C=w, D=w, E=w]   (g = gray)

扫描 A 的引用：A → B → C
[A=b, B=g, C=g, D=w, E=w]   (b = black)

扫描 B 的引用：B 已经没有引用
[A=b, B=b, C=g, D=w, E=w]

扫描 C 的引用：C → D
[A=b, B=b, C=b, D=g, E=w]

扫描 D 的引用：D → E
[A=b, B=b, C=b, D=b, E=g]

扫描 E 的引用：E 无引用
[A=b, B=b, C=b, D=b, E=b]

终止：白色对象 = []（无），全部存活
```

### 1.2.4 ART 中的三色实现

```cpp
// art/runtime/gc/collector/mark_sweep.h
class MarkBitmap {
 public:
  // 标记为灰色（实际是 Mark Bit = 1）
  bool Set(const mirror::Object* obj) {
    return Set(reinterpret_cast<uintptr_t>(obj));
  }
  
  // 测试是否标记
  bool Test(const mirror::Object* obj) {
    return Test(reinterpret_cast<uintptr_t>(obj));
  }
  
 private:
  // Mark Bitmap 是一个位图，每个对象占 1 bit
  // 1 = 标记（存活）
  // 0 = 未标记（死亡）
  std::unique_ptr<uint8_t[]> bitmap_;
};
```

**关键点**：
- ART 用 **位图（Bitmap）** 而非对象头 Mark Word 存储标记信息
- 原因：**并发标记 + 对象移动** 场景下，对象头 Mark Word 会被其他机制使用（如 hash、lock、age），不能"挪用"

## 二、强三色不变式 vs 弱三色不变式

### 1.2.5 强三色不变式（Strong Tri-Color Invariant）

**定义**：**黑色对象不允许引用白色对象**。

```
不允许的状态：
    ┌──────────┐         ┌──────────┐
    │  黑色    │────────→│  白色    │
    │ (Black)  │  field  │ (White)  │
    └──────────┘         └──────────┘
```

**这是 CMS 写屏障维护的不变式**。

### 1.2.6 弱三色不变式（Weak Tri-Color Invariant）

**定义**：**黑色对象可以引用白色对象，但白色对象必须被某个灰色对象可达**（"被保护可达"）。

```
允许的状态：
    ┌──────────┐         ┌──────────┐
    │  黑色    │────────→│  白色    │
    │ (Black)  │  field  │ (White)  │
    └──────────┘         └──────────┘
                                 ↑
                            被某个灰色引用
                            ┌──────────┐
                            │  灰色    │
                            │ (Gray)   │
                            └──────────┘
```

**这是 CC GC 读屏障维护的不变式**。

### 1.2.7 两种不变式的对比

| 维度 | 强三色不变式 | 弱三色不变式 |
|:---|:---|:---|
| 严格性 | 严格（黑色不许引用白色） | 宽松（黑色可引用白色，但白色必须被灰色保护） |
| 屏障成本 | 写屏障（每次指针赋值） | 读屏障（每次读对象引用） |
| 性能 | 写屏障轻（写少） | 读屏障重（读多） |
| STW 时间 | 长（必须 STW 处理脏对象） | 极短（< 1ms） |
| ART 中的实现 | CMS（Android 5.0-7.0） | CC / GenCC（Android 8.0+） |
| 演进方向 | 已被淘汰 | 当前主流 |

**为什么读屏障重于写屏障**：
- 写操作占比：~10-20%（业务代码中赋值相对少）
- 读操作占比：~80-90%（每行代码都可能多次读字段）

→ 强三色不变式（用写屏障）虽然"写屏障轻"，但 **STW 时间长**；弱三色不变式（用读屏障）"读屏障重"，但 **STW 时间短**。ART 选择了后者。

## 三、漏标（漏活对象）的本质

### 1.2.8 经典漏标场景

漏标 = **本应存活的对象被错误地判定为死亡**。

```
时刻 T1: GC 扫描到 A → A 染黑（已扫描）
时刻 T2: GC 扫描到 B → B 是 A 的引用 → B 染灰
时刻 T3: GC 继续扫描，标记 B 的引用 C
时刻 T4: 业务线程执行 A.field = null（断开 A → C）
时刻 T5: 业务线程执行 B.field = C（建立 B → C）  ← 此时 C 是白色
时刻 T6: GC 扫描 B，发现 B → C，C 还是白色，C 是新增引用
时刻 T7: GC 重新扫描 B 的引用？还是不重新扫描？
         - 如果不重新扫描：C 永远停留在白色 → 漏标
         - 如果重新扫描：开销巨大
```

**漏标的本质**：
- **A.field = null** + **B.field = C** 同时发生
- C 同时被 A 删除引用、被 B 添加引用
- 但 GC 已经扫过 A（A 染黑），不会重新扫描
- 如果 GC 已经扫过 B（B 染黑），也不会重新扫描
- **C 就停留在白色 → 被错误回收**

### 1.2.9 Wilson 的证明（1992）

> **理论保证**：只要维护住任一三色不变式（强或弱），并发 GC 就能正确工作。

论文：
- "Uniprocessor Garbage Collection Techniques" (1992)
- "The Tricolor Marking Problem" 后续工作

**核心结论**：漏标发生的充要条件是 **同时满足两个条件**：
1. 业务线程**插入**了一个从黑色对象到白色对象的新引用（写屏障需要拦截）
2. 业务线程**删除**了从灰色对象到这个白色对象的所有可达路径（让白色对象变成孤儿）

只要 **写屏障** 或 **读屏障** 能拦截其中任意一个条件，漏标就不会发生。

### 1.2.10 三种修法

| 修法 | 原理 | ART 中的实现 |
|:---|:---|:---|
| **Incremental Update**（增量更新） | 在写屏障里把"被删除引用"的对象重新染灰 | CMS 的 Pre-Write Barrier |
| **SATB**（Snapshot At The Beginning） | 在写屏障里把"新引用"的对象记录下来，Remark 阶段重新扫描 | G1 GC 的 Pre-Write Barrier |
| **Read Barrier**（读屏障） | 在读屏障里检查对象是否被移动，如果是则更新指针 | CC GC 的 Read Barrier |

## 四、CMS 的 Incremental Update 详解

### 1.2.11 Pre-Write Barrier 实现

```cpp
// art/runtime/gc/collector/mark_sweep.cc
class PreWriteBarrier {
 public:
  // 在 obj.offset 被赋值之前执行
  void Invoke(mirror::Object* obj, MemberOffset offset, mirror::Object* new_value) {
    // 1. 读出旧值
    mirror::Object* old_value = obj->GetFieldObject<mirror::Object>(offset);
    
    // 2. 如果旧值非 null，把它重新染灰
    if (old_value != nullptr && !IsMarked(old_value)) {
      // 经典的 Incremental Update
      MarkObject(old_value);
    }
    
    // 3. 真正的赋值（obj.offset = new_value）
  }
};
```

**为什么 Incremental Update 能防漏标**：
- 业务线程执行 `A.field = null` 时，Pre-Write Barrier 拦截
- 旧值（被删除引用的对象）被重新染灰
- 下一轮 GC 扫描会重新扫描这个旧值
- 即使这个旧值已经不被 A 引用，也会被重新检查是否被其他灰色对象引用

### 1.2.12 CMS 写屏障的局限

**问题 1：写屏障需要处理所有指针赋值**

```cpp
// 编译码（AOT/JIT）模式下，编译器要在每个指针赋值前插入 call 指令
// 例：把 obj.field = new_value 编译成：
//   call PreWriteBarrier    ; 写屏障
//   str r1, [r0, #offset]   ; 真正的赋值
```

→ **每次指针赋值都有额外开销**。

**问题 2：写屏障依然无法彻底解决 STW 时间**

CMS 的 STW 时间包括：
- Initial Mark：~5ms
- Concurrent Mark：0ms（并发）
- Remark：**~50ms**（要重新扫描被 Incremental Update 标记为脏的对象）
- Concurrent Sweep：0ms（并发）

→ Remark 阶段的 STW 是 CMS 的硬伤（用户能感知到卡顿）。

## 五、CC GC 的 Read Barrier 详解

### 1.2.13 弱三色不变式 + 读屏障的防漏标逻辑

CC GC 的策略：
- **允许黑色对象引用白色对象**
- **但读屏障保证**：业务线程每次读白色对象的引用，都会触发检查
- 如果白色对象已经被移动（说明它已经"复活"了），读屏障会更新指针

```cpp
// art/runtime/read_barrier.h
template <typename T>
inline T ReadBarrier(T* field) {
  mirror::Object* obj = *field;
  
  // 1. 检查对象是否在 from-space（已被移动）
  if (IsInFromSpace(obj)) {
    // 2. 获取 to-space 的新地址
    mirror::Object* new_obj = GetForwardingAddress(obj);
    
    // 3. 自愈（Self-Healing）：更新 field 指向新地址
    *field = new_obj;
    
    return new_obj;
  }
  
  return obj;  // 快速路径：直接返回
}
```

**为什么读屏障能防漏标**：
- 漏标的本质是"对象被错误判定为不可达"
- CC GC 移动对象时，会在 from-space 留下"forwarding address"
- 业务线程读旧地址时，读屏障检测到 from-space，自动跳转到 to-space
- **即使漏标了"对象不该被回收"，CC GC 也不会真正回收——对象已经被复制到 to-space**

→ CC GC 用 **移动对象 + 读屏障** 把"漏标"从"安全 bug"变成"性能浪费"。

### 1.2.14 CC GC 的 STW 分布

CC GC 的 STW 时间：
- **Initialize**：~2ms（栈扫描 + 引用初始化）
- **Copying**：0ms（并发）
- **Reclaim**：~1ms（清理 from-space）

→ **总 STW 时间 < 5ms**，通常 < 1ms（用户几乎无感知）。

## 六、不变式的形式化证明

### 1.2.15 强三色不变式的证明

**命题**：只要维护强三色不变式（黑色不引用白色），并发 GC 不会漏标。

**证明**（反证法）：
1. 假设漏标发生：存在白色对象 C，C 是 GC Root 可达的，但被回收
2. C 是 GC Root 可达 → C 的可达路径上至少有一个对象 G（离 C 最近的灰色对象）
3. C 被回收 → C 在 GC 结束时仍是白色 → C 从未被任何灰色对象染灰
4. 矛盾：G 是离 C 最近的灰色对象 → G 引用 C → G 扫描时会染灰 C
5. 除非 G 扫描 C 之前被染黑（违反不变式"灰色 → 黑色"前必须扫描完引用）
6. 但强三色不变式允许"黑色不引用白色" → G 染黑时已扫完 C → C 应被染灰
7. 矛盾 → 漏标不可能

**实际意义**：只要写屏障能保证"黑色不引用白色"（每次黑色要引用白色时，写屏障把白色染灰），漏标不会发生。

### 1.2.16 弱三色不变式的证明

**命题**：只要维护弱三色不变式（白色被某个灰色可达），并发 GC 不会漏标。

**证明**（反证法）：
1. 假设漏标发生：白色对象 C 被回收
2. 弱三色不变式要求 C 被某个灰色对象可达 → 存在灰色 G，G 引用 C
3. C 在 GC 结束时仍是白色 → G 染黑时未扫描 C 的引用
4. 矛盾：弱三色不变式要求 G 在染黑前必须经过 C 的可达路径
5. 除非业务线程在 G 染黑前执行 `G.field = null`（删除引用）
6. 但读屏障保证：业务线程读 C 时，C 已被移动到 to-space → C 不再是白色
7. 矛盾 → 漏标不可能

**实际意义**：弱三色不变式允许"黑色引用白色"（性能更好），但通过"移动对象 + 读屏障"保证"白色最终会被处理"。

## 七、稳定性关联：漏标的真实案例

### 1.2.17 案例 1：CMS 时代的"幽灵对象"复活 Bug

**场景**（Android 5.0-6.0 真实案例）：
```java
// 业务代码
public class Container {
    private List<Object> list = new ArrayList<>();
    
    public void removeFirst() {
        // 业务线程
        if (!list.isEmpty()) {
            list.remove(0);  // 断开 list[0] 的引用
        }
    }
}

// GC 线程（并发）
// 正在标记 list[0]
// 业务线程 removeFirst → list[0] 引用断开 → 但 GC 已经扫过 list（黑色）
// list[0] 是孤立对象，被回收
// 但 list[0] 还在使用中 → 业务崩溃
```

**根因**：CMS 写屏障在某些场景下漏拦截（极端并发场景）。

**修复**：升级到 Android 8.0+ CC GC。

### 1.2.18 案例 2：CC GC 下 Hook 框架的对象复活 Bug

**场景**（ART 8.0-9.0 真实案例）：
```cpp
// Hook 框架代码（绕过读屏障）
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    // 直接修改 ArtMethod.entrypoint，绕过读屏障
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}

// CC GC 移动 ArtMethod 到新地址
// 业务线程调用 method.entrypoint → 触发读屏障 → 但 Hook 框架没经过读屏障
// → 调用旧地址 → 崩溃
```

**根因**：Hook 框架绕过了 CC GC 的读屏障。

**修复**：Hook 框架必须用 `ReadBarrier::Barrier` 包裹所有 ArtMethod 访问（08 篇详细讲）。

## 八、ART 中的不变式实现源码

### 1.2.19 Mark Bitmap 的实现

```cpp
// art/runtime/gc/collector/mark_sweep.h
class MarkBitmap {
 public:
  // 每个对象对应 1 bit
  // 1 = 标记（存活）
  // 0 = 未标记（死亡）
  
  bool Set(const mirror::Object* obj) {
    uintptr_t addr = reinterpret_cast<uintptr_t>(obj);
    size_t index = (addr - base_addr_) / kAlignment;  // 每个对象 8 字节对齐
    size_t byte_offset = index / 8;
    uint8_t mask = 1u << (index % 8);
    
    uint8_t old = bitmap_[byte_offset];
    bitmap_[byte_offset] = old | mask;
    return (old & mask) == 0;  // 返回 true 表示第一次标记
  }
  
  bool Test(const mirror::Object* obj) {
    uintptr_t addr = reinterpret_cast<uintptr_t>(obj);
    size_t index = (addr - base_addr_) / kAlignment;
    size_t byte_offset = index / 8;
    uint8_t mask = 1u << (index % 8);
    
    return (bitmap_[byte_offset] & mask) != 0;
  }
  
 private:
  uintptr_t base_addr_;
  std::unique_ptr<uint8_t[]> bitmap_;
};
```

### 1.2.20 Mark Stack 的实现

```cpp
// art/runtime/gc/collector/mark_sweep.cc
class MarkStack {
 public:
  void Push(mirror::Object* obj) {
    if (top_ >= capacity_) {
      Resize();
    }
    stack_[top_++] = obj;
  }
  
  mirror::Object* Pop() {
    return stack_[--top_];
  }
  
 private:
  std::vector<mirror::Object*> stack_;
  size_t top_ = 0;
  size_t capacity_;
};
```

**灰色对象 = Mark Stack 里的对象**。

## 九、本节小结

1. **三色标记**：白色（未访问）/ 灰色（已访问待扫描）/ 黑色（已扫描）
2. **强三色不变式**：黑色不许引用白色 → 用写屏障维护（CMS）
3. **弱三色不变式**：黑色可引用白色，但白色必须被灰色保护 → 用读屏障维护（CC / GenCC）
4. **漏标的本质**：删除旧引用 + 添加新引用同时发生，绕过 GC 的扫描
5. **ART 的演进**：CMS（写屏障）→ CC（读屏障）→ GenCC（读屏障 + 分代）

→ **不理解三色不变式，就看不懂 CMS / CC / GenCC 的差异**。
