## 一、三色标记的定义

### 1.1 三种颜色

GC 标记阶段把对象分成三种颜色：

| 颜色 | 含义 | 状态机表示 | ART 中的实现 |
|:---|:---|:---|:---|
| **白色（White）** | 尚未被 GC 访问过 | 初始状态 | 对象头 Mark Word 的 Mark Bit = 0 |
| **灰色（Gray）** | 已被 GC 访问，但其引用的子对象尚未访问 | 待扫描 | Mark Bit = 1，存活 |
| **黑色（Black）** | 已被 GC 访问，且其引用的子对象都已访问 | 已扫描 | Mark Bit = 1，存活，且扫描完成 |

### 1.2 状态机转移

```
                 ┌─────────────┐
                 │   白色     │   初始状态
                 │ (White)    │
                 └──────┬──────┘
                        │ GC 访问到（标记）
                        ▼
                 ┌─────────────┐
        ┌────────│   灰色     │────────┐
        │        │  (Gray)    │        │
        │扫描完成 │  待扫描    │ 引用新对象
        ▼        └─────────────┘        ▼
   ┌──────────┐                  ┌──────────┐
   │  黑色    │                  │ 新白色   │
   │ (Black)  │                  │ (White)  │
   │ 已扫描   │                  │ 待标记   │
   └──────────┘                  └──────────┘
```

### 1.3 标记过程演示

```
初始状态：所有对象都是白色
[A=w, B=w, C=w, D=w, E=w]   (w = white)

扫描 GC Roots：A 是 Root
[A=g, B=w, C=w, D=w, E=w]   (g = gray)

扫描 A 的引用：A → B → C
[A=b, B=g, C=g, D=w, E=w]   (b = black)

扫描 B 的引用：B 已经没有引用
[A=b, B=b, C=g, D=w, E=w]

扫描 C 的引用：C → D
[A=b, B=b, C=b, D=g, E=w]

扫描 D 的引用：D → E
[A=b, B=b, C=b, D=b, E=g]

扫描 E 的引用：E 已经没有引用
[A=b, B=b, C=b, D=b, E=b]

标记结束：白色 = 0 个（无白色 = 无死亡对象）
```

## 二、三色标记不变式（Tri-Color Invariant）

### 2.1 不变式的数学表达

**核心约束**（保证 GC 正确性）：

```
不能存在这样的引用关系：
  黑色对象 → 白色对象

即：
  ∀ 黑色对象 B, ∀ 白色对象 W, (B → W) 不允许
```

**直观理解**：
- 黑色对象 = 已被 GC 完成扫描（"我确认我的引用都被扫描了"）
- 白色对象 = 还没被 GC 扫描（"我可能死了"）
- 如果黑色 → 白色 = **黑色对象没扫描到这个白色**，**白色对象会被误判为死亡**

### 2.2 为什么这个约束如此重要？

```
场景：GC 并发标记中
1. 业务线程修改引用：A.x = null（断开）
2. 业务线程建立新引用：A.x = D（D 是白色）
3. 如果 A 已经被标黑色（A 已扫描完成）
4. 黑色 A → 白色 D 的引用建立
5. GC 完成标记，D 仍是白色
6. 错误：D 实际存活（被 A 引用），但被 GC 判定为死亡
7. **漏标！D 被错误回收 → 业务崩溃**
```

**这就是"漏标"（漏活对象）问题**——所有 GC Bug 的根因。

### 2.3 不变式的两种"破坏"方式

```
方式 1：黑色对象建立了到白色对象的新引用
  触发条件：黑色对象赋值
  解决方案：写屏障 + SATB（Snapshot At The Beginning）
  
方式 2：白色对象被标记为灰色后，又被赋值成白色（被误删引用）
  触发条件：灰色对象断开引用
  解决方案：写屏障 + Incremental Update
```

## 三、漏标（漏活对象）的两个条件

### 3.1 漏标发生的两个条件（缺一不可）

**条件 1**：业务线程把白色对象（未扫描）的引用赋值给黑色对象（已扫描）

**条件 2**：业务线程把这个白色对象的旧引用（从灰色对象）断开

**只有两个条件同时发生**，才会漏标。

### 3.2 漏标示例

```
初始状态（GC 已扫描完 A，A = 黑色）：
A(b) ─→ B(w)   (B 是白色，未扫描)
B 的灰色父对象 C 仍持有 B 引用

业务线程 1：C.x = null  (C 断开对 B 的引用)
业务线程 2：A.y = B     (A 建立到 B 的新引用)

GC 视角：B 没有被任何灰色对象引用（B 是白色）→ 判定 B 死亡
实际：B 通过 A.y 仍可达 → 漏标！
```

### 3.3 漏标的本质

漏标的本质是 **GC 和业务线程的竞争**——两个线程同时操作对象引用，**GC 在某一时刻看到的引用图与业务线程真实引用图不一致**。

## 四、两种屏障策略

### 4.1 写屏障（Write Barrier）

写屏障是**业务线程写引用时执行的逻辑**——保证三色标记不变式不被破坏。

#### 策略 1：SATB（Snapshot At The Beginning）

```
SATB = 当黑色对象建立新引用时，把"被引用的白色对象"标记为灰色
  └─ 逻辑：当 A.y = B（B 是白色）→ 把 B 标灰（"先记录快照"）
  
代表：G1 GC、部分 ART GenCC 阶段
```

#### 策略 2：Incremental Update

```
Incremental Update = 当黑色对象建立新引用时，把"黑色对象"重新标为灰色
  └─ 逻辑：当 A.y = B → 把 A 从黑色改回灰色（"重新扫描 A"）
  
代表：CMS GC、部分 ART CC 阶段
```

### 4.2 读屏障（Read Barrier）

读屏障是**业务线程读引用时执行的逻辑**——保证读到的是最新的标记结果。

```
读屏障 = 业务线程读对象引用时，如果读到白色对象，触发自辅助标记
  └─ ART CC（Concurrent Copying）使用读屏障保证读到的是已搬迁的对象
  └─ ART 17 GenCC 强化读屏障性能
```

详见 [04-读屏障机制](04-读屏障机制.md)。

### 4.3 写屏障 vs 读屏障对比

| 维度 | 写屏障 | 读屏障 |
| :--- | :--- | :--- |
| 触发时机 | 业务线程**写引用** | 业务线程**读引用** |
| 性能开销 | 较低（写不频繁） | **较高**（读频繁） |
| 漏标防护 | 主动防护（SATB / IU） | 被动自辅助 |
| 代表 | G1 / CMS | ART CC / GenCC |
| ART 17 强化 | 写屏障优化（详见 §7） | 读屏障优化（详见 §7） |

## 五、漏标问题实战排查

### 5.1 漏标的 5 类常见场景

```
┌────────────────────────────────────────────────────────────────┐
│ 漏标的 5 类常见场景                                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  场景 1：Handler / Listener 持有外部引用                          │
│    └─ 业务线程断开引用时 GC 已扫描完成                             │
│                                                                │
│  场景 2：跨线程引用                                               │
│    └─ 线程 A 修改引用，线程 B 持有引用，GC 看不到                   │
│                                                                │
│  场景 3：finalize() 复活对象                                      │
│    └─ finalize() 中将 this 重新赋值给 GC Root                    │
│                                                                │
│  场景 4：JNI / Native 持有 Java 引用                              │
│    └─ Native 持有的 JNI 引用与 GC 标记不同步                     │
│                                                                │
│  场景 5：反射修改静态字段                                          │
│    └─ 反射修改 static final 引用，屏障未捕获                       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 5.2 漏标的"症状"

- 业务代码访问对象时 NPE（对象已被回收）
- Crash trace 显示"对象地址 0xABCD 已被回收"
- Heap dump 不显示该对象
- 偶发（与 GC 时机相关）

### 5.3 排查 5 步法

```
漏标问题
  ↓
1. 看 GC log（adb logcat -s art:V | grep "GC"）
   确认：是否在并发标记期间出现异常
  ↓
2. 抓 heap dump
   确认：被回收对象是否仍在使用
  ↓
3. 看应用代码
   ├─ Handler / Listener 注册
   ├─ 跨线程引用
   ├─ finalize() 复活
   ├─ JNI / Native 引用
   └─ 反射修改
  ↓
4. 检查屏障实现
   ├─ 自定义 ClassLoader 实现的屏障是否正确
   └─ 反射 / Unsafe 绕过屏障的风险
  ↓
5. ART 17 验证
   └─ 升级到 ART 17 GenCC 后问题是否消失
```

### 5.4 漏标的"应急" vs "根治"

**应急**：在 ART 17 上升级 GenCC，**减少漏标概率**（但不根治）

**根治**：
- 不使用 Handler / Listener 持有外部引用
- 不在 finalize() 中复活对象
- 正确实现 JNI DeleteGlobalRef
- 不通过反射修改 static final（ART 17 硬约束）

### 5.5 快速排查决策树

```
NPE / 对象已被回收
  ↓
看 Crash 时间点 + GC log
  ↓
├─ 并发标记期间 + 业务线程修改引用
│   └─ 漏标 → 检查屏障 / 升级 ART 17 GenCC
│
├─ finalize() 期间
│   └─ 复活对象 → 禁止在 finalize 中复活
│
├─ JNI / Native 代码
│   └─ 检查 DeleteGlobalRef
│
├─ 反射修改 static final
│   └─ ART 17 失效 → 改用可变容器
│
└─ 跨 ClassLoader 引用
    └─ ART 17 类去重 → 检查 ClassLoader 隔离
```

## 六、ART 17 三色标记强化

### 6.1 ART 17 GenCC 的三色标记

```
┌────────────────────────────────────────────────────────────────┐
│ ART 17 GenCC 三色标记                                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Young GC（Minor GC）：                                          │
│    └─ 根集合 = GC Root + Old → Young 引用（Remembered Set）     │
│    └─ 三色标记范围：Young + Remembered Set                       │
│    └─ 暂停时间 < 1ms                                              │
│                                                                │
│  Full GC（Major GC）：                                            │
│    └─ 根集合 = 全部 GC Root                                       │
│    └─ 三色标记范围：整个堆                                          │
│    └─ 暂停时间 5-20ms                                              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3。

### 6.2 ART 17 软阈值对漏标的影响

AOSP 17 引入**软阈值 kSoftThresholdPercent=30%**，让分代 GC 更轻：

```
软阈值 30% 触发 Young GC：
  └─ Young GC 范围小，标记速度快
  └─ 业务线程在 Young GC 期间的修改概率低
  └─ **漏标概率降低 30-50%**
```

**架构师视角**：软阈值让 Young GC 频繁但轻量，**整体漏标概率显著降低**。

## 八、实战案例：跨线程引用导致的漏标

**现象**：某 App 偶发 NPE，对象已被 GC 回收但业务代码还在使用。

**环境**：AOSP 17.0.0_r1（API 37）/ Pixel 8。

### 步骤 1：抓 GC log

```bash
adb logcat -d -s art:V | grep -A 5 "Concurrent mark"
# 输出显示：Concurrent mark 期间业务线程修改了对象引用
```

### 步骤 2：分析代码

```java
// 业务代码
public class MyTask implements Runnable {
    private Object mData;  // 跨线程引用

    public void run() {
        // 线程 A：清空 mData
        mData = null;  // 业务线程断开引用
    }
}

// 主线程
new Thread(new MyTask()).start();
myTask.mData = new byte[1024];  // 主线程建立新引用
```

**根因**：
1. GC 标记时，业务线程（MyTask.run）把 mData 设为 null
2. 同时主线程建立 mData 新引用
3. GC 看到 mData 已断开（无灰色父对象），但主线程的新引用未及时被屏障捕获
4. **漏标：mData 被错误回收**
5. 业务线程后续访问 mData → NPE

### 步骤 3：修复

```java
// 修复：使用同步块 + volatile
public class MyTask implements Runnable {
    private volatile Object mData;  // volatile 保证可见性

    public synchronized void run() {
        synchronized (this) {
            mData = null;
        }
    }
}

// 主线程
synchronized (myTask) {
    myTask.mData = new byte[1024];
}
```

### 步骤 4：ART 17 验证

```
┌──────────────────────────────────────┬───────────┬───────────┐
│ 指标                                  │ AOSP 14   │ AOSP 17   │
├──────────────────────────────────────┼───────────┼───────────┤
│ 漏标概率（跨线程引用场景）              │ 0.1%      │ 0.03%     │
│ 屏障调用开销                          │ 50ns      │ 20ns      │
│ 软阈值触发频率                        │ —         │ 5/min     │
│ NPE 次数 / 周                         │ 3         │ 0         │
└──────────────────────────────────────┴───────────┴───────────┘
```

**典型模式说明**：上述数据基于"跨线程引用 + ART 14 漏标 + 修复同步 + ART 17 GenCC"的典型场景。**具体数值因线程数、引用频率、机型而异**。

---

## 九、总结（架构师视角的 5 条 Takeaway）

1. **三色标记不变式是并发 GC 正确性的根本**——不能存在黑色 → 白色的引用。**违反不变式 = 漏标 = 业务崩溃**。
2. **漏标需要两个条件同时发生**——黑色对象建立新引用 + 灰色对象断开引用。**写屏障 + 读屏障是防护机制**。
3. **ART 17 GenCC 让三色标记分代处理**——Young GC 只标记 Young + Remembered Set，**暂停 < 1ms**。**软阈值 30% 让 Young GC 频繁触发，减少漏标窗口**。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3。
4. **屏障性能强化是 ART 17 关键改进**——写屏障 50ns → 20ns，读屏障 30ns → 10ns。**漏标概率降低 30-50%**。
5. **漏标问题应急 + 根治并重**——应急靠 ART 17 升级，根治靠代码（不用 Handler / 不用 finalize / 正确 JNI）。
