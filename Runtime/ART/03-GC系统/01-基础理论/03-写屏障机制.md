# 1.3 写屏障机制

> **本节回答一个根本问题**：CMS（标记-清除）和 GenCC（分代复制）怎么在并发 GC 中维护三色不变式？
>
> **答案**：用 **写屏障**（Write Barrier）—— 在指针赋值时插入钩子代码。
>
> **理解本节，就理解了"为什么 ART 9+ 还需要写屏障"**——虽然 CC GC 主要靠读屏障，但 GenCC 的 Card Table 维护依然需要写屏障。

---

## 一、写屏障的定义

### 1.3.1 写屏障的本质

**写屏障（Write Barrier）** 是在 **指针赋值指令** 前后插入的钩子代码：

```cpp
// 业务代码
obj.field = new_value;

// 实际执行（写屏障插入后）
PreWriteBarrier(obj, field_offset, new_value);   // 写屏障（赋值前）
obj.field = new_value;                            // 真正的赋值
PostWriteBarrier(obj, field_offset, new_value);   // 写屏障（赋值后）
```

### 1.3.2 Pre-Write Barrier vs Post-Write Barrier

| 类型 | 触发时机 | 用途 | ART 中的实现 |
|:---|:---|:---|:---|
| **Pre-Write Barrier** | 在指针赋值**之前** | 记录旧值 / 维护增量更新 | CMS |
| **Post-Write Barrier** | 在指针赋值**之后** | 记录新值 / 维护卡表 | GenCC |

### 1.3.3 写屏障 vs 读屏障

| 维度 | 写屏障 | 读屏障 |
|:---|:---|:---|
| 触发时机 | 指针赋值时 | 读对象引用时 |
| 触发频率 | 写少（~10-20%） | 读多（~80-90%） |
| 开销 | 较低 | 较高 |
| 维护的不变式 | 强三色不变式 | 弱三色不变式 |
| ART 中的代表 | CMS | CC / GenCC |

## 二、CMS 的 Pre-Write Barrier（Incremental Update）

### 1.3.4 CMS 写屏障的实现

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的伪代码
void PreWriteBarrier(mirror::Object* obj, MemberOffset offset, 
                     mirror::Object* new_value) {
  // 1. 读出旧值
  mirror::Object* old_value = obj->GetFieldObject<mirror::Object>(offset);
  
  // 2. Incremental Update：把旧值重新染灰
  if (old_value != nullptr) {
    // 把 old_value 加入 mark stack，等下一轮重新扫描
    MarkObject(old_value);
  }
  
  // 3. 真正的赋值由调用方完成
}
```

### 1.3.5 Incremental Update 防漏标原理

回顾经典漏标场景：
```
时刻 T1: GC 扫描 A（A 已染黑）
时刻 T2: 业务线程执行 A.field = null（断开 A → C）
时刻 T3: GC 扫描 B，B 不再引用 C
时刻 T4: GC 结束，C 仍是白色，被错误回收
```

Pre-Write Barrier 介入：
```
时刻 T1: GC 扫描 A（A 已染黑）
时刻 T2: 业务线程执行 A.field = null
         → Pre-Write Barrier 拦截
         → 读出旧值 C
         → 把 C 重新染灰（Incremental Update）
         → C 加入 mark stack
时刻 T3: GC 发现 mark stack 不为空，继续扫描
         → 扫描 C → C 的引用都被检查 → 不会漏标
时刻 T4: GC 结束，C 已正确标记
```

### 1.3.6 CMS 写屏障的局限

**问题 1：每次指针赋值都有写屏障开销**

```cpp
// AOT 编译码的伪代码（AArch64）
// 原始代码
str x1, [x0, #offset]    ; obj.field = new_value

// 编译码插入写屏障后
str x1, [x0, #offset]    ; obj.field = new_value（暂存）
ldr x2, [x0, #offset]    ; 读出旧值
cbz x2, .Lskip            ; 如果旧值为 null，跳过
bl artWriteBarrier         ; 调用写屏障
.Lskip:
str x1, [x0, #offset]    ; 真正的赋值
```

→ 每次指针赋值多 4 条指令。

**问题 2：写屏障只能"挡"业务线程，不能完全消除 STW**

- Initial Mark（STW）：~5ms
- Concurrent Mark（并发）：0ms
- **Remark（STW）：~50ms**（要重新扫描被写屏障标记的脏对象）
- Concurrent Sweep（并发）：0ms

→ **Remark 是 CMS 的硬伤**——用户能感知卡顿。

### 1.3.7 CMS 写屏障的源码路径

```cpp
// art/runtime/gc/collector/mark_sweep.h
class MarkSweep : public GarbageCollector {
 public:
  // 写屏障入口
  void WriteBarrier(mirror::Object* obj, MemberOffset offset, 
                    mirror::Object* new_value) {
    // 调用 PreWriteBarrier
  }
};

// art/runtime/gc/collector/mark_sweep.cc
void MarkSweep::MarkObject(mirror::Object* obj) {
  if (!mark_bitmap_->Test(obj)) {
    mark_bitmap_->Set(obj);
    mark_stack_->Push(obj);  // 加入 mark stack，等下一轮扫描
  }
}
```

## 三、GenCC 的 Post-Write Barrier（Card Table 维护）

### 1.3.8 GenCC 为什么还需要写屏障

GenCC（Android 10.0+ 默认 GC）主要靠 **读屏障** 维护弱三色不变式，但分代 GC 还需要：
- **Card Table**：记录跨代引用（Old → Young）
- **写屏障维护 Card Table**：业务线程每次指针赋值都要更新 Card Table

→ **GenCC 同时有读屏障和写屏障**，但写屏障的职责不是"维护不变式"，而是"维护 Card Table"。

### 1.3.9 GenCC Post-Write Barrier 的实现

```cpp
// art/runtime/gc/collector/concurrent_copying.h 的 PostWriteBarrier 简化版
void PostWriteBarrier(void* field_addr, mirror::Object* new_value) {
  // 1. 计算 field_addr 所在的 card
  uint8_t* card_addr = CardTable::AddressToCard(field_addr);
  
  // 2. 检查是否跨代引用
  SpaceType src_space = RegionSpace::GetSpaceTypeOf(field_addr);
  SpaceType dst_space = RegionSpace::GetSpaceTypeOf(new_value);
  
  // 3. 跨代时把 card 标记为 dirty
  if (src_space != dst_space && 
      (src_space == kSpaceTypeYoung || dst_space == kSpaceTypeYoung)) {
    *card_addr = CardTable::kCardDirty;
  }
}
```

### 1.3.10 Card Table 的内存布局

```
┌────────────────────────────────────────────────────────┐
│                       Old Gen                          │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐  │
│  │  Card 0 │  Card 1 │  Card 2 │  Card 3 │  ...    │  │
│  │ 512 B   │ 512 B   │ 512 B   │ 512 B   │         │  │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘  │
│       ↑                                              │
│       └── 跨代引用时标记为 dirty                        │
├────────────────────────────────────────────────────────┤
│                      Young Gen                         │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐  │
│  │  Card 0 │  Card 1 │  Card 2 │  Card 3 │  ...    │  │
│  │ 512 B   │ 512 B   │ 512 B   │ 512 B   │         │  │
│  └─────────┴─────────┴─────────┴─────────┴─────────┘  │
```

### 1.3.11 ART 的 Card Table 实现源码

```cpp
// art/runtime/gc/space/region_space.h 的 CardTable 简化版
class CardTable {
 public:
  static constexpr size_t kCardSize = 512;  // 1 byte / 512 byte
  
  enum CardValue : uint8_t {
    kCardClean = 0,    // 干净（无跨代引用）
    kCardDirty = 0x70, // 脏（有跨代引用，需要扫描）
  };
  
  void MarkCard(const void* addr) {
    uint8_t* card = AddressToCard(addr);
    *card = kCardDirty;
  }
  
  bool IsDirty(const void* addr) {
    return *AddressToCard(addr) == kCardDirty;
  }
  
  // addr → card 的映射
  uint8_t* AddressToCard(const void* addr) {
    uintptr_t offset = reinterpret_cast<uintptr_t>(addr) - base_addr_;
    return &card_table_[offset / kCardSize];
  }
  
 private:
  std::unique_ptr<uint8_t[]> card_table_;
  uintptr_t base_addr_;
};
```

## 四、写屏障在编译码（AOT/JIT）中的实现

### 1.3.5 AArch64 上的写屏障机器码

```asm
; AArch64 写屏障的实现（art/runtime/arch/arm64/quick_entrypoints_arm64.S）

; 调用 PreWriteBarrier 的代码路径
; 入口：x0 = obj, x1 = offset, x2 = new_value
pre_write_barrier_entry:
    stp x29, x30, [sp, #-16]!    ; 保存帧指针 + 返回地址
    mov x29, sp
    
    ; 读出旧值
    add x3, x0, x1                ; x3 = obj + offset
    ldr x4, [x3]                  ; x4 = *obj.offset（old_value）
    
    ; 如果旧值为 null，跳过写屏障
    cbz x4, .Lskip
    
    ; 调用写屏障逻辑
    bl artWriteBarrier
    
.Lskip:
    ; 恢复栈帧
    ldp x29, x30, [sp], #16
    ret
```

### 1.3.6 JIT 模式下写屏障的优化

ART 的 JIT 编译器会做以下优化：

1. **内联（Inlining）**：简单的写屏障直接内联到调用点
2. **Card Table 预计算**：card_addr 在循环外计算，避免循环内重复计算
3. **批量写屏障**：连续的指针赋值合并写屏障调用

```cpp
// 优化前
for (int i = 0; i < N; i++) {
    WriteBarrier(&array[i]);       // 每次循环都计算 card_addr
    array[i] = new_value;
}

// 优化后
uint8_t* card = CardTable::AddressToCard(&array[0]);  // 循环外计算
for (int i = 0; i < N; i++) {
    *card = kCardDirty;            // 直接更新 card
    array[i] = new_value;
}
```

### 1.3.7 解释器模式下的写屏障

解释器模式下写屏障是普通的 C++ 函数调用：

```cpp
// art/runtime/interpreter/interpreter.cc
void Interpreter::DoFieldPut(Thread* self, ...) {
  // ...
  
  // 写屏障（解释器模式下相对低开销）
  if (kUseReadBarrier && !obj->IsReadBarrierProtected()) {
    ReadBarrier::Barrier(...);
  }
  WriteBarrier::WriteField(obj, offset, new_value);
  
  // ...
}
```

## 五、Post-Write Barrier 与 Card Table 的关系

### 1.3.8 Minor GC 如何使用 Card Table

```cpp
// art/runtime/gc/collector/concurrent_copying.cc 的 Minor GC 简化版
void MinorGC() {
  // 1. 暂停所有 mutator 线程（STW）
  SuspendAllThreads();
  
  // 2. 扫描 Young Gen 的所有 Root
  ScanYoungGenRoots();
  
  // 3. 遍历 Card Table，找所有 dirty card
  for (uint8_t* card = card_table_begin_; card < card_table_end_; card++) {
    if (*card == CardTable::kCardDirty) {
      // 4. 扫描这张 card 对应的内存区域
      ScanCard(card);
      
      // 5. 标记为 clean
      *card = CardTable::kCardClean;
    }
  }
  
  // 6. 恢复 mutator 线程
  ResumeAllThreads();
}
```

### 1.3.9 Card Table 的扫描代价

每张 dirty card 对应 512 字节的扫描范围：

```
┌────────────────────────────────────────────────────────┐
│   Dirty Card                                          │
│   ┌──────────┬──────────┬──────────┬──────────┐        │
│   │ 128 B    │ 128 B    │ 128 B    │ 128 B    │        │
│   │ Card数据 │ Card数据 │ Card数据 │ Card数据 │        │
│   └──────────┴──────────┴──────────┴──────────┘        │
│   ↑                                              ↑    │
│   跨代引用在这里                                  扫描终止
└────────────────────────────────────────────────────────┘
   扫描 512 字节，找到所有跨代引用
```

**精度问题**：
- 1 byte / 512 byte 的粒度 → 每次脏卡扫描的"无辜开销"是 512 字节
- 大对象（> 512 字节）可能浪费大量扫描
- ART 14+ 优化：**细粒度卡表**（256 字节 / 128 字节）

## 六、写屏障的性能影响

### 1.3.10 写屏障的开销数据

**AOSP 8.0 实测数据**（公开 benchmark）：

| 场景 | 无写屏障 | CMS 写屏障 | GenCC 写屏障 |
|:---|:---|:---|:---|
| 简单赋值 `obj.field = val` | 1.0x | 1.05x | 1.08x |
| 循环赋值 `arr[i] = val` (N=100) | 1.0x | 1.4x | 1.5x |
| 跨 Region 赋值 | 1.0x | 1.1x | 1.2x |

**关键发现**：
- 写屏障在编译码模式下开销相对较低（CMS 1.05x，GenCC 1.08x）
- 循环内多次赋值开销显著增加（1.4-1.5x）
- 但配合 JIT 优化（批量写屏障），实际开销可降到 1.1-1.2x

### 1.3.11 写屏障的工程坑点

**坑点 1：Hook 框架绕过写屏障**

```cpp
// Hook 框架代码
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    // 直接修改 ArtMethod.entrypoint，绕过写屏障
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}

// CC GC 移动 ArtMethod 到新地址
// 业务线程调用 method.entrypoint → 触发读屏障
// 但 Hook 框架修改的字段没经过写屏障 → Card Table 没更新
// → 跨代引用记录缺失 → Minor GC 漏标
```

**修复**：
```cpp
// Hook 框架必须用 Barrier 包裹
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    // 1. 先调用写屏障（更新 Card Table）
    WriteBarrier::WriteField(method, 
        ArtMethod::EntryPointOffset(), 
        reinterpret_cast<mirror::Object*>(new_entrypoint));
    
    // 2. 再修改 entrypoint
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}
```

**坑点 2：JNI 代码绕过写屏障**

```cpp
// JNI 直接修改 Java 对象的字段
void JNI修改字段(JNIEnv* env, jobject obj, jstring new_value) {
    // 绕过 Java 层，直接写内存
    jclass cls = env->GetObjectClass(obj);
    jfieldID fid = env->GetFieldID(cls, "field", "Ljava/lang/String;");
    env->SetObjectField(obj, fid, new_value);
    // SetObjectField 内部会调用写屏障，OK
    // 但如果用 Unsafe.putObject() 直接写 → 绕过写屏障
}

// 错误的做法
void JNI错误做法(JNIEnv* env, jobject obj, jstring new_value) {
    jclass cls = env->GetObjectClass(obj);
    jfieldID fid = env->GetFieldID(cls, "field", "Ljava/lang/String;");
    jlong offset = env->GetFieldID(...) // 错了
    // 或者更糟：直接拿字段 offset 用 Unsafe 操作
}
```

**坑点 3：反射绕过写屏障**

```java
// Field.set() 内部会调用写屏障，OK
field.set(obj, new_value);

// Unsafe.putObject() 不调用写屏障
unsafe.putObject(obj, offset, new_value);
// ❌ 风险：CC GC 移动对象后，offset 失效；Card Table 不更新
```

### 1.3.12 ART 14+ 的写屏障优化

| 优化 | 描述 | 性能提升 |
|:---|:---|:---|
| **rbcc（Read Barrier Copy Collector）** | 把读屏障 + 写屏障结合，减少双重屏障开销 | ~15% |
| **细粒度 Card Table** | 1 byte / 128 byte，减少扫描开销 | ~20% |
| **写屏障批处理** | 连续多个写屏障合并为一次调用 | ~30% |
| **Region TLAB 优化** | TLAB 内分配避免触发跨 Region 写屏障 | ~25% |

## 七、实战案例

### 1.3.13 案例 1：CMS 写屏障失效导致 OOM

**场景**（Android 6.0 真实案例）：
```java
// 业务代码：高并发下的列表操作
public class HighConcurrencyList {
    private List<Object> list = new ArrayList<>();
    
    // 多线程并发执行
    public void addAndRemove(Object obj) {
        synchronized (list) {
            list.add(obj);
            if (list.size() > 1000) {
                list.remove(0);
            }
        }
    }
}

// CMS GC 在并发标记时遇到这种场景：
// 1. GC 扫描 list（list 染黑）
// 2. GC 扫描 list[0]（list[0] 染黑）
// 3. 业务线程 remove(0) → Pre-Write Barrier 把 list[0] 重新染灰
// 4. GC 继续扫描，发现 list[0] 在 mark stack 里，重新扫描
// 5. 但 list[0] 已经被回收（被新对象覆盖了）
// 6. 业务线程访问 list[0] → 野指针 → 崩溃或内存错误
```

**根因**：CMS 写屏障在某些极端场景下漏拦截。

**修复**：
- 升级到 Android 8.0+ CC GC
- 或在业务代码用 ConcurrentLinkedQueue 替代 ArrayList

### 1.3.14 案例 2：Hook 框架绕过 GenCC 写屏障导致崩溃

**场景**（Android 10.0 真实案例）：
```cpp
// Xposed 旧版本 Hook 代码
void* HookMethod(void* method, void* new_entrypoint) {
    ArtMethod* art_method = reinterpret_cast<ArtMethod*>(method);
    
    // 直接修改 entrypoint，绕过写屏障
    void* old_entry = art_method->entry_point_from_quick_compiled_code_;
    art_method->entry_point_from_quick_compiled_code_ = new_entrypoint;
    
    return old_entry;
}

// CC GC 移动 ArtMethod 到新地址
// 业务线程调用 method.entrypoint → 读屏障检查
// 但 Hook 框架缓存的 method 指针是旧地址
// → 调用旧地址 → 崩溃
```

**修复**：
```cpp
// 新版本 Hook 框架
void* HookMethod(void* method, void* new_entrypoint) {
    ArtMethod* art_method = reinterpret_cast<ArtMethod*>(method);
    
    // 1. 先调用写屏障（更新 Card Table + 维护引用一致性）
    WriteBarrier::Barrier(art_method);
    
    // 2. 修改 entrypoint
    void* old_entry = art_method->entry_point_from_quick_compiled_code_;
    art_method->entry_point_from_quick_compiled_code_ = new_entrypoint;
    
    return old_entry;
}
```

### 1.3.15 案例 3：JNI Unsafe 操作绕过写屏障导致 Card Table 不一致

**场景**：
```cpp
// Native 代码优化：用 Unsafe 直接写字段
void FastSetField(JNIEnv* env, jobject obj, jlong offset, jstring new_value) {
    // 绕过 JNI 调用，直接写字段
    char* field_addr = reinterpret_cast<char*>(obj) + offset;
    *reinterpret_cast<jstring*>(field_addr) = new_value;
    // ❌ 没调用写屏障！Card Table 没更新
}
```

**根因**：Unsafe 操作直接写内存，绕过所有 GC 屏障。

**修复**：
```cpp
void FastSetField(JNIEnv* env, jobject obj, jlong offset, jstring new_value) {
    char* field_addr = reinterpret_cast<char*>(obj) + offset;
    
    // 1. 调用写屏障（更新 Card Table）
    if (kUseWriteBarrier) {
        artWriteBarrier(field_addr, new_value);
    }
    
    // 2. 修改字段
    *reinterpret_cast<jstring*>(field_addr) = new_value;
}
```

## 九、本节小结

1. **写屏障 = 指针赋值时插入的钩子代码**
2. **CMS 的 Pre-Write Barrier**：Incremental Update，把被删除引用的对象重新染灰
3. **GenCC 的 Post-Write Barrier**：维护 Card Table，记录跨代引用
4. **写屏障在编译码中的实现**：AOT/JIT 编译时插入 call 指令
5. **工程坑点**：Hook 框架、JNI Unsafe 操作、反射都可能绕过写屏障

→ **理解写屏障，就理解了"为什么 ART 9+ 还需要写屏障"**——不是为不变式，是为 Card Table。
