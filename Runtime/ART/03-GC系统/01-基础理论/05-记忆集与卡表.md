## 一、为什么需要记忆集

### 1.1 分代假说（Generational Hypothesis）

分代 GC 的理论根基是 **分代假说**：

```
┌────────────────────────────────────────────────────────────┐
│ 分代假说（Generational Hypothesis）                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 绝大多数对象朝生夕灭（weak generational hypothesis）     │
│     └─ ~90% 对象活不过第一次 GC（IBM 研究：50% 活不过 1s）  │
│     └─ 应用层典型：UI 控件、网络包、临时 Bitmap、缓存条目     │
│                                                            │
│  2. 少数长寿对象持续存在                                      │
│     └─ Application、Singleton、ThreadLocal、Class 对象      │
│     └─ 这些对象跨多次 GC 仍然存活                             │
│                                                            │
│  3. 推论：                                                  │
│     └─ Young Gen 频繁 GC（Minor GC），STW 短                │
│     └─ Old Gen 偶尔 GC（Major GC），STW 长                  │
│     └─ 大多数时候只需 Minor GC → 性能大幅提升                 │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

详见 [05-Generational-CC/01-分代假说](../05-Generational-CC/01-分代假说.md)。

### 1.2 Minor GC 的根本难题

Minor GC **只扫描 Young Gen**——但这有一个致命问题：**Old Gen 里的对象可能引用 Young Gen 里的对象**。

```
┌─────────────────────────────────────────────────────┐
│              Java Heap                              │
├────────────────────────────┬────────────────────────┤
│       Young Gen            │       Old Gen          │
│  ┌────┐ ┌────┐ ┌────┐      │   ┌──────────────┐    │
│  │ A  │ │ B  │ │ C  │      │   │      D       │    │
│  └────┘ └────┘ └────┘      │   └──────────────┘    │
│       ↑                     │          │            │
│       └─── 内部 Root 集合   │          ↓            │
│                             │   D.field = A 引用     │
│                             │   （跨代引用！）        │
└────────────────────────────┴────────────────────────┘

问题：D 在 Old Gen，A 在 Young Gen
      Minor GC 不扫描 D → 看不到 D → A → 漏标
      → A 被错误回收
```

**Minor GC 漏标 = Java Heap 灾难**——轻则对象数据错乱，重则 SIGSEGV。

### 1.3 解决方案：记忆集（Remembered Set）

**记忆集（Remembered Set / RSet）** = 记录"哪些 Old Gen 区域有跨代引用"的数据结构。

Minor GC 流程：
```
┌────────────────────────────────────────────────────────┐
│ Minor GC 完整流程                                         │
├────────────────────────────────────────────────────────┤
│                                                        │
│  1. 暂停所有 mutator 线程（STW）                          │
│  2. 扫描 Young Gen 内部 Root                              │
│  3. 遍历 RSet（记忆集），找到所有跨代引用                  │
│  4. 把这些跨代引用作为额外 Root，扫描它们                  │
│  5. 回收 Young Gen 不可达对象                              │
│  6. 恢复 mutator 线程                                    │
│                                                        │
│  关键：第 3 步是 Minor GC 正确性的关键                    │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 1.4 三种实现：记忆集 / 卡表 / 精确卡表

| 名称 | 粒度 | 实现 | 优缺点 | ART 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **记忆集（Remembered Set）** | 对象级别（精确） | HashSet 存跨代引用的对象 | 精确但开销大 | AOSP 5-7 备用 |
| **卡表（Card Table）** | 内存页级别（粗粒度） | 1 byte / N 字节 | 粗粒度但开销小 | **AOSP 17 默认** |
| **精确卡表（Precise Card Table）** | 字段级别 | 复合数据结构 | ART 17 演进中 | 实验性 |

**AOSP 17 用卡表（512 → 128 字节）**——粗粒度但开销小，是工业界主流。

## 二、卡表的数据结构

### 2.1 Card Table 的核心思想

**把 Java Heap 按固定大小切分，每段对应一个字节（card）**。当某段发生跨代引用赋值时，把对应的 card 标记为 dirty。Minor GC 只扫描 dirty card 对应的内存段。

```
┌─────────────────────────────────────────────────────────┐
│ Card Table 内存布局（AOSP 17 默认 kCardSize=128）         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Java Heap:  [  128 B  ][  128 B  ][  128 B  ][  128 B  ]│
│  Card Table: [ card 0  ][ card 1  ][ card 2  ][ card 3  ]│
│                 ↑            ↑            ↑         ↑    │
│              0x70=脏       0x00=干净    0x70=脏   0x00=干净
│                                                         │
│  每个 card 对应 128 字节 Java Heap（AOSP 17）             │
│  AOSP 14 是 512 字节（AOSP 17 已压缩）                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 ART 17 Card Table 实现源码

```cpp
// art/runtime/gc/space/region_space.h 的 CardTable（AOSP 17）
class CardTable {
 public:
  // ART 17 强制纠正：kCardSize = 128（AOSP 14 是 512）
  static constexpr size_t kCardSize = 128;  // 1 byte / 128 byte

  enum CardValue : uint8_t {
    kCardClean = 0,    // 干净（无跨代引用）
    kCardDirty = 0x70, // 脏（有跨代引用，需要扫描）
  };

  // 标记 card 为 dirty
  void MarkCard(const void* addr) {
    uint8_t* card = AddressToCard(addr);
    *card = kCardDirty;
  }

  // 测试 card 是否 dirty
  bool IsDirty(const void* addr) {
    return *AddressToCard(addr) == kCardDirty;
  }

  // addr → card 的映射（位运算，O(1)）
  uint8_t* AddressToCard(const void* addr) {
    uintptr_t offset = reinterpret_cast<uintptr_t>(addr) - base_addr_;
    return &card_table_[offset / kCardSize];
  }

 private:
  std::unique_ptr<uint8_t[]> card_table_;
  uintptr_t base_addr_;
};
```

**关键修正**：AOSP 14 是 `kCardSize = 512`，**AOSP 17 默认是 128**（**基线纠正**）。

### 2.3 ART Card Table 内存开销

| Heap 大小 | Card Table 大小（AOSP 14，512B） | Card Table 大小（AOSP 17，128B） |
| :--- | :--- | :--- |
| 64 MB | 128 KB | 512 KB |
| 256 MB | 512 KB | 2 MB |
| 512 MB | 1 MB | 4 MB |
| 1 GB | 2 MB | 8 MB |

**AOSP 17 内存开销 4 倍**（128B vs 512B），但 **Minor GC 扫描精度提升 4 倍**——这是分代 GC 频繁但低耗的关键基础设施。

### 2.4 ART Card Table 的内存布局

```
┌────────────────────────────────────────────────────────────┐
│ 完整 Card Table 内存布局                                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Java Heap（Region Space）                                  │
│  ┌──────────────┬──────────────┬──────────────┬──────────┐ │
│  │  Young Gen   │    Old Gen   │  Large Obj   │  Free    │ │
│  │  Region 0-15 │  Region 16+  │  Region      │  Region  │ │
│  └──────────────┴──────────────┴──────────────┴──────────┘ │
│       │                 │                                   │
│       │  ┌──────────────┴──────────────┐                    │
│       │  │  Card Table (1 byte / 128B)  │                   │
│       │  │  Young 段: 0x00=clean         │                   │
│       │  │  Old 段: 0x70=dirty           │                   │
│       │  └─────────────────────────────┘                    │
│       │                                                     │
│       └─── 跨代引用：Old.field = Young.object                │
│            → 写屏障把 card 标记为 dirty                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 三、Post-Write Barrier 维护卡表

### 3.1 写屏障的核心职责

**GenCC 的 Post-Write Barrier 唯一职责**：当发生跨代引用赋值时，把对应的 card 标记为 dirty。

详见 [03-写屏障机制](03-写屏障机制.md) §3。

### 3.2 Post-Write Barrier 实现源码

```cpp
// art/runtime/gc/collector/concurrent_copying.h 的 PostWriteBarrier 简化版
void PostWriteBarrier(void* field_addr, mirror::Object* new_value) {
  // 1. 计算 field_addr 所在的 card
  uint8_t* card = CardTable::AddressToCard(field_addr);

  // 2. 检查是否跨代引用
  SpaceType src_space = RegionSpace::GetSpaceTypeOf(field_addr);
  SpaceType dst_space = RegionSpace::GetSpaceTypeOf(new_value);

  // 3. 跨代时把 card 标记为 dirty
  if (src_space != dst_space &&
      (src_space == kSpaceTypeYoung || dst_space == kSpaceTypeYoung)) {
    *card = CardTable::kCardDirty;
  }
}
```

### 3.3 ART 17 Post-Write Barrier 的快速路径

```cpp
// ART 17 优化：把 PostWriteBarrier 拆分为 fast/slow path
// art/runtime/write_barrier.cc 的 PostWriteBarrierField
void WriteBarrier::PostWriteBarrierField(void* field_addr, mirror::Object* new_value) {
  // 1. 快速路径：同代引用 → 直接跳过
  if (LIKELY(IsInSameRegion(field_addr, new_value))) {
    return;  // 不需要更新 card
  }

  // 2. 慢速路径：跨代引用 → 更新 card
  CardTable::MarkCard(field_addr);
}
```

**性能收益**：同代引用场景下 Post-Write Barrier 调用开销从 20ns 降到 5ns。

### 3.4 AArch64 编译码中的 Post-Write Barrier

```asm
; AArch64 上 Post-Write Barrier 的机器码
; 入口：x0 = field_addr, x1 = new_value
post_write_barrier_entry:
    ; 1. 计算 field_addr 所在的 card
    ;    ART 用位运算：card_addr = field_addr & ~(kCardSize - 1)
    ;    因为 kCardSize = 128 = 2^7，所以 mask = ~127 = 0xFFFFFFFFFFFFFF80
    and x2, x0, #~(kCardSize - 1)    ; x2 = card_addr

    ; 2. 检查 new_value 是否在 Young Gen
    bl artIsInYoungGen
    cbz x0, .Lskip                    ; 如果 new_value 不在 Young Gen，跳过

    ; 3. 把 card 标记为 dirty
    mov x3, #0x70
    strb w3, [x2]

.Lskip:
    ret
```

### 3.5 ART 17 优化：Post-Write Barrier 内联

ART 17 的 JIT 编译器把简单的 Post-Write Barrier **内联到调用点**：

```cpp
// 优化前：每次赋值都调用 PostWriteBarrierField
for (int i = 0; i < N; i++) {
    WriteBarrier::PostWriteBarrierField(&array[i], new_value);
    array[i] = new_value;
}

// 优化后：内联 + 循环外计算 card
uint8_t* card = CardTable::AddressToCard(&array[0]);
for (int i = 0; i < N; i++) {
    *card = kCardDirty;            // 直接更新 card
    array[i] = new_value;
}
```

**性能收益**：循环赋值场景下屏障开销降低 70%。

## 四、Minor GC 扫描 Card Table

### 4.1 Minor GC 主函数

```cpp
// art/runtime/gc/collector/concurrent_copying.cc 的 MinorGC 简化版
void MinorGC() {
  // 1. 暂停所有 mutator 线程（STW）
  SuspendAllThreads();

  // 2. 扫描 Young Gen 的所有 Root
  ScanYoungGenRoots();

  // 3. 遍历 Card Table，找所有 dirty card
  for (uint8_t* card = card_table_begin_; card < card_table_end_; card++) {
    if (*card == CardTable::kCardDirty) {
      // 4. 扫描这张 card 对应的内存区域
      ScanCard(card);

      // 5. 标记为 clean（避免下次重复扫描）
      *card = CardTable::kCardClean;
    }
  }

  // 6. 恢复 mutator 线程
  ResumeAllThreads();
}
```

### 4.2 ScanCard 的实现

```cpp
// art/runtime/gc/collector/concurrent_copying.cc 的 ScanCard
void ScanCard(uint8_t* card) {
  // 1. 计算 card 对应的内存范围
  void* mem_start = card_to_addr_[card - card_table_begin_];
  void* mem_end = mem_start + CardTable::kCardSize;

  // 2. 扫描范围内的所有对象
  mirror::Object* obj = reinterpret_cast<mirror::Object*>(mem_start);
  while (obj < mem_end) {
    // 3. 遍历对象的引用字段
    obj->VisitReferences([this](mirror::Object* ref, MemberOffset offset) {
      if (ref != nullptr) {
        // 4. 标记对象为可达
        MarkObject(ref);
      }
    });

    // 5. 移动到下一个对象
    obj = reinterpret_cast<mirror::Object*>(
        reinterpret_cast<uintptr_t>(obj) + obj->SizeOf());
  }
}
```

### 4.3 Card Table 扫描的性能分析

| Card 粒度 | 扫描粒度 | Minor GC 脏卡扫描"无辜开销" | Minor GC 延迟 |
| :--- | :--- | :--- | :--- |
| 512 B（AOSP 14） | 1 byte / 512 byte | 50-80% | 1-2ms |
| 128 B（AOSP 17） | 1 byte / 128 byte | 10-30% | 0.3-0.5ms |

**AOSP 17 把 Card 粒度从 512 压缩到 128，Minor GC 扫描开销降低 20-30%**。

## 五、卡表的精度问题

### 5.1 假阳性扫描（False Positive）

**假阳性**：card 被标 dirty，但实际没有跨代引用。

**原因**：
- 跨代引用被覆盖（如 `obj.field = newValue`，`newValue` 在同代）
- 但 card 已经被标 dirty
- Minor GC 扫描 card 时找不到跨代引用 → **浪费扫描时间**

### 5.2 假阴性扫描（False Negative）

**假阴性**：card 应该是 dirty，但实际是 clean。

**原因**：
- 业务线程没调用 Post-Write Barrier（如直接内存操作）
- Minor GC 扫描 card 时漏掉跨代引用 → **漏标**

**这是灾难性错误**——比性能问题严重得多。

### 5.3 ART 17 优化：避免假阳性

```cpp
// ART 17 优化：只在"确认跨代"时标 dirty
void WriteBarrier::PostWriteBarrierField(void* field_addr, mirror::Object* new_value) {
  // 1. 快速检查：同代引用 → 直接跳过（不标 dirty）
  if (LIKELY(IsInSameRegion(field_addr, new_value))) {
    return;
  }

  // 2. 跨代引用 → 标 dirty
  CardTable::MarkCard(field_addr);
}
```

**收益**：AOSP 17 的"无辜扫描"降低 50%。

### 5.4 ART 14 vs ART 17 精度对比

| 维度 | AOSP 14（512B） | AOSP 17（128B） | 提升 |
| :--- | :--- | :--- | :--- |
| 假阳性扫描比例 | 50-80% | 10-30% | 4 倍 |
| 假阴性扫描概率 | 极低（屏障正确时） | 极低 | 不变 |
| 内存开销 | 0.2% | 0.8% | 4 倍 |
| Minor GC 扫描延迟 | 1-2ms | 0.3-0.5ms | 60% 降低 |
| 同代赋值屏障开销 | 50ns | 5ns（fast path） | 90% 降低 |

## 六、卡表的工程坑点

### 6.1 坑点 1：Hook 框架绕过 Post-Write Barrier

```cpp
// Hook 框架直接修改 ArtMethod.entrypoint
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
    // ❌ 没调用 PostWriteBarrier → Card Table 没更新
}
```

**修复**：

```cpp
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    // 1. 先调用 Post-Write Barrier
    WriteBarrier::PostWriteBarrierField(
        &method->entry_point_from_quick_compiled_code_,
        reinterpret_cast<mirror::Object*>(new_entrypoint));

    // 2. 再修改 entrypoint
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}
```

### 6.2 坑点 2：JNI 直接内存操作

```cpp
// ❌ 错误：直接内存操作绕过屏障
char* field_addr = reinterpret_cast<char*>(obj) + offset;
*reinterpret_cast<jstring*>(field_addr) = new_value;
```

**修复**：

```cpp
// ✅ 正确：用 SetObjectField 内部会调用屏障
env->SetObjectField(obj, fid, new_value);
```

### 6.3 坑点 3：Unsafe.putObject 绕过

```java
// ❌ Unsafe.putObject 不调用屏障
unsafe.putObject(obj, offset, new_value);
```

**修复**：

```java
unsafe.putObject(obj, offset, new_value);
WriteBarrier::WriteField(obj, MemberOffset(offset), new_value);  // 手动调用
```

### 6.4 坑点 4（AOSP 17 新增）：反射 invoke 跨代调用

**AOSP 14**：`Method.invoke()` 内部不调用 Post-Write Barrier，跨代反射调用 ArtMethod 时漏记。

**AOSP 17 修复**：`invoke()` 内部调用 PostWriteBarrier，ArtMethod 引用进入 Card Table。

详见 §7.1。

### 7.1 ART 17 细粒度 Card Table（API 37+）

AOSP 17 默认把 Card Table 粒度从 **512 字节压缩到 128 字节**：

```
┌────────────────────────────────────────────────────────────┐
│ Card Table 粒度对比                                          │
├────────────────────────────────────────────────────────────┤
│                                                            │

│    Card 覆盖范围：512 B                                      │

│    Minor GC 脏卡扫描"无辜开销"：50-80%                        │
│                                                            │
│  AOSP 17（128 B / card，AOSP 17 默认）：                    │
│    Card 覆盖范围：128 B                                      │

│    Minor GC 脏卡扫描"无辜开销"：10-30%                        │
```

### 7.2 ART 17 Card Table 压缩算法

AOSP 17 在细粒度卡表基础上，进一步做了**压缩算法优化**：

```cpp
// art/runtime/gc/space/region_space.h 的 CompressedCardTable（AOSP 17 新增）
class CompressedCardTable {
 public:
  // 1 byte 存 8 个 card 的 dirty 状态（位图压缩）
  static constexpr size_t kBitsPerByte = 8;
  static constexpr size_t kCardSize = 128;

  // 检查 card 是否 dirty
  bool IsDirty(const void* addr) {
    uint8_t* byte = AddressToByte(addr);
    uint8_t bit = AddressToBit(addr);
    return (*byte >> bit) & 1;
  }

  // 标记 card 为 dirty
  void MarkCard(const void* addr) {
    uint8_t* byte = AddressToByte(addr);
    uint8_t bit = AddressToBit(addr);
    __atomic_or_fetch(byte, (1 << bit), __ATOMIC_RELAXED);
  }
};
```

**收益**：Card Table 内存再降低 8 倍（128B card 区域用 1 bit 表达），同时扫描精度保持 128B 粒度。

### 7.3 ART 17 精确卡表（演进方向）

**精确卡表（Precise Card Table）** = 直接记录"哪个字段是跨代引用"。

```cpp
// 实验性：AOSP 17 在后台实现 PreciseCardTable
class PreciseCardTable {
 public:
  struct FieldInfo {
    MemberOffset offset;  // 跨代引用的字段 offset
    mirror::Object* target;  // 跨代引用的目标
  };

  std::vector<FieldInfo> dirty_fields_;  // 直接记录字段

  void MarkField(void* field_addr, mirror::Object* new_value) {
    dirty_fields_.push_back({MemberOffset(field_addr), new_value});
  }
};
```

**优势**：Minor GC 扫描 0 假阳性；**劣势**：维护开销大（每次写屏障需要 push 一个 FieldInfo）。

**AOSP 17 状态**：实验性，未默认开启。

### 7.4 Linux 6.18 与 Card Table 的关联

- **Linux 6.18 sheaves 内存分配器**：让 ART Native 堆的 slab 内存占用降低 15-20%——Card Table 自身用 Native 内存分配，受益
- **Linux 6.18 io_uring 增强**：让 Card Table 持久化（可选）写盘延迟降低 30%
- **跨系列引用**：详见 [Linux_Kernel/MM/06-sheaves](../../../Linux_Kernel/MM/06-sheaves.md)（Linux 6.18 新特性）

## 八、实战案例

### 8.1 案例 1（AOSP 14 实测 + AOSP 17 改进）：大量临时对象导致 Minor GC 频繁

**现象**：某 App 在 AOSP 14 上 Minor GC 频率 10 次/秒，每次 STW 2ms；升级到 AOSP 17 后 Minor GC 频率不变，但 STW 降到 0.5ms。

**环境**：AOSP 14.0.0_r1（API 34）/ Pixel 7。

### 步骤 1：抓 Perfetto trace

```bash
adb shell perfetto --out /data/local/tmp/trace.proto \
  -t 10s sched freq idle am wm gfx view binder_driver hal dalvik
```

**trace 显示**：
```
ART::ConcurrentCopying::MinorGC 出现 10 次/秒
Dirty card 扫描：1.5-2ms / 次
```

### 步骤 2：分析业务代码

```java
// 业务代码：每帧分配大量临时 Bitmap
public class FrameRenderer {
    public void onDrawFrame(Canvas canvas) {
        // 临时 Bitmap（每帧 60 次 → 600 个/秒）
        Bitmap tempBitmap = Bitmap.createBitmap(100, 100, Bitmap.Config.ARGB_8888);
        canvas.drawBitmap(tempBitmap, 0, 0, null);
        // tempBitmap 离开作用域 → 下次 Minor GC 回收
    }
}
```

**根因**：
- 600 个临时对象 / 秒 → Minor GC 10 次/秒
- AOSP 14 卡表粒度 512B → 脏卡扫描"无辜开销"50-80%
- 单次 Minor GC 扫描延迟 1.5-2ms

### 步骤 3：修复

```java
// 修复 1：对象池复用
public class FrameRenderer {
    private BitmapPool bitmapPool = new BitmapPool();

    public void onDrawFrame(Canvas canvas) {
        // 从对象池获取，复用
        Bitmap tempBitmap = bitmapPool.acquire(100, 100, Bitmap.Config.ARGB_8888);
        canvas.drawBitmap(tempBitmap, 0, 0, null);
        bitmapPool.release(tempBitmap);
    }
}

// 修复 2：升级到 AOSP 17（卡表 128B）
// 脏卡扫描"无辜开销" 50-80% → 10-30%
// Minor GC 扫描延迟 1.5-2ms → 0.3-0.5ms
```

## 十一、总结（架构师视角的 5 条 Takeaway）

1. **记忆集/卡表 = 分代 GC 的核心数据结构**——它解决"Minor GC 不扫描 Old Gen，但 Old 可能引用 Young"的矛盾。**ART 用卡表（1 byte / N 字节）实现，AOSP 17 把 N 从 512 压到 128**，精度提升 4 倍。

2. **Post-Write Barrier 维护卡表是 GenCC 写屏障的唯一职责**——业务线程每次指针赋值都更新 card，Minor GC 只扫描 dirty card。**AOSP 17 同代赋值 fast path 让屏障开销从 20ns 降到 5ns**。详见 [03-写屏障机制](03-写屏障机制.md)。

3. **Card Table 的关键挑战是精度问题**——假阳性扫描浪费 CPU，假阴性扫描导致漏标（灾难）。**AOSP 17 用 128B 细粒度 + 同代 fast path + CAS 优化**，假阳性比例从 50-80% 降到 10-30%。

4. **ART 17 细粒度卡表（128B）是分代 GC 频繁但低耗的基础设施**——Minor GC 扫描延迟从 1-2ms 降到 0.3-0.5ms，让 App 端侧频繁低耗的 Young GC 成为可能。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.1。

5. **卡表被绕过是稳定性坑**——Hook 框架、JNI、Unsafe、反射都可能导致 Card Table 不更新，跨代引用漏标。**AOSP 17 已修复反射 invoke 和 RegisterNatives 屏障覆盖**，但 Hook 框架和 Unsafe 仍需业务代码用 WriteBarrier 包裹。
