# 1.5 记忆集与卡表

> **本节回答一个根本问题**：分代 GC（GenCC）怎么做到 Minor GC 只扫描 Young Gen？
>
> **答案**：用 **卡表（Card Table）**——一个 1 byte / 512 byte 的内存粒度记录器，记录 Old → Young 的跨代引用。
>
> **理解本节，就理解了"分代 GC 是怎么优化性能的"**——这是 ART 10.0+ 默认 GC 性能的关键。

---

## 一、为什么需要记忆集

### 1.5.1 分代假说

分代 GC 的理论根基是 **分代假说（Generational Hypothesis）**：
- **绝大多数对象朝生夕灭**（~90% 对象活不过第一次 GC）
- **少数长寿对象会持续存在**（如 Application、单例）

基于这个假说：
- 频繁 GC Young Gen（Minor GC），STW 时间短
- 偶尔 GC Old Gen（Major GC），STW 时间长
- **大多数时候只需 Minor GC**，性能大幅提升

### 1.5.2 Minor GC 的难题

Minor GC 只扫描 Young Gen：

```
┌─────────────────────────────────────────────┐
│              Java Heap                       │
├──────────────────────────────┬──────────────┤
│         Young Gen            │   Old Gen    │
│  ┌────┐ ┌────┐ ┌────┐        │  ┌────────┐  │
│  │ A  │ │ B  │ │ C  │        │  │   D    │  │
│  └────┘ └────┘ └────┘        │  └────────┘  │
│       ↑                       │      ↑      │
│       └─── Young Gen 内的对象  │      │      │
│              Root 集合         │  Old Gen    │
│                              │  内的对象   │
└──────────────────────────────┴──────────────┘
```

**问题**：如果 Old Gen 里的 D 引用了 Young Gen 里的 A，Minor GC 不扫描 D，会漏标 A（A 应该存活）。

**解决方案**：需要一个数据结构记录"Old Gen 里哪些地方有跨代引用"。这就是 **记忆集（Remembered Set / RSet）**。

### 1.5.3 记忆集 vs 卡表

| 名称 | 粒度 | 实现 | 优缺点 |
|:---|:---|:---|:---|
| **记忆集（Remembered Set）** | 对象级别（精确） | HashSet 存跨代引用的对象 | 精确但开销大 |
| **卡表（Card Table）** | 内存页级别（粗粒度） | 1 byte / 512 byte 字节数组 | 粗粒度但开销小 |
| **精确卡表（Precise Card Table）** | 字段级别（ART 14+） | 复合数据结构 | ART 14+ 优化中 |

ART 用 **卡表**（512 byte 粒度）。

## 二、卡表的实现

### 1.5.4 卡表的数据结构

```cpp
// art/runtime/gc/space/region_space.h 的 CardTable 简化版
class CardTable {
 public:
  static constexpr size_t kCardSize = 512;  // 1 byte / 512 byte
  
  enum CardValue : uint8_t {
    kCardClean = 0,    // 干净（无跨代引用）
    kCardDirty = 0x70, // 脏（有跨代引用，需要扫描）
  };
  
  // 标记 card 为 dirty
  void MarkCard(const void* addr) {
    uint8_t* card = AddressToCard(addr);
    *card = kCardDirty;
  }
  
  // 测试 card 是否 dirty
  bool IsDirty(const void* addr) {
    return *AddressToCard(addr) == kCardDirty;
  }
  
  // addr → card 的映射
  uint8_t* AddressToCard(const void* addr) {
    uintptr_t offset = reinterpret_cast<uintptr_t>(addr) - base_addr_;
    return &card_table_[offset / kCardSize];
  }
  
 private:
  std::unique_ptr<uint8_t[]> card_table_;
  uintptr_t base_addr_;
};
```

### 1.5.5 卡表的内存布局

```
Java Heap:   [  512 B  ][  512 B  ][  512 B  ][  512 B  ][  512 B  ] ...
Card Table:  [ card 0  ][ card 1  ][ card 2  ][ card 3  ][ card 4  ] ...
                ↑           ↑           ↑           ↑           ↑
             0x70=脏     0x00=干净    0x00=干净    0x70=脏     0x00=干净

每个 card 对应 512 字节 Java Heap
```

### 1.5.6 ART 的 Card Table 内存开销

| Heap 大小 | Card Table 大小 |
|:---|:---|
| 64 MB | 128 KB（64 MB / 512） |
| 256 MB | 512 KB |
| 512 MB | 1 MB |
| 1 GB | 2 MB |

→ **内存开销约 0.2%**，可接受。

## 三、写屏障维护卡表

### 1.5.7 Post-Write Barrier 维护 Card Table

```cpp
// art/runtime/gc/collector/concurrent_copying.h 的 PostWriteBarrier 简化版
void PostWriteBarrier(void* field_addr, mirror::Object* new_value) {
  // 1. 计算 field_addr 所在的 card
  uint8_t* card = CardTable::AddressToCard(field_addr);
  
  // 2. 检查是否跨代引用
  SpaceType src_space = RegionSpace::GetSpaceTypeOf(field_addr);
  SpaceType dst_space = RegionSpace::GetSpaceTypeOf(new_value);
  
  // 3. 跨代时把 card 标记为 dirty
  if (src_space != dst_space && 
      (src_space == kSpaceTypeYoung || dst_space == kSpaceTypeYoung)) {
    *card = CardTable::kCardDirty;
  }
}
```

### 1.5.8 PostWriteBarrier 的伪代码逻辑

```cpp
// 简化版
void PostWriteBarrierSlow(void* field_addr, Object* new_value) {
    // 假设：
    // - field_addr 在 Old Gen
    // - new_value 在 Young Gen
    // → 跨代引用，需要标记 card
    
    uint8_t* card = (uint8_t*)((uintptr_t)field_addr & ~(kCardSize - 1));
    *card = kCardDirty;  // 标记为 dirty
}
```

### 1.5.9 编译码中的 PostWriteBarrier 实现

```asm
; AArch64 上 PostWriteBarrier 的机器码
; 入口：x0 = field_addr, x1 = new_value
post_write_barrier_entry:
    ; 1. 计算 field_addr 所在的 card
    ;    ART 用位运算：card_addr = field_addr & ~(kCardSize - 1)
    ;    因为 kCardSize = 512 = 2^9，所以 mask = ~511 = 0xFFFFFFFFFFFFFE00
    and x2, x0, #~(kCardSize - 1)    ; x2 = card_addr
    
    ; 2. 检查 new_value 是否在 Young Gen
    ;    ART 通过 region space 类型判断
    bl artIsInYoungGen
    cbz x0, .Lskip                    ; 如果 new_value 不在 Young Gen，跳过
    
    ; 3. 把 card 标记为 dirty
    mov x3, #0x70
    strb w3, [x2]
    
.Lskip:
    ret
```

### 1.5.10 JIT 模式下的 PostWriteBarrier 优化

JIT 编译器做以下优化：

1. **跨代判断提到循环外**：
   ```cpp
   // 优化前
   for (int i = 0; i < N; i++) {
       PostWriteBarrier(&arr[i], value);  // 每次循环都判断
       arr[i] = value;
   }
   
   // 优化后
   bool is_young = IsInYoungGen(value);  // 循环外判断
   for (int i = 0; i < N; i++) {
       if (is_young) {
           PostWriteBarrier(&arr[i], value);
       }
       arr[i] = value;
   }
   ```

2. **Card Table 预计算**：
   ```cpp
   // 优化后
   uint8_t* card = CardTable::AddressToCard(&arr[0]);  // 循环外计算
   for (int i = 0; i < N; i++) {
       *card = kCardDirty;  // 直接更新 card
       arr[i] = value;
   }
   ```

## 四、Minor GC 如何使用卡表

### 1.5.11 Minor GC 的完整流程

```cpp
// art/runtime/gc/collector/concurrent_copying.cc 的 Minor GC 简化版
void MinorGC() {
  // 1. 暂停所有 mutator 线程（STW）
  SuspendAllThreads();
  
  // 2. 扫描 Young Gen 的所有 Root
  ScanYoungGenRoots();
  
  // 3. 遍历 Card Table，找所有 dirty card
  for (uint8_t* card = card_table_begin_; card < card_table_end_; card++) {
    if (*card == CardTable::kCardDirty) {
      // 4. 扫描这张 card 对应的 512 字节内存
      ScanCard(card);
      
      // 5. 标记为 clean
      *card = CardTable::kCardClean;
    }
  }
  
  // 6. 恢复 mutator 线程
  ResumeAllThreads();
}
```

### 1.5.12 ScanCard 的实现

```cpp
void ScanCard(uint8_t* card) {
  // 1. 计算 card 对应的 512 字节范围
  void* region_start = (void*)((uintptr_t)card - card_table_base_ + heap_base_);
  void* region_end = (char*)region_start + kCardSize;
  
  // 2. 遍历范围内的所有对象
  for (mirror::Object* obj = (mirror::Object*)region_start; 
       obj < region_end; 
       obj = NextObject(obj)) {
    // 3. 遍历对象的字段，找跨代引用
    obj->VisitReferences([&](mirror::Object* ref) {
      if (ref != nullptr && IsInYoungGen(ref)) {
        // 4. 把 Young Gen 的对象染灰
        MarkObject(ref);
      }
    });
  }
}
```

### 1.5.13 性能对比：有无卡表

**场景**：Old Gen 256 MB，Young Gen 64 MB，1% 的 Old Gen 区域有跨代引用

| 方案 | 扫描范围 | STW 时间（相对值） |
|:---|:---|:---|
| **无卡表**（扫描整个 Old Gen） | 256 MB | 100% |
| **有卡表**（只扫描 dirty cards） | ~2.5 MB（1% × 256 MB） | ~1% |

→ **卡表让 Minor GC 扫描范围减少 99%**，STW 时间降低 99%。

## 五、卡表的精度问题

### 1.5.14 卡表的粗粒度代价

每张 dirty card 对应 512 字节的扫描范围：

```
┌────────────────────────────────────────────────────────┐
│   Dirty Card                                          │
│   ┌──────────┬──────────┬──────────┬──────────┐        │
│   │ 128 B    │ 128 B    │ 128 B    │ 128 B    │        │
│   │ Card数据 │ Card数据 │ Card数据 │ Card数据 │        │
│   └──────────┴──────────┴──────────┴──────────┘        │
│   ↑                                              ↑    │
│   跨代引用在这里                                  扫描终止
└────────────────────────────────────────────────────────┘
   扫描 512 字节，找到所有跨代引用
```

**问题**：
- 1 byte / 512 byte 的粒度 → 每次脏卡扫描的"无辜开销"是 512 字节
- 大对象（> 512 字节）可能浪费大量扫描
- 多个对象的多个引用集中在同一 card → 整张 card 都扫描

### 1.5.15 ART 14+ 的优化：细粒度卡表

ART 14+ 引入 **细粒度卡表**：

| 粒度 | 实现 | 适用场景 |
|:---|:---|:---|
| 512 byte | 经典卡表 | 大对象少 |
| 256 byte | 两级卡表 | 大对象中等 |
| 128 byte | 三级卡表 | 大对象多 |

**示例**：
```cpp
// art/runtime/gc/space/region_space.h 的细粒度卡表
class FineGrainedCardTable {
 public:
  // 两级卡表：
  // - 第一级：1 byte / 8 KB（汇总）
  // - 第二级：1 byte / 1 KB（细粒度）
  
  void MarkCard(const void* addr) {
    uint8_t* coarse_card = CoarseAddressToCard(addr);
    uint8_t* fine_card = FineAddressToCard(addr);
    
    *fine_card = kCardDirty;
    *coarse_card = kCardDirty;  // 汇总标记
  }
  
  // 扫描时先看第一级，只扫描 dirty 的第二级
  void ScanDirtyCards() {
    for (auto* coarse_card : coarse_cards_) {
      if (*coarse_card == kCardDirty) {
        // 扫描这个粗粒度范围内的细粒度 cards
        ScanFineCards(coarse_card);
      }
    }
  }
};
```

### 1.5.16 ART 14+ 的 Hot Card 优化

**Hot Card 问题**：某些 card 被频繁标脏（如被高并发线程写入）。

```cpp
// Hot Card 优化（ART 14+）
void PostWriteBarrier(void* field_addr, Object* new_value) {
  uint8_t* card = AddressToCard(field_addr);
  
  // 检查 card 是否被频繁标脏
  if (card_hotness_[card] > kHotCardThreshold) {
    // Hot Card：直接扫描，不等 Minor GC
    ScanCardImmediately(card);
  }
  
  *card = kCardDirty;
}
```

→ Hot Card 优化可以避免 Minor GC 时扫描大量 dirty card 导致的"GC 卡顿"。

## 六、Remembered Set 的其他实现

### 1.5.17 ART Region Space 的 RSet

ART 的 Region Space 还用了 **Remembered Set** 作为补充：

```cpp
// art/runtime/gc/space/region_space.h 的 RSet
class Region {
 public:
  // Region 的 Remembered Set
  // 记录这个 Region 被哪些其他 Region 引用
  std::vector<Region*> inbound_refs_;
  
  void AddInboundRef(Region* from_region) {
    inbound_refs_.push_back(from_region);
  }
  
  // Minor GC 时只扫描 inbound_refs_ 里的 Region
  void ScanInboundRefs() {
    for (Region* region : inbound_refs_) {
      region->ScanReferences();
    }
  }
};
```

**RSet vs Card Table**：
- **Card Table**：粗粒度（512 byte），全堆统一维护
- **RSet**：细粒度（Region 级别），按 Region 维护
- ART **同时使用两者**：Card Table 是主要机制，RSet 是补充优化

### 1.5.18 G1 GC 的 RSet（对比）

G1 GC（Java HotSpot）也用 RSet，但实现更复杂：

```cpp
// G1 GC 的 RSet 简化版
class G1RSet {
 public:
  // 每个 Region 有一个 RSet
  // RSet 是 Hash Table，key 是 Card，value 是引用源
  std::unordered_map<Card*, std::vector<Region*>> rset_;
  
  void AddRef(Card* card, Region* from) {
    rset_[card].push_back(from);
  }
};
```

G1 的 RSet 精确度更高，但维护开销也更大。

## 七、稳定性关联：卡表导致的问题

### 1.5.19 问题 1：Hook 框架绕过 Card Table

**场景**：
```cpp
// Hook 框架绕过 PostWriteBarrier
void HookArtMethod(ArtMethod* method, void* new_entrypoint) {
    // 直接修改 ArtMethod.entrypoint，不调用 PostWriteBarrier
    method->entry_point_from_quick_compiled_code_ = new_entrypoint;
}

// CC GC 移动 ArtMethod 到新 Region（Old Gen）
// 业务线程调用 method.entrypoint → 读屏障
// 但 Card Table 没更新（PostWriteBarrier 被绕过）
// → Minor GC 扫描这个 Region 时，不知道有跨代引用
// → 漏标 ArtMethod → 错误回收 → 崩溃
```

**修复**：
- ART 13+ 引入了 **JIT 代码校验**，部分解决此问题
- Hook 框架必须用 `WriteBarrier::WriteField()` 包裹修改

### 1.5.20 问题 2：卡表粒度导致 STW 抖动

**场景**：
```java
// 大对象分配
byte[] bigArray = new byte[10 * 1024 * 1024]; // 10 MB
// 这张大对象占用 20 张 card（10 MB / 512 KB）

// 大对象触发 PostWriteBarrier，把大量 card 标脏
// Minor GC 时扫描这些 dirty card → STW 时间抖动
```

**修复**：
- ART 14+ 的 Hot Card 优化
- 大对象单独处理（Large Object Space）

### 1.5.21 问题 3：跨代引用过度频繁

**场景**：
```java
// 高频跨代引用
class Cache {
    private static final Map<Key, Value> cache = new HashMap<>();  // Old Gen
    
    public void put(Key key, Value value) {
        cache.put(key, value);  // Young Gen → Old Gen
        // 每次 put 都触发 PostWriteBarrier，标脏 card
    }
}
```

**问题**：
- 频繁 PostWriteBarrier → 卡表频繁 dirty → Minor GC 扫描开销大
- **分代假说失效**：大量长寿对象污染 Young Gen

**修复**：
- 让 Cache 自身在 Old Gen（避免跨代引用）
- 用 `ConcurrentHashMap` 替代 `HashMap`

## 九、本节小结

1. **记忆集 = 跨代引用的记录器**，让 Minor GC 不扫描整个 Old Gen
2. **卡表 = 1 byte / 512 byte 的粗粒度记忆集**，ART 的实际实现
3. **PostWriteBarrier 维护卡表**：每次指针赋值都更新卡表
4. **Minor GC 扫描 dirty cards**：只扫描跨代引用所在的 512 字节
5. **卡表的精度问题**：1 byte / 512 byte 粒度 → ART 14+ 优化为细粒度

→ **理解卡表，就理解了 GenCC 性能优化的关键**——99% 的 Minor GC 性能提升来自卡表。
