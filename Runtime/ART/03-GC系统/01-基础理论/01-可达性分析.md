# 1.1 可达性分析：为什么不是引用计数

> **本节回答一个根本问题**：ART（以及所有现代 JVM）用什么方式判定"对象死还是活"？
>
> **答案**：可达性分析（Reachability Analysis），不是引用计数（Reference Counting）。
>
> **理解本节，就理解了 GC Root 的 12 种来源**——这是排查所有 Java 堆泄漏的起点。

---

## 一、为什么不用引用计数

### 1.1.1 引用计数的致命缺陷

引用计数的逻辑很简单：
```
每个对象维护一个计数器
每被一个指针引用 +1
每断开一个引用 -1
计数器 = 0 → 回收
```

听起来很完美。但有两个致命缺陷：

#### 缺陷 1：循环引用无法回收

```java
public class Node {
    public Node other;
}

public static void main(String[] args) {
    Node a = new Node();   // a.count = 1
    Node b = new Node();   // b.count = 1
    a.other = b;           // b.count = 2
    b.other = a;           // a.count = 2
    a = null;              // a.count = 1（但 b.other 还引用着 a）
    b = null;              // b.count = 1（但 a.other 还引用着 b）
    // 引用计数算法：a.count = 1, b.count = 1，都 ≠ 0，不回收
    // 实际：a 和 b 都已经从 root 不可达，是垃圾
}
```

**这就是循环引用**——a 引用 b，b 引用 a，计数永远不为 0。

#### 缺陷 2：计数操作的开销太大

每次指针赋值都要更新两个对象的计数（被赋值的对象 -1，新值对象 +1）：

```java
public void reassign() {
    this.field = newValue;  // 实际机器码层面要执行：
                            //   1. 读 this.field（old_value）
                            //   2. old_value.count--
                            //   3. newValue.count++
                            //   4. this.field = newValue
                            //   5. 写屏障（如果有）
}
```

与业务读写严重耦合，**性能差 + 正确性难保证**（多线程下计数要原子操作）。

#### Java / Kotlin 对象图的循环无处不在

- **双向链表**：Node.prev ↔ Node.next
- **树形 UI 节点**：ViewGroup.parent ↔ View.children
- **图结构**：Graph.vertex.neighbors
- **Window/View 体系**：Window ↔ DecorView ↔ ViewRootImpl ↔ Window
- **Binder 引用**：Service ↔ ServiceConnection
- **EventBus / RxJava 订阅链**：Publisher ↔ Subscriber

→ **引用计数根本无法 hold 住 Java 生态**。

### 1.1.2 可达性分析的定义

```
GC Roots（根对象集合）
    │
    │  从 GC Roots 出发，沿着引用链（Reference Chain）遍历
    ▼
可达对象（Reachable）= 存活
不可达对象（Unreachable）= 死亡，下一轮 GC 回收
```

**核心思想**：不是"看一个对象被多少指针引用"，而是"看一个对象能不能从某个起点走通"。

类比：
- 引用计数 = "看一个岛屿上有多少人"
- 可达性分析 = "看一个岛屿能不能通过桥梁到达大陆"

> 哪怕岛屿上有很多人，只要没有桥与大陆相连，就是孤岛（垃圾）。

### 1.1.3 可达性分析 vs 引用计数对比

| 维度 | 引用计数 | 可达性分析 |
|:---|:---|:---|
| 循环引用 | ❌ 致命缺陷 | ✅ 天然解决 |
| 计数开销 | ❌ 每次指针赋值都要更新 | ✅ 只在 GC 时遍历 |
| 多线程支持 | ❌ 需要原子操作 | ✅ GC 时 STW |
| 实时性 | ✅ 计数为 0 立即回收 | ❌ GC 时机决定 |
| 内存局部性 | ✅ 回收的对象相邻 | ❌ 可能碎片化 |
| 工程实现 | ❌ 复杂 | ✅ 简单（DFS / BFS） |
| 使用场景 | Objective-C / Swift（arc）/ COM / 文件系统 | Java / Kotlin / Go / Python |

**关键 insight**：可达性分析的"实时性差"恰好被 **分代假说 + 并发 GC** 弥补——这是 ART GenCC 的设计哲学。

## 二、ART 的 GC Root 来源全景

### 1.1.4 ART 的 RootType 枚举

ART 把"可能的起点"分成 **12 种类型**，每种对应一个 Visit 函数：

```cpp
// art/runtime/gc/root_visitor.h（精简版）
enum RootType {
  kRootUnknown = 0,
  kRootJniGlobal,           // JNI Global Reference
  kRootJniLocal,            // JNI Local Reference
  kRootJavaFrame,           // Java 栈帧
  kRootNativeStack,         // Native 栈
  kRootStickyClass,         // 永久类（Boot ClassLoader 加载的）
  kRootThreadBlock,         // 阻塞的 Thread 对象
  kRootMonitorUsed,         // 被用的 Monitor
  kRootInternedString,      // String 常量池
  kRootFinalizing,          // 正在 Finalize 的对象
  kRootSystemServer,        // System Server 特殊 Root
  kRootReferenceCleanup,    // Reference 清理列表
  kRootNone
};
```

下面对每种 Root 做详细解读。

### 1.1.5 12 种 GC Root 详解

#### Root 1：`kRootJniGlobal`（JNI Global Reference）

```cpp
// JNI Global Reference 是 ART 强引用的对象
// 任何 jni->NewGlobalRef() 都不会被 GC 回收（直到 DeleteGlobalRef）

// 经典泄漏场景
extern "C" JNIEXPORT void JNICALL
Java_com_example_NativeLeak_nativeInit(JNIEnv* env, jobject thiz) {
    // 把 thiz 存到全局变量，没释放 → thiz 永远不回收
    g_obj = env->NewGlobalRef(thiz);  // 泄漏！
}
```

**稳定性排查**：JNI Global Ref 泄漏 = Java 堆泄漏。

**排查工具**：
```bash
# dumpsys meminfo 查看 JNI 引用数
adb shell dumpsys meminfo <package_name> | grep -i "JNI"
```

如果 JNI Global References 持续增长且不收敛，就是泄漏。

#### Root 2：`kRootJniLocal`（JNI Local Reference）

```cpp
// JNI Local Reference 在 native 函数返回时自动释放
// 但循环 / 长生命周期函数里如果不手动 DeleteLocalRef，会撑爆 LocalRef 表

void JNI循环() {
    for (int i = 0; i < 1000000; i++) {
        // 每次循环都 New 一个 Local Ref
        jstring jstr = env->NewStringUTF("hello");
        // 1百万次循环后：Local Ref 表溢出 → 崩溃
        // 错误：JNI ERROR (app bug): local reference table overflow
    }
}
```

**稳定性坑点**：Local Ref 表默认容量是 512 个引用（`art/runtime/jni/jni_internal.cc` 的 `kMaxLocalRefTableSize`）。

**修复方案**：
```cpp
void JNI循环() {
    for (int i = 0; i < 1000000; i++) {
        jstring jstr = env->NewStringUTF("hello");
        env->DeleteLocalRef(jstr);  // 用完就释放
    }
}
```

#### Root 3：`kRootJavaFrame`（Java 栈帧）

```cpp
// Java 方法栈帧里的局部变量表 + 操作数栈
// 每个栈帧引用的对象都是 GC Root

void example() {
    Object a = new Object();   // a 在栈帧的局部变量表里
    Object b = new Object();   // b 在栈帧里
    // a 和 b 都是 GC Root（存活）
}
```

**稳定性坑点**：
- **栈帧引用过多** → GC Root 扫描变慢 → STW 时间变长
- **长生命周期线程**（如 SystemServer 主线程）持有大量栈帧引用 → 影响 GC 效率

#### Root 4：`kRootNativeStack`（Native 栈）

```cpp
// Native 栈（机器栈）上如果有 mirror::Object* 指针，也是 GC Root
// 通常通过 ART 的 "stack walking" 机制枚举

// 经典场景：ART 8+ 的 nterp / JIT 代码可能把对象引用放在 native 栈上
```

#### Root 5：`kRootStickyClass`（永久类）

```cpp
// Boot ClassLoader 加载的类不会被回收（永久）
// 这些类对象是 GC Root

// 例子：String.class、Integer.class、Activity.class
// 这些类对象在进程生命周期内永远存在
```

**为什么"sticky"**：这些类加载得太早，回收成本远大于收益。直接当 Root 处理。

#### Root 6：`kRootThreadBlock`（阻塞的 Thread 对象）

```cpp
// 所有 Thread 对象（包括 Live + Daemon）都是 GC Root
// 直到 Thread.run() 执行完毕

// 经典场景：忘记调用 Thread.stop() / interrupt()
// → Thread 对象一直存活 → 持有的所有对象都不回收
```

#### Root 7：`kRootMonitorUsed`（被用的 Monitor）

```cpp
// 被 synchronized / Object.wait() / Object.notify() 占用的对象
// 这些对象上的 Monitor 不能被 GC 回收

// 经典场景：长生命周期的 Monitor 持有 → 死锁或泄漏
```

#### Root 8：`kRootInternedString`（String 常量池）

```cpp
// String.intern() 加入 String Table 的字符串是 GC Root

String s1 = "hello";        // 字面量自动 intern
String s2 = new String("hello").intern();  // 手动 intern

// s1 和 s2 都指向 String Table 里的同一个对象
// 这个对象是 GC Root，永远不回收（除非 String Table 清理，Android 8+ 才有）
```

**稳定性坑点**：
- **String 重复 intern** → String Table 持续增长 → 内存泄漏
- **Android 8+ 的优化**：String Table 可以 GC 了（按使用频率清理）

#### Root 9：`kRootFinalizing`（正在 Finalize 的对象）

```cpp
// 正在被 FinalizerDaemon 处理 finalize() 方法的对象
// 这些对象暂时不能回收（要等 finalize() 执行完）

// 06 篇详细讲：FinalizerDaemon + FinalizerWatchdogDaemon
```

#### Root 10：`kRootSystemServer`（System Server 特殊 Root）

```cpp
// System Server 进程的特殊 GC Root
// System Server 不能轻易 OOM（会系统重启）
// 所以很多关键对象被标记为 Root
```

**稳定性影响**：System Server OOM = 系统重启（重启 System Server = 重启整个 Android Framework）。

#### Root 11：`kRootReferenceCleanup`（Reference 清理列表）

```cpp
// java.lang.ref.Reference 的 pending list
// 待清理的 Soft / Weak / Phantom / Final Reference

// 06 篇详细讲：Reference 体系与 pending list 处理
```

#### Root 12：`kRootUnknown`（兜底 Root）

理论上不存在的 Root，用于调试和日志。

### 1.1.6 RootType 实战对照表

| RootType | 排查工具 | 常见问题 | 修复方案 |
|:---|:---|:---|:---|
| `kRootJniGlobal` | `dumpsys meminfo` | JNI Global Ref 泄漏 | 检查所有 `NewGlobalRef`，确保配对 `DeleteGlobalRef` |
| `kRootJniLocal` | logcat `JNI ERROR` | Local Ref 表溢出 | 循环 / 长函数里手动 `DeleteLocalRef` |
| `kRootJavaFrame` | Perfetto / systrace | 栈帧引用过多导致 STW 慢 | 减少长生命周期线程的栈帧深度 |
| `kRootStickyClass` | 静态分析 | 动态加载类不释放 | 用自定义 ClassLoader，避免类泄漏 |
| `kRootInternedString` | heap dump | String.intern 重复 | 改用 WeakHashMap / LRU 缓存 |
| `kRootThreadBlock` | `jstack` | Thread 不退出 | 检查 Thread 是否忘了 stop / interrupt |
| `kRootFinalizing` | `FinalizerWatchdog` 日志 | finalize() 阻塞 | 完全禁止 finalize()，改用 PhantomRef + Cleaner |

## 三、ART 的 Root 访问机制

### 1.1.7 `Heap::VisitRoots` 总入口

```cpp
// art/runtime/gc/heap.cc
void Heap::VisitRoots(RootVisitor* visitor) {
  // 1. 切换到 GC 上下文（暂停所有 mutator 线程）
  ThreadList* tl = ThreadList::Current();
  tl->SuspendAll();
  
  // 2. 访问 JNI Global References
  jni_globals->VisitRoots(visitor);
  
  // 3. 访问 JNI Local References（每个线程的 JNIEnv）
  tl->VisitRoots(visitor);
  
  // 4. 访问 Interned Strings
  intern_table_->VisitRoots(visitor);
  
  // 5. 访问 Class Objects
  class_linker_->VisitRoots(visitor);
  
  // 6. 访问 Reference Cleaner
  ref_processor_->VisitRoots(visitor);
  
  // 7. 恢复 mutator 线程
  tl->ResumeAll();
}
```

**关键点**：
- `SuspendAll()` → 进入 STW（Stop-The-World）状态
- 所有 Root 访问都在 STW 期间进行（线程全部暂停）
- STW 时间 = 12 种 Root 的访问时间 + 标记时间

### 1.1.8 RootVisitor 模式

```cpp
// art/runtime/gc/root_visitor.h
class RootVisitor {
 public:
  // 访问一个对象引用
  virtual void VisitRoot(mirror::Object** obj, size_t vreg, const RootInfo& info) = 0;
  virtual void VisitRoot(mirror::CompressedReference<mirror::Object>* ref, ...) = 0;
  
  // 批量访问优化
  virtual void VisitRoots(mirror::Object*** roots, size_t count, ...) {}
  virtual void VisitRoots(mirror::CompressedReference<mirror::Object>** roots, ...) {}
};

// GC 实现：MarkingRootVisitor
class MarkingRootVisitor : public RootVisitor {
  void VisitRoot(mirror::Object** obj, size_t vreg, const RootInfo& info) override {
    // 把对象染灰
    if (*obj != nullptr) {
      mark_stack_->Push(*obj);  // 加入 mark stack
    }
  }
};
```

### 1.1.9 线程栈扫描：`Thread::VisitRoots`

栈扫描是 GC Root 访问里**最耗时的部分**之一：

```cpp
// art/runtime/thread.cc 的简化版
void Thread::VisitRoots(RootVisitor* visitor) {
  // 1. 扫描 Java 栈
  for (StackFrame<mirror::Object>* frame = stack_; frame != nullptr; frame = frame->next_) {
    frame->VisitRoots(visitor);
  }
  
  // 2. 扫描本地变量表 + 操作数栈
  // ...
  
  // 3. 扫描 Native 栈（如果有对象引用）
  // ...
  
  // 4. 扫描 Thread 对象本身的字段（如 thread.name、thread.target）
  VisitObjectReferences(visitor, this, &thread_obj_);
}
```

**栈扫描的优化**：
- **指针映射表（Stack Map）**：AOT/JIT 编译时记录栈帧每个 slot 的类型，避免扫描时逐 slot 试探
- **AArch64 上特殊处理**：用 `fp` 寄存器帧指针 + 固定偏移

## 四、稳定性排查实战

### 1.1.10 案例 1：JNI Global Ref 泄漏导致 OOM

**场景**：
```java
public class MyService extends Service {
    static {
        System.loadLibrary("native");
    }
    
    public native void nativeInit();
    public native void nativeRelease();
}
```

```cpp
// native 代码
static jobject g_service_ref = nullptr;  // 全局引用

extern "C" JNIEXPORT void JNICALL
Java_com_example_MyService_nativeInit(JNIEnv* env, jobject thiz) {
    g_service_ref = env->NewGlobalRef(thiz);  // 创建全局引用，没释放！
}

// 多次创建 MyService 实例后 → g_service_ref 被覆盖，但每个 MyService 都不回收
```

**排查过程**：

```bash
# 1. dumpsys meminfo 看 JNI 引用数
$ adb shell dumpsys meminfo com.example.app | grep -i jni
                       JNI:    12345     7890     6789
                                          ↑↑↑↑↑
                                          JNI Global References: 12345（异常高）
```

**修复**：
```cpp
extern "C" JNIEXPORT void JNICALL
Java_com_example_MyService_nativeInit(JNIEnv* env, jobject thiz) {
    if (g_service_ref != nullptr) {
        env->DeleteGlobalRef(g_service_ref);  // 先释放旧的
    }
    g_service_ref = env->NewGlobalRef(thiz);
}

extern "C" JNIEXPORT void JNICALL
Java_com_example_MyService_nativeRelease(JNIEnv* env, jobject thiz) {
    if (g_service_ref != nullptr) {
        env->DeleteGlobalRef(g_service_ref);
        g_service_ref = nullptr;
    }
}
```

### 1.1.11 案例 2：Thread 不退出导致 OOM

**场景**：
```java
public class WorkerThread extends Thread {
    private volatile boolean running = true;
    
    public void shutdown() {
        running = false;
    }
    
    @Override
    public void run() {
        while (running) {
            // ... 业务逻辑
        }
        // 忘了调用 interrupt()
    }
}
```

```java
// Activity.onDestroy 里调 WorkerThread.shutdown()
// 但 WorkerThread 的 run() 还在 while(running) 循环
// → Thread 对象一直是 GC Root → 持有 WorkerThread 引用的所有对象都不回收
```

**排查过程**：
```bash
# 1. dumpsys meminfo 看 Thread 数
$ adb shell dumpsys meminfo com.example.app | grep -i thread

# 2. jstack 看具体哪些 Thread
$ adb shell ps -T -p <pid> | wc -l
# 假设有 2000+ 个 Thread → 大量泄漏

# 3. 看 Thread 状态
$ adb shell kill -3 <pid>
# 在 logcat 里看 Thread 状态分布
```

**修复**：
```java
@Override
public void run() {
    while (running) {
        // ... 业务逻辑
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            break;  // 响应中断
        }
    }
}

public void shutdown() {
    running = false;
    interrupt();  // 强制中断
}
```

### 1.1.12 案例 3：String.intern() 滥用导致 OOM

**场景**：
```java
// 用 String.intern() 做去重
Map<String, String> map = new HashMap<>();
for (String key : keys) {
    map.put(key.intern(), value);  // 每个 key 都 intern
}
```

**根因**：String.intern() 加入 String Table 后是 GC Root，永远不回收（Android 7 及之前）。

**修复**：
```java
// 改用 LRU 缓存
LRUCache<String, String> cache = new LRUCache<>(1000);
for (String key : keys) {
    cache.put(key, value);  // 自动淘汰
}
```

## 七、本节小结

1. **可达性分析 vs 引用计数**：ART 用可达性分析，因为引用计数解决不了循环引用
2. **12 种 GC Root**：ART 把可能的起点分成 12 种类型，每种对应一个 Visit 函数
3. **栈扫描是 STW 的主要耗时**：AOT/JIT 用 Stack Map 优化栈扫描
4. **实战排查**：
   - `dumpsys meminfo` 看 JNI 引用数
   - `jstack` 看 Thread 状态
   - heap dump 分析 String Table

→ **不理解 12 种 Root，就无法定位"对象为什么不被 GC 回收"**。
