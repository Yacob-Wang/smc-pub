## 一、5 个机制 × 3 个算法的对应关系

### 1.7.1 总对照表

| 机制 | 在 CMS 中的角色 | 在 CC GC 中的角色 | 在 GenCC 中的角色 |
|:---|:---|:---|:---|
| **可达性分析** | 12 种 GC Root 遍历 | 12 种 GC Root 遍历 | 12 种 GC Root + 跨代扫描 |
| **三色不变式** | 强三色不变式 | 弱三色不变式 | 弱三色不变式 |
| **写屏障** | Pre-Write Barrier（Incremental Update） | 不需要 | Post-Write Barrier（维护 Card Table） |
| **读屏障** | 不需要 | Read Barrier（核心创新） | Read Barrier |
| **记忆集 / 卡表** | 不需要 | 不需要 | Card Table（1 byte / 512 byte） |
| **Reference 体系** | ReferenceProcessor | ReferenceProcessor | ReferenceProcessor + ReferenceQueueDaemon |

### 1.7.2 ART GC 算法演进史

```
Android 版本        默认 GC            关键特性
─────────────────────────────────────────────────────
Android 5.0-7.0    CMS               标记-清除 + 写屏障，STW ~50ms
Android 8.0-9.0    CC                标记-复制 + 读屏障，STW < 1ms
Android 10.0+      GenCC             CC + 分代 + Card Table，Minor GC < 0.5ms
Android 14+        GenCC + rbcc      进一步优化读屏障开销
```

### 1.7.3 ART GC 算法的本质权衡

| 维度 | CMS | CC | GenCC |
|:---|:---|:---|:---|
| **STW 时间** | 长（~50ms） | 极短（< 1ms） | 极短（Minor < 0.5ms） |
| **运行时开销** | 低（写屏障） | 中（读屏障） | 中（读屏障 + Card Table） |
| **内存碎片** | 高（标记-清除） | 低（标记-复制） | 低（标记-复制 + Region） |
| **GC 频率** | 低 | 中 | 高（Minor GC）+ 低（Major GC） |
| **适用场景** | 内存敏感 + 性能要求不高 | 性能优先 + 内存不紧张 | 通用（Android 10+ 默认） |
| **复杂度** | 低 | 中 | 高 |

## 三、稳定性排查决策树

### 1.7.6 OOM 排查路径

```
OOM 发生
    │
    ▼
1. dumpsys meminfo 看 Native / Dalvik / Graphics 哪个暴涨？
    │
    ├─── Dalvik 暴涨
    │    │
    │    ▼
    │   1.1 是不是 Reference 体系泄漏？（软/弱/虚引用配置不当）
    │   1.2 是不是 JNI Global Ref 泄漏？（dumpsys meminfo 看 JNI）
    │   1.3 是不是 String.intern() 滥用？（heap dump 看 String Table）
    │   1.4 是不是 Thread 不退出？（jstack 看 Thread 数）
    │   1.5 是不是 WeakHashMap value 泄漏？（heap dump 看 Entry）
    │
    ├─── Native 暴涨
    │    │
    │    ▼
    │   2.1 是不是 DirectByteBuffer 泄漏？（dumpsys meminfo 看 .so mmap）
    │   2.2 是不是 Bitmap 泄漏？（RecycledBitmap 检测）
    │   2.3 是不是第三方 .so 泄漏？（procrank + smaps）
    │
    └─── Graphics 暴涨
         │
         ▼
        3.1 是不是 Surface 泄漏？
        3.2 是不是 GL Texture 泄漏？
        3.3 是不是 HWUI display list 泄漏？
```

### 1.7.7 GC 卡顿排查路径

```
GC 卡顿（jank）发生
    │
    ▼
1. Systrace / Perfetto 找 GC 事件
    │
    ▼
2. 看 STW 时间分布
    │
    ├─── Remark 阶段长（CMS）
    │    │
    │    ▼
    │   2.1 是否并发标记期间对象修改量大？
    │   2.2 是否 Incremental Update 标脏对象过多？
    │   2.3 升级到 CC GC 试试
    │
    ├─── Copy 阶段长（CC / GenCC）
    │    │
    │    ▼
    │   3.1 是否 Live Set 过大？
    │   3.2 是否读屏障开销过大？
    │   3.3 是否 Thread TLAB 竞争激烈？
    │
    └─── Minor GC 频率过高（GenCC）
         │
         ▼
        4.1 卡表是否过于频繁标脏？（看 Card Table 状态）
        4.2 是否大量对象跨 Region？
        4.3 是否长寿对象污染 Young Gen？
```

### 1.7.8 内存泄漏排查路径

```
内存泄漏发生
    │
    ▼
1. LeakCanary 检测
    │
    ├─── WeakReference.get() != null
    │    │
    │    ▼
    │   2. Heap Dump（LeakCanary 自动）
    │   3. 用 Shark 库分析
    │   4. 找出泄漏路径（GC Root → 泄漏对象）
    │   5. 常见泄漏源：
    │      - Activity Context 泄漏
    │      - Handler / Runnable 引用泄漏
    │      - 静态变量持有 Activity
    │      - Listener / Callback 未注销
    │      - Thread / TimerTask 未停止
    │
    └─── 业务层未使用 LeakCanary
         │
         ▼
        2. dumpsys meminfo 看 Dalvik Heap 是否持续增长
        3. heap dump 手动分析（MAT / Shark）
        4. 对比多个 dump 找出增长的对象
```

## 五、关键 takeaway

### 1.7.11 5 个最重要的洞察

1. **可达性分析 > 引用计数**：解决循环引用 + 降低开销
2. **三色不变式 = 并发 GC 正确性的数学根基**：只要维护住不变式，漏标不会发生
3. **写屏障 vs 读屏障的权衡**：写屏障轻但 STW 长，读屏障重但 STW 短
4. **卡表 = 分代 GC 的工程优化**：99% 的 Minor GC 性能提升
5. **Reference 体系 = Java 引用语义的对象回收时机**：LeakCanary / Cleaner / finalize() 的根

### 1.7.12 ART GC 演进的核心驱动力

**驱动力 1：用户体验（卡顿 → STW 优化）**
- CMS Remark ~50ms → CC < 1ms → GenCC Minor < 0.5ms

**驱动力 2：内存效率（碎片化 → 复制算法）**
- CMS 标记-清除（碎片化）→ CC / GenCC 标记-复制（无碎片）

**驱动力 3：内存增长（分代假说 → 分代 GC）**
- CC 扫描全堆 → GenCC Minor GC 只扫描 Young Gen

**驱动力 4：应用规模（更多对象 → 更复杂的 GC Root）**
- 12 种 GC Root 来源，每种都有专门的 Visit 函数
