# 5.7 写屏障在分代 GC 中的双重角色

> **本节回答一个根本问题**：GenCC 中的写屏障既要维护弱三色不变式（读屏障职责），又要维护 Card Table（分代 GC 职责），怎么协调？
>
> **答案**：**写屏障在 GenCC 中同时维护两件事** —— 通过 Post-Write Barrier 同时更新 Card Table 和对象头状态。
>
> **理解本节，就理解了 GenCC 中写屏障的多重职责**。

---

## 一、GenCC 写屏障的双重职责

### 5.7.1 CC GC 的写屏障（用于分代 GC 时）

```
GenCC 的 PostWriteBarrier 维护两件事：
  1. 不变式维护（与 CC GC 类似）
     - 通过对象头标记跟踪对象状态
     - 但不依赖写屏障（用读屏障）
  
  2. Card Table 维护（分代 GC 特有）
     - 跨代引用时标记 Card Table
     - 让 Minor GC 能找到跨代引用
```

### 5.7.2 GenCC 中不需要写屏障维护不变式

**关键洞察**：GenCC 用 **读屏障**维护弱三色不变式，**不需要写屏障**来防漏标。

```
GenCC 中：
  - 弱三色不变式 = 由读屏障维护（详见 04 篇 4.4）
  - 不需要写屏障做 Incremental Update
  - 但 GenCC 仍然需要写屏障维护 Card Table
```

### 5.7.3 GenCC 中写屏障的唯一职责：Card Table

```
Post-Write Barrier 的唯一职责：
  - 检查跨代引用
  - 标记 Card Table 中的 dirty card

→ 不维护三色不变式
→ 不重新染灰对象
→ 极简实现
```

## 二、Post-Write Barrier 的完整实现

### 5.7.4 PostWriteBarrier 的实现

```cpp
// art/runtime/gc/collector/concurrent_copying.h
void PostWriteBarrier(void* field_addr, mirror::Object* new_value) {
    // 1. 计算 field_addr 所在的 card
    uint8_t* card = CardTable::AddressToCard(field_addr);
    
    // 2. 检查是否跨代引用
    SpaceType src_space = RegionSpace::GetSpaceTypeOf(field_addr);
    SpaceType dst_space = RegionSpace::GetSpaceTypeOf(new_value);
    
    // 3. 跨代时把 card 标记为 dirty
    if (src_space != dst_space && 
        (src_space == kSpaceTypeYoung || dst_space == kSpaceTypeYoung)) {
        *card = CardTable::kCardDirty;
    }
}
```

### 5.7.5 编译码中的 PostWriteBarrier

```asm
; AArch64 上 PostWriteBarrier 的机器码
post_write_barrier_entry:
    ; 1. 计算 card 地址
    and x2, x0, #~(kCardSize - 1)    ; x2 = card_addr
    
    ; 2. 检查 new_value 是否在 Young Gen
    bl artIsInYoungGen
    cbz x0, .Lskip                    ; 如果不在 Young Gen，跳过
    
    ; 3. 把 card 标记为 dirty
    mov x3, #0x70
    strb w3, [x2]
    
.Lskip:
    ret
```

### 5.7.6 JIT 模式下的优化

```cpp
// 优化前（每次循环都触发）
for (int i = 0; i < N; i++) {
    PostWriteBarrier(&arr[i], value);  // 写屏障
    arr[i] = value;
}

// 优化后（循环外判断）
bool need_barrier = IsInYoungGen(value);  // 循环外判断
for (int i = 0; i < N; i++) {
    if (need_barrier) {
        PostWriteBarrier(&arr[i], value);
    }
    arr[i] = value;
}
```

## 三、PostWriteBarrier 与读屏障的协作

### 5.7.7 写屏障 vs 读屏障

| 维度 | 读屏障（Read Barrier） | 写屏障（Write Barrier） |
|:---|:---|:---|
| **触发时机** | 读对象引用时 | 指针赋值时 |
| **维护职责** | 弱三色不变式（对象移动） | Card Table（跨代引用） |
| **触发频率** | 读多（80-90%） | 写少（10-20%） |
| **开销** | 中（自愈后极低） | 中（每次跨代指针赋值） |
| **GenCC 中** | 必须 | 视情况 |

### 5.7.8 协作流程

```
业务线程：obj.field = new_value（跨代引用）

PostWriteBarrier 触发：
  1. 检查跨代引用 → 是
  2. 标记 Card Table 的 dirty card
  3. 返回

业务线程：value = obj.field

ReadBarrier 触发：
  1. 检查对象是否在 from-space → 否（在 to-space 或 Young Gen）
  2. 快速路径返回
```

## 四、PostWriteBarrier 的工程优化

### 5.7.9 优化 1：减少跨代引用

```java
// ✅ 好：长寿对象一次性创建
private static final Map<String, Object> cache = new HashMap<>();
// cache 在 Old Gen，value 也在 Old Gen → 无跨代引用

// ❌ 不好：缓存中放 Young Gen 对象
public void put(String key, Object value) {
    cache.put(key, value);  // value 在 Young Gen，cache 在 Old Gen → 跨代引用
}
```

### 5.7.10 优化 2：批量数据集中处理

```java
// ✅ 好：批量加载数据
public void loadData(List<Data> dataList) {
    // dataList 一次性加载到 Old Gen
    cache.putAll(dataList);
}

// ❌ 不好：逐条加载
public void loadData(List<Data> dataList) {
    for (Data data : dataList) {
        cache.put(data.getKey(), data);  // 每次都是跨代引用
    }
}
```

### 5.7.11 优化 3：使用 TLAB 优化

```java
// TLAB（Thread Local Allocation Buffer）让同线程分配的对象在同一 Region
// → 同 Region 对象引用不触发跨代检查（如果都在 Old Gen）
// → 减少 PostWriteBarrier 触发次数
```

## 五、PostWriteBarrier 的工程影响

### 5.7.12 PostWriteBarrier 的开销

**每次跨代引用**：~5ns（计算 card 地址 + 检查空间 + 标记 dirty）

**非跨代引用**：~1ns（仅检查，不标记）

**业务代码影响**：
- 高频跨代引用 → PostWriteBarrier 开销大
- 优化建议：减少跨代引用

### 5.7.13 PostWriteBarrier 的监控

```bash
# 1. 看 dirty card 数量
adb logcat -s "art" | grep "Card"
# 输出示例：
# art : Dirty card count: 12345 / 524288 (2.3%)

# 2. 看 PostWriteBarrier 触发次数
adb logcat -s "art" | grep "WriteBarrier"
```

### 5.7.14 PostWriteBarrier 异常的诊断

**问题 1：dirty card 比例过高（> 10%）**

```
根因：
  - 跨代引用频繁
  - 缓存实现不当

诊断：
  adb logcat -s "art" | grep "Card"
  # 输出: Dirty card count: 100000 / 524288 (19%)  ← 异常

修复：
  - 优化缓存实现
  - 减少跨代引用
```

**问题 2：PostWriteBarrier 触发过多**

```
根因：
  - 业务代码高频赋值
  - 静态集合持有 Young Gen 对象

诊断：
  adb logcat -s "art" | grep "WriteBarrier"
  # 输出: WriteBarrier called 100000 times/second  ← 异常

修复：
  - 减少静态集合
  - 优化业务代码
```

## 七、本节小结

1. **GenCC 中写屏障的唯一职责**：维护 Card Table
2. **不维护三色不变式**（由读屏障负责）
3. **每次跨代引用触发**：~5ns 开销
4. **优化方向**：减少跨代引用 + 优化缓存实现
5. **监控指标**：dirty card 比例 + WriteBarrier 触发次数

→ **理解 PostWriteBarrier 的双重角色，就理解了 GenCC 中写屏障的多重职责**。
