# 5.8 实战案例：分代假说失效的"长寿对象"场景

> **本节是 05 篇的"压轴案例"** —— 用多个真实案例，把分代假说的所有知识串起来。
>
> **理解本节，就理解了"为什么业务代码要配合分代 GC 设计"**。

---

## 一、案例背景

### 5.8.1 案例概述

分代假说在大多数 Android App 中成立，但有些场景会失效：

| 案例 | App | 问题 | 根因 |
|:---|:---|:---|:---|
| 1 | 缓存 App | Minor GC 频率极高 | 静态 HashMap 持有 Young Gen 对象 |
| 2 | 图片 App | Major GC 频率高 | 大量 Bitmap 进入 Old Gen |
| 3 | 数据 App | 晋升率过高 | 大量跨代引用 |
| 4 | 长会话 App | Old Gen 填满 | 单例持有大对象 |

---

## 二、案例 1：缓存 App Minor GC 频率极高

### 5.8.2 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某数据缓存 App |
| **Android 版本** | Android 12 (API 31) |
| **问题** | Minor GC 频率 > 50/分钟 |
| **现象** | App 流畅度差，电量消耗高 |

### 5.8.3 业务代码

```java
public class DataCache {
    // 问题代码：用 HashMap 强引用缓存
    private static final Map<String, Object> cache = new HashMap<>();
    
    public void put(String key, Object value) {
        cache.put(key, value);
    }
    
    public Object get(String key) {
        return cache.get(key);
    }
}
```

### 5.8.4 问题分析

```
1. cache 是 static final，类加载时创建 → cache 本身在 Old Gen
2. cache.put(key, value)：
   - value 可能是 Young Gen 新创建的对象
   - cache.put 把 value 引用到 Old Gen 的 cache 中
   - 触发 PostWriteBarrier，标记 Card Table dirty
3. cache.get(key)：
   - 业务线程访问 cache
   - 但 value 可能已被 GC 移动（CC GC）
   - 读屏障保护
4. 问题：
   - cache 持有所有 value 的强引用
   - 即使 value 应该被 Minor GC 回收，也被 cache 保护
   - value 被"晋升"到 Old Gen
   - Young Gen 中大部分对象被"强行"留在 Old Gen
   - → 分代假说失效：Young Gen 中"短命对象"变多
   - → Minor GC 频率异常
```

### 5.8.5 监控数据

```bash
$ adb logcat -s "art" | grep "GC"
art : Background concurrent copying GC freed 0(0B) AllocSpace objects
art : Background concurrent copying GC freed 1048576(13MB) LOS objects
art : kGcCauseForAlloc triggered minor GC  ← 频繁触发
art : kGcCauseForAlloc triggered minor GC
art : kGcCauseForAlloc triggered minor GC
...（每分钟 > 50 次）
```

### 5.8.6 修复方案

**方案 1：用 LRU 缓存**

```java
public class DataCache {
    // 修复：用 LruCache（自动淘汰 + Old Gen 友好）
    private final LruCache<String, Object> cache = new LruCache<>(100);
    
    public void put(String key, Object value) {
        cache.put(key, value);  // 自动淘汰
    }
    
    public Object get(String key) {
        return cache.get(key);
    }
}
```

**方案 2：用 WeakHashMap**

```java
public class DataCache {
    // 修复：用 WeakHashMap（不持有强引用）
    private static final Map<String, WeakReference<Object>> cache = 
        new WeakHashMap<>();
    
    public void put(String key, Object value) {
        cache.put(key, new WeakReference<>(value));
    }
    
    public Object get(String key) {
        WeakReference<Object> ref = cache.get(key);
        return ref != null ? ref.get() : null;
    }
}
```

**方案 3：用 ConcurrentHashMap + 手动管理**

```java
public class DataCache {
    // 修复：手动管理 + 软引用
    private final Map<String, SoftReference<Object>> cache = 
        new ConcurrentHashMap<>();
    
    public void put(String key, Object value) {
        cache.put(key, new SoftReference<>(value));
    }
    
    public Object get(String key) {
        SoftReference<Object> ref = cache.get(key);
        if (ref == null) return null;
        Object value = ref.get();
        if (value == null) cache.remove(key);
        return value;
    }
    
    public void clearExpired() {
        cache.entrySet().removeIf(e -> e.getValue().get() == null);
    }
}
```

### 5.8.7 修复效果

```
修复前：
- Minor GC 频率：> 50/分钟
- App 流畅度：差

修复后（用 LruCache）：
- Minor GC 频率：~5/分钟
- App 流畅度：优秀
```

## 三、案例 2：图片 App Major GC 频率高

### 5.8.8 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某图片编辑 App |
| **Android 版本** | Android 13 (API 33) |
| **问题** | Major GC 频率 > 5/小时 |
| **现象** | App 偶发卡顿（Major GC STW ~100ms） |

### 5.8.9 业务代码

```java
public class ImageCache {
    private static final Map<String, Bitmap> bitmapCache = new HashMap<>();
    
    public void loadImage(String url) {
        Bitmap bitmap = loadFromNetwork(url);
        bitmapCache.put(url, bitmap);  // 强引用 Bitmap
    }
    
    public Bitmap getImage(String url) {
        return bitmapCache.get(url);
    }
}
```

### 5.8.10 问题分析

```
1. bitmapCache 是 static final → 在 Old Gen
2. 加载图片：
   - 创建 Bitmap → Bitmap 分配到 LOS（≥ 12 KB）
   - bitmapCache.put → Bitmap 被引用到 Old Gen
3. 长期累积：
   - 100 张全屏 Bitmap × 8 MB = 800 MB
   - bitmapCache 占满 Old Gen + LOS
4. Old Gen 满 → 触发 Major GC
5. Major GC STW ~100ms → 用户感知卡顿
```

### 5.8.11 监控数据

```bash
$ adb logcat -s "art" | grep "Major"
art : kGcCauseBackground triggered concurrent major GC  ← 频繁触发
art : kGcCauseBackground triggered concurrent major GC
art : kGcCauseBackground triggered concurrent major GC
...（每小时 > 5 次）

$ adb shell dumpsys meminfo com.example.imageeditor
  Native Heap      56789    40000    16789      200    67890   102400    56789    45611
  Dalvik Heap      34567    30000     4567      100    40000    98304    45000    53304  ← LOS 占满
```

### 5.8.12 修复方案

**方案 1：用 Glide / Fresco（专业图片库）**

```java
// 用 Glide 自动管理 Bitmap
Glide.with(context)
    .load(url)
    .placeholder(R.drawable.placeholder)
    .into(imageView);

// Glide 自动：
// - LRU 缓存
// - Bitmap 复用（inBitmap）
// - 内存压力感知
```

**方案 2：用 LruCache + 软引用**

```java
public class ImageCache {
    private final LruCache<String, Bitmap> cache;
    
    public ImageCache() {
        // 计算可用内存的 1/8 作为缓存大小
        final int maxMemory = (int) (Runtime.getRuntime().maxMemory() / 1024);
        final int cacheSize = maxMemory / 8;
        
        cache = new LruCache<String, Bitmap>(cacheSize) {
            @Override
            protected int sizeOf(String key, Bitmap bitmap) {
                return bitmap.getByteCount() / 1024;  // KB
            }
            
            @Override
            protected void entryRemoved(boolean evicted, String key, 
                                       Bitmap oldValue, Bitmap newValue) {
                if (evicted && oldValue != null && !oldValue.isRecycled()) {
                    oldValue.recycle();  // 回收 Bitmap
                }
            }
        };
    }
}
```

**方案 3：用 SoftReference 自动管理**

```java
public class ImageCache {
    private final Map<String, SoftReference<Bitmap>> cache = 
        new ConcurrentHashMap<>();
    
    public void put(String url, Bitmap bitmap) {
        cache.put(url, new SoftReference<>(bitmap));
    }
    
    public Bitmap get(String url) {
        SoftReference<Bitmap> ref = cache.get(url);
        if (ref == null) return null;
        Bitmap bitmap = ref.get();
        if (bitmap == null) {
            cache.remove(url);
        }
        return bitmap;
    }
}
```

### 5.8.13 修复效果

```
修复前：
- Major GC 频率：> 5/小时
- 用户感知：偶发卡顿

修复后（用 Glide）：
- Major GC 频率：~1/小时
- 用户感知：几乎无卡顿
```

## 四、案例 3：数据 App 晋升率过高

### 5.8.14 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某数据分析 App |
| **Android 版本** | Android 12 (API 31) |
| **问题** | Young Gen 晋升率过高（> 50%） |
| **现象** | Old Gen 快速填满，频繁触发 Major GC |

### 5.8.15 业务代码

```java
public class DataAnalyzer {
    private final List<DataRecord> records = new ArrayList<>();
    // records 在 Old Gen
    
    public void processData(List<DataRecord> newRecords) {
        for (DataRecord record : newRecords) {
            // record 是新创建的，在 Young Gen
            processRecord(record);
            records.add(record);  // records 引用 record → 跨代引用
        }
    }
    
    private void processRecord(DataRecord record) {
        // 处理逻辑
    }
}
```

### 5.8.16 问题分析

```
1. records 在 Old Gen（长寿字段）
2. records.add(record)：
   - record 在 Young Gen
   - records（在 Old Gen）引用 record（在 Young Gen）
   - 触发 PostWriteBarrier，标记 Card Table dirty
3. 问题：
   - 每次 processData 都产生大量跨代引用
   - Card Table 频繁标脏
   - 每次 Minor GC 都要扫描大量 dirty card
   - Minor GC 效率下降
   - 长期累积：record 被晋升到 Old Gen
```

### 5.8.17 修复方案

**方案 1：批量处理**

```java
public void processData(List<DataRecord> newRecords) {
    // 1. 一次性处理所有 record
    List<DataRecord> processed = new ArrayList<>();
    for (DataRecord record : newRecords) {
        processRecord(record);
        processed.add(record);
    }
    
    // 2. 批量加入 Old Gen（一次性跨代引用）
    records.addAll(processed);
}
```

**方案 2：分阶段处理**

```java
public void processData(List<DataRecord> newRecords) {
    // 1. 在 Young Gen 中处理
    for (DataRecord record : newRecords) {
        processRecord(record);
    }
    
    // 2. 一次性把处理结果加入 Old Gen
    records.addAll(newRecords);
}
```

**方案 3：用 ThreadLocal 缓存中间结果**

```java
public void processData(List<DataRecord> newRecords) {
    // 1. 在 Young Gen 中处理
    List<DataRecord> processed = newRecords;
    for (DataRecord record : processed) {
        processRecord(record);
    }
    
    // 2. 一次批量加入 Old Gen
    records.addAll(processed);
}
```

### 5.8.18 修复效果

```
修复前：
- 跨代引用次数：~10 万/分钟
- Minor GC 扫描 dirty card：~5 万张
- Minor GC 耗时：~5ms（接近 Major GC）

修复后（批量处理）：
- 跨代引用次数：~1 万/分钟（10x 减少）
- Minor GC 扫描 dirty card：~1 千张
- Minor GC 耗时：~0.3ms
```

## 五、案例 4：长会话 App Old Gen 填满

### 5.8.19 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某长会话 App（社交） |
| **Android 版本** | Android 13 (API 33) |
| **问题** | 长会话 2 小时后 OOM |
| **现象** | App 偶发 OOM，无法释放内存 |

### 5.8.20 业务代码

```java
public class SessionManager {
    private static final SessionManager INSTANCE = new SessionManager();
    // INSTANCE 单例，在 Old Gen
    
    private final Map<String, Object> sessionData = new HashMap<>();
    // sessionData 在 Old Gen
    
    public void put(String key, Object value) {
        sessionData.put(key, value);  // value 也在 Old Gen
    }
}
```

### 5.8.21 问题分析

```
1. SessionManager.INSTANCE 单例 → Old Gen
2. sessionData 持有所有会话数据 → Old Gen
3. 长会话累积：
   - 用户聊天记录、状态、配置等
   - 大量大对象进入 Old Gen
4. 4 小时后：
   - Old Gen 接近 max_allowed_footprint
   - Major GC 也无法释放（数据还在使用）
   - OOM
```

### 5.8.22 修复方案

**方案 1：分层存储**

```java
public class SessionManager {
    // 热数据：在 Old Gen
    private final Map<String, Object> hotData = new HashMap<>();
    
    // 冷数据：可以清理
    private final LruCache<String, Object> coldData = new LruCache<>(100);
    
    public void put(String key, Object value, boolean hot) {
        if (hot) {
            hotData.put(key, value);
        } else {
            coldData.put(key, value);
        }
    }
}
```

**方案 2：定期清理**

```java
public class SessionManager {
    private final Map<String, Object> sessionData = new HashMap<>();
    private long lastClearTime;
    
    public void put(String key, Object value) {
        sessionData.put(key, value);
        
        // 每 30 分钟清理一次过期数据
        if (System.currentTimeMillis() - lastClearTime > 30 * 60 * 1000) {
            clearExpired();
            lastClearTime = System.currentTimeMillis();
        }
    }
    
    private void clearExpired() {
        sessionData.entrySet().removeIf(e -> isExpired(e.getKey()));
    }
}
```

**方案 3：监听系统低内存**

```java
public class SessionManager {
    private final Map<String, Object> sessionData = new HashMap<>();
    
    public void onTrimMemory(int level) {
        if (level >= TRIM_MEMORY_RUNNING_LOW) {
            // 系统内存紧张，清理不必要数据
            sessionData.clear();
        }
    }
}
```

### 5.8.23 修复效果

```
修复前：
- 长会话 2 小时后 OOM
- Old Gen 使用率：~95%

修复后（分层 + 定期清理）：
- 长会话 4 小时仍稳定
- Old Gen 使用率：~50%
```

## 六、案例总结

### 5.8.24 4 个案例的共同根因

| 案例 | 共同根因 |
|:---|:---|
| 1-4 | **业务代码与分代 GC 设计不匹配** |

### 5.8.25 解决方案的统一原则

| 原则 | 说明 |
|:---|:---|
| **静态集合慎用** | 用 LRU / SoftReference / WeakHashMap |
| **长寿对象集中管理** | 避免 Young Gen 中存长寿对象 |
| **批量处理** | 减少跨代引用频率 |
| **监听内存压力** | onTrimMemory / onLowMemory |
| **使用专业库** | Glide / Fresco / LruCache |

### 5.8.26 分代假说失效的检测指标

```bash
# 1. Minor GC 频率
adb logcat -s "art" | grep "minor GC" | wc -l  # 1 分钟内的次数

# 2. 晋升率
adb logcat -s "art" | grep "Promote" | wc -l  # 1 分钟内的晋升次数

# 3. dirty card 比例
adb logcat -s "art" | grep "Card"
```

### 5.8.27 通用排查流程

```
1. dumpsys meminfo 看 Young/Old Gen 使用率
   │
2. ART Trace 看 Minor GC / Major GC 频率和 STW 时间
   │
3. 检查静态集合
   │
4. 检查跨代引用
   │
5. 优化业务代码
```
