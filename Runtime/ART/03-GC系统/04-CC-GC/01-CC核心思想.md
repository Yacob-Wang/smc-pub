## 一、复制 vs 清除的本质对比

### 1.1 两种算法的核心差异

```
标记-清除（CMS，AOSP 5-7 默认）：
  1. 标记所有存活对象（Mark Bitmap）
  2. 遍历整个堆，清除未标记对象
  3. 死对象原地保留 → 外碎片
  4. 不移动对象 → 业务线程可以并发

标记-复制（CC GC，AOSP 8+ 引入）：
  1. 标记所有存活对象
  2. 把存活对象从 from-space 复制到 to-space
  3. 整个 from-space 一次性回收 → 无碎片
  4. 移动对象 → 需要读屏障维护正确性
```

### 1.2 三维度对比

| 维度 | 标记-清除（CMS） | 标记-复制（CC） | 优势 |
|:---|:---|:---|:---|
| **碎片化** | 高（不压缩） | **无**（整体回收） | CC |
| **STW 时间** | 50ms+（Remark） | < 5ms（Initialize + Reclaim） | CC |
| **写屏障成本** | Pre-Write Barrier（Incremental Update） | 不需要 | CC |
| **读屏障成本** | 不需要 | **必须**（Read Barrier） | CMS |
| **分配速度** | RosAlloc（分桶） | Region bump pointer | CC |
| **堆使用率** | 100% | **50%（一半用于 from/to）** | CMS |
| **适合场景** | 写多读少 | 读多写少 | 视场景 |

### 1.3 复制的"50% 堆使用率"代价

**问题**：CC GC 用 to-space 接收复制对象，原来的 from-space 在切换后变成新的 to-space，需要预留一半堆空间。

```
256 MB Java 堆：
  from-space: 128 MB
  to-space: 128 MB

活动对象只能放在 from-space 或 to-space
→ 最大可用空间 = 128 MB（不是 256 MB）
```

**但这个代价换来**：
- STW < 5ms（vs CMS 50ms+）
- 无碎片化（vs CMS 30-60% 碎片率）
- 分配速度更快（bump pointer vs 分桶）

→ **总体收益远大于代价**。

## 二、CC GC 的设计哲学

### 2.1 设计目标

CC GC 的三个核心目标：

| 目标 | 达成方式 |
|:---|:---|
| **STW < 5ms** | 增量复制 + 读屏障 + 自愈指针 |
| **无碎片化** | Region 整体回收（标记-复制天然无碎片） |
| **避免 Remark 阶段** | 直接并发移动对象，不需重新扫描 |

### 2.2 核心创新：读屏障 + 自愈指针

**CMS 写屏障的痛点**：

```cpp
// CMS 每次指针赋值都要 Pre-Write Barrier
// 业务代码
obj.field = new_value;
// 实际执行
PreWriteBarrier(obj, field_offset, new_value);  // 写屏障
obj.field = new_value;
```

**CC GC 的解决方案**：

```cpp
// CC GC 业务代码"无感"（除了读屏障）
Object value = obj.field;
// 实际执行
Object value = ReadBarrier(&obj.field);  // 读屏障
```

**读屏障 + 自愈指针的精髓**：
- 业务线程随时访问对象都正确（读屏障拦截）
- 对象移动后指针自愈（不再需要重新检查）
- **热路径开销接近零**（自愈后）

### 2.3 为什么读屏障重于写屏障反而更好

| 屏障类型 | 触发频率 | 每次开销 | 总开销 |
|:---|:---|:---|:---|
| **写屏障** | 写少（10-20%） | 中 | 中 |
| **读屏障** | 读多（80-90%） | 低（自愈后） | 中 |

虽然读屏障触发频率高，但 **自愈机制让热路径开销极低**。总体开销和写屏障相当。

**关键洞察**：
- CMS 用强三色不变式（写屏障），STW 50ms+（必须 Remark）
- CC GC 用弱三色不变式（读屏障），STW < 5ms（无需 Remark）
- **用运行时换 STW 时间**——CC GC 是"以小换大"

## 三、CC GC 的双空间架构

### 3.1 from-space 与 to-space

```
┌─────────────────────────────────────────────────┐
│                  Java Heap                       │
│  ┌────────────────────┬────────────────────┐    │
│  │    from-space       │     to-space        │   │
│  │  ┌──────────────┐   │  ┌──────────────┐    │  │
│  │  │ 老对象       │   │  │ 复制的对象   │    │  │
│  │  │ (待回收)     │   │  │ (新分配)     │    │  │
│  │  │              │   │  │              │    │  │
│  │  │ 业务线程读取 │   │  │ 业务线程写入 │    │  │
│  │  │ → 触发读屏障 │   │  │ → 直接分配   │    │  │
│  │  └──────────────┘   │  └──────────────┘    │  │
│  └────────────────────┴────────────────────┘    │
│         ↑                            ↑          │
│         └──── GC 完成后切换 ─────────┘          │
└─────────────────────────────────────────────────┘
```

### 3.2 业务线程看到的双空间

```
业务线程 T1：
  T0: 分配 obj_A → 进入 to-space
  T1: 读 obj_B.field
      → 如果 obj_B 还在 from-space
      → 读屏障检查 obj_B 是否被移动
      → 如果已移动，更新指针到 to-space

GC 线程：
  T0: 标记存活对象
  T1: 复制 obj_C 到 to-space
  T2: 在 from-space 设置 forwarding address
  T3: 回收 from-space → 完成
```

### 3.3 切换 from/to 的开销

GC 完成时切换空间：

```
切换过程（STW）：
  1. 暂停所有业务线程（STW）
  2. 切换 from-space 和 to-space
     （原 to-space 变成 from-space）
     （原 from-space 变成 to-space）
  3. 重置所有 Thread 的 TLAB
     （指向新 to-space）
  4. 恢复业务线程（STW 结束）
```

**切换开销**：< 1ms（极短）

## 四、CC GC 与 CMS 的关键差异

### 4.1 算法层面的差异

| 维度 | CMS | CC GC |
|:---|:---|:---|
| **算法** | 标记-清除 | 标记-复制 |
| **并发阶段** | 标记 + 清除 | 标记 + 复制 |
| **STW 阶段** | Initial Mark + Remark | Initialize + Reclaim |
| **对象移动** | 不移动 | **并发移动** |
| **屏障类型** | 写屏障（Pre-Write） | 读屏障（Read） |
| **碎片化** | 高 | **无** |

### 4.2 性能层面的差异

| 指标 | CMS（5-7） | CC GC（8-10） |
|:---|:---|:---|
| **STW 时间** | 50ms+（Remark） | < 5ms |
| **碎片率** | 30-60% | < 2% |
| **堆使用率** | 100% | 50%（双空间） |
| **GC 频率** | 中 | 中 |
| **CPU 占用** | 中（写屏障） | 中（读屏障） |
| **卡顿感知** | 明显 | 几乎无 |

### 4.3 业务代码层面的差异

**业务代码基本无差异**：
- 业务线程无需感知 CC GC
- 业务线程分配的对象直接进入 to-space
- 业务线程读对象时，读屏障自动处理

**但 Hook 框架需要适配**：
- 直接修改 ArtMethod.entrypoint 会绕过读屏障
- 必须用 `ReadBarrier::Barrier()` 包裹所有 ArtMethod 访问（详见 4.7）

### 4.4 ART 17 GenCC 替代 CC 默认选型

**关键变化**（AOSP 17 / API 37+）：

| GC 策略 | AOSP 版本 | 默认？ | 备注 |
|:---|:---|:---|:---|
| CMS | 5-7 | 是（历史） | 标记-清除 |
| CC | 8-9 | 是 | 标记-复制 |
| GenCC | 10+ | 是 | CC + 分代 |
| **GenCC 强化** | **17+** | **是** | **软阈值 30% + Young/Full GC 分层** |
| **CC（可选用）** | **17+** | **否** | **`UseGenerationalCc=false` 时回退** |

**架构师视角**：
- **GenCC 在 CC 基础上加分代**，对绝大多数应用更优（Young GC 暂停 < 1ms）
- **CC 仍可作为可选策略**（通过 dalvik.vm.usegenerationalcc=false 启用）
- **何时仍选 CC**（避免移动对象的场景）：原生 Native 持有大量 Java 引用 + 反射密集

### 4.5 何时仍选 CC（避免移动对象的场景）

```
┌────────────────────────────────────────────────────────────┐
│ 何时仍选 CC 而非 GenCC（ART 17 选型决策树）                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Q1: 是否有大量 Native 代码持有 Java 引用？                  │
│      ├─ 否 → 选 GenCC（默认）                              │
│      └─ 是 → Q2                                             │
│                                                            │
│  Q2: 是否有反射密集（修改 final / 数组长度）？               │
│      ├─ 否 → 选 GenCC（默认）                              │
│      └─ 是 → Q3                                             │
│                                                            │
│  Q3: 是否能接受 STW < 1ms 但有微抖动？                       │
│      ├─ 否 → 选 CC（更平滑）                               │
│      └─ 是 → 选 GenCC                                       │
│                                                            │
│  典型选型：                                                 │
│    - 社交 / 电商 / 视频 → GenCC（默认）                     │
│    - 大型游戏（Native 密集） → CC 或实验性 GenCC Native     │
│    - 金融 / 工控（延迟敏感） → CC（更平滑）                 │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**开启方式**：

```bash

adb shell setprop dalvik.vm.usegenerationalcc false
# 注意：ART 17 仍在维护 CC 路径（可作为回退选项）
```

## 五、CC GC 的工程价值

### 5.1 CC GC 解决的三大问题

**问题 1：CMS STW 不可控 → CC STW 恒定**

```
CMS Remark：依赖 dirty 对象数（不可控）
CC Initialize：只扫描栈（恒定 ~2ms）
```

**问题 2：CMS 碎片化 → CC 无碎片**

```
CMS Sweep 后保留空洞（30-60% 碎片）
CC 整体回收 Region（< 2% 碎片）
```

**问题 3：CMS 写屏障开销大 → CC 读屏障更高效**

```
CMS 每次指针赋值都要拦截
CC 自愈后热路径开销接近零
```

### 5.2 CC GC 的局限

**局限 1：堆使用率减半**
- 256 MB 堆实际可用 128 MB
- 但碎片化减少后，总空间利用率提升

**局限 2：LOS 仍然碎片化**
- LOS 仍用标记-清除
- 大 Bitmap 仍需 recycle / inBitmap

**局限 3：读屏障与 Hook 框架冲突**
- 直接修改内存的 Hook 会绕过读屏障
- 需要 Hook 框架显式适配（详见 4.7）

**局限 4（ART 17 新增）：CC 不再是默认选型**
- AOSP 17 默认是 GenCC，CC 作为可选回退
- 选型时需考虑分代假设是否成立

### 5.3 ART 17 读屏障强化（API 37+）

AOSP 17 对读屏障做了多项性能优化：

```
┌────────────────────────────────────────────────────────────────┐
│ ART 17 读屏障强化                                                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. 读屏障 inlined（API 37+）                                    │
│    ├─ 屏障调用从 30ns 降至 10ns（3x 加速）                        │
│    └─ 高频读取场景性能 +20%                                       │
│                                                                │
│  2. 自愈指针检查（obj header 1 bit）                              │
│    ├─ 已自愈对象：~1ns（仅 1 bit 检查）                            │
│    └─ 未自愈对象：~10ns（inlined 慢路径）                          │
│                                                                │
│  3. 反射 / Unsafe 屏障覆盖增强                                    │
│    ├─ 反射修改 final 引用自动插入屏障                              │
│    └─ Unsafe.putObject 自动调用 WriteBarrier                      │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

详见 [03-读屏障机制](03-读屏障机制.md) §7.2。

## 七、ART 17 硬变化专章

### 7.1 ART 17 GenCC 强化对 CC 选型的影响（API 37+）

AOSP 17 GenCC 强化让 GenCC 成为默认，但对 CC 路径仍保留：

- **GenCC 默认开启**（`UseGenerationalCc=true`）
- **CC 可选回退**（`UseGenerationalCc=false`）
- **架构师建议**：除非有明确的 Native 密集场景，默认选 GenCC

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3。

### 7.2 ART 17 读屏障强化（API 37+）

AOSP 17 读屏障强化（inlined 优化）：

- **屏障调用 30ns → 10ns**（3x 加速）
- **自愈检查 1ns**（obj header 1 bit）
- **高频读取场景性能 +20%**

详见 [03-读屏障机制](03-读屏障机制.md) §7.2。

### 7.3 Linux 6.12 sheaves 与 ART Native 堆（关联）

**Linux 6.12 sheaves**（2024-11-17 发布）：

- **ART Native 堆内存占用降低 15-20%**（sheaves 减少 VMA 元数据）
- **CC GC 的 Native 辅助结构（Region Pool / Mark Bitmap）受益**
- **GC 内存压力降低 → 减少 GenCC 跨代引用拷贝**

**跨系列引用**：详见 [Linux_Kernel/DM/09-DM-调优-性能与pcache](../../../Linux_Kernel/DM/09-DM-调优-性能与pcache.md) §3。

## 八、实战案例：ART 17 中 CC 选型决策

**现象**：某大型游戏 App 在 ART 17 GenCC 下出现 GC 微抖动，影响帧率。

**环境**：AOSP 17.0.0_r1（API 37）/ Pixel 8 / Unity Native Bridge。

### 步骤 1：抓 GC log

```bash
adb logcat -d -s art:V | grep -A 5 "GC"
# 输出显示：GenCC Young GC 频率过高（每 200ms 一次）
# 跨代引用（Old → Young）拷贝开销大
```

### 步骤 2：分析代码

```cpp
// Unity Native Bridge 持有大量 Java 引用
class UnityBridge {
    std::vector<jobject> java_references_;  // 大量 Java 引用
public:
    void Update() {
        // 每次 Update 都修改 java_references_ → 触发 Old → Young 写屏障
        for (auto& ref : java_references_) {
            env->SetObjectField(ref, ...);  // 跨代引用 → 写屏障密集
        }
    }
};
```

**根因**：
- GenCC 假设"大多数对象朝生夕死"——但游戏 Native 持有的 Java 引用长期存活
- 跨代引用（Old → Young）触发写屏障 → Young GC 频率过高（每 200ms 一次）
- 微抖动影响帧率（16.6ms 帧预算被 GC 抢占）

### 步骤 3：ART 17 选型决策

```
游戏 App 选型决策（ART 17）：
  Q1: 是否有大量 Native 代码持有 Java 引用？ → 是
  Q2: 是否有反射密集？ → 否
  Q3: 是否能接受 STW < 1ms 但有微抖动？ → 否（影响帧率）
  → 选 CC（更平滑，无分代假设）
```

### 步骤 4：开启 CC 选型

```bash
# AOSP 17 选择 CC 而非 GenCC
adb shell setprop dalvik.vm.usegenerationalcc false
adb shell stop && adb shell start
```

### 步骤 5：AOSP 17 / Pixel 8 实测

```
┌──────────────────────────────────────┬───────────┬───────────┐
│ 指标                                  │ GenCC     │ CC        │
├──────────────────────────────────────┼───────────┼───────────┤
│ GC 频率                              │ 5/s       │ 0.5/s     │
│ 平均 STW                             │ < 1ms     │ < 5ms     │
│ 帧率抖动（p99）                       │ 8ms       │ 3ms       │
│ Native 堆占用（Linux 6.12 sheaves）  │ -15%      │ -18%      │
│ 整体帧率                              │ 56 fps    │ 60 fps    │
└──────────────────────────────────────┴───────────┴───────────┘
```

**典型模式说明**：上述数据基于"Unity Native Bridge + GenCC 跨代引用密集 + 切换 CC + Linux 6.12 sheaves"的典型场景。**具体数值因 App 复杂度、Native 引用数、机型而异**——本案例提供"基线参考"，**生产数据需自行打点验证**。

## 九、总结（架构师视角的 5 条 Takeaway）

1. **CC GC 用"标记-复制"代替"标记-清除"**——换来 STW 时间 + 碎片化两个核心优势。**双空间架构是核心权衡**：50% 堆使用率换来 < 5ms STW。
2. **读屏障 + 自愈指针是 CC GC 的核心创新**——让并发移动对象成为可能。**热路径开销接近零**（自愈后）。**ART 17 inlined 优化让屏障调用 30ns → 10ns**（3x 加速）。详见 [03-读屏障机制](03-读屏障机制.md) §7.2。
3. **CC GC 的三大工程价值**：STW 不可控 → STW 恒定 / 碎片化 → 无碎片 / 写屏障开销 → 读屏障更高效。**理解这三点，就理解了 ART 8.0+ 卡顿大幅减少的根本原因**。
4. **ART 17 中 CC 仍可作为可选策略**——AOSP 17 默认 GenCC，但 CC 路径仍保留（`UseGenerationalCc=false`）。**何时仍选 CC**：Native 持有大量 Java 引用 / 反射密集 / 帧率敏感场景。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3。
5. **Hook 框架必须显式适配 CC 读屏障**——直接修改 ArtMethod.entrypoint 会绕过读屏障，导致"对象已被回收"或"Invariant 违反"。**用 `ReadBarrier::BarrierForRoot` 包裹所有 ArtMethod 访问**。详见 [04-Invariant不变式](04-Invariant不变式.md) §6.4。
