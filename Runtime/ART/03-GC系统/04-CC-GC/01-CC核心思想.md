# 4.1 CC GC 的核心思想：复制 vs 清除

> **本节回答一个根本问题**：CC GC 为什么用"标记-复制"而不是"标记-清除"？复制的本质优势是什么？
>
> **答案**：复制在并发性 + 碎片化 + STW 三个维度都优于清除。
>
> **理解本节，就理解了 CC GC 的设计哲学** —— 是 ART 8.0+ 卡顿大幅减少的根本原因。

---

## 一、复制 vs 清除的本质对比

### 4.1.1 两种算法的核心差异

```
标记-清除（CMS）：
  1. 标记所有存活对象（Mark Bitmap）
  2. 遍历整个堆，清除未标记对象
  3. 死对象原地保留 → 外碎片
  4. 不移动对象 → 业务线程可以并发

标记-复制（CC GC）：
  1. 标记所有存活对象
  2. 把存活对象从 from-space 复制到 to-space
  3. 整个 from-space 一次性回收 → 无碎片
  4. 移动对象 → 需要读屏障维护正确性
```

### 4.1.2 三维度对比

| 维度 | 标记-清除（CMS） | 标记-复制（CC） | 优势 |
|:---|:---|:---|:---|
| **碎片化** | 高（不压缩） | **无**（整体回收） | CC |
| **STW 时间** | 50ms+（Remark） | < 5ms（Initialize + Reclaim） | CC |
| **写屏障成本** | Pre-Write Barrier（Incremental Update） | 不需要 | CC |
| **读屏障成本** | 不需要 | **必须**（Read Barrier） | CMS |
| **分配速度** | RosAlloc（分桶） | Region bump pointer | CC |
| **堆使用率** | 100% | **50%（一半用于 from/to）** | CMS |
| **适合场景** | 写多读少 | 读多写少 | 视场景 |

### 4.1.3 复制的"50% 堆使用率"代价

**问题**：CC GC 用 to-space 接收复制对象，原来的 from-space 在切换后变成新的 to-space，需要预留一半堆空间。

```
256 MB Java 堆：
  from-space: 128 MB
  to-space: 128 MB

活动对象只能放在 from-space 或 to-space
→ 最大可用空间 = 128 MB（不是 256 MB）
```

**但这个代价换来**：
- STW < 5ms（vs CMS 50ms+）
- 无碎片化（vs CMS 30-60% 碎片率）
- 分配速度更快（bump pointer vs 分桶）

→ **总体收益远大于代价**。

## 二、CC GC 的设计哲学

### 4.1.4 设计目标

CC GC 的三个核心目标：

| 目标 | 达成方式 |
|:---|:---|
| **STW < 5ms** | 增量复制 + 读屏障 + 自愈指针 |
| **无碎片化** | Region 整体回收（标记-复制天然无碎片） |
| **避免 Remark 阶段** | 直接并发移动对象，不需重新扫描 |

### 4.1.5 核心创新：读屏障 + 自愈指针

**CMS 写屏障的痛点**：
```cpp
// CMS 每次指针赋值都要 Pre-Write Barrier
// 业务代码
obj.field = new_value;
// 实际执行
PreWriteBarrier(obj, field_offset, new_value);  // 写屏障
obj.field = new_value;
```

**CC GC 的解决方案**：
```cpp
// CC GC 业务代码"无感"（除了读屏障）
Object value = obj.field;
// 实际执行
Object value = ReadBarrier(&obj.field);  // 读屏障
```

**读屏障 + 自愈指针的精髓**：
- 业务线程随时访问对象都正确（读屏障拦截）
- 对象移动后指针自愈（不再需要重新检查）
- **热路径开销接近零**（自愈后）

### 4.1.6 为什么读屏障重于写屏障反而更好

| 屏障类型 | 触发频率 | 每次开销 | 总开销 |
|:---|:---|:---|:---|
| **写屏障** | 写少（10-20%） | 中 | 中 |
| **读屏障** | 读多（80-90%） | 低（自愈后） | 中 |

虽然读屏障触发频率高，但 **自愈机制让热路径开销极低**。总体开销和写屏障相当。

**关键洞察**：
- CMS 用强三色不变式（写屏障），STW 50ms+（必须 Remark）
- CC GC 用弱三色不变式（读屏障），STW < 5ms（无需 Remark）
- **用运行时换 STW 时间**——CC GC 是"以小换大"

## 三、CC GC 的双空间架构

### 4.1.7 from-space 与 to-space

```
┌─────────────────────────────────────────────────┐
│                  Java Heap                       │
│  ┌────────────────────┬────────────────────┐    │
│  │    from-space       │     to-space        │   │
│  │  ┌──────────────┐   │  ┌──────────────┐    │  │
│  │  │ 老对象       │   │  │ 复制的对象   │    │  │
│  │  │ (待回收)     │   │  │ (新分配)     │    │  │
│  │  │              │   │  │              │    │  │
│  │  │ 业务线程读取 │   │  │ 业务线程写入 │    │  │
│  │  │ → 触发读屏障 │   │  │ → 直接分配   │    │  │
│  │  └──────────────┘   │  └──────────────┘    │  │
│  └────────────────────┴────────────────────┘    │
│         ↑                            ↑          │
│         └──── GC 完成后切换 ─────────┘          │
└─────────────────────────────────────────────────┘
```

### 4.1.8 业务线程看到的双空间

```
业务线程 T1：
  T0: 分配 obj_A → 进入 to-space
  T1: 读 obj_B.field
      → 如果 obj_B 还在 from-space
      → 读屏障检查 obj_B 是否被移动
      → 如果已移动，更新指针到 to-space

GC 线程：
  T0: 标记存活对象
  T1: 复制 obj_C 到 to-space
  T2: 在 from-space 设置 forwarding address
  T3: 回收 from-space → 完成
```

### 4.1.9 切换 from/to 的开销

GC 完成时切换空间：

```
切换过程（STW）：
  1. 暂停所有业务线程（STW）
  2. 切换 from-space 和 to-space
     （原 to-space 变成 from-space）
     （原 from-space 变成 to-space）
  3. 重置所有 Thread 的 TLAB
     （指向新 to-space）
  4. 恢复业务线程（STW 结束）
```

**切换开销**：< 1ms（极短）

## 四、CC GC 与 CMS 的关键差异

### 4.1.10 算法层面的差异

| 维度 | CMS | CC GC |
|:---|:---|:---|
| **算法** | 标记-清除 | 标记-复制 |
| **并发阶段** | 标记 + 清除 | 标记 + 复制 |
| **STW 阶段** | Initial Mark + Remark | Initialize + Reclaim |
| **对象移动** | 不移动 | **并发移动** |
| **屏障类型** | 写屏障（Pre-Write） | 读屏障（Read） |
| **碎片化** | 高 | **无** |

### 4.1.11 性能层面的差异

| 指标 | CMS（5-7） | CC GC（8-10） |
|:---|:---|:---|
| **STW 时间** | 50ms+（Remark） | < 5ms |
| **碎片率** | 30-60% | < 2% |
| **堆使用率** | 100% | 50%（双空间） |
| **GC 频率** | 中 | 中 |
| **CPU 占用** | 中（写屏障） | 中（读屏障） |
| **卡顿感知** | 明显 | 几乎无 |

### 4.1.12 业务代码层面的差异

**业务代码基本无差异**：
- 业务线程无需感知 CC GC
- 业务线程分配的对象直接进入 to-space
- 业务线程读对象时，读屏障自动处理

**但 Hook 框架需要适配**：
- 直接修改 ArtMethod.entrypoint 会绕过读屏障
- 必须用 `ReadBarrier::Barrier()` 包裹所有 ArtMethod 访问（详见 4.7）

## 五、CC GC 的工程价值

### 4.1.13 CC GC 解决的三大问题

**问题 1：CMS STW 不可控 → CC STW 恒定**
```
CMS Remark：依赖 dirty 对象数（不可控）
CC Initialize：只扫描栈（恒定 ~2ms）
```

**问题 2：CMS 碎片化 → CC 无碎片**
```
CMS Sweep 后保留空洞（30-60% 碎片）
CC 整体回收 Region（< 2% 碎片）
```

**问题 3：CMS 写屏障开销大 → CC 读屏障更高效**
```
CMS 每次指针赋值都要拦截
CC 自愈后热路径开销接近零
```

### 4.1.14 CC GC 的局限

**局限 1：堆使用率减半**
- 256 MB 堆实际可用 128 MB
- 但碎片化减少后，总空间利用率提升

**局限 2：LOS 仍然碎片化**
- LOS 仍用标记-清除
- 大 Bitmap 仍需 recycle / inBitmap

**局限 3：读屏障与 Hook 框架冲突**
- 直接修改内存的 Hook 会绕过读屏障
- 需要 Hook 框架显式适配（详见 4.7）

## 七、本节小结

1. **CC GC 用标记-复制代替标记-清除**：换来 STW 时间 + 碎片化两个核心优势
2. **读屏障 + 自愈指针**：CC GC 的核心创新，让并发移动对象成为可能
3. **双空间架构**：堆使用率减半，但总体收益更大
4. **三大问题解决**：STW 不可控 / 碎片化 / 写屏障开销
5. **Hook 框架需要适配**：直接修改内存会绕过读屏障

→ **理解 CC GC 的核心思想，就理解了 ART 8.0+ 卡顿大幅减少的根本原因**。
