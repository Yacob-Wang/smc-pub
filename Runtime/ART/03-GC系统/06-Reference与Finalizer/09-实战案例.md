## 二、案例 1：Cursor finalize() 链式阻塞

### 2.1 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某数据库 App |
| **Android 版本** | Android 12 (API 31) → AOSP 17 (API 37) |
| **问题** | 运行 30 分钟后 OOM 崩溃 |
| **崩溃信息** | `OutOfMemoryError: Failed to allocate` |

### 2.2 业务代码

```java
public class DatabaseHelper {
    private SQLiteDatabase database;
    
    public List<Data> queryData() {
        Cursor cursor = database.rawQuery("SELECT * FROM data", null);
        // 忘记 close cursor
        List<Data> result = parseCursor(cursor);
        return result;
    }
}
```

### 2.3 问题分析

```
1. 每次 queryData() 都创建 Cursor
2. Cursor 在 finalize() 中关闭
3. 如果 Cursor 有未完成的查询 → finalize() 阻塞
4. 多个 Cursor 阻塞 → finalize() 队列堆积
5. FinalizerWatchdogDaemon 警告
6. 队列中的 Cursor 占内存 → OOM
```

### 2.4 崩溃过程

```bash
# logcat 关键日志
FATAL EXCEPTION: main
java.lang.OutOfMemoryError: Failed to allocate a 24 byte allocation
  at android.database.CursorWindow.nativeGetCursorPosition(Native Method)
  at android.database.CursorWindow.getPosition(CursorWindow.java:519)
  at android.database.AbstractCursor.getPosition(AbstractCursor.java:42)
  at ...

W art : Finalizer watch dog timed out: 15000ms, count=1
W art : Finalizer watch dog timed out: 30000ms, count=1
W art : Finalizer watch dog timed out: 45000ms, count=1
```

### 2.5 修复方案

**方案 1：try-with-resources 关闭 Cursor（推荐）**

```java
public List<Data> queryData() {
    try (Cursor cursor = database.rawQuery("SELECT * FROM data", null)) {
        return parseCursor(cursor);
    }
    // close() 自动调用，不会进入 finalize
}
```

**方案 2：手动关闭 Cursor**

```java
public List<Data> queryData() {
    Cursor cursor = null;
    try {
        cursor = database.rawQuery("SELECT * FROM data", null);
        return parseCursor(cursor);
    } finally {
        if (cursor != null) {
            cursor.close();  // 显式关闭
        }
    }
}
```

**方案 3：升级 AOSP 17（自动收益）**

无需改代码，Finalizer 4 线程池化 + 慢对象提前标记，**风险大幅降低**。

### 2.6 修复效果

| 指标 | 修复前 | 修复后（AOSP 14） | 升级 AOSP 17 | + try-with-resources |
|:---|:---|:---|:---|:---|
| Cursor 数量 | ~5000 | 0 | ~5000 | 0 |
| Finalizer 队列 | ~500 | 0 | ~60 | 0 |
| Watchdog 警告 / h | ~50 | 0 | ~50 | 0 |
| OOM 崩溃率 | ~80% | 0% | ~5% | 0% |

**典型模式说明**：上述数据基于"数据库查询 + Cursor 关闭遗漏 + 修复为 try-with-resources"的典型场景。**生产数据需自行打点验证**。

## 三、案例 2：Bitmap finalize() 阻塞

### 3.1 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某图片编辑器 |
| **Android 版本** | Android 13 (API 33) → AOSP 17 (API 37) |
| **问题** | native 内存泄漏，最终 OOM |
| **现象** | dumpsys meminfo 显示 native 内存持续增长 |

### 3.2 业务代码

```java
public class ImageEditor {
    private List<Bitmap> undoStack = new ArrayList<>();
    
    public void addUndo(Bitmap bitmap) {
        undoStack.add(bitmap);
        // 忘记 recycle
    }
    
    public void undo() {
        if (!undoStack.isEmpty()) {
            Bitmap bitmap = undoStack.remove(undoStack.size() - 1);
            // 忘记 recycle
            applyBitmap(bitmap);
        }
    }
}
```

### 3.3 问题分析

```
1. Bitmap 在 finalize() 中释放 native 内存
2. 业务代码不 recycle → 等 GC finalize 释放
3. 大量 Bitmap 堆积 → finalize 队列长
4. 每个 Bitmap finalize 释放 native 内存
5. 但 finalize 处理速度 < Bitmap 创建速度
6. → native 内存持续增长 → OOM
```

### 3.4 监控数据

```bash
$ adb shell dumpsys meminfo com.example.imageeditor

# 关键数据
  Native Heap      123456   100000    23456      500  150000   204800   123456    81344
  Dalvik Heap      45678    40000     5678      100    51234    98304    45678    52626
  Graphics         23456    20000     3456      100    28000
  TOTAL            234567   200000    34567      600  280000

# Native Heap Alloc = 123 MB（异常高）
# Graphics = 23 MB（Bitmap native 占用）
```

### 3.5 修复方案

**方案 1：及时 recycle() Bitmap**

```java
public class ImageEditor {
    private List<Bitmap> undoStack = new ArrayList<>();
    
    public void undo() {
        if (!undoStack.isEmpty()) {
            Bitmap bitmap = undoStack.remove(undoStack.size() - 1);
            try {
                applyBitmap(bitmap);
            } finally {
                if (bitmap != null && !bitmap.isRecycled()) {
                    bitmap.recycle();  // 立即释放 native
                }
            }
        }
    }
}
```

**方案 2：用 Glide / Fresco 自动管理**

```java
// Glide 自动 recycle Bitmap
Glide.with(context)
    .load(url)
    .into(imageView);

// Glide 内部使用 BitmapPool + LRU
```

**方案 3：用 inBitmap 复用**

```java
public Bitmap decodeBitmap(String path) {
    BitmapFactory.Options options = new BitmapFactory.Options();
    options.inBitmap = reusableBitmap;  // 复用 native 内存
    options.inMutable = true;
    return BitmapFactory.decodeFile(path, options);
}
```

### 3.6 修复效果

| 指标 | 修复前 | 修复后（AOSP 14） | 升级 AOSP 17 | + Glide 管理 |
|:---|:---|:---|:---|:---|
| Native 内存 | ~200 MB | ~50 MB | ~150 MB | ~30 MB |
| Bitmap 数量 | ~500 | ~30 | ~400 | ~30 |
| finalize() 队列 | ~200 | ~10 | ~50 | 0 |
| OOM 次数 / 周 | 5 | 0 | 1 | 0 |

**典型模式说明**：上述数据基于"图片编辑器 + Bitmap 不 recycle + 修复为 Glide 自动管理"的典型场景。**生产数据需自行打点验证**。

## 四、案例 3：Theme finalize() 卡死

### 4.1 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 系统服务 |
| **Android 版本** | Android 12 (API 31) → AOSP 17 (API 37) |
| **问题** | FinalizerWatchdogDaemon 警告频发 |
| **现象** | `Finalizer watch dog timed out: 15000ms` |

### 4.2 业务代码

```java
public class Theme {
    private long nativeThemeHandle;
    
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        // native 释放
        if (nativeThemeHandle != 0) {
            nativeDestroy(nativeThemeHandle);
            nativeThemeHandle = 0;
        }
    }
}
```

### 4.3 问题分析

```
1. Theme 在 finalize() 中释放 native 资源
2. 如果 native 资源被占用 → finalize() 阻塞
3. 例如：Theme 的 native 资源还在 GPU 渲染中
4. GPU 渲染线程持有 native 资源
5. finalize() 等待 GPU 释放 → 阻塞
6. 多个 Theme finalize() 排队 → 系统卡顿
```

### 4.4 AOSP 14 监控数据

```bash
$ adb logcat -s "art" | grep "Finalizer"
W art : Finalizer watch dog timed out: 15000ms, count=1
W art : Finalizer watch dog timed out: 30000ms, count=1
# 仅能看到超时警告，无法定位慢对象
```

### 4.5 AOSP 17 慢对象堆栈

```bash
$ adb logcat -s "art" | grep "Slow finalizeable"
W art: Slow finalizer detected: 5234ms, count=1
W art: Slow finalizeable object: com.example.Theme
W art:   at com.example.Theme.finalize(Theme.java:42)
W art:   at com.example.Theme.nativeDestroy(Native Method)
W art:   at android.opengl.GLES20.glDeleteShader(GLES20.java:1234)
# 完整堆栈 + 类名 + 阻塞位置
```

### 4.6 修复方案

**方案 1：用 Cleaner + AutoCloseable 替代（推荐）**

```java
public class Theme implements AutoCloseable {
    private final Cleaner cleaner;
    private volatile boolean closed = false;
    private long nativeThemeHandle;
    
    public Theme() {
        this.nativThemeHandle = nativeCreate();
        this.cleaner = Cleaner.create(this, () -> {
            // 快速释放（< 1 秒）
            if (!closed && nativeThemeHandle != 0) {
                nativeDestroy(nativeThemeHandle);
                nativeThemeHandle = 0;
            }
        });
    }
    
    @Override
    public void close() {
        if (!closed) {
            closed = true;
            cleaner.clean();
        }
    }
}
```

**方案 2：升级 AOSP 17（自动收益）**

5s 软告警标记慢对象 + 10s 硬告警打印堆栈 + 30s 致命告警触发 heap dump。

### 4.7 修复效果

| 指标 | AOSP 14 修复前 | AOSP 14 修复后 | AOSP 17 升级 | AOSP 17 + Cleaner |
|:---|:---|:---|:---|:---|
| Watchdog 警告 / h | 30 | 0 | 30 | 0 |
| finalize() 队列 | 20 | 0 | 5 | 0 |
| 慢对象定位 | 困难（仅警告） | 0 | 数分钟（完整堆栈） | N/A |
| 系统卡顿 | 明显 | 无 | 较轻 | 无 |
| GPU 资源泄漏 | 高 | 低 | 中（5s 跳过） | 低 |

**典型模式说明**：上述数据基于"系统服务 Theme + finalize 释放 GPU 资源 + 修复为 Cleaner 模式"的典型场景。**生产数据需自行打点验证**。

## 五、案例 4：DirectByteBuffer 内存泄漏

### 5.1 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某图片处理 App |
| **Android 版本** | Android 13 (API 33) → AOSP 17 (API 37) |
| **问题** | native 内存持续增长 |
| **现象** | OOM 但 Java 堆使用率不高 |

### 5.2 业务代码

```java
public class ImageProcessor {
    public Bitmap processImage(Bitmap source) {
        // 分配 native 内存
        ByteBuffer inputBuf = ByteBuffer.allocateDirect(source.getByteCount());
        ByteBuffer outputBuf = ByteBuffer.allocateDirect(source.getByteCount());
        
        // 业务处理
        // ... GPU 处理 ...

    }

```

### 5.3 问题分析

```
1. DirectByteBuffer 用 Cleaner 释放 native 内存
2. 但业务代码持有 buf 的引用（如成员变量）
3. buf 不被 GC → Cleaner 不触发
4. native 内存持续增长 → OOM
```

### 5.4 监控数据

```bash
$ adb shell dumpsys meminfo com.example.imageprocessor

  Native Heap      234567   200000    34567      500  280000
  Dalvik Heap      45678    40000     5678      100    51234
  TOTAL            280245   240000    40245      600  331290

# Native 内存 234 MB（异常高）
# 但 Java 堆只有 45 MB（不高）
# → DirectByteBuffer 泄漏
```

### 5.5 修复方案

**方案 1：及时释放 ByteBuffer**

```java
public Bitmap processImage(Bitmap source) {
    ByteBuffer inputBuf = null;
    ByteBuffer outputBuf = null;
    try {
        inputBuf = ByteBuffer.allocateDirect(source.getByteCount());
        outputBuf = ByteBuffer.allocateDirect(source.getByteCount());
        
        // 业务处理
        // ...
        
        return resultBitmap;
    } finally {
        if (inputBuf != null) {
            ((DirectBuffer) inputBuf).cleaner().clean();
        }
        if (outputBuf != null) {
            ((DirectBuffer) outputBuf).cleaner().clean();
        }
    }
}
```

**方案 2：用对象池复用 ByteBuffer（推荐）**

```java
public class ByteBufferPool {
    private final ConcurrentLinkedQueue<ByteBuffer> pool = new ConcurrentLinkedQueue<>();
    
    public ByteBuffer acquire(int size) {
        ByteBuffer buf = pool.poll();
        if (buf == null || buf.capacity() < size) {
            buf = ByteBuffer.allocateDirect(size);
        } else {
            buf.clear();
        }
        return buf;
    }
    
    public void release(ByteBuffer buf) {
        if (buf != null) {
            pool.offer(buf);
        }
    }
}
```

**方案 3：升级 AOSP 17（自动收益）**

PhantomReference 延后 Reclaim + Cleaner 集成强化。

## 六、案例 5：ART 17 GenCC 软阈值 kSoftThresholdPercent=30% 联动（**新增**）

### 6.1 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某图片 App（大量使用软引用缓存 Bitmap） |
| **Android 版本** | AOSP 17 (API 37) |
| **问题** | 软引用命中率低，缓存效果差 |
| **现象** | 软引用频繁回收 → 缓存命中率 < 50% |

### 6.2 业务代码

```java
// 用 SoftReference 缓存 Bitmap（Glide 类似模式）
public class BitmapCache {
    private final LinkedHashMap<String, SoftReference<Bitmap>> cache = 
        new LinkedHashMap<>(100, 0.75f, true);
    
    public void put(String key, Bitmap bitmap) {
        cache.put(key, new SoftReference<>(bitmap));
    }
    
    public Bitmap get(String key) {
        SoftReference<Bitmap> ref = cache.get(key);
        return ref != null ? ref.get() : null;  // 软引用回收后返回 null
    }
}
```

### 6.3 问题分析

```
AOSP 14 软引用阈值：
  - dalvik.vm.softrefthreshold = 0.25（25%）
  - 当堆使用 > 25% 时，软引用开始被回收
  - 软引用命中率低（内存压力下大量回收）

AOSP 17 软阈值新增（kSoftThresholdPercent=30%）：
  - art/runtime/options.h 中的新增参数
  - 与 GenCC 软阈值联动
  - 软引用更"宽容"，命中率提升
```

// 软引用保留率公式
// retain_ratio = (heap_used / heap_max - kSoftThresholdPercent / 100)
//                / (1 - kSoftThresholdPercent / 100)

### 7.2 Reference 类型选择决策

```
4 种引用类型的工程选择决策：

1. 强引用（StrongReference）：
   - 用途：业务主对象
   - 决策：默认使用

2. 软引用（SoftReference）：
   - 用途：内存敏感缓存（Glide Bitmap / 业务数据缓存）
   - 决策：AOSP 17 联动 kSoftThresholdPercent=30%

3. 弱引用（WeakReference）：
   - 用途：WeakHashMap / LeakCanary / 不影响 GC 的弱缓存
   - 决策：需要业务层访问对象时使用

4. 虚引用（PhantomReference）：
   - 用途：Cleaner / 资源清理（DirectByteBuffer / FileDescriptor）
   - 决策：需要 native 资源释放时使用

5. FinalReference（finalize()）：
   - 用途：历史遗留代码
   - 决策：新代码不用，老代码升级 AOSP 17 自动收益
```

### 7.3 Netty 服务综合调优

```java
// Netty ByteBuf + Reference 综合调优
public class NettyByteBufCache {
    // 1. 强引用：活跃 ByteBuf
    private final Map<String, ByteBuf> activeBufs = new HashMap<>();
    
    // 2. 软引用：温数据缓存（30% 软阈值联动）
    private final Map<String, SoftReference<ByteBuf>> warmCache = 
        new LinkedHashMap<>(1000, 0.75f, true);
    
    // 3. 弱引用：监控 / 弱引用追踪
    private final Set<WeakReference<ByteBuf>> trackedBufs = 
        Collections.newSetFromMap(new WeakHashMap<>());
    
    // 4. Cleaner：释放 native 内存（兜底）
    public ByteBuf allocateDirect(int size) {
        ByteBuf buf = PooledByteBufAllocator.DEFAULT.directBuffer(size);
        // Cleaner 释放 native 内存（兜底）
        Cleaner.create(buf, () -> {
            // 快速释放（< 1 秒）
            if (buf.refCnt() > 0) {
                buf.release();
            }
        });
        return buf;
    }
    
    public void put(String key, ByteBuf buf) {
        activeBufs.put(key, buf);
        // 软引用缓存（温数据）
        warmCache.put(key, new SoftReference<>(buf));
    }
    
    public ByteBuf get(String key) {
        ByteBuf buf = activeBufs.get(key);
        if (buf != null) return buf;
        
        // 软引用缓存查询
        SoftReference<ByteBuf> ref = warmCache.get(key);
        if (ref != null) {
            buf = ref.get();
            if (buf != null) {
                // 命中：移回活跃
                activeBufs.put(key, buf);
                return buf;
            }
        }
        
        return null;  // 缓存未命中
    }
}
```

├────────────────────────────────────────────────────────────────┤
│                                                                │

## 十一、总结（架构师视角的 5 条 Takeaway）

1. **6 大案例展示 Reference 与 Finalizer 的工程全貌**——Cursor / Bitmap / Theme / DirectByteBuffer 4 大经典案例 + GenCC 软阈值联动 + Reference 调优综合实战。**理解这些案例就掌握了 Reference 治理的完整方法论**。详见 §1 案例总览。
2. **AutoCloseable + try-with-resources 是首选**——业务层显式释放（确定性）+ Cleaner 兜底（安全性）。**所有新代码应使用此模式**。详见 [04-FinalReference](04-FinalReference.md)（重写为 v2 升级版）§10 Finalizer 替代方案。
3. **升级 AOSP 17 是自动收益**——4 线程池化 + 慢对象跳过 + PhantomReference 延后 + 软阈值联动。**Young GC 暂停从 8-12ms 降至 1-2ms（-75-85%）**。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §综合收益。
4. **AOSP 17 kSoftThresholdPercent=30% 联动是软引用调优关键**——软引用命中率 +50%。**图片 / Bitmap 缓存类应用收益最大**。详见 §6 案例 5 GenCC 软阈值联动。
5. **慢对象堆栈让诊断从"猜"变"看"**——5s 软告警 + 10s 硬告警 + 30s 致命告警触发 heap dump。**慢对象定位耗时从"数小时"降至"数分钟"**。详见 [08-FinalizerWatchdog源码](08-FinalizerWatchdog源码.md)（重写为 v2 升级版）§4.1 ART 17 慢对象 dump 机制。
