# 6.1 Java 引用的可达性状态机

> **本节回答一个根本问题**：Java 的 4 种引用类型（强 / 软 / 弱 / 虚）的可达性等级是什么？GC 怎么决定对象的回收时机？
>
> **答案**：**强 > 软 > 弱 > 虚 的可达性等级** —— 等级越低，越容易被 GC 回收。
>
> **理解本节，就理解了 Java 引用的完整生命周期**。

---

## 一、可达性的定义

### 6.1.1 GC 如何判定对象的可达性

```
可达性（Reachability）：
  从 GC Roots 出发，沿着引用链（Reference Chain）能走到的对象

不可达（Unreachable）：
  从任何 GC Root 出发，都无法走到的对象

→ 不可达对象 = 死亡，下一轮 GC 回收
```

### 6.1.2 Java 的可达性等级

```
强可达（Strongly Reachable）：
  - 对象有强引用指向
  - 永远不会被 GC 回收

软可达（Softly Reachable）：
  - 对象只有软引用指向
  - 内存不足时回收

弱可达（Weakly Reachable）：
  - 对象只有弱引用指向
  - 下一次 GC 一定回收

虚可达（Phantomly Reachable）：
  - 对象已经被 finalize（如果重写了 finalize）
  - 只有虚引用指向
  - get() 永远返回 null
```

### 6.1.3 可达性的优先级

```
强引用 > 软引用 > 弱引用 > 虚引用
```

**重要原则**：如果一个对象有多种引用指向，按"最强"的引用类型决定可达性等级。

```java
Object strong = new Object();
WeakReference<Object> weak = new WeakReference<>(strong);
// strong 是强可达
// weak 是弱可达（但对象本身是强可达）

strong = null;
// 现在对象只有 weak 引用 → 弱可达
// 下次 GC 一定回收
```

## 二、4 种引用的对比

### 6.1.4 4 种引用类型

| 引用类型 | 类 | 回收时机 | 典型用途 |
|:---|:---|:---|:---|
| **强引用** | `Object obj = new Object()` | 永远不回收 | 普通对象 |
| **软引用** | `SoftReference<T>` | 内存不足时 | LRU 缓存 |
| **弱引用** | `WeakReference<T>` | 下次 GC 一定 | WeakHashMap、LeakCanary |
| **虚引用** | `PhantomReference<T>` | finalize 后入队 | Cleaner |

### 6.1.5 4 种引用的 API 对比

```java
// 1. 强引用
Object strong = new Object();
// get() 不适用（直接是对象本身）

// 2. 软引用
SoftReference<Object> soft = new SoftReference<>(new Object());
Object value1 = soft.get();  // 可能返回 null（内存不足时）

// 3. 弱引用
WeakReference<Object> weak = new WeakReference<>(new Object());
Object value2 = weak.get();  // 可能返回 null（下次 GC 后）

// 4. 虚引用
PhantomReference<Object> phantom = new PhantomReference<>(
    new Object(), new ReferenceQueue<>());
Object value3 = phantom.get();  // 永远返回 null
```

### 6.1.6 Reference 类的核心方法

```java
// libcore/ojluni/src/main/java/java/lang/ref/Reference.java
public abstract class Reference<T> {
    private T referent;         // 引用的对象
    
    public T get() {
        return referent;  // 子类可能重写
    }
    
    public void clear() {
        synchronized (this) {
            if (referent != null) {
                referent = null;
                // 通知 GC
            }
        }
    }
    
    public boolean enqueue() {
        // 把 Reference 加入 ReferenceQueue
        return queue.enqueue(this);
    }
}
```

## 三、可达性状态机

### 6.1.7 状态转换图

```
┌─────────────────────────────────────────────────────────┐
│              Java 对象可达性状态机                        │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  Active（活跃）                                          │
│  - referent 非空                                        │
│  - GC 不会回收                                          │
│       │                                                  │
│       │ clear() 或 referent 被回收                       │
│       ▼                                                  │
│  Pending（待入队）                                       │
│  - referent 被清空                                       │
│  - 等待加入 ReferenceQueue                                │
│       │                                                  │
│       │ enqueue()                                        │
│       ▼                                                  │
│  Enqueued（已入队）                                       │
│  - 已在 ReferenceQueue 中                                 │
│  - 业务线程可以 poll()                                    │
│       │                                                  │
│       │ ReferenceQueue.clear()                           │
│       ▼                                                  │
│  Inactive（不活跃）                                       │
│  - 已从 ReferenceQueue 移除                               │
│  - 永久不活跃                                            │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 6.1.8 状态转换详解

```
对象刚创建时：
  - Reference 对象处于 Active 状态
  - referent 指向被引用的对象

GC 决定回收 referent：
  - SoftReference：内存不足时
  - WeakReference：下次 GC 时
  - PhantomReference：finalize 后
  - FinalReference：需要 finalize 时

GC 回收 referent 后：
  - 清空 referent 字段
  - 把 Reference 加入 pending list
  - 状态变为 Pending

ReferenceQueueDaemon 处理 pending list：
  - 把 Reference 加入 ReferenceQueue
  - 状态变为 Enqueued

业务线程调用 ReferenceQueue.poll()：
  - 取出 Reference
  - 状态变为 Inactive
```

## 四、GC 处理 Reference 的时机

### 6.1.9 不同 GC 阶段的 Reference 处理

```
CMS 4 阶段：
  Initial Mark (STW) → 不处理 Reference
  Concurrent Mark → 不处理 Reference
  Remark (STW) → 处理 Reference
  Concurrent Sweep → 不处理 Reference

CC GC 3 阶段：
  Initialize (STW) → 不处理 Reference
  Concurrent Copying → 不处理 Reference
  Reclaim (STW) → 处理 Reference

GenCC 流程：
  Minor GC → 简化 Reference 处理
  Major GC → 完整 Reference 处理
```

### 6.1.10 ReferenceProcessor 入口

```cpp
// art/runtime/gc/reference_processor.h
class ReferenceProcessor {
public:
    void ProcessReferences(...) {
        // 1. 处理软引用（按内存压力决定）
        HandleSoftReferences(...);
        
        // 2. 处理弱引用（全部入队）
        HandleWeakReferences(...);
        
        // 3. 处理 Final 引用（加入 FinalizerDaemon 队列）
        HandleFinalReferences(...);
        
        // 4. 处理虚引用（入队）
        HandlePhantomReferences(...);
    }
};
```

### 6.1.11 软引用的回收阈值

```cpp
// 软引用保留率公式
// retain_ratio = (heap_used / heap_max - threshold) / (1 - threshold)
// heap_used：当前堆使用量
// heap_max：当前堆上限
// threshold：dalvik.vm.softrefthreshold（默认 0.25）
//
// 当 heap_used / heap_max > threshold 时，软引用开始被回收
// 当 heap_used / heap_max = 1 时，几乎所有软引用都被回收
```

## 五、Reference 类的实现

### 6.1.12 Reference 类源码入口

```java
// libcore/ojluni/src/main/java/java/lang/ref/Reference.java
public abstract class Reference<T> {
    // 引用队列（ReferenceQueue）
    private ReferenceQueue<? super T> queue;
    
    // 链表 next（用于 pending list）
    private Reference next;
    
    // 内部字段
    private T referent;  // 引用的对象
    
    // 关键方法
    public T get() {
        return referent;
    }
    
    public void clear() {
        synchronized (this) {
            referent = null;
        }
    }
    
    public boolean enqueue() {
        return queue.enqueue(this);
    }
    
    // GC 内部调用
    void enqueueInternal() {
        // 加入 pending list
        synchronized (pendingLock) {
            this.next = pendingHead;
            pendingHead = this;
            pendingLock.notifyAll();
        }
    }
}
```

### 6.1.13 ReferenceQueue 类

```java
// libcore/ojluni/src/main/java/java/lang/ref/ReferenceQueue.java
public class ReferenceQueue<T> {
    // 内部链表 head
    private Reference<? extends T> head;
    
    // 入队（由 GC 或业务线程调用）
    boolean enqueue(Reference<? extends T> r) {
        synchronized (this) {
            r.queue = this;  // 反向引用
            r.next = head;
            head = r;
            return true;
        }
    }
    
    // 出队（业务线程调用）
    public Reference<? extends T> poll() {
        synchronized (this) {
            if (head == null) return null;
            Reference<? extends T> r = head;
            head = head.next;
            r.next = null;
            return r;
        }
    }
    
    // 阻塞出队（业务线程调用）
    public Reference<? extends T> remove(long timeout) throws InterruptedException {
        // 等待直到有 Reference 入队
    }
}
```

### 6.1.14 ART 的 Reference 处理入口

```cpp
// art/runtime/gc/reference_processor.cc
void ReferenceProcessor::ProcessReferences(...) {
    // 1. 处理软引用
    SoftReferenceList soft_refs = CollectSoftReferences();
    ClearReferents(soft_refs);  // 清空 referent
    
    // 2. 处理弱引用
    WeakReferenceList weak_refs = CollectWeakReferences();
    EnqueuePendingReferences(weak_refs);  // 入队
    
    // 3. 处理 Final 引用
    FinalReferenceList final_refs = CollectFinalReferences();
    EnqueueFinalReferences(final_refs);  // 加入 FinalizerDaemon 队列
    
    // 4. 处理虚引用
    PhantomReferenceList phantom_refs = CollectPhantomReferences();
    EnqueuePendingReferences(phantom_refs);  // 入队
}
```

## 六、可达性的工程价值

### 6.1.15 可达性 vs 引用计数的工程优势

```
引用计数的问题：
  1. 循环引用无法回收
  2. 每次指针赋值都要更新计数
  3. 多线程下需要原子操作

可达性的优势：
  1. 天然解决循环引用（可达性图分析）
  2. 无需每次赋值更新（只在 GC 时遍历）
  3. STW 暂停即可解决并发问题
```

### 6.1.16 4 种引用的典型场景

| 引用类型 | 典型场景 | 推荐度 |
|:---|:---|:---|
| **强引用** | 普通对象 | 默认 |
| **软引用** | 内存敏感缓存（Glide Bitmap 缓存） | 推荐 |
| **弱引用** | WeakHashMap、LeakCanary | 推荐 |
| **虚引用** | Cleaner（DirectByteBuffer 释放） | 推荐 |

### 6.1.17 Reference 选择的决策

```
需要缓存数据 + 内存压力敏感？
  → SoftReference

需要弱缓存 + 不影响 GC？
  → WeakReference

需要真正析构（释放 native 资源）？
  → Cleaner（基于 PhantomReference）

需要 finalizer 机制？
  → 重写 finalize()（不推荐）/ 用 Cleaner（推荐）
```

## 七、本节小结

1. **可达性等级**：强 > 软 > 弱 > 虚
2. **回收时机**：强 = 永远不回收；软 = 内存不足时；弱 = 下次 GC 一定；虚 = finalize 后入队
3. **状态机**：Active → Pending → Enqueued → Inactive
4. **GC 处理**：通常在 Remark（CMS）或 Reclaim（CC GC）阶段处理 Reference
5. **软引用阈值**：由 `dalvik.vm.softrefthreshold` 决定（默认 0.25）

→ **理解可达性状态机，就理解了 Java 引用的完整生命周期**。
