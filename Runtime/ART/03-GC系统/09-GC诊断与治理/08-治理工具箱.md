# 9.8 GC 治理工具箱

> **本节回答一个根本问题**：GC 治理有哪些手段？怎么从工具箱选？
>
> **答案**：**四大治理手段：参数调优 / 代码优化 / 监控告警 / 工具链** —— 按问题匹配手段。

---

## 一、治理工具箱总览

### 9.8.1 工具箱分类

```
GC 治理工具箱：

1. 参数调优
   - 调整 GC 算法
   - 调整堆大小
   - 调整并发度
   - 调整晋升阈值

2. 代码优化
   - 减少对象分配
   - 对象池化
   - 避免大对象
   - 避免 Finalize

3. 监控告警
   - JVMTI 监听
   - Perfetto 分析
   - Heap Dump
   - 阈值告警

4. 工具链
   - LeakCanary（自动）
   - MAT（深度）
   - Profiler（实时）
   - Android Studio Inspector
```

### 9.8.2 工具箱选择决策树

```
GC 治理决策树：

问题：GC 频繁
├─ 检测：JVMTI 频率
├─ 定位：Heap Dump + LeakCanary
├─ 治理：
│  ├─ 代码优化（减少分配）
│  ├─ 参数调优（调大堆）
│  └─ 监控告警（持续观察）
└─ 验证：再次抓 trace

问题：GC 卡顿
├─ 检测：Perfetto trace
├─ 定位：找 STW 阶段
├─ 治理：
│  ├─ 减少对象分配（减少 STW 时间）
│  ├─ 调整并发度
│  └─ 升级到 GenCC
└─ 验证：再抓 trace

问题：内存泄漏
├─ 检测：LeakCanary
├─ 定位：Heap Dump + MAT
├─ 治理：
│  ├─ 修复引用
│  ├─ 解除强引用
│  └─ WeakReference
└─ 验证：再压测

问题：OOM
├─ 检测：Runtime 堆水位
├─ 定位：Heap Dump
├─ 治理：
│  ├─ 减少大对象
│  ├─ Bitmap 优化
│  └─ 排查泄漏
└─ 验证：压测不 OOM
```

## 二、参数调优

### 9.8.3 堆大小调优

```bash
# 通过 system property 调堆
adb shell setprop dalvik.vm.heapgrowthlimit 256m  # 默认 256MB
adb shell setprop dalvik.vm.heapsize 512m          # 最大 512MB
adb shell setprop dalvik.vm.heapminfree 4m         # 最小空闲 4MB
adb shell setprop dalvik.vm.heapmaxfree 8m         # 最大空闲 8MB
```

### 9.8.4 GC 算法调优

```bash
# 选择 GC 算法
adb shell setprop dalvik.vm.gctype CC              # CC GC
adb shell setprop dalvik.vm.gctype CMS              # CMS GC（Android 8-）
adb shell setprop dalvik.vm.gctype GenCC            # GenCC（Android 14+）

# Generational CC 参数（Android 14+）
adb shell setprop dalvik.vm.gencc.heap-size 256m
```

### 9.8.5 并发度调优

```bash
# CC GC 的并发度
adb shell setprop dalvik.vm.cc.parallel-cg-mark   # 标记并发
adb shell setprop dalvik.vm.cc.parallel-cg-sweep  # 回收并发

# GenCC 的并发度
adb shell setprop dalvik.vm.gencc.concurrency     # 并发线程数
```

### 9.8.6 晋升阈值调优

```bash
# 调整晋升阈值
adb shell setprop dalvik.vm.kPromotionThreshold 15  # 默认 15

# ART 14 自适应晋升阈值
adb shell setprop dalvik.vm.gencc.adaptive-promotion true
```

## 三、代码优化

### 9.8.7 减少对象分配

```java
// 1. 避免在循环中分配对象
// Bad
for (int i = 0; i < n; i++) {
    String s = String.valueOf(i);  // 每次创建 String
}

// Good
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) {
    sb.append(i);
}

// 2. 使用对象池
// Bad
new Thread(runnable).start();  // 每次创建 Thread

// Good
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>()
);
executor.execute(runnable);

// 3. 避免装箱
// Bad
Map<Integer, String> map = new HashMap<>();
for (int i = 0; i < n; i++) {
    map.put(i, "value");  // 装箱
}

// Good
Map<Integer, String> map = new HashMap<>();
int[] keys = new int[n];
String[] values = new String[n];
for (int i = 0; i < n; i++) {
    map.put(keys[i], values[i]);  // 不装箱
}

// 4. 避免 String 拼接
// Bad
String result = "";
for (String s : list) {
    result += s;  // 每次创建新 String
}

// Good
StringBuilder sb = new StringBuilder();
for (String s : list) {
    sb.append(s);
}
String result = sb.toString();
```

### 9.8.8 对象池化

```java
// Bitmap 池化（Android）
public class BitmapPool {
    private final Stack<Bitmap> pool = new Stack<>();
    
    public Bitmap obtain(int width, int height) {
        if (!pool.isEmpty()) {
            Bitmap bmp = pool.pop();
            if (bmp.getWidth() == width && bmp.getHeight() == height) {
                return bmp;
            }
            bmp.recycle();
        }
        return Bitmap.createBitmap(width, height, Bitmap.Config.ARGB_8888);
    }
    
    public void recycle(Bitmap bmp) {
        if (pool.size() < MAX_POOL_SIZE) {
            pool.push(bmp);
        } else {
            bmp.recycle();
        }
    }
}

// View 池化（RecyclerView）
public class MyViewHolder extends RecyclerView.ViewHolder {
    public MyViewHolder(View view) {
        super(view);
    }
}
```

### 9.8.9 避免大对象

```java
// 1. Bitmap 优化
// Bad
Bitmap bmp = BitmapFactory.decodeFile(path);  // 原始大小

// Good
BitmapFactory.Options options = new BitmapFactory.Options();
options.inSampleSize = 2;  // 缩小一半
options.inPreferredConfig = Bitmap.Config.RGB_565;  // 565 而非 8888
Bitmap bmp = BitmapFactory.decodeFile(path, options);

// 2. 集合大小限制
// Bad
List<String> list = new ArrayList<>();  // 无限制

// Good
List<String> list = new ArrayList<>(MAX_SIZE);  // 限制容量

// 3. 字符串拼接
// Bad
String s = largeString1 + largeString2 + largeString3;

// Good
StringBuilder sb = new StringBuilder();
sb.append(largeString1).append(largeString2).append(largeString3);
```

### 9.8.10 避免 Finalize

```java
// 1. 用 Cleaner 替代 finalize
// Bad
public class Resource {
    @Override
    protected void finalize() throws Throwable {
        close();
    }
}

// Good
public class Resource {
    private static final Cleaner CLEANER = Cleaner.create();
    
    private final Cleaner.Cleanable cleanable;
    
    public Resource() {
        cleanable = CLEANER.register(this, new CleanupAction());
    }
    
    private static class CleanupAction implements Runnable {
        @Override
        public void run() {
            close();
        }
    }
}

// 2. 用 AutoCloseable + try-with-resources
public class Resource implements AutoCloseable {
    @Override
    public void close() {
        // 释放资源
    }
}

try (Resource r = new Resource()) {
    // 使用资源
}
```

## 四、监控告警

### 9.8.11 JVMTI 实时监控

```cpp
// 实时监控 + 告警
void JNICALL OnGCFinish(jvmtiEnv* env) {
    long pause_time = currentTimeMillis() - g_gc_start_time;
    
    // 长 STW 告警
    if (pause_time > 100) {
        apmClient.alert("gc.pause.long", pause_time);
    }
    
    // 抓 trace
    if (pause_time > 200) {
        asyncCaptureTrace("long_gc");
    }
}
```

### 9.8.12 Perfetto 离线分析

```bash
# 抓取 trace 用于事后分析

  -t 30s sched freq idle am wm gfx view binder_driver hal dalvik
adb pull /data/local/tmp/trace.proto
```

## 六、治理实践

### 9.8.17 案例 1：GC 频繁治理

```
现象：App GC 频率 100 次/分钟
定位：Perfetto trace 显示 kGcCauseForAlloc 频繁

治理：
1. 代码优化（减少对象分配）
2. 对象池化
3. 调大堆
4. 监控告警

效果：GC 频率降到 20 次/分钟
```

### 9.8.18 案例 2：GC 卡顿治理

```
现象：滑动列表卡顿（每 1s 一次卡顿）
定位：Perfetto 显示 GC STW 50ms+

治理：
1. 减少 GC 期间对象分配
2. 升级到 GenCC
3. 调整并发度

效果：STW 降到 5ms，卡顿消失
```

### 9.8.19 案例 3：内存泄漏治理

```
现象：App 进入二级页面后内存持续增长
定位：LeakCanary 检测到 Activity 泄漏

治理：
1. Heap Dump
2. MAT 分析引用链
3. 修复引用
4. WeakReference 替代

效果：泄漏修复，内存稳定
```

### 9.8.20 案例 4：OOM 治理

```
现象：App 在大图片场景 OOM
定位：Bitmap 占用大 + Heap Dump 显示大对象

治理：
1. Bitmap 优化（inSampleSize / RGB_565）
2. Bitmap 复用（BitmapPool）
3. 排查泄漏
4. 监控告警

效果：OOM 不再发生
```

## 七、治理的优先级

### 9.8.21 优先级评估

```
治理优先级：

P0（立即）：
- OOM
- 长 STW（> 100ms）
- 严重内存泄漏

P1（紧急）：
- GC 频繁（> 50 次/分钟）
- 中等卡顿（50-100ms）

P2（重要）：
- GC 频率中等（10-50 次/分钟）
- 轻微卡顿（16-50ms）

P3（优化）：
- GC 频率低（< 10 次/分钟）
- 偶发卡顿

→ 按优先级分配治理资源
```

### 9.8.22 治理的 ROI

```
ROI（投入产出比）：

高 ROI（先做）：
- 减少对象分配（成本低，收益大）
- 避免 Finalize（成本低，收益大）
- 监控告警（成本中，收益高）

中 ROI（接着做）：
- 调大堆（成本低，收益中）
- 对象池化（成本中，收益中）

低 ROI（后做）：
- CC GC 调优（成本高，收益中）
- 升级到 GenCC（成本高，收益高，但需要平台升级）

→ 按 ROI 分配资源
```

## 八、本节小结

1. **四大治理手段**：参数调优 / 代码优化 / 监控告警 / 工具链
2. **代码优化**：减少分配、对象池化、避免大对象、避免 Finalize
3. **工具链协作**：LeakCanary + Perfetto + MAT + 自建 APM
4. **优先级**：P0 紧急到 P3 优化
5. **ROI**：先做高 ROI 的（成本低，收益大）

→ **理解治理工具箱，就掌握了"GC 治理"的完整方法论**。
