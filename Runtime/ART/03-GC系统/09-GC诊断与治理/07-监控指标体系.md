# 9.7 GC 监控指标体系

> **本节回答一个根本问题**：GC 监控应该关注哪些指标？关键指标 / 告警阈值 / 仪表盘设计？
>
> **答案**：**四大指标族：频率 / 时长 / 原因 / 堆水位** —— 配合三档阈值（警戒 / 告警 / 紧急）。

---

## 一、监控指标体系设计

### 9.7.1 为什么需要监控指标体系

```
为什么需要监控指标体系：

1. 真实场景
   - GC 是 App 性能的最关键因素之一
   - GC 频率高 / 停顿长 → 直接影响卡顿 / 耗电
   - 必须量化监控

2. 多个维度
   - 频率（每秒多少次 GC）
   - 时长（每次 GC 多长）
   - 原因（什么触发了 GC）
   - 堆水位（堆用了多少）
   - 单维度信息不完整

3. 业务对接
   - 业务方需要量化指标（不是"GC 多了"）
   - 指标 → 阈值 → 告警 → 处理

→ 必须设计完整的监控指标体系
```

### 9.7.2 指标族分类

```
GC 监控指标族分类：

1. 频率指标族
   - GC 次数（每秒）
   - Minor GC 次数
   - Major GC 次数

2. 时长指标族
   - STW 时间
   - 总 GC 时间（含 Concurrent）
   - 95 / 99 分位

3. 原因指标族
   - 各 GcCause 的次数（kGcCauseForAlloc / kGcCauseBackground 等）

4. 堆水位指标族
   - 堆当前使用 / 总量
   - Native 堆大小
   - OOM 次数

5. 衍生指标族
   - GC 分配速率（MB/s）
   - GC 暂停占总时长比例
```

---

## 二、四大核心指标

### 9.7.3 指标 1：GC 频率

```
GC 频率指标：

1. 每分钟 GC 次数（gc.count.per.min）
2. 每秒 GC 次数（gc.count.per.sec）
3. Minor GC 与 Major GC 的比例
4. 不同 GcCause 的频率

正常范围：
- 静止应用：< 1 次/分钟
- 轻度使用：1-5 次/分钟
- 重度使用：5-20 次/分钟
- 异常：> 50 次/分钟 → 异常

采样实现：
- 累计 + 周期上报（每秒/每分钟）
- 分位数上报
```

### 9.7.4 指标 2：GC 时长

```
GC 时长指标：

1. STW 时间（gc.pause）
   - 每次 GC 的 STW 时长
   - 平均 / 95 分位 / 99 分位 / 最大

2. 总 GC 时间（含 Concurrent）
   - GC 从开始到结束的总时长

3. 单次耗时分布
   - < 5ms（正常）
   - 5-10ms（轻微）
   - 10-50ms（中等）
   - 50-100ms（严重）
   - > 100ms（紧急）

卡顿阈值：
- 16ms（1 帧）→ 用户能感知
- 100ms → 明显卡顿
- 300ms → ANR 风险
```

### 9.7.5 指标 3：GC 原因

```
GC 原因指标：

1. 各 GcCause 的次数（ART 8+）
   - kGcCauseForAlloc
   - kGcCauseForNativeAlloc
   - kGcCauseBackground
   - kGcCauseExplicit
   - kGcCauseForTrim
   - kGcCauseForInspect
   - kGcCauseJitArenaFull
   - kGcCauseMax

2. 异常的 GcCause
   - kGcCauseForAlloc 频繁 → 分配过多
   - kGcCauseBackground 频繁 → 堆压力大
   - kGcCauseExplicit 频繁 → 手动 GC
   - kGcCauseForTrim 频繁 → 后台 trim

3. GcCause → 优化方向
   - kGcCauseForAlloc：减少对象分配
   - kGcCauseBackground：调大堆
   - kGcCauseForNativeAlloc：减少 Native 分配
```

### 9.7.6 指标 4：堆水位

```
堆水位指标：

1. Java 堆
   - Java Heap Total（kb）
   - Java Heap Used（kb）
   - Java Heap Free（kb）
   - Java Heap Max（kb）

2. Native 堆
   - Native Heap Used（kb）
   - Native Heap Max（kb）

3. 堆使用率
   - Java Heap Used / Java Heap Max
   - > 80% → 高风险
   - > 90% → 紧急

4. 增长趋势
   - 堆随时间增长
   - 持续增长 → 可能的泄漏
```

## 三、衍生指标

### 9.7.5 GC 分配速率

```
GC 分配速率（Allocation Rate）：

定义：每秒分配多少对象内存
计算：gc.allocated_bytes / gc.time_window

正常范围：
- < 10 MB/s（轻量）
- 10-50 MB/s（中等）
- 50-100 MB/s（重度）
- > 100 MB/s（异常）

→ 分配速率过高 → GC 频率会高 → 影响性能
```

### 9.7.6 GC 暂停占比

```
GC 暂停占比（GC Pause Ratio）：

定义：GC 暂停时间占运行时间的比例
计算：sum(gc.pause) / total.time

正常范围：
- < 1%（健康）
- 1-5%（可接受）
- 5-10%（需要优化）
- > 10%（异常）

→ GC 暂停占比高 → 直接影响 App 性能
```

### 9.7.7 OOM 指标

```
OOM 指标：

1. OOM 次数（oom.count）
2. OOM 时的堆大小（oom.heap.size）
3. OOM 时的 GcCause（oom.cause）
4. OOM 时的内存快照（可选）

→ OOM 是 GC 的极端失败 → 必须监控
```

### 9.7.8 Finalize 与 Reference 指标

```
Finalize 与 Reference 指标：

1. Finalize 耗时（finalize.duration）
2. FinalizerDaemon 队列长度（finalizer.queue.length）
3. Cleaner 队列长度（cleaner.queue.length）

→ Finalizer 队列堆积 → 可能影响 GC 性能
```

## 四、采集实现

### 9.7.9 JVMTI 采集

```cpp
// JVMTI 采集 GC 频率和时长
static int gc_count = 0;
static long total_pause_time = 0;

void JNICALL OnGCStart(jvmtiEnv* env) {
    g_gc_start_time = currentTimeMillis();
}

void JNICALL OnGCFinish(jvmtiEnv* env) {
    long pause_time = currentTimeMillis() - g_gc_start_time;
    gc_count++;
    total_pause_time += pause_time;
}

// 定时上报
void report() {
    apmClient.report("gc.count", gc_count);
    apmClient.report("gc.pause.avg", total_pause_time / gc_count);
    apmClient.report("gc.pause.total", total_pause_time);
}
```

### 9.7.10 Runtime 采集

```java
// 通过 Runtime 采集堆水位
public void collectHeapMetrics() {
    Runtime runtime = Runtime.getRuntime();
    long total = runtime.totalMemory();   // 堆总量
    long free = runtime.freeMemory();      // 堆空闲
    long max = runtime.maxMemory();        // 堆最大
    
    long used = total - free;
    double usage = (double) used / max;
    
    apmClient.report("java.heap.used", used);
    apmClient.report("java.heap.total", total);
    apmClient.report("java.heap.max", max);
    apmClient.report("java.heap.usage", usage);
}
```

### 9.7.11 Debug.MemoryInfo 采集

```java
// Debug.MemoryInfo 采集 native 堆等
public void collectMemoryInfo() {
    Debug.MemoryInfo memoryInfo = new Debug.MemoryInfo();
    Debug.getMemoryInfo(memoryInfo);
    
    // Pss Total
    long totalPss = memoryInfo.getTotalPss();
    apmClient.report("memory.pss.total", totalPss);
    
    // Native Heap
    long nativeHeap = memoryInfo.nativePss;
    apmClient.report("memory.pss.native", nativeHeap);
    
    // Java Heap
    long javaHeap = memoryInfo.dalvikPss;
    apmClient.report("memory.pss.java", javaHeap);
}
```

## 五、告警阈值设计

### 9.7.12 三档阈值

```
三档阈值设计：

1. 警戒（warning）
   - 监控异常趋势
   - 不紧急，积累中
   - 日志记录 + 不打扰用户

2. 告警（alert）
   - 立即关注
   - 可能影响用户
   - 日志 + 通知开发者

3. 紧急（critical）
   - 立即处理
   - 影响用户
   - 日志 + 告警通道 + 抓 trace
```

### 9.7.13 GC 频率告警阈值

```
GC 频率告警阈值：

1. 警戒：gc.count.per.min > 20
   - App 频率高，可能有问题
   - 持续观察

2. 告警：gc.count.per.min > 50
   - 异常频繁
   - 通知开发者

3. 紧急：gc.count.per.min > 100
   - 持续高频
   - 抓 trace + 紧急处理
```

### 9.7.14 GC 时长告警阈值

```
GC 时长告警阈值：

1. 警戒：gc.pause.p95 > 16ms
   - 影响帧率（1 帧）
   - 轻微卡顿

2. 告警：gc.pause.p95 > 50ms
   - 明显卡顿
   - 影响体验

3. 紧急：gc.pause.max > 100ms
   - 严重卡顿
   - 用户能感知
   - 抓 trace + 立即处理
```

### 9.7.15 堆水位告警阈值

```
堆水位告警阈值：

1. 警戒：java.heap.usage > 0.7
   - 堆使用率 70%
   - 即将达到上限

2. 告警：java.heap.usage > 0.85
   - 堆使用率 85%
   - OOM 风险

3. 紧急：java.heap.usage > 0.95
   - 堆使用率 95%
   - OOM 即将发生
   - 抓 hprof + 立即处理
```

## 六、仪表盘设计

### 9.7.16 仪表盘结构

```
GC 监控仪表盘设计：

1. 总览仪表盘
   - GC 频率（最近 1 小时趋势）
   - GC 时长（最近 1 小时趋势）
   - 堆水位（最近 1 小时趋势）
   - OOM 次数（最近 1 天累计）

2. 详细仪表盘
   - 各 GcCause 频率
   - STW 时长分布
   - 堆分配速率

3. 异常仪表盘
   - 告警事件列表
   - OOM 事件列表
   - 长 STW 事件列表
```

### 9.7.17 实时 vs 历史

```
实时 vs 历史指标：

1. 实时指标
   - 用于实时监控和告警
   - 采样周期：1 秒 - 10 秒
   - 数据量：适中

2. 历史指标
   - 用于趋势分析和回溯
   - 存储周期：1 天 - 30 天
   - 数据量：大
   - 降采样：原始 → 1 分钟 → 1 小时
```

## 七、APM 集成

### 9.7.18 自建 APM 集成

```
自建 APM 中的 GC 监控：

1. 数据采集层
   - JVMTI 监听 GC 事件
   - Runtime 采集堆水位
   - Debug.getMemoryInfo 采集 native

2. 数据处理层
   - 计算衍生指标（分配速率等）
   - 触发阈值告警

3. 数据上报层
   - 实时上报（每分钟）
   - 异常上报（立即）
   - 汇总上报（每天）
```

### 9.7.19 开源 APM 集成

```java
// LeakCanary + JVMTI + 自建 APM
public class MyApplication extends Application {
    @Override
    public void onCreate() {
        super.onCreate();
        
        // 启用 LeakCanary（debug）
        if (BuildConfig.DEBUG) {
            LeakCanary.INSTANCE.monitorObject("MyObject", myObject);
        }
        
        // 启用 JVMTI 监控
        JvmtiGcMonitor.enableGcEvents();
        
        // 启用自建 APM
        ApmClient.init(this);
    }
}
```

## 八、本节小结

1. **四大核心指标族**：频率 / 时长 / 原因 / 堆水位
