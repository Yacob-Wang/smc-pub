## 一、监控指标体系设计

### 9.7.1 为什么需要监控指标体系

```
为什么需要监控指标体系：

1. 真实场景
   - GC 是 App 性能的最关键因素之一
   - GC 频率高 / 停顿长 → 直接影响卡顿 / 耗电
   - 必须量化监控

2. 多个维度
   - 频率（每秒多少次 GC）
   - 时长（每次 GC 多长）
   - 原因（什么触发了 GC）
   - 堆水位（堆用了多少）
   - 单维度信息不完整

3. 业务对接
   - 业务方需要量化指标（不是"GC 多了"）
   - 指标 → 阈值 → 告警 → 处理

【AOSP 17 增强】
- dumpsys meminfo 新增 ART 内部状态输出
- 软阈值 kSoftThresholdPercent=30% 状态可见
- Native 堆分类更细（sheaves slab 统计）

→ 必须设计完整的监控指标体系
```

### 9.7.2 指标族分类（AOSP 17 增强）

```
GC 监控指标族分类（AOSP 17 增强）：

1. 频率指标族
   - GC 次数（每秒/每分钟）
   - Minor GC 次数 / Major GC 次数
   -【AOSP 17】Young GC 次数 / Old GC 次数（GenCC 区分）

2. 时长指标族
   - STW 时间（gc.pause）
   - 总 GC 时间（含 Concurrent）
   - 95 / 99 分位
   -【AOSP 17】GenCC 各阶段耗时

3. 原因指标族
   - 各 GcCause 的次数
   -【AOSP 17 增强】通过 JVMTI 直接获取 gc_cause

4. 堆水位指标族
   - Java 堆 当前使用 / 总量 / 最大
   - Native 堆 当前使用 / 最大
   - 堆使用率
   -【AOSP 17 新增】软阈值触发距离（kSoftThresholdPercent=30%）
   -【AOSP 17 新增】硬阈值触发距离（80%）

5. ART 内部状态指标族【AOSP 17 新增】
   - JIT Code Cache 大小
   - Loaded Classes 数
   - JNI Global Refs 数
   - ClassLoader 数

6.【AOSP 17 新增】Native 堆分类指标族
   - sheaves slab 占用
   - malloc 占用
   - mmap 占用

7. 衍生指标族
   - GC 分配速率（MB/s）
   - GC 暂停占总时长比例
   - 晋升速率（Young → Old）
```

## 二、四大核心指标

### 9.7.4 指标 1：GC 频率（AOSP 17 增强）

```
GC 频率指标（AOSP 17 增强）：

1. 每分钟 GC 次数（gc.count.per.min）
2. 每秒 GC 次数（gc.count.per.sec）
3.【AOSP 17 新增】Young GC 次数（gc.young.count）
4.【AOSP 17 新增】Old GC 次数（gc.old.count）
5.【AOSP 17 新增】Sticky GC 次数（gc.sticky.count）
6. Minor GC 与 Major GC 的比例
7. 不同 GcCause 的频率

正常范围（AOSP 17）：
- 静止应用：< 1 次/分钟
- 轻度使用：1-5 次/分钟
- 重度使用：5-20 次/分钟
- 异常：> 50 次/分钟 → 异常

采样实现：
- 累计 + 周期上报（每秒/每分钟）
- 分位数上报
-【AOSP 17 增强】通过 JVMTI 直接获取 GC 类型
```

### 9.7.5 指标 2：GC 时长（AOSP 17 增强）

```
GC 时长指标（AOSP 17 增强）：

1. STW 时间（gc.pause）
   - 每次 GC 的 STW 时长
   - 平均 / 95 分位 / 99 分位 / 最大

2. 总 GC 时间（含 Concurrent）
   - GC 从开始到结束的总时长

3.【AOSP 17 新增】GenCC 各阶段耗时
   - Initialize 耗时
   - Marking 耗时
   - Reclaim 耗时
   - Promotion 耗时

4. 单次耗时分布
   - < 5ms（正常）
   - 5-10ms（轻微）
   - 10-50ms（中等）
   - 50-100ms（严重）
   - > 100ms（紧急）

卡顿阈值：
- 16ms（1 帧）→ 用户能感知
- 100ms → 明显卡顿
- 300ms → ANR 风险
```

### 9.7.6 指标 3：GC 原因（AOSP 17 增强）

```
GC 原因指标（AOSP 17 增强）：

1. 各 GcCause 的次数
   - kGcCauseForAlloc
   - kGcCauseForNativeAlloc
   - kGcCauseBackground
   - kGcCauseExplicit
   - kGcCauseForTrim
   - kGcCauseForInspect
   - kGcCauseJitArenaFull
   - kGcCauseMax

2.【AOSP 17 增强】通过 JVMTI GetGarbageCollectionExtendedInfo 直接获取 gc_cause
   - 不需要额外采集
   - JVMTI 回调中即可拿到

3. 异常的 GcCause
   - kGcCauseForAlloc 频繁 → 分配过多
   - kGcCauseBackground 频繁 → 堆压力大
   - kGcCauseExplicit 频繁 → 手动 GC
   - kGcCauseForTrim 频繁 → 后台 trim

4. GcCause → 优化方向
   - kGcCauseForAlloc：减少对象分配
   - kGcCauseBackground：调大堆
   - kGcCauseForNativeAlloc：减少 Native 分配

详见 [06-JVMTI监控GC](06-JVMTI监控GC.md) §6.2。
```

### 9.7.7 指标 4：堆水位（AOSP 17 增强）

```
堆水位指标（AOSP 17 增强）：

1. Java 堆
   - Java Heap Total（kb）
   - Java Heap Used（kb）
   - Java Heap Free（kb）
   - Java Heap Max（kb）
   -【AOSP 17 新增】分代指标：
     - Young Gen Used / Size
     - Old Gen Used / Size
     - LOS Used / Size

2.【AOSP 17 增强】Native 堆
   - Native Heap Used（kb）
   - Native Heap Max（kb）
   -【AOSP 17 新增】Native 堆分类：
     - sheaves slab（Linux 6.12 新增，15-20% 节省）
     - malloc
     - mmap

3. 堆使用率
   - Java Heap Used / Java Heap Max
   - > 80% → 高风险
   - > 90% → 紧急

4.【AOSP 17 新增】软阈值触发距离
   - Distance to soft threshold（30%）
   - > -5% → 软阈值即将触发
   -【AOSP 17 新增】Distance to hard threshold（80%）

5. 增长趋势
   - 堆随时间增长
   - 持续增长 → 可能的泄漏
```

## 三、ART 17 新增指标族

### 9.7.8 【AOSP 17 新增】ART 内部状态指标族

AOSP 17 新增 ART 内部状态指标，通过 dumpsys meminfo -d 或 JVMTI 获取：

```
ART 内部状态指标（AOSP 17 新增）：

1. JIT 状态
   - jit.code_cache.size（JIT Code Cache 大小）
   - jit.code_cache.max（Code Cache 上限）
   - jit.compiled_methods（JIT 已编译方法数）
   → Code Cache 满会导致 JIT 失效

2. ClassLoader 状态
   - classloader.loaded_classes（已加载类数）
   - classloader.count（ClassLoader 实例数）
   → ClassLoader 泄漏是常见 OOM 根因

3. JNI 引用
   - jni.global_refs（Global refs 数）
   - jni.local_refs（Local refs 数）
   - jni.weak_global_refs（Weak global refs 数）
   → JNI 引用泄漏 = 引用泄漏

4. GC 累计
   - gc.cumulative_count（累计 GC 次数）
   - gc.cumulative_time（累计 GC 耗时）
   - gc.last_type（上次 GC 类型）
   - gc.last_time（上次 GC 时间）
```

**架构师解读**：
- **可观测性增强**：AOSP 14 看不到这些指标，只能从 ART 内部估算
- **泄漏排查利器**：JNI Global Refs 是 JNI 泄漏的直接证据
- **JIT 失效监控**：Code Cache 满会导致 JIT 失效，性能下降

**采集方式**：
- dumpsys meminfo -d（开发/测试）
- JVMTI GetGarbageCollectionExtendedInfo（生产）

详见 [01-dumpsys-meminfo详解](01-dumpsys-meminfo详解.md) §6.1。

### 9.7.9 【AOSP 17 新增】软阈值指标

AOSP 17 新增软阈值指标：

```
软阈值指标（AOSP 17 新增）：

1. soft_threshold.percent（kSoftThresholdPercent=30%）
   - 默认 30%（AOSP 17）
   - 触发 Young GC（轻量、频繁、暂停 < 1ms）

2. hard_threshold.percent（80%）
   - 默认 80%
   - 触发 Full GC（重量、罕见、暂停 5-20ms）

3. distance_to_soft（软阈值距离）
   - > -5% → 软阈值即将触发
   - 立即告警

4. distance_to_hard（硬阈值距离）
   - > -10% → 硬阈值即将触发
   - 紧急告警

5. soft_threshold_reached_count（软阈值触发次数）
   - 频繁触发 → 堆太小或分配过快

6. last_soft_threshold_time（上次软阈值触发时间）
   - 用于趋势分析
```

**架构师解读**：
- **软阈值是"轻量预警"**——Young GC 触发，暂停 < 1ms
- **硬阈值是"紧急预警"**——Full GC 触发，暂停 5-20ms
- **Distance 字段**：明确告诉运维"距离下次 GC 触发还有多远"

**采集方式**：
- dumpsys meminfo -d（显示 "Distance to soft threshold"）
- JVMTI GetGarbageCollectionExtendedInfo（AOSP 17 新增）
- Perfetto SoftThresholdReached 事件

详见 [10-ART17分代GC强化专章 v2](../../03-GC系统/10-ART17分代GC强化专章-v2.md) §3。

### 9.7.10 【AOSP 17 新增】Native 堆分类指标

AOSP 17 + Linux 6.12 新增 Native 堆分类：

```
Native 堆分类指标（AOSP 17 + Linux 6.12 新增）：

1. native.sheaves.slab（AOSP 17 + Linux 6.12 新增）
   - sheaves slab 占用
   - 小对象（< 8KB）的 slab 缓存
   - 节省 15-20% 内存

2. native.malloc
   - 传统 malloc 占用
   - 大对象（> 8KB）

3. native.mmap
   - mmap 分配
   - DirectByteBuffer 等

4. native.total
   - Native 堆总占用 = sheaves + malloc + mmap

5.【AOSP 17 增强】native.growth_rate
   - Native 堆增长率（KB/分钟）
   - 持续增长 → 可能的 native 泄漏
```

**架构师解读**：
- **sheaves slab 优势**：频繁分配/释放的小对象不产生碎片
- **节省 15-20%**：Linux 6.12 sheaves 让 Native 堆内存自然下降
- **不要把 sheaves 优化误判为泄漏修复**

详见 [02-procrank-smaps](02-procrank-smaps.md) §6.1。

## 四、衍生指标

### 9.7.11 GC 分配速率

```
GC 分配速率（Allocation Rate）：

定义：每秒分配多少对象内存
计算：gc.allocated_bytes / gc.time_window

正常范围：
- < 10 MB/s（轻量）
- 10-50 MB/s（中等）
- 50-100 MB/s（重度）
- > 100 MB/s（异常）

【AOSP 17 增强】
- 通过 JVMTI GetGarbageCollectionExtendedInfo 获取 freed_bytes
- 计算：allocation_rate = freed_bytes / gc_interval

→ 分配速率过高 → GC 频率会高 → 影响性能
```

### 9.7.12 GC 暂停占比

```
GC 暂停占比（GC Pause Ratio）：

定义：GC 暂停时间占运行时间的比例
计算：sum(gc.pause) / total.time

正常范围：
- < 1%（健康）
- 1-5%（可接受）
- 5-10%（需要优化）
- > 10%（异常）

→ GC 暂停占比高 → 直接影响 App 性能
```

### 9.7.13 OOM 指标

```
OOM 指标：

1. OOM 次数（oom.count）
2. OOM 时的堆大小（oom.heap.size）
3. OOM 时的 GcCause（oom.cause）
4. OOM 时的内存快照（可选）

【AOSP 17 增强】
- OOM 时的 ART 内部状态（dumpsys meminfo -d）
- OOM 时的软阈值距离（应该非常接近 0）

→ OOM 是 GC 的极端失败 → 必须监控
```

### 9.7.14 晋升速率指标（AOSP 17 新增）

```
晋升速率指标（AOSP 17 新增）：

定义：每秒从 Young 晋升到 Old 的对象数
计算：promoted_objects / gc_interval

【AOSP 17 增强】
- 通过 JVMTI GetGarbageCollectionExtendedInfo 获取 promoted_objects
- GenCC 暴露晋升数

正常范围：
- < 100 objects/s（轻量）
- 100-1000 objects/s（中等）
- > 1000 objects/s（异常 → 长期持有对象过多）

→ 晋升速率高 → Old 代会快速填满 → 触发 Full GC
```

### 9.7.15 Finalize 与 Reference 指标（AOSP 17 增强）

```
Finalize 与 Reference 指标（AOSP 17 增强）：

1. Finalize 耗时（finalize.duration）
2. FinalizerDaemon 队列长度（finalizer.queue.length）
3. Cleaner 队列长度（cleaner.queue.length）

【AOSP 17 增强】
- Finalizer 池化（4 线程）后，队列长度变化
- 可监控 finalizer.queue.length 异常

→ Finalizer 队列堆积 → 可能影响 GC 性能
→ AOSP 17 池化后，堆积风险降低
```

## 五、采集实现

### 9.7.16 JVMTI 采集（AOSP 17 增强版）

```cpp
//【AOSP 17 增强】JVMTI 采集
static int gc_count = 0;
static int young_gc_count = 0;  //【AOSP 17 新增】
static int old_gc_count = 0;    //【AOSP 17 新增】
static long total_pause_time = 0;
//【AOSP 17 新增】软阈值统计
static int soft_threshold_reached = 0;

void JNICALL OnGCStart(jvmtiEnv* env) {
    g_gc_start_time = currentTimeMillis();
}

void JNICALL OnGCFinish(jvmtiEnv* env) {
    long pause_time = currentTimeMillis() - g_gc_start_time;
    gc_count++;
    total_pause_time += pause_time;
    
    //【AOSP 17 新增】获取 GC 扩展元数据
    jvmtiGarbageCollectionInfo gc_info = {0};
    env->GetGarbageCollectionExtendedInfo(&gc_info);
    
    if (gc_info.gc_type == GC_TYPE_YOUNG) young_gc_count++;
    if (gc_info.gc_type == GC_TYPE_OLD) old_gc_count++;
    
    //【AOSP 17 新增】软阈值统计
    if (gc_info.distance_to_soft_threshold > -5) {
        soft_threshold_reached++;
    }
}

// 定时上报
void report() {
    apmClient.report("gc.count", gc_count);
    apmClient.report("gc.young.count", young_gc_count);  //【AOSP 17】
    apmClient.report("gc.old.count", old_gc_count);      //【AOSP 17】
    apmClient.report("gc.pause.avg", total_pause_time / gc_count);
    apmClient.report("gc.pause.total", total_pause_time);
    apmClient.report("soft.threshold.reached", soft_threshold_reached);  //【AOSP 17】
}
```

public class SoftThresholdAlerter {
    public void check(GcMetrics metrics) {

        if (metrics.softThresholdDistance > -5) {

                "Soft threshold triggered " + metrics.softThresholdReached + " times in 1 min");
        }

        if (metrics.hardThresholdDistance > 0) {
            // 硬阈值已越过
            apmClient.alert("gc.hard.threshold.exceeded", 
                "Hard threshold exceeded");
        }
    }
}

## 十一、总结（架构师视角的 5 条 Takeaway）

1. **监控指标体系是 GC 治理的"度量衡"**——四大核心族（频率/时长/原因/堆水位）+ 衍生指标。**没有指标就没有治理**——所有优化都要有量化依据。详见 §1 + [08-治理工具箱](08-治理工具箱.md)（重写为 v2 升级版）。

2. **AOSP 17 dumpsys 增强是最大亮点**——ART Internal State 段（GC/JIT/ClassLoader/JNI refs 全暴露）+ 软阈值 kSoftThresholdPercent=30% 状态显示。**AOSP 14 看不到这些数据**。详见 §6.1 + [01-dumpsys-meminfo详解](01-dumpsys-meminfo详解.md) §6.1。

3. **AOSP 17 软阈值 kSoftThresholdPercent=30% 是 GC 监控的"轻量预警"**——明确告诉运维"距离下次 GC 触发还有多远"。**软阈值是 Young GC，硬阈值是 Full GC**——别把软阈值当 OOM 信号。详见 §6.2 + [10-ART17分代GC强化专章 v2](../../03-GC系统/10-ART17分代GC强化专章-v2.md) §3。

4. **AOSP 17 Native 堆分类让 Native 内存分析更精准**——sheaves slab 单独统计（Linux 6.12 新增，15-20% 节省）。**Native 堆下降不一定是泄漏修复**——可能是 sheaves 优化。详见 §6.3 + [02-procrank-smaps](02-procrank-smaps.md) §6.1。

5. **AOSP 17 ART 内部状态指标是 JNI 泄漏排查利器**——JNI Global Refs > 1000 告警、> 5000 紧急。**AOSP 14 看不到 JNI refs 数字**——AOSP 17 通过 dumpsys meminfo -d 直接暴露。详见 §6.4 + 附录 A 源码索引。
