# 2.3 内存配额：growth_limit vs max_heap vs largeHeap

> **本节回答一个根本问题**：ART 的 Java 堆大小是怎么决定的？`dalvik.vm.heapgrowthlimit`、`dalvik.vm.heapsize`、`largeHeap` 三个参数怎么配合？
>
> **答案**：通过 **Heap::Heap() 构造函数** 的多级参数解析，按 **是否 largeHeap → 是否系统进程 → 是否定制** 的优先级决定。
>
> **理解本节，就理解了"为什么误用 largeHeap 反而被 LMK 杀得更快"**。

---

## 一、三个核心参数

### 2.3.1 参数对照表

| 参数 | 含义 | 默认值 | 生效条件 |
|:---|:---|:---|:---|
| `dalvik.vm.heapgrowthlimit` | 普通进程堆增长上限 | 256MB | 永远生效 |
| `dalvik.vm.heapsize` | `largeHeap=true` 时的堆上限 | 512MB | 仅 `largeHeap=true` |
| `dalvik.vm.heaptargetutilization` | 目标使用率 | 0.75 | 永远生效 |
| `dalvik.vm.heapminfree` | 最小空闲 | 2MB | 永远生效 |
| `dalvik.vm.heapmaxfree` | 最大空闲 | 8MB | 永远生效 |

### 2.3.2 三个参数的关系

```
┌─────────────────────────────────────────────────────┐
│                  Java Heap 配额                       │
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │  max_allowed_footprint_                       │    │
│  │  = max(heapgrowthlimit, heapsize)              │    │
│  │                                               │    │
│  │  ┌──────────────────────────────────────────┐ │    │
│  │  │  实际堆使用范围                           │ │    │
│  │  │  = growth_limit_ × heaptargetutilization  │ │    │
│  │  │  (默认: 256MB × 0.75 = 192MB)             │ │    │
│  │  │                                           │ │    │
│  │  │  - 最小空闲: heapminfree = 2MB            │ │    │
│  │  │  - 最大空闲: heapmaxfree = 8MB            │ │    │
│  │  └──────────────────────────────────────────┘ │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 2.3.3 参数优先级

| 进程类型 | 是否 `largeHeap=true` | 堆上限 | 备注 |
|:---|:---|:---|:---|
| 普通 App | false | `heapgrowthlimit`（256MB） | 常见情况 |
| 普通 App | true | `heapsize`（512MB） | 需 manifest 声明 |
| 系统服务 | — | 系统专用配置 | 如 `system_server` 特殊 |
| 厂商定制 | — | 厂商覆盖 | 可能在 prop 中调整 |

## 二、Heap::Heap() 的参数解析

### 2.3.4 Heap 构造函数的参数解析

```cpp
// art/runtime/gc/heap.cc 的 Heap::Heap 构造函数（精简版）
Heap::Heap(const RuntimeOptions& runtime_options, const std::vector<std::string>& boot_class_path, ...) {
    // 1. 读取 dalvik.vm.heapgrowthlimit
    {
        std::string heap_growth_limit;
        Runtime::GetCurrent()->GetSystemProperty("dalvik.vm.heapgrowthlimit", &heap_growth_limit);
        ParseAbsoluteMaxMemory(heap_growth_limit, &growth_limit_);
    }
    
    // 2. 读取 dalvik.vm.heapsize
    {
        std::string heap_size;
        Runtime::GetCurrent()->GetSystemProperty("dalvik.vm.heapsize", &heap_size);
        size_t size_value = 0;
        ParseAbsoluteMaxMemory(heap_size, &size_value);
        if (size_value != 0) {
            max_allowed_footprint_ = size_value;  // 仅当配置了才覆盖
        }
    }
    
    // 3. 如果 heapsize 没配置，但 growth_limit 有，用 growth_limit
    if (max_allowed_footprint_ == 0) {
        max_allowed_footprint_ = growth_limit_;
    }
    
    // 4. 读取 dalvik.vm.heaptargetutilization
    {
        std::string target_utilization;
        Runtime::GetCurrent()->GetSystemProperty("dalvik.vm.heaptargetutilization", &target_utilization);
        target_utilization_ = std::stof(target_utilization);  // 默认 0.75
    }
}
```

### 2.3.5 配额计算的核心公式

```cpp
// 计算初始堆大小
size_t initial_heap_size = growth_limit_ * target_utilization_;

// 默认: 256MB × 0.75 = 192MB

// 计算最大堆大小
size_t max_heap_size = max_allowed_footprint_;

// 普通 App: 256MB
// largeHeap App: 512MB
```

### 2.3.6 配额计算的源码

```cpp
// art/runtime/gc/heap.cc 的 Heap::CalculateGrowthLimit 简化版
size_t Heap::CalculateGrowthLimit() {
    // 1. growth_limit 来自 dalvik.vm.heapgrowthlimit
    size_t growth_limit = growth_limit_;
    
    // 2. 按 utilization 计算
    size_t utilization = target_utilization_ * 100;  // 75
    
    // 3. 计算最大堆大小（按需扩展）
    size_t max_allowed = max_allowed_footprint_;
    
    // 4. 检查是否需要扩展
    if (current_footprint_ < max_allowed * utilization / 100) {
        // 当前堆使用率低，可以扩展
        size_t new_footprint = std::min(max_allowed, 
                                         current_footprint_ * 2);
        return new_footprint;
    }
    
    // 5. 已达上限
    return current_footprint_;
}
```

## 三、largeHeap 的代价

### 2.3.7 largeHeap 的开启方式

```xml
<!-- AndroidManifest.xml -->
<application
    android:largeHeap="true"
    ...>
```

### 2.3.8 largeHeap 的代价

**代价 1：被 LMK 杀得更快**

```cpp
// system/core/lmkd/lmkd.c 的 LMK 杀进程决策（简化版）
// 普通 App: oom_score_adj = 0 ~ 1000 (随进程优先级)
// largeHeap App: oom_score_adj + 200 (因为占用更多内存)
```

LMK（Low Memory Killer）的杀进程优先级：
- **进程内存越大，oom_score_adj 越高 → 越先被杀**
- largeHeap App 占用更多内存 → **更容易被杀**

### 2.3.9 largeHeap 的代价对比

| 维度 | 普通 Heap | largeHeap |
|:---|:---|:---|
| 堆大小 | 256 MB | 512 MB |
| OOM 风险 | 中（256MB 满） | 低（512MB 不易满） |
| LMK 风险 | 低（占用少） | **高（占用多 → 先杀）** |
| GC 扫描 | 快（堆小） | 慢（堆大） |
| GC 频率 | 高（堆小 → 频繁 GC） | 低（堆大 → 偶尔 GC） |
| 启动速度 | 快（堆预分配小） | 慢（堆预分配大） |

**核心矛盾**：largeHeap 减少 OOM 风险，但 **增加被 LMK 杀死的风险**。

### 2.3.10 什么场景才该用 largeHeap？

| 场景 | 是否 largeHeap | 理由 |
|:---|:---|:---|
| 普通 App | ❌ 否 | 默认 256MB 足够 |
| 视频编辑 App | ✅ 是 | 处理大视频需要大堆 |
| 图片编辑 App | ✅ 是 | 处理大 Bitmap |
| 游戏 App | ⚠️ 视情况 | 取决于资源大小 |
| 浏览器 App | ⚠️ 视情况 | 多 Tab 时内存需求大 |
| 工具类 App | ❌ 否 | 默认足够 |

**经验法则**：能用 Bitmap 复用、对象池、内存缓存解决的，**绝不用 largeHeap**。

## 四、配额的运行时调整

### 2.3.11 堆扩展（Grow Heap）

```cpp
// art/runtime/gc/heap.cc 的 Heap::ChangeSoftReferenceLimit 简化版
bool Heap::ChangeSoftReferenceLimit(size_t new_footprint) {
    // 1. 检查是否超过上限
    if (new_footprint > max_allowed_footprint_) {
        return false;
    }
    
    // 2. 检查是否需要扩展
    if (new_footprint <= growth_limit_) {
        return false;
    }
    
    // 3. 调整 SoftReference 阈值
    // SoftReference 的回收阈值 = heap_used_ratio
    // 调大阈值 → SoftReference 更容易保留
    // 调小阈值 → SoftReference 更容易回收
    
    size_t new_threshold = (new_footprint - growth_limit_) * 
                           target_utilization_;
    
    soft_ref_threshold_ = std::min(soft_ref_threshold_, new_threshold);
    
    return true;
}
```

### 2.3.12 堆收缩（Trim Heap）

```cpp
// art/runtime/gc/heap.cc 的 Heap::Trim 简化版
void Heap::Trim() {
    // 1. 计算目标堆大小
    size_t target = current_footprint_ * target_utilization_;
    
    // 2. 收缩堆
    if (current_footprint_ > target) {
        // 3. 调整 SoftReference 阈值
        ChangeSoftReferenceLimit(target);
        
        // 4. 触发 GC 释放内存
        CollectGarbage(kGcCauseTrim, ...);
        
        // 5. 归还内存给系统
        allocation_space_->Trim();
    }
}
```

### 2.3.13 配额与 SoftReference 的关系

SoftReference 的回收时机由 `dalvik.vm.softrefthreshold` 决定：

```
保留率 = (heap_used / heap_max - threshold) / (1 - threshold)

其中：
- heap_used：当前堆使用量
- heap_max：当前堆上限
- threshold：dalvik.vm.softrefthreshold（默认 0.25）
```

```
示例：
- heap_used = 100 MB
- heap_max = 200 MB
- threshold = 0.25
- 保留率 = (100/200 - 0.25) / (1 - 0.25) = 0.33

→ SoftReference 对象有 33% 概率被保留
```

$ adb shell dumpsys meminfo com.example.app

                       Pss    Private   Private   SwapPss      Rss     Heap     Heap     Heap
                     Total    Dirty    Clean    Dirty    Total     Size    Alloc     Free
  Dalvik Heap      45678    40000     5678      200    51234    65536    45678    19858

### 2.3.15 Heap 字段的含义

| 字段 | 含义 | 警戒线 |
|:---|:---|:---|
| **Heap Size** | 当前堆总大小 | 接近 `max_allowed_footprint` 时告警 |
| **Heap Alloc** | 已分配（使用中） | 接近 `Heap Size` 时告警 |
| **Heap Free** | 空闲 | 接近 0 时频繁 GC |

### 2.3.16 配额告警阈值

| 指标 | 警告 | 严重 |
|:---|:---|:---|
| Heap Alloc / Heap Size | > 70% | > 85% |
| Heap Alloc / max_allowed_footprint | > 60% | > 80% |
| GC 频率（次/分钟） | > 10 | > 30 |
| GC STW 时间（ms） | > 10 | > 50 |

## 六、配额的厂商定制

### 2.3.17 常见厂商配置

```bash
# 小米 (MIUI)
dalvik.vm.heapgrowthlimit=256m
dalvik.vm.heapsize=512m
dalvik.vm.heaptargetutilization=0.75

# 华为 (EMUI)
dalvik.vm.heapgrowthlimit=192m  # 较小
dalvik.vm.heapsize=384m
dalvik.vm.heaptargetutilization=0.7  # 较低（更激进 GC）

# 三星 (OneUI)
dalvik.vm.heapgrowthlimit=256m
dalvik.vm.heapsize=512m
dalvik.vm.heaptargetutilization=0.75

# Pixel (原厂)
dalvik.vm.heapgrowthlimit=256m
dalvik.vm.heapsize=512m
dalvik.vm.heaptargetutilization=0.75
```

### 2.3.18 厂商定制的副作用

| 定制点 | 影响 |
|:---|:---|
| 调小 `heapgrowthlimit` | 更易 OOM，更不易被 LMK 杀 |
| 调小 `heapsize`（largeHeap） | largeHeap 效果减弱 |
| 调小 `heaptargetutilization` | 堆更早收缩，GC 更频繁 |
| 调大 `heapgrowthlimit` | 不易 OOM，更易被 LMK 杀 |

→ **ROM 优化方向各有不同**，需要针对性适配。

## 八、配额与 Zygote fork 的关系

### 2.3.22 Zygote 预加载的堆大小

```cpp
// art/runtime/gc/heap.cc 的 Zygote 初始化（精简版）
void Heap::CreateZygoteSpace() {
    // 1. Zygote 进程启动时，growth_limit 是固定值
    size_t zygote_growth_limit = 256 * MB;
    
    // 2. Zygote Space 从 boot.art 加载
    zygote_space_ = ZygoteSpace::Create("boot.art", zygote_growth_limit);
    
    // 3. Zygote 进程堆预分配
    AllocateZygoteSpace(zygote_growth_limit);
}
```

### 2.3.23 fork 后子进程的配额继承

```
Zygote 进程 (init 时):
  growth_limit = 256 MB
  max_allowed_footprint = 256 MB (普通)

fork() 后:
  App 进程 1: 继承所有配额，但有自己的实际堆使用
  App 进程 2: 同上
  ...

→ 子进程的 quota 不变，但内存使用从 0 开始
```

## 九、配额与 Native 内存的关系

### 2.3.24 Java 堆 vs Native 内存

```bash
$ adb shell dumpsys meminfo com.example.app

  Native Heap      12345     6789     1234      100    15000
  Dalvik Heap      45678    40000     5678      200    51234
  ...
```

| 维度 | Native Heap | Dalvik Heap |
|:---|:---|:---|
| 控制者 | libc malloc / .so | ART |
| 增长上限 | 无（受 LMK 限制） | `max_allowed_footprint` |
| 释放机制 | 手动释放 / GC 间接 | 自动 GC |
| 监控方式 | `dumpsys meminfo` | `dumpsys meminfo` |
| LMK 杀进程 | 考虑总内存 | 考虑总内存 |

→ **Java 堆大小影响 Native 内存**（堆用 mmap 分配）。

### 2.3.25 真实内存占用

```bash
# 一个 App 的真实内存占用 = Dalvik Heap + Native Heap + Graphics + Stack + ...

# 普通 App:
#   Dalvik: ~50 MB
#   Native: ~80 MB
#   Graphics: ~30 MB
#   Stack: ~5 MB
#   Other: ~10 MB
#   总计: ~175 MB

# largeHeap App:
#   Dalvik: ~200 MB (用了更多堆)
#   Native: ~120 MB (堆大了，伴随 Native 也多)
#   Graphics: ~50 MB
#   Stack: ~10 MB
#   Other: ~20 MB
#   总计: ~400 MB (LMK 更敏感)
```

## 十、配额的稳定性排查

### 2.3.26 案例 1：误用 largeHeap 导致频繁被杀

**场景**：
```xml
<!-- 普通工具 App 误用 largeHeap -->
<application android:largeHeap="true">
```

**症状**：
- App 在内存压力大的设备上频繁被杀
- 用户反馈"App 在后台待不久"

**排查**：
```bash
# 1. 查看 App 是否被 LMK 杀
adb logcat | grep "Lowmemorykiller"
# 看到 "Killing 'com.example.app' (12345)" → LMK 杀进程

# 2. 查看 App 的内存占用
adb shell dumpsys meminfo com.example.app | grep "TOTAL"
# TOTAL PSS: 350 MB (比其他工具类 App 高 2-3 倍)

# 3. 确认是 largeHeap 导致
# 对比同类型 App（不开 largeHeap）的内存占用
```

**修复**：
```xml
<!-- 关闭 largeHeap -->
<application android:largeHeap="false">
```

### 2.3.27 案例 2：堆配置过小导致频繁 GC

**场景**：
```bash
# 厂商 ROM 调小了 heapgrowthlimit
dalvik.vm.heapgrowthlimit=128m  # 比默认 256m 小
```

**症状**：
- App 频繁 GC（> 30 次/分钟）
- 卡顿明显

**排查**：
```bash
# 1. 抓取 systrace
adb shell perfetto ...

# 2. 看 GC 频率
# 找到 GC 事件，统计 1 分钟内 GC 次数
# 假设: 35 次/分钟 → 过高

# 3. 看堆使用
adb shell dumpsys meminfo com.example.app | grep "Dalvik Heap"
# Heap Size: 65 MB（远小于 128 MB → 频繁扩展触发 GC）
```

**修复**：
- 与厂商沟通，调大 `heapgrowthlimit`
- 减小 App 的堆使用（对象池 / LRU 缓存）

### 2.3.28 案例 3：SoftReference 配置导致缓存丢失

**场景**：
```bash
# 厂商 ROM 调小了 softrefthreshold
dalvik.vm.softrefthreshold=0.1  # 默认 0.25
```

**症状**：
- Glide Bitmap 缓存命中率低
- 频繁重新加载图片

**排查**：
```bash
# 1. 启用 ART 调试日志
adb shell setprop dalvik.vm.image-dex2oat-flags --debug
adb shell setprop dalvik.vm.usejit true
adb shell setprop dalvik.vm.dex2oat-Xms 256m

# 2. 看 GC 日志
adb logcat | grep "soft reference"
# 看到 "soft reference cleared" 频繁 → SoftRef 提前释放
```

**修复**：
- 减少 Glide 的内存缓存（从 50% 调到 25%）
- 或与厂商沟通，调整 `softrefthreshold`
