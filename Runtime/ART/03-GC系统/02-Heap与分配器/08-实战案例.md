# 2.8 实战案例：LOS 碎片化导致大 Bitmap 分配失败

> **本节是 02 篇的"压轴案例"** —— 用一个完整的真实案例，把前 7 节的所有知识串起来。
>
> **理解本节，就理解了"Java 堆 OOM 排查"的完整流程**。

---

## 一、案例背景

### 2.8.1 案例信息

| 维度 | 信息 |
|:---|:---|
| **App** | 某图片编辑器 |
| **Android 版本** | Android 12 (API 31) |
| **设备** | Pixel 5 |
| **问题** | 编辑 4K 图片时 OOM 崩溃 |
| **OOM 类型** | Java heap OOM（看似） |
| **真实根因** | LOS 碎片化 |

### 2.8.2 问题描述

用户反馈：
> 在编辑 4K 图片（3840×2160）时，App 偶发性崩溃，显示 `OutOfMemoryError: Failed to allocate a 4194304 byte allocation`。

但 dumpsys meminfo 显示 Java 堆还有很多空闲内存（看起来不可能 OOM）。

### 2.8.4 Step 2：分析数据

```
Dalvik Heap:
  Heap Size: 98304 KB (96 MB)
  Heap Alloc: 45000 KB (44 MB)  ← 使用了 44 MB
  Heap Free: 53304 KB (52 MB)   ← 空闲 52 MB
```

**看起来**：堆空闲 52 MB，分配 4 MB 应该没问题。

**实际**：OOM 发生了。

**矛盾！**

### 2.8.5 Step 3：分析 LOS

```bash
$ adb shell am gc  # 触发 GC

$ adb shell dumpsys meminfo com.example.imageeditor | grep -A 5 "LOS"
# 没有直接的 LOS 字段
```

**问题**：dumpsys meminfo 不显示 LOS 详细分布。

**解决方案**：使用 ART 调试命令。

### 2.8.6 Step 4：使用 art 命令查看 LOS

```bash
$ adb shell cmd activity dumpheap <pid> /data/local/tmp/large_objects.hprof
$ adb pull /data/local/tmp/large_objects.hprof

# 用 hprof-conv 转换
$ hprof-conv large_objects.hprof large_objects-conv.hprof

# 用 MAT 打开
# 找到 LOS 中的大对象（> 12 KB）
```

### 2.8.7 Step 5：分析 LOS 大对象

用 MAT 分析 hprof，发现 LOS 中的 Bitmap 分布：

```
LOS 中的 Bitmap（按大小排序）：
┌────────────────────────────────────────────────┐
│  Bitmap A (8 MB) - 缩略图缓存 1                │
│  Bitmap B (12 MB) - 原始图片                   │
│  [FREE 1 MB]                                   │
│  Bitmap C (4 MB) - 编辑缩略图                  │
│  [FREE 2 MB]                                   │
│  Bitmap D (16 MB) - 4K 编辑图层                │
│  [FREE 1 MB]                                   │
│  Bitmap E (8 MB) - 缩略图缓存 2                │
│  [FREE 0.5 MB]                                 │
│  Bitmap F (4 MB) - 滤镜预览                    │
└────────────────────────────────────────────────┘
总占用：52 MB
最大连续空洞：2 MB
```

### 2.8.8 Step 6：找到根因

```
需要分配的 Bitmap G: 4 MB（编辑图层）
LOS 状态：
  - Bitmap A (8 MB) 已用
  - [FREE 1 MB] 空洞
  - Bitmap C (4 MB) 已用
  - [FREE 2 MB] 空洞  ← 最大空洞
  - Bitmap D (16 MB) 已用
  - [FREE 1 MB] 空洞
  - Bitmap E (8 MB) 已用
  - [FREE 0.5 MB] 空洞
  - Bitmap F (4 MB) 已用

→ 最大空洞 2 MB < 需要 4 MB
→ 即使总空闲 4.5 MB，依然无法分配 4 MB 连续空间
→ OOM！
```

**根因**：**LOS 碎片化** —— 大量小空洞，最大空洞 < 单个 Bitmap 大小。

### 2.8.9 Step 7：确认业务代码

```java
// 用户的图片编辑器代码
public class ImageEditor {
    private List<Bitmap> layerCache = new ArrayList<>();
    
    public void addLayer(Bitmap layer) {
        layerCache.add(layer);  // Bitmap 进入 LOS
    }
    
    public void removeLayer(int index) {
        Bitmap removed = layerCache.remove(index);
        // ❌ 忘记调用 removed.recycle() 释放 native 像素
        // ❌ Bitmap 对象本身被 GC 回收，但 LOS 留下空洞
    }
}
```

**问题**：
- 删除图层时 **忘记 recycle()** → Bitmap 对象被 GC 回收，但 LOS 留下空洞
- 长期运行后，LOS 充满小空洞
- 即使总空闲足够，也无法分配大 Bitmap

---

## 三、修复方案

### 2.8.10 方案 1：及时回收 Bitmap

```java
// 修复后
public void removeLayer(int index) {
    Bitmap removed = layerCache.remove(index);
    if (removed != null && !removed.isRecycled()) {
        removed.recycle();  // 立即释放 native 像素
        // Bitmap 对象本身在下次 GC 时被回收 → LOS 空间也释放
    }
}
```

### 2.8.11 方案 2：使用 inBitmap 复用

```java
// 修复后：复用已有的 Bitmap
public void addLayer(BitmapFactory.Options options) {
    // 检查是否能复用
    if (canUseInBitmap(layerCache, options)) {
        options.inBitmap = findReusableBitmap(layerCache, options);
    }
    Bitmap layer = BitmapFactory.decodeFile(path, options);
    layerCache.add(layer);
}

private boolean canUseInBitmap(List<Bitmap> cache, BitmapFactory.Options options) {
    // 检查是否有兼容的 Bitmap 可复用
    for (Bitmap bitmap : cache) {
        if (BitmapUtilities.canUseForInBitmap(bitmap, options)) {
            return true;
        }
    }
    return false;
}
```

### 2.8.12 方案 3：分块 Bitmap

```java
// 修复后：大 Bitmap 分块
public Bitmap createTiledBitmap(int width, int height) {
    int tileSize = 1024;
    int cols = (width + tileSize - 1) / tileSize;
    int rows = (height + tileSize - 1) / tileSize;
    
    Bitmap[] tiles = new Bitmap[cols * rows];
    for (int i = 0; i < tiles.length; i++) {
        tiles[i] = Bitmap.createBitmap(tileSize, tileSize, Bitmap.Config.ARGB_8888);
        // 每个 tile 只有 4 MB，更易分配和回收
    }
    return new TiledBitmap(tiles, width, height);
}
```

### 2.8.13 方案 4：限制 LOS 大小

```java
// 修复后：主动管理 LOS
public class LRUBitmapCache {
    private LinkedHashMap<String, Bitmap> cache;
    private long maxLOSUsage;  // 限制 LOS 总占用
    private long currentLOSUsage;
    
    public void put(String key, Bitmap bitmap) {
        // 检查 LOS 占用
        long bitmapSize = bitmap.getByteCount();
        if (currentLOSUsage + bitmapSize > maxLOSUsage) {
            evictUntil(bitmapSize);  // 淘汰直到能放下
        }
        cache.put(key, bitmap);
        currentLOSUsage += bitmapSize;
    }
}
```

## 四、修复后的验证

### 2.8.14 验证 dumpsys meminfo

```bash
$ adb shell dumpsys meminfo com.example.imageeditor

  Dalvik Heap      23456    20000     3456       50    28000    65536    23456    42080
                                                              ↑        ↑       ↑
                                                              Size    Alloc   Free
                                                              64 MB   22 MB   41 MB (空闲)
```

**修复效果**：
- Heap Alloc 从 44 MB 降到 22 MB（及时回收）
- Heap Free 从 52 MB 增加到 41 MB
- LOS 占用从 52 MB 降到 12 MB

### 2.8.15 验证 Bitmap 复用

```bash
$ adb shell dumpsys meminfo com.example.imageeditor | grep "TOTAL"
  TOTAL PSS: 87654 KB  # 从 118 MB 降到 85 MB
```

**整体内存占用降低 30%**。

### 2.8.16 验证不 OOM

```bash
# 1. 重复编辑 4K 图片 50 次
# 2. 每次都成功，无 OOM
# 3. 内存占用稳定，不再单调增长
```

**根因解决**：LOS 不再有大量空洞，可以正常分配大 Bitmap。

## 五、案例总结

### 2.8.17 核心 takeaway

| Takeaway | 说明 |
|:---|:---|
| **OOM ≠ 堆用完** | 可能是 LOS 碎片化导致 |
| **dumpsys meminfo 不显示 LOS 详细分布** | 需要 hprof + MAT 分析 |
| **Bitmap 必须及时 recycle()** | 否则 LOS 留下空洞 |
| **大对象优先用 inBitmap 复用** | 减少 LOS 占用 |
| **大 Bitmap 考虑分块** | 避免单对象过大 |

### 2.8.18 OOM 排查决策树

```
Java heap OOM
    │
    ▼
1. dumpsys meminfo 看 Heap Alloc / Heap Size
    │
    ├─── Heap Alloc ≈ Heap Size
    │    │
    │    ▼
    │   真实 OOM（堆用完）→ 检查泄漏
    │
    └─── Heap Alloc << Heap Size
         │
         ▼
        2. 生成 hprof
         │
         ├─── 3. 用 MAT 分析 LOS 大对象
         │
         ├─── 4. 检查 Bitmap / byte[] / long[] 等
         │
         ├─── 5. 看最大空洞 vs 需要分配的大小
         │    │
         │    ├─── 最大空洞 < 需要 → LOS 碎片化
         │    │    │
         │    │    ▼
         │    │   - 检查 Bitmap 回收
         │    │   - 启用 inBitmap
         │    │   - 分块 Bitmap
         │    │
         │    └─── 最大空洞 >= 需要 → 其他原因
         │         │
         │         ▼
         │        - Allocation Space 满了
         │        - 线程 TLAB 不够
         │        - Region Pool 耗尽
```

### 2.8.19 完整排查命令

```bash
# 1. 看基本内存
adb shell dumpsys meminfo <package>

# 2. 看详细分类
adb shell dumpsys meminfo -d <package>

# 3. 生成 heap dump
adb shell am dumpheap <pid> /data/local/tmp/oom.hprof

# 4. 拉取 heap dump
adb pull /data/local/tmp/oom.hprof

# 5. 转换 hprof（Android 格式 → Java SE 格式）
hprof-conv oom.hprof oom-conv.hprof

# 6. 用 MAT 分析
# - Leak Suspects 报告
# - Top Consumers（找最大的对象）
# - Histogram（按类统计）
# - Dominator Tree（支配树）

# 7. 看 LOS 大对象
# MAT → Histogram → 过滤大小 > 12KB
# → 看 Bitmap / byte[] 等大对象
```
