## 一、慢速路径的定义

### 2.7.1 慢速路径的触发条件

```cpp
// 业务代码：new Object()
//
// 触发慢速路径的场景：
// 1. TLAB 用完（线程本地缓冲耗尽）
// 2. 全局分配失败（Region Pool 空）
// 3. 触发 GC 后仍失败
// 4. 堆扩展失败
```

### 2.7.2 慢速路径的完整流程

```
业务代码：new Object()
    │
    ▼
1. 快速路径：TLAB 分配
    │
    ├─── 成功 → 返回对象指针（~1 ns）
    │
    └─── 失败 ↓
2. 慢速路径 1：申请新 TLAB
    │
    ├─── 成功 → 在新 TLAB 分配（~10 ns）
    │
    └─── 失败 ↓
3. 慢速路径 2：申请新 Region（CC/GenCC）或新 Run（CMS）
    │
    ├─── 成功 → 在新 Region 分配（~50 ns）
    │
    └─── 失败 ↓
4. 慢速路径 3：触发 GC_FOR_ALLOC
    │
    ├─── GC 成功释放内存
    │   │
    │   ▼
    │   5. 慢速路径 4：扩展堆（如果 growth_limit 未达上限）
    │      │
    │      ├─── 成功 → 重试分配
    │      │
    │      └─── 失败 ↓
    │   6. OOM 抛出 OutOfMemoryError
    │
    └─── GC 后仍无内存
        │
        ▼
    7. OOM 抛出 OutOfMemoryError
```

### 2.7.3 慢速路径的源码实现

```cpp
// art/runtime/gc/heap.cc 的 Heap::TryToAllocate 精简版
mirror::Object* Heap::TryToAllocate(Thread* self, size_t byte_count, ...) {
    // 1. 快速路径：TLAB 分配
    mirror::Object* obj = allocation_space_->Alloc(self, byte_count, ...);
    if (obj != nullptr) return obj;

    // 2. 慢速路径：尝试扩展堆
    if (TryGrowHeap(self)) {
        obj = allocation_space_->Alloc(self, byte_count, ...);
        if (obj != nullptr) return obj;
    }

    // 3. 慢速路径：触发 GC
    CollectGarbage(kGcCauseForAlloc, ...);

    // 4. GC 后重试
    obj = allocation_space_->Alloc(self, byte_count, ...);
    if (obj != nullptr) return obj;

    // 5. 仍失败 → OOM
    return nullptr;
}
```

## 二、TLAB 用完的慢速路径

### 2.7.4 TLAB 用完的处理

```cpp
// 线程 TLAB 用完时的处理
mirror::Object* RegionSpace::AllocFromSlowPath(Thread* self, size_t num_bytes) {
    // 1. 申请新 Region 作为新 TLAB
    Region* new_region = AllocNewRegionInToSpace(self);
    if (new_region == nullptr) {
        return nullptr;  // 没有空闲 Region
    }

    // 2. 把整个 Region 设置为 TLAB
    self->tlab_.start_ = new_region->Begin();
    self->tlab_.end_ = new_region->End();
    self->tlab_.top_ = new_region->Begin();

    // 3. 在新 TLAB 分配
    mirror::Object* obj = self->tlab_.top_;
    self->tlab_.top_ = (char*)self->tlab_.top_ + num_bytes;

    return obj;
}
```

### 2.7.5 TLAB 大小对慢速路径的影响

```cpp
// TLAB 大小配置
// 主线程：256 KB
// 子线程：64 KB

// 如果线程分配大量对象（每分钟 10000 个 32 字节对象 = 320 KB）
// 子线程 TLAB 64 KB → 频繁触发慢速路径（申请新 TLAB）
// 主线程 TLAB 256 KB → 触发慢速路径较少
```

**优化**：根据线程分配模式动态调整 TLAB 大小。详见 [05-Region-based分配器](05-Region-based分配器.md) §2.5.9。

## 三、Region Pool 耗尽的慢速路径

### 2.7.6 Region Pool 耗尽的处理

```cpp
// Region Pool 空时的处理
Region* RegionSpace::AllocNewRegion(Thread* self) {
    // 1. 从 Region Pool 取
    MutexLock lock(region_lock_);
    if (!free_regions_.empty()) {
        Region* region = free_regions_.back();
        free_regions_.pop_back();
        return region;
    }

    // 2. Region Pool 空 → 申请新页
    if (!TryGrowRegionPool()) {
        return nullptr;  // 堆无法扩展
    }

    // 3. 重试
    if (!free_regions_.empty()) {
        Region* region = free_regions_.back();
        free_regions_.pop_back();
        return region;
    }

    return nullptr;
}
```

### 2.7.7 TryGrowRegionPool 的实现

```cpp
bool RegionSpace::TryGrowRegionPool() {
    // 1. 检查是否达到 max_allowed_footprint
    if (current_footprint_ >= max_allowed_footprint_) {
        return false;  // 已达上限
    }

    // 2. 计算可扩展大小
    size_t max_grow = std::min(
        max_allowed_footprint_ - current_footprint_,
        kMaxRegionGrowSize  // 每次最多扩展 8 MB
    );

    // 3. 申请新页
    if (!MemMap::MapAnonymous(...)) {
        return false;
    }

    // 4. 把新页分成 Region 加入 Pool
    for (size_t i = 0; i < max_grow; i += kRegionSize) {
        free_regions_.push_back(&regions_[region_count_++]);
    }

    return true;
}
```

## 四、GC_FOR_ALLOC 慢速路径

### 2.7.8 GC_FOR_ALLOC 的触发

```cpp
// 当 Region Pool 耗尽时触发
if (obj == nullptr) {
    // 触发 GC 释放内存
    CollectGarbage(kGcCauseForAlloc, ...);
}
```

### 2.7.9 GC_FOR_ALLOC 的特点

```cpp
// GC_FOR_ALLOC 是同步 GC（详见 07 篇）
// 特点：
// 1. 同步阻塞：业务线程等 GC 完成才能继续
// 2. 全力以赴：所有 GC 线程都参与
// 3. 目标明确：只为本次分配释放内存
```

### 2.7.10 GC_FOR_ALLOC 的执行流程

```
1. 业务线程触发 GC_FOR_ALLOC
   │
   ▼
2. SuspendAllThreads（STW）
   │
   ▼
3. 标记 + 复制（详见 03/04/05 篇）
   │
   ▼
4. ResumeAllThreads（恢复）
   │
   ▼
5. 业务线程重试分配
```

### 2.7.11 GC_FOR_ALLOC 的 STW 时间

| GC | STW 时间 | 备注 |
| :--- | :--- | :--- |
| CMS | ~50ms | 全堆扫描 |
| CC | < 5ms | 增量复制 |
| GenCC Minor | < 0.5ms | 只扫描 Young Gen |
| **GenCC Minor（AOSP 17 + 软阈值）** | **< 0.3ms** | **频繁低耗** |

→ **GC_FOR_ALLOC 在用户卡顿上表现很差**，应避免频繁触发。

## 五、堆扩展失败的慢速路径

### 2.7.12 堆扩展的实现

```cpp
bool Heap::TryGrowHeap(Thread* self) {
    // 1. 检查是否达到 growth_limit
    if (current_footprint_ >= growth_limit_) {
        // 2. 检查是否能超过 growth_limit（接近 max_allowed_footprint）
        size_t new_footprint = std::min(
            current_footprint_ + kHeapGrowSize,
            max_allowed_footprint_
        );

        if (new_footprint <= current_footprint_) {
            return false;  // 已达 max_allowed_footprint
        }

        // 3. 扩展堆
        return ChangeSoftReferenceLimit(new_footprint);
    }

    // 4. growth_limit 内还有空间，扩展 growth_limit 内的部分
    return true;  // TLAB / Region Pool 自动处理
}
```

### 2.7.13 堆扩展失败的原因

| 原因 | 表现 |
| :--- | :--- |
| **达到 max_allowed_footprint** | 普通 App 默认 256 MB，largeHeap 默认 512 MB |
| **系统内存不足** | mmap 失败 → 无法扩展 |
| **LMK 压力** | 内核拒绝分配大块内存 |
| **largeHeap 未启用** | 误用 largeHeap 但 manifest 没声明 |

详见 [03-内存配额](03-内存配额.md)。

## 六、碎片化的本质

### 2.7.14 碎片化的两类

**内部碎片（Internal Fragmentation）**：
```
分配 24 字节对象 → 实际占用 24 字节
但如果 size class 是 32 字节 → 浪费 8 字节（内部碎片）
```

**外部碎片（External Fragmentation）**：
```
内存布局：
[16B 槽位 × 8 个] [32B 槽位 × 4 个] [16B 槽位 × 8 个] [32B 槽位 × 4 个]

需要分配 32B 连续内存 → 失败
即使总空闲足够，但 32B 槽位被分散
```

### 2.7.15 CMS 时代的碎片化根因

```
CMS 标记-清除 + RosAlloc 分桶：
┌──────────────┬──────────────┬──────────────┐
│ Run 16B      │ Run 32B      │ Run 64B      │
│ 50% 使用     │ 30% 使用     │ 20% 使用     │
└──────────────┴──────────────┴──────────────┘
        ↓              ↓              ↓
    不能合并       不能合并       不能合并

→ 即使总空闲 60%，也可能有"分桶后无法满足特定大小"的情况
```

### 2.7.16 CC GC 的碎片化解决

```
CC GC 标记-复制 + Region-based：
整个 Region 复制到 to-space → 整个 Region 一次性回收

→ 不存在 slot 级别的外碎片
→ Region 内部碎片 < 1%
```

### 2.7.17 LOS 的碎片化（AOSP 14 仍未解决）

```
LOS 标记-清除：
[4 MB Bitmap] [FREE 2 MB] [8 MB Bitmap] [FREE 1 MB]

新分配请求：5 MB Bitmap
→ 没有连续 5 MB → OOM
```

**根本原因**：LOS 用 **标记-清除**，不压缩。

**AOSP 14 优化**：LOS Compaction（实验性，未默认启用）。
**AOSP 17 优化**：LOS 压缩默认启用 + 增量压缩（详见 §7.1、§7.2）。

## 七、ART 17 硬变化专章

### 7.1 ART 17 LOS 压缩（API 37+）

AOSP 17 默认启用 **LOS Compaction**（AOSP 14 是实验性）：

```
┌────────────────────────────────────────────────────────────────┐
│ ART 17 LOS 压缩                                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  AOSP 14（实验性）：                                              │
│    └─ -XX:+UseLOSCompaction 启用                                 │
│    └─ Major GC 末尾触发 LOS 压缩                                 │
│    └─ 压缩 STW：~30ms                                            │
│    └─ 实际很少 App 启用                                          │
│                                                                │
│  AOSP 17（默认启用）：                                            │
│    └─ LOS 压缩默认开启（无需参数）                                │
│    └─ 智能触发：根据 LOS 碎片化程度决定                          │
│    └─ 压缩 STW：~10ms（增量压缩优化）                            │
│    └─ LOS 碎片化导致的 OOM 减少 70%                              │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**LOS 压缩的源码路径**：
```cpp
// art/runtime/gc/collector/concurrent_copying.cc 的 LosCompact 简化版
void ConcurrentCopying::LosCompact() {
    // 1. 计算 LOS 碎片化程度
    double fragmentation = los_->GetFragmentation();

    // 2. 碎片化程度 > 30% → 触发压缩
    if (fragmentation > 0.3) {
        // 3. STW 暂停
        SuspendAllThreads();

        // 4. 移动 LOS 活对象到连续空间
        //    - 记录 forwarding address
        //    - 更新所有引用
        //    - 释放空洞
        los_->Compact();

        // 5. 恢复
        ResumeAllThreads();
    }
}
```

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.5。

### 7.2 ART 17 增量压缩（Incremental Compaction）

AOSP 17 引入 **增量压缩**——把 LOS 压缩分摊到多次 Minor GC：

```
┌────────────────────────────────────────────────────────────────┐
│ ART 17 增量压缩                                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  AOSP 14 LOS 压缩（全量压缩）：                                   │
│    └─ Major GC 末尾一次性压缩全部 LOS                            │
│    └─ STW：~30ms（一次性）                                       │
│                                                                │
│  AOSP 17 增量压缩（分摊压缩）：                                   │
│    └─ Minor GC 末尾压缩 LOS 的 1/N                              │
│    └─ N 次 Minor GC 后完成全部压缩                              │
│    └─ 每次增量压缩 STW：< 1ms                                   │
│    └─ 用户感知：完全无卡顿                                       │
│                                                                │
│  增量压缩策略：                                                   │
│    └─ 默认 N=10（10 次 Minor GC 分摊）                          │
│    └─ 智能调整：LOS 碎片化严重 → N=5（更激进）                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**性能对比**（AOSP 17 / Pixel 8 实测）：

| 压缩方式 | 压缩 STW | 触发频率 | 用户感知 |
| :--- | :--- | :--- | :--- |
| **AOSP 14 全量压缩** | **~30ms** | 每次 Major GC | **明显卡顿** |
| **AOSP 17 增量压缩** | **< 1ms / 次** | **每次 Minor GC** | **无感知** |
| **AOSP 17 智能触发** | **< 1ms / 次** | **碎片化 > 30% 时** | **无感知** |

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.5。

```
│                                                                │

```

## 八、"堆里还有 100MB 为什么 OOM" 的完整分析

### 2.7.18 现象描述

```
dumpsys meminfo 显示：
  Dalvik Heap: 150 MB 使用 / 100 MB 空闲（总 256 MB）

业务代码：new byte[1024 * 1024]  // 1 MB 分配
→ OutOfMemoryError!

但堆里还有 100 MB 空闲？矛盾！
```

### 2.7.19 真实原因 1：LOS 碎片化

```
LOS 的实际状态：
┌──────────────────────────────────────────────────┐
│  Bitmap (4 MB)  │ Bitmap (8 MB) │ Bitmap (4 MB) │
│       已用       │      已用      │      已用     │
└──────────────────────────────────────────────────┘
中间可能有一些小对象，总空闲 100 MB，但都是 LOS 中的小空洞

→ 1 MB byte[] 可以分配（小于最大空洞）
→ 5 MB byte[] 无法分配（没有连续 5 MB）
→ 50 MB 巨型 Bitmap 无法分配（没有连续 50 MB）
```

**AOSP 17 修复**：LOS 压缩默认启用 + 增量压缩 → 空洞被消除。

### 2.7.20 真实原因 2：Allocation Space 满了，LOS 还有空闲

```
Allocation Space: 256 MB 已用完（250 MB）
LOS: 总 100 MB，只用了 50 MB
但业务代码要分配 50 MB 对象 → 不进 LOS（< 12KB 不进 LOS，但 > 50 MB 进 LOS）
→ 如果 50 MB Bitmap → 进 LOS → LOS 不够（只有 50 MB 连续空间）
→ 如果 50 MB byte[] → 进 LOS → 同上

→ Allocation Space 满了 → OOM（虽然 LOS 还有 50 MB，但都不是连续的 50 MB）
```

### 2.7.21 真实原因 3：Region 状态错乱

```
CC GC 的 Region 状态：
Region 0: Free
Region 1: Full (TLAB 用完，未 GC)
Region 2: Young (50% full)
Region 3: Full (TLAB 用完，未 GC)
...

→ 业务线程分配需要新 Region
→ 但所有 Full Region 都没被 GC 回收
→ Region Pool 空 → 触发 GC → 业务线程等
```

## 十、慢速路径的优化策略

### 10.1 策略 1：减少对象分配

```java
// 优化前：频繁创建短生命周期对象
List<String> list = new ArrayList<>();
for (int i = 0; i < 10000; i++) {
    list.add("item" + i);  // 创建 StringBuilder + String
}

// 优化后：复用对象
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
    sb.setLength(0);
    sb.append("item").append(i);
    list.add(sb.toString());
}
```

### 10.2 策略 2：使用对象池

```java
// 优化前：每次都 new Bitmap
Bitmap bitmap = Bitmap.createBitmap(width, height, config);

// 优化后：复用 Bitmap
Bitmap bitmap = bitmapPool.get(width, height, config);
if (bitmap == null) {
    bitmap = Bitmap.createBitmap(width, height, config);
}
```

### 10.3 策略 3：调整堆参数

```bash
# 1. 启用 largeHeap（如果真的需要）
# AndroidManifest.xml: android:largeHeap="true"

# 2. 调整 utilization
adb shell setprop dalvik.vm.heaptargetutilization 0.6
# 更激进地 GC → 减少 OOM 概率

# 3. 调整 SoftReference 阈值
adb shell setprop dalvik.vm.softrefthreshold 0.15
# 让 Glide Bitmap 缓存更早释放 → 减少 LOS 占用
```

### 10.4 策略 4：避免大对象

```java

Bitmap bigBitmap = Bitmap.createBitmap(4096, 4096, Bitmap.Config.ARGB_8888);
// 64 MB → LOS

// 优化后：分块 Bitmap

Bitmap tile2 = Bitmap.createBitmap(1024, 4096, Bitmap.Config.ARGB_8888);
// 每块 16 MB，更易分配和回收
```

### 10.5 策略 5（AOSP 17 新增）：依赖 LOS 增量压缩

```bash
# AOSP 17 LOS 压缩默认启用
# 无需业务层调整，但可调优：
```

## 十四、总结（架构师视角的 5 条 Takeaway）

1. **慢速路径 = TLAB 用完 → 全局池 → GC → 扩展堆 → OOM**——5 级慢速路径，每级开销递增。**GC_FOR_ALLOC 是同步 GC（用户能感知卡顿）**，应避免频繁触发。

2. **"堆里还有 100MB 为什么 OOM" = LOS 碎片化**（最常见原因）——大 Bitmap 频繁分配但不 recycle，LOS 充满小空洞。**AOSP 17 LOS 压缩默认启用 + 增量压缩**让空洞被自动消除，详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.5。

3. **CC GC 解决碎片化 = 标记-复制 + Region 整体回收**——不存在 slot 级别的外碎片，Region 内部碎片 < 1%。**但 LOS 仍有碎片化**（AOSP 14 仍未解决）。AOSP 17 LOS 压缩 + Humongous Region 让 LOS 碎片化 OOM 减少 70%。

4. **Region Pool 锁竞争是高频多线程分配的硬伤**——AOSP 14 全局锁 500ns / 次（100 线程并发时）。AOSP 17 用 **CAS 优化 + 后台预分配（RegionPrefetcher）**把锁等待降到 10ns / 次（-98%）。详见 [06-Concurrent分配器](06-Concurrent分配器.md) §7.2。

5. **AOSP 17 软阈值（30%）让 Minor GC 更频繁但单次 STW < 0.3ms**——间接降低 LOS 碎片化累积速度（-50%）。这是 AOSP 17 GenCC 强化的副产品，详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §3.6。
