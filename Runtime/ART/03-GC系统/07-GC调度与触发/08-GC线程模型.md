# 7.8 GC 线程模型总图

> **本节回答一个根本问题**：ART GC 涉及的所有线程怎么协作？完整 GC 流程中各线程扮演什么角色？
>
> **答案**：**HeapTaskDaemon + ConcurrentCopying 线程 + FinalizerDaemon + ReferenceQueueDaemon + 业务线程** —— 5 类线程协作。

---

## 一、ART GC 线程全景

### 7.8.1 GC 涉及的所有线程

### 7.8.2 5 类线程的职责

| 线程 | 职责 | 触发时机 |
|:---|:---|:---|
| **业务线程** | 分配对象 / 执行 Java 代码 | 持续运行 |
| **HeapTaskDaemon** | 异步执行 GC | 接收 GC 任务 |
| **ConcurrentCopying** | GC 标记 + 复制 | GC 进行中 |
| **ReferenceQueueDaemon** | 处理 Reference + Cleaner | GC 后 / 持续 |
| **FinalizerDaemon** | 执行 finalize() | 持续 |

## 三、线程模型的工程影响

### 7.8.5 线程间的同步

```
ART GC 线程间的同步：

1. 业务线程 ↔ GC 线程
   - 通过 STW（SuspendAllThreads）
   - 通过读屏障（ReadBarrier）
   - 通过 Post-Write Barrier

2. HeapTaskDaemon ↔ GC 工作线程
   - 通过 task_queue
   - HeapTaskDaemon 调度，GC 线程执行

3. GC 线程 ↔ ReferenceQueueDaemon
   - 通过 pending list
   - GC 完成后通知 Reference 处理

4. ReferenceQueueDaemon ↔ FinalizerDaemon
   - 通过 Reference 类型
   - FinalReference 专门给 FinalizerDaemon
```

### 7.8.6 线程的性能影响

```
GC 线程对系统性能的影响：

1. 业务线程
   - 主要承担 UI / 业务逻辑
   - GC 影响：STW 时阻塞
   - 优化：使用后台 GC

2. HeapTaskDaemon
   - 独立线程，CPU 占用低
   - 主要等待 task 队列

3. ConcurrentCopying
   - 并发执行，与业务线程竞争 CPU
   - 优化：NUMA 亲和性

4. ReferenceQueueDaemon
   - 处理 Reference 时短暂占用
   - 优化：批量处理

5. FinalizerDaemon
   - 单线程，性能瓶颈
   - 优化：避免 finalize()
```

### 7.8.7 线程模型的调优

```bash
# 1. 调整 GC 线程优先级
adb shell setprop dalvik.vm.gc.priority -19  # 默认

# 2. 调整 GC 线程数量
adb shell setprop dalvik.vm.gc.threads 4  # 默认

# 3. 调整 Reference 处理线程优先级
adb shell setprop dalvik.vm.ref.priority -19
```

## 五、线程模型的工程价值

### 7.8.10 监控线程状态

```bash
# 1. 看所有线程
adb shell ps -T -p <pid>
# 输出示例：
# 12345 12346 12345 1 -19 0 0 0 HeapTaskDaemon
# 12345 12347 12345 1 -19 0 0 0 GC
# 12345 12348 12345 1 -19 0 0 0 ReferenceQueueDaemon
# 12345 12349 12345 1 -19 0 0 0 FinalizerDaemon
# 12345 12350 12345 1 -19 0 0 0 FinalizerWatchdogDaemon

# 2. 看 GC 线程 CPU 占用
adb shell top -H -p <pid>

# 3. 看各 daemon 的状态
adb logcat -s "art" | grep "Daemon"
```

### 7.8.11 线程异常的诊断

```
线程异常场景：

1. HeapTaskDaemon 一直不工作
   - 任务队列为空（正常）
   - 任务队列堆积（异常）
   - 检查：是否有 GC 任务被正确提交

2. ConcurrentCopying 卡住
   - STW 时间过长
   - 检查：堆大小、对象数量

3. FinalizerDaemon 卡死
   - finalize() 阻塞
   - 检查：是否有耗时的 finalize()
```

### 7.8.12 线程模型的演进

```
ART GC 线程模型的演进：

Android 5-7 (CMS):
  - GC 在业务线程执行
  - 没有独立的 HeapTaskDaemon

Android 8-9 (CC GC):
  - 引入 HeapTaskDaemon
  - CC GC 工作线程独立

Android 10+ (GenCC):
  - Minor GC + Major GC 分工
  - HeapTaskDaemon 更活跃

Android 14+ (GenCC + rbcc):
  - rbcc 优化
  - 线程调度更智能
```

## 六、本节小结

1. **5 类 GC 线程**：业务线程 / HeapTaskDaemon / ConcurrentCopying / ReferenceQueueDaemon / FinalizerDaemon
2. **线程协作**：通过 task_queue / pending list / Reference 等机制
3. **STW 协调**：通过 SuspendAllThreads / ResumeAllThreads
4. **完整流程**：GenCC Minor GC 和 CC Major GC 的线程时序
5. **监控**：线程状态 + CPU 占用 + 任务队列

→ **理解 GC 线程模型，就理解了"ART GC 的完整运作机制"**。
