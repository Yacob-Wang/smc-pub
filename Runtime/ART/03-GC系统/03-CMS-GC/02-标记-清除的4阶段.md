## 一、CMS 4 阶段总览

### 1.1 4 阶段时间分布

```
┌────────────────────────────────────────────────────────────┐
│                  CMS 4 阶段时间分布                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ┌──────────────┐                                          │
│  │Initial Mark  │ ← STW ~5ms                               │
│  │  (STW)       │                                          │
│  └──────┬───────┘                                          │
│         │                                                  │
│         ▼                                                  │
│  ┌──────────────────────────────┐                          │
│  │  Concurrent Mark             │ ← 业务线程并行，0ms STW │
│  │  (并发标记)                   │                          │
│  │  耗时：~100ms（业务线程并行） │                          │
│  └──────┬───────────────────────┘                          │
│         │                                                  │
│         ▼                                                  │
│  ┌──────────────┐                                          │
│  │    Remark    │ ← STW ~50ms（可能飙到 100ms+）            │
│  │  (STW)       │                                          │
│  └──────┬───────┘                                          │
│         │                                                  │
│         ▼                                                  │
│  ┌──────────────────────────────┐                          │
│  │  Concurrent Sweep             │ ← 业务线程并行，0ms STW │
│  │  (并发清除)                   │                          │
│  │  耗时：~100ms（业务线程并行） │                          │
│  └──────────────────────────────┘                          │
│                                                            │
└────────────────────────────────────────────────────────────┘

总 STW 时间：~55ms（理想情况）
实际可能：~100ms+（Remark 飙的情况）
```

### 1.2 4 阶段的目的

| 阶段 | 类型 | 目的 |
|:---|:---|:---|
| **Initial Mark** | STW | 标记 GC Root 直接引用的对象（必须暂停，否则根找不到） |
| **Concurrent Mark** | 并发 | 从 GC Root 出发，递归标记所有可达对象（耗时但可并发） |
| **Remark** | STW | 修正并发标记期间被业务线程修改的引用（必须暂停，确保标记正确） |
| **Concurrent Sweep** | 并发 | 遍历 Mark Bitmap，清除未标记的对象（耗时但可并发） |

### 1.3 必须 STW 的两个阶段

CMS 把工作拆成 4 个阶段，**只有 2 个必须 STW**：

```
必须 STW：Initial Mark（5ms） + Remark（50ms）
并发：Concurrent Mark（100ms） + Concurrent Sweep（100ms）
```

**为什么 Initial Mark 和 Remark 必须 STW？**
- 这两个阶段需要看到"一致状态"
- 不 STW 会导致漏标（详见 [03-写屏障的角色](03-写屏障的角色.md) §1）

## 二、阶段 1：Initial Mark（STW ~5ms）

### 2.1 Initial Mark 的定义

**Initial Mark** 阶段标记 **GC Root 直接引用的对象**——只标记一层，必须 STW。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::InitialMarkPhase
void MarkSweep::InitialMarkPhase() {
    // 1. 暂停所有 mutator 线程（STW）
    SuspendAllThreads();

    // 2. 标记 GC Root
    VisitRoots([this](mirror::Object* obj) {
        MarkObjectParallel(obj);  // 标记为灰色
    });

    // 3. STW 完成前不解暂停线程
    // （继续 Concurrent Mark，但已经标记的 Root 对象可以遍历了）
}
```

### 2.2 为什么 Initial Mark 必须 STW

如果 Initial Mark 不 STW：
```
业务线程：修改 obj.field（断开引用）
GC 线程：访问 obj.field（标记引用链）
→ 两者顺序不确定
→ 可能 GC 标记了已被断开的引用（漏标）
→ 或 GC 没标记新建立的引用（漏标）
```

**结论**：必须 STW，让 GC 单独工作一段时间。

### 2.3 Initial Mark 为什么只要 5ms

Initial Mark **只标记 GC Root 直接引用的对象**——不做递归遍历。

```cpp
// 12 种 GC Root
VisitRoots([this](mirror::Object* obj) {
    MarkObjectParallel(obj);  // 直接标记，不递归
});
```

| Root 来源 | 数量级 | 耗时 |
|:---|:---|:---|
| JNI Global Ref | ~100 | ~10μs |
| JNI Local Ref | ~1000 | ~100μs |
| Java Frame | ~10000 | ~1ms |
| Sticky Class | ~5000 | ~500μs |
| Interned String | ~10000 | ~1ms |
| 其他 7 种 | ~10000 | ~1ms |
| **总计** | **~30K 对象** | **~5ms** |

→ **GC Root 直接引用对象数量级是 ~10K-100K**，标记耗时是 ms 级。

### 2.4 Initial Mark 的并发类卸载（ART 17 优化）

ART 17 在 Initial Mark 阶段做了**并发类卸载**优化：

```
┌────────────────────────────────────────────────────────────────┐
│ Initial Mark 并发类卸载（ART 17 优化）                           │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  CMS 时代（AOSP 14）：                                          │
│    └─ Initial Mark 仅标记 Root → STW ~5ms                      │
│    └─ 类卸载在 Remark 阶段做 → 慢                                │
│                                                                │
│  ART 17 优化：                                                  │
│    ├─ Initial Mark 阶段并发卸载无用 Class                        │
│    ├─ 避免类元数据进入 Concurrent Mark                            │
│    ├─ Initial Mark STW 从 5ms 降至 1-2ms                        │
│    └─ 总 STW 时间减少 60-80%                                    │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**架构师视角**：并发类卸载让 Initial Mark 更快，**直接降低用户感知的卡顿**。

## 三、阶段 2：Concurrent Mark（并发 ~100ms）

### 3.1 Concurrent Mark 的定义

**Concurrent Mark** 阶段从 GC Root 出发，**递归标记所有可达对象**——可以并发（不 STW）。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::ConcurrentMarkPhase
void MarkSweep::ConcurrentMarkPhase() {
    // 1. 恢复 mutator 线程
    ResumeAllThreads();

    // 2. 并发标记（与业务线程并行）
    while (!mark_stack_->IsEmpty()) {
        // 从 mark stack 取灰色对象
        mirror::Object* obj = mark_stack_->Pop();

        // 标记对象引用的所有子对象
        obj->VisitReferences([this](mirror::Object* ref) {
            if (ref != nullptr) {
                MarkObjectParallel(ref);  // 染灰
            }
        });
    }
}
```

### 3.2 并发标记的并发性

**关键**：Concurrent Mark 与业务线程并行。

```
时间线：
GC 线程：标记对象 A → 标记对象 B → 标记对象 C → ...
业务线程：执行新对象分配、执行引用赋值

两者同时进行，互不干扰（理论上）
```

### 3.3 并发标记的并发问题

**问题**：业务线程可能修改引用，导致漏标。

```
GC 线程：扫描到 obj_A，已将 obj_A 染黑（已完成扫描）
业务线程：执行 obj_A.field = new_obj（建立新引用）
          → new_obj 是白色（未被标记）
          → GC 不会再扫描 obj_A（已染黑）
          → new_obj 永远不会被标记
          → 漏标！
```

**修复**：用 **写屏障（Pre-Write Barrier）** 拦截这种修改（详见 [03-写屏障的角色](03-写屏障的角色.md)）。

### 3.4 Concurrent Mark 的并行优化

**Mark Stack** 的多线程优化：

```cpp
// ART 用多个 GC 线程并行标记
void MarkSweep::ConcurrentMarkPhase() {
    // 1. 启动多个 GC 工作线程
    for (int i = 0; i < num_gc_threads_; i++) {
        gc_threads_[i] = std::thread(this {
            // 每个线程从 mark stack 取对象标记
            while (!mark_stack_->IsEmpty()) {
                mirror::Object* obj = mark_stack_->Pop();
                obj->VisitReferences([this](mirror::Object* ref) {
                    MarkObjectParallel(ref);
                });
            }
        });
    }

    // 2. 等待所有 GC 线程完成
    for (auto& thread : gc_threads_) {
        thread.join();
    }
}
```

### 3.5 Mark Bitmap 的并发标记

CMS 用 **Mark Bitmap** 记录标记状态：

```cpp
// 标记对象（原子操作）
bool MarkBitmap::Set(mirror::Object* obj) {
    // 1. 计算 bit 位置
    size_t index = (reinterpret_cast<uintptr_t>(obj) - base_addr_) / kAlignment;
    size_t byte_offset = index / 8;
    uint8_t mask = 1u << (index % 8);

    // 2. CAS 设置 bit（并发安全）
    uint8_t old = bitmap_[byte_offset];
    while ((old & mask) == 0) {
        // CAS 重试
        if (CAS(&bitmap_[byte_offset], old, old | mask)) {
            return true;  // 第一次标记
        }
        old = bitmap_[byte_offset];
    }
    return false;  // 已经被标记
}
```

### 3.6 ART 17 Concurrent Mark 增量优化

ART 17 引入**增量标记**（Incremental Marking）：

```
┌────────────────────────────────────────────────────────────────┐
│ Concurrent Mark 增量（ART 17 优化）                              │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  CMS 时代（AOSP 14）：                                          │
│    └─ Concurrent Mark 一次性跑完 → 业务线程长延迟                │
│    └─ 堆大（256MB+）时单次标记可达 300-500ms                    │
│                                                                │
│  ART 17 优化：                                                  │
│    ├─ 增量标记：分片执行，每片 ~10ms                              │
│    ├─ 与业务线程穿插：标记 10ms → 业务 20ms → 标记 10ms           │
│    ├─ 总标记时间不变，但业务线程延迟降低 50%                       │
│    └─ 适合 UI 渲染等延迟敏感场景                                  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**架构师视角**：增量标记让 Concurrent Mark 不再"霸占"CPU，**业务线程延迟降低 50%**。

## 四、阶段 3：Remark（STW ~50ms）

### 4.1 Remark 的定义

**Remark** 阶段**重新扫描**，修正并发标记期间被业务线程修改的引用——必须 STW。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::RemarkPhase
void MarkSweep::RemarkPhase() {
    // 1. 暂停所有 mutator 线程（STW）
    SuspendAllThreads();

    // 2. 处理 dirty 对象（写屏障记录的）
    for (mirror::Object* obj : dirty_objects_) {
        // 重新扫描这个对象
        obj->VisitReferences([this](mirror::Object* ref) {
            if (ref != nullptr) {
                MarkObjectParallel(ref);
            }
        });
    }

    // 3. 清理 dirty 队列
    dirty_objects_.clear();
}
```

### 4.2 为什么 Remark 必须 STW

如果 Remark 不 STW：
```
业务线程：执行 obj_A.field = new_obj（建立新引用）
GC 线程：扫描 obj_A，发现 new_obj 未标记
业务线程：执行 obj_A.field = null（断开引用）
GC 线程：扫描 obj_A，发现 new_obj 不再被引用
         → 跳过 new_obj 的标记
         → new_obj 永远不被标记
         → 漏标！
```

**结论**：必须 STW，确保 GC 单独完成"修正"工作。

### 4.3 Remark 为何可能 50ms+

Remark 阶段 STW 时间取决于：

| 因素 | 影响 |
|:---|:---|
| **dirty 对象数量** | 越多越慢（每个都要重新扫描） |
| **对象引用复杂度** | 复杂引用扫描更慢 |
| **Java 线程数** | 栈扫描时间 |
| **堆大小** | 扫描范围 |

**为什么 dirty 对象多？**
- 业务线程在并发标记期间疯狂创建对象
- 或修改大量引用
- 每次都触发写屏障记录 → dirty 队列膨胀

### 4.4 Remark 阶段的性能瓶颈

**瓶颈 1：dirty 对象重新扫描**
```cpp
// 每个 dirty 对象都要 VisitReferences
// 如果有 100 万个 dirty 对象，每个有 10 个引用
// → 1000 万次引用扫描
// → STW 时间可能 50ms+
```

**瓶颈 2：栈扫描**
```cpp
// 重新扫描 Java 栈
for (Thread* thread : thread_list_) {
    thread->VisitStack([this](mirror::Object* ref) {
        MarkObjectParallel(ref);  // 重新标记栈引用
    });
}
```

### 4.5 ART 17 Remark 写屏障强化

ART 17 对 Remark 阶段的写屏障做了强化：

```
┌────────────────────────────────────────────────────────────────┐
│ Remark 写屏障强化（ART 17 优化）                                 │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  CMS 时代（AOSP 14）：                                          │
│    └─ Pre-Write Barrier（Incremental Update）                  │
│    └─ 每次指针赋值 3 条额外指令                                  │
│    └─ dirty 队列膨胀 → Remark STW 50ms+                         │
│                                                                │
│  ART 17 强化：                                                  │
│    ├─ 写屏障 + Card Table（卡表）协同                            │
│    ├─ Card Table 压缩（256 byte → 64 byte）                    │
│    ├─ dirty 对象减少 50-70%                                     │
│    ├─ Remark STW 从 50ms+ 降至 20-30ms                          │
│    └─ 与 SATB 屏障配合，漏标概率降低 30-50%                       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**架构师视角**：卡表压缩让 dirty 队列更小，**Remark STW 显著降低**。

详见 [03-写屏障的角色](03-写屏障的角色.md) §4。

## 五、阶段 4：Concurrent Sweep（并发 ~100ms）

### 5.1 Concurrent Sweep 的定义

**Concurrent Sweep** 阶段**遍历 Mark Bitmap，清除未标记的对象**——可以并发。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::ConcurrentSweepPhase
void MarkSweep::ConcurrentSweepPhase() {
    // 1. 恢复 mutator 线程
    ResumeAllThreads();

    // 2. 并发清除（与业务线程并行）
    SweepLargeObjects();   // LOS 先 Sweep

    // 3. Sweep Allocation Space
    for (Page* page : allocation_space_->GetPages()) {
        for (Run* run : page->GetRuns()) {
            SweepRun(run);  // Sweep 单个 Run
        }
    }
}
```

### 5.2 Sweep 单个 Run

```cpp
void MarkSweep::SweepRun(Run* run) {
    // 遍历 Run 内所有 slot
    for (size_t i = 0; i < run->num_slots_; i++) {
        void* slot = run->slots_[i];
        if (!mark_bitmap_->Test(slot)) {
            // 未标记 → 加入 free list
            run->free_list_.Push(slot);
        } else {
            // 已标记 → 重置 mark bit（下一轮 GC 用）
            mark_bitmap_->Clear(slot);
        }
    }
}
```

### 5.3 Alloc-During-Sweep 的处理

**问题**：业务线程在 Sweep 阶段继续分配对象。

```cpp
// 业务线程 new Object()
// 但 Run 的 slot 还没被 Sweep（free list 没更新）
// → 慢速路径：申请新的 Run 或触发 GC
```

**ART 的处理**：Sweep 是增量的，业务线程分配的 Run 还没 Sweep 时，从其他已 Sweep 的 Run 分配。

### 5.4 Concurrent Sweep 的并发问题

**问题 1：业务线程分配对象时，引用 Run 中已 dead 的对象**

```
CMS Sweep 把 dead 对象加入 free list
业务线程从 free list 拿 slot → 写入新对象
但旧对象的引用可能还在 → 野指针
```

**修复**：CMS 不立即复用 free list 的 slot，而是延迟复用。

```cpp
// CMS Sweep 后，slot 不立即可用
// 等下一轮 GC 后才真正复用
// 避免野指针问题
```

### 5.5 LOS 的 Sweep

```cpp
void MarkSweep::SweepLargeObjects() {
    // 遍历 LOS 中所有对象
    for (LargeObject* obj : large_object_space_->GetObjects()) {
        if (!mark_bitmap_->Test(obj)) {
            // 未标记 → 回收
            large_object_space_->Free(obj);
        }
    }
}
```

**LOS 的特点**：
- 不移动（不复制）
- 不压缩
- 单独 Sweep
- **碎片化严重**（详见 [3.6 内存碎片化](./06-内存碎片化.md)）

### 5.6 ART 17 Sweep 增量优化

ART 17 引入**增量 Sweep**（Incremental Sweep）：

```
┌────────────────────────────────────────────────────────────────┐
│ Concurrent Sweep 增量（ART 17 优化）                             │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  CMS 时代（AOSP 14）：                                          │
│    └─ Concurrent Sweep 一次性跑完 → 业务线程长延迟                │

│                                                                │
│  ART 17 优化：                                                  │
│    ├─ 增量 Sweep：分片执行，每片 ~5ms                             │
│    ├─ 与业务线程穿插：Sweep 5ms → 业务 20ms → Sweep 5ms          │
│    ├─ 业务线程分配延迟降低 70%                                   │
│    └─ Alloc-During-Sweep 命中率提升 30%                          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**架构师视角**：增量 Sweep 让 CMS 时代的 Sweep 延迟不再"集中爆发"。

## 七、ART 17 硬变化专章

### 7.1 ART 17 Initial Mark 优化：并发类卸载

AOSP 17 在 Initial Mark 阶段新增**并发类卸载**：

- **CMS 时代**：类卸载在 Remark 阶段做，STW 期间卸载大量无用 Class
- **ART 17**：Initial Mark 阶段并发卸载无用 Class，避免进入 Concurrent Mark
- **优化效果**：Initial Mark STW 从 5ms 降至 1-2ms，**总 STW 时间减少 60-80%**

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §4。

### 7.2 ART 17 Concurrent Mark 增量优化

AOSP 17 引入**增量标记**：

- **CMS 时代**：Concurrent Mark 一次性跑完，堆大（256MB+）时单次标记 300-500ms
- **ART 17**：分片执行（每片 ~10ms），与业务线程穿插
- **优化效果**：业务线程延迟降低 50%，适合 UI 渲染等延迟敏感场景

详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §4。

### 7.3 ART 17 Remark 写屏障强化

AOSP 17 强化 Remark 阶段的写屏障：

- **CMS 时代**：Pre-Write Barrier（Incremental Update），每次指针赋值 3 条额外指令
- **ART 17**：写屏障 + Card Table（卡表）协同，卡表压缩（256 byte → 64 byte）
- **优化效果**：dirty 对象减少 50-70%，Remark STW 从 50ms+ 降至 20-30ms，**漏标概率降低 30-50%**

详见 [03-写屏障的角色](03-写屏障的角色.md) §4。

### 7.4 ART 17 Sweep 增量优化

AOSP 17 引入**增量 Sweep**：

- **CMS 时代**：Concurrent Sweep 一次性跑完，业务线程长延迟
- **ART 17**：分片执行（每片 ~5ms），与业务线程穿插
- **优化效果**：业务线程分配延迟降低 70%，Alloc-During-Sweep 命中率提升 30%

详见 [04-Sweep的实现](04-Sweep的实现.md) §4。

### 7.5 Linux 6.12 与 CMS 4 阶段的关联

- **Linux 6.12 sheaves**：让 ART Native 堆内存降低 15-20%，CMS 时代的 libart.so 占用也受益
- **Linux 6.12 io_uring 增强**：让 GC 写屏障的卡表刷盘加速 30%
- **Linux 6.12 内存屏障**：x86 / arm64 架构的内存屏障指令优化，提升 CMS 并发标记的可见性
- **跨系列引用**：详见 [Linux_Kernel/DM/09-DM-调优-性能与pcache](../../../Linux_Kernel/DM/09-DM-调优-性能与pcache.md) §3

## 十一、总结（架构师视角的 5 条 Takeaway）

1. **CMS 4 阶段 = Initial Mark (5ms) + Concurrent Mark (100ms) + Remark (50ms) + Concurrent Sweep (100ms)**。**必须 STW 的只有 Initial Mark 和 Remark**——因为这两个阶段必须看到"一致状态"。
2. **Concurrent Mark 和 Sweep 是耗时大头，但 0ms STW**——与业务线程并行。**Remark 是硬伤**：dirty 对象多时可能 50-200ms+。
3. **ART 17 对 4 阶段全面优化**——Initial Mark 并发类卸载（5ms → 1-2ms）、Concurrent Mark 增量（业务延迟 -50%）、Remark 写屏障强化（50ms → 20-30ms）、Sweep 增量（业务延迟 -70%）。**总 STW 从 55ms 降至 24ms**。详见 [10-ART17分代GC强化专章 v2](../10-ART17分代GC强化专章-v2.md) §4。
4. **Concurrent Sweep 不立即复用 free list slot**——避免野指针。**ART 17 增量 Sweep 让延迟更平滑**。详见 [04-Sweep的实现](04-Sweep的实现.md) §4。
5. **4 阶段设计是 CMS 的"承上启下"**——为 CC GC 提供了 4 阶段借鉴，为 GenCC 提供了分代基础。**理解 4 阶段是理解 CMS 演进的根本**。
