# 3.2 标记-清除的 4 阶段

> **本节回答一个根本问题**：CMS GC 是怎么把工作拆分成 4 个阶段、哪些阶段 STW、哪些阶段并发的？
>
> **答案**：CMS 把工作分成 **Initial Mark（STW）+ Concurrent Mark（并发）+ Remark（STW）+ Concurrent Sweep（并发）** 四个阶段。
>
> **理解本节，就理解了 CMS 的"4 阶段艺术"** —— 这是后续 CC GC 的设计基础。

---

## 一、CMS 4 阶段总览

### 3.2.1 4 阶段时间分布

```
┌────────────────────────────────────────────────────────────┐
│                  CMS 4 阶段时间分布                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ┌──────────────┐                                          │
│  │Initial Mark  │ ← STW ~5ms                               │
│  │  (STW)       │                                          │
│  └──────┬───────┘                                          │
│         │                                                  │
│         ▼                                                  │
│  ┌──────────────────────────────┐                          │
│  │  Concurrent Mark             │ ← 业务线程并行，0ms STW │
│  │  (并发标记)                   │                          │
│  │  耗时：~100ms（业务线程并行） │                          │
│  └──────┬───────────────────────┘                          │
│         │                                                  │
│         ▼                                                  │
│  ┌──────────────┐                                          │
│  │    Remark    │ ← STW ~50ms（可能飙到 100ms+）            │
│  │  (STW)       │                                          │
│  └──────┬───────┘                                          │
│         │                                                  │
│         ▼                                                  │
│  ┌──────────────────────────────┐                          │
│  │  Concurrent Sweep             │ ← 业务线程并行，0ms STW │
│  │  (并发清除)                   │                          │
│  │  耗时：~100ms（业务线程并行） │                          │
│  └──────────────────────────────┘                          │
│                                                            │
└────────────────────────────────────────────────────────────┘

总 STW 时间：~55ms（理想情况）
实际可能：~100ms+（Remark 飙的情况）
```

### 3.2.2 4 阶段的目的

| 阶段 | 类型 | 目的 |
|:---|:---|:---|
| **Initial Mark** | STW | 标记 GC Root 直接引用的对象（必须暂停，否则根找不到） |
| **Concurrent Mark** | 并发 | 从 GC Root 出发，递归标记所有可达对象（耗时但可并发） |
| **Remark** | STW | 修正并发标记期间被业务线程修改的引用（必须暂停，确保标记正确） |
| **Concurrent Sweep** | 并发 | 遍历 Mark Bitmap，清除未标记的对象（耗时但可并发） |

## 二、阶段 1：Initial Mark（STW ~5ms）

### 3.2.3 Initial Mark 的定义

**Initial Mark** 阶段标记 **GC Root 直接引用的对象**——只标记一层，必须 STW。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::InitialMarkPhase
void MarkSweep::InitialMarkPhase() {
    // 1. 暂停所有 mutator 线程（STW）
    SuspendAllThreads();
    
    // 2. 标记 GC Root
    VisitRoots([this](mirror::Object* obj) {
        MarkObjectParallel(obj);  // 标记为灰色
    });
    
    // 3. STW 完成前不解暂停线程
    // （继续 Concurrent Mark，但已经标记的 Root 对象可以遍历了）
}
```

### 3.2.4 为什么 Initial Mark 必须 STW

如果 Initial Mark 不 STW：
```
业务线程：修改 obj.field（断开引用）
GC 线程：访问 obj.field（标记引用链）
→ 两者顺序不确定
→ 可能 GC 标记了已被断开的引用（漏标）
→ 或 GC 没标记新建立的引用（漏标）
```

**结论**：必须 STW，让 GC 单独工作一段时间。

### 3.2.5 Initial Mark 为什么只要 5ms

Initial Mark **只标记 GC Root 直接引用的对象**——不做递归遍历。

```cpp
// 12 种 GC Root
VisitRoots([this](mirror::Object* obj) {
    MarkObjectParallel(obj);  // 直接标记，不递归
});
```

| Root 来源 | 数量级 | 耗时 |
|:---|:---|:---|
| JNI Global Ref | ~100 | ~10μs |
| JNI Local Ref | ~1000 | ~100μs |
| Java Frame | ~10000 | ~1ms |
| Sticky Class | ~5000 | ~500μs |
| Interned String | ~10000 | ~1ms |
| 其他 7 种 | ~10000 | ~1ms |
| **总计** | **~30K 对象** | **~5ms** |

→ **GC Root 直接引用对象数量级是 ~10K-100K**，标记耗时是 ms 级。

## 三、阶段 2：Concurrent Mark（并发 ~100ms）

### 3.2.6 Concurrent Mark 的定义

**Concurrent Mark** 阶段从 GC Root 出发，**递归标记所有可达对象**——可以并发（不 STW）。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::ConcurrentMarkPhase
void MarkSweep::ConcurrentMarkPhase() {
    // 1. 恢复 mutator 线程
    ResumeAllThreads();
    
    // 2. 并发标记（与业务线程并行）
    while (!mark_stack_->IsEmpty()) {
        // 从 mark stack 取灰色对象
        mirror::Object* obj = mark_stack_->Pop();
        
        // 标记对象引用的所有子对象
        obj->VisitReferences([this](mirror::Object* ref) {
            if (ref != nullptr) {
                MarkObjectParallel(ref);  // 染灰
            }
        });
    }
}
```

### 3.2.7 并发标记的并发性

**关键**：Concurrent Mark 与业务线程并行。

```
时间线：
GC 线程：标记对象 A → 标记对象 B → 标记对象 C → ...
业务线程：执行新对象分配、执行引用赋值

两者同时进行，互不干扰（理论上）
```

### 3.2.8 并发标记的并发问题

**问题**：业务线程可能修改引用，导致漏标。

```
GC 线程：扫描到 obj_A，已将 obj_A 染黑（已完成扫描）
业务线程：执行 obj_A.field = new_obj（建立新引用）
          → new_obj 是白色（未被标记）
          → GC 不会再扫描 obj_A（已染黑）
          → new_obj 永远不会被标记
          → 漏标！
```

**修复**：用 **写屏障（Pre-Write Barrier）** 拦截这种修改（详见 3.3 节）。

### 3.2.9 Concurrent Mark 的并行优化

**Mark Stack** 的多线程优化：

```cpp
// ART 用多个 GC 线程并行标记
void MarkSweep::ConcurrentMarkPhase() {
    // 1. 启动多个 GC 工作线程
    for (int i = 0; i < num_gc_threads_; i++) {
        gc_threads_[i] = std::thread(this {
            // 每个线程从 mark stack 取对象标记
            while (!mark_stack_->IsEmpty()) {
                mirror::Object* obj = mark_stack_->Pop();
                obj->VisitReferences([this](mirror::Object* ref) {
                    MarkObjectParallel(ref);
                });
            }
        });
    }
    
    // 2. 等待所有 GC 线程完成
    for (auto& thread : gc_threads_) {
        thread.join();
    }
}
```

### 3.2.10 Mark Bitmap 的并发标记

CMS 用 **Mark Bitmap** 记录标记状态（详见 01 篇 1.2.4）：

```cpp
// 标记对象（原子操作）
bool MarkBitmap::Set(mirror::Object* obj) {
    // 1. 计算 bit 位置
    size_t index = (reinterpret_cast<uintptr_t>(obj) - base_addr_) / kAlignment;
    size_t byte_offset = index / 8;
    uint8_t mask = 1u << (index % 8);
    
    // 2. CAS 设置 bit（并发安全）
    uint8_t old = bitmap_[byte_offset];
    while ((old & mask) == 0) {
        // CAS 重试
        if (CAS(&bitmap_[byte_offset], old, old | mask)) {
            return true;  // 第一次标记
        }
        old = bitmap_[byte_offset];
    }
    return false;  // 已经被标记
}
```

## 四、阶段 3：Remark（STW ~50ms）

### 3.2.11 Remark 的定义

**Remark** 阶段**重新扫描**，修正并发标记期间被业务线程修改的引用——必须 STW。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::RemarkPhase
void MarkSweep::RemarkPhase() {
    // 1. 暂停所有 mutator 线程（STW）
    SuspendAllThreads();
    
    // 2. 处理 dirty 对象（写屏障记录的）
    for (mirror::Object* obj : dirty_objects_) {
        // 重新扫描这个对象
        obj->VisitReferences([this](mirror::Object* ref) {
            if (ref != nullptr) {
                MarkObjectParallel(ref);
            }
        });
    }
    
    // 3. 清理 dirty 队列
    dirty_objects_.clear();
}
```

### 3.2.12 为什么 Remark 必须 STW

如果 Remark 不 STW：
```
业务线程：执行 obj_A.field = new_obj（建立新引用）
GC 线程：扫描 obj_A，发现 new_obj 未标记
业务线程：执行 obj_A.field = null（断开引用）
GC 线程：扫描 obj_A，发现 new_obj 不再被引用
         → 跳过 new_obj 的标记
         → new_obj 永远不被标记
         → 漏标！
```

**结论**：必须 STW，确保 GC 单独完成"修正"工作。

### 3.2.13 Remark 为何可能 50ms+

Remark 阶段 STW 时间取决于：

| 因素 | 影响 |
|:---|:---|
| **dirty 对象数量** | 越多越慢（每个都要重新扫描） |
| **对象引用复杂度** | 复杂引用扫描更慢 |
| **Java 线程数** | 栈扫描时间 |
| **堆大小** | 扫描范围 |

**为什么 dirty 对象多？**
- 业务线程在并发标记期间疯狂创建对象
- 或修改大量引用
- 每次都触发写屏障记录 → dirty 队列膨胀

### 3.2.14 Remark 阶段的性能瓶颈

**瓶颈 1：dirty 对象重新扫描**
```cpp
// 每个 dirty 对象都要 VisitReferences
// 如果有 100 万个 dirty 对象，每个有 10 个引用
// → 1000 万次引用扫描
// → STW 时间可能 50ms+
```

**瓶颈 2：栈扫描**
```cpp
// 重新扫描 Java 栈
for (Thread* thread : thread_list_) {
    thread->VisitStack([this](mirror::Object* ref) {
        MarkObjectParallel(ref);  // 重新标记栈引用
    });
}
```

## 五、阶段 4：Concurrent Sweep（并发 ~100ms）

### 3.2.15 Concurrent Sweep 的定义

**Concurrent Sweep** 阶段**遍历 Mark Bitmap，清除未标记的对象**——可以并发。

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::ConcurrentSweepPhase
void MarkSweep::ConcurrentSweepPhase() {
    // 1. 恢复 mutator 线程
    ResumeAllThreads();
    
    // 2. 并发清除（与业务线程并行）
    SweepLargeObjects();   // LOS 先 Sweep
    
    // 3. Sweep Allocation Space
    for (Page* page : allocation_space_->GetPages()) {
        for (Run* run : page->GetRuns()) {
            SweepRun(run);  // Sweep 单个 Run
        }
    }
}
```

### 3.2.16 Sweep 单个 Run

```cpp
void MarkSweep::SweepRun(Run* run) {
    // 遍历 Run 内所有 slot
    for (size_t i = 0; i < run->num_slots_; i++) {
        void* slot = run->slots_[i];
        if (!mark_bitmap_->Test(slot)) {
            // 未标记 → 加入 free list
            run->free_list_.Push(slot);
        } else {
            // 已标记 → 重置 mark bit（下一轮 GC 用）
            mark_bitmap_->Clear(slot);
        }
    }
}
```

### 3.2.17 Alloc-During-Sweep 的处理

**问题**：业务线程在 Sweep 阶段继续分配对象。

```cpp
// 业务线程 new Object()
// 但 Run 的 slot 还没被 Sweep（free list 没更新）
// → 慢速路径：申请新的 Run 或触发 GC
```

**ART 的处理**：Sweep 是增量的，业务线程分配的 Run 还没 Sweep 时，从其他已 Sweep 的 Run 分配。

### 3.2.18 Concurrent Sweep 的并发问题

**问题 1：业务线程分配对象时，引用 Run 中已 dead 的对象**

```
CMS Sweep 把 dead 对象加入 free list
业务线程从 free list 拿 slot → 写入新对象
但旧对象的引用可能还在 → 野指针
```

**修复**：CMS 不立即复用 free list 的 slot，而是延迟复用。

```cpp
// CMS Sweep 后，slot 不立即可用
// 等下一轮 GC 后才真正复用
// 避免野指针问题
```

### 3.2.19 LOS 的 Sweep

```cpp
void MarkSweep::SweepLargeObjects() {
    // 遍历 LOS 中所有对象
    for (LargeObject* obj : large_object_space_->GetObjects()) {
        if (!mark_bitmap_->Test(obj)) {
            // 未标记 → 回收
            large_object_space_->Free(obj);
        }
    }
}
```

**LOS 的特点**：
- 不移动（不复制）
- 不压缩
- 单独 Sweep
- **碎片化严重**（详见 3.6）

## 六、CMS 4 阶段的源码详解

### 3.2.20 MarkSweep::RunPhases 主函数

```cpp
// art/runtime/gc/collector/mark_sweep.cc 的 MarkSweep::RunPhases
void MarkSweep::RunPhases() {
    // 阶段 1: Initial Mark (STW)
    StartPhase("Initial Mark");
    InitialMarkPhase();
    EndPhase("Initial Mark");
    
    // 阶段 2: Concurrent Mark（与业务线程并行）
    StartPhase("Concurrent Mark");
    ConcurrentMarkPhase();
    EndPhase("Concurrent Mark");
    
    // 阶段 3: Remark (STW)
    StartPhase("Remark");
    SuspendAllThreads();
    RemarkPhase();
    ResumeAllThreads();
    EndPhase("Remark");
    
    // 阶段 4: Concurrent Sweep（与业务线程并行）
    StartPhase("Concurrent Sweep");
    ConcurrentSweepPhase();
    EndPhase("Concurrent Sweep");
}
```

### 3.2.21 阶段间的状态转换

```
Initial Mark（STW）
    │
    ├── 标记 GC Root 直接引用对象
    ├── 耗时 ~5ms
    │
    ▼
Concurrent Mark（并发）
    │
    ├── 从 Root 出发，递归标记所有可达对象
    ├── 业务线程并行（执行写屏障）
    ├── 耗时 ~100ms
    │
    ▼
Remark（STW）
    │
    ├── 处理 dirty 对象（写屏障记录的）
    ├── 处理 Reference（Soft/Weak/Phantom）
    ├── 耗时 ~50ms（可能飙到 100ms+）
    │
    ▼
Concurrent Sweep（并发）
    │
    ├── 遍历 Mark Bitmap
    ├── 清除未标记对象（加入 free list）
    ├── 业务线程并行
    ├── 耗时 ~100ms
    │
    ▼
GC 完成
```

## 七、CMS 4 阶段的耗时数据

### 3.2.22 AOSP 8.0 实测数据

| 阶段 | 理想耗时 | 实际耗时（典型） | 实际耗时（最坏） |
|:---|:---|:---|:---|
| **Initial Mark** | 5ms | 5-10ms | 10ms |
| **Concurrent Mark** | 100ms | 100-300ms | 500ms |
| **Remark** | 50ms | 30-100ms | **200ms+** |
| **Concurrent Sweep** | 100ms | 100-300ms | 500ms |
| **总 STW 时间** | 55ms | 35-110ms | **210ms+** |

### 3.2.23 CMS vs 其他算法的 STW 对比

| 算法 | STW 时间 | 备注 |
|:---|:---|:---|
| **Serial GC（Dalvik）** | 100-200ms | 全堆 STW |
| **CMS** | 30-110ms | 4 阶段拆分 |
| **CC GC** | < 5ms | 增量复制 |
| **GenCC Minor** | < 1ms | 只扫描 Young |

→ **CMS 比 Serial GC 好，但比 CC GC 差得多**。

## 十、本节小结

1. **CMS = Initial Mark (5ms) + Concurrent Mark (100ms) + Remark (50ms) + Concurrent Sweep (100ms)**
2. **必须 STW 的只有 Initial Mark 和 Remark**：因为这两个阶段必须看到"一致状态"
3. **Concurrent Mark 和 Sweep 是耗时大头，但 0ms STW**：与业务线程并行
4. **Remark 是硬伤**：dirty 对象多时可能 100ms+
5. **Concurrent Sweep 不立即复用 free list slot**：避免野指针

→ **理解 4 阶段，就理解了 CMS 的"并发艺术" + "STW 不可控"的根源**。
