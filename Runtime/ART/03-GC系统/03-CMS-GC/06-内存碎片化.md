# 3.6 CMS 的死穴：内存碎片

> **本节回答一个根本问题**：CMS 为什么必然导致严重的内存碎片化？"堆里还有 100MB 为什么 OOM" 在 CMS 时代是怎么发生的？
>
> **答案**：**CMS 不压缩 + RosAlloc 按 size class 分桶 + LOS 标记-清除** 三大因素共同导致。
>
> **理解本节，就理解了 CMS 的根本硬伤 + 为什么必须升级到 CC GC**。

---

## 一、CMS 碎片化的根源

### 3.6.1 碎片化的三大根源

```
┌────────────────────────────────────────────────────────────┐
│                  CMS 碎片化的三大根源                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  根源 1: CMS 不压缩                                         │
│  ─────────────────────────                                  │
│  CMS 是标记-清除算法，不移动对象                              │
│  → 死对象留下的空洞无法合并                                  │
│  → Allocation Space 外碎片严重                              │
│                                                            │
│  根源 2: RosAlloc 按 size class 分桶                        │
│  ──────────────────────────────────                         │
│  RosAlloc 把空间分成 36 个 size class                        │
│  → 同一 size class 的对象走同一 Run                          │
│  → 不同 size class 的 Run 不能跨桶合并                       │
│  → 即使总空闲足够，单个 size class 不够也 OOM                │
│                                                            │
│  根源 3: LOS 标记-清除                                       │
│  ──────────────────────                                     │
│  LOS 用标记-清除，不压缩                                    │
│  → 大 Bitmap / byte[] 释放后留下空洞                       │
│  → 没有连续的大空间分配新大对象                              │
│                                                            │
│  三大根源协同 → 碎片化是 CMS 时代的必然                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3.6.2 碎片化的具体表现

**Allocation Space 碎片化**：

```
CMS GC 多次后的 Allocation Space 状态：

Run A (16B):  [██████░░░░░░░░] 50% 使用
Run B (32B):  [██░░░░░░░░░░░░] 20% 使用
Run C (64B):  [██░░░░░░░░░░░░] 20% 使用
Run D (16B):  [██░░░░░░░░░░░░] 20% 使用
Run E (32B):  [████░░░░░░░░░░] 30% 使用
...

总空闲 = 60%
但单 size class 都不够分配特定大小的对象
→ 申请新 Run → 堆空间持续增长 → OOM
```

**LOS 碎片化**：

```
LOS 状态：
[4 MB alive] [8 MB alive] [4 MB alive] [FREE 8 MB] [4 MB alive]

→ 总空闲 8 MB
→ 但都是空洞，最大空洞 ~2 MB
→ 申请 5 MB Bitmap → 失败 → OOM
```

## 二、内部碎片 vs 外部碎片

### 3.6.3 内部碎片（Internal Fragmentation）

**定义**：分配的空间 > 实际需要的空间。

```cpp
// RosAlloc 按 size class 分配
// 对象实际 17 字节 → 分配 24 字节（size class 是 24）
// 内部碎片 = 24 - 17 = 7 字节

// 假设 Heap 1 GB，全是 17 字节对象
// 实际占用：1 GB
// RosAlloc 占用：~1.4 GB（size class 24）
// 内部碎片：~400 MB（40%）
```

### 3.6.4 外部碎片（External Fragmentation）

**定义**：空闲内存分散在多个不连续的小块中。

```
Allocation Space 状态：
[Used][Used][FREE 10 MB][Used][FREE 5 MB][Used][FREE 8 MB]

需要分配 15 MB 连续空间：
→ 没有连续 15 MB
→ 即使总空闲 23 MB，也分配失败
```

### 3.6.5 碎片的量化分析

**CMS 时代碎片化程度**：

| 指标 | CMS（Android 5-7） | CC GC（Android 8+） |
|:---|:---|:---|
| **内部碎片率** | ~10-20% | < 1% |
| **外部碎片率** | ~20-40% | < 1% |
| **总碎片率** | ~30-60% | < 2% |
| **可分配空间** | ~50% | ~98% |

→ **CMS 时代实际可用空间只有 ~50%**。

## 三、RosAlloc 分桶导致的碎片化

### 3.6.6 size class 的不可调和

```cpp
// RosAlloc 的 size class 列表
static const size_t kSizeClasses[] = {
    16, 24, 32, 40, 48, 56, 64,
    72, 80, 88, 96, 104, 112, 120, 128,
    // ...
    3072, 4096
};
```

**问题**：相邻 size class 之间不能复用空闲 slot。

```
Run A (32B): 100 个 slot，70 个空闲
Run B (40B): 100 个 slot，50 个空闲

需要分配 32B 对象 → Run A 有空间 → OK
需要分配 40B 对象 → Run B 有空间 → OK
需要分配 36B 对象 → 实际需要 40B size class → Run B
                  → 但如果 Run B 满了 → 申请新 Run → 碎片化
```

### 3.6.7 分桶碎片的数学分析

**假设**：业务对象大小服从均匀分布 U(16, 4096)

```cpp
// 36 个 size class，每个 size class 有独立的 Run
// 业务对象 1/N 概率落入每个 size class
// N = 36 个 size class

// 假设每个 size class 的 Run 平均使用率 30%
// 那么每个 Run 的空闲率 70%

// 但单个 size class 满的可能性很高（泊松分布）
// P(Run 满) = e^(-30% × N) → N 大时接近 0

// 但如果某 size class 突然热门（如短时间内大量同大小对象）：
// → 该 size class 的 Run 频繁申请 → 碎片化
```

### 3.6.8 分桶碎片的实际表现

**场景**：图片列表 App

```
图片大小变化：
  缩略图：~10 KB
  中等图：~50 KB
  大图：~500 KB

ROSAlloc 分桶：
  缩略图 → size class 12 KB（10 KB 对象 + 内部碎片）
  中等图 → size class 56 KB（50 KB 对象 + 内部碎片）
  大图 → size class 512 KB（500 KB 对象 + 内部碎片）

→ 3 个 size class 各自管理
→ 不能跨桶合并
→ 总碎片率 ~15%
```

## 四、LOS 碎片化详解

### 3.6.9 LOS 的不压缩特性

CMS 时代 LOS 的处理：
```cpp
void MarkSweep::SweepLargeObjects() {
    for (LargeObject* obj : large_object_space_->GetObjects()) {
        if (!mark_bitmap_->Test(obj)) {
            // 死亡 → 释放
            large_object_space_->Free(obj);
            // ❌ LOS 不压缩，释放后留下空洞
        }
    }
}
```

**LOS 空洞的形成**：
```
初始：LOS 连续空间 [0x10000, 0x100000] (1 MB)

分配 Bitmap A (4 MB) → 占 [0x10000, 0x500000]
分配 Bitmap B (8 MB) → 占 [0x500000, 0xD00000]
释放 Bitmap A → 占 [0x500000, 0xD00000]，空洞 [0x10000, 0x500000]

→ 空洞 4 MB
→ 新分配 3 MB Bitmap → 可以（空洞 4 MB 足够）
→ 新分配 5 MB Bitmap → 不可以（空洞只有 4 MB，不连续 5 MB）
```

### 3.6.10 LOS 碎片化的实际案例

**Glide 缓存场景**：

```
Glide 加载不同大小的图片：
- 全屏图：1080×1920 = 8 MB
- 半屏图：1080×960 = 4 MB
- 缩略图：540×960 = 2 MB

加载流程：
1. 加载全屏图 → 分配 8 MB LOS（LOS #1）
2. 加载半屏图 → 分配 4 MB LOS（LOS #2）
3. 加载缩略图 → 分配 2 MB LOS（LOS #3）
4. Glide 缓存淘汰 → 释放半屏图
5. 加载全屏图 → 分配 8 MB LOS

→ 但 LOS #2 释放后留下 4 MB 空洞
→ 全屏图 8 MB 找不到连续 8 MB 空间
→ 即使 LOS 总空闲 12 MB（4 MB 空洞 + 2 MB 缩略图 + 6 MB 其他）
→ OOM！
```

### 3.6.11 LOS 碎片化的解决方案

**方案 1：及时 recycle() Bitmap**

```java
// 优化前
public void onBitmapReleased(Bitmap bitmap) {
    // 忘记 recycle
    bitmap = null;
}

// 优化后
public void onBitmapReleased(Bitmap bitmap) {
    if (bitmap != null && !bitmap.isRecycled()) {
        bitmap.recycle();  // 立即释放
    }
    bitmap = null;
}
```

**方案 2：Bitmap inBitmap 复用**

```java
// 复用相同大小的 Bitmap
BitmapFactory.Options options = new BitmapFactory.Options();
options.inBitmap = reusableBitmap;  // 复用
options.inSampleSize = 2;
Bitmap bitmap = BitmapFactory.decodeFile(path, options);
```

**方案 3：分块大 Bitmap**

```java
// 把大 Bitmap 切成小块
// 避免单个大 Bitmap 占满 LOS
Bitmap[] tiles = new Bitmap[16];
int tileSize = 256;
for (int i = 0; i < 16; i++) {
    tiles[i] = Bitmap.createBitmap(tileSize, tileSize, Bitmap.Config.ARGB_8888);
}
```

**方案 4：主动管理 LOS 缓存**

```java
// LRU 缓存，控制 LOS 总占用
public class LRUBitmapCache {
    private LinkedHashMap<String, Bitmap> cache;
    private long maxLOSUsage;
    
    public void put(String key, Bitmap bitmap) {
        // 检查 LOS 占用
        long bitmapSize = bitmap.getByteCount();
        if (currentLOSUsage + bitmapSize > maxLOSUsage) {
            evictUntil(bitmapSize);  // 淘汰直到能放下
        }
        cache.put(key, bitmap);
    }
}
```

## 五、CMS 时代的 OOM 模式

### 3.6.12 模式 1：Allocation Space 满了

**症状**：
```
dumpsys meminfo:
  Dalvik Heap: 250 MB / 256 MB (97% 使用)
  → 真实 OOM
```

**根因**：业务对象持续创建，没有及时释放。

**修复**：
- 修复泄漏
- 减少对象创建
- 优化堆大小

### 3.6.13 模式 2：LOS 碎片化导致 OOM

**症状**：
```
dumpsys meminfo:
  Dalvik Heap: 150 MB / 256 MB (58% 使用)
  → 但分配 5 MB Bitmap 失败 → OOM
```

**根因**：LOS 碎片化，总空闲足够但没连续大空间。

**修复**：
- 及时 recycle() Bitmap
- 用 inBitmap 复用
- 分块大 Bitmap

### 3.6.14 模式 3：Allocation Space 满了 + LOS 也有空洞

**症状**：
```
dumpsys meminfo:
  Dalvik Heap: 250 MB / 256 MB
  LOS: 50 MB 总空闲，但都是小空洞
  → 任何大 Bitmap 分配都失败
```

**根因**：双重碎片化。

**修复**：
- 全面优化 Bitmap 生命周期
- 减小 Bitmap 大小
- 启用 largeHeap（不推荐）

### 3.6.15 模式 4：GC 频率过高

**症状**：
```
ART GC 日志：
  Background concurrent copying GC freed 0(0B) AllocSpace objects
  → 每分钟 GC 30+ 次，但释放 0 字节
```

**根因**：CMS Sweep 后碎片化，每次分配都触发 GC。

**修复**：
- 优化对象生命周期
- 减少分配频率

### 3.6.16 模式 5：触发 GC 后仍失败

**症状**：
```
art : Background concurrent copying GC freed 0(0B) AllocSpace objects
art : OutOfMemoryError: Failed to allocate a 4194304 byte allocation
```

**根因**：CMS GC 后仍然碎片化，没有连续大空间。

**修复**：
- 综合优化（业务 + Heap 参数）

## 六、CMS 时代 vs CC GC 时代

### 3.6.17 碎片化对比

| 维度 | CMS（Android 5-7） | CC GC（Android 8+） |
|:---|:---|:---|
| **Allocation Space** | 标记-清除 + 分桶 | 标记-复制 + Region |
| **LOS** | 标记-清除 | 标记-清除（仍碎片化） |
| **总碎片率** | 30-60% | < 2% |
| **可分配空间** | ~50% | ~98% |
| **OOM 风险** | 高 | 低（除 LOS） |
| **碎片化修复** | 只能预防（业务层） | 自动（GC 整理） |

### 3.6.18 CC GC 如何解决碎片化

**Region 整体回收**：
```
CC GC 把所有存活对象从 from-space 复制到 to-space
→ 整个 Region 一次性回收
→ 不存在 slot 级别的外碎片
→ Region 内部碎片 < 1%
```

**LOS 仍有碎片化**：
```
CC GC 时代 LOS 仍是标记-清除
→ LOS 仍然碎片化
→ 大 Bitmap 仍然要 recycle / inBitmap
→ 详见 3.6 节
```

## 七、CMS 碎片化的监控

### 3.6.19 dumpsys meminfo 解读

```bash
$ adb shell dumpsys meminfo com.example.app

# 关键字段
  Dalvik Heap      234567   200000    34567      500  280000    65536    23456    42080
                                                              ↑        ↑       ↑
                                                              Size    Alloc   Free
                                                              64 MB   22 MB   41 MB
```

**判断碎片化**：
- 如果 Alloc ≈ Size → 真实 OOM
- 如果 Alloc << Size → 可能是碎片化（需要 hprof 分析）

### 3.6.20 hprof + MAT 分析碎片化

```bash
# 1. 生成 hprof
adb shell am dumpheap <pid> /data/local/tmp/oom.hprof
adb pull /data/local/tmp/oom.hprof
hprof-conv oom.hprof oom-conv.hprof

# 2. 用 MAT 分析
# - Histogram：找最大的对象
# - Top Consumers：找占用最多的类
# - Dominator Tree：找保留堆最大的对象
```

## 九、本节小结

1. **CMS 碎片化是必然**：不压缩 + 分桶 + LOS 标记-清除
2. **碎片化导致"虚假 OOM"**：堆里还有空闲但分配失败
3. **LOS 碎片化是最大问题**：大 Bitmap / byte[] 释放后留下空洞
4. **CC GC 通过 Region 整体回收解决 Allocation Space 碎片化**
5. **LOS 即使在 CC GC 时代仍有碎片化**（业务层必须管理）

→ **理解 CMS 碎片化，就理解了"为什么必须升级到 Android 8.0+"**。
