## 0. 怎么用本路线图

**两个铁律**先立住：

1. **机制、症状、工具、治理，四视角必须打通** —— 单跑任何一条线都只是「半吊子」稳定性架构师。
2. **线上 → 机制 → 取证 → 治理 → 案例** 是闭环 —— 学完不能落地到「线上少 1 个 bug」「门禁少 1 个漏网」= 没学。

**与 [L00 稳定性+性能路线图](../02-Symptom/README-学习路线.md) 的关系**：

- **本篇**：以 **稳定性架构师** 视角出发，覆盖 8 大分类全内容地图，含治理/案例/前沿
- **L00**：以 **稳定性 + 性能** 视角出发，专注 02-Symptom + 1-Mechanism 关键子系统，更适合「性能架构师」或「稳定性 + 性能」复合角色
- **本篇是 L00 的超集**：L00 的 Phase 1-4 等价于本篇的 阶段 1-3，**本篇多了治理 / 案例 / 前沿**

## 1. 角色画像：稳定性架构师的 4 层能力模型

```
                ┌──────────────────────────────────────────┐
                │  稳定性架构师 = 4 层能力 + 1 条闭环           │
                └──────────────────────────────────────────┘

  第 1 层：机制理解 ─── 看得懂源码，定位到 syscall / 函数 / 行
         │
  第 2 层：症状识别 ─── 线上现象 → 根因模块（7 大症状 × 5 类子系统）
         │
  第 3 层：取证落地 ─── 抓得到 dump，解读出 stack / heap / trace
         │
  第 4 层：治理体系 ─── 防得住复发，量化得清楚，灰度得稳
         │
         └─────────→ 闭环：所有线上问题都要反哺到治理/门禁/工具 ──→
```

**与 5 个邻近角色的边界**（不要越界，但要知道边界）：

| 角色 | 关注点 | 区分点 | 互引关系 |
|:-----|:-------|:-------|:---------|
| **应用开发者** | 单 App 内存/卡顿/崩溃 | 只看自进程 | 把 App 问题按本文 §4 找到对应机制 |
| **性能架构师** | 启动 / 流畅度 / 电量 | 偏 SLI/SLO 量化 | 用 L00 路线图，本篇是补充 |
| **BSP / 系统集成** | HAL / Kernel / 驱动 | 偏底层 | 用 [06-Foundation/Build-System](../06-Foundation/Build-System/) 系列 |
| **oncall 工程师** | 7×24 应急响应 | 偏 playbook | 用 [03-Forensics](../03-Forensics/) 系列 |
| **稳定性架构师**（你） | 机制 + 症状 + 取证 + 治理 | 全栈闭环 | 上面 4 个角色都要懂，但不能只做一面 |

## 12. 学习方法建议

### 12.1 阅读节奏（推荐"3-2-1"节奏）

- **3**：每天 3 篇机制文（深度读，含源码）
- **2**：每天 2 篇工具/案例文（边读边练）
- **1**：每天 1 次实战（哪怕只是验证一个 dumpsys 命令）

### 12.2 实战与笔记

- **必做**：每个 Phase 结束时，写 1 篇 500-1500 字的"复盘笔记"
  - Phase 1 结束：写"我理解的 1 个子系统"（如：binder 红黑树）
  - Phase 2 结束：写"我的 7 大症状速查表"
  - Phase 3 结束：写"我的取证 SOP"
  - Phase 4 结束：写"我的 SLI 设计"
- **必做**：每个 Phase 至少 1 次实操（编译 / 刷机 / 抓 dump / 跑 trace）

### 12.3 复盘与输出

- **每季度**：挑 1 个真实线上问题，按 "症状 → 机制 → 取证 → 治理" 写 1 篇复盘
- **每半年**：回头 review 路线图，标记哪些已经吃透、哪些需要补
- **持续**：跟踪 AOSP / K release notes，更新 §11 缺项表

### 12.4 反例（来自项目本身的 [PROMPT-技术系列文章写作指南.md](../PROMPT-技术系列文章写作指南.md) §5）

- ❌ 只读不练（看完就忘）
- ❌ 跨阶段跳读（机制没立住就读症状）
- ❌ 收藏式阅读（收藏 = 没读）
- ❌ 孤立学习（不跟同事讨论 / 不进 oncall 群）
- ❌ 闭门造车（不读官方 release notes / 不读同行 PR）

```
Process_Exit 01 (全链路) → 02 (do_exit 9 步) → 03 (慢的根因) → 04 (监控治理)
  → Kernel Process 05 (do_exit 与资源回收) → Memory Management 09 (LMKD)
  → Process 13 (进程调试与稳定性关联)
```

### 13.3 想理解"一次 ANR 是怎么发生的"

```
S01-ANR 症状 → F01-ANR 取证 → ANR-Detection 3 篇 → Input 06 (Input ANR)
  → Activity 07 (Launch ANR) → Handler 06 (Handler ANR) → Service 07 (Service ANR)
  → ART 06 (Signal Catcher) → 治理 (APM / AI for Stability F04)
```

### 13.4 想理解"一次 Native Crash 是怎么发生的"
