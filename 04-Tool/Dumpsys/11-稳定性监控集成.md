# 1. 背景：`dumpsys dropbox` 是什么？

## 1.1 一句话定位

**`dumpsys dropbox` 是 80% 稳定性 P0 工单的"统一取证入口"——任何 NE/JE/SWT/HANG 都会在 dropbox 留痕，drops 拉取是稳定性架构师第一动作**。

## 1.2 4 大子命令全景

```
adb shell dumpsys dropbox [subcmd]
  ├─ (无参数)             → 标签列表
  ├─ --print <tag>        → 单标签输出
  ├─ --print --all <tag>  → 单标签 + 全部内容
  └─ --system             → 系统级强制 dump

adb shell dumpsys crash
  └─ 触发 crash（测试用）
```

## 1.3 与稳定性症状的对应关系（**核心**）

| 症状 | dropbox 标签 | 配合 dumpsys |
|:-----|:-------------|:-------------|
| **ANR** | `APP_ANR` | `dumpsys activity broadcasts` + `dumpsys input` |
| **NE 崩溃** | `SYSTEM_TOMBSTONE` | `dumpsys dropbox --print SYSTEM_TOMBSTONE` |
| **JE 崩溃** | `APP_CRASH` | `dumpsys dropbox --print APP_CRASH` |
| **SWT** | `SYSTEM_SERVER_WATCHDOG` | `dumpsys activity processes` |
| **系统重启** | `SYSTEM_RESTART` | `dumpsys batterystats` |
| **HANG（无 dropbox）** | （无）| `dumpsys input` + `dumpsys meminfo` |
| **KE** | `SYSTEM_TOMBSTONE` | `dumpsys dropbox --system` |

> **所以呢**：dropbox 是 7 大症状中 5 类（ANR/NE/JE/SWT/REBOOT/KE）的"统一入口"——`HANG` 是本系列独占视角（[Stability S05](../02-Symptom/S00-稳定性症状总览.md)）。

# 2. 边界：`dumpsys dropbox` vs `bugreport`

| 工具 | 看什么 | dumpsys dropbox 不能给什么 |
|:-----|:-------|:--------------------------|
| **`dumpsys dropbox`** | 历史事件 | 不能改 |
| **`bugreport`** | 全部系统状态 | 体积大（100MB+）|
| **`dumpsys dropbox --system`** | 强制 dump | — |

# 3. 机制：4 大子命令深挖

## 3.1 `dumpsys dropbox`（标签列表）

### 3.1.1 典型输出

```bash
$ adb shell dumpsys dropbox
```

```
Dropbox Service (dumpsys dropbox)
  
  Tags (dumpsys dropbox):
    APP_CRASH (5 entries)
    APP_ANR (3 entries)
    SYSTEM_TOMBSTONE (2 entries)
    SYSTEM_SERVER_WATCHDOG (1 entries)
    SYSTEM_RESTART (1 entries)
    BATTERY_DISCHARGE_INFO
    KERNEL_PANIC_CONSOLE
    ...
```

### 3.1.2 12 个核心标签

| 标签 | 触发场景 | 保留期 |
|:-----|:---------|:-------|
| **APP_CRASH** | Java Exception 未 catch | 7 天 |
| **APP_ANR** | 4 类 ANR | 7 天 |
| **SYSTEM_TOMBSTONE** | NE 崩溃（debuggerd）| 30 天 |
| **SYSTEM_SERVER_WATCHDOG** | SWT（杀 SystemServer）| 30 天 |
| **SYSTEM_RESTART** | 整机重启 | 30 天 |
| **BATTERY_DISCHARGE_INFO** | 放电统计 | 30 天 |
| **KERNEL_PANIC_CONSOLE** | KE | 30 天 |
| **KERNEL_PANIC_THREAD** | KE 多线程 | 30 天 |
| **KERNEL_OOPS** | oops（非 panic）| 30 天 |
| **HUNG_TASK_RECORDS** | hung_task | 30 天 |
| **SYSTEM_FSCK** | 文件系统检查 | 30 天 |
| **APP_FUNCTIONS** | AI 任务失败（AOSP 17+）| 7 天 |

## 3.2 `dumpsys dropbox --print <tag>`（单标签输出）

### 3.2.1 典型输出

```bash
$ adb shell dumpsys dropbox --print APP_CRASH
```

```
Dropbox: APP_CRASH
  Entry #0: 2026-07-18 10:23:45
  =================
  Process: com.example.app
  Thread: main
  Exception: java.lang.NullPointerException
    at com.example.app.MainActivity.onCreate(MainActivity.java:42)
    at android.app.Activity.performCreate(Activity.java:8000)
    ...
  
  Entry #1: 2026-07-18 10:25:12
  =================
  Process: com.example.app
  ...
```

### 3.2.2 关键字段

| 字段 | 含义 | 异常判定 |
|:-----|:-----|:---------|
| **timestamp** | 事件时间 | — |
| **Process** | 进程名 | — |
| **Exception** | 异常类型 | NPE / OOM / ConcurrentModification |
| **stack trace** | 栈轨迹 | 看是哪一行代码 |

## 3.3 `dumpsys dropbox --system`（系统级强制 dump）

### 3.3.1 用途

强制 dump 系统级 dropbox——用于系统级 NE / KE 取证。

### 3.3.2 典型输出

```bash
$ adb shell dumpsys dropbox --system
```

会强制 dump 所有 SYSTEM_* 标签。

## 3.4 `dumpsys crash`（触发 crash · 测试用）

### 3.4.1 用途

**测试用**——给指定应用发一个 native crash 信号，验证 APM 是否能采集到。

### 3.4.2 实战命令

```bash
# 触发指定应用 crash
adb shell am crash com.example.app

# 验证 dropbox
adb shell dumpsys dropbox --print APP_CRASH | tail -20
```

# 4. 风险地图与解读阈值

## 4.1 12 个标签与触发场景

| 标签 | 触发条件 | 保留期 | 治理 |
|:-----|:---------|:-------|:-----|
| **APP_CRASH** | Java Throwable 未 catch | 7 天 | 必查 |
| **APP_ANR** | 4 类 ANR | 7 天 | 必查 |
| **SYSTEM_TOMBSTONE** | NE 崩溃 | 30 天 | 必查 |
| **SYSTEM_SERVER_WATCHDOG** | SystemServer 卡死 | 30 天 | 必查 |
| **SYSTEM_RESTART** | 整机重启 | 30 天 | 必查 |
| **BATTERY_DISCHARGE_INFO** | 放电完成 | 30 天 | 参考 |
| **KERNEL_PANIC_CONSOLE** | Kernel panic | 30 天 | 必查 |
| **KERNEL_PANIC_THREAD** | Kernel panic 多线程 | 30 天 | 必查 |
| **KERNEL_OOPS** | oops | 30 天 | 必查 |
| **HUNG_TASK_RECORDS** | D 状态超阈值 | 30 天 | 必查 |
| **SYSTEM_FSCK** | fs 检查 | 30 天 | 参考 |
| **APP_FUNCTIONS** | AI 任务失败 | 7 天 | 必查 |

## 4.2 关键阈值

| 阈值 | 数值 | 含义 |
|:-----|:-----|:-----|
| **dropbox 保留** | 7-30 天 | 由系统决定 |
| **dropbox 满后** | 覆盖最早 | 高发期会丢 |
| **dropbox 单条大小** | < 1MB | 50-500KB |
| **dropbox 总大小** | 系统配额 | 100MB |

# 5. 治理：dropbox 接入 APM 体系

## 5.1 接入方式 A：客户端 SDK 主动读

```python
# APM SDK 伪代码
def on_user_complaint(package_name):
    # Step 1: 拉 dropbox 该应用的所有条目
    result = run_adb(f"dumpsys dropbox --print APP_CRASH | grep {package_name}")
    # Step 2: 拉 dropbox 该应用的 ANR
    anr = run_adb(f"dumpsys dropbox --print APP_ANR | grep {package_name}")
    # Step 3: 解析
    parse_and_upload(result, anr)
```

## 5.2 接入方式 B：服务端按需拉取

```python
# 服务端：收到工单后按需拉
def on_p0_incident(device_id, package_name):
    tags = ["APP_CRASH", "APP_ANR", "SYSTEM_TOMBSTONE"]
    for tag in tags:
        result = ssh_run(device_id, f"dumpsys dropbox --print {tag}")
        parse_and_upload(result, package_name)
```

## 5.3 接入方式 C：bugreport 解析

```python
# 服务端：解析 bugreport
def parse_bugreport(bugreport_zip):
    extract(bugreport_zip, "/tmp/bugreport")
    # 找 dropbox 段
    with open("/tmp/bugreport/dropbox.txt") as f:
        content = f.read()
    # 按 tag 拆分
    tags = parse_dropbox_tags(content)
    return tags
```

## 5.4 dropbox 自动采集频率

| 场景 | 频率 | 采集内容 | 数据量 |
|:-----|:-----|:---------|:-------|
| **P0 工单到达** | 1 次 | 全部 dropbox | ~10MB |
| **定时巡检** | 1 小时 | APP_CRASH / APP_ANR | ~500KB |
| **系统重启后** | 1 次 | SYSTEM_RESTART / SYSTEM_TOMBSTONE | ~5MB |

# 6. 实战案例

## 6.1 CASE-DUMPSYS-11-01 拉取用户所有崩溃

**场景**：用户报"应用频繁崩溃"，拉取所有崩溃历史。

**操作时序**：

```bash
# T+0s: 拉 APP_CRASH 段
$ adb shell dumpsys dropbox --print APP_CRASH | grep -B 2 -A 20 "com.example.app"
  Entry #0: 2026-07-18 10:23:45
    Process: com.example.app
    Exception: java.lang.NullPointerException
    at com.example.app.MainActivity.onCreate(MainActivity.java:42)
    # 5 次 NPE
  Entry #1: 2026-07-18 10:25:12
    Exception: java.lang.NullPointerException
    ...
  # 共 23 次崩溃，全部 NPE

# T+30s: 拉 APP_ANR
$ adb shell dumpsys dropbox --print APP_ANR | grep -B 2 -A 10 "com.example.app"
  Entry #0: 2026-07-18 10:30:00
    Process: com.example.app
    Reason: Input dispatching timed out
    # 5s ANR
```

**根因定位**：
- 23 次 NPE，全部在 MainActivity.onCreate
- 5s Input ANR = 主线程卡
- **根因**：onCreate 同步加载 NPE + 卡死

**修复方案**：
1. 加空值检查
2. onCreate 异步化

## 6.2 CASE-DUMPSYS-11-02 NE 现场取证

**场景**：用户报"应用闪退，弹 native crash 弹窗"。

**操作时序**：

```bash
# T+0s: 拉 SYSTEM_TOMBSTONE
$ adb shell dumpsys dropbox --print SYSTEM_TOMBSTONE
  Entry #0: 2026-07-18 10:23:45
    Build fingerprint: 'AOSP/...'
    Revision: '0'
    ABI: 'arm64-v8a'
    ...
    pid: 12345, tid: 12345, name: com.example.app
    signal 11 (SIGSEGV), code 1 (SEGV_MAPERR)
    fault addr 0x0
    ...
    backtrace:
      #00 pc 0x... /data/app/com.example.app/lib/arm64/libnative.so
      #01 pc 0x... /data/app/com.example.app/lib/arm64/libnative.so
      ...
```

**根因定位**：
- SIGSEGV 11 = 段错误
- fault addr 0x0 = 空指针
- 栈在 libnative.so

**修复方案**：
- 用 addr2line 符号化栈
- 检查 native 代码的 null pointer

# 7. 总结

## 7.1 核心要诀（背下来）

1. **`dumpsys dropbox` 是 80% 稳定性 P0 的统一入口**
2. **12 个核心标签**——5 类症状（ANR/NE/JE/SWT/REBOOT/KE）都有专属
3. **`HANG` 不在 dropbox**——见 [Stability S05](../02-Symptom/S00-稳定性症状总览.md)
4. **dropbox 保留 7-30 天**——满后覆盖
5. **`dumpsys crash` 是测试触发器**——验证 APM 链路

## 7.2 5 条 Takeaway

1. **`APP_CRASH` = Java 崩溃**
2. **`SYSTEM_TOMBSTONE` = NE 崩溃**
3. **`APP_ANR` = 4 类 ANR**
4. **`SYSTEM_SERVER_WATCHDOG` = SWT**
5. **`HANG` 需主动诊断**（dropbox 不会自动记录）
