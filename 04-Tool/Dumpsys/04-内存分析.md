# 1. 背景：内存分析三件套是什么？

## 1.1 一句话定位

**`dumpsys meminfo` / `procrank` / `procstats` 是 Android 内存分析的"三件套"——一个看当前、一个看实时排序、一个看历史，组合起来能 90% 覆盖线上内存问题**。

## 1.2 三件套全景

| 工具 | 类型 | 看什么 | 输出 | 典型场景 |
|:-----|:-----|:-------|:-----|:---------|
| **`dumpsys meminfo`** | 应用内 | 进程详细内存（Java/Native/Graphics/Stack/Code/Other）| 文本 | OOM 取证 |
| **`dumpsys meminfo -d`** | 应用内 | 更详细 Dalvik/ART 内存 | 文本 | 详细分析 |
| **`dumpsys meminfo --proto`** | 应用内 | protobuf 格式 | 二进制 | 程序化分析 |
| **`procrank`** | 外部命令 | 系统所有进程 PSS 排序 | 文本 | 谁占内存最多 |
| **`dumpsys procstats`** | 系统服务 | 进程内存使用历史 + trim level | 文本 | 内存长时间演变 |

## 1.3 三件套适用场景对比

```
                        实时性
                          ↑
                          │
              procrank ●  │
                          │  dumpsys meminfo
                          │  (看单包详情)
                          │
                          │           procstats
                          │           (看历史)
                          │
                          └────────────────────→ 详细度
```

| 场景 | 优先工具 |
|:-----|:---------|
| 1. 应用 OOM 崩溃 | `dumpsys meminfo <pkg>` 看详细 |
| 2. 系统占内存最多的进程 | `procrank` |
| 3. 内存使用历史 | `dumpsys procstats` |
| 4. 应用内存泄漏确认 | `dumpsys meminfo <pkg>` 看 Views/Activities 增长 |
| 5. GC 频繁诊断 | `dumpsys meminfo <pkg>` + `dumpsys gfxinfo <pkg>`（看 GC 时间）|
| 6. 大图内存占用 | `dumpsys meminfo <pkg>` 看 Native Heap + Graphics |

## 1.4 与稳定性症状的对应关系

| 症状 | 优先 dumpsys | 关键字段 | 异常判定 |
|:-----|:-------------|:--------|:---------|
| **OOM 崩溃** | `dumpsys meminfo <pkg>` | TOTAL PSS、Native Heap | PSS > 500MB |
| **内存泄漏** | `dumpsys meminfo <pkg>` | Views/Activities/Contexts | 单调增长 |
| **GC 频繁** | `dumpsys meminfo <pkg>` | GC 时间 | GC > 5% CPU |
| **Native 内存高** | `dumpsys meminfo <pkg>` | Native Heap | > 100MB |
| **大图** | `dumpsys meminfo <pkg>` | Graphics | > 50MB |
| **App 占用总 PSS 高** | `procrank` | PSS 排序 | 该包在前 10 |

> **所以呢**：OOM / 内存泄漏 80% 都走 `dumpsys meminfo <pkg>` 一个命令。

# 2. 边界：meminfo vs hprof vs proc

| 工具 | 看什么 | dumpsys meminfo 不能给什么 |
|:-----|:-------|:--------------------------|
| **`dumpsys hprof`**（其实是 `am dumpheap`）| 完整 Java 堆对象图 | dumpsys meminfo 只给计数，不给对象图 |
| **`procrank`** | 系统所有进程 | dumpsys meminfo 只给单包 |
| **`cat /proc/<pid>/status`** | Linux 内核视角 | dumpsys meminfo 不含线程数等内核字段 |
| **`dumpsys procstats`** | 历史 + trim | dumpsys meminfo 不含历史 |

> **所以呢**：内存泄漏的"根因定位"必须 `dumpsys meminfo`（看趋势）+ `am dumpheap`（看对象图）组合。

# 3. 机制：5 大子命令深挖

## 3.1 `dumpsys meminfo`（默认 · 6 大内存段）

### 3.1.1 6 大内存段

AOSP 17 的 `dumpsys meminfo` 把进程内存分成 **6 大段**：

```
1. Java Heap        ← Java 对象 + Dalvik/ART 运行时
2. Native Heap      ← C/C++ 分配（Bitmap 像素 / JNI 缓冲）
3. Graphics         ← GPU 缓冲 / Surface 缓冲 / Texture
4. Stack            ← 线程栈
5. Code             ← .so / .dex / .oat
6. Other            ← 字体 / 设备特定 / GL 命令
```

$ adb shell dumpsys meminfo com.example.app

### 3.1.3 关键字段表

#### A. 6 大段解读

| 段 | 单位 | 正常范围 | 异常判定 |
|:---|:-----|:---------|:---------|
| **Native Heap** | KB | <50MB | >100MB 异常（Bitmap 嫌疑）|
| **Java Heap** | KB | <30MB | >100MB 异常（对象泄漏）|
| **Graphics** | KB | <30MB | >50MB 异常（大图 / Surface 泄漏）|
| **Stack** | KB | <2MB | 线程数过多 = Stack 大 |
| **Code** | KB | 10-30MB | 固定，与安装包大小相关 |
| **Other** | KB | <5MB | 异常字体加载 = 风险 |

#### B. 关键对象计数

| 对象 | 正常范围 | 异常判定 |
|:-----|:---------|:---------|
| **Views** | <1000 | >5000 异常（View 泄漏）|
| **Activities** | <10 | >30 异常（Activity 泄漏）|
| **AppContexts** | <10 | >30 异常（Context 泄漏）|
| **ViewRootImpl** | <10 | >30 异常（窗口泄漏）|
| **WebViews** | <5 | 长期存在 = WebView 泄漏 |
| **Pending Keys** | 0 | >0 异常（InputManager 泄漏）|
| **Local Binders** | <50 | >200 异常（Binder 泄漏）|

#### C. 关键汇总

| 字段 | 含义 | 异常判定 |
|:-----|:-----|:---------|
| **TOTAL PSS** | 进程总内存 | >500MB 严重 |
| **TOTAL RSS** | 进程物理内存 | 与 PSS 差过大 = 共享库多 |
| **Heap Size** | 堆总大小 | 持续增长 = Java 堆泄漏 |
| **Heap Alloc** | 堆已用 | 长期高位 = GC 异常 |

### 3.1.4 实战命令

```bash
# 1. 看某包总览
adb shell dumpsys meminfo com.example.app

# 2. 看 PSS 前 10 段
adb shell dumpsys meminfo com.example.app | grep -E "PSS|Heap"

# 3. 看对象计数
adb shell dumpsys meminfo com.example.app | grep -A 10 "Objects"

# 4. 看 Native Heap 详细（Bitmap 嫌疑）
adb shell dumpsys meminfo com.example.app | grep -A 5 "Native Heap"
```

## 3.3 `dumpsys meminfo --proto`（protobuf 格式）

### 3.3.1 用途

`--proto` 输出 protobuf 格式，可直接用 protoc 解析，便于：
- 程序化分析
- 与 APM 平台集成
- 长期数据收集

### 3.3.2 使用方法

```bash
# 1. 抓取 proto 输出
adb shell dumpsys meminfo --proto com.example.app > meminfo.proto

# 2. 用 protoc 解析
protoc --decode_raw < meminfo.proto

# 3. 或者在 Python 里解析
import subprocess
result = subprocess.run(
    ["protoc", "--decode_raw"],
    stdin=open("meminfo.proto", "rb"),
    capture_output=True
)
print(result.stdout.decode())
```

### 3.3.3 集成示例

```python
# APM SDK 伪代码
import subprocess
import google.protobuf

def collect_meminfo_proto(package_name):
    """采集 meminfo protobuf 格式"""
    result = subprocess.run(
        ["adb", "shell", "dumpsys", "meminfo", "--proto", package_name],
        capture_output=True, text=True
    )
    return parse_meminfo(result.stdout)
```

## 3.4 `procrank`（外部命令 · PSS 排序）

### 3.4.1 用途

`procrank` 是 system/extras 里的工具，**按 PSS 排序输出所有进程的内存**——快速看"谁占内存最多"。

### 3.4.2 典型输出

```bash
$ adb shell procrank
  PID       Vss      Rss      Pss      Uss  Swap    SwapPSS   Name
  12345   234567   145678   123456   100000     0         0  com.example.app
   1234   167890   100000    89012    70000     0         0  system_server
   5678   123456    78901    67890    50000     0         0  com.android.systemui
   ...
```

### 3.4.3 关键字段

| 字段 | 含义 | 用途 |
|:-----|:-----|:-----|
| **Vss** | 虚拟内存 | 不可靠（共享库算多次）|
| **Rss** | 物理内存 | 总占用 |
| **Pss** | 按比例分摊的物理内存 | **最常用**（公平比较）|
| **Uss** | 独占物理内存 | 进程独占的（泄漏的"真实"大小）|
| **Name** | 进程名 | 排序后看哪包占内存 |

### 3.4.4 实战命令

```bash
# 1. 排序看 PSS 前 10
adb shell procrank | head -11

# 2. 找某包的 PSS
adb shell procrank | grep "com.example.app"

# 3. 找 PSS > 300MB 的进程
adb shell procrank | awk '$4 > 300000'

# 4. 持续监控（每秒刷新）
adb shell "while true; do procrank | head -3; sleep 1; done"
```

### 3.5.1 用途

`procstats` 是 ProcessStatsService 的 dump 接口——记录**过去 N 小时**每个进程的内存使用历史 + trim level 决策。

# 4. 风险地图与解读阈值

## 4.1 11 类内存异常模式

| # | 异常模式 | 关键字段 | 异常判定 | 修复方向 |
|:--|:--------|:--------|:---------|:---------|
| 1 | **Java 对象泄漏** | Java Heap + Views/Activities | 单调增长 | Hprof 找 root |
| 2 | **Native 内存泄漏** | Native Heap | 单调增长 | 排查 JNI / Bitmap |
| 3 | **Bitmap 像素占用大** | Native Heap + Graphics | >100MB | 排查 Glide / 缓存 |
| 4 | **Surface 泄漏** | Graphics | 单调增长 | 查 SurfaceView / TextureView |
| 5 | **Activity 泄漏** | Activities > 30 | 单调增长 | 查 Context 引用 |
| 6 | **View 泄漏** | Views > 5000 | 单调增长 | 查 Handler 持有 View |
| 7 | **Window 泄漏** | ViewRootImpl > 30 | 单调增长 | Dialog 没 dismiss |
| 8 | **WebView 泄漏** | WebViews | 长期存在 | 加 destroy() |
| 9 | **线程泄漏** | Threads (proc/<pid>/status) | >500 | 排查线程池 |
| 10 | **Binder 泄漏** | Local/Proxy Binders | >500 | 排查 unlinkToDeath |
| 11 | **大对象（LOS）** | .LOS | >10MB | 排查大 List / 大数组 |

## 4.2 OOM 6 大根因

| 根因 | dumpsys 表现 | 修复方向 |
|:-----|:-------------|:---------|
| **1. 堆上限不足** | `Heap Size` 触顶 | 调 largeHeap / Nougat split |
| **2. 内存碎片** | `Heap Size` 大、`Heap Alloc` 小 | 触发 GC / 重启进程 |
| **3. 大对象** | `.LOS` 段大 | 用 SparseArray / 拆分 |
| **4. Native 内存高** | Native Heap 大 | Bitmap 压缩 / 缓存上限 |
| **5. 内存泄漏** | 总 PSS 单调增长 | 查 Hprof |
| **6. 进程分配内存过多** | 长期高位 | 加 Trim 监听 / 主动释放 |

## 4.3 dumpsys meminfo 自身风险

| 风险 | 触发条件 | 后果 | 规避 |
|:-----|:---------|:-----|:-----|
| **跨进程 dump 阻塞** | `dumpsys meminfo <pkg>` | App 主线程暂停 100-500ms | 提前告知用户 |
| **Proto 输出大** | `--proto` | 输出 1-10MB | 用 `--proto` 时跳过人类可读 |
| **procrank 需 root** | `procrank` | 没 root 不能用 | 用 `dumpsys meminfo` 替代 |

# 6. 实战案例

## 6.1 CASE-DUMPSYS-04-01 内存泄漏（用户报"应用越来越卡"）

**场景**：某应用启动 30 分钟后明显卡顿，重启后恢复。

**操作时序**（10 分钟）：

```bash
# T+0s: 启动后第一次 dump
$ adb shell dumpsys meminfo com.example.app
  TOTAL PSS:    145678 kB  ← 145MB
  Views:        234
  Activities:   3
  ...

# T+10min: 第二次 dump
$ adb shell dumpsys meminfo com.example.app
  TOTAL PSS:    234567 kB  ← 234MB（增长 90MB）
  Views:        1234
  Activities:   5

# T+20min: 第三次 dump
$ adb shell dumpsys meminfo com.example.app
  TOTAL PSS:    345678 kB  ← 345MB（再增长 110MB）
  Views:        5678
  Activities:   12

# 关键观察：Views 和 Activities 单调增长
```

**根因定位**：
- Views 234 → 1234 → 5678 = 单调增长 = View 泄漏
- Activities 3 → 5 → 12 = 单调增长 = Activity 泄漏
- 拉 Hprof 确认：

```bash
$ adb shell am dumpheap com.example.app /data/local/tmp/heap.hprof
$ adb pull /data/local/tmp/heap.hprof
# 用 MAT 打开
# 发现：MainActivity 持有 Handler，Handler 持有 View 树
# 根因：Handler 是匿名内部类，引用了外部 Activity
```

**修复方案**：
```java
// 修复前
public class MainActivity extends Activity {
    private Handler mHandler = new Handler() {
        @Override
        public void handleMessage(Message msg) {
            updateUI();  // 隐式持有 Activity
        }
    };
}

// 修复后：使用 StaticHandler + WeakReference
public class MainActivity extends Activity {
    private static class SafeHandler extends Handler {
        private final WeakReference<MainActivity> mActivity;
        SafeHandler(MainActivity activity) {
            mActivity = new WeakReference<>(activity);
        }
        @Override
        public void handleMessage(Message msg) {
            MainActivity activity = mActivity.get();
            if (activity != null && !activity.isFinishing()) {
                activity.updateUI();
            }
        }
    }
}
```

## 6.2 CASE-DUMPSYS-04-02 Native 内存高（用户报"切后台被杀"）

**场景**：某应用切后台被 lmkd 杀掉，怀疑 Native 内存高。

**操作时序**：

```bash
# T+0s: 抓 meminfo
$ adb shell dumpsys meminfo com.example.app | head -30
  Native Heap    234567   234564        0        0   245678    290000   245000
  Java Heap       12345    12340        0        0   13567     25678    18901
  Graphics        56789    56785        0        0   58000         0         0
  TOTAL PSS:    345678   345617        0        0  356748   315678   263901
  # ⭐ Native Heap = 234MB，异常高

# Step 2: 看 Native Heap 详细
$ adb shell dumpsys meminfo -d com.example.app | grep -A 5 "Native Heap"
  # .Bitmap 大头

# Step 3: 看进程 adj（验证是不是因为 Native 内存高被 lmkd 干掉）
$ adb shell dumpsys activity processes | grep "com.example.app"
  ProcessRecord{... com.example.app:1000}
    userId=10000  pid=12345
    adj=900  ← ⭐ 已被 lmkd 标记为高优先级杀
    lastPss=345678  ← ⭐ PSS 345MB
```

**根因定位**：
- Native Heap 234MB = Bitmap 像素缓冲泄漏
- 配合 `dumpsys meminfo -d` 发现 `.Bitmap` 段大
- 拉 Hprof 看 Bitmap 引用链（用 LeakCanary 风格）

**修复方案**：
1. 检查 Glide / Picasso 缓存配置
2. 排查手动 `Bitmap.createBitmap` 是否没 recycle
3. 主动调用 `BitmapFactory.Options.inBitmap` 复用

## 7.1 核心要诀（背下来）

1. **OOM / 泄漏 80% 走 `dumpsys meminfo <pkg>`**——6 大段 + 对象计数是关键
