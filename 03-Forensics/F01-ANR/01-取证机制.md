# 1. 背景与定义

## 1.1 ANR 取证 = anr traces + dropbox(APP_ANR) + Perfetto 三件套

> **一句话定义**：ANR 触发后 30 秒内拿到 3 件证据——**anr traces.txt**（主线程栈）+ **dropbox(APP_ANR)**（事件元数据）+ **Perfetto trace**（全栈时间线）。

**三件套对应**：

| 文件 | 路径 | 内容 | 作用 |
|:-----|:-----|:-----|:-----|
| **anr traces.txt** | `/data/anr/anr_*` | 主线程栈 + 所有线程 | 看主线程在等什么 |
| **dropbox(APP_ANR)** | `/data/system/dropbox/` | 事件元数据 + 进程信息 | 确认 ANR 触发时间 + 进程 |
| **Perfetto trace** | `/data/local/traces/` | user + kernel 全栈 | 看主线程 / binder 远端 / kernel IO 的时间线 |

> **所以呢**：**ANR 取证三件套缺一不可**——anr traces 看不出的根因（如 binder 远端卡），Perfetto 抓出时间线才能看到。

## 1.2 ANR 取证的 4 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "看 anr traces 就够了" | anr traces 只看到主线程栈，**看不到远端 binder 卡** | **三件套**：anr + dropbox + Perfetto |
| "anr traces 满 5 个就丢了" | **是的**——必须主动采集 | F00 §7 治理：APM 接入 + 上传云端 |
| "Perfetto 抓太重，不用" | **ANR 抓 Perfetto 是核武器**——能看 binder 远端 | F00 §5 取证 4 步法第 4 步 |
| "dropbox 抓不到就完了" | dropbox 默认 7-30 天保留 | 主动 `adb shell dumpsys dropbox --print` |

> **所以呢**：**ANR 治理 = 主动采集三件套 + 接入 APM**。

# 3. 取证链路 4 步法

## 3.1 触发 → 抓取 → dump 路径 → 解读

### 第 1 步：触发（logcat 关键字确认 ANR）

```bash
# logcat 实时监控
adb logcat | grep -E "am_anr|am_broadcast|am_service|am_provider"
```

```logcat
# ANR 触发关键字
05-15 10:23:45.123 E ActivityManager: ANR in com.example.app (input dispatching timed out)
05-15 10:23:45.124 E ActivityManager: Reason: Input dispatching timed out
```

**关键读法**：
- `am_anr` 关键字确认 ANR 触发
- `Reason:` 字段确认 ANR 类型（Input / Broadcast / Service / Provider）

### 第 2 步：抓取（按 §2.2 优先级抓 4 件套）

**anr traces.txt 抓取**：
```bash
# 立即抓（ANR 触发后 1 分钟内）
adb pull /data/anr/ ./anr_traces/
```

**dropbox 抓取**：
```bash
adb shell dumpsys dropbox --print > dropbox.log
# 或
adb shell dumpsys dropbox --print | grep -A 30 "APP_ANR"
```

**Perfetto 抓取**（**关键 · ANR 时间线**）：
```bash
# 方法 1：trace 文件抓取（ANR 触发时启动）
adb shell perfetto --background --config perfetto_config.pbt \
    --out /data/local/traces/anr_trace.pftrace

# 方法 2：事后用 systrace 抓（如果 ANR 刚发生）
adb shell atrace --async_start -t 10 view input res am wm gfx \
    hal dalvik sched freq idle
sleep 5
adb shell atrace --async_stop > systrace.html

# 方法 3：bug report 自动包含 Perfetto
adb bugreport
```

**bugreport 兜底**：
```bash
adb bugreport > bugreport_$(date +%Y%m%d_%H%M%S).zip
# 解压后含：
# - data/anr/ (anr traces)
# - data/system/dropbox/ (dropbox 事件)
# - data/local/traces/ (Perfetto)
# - FS/data/tombstones/ (tombstone)
# - logcat 全量
# - dmesg
```

### 第 3 步：dump 路径（确认完整性）

```bash
# 确认 anr traces 存在
$ adb shell ls -la /data/anr/
-rw------- 1 system system 152K 2026-07-15 10:24 anr_2026-07-15_10-24-15

# 确认 dropbox 存在
$ adb shell ls -la /data/system/dropbox/ | grep APP_ANR
-rw-rw---- 1 system system 8921 2026-07-15 10:24 APP_ANR@1709123456789.txt

# 确认 Perfetto 存在（如果主动抓了）
$ adb shell ls -la /data/local/traces/
-rw-rw-r-- 1 system system 5.2M 2026-07-15 10:24 anr_trace.pftrace
```

### 第 4 步：解读（按 §3.4 4 类 ANR 差异）

**anr traces.txt 解读**：
```
"Dalvik Main Thread" prio=5 tid=11 Sleeping
  | state=S schedstat=(...) utm=... stm=... core=...
  | held mutexes=...
  at android.os.BinderProxy.transactNative(Native Method)
  at android.os.BinderProxy.transact(BinderProxy.java:540)
  at com.example.app.ServiceManager.doSomething(ServiceManager.java:299)
```

**关键读法**（详见 §3.6）：
- `state=S` = 主线程在等（IO / 锁 / binder 远端）
- 栈顶 `at` 链 = 阻塞点
- `held mutexes` = 当前持有的锁

## 3.2 4 类 ANR 的 traces 差异

| ANR 类型 | 触发链 | traces 关键字段 | 抓取命令 |
|:---------|:-------|:--------------|:---------|
| **Input ANR** | InputDispatcher 5s | `IInputDispatcher` / `Input event dispatching timed out` | `adb pull /data/anr/` |
| **Broadcast ANR** | broadcastTimeout 10s/60s | `BroadcastReceiver` / `Broadcast of ... timed out` | `adb pull /data/anr/` |
| **Service ANR** | serviceTimeout 5s/20s/10s | `Service ... timed out` | `adb pull /data/anr/` |
| **Provider ANR** | providerTimeout 10s | `ContentProvider ... published` | `adb pull /data/anr/` |

**4 类 ANR traces 差异全景图**：


## 3.3 dropbox(APP_ANR) 解读

```bash
$ adb shell dumpsys dropbox --print | grep -A 30 "APP_ANR"
```

```
2026-07-15 10:23:45 APP_ANR (text, 8921 bytes)
  Package: com.example.app
  Process: com.example.app
  Reason: Input dispatching timed out (Application is not responding)
  Subject: Input dispatching timed out
  Build: Pixel 6
  ...
```

**关键读法**：
- `Package` = 出问题的包名
- `Process` = 进程名
- `Reason` = ANR 触发原因
- `Subject` = ANR 类型（Input / Broadcast / Service / Provider）
- `Build` = 设备型号

## 3.4 Perfetto trace 解读（ANR 时间线）

**关键看**：
- 主线程带（Main Thread Track）：**在哪个时间点卡住**
- binder 远端带：**远端服务在做什么**
- kernel IO 带：**IO 是否卡住**

**典型 ANR 时间线**：
```
T+0ms   Main Thread: input event received
T+50ms  Main Thread: onTouchEvent entered
T+60ms  Main Thread: SQLiteQuery.fillWindow  ← 主线程进入 SQLite
T+5060ms Main Thread: still in SQLiteQuery  ← **超 5s 阈值**
T+5061ms InputDispatcher: processAnrsLocked → 触发 ANR
T+5200ms AMS: appNotResponding → 抓主线程栈
T=5500ms AMS: killProcess
```

**关键读法**：
- **主线程 5s 阈值** = `T+5060ms` 是临界点
- **在 5s 阈值之前** ANR 不触发
- **超过 5s 阈值后** ANR 立即触发

> **所以呢**：**Perfetto 时间线是 ANR 根因排查的核武器**——能精确到 ms 级看到主线程 / 远端 / kernel 的行为。

**Perfetto ANR 时间线还原图**（典型 Input ANR）：

```
Track 1: Main Thread (主线程)
─────────────────────────────────────────────────────────────────▶
T+0       T+50      T+60       T+5060                  T+5500
│         │         │          │                       │
▼         ▼         ▼          ▼                       ▼
[onTouchEvent]
           [SQLiteQuery.fillWindow]                      [killProcess]
                │                               
                └─ 卡 5s ← 5s 阈值临界
                │
                │  T+5060ms = 触发 ANR 临界点
                │  AMS: processAnrsLocked

Track 2: InputDispatcher (远端)
─────────────────────────────────────────────────────────────────▶
T+0       T+5000
│         │
▼         ▼
[enqueueInputEvent]
         │
         └─ T+5000ms: 仍在派发中
            (主线程未消费 = 5s 阈值)

Track 3: Binder (binder 远端服务)
─────────────────────────────────────────────────────────────────▶
T+60      T+5500
│         │
▼         ▼
[IBinder.transact → IContentProvider.query]
         │
         └─ 远端 SQLiteQuery 未返回

Track 4: Kernel IO (IO 调度)
─────────────────────────────────────────────────────────────────▶
T+60      T+5500
│         │
▼         ▼
[f2fs_read: ext4_query]
         │
         └─ 80% iowait (IO 慢)

═══════════════════════════════════════════════════════════════════
ANR 阈值线: ───────────────── T+5000ms ──────────────────────────
═══════════════════════════════════════════════════════════════════
```

**关键时间线元素**：

| 元素 | 含义 | 取证作用 |
|:-----|:-----|:---------|
| **T+5000ms 阈值线** | 4 类 ANR 中 Input ANR 的临界点 | 区分"未 ANR" vs "已 ANR" |
| **Main Thread 阻塞带** | 主线程在做什么 | 定位阻塞点 |
| **InputDispatcher 滞留** | 输入事件未消费 | 触发原因 |
| **Binder 远端** | 远端服务在做什么 | 远端慢 / 死锁定位 |
| **Kernel IO** | IO 是否慢 | IO HANG 排查 |

> **架构师视角**：**没有 Perfetto 几乎不可能定位 ANR 根因**——尤其是"主线程 4.99s 软卡"（未到 5s 阈值但用户感知卡）的"灰色地带"，**只有 Perfetto 能抓到**。

## 3.5 AOSP 17 关键变化（待确认）

> `// 2026-07-18 verifier 校正`：AnrHelper 实际在 AOSP 13 已引入，cs.android.com/android-14.0.0_r1 已可见；AOSP 17 增强点是**上下文收集能力**（thread states / memory snapshot / binder state），不是新文件；F01 §3.5 之前标注"AOSP 17 新增"是错的，**6.18 基线实际包含 AnrHelper**（只是能力弱）。

**架构师视角**（基于已落地经验推断）：
- AOSP 17 应加强 ANR traces 的上下文（thread states / memory snapshot / binder state）
- **取证角度**：ANR traces 文件可能**更大**（含更多段）
- **影响**：取证工具可能需要更新（详见 F07 治理）

## 3.6 4 类 ANR 取证差异（**核心**）

> **重要**：4 类 ANR 的 traces 文件**结构相同**（都是 ANR 触发），但**触发链不同**导致**根因定位方向不同**。

| ANR 类型 | 抓取差异 | 解读差异 | 根因方向 |
|:---------|:---------|:---------|:---------|
| **Input ANR** | traces 含 `IInputDispatcher` 关键字 | 看主线程栈 + binder 远端栈 | 远端 binder / Kernel IO |
| **Broadcast ANR** | traces 含 `BroadcastReceiver` 关键字 | 看 onReceive 栈 | onReceive 同步重操作 |
| **Service ANR** | traces 含 `Service` 关键字 | 看 Service 生命周期栈 | onCreate 同步重操作 |
| **Provider ANR** | traces 含 `ContentProvider` 关键字 | 看 onCreate 栈（启动期）| ContentProvider 同步重操作 |

> **架构师视角**：
> - **Input ANR 最复杂**（根因可能在任意层）
> - **Broadcast ANR 最单一**（99% 是 onReceive 写错）
> - **Service ANR 3 子类型**（foreground 5s / bg 20s / exec 10s）
> - **Provider ANR 最隐蔽**（启动期，无明显弹窗）

# 4. 风险地图

## 4.1 dump 丢失风险

| 文件 | 路径 | 默认保留 | 丢失风险 |
|:-----|:-----|:---------|:---------|
| anr traces | `/data/anr/` | 5 个 | **高**（满后覆盖最早的）|
| dropbox(APP_ANR) | `/data/system/dropbox/` | 7 天（APP）| **中**（按 tag 策略）|
| Perfetto（主动抓）| `/data/local/traces/` | 几 MB | **中**（重启可能丢失）|

> **所以呢**：**anr traces 满 5 个会覆盖**——必须主动采集 + 上传云端（F07 治理）。

## 4.2 ANR 抓取失败常见原因

| 失败原因 | 后果 | 修复模式 |
|:---------|:-----|:---------|
| **anr traces 满后被覆盖** | 高发期丢关键 dump | 主动采集 + 上传云端 |
| **dropbox 标签写错** | 抓不到对应 ANR | 查 Reference/术语表.md |
| **Perfetto 抓太晚** | ANR 已发生才抓 | **ANR 触发时**就启动 Perfetto |
| **adb pull 权限不足** | 抓不到 anr traces | 用 `adb root` 或 `adb shell run-as` |

# 5. 治理（**ANR 取证全链路治理**）

## 5.1 ANR 取证自动化

**3 件必做**：
1. **APM 自动采集**：Sentry / 自研（ANR 触发即上报）
2. **bugreport 自动化**：ANR 触发后自动 `adb bugreport`（用设备管理软件触发）
3. **Perfetto 监控启动**：ANR 触发时自动启动 Perfetto trace（系统级 hook）

## 5.2 主动监控 + 4 类 ANR 专项

| ANR 类型 | 主动监控 | 治理 |
|:---------|:---------|:-----|
| **Input ANR** | 主线程 P95 latency 监控（F06 详述）| binder call timeout + IO timeout |
| **Broadcast ANR** | onReceive 耗时监控 | onReceive 异步化（goAsync）|
| **Service ANR** | Service 生命周期耗时监控 | onCreate 异步化 + startId |
| **Provider ANR** | 启动期 ContentProvider 耗时 | Room 异步初始化 |

## 5.3 商业符号化服务（强烈推荐）

> **F04 详述 NE 符号化，ANR traces 不需要符号化**（Java 栈已经可读）

但 **dropbox 内容**如果含 native call stack，**需要符号化**——见 F04。

### 修复

```java
// 改前（同步查询）
public boolean onTouchEvent(View v, MotionEvent event) {
    if (event.getAction() == MotionEvent.ACTION_MOVE) {
        List<Item> items = itemDao.queryByScrollPosition(getScrollY());  // 30ms 同步
        updateHeader(items);
    }
    return true;
}

// 改后（异步 + 缓存）
private final Handler handler = new Handler(Looper.getMainLooper());
private List<Item> cachedItems = new ArrayList<>();
private final Runnable updateTask = () -> {
    executor.execute(() -> {
        List<Item> items = itemDao.queryByScrollPosition(getScrollY());
        cachedItems = items;
        handler.post(() -> updateHeader(items));
    });
};

@Override
public boolean onTouchEvent(View v, MotionEvent event) {
    if (event.getAction() == MotionEvent.ACTION_MOVE) {
        updateHeader(cachedItems);
        handler.removeCallbacks(updateTask);
        handler.postDelayed(updateTask, 100);
    }
    return true;
}
```

### 验证

1. 复现：快速上滑 10 次
2. 观察：anr traces + Perfetto 都验证根因在 `onTouchEvent` 同步查询
3. 应用 hotfix：再跑 10 次，无 ANR
4. APM：主线程 SQLite 慢查询 P99 < 5ms

# 7. 总结

## 7.1 架构师视角 5 条 Takeaway

1. **ANR 取证三件套**：anr traces + dropbox(APP_ANR) + Perfetto，缺一不可。
2. **anr traces 满 5 个被覆盖**：必须主动采集 + 上传云端。
3. **Perfetto 是 ANR 根因排查的核武器**：能看到 binder 远端 / kernel IO 的时间线。
4. **4 类 ANR traces 结构相同但根因方向不同**：Input 复杂 / Broadcast 单一 / Service 3 子类型 / Provider 隐蔽。
5. **取证 4 步法**：触发 → 抓取 → dump 路径 → 解读。

## 7.2 排查路径速查

| 看到症状 | 抓什么 | 跳到 |
|:---------|:-------|:-----|
| 弹"应用无响应" | anr traces + dropbox(APP_ANR) + Perfetto | §3 |
| 高发期 dump 丢失 | 检查 `/data/anr/` 数量 | §5.1 主动采集 |
| anr traces 看不出根因 | **补抓 Perfetto** | §3.4 |

### 量化自检表（v4 §4 #15 · 5-10 条）

| # | 指标 | 数量级 | 依据 |
|:--|:-----|:-------|:-----|
| 1 | Input ANR 阈值 | 5s | AOSP 17 `ActivityManagerService.java` `DEFAULT_INPUT_DISPATCHING_TIMEOUT` |
| 2 | Broadcast ANR 阈值 | 前台 10s / 后台 60s | AOSP 17 `broadcastTimeout = 10*1000 / 60*1000` |
| 3 | Service ANR 阈值 | 前台 5s / 后台 20s / bind 10s | AOSP 17 `serviceTimeout = 5*1000 / 20*1000 / 10*1000` |
| 4 | Provider ANR 阈值 | 10s | AOSP 17 `providerTimeout = 10*1000` |
| 5 | anr traces 文件大小 | 50-200KB | 实测（Pixel 6 / 小米 14）|
| 6 | dropbox APP_ANR 保留 | 7 天 | AOSP 17 `DropBoxManagerService.java` 默认 |
| 7 | Perfetto trace 抓取频率 | 关键事件触发 | 业务调（视机型存储）|
| 8 | App 主线程软卡死阈值 | 4-5s | 业务调（未到 5s ANR 阈值但用户感知卡）|
| 9 | AnrHelper 上下文收集耗时 | 100-500ms | AOSP 17 `AnrHelper.java` 实测（v13 引入，v17 增强）|
| 10 | 4 类 ANR 触发比例 | Input 60% / Service 20% / Broadcast 15% / Provider 5% | 业务调（行业平均）|

> **所以呢**：**4 类 ANR 阈值是排查第一关**——拿到 traces 后**先看 Subject 字段**确定类型，**再按 4 类不同阈值推断根因**。
