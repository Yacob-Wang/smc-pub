# 1. 背景与定义

## 1.1 NE 取证 = tombstone + dropbox(SYSTEM_TOMBSTONE) + 符号化服务 3 件套

> **一句话定义**：NE 触发后 30 秒内拿到 3 件证据——**tombstone**（16 段 native 崩溃现场）+ **dropbox(SYSTEM_TOMBSTONE)**（事件元数据）+ **符号化后的可读栈**（商业符号化服务）。

**三件套对应**：

| 文件 | 路径 | 内容 | 作用 |
|:-----|:-----|:-----|:-----|
| **tombstone_NN** | `/data/tombstones/` | 16 段 raw 现场（registers / 栈 / 内存映射 / 线程列表 / etc.）| 看崩溃现场（**需要符号化**）|
| **dropbox(SYSTEM_TOMBSTONE)** | `/data/system/dropbox/` | 事件元数据 + tombstone 路径 | 确认 NE 触发 + 时间 |
| **符号化栈** | 云端 / 自建服务 | 函数名 + 源码行 | **可读**栈（**符号化后**）|

> **所以呢**：**NE 取证 = 抓 tombstone + 符号化**——符号化是 NE 排查的**最大瓶颈**。

## 1.2 NE 取证的 3 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "看 tombstone 就够了" | raw tombstone 是 16 进制地址，**不符号化看不出意义** | **必须接入商业符号化服务** |
| "用 debug build 的 .so 符号化" | **必须用与发布版本一致**的 .so + 配套符号表 | 接入 build server 的 .so |
| "手动 addr2line 就够了" | 1 次 NE 触发可能 10+ 栈，**手动太低效** | 商业符号化服务（Sentry / Bugsnag）|

> **所以呢**：**商业符号化服务是 NE 排查的必备**——手动符号化是 NRE 时代的事。

# 2. 取证 4 步法

## 2.1 触发 → 抓取 → 符号化 → 解读

### 第 1 步：触发（logcat 关键字）

```bash
adb logcat | grep -E "DEBUG|tombstone written"
```

```logcat
DEBUG: signal 11 (SIGSEGV), fault addr 0x10
DEBUG:    backtrace:
DEBUG:      #00 pc 0x1234 libnative.so (Java_com_example_NativeHelper_nativeMethod+24)
DEBUG:      #01 pc 0x2345 libart.so (art_jni_dlsym_lookup_stub+10)
DEBUG:        [stack] tombstone written to /data/tombstones/tombstone_03
```

**关键读法**：
- `signal 11` = SIGSEGV
- `fault addr 0x10` = 崩溃地址
- `tombstone written to /data/tombstones/tombstone_03` = tombstone 路径

### 第 2 步：抓取（3 件套）

**tombstone 抓取**：
```bash
# 立即抓（NE 触发后 1 分钟内）
adb pull /data/tombstones/ ./tombstones/

# 多个 tombstone 时按时间排序
adb shell ls -la /data/tombstones/ | sort -k 6,7
```

**dropbox 抓取**：
```bash
adb shell dumpsys dropbox --print | grep -A 30 "SYSTEM_TOMBSTONE"
```

**符号化所需 .so 抓取**：
```bash
# 从 build server 拉对应 .so（debug symbol file）
# 路径示例：build_server/app-v1.2.3-2026-07-15/lib/arm64-v8a/libnative.so

# 或在设备上抓（仅 release .so，无 .debug）
adb pull /data/app/*/lib/arm64-v8a/libnative.so
```

**debuggerd 信号处理时序图**（NE 触发 → tombstone 生成的完整链路）：

```
T+0     用户 App 进程触发致命信号（如 SIGSEGV）
        │
        │   ┌──────────────────────────────────────────┐
        │   │  Linux Kernel (K 6.18)                   │
        │   │  force_sigsegv(fault_addr)  ← K 6.4-6.6 │
        │   │  → kernel/signal.c                       │
        │   │  → 投递信号到用户进程                      │
        └───┤                                          │
            ▼                                          │
T+1ms   signalfd / sigaction 接收信号                 │
        （用户态注册的 debuggerd_handler）              │
        │                                              │
        ▼                                              │
T+2ms   debuggerd 进程 (system/core/debuggerd/)        │
        debuggerd_handler.cpp: signal_handler()       │
        │                                              │
        ├── ptrace(PTRACE_ATTACH) 到崩溃进程           │
        │                                              │
        ▼                                              │
T+5ms   抓取崩溃进程上下文：                           │
        ├── [1] 全部线程 registers（getcontext）       │
        ├── [2] 全部线程 stack frames（unwind）        │
        ├── [3] 崩溃线程 memory map (/proc/pid/maps)  │
        ├── [4] 全部线程列表 (/proc/pid/task/)         │
        ├── [5] 进程内存使用 (/proc/pid/status)        │
        ├── [6] open files (/proc/pid/fd/)             │
        ├── [7] 崩溃指令 (memory read around PC)       │
        ├── [8] build ID + .so 列表                    │
        ├── [9] kernel 栈（仅 system_server 可见）     │
        ├── [10] abi / 寄存器位数                      │
        ├── [11] 进程名 / thread name                  │
        ├── [12] 命令行（cmdline）                     │
        ├── [13] SELinux context                      │
        ├── [14] FORTIFY / 栈 canary 信息              │
        ├── [15] crash type + abort message            │
        ├── [16] memory near fault addr                │
        │                                              │
        ▼                                              │
T+50ms  生成 tombstone 16 段文本                       │
        tombstone.cpp: engrave_tombstone()             │
        │                                              │
        ├── 写入 /data/tombstones/tombstone_XX         │
        ├── 写入 /data/system/dropbox/SYSTEM_TOMBSTONE  │
        │                                              │
        ▼                                              │
T+100ms 退出 debuggerd_handler，崩溃进程被 SIGKILL 结束
```

> **架构师视角**：**debuggerd 在 100ms 内完成全部 16 段抓取**——**这是 NE 取证的硬性时间窗**（tombstone 必须在 100ms 内写完，否则丢失上下文）。**AOSP 17 增强**：增加 [9] kernel 栈（system_server 时）和 [14] FORTIFY 上下文。

### 第 3 步：dump 路径

```bash
$ adb shell ls -la /data/tombstones/
-rw------- 1 system system 152K 2026-07-15 10:24 tombstone_03

$ adb shell ls -la /data/system/dropbox/ | grep TOMBSTONE
-rw-rw---- 1 system system 8921 2026-07-15 10:24 SYSTEM_TOMBSTONE@1709123456789.txt
```

### 第 4 步：符号化（**关键 · 见 §3**）

**商业符号化**：
- 上传 .so + .debug + tombstone 到 Sentry / Bugsnag / Backtrace.io
- 自动得到函数名 + 源码行

**手动符号化**（仅作为补充）：
```bash
# 用 NDK addr2line
$ANDROID_HOME/ndk/.../toolchains/llvm/prebuilt/linux-x86_64/bin/llvm-addr2line \
    -e libnative.so -f 0x1234

# 输出：
Java_com_example_NativeHelper_nativeMethod
/path/to/source.cpp:42
```

# 3. tombstone 16 段结构（**核心**）

> **取证角度**：tombstone 16 段是 NE 排查的核心数据。**raw tombstone 不直接可读**，必须符号化 + 解读。

| 段号 | 段名 | 内容 | 取证价值 |
|:-----|:-----|:-----|:---------|
| 1 | **header** | 信号类型 + 进程名 + PID + 时间 | 识别 NE 类型 |
| 2 | **registers** | 通用寄存器（PC / SP / LR / ...）| PC 指向崩溃指令 |
| 3 | **backtrace** | 栈回溯（FP / LR）| **核心**：崩溃调用链 |
| 4 | **stack** | 栈内容（raw bytes）| 看局部变量 |
| 5 | **memory near registers** | PC 附近的内存 | 看崩溃指令上下文 |
| 6 | **memory near stack pointer** | SP 附近的内存 | 看栈上下文 |
| 7 | **memory map** | 进程内存映射 | 看 .so 加载位置 |
| 8 | **threads** | 所有线程列表 | 看其他线程状态 |
| 9 | **signal info** | 信号详细信息 | si_code / si_addr |
| 10 | **abort message** | abort() 时的 message | SIGABRT 专属 |
| 11 | **open files** | 进程打开的文件 | 排查资源泄漏 |
| 12 | **command line** | 启动命令行 | 看启动参数 |
| 13 | **process status** | /proc/PID/status | 内存 / 线程数 |
| 14 | **memory usage** | VMS/RSS/PSS | 看内存 |
| 15 | **log buffer** | logcat snapshot | 排查日志上下文 |
| 16 | **build fingerprint** | 系统版本 | 排查设备 / 版本差异 |

> **架构师视角**：
> - **backtrace 段是核心**——必须符号化才能读
> - **registers 段是辅助**——PC 指向崩溃指令
> - **memory map 段**——确认 .so 加载位置

## 3.1 tombstone 保留期

| 文件 | 路径 | 保留 | 丢失风险 |
|:-----|:-----|:-----|:---------|
| tombstone | `/data/tombstones/` | 10 个（满后覆盖最早的）| **高** |
| dropbox(SYSTEM_TOMBSTONE) | `/data/system/dropbox/` | 30 天 | **中** |
| logcat -b crash | ring buffer | 几 MB | **中** |

> **所以呢**：**tombstone 满 10 个会覆盖**——必须主动采集 + 上传 APM。

**16 段关键看哪 5 段**（取证 4 步法的"解读"步骤）：

| 优先级 | 段 | 用途 |
|:------|:--|:-----|
| **P0** | [1] Header | 第一时间定位症状（信号 + 时间 + 进程）|
| **P0** | [3] Backtrace | 定位代码（符号化后）|
| **P1** | [5] Memory Map | 确认 .so 加载位置 |
| **P1** | [6] Thread List | 看其他线程在做什么（deadlock 排查）|
| **P2** | [2] Register | 看 PC / SP（汇编排查）|
| **P2** | [9] Kernel Stack | system_server NE 排查 |

> **架构师视角**：**16 段中 [1][3][5][6] 是必看**——其他段在 P1/P2 排查时才用。**AOSP 17 增强的 [14] FORTIFY 段是关键**——能直接看到 FORTIFY 失败的具体检查。

## 3.2 6 种信号取证差异

| 信号 | tombstone 关键字段 | 解读重点 | 修复方向 |
|:-----|:-----------------|:---------|:---------|
| **SIGSEGV (11)** | `signal 11` + `fault addr` | fault addr 0x0 = 空指针 | 空指针 / UAF / 越界 |
| **SIGABRT (6)** | `signal 6` + `Abort message:` | abort message 含触发原因 | 断言 / abort / fortify |
| **SIGBUS (7)** | `signal 7` | 总线错误 | 内存对齐 / mmap 失败 |
| **SIGFPE (8)** | `signal 8` | 算术错误 | 除零（编译器未优化时）|
| **SIGILL (4)** | `signal 4` | 非法指令 | .so 损坏 / 栈破坏 |
| **SIGSYS (31)** | `signal 31` + `seccomp` | seccomp 拦截 | 用了被禁的 syscall |

> **架构师视角**：**6 种信号的 tombstone 字段不同**——**取证时先看 signal 编号，再看 backtrace**。

# 4. 符号化服务（**核心 · NE 排查的瓶颈**）

## 4.1 为什么符号化是 NE 排查的瓶颈？

```
tombstone raw（不可读）：
  #00 pc 0x1234 libnative.so (offset 0x1234)
  #01 pc 0x2345 libart.so (offset 0x2345)

符号化后（可读）：
  #00 pc 0x1234 libnative.so (Java_com_example_NativeHelper_nativeMethod+24)  src/native.cpp:42
  #01 pc 0x2345 libart.so (art_jni_dlsym_lookup_stub+10)                         art/runtime/...
```

**关键**：
- raw address = 16 进制地址，**不可读**
- 符号化后 = 函数名 + 源码行 + 偏移量，**可读**
- **必须用与发布版本一致**的 .so + 配套 .debug 符号表

## 4.2 商业符号化服务（**强烈推荐**）

| 服务 | 优势 | 接入难度 |
|:-----|:-----|:---------|
| **Sentry** | Native + Java 完整支持 | 中 |
| **Bugsnag** | 自动符号化 + alert | 中 |
| **Backtrace.io** | 专门做 NE 排查 | 中 |
| **Firebase Crashlytics** | Google 官方（已支持 NDK）| 低 |
| **自建** | 完全可控 | 高 |

**接入方式**（以 Sentry 为例）：
1. 在 Sentry 后台上传 .so + .debug（debug symbol file）
2. 上传 tombstone
3. Sentry 自动符号化 + 聚合
4. 看到可读栈 + 聚合统计

> **架构师视角**：**强烈推荐接入 1 个商业符号化服务**——手动符号化是 NRE 时代的事。

**符号化服务链路图**（tombstone → 云端 → 可读栈）：

```
┌────────────────────────┐
│ 设备端                  │
│                        │
│ 1. NE 触发             │
│    debuggerd 抓 tombstone │
│ 2. APM SDK 自动采集    │
│    ┌──────────────────┐ │
│    │ 1. tombstone_03  │ │
│    │ 2. libnative.so  │ │
│    │ 3. build ID      │ │
│    │ 4. device info   │ │
│    └────────┬─────────┘ │
└────────────┼──────────┘
             │ HTTPS
             ▼
┌────────────────────────┐
│ APM 云端                │
│                        │
│ 3. 符号化服务          │
│    ┌──────────────────┐ │
│    │ Build Server     │ │
│    │ .so + .debug     │ │
│    │   ↕              │ │
│    │ Symbolicator     │ │
│    │ (addr2line 云端版)│ │
│    └────────┬─────────┘ │
│             │           │
│ 4. 聚合 / 去重          │
│    同一 signature 聚合  │
└────────────┼──────────┘
             │ Webhook / 邮件
             ▼
┌────────────────────────┐
│ 工程师界面              │
│                        │
│ 5. 看到可读栈           │
│    #00 com.example.NativeHelper.nativeMethod│
│        src/native.cpp:42                     │
│    #01 art_jni_dlsym_lookup_stub              │
│        art/runtime/jni/...                    │
│    + 同 issue 聚合 1000 次                   │
│    + 影响用户 50K                            │
│    + 修复版本 v1.2.4                          │
└────────────────────────────────────────────────┘
```

**符号化 4 件套（缺一不可）**：

1. **tombstone 文件**（设备抓取）
2. **对应 .so**（从 build server 拉）
3. **对应 .debug 符号表**（从 build server 拉）
4. **build ID 匹配**（tombstone 含 build ID，符号表含 build ID，必须匹配）

> **架构师视角**：**符号化 90% 失败原因 = .so 和 tombstone 的 build ID 不匹配**——很多团队 1 个版本只 pull 1 次 .so，后续发布 .so 更新后，**老 tombstone 符号化失败**。**正确做法**：每个版本都同步 .debug 到符号化服务。

## 4.3 手动符号化（仅作为补充）

```bash
# 1. 拿到 .so + 调试符号表（从 build server）
# .so 是 release build，.debug 是调试符号（分离的）

# 2. NDK addr2line
$ANDROID_HOME/ndk/.../toolchains/llvm/prebuilt/linux-x86_64/bin/llvm-addr2line \
    -e libnative.so -f 0x1234
# 输出：
Java_com_example_NativeHelper_nativeMethod
/path/to/source.cpp:42
```

**手动符号化的局限**：
- 1 次 NE 触发可能 10+ 栈，**手动 10+ 次**太低效
- 需要**完整 .so 库**（不能是剥离的 release .so）
- 需要**配套 .debug 符号表**

> **所以呢**：手动符号化**仅作为应急**——生产环境必须用商业符号化服务。

# 5. dropbox(SYSTEM_TOMBSTONE) 详解

## 5.1 抓取与解读

```bash
$ adb shell dumpsys dropbox --print | grep -A 30 "SYSTEM_TOMBSTONE"
```

```
2026-07-15 10:24:00 SYSTEM_TOMBSTONE (text, 8921 bytes)
  Package: com.example.app
  Process: com.example.app
  Subject: Native crash: signal 11 (SIGSEGV) at 0x10
  Build: Pixel 6
  ...

```

**关键读法**：
- `Subject` = 信号类型 + 崩溃地址
- `Tombstone` 字段 = tombstone 文件路径

## 5.2 保留期

| tag | 保留期 | 备注 |
|:----|:-------|:-----|
| SYSTEM_TOMBSTONE | 30 天 | 系统 NE 触发 |
| SYSTEM_TOMBSTONE_PROTO | 30 天 | proto 格式（AOSP 14+）|

> **所以呢**：dropbox 保留 30 天——比 tombstone（10 个满后覆盖）保留更久。

# 6. 治理

## 6.1 NE 取证自动化

**4 件必做**：
1. **APM 自动采集**：Sentry / Backtrace.io / Crashlytics（NE 触发即上报）
2. **tombstone 主动采集**：自动 `adb pull /data/tombstones/`（高发期防覆盖）
3. **dropbox 主动监控**：定时 `adb shell dumpsys dropbox --print`
4. **商业符号化服务接入**：Sentry / Bugsnag / Backtrace.io

## 6.2 符号化服务选型矩阵

| 服务 | 价格 | NE 支持 | 接入难度 | 推荐场景 |
|:-----|:-----|:--------|:---------|:---------|
| **Sentry** | 中 | ✅ | 中 | 中大型 App |
| **Bugsnag** | 中 | ✅ | 中 | 中大型 App |
| **Backtrace.io** | 中-高 | ✅✅（专门做 NE）| 中 | 大型 App（NE 多）|
| **Firebase Crashlytics** | 低 | ✅（NDK 支持）| 低 | 初创 / 中小 App |
| **自建** | 高 | 自定义 | 高 | 大型 App（定制需求）|

> **架构师视角**：**Sentry 或 Crashlytics 是主流选择**——Backtrace.io 适合 NE 占比高的 App。

## 6.3 预防机制

**4 个必做**：
1. **开 ASAN**（debug build）—— 提前发现 UAF / 越界
2. **静态分析**（clang-tidy）—— 编译期检查
3. **fuzzing**（libFuzzer）—— 边界条件
4. **JNI 类型检查**（`IsAssignableFrom`）—— 详见 Stability S03 案例 A

# 7. 实战案例

## 7.1 案例 A（CASE-FORENSICS-04-01）：JNI IsAssignableFrom 漏检 → 完整取证 4 步法

> **类型**：典型模式
>
> **环境**：AOSP 17.0.0_r1 / Kernel android17-6.18 / 设备 Pixel 6
>
> **症状**：调用 Java 方法时随机崩溃
>
> **根因**：JNI 未检查 IsAssignableFrom，传入错误类型对象

### 现象

```
用户操作：
  T+0s   App 启动
  T+5s   第一次调用 nativeMethod(obj)
  T+5.001s SIGSEGV（部分场景）/ 正常返回（其他场景）

**关键观察**：**随机崩溃**（取决于传入对象的实际类型）
```

### 取证 4 步法

**第 1 步：触发（logcat）**

```logcat
DEBUG: signal 11 (SIGSEGV), fault addr 0x10
DEBUG:    backtrace:
DEBUG:      #00 pc 0x1234 libnative.so (offset 0x1234)  ← 符号化前
DEBUG:      #01 pc 0x2345 libart.so (offset 0x2345)
DEBUG:        [stack] tombstone written to /data/tombstones/tombstone_03
```

**第 2 步：抓取**

```bash
# tombstone 抓取
$ adb pull /data/tombstones/tombstone_03
./tombstone_03

# dropbox 抓取
$ adb shell dumpsys dropbox --print | grep -A 30 "SYSTEM_TOMBSTONE"
2026-07-15 10:24:00 SYSTEM_TOMBSTONE (text, 8921 bytes)
  Package: com.example.app
  Process: com.example.app
  Subject: Native crash: signal 11 (SIGSEGV) at 0x10
  Tombstone: /data/tombstones/tombstone_03

# .so 抓取
$ adb pull /data/app/com.example.app-*/lib/arm64-v8a/libnative.so
```

**第 3 步：dump 路径**

```bash
$ adb shell ls -la /data/tombstones/
-rw------- 1 system system 152K 2026-07-15 10:24 tombstone_03
```

**第 4 步：符号化（**关键**）**

**手动符号化**：
```bash
$ llvm-addr2line -e libnative.so -f 0x1234
Java_com_example_NativeHelper_nativeMethod
/workspace/src/native.cpp:42
```

**符号化后**：
```
DEBUG: signal 11 (SIGSEGV), fault addr 0x10
DEBUG:    backtrace:  ← 符号化后
DEBUG:      #00 pc 0x1234 libnative.so (Java_com_example_NativeHelper_nativeMethod+24) src/native.cpp:42
DEBUG:      #01 pc 0x2345 libart.so (art_jni_dlsym_lookup_stub+10)                       art/runtime/jni_dlsym_lookup_stub.cc
DEBUG:      #02 pc 0x3456 libart.so (artQuickInvokeJNIMethod+78)                       art/runtime/entrypoints/quick/quick_jni_entrypoints.cc
```

**关键读法**：
- `fault addr 0x10` = 偏移 0x10（可能是 `obj + 0x10` 处的字段访问）
- 栈顶 `Java_com_example_NativeHelper_nativeMethod+24 src/native.cpp:42` ← JNI 函数第 42 行

### 修复

**短期**：
```cpp
JNIEXPORT void JNICALL
Java_com_example_NativeHelper_nativeMethod(JNIEnv* env, jobject obj) {
    jclass cls = env->GetObjectClass(obj);
    
    // **关键修复 1**：检查类型
    jclass expectedCls = env->FindClass("com/example/ExpectedClass");
    if (!env->IsAssignableFrom(cls, expectedCls)) {
        return;
    }
    
    // **关键修复 2**：检查 fieldID
    jfieldID fid = env->GetFieldID(cls, "data", "Lcom/example/Data;");
    if (fid == NULL) {
        return;
    }
    
    jobject data = env->GetObjectField(obj, fid);
    if (data == NULL) {
        return;
    }
    
    // ... 后续访问 data
}
```

**长期**：接入 Sentry 商业符号化服务（自动符号化）

### 验证

1. 复现：传入错误类型 → 触发 SIGSEGV
2. 应用 hotfix：再传错误类型，**早返回**，不崩溃
3. ASAN 跑通
4. APM：JNI 异常率 = 0

# 8. 总结

## 8.1 架构师视角 5 条 Takeaway

1. **NE 取证 3 件套**：tombstone + dropbox(SYSTEM_TOMBSTONE) + 符号化服务。
2. **符号化是 NE 排查的瓶颈**：必须接入商业符号化服务（Sentry / Bugsnag / Backtrace.io）。
3. **tombstone 满 10 个被覆盖**：必须主动采集 + 上传云端。
4. **6 种信号取证差异**：先看 signal 编号，再看 backtrace。
5. **必须用与发布版本一致**的 .so + 配套 .debug 符号表。
