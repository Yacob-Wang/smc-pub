# 1. 背景与定义

## 1.1 KE 取证 = dmesg + pstore + last_kmsg 3 件套（**核心难点：dmesg 重启丢失**）

> **一句话定义**：KE 触发后 30 秒内拿到 3 件证据——**dmesg**（运行时 ring buffer）+ **pstore**（持久化 log）+ **last_kmsg**（重启前最后一次 kernel log）。**核心难点**：**dmesg 重启丢失**——KE 触发即整机重启，dmesg 必须在重启后立即抓 pstore / last_kmsg。

**三件套对应**：

| 文件 | 路径 | 触发时机 | 保留 |
|:-----|:-----|:---------|:-----|
| **dmesg** | `/dev/kmsg`（ring buffer）| 运行时 | **重启丢失**（**极高风险**）|
| **pstore** | `/sys/fs/pstore/` | kernel 主动写 | **持久化**（**核心**）|
| **last_kmsg** | `/proc/last_kmsg` | 重启前最后一次 | 需编译时开 |

> **所以呢**：**KE 取证的关键不是 logcat（看不到）**，**是 pstore**——架构师必须**主动配置 pstore + 立即抓**。

## 1.2 KE 取证 vs Stability S07 视角

| 维度 | Stability S07 | Forensics F05 |
|:-----|:--------------|:--------------|
| **视角** | 机制（5 类 KE 触发）| 取证（dmesg / pstore / last_kmsg）|
| **关注** | 7 个机制子节 | 3 件套 + 5 类 KE 差异取证 |

## 1.3 KE 取证的 3 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "logcat 看到了 kernel log" | **logcat 默认看不到 dmesg**（安全考虑）| `adb shell dmesg` 或 pstore |
| "重启了 kernel log 就没了" | **pstore / ramoops 持久化** | 编译时正确配置 |
| "kernel log 重启后能恢复" | 重启前 last_kmsg **仅保留重启前最后一次** | 主动抓 `proc/last_kmsg`（需 root）|

> **所以呢**：**KE 取证是所有症状中最难的**——因为**重启即失 log**。

# 2. 取证 4 步法

## 2.1 触发 → 抓取 → dump 路径 → 解读

### 第 1 步：触发（整机重启 / dmesg 关键字）

```bash
# 实时监控 dmesg（KE 触发时）
adb shell dmesg -w | grep -E "Kernel panic|Oops|hung_task|soft lockup|hard lockup|RCU stall"
```

```bash
[1234.567890] INFO: task xxx:1234 blocked for more than 120 seconds.
[1234.567891] Kernel panic - not syncing: hung_task detected, killing process
[1234.567892] CPU: 0 PID: 1234 Comm: binder_rust Tainted: G        W       5.15.0
```

**关键读法**：
- `Kernel panic - not syncing` = **panic 触发**
- `Oops:` = oops 触发
- `blocked for more than 120 seconds` = hung_task 触发
- `soft lockup` / `hard lockup` = 软/硬死锁
- `RCU stall` = RCU 读侧超时

### 第 2 步：抓取（**重启后立即**抓 3 件套）

**dmesg 抓取**（**重启前**抓，重启丢失）：
```bash
# 运行时 dmesg 抓取（重启前）
adb shell dmesg -T > dmesg_$(date +%Y%m%d_%H%M%S).log
```

**last_kmsg 抓取**（**重启后**立即抓，**核心**）：
```bash
# 重启后立即抓（**第一步**）
adb shell cat /proc/last_kmsg > last_kmsg_$(date +%Y%m%d_%H%M%S).log
```

**pstore 抓取**（**重启后**抓，持久化）：
```bash
# 检查 pstore 是否有内容
adb shell ls /sys/fs/pstore/

# 抓取 pstore（dmesg-ramoops-0 是常见的）
adb shell cat /sys/fs/pstore/dmesg-ramoops-0 > pstore_$(date +%Y%m%d_%H%M%S).log
```

**bugreport 兜底**（**包含 dmesg + last_kmsg**）：
```bash
adb bugreport > bugreport_$(date +%Y%m%d_%H%M%S).zip
# 解压后含：
# - data/system/dropbox/ (dropbox 事件)
# - data/local/traces/ (Perfetto)
# - FS/console-ramoops/ (pstore 备份)
# - FS/last_kmsg (last_kmsg 备份)
```

### 第 3 步：dump 路径

```bash
# 重启后立即检查
$ adb shell ls /sys/fs/pstore/
dmesg-ramoops-0    dmesg-ramoops-1    ...

# 抓取大小（应该 > 0）
$ adb shell cat /sys/fs/pstore/dmesg-ramoops-0 | wc -l
5234
```

### 第 4 步：解读

| 文件 | 关键看 |
|:-----|:------|
| **dmesg / last_kmsg** | `Kernel panic` / `Oops` / `hung_task` / `soft lockup` 关键字 |
| **pstore** | 同上（持久化的 dmesg）|
| **dropbox(KE_*)** | 事件元数据 |

# 3. dmesg 详解（**核心 · 但重启丢失**）

## 3.1 抓取方式

```bash
# 实时监控
adb shell dmesg -w

# 抓取当前 ring buffer
adb shell dmesg -T > dmesg.log

# 带时间戳（推荐）
adb shell dmesg -T -w
```

## 3.2 为什么 dmesg 重启丢失？

```
kernel ring buffer（运行时）：
  - dmesg 写入 → ring buffer（几 MB）
  - KE 触发 → panic + emergency_restart
  - 重启 → ring buffer **清空**（**重启丢失**）
```

> **架构师视角**：**dmesg 是"活的"**——运行时能看到，**但 KE 触发即整机重启，dmesg 丢失**。

## 3.3 应对：3 个持久化机制

| 机制 | 触发 | 持久化 |
|:-----|:-----|:-------|
| **pstore / ramoops** | kernel 主动写 | **持久化 RAM**（即使整机重启）|
| **last_kmsg** | 重启前最后一次 dump | 持久化（需编译时开）|
| **dropbox(KE_*)** | Framework 抓 kernel log 摘要 | 持久化（30 天）|

> **所以呢**：**dmesg 是"现场"，pstore / last_kmsg 是"快照"**——KE 取证必须抓**快照**。

**dmesg → pstore → last_kmsg 持久化链路图**：

```
                    ┌────────────────────────────┐
                    │     Kernel Runtime          │
                    │     dmesg ring buffer       │
                    │     /dev/kmsg (live)        │
                    └────────────┬───────────────┘
                                 │
                                 │ KE 触发
                                 │ (panic / oops / hung_task)
                                 ▼
        ┌────────────────────────────────────────────┐
        │         Kernel 三条持久化路径                │
        └────────┬──────────────────┬───────────────┬┘
                 │                  │               │
        ┌────────▼─────┐  ┌────────▼─────┐  ┌──────▼──────────┐
        │ pstore       │  │ last_kmsg    │  │ Framework        │
        │ (ramoops)    │  │ (boot 启动时 │  │ (dropbox KE_*)  │
        │              │  │  dump)       │  │                  │
        │ 触发:        │  │              │  │ 触发:            │
        │ panic/oops   │  │ 触发:        │  │ kernel 抓摘要    │
        │ + console    │  │ 整机重启     │  │                  │
        │  + pmsg      │  │              │  │                  │
        │ 写入:        │  │ 写入:        │  │ 写入:            │
        │ 持久化 RAM   │  │ 持久化 RAM   │  │ /data/system/    │
        │ (CONFIG_     │  │ (CONFIG_     │  │ dropbox/         │
        │  PSTORE_RAM) │  │  PRINTK_     │  │ KE_*@*.txt       │
        │              │  │  LAST)       │  │                  │
        └────────┬─────┘  └────────┬─────┘  └──────┬──────────┘
                 │                  │               │
                 ▼                  ▼               ▼
        ┌────────────────┐ ┌────────────────┐ ┌────────────────┐
        │ /sys/fs/pstore/│ │ /proc/         │ │ /data/system/  │
        │ ├─ dmesg-ramoops-0│ │   last_kmsg  │ │   dropbox/     │
        │ ├─ console-ramoops-0│ │              │ │   KE_*.txt    │
        │ └─ pmsg-ramoops-0│ │              │ │   (30 天)     │
        │                │ │              │ │                │
        │ 主动读:        │ │ 主动读:        │ │ 主动读:        │
        │ cat /sys/fs/  │ │ cat /proc/   │ │ dumpsys        │
        │ pstore/*       │ │   last_kmsg  │ │ dropbox --print│
        └────────────────┘ └────────────────┘ └────────────────┘
                 │                  │               │
                 └──────────────────┼───────────────┘
                                    ▼
                    ┌────────────────────────────┐
                    │   KE 取证 3 路对账          │
                    │   dmesg (现场)              │
                    │   ↔ pstore (崩溃瞬间)       │
                    │   ↔ last_kmsg (重启前)      │
                    │   ↔ dropbox (kernel 摘要)   │
                    │   → 锁定根因               │
                    └────────────────────────────┘
```

> **架构师视角**：**4 路机制中，pstore 是"最可靠"**——因为是**kernel 主动写**（不依赖 Framework），即使 Framework 没启动也能保留。**last_kmsg 是"最易丢"**——需编译时开 `CONFIG_PRINTK_LAST`，很多厂商默认关闭。

# 4. pstore 详解（**核心 · 持久化机制**）

## 4.1 pstore 是什么？

> pstore（persistent storage）= kernel 用于**持久化存储 panic / oops 等关键 log** 的机制。

**pstore 后端**：
- **ramoops**：pstore 的 RAM 后端（**最常用**）
- **blk**：block device 后端
- **mtd**：MTD 设备后端

**关键特性**：
- 写入 RAM（即使整机重启也能保留）
- 重启后**从 `/sys/fs/pstore/` 读取**

**5 类 KE 取证差异表**：

| KE 类型 | 触发条件 | 关键 log 关键字 | 取证路径 | 严重度 |
|:--------|:---------|:---------------|:---------|:------|
| **Kernel panic** | 致命 oops / 同步异常 | `Kernel panic - not syncing` | dmesg / pstore / dropbox(KE_PANIC) | **极高**（整机重启）|
| **Kernel oops** | 非致命 oops | `Oops:` / `BUG:` | dmesg / pstore | **高**（可能不重启）|
| **hung_task** | 进程 D 状态 ≥ 120s | `task XXX blocked for more than N seconds` | dmesg / pstore / dropbox(KE_*) | **中**（IO HANG 标志）|
| **softlockup** | CPU 软锁 ≥ 20s | `BUG: soft lockup` | dmesg / pstore | **中**（CPU 调度问题）|
| **hardlockup** | CPU 硬锁（NMI 看门狗）| `NMI watchdog: Watchdog detected hard LOCKUP` | dmesg / pstore | **高**（CPU 完全死锁）|
| **RCU stall** | RCU grace period 过长 | `rcu: INFO: rcu_sched self-detected stall` | dmesg / pstore | **中**（RCU 调度问题）|
| **WARN+BUG** | 内核 WARN/BUG 宏 | `WARNING:` / `BUG:` | dmesg / pstore | **中**（可恢复）|

**5 类 KE 取证优先级**：

```
              高 ──── 整机重启 ──── 取证紧急
              │
              │  panic → dmesg / pstore / dropbox(KE_PANIC)
              │
              ├──── oops → pstore / dropbox(KE_OOPS)
              │
              │  hardlockup → pstore / dropbox(KE_HARDLOCKUP)
              │
              ├──── softlockup / hung_task → pstore
              │
              │  RCU stall → pstore / dropbox(KE_RCU)
              │
              低 ──── 可恢复 ──── 取证重要但不紧急
```

> **架构师视角**：**5 类 KE 取证优先级 = 严重度排序**——panic 优先查（整机重启），WARN 延后查（可恢复）。**取证时先看 dmesg 第一行关键字**（`panic` / `Oops` / `BUG` / `WARN` / `soft lockup`）**确定 KE 类型**。

## 4.2 pstore 配置（**架构师必修**）

```kconfig
# .config 关键配置（K 6.18）
CONFIG_PSTORE=y
CONFIG_PSTORE_RAM=y
CONFIG_PSTORE_CONSOLE=y
CONFIG_PSTORE_PMSG=y
CONFIG_PSTORE_FTRACE=y

# pstore 大小（厂商定制，通常 64K-1M）
CONFIG_PSTORE_RAM_SIZE=65536
```

## 4.3 抓取与解读

```bash
# 重启后检查 pstore 状态
$ adb shell ls /sys/fs/pstore/
dmesg-ramoops-0    dmesg-ramoops-1    ...

# 抓取 pstore 内容
$ adb shell cat /sys/fs/pstore/dmesg-ramoops-0 > pstore.log

# 解析（去除二进制头部）
$ sed 's/^.\{0,13\}//' pstore.log > pstore_clean.log
```

**pstore 关键内容**：
```
[1234.567890] INFO: task xxx:1234 blocked for more than 120 seconds.
[1234.567891] Kernel panic - not syncing: hung_task detected, killing process
[1234.567892] CPU: 0 PID: 1234 Comm: binder_rust Tainted: G        W       5.15.0
[1234.567893] Call Trace:
[1234.567894]  __schedule+0x123/0x456
[1234.567895]  schedule+0x45/0x89
[1234.567896]  __mutex_lock.isra.0+0x234/0x345
[1234.567897]  binder_alloc_rust_mutex_lock+0x45/0x67   ← Rust 版 Binder（**2026-07-18 verifier 校正**：实际 K 6.4-6.6 期间合入上游 Greg KH 推动，K 6.18/6.18 主要是生产化）
```

> **架构师视角**：**pstore 是 KE 取证的核心**——架构师**必须确认生产设备的 pstore 配置正确**。

## 4.4 pstore 丢失风险

| 风险 | 应对 |
|:-----|:-----|
| pstore 未开启 | **生产必开**（CONFIG_PSTORE=y）|
| pstore 大小太小 | 调大 CONFIG_PSTORE_RAM_SIZE（64K-1M）|
| pstore 内容被覆盖 | 主动抓（重启后立即 `adb pull`）|

# 5. last_kmsg 详解

## 5.1 last_kmsg 是什么？

> last_kmsg = kernel panic / oops 时，**内核自动 dump 的最后一次 log**。重启后从 `/proc/last_kmsg` 读取。

**关键特性**：
- **重启前自动 dump**（kernel 行为）
- 重启后从 `/proc/last_kmsg` 读取
- **需编译时开启**（默认开启，但部分厂商会关掉）

## 5.2 抓取方式

```bash
# 重启后立即抓
$ adb shell cat /proc/last_kmsg > last_kmsg.log

# 注意：需要 root 权限
$ adb root
$ adb shell cat /proc/last_kmsg > last_kmsg.log
```

## 5.3 与 pstore 的差异

| 维度 | last_kmsg | pstore |
|:-----|:----------|:-------|
| 触发 | kernel panic / oops 时**自动 dump** | kernel 主动写（持续） |
| 路径 | `/proc/last_kmsg`（重启后） | `/sys/fs/pstore/`（持续） |
| 持久化 | **重启保留 1 个** | **持久化多个** |
| 容量 | 几 MB | 64K-1M |
| 优先级 | **重启后第一抓** | 持续抓 |

> **架构师视角**：**last_kmsg 是 KE 重启后第一抓**——pstore 是补充。

# 6. 5 类 KE 差异取证（**核心**）

> 详细机制见 [Stability S07 §3 核心机制与源码](../Stability/S07-KE.md)。本节只讲**取证差异**。

| KE 类型 | dmesg 关键字 | 严重性 | 取证重点 |
|:-------|:------------|:-------|:--------|
| **Kernel Panic** | `Kernel panic - not syncing` | 🔴 极重 | panic banner + 寄存器 + 栈 |
| **Kernel Oops** | `Oops:` | 🟠 重 | oops 栈 + 寄存器 |
| **hung_task** | `blocked for more than X seconds` | 🟠 重 | 进程栈 + mutex 位置 |
| **softlockup** | `BUG: soft lockup - CPU#X stuck for Ys!` | 🟡 中 | watchdog 栈 + 持锁函数 |
| **hardlockup** | `hard lockup` | 🟠 重 | NMI 栈 |
| **WARN / BUG** | `WARNING:` / `Kernel BUG` | 🟡 中 | 告警的代码行 |
| **RCU stall** | `rcu: INFO: rcu_preempt detected stalls` | 🟡 中 | RCU 读侧栈 |

> **架构师视角**：**取证时先看 dmesg 关键字确定 KE 类型**，再看完整栈定位根因。

# 7. 治理

## 7.1 KE 取证自动化

**5 件必做**：
1. **生产设备 pstore 配置正确**（CONFIG_PSTORE=y + 足够大小）
2. **last_kmsg 开启**（编译时开关）
3. **panic_on_oops = 1**（生产推荐，避免 oops 累积）
4. **APM 自动抓取 dmesg / pstore**（重启后第一抓）
5. **lockdep + KASAN 开启**（debug build）

## 7.2 KE 重启后第一抓流程

```
整机突然重启
  ↓
【立即】adb root
【立即】adb shell cat /proc/last_kmsg > last_kmsg.log  ← **第一抓**
【立即】adb shell ls /sys/fs/pstore/                  ← 检查 pstore
【立即】adb shell cat /sys/fs/pstore/dmesg-ramoops-* > pstore.log
【立即】adb shell dumpsys dropbox --print | grep KE   ← dropbox 事件
【立即】adb bugreport > bugreport.zip                 ← 兜底
  ↓
解读：找 Kernel panic / Oops / hung_task 关键字
  ↓
逆向排查：找根因
```

> **所以呢**：**KE 重启后第一抓流程** = last_kmsg + pstore + dropbox + bugreport 4 件套。

## 7.3 预防机制

**5 个必做**：
1. **生产开启 pstore**（CONFIG_PSTORE=y）
2. **生产开启 last_kmsg**（编译时）
3. **生产 panic_on_oops = 1**（避免 oops 累积）
4. **debug build 开启 lockdep**（**必做**）
5. **debug build 开启 KASAN / KMSAN**（**必做**）

# 8. 实战案例

## 8.1 案例 A（CASE-FORENSICS-05-01）：binder 驱动 mutex 死锁 → 完整取证 4 步法

> **类型**：典型模式
>
> **环境**：AOSP 17.0.0_r1 / Kernel android17-6.18 / 设备 Pixel 6
>
> **症状**：App 启动后整机卡死，重启
>
> **根因**：binder 驱动 mutex 锁顺序错误 → hung_task → kernel 升级为 panic

### 现象

```
用户操作：
  T+0s   启动 App（多进程 binder call）
  T+30s  进程 A 进入 D 状态
  T+120s hung_task 触发（默认 120s）
  T+125s kernel 升级为 panic
  T+130s 整机重启
  T+180s 开机
```

### 取证 4 步法（**重启后立即**）

**第 1 步：触发（重启后看 last_kmsg）**

```bash
$ adb root
$ adb shell cat /proc/last_kmsg | head -50
```

```
[ 6000.000] INFO: task binder_thread_X:1234 blocked for more than 120 seconds.
[ 6000.001] "echo 0 > /proc/sys/kernel/hung_task_timeout_secs" disables this message.
[ 6000.002] Call Trace:
[ 6000.003]  __schedule+0x123/0x456
[ 6000.004]  schedule+0x45/0x89
[ 6000.005]  __mutex_lock.isra.0+0x234/0x345
[ 6000.006]  binder_alloc_mutex_lock+0x45/0x67
[ 6000.007]  binder_alloc_new_buffer_locked+0x89/0x123
[ 6000.008]  binder_thread_write+0x234/0x345
```

**关键读法**：
- `blocked for more than 120 seconds` ← **hung_task 触发**
- `binder_alloc_mutex_lock` ← binder 驱动 mutex 卡死
- 进程栈在 mutex_lock 状态

**第 2 步：抓取（3 件套）**

```bash
# last_kmsg 抓取（**第一抓**）
$ adb shell cat /proc/last_kmsg > last_kmsg_$(date +%Y%m%d_%H%M%S).log

# pstore 抓取
$ adb shell ls /sys/fs/pstore/
dmesg-ramoops-0
$ adb shell cat /sys/fs/pstore/dmesg-ramoops-0 > pstore.log

# dropbox 抓取
$ adb shell dumpsys dropbox --print | grep -A 30 "KE_"

# bugreport 兜底
$ adb bugreport > bugreport.zip
```

**第 3 步：dump 路径**

```bash
$ adb shell ls -la /proc/last_kmsg
-rw-r--r-- 1 root root 524288 2026-07-15 10:30 /proc/last_kmsg

$ adb shell ls -la /sys/fs/pstore/
-rw-r--r-- 1 root root 65536 2026-07-15 10:30 dmesg-ramoops-0
```

**第 4 步：解读**

**根因**：
- hung_task 触发 → 进程 A 在 `binder_alloc_mutex_lock` 状态卡死
- binder 驱动中两个 mutex 锁顺序不一致（路径 A vs 路径 B）→ **死锁**

### 修复

AOSP 上游 commit（简化）：
```c
// drivers/android/binder.c
// 修复：统一锁顺序为 alloc_mutex → mmap_lock

static int binder_alloc_new_buffer_locked(...) {
    // 锁顺序：先 alloc_mutex，再 mmap_lock
    mutex_lock(&alloc->mutex);
    if (mmap) {
        mutex_lock(&mmap_lock);  // 锁顺序与另一路径对齐
    }
    ...
}
```

**Linux 6.18 前瞻**：
- _前瞻_：K 6.18 LTS 主要是**生产化** Rust 版 Binder（**2026-07-18 verifier 校正**：Rust 版 Binder 实际 K 6.4-6.6 期间合入上游，K 6.18/6.18 主要是稳定化与生产部署）
- Rust 与 C 版并存的锁顺序问题在 K 6.18 已基本解决
- AOSP 17 推 6.18 分支后**锁顺序治理已基本完成**，但仍需观察

### 验证

1. 应用 patch
2. 复现脚本：高频 binder call 1000 次
3. 验证：不再触发 hung_task
4. APM：hung_task 频率 = 0

# 9. 总结

## 9.1 架构师视角 5 条 Takeaway

1. **KE 取证 3 件套**：dmesg（重启丢失）+ pstore（持久化）+ last_kmsg（重启前最后一次）—— **pstore 是核心**。
2. **dmesg 重启丢失**：KE 触发即整机重启，**必须靠 pstore 持久化**。
3. **重启后第一抓**：last_kmsg + pstore + dropbox + bugreport 4 件套。
4. **生产配置 3 件**：pstore 开启 + last_kmsg 开启 + panic_on_oops = 1。
5. **5 类 KE 差异取证**：先看 dmesg 关键字确定类型，再看完整栈定位根因。

## 9.2 排查路径速查

| 看到症状 | 抓什么 | 跳到 |
|:---------|:-------|:-----|
| 整机突然重启 | `last_kmsg` + pstore + dropbox(KE_*) | §2 / §3 / §4 |
| dmesg 看到 `Kernel panic` | 找 panic banner + 完整栈 | §6 |
| dmesg 看到 `hung_task` | 找进程栈 + mutex 位置 | §6 / §8 案例 A |
| dmesg 看到 `soft lockup` | 找 watchdog 栈 + 持锁函数 | §6 |
