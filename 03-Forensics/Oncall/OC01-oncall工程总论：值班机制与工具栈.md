# 1. 为什么需要 oncall 工程化？

## 1.1 现状：野生 oncall 的 5 大痛点

> **野生 oncall** = 没有标准化的 oncall 流程，全靠"老员工记忆 + Slack 群吼"

| # | 痛点 | 后果 | 根因 |
|:-:|:-----|:-----|:-----|
| 1 | **响应时长不可控** | 线上问题 1 小时才有人看 | 没有轮转 + 没有升级 |
| 2 | **取证不标准** | 每次都要从头摸索 | 没有 playbook |
| 3 | **新人不愿值班** | 老员工连续值班 burnout | 没有 backup + 培训 |
| 4 | **故障反复发生** | 同一个问题每周来一次 | 没有 postmortem 闭环 |
| 5 | **复盘无输出** | 故障过去了就过去了 | 没有度量 + 改进机制 |

## 1.2 工程化 oncall 的 3 大收益

| 收益 | 含义 | 量化 |
|:-----|:-----|:-----|
| **1. MTTR 缩短** | 平均恢复时间 | 30 分钟 → 5 分钟 |
| 2. **知识沉淀** | 故障处理经验变成团队资产 | 减少 80% "问老人" |
| 3. **oncall 健康** | 不再 burnout | 满意度 +50% |

> **架构师视角**：oncall 工程化 = **把个人能力变成组织能力**。没有 oncall 体系 = 团队稳定性全靠"几个关键人"。

## 1.3 oncall 体系的 4 大组成

```
值班轮转（Who）  +  升级矩阵（How）  +  工具栈（What）  +  复盘文化（Why）
   ↓                  ↓                  ↓                  ↓
 谁来响应          怎么升级          用什么工具          怎么持续改进
```

# 2. 值班轮转：4 种模式 + 选型

## 2.1 4 种轮转模式

| 模式 | 描述 | 适用 | 优点 | 缺点 |
|:-----|:-----|:-----|:-----|:-----|
| **1. 主备轮转** | 1 主 1 备 | 小团队 | 简单 | 主值班疲劳 |
| **2. 跟随轮转** | 1 主 + 1 跟随（新人）| 培养期 | 传帮带 | 主值班需分心 |
| **3. 7×24 轮转** | 1 主 + 1 备 + 1 升级，全天覆盖 | 关键业务 | 响应快 | 人力成本高 |
| **4. Follow-the-Sun** | 跨时区轮转 | 海外业务 | 24h 真人 | 跨时区协调 |

**推荐**：**主备 + 跟随**（10-20 人团队最佳实践）

## 2.2 7 天 5 人轮转示例

```
人员：A（主 R）/ B（备 R1）/ C（跟随）/ D（主 R）/ E（备 R1）
班次：周一-周日 7 天
主 R 负责当周 7×24 响应
备 R1 升级到 R1 才介入
跟随 不值班，只跟主 R 学习 1 周
```

| 周一 | 周二 | 周三 | 周四 | 周五 | 周六 | 周日 |
|:----:|:----:|:----:|:----:|:----:|:----:|:----:|
| A 主 | A 主 | A 主 | A 主 | A 主 | A 主 | A 主 |
| B 备 | B 备 | B 备 | B 备 | B 备 | B 备 | B 备 |
| C 跟 | C 跟 | C 跟 | C 跟 | C 跟 | - | - |

> **关键规则**：
> - 主 R 周末允许"非紧急情况延迟 1 小时响应"
> - 备 R1 周末 1 小时内必须能远程连入
> - 跟随只在工作日 9-18 点跟学
> - 每 4 周轮换 1 次（避免疲劳）

## 2.3 轮转工具

| 工具 | 类型 | 推荐 |
|:-----|:-----|:-----|
| PagerDuty | 商业 | 海外大厂 |
| OpsGenie | 商业 | 国内可用 |
| 自研（Webhook + 飞书/钉钉）| 自研 | 成本敏感 |

# 3. 升级矩阵：3 级升级 + 5 分钟黄金窗口

## 3.1 黄金 5 分钟 + 白银 15 分钟 + 黄金 30 分钟

> **oncall 黄金法则**：**故障时间是分秒必争的黄金**

```
00:00  告警触发
00:01  oncall 收到告警
00:05  oncall 介入（黄金 5 分钟内）
       ↓ 如未响应，升级到备 R1
00:15  定位根因（白银 15 分钟内）
       ↓ 如未定位，升级到 Tech Lead
00:30  启动回滚或修复（黄金 30 分钟内）
       ↓ 如未恢复，升级到架构师 + 启动全团队应急
```

## 3.2 3 级升级矩阵

| 级别 | 触发条件 | 责任人 | 通知方式 |
|:-----|:---------|:-------|:---------|
| **L1 oncall** | 告警触发 | 主 R | 短信 + 飞书 |
| **L2 备 R1** | L1 5 分钟未响应 | 备 R1 | 电话（强制）|
| **L3 Tech Lead** | 15 分钟未定位 | TL | 飞书 + 电话 |
| **L4 架构师** | 30 分钟未恢复 | 稳定性架构师 | 飞书群 + 电话 |
| **L5 全员应急** | 1 小时未恢复 | 全团队 | 紧急群 + 拉视频会议 |

> **红线**：**L1 5 分钟未响应 = 必须升级**（不升级 = 事故定性更严重）

## 3.3 升级通知模板

```yaml
# 飞书/Lark 卡片（oncall 自动发）
告警级别: P0
告警名称: 启动成功率低于 99%
触发时间: 2026-07-22 14:30:00
影响范围: 全量用户（约 100 万 DAU）
当前 oncall: @A
未响应时长: 5 分钟
建议动作: 
  1. 拉 traces.txt 看 1% 异常 Session
  2. 跑 OC02-ANR 剧本
  3. 启动 P0 应急
升级路径: A (主) → B (备 R1) → @Tech_Lead → @架构师
```

# 4. 工具栈 5 层模型

> **oncall 工具栈 = 5 层金字塔**，从下到上：**监控 → 取证 → 协同 → 知识库 → 度量**

```
        ┌─────────┐
        │  度量  │  APM 看板 / oncall 报告
        ├─────────┤
        │ 知识库  │  Runbook / postmortem / F 系列
        ├─────────┤
        │  协同  │  飞书 / Slack / PagerDuty
        ├─────────┤
        │  取证  │  dumpsys / perfetto / hprof / 04-Tool 全系列
        ├─────────┤
        │  监控  │  APM A01-A10 / 告警 / 拨测
        └─────────┘
```

## 4.1 第 1 层：监控（3 大类工具）

| 类别 | 工具 | 推荐 |
|:-----|:-----|:-----|
| **APM 客户端** | 自研 SDK / Sentry / Firebase Crashlytics | [A01-APM 体系总览](../../05-Governance/APM/A01-APM体系总览.md) |
| **APM 后端** | 自研（Kafka+Flink+ClickHouse）/ Datadog / New Relic | [A07-度量后端设计](../../05-Governance/APM/A07-度量后端设计.md)（待补）|
| **告警** | Prometheus AlertManager / Grafana / PagerDuty | 自研对接 |

## 4.2 第 2 层：取证（4 类工具）

| 类别 | 工具 | 场景 |
|:-----|:-----|:-----|
| **基础** | dumpsys / logcat / bugreport | 所有场景 |
| **性能** | Perfetto / atrace / systrace | 启动/卡顿/ANR |
| **内存** | hprof / LeakCanary / procrank | OOM/泄漏 |
| **原生** | debuggerd / tombstone / addr2line | NE/KE |

详见 [04-Tool 全系列](../../04-Tool/)。

## 4.3 第 3 层：协同（3 大系统）

| 类别 | 工具 | 强制要求 |
|:-----|:-----|:---------|
| **告警通道** | 飞书机器人 / PagerDuty | 5 分钟内必达 |
| **应急群** | 飞书群 / 钉钉群 | 全员 oncall 在线 |
| **视频会议** | 飞书会议 / Zoom / 腾讯会议 | 30 分钟升级时启用 |

## 4.4 第 4 层：知识库（4 类资产）

| 类别 | 例子 | 维护人 |
|:-----|:-----|:-------|
| **Runbook** | OC02-OC08 7 大症状剧本 | oncall Lead |
| **Postmortem** | 历次故障复盘文档 | 当次 oncall |
| **症状库** | S00-S10 + F00-F07 | 稳定性架构师 |
| **源码索引** | 01-Mechanism 全部 | 全员 |

## 4.5 第 5 层：度量（4 大指标）

| 指标 | 公式 | 目标 |
|:-----|:-----|:-----|
| **MTTR** | 故障发生到恢复时间 | ≤ 30 分钟 |
| **MTTA** | 故障发生到首次响应 | ≤ 5 分钟 |
| **MTTD** | 故障发生到首次检测 | ≤ 2 分钟 |
| **oncall 满意度** | 月度匿名调研 | ≥ 7/10 |

> **关键公式**：
> ```
> MTTD ≤ MTTA ≤ MTTR
>   ↓        ↓        ↓
> 检测    介入     恢复
> ```

# 5. oncall 5 大红线

> **红线 = 绝对不能违反的规则**，违反 = 团队级 P0 故障

| # | 红线 | 含义 | 违反后果 |
|:-:|:-----|:-----|:---------|
| **1** | **不追责** | oncall 处理不当时**不追责个人** | 追责 = 隐藏问题 = 更大事故 |
| **2** | **必须交接** | 班次切换必须面对面 + Runbook 更新 | 漏交 = 下一个 oncall 抓瞎 |
| **3** | **必须 postmortem** | 所有 P0/P1 必须 24h 内出 postmortem | 不写 = 故障重演 |
| **4** | **必须有 backup** | 每个 oncall 必须有 backup | 没 backup = 失联 = 重大事故 |
| **5** | **疲劳红线** | 连续 oncall 超过 7 天强制换班 | 疲劳 = 响应变慢 = 二次事故 |

> **架构师视角**：**oncall 是"高强度脑力劳动"，疲劳管理不是福利，是风险控制**。

# 6. oncall 5 步落地法

## Step 1：建 Runbook（1 周）

```
任务：把 S00-S10 + F00-F07 整理成"5 分钟可用"的 Runbook
产出：OC02-OC08 7 大症状剧本（OC01 本篇 + OC02-OC08 共 8 篇）
验收：新人按 Runbook 能在 5 分钟内找到所需信息
```

## Step 2：定轮转（1 周）

```
任务：按团队规模设计轮转表
产出：oncall 轮转表 + 升级矩阵
验收：所有 oncall 知道"我是谁 + 升级到谁"
```

## Step 3：接监控（1-2 周）

```
任务：把 APM 告警接到 oncall 通道
产出：5 大 SLI 告警规则
验收：oncall 能在告警 5 分钟内收到通知
```

## Step 4：演练（2 周）

```
任务：每两周做 1 次故障演练（故障注入）
产出：演练 postmortem
验收：MTTR ≤ 30 分钟
```

## Step 5：度量 + 迭代（持续）

```
任务：每月输出 oncall 月报
产出：MTTA / MTTR / 满意度趋势
验收：连续 3 月无重大故障
```

# 8. oncall 反例清单

| # | 反例 | 错误做法 | 正确做法 |
|:-:|:-----|:---------|:---------|
| 1 | **oncall 没培训** | 直接排班 | **上岗前 1 周跟班** |
| 2 | **追责文化** | 故障后扣绩效 | **只对事不对人** |
| 3 | **告警疲劳** | 100 个告警一齐来 | **分级 + 静默规则** |
| 4 | **没 backup** | 1 个人 oncall | **每班至少 2 人** |
| 5 | **疲劳值班** | 连续值班 1 个月 | **最多 1 周轮换** |
| 6 | **不写 postmortem** | 故障过了就过 | **24h 内必须写** |
| 7 | **Runbook 过期** | 1 年前写的文档 | **每季度 review** |
| 8 | **没演练** | 真实故障才暴露问题 | **每 2 周演练** |
| 9 | **跨时区不友好** | 全用北京时间 | **Follow-the-Sun** |
| 10 | **告警不可达** | 飞书宕机没人收到 | **多通道 fallback** |
| 11 | **没升级机制** | 5 分钟没人就放着 | **5/15/30 升级** |
| 12 | **oncall 没奖励** | 当成负担 | **值班津贴 + 调休** |

# 9. 5 条 Takeaway

1. **oncall 工程化 = 4 大组成**（值班轮转 + 升级矩阵 + 工具栈 + 复盘文化）—— 缺一不可
2. **黄金 5 + 白银 15 + 黄金 30** 是 oncall 时间线的 3 个关键节点 —— 超时必须升级
3. **工具栈 5 层金字塔**（监控 → 取证 → 协同 → 知识库 → 度量）—— 自下而上建设
4. **5 大红线**（不追责 / 必须交接 / 必须 postmortem / 必须有 backup / 疲劳红线）—— 违反 = 团队级 P0
5. **5 步落地法**（Runbook → 轮转 → 监控 → 演练 → 度量）—— 1-2 个月可建成
