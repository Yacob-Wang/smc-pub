# 1. 背景与定义

## 1.1 HANG vs OOM 取证的根本差异

> **关键洞察**：HANG 和 OOM 是两个**完全不同**的取证场景。

| 症状 | 是否有自动 dump | 取证核心 |
|:-----|:---------------|:---------|
| **HANG** | ❌ **没有**（未被任何机制捕获）| **主动抓**（systrace / ftrace / Perfetto）|
| **OOM** | ✅ **有**（hprof 自动生成）| **抓 + 解读**（hprof + smaps + meminfo）|

> **所以呢**：**HANG 取证 = 主动监控 + 主动抓**；**OOM 取证 = 自动生成 + 抓 + 解读**。**完全不同的思路**。

## 1.2 HANG 取证 vs Stability S05 视角

| 维度 | Stability S05 | Forensics F06 |
|:-----|:--------------|:--------------|
| **视角** | 机制（4 层 HANG + 主动监控）| 取证（主动抓 trace + 主动监控工具）|
| **关注** | 5 个机制子节 | 主动抓 + OOM hprof 完整链路 |

## 1.3 HANG 取证的 3 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "HANG 没有 dump" | 是的，**但可以主动抓** | systrace / ftrace / Perfetto 主动抓 |
| "OOM 自动生成 hprof 就够了" | hprof 是**Java 堆**，**Native 内存看 smaps** | hprof + smaps + meminfo 联动 |
| "systrace 太重" | systrace 抓 5-10s 是合理的 | 关键场景下开启（ANR 触发时）|

> **所以呢**：**HANG + OOM 取证 = 主动抓 + 主动监控 + 联动解读**。

# 2. HANG 取证 4 步法（**核心 · 主动抓**）

## 2.1 触发 → 主动抓 → dump 路径 → 解读

### 第 1 步：触发（主动监控发现 HANG）

```bash
# 主线程 P95 latency 监控
# 业务代码中实现（参考 Stability S05 §5.1）
public void startMainThreadMonitor() {
    Choreographer.getInstance().postFrameCallback(frameTimeNanos -> {
        long latencyMs = (System.nanoTime() - frameTimeNanos) / 1_000_000;
        if (latencyMs > 200) {
            APM.report("main_thread_hang", Map.of(
                "latency_ms", latencyMs,
                "stack", Thread.currentThread().getStackTrace()
            ));
        }
    });
}
```

### 第 2 步：主动抓（**systrace / Perfetto / ftrace**）

**systrace 抓取**（**ANR 触发时主动开启**）：
```bash
# 异步抓 10s
adb shell atrace --async_start -t 10 view input res am wm gfx hal dalvik sched freq idle

# 或者用 perfetto（推荐，AOSP 17 默认）
adb shell perfetto --background --config /data/local/tmp/perfetto.cfg \
    --out /data/local/traces/hang_trace.pftrace
```

**ftrace 抓取**（**kernel 层**）：
```bash
# 开启调度追踪
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_stat_blocked/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_stat_wait/enable"

# 抓 30s
adb shell timeout 30 cat /sys/kernel/debug/tracing/trace > ftrace.log
```

**bugreport 兜底**（**含 systrace + ftrace**）：
```bash
adb bugreport > bugreport_$(date +%Y%m%d_%H%M%S).zip
# 解压后含：
# - data/local/traces/ (Perfetto)
# - kernel/tracing/ (ftrace)
# - data/system/dropbox/ (dropbox 事件)
# - logcat 全量
```

### 第 3 步：dump 路径

```bash
$ adb shell ls -la /data/local/traces/
-rw-rw-r-- 1 system system 5.2M 2026-07-15 10:23 hang_trace.pftrace
```

### 第 4 步：解读（**关键 · 用 Perfetto UI 打开**）

打开 hang_trace.pftrace 在 Perfetto UI（https://ui.perfetto.dev/）：

**关键看**：
- **主线程带（Main Thread Track）**：在哪个时间点卡住
- **binder 远端带**：远端服务在做什么
- **kernel IO 带**：IO 是否卡住
- **Choreographer 帧时间**：主线程 P95 latency

# 3. HANG 4 层取证（**核心**）

> HANG 4 层结构详细见 [Stability S05 §3 核心机制与源码](../Stability/S05-HANG.md)。本节只讲**取证差异**。

| 层 | 取证工具 | 抓取命令 | 关键看 |
|:---|:---------|:---------|:------|
| **Layer 1: App 主线程软卡死** | systrace / Perfetto | `adb shell perfetto --background --config ...` | 主线程带 + Choreographer 帧时间 |
| **Layer 2: IO HANG** | iostat / Perfetto / ftrace | `adb shell iostat -x 1` + Perfetto | IO await > 30s 但未达 hung_task 120s |
| **Layer 3: Binder HANG** | binder stats + Perfetto | `adb shell cat /sys/kernel/debug/binder/stats` | binder 排队 / transaction 满 |
| **Layer 4: Kernel HANG（升级为 KE）** | 已升级为 KE | [F05 KE 取证](../F05-KE/01-取证机制.md) | hung_task 触发后**不是 HANG**，是 KE |

> **架构师视角**：**Layer 1-3 才是 HANG**（未被任何机制捕获），**Layer 4 已经升级为 KE**（不是 HANG）。

## 3.1 Layer 1: App 主线程软卡死取证

**触发条件**：主线程 4-5s（未达 ANR 阈值 5s）

**取证**：
1. **Perfetto trace 抓取**（ANR 触发时）
2. **Choreographer 帧时间监控**（主动发现）
3. **主线程栈抓取**（如果 HANG 时正好在 ANR 阈值边缘）

**Perfetto 关键看**：
```
T+0ms   Main Thread: Choreographer#doFrame
T+50ms  Main Thread: SQLiteQuery.fillWindow
T+4050ms Main Thread: still in SQLiteQuery  ← 4s 软卡（未到 5s ANR 阈值）
T+4051ms Main Thread: SQLiteQuery 结束
T+4100ms Main Thread: Choreographer#doFrame  ← 帧恢复
```

**关键读法**：**4-5s 的卡**——用户感知但 ANR 不触发，这就是 **HANG Layer 1**。

## 3.2 Layer 2: IO HANG 取证

**触发条件**：IO 30s+（未达 hung_task 120s）

**取证**：
1. **iostat 持续监控**：
   ```bash
   adb shell iostat -x 1
   # 关键看：await > 30s 但未达 120s
   ```
2. **Perfetto trace 抓取**（IO 调用栈）
3. **ftrace 抓取**（block layer 路径）

**典型 IO HANG 时间线**：
```
T+0ms    App: fileRead(path)
T+10ms   VFS: vfs_read
T+20ms   f2fs: f2fs_read  ← f2fs journal 卡
T+30000ms f2fs: still in f2fs_read  ← 30s IO HANG
T+120000ms hung_task: triggered  ← **升级为 KE**（不再是 HANG）
```

## 3.3 Layer 3: Binder HANG 取证

**触发条件**：binder 死锁 / 排队 / transaction 满

**取证**：
1. **binder stats 抓取**：
   ```bash
   adb shell cat /sys/kernel/debug/binder/stats
   # 关键看：transaction 满 / transaction_free < threshold
   ```
2. **Perfetto trace 抓取**（binder 远端栈）
3. **ftrace 抓取**（binder driver 路径）

**Binder HANG 时间线**：
```
T+0ms    App: binder.call(remote_service)
T+10ms   binder driver: transaction queued
T+20ms   remote service: 卡死（CPU 0%）
T+1000ms App: still waiting  ← 1s HANG
T+30000ms App: still waiting  ← 30s HANG
```

# 4. OOM 取证 4 步法（**核心 · 自动 dump**）

## 4.1 触发 → 抓取 → 解读 → 修复

### 第 1 步：触发（hprof 自动生成）

```
App 触发 OOM
  ↓
ActivityManagerNative.getDefault().setProcessStateSummary()
  ↓
触发 hprof dump（默认 5s 阈值）
  ↓
hprof 写入 /data/misc/heap-dump/*.hprof
```

> **架构师视角**：**hprof 是自动生成的**（与 HANG 相反），但**默认是 Java 堆**，**Native 内存需要 smaps**。

### 第 2 步：抓取（3 件套）

**hprof 抓取**：
```bash
# 立即抓（OOM 触发后 1 分钟内，hprof 满 1-3 个会覆盖）
adb pull /data/misc/heap-dump/ ./oom_dumps/

# 或主动触发（不依赖自动）
adb shell am dumpheap <package> /data/local/tmp/oom.hprof
adb pull /data/local/tmp/oom.hprof
```

**smaps 抓取**（**Native 内存取证**）：
```bash
# Native 内存映射（关键看 .so / .jar / 匿名映射）
adb shell dumpsys meminfo <package>
# 或
adb pull /proc/<pid>/smaps ./smaps/
```

**proc/meminfo 抓取**（**系统内存**）：
```bash
adb shell cat /proc/meminfo > meminfo.log
```

**dropbox 抓取**：
```bash
adb shell dumpsys dropbox --print | grep -A 30 "LOW_MEM\|NATIVE_MEMORY"
```

### 第 3 步：dump 路径

```bash
$ adb shell ls -la /data/misc/heap-dump/
-rw------- 1 system system 234M 2026-07-15 10:24 heap-dump-tm12458923-9342.hprof

$ adb shell dumpsys meminfo com.example.app
...
TOTAL PSS:    245000
TOTAL RSS:    320000
TOTAL SWAP PSS:    0
```

### 第 4 步：解读

| 文件 | 关键看 |
|:-----|:------|
| **hprof** | Java 堆对象引用（LeakCanary / Memory Analyzer）|
| **smaps** | .so 内存 + 匿名内存 + VSS/RSS/PSS |
| **proc/meminfo** | 系统整体内存状态 |
| **dropbox(LOW_MEM)** | 系统级低内存告警 |

# 5. hprof 详解（**核心 · OOM 排查**）

## 5.1 hprof 是什么？

> hprof = **Java heap profile**——Java 堆的完整快照（包含所有对象 + 引用关系）。

**关键特性**：
- **只包含 Java 堆**（**不包含 Native 内存**）
- 通常 50-500MB
- 自动生成（OOM 触发时）
- 默认保留 1-3 个（**满后覆盖**）

## 5.2 hprof 抓取

```bash
# 1. 自动生成（OOM 触发）
adb pull /data/misc/heap-dump/

# 2. 主动触发（不依赖自动）
adb shell am dumpheap <package> /data/local/tmp/oom.hprof
adb pull /data/local/tmp/oom.hprof
```

## 5.3 hprof 解读

**工具**：
- **Eclipse MAT**（Memory Analyzer Tool）—— 桌面工具
- **LeakCanary** —— App 内自动检测
- **Android Studio Profiler** —— 集成开发工具

**关键看**：
- **Leak suspects**（泄漏嫌疑）
- **Dominator tree**（支配树，**看哪个对象占用最大**）
- **GC roots**（GC 根，**看谁在持有**）

> **架构师视角**：**hprof 解读需要专业工具**——MAT 是工业标准。

## 5.4 hprof 限制

| 限制 | 应对 |
|:-----|:-----|
| **只含 Java 堆** | Native 内存用 smaps |
| **大文件**（50-500MB）| 主动抓 + 上传云端 |
| **保留 1-3 个**（满后覆盖）| 主动采集 |
| **解读复杂** | 接入 LeakCanary 自动分析 |

# 6. smaps 详解（**Native 内存取证**）

## 6.1 smaps 是什么？

> smaps = `/proc/<pid>/smaps` —— 进程的**详细内存映射**（按段列出每段的大小、权限、RSS、PSS 等）。

**关键特性**：
- 包含 .so / .jar / 匿名映射 / 堆栈 / 设备内存
- 比 proc/meminfo 更详细
- **Native 内存取证必备**

## 6.2 smaps 抓取

```bash
# 单个进程
adb shell cat /proc/<pid>/smaps > smaps.log

# 或用 dumpsys meminfo（更易读）
adb shell dumpsys meminfo com.example.app
```

## 6.3 smaps 解读

```
7f1234567000-7f123456b000 r--p 00000000 fd:01 12345  /data/app/com.example.app/lib/arm64/libnative.so
Size:               12 kB
KernelPageSize:     4 kB
MMUPageSize:        4 kB
Rss:                12 kB
Pss:                12 kB
...
```

**关键看**：
- 哪些 .so 占内存
- 哪些匿名映射占内存（**Native 内存泄漏的关键**）
- PSS 总量（**进程实际占内存**）

# 7. 主动监控工具（**HANG 取证的核心**）

## 7.1 主线程 P95 latency 监控

**代码**（业务内）：
```java
// Choreographer 监控（参考 Stability S05 §5.1）
public class MainThreadMonitor {
    private static final long HANG_THRESHOLD_MS = 200;
    private final Choreographer.FrameCallback callback = frameTimeNanos -> {
        long latencyMs = (System.nanoTime() - frameTimeNanos) / 1_000_000;
        if (latencyMs > HANG_THRESHOLD_MS) {
            // 上报
            APM.report("main_thread_hang", Map.of(
                "latency_ms", latencyMs,
                "stack", Thread.currentThread().getStackTrace()
            ));
        }
        Choreographer.getInstance().postFrameCallback(this.callback);
    };
}
```

**抓取触发**：
```bash
# 当 APM 收到 main_thread_hang 报告时，自动抓 Perfetto
adb shell perfetto --background --config hang_monitor.cfg \
    --out /data/local/traces/hang_trace.pftrace
```

## 7.2 iostat 持续监控

```bash
# 持续 30s
adb shell iostat -x 1 30 > iostat.log
# 关键看：await > 30s 但未达 120s（IO HANG）
```

## 7.3 binder stats 监控

```bash
# 持续 30s
while true; do
    adb shell cat /sys/kernel/debug/binder/stats >> binder_stats.log
    sleep 1
done
# 关键看：transaction_free < threshold
```

## 7.4 ftrace 关键路径监控

```bash
# 监控主线程调度延迟
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_stat_blocked/enable"
adb shell "echo 1 > /sys/kernel/debug/tracing/events/sched/sched_stat_wait/enable"

# 抓 30s
adb shell timeout 30 cat /sys/kernel/debug/tracing/trace > ftrace.log
```

# 8. 治理

## 8.1 HANG 取证自动化

**5 件必做**：
1. **主线程 P95 latency 监控**（**必做**）
2. **所有 binder call 加 timeout**（**必做**）
3. **所有 IO 加 timeout**（**必做**）
4. **ftrace 关键路径接入**（推荐）
5. **APM 接入 Sentry / 自研**（**必做**）

## 8.2 OOM 取证自动化

**4 件必做**：
1. **hprof 主动采集**（自动 + 主动触发）
2. **smaps 持续监控**（dumpsys meminfo 接入 APM）
3. **LeakCanary 自动分析**（debug build）
4. **APM 接入**（Sentry / 自研）

# 9. 实战案例

## 9.1 案例 A（CASE-FORENSICS-06-01）：Volley 4.5s 软卡 → 主动抓 Perfetto

> **类型**：典型模式
>
> **环境**：AOSP 17.0.0_r1 / Kernel android17-6.18 / 设备 Pixel 6
>
> **症状**：用户报"App 列表滚动偶尔卡，但没弹 ANR"
>
> **根因**：Volley 回调在主线程同步做 4.5s JSON 解析，**未达 ANR 5s 阈值**

### 现象

```
用户操作：
  T+0s   滚动列表
  T+0.5s 网络回调
  T+5s   JSON 解析阻塞主线程 4.5s
  T+5s   **未触发 ANR**（差 0.5s）
  T+5.1s 解析完成
  T+5.5s 列表恢复滚动

**关键观察**：用户感知卡（4.5s 静止）但 ANR 没弹
```

### 取证 4 步法（**主动抓**）

**第 1 步：触发（主动监控发现 HANG）**

主线程 P95 latency 监控报告：
```json
{
  "event": "main_thread_hang",
  "latency_ms": 4500,
  "stack": "at com.example.app.VolleyResponseDelivery.deliverResponse..."
}
```

**第 2 步：主动抓（**关键 · 主动触发 Perfetto**）**

```bash
# 当 APM 收到 hang 报告时，自动触发
$ adb shell perfetto --background --config hang_monitor.cfg \
    --out /data/local/traces/hang_$(date +%Y%m%d_%H%M%S).pftrace
[Perfetto started in background, output: /data/local/traces/hang_2026-07-15_10-23-45.pftrace]

# 10s 后抓取
$ sleep 10
$ adb pull /data/local/traces/hang_2026-07-15_10-23-45.pftrace
./hang_trace.pftrace
5.2M pulled in 0.5s
```

**第 3 步：dump 路径**

```bash
$ adb shell ls -la /data/local/traces/
-rw-rw-r-- 1 system system 5.2M 2026-07-15 10:23 hang_2026-07-15_10-23-45.pftrace
```

**第 4 步：解读（用 Perfetto UI 打开 hang_trace.pftrace）**

```
T+0ms    Main Thread: Choreographer#doFrame
T+50ms   Main Thread: VolleyResponseDelivery.deliverResponse
T+60ms   Main Thread: JSON.parse（开始 4.5s 卡顿）
T+5000ms Main Thread: JSON.parse（**4.5s 后**）
T+5050ms Main Thread: Choreographer#doFrame  ← 帧恢复
```

**关键发现**：**JSON.parse 占了 4.5s 主线程**——但 ANR 没触发（5s 阈值）。

### 修复

**短期**（同 [Stability S05 案例 A §6.1 修复方案](../Stability/S05-HANG.md)）：
```java
// 改前（主线程解析）
@Override
public void onResponse(JSONObject response) {
    dataList = JsonParser.parseList(response);  // 4.5s 同步
    adapter.notifyDataSetChanged();
}

// 改后（异步 + 线程池）
private final ExecutorService parseExecutor = Executors.newFixedThreadPool(2);

@Override
public void onResponse(JSONObject response) {
    parseExecutor.execute(() -> {
        List<Item> items = JsonParser.parseList(response);
        handler.post(() -> {
            dataList = items;
            adapter.notifyDataSetChanged();
        });
    });
}
```

### 验证

1. 复现：滚动 + 大 JSON
2. Perfetto 验证：JSON.parse 不再占主线程
3. APM：主线程 P95 latency < 200ms
4. 用户报障率 = 0

## 9.2 案例 B（CASE-FORENSICS-06-02）：OOM 频繁 → hprof + smaps 联动取证

> **类型**：典型模式
>
> **环境**：AOSP 14.0.0_r1 / Kernel 5.10 / 设备 Pixel 6
>
> **症状**：用户报"App 经常 OOM 闪退"
>
> **根因**：Bitmap 缓存未 LRU，导致 Native 内存累积

### 现象

```
用户操作：
  T+0s   打开 App
  T+30s  浏览图片列表（连续加载 100 张大图）
  T+60s  OOM 抛出
  T+60.1s 进程被杀
  T+60.2s dropbox(APP_CRASH) 写入

**关键观察**：**每次浏览大量图片后 OOM**
```

### 取证 4 步法

**第 1 步：触发（hprof 自动生成）**

```logcat
07-15 10:23:45.123 E AndroidRuntime: FATAL EXCEPTION: main
07-15 10:23:45.124 E AndroidRuntime: java.lang.OutOfMemoryError: Failed to allocate a 8MB byte array
07-15 10:23:45.125 E AndroidRuntime:    at com.example.app.ImageLoader.load(ImageLoader.java:42)
```

**第 2 步：抓取（hprof + smaps + meminfo）**

```bash
# hprof 抓取（自动生成在 /data/misc/heap-dump/）
$ adb pull /data/misc/heap-dump/
./heap-dump-tm12458923-9342.hprof
234M pulled in 5.0s

# smaps 抓取（关键 · Native 内存）
$ adb shell dumpsys meminfo com.example.app > meminfo.log
$ adb shell cat /proc/$(adb shell pidof com.example.app)/smaps > smaps.log

# dropbox 抓取
$ adb shell dumpsys dropbox --print | grep -A 30 "APP_CRASH"
```

**第 3 步：dump 路径**

```bash
$ adb shell ls -la /data/misc/heap-dump/
-rw------- 1 system system 234M 2026-07-15 10:23 heap-dump-tm12458923-9342.hprof

$ head -50 meminfo.log
App Summary
...
Native Heap    156000  145000  145000  ...  ← **Native 堆 145MB**（过高）
Java Heap      78000   65000   65000
Code           12000   12000   12000
Stack            500     500     500
Graphics       10000   10000   10000
...
TOTAL PSS:    245000
```

**第 4 步：解读（hprof + smaps 联动）**

**hprof 解读**（Eclipse MAT）：
- Dominator tree：Bitmap 占 Java 堆 65MB（其中 50MB 是同一张图的多个引用）
- GC roots：`ImageCache`（static）持有所有 Bitmap

**smaps 解读**（关键 · **Native 内存**）：
```
7f1234567000-7f123456b000 r--p ...  /data/app/com.example.app/lib/arm64/libimage.so
Size:               120 MB  ← libimage.so 段
...
```

**关键发现**：
- **Java 堆 78MB**（hprof 显示 Bitmap 重复引用）—— 内存泄漏
- **Native 堆 145MB**（libimage.so 占大头）—— Native 内存泄漏
- **联动结论**：Bitmap 缓存既泄漏 Java 堆，又泄漏 Native 堆

### 修复

**短期**：
```java
// 改前（无 LRU 限制）
private static final Map<String, Bitmap> cache = new HashMap<>();

// 改后（LRU + 软引用）
private static final LruCache<String, Bitmap> cache = new LruCache<>(50 * 1024 * 1024);  // 50MB
```

**长期**：用 Glide / Picasso（自动 LRU + 内存优化）

### 验证

1. 复现：浏览 100 张大图
2. 应用 hotfix：再浏览 100 张，无 OOM
3. dumpsys meminfo：Native 堆 < 50MB
4. hprof：Bitmap 缓存 < 50MB

# 10. 总结

## 10.1 架构师视角 5 条 Takeaway

1. **HANG vs OOM 取证思路完全不同**：HANG 主动抓 / OOM 自动生成。
2. **HANG 3 件套**：主线程 P95 latency 监控 + Perfetto 主动抓 + 喂狗链路监控。
3. **OOM 3 件套**：hprof（Java 堆）+ smaps（Native 内存）+ meminfo（系统）。
4. **hprof 只含 Java 堆**：Native 内存必须用 smaps——**联动解读**。
5. **主动监控是 HANG 取证的核心**：**不能等 ANR / KE**——HANG 是 ANR 之前 100s+ 的"沉默期"。

## 10.2 排查路径速查

| 看到症状 | 抓什么 | 跳到 |
|:---------|:-------|:-----|
| 用户报"卡"无 ANR | 主线程 P95 latency + Perfetto 主动抓 | §2 / §3 |
| IO 慢 | iostat + Perfetto | §3.2 |
| binder 排队 | binder stats + Perfetto | §3.3 |
| 频繁 OOM | hprof + smaps + meminfo | §4 / §5 / §6 |
