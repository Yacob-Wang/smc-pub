## 1. 背景与定义：Native Tombstone 在稳定性治理中的位置

### 1.1 什么是 Native Tombstone

**Native Crash**（SIGSEGV / SIGBUS / SIGABRT / SIGFPE 等）发生时，系统生成 **Tombstone 文件**记录崩溃现场信息。

**Tombstone 文件位置**：
- `/data/tombstones/tombstone_XX`（XX = 00-09，循环覆盖）
- tombstone 写入 dropbox（`SYSTEM_TOMBSTONE` 标签）

**Tombstone 价值**：
- 唯一包含崩溃现场寄存器 / 内存映射 / 信号信息的文件
- 比 logcat 更详细（logcat 只有错误消息）
- 比 bugreport 更聚焦（只针对单次崩溃）

### 1.2 Tombstone 16 段结构

```
┌────────────────────────────────────────────────────────────────┐
│ Tombstone 16 段结构（AOSP 14 标准）                              │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. Header（时间 / 进程 / 信号 / 线程）                          │
│  2. Signal（信号编号 + 代码 + 地址）                             │
│  3. Register（崩溃时寄存器值）                                   │
│  4. Backtrace（崩溃时调用栈）                                   │
│  5. Stack（栈内存内容）                                         │
│  6. Memory Map（进程内存映射）                                  │
│  7. Threads（其他线程状态）                                     │
│  8. Open Files（打开的文件）                                    │
│  9. Network Info（网络信息）                                    │
│  10. Build Info（设备 / 系统信息）                              │
│  11. Command Line（启动参数）                                   │
│  12. SELinux Context（SELinux 上下文）                          │
│  13. Kernel Version（内核版本）                                 │
│  14. ABI（架构信息）                                            │
│  15. FORTIFY（编译选项）                                        │
│  16. MTE / Scudo（内存安全扩展）                                │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 2. Native Crash 解读挑战

### 2.1 4 大核心挑战

**挑战 1：寄存器编码**

```
寄存器 r0 = 0x0000000000000000
  ├─ 可能是 NULL（空指针解引用）
  ├─ 可能是未初始化
  ├─ 可能是 API 返回值（特定错误码）
  └─ 需要结合上下文判断

需要 LLM 理解：
- API 调用约定（ARM64 ABI）
- 函数调用栈（如何从 r0 反推到调用方）
- 平台特定的寄存器语义
```

**挑战 2：优化栈（编译优化导致栈帧缺失）**

```
原始栈（debug 编译）：
  #00 com.example.app.NativeHelper.processData (NativeHelper.java:42)
  #01 com.example.app.NativeHelper.crash (NativeHelper.java:88)

优化栈（release 编译，O2 优化）：
  #00 libexample.so (offset 0x123456)
  #01 libexample.so (offset 0x123012)
  #02 libart.so
  #03 boot-framework.oat
  #04 ???
```

**优化栈问题**：
- release 编译栈帧地址是绝对地址，无法直接对应源码
- 需要 unstripped `.so` 文件 + addr2line + 符号表
- 部分栈帧可能完全丢失（inlined 函数）

**解决方案**：
- 保留 unstripped `.so`（debug build）
- 使用 LLDB / ndk-stack 工具
- LLM 解读时附加"建议人工确认"

**挑战 3：异步信号**

```
信号处理流程：

  T0: 异步信号产生（如 SIGSEGV）
    ↓
  T1: 内核投递信号到进程
    ↓
  T2: 信号处理函数（sigaction 注册）执行
    ├─ 打印 Tombstone
    └─ 终止进程

  问题：信号处理函数运行时的栈 ≠ 崩溃时的栈
```

**挑战 4：内核态崩溃**

```
内核态 Native Crash 表现：
  ├─ kernel panic（设备重启）
  ├─ 栈完全丢失（kernel stack）
  └─ 无法用普通 addr2line 解读

需要：
- kernel crash dump
- RAM dump
- 厂商 GKI 内核符号表
```

### 2.2 LLM 解读的局限性

**LLM 不能做的事**：
1. **不能直接反汇编机器码**（需要 native 工具支持）
2. **不能保证 100% 准确率**（LLM 幻觉风险）
3. **不能解读内核态 crash**（缺少训练数据）
4. **不能替代人工对罕见崩溃的深入分析**

**LLM 能做的事**：
1. **快速结构化提取**（16 段 JSON）
2. **基于已知模式推断**（常见崩溃类型）
3. **结合源码上下文**（Function Calling 调用代码搜索）
4. **相似历史 case 检索**（RAG）

## 3. LLM 解读 Tombstone（Prompt + Few-shot + Function Calling）

### 3.1 Tombstone 解析器（端侧预处理）

```python
import re

def parse_tombstone(tombstone_text):
    """Tombstone 结构化解析（16 段）"""
    result = {}
    
    # 1. Header
    result['header'] = {
        'process': extract_field(tombstone_text, r'pid: (\d+), tid: \d+, name: (\S+)'),
        'signal': extract_signal(tombstone_text),
        'fault_addr': extract_fault_addr(tombstone_text),
    }
    
    # 2. Register
    result['registers'] = extract_registers(tombstone_text)
    
    # 3. Backtrace
    result['backtrace'] = extract_backtrace(tombstone_text)
    
    # 4. Memory Map
    result['memory_map'] = extract_memory_map(tombstone_text)
    
    # ... 其他段
    
    return result

def extract_field(text, pattern):
    match = re.search(pattern, text)
    return match.group(1) if match else None

def extract_signal(text):
    match = re.search(r'signal (\d+) \((\w+)\), code (\d+) \(([^)]+)\)', text)
    if match:
        return {
            'number': int(match.group(1)),
            'name': match.group(2),
            'code': int(match.group(3)),
            'code_meaning': match.group(4)
        }
    return None
```

### 3.2 addr2line 反汇编（端侧预处理）

```python
import subprocess

def resolve_addresses(backtrace, so_files):
    """addr2line 反汇编：栈帧地址 → 源码位置"""
    resolved = []
    for frame in backtrace:
        if frame['library'] in so_files:
            cmd = [
                'llvm-addr2line',
                '-e', so_files[frame['library']],
                '-f',  # 函数名
                '-C',  # demangle
                hex(frame['address'])
            ]
            result = subprocess.run(cmd, capture_output=True, text=True)
            function_name, source_location = result.stdout.strip().split('\n')
            frame['function'] = function_name
            frame['source'] = source_location
        resolved.append(frame)
    return resolved
```

**关键**：
- 需要 unstripped `.so`（含调试符号）
- NDK 自带 llvm-addr2line
- 端侧 SDK 集成（首次启动反汇编 + 缓存）

```python
def analyze_native_crash_with_llm(tombstone_json):
    """LLM 解读 Native Crash"""
    prompt = NATIVE_CRASH_PROMPT.format(
        tombstone_header=format_header(tombstone_json['header']),
        signal_info=format_signal(tombstone_json['header']['signal']),
        key_registers=format_registers(tombstone_json['registers']),
        resolved_backtrace=format_backtrace(tombstone_json['backtrace'])

    response = openai.ChatCompletion.create(
        model="gpt-4o",

        response_format={"type": "json_object"},
        temperature=0
    )
    return json.loads(response.choices[0].message.content)
```

### 3.4 Function Calling + 多步解读

**关键场景**：LLM 需要查询代码上下文。

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "search_native_code",
            "description": "在 Native 代码（C/C++）中搜索函数定义和调用",
            "parameters": {
                "type": "object",
                "properties": {
                    "library": {"type": "string"},
                    "function_name": {"type": "string"}
                }
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "check_thread_safety",
            "description": "检查函数是否有锁保护（线程安全分析）",
            "parameters": {
                "type": "object",
                "properties": {
                    "function_name": {"type": "string"}
                }
            }
        }
    }
]

def analyze_with_function_calling(tombstone_json):
    """Function Calling 驱动的 Native Crash 解读"""
    messages = [
        {"role": "user", "content": f"分析这份 Native Crash:\n{json.dumps(tombstone_json)}"}
    ]
    
    for i in range(5):  # 最多 5 轮
        response = openai.ChatCompletion.create(
            model="gpt-4o",
            messages=messages,
            tools=tools
        )
        
        message = response.choices[0].message
        if not message.tool_calls:
            return message.content
        
        messages.append(message)
        for tool_call in message.tool_calls:
            # 执行工具：搜索 Native 代码 / 检查线程安全
            if tool_call.function.name == "search_native_code":
                args = json.loads(tool_call.function.arguments)
                result = search_native_code(args["library"], args["function_name"])
            elif tool_call.function.name == "check_thread_safety":
                args = json.loads(tool_call.function.arguments)
                result = check_thread_safety(args["function_name"])
            
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": json.dumps(result)
            })
```

## 5. 端云协同

### 5.1 端云协同的价值

**挑战**：
- Native Crash 排查需要**大模型推理**（>= 7B 参数）
- 端侧部署大模型成本高（1B+ 模型 ~1GB 内存）
- 纯云端成本高（每次 LLM 调用 $0.01+）

**端云协同方案**：

| 任务 | 端侧 | 云端 |
| :--- | :--- | :--- |
| **Tombstone 解析** | ✅（小模型 / 规则） | — |
| **addr2line 反汇编** | ✅（必须有 .so） | — |
| **常见模式识别** | ✅（小模型） | — |
| **复杂推理（结合代码 / 历史）** | ❌ | ✅（大模型） |
| **多模态融合** | ❌ | ✅（GPT-4o vision） |

### 5.2 端云协同架构

```
┌────────────────────────────────────────────────────────────────┐
```

### 5.3 成本优化

**优化策略**：

| 策略 | 效果 |
| :--- | :--- |
| **端侧兜底常见模式** | 减少 70% 云端调用 |
| **云端调用限流** | 单用户每天最多 5 次云端调用 |
| **批量处理** | 多条 Crash 合并为一次云端调用 |
| **缓存结果** | 相同 Crash 模式直接返回缓存 |

**成本估算**：

- 某 App 日均 10000 条 Native Crash
- 端侧兜底 70% → 3000 条进云端
- 云端 $0.01 / 次 → $30 / 天 = $900 / 月
- 对比纯云端：10000 × $0.01 = $100 / 天 = $3000 / 月
- **节省 70%**

## 9. 总结（架构师视角的 5 条 Takeaway）

1. **Native Tombstone 是稳定性分析的"金矿"**——16 段结构化信息 + 寄存器 + 栈 + 内存映射，比 logcat / bugreport 都详细。**传统 4h 排查时间必须压缩**。
2. **端侧预处理 + 云端深度分析是端云协同最优解**——端侧小模型兜底常见模式（70%），云端大模型处理复杂模式（30%）。**成本节省 70%**。
3. **addr2line + LLM 是黄金组合**——addr2line 把栈帧地址反汇编成源码位置，LLM 基于源码位置推断根因。**纯人工解读 = 4h，组合 = 30min**。
4. **多模态融合是 Native Crash 解读的进阶方向**——trace + log + 截图 + 代码的多模态融合，可还原"从用户操作到崩溃"的完整链路。**这超越传统 addr2line 边界**。
5. **unstripped `.so` 是必备前提**——release 包必须保留 unstripped `.so`（含调试符号），否则 addr2line 失效。**构建流程必须配合**。

**Native Crash AI 解读决策树**：

```
新项目要做 Native Crash AI 解读
  ↓
构建流程支持？
  ├─ 是（保留 unstripped .so） → 可以做
  └─ 否（stripped .so） → 先改构建流程
  ↓
设备性能？
  ├─ 旗舰 → 端侧 Gemini Nano
  ├─ 中端 → 端侧 Qwen2.5-1.5B
  └─ 入门 → 仅端侧规则 + 云端
  ↓
预算？
  ├─ 高 → 云端 LLM 为主
  ├─ 中 → 端云协同（70% 端侧 + 30% 云端）
  └─ 低 → 仅端侧规则
  ↓
多模态需求？
  ├─ 是 → GPT-4o vision + trace + log + 截图
  └─ 否 → 仅文本 LLM
  ↓
历史 case 库？
  ├─ 丰富（> 1000 条） → RAG
  └─ 少 → 无 RAG 模式
```
