# 技术系列文章写作 Prompting 指南

> 给资深架构师看的 AI 提示词模板,用于生成有深度、有广度、有源码依据、有工程基线、可验证的技术系列文章。
> 适用于任何技术模块(内核子系统、Framework 服务、Native 库、协议层等)的系列化学习输出。
> 所有约束以"可治理、可机制化、可追溯"为标准。

---

## 一、核心原则

````
你写的不是科普文,不是面试八股,而是给【资深架构师】看的技术深度文章。

1. 言必有据,码必有源:每个技术论点标注源码文件路径,关键流程贴核心源码片段;
   每篇文章末尾给出"源码路径对账表",逐条标注【已校对 / 待确认】。
2. 自顶向下:每个章节先讲"是什么、为什么需要它",建立上下文后再深入源码。禁止上来就贴代码。
3. 关联实战:每个知识点关联到真实工程问题(Crash/ANR/OOM/性能退化等),
   说清楚"这个机制会在什么场景下咬你一口"。
4. 实战案例收尾:每篇末尾 1-2 个完整线上问题排查案例(现象→分析→根因→修复),
   案例可验证(含 dump/日志片段、版本号、复现步骤)。
5. 内容量级:每篇不少于 300 行(约 8000-15000 字)。
6. 版本基线(多版本矩阵):
   - 应用层 / Framework:以 Android 14(AOSP android-14.0.0_r1)为主线
   - Linux 内核:以 android14-5.10 / android14-5.15 / android15-6.1 / android15-6.6 为基线
   - 涉及 GKI 厂商分支差异时,标注"厂商 GKI"或具体 vendor tag
   - 涉及历史演进或多版本差异时,标注版本范围
7. 工程基线:涉及可调参数(timeout、buffer size、maxevents、优先级等)时,
   必须给出"工程默认值"与"选用准则",说清楚"线上我应该用啥、为啥"。
````

## 二、系列文章设计流程

### 第一步:定义角色与目标

````
我的角色是 Android 稳定性高级工程师。

我要系统学习【填入模块名称,如:Binder / SurfaceFlinger / Linux 调度器 / epoll】。

学习目标不是"了解原理",而是:
- 能在遇到线上问题时,快速定位到该模块的对应子系统和源码文件
- 能理解问题的根因机制
- 能给出治理方案或监控手段
````

### 第二步:让 AI 先输出大纲(不要直接写文章)

````
请按架构师思维链为【模块名称】设计系列文章大纲:

  它是什么?解决什么问题?(定位)
      ↓
  它在系统中处于什么位置?和谁协作?(边界与交互)
      ↓
  它内部是怎么运转的?(核心机制)
      ↓
  它会在什么地方出问题?(风险地图)
      ↓
  出了问题我怎么查?怎么防?(诊断与治理)

大纲要求:
1. 分为 3-5 个篇章(全局观 → 核心机制 → 跨模块交互 → 实战/治理)
2. 每篇文章列出具体章节,说清楚:讲什么内容、对应哪个源码文件/目录(含内核版本基线)、关联什么工程问题
3. 用表格形式呈现每篇文章的章节规划
4. 明确标注与其他系列(如已有的 Input / ART / MM_v2 / IO / Socket / Syscalls)的交叉引用点
5. 明确标注"本篇定位":在本系列中属于什么角色(全局观/核心机制/横切专题/风险/治理),
   强依赖哪几篇已发布文章
````

### 第三步:生成系列 README

大纲审核通过后,生成系列 README(为什么要写 / 系列设计思路 / 章节规划表 / 跨系列引用 / 阅读建议)。模板按需展开,非核心。

### 第四步:逐篇撰写(使用一站式模板)

大纲和 README 审核通过后,用下方**一站式写作模板**逐篇撰写。每次只需替换 `【】` 中的内容。

# 写作标准
## 硬性要求
1. 目标读者:资深架构师,不是初学者。不解释基础概念(如什么是线程、什么是指针),
   解释该模块特有的术语和机制。
2. 每个章节先讲"这个东西是什么、为什么需要它、解决什么问题",然后再深入源码。
3. 涉及源码时:
   - 标注源码文件路径(如 `frameworks/native/services/inputflinger/dispatcher/InputDispatcher.cpp`
     或 `fs/eventpoll.c`),**并标注内核版本基线**
   - 只贴核心逻辑,不要全贴
   - 贴代码前先用自然语言解释这段代码要干什么
   - 贴代码后紧跟"稳定性架构师视角"分析
   - 不确定的路径用 `// 路径待确认` 标注,并在"源码路径对账表"中标记【待确认】
4. 每个技术点关联到实际工程问题(Crash、ANR、OOM、JE、NE、Watchdog、性能退化、兼容性问题),
   说清楚"它会在什么场景下咬你一口"。
5. 涉及量化描述时,给出数量级("ms 级""秒级""占比 X%"),禁止"大约""通常"等模糊用词。
   量化依据填入"量化数据自检表"。
6. 源码版本基线:同核心原则 #6
7. 工程基线要求:涉及可调参数时,给出"工程默认值"与"选用准则"。
8. 文章长度不少于 300 行(约 8000-15000 字)。

## 章节结构

```
┌─ 背景与定义(是什么、为什么需要它、解决什么问题)
│     不要假设读者已经知道,先用 1-2 段建立上下文
│
├─ 架构与交互(在系统中的位置、上下游依赖、数据流向)
│     用 ASCII Art 画架构图或层级图,说清楚边界
│
├─ 核心机制与源码(关键数据结构、核心流程、源码走读)
│     每段源码标注 AOSP/内核 源码路径 + 内核版本基线
│     贴代码前用自然语言解释这段代码要干什么
│
├─ 风险地图(这个机制会在哪些场景下出问题)
│     列出具体的异常类型、日志特征、触发条件
│
├─ 实战案例(1-2 个线上问题的完整排查过程)
│     **案例 5 件套**(每条案例都必填,缺一不可):
│     1. **环境**:Android 版本 / 内核版本 / 设备 / 复现步骤
│     2. **现象**:关键 logcat / dmesg / systrace 片段
│     3. **分析思路**:排查路径(从现象到根因的推理链)
│     4. **根因**:具体到源码文件 + 方法 + 行
│     5. **修复**:修复 commit 或配置 diff + 修复原理
│     案例标注类型:"典型模式" 或 "真实案例(来源:...)"
│
├─ 总结(3-5 条架构师视角的关键 Takeaway)
│
├─ 附录 A:核心源码路径索引(表格)
├─ 附录 B:源码路径对账表(强制)
├─ 附录 C:量化数据自检表(强制)
├─ 附录 D:工程基线表(若涉及可调参数,4 列强制)
│     **4 列定义**(砍 1 列都属违反硬性要求):
│     | 参数 | 典型默认 | 选用准则 | 踩坑提醒 |
│     详见 §5 反例 #7(反例库指向此处)
└─ 篇尾衔接:用 1-2 句说明下一篇将深入什么内容
```

## 图表格式
- 架构图/层级图/流程图:统一使用 ASCII Art(纯文本方框图)
- 对比信息:Markdown 表格
- 时序说明:缩进文本 + 箭头符号

## 图表密度规则
- **平均每 1500 字 1 张图**
- **单篇 4-6 张图(不含源码中的小型 ASCII 示意)**
- 上限 6,下限 4,表格不计入图数
- **例外**(破例适用,按 §8 处理):横切专题型可放宽到 3 张;演进型可放宽到 3 张 + 1-2 张对比表

## 跨模块引用规范
- 涉及本系列其他篇章时,用 `[文章标题](文件名.md)` 引用,不重复展开
- 涉及其他系列时,用 `[系列名-文章标题](相对路径)` 引用,只用 1-2 句概述核心结论
- 涉及其他系列未覆盖的内容时,简要说明核心逻辑(3-5 句),不要深入展开
- 涉及引用时,必须在文章开头"本篇定位"中显式声明

## 禁止事项
1. 禁止挖坑不填("我们将在后续文章详细讲"→当场讲清或显式指向)
2. 禁止数据堆砌(每个数字后必须有"所以呢")
3. 禁止 AI 自嗨("非常精妙""体现了……深度融合"→删)
4. 禁止模糊量化("通常""大约"→给具体数字 + 来源)
5. 禁止跨篇重复(已在其他系列讲过的内容,本系列只引用不展开)

### 5.2 错例全文(❌ 警示)

#### 反例 #1:纯科普模式
```
❌ "ART 是 Android 的运行时环境,负责执行 Java 代码。它支持 JIT 和 AOT 两种编译模式..."
```
**问题**:没有源码、没有数据结构、没有关联问题,停留在百科词条级别。**防范**:硬性要求每章三件套(定义/源码/风险)。

#### 反例 #2:代码堆砌模式
```
❌ "## 1. 源码结构
   源码路径:art/runtime/runtime.cc
   bool Runtime::Init(...) {
     heap_ = new gc::Heap(...);
   }"
```
**问题**:上来就贴源码,没有"为什么要看这段代码""它和我的日常工作有什么关系"。**防范**:见 §3"贴代码前自然语言+贴代码后视角"。

#### 反例 #3:源码路径幻觉
```
❌ 源码路径:frameworks/base/services/core/java/com/android/server/wm/WindowManagerPolicy.java
```
**问题**:这个文件在 Android 10+ 已被移除/重构,AI 基于旧版本训练数据生成了不存在的路径。**防范**:附录 B 全量对账,逐条在 cs.android.com / Elixir 上验证。

#### 反例 #4:版本混用
```
❌ "InputDispatcher 通过 InputMonitor.updateInputWindowsLw() 同步窗口信息..."
   (后面又写)
   "InputDispatcher 通过 InputWindowHandle.setInputWindows() 更新..."
```
**问题**:第一种写法是 Android 11 之前的,第二种是 Android 12+ 的,混在一起让读者困惑。**防范**:在 §3 模板中指定多版本基线矩阵,版本差异用对比表。

#### 反例 #5:模糊量化
```
❌ "InputDispatcher 通常在几毫秒内完成事件分发"
❌ "大约 60% 的 ANR 与 Input 相关"
```
**问题**:"通常""大约"没有依据。**防范**:要么给源码中的常量值,要么给具体统计数据(见 §3 硬性要求 #5)。

#### 反例 #6:图表密度失控
```
❌ 全文 8000 字只有 1 张架构图 → 信息密度跟不上
❌ 全文 8000 字塞了 12 张图 → 图淹没文字
```
**防范**:4-6 张核心图,平均 1500 字/张(见 §3 图表密度规则)。

#### 反例 #7:工程参数无基线
```
❌ "epoll_wait 第二个参数 maxevents 应该设置一个合适的值"
```
**问题**:读者不知道"合适"是多大。**防范**:见 §3 附录 D 4 列定义(参数 / 典型默认 / 选用准则 / 踩坑提醒)。

#### 反例 #8:案例不可验证
```
❌ "曾经遇到一个线上问题,InputDispatcher 阻塞导致 ANR"
```
**问题**:没有版本、没有 logcat、无法复现。**防范**:见 §3 实战案例 5 件套(环境 / 现象 / 分析思路 / 根因 / 修复)。

#### 反例 #9:跨篇重复造内容
```
❌ 在 epoll 文章中详细讲了"什么是 fd、什么是 poll/select",但同系列 socket-04 已经详细讲过
```
**防范**:本篇定位段强制声明"不重复内容",跨篇用 Markdown 引用。

#### 反例 #10:挖坑不填
```
❌ "关于 epoll 在 Android 中的实际应用,限于篇幅我们将在后续文章中详细讨论。"
```
**问题**:架构师读者最关心"实际应用","后续"是最严重的反例。**防范**:要么当场讲清,要么明确写"本篇不涉及,详见 [系列名-文章](path)"。

### 5.3 关键警示(#11 #12 是核心洞察,完整保留)

#### 反例 #11:数据堆砌模式

```
❌ "epoll_wait 第二个参数 maxevents 默认 64;
    InputDispatcher timeout 默认 5000ms;
    binder thread pool 默认 15;
    Looper slow dispatch 阈值 2000ms;
    ……(继续列 20 条)"
```

**问题**:列了一堆数字,但读者看完只知道"有这么些参数",不知道"所以呢"。

**正确姿势**:每个数据后必须有"**所以呢**"——
```
✅ "epoll_wait maxevents 默认 64(AOSP 14 frameworks/native/services/inputflinger/EventHub.cpp),
    原因:单次事件处理平均 1-3ms,64 个事件在 200ms 内能消化完;
    线上建议:高吞吐场景(InputDispatcher)保持 64,反过来低频唤醒(alarm timer)可降到 16-32"
```

**判定标准**:如果把数字删掉,文章的"洞察"还在吗?不在 → 数据堆砌。

#### 反例 #12:AI 自嗨模式

```
❌ "ART 的 GC 实现非常精妙,它使用了分代回收算法,结合了标记-清除和标记-整理的混合策略,
    通过 Card Table 维护跨代引用,通过 Remembered Set 跟踪老年代对新生代的引用,
    整体设计体现了编译原理、操作系统、硬件体系结构的深度融合……"
```

**问题**:通篇在描述"这个东西多精妙",但架构师读者关心的是"**我能拿来干嘛**"——
"那这意味着我在线上遇到 OOM 时该怎么办?""什么样的代码会触发它?"

**正确姿势**:每个技术机制后必须有"**对读者有什么用**"——
```
✅ "ART 的 GC 使用分代回收(源码 art/runtime/gc/heap.cc):
    新生代用 Copying(快),老年代用 Mark-Compact(省内存)
    → 所以:短命对象(如 Activity 转屏创建的临时对象)几乎无成本
    → 所以:长命大对象(如 Bitmap 缓存)尽量复用,避免进老年代
    → 所以:内存泄漏检测主要盯老年代增长速度,不是新生代
    线上场景:某 App 频繁 OOM → 看 hprof 老年代占比 → 90% 来自某静态 Map 缓存"
```

**判定标准**:把"非常精妙""体现了……深度融合"这种话删掉,文章还剩多少干货?剩得少 → AI 自嗨。

## 六、校准与迭代机制

> 架构师交付的是**机制**(可复用、可治理),不是"一篇文章"。
> 一篇文章质量从 70 分到 90 分靠的是**校准机制设计得更合理**。

### 6.1 AI 输出的三个天然倾向

1. **平滑化**:倾向于写"看起来都对"的内容,但缺乏锐度
2. **冗余化**:同一个意思换 3 种说法重复 3 次
3. **自信化**:即使不确定也写得很笃定("一定是……")

### 6.2 校准的三轮节奏

| 轮次 | 目标 | 触发 | 动作 |
|------|------|------|------|
| **第 1 轮 · 结构** | 让骨架立住 | 26 项清单扫描后 | 缺本篇定位 / 总结 / 附录 → 补;章节顺序乱 → 调整;跨篇重复 → 改引用 |
| **第 2 轮 · 硬伤** | 经得起查 | 路径对账 / 量化自检 / API 版本核对后 | 路径幻觉 → 重查或删;量化没依据 → 补来源;API 签名不对 → 重读源码 |
| **第 3 轮 · 锐度** | 读完有收获 | 通读全文,凭"读起来是否废话"判断 | 数据堆砌 → 加"所以呢";AI 自嗨 → 删"非常精妙";模糊量化 → 改具体数字;冗余 → 合并 |

**铁律**:每轮校准**只动一类问题**,不交叉。否则丢失已校准的进度。

### 6.3 校准决策日志

每轮校准后,在文章开头(本篇定位段下方)记录决策日志(用 `<!-- AUTHOR_ONLY -->` marker 包裹,见 §9):

```markdown
| 轮次 | 类别 | 决策 | 理由 | 影响范围 |
|------|------|------|------|----------|
| 1    | 结构 | 附录 D 改为按需 | §8 破例:本篇无参数表 | 仅本篇 |
| 2    | 硬伤 | 删除路径 fs/foo/bar.c | 校对发现该文件不存在 | 全文 3 处 |
| 3    | 锐度 | 删"非常精妙"段落 | 反例 #12 | 全文 2 处 |
```

**作用**:6 个月后回头复盘、跨系列共享经验、挡住 AI 之后"建议"把已校准段落改回去。

### 6.4 校准的边际效用与停止准则

| 轮次 | 典型分 | 边际收益 | 何时停 |
|------|--------|---------|--------|
| 第 1 稿 | 65-75 | —— | —— |
| 校准 1 轮 | 75-82 | +7-10 | 26 项清单基本通过 |
| 校准 2 轮 | 82-88 | +5-7 | 附录 B/C 全量核对完毕 |
| 校准 3 轮 | 88-92 | +3-4 | 通读没有废话 |
| 校准 4 轮 | 边际递减 | <+2 | 改的比改坏的多 |

**停止信号**:当一轮校准里,AI 的修改有 ≥30% 引发新问题,**立即停止校准**——这是 AI 已经"理解饱和"的信号,再压会写崩。改为人工润色。

### 6.5 校准时的提示词模板(定向修正,避免整篇重写)

````markdown
# 校准第 N 轮(结构 / 硬伤 / 锐度)

只修改以下指出的问题,**其他段落完全保留**:

## 必须修改的(按优先级)
1. 【第 X 章 第 Y 节】"那段话" → 改为"……"(理由:反例 #5 模糊量化)
2. 【附录 B 第 Z 行】路径 fs/eventpoll.c → 实际不存在,请重新确认或标注"路径待确认"

## 可以改进的(不强制)
- 【第 3 章】"非常精妙"这种话可以删

## 绝对不要动
- 第 1-2 章(已经校准过)
- 附录 A 索引(已经核对)

修改后请在末尾输出"本轮修改清单",并更新文首"校准决策日志"。
````

## 七、跨系列一致性治理

> 系列数 ≥ 3(如 ART + Input + MM_v2)时,"术语漂移""基线漂移""引用断裂"会指数级恶化。
> 10+ 系列规模下,**一致性治理是架构师的核心责任**。

### 7.1 治理对象与手段

| 治理对象 | 漂移风险 | 治理手段 |
|---------|---------|---------|
| **术语** | 同一概念在不同系列用不同名字 | 全局术语表 |
| **版本基线** | 新 AOSP 版本出来,旧文章基线过时 | 基线升级流程 |
| **跨篇引用** | 文章删除/移动,引用断裂 | 引用矩阵治理 |
| **架构图风格** | 不同系列画的架构图风格不一致 | 模板规范 |
| **案例编号** | 案例无法跨系列检索 | 案例索引 |

### 7.2 全局术语表(强制维护)

`Reference/术语表.md` 维护一份全局术语,所有系列必须遵守:

```markdown
# 全局术语表
| 中文 | 英文 | 别名(禁止使用) | 首次定义于 | 备注 |
|------|------|----------------|-----------|------|
| 主线程 | main thread | UI 线程、UI thread | ART 01 | 明确不用"UI 线程"避免和 UI 渲染混淆 |
| 焦点窗口 | focused window | 当前焦点、front window | Input 05 | |
| 分代回收 | generational GC | 分代收集 | ART 05 | 统一用"回收" |
```

**使用规则**:任何系列新引入术语前先查术语表;已有术语**必须使用**,禁止"别名"。

### 7.3 版本基线升级流程

```
步骤 1:基线评估 - 新版本相对旧版本的关键 API 变化(cs.android.com diff)
步骤 2:批次升级 - 优先升级引用次数最多的文章
步骤 3:跨系列一致性回归 - 升级后跑一遍 26 项清单
步骤 4:基线切换公告 - 在 README 顶部加"本系列基线:自 YYYY-MM-DD 起"
```

**反模式**:一次性把所有系列基线全切——会导致"50% 文章是 A14,50% 是 A15"。

### 7.4 引用矩阵治理

每篇 README 维护一份**本系列 → 跨系列**的引用矩阵,字段:`本系列文章 | 引用其他系列 | 引用章节 | 引用原因 | 链接有效性最后核查`。

**治理动作**:
- 每月跑一次"链接有效性核查"(点击所有引用链接,确认未 404)
- 目标链接文章删除/移动前,先在引用矩阵里找到所有引用方,逐个更新
- 用脚本辅助(`grep -r "ART 05" --include="*.md"` 找到所有引用方)

### 7.5 架构图风格规范

- 方向:全仓库只用左→右 或 上→下
- 边框:`┌─┐│└─┘`(不用 `+-|+`)
- 颜色:Markdown 不支持,统一用文字标注层级(`[FW]`、`[Native]`、`[Kernel]`)

### 7.6 案例索引

按主题分类的全局表:`案例 ID | 主题 | 系列 | 文章 | 环境 | 案例类型`。同一症状(如 OOM)能在多个系列里找到不同视角的案例,避免同一案例被多个系列各自构造。

## 八、破例准则

> 模板不是"必须逐字遵循的教条",但**破例必须有决策记录**。

### 8.1 合法破例

| 场景 | 允许的破例 | 决策记录位置 |
|------|-----------|----------------|
| **横切专题型**(epoll、信号横切) | 图表数 → 3 张;案例 1 个即可;章节合并 | README 的"系列设计思路"段 |
| **演进型**(ART vs JVM、Kernel 版本演进) | 图表数 3 张 + 1-2 张对比表;源码片段可精简 | README 的"系列设计思路"段 |
| **总览篇**(系列第 1 篇) | 风险地图可只列 3 类常见问题;实战案例可省略 | 文章开头"本篇定位"段 |
| **诊断工具型**(trace 工具、内存诊断) | 章节结构重排为"工具使用 → 工具源码 → 实战" | README 的"系列设计思路"段 |
| **小众模块**(kworker、PSI) | 实战案例可标注"行业案例"代替"线上案例" | 文章开头"本篇定位"段 |

### 8.2 破例决策记录格式

```markdown
## 破例决策记录
| 破例项 | 破例内容 | 破例理由 | 影响范围 | 是否传染 |
|--------|---------|---------|---------|--------|
| 图表密度 | 3 张图(规则要求 4-6 张) | 横切专题型,机制+案例一篇覆盖 | 仅本篇 | 否 |
| 实战案例 | 1 个行业案例(规则要求 1-2 个真实案例) | 该模块线上案例数据不足 | 仅本篇 | 否 |
```

**字段说明**:
- **影响范围**:破例只影响本篇,还是整个系列都会这么写?
- **是否传染**:是否会成为后续文章的"先例",导致模板整体松弛?
  - 若 `是` → 必须升级到 README 的"系列设计思路"段登记
  - 若 `否` → 写在本篇即可

### 8.3 不允许的破例

| 项 | 不允许原因 |
|----|----------|
| **本篇定位段** | 没有定位就不知道这文章在整个系列里的角色,跨篇引用一定乱 |
| **附录 B 路径对账** | AI 幻觉的最大风险点,省了就是赌运气 |
| **数据后必须有洞察**("所以呢"原则) | 反例 #11 的核心防御 |
| **多版本基线矩阵** | 没有基线声明,读者无法判断"我这版本是不是这样" |
| **跨篇引用用 Markdown 链接** | 不引就是抄,破坏可治理性 |

### 8.4 破例传染性治理

破例出现"破例其实适用于一类文章"时,**必须升级为模板规则**:

```
本篇破例:横切 epoll 文章用 3 张图
  ↓
3 篇横切文章都用了 3 张图
  ↓
发现 §4 质量清单 #26 写"4-6 张"是个一刀切
  ↓
升级为:横切专题型 → 3 张图;普通型 → 4-6 张图
  ↓
更新 §4 质量清单,更新 §8.1 破例表
```

**反模式**:破例→传染→破例→传染——最后模板完全失效。

## 九、读者视图规范(双层结构)

> 源文件必须同时服务两群人:作者/AI(写稿时需要 5 段提示)和读者(只看到基线 blockquote + 正文)。
> 解决 Symptom / Forensics 出现的"作者前言淹没读者"问题。

### 9.1 双层结构

| 视图 | 看到什么 | 用途 |
|------|---------|------|
| **作者视图**(源文件 / 编辑器 / 本地预览) | 完整内容(含 5 段作者前言) | AI 写稿 / 作者自审 / 决策回溯 |
| **读者视图**(公开站 / 公开仓) | 仅基线 blockquote + 正文 | 真实读者阅读,不被作者前言淹没 |

源文件**必须**包含完整作者前言(便于 AI 写稿),但**必须**用 marker 包裹(确保可剥离)。

### 9.2 marker 格式

5 段作者前言(本篇定位 / 校准决策日志 / 角色设定 / 上下文 / 写作标准)用 HTML 注释 marker 包裹:

```
<!-- AUTHOR_ONLY:START -->
# 本篇定位
- 本篇系列角色:...
- 强依赖:...
...

# 校准决策日志
| 轮次 | 类别 | 决策 | 理由 | 影响范围 |
|------|------|------|------|----------|
...

# 角色设定
我是一名 Android 稳定性架构师,正在系统学习...

# 上下文
- 上一篇:...
- 下一篇:...

# 写作标准
- 本指南规范
...
<!-- AUTHOR_ONLY:END -->
```

**为什么用 HTML 注释**:
1. **零侵入** — 所有 Markdown 渲染器直接忽略,vscode/Typora/git diff 全无副作用
2. **行级粒度** — 比 frontmatter 灵活,可以包多段、部分剥
3. **AI 友好** — LLM 写稿时看见 marker 不会跳过内容(它读全文)

### 9.3 剥离范围(明确"剥什么 / 留什么")

| 段 | 是否剥 | 理由 |
|----|-------|------|
| 标题 | ❌ 不剥 | 读者需要 |
| 顶部 blockquote(基线 / 本篇角色 / 强依赖 / 承接自 / 衔接去) | ❌ 不剥 | 读者元信息 |
| `# 本篇定位` | ✅ **剥** | 已在 blockquote 折合,正文不需要 |
| `# 校准决策日志` | ✅ **剥** | 纯作者元信息,校准过程不是读者关心的 |
| `# 角色设定` | ✅ **剥** | 给 AI 的提示词,读者不需要 |
| `# 上下文` | ✅ **剥** | 给作者避免重复的指令,不是读者元信息 |
| `# 写作标准` | ✅ **剥** | 规范要点(已在 §3 模板),重复 |
| `## 破例决策记录` | 🟡 **保留(可读)** | 读者需要知道"本篇为什么这样写" |
| `## 一、背景与定义` 之后正文 | ❌ 不剥 | 文章主体 |

### 9.4 构建 hook(mkdocs)

```python
# scripts/mkdocs_strip_author_meta.py
import re
from mkdocs.plugins import get_plugin_logger

log = get_plugin_logger(__name__)
PATTERN = re.compile(
    r'<!--\s*AUTHOR_ONLY:START\s*-->.*?<!--\s*AUTHOR_ONLY:END\s*-->\n?',
    re.DOTALL
)

def on_page_markdown(markdown, page, config, files):
    cleaned = PATTERN.sub('', markdown)
    if cleaned != markdown:
        log.info(f"stripped AUTHOR_ONLY from {page.file.src_path}")
    return cleaned
```

注册:`mkdocs.yml` 的 `hooks` 段加 `- scripts/mkdocs_strip_author_meta.py`。
