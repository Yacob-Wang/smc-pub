# 1. 背景与定义

## 1.1 REBOOT 的本质：**结果态** —— 不是独立的触发症状

> **一句话定义**：REBOOT = 非预期的进程或系统重启。**REBOOT 不是一种症状，而是一种结果**——任何上层症状（SWT / KE / NE / 异常累积）都可能演变为 REBOOT。

**关键洞察**：
- **REBOOT 是结果态，不是原因态**（和 ANR / JE / NE 不同）
- 所有 REBOOT 都能**倒推**到其他 6 类之一（SWT / KE / NE / 异常累积 / 系统级）
- 4 类重启源（按严重性递增）：
  - **App 进程重启**（crash 后的 Zygote fork）
  - **SystemServer 重启**（杀 SystemServer）
  - **Zygote 重启**（Zygote 死 → 整机重启）
  - **整机重启**（Kernel panic / reboot 命令）
- **cascade 链路**：REBOOT 是所有严重症状的**最终态**（S00 §2.3 已讲）

> **所以呢**：REBOOT 排查 = **重启溯源**（last_kmsg / pstore / dropbox / bootstat）——**找到上一级症状，倒推根因**。

## 1.2 REBOOT 触发的代价

| 代价 | 严重性 | 量化 |
|:-----|:-------|:-----|
| **L1 极强**：整机重启 | 极强 | 整机不可用 30-60s |
| **L2 强**：SystemServer 重启 | 强 | Zygote 重新拉起 + SystemServer 启动 = 10-20s |
| **L3 中**：App 反复重启 | 中 | 用户感知"App 又被重新打开了" |
| **L4 弱**：bootstat 累积 | 弱 | 数据分析价值 |

> **所以呢**：REBOOT 的代价按"层"递增——App 重启 < SystemServer 重启 < Zygote 重启 < 整机重启。**架构师必须监控所有 4 层**。

## 1.3 排查 REBOOT 的 3 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "REBOOT 是用户手机问题" | 可能是 **cascade 链路**（SWT → 杀 SystemServer → 整机重启）| 用 last_kmsg / pstore 溯源 |
| "App 重启不算问题" | **App 反复重启 = 用户体验灾难** | 监控 bootstat + dropbox(APP_CRASH) |
| "bootstat 不准" | bootstat 是**官方统计源** | 配合 pstore + last_kmsg 交叉验证 |

> **所以呢**：REBOOT 排查 = **重启溯源**（不只看现象，要追到根因）。

# 2. 边界声明

## 2.1 REBOOT vs 其他 6 类（**结果态 vs 原因态**）

| 类别 | 角色 | 触发源 | 关键 logcat |
|:-----|:-----|:-------|:------------|
| **ANR** | 原因态 | AMS 检测 App 主线程阻塞 | `am_anr` |
| **JE** | 原因态 | Throwable 未 catch | `AndroidRuntime` |
| **NE** | 原因态 | Kernel 投递致命信号 | `DEBUG` |
| **SWT** | 原因态 | Watchdog 检测 SystemServer 卡死 | `Watchdog ... KILLING` |
| **HANG** | 原因态 | 未被任何机制捕获 | 无统一关键字 |
| **KE** | 原因态 | Kernel 自身 panic / oops / hung_task | `Kernel panic` / `Oops` |
| **REBOOT** | **结果态** | **任一上层症状严重时** | `bootstat` / `last_kmsg` |

> **架构师防混淆**：
> - **REBOOT 是结果态**——**不是独立原因**
> - **所有 REBOOT 都能倒推**到其他 6 类之一
> - **排查 REBOOT = 倒推**到上层症状 + 找根因

## 2.2 REBOOT 的 4 类源（按严重性递增）

| 类别 | 严重性 | 触发条件 | 恢复时间 | 数据丢失风险 |
|:-----|:-------|:---------|:---------|:------------|
| **App 进程重启** | 🟡 中 | App crash 后 Zygote fork | 1-3s | 低 |
| **SystemServer 重启** | 🟠 重 | Watchdog 杀 SystemServer | 10-20s | 中 |
| **Zygote 重启** | 🔴 极重 | Zygote 死 → 整机重启 | 30-60s | 高 |
| **整机重启** | 🔴 极重 | Kernel panic / reboot 命令 | 30-60s | 高 |

> **架构师视角**：
> - **App 重启最常见**（用户级故障）
> - **SystemServer 重启严重**（整机级）
> - **Zygote 重启和整机重启几乎等价**（用户级不可恢复）

## 2.3 REBOOT 边界决策表

| 看到症状 | 关键证据 | 分类 | 跳到 |
|:---------|:---------|:-----|:-----|
| App 反复重启 | bootstat 异常 + dropbox(APP_CRASH) | **App 重启** | §3.1 |
| logcat `Killing system server` + Zygote 重启 | last_kmsg + pstore | **SystemServer 重启** | §3.2 |
| logcat `Zygote died` + 整机重启 | bootstat + pstore | **Zygote 重启** | §3.3 |
| 整机重启 + last_kmsg `Kernel panic` | pstore | **整机重启（KE 触发）** | §3.4 / S07 |
| 整机重启 + last_kmsg `Watchdog ... KILLING` | pstore | **整机重启（SWT 触发）** | §3.4 / S04 |

# 3. 核心机制与源码（5 个子节深挖）

## 3.1 App 进程重启（最高频 · 用户级）

### 3.1.1 触发链

```
App 触发 crash（JE / NE）
  ↓
AMS.appDiedLocked 接收进程死亡通知
  ↓
AMS 检查是否在 mPersistentStartingProcesses（持久进程）
  ├─ 否 → 进程被 Zygote 清理（不重启）
  └─ 是 → Zygote fork 新进程（重启）
  ↓
App 重新启动

图 3.1.1：App 进程重启触发链
```

### 3.1.2 源码（ActivityManagerService.java）

```java
// frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java
// 路径：AOSP 17.0.0_r1
// 关键：appDiedLocked() - 进程死亡处理

private final void appDiedLocked(ProcessRecord app, ...) {
    // 1. 清理进程资源
    handleAppDiedLocked(app, ...);
    
    // 2. 检查是否需要重启
    if (app.isPersistent()) {
        // 持久进程（如 system_app）→ Zygote fork
        addProcessNameLocked(app);
        mProcessList.startProcessLocked(app, ...);
    } else {
        // 普通进程 → 不重启
    }
}
```

**架构师视角**：
- **持久进程**（manifest `android:persistent="true"`）会自动重启
- **普通进程**（用户 App）通常**不自动重启**——除非有 Service / Provider 绑定
- **App 重启通常** = 用户主动开 + 普通进程被 Zygote 重新 fork
- **bootstat 统计** 每次启动（包括用户主动和异常重启）

## 3.2 SystemServer 重启（系统级）

### 3.2.1 触发链

```
Watchdog 杀 SystemServer（SWT 触发）
  ↓
system_server 进程被 SIGKILL
  ↓
init 检测到 system_server 死 → 重启
  ↓
Zygote 重新 fork system_server
  ↓
SystemServer.main() 重新启动（耗时 10-20s）
  ↓
所有 system_server 管理的服务重启

图 3.2.1：SystemServer 重启触发链
```

### 3.2.2 源码（init.rc + ActivityManagerService.java）

```rc
# system/core/rootdir/init.rc
# 路径：AOSP 17.0.0_r1
# 关键：system_server 死亡后重启

service system_server /system/bin/app_process -Xzygote ...
    class core
    user root
    group root
    # **关键**：onrestart 重启所有依赖服务
    onrestart restart zygote
    onrestart restart media
    onrestart restart surfaceflinger
    onrestart restart inputflinger
    ...
```

**架构师视角**：
- **system_server 死亡 = init 主动重启**（AOSP init.rc 配置）
- **重启耗时 10-20s**（SystemServer.main() 启动慢）
- **期间整机不可用**——所有系统服务都不可用
- **Zygote 不重启**（仅 system_server 重启）——App 进程不受影响（如果有持久进程会重新绑定）

## 3.3 Zygote 重启（整机级 · 极重）

### 3.3.1 触发链

```
Zygote 进程死（罕见但极重）
  ├─ Zygote 自身 bug
  ├─ Zygote 资源耗尽
  └─ init 检测到 Zygote 死 → 整机重启
  ↓
init 触发 kernel.reboot
  ↓
整机重启（30-60s）

图 3.3.1：Zygote 重启触发链
```

### 3.3.2 源码（init.rc）

```rc
# system/core/rootdir/init.rc
# 关键：zygote 死亡触发整机重启

service zygote /system/bin/app_process -Xzygote /system/bin --zygote ...
    class core
    user root
    group root
    # **关键**：zygote 死亡 = 整机不可用
    # 默认 init 会重启 zygote（但 AOSP 中 zygote 死通常伴随整机重启）
```

**架构师视角**：
- **Zygote 死 = 整机重启**（AOSP 默认行为）
- 极罕见（Zygote 是 Android 启动期的核心，稳定性极高）
- **Zygote 死通常伴随 kernel panic**（实际就是 REBOOT 由 KE 触发）

## 3.4 整机重启（Kernel 级 · 极重）

### 3.4.1 触发链

```
Kernel panic 触发（kernel/panic.c）
  ↓
emergency_restart() 调用
  ↓
kernel 同步 + 重新启动
  ↓
整机重启（30-60s）
  ↓
开机后：last_kmsg / pstore 保留重启前 kernel log

图 3.4.1：整机重启触发链
```

### 3.4.2 源码（kernel/reboot.c + kernel/panic.c）

```c
// kernel/reboot.c
// 路径：K 6.18
// 关键：emergency_restart()

void emergency_restart(void) {
    // 1. 禁用中断
    // 2. 调用架构特定的重启代码
    machine_emergency_restart();
}
```

```c
// kernel/panic.c
// 关键：panic() 调用 emergency_restart

void panic(const char *fmt, ...) {
    pr_emerg("Kernel panic - not syncing: %s\n", ...);
    emergency_sync();
    emergency_restart();  // 整机重启
}
```

**架构师视角**：
- 整机重启 = **kernel 主动调用 emergency_restart()**
- **last_kmsg 保留重启前 kernel log**（如果开启）
- **pstore 持久化**（如果开启）
- **bootstat 统计** 此次重启（写入 data/system/bootstat）

## 3.5 cascade 链路（**与 S00 §2.3 呼应**）

### 3.5.1 触发链

```
单一根因 → 多级 cascade → REBOOT

最常见 cascade 模式：
  Binder 死锁（5s）
    ↓ App 主线程阻塞 → ANR 5s 触发
    ↓ AMS binder call 阻塞 → SystemServer 主线程阻塞
    ↓ Watchdog 30s 触发 → SWT
    ↓ 杀 SystemServer
    ↓ SystemServer 重启（init.rc）
    ↓ 期间 SystemServer 不可用
    ↓ 整机重启（如果 init 检测到异常）

次常见 cascade 模式：
  Camera HAL IO 卡死（5s+）
    ↓ CameraService 阻塞
    ↓ App 主线程阻塞 → ANR
    ↓ SystemServer 卡死 → SWT
    ↓ 整机重启
```

**架构师视角**：
- **cascade 是 REBOOT 的常见根因**（占 50-60% 的 REBOOT）
- **逆向排查** REBOOT 时必须**找根因**（不只是看现象）
- **主动监控**喂狗链路（input / VSYNC / binder）+ ANR + SWT

### 3.5.2 cascade 链路阻断

| 阻断点 | 阻断措施 | 实施位置 |
|:-------|:---------|:---------|
| **主线程阻塞** | 主线程 P95 latency 监控 | App 端 |
| **binder call 阻塞** | binder call timeout | App 端 + Framework |
| **SystemServer 阻塞** | 喂狗链路监控 | Framework |
| **Kernel panic** | panic_on_oops + pstore | Kernel |

> **架构师视角**：
> - **cascade 链路每个节点都可阻断**——**架构师必须在每个节点都加防护**
> - **只防一处 = 形同虚设**（cascade 会绕过）

## 3.6 AOSP 17 关键变化

- **bootstat 增强**：AOSP 17 增加更多重启源分类
- _前瞻_：**pstore 与 K 6.18 集成增强**（可能）

# 4. 风险地图

## 4.1 REBOOT 的高频触发场景

| 场景 | 占比 | 根因 |
|:-----|:-----|:-----|
| **App 反复重启** | 50-60% | App crash 后被 Zygote 重新 fork |
| **SystemServer 重启** | 20-30% | SWT 触发杀 SystemServer |
| **Zygote 重启** | 5-10% | 罕见（Zygote 死通常伴随整机重启）|
| **整机重启** | 10-15% | Kernel panic / reboot 命令 |

> **所以呢**：**App 重启最常见**（用户级），但**整机重启最严重**（系统级）。

## 4.2 logcat 关键字段

| 字段 | 含义 |
|:-----|:-----|
| `Killing system server` | SystemServer 重启 |
| `Zygote died` | Zygote 重启（极罕见）|
| `emergency_restart` | 整机重启（kernel）|
| `Kernel panic - not syncing` | panic 触发整机重启 |
| `Watchdog ... Rebooting device` | SWT 触发整机重启 |

## 4.3 dump 文件分布

| 文件 | 路径 | 大小 | 保留 |
|:-----|:-----|:-----|:-----|
| **last_kmsg** | `/proc/last_kmsg` | 几 MB | **重启保留**（需编译时开）|
| **pstore** | `/sys/fs/pstore/` | 厂商定制 | **持久化** |
| **dropbox** | `/data/system/dropbox/` | 几 KB | 30 天 |
| **bootstat** | `/data/system/bootstat/` | 几 KB | 永久 |

> **所以呢**：**last_kmsg + pstore + dropbox + bootstat** = REBOOT 排查 4 件套。

# 5. 治理（**重启溯源是 REBOOT 治理的核心**）

## 5.1 重启溯源 4 步法（**架构师必修**）

| 步骤 | 关键 | 工具 |
|:-----|:-----|:-----|
| **第 1 步**：识别重启类型 | bootstat / dropbox | `adb shell dumpsys bootstat` |
| **第 2 步**：查 last_kmsg | 重启前 kernel log | `adb shell cat /proc/last_kmsg` |
| **第 3 步**：查 pstore | kernel 持久化 log | `adb shell cat /sys/fs/pstore/dmesg-ramoops-*` |
| **第 4 步**：倒推根因 | last_kmsg 找 panic / Watchdog / hung_task | 关键字搜索 |

> **所以呢**：**重启后第一时间** last_kmsg / pstore 抓取——**dmesg 重启丢失**。

## 5.2 dump 取证

**取证步骤**（重启后）：
1. **adb shell cat /proc/last_kmsg > last_kmsg.log** ← **必做**（重启前 kernel log）
2. **adb shell cat /sys/fs/pstore/dmesg-ramoops-0 > pstore.log** ← **必做**（持久化）
3. **adb shell dumpsys dropbox --print | grep -E "ANR|CRASH|WATCHDOG"** ← dropbox 备份
4. **adb shell dumpsys bootstat** ← 重启统计
5. **adb bugreport** ← 全量 dump

## 5.3 修复模式（4 类各 1 个）

| 类别 | 典型反模式 | 修复模式 |
|:-----|:----------|:---------|
| **App 重启** | App crash 不处理 | 修复 root cause + 加 catch |
| **SystemServer 重启** | 喂狗链路断 | input/VSYNC/binder 喂狗监控 |
| **Zygote 重启** | Zygote 资源耗尽 | 监控 Zygote 内存 + 升级 GKI |
| **整机重启** | kernel panic | panic_on_oops + pstore + last_kmsg |

## 5.4 预防机制（架构师必修）

**5 个必做**：
1. **bootstat 接入 APM**（**必做**）—— 监控重启频率
2. **last_kmsg 开启**（**必做**）—— 重启取证
3. **pstore 配置正确**（**必做**）—— 持久化
4. **panic_on_oops = 1**（生产）—— 避免 oops 累积
5. **喂狗链路监控**（推荐）—— input/VSYNC/binder

# 6. 实战案例

## 6.1 案例 A（CASE-STAB-06-01）：SystemServer 反复重启 → pstore 抓 kernel log → GPU driver 卡死

> **类型**：典型模式
>
> **环境**：AOSP 17.0.0_r1 / Kernel android17-6.18 / 设备 Pixel 6
>
> **症状**：用户报"手机反复重启"
>
> **根因**：GPU driver 卡死 → cascade 到 SWT → 杀 SystemServer → 重启

### 现象

```
用户操作：
  T+0s    启动相机
  T+5s    GPU driver 卡死（GPU hang）
  T+5s    SurfaceFlinger 阻塞
  T+5s    VSYNC 喂狗断
  T+30s   Watchdog 触发（30s 周期）
  T+30s   SWT 杀 SystemServer
  T+40s   SystemServer 重启
  T+50s   重启后 GPU driver 还是卡死
  T+50s   再次触发 SWT
  T+50s   杀 SystemServer
  ... **反复重启**

**关键观察**：**反复重启**（不能恢复）
```

### 分析（last_kmsg + pstore）

```bash
# 重启后立即抓 last_kmsg
$ adb shell cat /proc/last_kmsg > last_kmsg.log

# last_kmsg 关键内容
[   30.000] Watchdog: *** WATCHDOG KILLING SYSTEM PROCESS: Blocked in handler on WindowManager
[   30.001] Watchdog: Input event dispatching timed out
[   30.002] Watchdog: VSYNC event delivery timed out
[   30.003] Watchdog: Killing system server due to blocked handler in WindowManager
```

**关键读法**：
- `Blocked in handler on WindowManager` ← WM 卡死
- `VSYNC event delivery timed out` ← **VSYNC 喂狗断**（根因）
- `Input event dispatching timed out` ← input 喂狗断（级联）

### 根因（逆向 cascade）

```
GPU driver hang（kernel 层）
  ↓ SurfaceFlinger 等 GPU
  ↓ VSYNC 喂狗断（30s）
  ↓ WindowManager 卡死（SurfaceFlinger 在 WM 路径上）
  ↓ Watchdog 触发 SWT
  ↓ 杀 SystemServer
  ↓ SystemServer 重启
  ↓ 启动时 GPU driver 还是 hang
  ↓ 再次触发 SWT
  ↓ **反复重启**（GPU driver 不修复 = 永远重启）
```

### 修复

**短期**（绕过 GPU）：
```bash
# 临时禁用 GPU 硬件加速
adb shell setprop debug.hwui.renderer skiagl
# 或降级到 OpenGL ES 2.0
adb shell setprop ro.opengles.version 131072
```

**长期**（修复 GPU driver）：
- 联系 GPU 厂商（Qualcomm / Mali / PowerVR）
- 升级 GPU driver 版本
- 监控 GPU hang（`/sys/kernel/debug/dri/0/hang`）

### 验证

1. 复现：触发 GPU hang
2. 短期修复：禁用 GPU 加速 → 不再反复重启
3. 长期修复：升级 GPU driver
4. APM：反复重启率 = 0

# 7. 总结

## 7.1 架构师视角 5 条 Takeaway

1. **REBOOT 是结果态，不是原因态**：所有 REBOOT 都能倒推到 SWT / KE / NE / 异常累积。
2. **4 类重启源按严重性递增**：App < SystemServer < Zygote ≈ 整机。
3. **cascade 链路是 REBOOT 的常见根因**：占 50-60%，架构师必须在每个节点阻断。
4. **重启溯源 4 件套**：last_kmsg + pstore + dropbox + bootstat。
5. **预防 5 件**：bootstat 接入 + last_kmsg 开启 + pstore 配置 + panic_on_oops + 喂狗链路监控。
