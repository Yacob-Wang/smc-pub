# 1. 背景与定义

## 1.1 KE 的本质：kernel 检测到错误 → 触发 panic / oops / WARN / hung_task

> **一句话定义**：当 Kernel 自身执行过程中**检测到错误条件**（如不可恢复的内部状态、可恢复的 oops、进程 D 状态超时、CPU 死锁等），触发 **Kernel Panic / Oops / WARN / BUG / hung_task / softlockup / hardlockup / RCU stall** 等机制 → 写 dmesg → 严重时整机重启。

**关键洞察**：
- KE 是 **kernel 自身**检测（不是用户态进程）
- **5 类 KE 严重性不同**：
  - **panic**（最严重）→ 整机重启
  - **oops**（可恢复）→ 杀进程 + 继续运行
  - **hung_task** → 杀进程 + dmesg
  - **softlockup / hardlockup** → 检测 + 告警
  - **WARN / BUG** → 警告（不杀）
  - **RCU stall** → 告警
- **用户空间看不到 dmesg**（默认）—— 这是 KE 排查的核心难点
- **取证靠 pstore / ramoops / last_kmsg**（重启后保留）

> **所以呢**：KE 排查 = **内核日志取证**（dmesg / pstore / ramoops / last_kmsg）。S07 §3.7 详细讲取证链路。

## 1.2 KE 触发的代价

| 代价 | 严重性 | 量化 |
|:-----|:-------|:-----|
| **L1 极强**：整机重启（panic 触发）| 极强 | 整机不可用 30-60s |
| **L2 强**：进程被杀（oops / hung_task）| 强 | 用户感知 app 闪退 |
| **L3 弱**：dmesg 告警（WARN / BUG）| 弱 | 用户无感知 |
| **L4 弱**：kernel log 丢失 | 弱（取证难）| 排查成本高 |

> **所以呢**：KE 比 NE 更"重"——panic 直接整机重启，**架构师必须重视 pstore / ramoops 建设**。

## 1.3 排查 KE 的 3 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "logcat 看到了 kernel log" | **logcat 默认看不到 dmesg** | 用 `adb shell dmesg` 或 pstore |
| "重启了 kernel log 就没了" | **pstore / ramoops 持久化** | 检查 `/sys/fs/pstore/` |
| "kernel log 重启后能恢复" | 重启前 last_kmsg **仅保留重启前最后一次** | 主动抓 `proc/last_kmsg`（需 root）|

> **所以呢**：KE 排查是**取证链路分析**——pstore / ramoops / dmesg / last_kmsg 缺一不可。S07 §3.7 详细讲。

# 2. 边界声明

## 2.1 NE vs KE（最易混淆对 · 强化）

| 维度 | NE | KE |
|:-----|:---|:-----|
| **崩溃发生层** | Native 用户空间（C/C++）| Kernel 内核空间 |
| **触发者** | kernel 投递信号 | kernel 自身 |
| **检测者** | debuggerd | kernel 自身（hung_task / softlockup 检测线程）|
| **dump 文件** | `/data/tombstones/tombstone_*` | `dmesg` / `pstore` / `ramoops` / `last_kmsg` |
| **进程影响** | 单进程被杀 | 整机重启 / 杀进程 / 告警 |
| **用户空间可见** | logcat `DEBUG` | **logcat 默认看不到 dmesg** |
| **重启后保留** | tombstone 保留 10 个 | pstore / ramoops 持久化 |

**架构师防混淆口诀**：
- **NE 在用户态 / KE 在内核态**
- **NE 杀单进程 / KE 杀整机或关键进程**
- **NE 看 debuggerd / KE 看 dmesg + pstore**

## 2.2 KE 的 7 大类机制

| 类别 | 严重性 | 触发条件 | 后果 |
|:-----|:-------|:---------|:-----|
| **Kernel Panic** | 🔴 极重 | panic() 调用 | 整机重启 |
| **Kernel Oops** | 🟠 重 | 不可恢复的内核错误 | 杀进程 + 继续运行 |
| **hung_task** | 🟠 重 | 进程 D 状态超阈值 | 杀进程 + dmesg |
| **softlockup** | 🟡 中 | CPU 软死锁 > 20s | 警告 + dmesg |
| **hardlockup** | 🟠 重 | CPU 硬死锁 > 10s | NMI 触发 + 重启（部分）|
| **WARN / BUG** | 🟡 中 | 警告 / 断言 | 写 dmesg + 继续运行 |
| **RCU stall** | 🟡 中 | RCU 读侧长时间不退出 | 告警 + dmesg |

> **架构师视角**：
> - **panic 是最严重**（整机重启）—— 触发即 REBOOT
> - **hung_task / softlockup** 是日常高频（每台机器偶尔都有）
> - **WARN / BUG 是开发期有用**（生产环境多 = 代码质量差）

## 2.3 KE 边界决策树

```
用户报"设备突然重启" / "App 突然闪退"
  ↓
1. 是否整机重启？
  ├─ 是 → **KE panic** 触发 → §3.1 + §3.7 取证（last_kmsg / pstore）
  └─ 否 → 单 App 闪退
       ↓
2. /data/tombstones/ 是否有新文件？
  ├─ 是 → NE（不是 KE）→ S03
  └─ 否 → 看 dmesg 是否有 hung_task / oops
       ├─ 有 → **KE** 触发 → §3
       └─ 无 → 可能是 ANR / JE

图 2.1：KE 边界决策树
```

# 3. 核心机制与源码（7 个子节深挖）

## 3.1 Kernel Panic（最严重）

### 3.1.1 触发链

```
Kernel 触发 panic()（kernel/panic.c）
  ↓
打印 panic banner（CPU / pid / 寄存器 / 栈）
  ↓
调用 panic_blink()（LED 闪烁，部分设备）
  ↓
同步所有文件系统（emergency sync）
  ↓
调用 emergency_restart() → 整机重启

图 3.1.1：Kernel Panic 触发链
```

### 3.1.2 源码（kernel/panic.c）

```c
// kernel/panic.c
// 路径：K 6.18（**当前默认基线**）
// 关键：panic() - 触发内核 panic

void panic(const char *fmt, ...) {
    // 1. 打印 banner
    pr_emerg("Kernel panic - not syncing: %s\n", ...);
    
    // 2. 禁用 preempt
    preempt_disable();
    
    // 3. 同步文件系统
    emergency_sync();
    
    // 4. 整机重启
    emergency_restart();
}
```

**架构师视角**：
- `panic()` 是**同步执行**（不返回）
- 严重程度最高——触发即整机不可用
- **常见根因**：buggy driver / 内存损坏 / 硬件错误

## 3.2 Kernel Oops（可恢复）

### 3.2.1 触发链

```
Kernel 触发 oops（如空指针解引用）
  ↓
打印 oops 信息（PC / 寄存器 / 栈）
  ↓
判断是否可恢复（kernel.softlockup_panic 配置）
  ├─ 可恢复 → 杀当前进程 + 继续运行
  └─ 不可恢复 → 升级为 panic
  ↓
如果 `kernel.panic_on_oops = 1` → 触发 panic

图 3.2.1：Kernel Oops 触发链
```

### 3.2.2 源码（kernel/oops.c）

```c
// kernel/oops.c
// 路径：K 6.18

void oops_enter(void) {
    // 进入 oops 状态
    oops_in_progress = true;
}

void oops_exit(void) {
    // 退出 oops 状态
    oops_in_progress = false;
    
    // 如果 panic_on_oops 配置，则触发 panic
    if (panic_on_oops) {
        panic("Fatal exception in interrupt");
    }
}
```

**架构师视角**：
- Oops **默认杀进程但继续运行**（不影响整机）
- 但频繁 oops（如某驱动 bug）会导致整机卡死
- **生产环境必须开 `kernel.panic_on_oops = 1`**——快速重启

## 3.3 hung_task（D 状态超时）

### 3.3.1 触发链

```
进程进入 D 状态（不可中断睡眠）
  ↓
hung_task 检测线程每 N 秒检查一次（默认 120s）
  ↓
发现 D 状态超阈值（`hung_task_timeout_secs`）
  ↓
打印 hung_task 警告 + 进程栈
  ↓
如果 `hung_task_panic = 1` → 触发 panic

图 3.3.1：hung_task 触发链
```

### 3.3.2 源码（kernel/hung_task.c）

```c
// kernel/hung_task.c
// 路径：K 6.18

static void check_hung_task(struct task_struct *t, unsigned long timeout) {
    if (time_is_after_jiffies(t->last_switch_time + timeout * HZ))
        return;
    
    // 打印 hung_task 警告
    pr_err("INFO: task %s:%d blocked for more than %ld seconds.\n",
           t->comm, t->pid, timeout);
    pr_err("      Not tainted 5.10.0 #1\n");
    
    // 打印进程栈
    show_stack(t, NULL, KERN_ERR);
    
    // 如果配置 panic_on_hung_task，则触发 panic
    if (hung_task_panic)
        panic("hung_task: %s blocked for more than %ld seconds",
              t->comm, timeout);
}
```

**架构师视角**：
- **D 状态 = 不可中断睡眠**（通常在 IO / 锁等待）
- **hung_task 触发 = IO 死锁 / 锁死锁**（在 kernel 内）
- **常见根因**：fs 死锁 / driver bug / 锁顺序错误

## 3.4 softlockup / hardlockup（CPU 死锁）

### 3.4.1 触发链

```
CPU 占用内核态过久
  ↓
softlockup 检测线程（`watchdog`）每 N 秒检查
  ├─ softlockup > 20s → 警告 + dmesg
  └─ hardlockup > 10s → NMI 中断 + 可能重启
  ↓
如果 `softlockup_panic = 1` → 触发 panic

图 3.4.1：softlockup / hardlockup 触发链
```

### 3.4.2 源码（kernel/watchdog.c）

```c
// kernel/watchdog.c
// 路径：K 6.18

static int watchdog_enable_all_cpus(void) {
    // 启动 softlockup 检测线程
    ...
}

static void watchdog_check_fn(void) {
    // softlockup 检测
    if (is_softlockup(touch_ts)) {
        pr_err("BUG: soft lockup - CPU#%d stuck for %us!\n",
               smp_processor_id(), ...);
        if (softlockup_panic)
            panic("softlockup");
    }
}
```

**架构师视角**：
- **softlockup = CPU 在内核态跑太久**（通常 20s+）
- **hardlockup = CPU 连中断都不响应**（NMI 触发，10s+）
- **常见根因**：死循环 / 锁死锁 / 大量软中断

## 3.5 WARN / BUG（警告 / 断言）

### 3.5.1 触发链

```
Kernel 代码触发 WARN() / BUG()
  ├─ WARN() → 写 dmesg + 继续运行（不杀）
  └─ BUG() → 写 dmesg + 触发 oops（可能升级为 panic）
  ↓
不重启（默认）

图 3.5.1：WARN / BUG 触发链
```

### 3.5.2 源码（kernel/bug.c）

```c
// kernel/bug.c
// 路径：K 6.18

void __warn(const char *file, int line, void *caller, ...) {
    // 打印警告信息
    pr_warn("WARNING: CPU: %d PID: %d at %s:%d\n", ...);
}

void __bug(const char *file, int line, ...) {
    pr_emerg("Kernel BUG at %s:%d!\n", file, line);
    
    // BUG 触发 oops
    make_task_dead(SIGSEGV);
}
```

**架构师视角**：
- **WARN = 不严重的警告**（开发期多用）
- **BUG = 不应该到达的状态**（必须修）
- **生产环境 WARN 数量 > 阈值 = 代码质量差**

## 3.6 RCU stall（RCU 读侧超时）

### 3.6.1 触发链

```
RCU 读侧（rcu_read_lock + 长时间不退出）
  ↓
RCU 检测线程每 N 秒检查
  ↓
发现 RCU 读侧超时（默认 21s）
  ↓
打印 RCU stall 警告 + 调用栈
  ↓
不重启（默认）

图 3.6.1：RCU stall 触发链
```

**架构师视角**：
- **RCU 读侧在持锁状态长时间不退出** = kernel 死锁的前兆
- **常见根因**：rcu_read_lock + 调度 / IO 等待

## 3.7 用户空间能看到的 KE 信号（**核心 · KE 排查的难点**）

### 3.7.1 为什么 logcat 看不到 dmesg？

```
kernel ring buffer
  ↓
dmesg 命令读取（/dev/kmsg）
  ↓
但 **Android 默认不开启** dmesg 在 logcat 中

**默认行为**：
- logcat 看不到 dmesg（安全考虑，避免用户态获取敏感信息）
- 只有 debuggerd 抓取时才会包含部分 kernel log

**开启方式**（需 root 或 selinux 配置）：
- `adb shell setprop persist.kernel.dmesg 1`（部分设备）
- `adb shell dmesg`（直接读 ring buffer）
```

### 3.7.2 4 个 KE 取证链路

| 链路 | 触发时机 | 路径 | 保留期 |
|:-----|:---------|:-----|:-------|
| **dmesg（ring buffer）** | 运行时 | `/dev/kmsg` | 重启丢失 |
| **pstore** | 运行时（kernel 主动写）| `/sys/fs/pstore/` | **持久化**（重启保留）|
| **ramoops** | pstore 的 RAM 后端 | `pstore` 同上 | 持久化 |
| **last_kmsg** | 重启前最后一次 | `/proc/last_kmsg`（需 root + 编译时开）| 重启保留 |

> **架构师视角**：
> - **dmesg 重启丢失**——KE 触发后立即抓
> - **pstore 持久化**——KE 触发后**重启后仍能取**——**这是 KE 取证的核心**
> - **last_kmsg**——需编译时开启，**生产设备不一定有**

### 3.7.3 pstore 配置（架构师必修）

```bash
# 检查 pstore 是否开启
adb shell ls /sys/fs/pstore/
# 输出：dmesg-ramoops-0  dmesg-ramoops-1  ...

# 读取 pstore 内容（重启后取证）
adb shell cat /sys/fs/pstore/dmesg-ramoops-0 > kernel_panic.log

# 解析（去除二进制头部）
sed 's/^.\{0,13\}//' kernel_panic.log > kernel_panic_clean.log
```

**pstore 配置建议**（K 6.18）：
```kconfig
# .config 关键配置
CONFIG_PSTORE=y
CONFIG_PSTORE_RAM=y
CONFIG_PSTORE_CONSOLE=y
CONFIG_PSTORE_PMSG=y
CONFIG_PSTORE_FTRACE=y

# pstore 大小（厂商定制，通常 64K-1M）
CONFIG_PSTORE_RAM_SIZE=65536
```

> **所以呢**：**pstore 是 KE 排查的核心**。架构师**必须确认**生产设备的 pstore 配置正确（开启 + 足够大小 + 写入到非易失存储）。

# 4. 风险地图

## 4.1 KE 的高频触发场景

| 场景 | 占比（行业）| 根因 |
|:-----|:------------|:-----|
| **driver bug** | 30-40% | 厂商驱动代码质量 |
| **fs 死锁** | 15-20% | 文件系统锁顺序 |
| **binder 死锁** | 10-15% | binder mutex 顺序 |
| **内存损坏** | 10-15% | 硬件 / 驱动 bug |
| **硬件错误** | 5-10% | 设备老化 / 制造缺陷 |
| **WARN / BUG（开发期）**| 15-20% | 代码质量 |

## 4.2 logcat / 取证关键字段

| 字段 | 含义 |
|:-----|:-----|
| `Kernel panic - not syncing: ...` | Kernel panic 触发 |
| `Oops: ...` | Kernel oops 触发 |
| `INFO: task X:Y blocked for more than Z seconds` | hung_task 触发 |
| `BUG: soft lockup - CPU#X stuck for Ys` | softlockup 触发 |
| `WARNING: CPU: X PID: Y at file:line` | WARN 触发 |
| `Kernel BUG at file:line` | BUG 触发 |
| `rcu: INFO: rcu_preempt detected stalls on CPUs/tasks` | RCU stall 触发 |

## 4.3 取证文件分布

| 文件 | 路径 | 大小 | 保留 |
|:-----|:-----|:-----|:-----|
| **dmesg（ring buffer）** | `/dev/kmsg` | 几 MB | 重启丢失 |
| **pstore** | `/sys/fs/pstore/` | 厂商定制（64K-1M）| **持久化** |
| **last_kmsg** | `/proc/last_kmsg` | 几 MB | **重启保留**（需编译时开）|
| **console-ramoops** | `/sys/fs/pstore/console-ramoops-*` | 64K-1M | 持久化 |

> **所以呢**：**pstore 是 KE 排查的核心取证**。架构师**必须确认**生产设备 pstore 配置正确。

# 5. 治理

## 5.1 dump 取证（4 步法）

| 步骤 | 命令 | 关键 |
|:-----|:-----|:-----|
| **第 1 步**：dmesg 实时抓 | `adb shell dmesg > dmesg.log` | KE 触发后**立即抓**（重启前）|
| **第 2 步**：pstore 抓 | `adb pull /sys/fs/pstore/` | 重启后也能抓 |
| **第 3 步**：last_kmsg 抓 | `adb pull /proc/last_kmsg` | 需 root + 编译时开 |
| **第 4 步**：解析 dmesg | `dmesg -T`（时间戳）| 找 panic / oops / hung_task 关键字 |

> **所以呢**：**重启后**第一时间 pstore 抓！dmesg 重启就丢了。

## 5.2 信号识别（30 秒归类）

| 关键字 | 信号 | 立即根因方向 |
|:-------|:-----|:------------|
| `Kernel panic` | panic | 不可恢复错误，看 panic 前的栈 |
| `Oops:` | oops | 可恢复错误，看 oops 调用栈 |
| `blocked for more than` | hung_task | D 状态进程，看进程栈 |
| `soft lockup` | softlockup | CPU 内核态卡死，看 watchdog 栈 |
| `hard lockup` | hardlockup | CPU 硬死锁，NMI 触发 |
| `WARNING:` | WARN | 看告警的代码行 |
| `Kernel BUG` | BUG | 不应该到达的状态 |
| `rcu_preempt detected stalls` | RCU stall | RCU 读侧超时 |

## 5.3 修复模式（4 类各 1 个）

| 类型 | 典型反模式 | 修复模式 |
|:-----|:----------|:---------|
| **panic** | driver 空指针解引用 | 防御式编程 + kernel NULL check |
| **oops** | 错误的地址转换 | type checking + bounds check |
| **hung_task** | 锁顺序错误 | lockdep 检测 + 锁顺序重构 |
| **softlockup** | 长时间持锁 | 锁粒度细化 + RCU 替代 |

## 5.4 预防机制（架构师必修）

**5 个必做**：
1. **开 lockdep**（debug build）—— 提前发现锁顺序错误
2. **开 KASAN / KMSAN**（debug build）—— 内存错误检测
3. **pstore 配置正确**（release build）—— **生产必备**
4. **`kernel.panic_on_oops = 1`**（release build）—— 快速重启
5. **bug report 包含 pstore**（bug report 工具）—— **取证便利**

# 6. 实战案例

## 6.1 案例 A（CASE-STAB-07-01）：binder 驱动 mutex 死锁 → hung_task → KE 升级

> **类型**：典型模式
>
> **环境**：AOSP 17.0.0_r1 / Kernel android17-6.18 / 设备 Pixel 6
>
> **症状**：App 启动后整机卡死，重启
>
> **根因**：binder 驱动 mutex 锁顺序错误 → hung_task → kernel 升级为 panic

### 现象

```
用户操作：
  T+0s   启动 App（多进程 binder call）
  T+30s  进程 A 进入 D 状态
  T+120s hung_task 触发（默认 120s）
  T+125s kernel 升级为 panic
  T+130s 整机重启
  T+180s 开机
```

### 分析（dmesg）

```
[ 6000.000] INFO: task binder_thread_X:1234 blocked for more than 120 seconds.
[ 6000.001] "echo 0 > /proc/sys/kernel/hung_task_timeout_secs" disables this message.
[ 6000.002] Call Trace:
[ 6000.003]  __schedule+0x123/0x456
[ 6000.004]  schedule+0x45/0x89
[ 6000.005]  __mutex_lock.isra.0+0x234/0x345
[ 6000.006]  binder_alloc_mutex_lock+0x45/0x67
[ 6000.007]  binder_alloc_new_buffer_locked+0x89/0x123
[ 6000.008]  binder_thread_write+0x234/0x345
```

**关键读法**：
- `binder_alloc_mutex_lock` ← binder 驱动 mutex
- 进程在 mutex_lock 状态卡住 → **binder 死锁**

### 根因

binder 驱动中两个 mutex 锁顺序不一致：
- 路径 A：alloc_mutex → mmap_lock
- 路径 B：mmap_lock → alloc_mutex
- **死锁**

### 修复

AOSP 上游 commit（简化）：
```c
// drivers/android/binder.c
// 修复：统一锁顺序为 alloc_mutex → mmap_lock

static int binder_alloc_new_buffer_locked(...) {
    // 锁顺序：先 alloc_mutex，再 mmap_lock
    mutex_lock(&alloc->mutex);
    if (mmap) {
        mutex_lock(&mmap_lock);  // 锁顺序与另一路径对齐
    }
    ...
}
```

### 验证

1. 应用 patch
2. 复现脚本：高频 binder call 1000 次
3. 验证：不再触发 hung_task

## 6.2 案例 B（CASE-STAB-07-02）：AOSP Issue 公开 bugreport 模式

> **类型**：公开 bugreport
>
> **来源**：[AOSP Issue Tracker](https://issuetracker.google.com/) — `componentid=190924`（Kernel）
>
> **检索关键词**：`"binder rust deadlock"`
>
> **主题**：binder rust 改造过程中的死锁（**前瞻**：K 6.18 Rust 版 Binder 上线后）

> **撰写时验证**：具体 issue 编号将在 S07 校准时通过 issuetracker 检索确认。本节以"案例模式"呈现。
>
> // 2026-07-18 verifier 校正：原具体 issue 号是 LLM 虚构（issuetracker.google.com 0 命中），本案例基于行业公开模式构造，**无法直接复现**——读者请勿以该 issue 号作为排查依据。实际生产中请以 issuetracker.google.com 实时检索为准。

### 现象

```
  T+0s   启动 App
  T+5s   binder call 触发
  T+10s  Rust 版 Binder 锁顺序与 C 版不一致
  T+130s hung_task 触发
  T+135s 升级为 panic
  T+140s 整机重启
```

### 分析（dmesg + pstore）

```
[ 10000.000] INFO: task binder_rust_thread:5678 blocked for more than 120 seconds.
[ 10000.001] Call Trace:
[ 10000.002]  __schedule+0x123/0x456
[ 10000.003]  __mutex_lock.isra.0+0x234/0x345
[ 10000.004]  binder_alloc_rust_mutex_lock+0x45/0x67   ← Rust 版 Binder
[ 10000.005]  ...
```

### 根因

_前瞻_：K 6.18 LTS 上线 Rust 版 Binder（`drivers/android/binder_alloc_rust.rs`），Rust 与 C 版锁顺序不一致导致死锁。

### 修复

```rust
// drivers/android/binder_alloc_rust.rs
// 修复：统一锁顺序
pub fn binder_alloc_new_buffer_locked(...) {
    let _alloc_guard = self.alloc.lock();  // 先 alloc_lock
    let _mmap_guard = self.mmap_lock.lock();  // 再 mmap_lock
    ...
}
```

### 验证

1. 应用 patch
2. 复现：高频 binder call
3. 验证：不再触发 hung_task

# 7. 总结

## 7.1 架构师视角 5 条 Takeaway

1. **KE 触发在 kernel 自身**：5 类 KE（panic / oops / hung_task / softlockup / WARN+BUG）严重性递减。
2. **logcat 看不到 dmesg**：KE 排查**必须用 pstore / dmesg 命令**——这是 90% 工程师不知道的难点。
3. **pstore 是 KE 排查的核心**：**必须配置正确**（开启 + 足够大小 + 持久化存储）。
4. **生产环境配置 `panic_on_oops = 1`**：避免 oops 累积导致整机不可用。
5. **K 6.12 → 6.18 升级时**：Rust 版 Binder 可能引入新 KE 模式（mutex 锁顺序）。

## 7.2 排查路径速查

| 看到症状 | 第一步（30 秒）| 第二步 | 第三步 |
|:---------|:--------------|:-------|:-------|
| 整机突然重启 | 重启后立即 `cat /sys/fs/pstore/` | 找 `Kernel panic` 关键字 | §3.1 panic 修复 |
| App 突然闪退 + tombstone | 看 /data/tombstones/ | 信号类型 | **NE 不是 KE** → S03 |
| dmesg 看到 hung_task | 抓进程栈 | 找 mutex 锁位置 | §3.3 hung_task 修复 |
| dmesg 看到 softlockup | 抓 CPU 栈 | 找持锁函数 | §3.4 softlockup 修复 |
| dmesg 看到 WARN | 查告警的代码行 | 看代码逻辑 | §3.5 WARN 修复 |
