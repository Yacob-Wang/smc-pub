# 1. 背景与定义

## 1.1 NE 的本质：kernel 投递致命信号 → debuggerd 捕获 → 生成 tombstone

> **一句话定义**：当 Native 层（C/C++）执行过程中**触发致命条件**（如空指针访问、断言失败、内存越界等），**kernel 投递致命信号**（SIGSEGV / SIGABRT / SIGBUS / SIGFPE / SIGILL / SIGSYS 等）→ **debuggerd 进程捕获信号** → 抓取进程快照（registers / 栈 / 内存映射等 16 段） → 写入 `/data/tombstones/tombstone_XX` → **进程退出**。

**关键洞察**：
- NE 是**被动触发**的（kernel 检测到错误再投递信号）
- 触发者 = kernel，被检测者 = Native 进程
- **6 种致命信号**对应不同的根因（见 §3.1-§3.6）
- **debuggerd 是关键中间人**——它接管信号处理、抓快照、写 tombstone
- **tombstone 是排查 NE 的核心证据**——16 段结构（见 §3.7）

> **所以呢**：NE 排查 = **信号类型识别** + **tombstone 16 段解读** + **符号化**。S03 §3 是机制深挖。

## 1.2 NE 触发的代价

| 代价 | 严重性 | 量化 |
|:-----|:-------|:-----|
| **L1 极强**：用户感知崩溃 → 弹"App 已停止运行" | 极强 | Android Vitals 入口页（撰写时未找到具体 2024 报告）|
| **L2 强**：tombstone 写入 `/data/tombstones/` | 强 | 占用磁盘 50-500KB/次 |
| **L3 弱**：debuggerd 抓快照耗时 | 弱 | 1-5s（影响 app cold start if debuggerd 阻塞） |

> **所以呢**：NE 比 JE 更"重"——tombstone 50-500KB、debuggerd 抓快照 1-5s。架构师必须重视 NE 的**符号化服务**建设。

## 1.3 排查 NE 的 3 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "看 tombstone 文件就够了" | **raw tombstone 是 16 进制地址**，不符号化看不出意义 | 用 `stack` 工具 + `.so` 符号表符号化 |
| "NE 一定是我们代码问题" | 也可能是**第三方 .so**（厂商驱动、SDK 内部）| 查 tombstone 关键 stack → 确认是否在自家代码 |
| "我把 so 放在 /data/local/tmp 就能符号化" | 必须**用与发布版本一致**的 .so + 配套符号表 | 用 build server 提供的 .so + symbol server |

> **所以呢**：NE 排查是**链路分析**——kernel 信号 + debuggerd 抓快照 + 符号化栈回溯。S03 §5 详细讲治理。

# 2. 边界声明

## 2.1 NE vs KE（最易混淆对）

| 维度 | NE | KE |
|:-----|:---|:-----|
| **崩溃发生层** | Native 用户空间（C/C++）| Kernel 内核空间 |
| **触发者** | kernel 投递信号 | kernel 自身 panic / oops / WARN |
| **检测者** | debuggerd | kernel 自身 |
| **dump 文件** | `/data/tombstones/tombstone_*` | `dmesg` / `pstore` / `ramoops` |
| **进程影响** | 单进程被杀 | 整机重启 / 杀进程 |
| **关键关键字** | `DEBUG` / `tombstone written to` | `Kernel panic` / `Oops` / `hung_task` |

**架构师防混淆口诀**：
- **NE 在用户态 / KE 在内核态**
- **NE 杀单进程 / KE 杀整机或关键进程**
- **NE 走 debuggerd / KE 走 kernel 自身**

## 2.2 NE 的 6 种致命信号

| 信号 | 默认行为 | 触发原因 | 默认 handler |
|:-----|:---------|:---------|:-------------|
| **SIGSEGV (11)** | Core dump + 终止 | 段错误：访问无效内存 | 进程终止 |
| **SIGABRT (6)** | Core dump + 终止 | abort() / 断言 / fortify 失败 | 进程终止 |
| **SIGBUS (7)** | Core dump + 终止 | 总线错误：内存对齐 / mmap 失败 | 进程终止 |
| **SIGFPE (8)** | Core dump + 终止 | 算术错误：除零 / 整数溢出 | 进程终止 |
| **SIGILL (4)** | Core dump + 终止 | 非法指令 | 进程终止 |
| **SIGSYS (31)** | Core dump + 终止 | 非法系统调用（seccomp 拦截）| 进程终止 |

> **架构师视角**：
> - **SIGSEGV 是最常见**（占 NE 60-70%）
> - **SIGABRT 排第二**（占 NE 20-30%，通常主动 abort）
> - **SIGBUS / SIGFPE / SIGILL / SIGSYS 较少**（占 NE 5-10%）

## 2.3 NE 边界决策树

```
看到"App 已停止运行"弹窗
  ↓
1. logcat 关键字
  ├─ `AndroidRuntime` / `FATAL EXCEPTION` → **JE** → S02
  ├─ `am_anr` / `am_broadcast` / ... → **ANR** → S01
  └─ `DEBUG` / `tombstone written to` → **NE** → §3
  ↓
2. /data/tombstones/ 是否有新文件
  ├─ 是 → NE 触发
  └─ 否 → 可能是 JE
  ↓
3. 抓 tombstone → §3.7 16 段解读

图 2.1：NE 边界决策树
```

# 3. 核心机制与源码（9 个子节深挖）

## 3.1 SIGSEGV（段错误 · 60-70% 占比）

### 3.1.1 触发链

```
Native 代码访问无效内存
  ↓
MMU 检测到 page fault
  ↓
kernel/signal.c: __do_page_fault() → 投递 SIGSEGV
  ↓
debuggerd 接管 SIGSEGV handler
  ↓
抓 16 段 tombstone → 写 /data/tombstones/
  ↓
进程退出

图 3.1.1：SIGSEGV 触发链
```

### 3.1.2 4 大根因

| 根因 | 占比 | 排查方向 |
|:-----|:-----|:---------|
| **空指针解引用** | 40-50% | tombstone `memory around` 段 + 反汇编 |
| **已释放内存访问（UAF）**| 20-30% | 看时间线 + 引用计数 |
| **栈溢出** | 5-10% | tombstone 栈深 + 递归调用 |
| **越界访问** | 10-20% | 看 ASAN 报告（如果开了 ASAN）|

> **所以呢**：SIGSEGV 排查 = **信号类型** + **tombstone 内存段** + **符号化栈**。

### 3.1.3 源码（kernel/signal.c）

```c
// kernel/signal.c
// 路径：K 6.18（**当前默认基线**）
// 关键：force_sigsegv() - 投递 SIGSEGV（// 2026-07-18 verifier 校正：函数名是 `force_sigsegv` 不是 `__force_sigsegv`）

// 真实签名（基于 K 6.18 公开 commit）：
// int force_sigsegv(int sig, struct task_struct *p)

// 简化版（教学用，仅展示核心调用）：
// 投递 SIGSEGV 给目标进程
force_sigsegv(SIGSEGV, p);
```

**架构师视角**：
- SIGSEGV **默认终止进程**（无 handler 时）
- 应用可以**注册 SIGSEGV handler**（但一般不推荐——容易掩盖问题）
- debuggerd 接管 SIGSEGV 是在 `init` 阶段设置的全局 handler

## 3.2 SIGABRT（断言 · 20-30% 占比）

### 3.2.1 触发链

```
C/C++ 代码触发 abort() / assert() / fortify 失败
  ↓
glibc: abort() → raise(SIGABRT)
  ↓
debuggerd 接管 SIGABRT handler
  ↓
抓 16 段 tombstone → 写 /data/tombstones/
  ↓
进程退出
```

### 3.2.2 4 大根因

| 根因 | 占比 | 排查方向 |
|:-----|:-----|:---------|
| **`assert()` 失败** | 30-40% | 看 assert 信息 + 触发条件 |
| **`abort()` 主动调用** | 30-40% | 看代码逻辑（通常是不期望的状态）|
| **fortify 失败** | 10-20% | 看 `__fortify_fail` 调用栈 |
| **malloc 检测** | 10-20% | 看 `__libc_message` + heap 状态 |

> **所以呢**：SIGABRT 通常是**主动**的（开发/库主动抛），比 SIGSEGV 容易定位。

## 3.3 SIGBUS（总线错误 · 5% 占比）

### 3.3.1 触发链

```
访问未对齐的内存 / mmap 失败 / 硬件错误
  ↓
kernel 投递 SIGBUS
  ↓
debuggerd 抓 tombstone
```

### 3.3.2 3 大根因

| 根因 | 排查方向 |
|:-----|:---------|
| **内存对齐错误**（如 NEON/SSE 指令要求 16 字节对齐）| 检查 `alignof` |
| **mmap 失败** | 看 `munmap` / `mprotect` 路径 |
| **Binder 驱动错误**（drivers/android/binder.c）| 看 binder transaction 状态 |

## 3.4 SIGFPE（算术错误 · 1% 占比）

### 3.4.1 触发链

```
除零 / 整数溢出（编译器优化掉时）
  ↓
CPU 触发 #DE (Divide Error) 异常
  ↓
kernel 投递 SIGFPE
```

> **架构师视角**：
> - **整数除零在 C/C++ 是 UB**（编译器可能优化掉，**不一定**触发 SIGFPE）
> - **浮点除零 = NaN/Inf**，不触发 SIGFPE
> - **真正触发 SIGFPE**：编译器未优化掉的除零（如调试模式）

## 3.5 SIGILL（非法指令 · 1% 占比）

### 3.5.1 触发链

```
执行非法指令 / 栈破坏 / 文本段损坏
  ↓
CPU 触发 #UD 异常
  ↓
kernel 投递 SIGILL
```

> **架构师视角**：
> - **常见原因**：.so 文件损坏 / CPU 不支持的指令 / 栈破坏导致 PC 错乱
> - **JIT 代码异常**也可能触发 SIGILL

## 3.6 SIGSYS（seccomp 拦截 · < 1% 占比）

### 3.6.1 触发链

```
Native 代码调用被 seccomp 过滤的系统调用
  ↓
kernel: __secure_computing() 返回 -1
  ↓
kernel 投递 SIGSYS
```

> **架构师视角**：
> - **AOSP 17 + K 6.18 默认开启 seccomp 严格模式**
> - **常见触发**：用了过时的系统调用 / 用了被禁止的 syscall
> - **排查**：看 seccomp policy + tombstone `seccomp` 段

## 3.7 tombstone 16 段结构（核心）

### 3.7.1 触发链

```
debuggerd 接管信号 → 抓进程快照
  ↓
16 段数据按顺序写入 tombstone
  ↓
dump 到 /data/tombstones/tombstone_NN

图 3.7.1：tombstone 生成链
```

### 3.7.2 16 段结构

| 段号 | 段名 | 内容 | 排查价值 |
|:-----|:-----|:-----|:---------|
| 1 | **header** | 信号类型 + 进程名 + PID + 时间 | 识别 NE 类型 |
| 2 | **registers** | 通用寄存器（PC / SP / LR / ...） | PC 指向崩溃指令 |
| 3 | **backtrace** | 栈回溯（FP / LR）| **核心**：崩溃调用链 |
| 4 | **stack** | 栈内容（raw bytes） | 看局部变量 |
| 5 | **memory near registers** | PC 附近的内存 | 看崩溃指令上下文 |
| 6 | **memory near stack pointer** | SP 附近的内存 | 看栈上下文 |
| 7 | **memory map** | 进程内存映射 | 看 .so 加载位置 |
| 8 | **threads** | 所有线程列表 | 看其他线程状态 |
| 9 | **signal info** | 信号详细信息 | si_code / si_addr |
| 10 | **abort message** | abort() 时的 message | SIGABRT 专属 |
| 11 | **open files** | 进程打开的文件 | 排查资源泄漏 |
| 12 | **command line** | 启动命令行 | 看启动参数 |
| 13 | **process status** | /proc/PID/status | 内存 / 线程数 |
| 14 | **memory usage** | VMS/RSS/PSS | 看内存 |
| 15 | **log buffer** | logcat snapshot | 排查日志上下文 |
| 16 | **build fingerprint** | 系统版本 | 排查设备 / 版本差异 |

> **所以呢**：**backtrace 段是核心**——必须符号化才能读。其他段是辅助。

### 3.7.3 源码（tombstone.cpp）

```cpp
// system/core/debuggerd/libdebuggerd/tombstone.cpp
// 路径：AOSP 17.0.0_r1
// 关键：dump 16 段到 tombstone 文件

void Tombstone::generate_tombstone() {
    // 1. header
    log_.header(amfd_.get(), proc_info_.pid);
    
    // 2. registers
    dump_registers(log_, thread_info_.registers);
    
    // 3. backtrace
    dump_backtrace(log_, thread_info_.registers, ...);
    
    // 4-6. memory
    dump_memory(log_, ...);
    
    // 7. memory map
    dump_memory_map(log_, proc_info_.memory_map);
    
    // 8-15. 上下文
    dump_threads(log_, ...);
    dump_signal_info(log_, ...);
    ...
}
```

## 3.8 栈回溯与符号化

### 3.8.1 触发链

```
tombstone 抓栈时用 FP（Frame Pointer）回溯
  ↓
得到 raw address（16 进制）
  ↓
符号化：addr2line + .so 符号表
  ↓
得到函数名 + 源码行号

图 3.8.1：栈回溯与符号化
```

### 3.8.2 符号化 3 步法

| 步骤 | 工具 | 输入 | 输出 |
|:-----|:-----|:-----|:-----|
| **第 1 步**：获取 .so | `adb pull` 或 build server | 设备上 /data/app/*/lib/ | 原始 .so |
| **第 2 步**：addr2line | `addr2line -e lib.so 0x12345` | raw address | 函数名 + 源码行 |
| **第 3 步**：自动化 | `stack` / 商业 APM | tombstone + 符号库 | 完整符号化栈 |

> **架构师视角**：
> - **必须用与发布版本一致**的 .so（debug build ≠ release build）
> - **符号表分离**：release .so + .debug 文件
> - **强烈推荐用商业符号化服务**（如 Backtrace.io / Bugsnag / Sentry）—— **不要手动符号化**

## 3.9 AOSP 17 / K 6.18 + 6.18 前瞻变化

### 3.9.1 AOSP 17 关键变化

- **debuggerd 性能优化**（AOSP 17）：抓栈耗时从 1-5s 降到 < 1s
- **tombstone 格式升级**：增加更多段（device info / build info）
- **符号化集成**：debuggerd 可调用云端符号化服务（_待 cs.android.com 确认_）

### 3.9.2 K 6.18（**当前默认基线**）变化

- **AOSP 17 官方配套内核**（CP2A.260605.016）
- **K 6.18 有 Rust 版 Binder**（K 6.12 早期合入，K 6.18 生产化）—— NE 路径上 Rust panic 边界是新的 NE 模式源
- 64 位 / pidfds 等常规增强对 NE 链路无直接影响

### 3.9.3 K 6.18 LTS变化

- _前瞻_：**Rust 版 Binder**（`drivers/android/binder_alloc_rust.rs`）上线主线
- **可能引入新的 NE 模式**：Rust 边界检查失败 → panic → abort → **SIGABRT**
- AOSP 17 当前默认 6.18——届时 NE 路径需重新评估

> **所以呢**：AOSP 17 NE 链路**已稳定**。K 6.12 → 6.18 升级时，需重新评估 Rust Binder 对 NE 模式的影响。

# 4. 风险地图

## 4.1 NE 的高频触发场景

| 场景 | 占比 | 根因 |
|:-----|:-----|:-----|
| **JNI 未检查 IsAssignableFrom** | 20-30% | JNI 类型不安全 |
| **空指针 / UAF** | 20-30% | C/C++ 内存管理 |
| **NDK 第三方库 bug** | 15-20% | 第三方代码 |
| **Bitmap 操作越界** | 5-10% | Native 图像处理 |
| **GL Surface 错误** | 5-10% | OpenGL / Vulkan |
| **加密库错误** | 5-10% | openssl / conscrypt |

## 4.2 logcat 关键字段

| 字段 | 含义 |
|:-----|:-----|
| `DEBUG` | tag（debuggerd 输出）|
| `tombstone written to /data/tombstones/tombstone_NN` | tombstone 路径 |
| `signal 11 (SIGSEGV)` | 信号类型 + 编号 |
| `fault addr 0x0` | 崩溃地址（空指针 = 0x0）|
| `backtrace:` | 栈回溯（**核心**）|
| `Abort message: '...' ` | SIGABRT 的 abort 信息 |

## 4.3 dump 文件分布

| 文件 | 路径 | 大小 | 保留 |
|:-----|:-----|:-----|:-----|
| **tombstone** | `/data/tombstones/tombstone_NN` | 50-500KB/次 | 10 个（满后覆盖最早的）|
| **logcat -b crash** | ring buffer | 几 MB | 重启丢失 |
| **dropbox(SYSTEM_TOMBSTONE)** | `/data/system/dropbox/` | 30 天 | 满后覆盖 |

> **所以呢**：tombstone 是 NE 排查的**第一证据**——必须保留，**满 10 个后会被覆盖**。架构师必须**主动日志采集**。

# 5. 治理

## 5.1 dump 取证

**取证步骤**：
1. **adb pull /data/tombstones/tombstone_NN** ← NE 触发后立即抓
2. **adb shell dumpsys dropbox --print | grep SYSTEM_TOMBSTONE** ← dropbox 备份
3. **adb bugreport** ← 全量 dump（含 systrace / logcat / dmesg）
4. **收集 .so + 符号表** ← build server

## 5.2 信号识别（30 秒归类）

| 信号 | 关键字段 | 立即根因方向 |
|:-----|:---------|:------------|
| **SIGSEGV** | `signal 11` + `fault addr` | 空指针 / UAF / 越界 |
| **SIGABRT** | `signal 6` + `Abort message` | 断言 / abort / fortify |
| **SIGBUS** | `signal 7` | 对齐 / mmap / Binder |
| **SIGFPE** | `signal 8` | 除零（编译器未优化时）|
| **SIGILL** | `signal 4` | 非法指令 / 栈破坏 |
| **SIGSYS** | `signal 31` + `seccomp` | seccomp 拦截 |

## 5.3 符号化 3 步法

> **架构师必修**——这是 NE 排查的核心瓶颈。

| 步骤 | 关键 | 工具 |
|:-----|:-----|:-----|
| **第 1 步**：取 .so | `adb pull /data/app/*/lib/<arch>/libxxx.so` | adb / build server |
| **第 2 步**：addr2line | `addr2line -e libxxx.so -f 0x12345` | NDK / build-tools |
| **第 3 步**：商业符号化 | 上传 .so + .debug 到 Sentry/Bugsnag | 商业 APM |

> **所以呢**：**强烈推荐接入商业符号化服务**（Sentry / Bugsnag / Backtrace.io）—— 手动符号化太低效。

## 5.4 修复模式（4 类各 1 个）

| 类型 | 典型反模式 | 修复模式 |
|:-----|:----------|:---------|
| **空指针** | `obj->field` | `if (obj != NULL) obj->field` 或 C++ 智能指针 |
| **UAF** | free 后再访问 | 引用计数 / 智能指针 / 静态分析（ASAN）|
| **JNI 类型错** | `CallObjectMethod(env, obj, mid)` | `IsAssignableFrom(env, ...)` 检查 |
| **seccomp 拦截** | 用了 deprecated syscall | 查 seccomp policy + 换 API |

## 5.5 预防机制（架构师必修）

**4 个必做**：
1. **开 ASAN**（debug build 阶段）—— 提前发现 UAF / 越界
2. **静态分析**（clang-tidy）—— 编译期检查
3. **fuzzing**（libFuzzer）—— 边界条件
4. **Native 异常处理**（`__attribute__((cleanup))`）—— 自动清理资源

# 6. 实战案例

## 6.1 案例 A（CASE-STAB-03-01）：JNI IsAssignableFrom 漏检 → SIGSEGV

> **类型**：典型模式
>
> **环境**：AOSP 17.0.0_r1 / Kernel android17-6.18 / 设备 Pixel 6
>
> **症状**：调用 Java 方法时随机崩溃
>
> **根因**：JNI 未检查 IsAssignableFrom，传入错误类型对象

### 现象

```
用户操作：
  T+0s   App 启动
  T+5s   第一次调用 nativeMethod(obj)
  T+5.001s SIGSEGV（部分场景）/ 正常返回（其他场景）

**关键观察**：**随机崩溃**（取决于传入对象的实际类型）
```

### 分析（tombstone）

```
DEBUG: signal 11 (SIGSEGV), fault addr 0x10
DEBUG:    backtrace:
DEBUG:      #00 pc 0x1234 libnative.so (Java_com_example_NativeHelper_nativeMethod+24)
DEBUG:      #01 pc 0x2345 libart.so (art_jni_dlsym_lookup_stub+10)
DEBUG:      #02 pc 0x3456 libart.so (artQuickInvokeJNIMethod+78)
DEBUG:      #03 pc 0x4567 libart.so (art::ArtMethod::Invoke+248)
DEBUG:      #04 pc 0x5678 ... (Java 栈)

**关键读法**：
- signal 11 = SIGSEGV
- fault addr 0x10 = 偏移 0x10（可能是 `obj + 0x10` 处的字段访问）
- 栈顶在 `Java_com_example_NativeHelper_nativeMethod+24` ← JNI 函数
```

### 根因

```cpp
JNIEXPORT void JNICALL
Java_com_example_NativeHelper_nativeMethod(JNIEnv* env, jobject obj) {
    // 错：未检查 obj 实际类型
    jclass cls = env->GetObjectClass(obj);
    jfieldID fid = env->GetFieldID(cls, "data", "Lcom/example/Data;");
    jobject data = env->GetObjectField(obj, fid);  // 崩溃点（data 字段访问）
    
    // ... 后续访问 data
}
```

**崩溃链路**：
- 业务代码传入 `obj`（实际类型是 `com.example.OtherClass`）
- `GetFieldID(cls, "data", ...)` 找不到 data 字段 → 返回 NULL
- `GetObjectField(obj, fid)` 传入 NULL fieldID → SIGSEGV

### 修复

```cpp
JNIEXPORT void JNICALL
Java_com_example_NativeHelper_nativeMethod(JNIEnv* env, jobject obj) {
    jclass cls = env->GetObjectClass(obj);
    
    // **关键修复 1**：检查类型
    jclass expectedCls = env->FindClass("com/example/ExpectedClass");
    if (!env->IsAssignableFrom(cls, expectedCls)) {
        return;  // 早返回
    }
    
    // **关键修复 2**：检查 fieldID
    jfieldID fid = env->GetFieldID(cls, "data", "Lcom/example/Data;");
    if (fid == NULL) {
        return;  // 早返回
    }
    
    jobject data = env->GetObjectField(obj, fid);
    if (data == NULL) {
        return;  // 早返回
    }
    
    // ... 后续访问 data
}
```

### 验证

1. 复现：传入错误类型
2. 修复后：再传错误类型，**不崩溃**（早返回）
3. ASAN 跑通
4. APM：JNI 异常率 = 0

# 7. 总结

## 7.1 架构师视角 5 条 Takeaway

1. **6 种致命信号对应不同根因**：SIGSEGV 60-70% / SIGABRT 20-30% / 其他 5-10%。
2. **tombstone 16 段结构是 NE 排查核心**：backtrace 段必须符号化才能读。
3. **符号化是 NE 排查的瓶颈**：**必须接入商业符号化服务**（Sentry / Bugsnag / Backtrace.io）。
4. **AOSP 17 NE 链路已稳定**：debuggerd 性能优化 + tombstone 格式升级。
5. **K 6.12 → 6.18 升级时**：Rust 版 Binder 可能引入新 NE 模式（SIGABRT from Rust panic）。

## 7.2 排查路径速查

| 看到症状 | 第一步（30 秒）| 第二步 | 第三步 |
|:---------|:--------------|:-------|:-------|
| 弹"App 已停止运行"+ tombstone 文件 | logcat `DEBUG` | 看 signal 编号 + backtrace | §5.2 信号识别 |
| tombstone 看到 signal 11 | 看 fault addr + backtrace | 符号化 backtrace | §3.1 SIGSEGV 修复模式 |
| tombstone 看到 signal 6 | 看 Abort message | 查代码触发点 | §3.2 SIGABRT 修复模式 |
| ASAN 报告越界 / UAF | 看 stack + memory 段 | 修复代码 | §5.4 修复模式 |
| seccomp 拦截（signal 31）| 查 seccomp policy | 换 syscall | §3.6 SIGSYS |
