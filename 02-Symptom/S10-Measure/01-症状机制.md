# 3. 稳定性度量学：5 大度量

> **本节是 S10 的核心**——5 大度量是稳定性治理的"硬基础"。

## 3.1 度量 #1：MTBF（Mean Time Between Failures）

### 3.1.1 定义

**MTBF**（平均故障间隔时间）= 总运行时间 / 故障次数

**稳定性 MTBF 公式**：
```
MTBF = 总运行时间 / 致命故障次数
```

**致命故障**（Fatal Failure）定义：
- App 闪退（JE）
- App 卡死超过 30s（ANR / HANG）
- App 频繁 OOM 被系统杀
- App 整机重启（KE / REBOOT）

**示例**：
- 100 万 DAU 一天，10 万小时运行，发生 100 次闪退
- MTBF = 100,000 小时 / 100 = **1000 小时 / 次**
- 含义：平均每 1000 小时发生 1 次闪退

### 3.1.2 行业基准

| 行业 | 优秀 MTBF | 良好 MTBF | 及格 MTBF |
|:-----|:----------|:----------|:----------|
| 金融 / 工具 App | > 10,000 小时 | > 5,000 小时 | > 1,000 小时 |
| 社交 / 内容 App | > 5,000 小时 | > 2,000 小时 | > 500 小时 |
| 游戏 App | > 2,000 小时 | > 1,000 小时 | > 200 小时 |
| 系统 ROM | > 50,000 小时 | > 20,000 小时 | > 5,000 小时 |

> **所以呢**：**MTBF 越高越好**——但要按行业基准对齐，**金融 App 的 1000 小时 MTBF 可能是游戏 App 的"优秀"水平**。

### 3.1.3 MTBF 的 3 大局限

| 局限 | 说明 | 应对 |
|:-----|:-----|:-----|
| **1. 不区分严重性** | 闪退 1 次 = ANR 1 次 = HANG 1 次（实际严重性不同）| 配合"严重性加权"使用 |
| **2. 不区分频次用户** | 1 个用户 1 次闪退 = 100 个用户 1 次闪退（实际不同）| 配合"崩溃率"使用 |
| **3. 不区分首次/回归** | 首次发现 = 上线回归（实际不同）| 配合"回归率"使用 |

## 3.2 度量 #2：崩溃率 / ANR 率 / OOM 率

### 3.2.1 定义

**崩溃率**（Crash Rate）= 闪退次数 / 启动次数 × 100%

**ANR 率**（ANR Rate）= ANR 次数 / 启动次数 × 100%

**OOM 率**（OOM Rate）= OOM 次数 / 启动次数 × 100%

**HANG 率**（HANG Rate）= HANG 次数 / 启动次数 × 100%

**示例**：
- 100 万次启动，发生 5000 次闪退
- 崩溃率 = 5000 / 1,000,000 × 100% = **0.5%**
- 含义：每 1000 次启动有 5 次闪退

### 3.2.2 行业基准

| App 类型 | 崩溃率 | ANR 率 | OOM 率 |
|:---------|:-------|:-------|:-------|
| 金融 / 工具 | < 0.1% | < 0.05% | < 0.05% |
| 社交 / 内容 | < 0.3% | < 0.1% | < 0.1% |
| 游戏 | < 0.5% | < 0.2% | < 0.2% |
| 系统 ROM | < 0.05% | < 0.02% | < 0.02% |

> **所以呢**：**崩溃率 < 0.3% 是"良好"基线，< 0.1% 是"优秀"**——超过基线必须立即优化。

### 3.2.3 崩溃率分母选择

**分母 1：启动次数**（最常用）
- 优点：简单直观
- 缺点：长会话 App（如 IM / 阅读）低估

**分母 2：DAU 活跃次数**
- 优点：覆盖所有会话
- 缺点：DAU 计算复杂

**分母 3：会话次数**
- 优点：粒度适中
- 缺点：会话定义难

**推荐**：**启动次数 + 活跃次数** 两种分母并行使用，覆盖不同场景。

## 3.3 度量 #3：严重性加权

### 3.3.1 定义

**严重性加权**（Severity Weight）= 不同严重程度的故障用不同权重计算

**权重表**（**业界常用**）：

| 严重性 | 权重 | 包含症状 |
|:-------|:-----|:---------|
| **Fatal** | 100 | 整机重启（KE / REBOOT）|
| **Critical** | 10 | NE / 严重 OOM / 持久 HANG |
| **High** | 5 | JE / ANR / 中度 OOM |
| **Medium** | 1 | 短暂 HANG / 软卡死 |
| **Low** | 0.1 | UI 抖动 / 微卡 |

**示例**：
- 1000 个故障：500 个 High（JE）+ 300 个 Medium（HANG）+ 200 个 Critical（NE）
- 严重性得分 = 500×5 + 300×1 + 200×10 = **4,800**
- 归一化得分 = 4,800 / 1000 = **4.8**（说明：平均每个故障是"High"级别）

### 3.3.2 行业基准

| 行业 | 严重性得分（平均）| 含义 |
|:-----|:-----------------|:-----|
| 金融 / 工具 | < 1 | 几乎都是 Low 级别 |
| 社交 / 内容 | < 5 | 主要是 High 级别 |
| 游戏 | < 10 | 包含 Critical 级别 |
| 系统 ROM | < 0.5 | 几乎都是 Low 级别 |

> **所以呢**：**严重性加权 = 故障的"质量"**——单看"故障数"会掩盖严重性。

## 3.4 度量 #4：回归率

### 3.4.1 定义

**回归率**（Regress Rate）= 已知问题再次发生的比例

**回归率公式**：
```
回归率 = (本期已知问题再次发生数) / (本期已知问题总数) × 100%
```

**示例**：
- 已知问题库 100 个，本期发生 50 个，其中 20 个是上版本修过但本期又发生的
- 回归率 = 20 / 50 × 100% = **40%**
- 含义：50 个故障里 40% 是回归

### 3.4.2 行业基准

| 行业 | 优秀回归率 | 良好回归率 | 及格回归率 |
|:-----|:----------|:----------|:----------|
| 任何 | < 5% | < 15% | < 30% |

> **所以呢**：**回归率 > 30% = 测试体系失效**——说明修复没真正落地或没覆盖到。

## 3.5 度量 #5：稳定性综合指数（**架构师新工具**）

### 3.5.1 定义

**稳定性综合指数**（Stability Score）= 多度量加权综合得分

**公式**：
```
Stability Score = 0.3 × (1 - 崩溃率/0.5%) + 0.3 × (1 - ANR率/0.1%) +
                  0.2 × (1 - OOM率/0.1%) + 0.1 × (1 - 严重性得分/10) +
                  0.1 × (1 - 回归率/30%)
```

**得分范围**：0-1，越高越好
- 0.9-1.0：**优秀**
- 0.7-0.9：**良好**
- 0.5-0.7：**及格**
- < 0.5：**差**

### 3.5.2 示例

某 App 7 天指标：
- 崩溃率 = 0.2%
- ANR 率 = 0.08%
- OOM 率 = 0.05%
- 严重性得分 = 3
- 回归率 = 15%

```
Stability Score = 0.3 × (1 - 0.2/0.5) + 0.3 × (1 - 0.08/0.1) +
                  0.2 × (1 - 0.05/0.1) + 0.1 × (1 - 3/10) +
                  0.1 × (1 - 15/30)
                = 0.3 × 0.6 + 0.3 × 0.2 + 0.2 × 0.5 + 0.1 × 0.7 + 0.1 × 0.5
                = 0.18 + 0.06 + 0.10 + 0.07 + 0.05
                = 0.46
```

**得分 0.46 = 差**——架构师必须立即优化。

> **所以呢**：**Stability Score 是单一数字反映综合稳定性**——适合做月度报表 / 季度 OKR。

**Stability Score 计算示例图**（某 App 7 天指标）：

```
【输入：5 大度量】
  崩溃率     = 0.2%   （行业基准 0.3%）
  ANR 率    = 0.08%  （行业基准 0.1%）
  OOM 率    = 0.05%  （行业基准 0.1%）
  严重性    = 3      （行业基准 5）
  回归率    = 15%    （行业基准 30%）

═══════════════════════════════════════════════════════════════════

【计算：加权综合】

  崩溃率得分   = 1 - 0.2/0.3  = 0.33
  ANR 率得分   = 1 - 0.08/0.1 = 0.20
  OOM 率得分   = 1 - 0.05/0.1 = 0.50
  严重性得分   = 1 - 3/5      = 0.40
  回归率得分   = 1 - 15/30    = 0.50

  权重：崩溃率 0.3 / ANR 0.3 / OOM 0.2 / 严重性 0.1 / 回归率 0.1

  Stability Score = 0.3×0.33 + 0.3×0.20 + 0.2×0.50 + 0.1×0.40 + 0.1×0.50
                  = 0.099 + 0.060 + 0.100 + 0.040 + 0.050
                  = 0.349

═══════════════════════════════════════════════════════════════════

【输出：决策建议】

  Stability Score = 0.349

  ┌──────────────────────────────────────┐
  │ 0.349 < 0.5  →  【差】                │
  │                                      │
  │ 决策：暂停新功能，全员优化稳定性       │
  │ 行动：2 周稳定性冲刺                  │
  │ 资源：100% 投入                       │
  │ 期望：1 月内达到 0.5+                 │

  对标：0.7+ 是"良好"基线，0.349 远低于基线
```

# 4. 度量体系：4 大类指标如何构建

> **本节把 5 大度量组合成"指标体系"**——单看 1 个指标会偏，**4 大类指标组合才能反映全貌**。

## 4.1 指标分类 1：单点指标（Point Metrics）

**定义**：某一刻的瞬时值

**示例**：
- 主线程响应时间 P95 = 200ms
- GC 暂停 P99 = 100ms
- binder call 等待 P99 = 50ms

**适用**：实时监控 + 即时告警

## 4.2 指标分类 2：聚合指标（Aggregate Metrics）

**定义**：一段时间的聚合值

**示例**：
- 崩溃率（日聚合）= 0.2%
- MTBF（月聚合）= 5000 小时
- Stability Score（周聚合）= 0.75

**适用**：日报 / 周报 / 月报

## 4.3 指标分类 3：趋势指标（Trend Metrics）

**定义**：连续 2+ 周期的变化率

**示例**：
- 崩溃率趋势：本周 0.2% vs 上周 0.15%（**+33% 上升** ← 告警）
- ANR 率趋势：本周 0.08% vs 上周 0.10%（**-20% 下降** ← 好）
- Stability Score 趋势：本周 0.75 vs 上周 0.70（**+7% 上升** ← 好）

**适用**：周报对比、月度 OKR 评估

## 4.4 指标分类 4：分层指标（Layered Metrics）

**定义**：按"系统层 / 进程层 / 函数层"分层

| 分层 | 示例 | 适用 |
|:-----|:-----|:-----|
| **系统层** | SystemServer crash 率、整机重启率 | ROM 厂商 |
| **进程层** | App 进程 crash 率、App 进程 ANR 率 | App 团队 |
| **函数层** | 特定函数（如 ChatManager.send）调用失败率 | 具体业务方 |
| **用户层** | VIP 用户崩溃率、新用户崩溃率 | 业务方 |

**适用**：精准定位到具体责任方

> **架构师视角**：**4 大类指标 = 实时（单点）+ 周期（聚合）+ 趋势（变化）+ 责任（分层）**——4 类组合才能反映全貌。

# 5. 发布门禁：CI/CD 中如何卡稳定性

> **本节是 S10 的落地价值**——度量是"看"，门禁是"挡"。**没门禁 = 没治理**。

## 5.1 6 大门禁维度

### 5.1.1 维度 1：崩溃率门禁

**阈值**：
- 主版本：崩溃率 < 0.3%
- 子版本：崩溃率 < 0.5%
- 修复版本：崩溃率 < 1.0%

**门禁动作**：
- 超过阈值 → release blocker
- 超过 2x 阈值 → 直接拒绝

### 5.1.2 维度 2：ANR 率门禁

**阈值**：
- 主版本：ANR 率 < 0.1%
- 子版本：ANR 率 < 0.2%
- 修复版本：ANR 率 < 0.5%

**门禁动作**：
- 超过阈值 → release blocker
- 超过 2x 阈值 → 直接拒绝

### 5.1.3 维度 3：性能门禁

**阈值**：
- 主线程 P95 响应 < 200ms
- GC 暂停 P99 < 100ms
- 启动时间 P95 < 2000ms

**门禁动作**：
- 超过阈值 → release blocker
- 超过 1.5x 阈值 → 直接拒绝

### 5.1.4 维度 4：兼容性门禁

**阈值**：
- Android 8.0+ 启动成功率 > 99.5%
- 不同 ROM（MIUI / EMUI / OriginOS）崩溃率差异 < 0.1%
- 不同芯片（高通 / 联发科 / 麒麟）崩溃率差异 < 0.1%

**门禁动作**：
- 超过阈值 → 兼容性告警（不阻止 release，但要 follow-up）

### 5.1.5 维度 5：安全门禁

**阈值**：
- 高危漏洞 = 0
- 中危漏洞 < 5
- 数据合规 100%（**详见 [S08](../S08-AOSP17-K618/01-症状机制.md) §6.5**）

**门禁动作**：
- 高危漏洞 = 0 → release blocker
- 中危漏洞 > 5 → 拒绝

### 5.1.6 维度 6：业务门禁

**阈值**：
- 核心业务路径崩溃率 < 0.05%
- 核心业务路径 ANR 率 < 0.02%
- 业务方满意度 > 90%

**门禁动作**：
- 核心业务崩溃率 > 0.1% → release blocker
- 业务方满意度 < 80% → 业务方签字

> **架构师视角**：**6 大门禁维度 = 崩溃 / ANR / 性能 / 兼容 / 安全 / 业务**——缺一不可。

**6 大门禁维度的"否决权"时序图**：

```
[代码 push]
    │
    ▼
[门禁 #1: 单元测试覆盖率 ≥ 80%]   ← 失败 → 立即返回
    │
    ▼
[门禁 #2: 静态分析（SonarQube）]   ← 失败 → 立即返回
    │
    ▼
[门禁 #3: 集成测试（核心路径 100%）]  ← 失败 → 立即返回
    │
    ▼
[构建 APK]
    │
    ▼
[灰度发布 1%]
    │
    ▼
[门禁 #4: 24h 崩溃率 < 0.3%]   ← 失败 → 立即回滚
    │                          ← 失败 → 72h 内禁止 release
    ▼
[灰度发布 20%]
    │
    ▼
[门禁 #5: 7d Stability Score > 0.7]  ← 失败 → 立即回滚
    │                              ← 失败 → 1 月内禁止 release
    ▼
[全量发布]
    │
    ▼
[门禁 #6: 月度 Stability Score 不下降]  ← 下降 > 5% → 警告
                                          ← 下降 > 10% → 复盘
```

**6 大门禁的"否决权"原则**：
- **门禁 #1-3**：开发期，**失败立即返回**（无法构建）
- **门禁 #4-5**：发布期，**失败立即回滚 + 禁止 release**（已发布则回滚）
- **门禁 #6**：运营期，**下降超阈值告警**（月度复盘）

> **架构师视角**：**6 大门禁 = 6 个"硬红线"**——任何一个失败都阻止 release，**没有"软门禁"**。

## 5.2 门禁流程（CI/CD 集成）

```
[代码 push]
    ↓
[单元测试]  ← 门禁 #1：测试覆盖率 > 80%
    ↓
[静态分析]  ← 门禁 #2：SonarQube 质量门
    ↓
[构建 APK]
    ↓
[集成测试]  ← 门禁 #3：核心业务路径 100% 通过
    ↓
[灰度发布 1%]
    ↓
[稳定性指标监控 24h]  ← 门禁 #4：崩溃率 < 0.3%
    ↓
[灰度发布 20%]
    ↓
[稳定性指标监控 7 天]  ← 门禁 #5：Stability Score > 0.7
    ↓
[全量发布]
    ↓
[月度 Stability Score 监控]  ← 门禁 #6：月度得分不下降
```

> **架构师视角**：**6 大门禁对应 6 个 CI/CD 卡点**——任何一个卡点失败都阻止 release。

## 5.3 门禁的 3 大注意事项

| 注意事项 | 说明 |
|:---------|:-----|
| **1. 阈值不能太严** | 太严 → 永远 release 不了 |
| **2. 阈值不能太松** | 太松 → 形同虚设 |
| **3. 阈值要随业务调整** | 业务变化时同步调整阈值 |

# 6. 治理闭环：度量 → 决策 → 行动 → 回归

> **本节是 S10 的终极价值**——**没有闭环 = 治理失败**。

## 6.1 闭环 4 步

### 第 1 步：度量（Measure）

**做什么**：建立 5 大度量 + 4 类指标

**关键输出**：
- 周报：Stability Score 趋势
- 月报：5 大度量行业基准对齐
- 季报：跨团队对比

### 第 2 步：决策（Decide）

**做什么**：根据度量结果决定下一步

**决策类型**：

| 度量结果 | 决策 |
|:---------|:-----|
| Stability Score < 0.5 | **重大事故**——暂停新功能，全员优化稳定性 |
| Stability Score 0.5-0.7 | **需改进**——稳定性优化优先于新功能 |
| Stability Score 0.7-0.9 | **良好**——稳定性 + 新功能并行 |
| Stability Score > 0.9 | **优秀**——稳定性维护 + 创新 |

### 第 3 步：行动（Act）

**做什么**：根据决策执行

**行动类型**：

| 决策 | 行动 |
|:-----|:-----|
| 重大事故 | 全员 2 周冲刺 + 紧急修复 + 客户补偿 |
| 需改进 | 1 月稳定性专项 + 加 APM 监控 + 加单元测试 |
| 良好 | 持续优化 + 季度复盘 |
| 优秀 | 探索创新方向 |

### 第 4 步：回归（Regress）

**做什么**：行动后验证效果

**关键动作**：
- 行动后 1 周：再测 5 大度量
- 行动后 1 月：再测 Stability Score
- 行动后 1 季：跨团队对标

**回归验证三问**：
1. 行动是否落地？
2. 度量是否改善？
3. 没有新引入其他问题？

> **架构师视角**：**没有回归 = 行动可能无效**——**回归是闭环的"硬终点"**。

**度量 → 决策 → 行动 → 回归 循环时序图**：

```
【完整闭环 4 步 · 以某 App 7 月 1 日发现崩溃率上升为例】

  T+0  7/1 09:00
       │
       ▼
  ┌─────────────┐
  │  第 1 步     │   5 大度量 → 崩溃率 0.2% → 0.4%（+100%）
  │  度量        │   严重性 = 4（上升 33%）
  │  (Measure)   │   回归率 = 20%（上升 33%）
  └──────┬──────┘
         │ 异常告警（Stability Score 0.42 → 0.30）
         ▼
  ┌─────────────┐
  │  第 2 步     │   0.30 < 0.5  →  决策：【差】
  │  决策        │   行动：暂停新功能，全员优化稳定性
  │  (Decide)    │   资源：100% 投入
  └──────┬──────┘
         │ 决策输出
         ▼
  ┌─────────────┐
  │  第 3 步     │   7/1-7/7：2 周稳定性冲刺
  │  行动        │   ├─ 修复 Top 10 崩溃堆栈
  │  (Act)       │   ├─ 优化 Top 5 ANR 路径
  │              │   ├─ 加 APM 监控
  │              │   └─ 加单元测试覆盖率
  └──────┬──────┘
         │ 行动完成
         ▼
  ┌─────────────┐
  │  第 4 步     │   7/8 验证：
  │  回归        │   ├─ 1 周：崩溃率 0.4% → 0.25%（-37%）✓
  │  (Regress)   │   ├─ 1 月：Stability Score 0.30 → 0.55（+83%）✓
  │              │   └─ 1 季：持续监控无新引入 ✓
  └──────┬──────┘
         │ 回归通过
         ▼
       回到度量层（持续循环）
       
═══════════════════════════════════════════════════════════════════

【关键时序元素】

  异常告警（度量 → 决策）       → 5 分钟内响应
  决策输出（决策 → 行动）         → 24 小时内启动
  行动完成（行动 → 回归）         → 2-4 周
  回归验证（回归 → 下一轮度量）    → 1 周 / 1 月 / 1 季
  循环周期（整个 4 步）           → 1-3 月
```

## 6.3 度量-决策-行动-回归的 5 大红线

|:-----|:-------|:-------|:-------|:-------|

| **4. 不回归** | 有度量 | 决定 | 行动 | 不验证效果 |
| **5. 不循环** | 一次性 | — | — | 闭环失败 |

# 7. 实战案例

## 7.1 案例 A（CASE-STABILITY-S10-01）：某电商 App 稳定性治理体系建设

> **典型模式**：从"无量度无门禁"到"完整治理闭环"

**背景**：
- 某电商 App，日活 500 万
- 治理前：每周 release，无任何稳定性门禁
- 治理后：建立完整度量 + 门禁 + 闭环

**治理前 6 个月指标**：
- 崩溃率：0.6%（高于行业基准 0.3%）
- ANR 率：0.25%（高于行业基准 0.1%）
- 回归率：35%（行业优秀 < 5%）
- Stability Score：0.32（**差**）

**第 1 步：建立度量**（2 周）
- 建立 5 大度量（MTBF / 崩溃率 / ANR 率 / 严重性 / 回归率）
- 建立 4 类指标（单点 / 聚合 / 趋势 / 分层）
- 引入 APM（Sentry + 自研）

**第 2 步：建立门禁**（2 周）
- 6 大门禁维度（崩溃 / ANR / 性能 / 兼容 / 安全 / 业务）
- 6 个 CI/CD 卡点
- 阈值：崩溃率 < 0.3% / ANR 率 < 0.1% / Stability Score > 0.7

**第 3 步：建立闭环**（4 周）
- 周报：Stability Score 趋势
- 月报：5 大度量行业基准对齐
- 季报：跨团队对比
- 度量 → 决策 → 行动 → 回归 4 步流程

**第 4 步：执行**（持续）
- 第 1-3 月：Stability Score 0.32 → 0.55（从"差"到"及格"）
- 第 4-6 月：Stability Score 0.55 → 0.78（从"及格"到"良好"）
- 第 7-12 月：Stability Score 0.78 → 0.85（持续优化到"良好+")

**治理后 6 个月指标**：
- 崩溃率：0.18%（**-70%**）
- ANR 率：0.07%（**-72%**）
- 回归率：8%（**-77%**）
- Stability Score：0.78（**+144%**）

> **架构师视角**：**这是"完整治理闭环"的典型案例**——6 个月从"差"到"良好+"，**5 大度量全部改善 70%+**。

## 7.2 案例 B（CASE-STABILITY-S10-02）：某社交 App 发布门禁的"红线"事件

> **典型模式**：门禁被绕过导致严重线上事故

**背景**：
- 某社交 App，2026-03 release
- 业务方施压要赶"春节活动"
- 跳过崩溃率门禁直接 release

**跳过门禁的决策**：
- 业务方：春节活动 > 稳定性
- 架构师：业务压力大，被迫同意
- 结果：release 后 1 周发现线上崩溃率 0.8%（**超过阈值 0.3% 的 2.67x**）

**线上事故**：
- 春节活动期间，500 万 DAU 崩溃
- 投诉量上升 300%
- 业务损失：DAU 下降 5%（约 25 万用户）
- 紧急回滚 + 修复 2 周
- 总体损失估算：**直接 + 间接损失 2000 万**

**门禁恢复**：
- 之后重新确立"门禁不可绕过"原则
- 业务方签字：必须先优化稳定性才能 release
- 增加 6 大门禁维度的硬阈值（不可临时调整）

**架构师反思**：
- **门禁必须有"否决权"**——业务方不应能绕过
- **门禁绕过 = 短期业务 vs 长期稳定性**的赌注
- **架构师的职责是"守住门禁"**——这是"治理"的核心

> **架构师视角**：**这是"门禁被绕过"的反面案例**——**短期业务压力下，门禁必须坚守**。**2000 万损失 = 1 个门禁决策失误**。

# 8. 风险地图（治理决策视角）

## 8.1 高 ROI 治理场景

- **用户基数大（亿级）**：Stability Score 提升 0.1 = 1% 留存提升 → **ROI 极高**
- **App 对稳定性要求高（金融 / 工具）**：门禁必须严格 → **ROI 高**
- **业务迭代快（每周 release）**：门禁自动化 → **ROI 高**
- **跨团队协作多（多 App 共享代码）**：统一度量 + 门禁 → **ROI 高**

## 8.2 低 ROI / 高风险治理场景

- **App 用户基数小（万级）**：度量成本 > 收益 → **ROI 低**
- **业务极不稳定（创业期）**：度量会拖慢迭代 → **ROI 低**
- **单点 App（无代码共享）**：度量 + 门禁可简化 → **ROI 中**
- **业务方强势（无否决权）**：门禁形同虚设 → **ROI 0**

## 8.3 不要治理场景

- **业务起步期（前 3 月）**：先活下来，再治理
- **App 即将下线**：无需投入
- **业务方完全不配合**：无法建立闭环

# 9. 度量与门禁的"5 大反模式"

## 9.1 反模式 1：度量只为"好看"

**表现**：度量 + 粉饰 + 隐瞒问题
**问题**：失去客观性
**应对**：度量透明化，跨团队监督

## 9.2 反模式 2：门禁只为"挡事"

**表现**：门禁 = 找借口阻止 release
**问题**：失去工程意义
**应对**：门禁 = 解决问题，不是挡 release

## 9.3 反模式 3：闭环只为"走形式"

**表现**：度量 → 决策 → 行动 → 回归 走过场
**问题**：失去改进作用
**应对**：闭环必须产出具体改进行动

## 9.4 反模式 4：基准只为"对标"

**表现**：盲目对标大厂基准
**问题**：大厂基准不一定适合你
**应对**：基于自己业务定制基准

## 9.5 反模式 5：治理只为"汇报"

**表现**：度量 + 门禁只为应付检查
**问题**：失去实际价值
**应对**：治理必须服务业务

# 10. 架构师视角 5 条 Takeaway

1. **5 大度量是稳定性的"客观定义"**——MTBF / 崩溃率 / ANR 率 / 严重性 / 回归率 + Stability Score 综合指数，**没有度量就没有治理**。
2. **6 大门禁维度 = CI/CD 卡点**——崩溃 / ANR / 性能 / 兼容 / 安全 / 业务，**任何一个卡点失败都阻止 release**（**门禁必须坚守**）。
3. **度量 → 决策 → 行动 → 回归 闭环是治理的"硬终点"**——**没有闭环 = 治理失败**（案例 B 损失 2000 万就是闭环失败的代价）。
4. **AOSP 17 + K 6.18 升级 + 5 大根因优化**是提升 Stability Score 的 2 大杠杆——**单 ART 17 GenCC 升级可提升 0.05-0.10**（详见 [S08](../S08-AOSP17-K618/01-症状机制.md) §5）。
5. **Stability Score 0.7+ 是"良好"基线**——低于 0.5 立即全公司优化，高于 0.9 才能探索创新方向，**度量驱动决策**。
