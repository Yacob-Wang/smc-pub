# 1. 什么是 Stability Budget？

> **Stability Budget = Stability Score 的"可消耗余量"**——类比 Google SRE 的 Error Budget
>
> **核心思想**：稳定性**不是越高越好**——一个 100 分稳定性的代价 = 0 业务变更 = 死亡

```
Stability Score 越高    →  业务变更越谨慎   →  业务发展越慢
Stability Score 越低    →  业务变更越激进   →  业务发展越快
                       ↑
                  平衡点 = 行业平均（~85）
```

> **Stability Budget = 行业平均 85 分 - 当前实际 Score** = 你还有多少"变更余量"

# 2. 5 维预算

> 每个维度都有独立预算——**单维度消耗完不阻断，但综合分消耗完要阻断**

## 2.1 5 维预算公式

```
崩溃预算 = 100 - 实际崩溃分（满分 100）
ANR 预算 = 100 - 实际 ANR 分
性能预算 = 100 - 实际性能分
兼容预算 = 100 - 实际兼容分
安全预算 = 100 - 实际安全分
业务预算 = 100 - 实际业务分
```

## 2.2 行业基线（预算参考）

| 维度 | 行业基线 | 实际值 | 预算 |
|:-----|:---------|:-------|:-----|
| 崩溃 | 90 | 88 | 2 |
| ANR | 88 | 85 | 3 |
| 性能 | 85 | 80 | 5 |
| 兼容 | 85 | 90 | -5（实际超基线 = 无预算）|
| 安全 | 95 | 100 | -5 |
| 业务 | 90 | 95 | -5 |

> **注意**：实际值 > 行业基线 = 预算 = 0（不能再消耗）

# 3. 预算扣分 3 大场景

## 3.1 场景 1：日常变更（每发版扣 1-2 分）

| 变更类型 | 扣分 |
|:---------|:-----|
| 普通 feature | -1 |
| 重大 feature | -2 |
| 重构 | -1.5 |
| 紧急修复 | -0.5 |
| 纯文案 | -0.1 |

## 3.2 场景 2：性能回归（按实际恶化扣分）

| 指标恶化 | 扣分 |
|:---------|:-----|
| 启动 P95 +10% | -2 |
| 启动 P95 +30% | -5 |
| Crash-free Session -0.1% | -5 |
| Crash-free Session -0.5% | -15 |
| ANR-free Session -0.5% | -10 |

## 3.3 场景 3：故障（按严重程度扣分）

| 故障类型 | 扣分 |
|:---------|:-----|
| P0 故障 | -10 |
| P1 故障 | -5 |
| P2 故障 | -2 |
| P3 故障 | -0.5 |
| 单一崩溃堆栈 | -0.1 |
| Top 10 崩溃 | -1 |

# 4. 4 大扣分动作

## 4.1 动作 1：扣分（实时）

> **铁律**：每次变更 / 故障**实时扣分**——不能延后

| 触发 | 扣分 |
|:-----|:-----|
| 发版完成 | -1 ~ -2 |
| 故障恢复 | -2 ~ -10 |
| 性能回归 | -2 ~ -15 |

## 4.2 动作 2：告警（按比例）

| 消耗 | 告警 |
|:-----|:-----|
| 消耗 ≤ 30% | 无 |
| 消耗 30-60% | P2 通知 |
| 消耗 60-90% | P1 警告 |
| 消耗 90-100% | P0 阻断 |

## 4.3 动作 3：暂停变更（按综合分）

| 综合分 | 状态 | 动作 |
|:-------|:-----|:-----|
| ≥ 90 | 优秀 | 正常 |
| 85-90 | 良好 | 关注 |
| 75-85 | 警告 | 谨慎变更 |
| 60-75 | 风险 | **暂停非紧急变更** |
| < 60 | 危险 | **冻结所有变更** |

## 4.4 动作 4：专项治理（耗尽时）

> **触发条件**：某维度预算 ≤ 0

- 拉专项 PM + TL + oncall
- 输出 postmortem
- 72h 内出短期项
- 2 周内出中期项
- 季度内出长期项

# 5. 预算与 SLO 的关系

| 概念 | Google SRE | Stability Budget |
|:-----|:-----------|:------------------|
| 衡量对象 | 单服务 | 整体 App |
| 目标 | SLO（99.9%）| Stability Score（85+）|
| 消耗单位 | downtime | 分数 |
| 触发动作 | 冻结发版 | 暂停变更 |
| 适合 | 服务级 SLA | 整体稳定性 |

> **关系**：Stability Budget = SLO Error Budget 的"整体版"——SLO 关心服务级，Stability Budget 关心整体

详见 [IB01-Google SRE §4 Error Budget](../../00-Meta/Industry-Benchmark/IB01-Google SRE稳定性白皮书解读.md)。

# 6. 真实案例：某 App 6 月预算消耗

## 6.1 起点（M0）

| 维度 | 分数 | 预算 |
|:-----|:-----|:-----|
| 崩溃 | 90 | 0 |
| ANR | 88 | 0 |
| 性能 | 85 | 0 |
| 兼容 | 90 | -5 |
| 安全 | 95 | -5 |
| 业务 | 95 | -5 |
| **综合** | **92** | -7（**超预算**）|

## 6.2 6 月变更（M1-M6）

| 月份 | 主要变更 | 扣分 |
|:-----|:---------|:-----|
| M1 | 启动期优化 | -1 |
| M2 | Top 10 崩溃修复 | -1 |
| M3 | ANR 治理 | -1 |
| M4 | 性能治理 | -2 |
| M5 | 低端机专项 | -1 |
| M6 | 双 11 准备 | -2 |
| **累计** | - | **-8** |

## 6.3 故障扣分

| 故障 | 扣分 |
|:-----|:-----|
| P1 × 2 | -10 |
| P2 × 5 | -10 |
| P3 × 10 | -5 |
| **累计** | **-25** |

## 6.4 6 月底

| 维度 | 起点 | 扣分 | 实际 |
|:-----|:-----|:-----|:-----|
| 崩溃 | 90 | -10 | 80 |
| ANR | 88 | -8 | 80 |
| 性能 | 85 | -10 | 75 |
| 兼容 | 90 | -3 | 87 |
| 安全 | 95 | -1 | 94 |
| 业务 | 95 | -1 | 94 |
| **综合** | **92** | **-33** | **85（行业基线）**|

> **结论**：6 月大量变更 + 故障后，综合分从 92 降至 85（行业基线），**预算已经用完**

# 7. 预算耗尽怎么办？

## 7.1 综合分预算耗尽（< 60）

| 动作 | 时限 |
|:-----|:-----|
| **冻结所有变更** | 立即 |
| 启动专项治理 | 24h |
| 拉 PM + TL + 架构师 | 24h |
| 输出 postmortem | 24h |
| 制定恢复计划 | 1 周 |
| 恢复 ≥ 60 分后解冻 | - |

## 7.2 单维度预算耗尽

| 维度 | 动作 |
|:-----|:-----|
| 崩溃 | **禁止发版**直到崩溃率回到基线 |
| ANR | **禁止发版**直到 ANR 率回到基线 |
| 性能 | 限制变更范围（仅修性能）|
| 兼容 | OEM 专项 |
| 安全 | **强制修复**（无选项）|
| 业务 | 业务 KPI 优先 |

# 8. 预算的"政治化"

## 8.1 谁决定预算大小？

- **架构师**：技术预算（崩溃/ANR/性能/兼容/安全）
- **PM**：业务预算（业务 KPI）
- **CEO**：综合预算（最终仲裁）

## 8.2 谁决定扣分动作？

- 架构师：技术扣分
- PM：业务扣分
- **架构师一票否决**：稳定性预算耗尽 = 冻结变更

## 8.3 谁决定恢复？

- **架构师**（技术恢复）
- **PM + TL**（业务恢复）

# 9. 6 反例

| # | 反例 | 错误做法 | 正确做法 |
|:-:|:-----|:---------|:---------|
| 1 | **没预算概念** | "稳定性越高越好" | **平衡点 85 分** |
| 2 | **预算不量化** | "大概能改" | **每变更必带扣分** |
| 3 | **预算不公开** | 架构师自己算 | **看板上公开** |
| 4 | **耗尽不冻结** | 继续发版 | **必冻结变更** |
| 5 | **预算不 review** | 1 年不动 | **季度 review** |
| 6 | **追责扣分** | "X 这次扣了 5 分" | **只对事不对人** |

# 10. 5 条 Takeaway

1. **Stability Budget = 行业基线 85 - 当前 Score** = 可消耗余量
2. **5 维独立预算** —— 单维度耗尽不阻断，综合分耗尽必冻结
3. **3 大扣分场景**（日常变更 / 性能回归 / 故障）+ 4 大扣分动作
4. **耗尽 = 冻结** —— 综合分 < 60 = 必冻结
5. **季度 review** —— 业务变了预算要变
