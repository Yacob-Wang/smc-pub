# 1. 背景：为什么"启动死锁"是头号 P0 工单

## 1.1 一句话定位

**启动期死锁 = 启动过程中关键路径卡死**——SystemServer / Zygote / Binder 三大类死锁**任一触发 = 整机卡死 = Watchdog 30s 杀进程 = 重启**。

## 1.2 启动期死锁的 4 个独特性

| 独特性 | 表现 | 后果 |
|:-------|:-----|:-----|
| **Watchdog 兜底** | 30s 杀 SystemServer | 整机重启 |
| **卡在固定进度** | 30%-80% 进度卡死 | 用户可感知 |
| **重启循环** | 5 次/5min = BootLoop | 工厂重置 |
| **OEM 主导** | 启动早期 OEM 主导 | 排查困难 |

## 1.3 行业数据

| 指标 | 数据 | 来源 |
|:-----|:-----|:-----|
| **启动期死锁占比** | 10-15% 启动问题 | 字节 / 阿里内部数据 |
| **SystemServer 卡死占比** | 60% 启动期死锁 | 5 大厂内部数据 |
| **Zygote 死锁占比** | 20% 启动期死锁 | 5 大厂内部数据 |
| **Binder 死锁占比** | 15% 启动期死锁 | 5 大厂内部数据 |
| **Watchdog 周期** | 30s | AOSP 17 不可调 |
| **BootLoop 阈值** | 5 次 / 5min | AOSP 17 默认 |

> **所以呢**：启动期死锁 = 10-15% 启动问题 + Watchdog 30s 兜底 = 头号 P0。

# 2. 边界：启动期死锁 vs 稳态死锁

| 维度 | 启动期死锁 | 稳态死锁 |
|:-----|:----------|:---------|
| **持续时间** | 30s+ | 任意 |
| **Watchdog** | 30s 杀 | 不影响 |
| **重启** | 5 次/5min = BootLoop | 不会自动重启 |
| **卡死位置** | SystemServer / Zygote | 任意 |
| **占死锁总数** | 10-15% | 85-90% |

# 3. 4 类启动期死锁详解

## 3.1 4 类死锁总览

```
   ┌────────────────────────────────────────────────────────────┐
   │  启动期死锁 4 大类（按占比排序）                            │
   └────────────────────────────────────────────────────────────┘

   1. SystemServer 卡死（60%）
      └─ 表现：卡在 30%-80% 进度
      └─ 原因：某 service 卡 / 依赖死锁 / 资源等待

   2. Zygote 死锁（20%）
      └─ 表现：卡在 30%-50% 进度
      └─ 原因：ART 类加载死锁 / GC 死锁 / fork 死锁

   3. Binder 死锁（15%）
      └─ 表现：卡在 50%-80% 进度
      └─ 原因：Binder 通信死锁 / servicemanager 卡

   4. 资源死锁（5%）
      └─ 表现：卡在任意阶段
      └─ 原因：文件锁 / IO 等待 / 内存 OOM
```

## 3.2 4 类死锁详解

### 类型 1 · SystemServer 卡死（占比 60%）

**典型表现**：
- 启动卡在 30%-80% 进度
- 屏幕显示 Boot Logo 或 Boot Animation 不动
- Watchdog 30s 后杀 SystemServer

**典型根因**：
- 某 service 启动时等待其他 service
- 某 service 启动时死锁
- 某 service 启动时 GC 暂停
- 某 service 启动时 IO 等待

**关键源码**：
- `frameworks/base/services/java/com/android/server/SystemServer.java`
- `frameworks/base/services/core/java/com/android/server/Watchdog.java`

### 类型 2 · Zygote 死锁（占比 20%）

**典型表现**：
- 启动卡在 30%-50% 进度
- 屏幕显示 "Starting Android" 不动

**典型根因**：
- ART 类加载死锁
- ART GC 死锁（GC 等待业务线程释放对象）
- Zygote fork 死锁
- preloadClasses 死锁

**关键源码**：
- `frameworks/base/core/java/com/android/internal/os/ZygoteInit.java`
- `frameworks/base/core/java/com/android/internal/os/Zygote.java`
- `art/runtime/gc/`

### 类型 3 · Binder 死锁（占比 15%）

**典型表现**：
- 启动卡在 50%-80% 进度
- 屏幕显示 Boot Animation 不动
- servicemanager 卡死

**典型根因**：
- servicemanager 卡死（Binder 中心）
- 某 service Binder 注册死锁
- SystemServer 等待 Binder 调用返回

**关键源码**：
- `frameworks/native/cmds/servicemanager/main.cpp`
- `frameworks/base/services/core/java/com/android/server/ServiceManager.java`

### 类型 4 · 资源死锁（占比 5%）

**典型表现**：
- 启动卡在任意阶段
- 资源（文件 / IO / 内存）竞争

**典型根因**：
- 文件锁死锁
- 磁盘 IO 等待
- 内存 OOM（GC 频繁）

> **所以呢**：4 类死锁中 SystemServer 占 60%——**头号排查目标**。

# 4. 启动期死锁的 5 大根因

## 4.1 5 大根因总览

```
   ┌────────────────────────────────────────────────────────────┐
   │  启动期死锁 5 大根因                                        │
   └────────────────────────────────────────────────────────────┘

   1. 某 service 启动卡（30%）
      └─ 表现：SystemServer 卡在某 service
      └─ 原因：service 初始化死锁

   2. 服务依赖死锁（25%）
      └─ 表现：service A 等 service B，B 等 A
      └─ 原因：服务依赖关系错

   3. 资源等待（20%）
      └─ 表现：service 等 IO / 锁
      └─ 原因：磁盘 / Binder / 内存

   4. GC 卡顿（15%）
      └─ 表现：service 启动时 GC 暂停
      └─ 原因：启动期大对象分配

   5. OEM 定制 BUG（10%）
      └─ 表现：OEM 定制 service 卡
      └─ 原因：厂商定制服务 BUG
```

## 4.2 5 大根因详解

### 根因 1 · 某 service 启动卡

**反例**：
```java
// 某 OEM service 启动时同步等待
public class OEMService extends SystemService {
    @Override
    public void onStart() {
        // 🔴 反例：同步等待 + IO
        String config = readConfigFile();  // 30s+
        // ...
    }
}
```

**正例**：
```java
public class OEMService extends SystemService {
    @Override
    public void onStart() {
        // 🟢 异步加载
        CompletableFuture.runAsync(() -> {
            String config = readConfigFile();
        });
    }
}
```

### 根因 2 · 服务依赖死锁

**反例**：
```java
// service A 等待 service B
public class ServiceA {
    @Override
    public void onStart() {
        ServiceManager.getService("B");  // 🔴 等 B ready
    }
}

// service B 等待 service A
public class ServiceB {
    @Override
    public void onStart() {
        ServiceManager.getService("A");  // 🔴 等 A ready
    }
}
```

**正例**：
```java
// service A 异步等待 service B
public class ServiceA {
    @Override
    public void onStart() {
        CompletableFuture.runAsync(() -> {
            ServiceManager.getService("B");  // 🟢 异步
        });
    }
}
```

### 根因 3 · 资源等待

**反例**：
```java
// service 启动时等 IO
public class MyService {
    @Override
    public void onStart() {
        // 🔴 等磁盘 IO
        while (!isFileReady("/data/config.txt")) {
            try { Thread.sleep(100); } catch (Exception e) {}
        }
    }
}
```

**正例**：
```java
// service 启动时不阻塞
public class MyService {
    @Override
    public void onStart() {
        // 🟢 不阻塞
    }
    
    @Override
    public void onBootComplete() {
        // bootCompleted 后再检查文件
    }
}
```

### 根因 4 · GC 卡顿

**反例**：
```java
// service 启动时分配大对象
public class MyService {
    @Override
    public void onStart() {
        // 🔴 分配 100MB 对象 → 触发 Full GC
        List<Data> data = new ArrayList<>();
        for (int i = 0; i < 1000000; i++) {
            data.add(new Data(i));
        }
    }
}
```

**正例**：
```java
// service 启动时分配小对象
public class MyService {
    @Override
    public void onStart() {
        // 🟢 分配小对象
    }
    
    // 大对象延迟到 onBootComplete
    public void onBootComplete() {
        List<Data> data = new ArrayList<>();
        // ...
    }
}
```

### 根因 5 · OEM 定制 BUG

**反例**：
- OEM 定制 service 启动时 crash
- OEM 定制 service 启动时等待硬件
- OEM 定制 service 启动时 IO 阻塞

**正例**：
- OEM 定制 service 加 try-catch
- OEM 定制 service 加超时
- OEM 定制 service 异步初始化

> **所以呢**：5 大根因覆盖 95% 启动期死锁——按根因对症下药。

# 5. 5 大排查剧本

## 5.1 排查剧本 1 · SystemServer 卡死

**症状**：
- 启动卡在 30%-80% 进度
- Watchdog 30s 后杀 SystemServer
- 屏幕显示 Boot Logo 或 Boot Animation

**4 步排查法**：
```bash
# Step 1: 看 SystemServer 状态
adb shell ps -A | grep system_server
# 异常：system_server 不存在 → 已被 Watchdog 杀

# Step 2: 看 SystemServer 卡在哪个 service
adb shell logcat -d -s SystemServer:V
# 关键：找最后一条 service 启动日志

# Step 3: 看 Watchdog dropbox
adb shell dumpsys dropbox --print SYSTEM_SERVER_WATCHDOG
# 关键：找最近 5 条 Watchdog 记录

# Step 4: 看 traces.txt
adb shell cat /data/anr/traces.txt | grep -A 50 "system_server"
# 关键：看 system_server 主线程 stack
```

**解决方案**：
- 找到卡死的 service
- 优化 service 启动（异步化 / 加超时）
- 关闭非必要 service

## 5.2 排查剧本 2 · Zygote 死锁

**症状**：
- 启动卡在 30%-50% 进度
- 屏幕显示 "Starting Android" 不动
- 进程列表无 system_server

**4 步排查法**：
```bash
# Step 1: 看 Zygote 进程
adb shell ps -A | grep zygote
# 异常：无 zygote 进程 → Zygote 死

# Step 2: 看 Zygote 日志
adb shell logcat -d -s ZygoteInit:V Zygote:V
# 关键：找最后一条 Zygote 日志

# Step 3: 看 ART 日志
adb shell logcat -d -s art:V AndroidRuntime:V
# 关键：找 ART 类加载 / GC 日志

# Step 4: 看 dmesg（preload 死锁）
adb shell dmesg | grep -i "zygote"
# 关键：找 Zygote 启动失败信息
```

**解决方案**：
- 检查 preloaded-classes（删除错误类）
- 优化 ART 启动参数
- 检查 preloaded-resources

## 5.3 排查剧本 3 · Binder 死锁

**症状**：
- 启动卡在 50%-80% 进度
- servicemanager 卡死
- 所有 Binder 通信失败

**4 步排查法**：
```bash
# Step 1: 看 servicemanager 状态
adb shell getprop init.svc.servicemanager
# 异常：stopped / restarting → servicemanager 卡

# Step 2: 看 SystemServer 是否等待 Binder
adb shell logcat -d -s ActivityManager:V
# 关键：找 AMS 等待 Binder 日志

# Step 3: 看 Binder 状态
adb shell cat /proc/binder/stats
# 关键：看 Binder 事务数

# Step 4: 看 system_server 等待
adb shell cat /data/anr/traces.txt | grep -A 30 "system_server"
# 关键：看 system_server 等待哪个 Binder
```

**解决方案**：
- 检查 servicemanager 启动
- 优化 SystemServer 启动（关闭不必要 service）
- 优化 Binder 通信（异步化）

## 5.4 排查剧本 4 · 资源死锁

**症状**：
- 启动卡在任意阶段
- 资源（文件 / IO / 内存）竞争

**4 步排查法**：
```bash
# Step 1: 看磁盘 IO
adb shell iostat
# 关键：看 IO 等待

# Step 2: 看内存
adb shell dumpsys meminfo
# 关键：看内存占用

# Step 3: 看文件锁
adb shell lsof | grep "/data"
# 关键：找文件锁

# Step 4: 看 GC
adb shell logcat -d -s art:V | grep "GC"
# 关键：找 GC 暂停
```

**解决方案**：
- 优化磁盘 IO（异步化）
- 优化内存使用（减少对象分配）
- 优化文件锁（加超时）

## 5.5 排查剧本 5 · BootLoop

**症状**：
- 设备连续重启 5+ 次
- 触发工厂重置
- 屏幕卡在 Boot Logo 5min+

**4 步排查法**：
```bash
# Step 1: 看重启历史
adb shell dumpsys bootstat --history | head -50
# 关键：看重启次数 + 重启原因

# Step 2: 看 SystemServer crash
adb shell dumpsys dropbox --print SYSTEM_TOMBSTONE
# 关键：找 system_server crash 原因

# Step 3: 看 Kernel panic
adb shell dumpsys dropbox --print KERNEL_PANIC_CONSOLE
# 关键：找 Kernel panic 日志

# Step 4: 看 AVB 状态
adb shell getprop ro.boot.veritymode
# 异常：veritymode=enforcing + 校验失败 → AVB 触发 BootLoop
```

**解决方案**：
- 修复 SystemServer crash
- 修复 Kernel panic
- 关闭 AVB（OEM 量产事故）
- 工厂重置

> **所以呢**：5 大排查剧本覆盖 95% 启动期死锁——按剧本走 = 高效定位。

# 6. 风险地图（与 Stability S04 联动 · 强制）

> **本节是 v4 强制要求**——启动期死锁是 S04 启动子集。

## 6.1 4 类死锁风险

| 死锁类型 | 触发条件 | Watchdog 兜底 | 5 次 BootLoop 触发 |
|:---------|:---------|:-------------|:------------------|
| **SystemServer 卡** | service 启动卡 | ✅ 30s 杀 | ✅ |
| **Zygote 死锁** | ART 类加载死锁 | ❌（Zygote 不在 Watchdog）| ❌ |
| **Binder 死锁** | servicemanager 卡 | ✅ 30s 杀 | ✅ |
| **资源死锁** | 文件 / IO / 内存 | ✅ 30s 杀 | ✅ |

## 6.2 启动期死锁 5 大根因

| 根因 | 占比 | 表现 |
|:-----|:----:|:-----|
| **某 service 启动卡** | 30% | SystemServer 卡在某 service |
| **服务依赖死锁** | 25% | service A 等 B，B 等 A |
| **资源等待** | 20% | service 等 IO / 锁 |
| **GC 卡顿** | 15% | service 启动时 GC 暂停 |
| **OEM 定制 BUG** | 10% | OEM 定制 service 卡 |

## 6.3 4 大根治方案

| 方案 | 原理 | 收益 | 难度 |
|:-----|:-----|:----:|:----:|
| **服务按需启动** | 关闭不必要 service | 30-50% | 🟡 中 |
| **服务异步化** | 启动时异步 IO | 20-30% | 🟢 低 |
| **加超时机制** | service 启动超时 | 10-20% | 🟡 中 |
| **Watchdog 监控** | 自动化检测 | 5-10% | 🟡 中 |

# 7. 5 实战案例

## 7.1 案例 1：SystemServer 卡在某 OEM service

**症状**：
- 启动卡在 60% 进度
- Watchdog 30s 杀 SystemServer
- 设备连续重启 5 次 → 工厂重置

**根因**：
- 某 OEM 定制 service 启动时同步读取 config 文件
- config 文件 30s 还没读完
- 触发 Watchdog

**排查过程**：
```bash
# 1. 看 SystemServer 卡在哪个 service
adb shell logcat -d -s SystemServer:V | tail -50
# 关键：看最后一条 service 启动日志

# 2. 看 Watchdog dropbox
adb shell dumpsys dropbox --print SYSTEM_SERVER_WATCHDOG
# 关键：看卡死的 service
```

**解决方案**：
- OEM service 启动异步化
- 加 try-catch
- 关闭非关键 service

**收益**：卡死 → 正常启动

## 7.2 案例 2：Zygote ART 类加载死锁

**症状**：
- 启动卡在 40% 进度
- 屏幕显示 "Starting Android" 不动
- 进程列表无 system_server

**根因**：
- preloaded-classes 中有错误类
- ART 类加载死锁

**排查过程**：
```bash
# 1. 看 Zygote 日志
adb shell logcat -d -s ZygoteInit:V
# 关键：找类加载失败

# 2. 看 ART 日志
adb shell logcat -d -s art:V
# 关键：找 ART 类加载错误
```

**解决方案**：
- 修正 preloaded-classes
- 删除错误类

**收益**：卡死 → 正常启动

## 7.3 案例 3：servicemanager 卡死

**症状**：
- 启动卡在 70% 进度
- 所有 Binder 通信失败
- 屏幕显示 Boot Animation

**根因**：
- servicemanager 启动时 crash
- 所有 service 无法注册

**排查过程**：
```bash
# 1. 看 servicemanager 状态
adb shell getprop init.svc.servicemanager
# 异常：stopped

# 2. 看 servicemanager 日志
adb shell logcat -d -s servicemanager:V
# 关键：找 crash 原因
```

**解决方案**：
- 修复 servicemanager crash
- 增加 servicemanager 启动重试

**收益**：卡死 → 正常启动

## 7.4 案例 4：service 启动时 GC 暂停 5s

**症状**：
- 启动卡在 50% 进度
- 屏幕显示 Boot Animation
- GC 暂停 5s+

**根因**：
- service 启动时分配 100MB 对象
- 触发 Full GC，暂停 5s

**排查过程**：
```bash
# 1. 看 GC 日志
adb shell logcat -d -s art:V | grep "GC"
# 关键：找 Full GC 暂停 5s

# 2. 看哪个 service 分配
adb shell logcat -d -s SystemServer:V
# 关键：找哪个 service
```

**解决方案**：
- service 启动时分配小对象
- 大对象延迟到 onBootComplete

**收益**：GC 5s → 100ms

## 7.5 案例 5：AVB BootLoop

**症状**：
- 设备连续重启 5+ 次
- 屏幕卡在 Boot Logo
- 触发工厂重置

**根因**：
- AVB 校验失败
- OEM 触发工厂重置
- 工厂重置又失败 → BootLoop

**排查过程**：
```bash
# 1. 看 AVB 状态
adb shell getprop ro.boot.veritymode
# 异常：veritymode=enforcing

# 2. 看重启历史
adb shell dumpsys bootstat --history | head -50
# 关键：看重启原因
```

**解决方案**：
- 关闭 AVB（OEM 主导）
- 修复 AVB 校验失败
- 工厂重置 + 重刷

**收益**：BootLoop → 正常启动

# 8. dumpsys 怎么取证（与 Dumpsys D11 联动 · 强制）

## 8.1 启动期死锁 4 步取证法

```bash
# 场景：启动卡在 60% 进度

# 异常：system_server 不存在 → 已被 Watchdog 杀

# 步骤 2: 看 SystemServer 卡在哪个 service
adb shell logcat -d -s SystemServer:V | tail -50

# 步骤 3: 看 Watchdog dropbox

# 关键：找卡死的 service

# 步骤 4: 看 AVB 状态
adb shell getprop ro.boot.veritymode
# 异常：veritymode=enforcing + 校验失败
```

# 9. 关键阈值与性能基准

## 9.1 启动期死锁阈值（AOSP 17 不可调）

| 参数 | 阈值 | 含义 |
|:-----|:-----|:-----|
| **Watchdog 周期** | 30s | SystemServer 30s 不响应 = 杀 |
| **BootLoop 阈值** | 5 次 / 5min | 5min 内 5 次重启 = BootLoop |
| **BootLoop 触发** | 5 次 | 触发工厂重置 |
| **AVB 校验失败** | 1 次 | OEM 触发工厂重置 |

## 9.2 4 类死锁判定基线

| 指标 | 优秀 | 良好 | 异常 |
|:-----|:----:|:----:|:----:|
| **SystemServer 启动** | < 5s | < 10s | > 15s |
| **Zygote 启动** | < 1s | < 1.5s | > 3s |
| **servicemanager 启动** | < 100ms | < 200ms | > 500ms |
| **ART 启动** | < 500ms | < 1s | > 1.5s |
| **启动期死锁占比** | < 0.1% | < 1% | > 5% |

## 9.3 4 大根治方案综合收益

| 方案 | 收益范围 | 平均收益 |
|:-----|:---------|:--------:|
| **服务按需启动** | 30-50% | 40% |
| **服务异步化** | 20-30% | 25% |
| **加超时机制** | 10-20% | 15% |
| **Watchdog 监控** | 5-10% | 7% |
| **总收益** | **30-80% 死锁降低** | **50%** |
