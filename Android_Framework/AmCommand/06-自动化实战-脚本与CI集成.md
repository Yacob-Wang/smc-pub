## 1. 一句话定位:把 am 命令升级为"稳定性工具链"

### 1.1 为什么需要 amlib.py

前 5 篇讲了所有 am 命令的"命令级"用法——但稳定性工程师真正想要的不是"敲 10 行命令",而是"一句 Python 代码触发完整流程":

| 痛点 | 裸 adb 命令的局限 | amlib.py 的解决方案 |
|------|------------------|-------------------|
| "每次写冷启动脚本都要复制粘贴 50 行 adb 命令" | 重复造轮子 | `cold_start_check("com.app", wait=5)` 一行调用 |
| "trace 拉回来要手写 hprof-conv + Studio 分析" | 人工处理 | `profile_and_pull()` 自动归档 |
| "CI 里 adb 设备掉线,脚本一直卡死" | 没错误恢复 | 自动重连 + 超时控制 |
| "多人协作脚本风格不统一" | 复制粘贴各改各的 | 统一 SDK 入口 |
| "线上出问题,不知道跑哪个命令" | 命令散落 | 统一的 `forensic_collect()` 三段式保留 |

> **amlib.py 是把"散落的 adb shell 命令"升级为"工程化 Python SDK",**让稳定性脚本从"一次性脚本"变成"可复用的工具集"。

### 1.2 三大自动化场景

```
                    06 篇三大巡检场景
                          │
        ┌─────────────────┼─────────────────┐
        ↓                 ↓                 ↓
   冷启动性能           内存压力           进程稳定性
   (性能维度)           (内存维度)         (崩溃维度)
        │                 │                 │
        ↓                 ↓                 ↓
   am start -W         am dumpheap         am crash
   + am profile        + dumpsys           + am monitor
   + 解析 trace         + hprof-conv        + 拉 tombstone
   + 报告              + MAT 报告          + 归档 dropbox
```

**场景 1 - 冷启动性能巡检**:

- **目标**:每次发版自动测 5 个核心 Activity 的冷启动耗时,劣化超过基线 20% 告警
- **触发**:`am start -W -n <pkg>/<activity>`,**WaitTime** 是核心指标
- **可选**:同步 `am profile` 采 5 秒 trace,定位启动慢的根因函数

**场景 2 - 内存压力巡检**:

- **目标**:模拟压测 10 分钟,定时 dump heap,看对象增长趋势
- **触发**:压测期间每 2 分钟 `am dumpheap`,事后用 MAT / 脚本对比
- **可选**:`am profile` 同期采 trace,关联 CPU 热点和内存分配

**场景 3 - 进程稳定性巡检**:

- **目标**:模拟 1000 次冷启动,捕获 Crash / ANR / Native crash,统计崩溃率
- **触发**:`am start` + `am monitor` + 监测 `Process died` 事件
- **可选**:`am crash` 主动制造一次 Crash,验证崩溃捕获链路是否正常

   ↓              ↓                ↓                ↓                ↓                ↓

## 2. amlib.py 设计:Python SDK 封装

### 2.1 设计原则

| 原则 | 落地方式 |
|------|---------|
| **薄封装** | amlib 不重新发明 adb,只在 `subprocess` 上加错误处理、超时、重试 |
| **幂等** | 每个函数都可重复调用,不会污染设备状态(自己启的 profile 自己关) |
| **自动归档** | 拉回的文件自动按时间戳+场景归档,绝不丢到当前目录 |
| **可恢复** | adb 设备掉线自动重连,profile start 失败自动重试 |
| **人类可读错误** | 异常带根因 + 解决方案 + 文档链接,而不是裸 `subprocess.CalledProcessError` |

### 2.2 模块划分

```
amlib/
├── __init__.py
├── device.py            # 设备管理(adb 设备发现 + 重连 + 超时)
├── am.py                # am 命令封装(profile/dumpheap/crash/hang/monitor 等)
├── artifact.py          # 现场保留(trace/hprof/tombstone 拉取 + 归档)
├── report.py            # 报告生成(Jinja2 模板 + Markdown 输出)
├── exceptions.py        # 异常类(AMError, DeviceOfflineError, ProfileError)
└── utils.py             # 工具函数(adb 命令封装、设备信息采集)
```

**模块依赖图**:

```
device.py ─┬─→ am.py ─┬─→ artifact.py ─┬─→ report.py
           │          │                │
           └──────────┴────────────────┴─→ utils.py
```

### 2.4 与裸 adb 命令的对比

| 场景 | 裸 adb 命令 | amlib.py |

## 3. 巡检场景 1:冷启动性能巡检

### 3.1 巡检 SOP

```
每次发版 / 每日定时
   ↓
1. 选 5 个核心 Activity(Launcher/Mine/Order/Detail/Setting)
   ↓
2. 每个 Activity 测 3 次冷启动,取中位数
   ↓
3. 同步 am profile 采 5 秒 trace,定位启动慢的根因函数
   ↓
4. 对比基线(上次发版 / 历史均值)
   ↓
5. 劣化 > 20% 触发告警 / 失败 / Slack 通知
```

### 3.2 数据流图

```
[CI Trigger]
    ↓
[amlib.Device] ─→ shell getprop(确认设备/版本)
    ↓
[amlib.AM.cold_start_time()] ─→ am start -W -n <pkg>/<activity>
    ↓                              (返回 WaitTime / ThisTime / TotalTime)
    ├─→ [amlib.ArtifactCollector] ─→ trace.trace(自动归档)
    └─→ [amlib.ColdStartReport] ─→ report.md
                                   ↓
                              [Slack / Email / Issue]
```

### 3.3 脚本实现关键点

完整脚本见 `scripts/check_cold_start.py`,核心逻辑:

```python
from amlib.device import Device
from amlib.am import AM
from amlib.artifact import ArtifactCollector
from amlib.report import ColdStartReport

def check_cold_start(pkg: str, activities: list, baseline_ms: int, threshold_pct: float = 20):
    dev = Device()
    am = AM(dev)
    collector = ArtifactCollector(dev, output_dir='./reports/cold_start')

    results = []
    for activity in activities:
        component = f"{pkg}/{activity}"

        # 1. 强制停止,确保冷启动
        am.force_stop(pkg)
        time.sleep(2)

        # 2. 测 3 次取中位数
        timings = []
        for i in range(3):
            t = am.cold_start_time(component)
            timings.append(t['WaitTime'])
            time.sleep(2)
        median_ms = sorted(timings)[1]

        # 3. 劣化检查
        delta_pct = (median_ms - baseline_ms) / baseline_ms * 100

        # 4. 同步采 profile(可选)
        pid = am.get_pid(pkg)
        trace_path = collector.collect_profile(pid, duration_sec=5, scene=f'cold_start_{activity}')

        results.append({
            'activity': activity,
            'median_ms': median_ms,
            'baseline_ms': baseline_ms,
            'delta_pct': delta_pct,
            'status': 'FAIL' if delta_pct > threshold_pct else 'PASS',
            'trace': trace_path,
        })

    # 5. 生成报告
    report = ColdStartReport(results)
    report.render('./reports/cold_start/latest.md')

    return results
```

**关键点**:

- **3 次中位数**:避免冷启动波动(磁盘缓存、GC 抖动等)
- **force_stop 后 sleep 2s**:确保进程完全死掉,避免热启动混入
- **劣化阈值 20%**:经验值——< 10% 是噪声,> 30% 是事故
- **同步 profile**:性能劣化时,自动归档 trace 文件,便于定位根因

## 4. 巡检场景 2:内存压力巡检

### 4.1 巡检 SOP

```
每日定时 / 压测期间
   ↓
1. 启动 app,跑压测 monkey 10 分钟
   ↓
2. 每 2 分钟 dump 一次 heap(共 5 次)
   ↓
3. 同步 am profile 采 trace,关联 CPU 热点
   ↓
4. 对比 5 次 heap 报告,看对象增长趋势
   ↓
5. 单类对象 > 1000 个或总内存 > 200MB 告警
```

### 4.2 数据流图

```
[CI / 定时任务]
    ↓
[am start + monkey 压测 10min]
    ↓
[每 2min: am.dumpheap_and_pull()]
    ├─→ heap_0.hprof(归档 0min)
    ├─→ heap_2.hprof(归档 2min)
    ├─→ heap_4.hprof(归档 4min)
    ├─→ heap_6.hprof(归档 6min)
    └─→ heap_8.hprof(归档 8min)
    ↓
[同步 am profile + 解析 → trace.trace]
    ↓
[对比 5 个 heap 报告 → 增长曲线]
    ↓
[判定:对象增长 / 内存增长 → 告警]
```

### 4.3 脚本实现关键点

完整脚本见 `scripts/check_memory_pressure.py`,核心逻辑:

```python
from amlib.am import AM
from amlib.artifact import ArtifactCollector
from amlib.report import MemoryReport

def check_memory_pressure(pkg: str, duration_sec: int = 600, dump_interval_sec: int = 120):
    dev = Device()
    am = AM(dev)
    collector = ArtifactCollector(dev, output_dir='./reports/memory_pressure')

    # 1. 启动压测
    am.start(pkg, '.MainActivity')
    pid = am.get_pid(pkg)

    # 2. 同步采 profile(全程)
    collector.collect_profile_async(pid, duration_sec=duration_sec, scene='memory_pressure')

    # 3. 每 N 分钟 dump heap
    heaps = []
    for i in range(0, duration_sec, dump_interval_sec):
        time.sleep(dump_interval_sec)
        heap_path = collector.collect_heap(pid, scene=f'memory_t{i}')
        heaps.append((i, heap_path))

    # 4. 解析每个 heap,统计 Top 10 类对象
    snapshots = []
    for time_sec, heap_path in heaps:
        stats = analyze_hprof(heap_path)  # 调用 hprof-conv + Python 解析
        snapshots.append({'time_sec': time_sec, 'stats': stats})

    # 5. 对比增长趋势
    report = MemoryReport(snapshots)
    report.render('./reports/memory_pressure/latest.md')

    # 6. 告警判定
    return check_memory_alerts(snapshots)
```

**关键点**:

- **dump 间隔 2 分钟**:太频繁(30s)会让 app 卡死;太稀疏(5min)又漏掉泄漏点
- **全程采 profile**:`collect_profile_async` 后台跑,关联 CPU 热点和内存分配
- **hprof 解析**:用 `hprof-conv` 转 + Python 解析(或集成 MAT 的脚本化调用)

## 5. 巡检场景 3:进程稳定性巡检

### 5.1 巡检 SOP

```
每日定时 / 发版前
   ↓
1. 启动 app,跑 1000 次冷启动(模拟用户操作)
   ↓
2. 后台 am monitor 监控 Crash / ANR / Process died 事件
   ↓
3. 每次 Crash 立即三段式现场保留(trace + heap + logcat)
   ↓
4. 统计崩溃率 / ANR 率
   ↓
5. 崩溃率 > 0.1% 告警 / 创建 issue
```

### 5.2 数据流图

```
[CI / 定时]
    ↓
[am monitor &] ←─ 后台事件流(GC/Crash/LMK/Process died)
    ↓
[monkey -p <pkg> 1000] ←─ 1000 次冷启动压测
    ↓
[事件检测]
    ├─ Crash detected → [ArtifactCollector.collect_full()] → 归档
    ├─ ANR detected → [拉 /data/anr/] → 归档
    └─ Process died → [拉 tombstone] → 归档
    ↓
[统计报告] → 崩溃率 / ANR 率 / 死亡原因分布
```

### 5.3 脚本实现关键点

完整脚本见 `scripts/check_process_stability.py`,核心逻辑:

```python
from amlib.am import AM
from amlib.artifact import ArtifactCollector
from amlib.report import StabilityReport

def check_process_stability(pkg: str, iterations: int = 1000):
    dev = Device()
    am = AM(dev)
    collector = ArtifactCollector(dev, output_dir='./reports/stability')

    # 1. 启动 am monitor 后台监控
    monitor = am.monitor_start()

    crashes = []
    anrs = []

    # 2. 跑 1000 次 monkey
    monkey_output = am.monkey(pkg, iterations=iterations, throttle_ms=100)

    # 3. 解析 monitor 输出,捕获事件
    for event in monitor.parse_events():
        if event.type == 'crash':
            crashes.append(event)
            # 三段式现场保留
            collector.collect_full(pkg, scene=f'crash_{event.timestamp}', include_profile=False)
        elif event.type == 'anr':
            anrs.append(event)
            collector.collect_anr_traces(scene=f'anr_{event.timestamp}')

    # 4. 停止 monitor
    am.monitor_stop(monitor)

    # 5. 统计报告
    report = StabilityReport(crashes=crashes, anrs=anrs, iterations=iterations)
    report.render('./reports/stability/latest.md')

    return {
        'crash_rate': len(crashes) / iterations * 100,
        'anr_rate': len(anrs) / iterations * 100,
        'crashes': crashes,
        'anrs': anrs,
    }
```

**关键点**:

- **am monitor 后台 + 事件回调**:必须后台跑,否则会和 monkey 抢 stdout
- **三段式现场保留**:每次 crash 立即归档,而不是等压测结束统一处理(避免进程死了 logcat 被覆盖)
- **崩溃率 0.1%**:行业基线,> 0.1% 通常有系统性问题

## 7. 关键坑位图

### 7.1 adb 设备掉线导致脚本卡死

**症状**:`adb shell sleep 30` 中途 USB 断开,Python 脚本卡死 30 秒不退出。

**根因**:`subprocess.run` 默认无超时控制。

**解决**(amlib 已处理):

```python
# amlib 的 Device.shell 默认 30s 超时
result = dev.shell('sleep 30', timeout=5)
# 超时后抛 DeviceTimeoutError,而不是卡死
```

**经验值**:`am profile start` 等操作设 5-10s 超时,`am dumpheap` 等重操作设 30-60s 超时。

### 7.2 多设备并行执行时端口冲突

**症状**:多台设备同时 `adb forward tcp:8000 tcp:8000`,后面的设备覆盖前面的。

**根因**:`adb forward` 是全局的,不是 per-device 的。

**解决**:

```python
# ❌ 错误:固定端口
for dev in devices:
    dev.forward(8000, 8000)  # 冲突!

# ✅ 正确:用设备 serial 区分端口
for dev in devices:
    port = 8000 + hash(dev.serial) % 100
    dev.forward(port, 8000)
```

### 7.3 trace/pull 文件路径污染 CI workspace

**症状**:CI 多次跑后,workspace 里堆满 trace 文件,git pull 失败。

**根因**:trace 文件没归档,直接堆在当前目录。

**解决**(amlib 已处理):

```python
# amlib 默认按时间戳归档到 ./reports/<scene>/<timestamp>/
collector = ArtifactCollector(dev, output_dir='./reports')
collector.collect_profile(...)

# CI 上加 .gitignore
# reports/
# traces/
# *.trace
# *.hprof
```

### 7.4 CI runner 无 adb 权限

**症状**:GitHub Actions 官方 runner 没真机,`adb devices` 空。

**根因**:官方 runner 是云端虚拟机,没物理 USB 接口。

**解决**:

1. **自托管 runner**:在带真机的机器上装 GitHub Actions runner
2. **Device Farm 服务**:用 BrowserStack / Sauce Labs / Firebase Test Lab(云端真机)
3. **Mock 设备**:用 Android emulator(性能差,只适合开发自测)

## 9. 总结:架构师视角的 5 条 Takeaway

1. **amlib.py 是 am 命令的"工程化封装层"**——前 5 篇的命令能力,要靠 SDK + 巡检脚本 + CI 模板才能在生产环境真正落地。**裸 adb 命令是一次性的,SDK 才是可复用的**。

2. **三大巡检场景覆盖稳定性最关键的三个维度**——冷启动(性能)、内存(资源)、进程稳定性(崩溃)。**每天定时跑、每次发版前跑**,稳定性问题不会漏到线上。

3. **CI 集成必须用自托管 runner**——云端 runner 没有 adb 物理设备。**预算一次性硬件投入,换长期自动化收益**——一台真机 + 自托管 runner 能顶半个 QA 工程师。

4. **报告归档 + 告警渠道是闭环**——只跑巡检不够,**必须把结果落到 Slack / Issue / Email**,让团队感知。否则巡检报告躺在 reports/ 目录里,和没跑一样。

5. **多设备并发是规模化的必经之路**——单台设备只能验证"功能在 Pixel 上 OK",多设备并行才能验证"功能在小米/华为/三星上都没问题"。**DevicePool + parallel_run 是关键基础设施**。

**AmCommand 系列 6 篇全部完结**:

```
01 am 全景           入门(命令本质 + AMS 调用栈)
02 进程三件套        稳定性触发(kill/crash/restart)
03 profile           性能触发(采样 trace)
04 dumpheap          内存触发(堆 dump)
05 hang/monitor      诊断触发(ANR + 事件流)
06 自动化实战        工程化(SDK + 巡检 + CI)  ← 系列收官
```

**系列与其他系列的衔接**:

- **Hprof 系列**:`am dumpheap` 是 Hprof 系列的"触发入口"
- **Perfetto 系列**:`am profile` 与 Perfetto atrace 是互补
- **Process 系列**:`am kill` / `am crash` 触发进程死亡,与 Process 系列的死亡链路分析联动
- **Dumpsys 系列**:巡检脚本里大量使用 `dumpsys meminfo` / `dumpsys gfxinfo` 做诊断
- **Tools 系列**:amlib.py 是 Tools 的"工程化案例"
