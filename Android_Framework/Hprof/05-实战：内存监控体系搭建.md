## 1. 背景:从"被动排查"到"主动监控"

### 1.1 现状:90% 团队靠"被动响应"

```
当前大多数团队的内存问题处理流程:
    
  客服/用户报障 ─→ 开发 dump ─→ 找原因 ─→ 修复 ─→ 发布
       ↑                                         │
       └──────────── 下次再出问题 ←───────────────┘
       
特点:
  - 被动:用户先发现
  - 滞后:出问题几天后才看到
  - 重复:同类问题反复出现
```

> **问题**:线上 OOM 已经影响 DAU 了,开发才开始排查。**这是"事故驱动",不是"工程体系"**。

### 1.2 目标:四层防御体系

```
         内存稳定性的"四层防御体系"
         
┌─────────────────────────────────────────┐
│ 第 4 层:回归预防(CI 集成 + LeakCanary 自测) │
├─────────────────────────────────────────┤
│ 第 3 层:智能归因(AI 自动定位 + 自动提单)    │
├─────────────────────────────────────────┤
│ 第 2 层:主动监控(灰度包 LeakCanary + perfetto) │
├─────────────────────────────────────────┤
│ 第 1 层:基础监控(线上 statsd + OOM 报警)     │
└─────────────────────────────────────────┘
         ↑ 越往上,越主动,成本越高
```

**本篇重点**:第 1-2 层(基础 + 主动)。第 3-4 层需要 AI 能力,不在本篇范围。

## 2. 内存监控的整体架构

### 2.1 四层架构:Client / Edge / Server / Dashboard

```
┌─────────────────────────────────────────────────────────────┐
│ 第 1 层:Client(app 进程)                                    │
│  ┌──────────────┬──────────────┬─────────────────┐          │
│  │ LeakCanary   │ statsd       │ perfetto_       │          │
│  │ (debug/灰度) │ (内存指标)   │ heapprofd       │          │
│  └──────┬───────┴──────┬───────┴────────┬────────┘          │
└─────────┼──────────────┼────────────────┼──────────────────┘
          │              │                │
          ↓              ↓                ↓
┌─────────────────────────────────────────────────────────────┐
│ 第 2 层:Edge(数据上报通道)                                   │
│  - HTTPS 上传 hprof / LeakCanary 报告                        │
│  - statsd 推送到 statsd 服务                                  │
│  - perfetto trace 落盘                                      │
└─────────┬──────────────┬────────────────┬──────────────────┘
          │              │                │
          ↓              ↓                ↓
┌─────────────────────────────────────────────────────────────┐
│ 第 3 层:Server(服务端处理)                                   │
│  ┌──────────────┬──────────────┬─────────────────┐          │
│  │ OStorage     │ MemoryMgr    │ Perfetto        │          │
│  │ (hprof)      │ (统计)       │ Analyzer        │          │
│  └──────┬───────┴──────┬───────┴────────┬────────┘          │
└─────────┼──────────────┼────────────────┼──────────────────┘
          │              │                │
          ↓              ↓                ↓
┌─────────────────────────────────────────────────────────────┐
│ 第 4 层:Dashboard(可视化 + 告警)                              │
│  ┌──────────────┬──────────────┬─────────────────┐          │
│  │ 实时大屏     │ 趋势图       │ 告警通知        │          │
│  │ (Grafana)    │ (InfluxDB)   │ (PagerDuty)     │          │
│  └──────────────┴──────────────┴─────────────────┘          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 数据流时序:从客户端检测到 dashboard 展示

```
[Client 进程]                    [Edge]              [Server]            [Dashboard]
     │                              │                    │                     │
     ├─ OOM 触发 ─→ onLowMemory() ─→│                    │                     │
     │                              ├─ HTTPS POST ──────→├─ 存储 hprof         │
     │                              │                    ├─ 解析(LeakCanary)  │
     │                              │                    ├─ 触发告警 ─────────→├─ 飞书/邮件通知
     │                              │                    │                     │
     ├─ statsd 定时(每 60s) ────────→├─ UDP push ────────→├─ 累加到 InfluxDB    │
     │                              │                    │                     ├─ Grafana 实时刷新
     │                              │                    │                     │
     ├─ LeakCanary 触发 ─→ 上传 ────→├─ HTTPS POST ──────→├─ 归类:已知/未知     │
     │                              │                    ├─ 自动提单(Bug)    ─→├─ 飞书
     │                              │                    │                     │
     ├─ perfetto 定时抓取 ──────────→│                    ├─ 落盘 + 解析       │
     │                              │                    │                     ├─ 高级分析视图
     ↓                              ↓                    ↓                     ↓
   < 5-30s 检测                        < 1-5min 告警                          < 实时可视化
```

**关键时延**:
- OOM 触发 → 告警:**1-5 分钟**
- statsd 指标更新:**60 秒**
- LeakCanary 分析 → 归类:**5-10 分钟**

### 2.3 关键技术选型矩阵

| 组件 | 推荐方案 | 备选方案 | 决策依据 |
|------|---------|---------|---------|
| **指标采集** | statsd(原生) | LeakCanary APM | statsd 系统级,无侵入 |
| **Java 泄漏** | LeakCanary(灰度) | Perfetto heapprofd java mode | LeakCanary 成熟,perfetto 进阶 |
| **Native 监控** | perfetto_heapprofd native | simpleperf + malloc hook | perfetto 是 Google 官方演进 |
| **数据上报** | HTTPS POST | mqtt | HTTPS 简单 + 安全 |
| **指标存储** | InfluxDB | Prometheus | 时序场景 InfluxDB 更合适 |
| **可视化** | Grafana | Kibana | Grafana 社区成熟 + 模板多 |
| **告警** | PagerDuty / 飞书机器人 | 邮件 | 飞书适配国内团队 |
| **hprof 存储** | OSS / S3 + 元数据进 MySQL | 内部 MinIO | 成本 + 规模 |

### 3.2 内部灰度包:启用 + 上传

**配套配置模板**见 `hprof_configs/leakcanary_config.gradle`。

```kotlin
// build.gradle.kts
android {
    flavorDimensions += "stage"
    productFlavors {
        create("internal") {
            dimension = "stage"
            applicationIdSuffix = ".internal"
            buildConfigField("boolean", "ENABLE_LEAK_CANARY", "true")
        }
        create("production") {
            dimension = "stage"
            // 默认不启用
        }
    }
}

// internal 包启用 + 上传
dependencies {
    internalImplementation("com.squareup.leakcanary:leakcanary-android:2.14")
    internalImplementation("com.squareup.leakcanary:leakcanary-android-upload-to-file:2.14")
}
```

**自定义上传**:

```kotlin
class LeakCanaryConfig {
    fun setup() {
        LeakCanary.config = LeakCanary.config.apply {
            // 自定义:报告也上传到服务端
            dumpHeap(HeapDumpStrategy.All)
            
            // 上传到内部平台
            onHeapAnalyzedListener = { heapAnalysis ->
                if (heapAnalysis is HeapAnalysisSuccess) {
                    val report = heapAnalysis.toJson()
                    LeakReportUploader.upload(report)  // 自定义上传器
                }
            }
        }
    }
}

object LeakReportUploader {
    fun upload(reportJson: String) {
        // HTTPS POST 到内部平台
        CoroutineScope(Dispatchers.IO).launch {
            val client = OkHttpClient()
            val body = reportJson.toRequestBody("application/json".toMediaType())
            val request = Request.Builder()
                .url("https://apm.example.com/leak/upload")
                .post(body)
                .build()
            client.newCall(request).execute().use { it.close() }
        }
    }
}
```

### 3.3 Release 包:不启用(只接 statsd / 远程配置开关)

```kotlin
// production 包不引入 LeakCanary
dependencies {
    productionImplementation(libs.androidx.core.ktx)  // 不加 leakcanary
}
```

**远程配置开关**(可选):
- 通过远程配置(如 Firebase Remote Config / 自建配置中心)动态开启 LeakCanary
- 紧急 OOM 问题可以远程开 1% 流量验证
- ⚠️ 风险:远程开启需要兜底(开启失败不影响业务)

### 3.4 自定义排除规则(降低误报)

```kotlin
// LeakCanary 自定义排除
class App : Application() {
    override fun onCreate() {
        super.onCreate()
        
        LeakCanary.config = LeakCanary.config.apply {
            // 排除已知系统类(避免误报)
            excludedLeakClasses(
                Toast::class.java,
                Dialog::class.java,
                // 第三方 SDK 类(等待修复)
                "com.example.thirdparty.SdkManager".toClass()
            )
            
            // 自定义引用匹配器(更精细)
            object : ReferenceMatcher {
                override fun matches(ref: Reference): Boolean {
                    return false  // 默认不排除
                }
            }
        }
    }
}
```

**常见排除项**:
- 系统类:`Toast`、`Dialog`、`InputMethodManager`
- 第三方 SDK:`XxxSdk.getInstance()` 持有
- 已知问题:WebView(用独立进程方案)

### 3.5 工程坑位图:接入 5 大问题

| 问题 | 症状 | 解决方案 |
|------|------|---------|
| **Debug 包引入 LeakCanary 后启动慢 3-5s** | 启动 STW | 在 `Application.onCreate` 之后异步初始化 |
| **LeakCanary 误报很多** | 每天 100+ 报告 | 配置 `excludedLeakClasses` + 调高 retained size 阈值 |
| **Release 包接入后 ANR** | 用户卡顿 | 不在 Release 包启用(只 internal 灰度) |
| **LeakCanary dump 让用户卡顿** | 用户反馈 | 默认禁用 dump,只标记 + 后台分析 |
| **hprof 文件太大(>500MB)** | 上传失败 | 配置 `HeapDumpStrategy.All` + 限制最大堆 |

> **关键经验**:**默认配置 + 简单自定义**比"重写 LeakCanary"更稳。LeakCanary 2.14 已经做了很多性能优化,自己改反而容易踩坑。

## 4. 线上 OOM 监控 + hprof 上传链路

### 4.1 OOM 监控:三种触发方式

**方式 1:Application.onLowMemory()**(传统方式)

```kotlin
class MyApplication : Application() {
    override fun onLowMemory() {
        super.onLowMemory()
        // 系统内存低时调用(可能在 OOM 前几秒)
        reportLowMemory()
    }
    
    override fun onTrimMemory(level: Int) {
        super.onTrimMemory(level)
        when (level) {
            TRIM_MEMORY_RUNNING_CRITICAL -> reportLowMemory(level)
            TRIM_MEMORY_COMPLETE -> reportLowMemory(level)
            // ...
        }
    }
}
```

**方式 2:CrashHandler 捕获 OOM**(最稳)

```kotlin
class MyCrashHandler : Thread.UncaughtExceptionHandler {
    override fun uncaughtException(t: Thread, e: Throwable) {
        if (e is OutOfMemoryError) {
            // 上报 + dump hprof
            OomReporter.report(t, e)
        }
        // 继续走默认 handler
        defaultHandler.uncaughtException(t, e)
    }
}
```

**方式 3:Native 监听 LMKD events**(底层)

```bash
# 监听 LMKD 杀进程事件
adb shell setprop persist.debug.lmkd.stats.enable 1
# 然后通过 statsd / logcat 拿到事件
```

### 4.2 远程上传链路:客户端 → 服务端

```
┌────────────────────────────────────────────────────────────┐
│ Client 端(Android app)                                     │
│                                                            │
│  [OOM 触发 / onLowMemory]                                  │
│       ↓                                                    │
│  [CrashHandler 拦截]                                        │
│       ↓                                                    │
│  [1. dump hprof(Debug.dumpHprofData,后台线程)]              │
│       ↓                                                    │
│  [2. 压缩:7zip 最大压缩率,200MB → 30MB]                   │
│       ↓                                                    │
│  [3. 上传:分片上传(每片 5MB,断点续传)]                     │
│       ↓                                                    │
│  [4. 服务端返回 hprof_id]                                  │
│       ↓                                                    │
│  [5. 上报 metadata:device/os/appVersion/场景]              │
└────────────────────────────────────────────────────────────┘
                           ↓ HTTPS
┌────────────────────────────────────────────────────────────┐
│ Server 端                                                  │
│                                                            │
│  [1. 接收 hprof 落 OSS]                                     │
│       ↓                                                    │
│  [2. 接收 metadata 入库 MySQL]                              │
│       ↓                                                    │
│  [3. 触发 LeakCanary / MAT 分析(异步任务)]                  │
│       ↓                                                    │
│  [4. 解析结果入 ES / ClickHouse]                            │
│       ↓                                                    │
│  [5. 触发告警(飞书机器人 / PagerDuty)]                      │
└────────────────────────────────────────────────────────────┘
```

### 4.3 服务端:接收 + 解析 + 告警

**服务端伪代码**(Python + Flask):

```python
@app.route("/leak/upload", methods=["POST"])
def upload_leak():
    metadata = request.form.get("metadata")  # JSON
    hprof_file = request.files["hprof"]
    
    # 1. 落 OSS
    hprof_id = oss.put(f"leak/{metadata['device_id']}/{timestamp}.hprof", hprof_file)
    
    # 2. 入库
    db.execute(
        "INSERT INTO leak_reports (id, device_id, app_version, scene, ...) VALUES (%s, ...)",
        (hprof_id, ...)
    )
    
    # 3. 异步分析
    celery_app.send_task("analyze_hprof", args=[hprof_id])
    
    # 4. 触发告警(如果严重)
    if metadata["retained_size_mb"] > 50:
        send_feishu_alert(metadata, hprof_id)
    
    return {"hprof_id": hprof_id}
```

### 4.4 工程资产:配套脚本(`oom_log_analyzer.sh` / `.ps1`)

**核心功能**:解析 logcat 中的 OOM 日志,提取关键信息。

```bash
# scripts/oom_log_analyzer.sh(Linux/Mac)
#!/bin/bash
# 用法:./oom_log_analyzer.sh <logcat_file>

LOG_FILE=$1
OUTPUT_DIR="oom_analysis_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$OUTPUT_DIR"

echo "=== OOM Log 分析器 ==="
echo "输入文件: $LOG_FILE"
echo "输出目录: $OUTPUT_DIR"
echo ""

# 1. 提取 OOM 异常
grep -E "OutOfMemoryError|Failed to allocate" "$LOG_FILE" > "$OUTPUT_DIR/oome_exceptions.txt"
echo "✓ OOM 异常数: $(wc -l < "$OUTPUT_DIR/oome_exceptions.txt")"

# 2. 提取进程死亡记录
grep -E "Process.*has died.*OOM|kill.*oom_score" "$LOG_FILE" > "$OUTPUT_DIR/oom_kills.txt"
echo "✓ OOM kill 事件数: $(wc -l < "$OUTPUT_DIR/oom_kills.txt")"

# 3. 提取 LMKD 杀进程
grep -i "lowmemorykiller\|lmkd" "$LOG_FILE" > "$OUTPUT_DIR/lmkd_events.txt"
echo "✓ LMKD 事件数: $(wc -l < "$OUTPUT_DIR/lmkd_events.txt")"

# 4. 统计 TOP 10 OOM 异常类型
echo ""
echo "=== TOP 10 OOM 异常类型 ==="
grep -oE "OutOfMemoryError[^:]*" "$LOG_FILE" | sort | uniq -c | sort -rn | head -10

# 5. 统计被杀进程 TOP 10
echo ""
echo "=== TOP 10 OOM kill 进程 ==="
grep -oE "Process [a-zA-Z0-9.]+ has died" "$LOG_FILE" | sort | uniq -c | sort -rn | head -10

echo ""
echo "✓ 分析完成,结果在: $OUTPUT_DIR"
```

> **配套 Windows 版本**:`scripts/oom_log_analyzer.ps1`(同功能)。

### 4.5 工程资产:LeakCanary 报告解析脚本

**Python 脚本**:`scripts/leakcanary_report_parse.py`

```python
#!/usr/bin/env python3
"""
LeakCanary 报告批量解析工具
解析 JSON 报告,提取关键字段,生成汇总表
"""

import json
import sys
from pathlib import Path
from typing import List, Dict
import csv

def parse_leak_report(report_path: Path) -> List[Dict]:
    """解析单个 LeakCanary 报告"""
    data = json.loads(report_path.read_text(encoding="utf-8"))
    
    results = []
    for leak in data.get("leaks", []):
        results.append({
            "report": report_path.name,
            "class": leak.get("className", "Unknown"),
            "retained_size_mb": leak.get("retainedSize", 0) / 1024 / 1024,
            "shallow_size_mb": leak.get("shallowSize", 0) / 1024 / 1024,
            "root_chain": " → ".join(leak.get("referenceChain", [])),
            "is_real_leak": leak.get("retainedSize", 0) > 1_000_000,
        })
    return results

def batch_parse(report_dir: Path, output_csv: Path):
    """批量解析报告目录"""
    all_results = []
    
    for report_file in report_dir.glob("*.json"):
        try:
            results = parse_leak_report(report_file)
            all_results.extend(results)
            print(f"✓ {report_file.name}: {len(results)} 个泄漏")
        except Exception as e:
            print(f"✗ {report_file.name}: {e}")
    
    if not all_results:
        print("无泄漏报告")
        return
    
    # 写 CSV
    with output_csv.open("w", newline="", encoding="utf-8") as f:
        writer = csv.DictWriter(f, fieldnames=all_results[0].keys())
        writer.writeheader()
        writer.writerows(all_results)
    
    # 统计
    real_leaks = [r for r in all_results if r["is_real_leak"]]
    total_retained = sum(r["retained_size_mb"] for r in real_leaks)
    
    print(f"\n=== 汇总 ===")
    print(f"总泄漏数: {len(all_results)}")
    print(f"真泄漏数(retained > 1MB): {len(real_leaks)}")
    print(f"累计 retained: {total_retained:.1f} MB")
    print(f"输出文件: {output_csv}")

if __name__ == "__main__":
    if len(sys.argv) < 3:
        print("用法: leakcanary_report_parse.py <报告目录> <输出CSV>")
        sys.exit(1)
    
    batch_parse(Path(sys.argv[1]), Path(sys.argv[2]))
```

**用法**:
```bash
# 批量解析 LeakCanary 报告
python3 scripts/leakcanary_report_parse.py reports/ summary.csv

# 输出:
# ✓ leakcanary-2026-01-15.json: 3 个泄漏
# ✓ leakcanary-2026-01-16.json: 1 个泄漏
# 
# === 汇总 ===
# 总泄漏数: 4
# 真泄漏数(retained > 1MB): 2
# 累计 retained: 87.3 MB
# 输出文件: summary.csv
```

## 6. 与 perfetto_hprof / statsd 的协同

### 6.1 statsd 内存指标:PSS / Java Heap / Native Heap

**为什么先接 statsd?**
- statsd 是 **系统级**,所有 app 统一采集,无需 app 端 SDK 改动
- perfetto_hprof 需要 app 主动配置,**侵入性更大**
- 先用 statsd 摸底,再用 perfetto_hprof 深挖

**协同模式**:

```
statsd 提供 "是什么"(指标)
    ↓
perfetto_heapprofd 提供 "为什么"(调用栈)
```

### 6.2 perfetto_hprof 联动:从指标到堆栈

**自动触发链路**:

```python
# 服务端:statsd 检测到异常 → 自动触发 perfetto_hprof
def on_memory_alert(alert: Alert):
    if alert.level == "P1" and alert.is_native_growth:
        # 触发 perfetto_heapprofd
        trigger_perfetto_heapprofd(
            process_name=alert.process_name,
            duration_ms=300000,  # 5 分钟
            sampling_interval_bytes=2048
        )
        # 5 分钟后拉 trace,SQL 查询 native 增长热点
```

**对应配置**(`hprof_configs/perfetto_hprof.pbtxt`):

```protobuf
duration_ms: 300000  # 5 分钟
data_sources: {
  config: {
    name: "android.heapprofd"
    heapprofd_config: {
      sampling_interval_bytes: 2048
      process_dump_config: {
        process_name: "com.example.app"
        sampling_interval_bytes: 2048
      }
    }
  }
}
```

### 6.3 statsd 推送规则配置

**详细配置参考 [Perfetto 系列 03 §3](../../Perfetto/03-Perfetto与statsd联动机制.md)**,本篇不重复。

## 9. 总结:架构师视角的 5 条 Takeaway

### Takeaway 1:四层防御体系是目标
- 第 1 层:statsd 基础监控(看到)
- 第 2 层:LeakCanary 主动监控(找到)
- 第 3 层:AI 智能归因(自动)
- 第 4 层:CI 回归预防(拦截)

### Takeaway 2:LeakCanary 不在 Release 包启用
- Debug 默认 / Internal 灰度 / Production 不接
- 远程配置开关做应急,但不是常态

### Takeaway 3:hprof 上传链路要"分片 + 压缩 + 异步"
- hprof 文件大(50-500MB),必须压缩 + 分片
- 异步上传不阻塞业务

### Takeaway 4:perfetto_hprof 是 Native 监控的"必备"
- 传统 hprof 看不见 native,perfetto_heapprofd 补齐
- statsd 触发 → perfetto 联动 是标准模式

### Takeaway 5:体系化 > 工具化
- 单工具 = 救火
- 体系化 = 主动防御 + 自动化 + 数据驱动
