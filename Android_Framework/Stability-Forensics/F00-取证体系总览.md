# 1. 背景与定义

## 1.1 Forensics 是 Stability 的"取证侧"——**机制 + 取证双视角**

> **一句话定义**：Forensics 系列 = 把 [Stability S00-S07](../Stability/) 讲的"症状机制"反向落到"取证抓取"——症状触发后，dump 文件在哪里、怎么抓、怎么解读。

**双视角对应**：

| 视角 | 系列 | 核心问题 |
|:-----|:-----|:---------|
| **机制** | [Stability](../Stability/) | 症状怎么发生？怎么修？|
| **取证** | **Forensics（本系列）** | 症状发生后怎么抓证据？怎么分析？|

> **所以呢**：**两个系列读完后，稳定性治理全栈闭环**——Stability 给你"症状是什么"，Forensics 给你"症状怎么取证"。

## 1.2 为什么要建取证体系

**3 个核心价值**：

1. **30 秒拿到 dump**：稳定性 oncall 最宝贵的是时间，取证体系把"找 dump"压缩到 30 秒
2. **避免 dump 丢失**：很多 dump 文件**满后会被覆盖**（anr traces 5 个 / tombstone 10 个），不主动采集会丢
3. **cascade 推演**：1 个根因触发 4 类症状（见 [Stability S00 §2.3](../Stability/S00-稳定性症状总览.md)），取证链路也是 cascade 的——多个 dump 文件相互印证

> **所以呢**：**架构师必须建取证体系**——oncall 群一响，30 秒内知道"在哪个路径下抓什么文件"。

## 1.3 3 个常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "看 logcat 就够了" | logcat 是**应用层 log**，**看不到 native / kernel / 异步线程** | 按 §4 矩阵针对性抓 |
| "我把 dump 都抓了" | **dump 文件满后会被覆盖** | 主动采集 + 上传云端 |
| "重启后 log 还在" | **dmesg 重启丢失**（除非 pstore） | 重启后立即抓 last_kmsg / pstore |

# 3. 取证层级（横向 7 层）

> **核心洞察**：Android 取证体系分 7 层，每层有专属工具和 dump 文件。

```
┌─────────────────────────────────────────────────────────────────────┐
│  Level 1: 应用层日志（dropbox / traces / logcat -b crash）         │
│  ├─ dropbox(APP_*) ← 统一存储层                                     │
│  ├─ anr traces.txt（ANR 专用）                                       │
│  ├─ logcat -b crash（crash ring buffer）                            │
│  └─ 路径：/data/system/dropbox/ + /data/anr/ + logcat ring buffer    │
├─────────────────────────────────────────────────────────────────────┤
│  Level 2: Native 层日志（tombstone / debuggerd）                  │
│  ├─ tombstone_NN（NE 触发后生成）                                    │
│  ├─ debuggerd 信号处理                                              │
│  └─ 路径：/data/tombstones/                                          │
├─────────────────────────────────────────────────────────────────────┤
│  Level 3: Kernel 层日志（dmesg / pstore / last_kmsg）             │
│  ├─ dmesg（运行时 ring buffer）                                      │
│  ├─ pstore（持久化）                                                │
│  ├─ last_kmsg（重启前最后一次）                                      │
│  └─ 路径：/dev/kmsg + /sys/fs/pstore/ + /proc/last_kmsg             │
├─────────────────────────────────────────────────────────────────────┤
│  Level 4: 全栈追踪（Perfetto / systrace / ftrace）                 │
│  ├─ Perfetto trace（user + kernel 全栈）                            │
│  ├─ systrace（Perfetto 旧版）                                       │
│  ├─ ftrace（kernel 函数级）                                         │
│  └─ 路径：/data/local/traces/ + /sys/kernel/debug/tracing/         │
├─────────────────────────────────────────────────────────────────────┤
│  Level 5: 内存取证（hprof / smaps / proc/meminfo）                 │
│  ├─ hprof（Java 堆快照）                                            │
│  ├─ smaps（Native 内存映射）                                        │
│  ├─ proc/meminfo（系统内存）                                        │
│  └─ 路径：/data/misc/heap-dump/ + /proc/meminfo                     │
├─────────────────────────────────────────────────────────────────────┤
│  Level 6: 全量 dump（bugreport）                                   │
│  ├─ bugreport（包含上述所有 + dmesg + logcat 全量）                │
│  └─ 路径：adb bugreport > xxx.zip                                   │
├─────────────────────────────────────────────────────────────────────┤
│  Level 7: 商业化（APM / Sentry / Backtrace.io 接入）                │
│  ├─ Sentry / Bugsnag / Backtrace.io 符号化服务                       │
│  ├─ 自研 APM（自动采集 + 上报）                                      │
│  └─ 路径：云端（无本地路径）                                        │
└─────────────────────────────────────────────────────────────────────┘

图 3.1：取证 7 层结构
```

> **架构师视角**：
> - **Level 1-3 是症状自带的 dump**（触发后自动生成）
> - **Level 4-5 是主动抓的**（HANG 必备）
> - **Level 6 是兜底**（bugreport 全量）
> - **Level 7 是商业化**（推荐但非必须）

# 4. 症状 × 抓取矩阵（核心 · 二维交叉）

> **本篇最核心的内容**——把 7 大症状 × 7 类日志 = 49 个交叉点列清楚。

| 症状 \\ 日志层 | Level 1: dropbox | Level 1: traces | Level 2: tombstone | Level 3: kernel | Level 4: 全栈 | Level 5: 内存 | 关键路径 |
|:--------------|:-----------------|:----------------|:-------------------|:-----------------|:--------------|:--------------|:---------|
| **ANR** | `dropbox(APP_ANR)` | `anr traces.txt` | — | — | Perfetto 主动抓 | — | `/data/anr/` + `/data/system/dropbox/` + `/data/local/traces/` |
| **SWT** | `dropbox(SYSTEM_SERVER_WATCHDOG)` | `watchdog traces` | — | — | SystemServer Perfetto（AOSP 17 自动）| — | `/data/anr/` + `/data/system/dropbox/` + `/data/local/traces/` |
| **JE** | `dropbox(APP_CRASH)` | — | — | — | — | — | `/data/system/dropbox/` + logcat -b crash |
| **NE** | `dropbox(SYSTEM_TOMBSTONE)` | — | `tombstone_NN`（16 段）| — | — | — | `/data/tombstones/` + `/data/system/dropbox/` |
| **KE** | `dropbox(KE_*)` | — | — | `dmesg` + `pstore` + `last_kmsg` | ftrace 主动抓 | — | `/sys/fs/pstore/` + `/proc/last_kmsg` + `/dev/kmsg` |
| **HANG** | — | — | — | — | **systrace / Perfetto 主动抓** | — | `/data/local/traces/`（**主动抓**）|
| **OOM** | `dropbox(LOW_MEM)` | — | — | — | — | `hprof` + `smaps` + `proc/meminfo` | `/data/misc/heap-dump/` + `/proc/meminfo` |

> **所以呢**：
> - **ANR / SWT / NE / JE** 都有**自动 dump**（触发即生成）
> - **HANG 没有自动 dump**（必须主动抓，详见 F06）
> - **KE 触发即整机重启**（必须靠 pstore / last_kmsg 持久化）

## 4.1 抓取优先级速查（**30 秒归类**）

```
看到症状
  ↓
1. logcat 关键字判断
  ├─ `am_anr` → ANR → F01
  ├─ `AndroidRuntime` / `FATAL EXCEPTION` → JE → F03
  ├─ `DEBUG` / `tombstone written to` → NE → F04
  ├─ `Watchdog ... KILLING` → SWT → F02
  └─ 无关键字（用户报"卡"）→ HANG → F06
  ↓
2. dump 文件路径
  ├─ /data/anr/ → ANR / SWT traces

  ↓
3. 主动抓
  ├─ ANR/SWT：adb pull + Perfetto
  ├─ HANG：systrace / ftrace
  └─ OOM：hprof + smaps
  ↓
4. 兜底
  └─ adb bugreport（含所有）
```

# 5. 取证路径 4 步法（**架构师必修**）

> **核心**：任何症状的取证都遵循 **触发 → 抓取 → dump 路径 → 解读** 4 步法。

| 步骤 | 关键 | 工具 |
|:-----|:-----|:-----|
| **第 1 步：触发** | logcat 关键字确认症状类型 | logcat / dropbox |
| **第 2 步：抓取** | 按 §4 矩阵对应路径抓 dump 文件 | adb pull / dumpsys dropbox / adb bugreport |
| **第 3 步：dump 路径** | 确认 dump 文件存在 + 完整性 | ls / stat / wc -l |
| **第 4 步：解读** | 按症状对应文章解读 | F01-F07 |

> **所以呢**：**取证 4 步法 = 触发 + 抓取 + 路径 + 解读**——本系列 F01-F07 讲"如何解读"，F00 讲"如何走通 4 步"。

## 5.1 4 步法示例（ANR）

| 步骤 | 操作 | 输出 |
|:-----|:-----|:-----|
| **第 1 步：触发** | logcat 看到 `am_anr` | 确认 ANR 触发 |
| **第 2 步：抓取** | `adb pull /data/anr/anr_*.txt` | anr traces 文件 |
| **第 3 步：dump 路径** | `ls -la /data/anr/` | 确认文件存在 |
| **第 4 步：解读** | 看主线程栈 → 找阻塞点 | 详见 F01 |

## 5.2 4 步法示例（KE 整机重启）

| 步骤 | 操作 | 输出 |
|:-----|:-----|:-----|
| **第 1 步：触发** | 整机突然重启 | 无 logcat（已丢失）|
| **第 2 步：抓取** | **重启后立即** `cat /proc/last_kmsg` | last_kmsg 文件 |
| **第 3 步：dump 路径** | `cat /sys/fs/pstore/dmesg-ramoops-*` | pstore 持久化 log |
| **第 4 步：解读** | 找 panic / Watchdog / hung_task 关键字 | 详见 F05 |

# 6. 风险地图

## 6.1 dump 文件丢失风险

| 文件 | 路径 | 默认保留 | 丢失风险 |
|:-----|:-----|:---------|:---------|
| anr traces | `/data/anr/` | 5 个 | **高**（满后覆盖）|
| watchdog traces | `/data/anr/` | 几个 | **高** |
| tombstone | `/data/tombstones/` | 10 个 | **高** |
| dropbox(APP_*) | `/data/system/dropbox/` | 7-30 天 | **中**（按 tag 策略覆盖）|
| dropbox(SYSTEM_*) | `/data/system/dropbox/` | 30 天 | **中** |
| hprof | `/data/misc/heap-dump/` | 1-3 个 | **高**（大文件，易覆盖）|
| Perfetto trace | `/data/local/traces/` | 几 MB | **中**（重启可能丢失）|
| dmesg | `/dev/kmsg` | ring buffer | **极高**（**重启丢失**）|
| pstore | `/sys/fs/pstore/` | 厂商定制 | **低**（持久化）|
| last_kmsg | `/proc/last_kmsg` | 1 个 | **中**（需编译时开）|

> **所以呢**：**dmesg 重启丢失**——KE 触发后**立即抓** last_kmsg / pstore。

## 6.2 取证失败常见原因

| 失败原因 | 后果 | 修复模式 |
|:---------|:-----|:---------|
| **dump 满后被覆盖** | 高发期丢关键 dump | 主动采集 + 上传云端 |
| **dmesg 重启丢失** | KE 触发后无 log | 开启 pstore + last_kmsg |
| **tombstone 未符号化** | NE 排查效率低 | 接入商业符号化服务 |
| **dropbox 标签写错** | 抓不到对应症状 | 查 Reference/术语表.md |
| **last_kmsg 编译时未开** | 重启前 log 丢失 | 编译时开启 last_kmsg |
| **hprof 未触发** | OOM 无内存快照 | 主动 am dumpheap |

# 8. 实战案例

> 案例引用规则（v4 §4 #8 案例可验证性）：每个案例必须含 logcat / dmesg / systrace 片段 + 环境版本 + 复现步骤 + **取证路径 4 步法** + 修复 diff。

## 8.1 案例 A（CASE-FORENSICS-00-01）：cascade 链路完整取证 4 步法

> **类型**：典型模式
>
> **环境**：AOSP 14.0.0_r1 / Kernel 5.10 / 设备 Pixel 6（**AOSP 17 / K 6.12 验证版准备中**）
>
> **症状**：用户报"App 卡死 10 秒后系统重启"
>
> **根因**：binder 远端服务死锁 → cascade 触发 ANR → SWT → REBOOT
>
> **取证路径**：4 步法完整还原

### 现象

```
用户操作：
  T+0s   用户点击"打开相机"按钮
  T+2s   App 切换到相机预览（黑屏）
  T+5s   App 弹"应用无响应"
  T+10s  系统开始卡顿（触摸无响应）
  T+30s  屏幕黑屏
  T+35s  设备自动重启
  T+60s  开机完成
```

### 取证 4 步法

**第 1 步：触发（logcat 关键字）**

```logcat
# logcat 关键字 → 归类到 ANR
05-15 10:23:45.123  1000  1234  1234 E ActivityManager: ANR in com.example.camera (input dispatching timed out)

# logcat 关键字 → 归类到 SWT
05-15 10:24:15.000  1000  1000  1010 W Watchdog: *** WATCHDOG KILLING SYSTEM PROCESS: Blocked in handler on ActivityManager

# logcat 关键字 → 归类到 REBOOT
05-15 10:24:20.000  1000  1000  1010 I SystemServer: System server died, restarting
```

**第 2 步：抓取（按 §4 矩阵对应路径）**

```bash
# ANR traces 抓取（F01）
adb pull /data/anr/anr_*.txt

# dropbox 抓取（F01/F02）
adb shell dumpsys dropbox --print > dropbox.log

# 整机重启后立即抓 last_kmsg（F05）
adb shell cat /proc/last_kmsg > last_kmsg.log

# pstore 抓取（F05）
adb shell cat /sys/fs/pstore/dmesg-ramoops-* > pstore.log
```

**第 3 步：dump 路径（确认完整性）**

```bash
$ ls -la /data/anr/
-rw------- 1 system system 152K 2026-07-15 10:24 anr_2026-07-15_10-24-15

$ ls -la /data/system/dropbox/ | head -5
-rw-rw---- 1 system system 8921 2026-07-15 10:24 APP_ANR@1709123456789.txt
-rw-rw---- 1 system system 12450 2026-07-15 10:24 SYSTEM_SERVER_WATCHDOG@1709123456789.txt

$ ls -la /sys/fs/pstore/
-rw-r--r-- 1 root root 65536 2026-07-15 10:24 dmesg-ramoops-0
```

**第 4 步：解读（按症状对应文章）**

| dump 文件 | 关键信息 | 解读文章 |
|:---------|:---------|:---------|
| anr traces | 主线程栈在 `BinderProxy.transact` | [F01 §5 解读](../Stability/S01-ANR.md) |
| dropbox(APP_ANR) | ANR 触发时间 + 进程 | [F01 §3 抓取链路](F01-ANR取证.md) |
| dropbox(SYSTEM_SERVER_WATCHDOG) | SystemServer 杀进程决策 | [F02 §3 抓取链路](F02-SWT取证.md) |
| last_kmsg | kernel panic / Watchdog 触发 | [F05 §3 抓取链路](F05-KE取证.md) |
| pstore | 持久化 kernel log | [F05 §3 抓取链路](F05-KE取证.md) |

> **根因**：5 个 dump 文件相互印证 → **单一根因：Camera HAL 阻塞** → cascade 触发 ANR → SWT → REBOOT

### 修复

**短期**：
```java
// 业务加 timeout
private void openCameraAsync() {
    Future<?> future = executor.submit(() -> cameraDevice.openCamera(...));
    try {
        future.get(2, TimeUnit.SECONDS);  // 2s 超时
    } catch (TimeoutException e) {
        future.cancel(true);
        showRetryDialog();
    }
}
```

**长期**：
- 在 CameraService 加 binder call 超时保护
- 接入 Sentry 商业符号化服务（F07）
- HANG 主动监控（F06）

### 验证

1. 复现：低性能设备 + 高频相机调用
2. 观察：cascade 链路（ANR → SWT → REBOOT）是否触发
3. 应用 hotfix：再跑 100 次，无 cascade
4. APM：binder call 失败率 < 0.1%

# 9. 总结

## 9.1 架构师视角 5 条 Takeaway

1. **Forensics 是 Stability 的"取证侧"**：机制 + 取证双视角 = 稳定性治理全栈闭环。
2. **症状 × 日志类型 = 49 交叉点**：7 大症状 × 7 类日志，不是每格都有内容（详见 §4 矩阵）。
3. **取证 4 步法**：触发 → 抓取 → dump 路径 → 解读。
4. **HANG 是特殊的**：没有自动 dump，必须主动抓（详见 F06）。
5. **dmesg 重启丢失**：KE 触发后立即抓 last_kmsg / pstore。
