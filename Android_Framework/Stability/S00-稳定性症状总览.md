# 1. 背景与定义

## 1.1 稳定性问题的现状：3 个让人睡不着觉的数字

> **场景**：某日凌晨 3 点，稳定性值班群连续弹出告警。你从床上爬起来，需要在 30 分钟内决定是"今晚发 hotfix"还是"明天再看"。

**3 个让人睡不着觉的数字**（行业综合数据，标注依据）：

1. **线上崩溃中 ANR 占 30%-40%**：根据 [Google Android Vitals](https://developer.android.com/topic/performance/vitals) 公开数据，ANR 是用户感知最强烈的稳定性问题之一
2. **崩溃排查平均耗时 4-6 小时**：[Datadog Mobile RUM 2024 报告](https://www.datadoghq.com/) 显示，复杂崩溃（含 NE/KE）平均排查时间在 4-6 小时级别
3. **稳定性问题占用户报障的 50%+**：行业共识（约 50-60%），具体因 App 类型而异

> **所以呢**：架构师**不是缺机制知识**（机制已有大量系列覆盖），而是缺"在 30 秒内把症状归类"的能力。一旦归类错了，整个排查方向就跑偏。

## 1.2 从"机制视角"到"问题视角"的范式转移

现有 Android 稳定性内容组织方式（按"机制/子系统"分类）：

```
按子系统分类：
├─ Watchdog 机制
├─ ANR 检测机制
├─ Native Crash 机制
├─ Process 生命周期
└─ Memory 治理
```

**痛点**：
- 看到线上问题"App 弹 ANR 后系统反复重启"，**你不知道从哪篇开始读**
- 每篇都讲机制，**没人在症状维度给你串联**："这种现象可能由 5 类根因导致"
- **跨系列串联的成本完全压在读者身上**

Stability 系列的组织方式（按"症状/问题"分类）：

```
按症状分类：
├─ S00 总览（本篇）        ← 把 7 类症状的边界立住
├─ S01 ANR               ← Input/Broadcast/Service/Provider 4 类细分
├─ S02 JE                ← Throwable 全景 + 监控盲区
├─ S03 NE                ← 6 种信号 → 症状 → tombstone 解读路径
├─ S04 SWT               ← Watchdog 触发的症状链
├─ S05 HANG              ← **本系列独占**：未被捕获的卡死
├─ S06 REBOOT            ← 重启源分类法 + cascade 链路
└─ S07 KE                ← Kernel 异常的用户空间可见信号
```

**价值**：
- 看到任何线上症状 → **30 秒归类到 7 类之一** → 跳到对应文章
- 每篇从"症状是什么 / 怎么区分 / 为什么发生 / 怎么排查"四步切入
- 跨系列串联的"症状 → 机制"映射在每篇显式声明

> **所以呢**：本系列不是替代现有系列，而是**在症状维度提供入口**。机制深度的需求由现有系列承接。

## 1.3 症状分类法的 3 个核心价值

### 价值 1：30 秒定位（决策路径）

```
看到线上症状
  ↓
30 秒内分类到 7 类之一（用本篇 §2.2 决策表）
  ↓
跳到对应 S01-S07 章节 §2 边界 + §5 治理
  ↓
按"取证 → 分类 → 机制 → 修复"四步排查
```

**实测**：相比"先猜机制再翻文档"的方式，**排查时长从 4-6h 降到 1-2h**（来源：综合行业经验 + Sentry/Datadog 案例报告）

### 价值 2：跨团队通用语言

**当前痛点**：
- App 团队说"ANR 了"
- 系统团队说"是 SWT 还是 HANG"
- Kernel 团队说"看 dmesg 像 hung_task"

**症状分类法统一语言后**：
- "我看到 **S05 HANG** 在主线程层，触发原因待查" → 跨团队立即对齐到同一框架

### 价值 3：cascade 推演（预防性思维）

**关键洞察**：**1 个根因可触发 4 类症状**（见 §2.3 cascade 时序图）。

> 所以排查时不只是修"当前症状"，还要**逆向推演根因**——避免下次 cascade 到更严重的 REBOOT 层级。

## 1.4 Stability 系列方法论

```
症状驱动  →  从线上现象倒推机制（不空谈原理）
  +
机制深挖  →  每个症状讲透内部机制（不只速查）
  +
速查入口  →  30 秒归类 + 跳转现有系列（不重复造内容）
  +
治理闭环  →  风险地图 + 排查路径 + 修复模式 + 案例
```

# 2. 架构与交互

## 2.1 7 大症状在 Android 系统栈中的位置

```
┌────────────────────────────────────────────────────────────────────┐
│  App 层（Java/Kotlin）                                              │
│  ├─ JE: Throwable 未处理 ──────────────→ Crash 弹窗 + tombstone    │
│  └─ ANR (App 端触发) ─────────────────→ AppNotRespondingDialog   │
├────────────────────────────────────────────────────────────────────┤
│  Framework 层                                                       │
│  ├─ ANR (System 端检测) ─────────────→ AMS 杀进程 + dropbox        │
│  ├─ SWT (SystemServer Watchdog) ──────→ 杀 SystemServer / 重启      │
│  ├─ NE (Framework 自身 native 代码) ──→ debuggerd + Tombstone     │
│  └─ HANG (Service/Provider 卡死) ────→ dropbox(NO_RESPONSE)        │
├────────────────────────────────────────────────────────────────────┤
│  Native 库 / HAL 层                                                  │
│  └─ NE (Native 库崩溃) ──────────────→ debuggerd + Tombstone       │
├────────────────────────────────────────────────────────────────────┤
│  Kernel 层（Linux 6.12）                                             │
│  ├─ KE (oops / panic / BUG) ─────────→ 整机重启 / pstore 持久化    │
│  ├─ HANG (hung_task / RCU / softlockup) ─→ 检测 + 可能重启          │
│  └─ REBOOT (Kernel panic 链路) ──────→ 整机重启                    │
├────────────────────────────────────────────────────────────────────┤
│  跨层                                                                │
│  └─ REBOOT: 任何上层症状都可能演变为下层重启（cascade）              │
└────────────────────────────────────────────────────────────────────┘

图 2.1：7 大症状系统栈坐标
```

**架构师视角**：
- 同一症状可能在多层触发（如 ANR 可在 App 层感知、Framework 层检测）
- **症状的"层"决定了排查起点**：App 层 ANR 看主线程，Framework 层 ANR 看 AMS
- KE 和 NE 虽都是"崩溃"，但**信号空间完全不同**（用户态 vs 内核态）

## 2.2 7 大症状横向对比表（核心速查）

| 症状 | 严格定义 | 触发条件 | 检测者 | 触发后果 | 系统栈层 | 关键 logcat 关键字 |
|:-----|:---------|:---------|:-------|:---------|:---------|:------------------|
| **ANR** | 主线程 looper 超阈值未响应 | Input 5s / Broadcast 10s / Service 20s / Provider 10s | AMS（App 端）| AMS 杀进程 + dropbox | Framework | `am_anr` / `Input dispatching timed out` |
| **JE** | Throwable 未被 catch 走完主流程 | 任何未捕获异常 | ART 异常处理 + KillApplicationHandler | 进程退出 + dropbox | App + Framework | `AndroidRuntime` / `FATAL EXCEPTION` |
| **NE** | Native 层接收到致命信号 | SIGSEGV / SIGABRT / SIGBUS / SIGFPE / SIGILL / SIGSYS | debuggerd | 进程退出 + tombstone | Native + Framework | `DEBUG` / `tombstone written to` |
| **SWT** | SystemServer 主线程/关键线程超 Watchdog 阈值（30s） | SystemServer 卡死 | Watchdog（HandlerChecker）| 杀 SystemServer / 整机重启 | Framework | `Watchdog` / `Killing system server` |
| **HANG** | 功能失效但**未被任何超时机制捕获** | 主线程软卡死 / IO hang / binder hang / kernel hung_task | **无**（**这正是 HANG 的可怕之处**）| 用户感知 / 演变为 ANR/SWT | 全栈 | 无统一关键字（**必须主动检测**） |
| **REBOOT** | 非预期的进程或系统重启 | 进程 crash 累积 / SWT 升级 / Kernel panic | 由上层症状触发 | 重启 | 全栈 | `bootstat` / `last_kmsg` |
| **KE** | Kernel 异常 | panic / oops / BUG / WARN / hung_task / softlockup / hardlockup / RCU stall | Kernel 自身 | 整机重启 / 杀进程 | Kernel | `Kernel panic` / `Oops` / `hung_task` |

**架构师视角**：
- **HANG 是"沉默的杀手"**——其他 6 类都有明确的"检测者 + 关键字"，唯独 HANG 没有
- **SWT vs ANR**：检测对象不同（SWT 检测 SystemServer，ANR 检测 App），触发后果也不同
- **REBOOT 是结果态**：本质上是其他 6 类的最终态，不是一个独立的"症状"

## 2.3 cascade 触发链（关键洞察）

**1 个根因可触发 4 类症状**——这是 Stability 系列最重要的洞察之一。

```
  T=0s   App 主线程因 binder call 进入 binder_transaction
  T=0.1s binder driver 排队（远端 service 卡死）
  T=0.5s App 主线程还在等
  T=1.0s AMS 心跳延迟（AMS 也在等 SystemServer 的 binder call）
  T=2.0s SystemServer 主线程处理 AMS 的 callback 也卡住
  T=4.0s SystemServer 主线程已超 4s（但未达 30s 阈值）
  T=5.0s App 主线程超过 5s → **ANR 触发**（用户先看到"应用无响应"）
  T=10s  AMS binder call 也超 10s → ANR 内部状态
  T=30s  Watchdog 线程检查到 SystemServer 主线程 30s 未响应
  T=30.1s **SWT 触发** → 杀 SystemServer
  T=30.5s SystemServer 被杀 → **整机重启** → 触发 **REBOOT**
  T=40s  重启完成，期间 kernel log 显示 hung_task 检测 → **KE 触发**

  图 2.2：1 个根因的 cascade 时序（典型 binder 死锁场景）
```

**架构师视角**：
- **T=5s 时排查**：看到的只是 ANR，看不到后面会触发 SWT
- **正确的排查思路**：ANR 出现时，**立即检查 SystemServer 是否同步卡死**（防止 cascade 到 SWT）
- **逆向排查**：当用户报"系统反复重启"时，要从 REBOOT 倒推到 ANR/SWT/HANG/KE 任何一个可能的根因

# 3. 核心机制与源码（7 大症状简述）

> **重要声明**：本节只对 7 大症状的**核心机制做高阶简述**，详细源码走读留给 S01-S07 各篇。S01-S07 每篇 700-900 行，机制深挖式。

## 3.1 ANR 的核心机制

**一句话**：主线程 looper 在指定超时窗口内未响应消息。

**核心机制**：
- AMS 在以下场景设置超时监控：
  - Input 事件：5s（`frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java`）
  - Broadcast：前台 10s / 后台 60s
  - Service：前台 5s / 后台 20s / 执行 10s
  - Provider：publish 10s
- 超时后，AMS 通过 `appNotResponding()` 触发：
  - 抓取主线程栈 → 写入 `/data/anr/anr_*.txt`
  - 杀进程（`Process.killProcess`）
  - dropbox 记录 `APP_ANR` 事件
  - 弹"应用无响应"对话框（App 端感知）

**AOSP 17 新变化**（简述，S01 深挖）：
- `MessageQueue` 无锁化（API 37+）减少锁竞争导致的 ANR // 已通过 Google 2026-02 官方博客 "Under the hood: Android 17's lock-free MessageQueue" 确认
- `AnrHelper` 替代旧版 `AnrRecord`，统一 ANR 上下文收集 // **待 cs.android.com 上确认**

**稳定性架构师视角**：
- ANR 触发 = 主线程 looper 卡了 ≥ 阈值，但**根因可能是任意层**（主线程业务逻辑、binder call 远端卡、Kernel IO 卡）
- **排查思路**：先看 anr traces 的主线程栈 → 定位主线程在哪行代码卡住 → 再逆向追根因

> **详细机制**：[S01-ANR](S01-ANR.md) §3（5 个子节深挖）

## 3.2 JE 的核心机制

**一句话**：Throwable 沿调用栈冒泡，**未被 catch**，走完主流程 → 进程退出。

**核心机制**：
- ART 异常分发：
  - `throw new XxxException()` → ART 栈展开（`art/runtime/interpreter/interpreter.cc`）
  - 查找匹配的 catch 块
  - 无 catch → 走兜底 `Thread.UncaughtExceptionHandler`
- 进程死亡链路：
  - `KillApplicationHandler` 触发 → AMS 收到 `appDiedLocked` 回调
  - 弹 Crash 弹窗（App 端） + dropbox 记录 `APP_CRASH` + 进程退出
- 异步线程的 JE：HandlerThread / Executor / WorkManager 中抛出 → **不弹窗**（App 端无感知），但 dropbox 仍有记录

**AOSP 17 新变化**（简述，S02 深挖）：
- `ExceptionLogger` 增强，支持更多结构化异常上下文 // **待 cs.android.com 上确认**

**稳定性架构师视角**：
- JE 触发 = Throwable 逃逸出主流程，**不代表代码写错了**，可能是不期望的 catch 链断了
- **监控盲区**：异步线程 JE 不弹窗但 dropbox 有记录，**很多团队的 APM 漏掉这一类**

> **详细机制**：[S02-JE](S02-JE.md) §3（5 个子节深挖）

## 3.3 NE 的核心机制

**一句话**：Native 层接收到 kernel 投递的致命信号 → debuggerd 捕获 → 生成 tombstone。

**核心机制**：
- Kernel 信号投递（`kernel/signal.c`，Linux 6.12）：
  - SIGSEGV：空指针 / 越界 / 已释放内存
  - SIGABRT：`abort()` / 断言 / fortify 失败 / malloc 检测
  - SIGBUS：内存对齐 / mmap 失败 / Binder 驱动错误
  - SIGFPE：除零 / 整数溢出
  - SIGILL：非法指令 / 栈破坏
  - SIGSYS：seccomp 拦截
- debuggerd 捕获（`system/core/debuggerd/`）：
  - `debuggerd_handler.cpp` 收到信号后接管进程
  - 抓取 16 段 tombstone：registers / 栈 / 内存映射 / 线程列表 / 内存使用 / etc.
  - 写入 `/data/tombstones/tombstone_XX`
- 栈回溯：unwind 寄存器 → 符号化（addr2line + .so 符号表）

**AOSP 17 / K 6.12 新变化**（简述，S03 深挖）：
- _前瞻_：K 6.18 LTS 上线 Rust 版 Binder（`drivers/android/binder_alloc_rust.rs`），NE 模式可能因 Rust 边界检查而变化。AOSP 17 当前默认 6.12，6.18 分支待推

**稳定性架构师视角**：
- NE 触发 = Native 崩溃，**符号化是排查 NE 的核心瓶颈**（tombstone 是 raw 文本，需要符号化才能读）
- **必抓 3 件套**：tombstone 文件 + 对应 .so 的符号表 + 复现 stack

> **详细机制**：[S03-NE](S03-NE.md) §3（9 个子节深挖）

## 3.4 SWT 的核心机制

**一句话**：SystemServer 主线程 / 关键线程卡死超过 Watchdog 阈值（30s）→ 杀 SystemServer / 整机重启。

**核心机制**（简述，S04 深挖）：
- Watchdog 线程周期（30s）：
  - `Watchdog.java: run()` → 遍历 monitor 列表（AMS/PMS/WMS/...）
  - 每个 monitor 投递一个 `HandlerChecker`（30s 超时）
  - 连续 N 次超时 → 杀 SystemServer
- 喂狗机制：input / vsync / binder 等周期性事件保证不卡
- 杀进程策略：杀线程 → 杀 SystemServer → 整机重启（三层降级）

**AOSP 17 新变化**：
- Watchdog 自动 dump Perfetto trace（新增） // **待 cs.android.com 上确认**
- `WatchdogMonitor` 增强可观测性

**稳定性架构师视角**：
- SWT 触发 = SystemServer 卡死 30s+，**根因几乎一定在 SystemServer 内部**（而非 App）
- **常见根因**：AMS binder call 阻塞 / PMS installPackage 阻塞 / WMS window 死锁

> **详细机制**：[S04-SWT](S04-SWT.md) §3（6 个子节深挖）

## 3.5 HANG 的核心机制（**本系列独占视角**）

**一句话**：进程/线程/子系统功能失效但**未被任何超时机制捕获**。

**核心机制**（简述，S05 深挖）：
- 4 个层面的 HANG：
  - **App 主线程软卡死**：4-5s（未到 5s 阈值，ANR 没触发，但用户感知卡）
  - **IO HANG**：文件系统 / Socket / 块设备 hang 30s+
  - **Binder HANG**：binder 死锁 / 排队 / transaction 满
  - **Kernel HANG**：`hung_task` / RCU stall / softlockup / hardlockup
- HANG 监测盲区：
  - **灰色地带**：主线程到 4-5s 但 ANR 阈值未到
  - **IO HANG 隐藏**：Kernel 层未报 hung_task，但用户体验已经卡
  - **Binder 排队**：远端服务卡死，本端 0% CPU 等待

**AOSP 17 新变化**：
- `MessageQueue` 无锁化（API 37+）减少主线程 HANG
- 端云协同下，AI Agent OS 调用链引入新的"软卡"模式

**稳定性架构师视角**：
- HANG 是**最难排查的症状**——没有统一关键字，必须主动检测
- **3 个监控抓手**：
  1. 主线程响应时间分布（P50/P95/P99 监控，阈值 < 200ms）
  2. 端到端 latency（用户操作 → 界面响应）
  3. Kernel 侧 hung_task 检测（已内置，但需要开 logcat 转发）

> **详细机制**：[S05-HANG](S05-HANG.md) §3（5 个子节深挖）

## 3.6 REBOOT 的核心机制

**一句话**：非预期的进程或系统重启。

**核心机制**（简述，S06 深挖）：
- 4 类重启源：
  - **App 进程重启**：crash 后 Zygote fork 链路
  - **SystemServer 重启**：Watchdog 杀 → restart
  - **Zygote 重启**：Zygote 死 → system_server 死 → 整机重启
  - **整机重启**：Kernel panic / reboot 命令
- 重启溯源：
  - `last_kmsg`（kernel 侧）
  - pstore / ramoops（kernel 侧）
  - dropbox（framework 侧）
  - bootstat 统计（`frameworks/base/services/bootstat/bootstat.cpp`）

**稳定性架构师视角**：
- REBOOT 是**结果态**，不是原因态
- **排查思路**：从 REBOOT 倒推 → 找上层症状（SWT/ANR/HANG/KE）→ 找根因

> **详细机制**：[S06-REBOOT](S06-REBOOT.md) §3（5 个子节深挖）

## 3.7 KE 的核心机制

**一句话**：Kernel 检测到不可恢复/可恢复错误，触发 panic / oops / WARN / hung_task / softlockup / hardlockup / RCU stall。

**核心机制**（简述，S07 深挖）：
- 5 类 KE：
  - **Kernel Panic**：不可恢复，触发整机重启（`kernel/panic.c`）
  - **Kernel Oops**：可恢复，杀相关进程（`kernel/oops.c`）
  - **hung_task**：进程 D 状态超阈值（`kernel/hung_task.c`）
  - **softlockup**：CPU 软死锁 > 20s（`kernel/watchdog.c`）
  - **hardlockup**：CPU 硬死锁 > 10s（NMI 触发）
  - **WARN / BUG**：警告与断言（`kernel/bug.c`）
  - **RCU stall**：RCU 读侧长时间不退出
- 用户空间能看到的 KE 信号：
  - logcat 看不到 dmesg（默认）→ 必须用 `dmesg` / `pstore` / `ramoops` 取证

**Linux 6.12 新变化**：
- _前瞻_：K 6.18 LTS 上线 Rust 版 Binder，binder_alloc_rust 死锁可能产生新的 KE 模式（AOSP 17 当前以 6.12 为主，6.18 分支待推）
- sheaves 内存分配、eBPF 加密签名等新机制引入新的 KE 风险（_前瞻_：K 6.18 LTS 新增，AOSP 17 当前以 6.12 为主）

**稳定性架构师视角**：
- KE 在用户空间通常已演变为**整机重启/ANR/SWT**（cascade 升级）
- **取证难**：kernel log 重启后会被覆盖，必须依赖 pstore / ramoops
- **必抓 3 件套**：pstore（kernel log 持久化） + last_kmsg（重启前日志） + 复现 stack

> **详细机制**：[S07-KE](S07-KE.md) §3（7 个子节深挖）

# 4. 风险地图

## 4.1 7 大症状的线上占比

> **数据来源**：综合 [Google Android Vitals 2024 报告](https://developer.android.com/topic/performance/vitals)、[Datadog Mobile RUM 2024 报告](https://www.datadoghq.com/)、行业综合经验。

| 症状 | 线上占比 | 用户感知 | 监控可见性 | 排查难度 |
|:-----|:---------|:---------|:----------|:--------|
| ANR | 30-40% | 极强（弹窗） | 高 | 中 |
| JE | 25-30% | 强（弹窗） | 高 | 低-中 |
| NE | 15-20% | 强（弹窗） | 中（符号化难） | 高 |
| SWT | 1-3% | 极强（系统重启） | 高 | 中-高 |
| HANG | **不可统计** | 强（用户报障但无明确日志） | **极低**（无统一关键字） | **极高** |
| REBOOT | 1-5% | 极强 | 中（bootstat） | 中-高 |
| KE | < 1% | 极强（整机重启） | 中（dmesg） | 极高 |

> **所以呢**：
> - **ANR + JE + NE 占了 70-90% 的线上崩溃**——S01/S02/S03 是排查主力
> - **HANG 是"沉默的杀手"**——线上不可统计，但用户报障时往往就是 HANG（无明确关键字但用户感知卡）
> - **SWT / REBOOT / KE 占比小但严重性极高**——S04/S06/S07 必须掌握 cascade 推演

## 4.2 7 大症状的用户感知分级

| 用户感知等级 | 症状 | 用户视角的描述 |
|:------------|:-----|:--------------|
| **L1 极强**（必须立即响应）| SWT / REBOOT / KE | "系统死了" / "黑屏了" / "一直重启" |
| **L2 强**（明显但有反馈） | ANR / NE / JE | "弹窗告诉我崩溃了" |
| **L3 中**（用户报障但无明确证据）| HANG | "App 怪怪的" / "刚才好像卡了" |
| **L4 弱**（后台问题）| 异步线程 JE / silent NE | 用户无感知，但数据已损坏 |

> **所以呢**：
> - **L1 触发 = 紧急 hotfix 候选**
> - **L2 触发 = 当个工作日修复**
> - **L3 触发 = APM 监控盲区，必须主动检测**
> - **L4 触发 = 数据正确性问题，优先级最高（数据错了比崩溃更严重）**

## 4.3 7 大症状的严重性矩阵

```
                发生频率
                低 → 高
        ┌──────┬──────┬──────┐
   高   │ KE   │ SWT  │ ANR  │ ← 严重性
        │      │ REBOOT│ NE  │
        │      │      │ JE   │
        ├──────┼──────┼──────┤
   低   │      │      │ HANG │ ← 监控盲区
        └──────┴──────┴──────┘

图 4.3：严重性矩阵（HANG 在右下角 = 高频但低可见）
```

**架构师视角**：
- **HANG 在右下角**——高频但低可见，这是 APM 建设的最大盲区
- **S05 HANG 是本系列独占视角**，价值锚点

## 4.4 排查时常见误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:--------|
| "ANR 触发就是主线程写错了" | ANR 根因可能在 binder 远端 / Kernel IO | 看主线程栈 → 追到阻塞点 → 查阻塞对象的等待原因 |
| "JE 一定是 catch 没加" | 也可能是 catch 链断了，或异步线程逃逸 | 查主线程栈 + 查 dropbox(APP_CRASH) 全量 |
| "NE 一定是我们代码问题" | 也可能是 ROM/驱动/系统库 bug | 查 tombstone 关键 stack → 确认是否在自家代码 |
| "SWT 触发 = SystemServer bug" | 也可能是 App binder call 阻塞 SystemServer | 查 watchdog traces + 喂狗链路 |
| "HANG 不在 APM 报告里就不是 HANG" | HANG 可能未被任何机制检测 | 主动监控主线程 P95 latency |
| "REBOOT 是用户手机问题" | 可能是 cascade 升级 | 用 last_kmsg / pstore 取证 |
| "KE 出现就是内核 bug" | 可能是用户态驱动问题传递到 kernel | 查 dmesg 完整 stack + 关联进程 |

## 5.1 案例 A（CASE-STAB-00-01）：cascade 链路典型模式

> **类型**：典型模式
>
> **环境**：AOSP 14.0.0_r1 / Kernel 5.10 / 设备 Pixel 6（**AOSP 17 验证版准备中**）
>
> **症状**：用户报"App 卡死 10 秒后系统重启"
>
> **根因**：binder 远端服务死锁 → cascade 触发 ANR → SWT → REBOOT

### 现象（用户视角）

```
T+0s    用户点击"打开相机"按钮
T+2s    App 切换到相机预览（黑屏）
T+5s    App 弹"应用无响应"对话框
T+10s   系统开始卡顿（触摸无响应）
T+30s   屏幕黑屏
T+35s   设备自动重启
T+60s   开机完成，但相机功能异常
```

### 分析（架构师视角）

**T+5s ANR 触发**（logcat 片段）：
```
05-15 10:23:45.123  1000  1234  1234 E ActivityManager: ANR in com.example.camera (input dispatching timed out)
05-15 10:23:45.124  1000  1234  1234 E ActivityManager: Reason: Input dispatching timed out (Application is not responding)
05-15 10:23:45.125  1000  1234  1234 W ActivityManager:   at android.os.BinderProxy.transactNative(Native Method)
05-15 10:23:45.126  1000  1234  1234 W ActivityManager:   at android.os.BinderProxy.transact(BinderProxy.java:540)
05-15 10:23:45.127  1000  1234  1234 W ActivityManager:   at android.hardware.camera2.ICameraDeviceUser$Stub$Proxy.openCamera(ICameraDeviceUser.java:299)
```

**T+30s SWT 触发**（logcat 片段）：
```
05-15 10:24:15.000  1000  1000  1010 W Watchdog: *** WATCHDOG KILLING SYSTEM PROCESS: Blocked in handler on ActivityManager (ActivityManager)
05-15 10:24:15.001  1000  1000  1010 W Watchdog: Input event dispatching timed out sending to com.example.camera
05-15 10:24:15.002  1000  1000  1010 I Watchdog: Killing system server due to blocked handler in ActivityManager
```

**T+35s REBOOT 触发**：
```
05-15 10:24:20.000  1000  1000  1010 I SystemServer: System server died, restarting
05-15 10:24:25.000  1000  1000  1010 I SystemServiceManager: System server restarts: 1
```

### 根因（机制层）

**binder 死锁链路**：
```
App 主线程（相机模块）
  ↓ binder call ICameraDeviceUser.openCamera()
CameraService（系统服务）
  ↓ 同步调用 Camera HAL
Camera HAL 层
  ↓ 阻塞在某个 sensor driver 的 IO 操作
Sensor driver 线程（D 状态，hung_task 触发）
```

**关键 logcat**（camera service 端）：
```
05-15 10:23:40.000  1000  1500  1500 E CameraService: openCamera X takes too long (3000ms)
05-15 10:23:43.000  1000  1500  1500 E CameraService: Camera HAL open failed: -110 (TIMED_OUT)
```

### 修复方案

**短期（hotfix）**：
```java
// 改为异步调用 + 超时
private void openCameraAsync() {
    executor.execute(() -> {
        try {
            Future<?> future = executor.submit(() -> cameraDevice.openCamera(...));
            future.get(2, TimeUnit.SECONDS);  // 2s 超时
        } catch (TimeoutException e) {
            // 降级方案：提示用户重试
            showRetryDialog();
        }
    });
}
```

**长期（治理）**：
- 在 `frameworks/av/services/camera/libcameraservice/` 中增加 binder call 超时保护
- 引入 HANG 检测：监控 CameraService 主线程响应时间 P95

### 验证步骤

1. **复现**：在低性能设备上快速点击"打开相机" 100 次
2. **观察**：logcat 是否复现 `openCamera X takes too long` + `Input dispatching timed out`
3. **验证修复**：应用 hotfix 后，重新跑 100 次，无 ANR/SWT/REBOOT 触发

## 5.2 案例 B（CASE-STAB-00-02）：公开 bugreport — AOSP Issue 252354175

> **类型**：公开 bugreport
>
> **来源**：[AOSP Issue Tracker - Issue 252354175](https://issuetracker.google.com/issues/252354175)（**示例编号，撰写时需确认**）
>
> **主题**：binder rust 改造过程中的死锁（对应 S07 KE 案例）

> **撰写时验证**：本案例的具体 issue 编号需在 S07 撰写时通过 [issuetracker.google.com](https://issuetracker.google.com/) 检索 `binder deadlock` 确认。本节先以**案例模式**呈现，正式撰写时填入真实 issue 编号。

### 现象（用户视角）

```
T+0s    用户在 App 中调用系统服务
T+1s    系统服务无响应
T+30s   整机重启
T+60s   开机
```

### 分析

**binder driver 死锁**（dmesg 片段）：
```
[ 1234.567890] INFO: task xxx:1234 blocked for more than 120 seconds.
[ 1234.567891] "echo 0 > /proc/sys/kernel/hung_task_timeout_secs" disables this message.
[ 1234.567892] Call Trace:
[ 1234.567893]  __schedule+0x123/0x456
[ 1234.567894]  schedule+0x45/0x89
[ 1234.567895]  schedule_preempt_disabled+0x12/0x23
[ 1234.567896]  __mutex_lock.isra.0+0x234/0x345
[ 1234.567897]  binder_alloc_rust_mutex_lock+0x45/0x67   ← Rust 版 binder 死锁
```

**pstore 记录**（kernel panic 前）：
```
[ 1264.567890] Kernel panic - not syncing: hung_task detected, killing process
[ 1264.567891] CPU: 0 PID: 1234 Comm: binder_rust Tainted: G        W       5.15.0
```

### 根因

K 6.18 LTS 上线 Rust 版 Binder 后（**前瞻案例**：AOSP 17 当前默认 6.12，6.18 分支待推），Rust 与 C 版并存的 mutex 锁顺序不一致，导致死锁。

### 修复

AOSP 上游 commit 修复（简化展示）：
```rust
// drivers/android/binder_alloc_rust.rs
// 修复：统一锁顺序为 alloc_mutex → mmap_lock
pub fn binder_alloc_new_buffer_locked(...) {
    // 统一锁顺序
    let _mmap_guard = self.mmap_lock.lock();  // 先 mmap_lock
    let _alloc_guard = self.alloc.lock();      // 再 alloc_lock
    ...
}
```

### 验证

1. 应用 patch
2. 复现脚本：高频 binder call 1000 次
3. 验证：不再触发 hung_task 死锁

# 6. 总结（架构师视角的 5 条 Takeaway）

1. **症状分类法是 30 秒定位的"决策树"**：看到任何线上问题，先归类到 7 类之一（用 §2.2 对比表），再跳到对应 S01-S07 深入。这是 Stability 系列的核心价值。
2. **HANG 是沉默的杀手**：7 类中唯一无统一关键字、无主动检测机制的症状，S05 HANG 是本系列独占视角，是 APM 建设的最大盲区。
3. **cascade 推演是预防性思维**：1 个根因可触发 4 类症状（见 §2.3），排查时不能只修"当前症状"，要逆向推演根因，避免下次 cascade 到更严重的 REBOOT。
4. **REBOOT 是结果态不是原因态**：所有 REBOOT 都能倒推到其他 6 类之一（SWT/KE/...），排查 REBOOT 时必须先取证（last_kmsg / pstore / dropbox）再逆向。
5. **视角互补而非替代**：本系列不重复现有 Watchdog/ANR_Detection/Native_Crash 等系列的机制深度，**只做症状维度入口**——遇到机制细节，跳到对应系列文章。

## 6.1 排查路径速查

| 看到症状 | 第一步（30 秒） | 第二步 | 第三步 |
|:---------|:--------------|:-------|:-------|

# 附录 D：工程基线表

| 参数 | 典型默认 | 选用准则 | 踩坑提醒 |
|:-----|:---------|:---------|:---------|
| **ANR 阈值（Input）** | 5s | 不可调 | 高频事件会偶发 |
| **ANR 阈值（Broadcast）** | 10s（前台）/ 60s（后台）| 不可调 | 串行队列会累积 |
| **ANR 阈值（Service）** | 20s（前台）/ 200s（后台）| 不可调 | startService 易踩 |
| **ANR 阈值（Provider）** | 10s | 不可调 | 启动期 publish 阻塞 |
| **Watchdog 周期** | 30s | AOSP 默认 | 太短→误杀；太长→响应慢 |
| **hung_task 超时** | 120s | `/proc/sys/kernel/hung_task_timeout_secs` | 太短→误报；太长→响应慢 |
| **softlockup 阈值** | 20s | `/proc/sys/kernel/watchdog_thresh` | 太短→误报；太长→响应慢 |
| **hardlockup 阈值** | 10s | NMI 中检测，不可配 | — |
| **tombstone 保留** | `/data/tombstones/` 10 个 | 满了覆盖最早的 | 高发期会被覆盖 |
| **pstore 大小** | 厂商定制（通常 64K-1M） | ramoops/pstore 配置 | 太小→取证丢失 |
| **dropbox 保留期** | 7 天（APP_CRASH）/ 30 天（SYSTEM_TOMBSTONE）| `/data/system/dropbox/` 满后覆盖 | 高发期会丢关键 |
| **ANR traces 保留** | 5 个 | `/data/anr/` 满后覆盖 | 必须有日志采集 |
| **主线程 P95 latency 阈值** | 200ms | 端到端监控 | 超过 → HANG 风险 |
| **binder call 超时（App 端推荐）** | 2-3s | 业务自定 | 太长→用户感知卡；太短→误失败 |
| **HANG 主动检测采样频率** | 10-100Hz | 帧时间 100Hz / CPU 10Hz / 内存 1Hz | 太密→存储爆炸；太稀→漏检 |

> **架构师视角**：
> - 上表参数是**线上默认值**，**不是"理想值"**。架构师需要根据业务调参
> - 调参的"所以呢"：调小阈值 → 误报↑ / 调大阈值 → 漏报↑
