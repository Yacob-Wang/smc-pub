## 2. Crash 堆栈聚类（Sentence-BERT + HDBSCAN）

### 2.1 为什么需要聚类

**场景**：某 App 日均 10000 条 Java Crash，直接 LLM 解读成本高（$0.01 × 10000 = $100/天）。

**聚类的好处**：
- 10000 条 → 50 个簇 → 50 次 LLM 调用 → $0.5/天（-99.5%）
- 自动发现"哪些 Crash 是同一个根因"
- 监控 Crash 簇的变化（新增簇 / 簇增长）

### 2.2 Sentence-BERT 嵌入

**原理**：把 Java 堆栈文本转成 768 维向量，相似堆栈在向量空间中距离近。

```python
from sentence_transformers import SentenceTransformer

def embed_stack_traces(stacks, model_name='sentence-transformers/all-MiniLM-L6-v2'):
    """Java 堆栈 Sentence-BERT 嵌入"""
    model = SentenceTransformer(model_name)
    
    # 预处理：提取"关键帧"（去掉行号、参数等噪声）
    processed_stacks = [preprocess_stack(s) for s in stacks]
    
    # 批量嵌入
    embeddings = model.encode(processed_stacks, 
                               batch_size=64,
                               show_progress_bar=True)
    return embeddings

def preprocess_stack(stack):
    """堆栈预处理：标准化 + 关键帧提取"""
    lines = stack.split('\n')
    # 保留类名 + 方法名，去掉行号 / 参数
    normalized = []
    for line in lines:
        if 'at ' in line:
            # 提取 "at com.example.Class.method(File.java:123)" → "com.example.Class.method"
            match = re.search(r'at ([\w.$]+)\(([\w.]+):(\d+)\)', line)
            if match:
                cls_method = match.group(1)
                normalized.append(cls_method)
    return ' '.join(normalized[:10])  # 只保留前 10 帧
```

**关键步骤**：

1. **预处理**：去行号 / 去参数 / 去包名前缀噪声（保留业务类）
2. **截断**：只保留前 5-10 帧（最相关的栈帧）
3. **批量嵌入**：GPU 加速时 64-128 batch

### 2.3 HDBSCAN 聚类

**原理**：层次密度聚类，自动确定簇数量 + 识别噪声点。

**为什么选 HDBSCAN 而不是 K-Means**：
- K-Means 需要预先指定簇数量（实际未知）
- HDBSCAN 自动发现簇 + 识别噪声点（不适合聚类的异常 crash）
- HDBSCAN 对簇形状无假设（K-Means 假设球形）

```python
import hdbscan

def cluster_embeddings(embeddings, min_cluster_size=50, min_samples=5):
    """HDBSCAN 聚类"""
    clusterer = hdbscan.HDBSCAN(
        min_cluster_size=min_cluster_size,  # 最小簇大小
        min_samples=min_samples,            # 核心点邻居数
        metric='euclidean',
        cluster_selection_method='eom'      # Excess of Mass（推荐）
    )
    cluster_labels = clusterer.fit_predict(embeddings)
    
    # cluster_labels: -1 = 噪声 / 0+ = 簇编号
    n_clusters = len(set(cluster_labels)) - (1 if -1 in cluster_labels else 0)
    n_noise = list(cluster_labels).count(-1)
    
    return cluster_labels, n_clusters, n_noise
```

**关键参数**：
- `min_cluster_size`：最小簇大小（通常 50-100）
- `min_samples`：核心点最少邻居数（影响噪声识别）
- `metric`：距离度量（euclidean / cosine）

### 2.4 聚类效果评估

**评估指标**：

| 指标 | 含义 | 目标 |
| :--- | :--- | :--- |
| **轮廓系数（Silhouette）** | 簇内紧密 + 簇间分离 | > 0.5 |
| **簇大小分布** | 簇是否均匀 | 头部簇占比 < 30% |
| **噪声点占比** | 不适合聚类的比例 | < 10% |

**实战数据**（某 App 10000 条 Java Crash）：
- HDBSCAN 输出：50 个簇 + 200 个噪声点（2%）
- 最大簇：1200 条（同一 NPE 根因）
- 轮廓系数：0.62（优秀）

## 3. LLM 解析 trace（Prompt Engineering + Few-shot + CoT）

### 3.1 Prompt Engineering 基础

**原则**：
1. **明确角色**：LLM 扮演"Android 稳定性工程师"
2. **结构化输出**：强制 JSON 输出
3. **Few-shot**：提供 3-5 个示例
4. **Chain-of-thought（CoT）**：让 LLM 一步步推理

### 3.2 Java Crash 根因解读 Prompt 模板

```python
JAVA_CRASH_PROMPT = """
你是一名资深的 Android 稳定性工程师。请分析以下 Java Crash 堆栈，输出根因标签和修复建议。

【堆栈文本】
{stack_trace}

【异常类型】
{exception_type}

【设备信息】
Android {android_version} / Kernel {kernel_version} / 设备 {device_model}

【常见根因标签（请从中选择或自定义）】
1. NPE_NULL_CHECK（空指针未检查）
2. NPE_VIEW_NULL（View 为空）
3. CLASS_CAST（类型转换异常）
4. INDEX_OUT_OF_BOUNDS（数组越界）
5. NETWORK_IO（网络 IO 异常）
6. DB_CORRUPTED（数据库损坏）
7. CONCURRENT_MODIFICATION（并发修改）
8. OOM_HEAP（堆内存溢出）
9. SECURITY_PERMISSION（权限异常）
10. OTHER（其他）

【输出格式（JSON）】
{{
  "root_cause": "选择的标签或自定义",
  "confidence": "high/medium/low",
  "reasoning": "基于堆栈的推理过程（链式思考）",
  "affected_code": "可能出错的代码位置（类名.方法名）",
  "fix_suggestion": "修复建议（具体可执行）",
  "severity": "critical/high/medium/low"
}}

【Few-shot 示例】
示例 1:
堆栈：java.lang.NullPointerException: Attempt to invoke virtual method 'int java.lang.String.length()' on a null object reference
   at com.example.app.User.getName(User.java:42)
   at com.example.app.ProfileActivity.onCreate(ProfileActivity.java:78)
输出：
{{"root_cause": "NPE_NULL_CHECK", "confidence": "high", "reasoning": "User.java:42 调用 getName() 时 userName 为 null", "affected_code": "User.getName", "fix_suggestion": "在 getName() 中添加 null 检查或确保 userName 在使用前已赋值", "severity": "high"}}

示例 2:
堆栈：java.lang.ClassCastException: android.widget.TextView cannot be cast to android.widget.ImageView
   at com.example.app.HomeAdapter.onBindViewHolder(HomeAdapter.java:55)
输出：
{{"root_cause": "CLASS_CAST", "confidence": "high", "reasoning": "HomeAdapter.java:55 误将 TextView 强转为 ImageView，可能是 RecyclerView item 布局错误", "affected_code": "HomeAdapter.onBindViewHolder", "fix_suggestion": "检查 item_home.xml 中对应位置的 View 类型是否正确", "severity": "high"}}

现在请分析【堆栈文本】并输出 JSON：
"""

def parse_java_crash_with_llm(stack_trace, exception_type, device_info):
    """LLM 解析 Java Crash"""
    prompt = JAVA_CRASH_PROMPT.format(
        stack_trace=stack_trace,
        exception_type=exception_type,
        android_version=device_info['android'],
        kernel_version=device_info['kernel'],
        device_model=device_info['model']
    )
    response = openai.ChatCompletion.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"},
        temperature=0  # 降低随机性
    )
    return json.loads(response.choices[0].message.content)
```

## 4. Function Calling（让 LLM 调用工具）

### 4.1 为什么需要 Function Calling

**问题**：LLM 只能处理自然语言，不能直接：
- 解析二进制 trace 文件
- 查询数据库
- 搜索代码
- 调用 RPC

**Function Calling 解决方案**：让 LLM 输出"调用哪个函数 + 参数"，代码层执行函数，把结果喂回 LLM。

### 4.2 Function Calling 实战示例

**定义工具函数**：

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "search_code",
            "description": "在代码仓库中搜索包含指定关键词的代码段",
            "parameters": {
                "type": "object",
                "properties": {
                    "keyword": {"type": "string", "description": "搜索关键词（类名或方法名）"},
                    "file_pattern": {"type": "string", "description": "文件名模式（如 *.java）"}
                },
                "required": ["keyword"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "parse_anr_trace",
            "description": "解析 ANR trace 文件，提取关键线程状态",
            "parameters": {
                "type": "object",
                "properties": {
                    "trace_content": {"type": "string", "description": "ANR trace 文件内容"}
                },
                "required": ["trace_content"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "get_recent_commits",
            "description": "获取指定文件最近的提交记录",

                "type": "object",
                "properties": {

            }
        }
    }

```

**LLM 自动调用流程**：

```
用户：分析这份 ANR trace
   ↓
LLM 思考：需要先解析 trace
   ↓
LLM 输出：调用 parse_anr_trace(trace_content=...)
   ↓
代码层：解析 trace，提取 main 线程栈
   ↓
代码层：把结果喂回 LLM
   ↓
LLM 思考：需要找到阻塞的具体代码
   ↓
LLM 输出：调用 search_code(keyword="UserService.getUser")
   ↓
代码层：搜索代码，找到 UserService.java:78
   ↓
代码层：把代码段喂回 LLM
   ↓
LLM 输出：根因分析 + 修复建议
```

### 4.3 Function Calling 实战代码

```python
def analyze_with_function_calling(anr_trace):
    """Function Calling 驱动的 ANR 分析"""
    messages = [
        {"role": "user", "content": f"分析这份 ANR trace:\n{anr_trace}"}
    ]
    
    max_iterations = 5
    for i in range(max_iterations):
        response = openai.ChatCompletion.create(
            model="gpt-4o",
            messages=messages,
            tools=tools,
            tool_choice="auto"
        )
        
        message = response.choices[0].message
        
        # LLM 不需要调用工具 → 返回最终结果
        if not message.tool_calls:
            return message.content
        
        # LLM 调用工具 → 执行 + 喂回
        messages.append(message)
        
        for tool_call in message.tool_calls:
            function_name = tool_call.function.name
            arguments = json.loads(tool_call.function.arguments)
            
            # 执行工具
            if function_name == "parse_anr_trace":
                result = parse_anr_trace(arguments["trace_content"])
            elif function_name == "search_code":
                result = search_code(arguments["keyword"])
            elif function_name == "get_recent_commits":
                result = get_recent_commits(arguments["file_path"])
            
            # 喂回 LLM
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": json.dumps(result)
            })
    
    return "达到最大迭代次数"
```

### 4.4 Function Calling 适用场景

| 场景 | 适用度 | 理由 |
| :--- | :--- | :--- |
| **多步归因（解析 → 搜索 → 推断）** | ✅ 高 | LLM 自主决策工具调用 |
| **复杂数据查询** | ✅ 高 | LLM 把自然语言转 SQL/API |
| **简单问答** | ❌ 低 | 不需要工具 |
| **实时决策（< 100ms）** | ❌ 低 | 多轮调用延迟高 |

## 5. RAG 检索历史 case（向量库 + 相似度检索）

### 5.1 为什么需要 RAG

**问题**：
- LLM 没有"历史经验"
- 同样的 Crash 反复发生，但每次 LLM 都从零分析
- 历史工程师积累的"case 库"没有被利用

**RAG 解决方案**：
- 把历史 case 库存入向量库
- 新 crash 进来时，先检索 top-K 相似历史 case
- 把历史 case 作为上下文喂给 LLM

### 5.2 RAG 架构

```
┌──────────────────────────────────────────────────────────┐
│ RAG 检索架构                                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  历史 case 库                                              │
│    ├─ 标题、堆栈、根因、修复方案、commit hash              │
│    └─ Sentence-BERT 嵌入 → 向量库（Chroma / Milvus）      │
│                                                          │
│  新 crash 进来                                            │
│    ↓                                                     │
│  Sentence-BERT 嵌入                                       │
│    ↓                                                     │
│  向量相似度检索（top-K = 5）                               │
│    ↓                                                     │
│  把 top-5 相似 case 作为上下文喂给 LLM                    │
│    ↓                                                     │
│  LLM 输出：根因 + 修复建议 + 相似历史 case 引用            │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 5.3 向量库选型

| 向量库 | 类型 | 适用规模 | 部署方式 |
| :--- | :--- | :--- | :--- |
| **Chroma** | 嵌入式 | < 100 万 | 本地 / 云端 |
| **Milvus** | 分布式 | > 100 万 | 自建 |
| **Pinecone** | SaaS | 任意 | 云端 |
| **Weaviate** | 分布式 | > 100 万 | 自建 / 云端 |
| **FAISS** | 库 | < 1000 万 | 本地 |

**推荐**：中小规模（< 100 万 case）用 **Chroma**；大规模用 **Milvus**。

### 5.4 RAG 实战代码

```python
import chromadb
from sentence_transformers import SentenceTransformer

class CaseRAG:
    def __init__(self, persist_dir="./chroma_db"):
        self.client = chromadb.PersistentClient(path=persist_dir)
        self.collection = self.client.get_or_create_collection("crash_cases")
        self.embedder = SentenceTransformer('sentence-transformers/all-MiniLM-L6-v2')
    
    def add_case(self, case_id, stack, root_cause, fix):
        """添加历史 case 到向量库"""
        embedding = self.embedder.encode(preprocess_stack(stack)).tolist()
        self.collection.add(
            ids=[case_id],
            embeddings=[embedding],
            documents=[f"堆栈：{stack}\n根因：{root_cause}\n修复：{fix}"]
        )
    
    def search_similar(self, stack, k=5):
        """检索相似历史 case"""
        embedding = self.embedder.encode(preprocess_stack(stack)).tolist()
        results = self.collection.query(
            query_embeddings=[embedding],
            n_results=k
        )
        return results

def analyze_with_rag(stack_trace, exception_type):
    """RAG 驱动的智能归因"""
    rag = CaseRAG()
    
    # 步骤 1：检索相似历史 case
    similar_cases = rag.search_similar(stack_trace, k=5)
    
    # 步骤 2：构造 Prompt
    prompt = f"""
你是 Android 稳定性工程师。请分析以下 Java Crash，结合历史相似 case 给出根因。

【当前堆栈】
{stack_trace}

【异常类型】
{exception_type}

【历史相似 case（top 5）】
"""
    for i, case in enumerate(similar_cases['documents'][0]):
        prompt += f"\n--- 相似 case {i+1} ---\n{case}\n"
    
    prompt += """
【输出格式】
{
  "root_cause": "根因",
  "confidence": "high/medium/low",
  "similar_case_ids": ["引用最相关的历史 case ID"],
  "fix_suggestion": "修复建议"
}
"""
    
    # 步骤 3：LLM 推理
    response = openai.ChatCompletion.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"}
    )
    return json.loads(response.choices[0].message.content)
```

### 5.5 RAG 实战效果

**对比实验**（某 App 500 条 Crash 测试集）：

| 方案 | 根因准确率 | 输出含历史 case 引用 |
| :--- | :--- | :--- |
| LLM 无 RAG | 72% | 0% |
| LLM + RAG（top-5） | **84%** | **100%** |
| LLM + RAG（top-10） | 85% | 100% |

**结论**：RAG 显著提升准确率（+12%），并提供"历史 case 引用"增强可解释性。

## 6. 根因标签体系（规则 + LLM 双轨）

### 6.1 根因标签的价值

**作用**：
- 自动归类（聚类 → 标签）
- 自动派单（标签 → 对应团队）
- 趋势监控（标签分布变化 → 早期预警）

### 6.2 标签体系设计

**设计原则**：
- **MECE 原则**：相互独立、完全穷尽
- **可操作性**：标签必须能直接对应修复团队
- **稳定性**：标签不能频繁变化

**示例标签体系**（Java Crash）：

| 一级标签 | 二级标签 | 修复团队 |
| :--- | :--- | :--- |
| **NPE** | NPE_NULL_CHECK / NPE_VIEW_NULL / NPE_BUNDLE_GET | 业务开发 |
| **类型转换** | CLASS_CAST / ILLEGAL_ARGUMENT | 业务开发 |
| **数组越界** | INDEX_OUT_OF_BOUNDS | 业务开发 |
| **网络** | NETWORK_IO / SOCKET_TIMEOUT | 网络团队 |
| **数据库** | DB_CORRUPTED / DB_LOCK_TIMEOUT | 存储团队 |
| **并发** | CONCURRENT_MODIFICATION / DEAD_LOCK | 业务开发 |
| **内存** | OOM_HEAP / OOM_NATIVE | 性能团队 |
| **安全** | SECURITY_PERMISSION | 安全团队 |
| **第三方** | THIRD_PARTY_SDK | 第三方对接 |
| **其他** | OTHER | 兜底 |

### 6.3 规则引擎 + LLM 双轨归因

**为什么双轨**：
- 规则引擎：100% 准确率，但只能覆盖已知模式（30-50%）
- LLM：覆盖率 90%+，但准确率 75-85%
- 双轨：先用规则（高准确），规则未覆盖时用 LLM（高覆盖）

```python
def classify_crash_with_hybrid(stack, exception_type):
    """规则引擎 + LLM 双轨归因"""
    # 步骤 1：规则引擎（先尝试）
    rule_result = apply_rules(stack, exception_type)
    if rule_result.confidence == 'high':
        return rule_result
    
    # 步骤 2：LLM 兜底
    llm_result = analyze_with_llm(stack, exception_type)
    
    # 步骤 3：合并结果
    return merge_results(rule_result, llm_result)
```

## 7. 风险地图

| 风险类型 | 触发条件 | 现象 | 防范 |
| :--- | :--- | :--- | :--- |
| **LLM 幻觉** | Prompt 不严谨 / Few-shot 错误 | 根因标签错误 | 人工复核 + 规则兜底 |
| **聚类过细** | min_cluster_size 太小 | 簇过多，重复分析 | 调参 + 自动合并 |
| **聚类过粗** | min_cluster_size 太大 | 不同根因混合 | 调参 + 人工拆解 |
| **RAG 检索不准** | 嵌入模型不匹配 | 相似 case 召回低 | 用 CodeBERT / 调整相似度阈值 |
| **数据泄露** | 堆栈含用户数据 | 隐私风险 | 脱敏 + 私有化部署 |
| **成本失控** | LLM 高频调用 | 账单爆炸 | 聚类降频 + 限流 |
| **冷启动数据不足** | 新上线 case 库为空 | RAG 失效 | 冷启动策略（无 RAG 模式） |
| **根因标签过时** | 业务重构 | 标签与代码脱节 | 定期回顾 + 自动更新 |

- 根因标签 → 对应修复团队（如 `ANR_BROADCAST_RECEIVER` → 业务开发）

## 9. 总结（架构师视角的 5 条 Takeaway）

1. **智能归因 = 聚类 + LLM + RAG + 标签体系**——单一技术不够，**四件套组合拳**才是工程答案。聚类降本、LLM 解读、RAG 复用、标签派单。
2. **Sentence-BERT + HDBSCAN 是当前最优聚类组合**——K-Means 需要指定簇数（实际未知），HDBSCAN 自动发现 + 识别噪声。**实战覆盖率 95%+**。
3. **Few-shot + CoT 是 Prompt 黄金组合**——比 Zero-shot 准确率高 16-20%，且输出格式合规率高。**3-5 个示例是最佳平衡**。
4. **Function Calling 让 LLM 真正"动手"**——多步归因（解析 → 搜索 → 推断）必须用 Function Calling，**单轮 LLM 无法处理复杂任务**。
5. **规则引擎 + LLM 双轨是工程最优解**——规则 100% 准确（覆盖 30-50%），LLM 85% 准确（覆盖 90%+），双轨组合 = 95%+ 准确 + 90%+ 覆盖。

**智能归因决策树**：

```
新项目要做智能归因
  ↓
数据规模？
  ├─ 小（< 1000 条/天） → 直接 LLM 解读（无需聚类）
  ├─ 中（1000-10000 条/天） → Sentence-BERT + HDBSCAN 聚类
  └─ 大（> 10000 条/天） → 聚类 + LLM + RAG 全套
  ↓
是否需要 RAG？
  ├─ 历史 case 库丰富（> 1000 条） → RAG
  └─ 历史 case 库少 → 无 RAG 模式
  ↓
是否需要 Function Calling？
  ├─ 多步归因（解析 → 搜索 → 推断） → Function Calling
  └─ 单步归因 → 不需要
  ↓
归因标签体系？
  ├─ 业务稳定（标签 1-2 年不变） → 完整标签体系
  └─ 业务快速迭代 → 简化标签 + 自动更新
```

| **LLM Few-shot 示例数** | 3-5 | 太多→token 浪费 | < 2→效果差 |
| **Function Calling 最大迭代** | 5 | 防止死循环 | 太多→成本高 |
| **RAG top-K** | 5 | 太多→噪声 | < 3→召回低 |
| **聚类降频阈值** | 1000+ 条/天启用聚类 | 数据量小直接 LLM | 不分类直接 LLM 成本高 |
| **规则覆盖率目标** | ≥ 30% | 业务稳定→提高 | 不维护→失效 |
| **LLM 兜底覆盖率目标** | ≥ 90% | 经验值 | 不兜底→长尾漏检 |
| **标签准确率目标** | ≥ 80% | 实战标准 | < 70%→不可用 |
| **自动归因率目标** | ≥ 70% | 实战标准 | < 50%→人工工作量大 |
| **RAG 数据脱敏** | 100% | 强制 | 数据泄露必触发监管 |
| **LLM 调用频率限制** | 异常时 + 节流 | 高频→成本爆炸 | 不限流→账单失控 |
