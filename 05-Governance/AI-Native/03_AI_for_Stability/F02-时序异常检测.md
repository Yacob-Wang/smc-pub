## 1. 背景与定义：时序异常检测在稳定性治理中的位置

### 1.1 什么是时序异常检测

**定义**：从按时间顺序排列的指标数据中，自动识别出"偏离正常模式"的点或段。

```
时序异常检测示意：

时间 →
   ┌──────────────────────────────────────────────────────┐
   │  CPU 占用率（%）                                       │
   │ 100 ┤                                  ▲             │
   │  80 ┤                                  │             │
   │  60 ┤                                  │  ← 异常段    │
   │  40 ┤──────────────────┐               │             │
   │  20 ┤                  └───────────────┘             │
   │   0 └──────────────────────────────────────────────────│
   └──────────────────────────────────────────────────────┘
   正常基线：20-40%           异常：>60% 持续 30s+
```

**稳定性场景中的"异常"分类**：

| 异常类型 | 数据特征 | 示例 |
| :--- | :--- | :--- |
| **点异常（Point Anomaly）** | 单点偏离基线 | 主线程帧时间从 16ms 突然到 200ms |
| **段异常（Contextual Anomaly）** | 一段时间内的整体偏离 | CPU 持续 90%+ 5s+ |
| **频率异常（Frequency Anomaly）** | 事件频率异常 | Binder 调用频率突增 10x |
| **模式异常（Pattern Anomaly）** | 时序模式变化 | 主线程任务从"短而密"变成"长而稀" |

### 1.2 为什么稳定性治理需要时序异常检测

**3 大业务价值**：

1. **实时发现**：从"用户反馈 → 工单 → 排查"变成"系统自动发现 → 自动告警"，预警时间从小时级降到秒级。
2. **量化趋势**：把"系统稳不稳"从主观感受变成可量化的指标（如"P99 主线程帧时间 < 16ms 的占比 99.5%"）。
3. **早期预警**：异常点往往在 ANR/Crash 之前出现 → 提前 5-10s 预警（详见 [F04 AI 预测 ANR](F04-AI预测ANR.md)）。

### 1.3 稳定性场景的核心指标

| 指标类别 | 具体指标 | 采样频率 | 数据源 |
| :--- | :--- | :--- | :--- |
| **主线程** | 帧时间 / 卡顿帧率 / Message 处理时长 | 60Hz / 120Hz | Choreographer / Looper |
| **CPU** | 进程 CPU 占用 / 主线程 CPU / 系统 CPU | 1Hz | /proc/stat |
| **内存** | PSS / RSS / Java Heap / Native Heap | 1Hz | Debug.MemoryInfo |
| **IO** | 磁盘读 / 写 / 异步 IO 队列 | 1Hz | /proc/diskstats |
| **网络** | 收发字节数 / 连接数 / 重传率 | 1Hz | /proc/net |
| **Binder** | 调用频次 / 活跃线程数 / buffer 使用率 | 1Hz | debugfs |
| **GPU** | 渲染耗时 / 掉帧率 | 60Hz | SurfaceFlinger |
| **耗电** | 电流 / 电压 / 温度 | 1Hz | BatteryStats |

## 2. 统计方法：3σ / IQR / Z-Score

### 2.1 3σ 准则（高斯分布）

**原理**：假设数据服从正态分布 N(μ, σ²)，约 99.7% 的数据落在 [μ-3σ, μ+3σ] 区间。超出此区间的点视为异常。

```
3σ 准则示意：

                 异常 ←3σ──┐   ┌──3σ→ 异常
                          ▼   ▼
        ┌────────────────────────────────────────┐
        │      ╱╲                                │
        │     ╱  ╲                               │
        │    ╱    ╲                              │
        │   ╱      ╲     ← 99.7% 数据             │
        │  ╱        ╲                            │
        │ ╱          ╲                           │
        │╱            ╲                          │
        └────────────────────────────────────────┘
        μ-3σ          μ          μ+3σ
```

**实现**：

```python
import numpy as np

def detect_3sigma(data, window_size=300, threshold=3.0):
    """3σ 准则异常检测（滑动窗口）"""
    anomalies = []
    for i in range(window_size, len(data)):
        window = data[i-window_size:i]
        mean = np.mean(window)
        std = np.std(window)
        if std == 0:
            continue
        z_score = (data[i] - mean) / std
        if abs(z_score) > threshold:
            anomalies.append((i, data[i], z_score))
    return anomalies
```

**适用场景**：
- 单指标 / 简单分布
- 实时性要求高（O(n) 时间复杂度）
- 资源受限（无训练 / 无存储）

**局限**：
- 假设正态分布（实际数据多偏态 / 长尾）
- 单指标（不能处理多指标联合异常）
- 对突变不敏感（窗口大小敏感）

### 2.2 IQR 方法（任意分布）

**原理**：使用四分位数定义异常边界，超出 Q1 - k×IQR 或 Q3 + k×IQR（k 通常取 1.5）的点视为异常。**不假设数据分布**。

```
IQR 方法示意：

        ┌────────────────────────────────────────┐
        │                                        │
        │    Q1           Q2(中位数)      Q3     │
        │    │             │              │     │
        │    ▼             ▼              ▼     │
        │    ├─────────────┼──────────────┤     │
        │    │             │              │     │
        └────┼─────────────┼──────────────┼─────┘
             │             │              │
       Q1-1.5×IQR                    Q3+1.5×IQR
        ←── 异常边界 ──→            ←── 异常边界 ──→
```

**实现**：

```python
def detect_iqr(data, k=1.5):
    """IQR 方法异常检测"""
    q1 = np.percentile(data, 25)
    q3 = np.percentile(data, 75)
    iqr = q3 - q1
    lower_bound = q1 - k * iqr
    upper_bound = q3 + k * iqr
    anomalies = [(i, v) for i, v in enumerate(data) 
                 if v < lower_bound or v > upper_bound]
    return anomalies
```

**适用场景**：
- 偏态分布 / 长尾分布（如响应时间通常右偏）
- 数据分布未知

**局限**：
- 仍然是单指标
- 对短期异常不敏感（k 值固定）

### 2.3 Z-Score（带时间衰减）

**原理**：在 3σ 基础上，引入**指数移动平均（EMA）** 让基线随时间漂移，适应数据分布变化。

```python
def detect_zscore_ema(data, alpha=0.1, threshold=3.0):
    """带 EMA 的 Z-Score 异常检测"""
    ema_mean = data[0]
    ema_var = 0
    anomalies = []
    
    for i, x in enumerate(data):
        # 计算 Z-Score
        std = np.sqrt(ema_var) if ema_var > 0 else 1
        z_score = (x - ema_mean) / std
        if abs(z_score) > threshold:
            anomalies.append((i, x, z_score))
        
        # 更新 EMA
        ema_mean = alpha * x + (1 - alpha) * ema_mean
        ema_var = alpha * (x - ema_mean)**2 + (1 - alpha) * ema_var
    
    return anomalies
```

**适用场景**：
- 数据分布随时间漂移（如 App 升级后基线变化）
- 在线学习（无需重新训练）

**局限**：
- 仍然单指标
- α 值需要调参

### 2.4 统计方法选型矩阵

| 场景 | 推荐方法 | 理由 |
| :--- | :--- | :--- |
| **CPU 占用率（接近正态）** | 3σ | 简单有效 |
| **响应时间（右偏）** | IQR | 不假设分布 |
| **用户活跃度（漂移）** | Z-Score + EMA | 自适应基线 |
| **主线程帧时间** | IQR | 偏态分布 |
| **内存使用（漂移）** | Z-Score + EMA | App 升级基线变 |

## 3. 机器学习方法：Isolation Forest / One-Class SVM

### 3.1 Isolation Forest（孤立森林）

**原理**：异常点"易被孤立"——随机切分特征空间时，异常点需要的切分次数更少（路径更短）。

```
Isolation Forest 原理示意：

正常点：          异常点：
  ┌──────────┐     ┌──────────┐
  │   ╱╲     │     │   ╱╲     │
  │  ╱  ╲    │     │  ╱  ╲    │
  │ ╱    ╲   │     │ ╱    ╲   │
  │╱      ╲  │     │╱      ╲  │
  └──────────┘     └──────────┘
  路径长度：10     路径长度：3
  
  → 异常点路径短 = 异常分数高
```

**实现**：

```python
from sklearn.ensemble import IsolationForest

def detect_isolation_forest(X, contamination=0.01):
    """Isolation Forest 多指标联合异常检测"""
    clf = IsolationForest(
        n_estimators=100,
        contamination=contamination,  # 预期异常比例
        random_state=42
    )
    clf.fit(X)
    # predict 返回 -1（异常）/ 1（正常）
    labels = clf.predict(X)
    # decision_function 返回异常分数（越小越异常）
    scores = clf.decision_function(X)
    return labels, scores
```

**关键参数**：
- `n_estimators`：树的数量（通常 100）
- `contamination`：预期异常比例（0.01-0.1）
- `max_samples`：每棵树的样本数（通常 256）

**适用场景**：
- 多指标联合异常（CPU + 内存 + IO + Binder 同时异常）
- 异常比例可估计
- 中等数据量（< 100 万条）

**优势**：
- 无监督（无需标注）
- 计算高效（O(n log n)）
- 多指标联合

**局限**：
- 异常比例参数敏感
- 对局部异常不敏感（全局视角）

### 3.2 One-Class SVM

**原理**：学习正常数据的边界（超平面），超出边界视为异常。

```
One-Class SVM 原理示意：

              决策边界
                ↑
   正常数据 ────┼──── 正常数据
                │
                │
              ↓ 异常
```

**实现**：

```python
from sklearn.svm import OneClassSVM

def detect_one_class_svm(X, nu=0.01):
    """One-Class SVM 异常检测"""
    clf = OneClassSVM(kernel='rbf', gamma='scale', nu=nu)
    clf.fit(X)
    labels = clf.predict(X)  # -1 = 异常, 1 = 正常
    return labels
```

**适用场景**：
- 正常数据丰富、异常数据稀少
- 数据维度高（多指标）
- 边界明确（正常 vs 异常）

**局限**：
- 计算成本高（O(n²)）
- 不适合大数据量
- nu 参数敏感

### 3.3 机器学习方法选型

| 场景 | 推荐方法 | 理由 |
| :--- | :--- | :--- |
| **多指标联合 + 数据量中等** | Isolation Forest | 高效、效果稳定 |
| **高维 + 边界明确** | One-Class SVM | 边界刻画准确 |
| **在线学习（持续更新）** | Isolation Forest（增量） | 支持增量训练 |
| **异常比例极低（< 0.1%）** | One-Class SVM | 边界刻画更细 |

## 4. 深度学习方法：AutoEncoder / LSTM / Transformer

### 4.1 AutoEncoder（自编码器）

**原理**：训练一个"重建网络"，正常数据重建误差小，异常数据重建误差大。

```
AutoEncoder 异常检测原理：

   输入数据                重建数据
   ┌─────┐                ┌─────┐
   │ x_1 │ → Encoder → z → Decoder → x̂_1
   │ x_2 │                              │
   │ ... │                              │
   │ x_n │                              │
   └─────┘                              │
       ↓                                ↓
   正常数据：x̂ ≈ x（重建误差小）
   异常数据：x̂ ≠ x（重建误差大）→ 异常
```

**实现**：

```python
import torch
import torch.nn as nn

class AutoEncoder(nn.Module):
    def __init__(self, input_dim, hidden_dim=64):
        super().__init__()
        self.encoder = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, hidden_dim // 2),
            nn.ReLU()
        )
        self.decoder = nn.Sequential(
            nn.Linear(hidden_dim // 2, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, input_dim),
            nn.Sigmoid()
        )
    
    def forward(self, x):
        z = self.encoder(x)
        x_hat = self.decoder(z)
        return x_hat

def train_autoencoder(model, data, epochs=100):
    """训练 AutoEncoder（只用正常数据）"""
    optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
    criterion = nn.MSELoss()
    
    for epoch in range(epochs):
        for x in data:
            x_hat = model(x)
            loss = criterion(x_hat, x)
            optimizer.zero_grad()
            loss.backward()
            optimizer.step()

def detect_autoencoder(model, data, threshold):
    """AutoEncoder 异常检测"""
    anomalies = []
    for i, x in enumerate(data):
        x_hat = model(x)
        recon_error = torch.mean((x - x_hat) ** 2).item()
        if recon_error > threshold:
            anomalies.append((i, recon_error))
    return anomalies
```

**适用场景**：
- 多指标联合
- 数据分布复杂
- 正常数据丰富（用于训练）

**优势**：
- 自动学习特征
- 复杂模式捕捉

**局限**：
- 需要训练数据
- 黑盒（难解释）
- 训练成本高

### 4.2 LSTM（长短期记忆网络）

**原理**：时序数据建模，预测下一个时间步；预测误差大的视为异常。

```
LSTM 时序异常检测原理：

时间步：  t-2       t-1       t        t+1
        ┌────┐    ┌────┐    ┌────┐    ┌────┐
        │ x_1 │ →  │ x_2 │ →  │ x_3 │ →  │ x_4 │  ← 输入
        └────┘    └────┘    └────┘    └────┘
            ↓        ↓        ↓        ↓
        ┌────┐    ┌────┐    ┌────┐    ┌────┐
        │ h_1 │ →  │ h_2 │ →  │ h_3 │ →  │ h_4 │  ← 隐藏状态
        └────┘    └────┘    └────┘    └────┘
                                    ↓
                              ┌──────────┐
                              │ predict x_4 │
                              └──────────┘
                              ↓
                         与真实 x_4 对比
                         误差大 = 异常
```

**实现**：

```python
class LSTMPredictor(nn.Module):
    def __init__(self, input_dim, hidden_dim=64, num_layers=2):
        super().__init__()
        self.lstm = nn.LSTM(input_dim, hidden_dim, num_layers, batch_first=True)
        self.linear = nn.Linear(hidden_dim, input_dim)
    
    def forward(self, x):
        # x: (batch, seq_len, input_dim)
        lstm_out, _ = self.lstm(x)
        # 预测最后一个时间步
        prediction = self.linear(lstm_out[:, -1, :])
        return prediction

def detect_lstm(model, sequence, threshold):
    """LSTM 时序异常检测"""
    anomalies = []
    for i in range(len(sequence) - sequence_length):
        x = sequence[i:i+sequence_length]
        y_true = sequence[i+sequence_length]
        y_pred = model(x)
        error = torch.mean((y_true - y_pred) ** 2).item()
        if error > threshold:
            anomalies.append((i, error))
    return anomalies
```

**适用场景**：
- 长序列依赖
- 时间模式重要
- 数据充足

**优势**：
- 捕捉长期依赖
- 时序特征自动学习

**局限**：
- 训练成本高
- 长序列计算慢
- 黑盒（难解释）

### 4.3 Transformer 时序模型

**原理**：用自注意力机制替代 LSTM 的循环结构，**并行训练 + 长序列建模**。

```python
class TransformerPredictor(nn.Module):
    def __init__(self, input_dim, d_model=64, nhead=4, num_layers=3):
        super().__init__()
        self.embedding = nn.Linear(input_dim, d_model)
        encoder_layer = nn.TransformerEncoderLayer(
            d_model=d_model, nhead=nhead, batch_first=True
        )
        self.transformer = nn.TransformerEncoder(encoder_layer, num_layers)
        self.output = nn.Linear(d_model, input_dim)
    
    def forward(self, x):
        x = self.embedding(x)
        x = self.transformer(x)
        return self.output(x[:, -1, :])  # 预测最后一个时间步
```

**适用场景**：
- 长序列（> 1000 时间步）
- 多维时序
- 训练资源充足

**优势**：
- 并行训练（比 LSTM 快）
- 长序列建模能力强
- 自注意力可解释

**局限**：
- 计算资源大
- 训练数据需求大
- 超参敏感

### 4.4 深度学习方法选型

| 场景 | 推荐方法 | 理由 |
| :--- | :--- | :--- |
| **多指标 + 中等序列长度** | AutoEncoder | 简单、稳定 |
| **长序列 + 时间模式** | LSTM | 经典、稳定 |
| **超长序列 + 多维** | Transformer | 长序列能力强 |
| **实时性要求** | AutoEncoder | 推理快 |
| **可解释性要求** | Transformer（注意力可视化） | 注意力可解释 |

## 5. 大模型方法：LLM-as-detector

### 5.1 LLM-as-detector 原理

**核心思想**：把时序数据**转化为自然语言 / 表格描述**，让 LLM 直接判断"是否异常"。

**示例 Prompt**：

```
你是一名 Android 稳定性工程师。请判断以下主线程帧时间序列是否异常。

时间窗口：最近 60 秒（每 100ms 一个采样点，共 600 个点）

帧时间统计：
- 平均：16.2ms
- P50：16.0ms
- P95：18.5ms
- P99：25.0ms
- 最大：250.0ms（异常突增）
- 超过 16ms 的占比：5.2%
- 超过 32ms 的占比：0.5%

最近 5 个采样点（ms）：16, 16, 17, 16, 250

请输出：
1. 是否异常（是 / 否）
2. 异常等级（低 / 中 / 高）
3. 可能原因（基于 Android 稳定性经验）
4. 建议排查方向
```

**输出**：
```
1. 异常：是
2. 等级：高
3. 可能原因：单帧卡死（250ms 远超正常 16ms），可能是主线程长任务 / IO 阻塞 / GC 暂停
4. 建议排查：
   - 检查主线程 Message 队列是否有长任务
   - 查看 Choreographer 跳帧原因
   - 抓取 Perfetto trace 看阻塞事件
```

### 5.2 LLM-as-detector 实现

```python
import openai

def detect_llm(metrics_summary, recent_samples, model="gpt-4o"):
    """LLM-as-detector 时序异常检测"""
    prompt = f"""
    你是一名 Android 稳定性工程师。请判断以下主线程帧时间序列是否异常。

    时间窗口：最近 60 秒
    帧时间统计：
    - 平均：{metrics_summary['mean']}ms
    - P95：{metrics_summary['p95']}ms
    - P99：{metrics_summary['p99']}ms
    - 最大：{metrics_summary['max']}ms
    最近 5 个采样点（ms）：{', '.join(map(str, recent_samples))}

    请输出（JSON）：
    {{
      "is_anomaly": true/false,
      "severity": "low/medium/high",
      "likely_causes": ["原因1", "原因2"],
      "next_steps": ["排查方向1", "排查方向2"]
    }}
    """
    response = openai.ChatCompletion.create(
        model=model,
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"}
    )
    return json.loads(response.choices[0].message.content)
```

### 5.3 优缺点

**优势**：
- 自然语言输出（可解释性强）
- 综合多种指标
- 无需训练数据
- 可结合历史经验

**局限**：
- 单次调用成本高（$0.005-0.01）
- 延迟高（1-5s）
- 不能处理长序列（上下文窗口限制）
- LLM 幻觉风险

### 5.4 LLM-as-detector 适用场景

| 场景 | 适用度 | 理由 |
| :--- | :--- | :--- |
| **事后分析（人工确认）** | ✅ 高 | 可解释性强 |
| **异常解释 / 工单生成** | ✅ 高 | 自然语言输出 |
| **实时检测（高频）** | ❌ 低 | 成本高 + 延迟高 |
| **首次出现的未知异常** | ✅ 高 | 依赖经验推理 |

### 5.5 LLM 与其他方法的混合架构

**实战推荐**：

```

├──────────────────────────────────────────────────────────┤
```

**误报治理**：

```
误报治理策略：

1. 多指标联合确认
   - 单指标异常 → 等待 30s → 多指标确认
   - 减少瞬时波动误报

2. 时间窗口平滑
   - 异常必须持续 N 秒才告警

4. 用户反馈闭环
   - 用户反馈"系统卡" → 标记为 TP（True Positive）
   - 用户无反馈 → 标记为 FP（False Positive）
   - 在线学习调整阈值
```

## 8. 实战案例：某 App 主线程帧时间异常检测（误报率 < 5%）

**现象**：某 IM App 上线后频繁收到"卡顿"反馈，但稳定性团队无法快速定位是哪个时段、哪个页面卡顿。

**环境**：Android 14 (AOSP 14.0.0_r1) / Kernel 5.10 / 设备 1000 万 DAU。

### 阶段 1：数据采集

**采集方案**：
- Choreographer.FrameCallback → 主线程帧时间（vsync 时间）
- 采样频率：60Hz（与系统 vsync 对齐）
- 数据落盘：1Hz 聚合（P50 / P95 / P99 / Max）

```java
// Android 端采样代码
Choreographer.getInstance().postFrameCallback(new Choreographer.FrameCallback() {
    @Override
    public void doFrame(long frameTimeNanos) {
        long vsyncTimeNanos = SystemClock.elapsedRealtimeNanos();
        long frameDuration = vsyncTimeNanos - lastFrameTime;
        // frameDuration 单位：纳秒
        frameTimes.add(frameDuration);
        lastFrameTime = vsyncTimeNanos;
        // 1 秒聚合一次上报
        ...
    }
});
```

### 阶段 2：异常检测（Isolation Forest + Z-Score）

**多指标联合**：

| 指标 | 类型 | 异常条件 |
| :--- | :--- | :--- |
| P99 帧时间 | 主线程卡顿 | > 32ms 占比 > 5% |
| P99 vs P50 比值 | 抖动检测 | > 5x |
| 主线程 CPU 占用 | 资源压力 | > 80% 持续 30s+ |
| Binder 调用频次 | IPC 压力 | 突增 5x+ |

**检测架构**：
- Isolation Forest 多指标联合（4 个特征）
- Z-Score + EMA 单指标基线漂移检测
- 两者结果 AND 起来作为最终异常事件

### 阶段 3：告警治理

**告警合并**：
- 5min 窗口内相同根因的异常 → 合并为 1 条告警
- 告警量从 1000+ / 故障 → 100-200 / 故障（-80%）

**告警优先级**：
- 高：主线程长任务 > 200ms + 持续 30s+
- 中：P99 帧时间 > 50ms 持续 1min+
- 低：单指标轻微异常

**告警抑制**：
- 同一异常 1h 内不重复告警
- 用户无反馈的告警自动降级

### 阶段 4：效果验证

**修复前后对比**：

```

│ 指标                                  │ 修复前     │ 修复后     │
├──────────────────────────────────────┼───────────┼───────────┤
│ 异常检测覆盖率                         │ 60%       │ 92%       │
│ 误报率                                │ 15%       │ 4.5%      │
│ 告警量 / 故障                          │ 1000+     │ 150       │
```

## 9. 总结（架构师视角的 5 条 Takeaway）

1. **时序异常检测是 AI for Stability 的基础**——没有异常检测就没有早期预警和智能归因。**F02 是 F03/F04/F06 的输入**。
2. **4 类算法各有适用场景**——统计方法（简单 / 实时）、机器学习（多指标联合）、深度学习（复杂模式）、LLM（可解释 / 兜底）。**实战推荐 Isolation Forest + Z-Score 组合 + LLM 兜底**。
3. **工程化落地比算法选择更重要**——数据采集、降采样、特征工程、阈值调优、告警治理每一个环节都可能让模型失效。**70% 的工作量在工程化**。
4. **误报率 < 5% 是工程目标，不是算法目标**——单纯调模型无法达到，需要多指标联合 + 告警合并 + 用户反馈闭环。
5. **LLM-as-detector 是"最后 1 公里"**——传统算法负责高频实时检测，LLM 负责低频可解释输出。**混合架构是当前最优解**。

**时序异常检测决策树**：

```
新项目要做时序异常检测
  ↓
数据规模？
  ├─ 小（< 100 万条） → 统计方法（3σ / IQR）
  ├─ 中（100 万-1 亿） → 机器学习（Isolation Forest）
  └─ 大（> 1 亿） → 深度学习（LSTM / Transformer）
  ↓
实时性要求？
  ├─ 强（秒级） → Isolation Forest + 滑动窗口
  ├─ 中（分钟级） → LSTM 在线预测
  └─ 弱（小时级） → Transformer 离线训练
  ↓
可解释性要求？
  ├─ 强 → Transformer（注意力可视化）+ LLM 解释
  ├─ 中 → Isolation Forest（特征重要性）
  └─ 弱 → AutoEncoder（黑盒）
  ↓
成本预算？
  ├─ 低 → 端侧推理（量化模型 < 10MB）
  ├─ 中 → 云端 GPU 推理
  └─ 高 → 云端 LLM 兜底
```

| **误报率目标** | < 5% | 行业标准 | > 10%→不可用 |
| **异常检测延迟** | < 1s（实时）/ < 1min（近实时） | 视场景 | 太慢→失去预警价值 |
| **冷启动策略** | 通用模型兜底 + 1 周数据采集 | 视业务 | 无兜底→新功能漏检 |
