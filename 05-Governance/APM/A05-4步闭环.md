# 1. 为什么需要 4 步闭环？

> **铁律**：**没有闭环的治理 = 没有治理**——APM 平台建完不用 = 0 价值

| 现状 | 后果 |
|:-----|:-----|
| 只度量不决策 | 数据成了"看板摆设" |
| 只决策不行动 | 决策成了一纸空文 |
| 只行动不回归 | 同一个问题下周再发 |

**4 步闭环 = 度量 → 决策 → 行动 → 回归** —— 每一步都有 owner、时限、可验收

# 2. 4 步闭环总览

```
┌────────────────────────────────────────────────────┐
│  Step 1 度量（持续）                                  │
│  - 5 大度量（A02）+ 6 大门禁（A03）                 │
│  - owner: APM 团队                                  │
│  - 时限: 实时 / 5 分钟 / 1 小时                     │
└────────────────────────────────────────────────────┘
              ↓
┌────────────────────────────────────────────────────┐
│  Step 2 决策（小时级）                                │
│  - 触发门禁：P0 阻断 / P1 警告 / P2 通知            │
│  - owner: oncall + TL                               │
│  - 时限: 5 分钟决策                                  │
└────────────────────────────────────────────────────┘
              ↓
┌────────────────────────────────────────────────────┐
│  Step 3 行动（天级）                                  │
│  - 修复 / 回滚 / 灰度 / 应急发版                    │
│  - owner: 业务团队 + oncall                         │
│  - 时限: P0 ≤ 4h / P1 ≤ 1d / P2 ≤ 1 周            │
└────────────────────────────────────────────────────┘
              ↓
┌────────────────────────────────────────────────────┐
│  Step 4 回归（周/月级）                               │
│  - postmortem + Action Items + 门禁更新            │
│  - owner: 稳定性架构师                              │
│  - 时限: 24h postmortem / 季度 review              │
└────────────────────────────────────────────────────┘
              ↓
       反馈到 Step 1（数据驱动）
```

# 3. Step 1：度量

## 3.1 度量什么

| 度量 | 来源 | 频次 | owner |
|:-----|:-----|:-----|:------|
| 5 大 SLI | [A02-5 大度量](A02-5大度量后端实现.md) | 实时 | APM 团队 |
| 6 大门禁 | [A03-6 大门禁](A03-6大门禁维度.md) | 每次发版 | QA + APM |
| Stability Score | [A04-Stability Score](A04-StabilityScore综合指数.md) | 实时 | APM 团队 |

## 3.2 度量 SOP

```
SOP 1: 每 5 分钟 APM 自动采集
  - 5 大 SLI 计算
  - 与基线对比
  - 异常触发告警

SOP 2: 每天 9:00 输出日报
  - 综合分 + 6 维度分
  - Top 10 异常
  - 趋势对比

SOP 3: 每周一输出周报
  - Stability Score 周环比
  - 跨版本对比
  - 改进项
```

# 4. Step 2：决策

## 4.1 决策树

```
告警触发
   │
   ├── P0 阻断
   │     │
   │     ├── 立即升级到架构师
   │     ├── 决定：修复 / 回滚 / 灰度
   │     └── 时限: 5 分钟
   │
   ├── P1 警告
   │     │
   │     ├── 升级到 TL
   │     ├── 决定：下个版本修 / 紧急发版
   │     └── 时限: 1 小时
   │
   └── P2 通知
         │
         ├── 通知业务团队
         ├── 决定：跟踪 / 下季度
         └── 时限: 1 周
```

## 4.2 决策 SOP

```
SOP 1: oncall 收到 P0 告警
  - 5 分钟内必介入
  - 拉群 + 抓数据
  - 15 分钟内出二报
  - 30 分钟内给决策

SOP 2: 决策矩阵（oncall 必背）
  - 已知 bug + 紧急 → 热修
  - 未知 bug + 影响大 → 回滚
  - 第三方 bug → 降级 + 切回旧版
  - 机制问题 → 绕开 + 报告

SOP 3: 升级机制
  - 5 分钟未响应 → 备 R1
  - 15 分钟未定位 → TL
  - 30 分钟未恢复 → 架构师
  - 1 小时未恢复 → 全员应急
```

详见 [OC01-oncall 工程总论](../Oncall/OC01-oncall工程总论：值班机制与工具栈.md)。

# 5. Step 3：行动

## 5.1 4 类行动

| 行动 | 触发 | 时限 | owner |
|:-----|:-----|:-----|:------|
| **修复** | 已知根因 + 可热修 | 4h | 业务团队 |
| **回滚** | 未知根因 + 影响大 | 30min | oncall + 平台 |
| **灰度** | 中等风险 | 1-3 天 | 业务团队 + 平台 |
| **应急发版** | 紧急 + 已有 fix | 2h | 业务团队 + CI |

## 5.2 行动 SOP

```
SOP 1: 修复
  - 5 Whys 定位根因
  - 写修复 + Code Review
  - 单测 + 灰度
  - 全量

SOP 2: 回滚
  - 灰度期间：降到 0%
  - 全量期间：一键回滚
  - 监控验证恢复

SOP 3: 灰度
  - 1% → 10% → 50% → 100%
  - 每阶段观察 SLO
  - 满足后放量

SOP 4: 应急发版
  - 走紧急 CI 通道
  - 30-60 分钟出包
  - 1% → 10% → 100%
```

## 5.3 行动 owner 表

| 类型 | owner | 必通知 |
|:-----|:-------|:-------|
| Java 崩溃 | Java 团队 | oncall + TL |
| Native 崩溃 | Native 团队 | oncall + TL + 芯片厂 |
| ANR | Framework 团队 | oncall + TL |
| 性能 | 性能团队 | oncall + TL |
| 兼容 | OEM 团队 | oncall + TL + OEM |
| 安全 | 安全团队 | oncall + TL + 法务 |

# 6. Step 4：回归

## 6.1 回归 3 件套

```
件套 1: postmortem
  - 24h 内出
  - 5 Whys + Action Items
  - 不追责

件套 2: Action Items
  - 72h 内出短期项
  - 2 周内出中期项
  - 季度内出长期项
  - 必带 owner + deadline

件套 3: 门禁更新
  - 补充新检查项
  - 调阈值
  - 加自动化测试
```

## 6.2 postmortem 模板

详见 [S10-02-SLI与SLO设计 §6 postmortem 模板](../../02-Symptom/S10-Measure/02-SLI与SLO设计：从指标到门禁.md)。

## 6.3 门禁更新 SOP

```
SOP 1: Action Items 必转门禁
  - P0 短期项 → 加 lint / 单测
  - P0 中期项 → 加准出门禁
  - P0 长期项 → 加灰度门禁

SOP 2: 季度门禁 review
  - 当前门禁覆盖度
  - 漏网事件分析
  - 阈值调整
```

# 7. 5 类真实场景

## 7.1 场景 1：JE 率突增 0.05% → 0.15%

**Step 1 度量**：APM 告警 JE 率 > 0.1%
**Step 2 决策**：P0 阻断 + 决定回滚
**Step 3 行动**：回滚到上一个稳定版本
**Step 4 回归**：postmortem 写"未跑端到端测试导致" → 加 E2E 门禁

## 7.2 场景 2：ANR 率持续 0.2%

**Step 1 度量**：ANR 率稳定在 0.2%（不恶化）
**Step 2 决策**：P1 警告（不阻断）
**Step 3 行动**：定位 Input ANR 占 70% → 修主线程
**Step 4 回归**：4 周后 ANR 率降至 0.05% → 加 Input ANR 阈值告警

## 7.3 场景 3：高端机启动 P95 突然 3s

**Step 1 度量**：启动 P95 突增 50%
**Step 2 决策**：P0 阻断（影响核心体验）
**Step 3 行动**：灰度回滚 + 修启动期同步 IO
**Step 4 回归**：加启动期 IO 静态扫描

## 7.4 场景 4：低端机 NE 率高

**Step 1 度量**：低端机 NE 率 0.5%（高端机 0.05%）
**Step 2 决策**：P1 警告
**Step 3 行动**：针对性优化低端机
**Step 4 回归**：低端机 NE 率降至 0.1% → 加低端机自动化测试

## 7.5 场景 5：ANR-free Session 缓慢下降

**Step 1 度量**：ANR-free Session 6 周从 99.9% 降至 99.5%
**Step 2 决策**：P2 通知
**Step 3 行动**：分析 6 周趋势
**Step 4 回归**：发现是 Input ANR 渐进恶化 → 修 1 个底层问题，回归 99.9%

# 8. 4 步闭环的反例

| # | 反例 | 错误做法 | 正确做法 |
|:-:|:-----|:---------|:---------|
| 1 | **只度量不决策** | 看数据但不行动 | **每数据必决策** |
| 2 | **只决策不行动** | 写决策但没执行 | **每决策必执行** |
| 3 | **只行动不回归** | 修完不验证 | **每行动必验证** |
| 4 | **没有 owner** | 谁都管谁都不管 | **4 步必有 owner** |
| 5 | **没时限** | 慢慢做 | **P0 4h / P1 1d / P2 1w** |
| 6 | **postmortem 走过场** | 不写 5 Whys | **必写 5 Whys** |
| 7 | **Action Items 没 deadline** | 永远拖 | **必带 deadline** |
| 8 | **不更新门禁** | 不沉淀 | **Action Items 必转门禁** |
| 9 | **没人 review** | 自己 review 自己 | **架构师 review** |
| 10 | **不复盘同类** | 单点修复 | **横向 review 同类** |

# 9. 4 步闭环的度量

| 度量 | 公式 | 目标 |
|:-----|:-----|:-----|
| **闭环率** | 闭环数 ÷ 度量事件数 | ≥ 90% |
| **决策时长** | 告警到决策的时间 | P0 ≤ 5min |
| **行动时长** | 决策到行动完成 | P0 ≤ 4h |
| **回归时长** | 行动到 postmortem | ≤ 24h |
| **Action Items 关闭率** | 关闭 ÷ 总数 | ≥ 80% |
| **Stability Score 提升** | 当前 - 上月 | ≥ +1 分 |

---

# 10. 5 条 Takeaway

1. **4 步闭环 = 度量 → 决策 → 行动 → 回归** —— 缺一不可
2. **每步必带 owner + 时限** —— P0 5min 决策 / 4h 行动 / 24h postmortem
3. **postmortem 不追责** —— 必写 5 Whys + Action Items
4. **Action Items 必转门禁** —— 否则同类下周再发
5. **闭环率必 ≥ 90%** —— 否则就是"有度量无治理"
