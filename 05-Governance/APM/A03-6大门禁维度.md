# 1. 6 大门禁维度全景

> **铁律**：稳定性门禁**不是 1 个维度**，是 **6 大维度 × 4 阶段** 的矩阵

```
   准入 (Pre)     准出 (Gate)     灰度 (Canary)     全量 (GA)
   ↓              ↓               ↓                 ↓
  代码质量        集成完成         1% / 10% / 50%    100%
  + Lint          + 端到端         + 监控            + 监控
                  + 性能基线
```

| # | 维度 | 关注点 | 阻断类型 |
|:-:|:-----|:-------|:---------|
| 1 | **崩溃** | JE/NE/KE | 任何崩溃率恶化 ≥ 5% 阻断 |
| 2 | **ANR** | Input/Service/Broadcast | ANR 率恶化 ≥ 3% 阻断 |
| 3 | **性能** | 启动/帧率/内存/电量 | 启动 P95 恶化 ≥ 10% 阻断 |
| 4 | **兼容** | 设备/版本/SDK | 崩溃率在 ≥ 2% 设备上恶化阻断 |
| 5 | **安全** | 漏洞/权限/数据 | 任何 P0 漏洞阻断 |
| 6 | **业务** | 核心 KPI | 业务转化率恶化 ≥ 5% 阻断 |

# 2. 4 阶段门禁流程

## 2.1 阶段 1：准入（Pre-Gate）

**触发点**：代码合入 main 分支
**自动化**：CI 流水线

```
代码 commit → Lint + 单测 + 静态扫描 + 安全扫描
   ↓ (全部通过)
触发构建
   ↓ (构建成功)
进入准出门禁
```

## 2.2 阶段 2：准出（Gate）

**触发点**：构建产物出包后
**自动化**：CI 流水线

```
构建产物 → 端到端测试 + 性能基线 + 兼容性测试
   ↓ (全部通过)
进入灰度发布
   ↓
灰度门禁
```

## 2.3 阶段 3：灰度（Canary）

**触发点**：灰度发布平台
**自动化**：灰度发布系统 + 监控

```
1% 灰度 → 10% 灰度 → 50% 灰度 → 100%
   ↓            ↓            ↓
 24h 观察     48h 观察     72h 观察
 满足后放量   满足后放量   满足后放量
```

## 2.4 阶段 4：全量（GA）

**触发点**：灰度完成
**自动化**：监控 + 告警 + 自动回滚

```
全量发布
   ↓
持续监控（7 天）
   ↓
发现问题 → 自动回滚 + 5 分钟告警
```

# 3. 维度 1：崩溃门禁

## 3.1 准入（Pre）

| 检查项 | 工具 | 阈值 | 失败处理 |
|:-------|:-----|:-----|:---------|
| 静态 crash 关键字扫描 | 自研脚本 | 0 命中 | 阻断 |
| 单元测试崩溃场景 | JUnit | 100% 通过 | 阻断 |
| Native crash 复现测试 | AddressSanitizer | 0 命中 | 阻断 |

## 3.2 准出（Gate）

| 检查项 | 工具 | 阈值 | 失败处理 |
|:-------|:-----|:-----|:---------|
| 端到端崩溃测试 | UI Automator | 0 崩溃 | 阻断 |
| JE 率基线 | APM | ≤ baseline × 1.1 | 阻断 |
| NE 率基线 | APM | ≤ baseline × 1.1 | 阻断 |
| Crash 关键字回归 | 自研脚本 | 0 新关键字 | 阻断 |

## 3.3 灰度（Canary）

| 阶段 | 比例 | 观察时长 | 放行条件 | 回滚条件 |
|:-----|:----:|:--------:|:---------|:---------|
| 1% | 1% | 24h | JE/NE 率不恶化 | 恶化 ≥ 50% 立即回滚 |
| 10% | 10% | 48h | 持平 | 恶化 ≥ 5% 回滚 |
| 50% | 50% | 72h | 持平 | 恶化 ≥ 3% 回滚 |

## 3.4 全量（GA）

| 触发 | 动作 | 响应时长 |
|:-----|:-----|:---------|
| 任意崩溃率恶化 ≥ 1% | 自动回滚 | 5 分钟内 |
| 单一崩溃堆栈占新增 50%+ | 告警 + 人工 review | 30 分钟 |
| 7 天内崩溃率累计恶化 ≥ 0.5% | 暂停变更 | 1 天 |

# 9. 门禁分级模型

> **3 级门禁** —— 不是所有检查项都阻断

| 级别 | 含义 | 触发 | 动作 |
|:-----|:-----|:-----|:-----|
| **P0 阻断** | 致命问题 | 任何 1 项不通过 | 阻断发布 |
| **P1 警告** | 重要问题 | 任意 1 项不通过 | 警告 + 人工 review |
| **P2 通知** | 优化建议 | 任意 1 项不通过 | 通知 + 跟踪 |

## 9.1 6 维度的门禁级别分配

| 维度 | P0 阻断 | P1 警告 | P2 通知 |
|:-----|:--------|:--------|:--------|
| 崩溃 | ✓ | | |
| ANR | ✓ | | |
| 性能 | ✓ | | |
| 兼容 | | ✓（部分）| |
| 安全 | ✓（P0 漏洞）| ✓（中危）| ✓（低危）|
| 业务 | | ✓（核心）| ✓（边缘）|

## 9.2 4 阶段的门禁级别

| 阶段 | 阻断条件 |
|:-----|:---------|
| 准入 | 1 项 P0 不通过 = 阻断 |

# 11. 6 反例清单

# 12. 5 条 Takeaway

1. **6 维门禁矩阵**（崩溃 / ANR / 性能 / 兼容 / 安全 / 业务）—— 单维度不够
2. **4 阶段门禁流程**（准入 / 准出 / 灰度 / 全量）—— 灰度不严 = 风险
3. **3 级门禁分级**（P0 阻断 / P1 警告 / P2 通知）—— 一刀切 = 没门禁
4. **每维度给"具体阈值"** —— 不要"通常""大概"
5. **每季度 review 门禁** —— 业务变了，阈值要变
