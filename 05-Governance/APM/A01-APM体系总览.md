# 1. 为什么需要 APM 平台？

## 1.1 单症状视角的局限

[S00-S10](../../02-Symptom/) 11 篇 + [F00-F07](../../03-Forensics/) 8 篇讲了**怎么发现 + 怎么修**——但**没有讲"数据怎么回流到架构师"**：

| 已有系列 | 解决的问题 | 留下的空白 |
|:---------|:---------|:----------|
| S00-S10 | 怎么定义 / 排查 / 量化症状 | **数据怎么采**？|
| F00-F07 | 怎么取证 + 怎么治理 | **数据怎么存 + 怎么展示**？|

**问题**：架构师做稳定性决策时，**需要"全量数据视图"**——**没有 APM 平台 = 瞎子摸象**。

## 1.2 APM 平台的 3 大价值

| 价值 | 含义 | 收益 |
|:-----|:-----|:-----|
| **1. 全量数据** | 100% 用户 100% 设备的稳定性数据 | 1 个崩溃可见 100% 用户 |
| **2. 实时告警** | 异常指标超阈值立即告警 | 5 分钟响应 vs 24 小时发现 |
| **3. 趋势分析** | 跨时间 / 跨版本 / 跨用户群对比 | 回归发现 / 优化验证 |

> **架构师视角**：**没有 APM 平台 = 治理闭环断在"度量"环节**（S10 §6 闭环的第一步就缺了）。

## 1.3 APM 平台的 4 大误区

| 误区 | 错在哪 | 正确做法 |
|:-----|:-------|:---------|
| **1. 商业 SaaS 一劳永逸** | 商业 SaaS 不解决"自研指标 + 自定义告警" | 商业为主 + 自研补充 |
| **2. 采集越多越好** | 性能损耗 + 存储爆炸 | 关键指标 + 抽样采集 |
| **3. 实时 = 一切** | 实时指标会爆存储 | 实时 + 准实时 + 离线分层 |
| **4. UI 越炫越好** | 工程师用不上炫 UI | **oncall 第一反应表优先** |

# 2. APM 3 大模块：采集 / 后端 / 治理

> **本节是 A01 的核心**——3 大模块是 APM 平台架构骨架。

## 2.1 模块 1：数据采集（Data Collection）

**做什么**：从设备端采集稳定性相关数据

**5 类采集源**：

| 采集源 | 触发时机 | 数据类型 | 工具 |
|:-------|:---------|:---------|:-----|
| **崩溃 dump** | 崩溃触发时 | tombstone / dropbox / logcat | debuggerd / AnrHelper |
| **ANR traces** | ANR 触发时 | anr traces + dropbox | ActivityManager |
| **性能指标** | 周期 / 事件 | FPS / 启动时间 / 内存 | Choreographer / Perfetto |
| **业务事件** | 业务触发 | 业务路径成功率 | 自埋点 |
| **环境信息** | App 启动时 | 设备型号 / OS / 网络 | Build / ConnectivityManager |

**关键指标**（**采集优先级**）：

| 优先级 | 指标 | 采集频率 | 存储 |
|:-------|:-----|:---------|:-----|
| **P0** | 崩溃 / ANR / 启动时间 | 事件触发 | 全量 |
| **P1** | FPS / 内存 / 电量 | 周期 10s | 抽样 1% |
| **P2** | 业务事件 | 业务触发 | 抽样 0.1% |

> **架构师视角**：**P0 全量 + P1 抽样 + P2 抽样更低**——这是"采集不爆存储"的铁律。

**5 类采集源触发时序图**：

```
T+0      T+1s      T+10s     T+30s     T+5min    T+1h
│        │         │         │         │         │
▼        ▼         ▼         ▼         ▼         ▼
[App 启动]
         │
         ├─ [T+0: 环境信息采集]
         │   ├─ 设备型号 / OS / 网络
         │   └─ Build / ConnectivityManager
         │
         ├─ [T+1s: 性能指标启动]
         │   ├─ Choreographer 启动
         │   ├─ Perfetto 启动
         │   └─ 周期 10s 采集 FPS / 内存
         │
         ├─ [T+10s: 业务事件]
         │   └─ 自埋点（业务路径）
         │
         ├─ [周期 10s: 持续性能采集]
         │   ├─ FPS
         │   ├─ 内存
         │   └─ 电量
         │
         └─ [事件触发: 崩溃 / ANR / OOM]
             ├─ debuggerd → tombstone
             ├─ AnrHelper → dropbox
             └─ SDK FileObserver → 抓 dump

【5 类采集源 + 触发时机 + 频率总览】

  ┌──────────────────┬──────────────┬──────────┬──────────┐
  │ 采集源            │ 触发时机     │ 频率    │ 优先级  │
  ├──────────────────┼──────────────┼──────────┼──────────┤
  │ 环境信息          │ App 启动时   │ 1 次    │ P0      │
  │ 崩溃 dump         │ 崩溃触发时   │ 事件     │ P0      │
  │ ANR traces        │ ANR 触发时   │ 事件     │ P0      │
  │ 性能指标          │ 周期 10s    │ 周期     │ P1      │
  │ 业务事件          │ 业务触发     │ 业务     │ P2      │
  └──────────────────┴──────────────┴──────────┴──────────┘
```

## 2.2 模块 2：后端处理（Backend Processing）

**做什么**：从设备上报到数据可视化

**5 大组件**：

| 组件 | 职责 | 选型 |
|:-----|:-----|:-----|
| **数据接入** | 接收 SDK 上报 | Kafka（高吞吐）/ Pulsar |
| **流处理** | 实时聚合 + 异常检测 | Flink / Spark Streaming |
| **批处理** | 离线分析 + 报表 | Spark / Hive |
| **存储** | 时序数据 + 维度数据 | ClickHouse / Druid / ElasticSearch |
| **告警** | 阈值检测 + 通知 | Prometheus AlertManager / 自研 |

**关键性能指标**：

- **数据接入吞吐**：单 Kafka topic > 10 万 msg/s（**亿级用户必备**）
- **流处理延迟**：P99 < 10s（**实时告警必备**）
- **存储压缩比**：> 10:1（**成本控制**）
- **查询响应**：P99 < 1s（**oncall 体验**）

## 2.3 模块 3：治理应用（Governance Application）

**做什么**：把数据转化为治理决策

**4 类应用**：

| 应用 | 目标用户 | 关键功能 |
|:-----|:---------|:---------|
| **oncall 看板** | oncall 工程师 | 实时指标 + Top 崩溃 + 告警 |
| **稳定性报表** | 架构师 / PM | 周报 / 月报 / 季度 OKR |
| **告警系统** | oncall + 架构师 | 阈值告警 / 异常检测告警 |
| **CI/CD 卡点** | CI / 发布 | release 门禁 |

> **架构师视角**：**3 大模块中"治理应用"是用户感知的**——oncall 看板 + 告警系统 = 团队每天 80% 的使用场景。

## 2.4 3 大模块的关系

**采集 → 后端 → 治理 = 单向数据流**：

```
【设备端】                     【后端】                     【用户】
┌─────────┐                 ┌────────────┐               ┌─────────┐
│  APM SDK │ ──HTTP/MQTT──> │  Kafka     │ ──Flink──>   │ oncall  │
│  - 崩溃  │                 │  - 接入     │               │ 看板    │
│  - ANR   │                 │  - 聚合     │               │         │
│  - 性能  │                 │  - 告警     │               │ 报表    │
└─────────┘                 └────────────┘               │         │
       │                       │                          │ 告警    │
       │                       │                          │         │
       │                       ▼                          │ CI 卡点 │
       │                  ┌──────────┐                    └─────────┘
       │                  │ClickHouse│
       │                  │ 时序存储  │
       │                  └──────────┘
       │
       └─ 商业 SaaS：Sentry / Crashlytics
```

# 3. APM 完整数据流（设备 → SDK → 后端 → 展示）

> **本节讲完整数据流**——从崩溃发生到架构师看到数据，**每一步都关键**。

## 3.1 数据流 5 步

**Step 1：设备端崩溃发生**
- 崩溃触发（Java / Native / ANR）
- 设备端 debuggerd / AnrHelper / ActivityManager 抓取 dump
- dump 文件写入 /data/tombstones / /data/anr / /data/system/dropbox

**Step 2：APM SDK 抓取**
- SDK 注册 FileObserver 监听 /data/anr / /data/tombstones 变化
- SDK 读取 dump + .so（NE 时）+ 设备信息
- SDK 加密 + 压缩（**性能考虑**）
- SDK 上报到后端（HTTP POST 或 MQTT）

**Step 3：后端接入**
- API Gateway 接收 + 鉴权
- 写入 Kafka（**削峰 + 解耦**）
- 上报成功 → SDK 标记 dump 文件已上传 → 设备清理 dump

**Step 4：流处理**
- Flink 消费 Kafka
- 数据清洗 + 去重 + 聚合
- 维度关联（设备型号 / OS / 网络 / 用户群）
- 异常检测（**同比 / 环比**）

**Step 5：存储 + 告警 + 展示**
- ClickHouse 存储（时序 + 维度）
- 告警规则检测（阈值 / 异常）
- 看板 / 报表 / 告警系统
- CI/CD 门禁（release 决策）

## 3.2 数据流关键时序图

```
T+0      T+50ms    T+100ms   T+200ms   T+1s      T+5s      T+10s
│        │         │         │         │         │         │
▼        ▼         ▼         ▼         ▼         ▼         ▼
[崩溃发生]
         │
         ├─ [T+50ms: debuggerd 抓 NE dump]
         │
         ├─ [T+100ms: dropbox 写入]
         │
         ├─ [T+200ms: SDK FileObserver 触发]
         │   ├─ 读取 dump + .so
         │   ├─ 加密 + 压缩
         │   └─ 标记"待上传"
         │
         ├─ [T+1s: SDK HTTP POST 到 API Gateway]
         │   ├─ 鉴权
         │   ├─ 写入 Kafka
         │   └─ 返回 200
         │
         ├─ [T+1.5s: Flink 消费]
         │   ├─ 清洗 + 去重
         │   ├─ 维度关联
         │   └─ 异常检测
         │
         ├─ [T+5s: ClickHouse 写入]
         │   ├─ 时序数据
         │   └─ 维度数据
         │
         └─ [T+10s: 告警检测 + 看板刷新]
             ├─ 阈值检测（崩溃率 > 0.3% 告警）
             ├─ 异常检测（同环比 +50% 告警）
             └─ 看板 / 报表
```

> **架构师视角**：**端到端延迟 < 10s** 是 oncall 体验的硬指标——超出 10s 用户感知延迟，**崩溃排查效率下降 30%**。

## 3.3 数据流 3 大瓶颈

| 瓶颈 | 原因 | 优化 |
|:-----|:-----|:-----|
| **1. 设备端采集** | SDK 在主线程运行 | 异步化 + WorkManager |
| **2. 网络上报** | 弱网 / 切网 | 离线缓存 + 弱网降级 |
| **3. 后端聚合** | 千万级用户聚合慢 | 预聚合 + 维度剪枝 |

# 4. APM 平台选型：5 大方案对比

> **本节是 APM 落地的"决策矩阵"**——选商业还是自研？这是 5 大团队最常问的问题。

## 4.1 5 大选型对比

| 方案 | 上手成本 | 月成本（亿级用户）| 自定义能力 | 数据合规 | 适用场景 |
|:-----|:---------|:------------------|:----------|:---------|:---------|
| **Sentry** | 中 | $500-2000 | 中 | 中（云端）| 中小团队 + 快速接入 |
| **Firebase Crashlytics** | 低 | 免费 / 中等 | 低 | 中（Google 云）| 中小团队 + 简单场景 |
| **Bugsnag** | 中 | $500-2000 | 中 | 中 | 中小团队 + 多语言 |
| **Backtrace.io** | 高 | $2000+ | 高 | 中（云端）| 专注 NE 排查 |
| **自研（ClickHouse + Flink）** | 高 | $5000-20000 | 极高 | **高**（私有部署）| 大团队 / 金融 / 政府 |

## 4.2 选型决策树

```
【选型决策树】

  你的团队规模？
  │
  ├─ < 10 人 ──> Firebase Crashlytics（免费 + 简单）
  │
  ├─ 10-50 人 ─┬─ 强 NE 排查 ─> Backtrace.io
  │            └─ 通用 + 中等成本 ─> Sentry
  │
  ├─ 50-200 人 ─┬─ 重视数据合规 ─> Sentry 自托管 / 自研
  │             └─ 重视效率 + 通用 ─> Sentry SaaS
  │
  └─ > 200 人 + 金融/政府 ──> 自研（必须）
```

> **架构师视角**：**数据合规是 5 大选型的"分水岭"**——涉及 PII（个人身份信息）必须自研或私有部署。

## 4.3 商业 vs 自研的核心权衡

| 维度 | 商业 SaaS | 自研 |
|:-----|:---------|:-----|
| **上手时间** | 1-2 周 | 3-6 月 |
| **月成本** | $500-2000 | $5000-20000 + 1-3 人月 |
| **数据合规** | 中（云端）| **高（私有部署）** |
| **自定义告警** | 受限 | **无限** |
| **自定义指标** | 受限 | **无限** |
| **崩溃聚合算法** | 黑盒 | **完全可控** |
| **可观察性** | 文档 + 工单 | **源码可读** |
| **扩展性** | 受限 | **完全可控** |

**经验法则**：
- **团队 < 50 人 + 数据合规要求低** → 商业 SaaS（Sentry / Crashlytics）
- **团队 > 50 人 + 重视自定义** → Sentry 自托管 + 补充自研
- **团队 > 200 人 + 金融/政府** → 全面自研

# 5. APM 5 大红线（平台建设硬约束）

> **本节是 APM 平台建设的"硬底线"**——5 大红线 = 平台验收的最低标准。

## 红线 1：性能损耗

**红线**：APM SDK 在设备端的性能损耗 **< 3%**

**测算**：
- SDK 启动耗时：< 100ms
- 内存占用：< 10MB
- CPU 占用：< 1%
- 上报流量：< 1MB/天/用户

**超出红线的后果**：
- 用户报卡
- 用户主动卸载
- 平台被团队抵制

## 红线 2：数据合规

**红线**：PII（个人身份信息）100% 脱敏

**必做**：
- 设备 ID 哈希（**不能明文**）
- 用户名 / 手机号 / 邮箱 / 地址 100% 脱敏
- 地理位置（粗粒度到省 / 城市级，**不要 GPS 精坐标**）
- 跨境数据传输 合规审查

**违反红线的后果**：
- **GDPR / 中国个保法 罚款**（最高数亿）
- 用户信任崩溃

## 红线 3：误报率

**红线**：告警误报率 **< 1%**

**测算**：
- 月告警总数：1000
- 月误报数：< 10
- 否则团队会**关掉告警通道**（"狼来了"效应）

**优化**：
- 阈值要严格（**宁可不告**）
- 异常检测要过滤"已知 noise"（如节假日流量低）
- 告警分级（**Critical 一定人工，Medium 自动**）

## 红线 4：漏报率

**红线**：真实异常漏报率 **< 0.1%**

**测算**：
- 真实异常 1000 次
- 漏报 < 1 次
- 否则 oncall 错过 P0 事故

**优化**：
- 多维度关联（崩溃率 + 用户报卡 + 客服反馈）
- 同比 / 环比 + 阈值告警
- 关键指标 P0 全量采集

## 红线 5：成本

**红线**：APM 平台月成本 < 团队总预算 5%

**测算**：
- 团队总预算 100 万
- APM 平台 < 5 万
- 超出 → **平台被砍**

**优化**：
- 商业 SaaS → 自研
- 自研 → 预聚合 + 维度剪枝 + 老数据归档
- 实时指标 → 准实时

> **架构师视角**：**5 大红线中 #2（数据合规）是法律底线，#4（漏报率）是 oncall 底线**——其他 3 项是工程权衡。

# 6. APM 与 S10 / F07 的联动

> **本节讲 APM 平台如何与 S10 度量学、F07 治理体系联动**——形成"度量 → 平台 → 治理"完整闭环。

## 6.1 联动 #1：S10 度量学 → APM 后端实现

| S10 度量 | APM 后端实现 |
|:---------|:-------------|
| **MTBF** | 实时聚合（运行时间 / 故障次数）|
| **崩溃率** | 实时聚合（崩溃数 / 启动数）|
| **ANR 率** | 实时聚合（ANR 数 / 启动数）|
| **严重性** | 离线批处理（多维度加权）|
| **回归率** | 离线批处理（与已知问题库匹配）|
| **Stability Score** | 实时 + 离线混合（综合指数）|

**实现要点**：
- 实时指标走 Flink（**5 大度量中 3 个**）
- 离线指标走 Spark（**严重性 + 回归率**）

# 7. APM 5 步落地法

> **本节是 APM 平台落地的"5 步走"**——从 0 到 1 建平台的标准流程。

# 8. 实战案例

## 8.1 案例 A（CASE-APM-01）：某社交 App 接入 Sentry 实现 APM 平台

> **典型模式**：50-200 人团队 + 重视效率 + 通用场景 → 选 Sentry SaaS

**背景**：
- 某社交 App，5000 万 DAU，团队 80 人
- 已有稳定性度量（S10）+ 治理体系（F07）
- 缺 APM 平台：oncall 靠人工统计崩溃 + 邮件告警

**选型决策**：
- 80 人团队：商业 SaaS 足够
- 重视效率：选 Sentry（生态最好）
- 通用场景：不需要 Backtrace.io 的 NE 专项
- 数据合规：接受 Sentry 托管

**实施 4 个月**：
- 第 1-2 周：Sentry 接入 + 团队培训
- 第 3-6 周：崩溃 / ANR / 性能 5 类采集源接入
- 第 7-12 周：Sentry 看板定制 + 告警规则
- 第 13-16 周：CI/CD 卡点集成（崩溃率 > 0.3% 拒绝 release）

**实施效果**：
- oncall 响应时间：从 4 小时 → 10 分钟（**-96%**）
- 崩溃排查时间：从 60 分钟 → 10 分钟（**-83%**）
- 团队满意度：从"抱怨稳定性" → "信任稳定性"（**质变**）
- 月成本：$1500（Sentry Team plan）

> **架构师视角**：**商业 SaaS 选型 = 快速 + 成本可控**——50-200 人团队 80% 都应该选这条路。

## 8.2 案例 B（CASE-APM-02）：某金融 App 自研 APM 平台（ClickHouse + Flink）

> **典型模式**：> 200 人团队 + 金融 + 数据合规 → 自研

**背景**：
- 某金融 App，2 亿用户，团队 300 人
- **必须自研**（数据合规：金融 PII）
- 已有 5 大度量 + 4 步闭环

**自研架构**（**4 层栈**）：
- **采集层**：APM SDK（含 Sentry 兼容协议）+ FileObserver + AnrHelper
- **接入层**：Kafka（10 万 msg/s）+ API Gateway（鉴权 + 限流）
- **处理层**：Flink（实时聚合 5 大度量）+ Spark（离线批处理严重性 / 回归率）
- **存储层**：ClickHouse（时序数据）+ Druid（维度聚合）+ HDFS（冷数据归档）
- **应用层**：oncall 看板（自研）+ 报表（自研）+ 告警（自研）+ CI/CD 卡点

**实施 12 个月 + 4 人团队**：
- 第 1-3 月：架构设计 + 选型 + 技术验证
- 第 4-6 月：采集 SDK v1.0 + 接入层 v1.0
- 第 7-9 月：处理层 v1.0 + 存储层 v1.0
- 第 10-12 月：应用层 v1.0 + 5 大红线验收

**实施效果**：
- 数据合规：100% 私有部署（**金融合规通过**）
- 自定义能力：完全可控（**支持 5 大度量 + 4 步闭环 + 6 大门禁**）
- 性能损耗：< 2%（**优于商业 SaaS**）
- 月成本：$10000（自建机房 + 4 人维护成本）

> **架构师视角**：**自研不是"成本高"——是"合规 + 定制"的必要投资**。金融 / 政府 / 强数据合规场景，**自研是唯一选择**。

| 风险 | 触发条件 | 后果 | 应对 |
|:-----|:---------|:-----|:-----|
| **1. SDK 性能损耗超 3%** | SDK 在主线程同步执行 | 用户报卡 | 异步化 + WorkManager |

| **4. oncall 告警疲劳** | 误报率高 | 团队关掉告警 | 严格阈值 + 告警分级 |
| **5. 真实异常漏报** | 多维度异常未检测 | 错过 P0 事故 | 多维度关联 + 异常检测 |

> **架构师视角**：**5 大风险中 #2（合规）和 #5（漏报）是"硬底线"——其他 3 项是工程权衡**。

# 10. 架构师视角 5 条 Takeaway

1. **APM 平台 = S10 度量学的工程化落地**——没有 APM 平台，**5 大度量 + 4 步闭环 + 6 大门禁 = 空中楼阁**（详见 [S10-Measure](../../02-Symptom/S10-Measure/01-症状机制.md) §3 / §5 / §6）。
2. **3 大模块 = 采集 / 后端 / 治理**——后端核心是 Kafka + Flink + ClickHouse，**端到端延迟 < 10s** 是硬指标。
3. **5 大红线 = 性能 / 合规 / 误报 / 漏报 / 成本**——**数据合规是法律底线，漏报是 oncall 底线**，其他 3 项是工程权衡。
4. **选型决策：50 人以下商业 SaaS，50-200 人 Sentry 自托管 + 自研，200+ 全面自研**——**数据合规是分水岭**。
5. **5 步落地法 = 选型 / 采集 / 后端 / 告警 / AI 协同**（**16 周 v1.0**）——复用 04-Tool 工具是捷径，**不要从 0 造数据采集轮子**。
