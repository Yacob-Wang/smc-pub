# 1. 4 级告警体系

> **铁律**：**告警不是越多越好** —— 没分级的告警 = 噪音 = 没人看

| 级别 | 含义 | 响应时长 | 通知方式 |
|:-----|:-----|:---------|:---------|
| **P0 阻断** | 严重事故 | 5 分钟 | 电话 + 飞书群 + 全员 |
| **P1 警告** | 重要问题 | 1 小时 | 飞书 + 短信 |
| **P2 通知** | 优化建议 | 1 天 | 邮件 + 飞书 |
| **P3 信息** | 趋势变化 | 1 周 | 邮件 + 周报 |

---

# 2. P0 阻断告警

## 2.1 触发条件

| 条件 | 阈值 |
|:-----|:-----|
| Crash-free Session | < 99.5% |
| ANR-free Session | < 99% |
| 启动成功率 | < 99% |
| Stability Score | 跌至 C 段（< 75）|
| Kernel Panic | 任意 1 次 |
| system_server 重启 | 任意 1 次 |

## 2.2 通知 SOP

```
00:00  告警触发
00:01  oncall 收到电话（强制接听）
00:03  oncall 确认收到（短信回复 "ACK"）
00:05  oncall 介入（拉群 + 抓数据）
```

**通知通道优先级**：
1. 电话（PagerDuty / 自研）
2. 飞书群（@oncall）
3. 短信（备用）
4. 邮件（兜底）

## 2.3 P0 必做 3 件

- 5 分钟内介入
- 30 分钟内给决策（修复 / 回滚 / 灰度）
- 24 小时内出 postmortem

# 3. P1 警告告警

## 3.1 触发条件

| 条件 | 阈值 |
|:-----|:-----|
| Crash-free Session | 99.5% - 99.9% |
| ANR-free Session | 99% - 99.5% |
| 单维度分 | 跌至 C 段 |
| 某机型崩溃率 | > 1% |
| Error Budget 消耗 | 50-80% |

## 3.2 通知 SOP

```
00:00  告警触发
00:05  oncall 收到飞书
01:00  oncall 必介入（如未响应升级）
```

**通知通道**：
1. 飞书群（@oncall）
2. 飞书机器人（IM 通知）
3. 短信（升级时）

## 3.3 P1 必做 3 件

- 1 小时内确认
- 24 小时内给决策
- 72 小时内出短期 Action Items

# 4. P2 通知告警

## 4.1 触发条件

| 条件 | 阈值 |
|:-----|:-----|
| 性能分 | 持续 7 天 < 60 |
| 兼容性 | 新机型通过率 < 90% |
| Error Budget 消耗 | 30-50% |

## 4.2 通知 SOP

```
00:00  告警触发
01d  oncall 收到邮件
07d  oncall review 后处理
```

**通知通道**：
1. 邮件
2. 飞书机器人（@团队）
3. 周报汇总

# 5. P3 信息告警

## 5.1 触发条件

- 趋势变化（如某指标 30 天下降 5%）
- 新版本发布对比
- 行业基线对比

## 5.2 通知 SOP

- 邮件 + 周报
- 季度 review 时处理

# 6. 告警路由

## 6.1 路由表

```yaml
# alertmanager.yml
route:
  receiver: 'default'
  group_by: ['alertname', 'app', 'severity']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  routes:
    # P0 阻断：电话 + 飞书
    - match:
        severity: P0
      receiver: 'pagerduty-critical'
      continue: true
    - match:
        severity: P0
      receiver: 'feishu-p0-group'
      
    # P1 警告：飞书 + 短信
    - match:
        severity: P1
      receiver: 'feishu-p1-group'
      continue: true
    - match:
        severity: P1
      receiver: 'sms-p1'
      
    # P2 通知：邮件
    - match:
        severity: P2
      receiver: 'email-p2'
      
    # P3 信息：周报
    - match:
        severity: P3
      receiver: 'weekly-report'

receivers:
  - name: 'pagerduty-critical'
    pagerduty_configs:
      - service_key: '<PAGERDUTY_KEY>'
    webhook_configs:
      - url: 'http://phone-gateway/call'
  - name: 'feishu-p0-group'
    webhook_configs:
      - url: 'http://feishu-webhook/p0'
```

## 6.2 路由决策树

```
告警触发
   │
   ├── severity: P0
   │     → 电话（5 秒内）
   │     → 飞书群（@oncall）
   │     → 30 秒后未确认 → 重拨
   │
   ├── severity: P1
   │     → 飞书群（@oncall）
   │     → 10 分钟未响应 → 升级到 TL
   │
   ├── severity: P2
   │     → 邮件
   │     → 飞书机器人
   │
   └── severity: P3
         → 周报
```

# 7. 告警抑制（避免噪音）

## 7.1 抑制规则

```yaml
# 抑制规则 1：相同指标的 P0 抑制 P1
inhibit_rules:
  - source_match:
      severity: 'P0'
    target_match:
      severity: 'P1'
    equal: ['alertname', 'app']

# 抑制规则 2：已知故障抑制
- source_match:
    alertname: 'SystemServerRestart'
  target_match:
    alertname: 'StabilityScore'
  equal: ['app']
```

## 7.2 静默规则

```yaml
# 静默：维护窗口
- matchers:
    - alertname=~"Maintenance.*"
  starts_at: '2026-08-01T02:00:00+08:00'
  ends_at: '2026-08-01T04:00:00+08:00'
  created_by: 'platform-team'
  comment: '定期维护'
```

## 7.3 去重规则

```yaml
# 5 分钟内相同告警只发 1 次
group_wait: 30s
group_interval: 5m
```

# 8. 告警度量

## 8.1 5 大告警指标

| 指标 | 公式 | 目标 |
|:-----|:-----|:-----|
| **告警准确率** | 真实告警 ÷ 总告警 | ≥ 80% |
| **MTTA** | 告警触发到首次响应 | P0 ≤ 5min / P1 ≤ 1h |
| **MTTR** | 告警触发到恢复 | P0 ≤ 30min |
| **告警疲劳** | 每天告警数 | ≤ 10 条 / 团队 |
| **漏报率** | 漏报 P0 数 ÷ 总 P0 数 | ≤ 5% |

## 8.2 告警健康度

| 等级 | 标准 |
|:-----|:-----|
| 优秀 | 准确率 ≥ 90% + 漏报率 ≤ 1% |
| 良好 | 准确率 ≥ 80% + 漏报率 ≤ 5% |
| 警告 | 准确率 < 80% 或漏报率 > 5% |
| 危险 | 每天告警 > 50 条 |

# 9. 真实案例

## 9.1 案例 1：告警疲劳

**问题**：每天 200+ 告警，oncall 麻木
**原因**：告警没分级，琐碎问题也 P0
**修复**：
- 重新分级（只有真正严重才 P0）
- 抑制规则
- 静默规则

**结果**：每天 8 条告警，oncall 满意度 +50%

## 9.2 案例 2：漏报

**问题**：P0 故障但 oncall 不知
**原因**：告警通道单一（只用飞书）
**修复**：
- 多通道（电话 + 飞书 + 短信）
- 30 秒未确认自动重拨
- 5 分钟未响应升级

**结果**：漏报率 5% → 0.5%

# 10. 8 反例

| # | 反例 | 错误做法 | 正确做法 |
|:-:|:-----|:---------|:---------|
| 1 | **没分级** | 所有都 P0 | **必 4 级** |
| 2 | **告警疲劳** | 每天 200+ | **必抑制 + 静默** |
| 3 | **单通道** | 只用飞书 | **必多通道** |
| 4 | **无升级** | oncall 不响应就完 | **必升级机制** |
| 5 | **不度量** | 凭感觉 | **必度量 5 大指标** |
| 6 | **告警无人 review** | 触发就完 | **每周 review** |
| 7 | **不抑制** | 触发 100 次 | **必抑制 + 去重** |
| 8 | **不静默** | 维护期也告 | **维护期必静默** |

---

# 11. 5 条 Takeaway

1. **4 级告警**（P0 阻断 / P1 警告 / P2 通知 / P3 信息）—— 没分级 = 噪音
2. **多通道**（电话 / 飞书 / 短信 / 邮件）—— 单一通道 = 漏报
3. **5 分钟响应 P0** —— 升级机制必带
4. **抑制 + 静默 + 去重** —— 防止告警疲劳
5. **告警健康度必度量** —— 准确率 ≥ 80% + 漏报率 ≤ 5%
