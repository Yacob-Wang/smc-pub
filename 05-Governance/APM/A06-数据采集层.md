# 1. APM 客户端核心问题

> **铁律**：**APM SDK 的性能损耗必须 ≤ 1%（CPU/内存/电量）**——否则变成"治理工具反而是问题源"

| 维度 | 性能损耗上限 |
|:-----|:-------------|
| CPU | < 1% |
| 内存 | < 5MB |
| 电量 | < 0.5% / 小时 |
| 网络 | < 1KB/s |
| 包大小 | < 500KB |

# 4. Layer 1-2：数据源 + 采集层

## 4.1 崩溃采集

### 4.1.1 Java 崩溃

```java
// Thread.setDefaultUncaughtExceptionHandler
Thread.setDefaultUncaughtExceptionHandler((thread, throwable) -> {
    String stack = StackTraceUtils.getStackTrace(throwable);
    APMCollector.report("je_exception", new CrashEvent(
        thread.getName(),
        throwable.getClass().getName(),
        stack,
        System.currentTimeMillis()
    ));
    // 调用原 handler
    defaultHandler.uncaughtException(thread, throwable);
});
```

**性能损耗**：0（崩溃瞬间）
**关键点**：必须先调原 handler（否则破坏系统默认行为）

## 4.2 性能采集

### 4.2.1 帧率

```java
// Choreographer + FrameMetrics
Choreographer.getInstance().postFrameCallback(frameTimeNanos -> {
    long cost = (System.nanoTime() - frameTimeNanos) / 1_000_000;
    APMCollector.report("frame_cost", new FrameEvent(cost));
    Choreographer.getInstance().postFrameCallback(this);  // 链式注册
});
```

**性能损耗**：< 0.5% CPU
**关键点**：必须链式注册，否则只触发 1 次

### 4.2.2 启动时间

```java
// 启动期打点
public class StabilityApp extends Application {
    private long coldStartTime;
    
    @Override
    public void onCreate() {
        super.onCreate();
        coldStartTime = SystemClock.uptimeMillis();
    }
    
    public void onFirstFrame() {
        long cost = SystemClock.uptimeMillis() - coldStartTime;
        APMCollector.report("cold_start", new StartEvent(cost));
    }
}
```

## 4.3 网络采集

```java
// OkHttp Interceptor
class APMInterceptor implements Interceptor {
    @Override
    public Response intercept(Chain chain) throws IOException {
        Request req = chain.request();
        long start = SystemClock.uptimeMillis();
        Response resp = null;
        try {
            resp = chain.proceed(req);
            return resp;
        } finally {
            long cost = SystemClock.uptimeMillis() - start;
            APMCollector.report("network", new NetEvent(
                req.url().toString(),
                resp != null ? resp.code() : -1,
                cost
            ));
        }
    }
}
```

**性能损耗**：< 0.3% CPU

# 6. Layer 4：上报层

## 6.1 批量上报

```java
class Reporter {
    private final BlockingQueue<Event> queue = new LinkedBlockingQueue<>(1000);
    
    public void report(Event event) {
        if (!queue.offer(event)) {
            // 队列满，丢最早的
            queue.poll();
            queue.offer(event);
        }
    }
    
    // 定时器每 30s 批量上报
    void flush() {
        List<Event> batch = new ArrayList<>();
        queue.drainTo(batch, 100);  // 一次最多 100 条
        if (!batch.isEmpty()) {
            httpClient.post("/apm/events", batch);
        }
    }
}
```

## 6.2 压缩 + 加密

```java
// 压缩（GZIP）
byte[] compressed = gzipCompress(jsonBytes);

// 加密（AES）
byte[] encrypted = aesEncrypt(compressed, publicKey);
```

## 6.3 失败重试 + 降级

```java
class Reporter {
    private int retryCount = 0;
    
    void sendWithRetry(byte[] data) {
        try {
            httpClient.post("/apm/events", data);
            retryCount = 0;  // 成功后重置
        } catch (IOException e) {
            retryCount++;
            if (retryCount <= 3) {
                // 指数退避
                Thread.sleep(1000 * (1 << retryCount));
                sendWithRetry(data);
            } else {
                // 降级：存本地
                localCache.save(data);
            }
        }
    }
}
```

# 7. 性能损耗控制 5 大策略

| # | 策略 | 节省 | 副作用 |
|:-:|:-----|:-----|:-------|
| 1 | **采样** | 90% 数据量 | 漏报少量事件 |
| 2 | **聚合** | 50% 网络 | 实时性差 |
| 3 | **压缩** | 70% 网络 | CPU |
| 4 | **限流** | 防止雪崩 | 关键事件可能被限 |
| 5 | **异步** | 不阻塞 UI | 内存累积 |

# 8. 真实案例

## 8.1 案例 1：自研 SDK 包大小控制

**目标**：< 500KB
**方法**：
- 拆 core + 可选模块
- 启用 R8 混淆
- 用 Protobuf 替代 JSON

**结果**：280KB（满足要求）

## 8.2 案例 2：网络数据降级

**问题**：低端机网络差，每秒 1KB 都不行
**方案**：
- 帧率数据 10% 采样
- 网络数据 1% 采样
- 启动期 60s 后才上报
- 网络断时存本地

**结果**：< 0.3KB/s

## 8.3 案例 3：内存累积

**问题**：SDK 内存泄漏 5MB
**方案**：
- 用 WeakReference
- 队列大小限制
- 主动 GC 触发

**结果**：< 2MB

# 9. 反例清单

| # | 反例 | 错误做法 | 正确做法 |
|:-:|:-----|:---------|:---------|
| 1 | **采所有数据** | 100% 全采 | **关键 100% / 边缘 1%** |
| 2 | **实时单条上报** | 每事件 HTTP | **批量 + 压缩** |
| 3 | **无失败重试** | 失败就丢 | **3 次重试 + 降级** |
| 4 | **无队列限制** | 内存无限增长 | **队列大小限制** |
| 5 | **阻塞 UI** | 同步处理 | **异步队列** |
| 6 | **不脱敏** | 隐私泄露 | **必脱敏** |
| 7 | **不压缩** | 流量爆炸 | **GZIP + Protobuf** |
| 8 | **不加密** | 中间人攻击 | **AES 加密** |
| 9 | **没降级** | 网络差就崩 | **本地缓存 + 降级** |
| 10 | **不删老代码** | SDK 体积膨胀 | **模块化 + 选采** |

# 10. 5 条 Takeaway

1. **4 类数据源**（崩溃/性能/网络/自定义）+ **性能损耗 ≤ 1%**
2. **SDK 4 层架构**（数据源/采集/处理/上报）—— 分层清晰
3. **采样是关键**（崩溃 100% / 帧率 10% / 网络 1%）
4. **批量 + 压缩 + 加密** —— 网络开销 < 0.3KB/s
5. **失败重试 + 降级** —— 防止 SDK 自身成为问题
