# 1. 5 大度量全景与依赖关系

## 1.1 为什么是这 5 个？

> **铁律**：稳定性架构师**不需要 50 个指标**——5 个就够覆盖 90% 决策场景。

| 度量 | 业务问题 | 决策权重 |
|:-----|:---------|:---------|
| **MTBF** | "平均多久崩一次？" | 长期健康度 |
| **崩溃率** | "今天有多少 Session 崩了？" | 短期告警 |
| **ANR 率** | "今天卡死了多少次？" | 体验红线 |
| **严重性** | "哪些崩溃最重要？" | 优先级排序 |
| **回归率** | "新版本是不是比旧版本差？" | 发版门禁 |

**5 度量的依赖**：

```
MTBF ──→ 崩溃率 ──→ 严重性
            ↓
          ANR 率 ──→ 严重性
            ↓
          回归率（横切所有）
```

## 1.2 5 度量的数据源

| 度量 | 客户端埋点 | 触发时机 | 上报通道 |
|:-----|:-----------|:---------|:---------|
| MTBF | Session 边界 | App 启动/退后台 | 实时流 |
| 崩溃率 | Session ID + 异常堆栈 | UncaughtException / NE 信号 | 实时流 |
| ANR 率 | ANR 检测回调 | Input 5s / Service 20s / Broadcast 10s | 实时流 |
| 严重性 | 影响用户数 | 后端聚合 | 离线批 |
| 回归率 | 版本号 + 指标 | 每次发版 | 准实时 |

# 2. MTBF（Mean Time Between Failures）

> **业务定义**：平均两次失败之间的间隔时间

## 2.1 三种计算方式

| 方式 | 公式 | 适用场景 | 优缺点 |
|:-----|:-----|:---------|:-------|
| **用户 MTBF** | 1 个用户两次崩溃间隔均值 | 长期健康度 | ✅ 反映用户体验 / ❌ 长尾用户影响大 |
| **设备 MTBF** | 1 个设备两次崩溃间隔均值 | OEM 视角 | ✅ 反映设备质量 / ❌ 装机量少时不准 |
| **Session MTBF** | 两次 Session 崩溃间隔均值 | App 稳定性 | ✅ 反映 App 质量 / ❌ 短 Session 难统计 |

**推荐**：**Session MTBF**（最贴近 App 视角，业界主流）

## 2.2 算法（伪代码）

```python
# 1. 取出所有 Session 及其崩溃状态
sessions = SELECT
    session_id,
    user_id,
    device_id,
    start_time,
    end_time,
    is_crashed
FROM session_table
WHERE start_time >= NOW() - INTERVAL '30 DAY';

# 2. 按 user_id 排序，计算相邻崩溃间隔
intervals = []
for user_group in sessions.groupby('user_id'):
    crashes = user_group[user_group.is_crashed == True]
    for i in range(1, len(crashes)):
        intervals.append(crashes[i].start_time - crashes[i-1].start_time)

# 3. 计算均值（小时）
mtbf_hours = mean(intervals) / 3600
```

## 2.3 ClickHouse SQL 实现

```sql
-- 30 天 Session MTBF（按用户聚合）
WITH user_crashes AS (
    SELECT
        user_id,
        toStartOfDay(start_time) AS crash_date,
        countIf(exception_type != '') AS crash_count
    FROM events
    WHERE event_type = 'session_start'
      AND start_time >= now() - INTERVAL 30 DAY
    GROUP BY user_id, crash_date
    HAVING crash_count > 0
)
SELECT
    avg(dateDiff('hour', lag(crash_date) OVER (PARTITION BY user_id ORDER BY crash_date), crash_date)) AS mtbf_hours
FROM user_crashes;
```

## 2.4 业界参考值

| 行业 | 优秀 | 平均 | 较差 |
|:-----|:----:|:---:|:----:|
| 社交 | > 200h | 50-200h | < 50h |
| 电商 | > 150h | 30-150h | < 30h |
| 工具 | > 300h | 100-300h | < 100h |
| 游戏 | > 100h | 20-100h | < 20h |

> **所以呢**：MTBF 一定要**按行业基线对标**，不要跟自家历史比（会自我感觉良好）。

# 3. 崩溃率（Crash Rate）

> **业务定义**：发生崩溃的 Session 占总 Session 的比例

## 3.1 关键设计决策：分子分母怎么去重？

| 设计 | 分子 | 分母 | 优缺点 |
|:-----|:-----|:-----|:-------|
| **严格版** ✅ | 1 个 Session 内多次崩溃只算 1 次 | 所有 Session | ✅ 反映用户感知 / **业界主流** |
| **宽松版** | 每次崩溃都算 | 所有 Session | ❌ 一个用户多次崩会让崩溃率虚高 |
| **设备版** | 设备级去重 | 设备日活 | ✅ OEM 视角 / ❌ 用户粒度粗 |

**推荐**：**严格版（用户感知对齐）**

## 3.2 Java + Native 合并算法

```sql
-- 30 天 Crash-free Session Rate（合并 JE + NE）
WITH session_crashes AS (
    SELECT
        session_id,
        countIf(event_type IN ('je_exception', 'ne_tombstone')) AS crash_count
    FROM events
    WHERE start_time >= now() - INTERVAL 30 DAY
    GROUP BY session_id
)
SELECT
    1 - (countIf(crash_count > 0) / count(*)) AS crash_free_session_rate
FROM session_crashes;
```

## 3.3 业界参考值

| 指标 | 优秀 | 行业 | 较差 |
|:-----|:----:|:----:|:----:|
| Crash-free Session | > 99.95% | 99.9% | < 99.5% |
| Crash-free User | > 99.99% | 99.95% | < 99.9% |

> **所以呢**：崩溃率 0.05% 提升到 0.01% 看似很小，但**意味着少了 80% 崩溃用户**（绝对数）。

## 3.4 告警规则

```yaml
# 告警规则示例（Prometheus 风格）
- alert: HighCrashRate
  expr: |
    1 - (
      countIf(event_type IN ('je_exception', 'ne_tombstone'), session_id != '')
      / countIf(event_type == 'session_start')
    ) < 0.999
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "崩溃率超过 0.1%"
```

# 4. ANR 率

> **业务定义**：发生 ANR 的 Session 占总 Session 的比例

## 4.1 三类 ANR 单独计算

| ANR 类型 | 触发条件 | 检测点 | 业界占比 |
|:---------|:---------|:-------|:---------|
| **Input** | 5s 无响应 | InputDispatcher | 60% |
| **Service** | 20s 无响应（前台）/ 200s（后台）| ActiveServices | 20% |
| **Broadcast** | 10s（前台）/ 60s（后台）| BroadcastQueue | 15% |
| **ContentProvider** | 10s | ContentProvider | 5% |

## 4.2 分桶算法

```sql
-- 三类 ANR 分别计算 7 天趋势
SELECT
    toDate(start_time) AS dt,
    countIf(anr_type = 'input' AND session_id IN (
        SELECT session_id FROM events
        WHERE event_type = 'session_start' AND start_time >= now() - INTERVAL 7 DAY
    )) AS input_anr,
    countIf(anr_type = 'service') AS service_anr,
    countIf(anr_type = 'broadcast') AS broadcast_anr
FROM events
WHERE event_type = 'anr'
  AND start_time >= now() - INTERVAL 7 DAY
GROUP BY dt
ORDER BY dt;
```

## 4.3 业界参考值

| ANR 类型 | 优秀 | 行业 | 较差 |
|:---------|:----:|:----:|:----:|
| ANR-free Session | > 99.9% | 99.85% | < 99.5% |
| Input ANR 占比 | < 30% | 50% | > 70% |
| Service ANR 占比 | < 10% | 20% | > 30% |

> **所以呢**：**Input ANR 占比 > 70% 一定是 InputDispatcher 卡了**（要查 Input 系列），Service ANR 占比 > 30% 一定是后台 service 太重。

# 5. 严重性评分（Severity Score）

> **业务定义**：单个崩溃问题的影响程度，用于优先级排序

## 5.1 5 维加权模型

```
severity_score = 0.30 × 频次分
              + 0.30 × 影响用户分
              + 0.20 × 业务损失分
              + 0.10 × 恢复时间分
              + 0.10 × 复发率分
```

### 5.1.1 频次分（0-100）

| 频次 | 分值 |
|:-----|:----:|
| > 10000 次/日 | 100 |
| 1000-10000 | 80 |
| 100-1000 | 60 |
| 10-100 | 40 |
| < 10 | 20 |

### 5.1.2 影响用户分（0-100）

| 影响用户 | 分值 |
|:---------|:----:|
| > 100 万 | 100 |
| 10-100 万 | 80 |
| 1-10 万 | 60 |
| 1000-1 万 | 40 |
| < 1000 | 20 |

### 5.1.3 业务损失分（0-100）

| 损失类型 | 分值 |
|:---------|:----:|
| 支付/交易链路崩溃 | 100 |
| 核心流程崩溃 | 80 |
| 边缘流程崩溃 | 60 |
| UI 渲染崩溃 | 40 |
| 后台崩溃 | 20 |

### 5.1.4 恢复时间分（0-100）

> **恢复时间 = P0 修复时长**（影响 SLA 评分）

| 恢复时间 | 分值 |
|:---------|:----:|
| < 1h | 100 |
| 1-4h | 80 |
| 4-24h | 60 |
| 1-7d | 40 |
| > 7d | 20 |

### 5.1.5 复发率分（0-100）

| 复发次数 | 分值 |
|:---------|:----:|
| > 5 次 | 100 |
| 3-5 次 | 80 |
| 2 次 | 60 |
| 1 次 | 40 |
| 0 次（新问题）| 20 |

## 5.2 优先级划分

| 严重性分 | 优先级 | 响应时长 |
|:---------|:------:|:---------|
| 80-100 | P0 | 1h |
| 60-79 | P1 | 4h |
| 40-59 | P2 | 24h |
| 20-39 | P3 | 1 周 |
| 0-19 | P4 | 1 月 |

## 5.3 算法实现

```python
def calculate_severity(crash_event):
    freq_score = freq_to_score(crash_event.daily_count)
    user_score = user_to_score(crash_event.affected_users)
    biz_score = biz_to_score(crash_event.business_path)
    recovery_score = recovery_to_score(crash_event.recovery_hours)
    recurrence_score = recurrence_to_score(crash_event.recur_count)

    severity = (
        0.30 * freq_score +
        0.30 * user_score +
        0.20 * biz_score +
        0.10 * recovery_score +
        0.10 * recurrence_score
    )
    return severity
```

# 6. 回归率（Regression Rate）

> **业务定义**：新版本相对基线版本的指标恶化程度

## 6.1 三种检测算法

| 算法 | 原理 | 适用场景 | 复杂度 |
|:-----|:-----|:---------|:-------|
| **差分法** | `new_rate - baseline_rate > threshold` | 灰度期 | O(1) |
| **时序法** | 移动平均 ± 3σ | 长期趋势 | O(n) |
| **统计检验** | T 检验 / Mann-Whitney U | 严格发布门禁 | O(n log n) |

## 6.2 推荐方案：T 检验

```python
from scipy import stats

def detect_regression(baseline_data, new_data, alpha=0.01):
    """T 检验检测回归：p < alpha 则拒绝原假设（新数据均值显著大于基线）"""
    t_stat, p_value = stats.ttest_ind(baseline_data, new_data)
    is_regression = (p_value < alpha) and (new_data.mean() > baseline_data.mean())
    return {
        't_stat': t_stat,
        'p_value': p_value,
        'is_regression': is_regression,
        'effect_size': (new_data.mean() - baseline_data.mean()) / baseline_data.mean()
    }
```

## 6.3 灰度发布门禁应用

| 阶段 | 样本量 | 检验方式 | 阈值 |
|:-----|:-------|:---------|:-----|
| **1% 灰度** | 1 万 | 差分法 | 崩溃率恶化 > 50% 立即回滚 |
| **10% 灰度** | 10 万 | T 检验 | p < 0.01 且 effect_size > 5% |
| **50% 灰度** | 50 万 | T 检验 | p < 0.001 且 effect_size > 2% |
| **全量** | 100 万+ | 持续监测 | Effect Size > 1% 持续 1 小时告警 |

## 6.4 行业基线

| 指标 | 灰度通过率 | 全量后回归率 |
|:-----|:----------:|:------------:|
| 启动成功率 | 95% | 1% |
| 崩溃率 | 92% | 2% |
| ANR 率 | 93% | 1.5% |

> **所以呢**：**回归率 1% 看似很低，但月发 4 次 = 月 4% 累积**，长期会拖垮稳定性。

# 7. 5 度量的 Flink 实时计算骨架

> **本节**：把 5 大度量串成一段 Flink 作业骨架

```java
DataStream<CrashEvent> crashes = env
    .addSource(new KafkaSource<>("crash-events"))
    .keyBy(event -> event.sessionId)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)));

// MTBF 计算
crashes
    .process(new MtbfCalculator())
    .addSink(new ClickHouseSink<>("mtbf_metrics"));

// 崩溃率计算
crashes
    .keyBy(event -> event.appVersion)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new CrashRateAggregator())
    .addSink(new ClickHouseSink<>("crash_rate_metrics"));

// 严重性评分（离线批）
crashes
    .keyBy(event -> event.exceptionHash)
    .window(TumblingEventTimeWindows.of(Time.hours(1)))
    .process(new SeverityScorer())
    .addSink(new KafkaSink<>("severity-events"));
```

# 9. 反例清单（避免踩坑）

| # | 反例 | 错误做法 | 正确做法 |
|:-:|:-----|:---------|:---------|
| 1 | **指标虚高** | 1 Session 多次崩溃都计入分子 | **严格去重**（1 Session 算 1 次）|
| 2 | **分母膨胀** | 把"启动尝试"当分母（容易被刷量）| 用"实际 Session"做分母 |
| 3 | **时区错乱** | UTC 算当日 | 用**用户本地时区** |
| 4 | **离线数据混入** | 把昨天 23:59 数据算进今天 | 用**事件时间** + 允许延迟 |
| 5 | **回归误报** | 1 个异常值就告警 | 用 **T 检验 + 持续时长** |
| 6 | **MTBF 长尾污染** | 含 0 天活跃用户拉低均值 | 过滤 `active_days >= 7` |
| 7 | **ANR 类型混淆** | 把 native 挂起当 ANR | **严格按 framework 检测** |
| 8 | **严重性主观** | 工程师凭感觉给 P0/P1 | **5 维加权 + 客观分** |
| 9 | **基线过期** | 6 个月前的基线当"正常" | **每季度刷新基线** |
| 10 | **只看绝对值** | 崩溃 1000 次 = 严重（用户 1 万 = 10%）| 看**比例 + 绝对值** |
| 11 | **告警风暴** | 每次指标波动都告警 | 用 **P2 警告 / P0 阻断**分级 |
| 12 | **无灰度基线** | 全量对比 1% 灰度（样本不足）| **同比例 + 同时段对比** |

---

# 10. 5 条 Takeaway

1. **5 大度量**（MTBF / 崩溃率 / ANR 率 / 严重性 / 回归率）覆盖 90% 决策场景——不要被 50 个指标淹没
2. **崩溃率必须 Session 去重**（1 Session 多次崩溃只算 1 次）——否则分母虚高
3. **ANR 必须 3 类分开算**（Input / Service / Broadcast）——占比是定位的关键
4. **严重性 5 维加权**（频次 30% + 用户 30% + 业务 20% + 恢复 10% + 复发 10%）——避免主观判断
5. **回归检测用 T 检验**（不要只看差分）——避免灰度期误回滚
